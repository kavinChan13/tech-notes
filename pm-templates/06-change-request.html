Change Request + CCB 变更请求

"不口头答应"是 PgM 最重要的纪律 · 所有变更走 CR 流程留痕

Cadence事件触发（每个变更 1 份 CR）
AudienceCCB 成员 + Sponsor + 受影响 stakeholder
Lifetime永久存档（合规留痕）
Visibility公开（CCB 决议后）

01是什么 / 为什么

CR (Change Request) 是项目变更的"合规通道"。任何对 Charter 范围 / 进度 / 预算 / 关键质量标准的修改都必须走 CR：

  • 客户加范围 → CR
  • 关键人员变更 → CR
  • 预算追加 → CR
  • Assumption Broken → CR（来自 RAID Log A）
  • 关键路径上的活动估算翻倍 → CR
没 CR 的变更，等于没发生 —— 最终在审计 / 复盘 / 客户对账时炸雷。

02CR 必备字段

字段为什么存在
CR ID + 版本唯一编号 + 项目内有序追踪
提出人 + 日期谁提出、什么时候
变更类型范围 / 进度 / 预算 / 质量 / 关键人 / 其他
变更内容1 句话变更描述（必须具体）
提出原因为什么需要这个变更
影响分析进度（±天）/ 成本（±钱）/ 范围 / 风险 / 质量 5 维度
建议方案给 CCB 至少 3 个选项（接 / 部分 / 拒绝）
不批准的后果明示风险 —— 最容易被忽略
CCB 决议批准 / 拒绝 / 推迟 / 部分批准 + 决议日期
新基线版本号批准后基线递增（Charter v1 → v2）

03CCB 节奏建议

项目规模常规 CCB紧急 CR
小（< 10 人）每周 30 分钟（PM + Sponsor + 关键 stakeholder）异步邮件批准
中（10-50 人）双周 1 小时Async approval < 24h
大（> 50 / 合规重）双周 1 小时 + 紧急流程2h 紧急会议 + Sign-off 留痕

📝填好示例

Project Apex 第 6 周触发的真实 CR。

Change Request · CR-007 · Project Apex

Status: Approved · Submitted by: Tomás (CSM Head) · Date: 2026-06-12 · Type: 范围 + 预算

1. 变更内容

客户 X 要求在 NIS2 合规升级同时，额外纳入"客户专属审计 dashboard"，使其能自服务查看每日合规事件。

2. 提出原因

  • 客户 X CTO Pavel 在 5/22 现场会议上提出（属于商务让利后的反向条件）
  • 客户 X 内部合规审计要求 7×24 自查能力
  • 客户 W 听说后也表示如果给 X 就要给 W（"Equal Treatment"）

3. 影响分析

维度当前基线变更后偏差
进度Go-Live 10/31Go-Live 11/15+15 天
成本€2.4M€2.7M+€300K
范围合规升级 + 4 客户迁移+ 自服务审计 dashboard ×2 客户（X, W）+ 1 个工作包
关键路径1.1.2 → 1.5 → 1.3.11.1.2 → 1.5 → 1.3.1 → 1.6 (新增)+15 天关键路径
风险R-001 ~ R-005新增 R-006 多客户对等待遇+1 风险
质量P99 +5% 阈值不变—

4. 建议方案

方案 A：全接受（X + W 都做）

  • 影响：+15 天 + €300K
  • 优点：保住客户 X 的 9/30 sandbox 接入；同时给 W 一个升级理由（巩固关系）
  • 缺点：超 PM 授权范围（>5% 预算 + >1 周延期）；触发 Charter v1 → v2

方案 B：部分接受（仅 X 做，W 推到 v2）

  • 影响：+10 天 + €180K
  • 优点：商务承诺兑现；预算可控
  • 缺点：W 可能也要求商务让利，引发连锁反应

方案 C：拒绝（X 接受承诺但等 v2）

  • 优点：守住原基线 + 原预算
  • 缺点：客户 X 可能再次拒绝 9/30 sandbox（回到 I-001 状态）；€80M 年合同存在风险

推荐：方案 A。理由：①保住客户 X 商务承诺；②给客户 W 提供"高价值客户待遇"，反而加固关系；③ €300K 在储备金 €360K 内（部分），剩余从下季度预算调拨。

5. 不批准的后果

  • 客户 X 会在 9/30 sandbox 验收时索要"等价代偿"，最坏情况退订（损失 €25M / 年）
  • 客户 W 听到 X 被拒会反过来怀疑承诺真实性
  • NIS2 审计无法获得 Tier 2 推荐认证（影响下一轮采购）

⚖️DO & DON'T

✅ DO

  • 所有变更走 CR，无一例外
  • 影响分析 5 维度（进度 / 成本 / 范围 / 关键路径 / 风险 / 质量）
  • 至少给 CCB 3 个选项
  • "不批准的后果"必填
  • 批准后立刻刷新基线版本号

❌ DON'T

  • 不要口头答应 — 最容易翻车
  • 不要"积攒一堆 CR 一次过审" — 时机错过
  • 不要 CR 写完不通知受影响 stakeholder
  • 不要在 PRD 里悄悄改范围 — 必须走 CR
  • 不要 CR 决议后不更新 Charter / WBS / Schedule
