01

编排解决什么

从"一堆 crontab"到"可靠流水线"

数据管道由许多有依赖关系的任务组成：抽数 → 清洗 → 聚合 → 产出报表。最原始的做法是一堆 crontab 定时脚本，但它很快失控：任务 B 依赖 A 完成，可 cron 只会「到点就跑」不管 A 好没好；A 失败了 B 拿到脏数据；某天挂了想补跑三天前的数据不知从何下手；出错没人知道。 编排器就是来解决这些的。

⛓ 依赖管理

用 DAG 显式表达「谁依赖谁」，上游成功才跑下游。

🔁 重试与回填

失败自动重试、可精确重跑某个历史时间段（backfill）。

👁 可观测

每次运行有状态、日志、耗时、告警；一眼看清管道健康。

生活类比：菜谱工序 vs 定闹钟

用 cron 做数据管道，就像做一桌菜只会「定闹钟」：8:00 切菜、8:10 下锅、8:20 装盘——可万一切菜慢了，闹钟一响照样下锅（下游拿到没准备好的数据）。编排器像一个会看工序依赖的厨师长：装盘一定等炒好、炒一定等切好；哪道工序烧糊了会重做并叫你，还能「把昨天那道菜按同样工序重做一遍」（回填）。

映射：工序依赖=DAG · 等上一步好=依赖触发 · 重做=重试/回填 · 叫你=告警
02

DAG 与调度器

有向无环图是数据管道的通用语言

DAG（有向无环图）：节点是任务，边是依赖，无环保证不会循环等待。编排器按 DAG 拓扑顺序调度：一个任务的所有上游成功后它才变为可运行。

调度器核心概念

  • Task / Operator：一个工作单元（跑 SQL、调 Spark、传文件）。
  • Schedule：触发节奏（cron 表达式 / 数据到达 / 手动）。
  • Run（DAG Run）：一次具体执行实例，绑定一个逻辑数据区间（如「2026-07-12 这一天的数据」）。
  • 依赖与并发：上游成功才跑下游；同一 DAG 不同时间区间可并行，需控并发与资源池。
  • 状态机：每个 task 有 queued/running/success/failed/retry/skipped 等状态，可视化追踪。
关键区分「逻辑执行时间」与「真实运行时间」：处理「7 月 12 日」的数据，可能实际在 7 月 13 日凌晨才跑。任务逻辑必须基于逻辑区间参数（如 {{ ds }}），绝不能用 now()——否则重跑历史会算错、结果不可复现。这是数据编排最容易踩的坑。
03

ETL vs ELT

湖仓时代的范式反转

传统 ETL：抽取（Extract）→ 在独立引擎里转换（Transform）→ 加载（Load）进数仓。现代 ELT：先加载原始数据进数仓/湖仓，再用数仓强大的算力在库内转换。

维度ETL（传统）ELT（现代）
转换位置数仓外的独立 ETL 引擎数仓/湖仓内（SQL 算力）
原始数据转换后才入仓，原始不留原始先入仓，可随时重转
灵活性改需求要重跑抽取原始还在，改逻辑重跑转换即可
算力依赖ETL 集群依赖数仓弹性算力（Snowflake/BQ/StarRocks）
典型工具Informatica、KettleFivetran/Airbyte 抽取 + dbt 转换
ELT 兴起的根因：云数仓/湖仓算力变得又便宜又弹性，「把转换放进数仓用 SQL 做」比维护独立 ETL 引擎更简单、更灵活、更易协作。先存原始数据还带来可回溯性——需求变了重跑转换即可，不用重新抽数。
04

dbt：让 SQL 变成软件工程

ELT 里 "T" 的事实标准

dbt（data build tool）把「写一堆 SQL 转换脚本」变成规范的软件工程：每个模型是一个 SELECT，dbt 负责推导依赖、按序执行、建表/视图、测试、生成文档与血缘。

dbt 核心能力

  • 模型（model）：一个 .sql 文件 = 一个 SELECT；dbt 负责物化成表/视图/增量表。
  • ref() 自动建依赖：模型间用 {{ ref('stg_orders') }} 引用，dbt 据此推出 DAG、决定执行顺序——不用手写依赖。
  • 测试：内置 unique/not_null/relationships/accepted_values + 自定义断言，数据质量测试即代码。
  • 增量模型：只处理新增/变化数据，省算力；配 is_incremental()。
  • 文档 + 血缘：自动生成字段级文档与依赖图；版本化（Git）+ CI 可测试。
dbt 像给 SQL 装上了「编译器 + 单测 + 依赖管理」

过去写数仓 SQL 像手抄一叠彼此引用的稿子：改一处得手动找出所有受影响的、还得记住先跑哪个后跑哪个。dbt 像给这堆 SQL 装了个构建系统（类比代码里的 Maven/npm）：声明 ref 依赖，它自动算出编译顺序、增量构建、跑测试、出文档。SQL 从「一次性脚本」升级成「可维护的工程资产」。

05

调度语义：幂等与回填

管道可靠性的地基

数据管道一定会失败、一定要重跑（补数据、修 bug、加历史）。所以每个任务必须幂等——同一区间跑一次和跑十次结果相同，否则重跑就产生重复/错误数据。

幂等的实现

  • 分区覆盖写（overwrite partition）：按逻辑日期分区，重跑时先删该分区再写（INSERT OVERWRITE），而非追加——最常用的幂等手段。
  • MERGE / Upsert：按主键合并，湖仓表（Iceberg/Hudi）支持。
  • 绝不用 now()：一切基于逻辑区间参数，保证可复现。

回填（Backfill）

  • 定义：为过去的时间区间重跑管道（新上线要灌历史、修了 bug 要重算、补漏跑的天）。
  • 前提是幂等：否则回填会叠加出错误数据。
  • 要控并发：一次回填几百天别把集群/数仓打爆，限制并行区间数。
  • Airflow catchup、dbt 增量重建都服务于回填场景。
06

数据感知调度

从"按时间触发"到"按数据就绪触发"

传统调度是时间驱动（cron：到点就跑）。但真实依赖的是「数据到齐了没」而非「几点了」。新一代编排强调数据感知（data-aware）/ 资产（asset）调度：以「数据资产」为一等公民，上游资产更新后自动触发下游。

范式触发依据代表
任务导向（时间）cron 到点 / 上游 task 成功经典 Airflow DAG
数据/资产导向数据资产就绪/更新Dagster（software-defined assets）、Airflow Datasets
资产导向的好处：你声明「报表依赖哪些表」，系统自动维护「哪张表变了就重算哪些下游」，还天然知道每份数据的血缘、新鲜度（freshness）与物化状态。它把关注点从「怎么调度任务」转到「保证数据资产正确、及时」——更贴近数据工程的本质。
07

数据契约与质量门

把质量左移到管道里

管道跑成功 ≠ 数据正确。要在编排里嵌入质量检查，坏数据不该流向下游。

手段

  • 数据契约（data contract）：上下游约定 schema、字段含义、SLA、允许的取值——破坏即告警/拦截（呼应 Schema Registry）。
  • 质量门（quality gate）：转换后跑断言（行数不为 0、主键唯一、空值率阈值、金额非负），不达标阻断下游——dbt test / Great Expectations / Soda。
  • 阻断 vs 告警：核心表不达标就阻断，边缘表告警放行——按数据重要性分级，别一刀切卡死发布。
  • 新鲜度检查：数据是否按时到达（freshness），迟到即告警。

完整的质量六维、血缘与治理平台见 数据治理与平台工程。

08

编排工具选型

没有最好，只有最合适
工具定位特点 / 适用
Airflow通用任务编排生态最大、Operator 丰富；任务导向、Python 定义 DAG；重运维
Dagster数据资产编排软件定义资产、强类型、内建血缘/测试；开发体验好
dbt数仓内转换只管「T」，需配调度器触发；SQL 工程化事实标准
Prefect轻量 Python 编排动态工作流、Python 原生、上手快
DolphinScheduler可视化任务调度国内常用、可视化拖拽、适合传统数仓团队

典型组合

  • Airbyte/Fivetran（EL）+ dbt（T）+ Airflow/Dagster（调度） 是现代 ELT 栈的黄金三角。
  • 调度器负责「何时、按什么依赖触发」，dbt 负责「库内怎么转换 + 测试」，各司其职。
  • 选型看团队：SQL 强 + 云数仓 → dbt 为核心；复杂异构任务 → Airflow；重数据血缘/资产 → Dagster。
09

checklist

数据管道工程速查

  • 任务全部幂等（分区覆盖写 / Upsert），基于逻辑区间参数，禁用 now()。
  • 依赖用 DAG/ref 显式声明，别靠「错开时间」隐式假设上游好了。
  • 失败可重试 + 可精确回填历史区间，且回填有并发上限。
  • 转换后跑质量门（dbt test / GE），核心表不达标阻断下游。
  • 定义数据契约与新鲜度 SLA，破坏即告警。
  • 管道可观测：运行状态、耗时、血缘、失败告警齐全。
  • 转换逻辑版本化 + CI（dbt + Git），像代码一样评审与测试。
一句话记忆：编排器管依赖与调度，dbt 管库内转换与测试，幂等 + 回填是可靠地基，质量门 + 数据契约让坏数据止步于管道内。
