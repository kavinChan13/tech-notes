数据 × AI 桥梁

2026 年，大数据与 AI 的边界正在消失：模型再强，喂进去的数据才是天花板。这一篇站在数据工程师的视角，打通通往 AI 的四座桥：特征平台（Feature Store）——解决「训练用的特征和线上服务的特征不一致」这一 ML 头号坑；向量与向量数据库——把文本/图像变成 embedding，做相似检索；RAG 数据管道——给大模型外挂知识库的工程链路；Lakehouse for ML——用湖仓的版本/时间旅行做可复现的训练数据。你会发现：这些「AI 基础设施」的内核，依然是你熟悉的批流处理、存储、一致性、数据质量。

数据是 AI 真正的瓶颈

模型开源人人能用，但数据与数据基础设施才是护城河

业界共识：「Garbage in, garbage out」在 AI 时代被放大了。模型架构（见 ai-native/transformer_llm_architecture.html）日益标准化、开源，真正拉开差距的是数据的质量、规模、新鲜度，以及把数据可靠地喂给模型的基础设施。这恰恰是大数据工程师的主场——你在 D01–D12 学的存储、批流、治理、服务层，正是 AI 数据基建的底座。

如果说大模型是一位天赋异禀的厨师，那数据就是食材，数据基础设施是供应链与厨房。再好的厨师，食材不新鲜、配送混乱、训练时尝的味道和上菜时不一样，照样做不出好菜。数据工程师干的，就是把这条「从田间到餐桌」的供应链建好——这活儿在 AI 时代不是变少了，而是更关键了。

01

特征平台 Feature Store

解决 ML 工程头号坑：训练和线上用的特征对不上

机器学习模型吃的不是原始数据，是特征（feature）——「用户近 7 天下单数」「商品 30 天点击率」这类加工过的信号。特征工程本质就是你熟悉的 SQL 聚合 + 窗口（D05）。但 ML 落地时有个臭名昭著的坑——训练-服务偏差，特征平台就是为治它而生。

1.1 训练-服务一致性（Train-Serve Skew）

训练-服务偏差（Train-Serving Skew）：训练时用「离线批计算」算特征（如用 Spark 跑历史数据），线上预测时用「另一套实时代码」算同名特征。两套代码逻辑稍有不同（口径、时间窗口、null 处理），模型线上效果就崩——而且极难排查。这是 ML 工程最隐蔽的事故之一。
就像备考时用的练习题和真正考试的题出题口径不一样：你练得再好，一上考场发现题型变了，照样考砸。训练用的特征 = 练习题，线上服务的特征 = 考题，两者必须同源同口径，模型才靠谱。
Point-in-Time Correctness（时间点正确性 / 防数据穿越）：构造训练样本时，每条样本的特征只能用该样本时间点之前就已知的数据。若不小心用了「未来」的信息（比如用今天的总下单数预测昨天的行为），就是特征穿越/数据泄露——离线评估好得离谱、上线一塌糊涂。特征平台用 point-in-time join 自动避免它。这与数仓的「拉链表/快照」（D02）、湖仓「时间旅行」（D11）是同一类「按时间点取正确版本」的问题。

1.2 离线 + 在线双存储：一次定义，两处一致供给

特征平台的核心：同一份特征定义同时物化到「离线存储（训练）」和「在线存储（推理）」，从根上消除训练-服务偏差。在线存储正是 D12 的服务层。

一致性

一处定义、两处物化，训练与线上特征同源同口径。

复用

特征注册成「资产」可被多模型/团队复用，免重复造轮子。

新鲜度

实时特征经 Flink（D04）算好写在线存储，秒级更新供推理。

代表实现：Feast（开源）、Tecton、Databricks Feature Store。它们底层正是你熟悉的组件——离线存储 = 湖仓/数仓（D11/D02），在线存储 = Redis/HBase（D12），实时特征 = Flink/Spark Streaming（D03/D04）。特征平台不是新魔法，而是把这些组件按「ML 特征供给」这一目标编排起来。
02

向量与向量数据库

把万物变成坐标 · 用「距离近」表示「语义像」

2.1 Embedding：把文本/图像变成一串数字

Embedding（嵌入向量）是 AI 时代的「数据新形态」：用一个模型把一段文本、一张图、一个商品映射成一个高维向量（如 768 维），使得语义相近的东西，向量距离也近。于是「找相似」就变成了「找向量空间里最近的邻居」。

想象把每个词/句子放到一张巨大的地图上：「猫」和「狗」离得近（都是宠物），离「汽车」远。Embedding 就是这张地图的坐标。「找语义相似」= 在地图上找距离最近的点——这就是向量检索。
相似度度量：常用余弦相似度（cosine）（看方向夹角，最常用于文本）或欧氏距离 / 内积。向量库就是为「在亿级向量里快速找 Top-K 最近邻」而优化的存储。

2.2 ANN：在亿级向量里「足够快地」找最近邻

暴力比对每个向量（精确最近邻）在亿级规模下太慢。实际用近似最近邻（ANN）——牺牲一点精度换巨大的速度。点下面的画布任意位置放一个「查询点」，看它的 K 个最近邻被高亮：

点击画布任意处放一个「查询向量」，绿色高亮的就是离它最近的 K 个邻居——这正是向量检索在做的事（只是真实是几百维、上亿个点）。
ANN 索引算法：HNSW（分层小世界图，查询快、最常用）、IVF（倒排聚类，先定位簇再搜）、PQ（乘积量化，压缩向量省内存）。它们都在「召回率 / 速度 / 内存」之间取舍——又是老兵熟悉的工程权衡。
向量库 / 方案定位适用
Milvus / Zilliz专用向量数据库，规模大亿级以上向量、独立向量检索服务
pgvectorPostgres 扩展已有 PG、向量规模不大，省一套系统
FaissMeta 的库（非服务）嵌入应用内、自己管索引
Elasticsearch / OpenSearch搜索引擎加向量已有 ES、要「关键词 + 向量」混合检索
云托管Pinecone / 各云向量服务免运维、快速起步
03

RAG 数据管道

给大模型外挂一个「随时可查的知识库」· 数据工程的活

RAG（检索增强生成）是当下企业落地大模型最主流的方式：不重训模型，而是在回答前先去知识库检索相关资料，连同问题一起喂给模型，让它「开卷答题」。原理深探见 ai-native/rag_deep_dive.html；这里讲构建 RAG 知识库的数据管道——这正是数据工程师的活。

RAG 的「离线建库」就是一条数据管道：文档抽取 → 切分 chunk → 调 embedding 模型 → 写向量库。数据工程的 ETL 思维直接复用。
Chunking（切分）：把长文档切成适合检索的小块（按段落/标题/固定 token 数，常带重叠）。切太大→检索不精准、塞不进上下文；切太小→丢失语境。切分策略 + 元数据（来源、标题）直接决定 RAG 质量——这是数据预处理的活，做得好坏差距巨大。
数据质量对 RAG 是生死攸关的：知识库里的脏数据、过期文档、重复内容，会被原样检索出来喂给模型 → 模型一本正经地胡说（基于错误资料）。所以 D09 的数据质量、血缘、新鲜度治理，在 RAG 场景不是锦上添花，而是底线。「检索到的就是答案的依据」，垃圾进、垃圾出被无限放大。
RAG 数据管道要持续运转：文档更新 → 增量 re-embedding → 更新向量库（呼应 D06 增量/CDC 思想）。embedding 模型升级时还要全量重建库。这又是一条需要编排、调度、监控（D06 Airflow）的数据流水线。
04

Lakehouse for ML：可复现的训练数据

湖仓的版本/时间旅行，正是 ML 复现性的答案

ML 工程的另一个老大难是可复现性：「三个月前训练出那个效果好的模型，用的是哪一版数据？」如果训练数据是一堆被反复覆盖的文件，根本说不清。湖仓表格式（D11）的快照 / 时间旅行 / 数据版本恰好解决这个问题。

数据版本 = 可复现

训练时记录用的是哪个快照 id（D11 time travel）。要复现/对比实验，按快照重新拉那一版数据即可。配合模型版本，实验完全可追溯。

统一存储训练数据

特征、标签、原始数据都在湖仓，用 Spark（D03）大规模准备训练集，省去专门的数据搬运。

DataOps × MLOps

数据管道（DataOps，D09）与模型管道（MLOps）在湖仓交汇：数据质量门、血缘一路延伸到模型，问题可端到端追溯。

多模态与非结构化

图片/音频/文本等非结构化数据也进湖（对象存储 + 元数据表），统一管理，喂给多模态训练。

数据血缘延伸到模型：D09 讲的元数据血缘，在 AI 场景要继续往下游连——「这个模型用了哪些表的哪些特征、哪个数据版本」。一旦上游某张表出问题，能立刻评估「影响了哪些模型、哪些线上预测」。这是 AI 治理（呼应 ai-native/ai_safety.html / ai_eval.html）的数据基础。
一句话收束：AI 基础设施的「数据侧」，本质是大数据基础设施的延伸与特化。特征平台 = 数仓建模 + 服务层；RAG 建库 = ETL 管道；训练数据管理 = 湖仓版本治理。数据工程师转 AI 数据方向，不是从零学起，而是把已有能力投射到新场景——这正是当下最具性价比的能力迁移。
05

小结 · 面试 · 术语表

收成判断力

一页纸总结

① 数据是瓶颈

模型趋同，数据质量/规模/新鲜度 + 基础设施才是护城河，是数据工程师的主场。

② 特征平台

一份定义→离线(训练)+在线(推理)双物化，治训练-服务偏差与时间点正确性。

③ 向量检索

Embedding 把语义变坐标，ANN(HNSW/IVF)在亿级里找最近邻，权衡召回/速度/内存。

④ RAG 管道

离线建库=ETL(抽取/切分/embedding/入库)；数据质量决定 RAG 生死。

⑤ Lakehouse for ML

快照/时间旅行=可复现训练数据；血缘延伸到模型，DataOps×MLOps 交汇。

⑥ 能力迁移

AI 数据基建是大数据基建的延伸，已有能力直接投射，性价比最高。

面试自测（点击展开）

基础什么是训练-服务偏差？特征平台怎么解决？
问题训练用离线代码算特征、线上用另一套实时代码，口径不一致 → 模型线上效果崩。
解法特征平台「一份定义、离线+在线双物化」，保证两处同源同口径。
附带还用 point-in-time join 防数据穿越（只用样本时间点之前已知的数据）。
核心为什么向量检索要用 ANN 而不是精确最近邻？
规模亿级、几百维向量，暴力逐个算距离太慢，线上扛不住。
ANNHNSW/IVF/PQ 用图/聚类/量化，牺牲少量召回率换巨大速度提升。
权衡在召回率、延迟、内存间取舍——和所有索引一样的工程权衡。
高频构建一个 RAG 知识库，数据侧要做哪些事？
管道文档抽取 → 切分 chunk（含重叠/元数据）→ 调 embedding → 写向量库。
质量去重/去过期/清洗——脏数据会被检索出来喂模型致幻觉。
运维文档更新需增量 re-embedding；embedding 模型升级要全量重建。需编排调度（Airflow）。
架构湖仓（Iceberg/Delta）怎么帮助 ML 可复现？
快照训练记录用的数据快照 id，按 time travel 可重拉那一版（D11）。
血缘D09 血缘延伸到模型：知道模型用了哪些表/特征/数据版本，上游出问题可评估影响。
统一特征/标签/非结构化都进湖，Spark 大规模准备训练集，DataOps×MLOps 交汇。

术语表速查 · Glossary

① 特征

特征 Feature喂给模型的加工信号（如近7天下单数）。特征工程本质是 SQL 聚合+窗口。
特征平台 Feature Store一份特征定义同时物化到离线(训练)与在线(推理)存储，保证一致、可复用、新鲜。Feast/Tecton。
训练-服务偏差训练与线上用不同代码算同名特征致口径不一致，模型线上崩。特征平台治本。
Point-in-Time Correctness构造样本只用其时间点之前已知的数据，防数据穿越/泄露。

② 向量

Embedding 嵌入向量把文本/图像映射为高维向量，使语义相近者向量距离近。
向量数据库为亿级向量的 Top-K 最近邻检索优化的存储。Milvus/pgvector/Faiss。
ANN 近似最近邻牺牲少量精度换速度的最近邻检索。HNSW(图)/IVF(聚类)/PQ(量化)。
余弦相似度用向量夹角衡量语义相似度，文本检索最常用。

③ RAG 与 ML 数据

RAG 检索增强生成回答前先检索知识库，把相关资料连同问题喂给 LLM，「开卷答题」。
Chunking 切分把长文档切成适合检索的小块（带重叠+元数据），策略直接影响 RAG 质量。
Lakehouse for ML用湖仓快照/时间旅行做可复现训练数据，血缘延伸到模型。
DataOps × MLOps数据管道与模型管道在湖仓交汇，质量门/血缘端到端打通。
