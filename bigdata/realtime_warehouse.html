为什么要实时数仓

T+1 隔天看数据，越来越不够用 · 实时决策等不了

传统离线数仓（D02）是 T+1 的：今天的数据，凌晨跑批，明天才能在报表里看到。但越来越多场景等不了一天：双 11 实时大屏、风控拦截、实时推荐、库存预警、运营实时调价……实时数仓就是把数仓的"产出节奏"从天级压到秒级。

离线数仓像每月一次的体检报告——全面、准确，但你拿到时身体状况已是一个月前的。实时数仓像手腕上的实时心率手表——此刻心跳多少立刻知道，能当场做决定（该歇了 / 该冲刺了）。两者不是替代关系：体检（离线，准）和心率表（实时，快）各管一摊、互为补充。

实时数仓（Real-time Data Warehouse）：以流处理为核心、能在秒级 ~ 分钟级产出分析结果的数仓体系。它沿用离线数仓的建模与分层思想（D02），但把"批加工"换成"流加工"（Flink），把"批存储"换成"可实时写入 + 实时查询"的载体（Kafka 中间态 + OLAP 引擎做服务层）。核心命题：在保证可接受的延迟下，尽量逼近离线的准确性。
本篇是系列的"架构集成篇"，几乎不引入全新的底层技术，而是把 D04 Flink、D06 Kafka/CDC、D07 OLAP 组合成一套能跑的实时数据平台。建议先有那几篇的基础。
01

Lambda 架构：批流双链路

实时数仓的第一代答案 · 强大但维护成本高

Lambda 架构是实时数仓最早、最经典的范式。思路很直接：批和流各跑一条链路，再合并——批层保证最终准确，流层保证即时但近似。

Lambda 架构三层：批处理层（Batch Layer）——用 Spark 等跑全量，结果准确但慢（T+1）；速度层（Speed Layer）——用 Flink 等处理增量实时数据，快但只是近似/增量；服务层（Serving Layer）——查询时把批层的"历史准确结果"和速度层的"今天实时增量"合并返回。批层会不断用全量重算覆盖速度层的近似值，保证最终一致。
同一份数据走两条链路：批层慢而准、速度层快而近似，服务层合并。批层定期用全量重算"修正"速度层的近似。
Lambda 的致命痛点：两套代码。 同一个指标（如 GMV），要在批（Spark SQL）写一遍、在流（Flink）再写一遍。两套逻辑必须严格对齐，否则实时和离线对不上数——而它们用不同引擎、不同语言、不同算子，对齐极难。维护翻倍、口径易漂、改一处要改两处。这正是 Kappa 架构想消灭的问题。
02

Kappa 架构：一切皆流

用一条流链路干掉双套代码 · 重算靠"重放"

2.1 一切皆流

Kappa 架构：干脆只保留流处理一条链路，砍掉批层。一切数据都先进 Kafka（可重放的日志），流处理引擎（Flink）消费产出结果。需要"重算"（改了逻辑、修了 bug）时，不另跑批，而是把 Kafka 的历史数据重新消费一遍（重放 replay）——用同一套流代码重跑历史。一套代码、一种语义，从根上消灭 Lambda 的双链路痛点。
Lambda 像同时养两个厨师：一个做"标准大餐"（批，准但慢）、一个做"快手菜"（流，快但糙），还得保证两人手艺一致。Kappa 说：只留快手菜厨师，要重做历史菜单就把老订单重新下一遍给他做——一个厨师、一本菜谱，省心，但"重放整月订单"会让他忙一阵（重算成本）。
维度LambdaKappa
链路批 + 流 双链路仅流单链路
代码维护两套（易不一致）一套
重算方式批层全量重跑重放 Kafka 历史
重算成本 / 历史擅长全量回算依赖消息长期保留，重放大历史较重
复杂度高（两套+合并）较低

03

实时数仓的分层

离线怎么分层，实时也怎么分层 · 只是把"批加工"换成"流加工"

实时数仓不是另起炉灶，它沿用 D02 的分层思想（ODS→DWD→DWS→ADS），只是每一层的"加工方式"和"载体"变了——从离线批表，变成 Kafka topic + Flink 流作业 + OLAP 表。

分层模型与离线一致（D02），但"表"变成了"流动的 Kafka topic"，"加工"变成了 Flink 作业，最终结果落到 OLAP 引擎（D07）供实时查询。
实时与离线的关系（实时离线一体）：成熟团队追求"一套模型、一套口径，两种时效"——同一个指标定义，离线用 Spark 跑全量保准确、实时用 Flink 跑增量保时效，但建模规范、维度、指标口径完全一致（靠 D02 的指标语义层统一）。这样实时大屏的数和次日离线报表的数才能对得上，业务才敢信。
实时数仓的中间层用 Kafka topic 承载，不像离线表那样能随便 SELECT 查看 / 回溯，排错更难。所以实时链路常同时把中间结果落一份到 OLAP/湖（可查、可对账）。可观测性在实时场景尤其重要。
04

端到端实时链路

CDC / Kafka → Flink → OLAP → 看板 · 把全系列拼起来

4.1 链路组件各司其职

① 源 + 摄取（D06）

业务库变更经 CDC（Debezium/Flink CDC）、埋点经 Kafka 实时进入链路。Kafka 是中枢与缓冲。

② 流计算（D04）

Flink 做清洗、维表关联、窗口聚合、去重——实时数仓的"加工车间"，有状态、Exactly-once。

③ 服务层（D07）

StarRocks/Doris/ClickHouse 既能实时写入又能高并发即席查询，是实时数仓的"门面"，直接喂大屏/API。

④ 应用

实时大屏、监控告警、实时推荐特征、运营看板——消费实时结果做即时决策。

为什么 OLAP 引擎适合做实时数仓的"服务层"？ 因为它同时满足两个苛刻要求：① 支持实时写入/更新（StarRocks 主键模型、ClickHouse 等支持高频写）；② 支持高并发、低延迟的即席聚合查询（D07 的列存+向量化+MPP）。Kafka 适合"流动"、不适合"被随机查询"，所以结果要落到 OLAP 才能给大屏/BI 用。

05

实时特有的硬骨头

实时比离线难在哪 · 这些是面试与实战的高频痛点

① 实时维表关联：流里的事件要补维度属性（订单流补商品/用户信息）。维表也在变——用 Lookup Join（查外部 Redis/HBase，配缓存）或 Temporal Join（时态表）关联到"事件发生时刻"的维度版本（呼应 D04 时态表、D02 SCD2）。难点是维表更新的时效与一致性。
② 实时去重：上游可能重发（at-least-once）、用户重复点击。实时去重靠 Flink 状态（State）记住"见过的 key"（配 State TTL 防膨胀），或用 OLAP 的主键模型（StarRocks Primary Key / ClickHouse ReplacingMergeTree）写入时去重。海量 key 去重还可用 HLL 近似（D05）。
③ 数据修正与撤回（Retract）：实时算出的值可能要改（订单退款、迟到数据更新窗口）。Flink 用 Changelog / Retract 流——发一条"撤回旧值（-）"再发"新值（+）"，下游据此更新。这让"流上的聚合结果可变"成为可能，是实时数仓正确性的关键机制。
④ 乱序与迟到：直接复用 D04 的 事件时间 + Watermark + allowed lateness。实时大屏要在"等迟到数据更准"和"尽快出数"间权衡。
⑤ 流批一体落地：用同一套 Flink SQL 既跑实时（无界流）又跑离线回算（有界流），从根上保证口径一致（Kappa 的理想）。存储侧，流式数据湖 Apache Paimon（前 Flink Table Store）/ Hudi 让数据湖既能流式写入/读取、又能批量分析——把"湖仓"和"实时"合二为一，是当下最热的方向。
撤回流像记账时写错了划掉重写：不是偷偷改原来那笔（流已经发出去了改不了），而是记一笔红冲（-99）再记一笔正确的（+105），账本最终对得上。实时系统里"已经发出的结果不能改，只能用新消息纠正"，撤回流就是这套纠错机制。

5.1 流式湖仓选型：Paimon vs Hudi vs Iceberg

「流批一体」存储侧的关键，是选一个能流式写入 + 流式读取 + 高频更新，又能被批量分析的湖仓表格式。这正是 D11 湖仓表格式在「实时」语境下的延伸——但三家对「流」的友好度差别很大，是实时数仓选型的高频考点。

为什么普通 Iceberg/Delta「流式更新」会吃力？ 它们最初为批分析设计：高频小批写入会产生海量小文件与频繁快照，更新（MERGE）成本不低，且没有为「按主键持续 upsert + 流式增量读」做专门优化。能用，但在秒级、高频、带主键更新的实时场景下，需要很重的 compaction 运维才跟得上。
Apache Paimon 凭什么更「流原生」？ Paimon（前 Flink Table Store）底层是一个 带 LSM-Tree 的湖存储（又见 D01/D12 的 LSM 老朋友）：数据按主键进 LSM，天然支持高频 Upsert、自动 compaction、且能产出 changelog 供下游流式消费。于是「DWD 流式更新 → DWS 流式读取再聚合」这种整条链路都在湖上流动的模式成为可能——这是它和 Flink 配合做流批一体的核心优势。
维度Apache PaimonApache HudiIceberg / Delta
底层结构LSM（为流而生）主键索引 + COW/MOR清单树 / 事务日志
高频流式 Upsert最强强（CDC 入湖老牌）可用，运维偏重
流式读 / changelog原生产出 changelog有限较弱（偏批读）
与 Flink 集成深度（同社区）较好好（引擎中立）
批分析生态成长中好最广
选型直觉：整条链路都要流式更新 + 流式读（流批一体的湖仓底座）→ Paimon；CDC 高频 Upsert 入湖、偏批分析 → Hudi；引擎中立、批分析为主、偶尔更新 → Iceberg/Delta（D11）。很多团队是「Iceberg 做离线主湖 + Paimon 做实时层」并存。一句话：Paimon 把"流"当一等公民、Iceberg 把"开放与批"当一等公民，按你的主战场选。
06

实战：实时 GMV 大屏

把前 5 章串成一条能跑的链路

需求：「双 11 实时大屏，秒级展示全站 GMV、各品类销售额、实时订单量。」 用 Kappa 风格搭一条链路。

① 源 / ODS
订单库 CDC → Kafka ods_orders
Flink CDC 实时捕获订单表的 INSERT/UPDATE（含退款改状态），灌入 Kafka。
② DWD
Flink 清洗 + 维表关联 → dwd_order_detail
过滤无效订单、用 Temporal Join 补商品类目（维表在 Redis），退化订单号。
③ DWS
Flink 实时聚合 → GMV / 品类销售额
按全站 / 品类做累计聚合；退款用 Retract 流红冲，保证 GMV 正确。
④ ADS / 服务
写入 StarRocks（主键模型）→ 大屏查询
StarRocks 实时写入 + 高并发查询；大屏每秒查一次，亚秒返回。
-- Flink SQL：从 CDC 订单流实时聚合全站 GMV（含退款撤回）
-- 源表声明为 changelog（带 +I/-U/+U/-D），聚合会自动产生 retract 维持正确
INSERT INTO ads_gmv_realtime        -- StarRocks 主键表，按 retract 更新
SELECT
    'ALL' AS scope,
    SUM(pay_amount) AS gmv,           -- 退款使 pay_amount 变更 → SUM 自动回退
    COUNT(DISTINCT order_id) AS order_cnt
FROM dwd_order_detail
WHERE order_status = 'paid';     -- 状态变退款 → 该行被撤回，GMV 准确下修
这个 case 把全系列焊在一起：CDC（D06）捕获变更、Kafka（D06）做中枢、Flink（D04）聚合 + 时态 join + retract、StarRocks（D07）主键模型实时写 + 高并发查、口径靠指标语义层（D02）与离线统一。这就是一套现代实时数仓的最小可用骨架。

小结 · 面试 · 术语表

收束成判断力 · 并接上 D09

一页纸总结

① 为何实时

T+1 等不了；实时数仓把产出压到秒级，与离线互补（快 vs 准）。

② Lambda

批+流双链路+合并。强但两套代码、口径易漂、维护翻倍。

③ Kappa

一切皆流，一套代码，重算靠重放 Kafka。流批一体后渐成主流。

④ 链路

CDC/Kafka(D06)→Flink(D04)→OLAP(D07)→大屏。OLAP 做服务层（实时写+高并发查）。

⑤ 硬骨头

维表 join（时态表）、实时去重（状态/主键）、撤回流纠错、乱序 Watermark。

⑥ 流批一体

一套 Flink SQL 跑流与批；Paimon/Hudi 流式湖仓统一存储。口径流批一致是命门。

面试自测（点击展开）

基础什么是实时数仓？和离线数仓什么关系？
实时数仓以流处理为核心、秒~分钟级产出的数仓，沿用离线的建模分层思想，把批加工换成流加工。
关系互补：离线（Spark，准、全、T+1）+ 实时（Flink，快、增量、秒级）；追求一套模型一套口径、两种时效。
核心Lambda 和 Kappa 架构的区别？各自优缺点？
Lambda批+流双链路+服务层合并。优：全量回算强；缺：两套代码、口径易不一致、维护翻倍。
Kappa仅流单链路，重算靠重放 Kafka 历史。优：一套代码一种语义；缺：重放大历史成本高、依赖消息长期保留。
趋势流批一体工具成熟后 Kappa 渐主流，但很多团队务实混合。无论哪种，流批口径统一是关键。
链路一条端到端实时链路由哪些组件构成？各自角色？
摄取CDC/Kafka（D06）：捕获变更/事件，做中枢与缓冲。
计算Flink（D04）：清洗、维表关联、窗口聚合、去重，有状态 Exactly-once。
服务OLAP 引擎（D07）：实时写入 + 高并发即席查询，喂大屏/API。
为何 OLAP 做服务层Kafka 适合流动不适合被随机查，结果必须落到能高并发查询的 OLAP。
实战实时 GMV，遇到退款怎么保证数字正确？
机制CDC 捕获订单状态变更（paid→refunded）；Flink 用 Retract/Changelog 流，撤回旧值（-）再发新值（+）。
落地聚合（SUM）自动随撤回回退；写入 StarRocks 主键模型按主键更新，GMV 准确下修。
本质"已发出的结果不能改，只能用新消息纠正"——红冲思想。
前沿"流批一体"到底解决什么问题？怎么落地？
问题消灭 Lambda 的双套代码 + 流批口径不一致。
计算侧用同一套 Flink SQL 跑无界流（实时）与有界流（离线回算），逻辑只写一遍。
存储侧流式数据湖 Paimon / Hudi：既支持流式读写、又支持批量分析，统一湖仓与实时。

术语表速查 · Glossary

本篇核心名词一次收齐：

① 架构范式

实时数仓 Real-time DW以流处理为核心、秒~分钟级产出的数仓，沿用离线分层思想，批加工换流加工。
Lambda 架构批层(准慢)+速度层(快近似)+服务层(合并)。强但两套代码、口径易漂。
Kappa 架构仅流单链路，一套代码；重算靠重放 Kafka 历史。流批一体后渐主流。
重放 Replay把 Kafka 历史消息重新消费一遍，用同一流代码重算（Kappa 的回算手段）。
实时离线一体同一套模型与指标口径，离线 Spark 保准、实时 Flink 保快，两数对得上。

② 链路组件
