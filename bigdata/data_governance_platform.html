数据治理与平台工程

00

为什么要治理 · 别让数据湖变数据沼泽

能跑 ≠ 可信赖 · 治理是数据从"成本"变"资产"的前提

技术管道搭好后，最常见的结局不是"数据驱动决策"，而是：表多到没人知道用哪张、指标对不上数没人敢信、改一张表线上报表突然崩了、敏感数据裸奔、云账单失控。数据湖（D01）一旦失管，就退化成"数据沼泽（data swamp）"——堆了海量数据却没人能用。

没治理的数据平台像一个没有图书管理员的巨型仓库：书（数据）堆成山，但没有目录、没有分类、不知道哪本是最新版、谁能借、借了会不会影响别人。治理就是请来管理员、立起目录与规则——让仓库从"杂物堆"变成"能高效借阅的图书馆"。技术决定数据"有没有"，治理决定数据"能不能放心用"。
数据治理（Data Governance）：围绕数据资产建立的一套规则、流程与工具，确保数据可发现、可信、安全、合规、成本可控、可追溯。本篇的六大支柱：① 元数据 & 血缘（找得到、看得清依赖）；② 数据质量（信得过）；③ 安全与隐私（合规）；④ 成本 / FinOps（花得值）；⑤ 平台工程 / 云原生（跑得稳、弹性）；⑥ DataOps / 组织（工程化协作）。
这一篇"工具名词"多，但别陷在工具里——重点是理解每类治理解决什么问题。工具会变（DataHub/Atlas/Ranger…），但"可发现、可信、安全、省钱"的诉求长青。
01

元数据与数据血缘

数据平台的"地图与导航" · 找得到数据、看得清依赖

1.1 元数据与数据目录

元数据（Metadata，"描述数据的数据"，呼应 D01）分三类：技术元数据（表结构、字段类型、分区、存储位置）、业务元数据（这张表是什么业务含义、指标口径、负责人）、操作元数据（谁在何时跑的、数据量、更新时间、运行状态）。把它们集中起来、可搜索，就是数据目录（Data Catalog）——数据平台的"搜索引擎 + 黄页"。代表工具：DataHub、Apache Atlas、OpenMetadata。
数据血缘（Data Lineage）：记录数据从哪来、流到哪去的依赖关系图——字段级 / 表级。两大用途：① 影响分析（下游）：我要改 / 下线这张表，会影响哪些下游表和报表？② 溯源（上游）：这个报表的数字算错了，往上追是哪张源表 / 哪段逻辑出的问题？没有血缘，改表就是"盲改"、排错就是"大海捞针"。
血缘像家谱 + 物流追踪：往上看（溯源）能查到这条数据的"祖先"是哪张源表；往下看（影响分析）能查到它的"子孙"有哪些报表依赖。改动前先看一眼下游、出错时先往上溯源——这是数据工程师的基本素养。

02

数据质量：让数据信得过

脏数据比没数据更可怕 · 它会安静地把决策带偏

脏数据像掺了水的汽油——车（分析/模型）照样能发动，但跑着跑着就出问题，而且你很难当场发现是油的问题。"没数据"你会警觉，"有脏数据"却让你带着错误的自信做决策，这才是最危险的。

2.1 数据质量六维

维度问什么例子规则
完整性 Completeness该有的值缺了吗？user_id 非空、关键字段不缺
唯一性 Uniqueness有重复吗？order_id 唯一、无重复行
有效性 Validity值合法吗？amount > 0、status ∈ 枚举集
准确性 Accuracy和真实世界一致吗？金额与订单系统对账一致
及时性 Timeliness按时到了吗？每天 6 点前数据就绪
一致性 Consistency跨表对得上吗？订单数 = 明细聚合数
数据质量工具与"质量门"：Great Expectations（Python，把规则写成"期望/断言"）、Deequ（Spark 上的质量校验，Amazon 开源）、dbt tests（D02）。核心实践：把质量校验嵌进数据管道当"质量门（quality gate）"——校验不通过就阻断下游、告警，而不是让脏数据继续往下游扩散污染报表。「宁可断、不可脏」。

03

数据安全与隐私

数据是资产也是负债 · 合规是不可逾越的底线

认证与授权（AuthN / AuthZ）：认证=你是谁（登录），授权=你能干什么。大数据常用 RBAC（基于角色的访问控制）——按角色授权而非逐人，并细化到库 / 表 / 列 / 行级别。代表：Apache Ranger（统一权限策略 + 审计，覆盖 Hive/HDFS/Kafka 等）、Apache Sentry。

列级 / 行级权限

分析师能看订单表，但看不到身份证号列（列级）；区域经理只能看自己区域的行（行级）。最小权限原则。

数据脱敏 Masking

把敏感字段遮蔽/替换：手机号 138****8888、身份证哈希、姓名打码。分静态脱敏（落地时改）和动态脱敏（查询时按权限改）。

数据分级分类

给数据打敏感级别标签（公开/内部/机密/绝密、是否含 PII 个人信息），再据此自动套用权限与脱敏策略。治理的基础动作。

合规与审计

GDPR / 个人信息保护法：用户有"被遗忘权"（要求删除数据）、知情同意。需审计日志记录谁访问了什么，备查。

PII（Personally Identifiable Information，个人身份信息）：能识别到具体个人的数据（姓名、手机号、身份证、人脸、精确位置等）。是隐私合规的核心保护对象——存储要加密、访问要授权、共享要脱敏、按法规该删要能删。识别并标记全平台的 PII，是数据安全治理的第一步（常借助数据目录 + 自动扫描分类）。
数据安全是"负债"视角：数据越多，泄露风险越大、合规义务越重。一次数据泄露的罚款 + 声誉损失，可能远超数据带来的收益。所以资深团队会主动"减负"：不必要的敏感数据不采集、过期数据按 TTL 删除、最小权限授权。"能不存就不存"也是一种治理。
04

成本治理 · FinOps

大数据 = 大账单 · 性价比是架构永恒的主线（呼应 D01 的 Value）

云上大数据最容易失控的就是成本：存储按量计费、计算按时计费，一个写错的查询全表扫几十 TB，账单就爆了。FinOps（云财务运营）就是让成本可见、可归因、可优化的工程实践。

FinOps：把"成本"当成一等工程指标来运营——可见（谁花了多少）→ 归因（哪个团队/任务/查询的成本）→ 优化（持续降本），并在"性能/时效"和"成本"间做理性权衡。核心心态：成本是每个数据工程师的责任，不是事后财务的事。

存储降本

  • 冷热分层：热数据 SSD、冷数据对象存储/归档
  • TTL 过期删除：日志只留 N 天
  • 小文件合并 + 压缩（呼应 D01 ZSTD、D06 小文件）
  • 纠删码替代多副本（冷数据）

计算降本

  • 弹性伸缩：按需拉起、用完释放（存算分离前提）
  • Spot/抢占式实例跑容错作业
  • 少 shuffle / 选对 join（D03）、分区裁剪（D05）
  • 预聚合/物化视图减少重复计算（D07）
成本归因（Cost Attribution）：把云账单拆到团队 / 项目 / 任务 / 甚至单条查询，让花钱的人看见自己花了多少。常用资源打标签（tagging） + 用量监控实现。看不见就治不了——可见性是 FinOps 的第一步，往往一上线就能砍掉一批"没人记得的僵尸任务"。
和性能调优一脉相承："先少干活"既快又省。D01 的列存压缩、D03 的少 shuffle、D05 的分区裁剪、D07 的预聚合——既是性能优化，也是成本优化。一个高效的查询，往往也是一个便宜的查询。
05

云原生大数据

从"自建 Hadoop 集群"到"跑在 K8s 上、按需弹性"

传统大数据是"自建一个常驻的 Hadoop/YARN 集群"——运维重、资源僵化、利用率低。云原生时代，大数据正在容器化、跑到 Kubernetes 上、走存算分离、按需弹性。

云原生大数据：把 Spark/Flink 等以容器形式跑在 Kubernetes 上，用 Operator（如 Spark Operator、Flink Operator）声明式地管理作业生命周期；存储走对象存储 + 存算分离（呼应 D01）。好处：弹性伸缩（用多少拉多少）、资源利用率高、与微服务统一调度、多云可移植。
大数据正在融入云原生底座：计算容器化、K8s 统一调度、存算分离。这也是 Spark/Flink 都重点投入 K8s 原生支持的原因。
代价与权衡：K8s 上跑大数据，shuffle 数据落本地盘、网络、有状态作业的存储等仍有挑战，且 K8s 本身有学习与运维成本。但方向明确——统一的弹性基础设施是大势所趋，尤其配合存算分离（D01）。

5.2 自建 vs 托管：怎么选

托管 / Serverless 更优当…

团队小、要快速上线、负载有明显波峰波谷、不想养专职平台团队。按量付费 + 近零运维，把精力放在业务与数据本身。

自建 / 半托管更优当…

规模极大（托管单价不划算）、有强合规/数据不出境要求、需要深度定制引擎、已有成熟平台团队。掌控力强但运维成本高。

厂商锁定（Vendor Lock-in）是托管服务的隐性成本：私有格式、私有 SQL 方言、迁移困难。当下的对冲手段是把数据以开放表格式（Iceberg/Delta）存在对象存储上，让计算引擎可插拔替换——这也是「湖仓 + 开放 catalog」被各云厂商集体拥抱的深层原因（详见湖仓表格式专篇）。成本侧则呼应本篇 FinOps（ch4）：托管不等于便宜，弹性用错姿势照样烧钱。
06

DataOps · 数据契约 · 数据网格

把软件工程纪律带进数据 · 以及组织层面的治理范式

DataOps：把 DevOps 的理念搬到数据——版本控制、自动化测试、CI/CD、可观测性、快速迭代。具体：数据管道代码进 Git、改动走 CI 自动跑测试（dbt test / GE）、自动部署、上线后持续监控数据 SLA。目标：让数据交付又快又稳，呼应 D02 的 dbt（"把软件工程纪律带进数据建模"）。
数据可观测性（Data Observability）：像监控服务一样监控"数据健康"——新鲜度（按时到了吗）、量级（行数突增突降？）、分布（值漂移？）、schema（结构悄悄变了？）、血缘。出问题主动告警、能快速定位（结合血缘溯源）。代表：Monte Carlo、datafold 等。它是数据质量的"运行时雷达"。
数据契约（Data Contract）：上游数据生产者与下游消费者之间的显式约定——schema、字段含义、质量保证、SLA、变更规则。把"上游随手改个字段、下游突然崩"变成"改动要遵守契约、破坏性变更要走流程"。呼应 D06 Schema Registry 的兼容性校验，是其组织/流程层面的延伸。
数据网格（Data Mesh）：一种组织范式（非具体技术）。核心是去中心化：不再由一个中央数据团队包揽一切，而是让各业务域（domain）自己拥有并以"数据产品"的方式对外提供高质量数据，平台团队提供自助式基础设施 + 统一治理标准。四原则：域所有权、数据即产品、自助平台、联邦式治理。适合大型组织数据团队成为瓶颈时。
中心化数据团队像全公司只有一个中央厨房——所有部门点菜都找它，很快排长队、成瓶颈。数据网格像每个部门有自己的小厨房（域自治），但共用同一套食品安全标准与供应链（联邦治理 + 自助平台）——既快又不失控。它解决的是"组织规模化"的问题，小团队不必上来就追求。
这些"软"实践常被工程师轻视，却是资深与初级数据团队的真正分水岭：初级比"会不会用 Flink/Spark"，资深比"能不能让一个几百人协作的数据平台可信、可治理、可持续"。技术是地基，治理是让大厦不塌的钢筋。
07

小结 · 面试 · 术语表

收束成判断力 · 并接上 D10 Capstone

一页纸总结

① 元数据 & 血缘

数据目录让数据可发现；血缘支持影响分析（改前查下游）+ 溯源（错了追上游）。

② 数据质量

六维（完整/唯一/有效/准确/及时/一致）；GE/Deequ 做质量门，宁可断不可脏。

③ 安全隐私

RBAC + 列/行级权限、脱敏、分级分类、PII 保护、GDPR/审计。能不存就不存。

④ FinOps

成本可见→归因→优化；存储分层/TTL、弹性计算；高效查询=便宜查询。

⑤ 云原生

Spark/Flink 容器化跑 K8s + Operator + 存算分离，弹性、高利用率。

⑥ DataOps / 组织

数据可观测性、数据契约、数据网格——把软件工程与组织治理带进数据。

合规GDPR/个保法：被遗忘权、知情同意、审计日志；能不存就不存。
实战云上大数据成本失控，怎么治理（FinOps）？
可见资源打标签，把账单拆到团队/任务/查询，先看见再治理。
存储冷热分层、TTL 删过期、小文件合并+压缩。
计算弹性伸缩+Spot、少 shuffle、分区裁剪、预聚合。
心态高效查询=便宜查询；成本是每个工程师的责任。
理念数据网格（Data Mesh）解决什么问题？什么时候用？
问题中央数据团队在大型组织里成为瓶颈、不了解各业务域。
方案去中心化：各业务域自治、把数据当产品对外提供；平台团队提供自助基建+联邦治理标准。
何时大型组织、数据团队成瓶颈时；小团队不必上来就追求，避免过度设计。

术语表速查 · Glossary

本篇核心名词一次收齐：

① 元数据与血缘

数据治理 Data Governance确保数据可发现/可信/安全/合规/成本可控/可追溯的规则、流程与工具。
数据沼泽 Data Swamp失管的数据湖：堆了海量数据却没人能找到、能信、敢用。
元数据 Metadata描述数据的数据：技术（结构/位置）、业务（含义/口径/负责人）、操作（运行/量级/时间）。
数据目录 Data Catalog集中可搜索的元数据，平台的"搜索引擎+黄页"。DataHub/Atlas/OpenMetadata。
数据血缘 Lineage数据从哪来到哪去的依赖图。用于影响分析（下游）与溯源（上游）。

② 质量与安全

数据质量六维完整性/唯一性/有效性/准确性/及时性/一致性。
质量门 Quality Gate把校验嵌进管道，不通过则阻断下游+告警。GE/Deequ/dbt test。宁可断不可脏。
RBAC / Ranger基于角色的访问控制，细到库/表/列/行；Ranger 统一权限策略+审计。
脱敏 Masking遮蔽/替换敏感字段（静态落地改 / 动态查询时按权限改）。
PII / 分级分类个人身份信息；按敏感级别打标签，据此自动套权限与脱敏策略。
GDPR / 个保法隐私合规法规：被遗忘权、知情同意、审计；驱动"能不存就不存"。

③ 成本与平台

FinOps让云成本可见→归因→优化的工程实践；成本是每个工程师的责任。
成本归因 / 冷热分层 / TTL账单拆到团队/任务；热数据快存冷数据廉价存；过期自动删。
云原生大数据Spark/Flink 容器化跑 K8s + Operator + 存算分离，弹性、高利用率。
OperatorK8s 上声明式管理有状态应用（Spark/Flink）生命周期的控制器。

④ DataOps 与组织

DataOps把 DevOps 搬到数据：版本控制、自动测试、CI/CD、可观测、快速迭代。
数据可观测性监控数据健康：新鲜度/量级/分布/schema/血缘，主动告警+定位。
数据契约 Data Contract生产者与消费者对 schema/含义/质量/SLA/变更规则的显式约定，防破坏性变更。
数据网格 Data Mesh组织范式：去中心化、域自治、数据即产品、自助平台、联邦治理。解决组织规模化瓶颈。
