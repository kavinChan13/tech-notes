给 C++ / 系统老兵的大数据地图

你已经掌握的，比你以为的多 · 大数据只是把熟悉的约束放大到「一个机房」

如果你写过 C++、调过内存泄漏、读过 Linux IO 路径，那么你其实已经懂了大数据 70% 的底层直觉。大数据不是一门全新的学问，而是把单机里那些你已经熟悉的权衡——顺序 IO vs 随机 IO、cache 命中、批处理、并发与一致性——放大到几十上百台机器协作的尺度。本系列就站在这个起点上展开。

单机就像你一个人在厨房做一桌菜：内存是案台，磁盘是冰箱，CPU 是你的双手。大数据则是一整个中央厨房：几十个厨师（节点）协作，食材（数据）放在共享冷库（分布式存储）里，难点从「怎么切菜」变成「怎么分工、怎么对账、有人请假怎么办」。烹饪技巧没变，协调成了新的主题。
本系列站在这些「已有知识」之上

system / IO 与内核

顺序 vs 随机 IO、page cache、零拷贝、文件描述符——大数据存储引擎的每个设计都在和这些较劲。

architect / 分布式 & 存储

CAP、副本、共识、B+Tree vs LSM、WAL——分布式文件系统和数仓直接复用这些原语。

cpp / 并发 & 性能

cache 局部性、SIMD、向量化、内存布局——正是列存与向量化执行引擎快的根因。

本篇遇到的分布式原理（CAP、副本、共识）只做「数据视角」的最小化引用，不重复展开。需要深入时，请看 architect/distributed_systems_fundamentals.html 与 architect/storage_and_data_architecture.html。

一句话定义

所谓「大数据」，并不是「数据很大」这么简单，而是：当数据的体量、速度或多样性，超过了单台机器在可接受时间 / 成本内处理的能力，必须用一组机器协作来存储与计算时，你就进入了大数据的领域。判据是「单机搞不定」，而不是某个绝对的 GB 数字。
01

大数据是什么 · 演进史

5V 模型 · 单机的物理天花板 · 从 Google 三驾马车到湖仓与云原生

1.1 5V：大数据的五个维度

业界常用 5 个以 V 开头的词来刻画大数据的特征。别把它当口号，每个 V 都对应一类真实的工程难题：

Volume · 体量

TB / PB / EB 级。一台机器的内存（几百 GB）和磁盘（几十 TB）都装不下 → 必须分片（sharding）+ 分布式存储。

Velocity · 速度

数据持续高速涌入（每秒百万条）→ 催生流处理与消息系统（Kafka / Flink）。

Variety · 多样性

结构化（表）、半结构化（JSON / 日志）、非结构化（图片 / 视频）混杂 → 催生数据湖与 schema-on-read。

Veracity · 真实性

数据有脏、有缺、有重复、有迟到 → 催生数据质量与治理（后续 D09 专题）。

Value · 价值

价值密度低——PB 的日志里也许只有几条有用。整个数据工程的终极目的，是用尽量低的成本把低密度的价值提炼出来。这也是为什么「成本 / 性价比」是大数据架构永恒的主线。

结构化 / 半结构化 / 非结构化数据——这是「多样性」里最常被问到的三个词：
  • 结构化（Structured）：有固定 schema、能放进二维表的数据。如数据库里的订单表。规整、好查询。
  • 半结构化（Semi-structured）：有一定结构但不固定，字段可增减、可嵌套。如 JSON 日志、XML。靠「自描述」携带结构信息。
  • 非结构化（Unstructured）：没有预定义结构。如图片、音频、视频、自由文本。占数据总量的大头，但最难直接分析。
分片 / 分区（Sharding / Partitioning）：把一份大数据按某个规则（如按用户 id 取模、按日期）切成多个小片，分散到不同机器，让它们能并行读写。这是 scale-out 的最基本动作——和单机里「把大任务拆成多线程」是同一种思想，只是从「多核」放大到了「多机」。后面几乎所有引擎的并行度，本质都来自分片。

1.2 单机为什么搞不定？——算一笔账

老兵最该建立的直觉是定量的。我们用一个最朴素的任务来感受单机的天花板：「扫描 1 TB 数据做一次聚合」。

单盘吞吐是物理常数；要更快，只能让更多盘同时读 —— 这就是 scale-out 的全部动机。
关键认知：单盘吞吐（HDD ~200MB/s、NVMe SSD ~3-7GB/s）是物理上限，靠换更贵的单机（scale-up）很快撞墙且价格指数上涨。把数据切片、让 N 台机器各读一份再汇总（scale-out），用数量换吞吐，是大数据的第一性原理。代价是：你引入了网络、协调、容错这三个新麻烦。

1.3 技术演进时间线

理解「为什么有这么多组件」，最好的方式是顺着历史走一遍。每一代技术，都是为了解决上一代的痛点而生：

2003–2004 · Google 三驾马车
GFS + MapReduce + BigTable（论文）
奠基之作：GFS = 分布式文件系统，MapReduce = 分布式批计算范式，BigTable = 分布式宽表。整个大数据世界都是这三篇论文的回声。
2006 · Hadoop
HDFS + MapReduce 开源
Doug Cutting 把 Google 论文落地为开源。从此普通公司也能搭分布式集群。HDFS 至今仍是很多体系的存储底座。
2010 · Spark
内存计算挑战 MapReduce
MapReduce 每步都落盘、慢。Spark 用内存 + DAG 把迭代计算提速 10–100x，逐渐成为批处理事实标准。
2011 · Kafka
高吞吐分布式消息
LinkedIn 开源。成为「数据管道的中枢神经」，把批处理世界推向流处理时代。
2014–2015 · Flink
真正的流优先引擎
事件时间、Watermark、状态、Exactly-once——把「流」做成一等公民，而非把批切小。实时数仓的引擎担当。
2014+ · 云数仓
Snowflake / BigQuery：存算分离
计算与存储解耦、按需弹性、按量计费。把「运维集群」变成「写 SQL」，重塑了行业。
2017–2020 · Lakehouse
Delta Lake / Iceberg / Hudi
给廉价的对象存储 + Parquet 加上 ACID、schema 演进、time travel。湖（灵活廉价）与仓（事务可靠）走向融合。
2023+ · 云原生 & AI
K8s 化 · 向量 / 特征 · 数据 ×AI
引擎跑在 Kubernetes 上；数据栈开始为 ML / LLM（特征平台、向量检索、RAG 管道）服务。
展开「Google 三驾马车」到底各自解决了什么问题？
GFSGoogle File System（2003）：一个跑在大量廉价机器上的分布式文件系统。核心思想——大文件切块、多副本、主从架构。解决「数据放哪、坏盘了怎么办」。开源对应物就是 HDFS（第 3 章详讲）。
MapReduce分布式批计算范式（2004）：把计算拆成 Map（各节点本地处理自己那片数据）+ Reduce（汇总）两阶段，框架自动负责分发、容错、汇总。解决「怎么让一群机器协作算一件大事」。
BigTable分布式宽表 / NoSQL（2006）：在 GFS 之上建一个能存千亿行、按 key 快速读写的「超大表」。解决「海量结构化数据的在线存取」。开源对应物是 HBase。
一句话GFS 管「存」、MapReduce 管「算」、BigTable 管「在线查」——后来整个大数据生态，几乎都是在这三条主线上不断演进、优化、替换零件。
这条时间线就是本系列的「地图」：D01 打地基，D03/D04 讲计算引擎（Spark/Flink），D05 讲 Kafka/摄取，D06 讲湖仓（Iceberg/Delta），D07 讲 OLAP 引擎内核，D08 讲实时数仓，D09 讲治理与平台，D10 是端到端 Capstone。
02

五个分水岭概念

这五对概念决定了「你该选哪种系统」 · 选错了，再努力调优也救不回来

2.1 批处理 vs 流处理

批处理像「每天晚上盘点一次仓库」——数据攒成一批，定时跑，吞吐高、延迟以小时计。流处理像「每进出一件货就立刻记一笔」——数据一来就处理，延迟以秒 / 毫秒计，但要时刻在线、处理乱序与迟到。
维度批处理 Batch流处理 Stream
数据边界有界（一批固定数据）无界（永不结束）
延迟分钟 ~ 小时毫秒 ~ 秒
吞吐极高高
典型场景T+1 报表、离线训练、对账实时大屏、风控、监控告警
代表引擎Spark、MapReduceFlink、Spark Structured Streaming
现代趋势是流批一体：用同一套引擎和 API 同时表达批与流（Flink、Spark 都在做）。但「批」和「流」的心智差异依然存在，D04 / D08 会深入。

2.2 OLTP vs OLAP——大数据主要服务后者

OLTP像银行柜台：一次只办一个人的一笔业务（转账），要快、要准、要事务。OLAP像统计局：一次扫描上亿条记录算「全国人均存款」，不在乎单条延迟，只在乎整体吞吐。
维度OLTP（事务）OLAP（分析）
典型操作点查 / 单行增删改大范围扫描 + 聚合
并发高并发、小事务低并发、大查询
数据量 / 查询几行百万 ~ 十亿行
存储取向行存（整行一起读写）列存（只读需要的列）
代表系统MySQL、PostgreSQLClickHouse、StarRocks、数仓
大数据体系绝大多数组件服务于 OLAP。记住这一点，第 4 章「为什么是列存」就顺理成章了——OLAP 查询的本质是「扫很多行，但只关心少数几列」。

2.3 Scale-up vs Scale-out

Scale-up（垂直扩展）

换更强的单机：更多 CPU / 内存 / 更快磁盘。简单，无需改架构，但有物理上限，且高端硬件价格指数上涨。

Scale-out（水平扩展）

加更多普通机器，让它们并行。理论上线性扩展、用廉价硬件，但引入网络、协调、容错的复杂度。大数据的主旋律。

Scale-out 不是免费午餐。一旦多机协作，你立刻要面对：数据怎么分片、节点挂了怎么办、机器间怎么对账（一致性）、网络成为瓶颈。这正是分布式系统的全部难点（详见 architect/ 板块）。

2.4 Schema-on-write vs Schema-on-read

Schema-on-write（写时模式 / 数仓）像「进图书馆前先按杜威分类编号上架」——入库严格、查得快，但前期规整成本高、不灵活。Schema-on-read（读时模式 / 数据湖）像「先把书全堆进仓库，要看时再翻」——入库零成本、什么都能存，但查询时要自己解析、容易乱。
Schema-on-write（数据仓库）Schema-on-read（数据湖）
何时定义结构写入前读取时
灵活性低（改结构代价大）高（先存了再说）
查询性能 / 可靠高、强一致取决于读时处理
典型载体数仓表对象存储上的文件
「湖仓一体（Lakehouse）」正是想兼得两者：在数据湖的廉价灵活之上，叠加数仓的事务与 schema 治理。这是 D06 的主题。
03

分布式存储底座

数据先得有地方放 · HDFS 的设计哲学 · 对象存储与存算分离的崛起

所有计算都建立在存储之上。大数据的存储底座要回答一个老兵熟悉、但被放大了的问题：当单块磁盘装不下、且随时可能坏掉时，怎么可靠地存 PB 级数据？

3.1 HDFS：分布式文件系统的范本

HDFS（Hadoop Distributed File System）是 GFS 的开源实现，核心思路朴素得惊人：把大文件切成固定大小的块（默认 128MB），分散存到很多台廉价机器上，每块存 3 份副本防丢。

主从架构：NameNode 是「目录管理员」（只记谁在哪），DataNode 是「货架」（真正存货）。数据流不经过 NameNode，避免它成为带宽瓶颈。

三个关键设计，背后都是老兵熟悉的权衡

  • 为什么块这么大（128MB）？ 减小元数据规模（块越大、块越少、NameNode 内存压力越小），并让磁盘读以顺序 IO 为主——这正是你在单机优化里追求的「大块顺序读 ≫ 小块随机读」，只是搬到了集群尺度。
  • 为什么 3 副本？ 廉价机器会坏。副本是最简单的容错：坏一份，从另两份恢复。代价是 3 倍存储成本（后来用 纠删码 Erasure Coding 把冷数据降到 ~1.5 倍）。
  • 为什么元数据和数据分离？ NameNode 只管「映射」放内存里、查得飞快；真实数据直连 DataNode，吞吐随节点数线性扩展。这与数据库里「索引 vs 数据页」的分工异曲同工。
元数据（Metadata）：描述「数据的数据」——不是文件内容本身，而是「有哪些文件、每个文件由哪些块组成、每个块存在哪几台机器上、文件多大、谁创建的」这类账本信息。HDFS 把元数据全放在 NameNode 内存里，所以元数据的规模直接决定了集群能管多少文件（这也是「小文件问题」的根源）。
纠删码（Erasure Coding, EC）：一种比「多副本」更省空间的容错方式。原理类似 RAID——把数据切成 k 个数据块，再算出 m 个校验块；任意丢失 ≤ m 个块都能从其余块反推恢复。例如 EC(6,3) 用 1.5 倍空间就能容忍坏 3 块，而 3 副本要 3 倍空间。代价是恢复时要做计算、读多个块，所以一般只对冷数据用 EC，热数据仍用副本。
NameNode 是单点：它一挂、元数据全失，整个集群瘫痪（早期 HDFS 的痛点）。生产用 HA 双 NameNode + ZooKeeper 自动切换 解决——这本质就是分布式共识问题（见 architect/distributed_systems_advanced.html 的 Raft / Paxos）。

3.2 写入流程：副本是怎么落地的

理解写流程，能解释很多「为什么 HDFS 适合一次写多次读、不适合频繁改」的问题：

1→Client 问 NameNode：「我要写文件 X，给我分配块和目标 DataNode」
2→NameNode 返回一组 DataNode（按机架感知策略挑选）组成写入管线（pipeline）
3→Client 把数据切成 packet，流式推给第 1 个 DataNode；它边收边转发给第 2 个，第 2 个转发给第 3 个（链式复制，省 Client 带宽）
4→三份都落盘后逐级 ACK 回到 Client，该块写入完成
机架感知（rack awareness）：默认把 3 副本放成「本机架 2 份 + 异机架 1 份」——既防单机坏（同机架另一份兜底），又防整机架断电（异机架那份兜底），同时尽量减少跨机架带宽。这是「容错」与「网络成本」之间的经典折中。

3.3 对象存储与存算分离

云时代，HDFS 的「存算耦合」（数据和计算在同一批机器）显出短板：扩容算力必须连带扩存储，反之亦然，弹性差。于是对象存储 + 存算分离成为新范式。

存算耦合像「自带食材的私厨」——厨师和冰箱绑在一起，要加厨师就得连冰箱一起买。存算分离像「中央冷库 + 按需叫的厨师」——食材统一放云冷库（对象存储），需要算力时临时拉一批计算节点，算完释放。冷库和厨师各自独立伸缩。
维度HDFS（存算耦合）对象存储 S3 / OSS / MinIO
数据模型文件 / 目录（可追加）对象 + 扁平 key（不可改，整体覆盖）
弹性存算绑定，弹性差存算分离，按需弹性
成本自建运维成本高极低（尤其冷数据）
一致性 / 延迟低延迟、强一致延迟较高、历史上最终一致
典型搭配本地 Hadoop 集群云上 Lakehouse（Iceberg/Delta + Spark/Trino）
趋势很明确：对象存储正在成为大数据新的统一底座。但对象存储「不能原地改、只能整体覆盖、list 慢」的特性，催生了表格式（Iceberg/Delta/Hudi）来管理「哪些文件属于当前表、怎么做 ACID 更新」——这是 D06 的核心。

3.4 数据视角的分布式基础速览

大数据的存储与计算，全建立在几条分布式原语之上。这里用数据语境把它们串成一页速查——不展开推导（那是架构板块的事），只讲"它在大数据里长什么样"。

分片 / 分区（Sharding / Partitioning）：把大数据切成多片散到不同机器，是并行度与扩展性的根。在不同层有不同名字：HDFS 叫块（Block）、Kafka 叫分区（Partition）、Spark/Flink 叫分区（Partition，= 一个 Task 的处理单元）、数仓表叫分区（按天/地区）。它们本质同源：用切分换并行。
副本（Replication）：同一份分片存多份在不同机器上，用于容错（坏一份从别处恢复）与分摊读压力。HDFS 默认 3 副本、Kafka 用 ISR（同步副本集）、对象存储内部也做多副本/纠删码。代价是存储成本与写放大。
CAP 定理：分布式系统在一致性（C）、可用性（A）、分区容错（P）三者中，网络分区发生时只能保 CP 或 AP（P 在分布式里必选）。数据视角：强一致存储（如 HBase）偏 CP，高可用缓存/对象存储常偏 AP（最终一致）。选型即在"读到的一定是最新"和"任何时候都能读写"之间取舍。（进阶 PACELC：即使没分区，也要在延迟 L 与一致性 C 间权衡。）
共识（Consensus）与 Raft：让多个节点对"谁是主、数据是什么"达成一致的算法。大数据里到处是它的身影：HDFS NameNode HA、Kafka Controller、各类元数据服务都靠共识选主、保证元数据一致。Raft 是目前最易懂、最常用的共识算法（Leader 选举 + 日志复制）。
这些只是"数据视角的速览"。完整推导（CAP/PACELC 证明、Raft 四子问题与陷阱、Paxos/BFT、向量时钟/TrueTime、CRDT、TLA+）请看架构板块：architect/distributed_systems_fundamentals.html 与 architect/distributed_systems_advanced.html。本系列只在用到时点到，不重复造轮子。

行存 vs 列存：大数据性能的命门

同样的数据，换个摆放方式，分析查询能快一个数量级 · 老兵的 cache 直觉在这里发光

如果说前三章是「数据放哪」，那从这一章开始是「数据怎么摆」。行存还是列存，是分析型系统最关键的一个决定。理解它，你就理解了 Parquet、ClickHouse、向量化执行为什么快。

4.1 同一张表的两种摆法

想象一张员工表。行存像一摞「个人档案袋」——每个人的所有信息装在一个袋子里，按人挨个排。要看某个人全部资料很方便；但要「统计全公司平均工资」，得把每个袋子都翻出来抽出工资那页。列存像「按字段归档的文件柜」——所有人的工资单放一个抽屉、所有人的姓名放另一个抽屉。算平均工资只需拉开「工资」抽屉，其余看都不看。
磁盘是一维的。「行存」沿行展开，「列存」沿列展开。摆法决定了「只关心几列」的查询要不要读无关数据。

4.3 为什么 OLAP 一定选列存？三个老兵能秒懂的理由

① 只读需要的列

分析查询常是「几百列里取 2 列做聚合」。列存只读这 2 列对应的磁盘段，投影下推直接砍掉绝大部分 IO。

② 压缩率碾压

同一列的数据类型相同、取值相近（比如「国家」列大量重复），压缩比远高于行存的混合数据。第 6 章详解。

③ 向量化 + SIMD 友好

同列连续 = 内存里一段同类型数组，正好喂给 SIMD 批量计算、cache 命中高。这正是你在 C++ 性能优化里追求的「数据导向设计」。

第 ③ 点直接呼应 cpp/concurrency_deep.html 与 perf-debug/ 里的 cache 局部性、SIMD 主题。列存 + 向量化执行引擎（D07 的 ClickHouse / Velox）本质就是把单机性能工程的「array of structs vs struct of arrays」之争，搬到了数据引擎尺度。
列存的代价：写入和「按行点查 / 改单行」变慢（要拆散到各列、或重写整个文件）。所以列存服务 OLAP（读多写少、批量追加），OLTP 仍然行存。没有银弹，只有匹配工作负载的选择。
05

列式文件格式：Parquet / ORC

列存的思想，落地成磁盘上的具体文件格式 · 自带「智能」让引擎少读数据

「列存」是思想，Parquet / ORC 是把它落到磁盘上的标准文件格式。它们是当今数据湖 / 湖仓的事实标准——你存进对象存储的，几乎都是 Parquet。理解它的内部布局，是理解「谓词下推为什么能跳过整段数据」的钥匙。

5.1 Parquet 的物理布局

Parquet 不是「整列从头存到尾」那么简单，而是先横切成若干行组（Row Group），每个行组内部再按列切块——这样既享受列存的好处，又能让一个行组成为并行处理的单位。

层级：File → Row Group（并行单位） → Column Chunk（列存单位） → Page（编码 / 压缩单位）。Footer 是「目录 + 统计摘要」，读取时先读它。
为什么 Footer 在末尾？ 写的时候数据是流式追加的，统计信息（每列 min/max、行数）要等全部写完才知道，所以放最后。读的时候引擎先 seek 到文件尾读 Footer，拿到「地图」后再精准跳读需要的列块——这就是下面三大优化的物理基础。

5.2 三大下推优化：让引擎「少读」

Parquet 之所以快，不只是列存，更因为它自带统计信息，让查询引擎在真正读数据前就能跳过大量无关内容：

投影下推 Projection

SELECT salary 只读 salary 那些 Column Chunk，其余列的磁盘段碰都不碰。

谓词下推 Predicate

WHERE salary > 100 时，先看每个 Row Group 的 salary max。max≤100 的整组直接跳过，不解压不扫描。

统计裁剪 Pruning

利用 min/max/null 计数、字典、Bloom Filter，把「肯定不含目标值」的块整段排除。

# PyArrow 读 Parquet：只取 2 列 + 谓词，引擎自动下推
import pyarrow.parquet as pq

table = pq.read_table(
    "employees.parquet",
    columns=["dept", "salary"],          # 投影下推：只读这 2 列的 Column Chunk
    filters=[("salary", ">", 100)],     # 谓词下推：salary max≤100 的 Row Group 整段跳过
)
# 100 列、10 亿行的文件里，这条查询可能只真正读了 <2% 的字节
这就像查一本带详细目录和索引的工具书：你不会从第一页读到最后一页，而是先翻目录（Footer），看「第 3 章工资在哪、范围多少」，确认相关章节后只翻那几页。Parquet 把这套「目录 + 索引」直接焊进了文件格式里。

5.3 格式对比：Parquet / ORC / Avro

编码与压缩：列存为何如此省空间

先「聪明地表示」（编码），再「无脑地压小」（压缩）· 两步叠加才是列存的省空间魔法

6.1 编码 ≠ 压缩：先分清两件事

很多人把「编码」和「压缩」混为一谈。在 Parquet 这类格式里，它们是先后两步，各司其职：

编码 Encoding（先做）

利用数据本身的规律换一种更紧凑的表示。无损、可直接在编码态上做计算。例：字典编码、RLE、Delta、Bit-packing。

压缩 Compression（后做）

对编码后的字节流再跑通用压缩算法，挤掉剩余冗余。与数据语义无关。例：Snappy、LZ4、ZSTD、GZIP。

列存之所以省空间，关键在于编码这一步在列存里特别有效——因为同一列的值类型相同、分布集中，规律性强。常见列编码：

  • 字典编码（Dictionary）：把重复的值（如「部门」只有十几种）建一张字典表，原列只存「字典下标」的小整数。基数低的列效果惊人。
  • 游程编码（RLE, Run-Length）：连续重复值压成「值 × 次数」。如 Eng,Eng,Eng,Eng → Eng×4。排序后的列效果最佳。
  • Delta 编码：存相邻值的差。对递增的 id / 时间戳列，差值都很小，能用更少 bit。
  • Bit-packing：若一列值域只需 3 bit，就别用 32 bit 存——按实际位宽紧凑打包。

6.3 压缩算法选型：速度 vs 比率的权衡

编码之后再叠加通用压缩。选哪个算法，本质是「CPU 时间」换「磁盘 / 网络字节」的权衡——又是一个老兵熟悉的折中：

算法压缩比压 / 解速度适用
Snappy中极快默认之选，热数据、追求吞吐
LZ4中极快（解压尤快）对延迟敏感的读
ZSTD高（可调级别）快（综合最佳）现代首选，比率 / 速度平衡好
GZIP高慢冷数据、归档、CPU 不紧张
经验法则：热数据（频繁查）用 Snappy/LZ4 保速度，冷数据（少查、长期存）用 ZSTD 高级别或 GZIP 省空间。如今 ZSTD 因为「比率接近 GZIP、速度接近 Snappy」，正在成为很多场景的新默认。这与你在网络 / 序列化里做的取舍完全同源。
别忽视「解压」开销：分析查询是读多的，解压速度往往比压缩速度更重要。一个压缩比高但解压慢的算法，可能让 CPU 成为查询瓶颈，反而比少压一点更慢。

一页纸总结

① 大数据的判据

「单机搞不定」而非某个 GB 数字。靠 scale-out 用数量换吞吐，代价是网络 / 协调 / 容错。

② 五个分水岭

批/流、OLTP/OLAP、up/out、写时/读时 schema。大数据主要服务 OLAP。

③ 存储底座

HDFS（切块 + 多副本 + 元数据分离）→ 对象存储 + 存算分离成为云时代新底座。

④ 列存是命门

OLAP 选列存：少读列、压得狠、向量化友好。Parquet 是事实标准。

⑤ 文件级智能

Parquet Footer + 统计 → 投影 / 谓词下推 / 裁剪，让引擎「少读」。

⑥ 编码 + 压缩

先编码（字典 / RLE / Delta）利用规律，再压缩（ZSTD/Snappy）挤冗余。

一个最小选型决策清单

Q1→是事务（点查改单行）还是分析（扫描聚合）？ → 事务走 OLTP 行存；分析走 OLAP 列存
Q2→延迟要求是秒级实时还是分钟 / 小时？ → 实时走流（Flink）；离线走批（Spark）
Q3→数据结构固定还是杂乱多变？ → 固定可入数仓；多变先入数据湖（Parquet on 对象存储）
Q4→数据冷热？ → 热数据 Snappy/LZ4 保速度，冷数据 ZSTD/GZIP 省空间

基础为什么大数据系统普遍用列式存储？至少说出两个理由。
投影分析查询常只取少数列，列存只读相关列的磁盘段，IO 大幅下降（投影下推）。
压缩同列数据类型相同、取值集中，编码 + 压缩比远高于行存的混合数据。
向量化同列连续 = 内存里同类型数组，天然适合 SIMD 批量计算、cache 友好。
理解Parquet 的「谓词下推」是怎么做到「不读就跳过」一整段数据的？
机制Parquet 在 Footer 里为每个 Row Group / Page 存了该列的 min/max/null 统计。
裁剪遇到 WHERE salary>100，引擎先看某 Row Group 的 salary max；若 max≤100，该组不可能有命中行，整段跳过——不解压、不扫描。
延伸数据若按谓词列排序 / 分区存放，跳过效率更高。这也是「数据布局优化」的核心套路（D06 会讲 Z-order / 聚簇）。
辨析HDFS 的块为什么默认设得很大（128MB）？小块不行吗？
元数据块越大、块数越少，NameNode 内存里要维护的「块→节点」映射就越少（NameNode 元数据全在内存，是稀缺资源）。
IO 模式大块促成顺序读，摊薄寻址 / 建连开销——和单机「大块顺序 IO ≫ 小块随机 IO」同理。
代价小文件多时（海量 KB 级文件）会撑爆 NameNode 内存、且每个都占一个块槽——这就是著名的「HDFS 小文件问题」。
进阶编码（encoding）和压缩（compression）有什么区别？为什么要分两步？
编码利用数据语义规律换紧凑表示（字典 / RLE / Delta），无损、常可在编码态直接计算（如字典下标比较）。
压缩对编码后的字节流跑通用算法（ZSTD/Snappy），与语义无关，挤掉剩余冗余。
为何分两步编码把「结构化冗余」先榨干（这是通用压缩看不懂的），再交给压缩处理「字节级冗余」，两步叠加效果远好于单独压缩原始数据。
架构都说「存算分离」好，那它相比 HDFS 的存算耦合，牺牲了什么？
收益计算 / 存储独立弹性伸缩、对象存储成本极低、运维简化。
代价数据不在本地，计算要跨网络拉取，延迟更高；对象存储「不可原地改、list 慢、历史上最终一致」带来新问题。
应对用缓存层（如本地 SSD 缓存热数据）补延迟，用表格式（Iceberg/Delta）在对象存储上补 ACID 与高效元数据——正是 D06 的主题。

术语表速查 · Glossary

本篇出现的名词一次性收齐 · 看到不认识的词，回这里查 · 也为后续 D02–D10 做铺垫

大数据领域名词密度高、缩写多，初学最容易被术语劝退。这里把全篇（及后续会反复出现）的核心概念按主题归类，给出「人话版」定义。不必一次记完，当字典用即可。

① 总览与基础

大数据 Big Data当数据的量 / 速度 / 多样性超过单机可承受的处理能力，必须用一组机器协作的场景。判据是「单机搞不定」，而非某个绝对数字。
5V刻画大数据的五个维度：Volume 体量、Velocity 速度、Variety 多样性、Veracity 真实性、Value 价值。
吞吐 vs 延迟 Throughput / Latency吞吐=单位时间能处理多少（每秒多少条）；延迟=单次处理多久出结果。大数据常以「牺牲单条延迟换整体吞吐」。
分片 / 分区 Sharding / Partitioning把大数据按规则切成多片分散到不同机器，以便并行读写。是 scale-out 并行度的来源。
副本 Replica同一份数据存多份在不同机器上，用于容错（坏一份从别的恢复）和分摊读压力。
一致性 Consistency多副本 / 多节点之间，数据是否「对得上账」。强一致=任何时刻读到的都是最新；最终一致=允许短暂不一致，但最终会收敛。
顺序 IO vs 随机 IO顺序读写（连续地址）比随机读写（跳来跳去）快得多。大数据存储格式的设计大量围绕「尽量顺序」展开。

② 计算引擎与范式

MapReduce最早的分布式批计算范式：Map（各节点本地处理）+ Reduce（汇总）。框架自动管分发与容错。慢在每步落盘，已被 Spark 取代。
DAG 有向无环图把一个计算任务表示成「算子→算子」的依赖图。Spark/Flink 用它来调度、优化、做容错恢复。
Shuffle分布式计算中，为了把「相同 key 的数据」聚到一起（如 group by），在节点之间大规模搬运、重分发数据的过程。往往是性能瓶颈，D03 详讲。
批处理 / 流处理 Batch / Stream批=攒一批有界数据定时跑，吞吐高延迟大；流=数据一来就处理无界数据，延迟低但要常驻、处理乱序。
流批一体用同一套引擎和 API 同时表达批与流，降低维护两套代码的成本。Flink / Spark 都在推。
事件时间 vs 处理时间 Event / Processing Time事件时间=数据「实际发生」的时间；处理时间=引擎「收到处理」的时间。网络延迟会让二者错位，流处理要按事件时间才准确。
Watermark 水位线流处理里的一个「时间进度标记」，用来判断「事件时间到此为止的数据应该都到齐了」，从而触发窗口计算、容忍一定迟到。D04 详讲。
状态 State流处理为了做累计 / 关联，需要在内存 / 磁盘里记住「到目前为止的中间结果」（如至今的总和），这就是状态。
Exactly-once 精确一次即使中途故障重试，每条数据对结果的影响也有且仅有一次（不丢不重）。是流处理可靠性的最高级别，对应还有 at-least-once（不丢但可能重）。
反压 Backpressure下游处理不过来时，把「慢」的信号逐级传回上游让它降速，避免数据堆爆内存。类似水管堵了往回顶。

③ 存储底座

HDFSHadoop 分布式文件系统，GFS 的开源实现。大文件切块、多副本、主从架构。很多大数据体系的存储底座。
NameNode / DataNodeHDFS 的主从两角色：NameNode（主）只管元数据（谁在哪）；DataNode（从）真正存数据块。
块 BlockHDFS 把文件切成的固定大小单位（默认 128MB）。块大→元数据少、利于顺序读。
机架感知 Rack Awareness放副本时，刻意把副本分散到「同机架 + 异机架」，同时兼顾「容错」与「跨机架带宽成本」。
元数据 Metadata「描述数据的数据」：有哪些文件、由哪些块组成、块在哪、多大、谁建的。不是内容本身，而是账本。
纠删码 Erasure Coding比多副本更省空间的容错法（类 RAID）：数据块 + 校验块，丢少量块可反推恢复。适合冷数据。
对象存储 Object Storage以「对象 + 扁平 key」组织数据的存储（S3 / OSS / MinIO）。便宜、近乎无限扩展，但不能原地改、list 慢。云时代新底座。
存算分离把存储（对象存储）和计算（临时拉起的节点）解耦，各自独立弹性伸缩。代价是计算要跨网络读数据。

④ 仓库 / 湖 / 表

OLTP / OLAPOLTP=在线事务处理（高并发小事务，如下单）；OLAP=在线分析处理（大范围扫描聚合，如报表）。大数据主要服务 OLAP。
数据仓库 Data Warehouse为分析而生、结构严谨（schema-on-write）的集中式数据存储。查得快、强治理，但灵活性低。
数据湖 Data Lake先把各种原始数据（结构化 / 非结构化）廉价地堆进对象存储，读时再定义结构（schema-on-read）。灵活但易乱。
湖仓一体 Lakehouse在数据湖的廉价灵活之上，叠加数仓的事务 / schema 治理，兼得两者。靠表格式（Iceberg/Delta/Hudi）实现。D06 主题。
Schema 模式数据的「结构定义」：有哪些列、各是什么类型。Schema 演进=随业务变化安全地增删改列。
ACID事务的四个保证：原子性 / 一致性 / 隔离性 / 持久性。让「要么全成功要么全失败」成为可能。湖仓表格式给对象存储补上了 ACID。
Time Travel 时间旅行表格式能保留历史版本，让你「查询某个过去时间点的数据」或回滚到旧版本。便于纠错与审计。
数仓分层 ODS/DWD/DWS/ADS把数据按加工程度分层：原始层→明细层→汇总层→应用层（或 Medallion 的 bronze/silver/gold）。D02 详讲。

⑤ 列存与文件格式

行存 / 列存 Row / Columnar行存=整行连续存（利于点查改单行，OLTP）；列存=同列连续存（利于扫描聚合、压缩、向量化，OLAP）。
Parquet / ORC / AvroParquet、ORC=列式分析格式（湖仓事实标准）；Avro=行式格式，适合逐行写入 / 流式传输。
Row Group / Column Chunk / PageParquet 的层级单位：行组（并行单位）→列块（列存单位）→页（编码/压缩单位）。Footer 记录各自的偏移与统计。
谓词下推 Predicate Pushdown把 WHERE 过滤条件「下推」到读数据这一层，利用块的 min/max 统计整段跳过不可能命中的数据。
投影下推 Projection Pushdown只读 SELECT 用到的列对应的磁盘段，其余列完全不读。列存的核心红利。
基数 Cardinality一列中不同取值的数量。如「性别」基数=2，「用户 id」基数极高。低基数列特别适合字典编码。
编码 Encoding利用数据规律换紧凑表示：字典编码（重复值→小整数下标）、RLE（连续重复→值×次数）、Delta（存差值）、Bit-packing（按实际位宽打包）。无损。
压缩 Compression对编码后的字节流再跑通用算法挤冗余：Snappy/LZ4（极快、比率中）、ZSTD（综合最佳）、GZIP（比率高但慢）。
向量化执行 Vectorized Execution一次处理一批同类型数据（而非一行一行），配合 CPU 的 SIMD 指令并行计算。列存天然适配，是 OLAP 引擎快的关键。
Bloom Filter 布隆过滤器一种极省空间的概率结构，能快速判断「某值一定不在集合里」（可能误判存在，绝不漏判不存在）。用于跳过不含目标值的数据块。
这些词在 D02–D10 会反复出现并各自展开。这里建立「混个脸熟」的程度即可——真正的理解会在后续看到它们如何协同工作时自然长出来。
