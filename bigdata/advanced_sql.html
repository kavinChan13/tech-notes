SQL 心智与逻辑执行顺序

SQL 是声明式的 · 搞懂"子句先后顺序"，一半的诡异 bug 就解释清了

SQL 是声明式语言：你描述「要什么结果」，不写「怎么一步步算」——引擎的优化器（如 D03 的 Catalyst）决定物理执行。但 SQL 有一个固定的逻辑执行顺序，理解它能解释一大票"为什么这么写报错/不对"的问题。

写 SQL 像在餐厅点菜：你说「要一份不辣的、川菜、按辣度排序的菜单前 3 道」（声明要什么），厨房（引擎）自己决定怎么备料下锅（物理执行）。你不用管火候顺序，但菜单的"逻辑处理顺序"是固定的：先确定有哪些菜（FROM）、筛掉不要的（WHERE）、分组、再排序、最后取前几道。
逻辑执行顺序（注意：和书写顺序不同！）：
① FROM / JOIN → ② WHERE → ③ GROUP BY → ④ HAVING → ⑤ SELECT（含窗口函数）→ ⑥ DISTINCT → ⑦ ORDER BY → ⑧ LIMIT

这个顺序立刻解释了几个高频困惑：

困惑原因（看执行顺序）
WHERE 里不能用 SELECT 取的别名WHERE（②）在 SELECT（⑤）之前执行，别名还不存在
ORDER BY 里却能用别名ORDER BY（⑦）在 SELECT（⑤）之后，别名已生成
WHERE 和 HAVING 区别WHERE 在分组前过滤原始行；HAVING 在分组后过滤聚合结果
窗口函数不能写在 WHERE 里窗口在 SELECT（⑤）阶段算，WHERE（②）时还没算 → 要过滤窗口结果得用子查询或 QUALIFY
逻辑顺序 ≠ 物理顺序：上面是"语义上"的处理顺序；真正执行时优化器会重排（谓词下推让 WHERE 早于 JOIN 发生等），只要结果等价即可。逻辑顺序帮你写对，物理优化帮你跑快（D03 Catalyst）。

窗口函数：分析型 SQL 的皇冠

组内排名、同比环比、累计、移动平均——离开窗口函数寸步难行

1.1 窗口函数 vs GROUP BY

GROUP BY 像把全班按小组合并成一行汇总——「Eng 组平均工资 83」，个人没了。窗口函数像在每个同学旁边贴一张便签，写上「你在组内排第 2 / 组累计到你是 170」——每一行都还在，只是多了一列基于"一组相关行"算出的值。这是两者最本质的区别。
窗口函数（Window Function）：在一个与当前行相关的「窗口」（一组行）上做计算，但不合并行——每行都保留，并附带一个计算结果列。语法核心是 OVER 子句： 
函数() OVER (
    PARTITION BY 分区列   -- 按它分窗（类似 group by，但不合并）
    ORDER BY 排序列        -- 窗内顺序（排名/累计/取前后行都依赖它）
    -- [frame: ROWS/RANGE BETWEEN ...]  窗口的"取值范围"
)

1.2 三类窗口函数

排名类

ROW_NUMBER（唯一序号）、RANK（并列跳号 1,2,2,4）、DENSE_RANK（并列不跳 1,2,2,3）、NTILE(n)（分 n 桶）。

取值类（偏移）

LAG（取前 N 行）、LEAD（取后 N 行）、FIRST_VALUE / LAST_VALUE / NTH_VALUE。同比环比、做差分全靠它。

聚合类（开窗）

SUM / AVG / COUNT / MAX / MIN OVER(...)。配 frame 可做累计（running total）和移动平均。

1.4 Frame 与几个经典套路

窗口框架（Frame）：在 ORDER BY 基础上进一步限定"窗口到底取哪些行"。ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW = 从头累计到当前行（running total）；ROWS BETWEEN 2 PRECEDING AND CURRENT ROW = 当前及前两行（3 行移动平均）。很多引擎在你不显式写 frame 时会给默认值，但具体默认行为可能因方言与数据类型（特别是 RANGE）存在差异，生产 SQL 建议写明 frame 以保证可移植与可复现。
-- 套路 1：每个部门工资 Top-3（top-N per group 的标准写法）
SELECT * FROM (
  SELECT name, dept, salary,
    ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn
  FROM emp
) t WHERE rn <= 3;   -- 支持 QUALIFY 的引擎可直接 QUALIFY rn <= 3

-- 套路 2：同比/环比（与上一期比较）
SELECT month, sales,
  sales - LAG(sales) OVER (ORDER BY month) AS mom_diff   -- 环比差
FROM monthly_sales;

-- 套路 3：累计销售额（running total）
SELECT day, amount,
  SUM(amount) OVER (ORDER BY day
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cum_amount
FROM daily_sales;
QUALIFY：专门用来过滤窗口函数结果的子句（之于窗口，正如 HAVING 之于 GROUP BY）。... QUALIFY ROW_NUMBER() OVER(...) = 1 一行搞定"取每组第一"，省去子查询嵌套。Snowflake / BigQuery / DuckDB 等支持；不支持的引擎用子查询 + WHERE 代替。

1.5 方言差异与兼容写法

能力常见支持情况兼容策略
QUALIFY部分云数仓与分析引擎支持改写为子查询后在外层 WHERE 过滤
窗口默认 frame不同引擎细节有差异显式写 ROWS/RANGE BETWEEN ...

高级聚合：多维小计一把梭

报表常要"分项 + 小计 + 总计"——别再 UNION 一堆子查询了

2.1 GROUPING SETS / ROLLUP / CUBE

普通 GROUP BY a, b 只给你「a×b 的明细」。但报表往往还要各维度小计 + 总计。手写要 UNION 好几段，啰嗦易错。这三个扩展一次搞定：

ROLLUP(a, b)

层级小计：(a,b) 明细 + 按 a 的小计 + 总计。适合有层级关系的维度（年→月、地区→城市）。

CUBE(a, b)

所有组合：(a,b) + 按 a + 按 b + 总计。维度无层级、要全交叉时用。

GROUPING SETS

最灵活：你指定要哪些分组组合。ROLLUP/CUBE 其实都是它的语法糖。

切换聚合方式，看结果行如何变化——尤其注意 ROLLUP/CUBE 自动多出来的小计行（绿色）和总计行（紫色）（被卷起的维度显示为「— 合计」）：

怎么区分"小计行"和"真有个叫 NULL 的值"？用 GROUPING(col) 函数：它对被卷起的维度返回 1、否则 0。报表里常 CASE WHEN GROUPING(region)=1 THEN '全部' ELSE region END 来美化小计标签。

2.3 FILTER：优雅的条件聚合

想在一次查询里算「总订单数」和「已支付订单数」？老写法是 SUM(CASE WHEN paid THEN 1 ELSE 0 END)。标准 SQL 的 FILTER 更清晰：

SELECT
    COUNT(*) AS total,
    COUNT(*) FILTER (WHERE status = 'paid') AS paid_cnt,
    SUM(amount) FILTER (WHERE region = '华北') AS north_sales
FROM orders;   -- 一次扫描，多个条件聚合。不支持 FILTER 的引擎退回 CASE WHEN
03

CTE 与递归查询

把"俄罗斯套娃式子查询"拆成可读的步骤 · 以及用 SQL 遍历层级

3.1 CTE：给子查询起名字

CTE（Common Table Expression，公共表表达式）：用 WITH 名字 AS (子查询) 给一段查询命名，后面直接当表用。它把深层嵌套的子查询拉平成自上而下的步骤，可读性、可维护性大增，还能在同一查询里多次复用同一中间结果。
-- 用 CTE 把"先算每店日销，再筛出高于全店均值的"拆成两步，清晰可读
WITH daily AS (
    SELECT store_id, SUM(amount) AS sales
    FROM orders GROUP BY store_id
),
avg_sales AS (
    SELECT AVG(sales) AS a FROM daily
)
SELECT d.* FROM daily d, avg_sales WHERE d.sales > avg_sales.a;
CTE 不一定被"物化"成临时表——多数引擎会把它内联（inline）展开优化（和直接写子查询等价）。少数引擎/场景会物化（被多次引用时）。需要强制复用时，部分引擎支持 MATERIALIZED 提示。别假设 CTE 自带缓存。

3.2 递归 CTE：用 SQL 走层级 / 图

递归 CTE（WITH RECURSIVE）：能引用自身的 CTE，用来处理层级 / 树 / 图结构——组织架构（找某人所有下属）、物料清单（BOM 展开）、评论树、图的可达性。结构固定为两部分：锚成员（起点，非递归）UNION ALL 递归成员（引用 CTE 自身，逐层展开），直到没有新行产生为止。
-- 找出员工 #1 的所有层级下属（组织架构树展开）
WITH RECURSIVE subordinates AS (
    -- 锚成员：起点
    SELECT id, name, manager_id, 1 AS lvl
    FROM emp WHERE id = 1
  UNION ALL
    -- 递归成员：把"下属的下属"一层层接上来
    SELECT e.id, e.name, e.manager_id, s.lvl + 1
    FROM emp e JOIN subordinates s ON e.manager_id = s.id
)
SELECT * FROM subordinates ORDER BY lvl;
递归 CTE 容易写出死循环（数据有环，如 A 管 B、B 又管 A）。生产里要么保证数据无环，要么加深度上限（用 lvl 列加 WHERE lvl < N 兜底）。大数据引擎（Spark SQL 早期）对递归 CTE 支持有限，用前先确认。

Join 进阶与子查询

semi / anti join、LATERAL、相关子查询、集合运算 · 还有 NULL 的坑

4.1 半连接 / 反连接（Semi / Anti Join）

半连接（Semi Join）：只想知道"左表的行在右表是否存在匹配"，不放大行数、不取右表列。用 WHERE EXISTS (...) 或 IN (...) 表达。反连接（Anti Join）：相反，要"左表中右表没有匹配的行"，用 NOT EXISTS。它们和普通 JOIN 的关键区别：普通 JOIN 会因右表多条匹配而放大左表行数，semi/anti 不会。
-- 找"下过单的用户"（semi join：存在即可，不关心下了几单）
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

-- 找"从没下过单的用户"（anti join）
SELECT * FROM users u
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);
NOT IN + NULL 是经典大坑：如果子查询结果里含 NULL，NOT IN 会返回空集（因为 x NOT IN (1, NULL) 在 SQL 三值逻辑里永远不为 true）。用 NOT EXISTS 代替 NOT IN 可避开此坑——这是 anti join 几乎总用 NOT EXISTS 的原因。

4.2 LATERAL / 相关子查询

LATERAL Join（横向连接）：让右侧子查询能引用左表当前行的列（普通子查询不行）。常用于"对每一行执行一个依赖该行的子查询"，如"每个用户取其最近 3 笔订单"，或配合展开嵌套数组。MySQL/PG 叫 LATERAL，SQL Server 叫 CROSS/OUTER APPLY。
相关子查询（Correlated Subquery）：子查询里引用了外层查询的列，因此对外层每一行都要重算一次（逻辑上）。表达力强但可能慢——优化器常能把它改写成 join。非相关子查询则只算一次。

4.3 集合运算

运算含义去重？
UNION并集去重（慢）
UNION ALL并集，保留重复不去重（快，优先用）
INTERSECT交集去重
EXCEPT / MINUS差集（A 有 B 无）去重
能用 UNION ALL 就别用 UNION——后者要做一次全局去重（排序/哈希，相当于一次 shuffle），数据大时代价不小。只有确实需要去重才用 UNION。

4.4 防行数放大模板（报表最常翻车点）

先聚合后 Join：明细表直接 join 明细表最容易把行数放大。先在各自粒度上聚合或去重，再 join，可以显著减少重复计数风险与 shuffle 体积。
-- 模板：先按订单聚合，再和维表关联，避免明细乘法放大
WITH fact AS (
  SELECT order_id, user_id, SUM(amount) AS amt
  FROM order_items
  GROUP BY order_id, user_id
)
SELECT f.order_id, f.amt, u.level
FROM fact f
LEFT JOIN users u ON f.user_id = u.id;
05

大数据 SQL 特色

复杂嵌套类型、数组炸裂、近似计算、采样——传统 OLTP SQL 里少见，大数据里天天用

5.1 复杂类型与数组炸裂

复杂类型（ARRAY / MAP / STRUCT）：大数据表常存嵌套结构（如一行用户带一个 tags ARRAY<string>、一个 profile STRUCT<age,city>）——这是半结构化数据（D01 Variety）落到表里的样子。访问：profile.city（struct）、tags[0]（array）、props['key']（map）。
explode / LATERAL VIEW（数组炸裂）：把一行里的数组/Map展开成多行——「一个用户 3 个标签」变成「3 行各 1 个标签」，之后就能正常 group by。Spark/Hive 用 LATERAL VIEW explode(tags)，Presto/Trino 用 CROSS JOIN UNNEST(tags)。是处理嵌套数据的核心动作。
-- 把每个用户的 tags 数组炸成多行，再统计每个标签的人数（Spark/Hive 语法）
SELECT tag, COUNT(*) AS cnt
FROM users
LATERAL VIEW explode(tags) t AS tag
GROUP BY tag;

5.2 近似计算：用精度换速度

近似聚合函数：在海量数据上，精确的 COUNT(DISTINCT) 极贵（要把所有值 shuffle 到一起去重）。approx_count_distinct 用 HyperLogLog 算法，以极小内存给出误差 ~1-2% 的去重计数，快几个数量级。同理 approx_percentile（近似分位数，用于 P50/P99）。大数据分析里，"差不多准 + 快" 常比 "绝对准 + 慢" 更有价值——这是和 OLTP SQL 思维的一大不同。
-- 精确去重（慢，大数据上可能跑挂）vs 近似去重（快，误差 ~1%）
SELECT COUNT(DISTINCT user_id) FROM events;          -- 精确，重 shuffle
SELECT approx_count_distinct(user_id) FROM events;    -- HLL，快得多
采样（TABLESAMPLE / TABLESAMPLE BERNOULLI(10)）也是大数据 SQL 常用招：先在 10% 样本上探索/调试，确认逻辑对了再上全量，省时省钱。
06

执行顺序 · 性能 · 常见陷阱

写对只是及格 · 写"快"且"不踩坑"才是高级

6.1 让优化器帮上忙（呼应 D01 / D03）

  • 分区裁剪（Partition Pruning）：表按天/地区分区时，WHERE dt='2026-06-25' 让引擎只扫该分区，跳过其余——海量表上这是数量级的差异。务必在分区列上加过滤。
  • 谓词下推 / 列裁剪：用列存（Parquet）+ 只 SELECT 需要的列 + 尽早 WHERE，让引擎少读数据（D01/D03 讲过的红利）。少写 SELECT *。
  • SARGable 条件：别在过滤/JOIN 列上套函数（如 WHERE date(ts)='...' 或 WHERE substr(id,1,3)=...），否则无法走索引/分区裁剪，退化成全扫。改写成对常量做变换（WHERE ts >= '...' AND ts < '...'）。

6.2 大数据 SQL 的性能要点

问题表现对策
COUNT(DISTINCT)重 shuffle、慢能容忍误差用 approx_count_distinct
Join key 数据倾斜个别 reduce task 卡住（D03）加盐 / 开 AQE 倾斜处理 / 热点单独处理
大表 join 大表巨量 shuffle小表侧 broadcast；或预聚合后再 join

6.3 容易翻车的陷阱清单

NULL 三值逻辑：NULL = NULL 结果是 NULL（不是 true）；NOT IN (含 NULL) 返回空集；聚合函数 SUM/AVG 忽略 NULL 但 COUNT(*) 不忽略、COUNT(col) 忽略。处理前用 COALESCE / IS NULL 想清楚。
其他常见坑：① GROUP BY 后 SELECT 非聚合列（多数引擎报错）；② 隐式类型转换（字符串 join 数字）导致全扫或错配；③ 漏 join 条件造成笛卡尔积（行数爆炸）；④ JOIN 后行被放大导致聚合重复计数（用 semi join 或先去重）；⑤ 浮点数等值比较。

6.4 去重模板（防重放、防多次入湖）

在日志回放、CDC 重试、离线重跑场景中，最稳妥的去重写法是：按业务唯一键分组，用窗口函数选最新一条。
SELECT * FROM (
  SELECT t.*,
         ROW_NUMBER() OVER (
           PARTITION BY biz_id
           ORDER BY event_time DESC, ingest_time DESC
         ) AS rn
  FROM fact_events t
) x
WHERE rn = 1;
若引擎支持 QUALIFY，可改写为 ... QUALIFY ROW_NUMBER() OVER(...) = 1，可读性更高。

小结 · 面试 · 术语表

把高级 SQL 收成肌肉记忆

一页纸总结

① 执行顺序

FROM→WHERE→GROUP BY→HAVING→SELECT→ORDER BY→LIMIT。解释别名/HAVING/窗口位置之谜。

② 窗口函数

OVER(PARTITION BY…ORDER BY…)，不合并行。排名/偏移/开窗聚合；frame 做累计与移动平均；QUALIFY 过滤。

③ 多维聚合

ROLLUP（层级小计）/CUBE（全交叉）/GROUPING SETS（自定义）+ FILTER 条件聚合。

④ CTE / 递归

WITH 拉平嵌套；WITH RECURSIVE 走层级/树/图（锚 + 递归成员，防死循环）。

⑤ Join / 子查询

semi/anti（EXISTS/NOT EXISTS，不放大行）；LATERAL；UNION ALL 优先；NOT IN+NULL 坑。

⑥ 大数据特色 + 性能

复杂类型/explode；近似函数换速度；分区裁剪/列裁剪/SARGable；NULL 三值逻辑。

面试自测（点击展开）

基础SQL 的逻辑执行顺序是什么？为什么 WHERE 里不能用 SELECT 的别名？
顺序FROM/JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT。
别名WHERE 在 SELECT 之前执行，那时别名还没生成，所以 WHERE 用不了；ORDER BY 在 SELECT 之后，能用。
核心窗口函数和 GROUP BY 的区别？怎么取"每个部门工资前 3"？
区别GROUP BY 合并行成聚合；窗口函数保留每一行、附加一列基于窗口的计算。
Top-NROW_NUMBER() OVER(PARTITION BY dept ORDER BY salary DESC) 编号，外层 WHERE rn≤3（或 QUALIFY）。
辨析ROW_NUMBER 唯一序号；RANK 并列跳号；DENSE_RANK 并列不跳。按需求选。
高频RANK、DENSE_RANK、ROW_NUMBER 有什么区别？
ROW_NUMBER严格唯一序号，并列也给不同号（1,2,3,4）。
RANK并列同号、跳号（1,2,2,4）。
DENSE_RANK并列同号、不跳号（1,2,2,3）。
实战为什么 NOT IN 有时返回空集？怎么避免？
原因子查询结果含 NULL 时，x NOT IN (…, NULL) 在三值逻辑下永不为 true → 整体空集。
避免改用 NOT EXISTS（anti join），它对 NULL 安全；或在子查询里过滤掉 NULL。
大数据大表上 COUNT(DISTINCT) 很慢，怎么优化？
原因精确去重要把所有值 shuffle 到一起，重 IO/网络。
优化能容忍 ~1% 误差就用 approx_count_distinct（HyperLogLog）；或预聚合、按维度预计算去重结果。
延伸若 distinct key 倾斜，配合加盐/AQE（D03）。

术语表速查 · Glossary

本篇核心名词一次收齐：

① 基础与窗口

逻辑执行顺序FROM→WHERE→GROUP BY→HAVING→SELECT→DISTINCT→ORDER BY→LIMIT。解释别名可用性、HAVING/WHERE 之别。
窗口函数 Window Function在相关行窗口上计算但不合并行，靠 OVER(PARTITION BY…ORDER BY…[frame]) 定义。
ROW_NUMBER / RANK / DENSE_RANK排名函数：唯一序号 / 并列跳号 / 并列不跳号。
LAG / LEAD取窗口内前/后第 N 行的值。做同比环比、差分。
Frame 窗口框架ROWS/RANGE BETWEEN… 限定窗口取值范围，实现累计、移动平均。
QUALIFY过滤窗口函数结果的子句（之于窗口如 HAVING 之于 GROUP BY）。

② 聚合与 CTE

ROLLUP / CUBE / GROUPING SETS多维聚合：层级小计 / 全交叉 / 自定义组合。一次出"明细+小计+总计"。
GROUPING()判断某行某维度是否被卷起（小计行），返回 1/0，用于美化标签。
FILTER (WHERE…)条件聚合的标准写法，比 SUM(CASE WHEN…) 清晰。
CTE WITH命名子查询，拉平嵌套、提升可读性、可复用。不一定物化。
递归 CTE WITH RECURSIVE引用自身遍历层级/树/图：锚成员 UNION ALL 递归成员。注意防死循环。

③ Join 与子查询

Semi / Anti Join存在性匹配（EXISTS）/ 不存在匹配（NOT EXISTS）。不放大行、不取右表列。
LATERAL / APPLY让右子查询引用左表当前行，对每行执行依赖该行的查询。
相关子查询 Correlated子查询引用外层列，逻辑上对外层每行重算一次。优化器常改写为 join。
UNION (ALL) / INTERSECT / EXCEPT集合并/交/差。UNION 去重（慢），UNION ALL 不去重（优先）。

④ 大数据特色与性能

ARRAY/MAP/STRUCT嵌套复杂类型，承载半结构化数据。点/下标/键访问。
explode / LATERAL VIEW / UNNEST把数组/Map 炸成多行，便于后续聚合。
approx_count_distinctHyperLogLog 近似去重计数，以 ~1% 误差换数量级提速。
分区裁剪 Partition Pruning在分区列上过滤，引擎只扫相关分区，海量表的数量级优化。
SARGable过滤/Join 列不套函数，才能走索引/分区裁剪，避免全扫。
三值逻辑 NULLNULL 比较结果是 NULL；NOT IN 含 NULL 返回空集；聚合多忽略 NULL。
