背景与需求

省属国企「华港集团」要建生产经营数仓 · 先把需求和约束说清楚

场景：省属综合性国企「华港集团」，下辖 12 家二级子公司（制造、能源、贸易），各子公司自建 ERP / MES / 财务系统，数据孤岛严重。集团信息中心要建统一生产经营数据平台，支撑 T+1 经营日报、月度国资监管报送、以及厂区生产运行监控大屏。原来靠子公司手工填报 Excel，现在要正经做离线数仓。好的架构始于把需求和约束讲清楚——国企项目尤其不能一上来就堆「互联网大厂全家桶」。

业务需求（功能）

  • 集团生产经营日报：各子公司产量、产值、能耗 T+1 汇总
  • 国资监管报送：按月产出标准化指标包，口径与集团一致
  • 多维经营分析：按组织/产品线/厂区的历史趋势与同比
  • 生产运行监控（二期）：关键产线设备状态、异常告警大屏

非功能需求（约束）

  • 延迟：经营分析 T+1；监控类分钟级可接受
  • 规模：日增百 GB 级明细，历史留存 5–10 年
  • 合规：等保二级、数据分级分类、审计留痕
  • 信创：优先 Hive/Spark 成熟栈，云上用 DataWorks 亦可
  • 团队：6 人数据小组，运维以稳为主、拒绝过度花哨
需求驱动架构（Requirement-Driven Architecture）：架构是约束的产物，不是技术的堆砌。同样"做大数据"，5 人小团队和 500 人大厂的最优解天差地别。先问清延迟要求、数据规模、成本预算、合规要求、团队能力这五个维度，再谈选型——这是架构师与"工具调包侠"的区别（呼应 architect/architecture_methodology.html 的 ATAM 质量属性思想）。
国企数仓项目像集团财务合并报表：各子公司先按统一科目记账（分层建模），月底总部汇总（DWS/ADS），监管机构和领导看的是同一套口径。最怕的是 A 厂用「产量」、B 厂用「产出吨数」——口径不一，汇报全乱。所以建模和指标定义，比选 Flink 还是 Spark 更重要。

02

关键架构决策（ADR）

架构 = 一连串有据可查的权衡 · 用 ADR 记录"为什么这么选"

ADR（Architecture Decision Record，架构决策记录）：用简短文档记录"我们面临什么问题、有哪些选项、最终选了什么、为什么"。它让架构决策可追溯、可复盘，避免半年后没人记得"当初为啥用 DolphinScheduler 不用 Airflow"（呼应 architect/ 板块）。下面是「华港集团」的 5 条核心 ADR。
ADR-1
Hive + ORC 分区表，而非一步到位上 Iceberg 湖仓
问题：存储底座选传统 Hive 数仓，还是对象存储 + Iceberg 湖仓？
决策：Hive on HDFS（ORC + 按日分区） 作为一期主存储；Iceberg 作为二期演进选项。
理由：国企现网、招标文档、运维团队对 Hive 最熟；信创环境与集团审计都偏好成熟栈；招聘与外包交接成本低。（D01）
ADR-2
离线优先，实时监控二期再上
问题：经营分析与产线监控，一期要不要同时上 Flink？
决策：一期 Spark SQL 批处理 T+1 满足集团日报与国资报送；二期对 MES/IoT 关键指标上 Flink + Kafka。
理由：国企核心诉求是口径统一、可审计、稳定跑批，不是毫秒级 GMV；6 人团队先把离线链路跑通更务实。（D08）
ADR-3
摄取以 DataX 批量同步为主
决策：子公司 ERP/财务库用 DataX 夜间全量/增量同步进 ODS；核心系统二期再评估 Flink CDC。
理由：DataX 在政企项目里部署经验多、可视化配置友好；T+1 场景不必强行实时 CDC。（D06）
ADR-4
服务层选 MySQL + ClickHouse（而非重型 OLAP 集群）
决策：ADS 结果 MySQL 供固定报表与监管接口；多维分析用 ClickHouse 承接查询。
理由：集团日报数据量在 ClickHouse 单机/小集群可承受；比上 StarRocks 集群运维更轻，符合信息中心能力。（D07）
ADR-5
编排 DolphinScheduler + Ranger 治理从第一天起
决策：DolphinScheduler 调度 Spark/Hive 任务；Apache Ranger 做库表级权限与审计；指标口径文档化。
理由：DS 是国内政企项目最常见调度器之一（杭州很多集成商默认栈）；治理与等保要求「谁能看什么表」必须可追溯。（D02/D09）

03

数据建模与分层落地

把 D02 落到生产经营主题 · 集团统一口径

「华港集团」围绕核心业务过程生产报工做维度建模（D02 的 Kimball 四步法），并落成 ODS→DWD→DWS→ADS 分层。组织架构（集团—子公司—厂区）用一致性维度贯穿各层，保证汇总到集团总部时口径一致。

层内容离线（Spark SQL / Hive）实时（二期 Flink）
ODS贴源：ERP 工单、MES 报工、能耗采集、组织主数据Hive 外部表 / 按日落分区Kafka 原始 topic（IoT）
DWD明细事实 fact_production + 维度（SCD2 组织架构）清洗去重、统一计量单位、关联维度Flink 清洗 + 时态 join
DWS按日×子公司×产品线：产量、产值、能耗汇总预聚合宽表分钟级产线指标（可选）
ADS集团生产经营日报、国资报送指标包、厂区排行MySQL / ClickHouse监控大屏（二期）
一套口径、多级汇总：fact_production 的粒度（每条报工记录）、维度（组织/产品/厂区）、指标定义（产量、产值、综合能耗）在集团信息中心文档化——各子公司 ETL 必须遵守。「产量」是完工合格品数量还是投料量，必须在 DWD 层统一，否则集团日报无法合并。这是国企数仓比互联网更「硬」的地方。
-- DWD 事实表：Spark SQL 写入 Hive 分区表（按业务日期 dt 分区）
INSERT OVERWRITE TABLE dwd_fact_production PARTITION (dt='${biz_date}')
SELECT
    dd.date_key, dorg.org_key, dprod.product_key,   -- 维度代理键（组织用 SCD2 拉链挂报工当日版本）
    p.work_order_id,                                 -- 退化维度
    p.output_qty,                                    -- 产量（已统一为「合格品吨数」）
    p.output_qty * dprod.unit_price AS output_value,
    p.energy_kwh                                     -- 能耗
FROM ods_mes_production p
JOIN dim_organization dorg ON p.plant_code = dorg.plant_code
   AND p.report_time >= dorg.start_date AND p.report_time < dorg.end_date
JOIN dim_product dprod ON p.product_code = dprod.product_code
JOIN dim_date dd ON date_format(p.report_time, 'yyyyMMdd') = dd.date_key
WHERE p.dt = '${biz_date}';

3.1 Hive 实操动手（分区表 · 外部表 · ORC · EXPLAIN）

下面是一套可直接在 Hive CLI / Beeline 或 Spark SQL（Hive Metastore） 里跑的 DDL 与 ETL 模板，对应华港集团 ODS→DWD。本地练习可用 Docker 镜像 apache/hive + bde2020/spark-master，或云厂商单机实验环境。

Managed Table（内部表） vs External Table（外部表）：内部表删表时数据一并删除；外部表只删元数据，HDFS 上的数据文件保留。ODS 贴源层常用外部表指向 DataX 落地的 HDFS 路径，方便重跑同步而不动 Hive 元数据。
-- ① ODS 外部表：指向 DataX 同步到 HDFS 的 CSV/ORC 目录
CREATE EXTERNAL TABLE IF NOT EXISTS ods_mes_production (
  work_order_id  STRING,
  plant_code     STRING,
  product_code   STRING,
  output_qty     DOUBLE,
  energy_kwh     DOUBLE,
  report_time    TIMESTAMP
)
PARTITIONED BY (dt STRING)          -- 按业务日期分区，T+1 跑批必备
STORED AS ORC                          -- 列存 + 高压缩，Hive 分析标配
LOCATION 'hdfs://ns1/warehouse/ods/mes_production';

-- ② 同步新分区后，让 Hive 感知（DataX 落盘后执行）
ALTER TABLE ods_mes_production ADD IF NOT EXISTS PARTITION (dt='2026-06-01')
  LOCATION 'hdfs://ns1/warehouse/ods/mes_production/dt=2026-06-01';

-- ③ DWD 内部管理表：ORC + 分区（下游 Spark 主要写这张）
CREATE TABLE IF NOT EXISTS dwd_fact_production (
  date_key      INT,
  org_key       INT,
  product_key   INT,
  work_order_id STRING,
  output_qty    DOUBLE,
  output_value  DOUBLE,
  energy_kwh    DOUBLE
)
PARTITIONED BY (dt STRING)
STORED AS ORC
TBLPROPERTIES ('orc.compress'='SNAPPY');
INSERT OVERWRITE：按分区覆盖写入，是离线数仓 T+1 跑批的标准写法——重跑同一天任务不会产生重复数据（幂等）。对应 Spark SQL 里同样语法；切忌对分区表用纯 INSERT INTO 追加导致重复。
-- ④ 覆盖写入 DWD 分区（与上文 SELECT 逻辑一致，这里强调 Hive 写法）
INSERT OVERWRITE TABLE dwd_fact_production PARTITION (dt='${biz_date}')
SELECT ... -- 见上文完整 SELECT
;

-- ⑤ 执行计划：排查慢 SQL、确认分区裁剪是否生效
EXPLAIN EXTENDED
SELECT plant_code, SUM(output_qty)
FROM ods_mes_production
WHERE dt = '2026-06-01'    -- 有 dt 条件 → 应只扫一个分区目录
GROUP BY plant_code;
EXPLAIN 关注点正常表现异常信号
分区裁剪Partition Filters: dt=...扫描全部分区 → WHERE 漏写 dt
文件格式ORC / VectorizedTextFile 大表 → 考虑改 ORC
Join小表 MapJoin / Broadcast大表 Shuffle Join → 看数据倾斜
Spark SQL on Hive Metastore：生产里常写 spark-sql -f dwd.sql 或 DolphinScheduler 调 Spark 任务，表存在 Hive Metastore 里，语法与 Hive 基本一致。本地可用 spark-shell 配 spark.sql.catalogImplementation=hive 练习。

05

治理 · 成本 · 可观测性落地

从第一天就做 · 别等变成数据沼泽再补

治理（D09）

  • Ranger 库/表/列级权限 + 操作审计
  • 调度任务前后置质量校验（行数、空值率、环比突变）
  • 敏感字段（产值细目、人员信息）分级脱敏

成本与资源（D09）

  • Hive 历史分区按年归档冷存储
  • Spark 动态分区裁剪 + 合理并行度
  • 按子公司/主题域打标签，成本可分摊

可观测（D09）

  • 批任务就绪 SLA（日报 08:00 前）
  • DolphinScheduler 失败告警 + 自动重试
  • 二期：Kafka lag / Flink 反压监控
治理"左移"在国企尤其重要：国资监管、等保测评、集团审计都会问「数据从哪来、谁改过、谁能看」。「华港」的做法是权限与审计从第一天内建——新表进 Ranger 策略、调度任务带质量门、报送指标有版本文档。这比互联网多一层「合规交付」——也是国企数据岗的核心价值。
06

演进路径与反模式

不要一步到位 · 也别踩这些坑

演进三阶段（国企项目典型节奏）

阶段一 · MVP（前 3–6 个月）
先把离线 T+1 跑通
DataX → Hive ODS → Spark SQL 分层 → MySQL/ClickHouse 日报。暂不上实时，先满足集团生产经营日报与统一口径，通过验收。
阶段二 · 扩展（6–12 个月）
接入更多子公司 + 国资报送
扩展数据源、完善组织拉链表、上线国资监管指标包；对重点厂区加 Kafka + Flink 产线监控。
阶段三 · 深化（12 个月+）
治理深化 + 湖仓演进（可选）
完善血缘/质量平台、指标自助查询；评估 Iceberg 平滑迁移或上云 DataWorks。

常见反模式（国企项目也踩）

① 照搬互联网湖仓全家桶

招标写 Iceberg + Flink + StarRocks，运维团队 hold 不住，最后只有 Hive 在跑。成熟栈优先。

② 口径不统一就急着接数

12 家子公司各写各的「产量」，集团日报永远对不上。先定指标字典和 DWD 规范。

③ 一期就上实时大屏

经营分析 T+1 够用，却先花半年做 Flink——领导要的是稳定日报，不是炫酷大屏。

④ 忽视权限与审计

等保测评才发现「谁都能查全集团产值」。Ranger + 审计日志从第一天起。

容量规划速算：日产报工记录 ~200 万条 × 每条约 0.5KB ≈ 1GB/天 明细；5 年留存 ORC 压缩后约数百 GB 级，单机 Hive + modest Spark 集群可承载。国企项目常见误区是过度预估规模——先把口径和流程跑通，再按需扩容。

面试综合题（点击展开）

系统设计请为一家省属国企设计生产经营离线数仓（多子公司数据汇聚）。
先问需求报送时效 T+1 还是实时？数据源有哪些？等保/权限要求？团队与现网技术栈？
架构DataX 批量同步 → Hive ODS → Spark SQL 分层(DWD/DWS/ADS) → MySQL/ClickHouse 报表；DolphinScheduler 调度。
关键集团统一指标口径；组织维度 SCD2/拉链表；Hive 分区 + ORC；Ranger 权限与审计。
演进一期离线跑通验收 → 二期扩展子公司与实时监控 → 可选 Iceberg/DataWorks 上云。
权衡国企项目为什么常见 Hive，而不是一上来 Iceberg？
成熟度Hive 在政企现网、招标、运维手册、人员招聘上生态最成熟。
需求匹配T+1 批处理为主，Hive + Spark 足够；湖仓优势在流批一体与 schema 演进，一期非刚需。
原则技术选型服从交付风险与团队能力，不是越新越好。
综合贯穿国企数仓的几条"第一性原理"是什么？
口径统一集团—子公司—厂区同一套指标定义，DWD 层落地（D02）。
稳定跑批T+1 SLA 比实时炫酷更重要；调度幂等、可回填（D06）。
可审计谁看了什么数据、任务何时跑完，必须留痕（D09）。
按需演进离线先行、实时后置；成熟栈先行、湖仓可选。
🎓 系列结语：十篇走完，你应该建立起一个完整的心智地图——大数据不是一堆孤立的时髦组件，而是围绕「把分散的业务数据，用可接受的成本与延迟、可信赖地汇聚成集团可用的指标」这一目标，层层咬合的工程体系。对国企大数据岗，Hive + Spark + 调度 + 建模 + 治理 是比追 Flink/Iceberg 更稳的求职基本盘。祝你项目顺利、面试顺利。
