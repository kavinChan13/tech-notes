背景与需求

电商「鲜购」要建数据平台 · 先把需求和约束说清楚

场景：生鲜电商「鲜购」，日订单 ~500 万、日活百万级。原来只有业务库 + 几张人肉跑的 Excel 报表，现在要建一套正经的数据平台。好的架构始于把需求和约束讲清楚——不是上来就选 Flink。

业务需求（功能）

  • 实时 GMV 大屏：大促期间秒级看销售
  • 离线经营分析：T+1 的多维报表、留存、复购
  • 实时风控：异常下单秒级拦截
  • 推荐特征：为推荐系统供实时 + 离线特征

非功能需求（约束）

  • 延迟：实时类秒级，分析类 T+1 可接受
  • 规模：日增 TB 级，历史留存 2 年
  • 成本：初期预算有限，要可控可弹性
  • 合规：用户 PII 要保护，满足个保法
  • 团队：5 人数据团队，运维能力有限
需求驱动架构（Requirement-Driven Architecture）：架构是约束的产物，不是技术的堆砌。同样"做大数据"，5 人小团队和 500 人大厂的最优解天差地别。先问清延迟要求、数据规模、成本预算、合规要求、团队能力这五个维度，再谈选型——这是架构师与"工具调包侠"的区别（呼应 architect/architecture_methodology.html 的 ATAM 质量属性思想）。
先定需求再选技术，就像先确定"住几口人、预算多少、通勤多远"再买房，而不是看到一套豪宅就冲动下单。很多数据平台失败，不是技术不行，而是一上来就照搬大厂架构——5 人团队硬上一套需要 50 人维护的复杂栈，最后没人 hold 得住。

基于需求，「鲜购」选了一套湖仓 + 实时离线一体的现代架构。下面这张端到端蓝图，每个组件都能在前 9 篇里找到出处：

源 → 摄取(D06) → 实时(D04)/离线(D03) 双链路 → 湖仓(D01/D02) → OLAP 服务(D07)，治理(D09)与编排贯穿全程。实时链路也可直供服务层（D08）。

02

关键架构决策（ADR）

架构 = 一连串有据可查的权衡 · 用 ADR 记录"为什么这么选"

ADR（Architecture Decision Record，架构决策记录）：用简短文档记录"我们面临什么问题、有哪些选项、最终选了什么、为什么"。它让架构决策可追溯、可复盘，避免半年后没人记得"当初为啥用 Kafka 不用 Pulsar"（呼应 architect/ 板块）。下面是「鲜购」的 5 条核心 ADR。
ADR-1
湖仓一体（Iceberg）而非传统数仓
问题：存储底座选传统 Hadoop 数仓，还是对象存储 + 表格式？
决策：对象存储 + Apache Iceberg（湖仓）。
理由：存算分离弹性、成本低、schema 演进 + ACID + time travel；小团队不想自维护 HDFS 集群。（D01）
ADR-2
实时走 Kappa 风格，不上重型 Lambda
问题：实时需求用 Lambda 双链路还是 Kappa？
决策：以 Flink 流为主（Kappa 风格），离线用 Spark 跑全量回算 / 复杂分析，口径靠指标层统一。
理由：5 人团队维护不起两套对齐的代码；实时离线一体降低心智负担。（D08）
ADR-3
摄取用 Flink CDC（不引入独立 Debezium 集群）
决策：业务库变更用 Flink CDC 直接进 Flink，埋点经 Kafka。
理由：少维护一个 Debezium/Connect 集群；Flink CDC 全量+增量无缝。（D06）
ADR-4
服务层选 StarRocks（而非 ClickHouse）
决策：OLAP 服务层用 StarRocks。
理由：电商分析多表 join 多（订单×商品×用户），StarRocks 的 CBO + colocate join 比 ClickHouse 更合适；主键模型支持实时 upsert。（D07）
ADR-5
编排 Airflow + 转换 dbt + 治理从第一天起
决策：Airflow 调度、dbt 管离线转换与测试、DataHub 做血缘/目录。
理由：治理不是"以后再补"——第一天就建血缘 + 质量门 + 成本标签，避免日后变数据沼泽。（D02/D09）

2.1 决策背后：没有放之四海皆准的架构

注意：以上每条决策都绑定「鲜购」的具体约束（5 人团队、预算有限、电商多表 join）。换个场景，最优解就变了。这正是下面这个演示要展示的——需求不同，推荐的技术栈不同。

03

数据建模与分层落地

把 D02 落到电商主题 · 一套模型，流批共用

「鲜购」围绕核心业务过程下单做维度建模（D02 的 Kimball 四步法），并落成离线 + 实时共用一套口径的分层。

层内容离线（Spark/dbt）实时（Flink）
ODS贴源订单/商品/用户/埋点Iceberg 原始表Kafka 原始 topic
DWD明细事实 fact_order_item + 维度（SCD2 客户）清洗去重、关联维度Flink 清洗 + 时态 join
DWS按天×品类 GMV、用户行为汇总预聚合宽表Flink 实时聚合（retract）
ADSGMV 大屏、经营报表、推荐特征StarRocks 报表表StarRocks 主键表（实时）
一套模型、两种时效（实时离线一体）：fact_order_item 的粒度、维度、指标口径离线和实时完全一致——「GMV」无论走 Spark 还是 Flink，定义都是同一个（D02 指标语义层兜底）。这样实时大屏的数和次日离线报表的数才对得上，业务才敢信。这是整套架构正确性的基石。
-- DWD 事实表：离线与实时共用同一套口径定义（这里是离线 dbt 版）
SELECT
    dd.date_key, dp.product_key, dc.cust_key,   -- 维度外键（dc 关联下单时刻的客户版本 SCD2）
    o.order_id,                                  -- 退化维度
    oi.qty, oi.qty * oi.price AS amount       -- 度量：口径与实时 Flink SQL 严格一致
FROM ods_order_item oi
JOIN ods_order o ON oi.order_id = o.order_id
JOIN dim_customer dc ON o.cust_id = dc.cust_id
   AND o.created_at >= dc.valid_from AND o.created_at < dc.valid_to;

端到端数据流

离线链路 + 实时链路 · 一笔订单的两条命运

4.1 离线链路 + 实时链路

离线链路（T+1 经营分析）

1→CDC/埋点 落 Iceberg ODS
2→Airflow 凌晨触发 Spark+dbt 跑 DWD→DWS→ADS
3→结果入 StarRocks，次日报表/留存/复购分析

实时链路（秒级大屏/风控）

1→Flink CDC 实时捕获订单变更
2→Flink 时态 join 补维 + 聚合 GMV（retract 处理退款）
3→写 StarRocks 主键表，大屏秒级刷新；风控订阅异常

05

治理 · 成本 · 可观测性落地

从第一天就做 · 别等变成数据沼泽再补

治理（D09）

  • DataHub 自动采集血缘 + 数据目录
  • dbt test / GE 做质量门（主键唯一、金额>0）
  • PII 字段（手机号/地址）打标 + 列级权限 + 脱敏

成本 FinOps（D09）

  • Iceberg 冷热分层 + 历史 2 年后归档
  • Spark 弹性 + Spot 实例跑离线
  • 资源打标签，成本归因到主题域

可观测（D09）

  • 数据新鲜度监控（大屏延迟、批就绪时间）
  • 质量看板 + 量级突变告警
  • Flink 反压 / Kafka lag 监控（D04/D06）
治理"左移"：很多团队把治理当"以后再补"的脏活，结果数据爆炸后追悔莫及。「鲜购」的做法是治理从第一天起、内建进管道——新表自动进目录、质量门挡在入库前、成本标签建表即打。这点额外成本，能避免日后指数级的清理代价。这也是资深团队最值钱的经验。
06

演进路径与反模式

不要一步到位 · 也别踩这些坑

演进三阶段（按需长大，别过度设计）

阶段一 · MVP（前 3 个月）
先把离线跑通
CDC/埋点 → Iceberg → Spark+dbt 分层 → StarRocks 报表。暂不上实时，先满足 T+1 经营分析，验证建模与口径。
阶段二 · 实时（3–6 个月）
加上 Flink 实时链路
大促前上实时 GMV 大屏 + 风控。复用阶段一的模型口径，Kappa 风格。
阶段三 · 平台化（6 个月+）
治理深化 + 自助化
完善血缘/质量/成本、指标平台、自助查询。团队/业务变多时考虑数据网格。

常见反模式（踩了就翻车）

① 一上来照搬大厂全家桶

5 人团队硬上 50 人才 hold 得住的复杂栈，运维压垮自己。按需长大。

② 治理留到"以后"

数据先堆着不管，半年后变数据沼泽，清理成本指数级。治理左移。

③ 实时离线两套口径

大屏和报表对不上数，业务失去信任。口径必须统一。

④ 唯实时论 / 唯新技术论

不是所有需求都要实时；T+1 够用就别上 Flink。需求驱动，别为技术而技术。

容量规划速算（老兵习惯）：日订单 500 万 × 每单 ~2KB ≈ 10GB/天 明细；2 年留存 ≈ 7TB（列存压缩后可能 ~1-2TB）。Kafka 分区数按峰值吞吐 / 单分区吞吐估，留 2-3x buffer。Flink 并行度 ≈ 分区数。先量化再选型——这又回到 D01 "单机搞不搞得定" 的第一性思考。

给 C++ / 系统老兵的转型建议

你的最大杠杆点：整条路线里，D03 Tungsten、D07 向量化/SIMD/列存引擎内核是你的主场——分布式分析引擎的单机性能优化，就是你熟悉的 cache 局部性、SoA、SIMD、避免虚调用那一套。从这里切入（如 ClickHouse / StarRocks / Velox 内核、或 AI Infra），比从"业务建模"切入更有竞争力（呼应 cpp/transition_to_ai_infra.html）。而 D02 建模、D09 治理这些"软"能力，则是你需要补的、决定你能否走到资深/架构的短板。

面试综合题（点击展开）

系统设计请为一个日订单千万级的电商设计实时 + 离线数据平台。
先问需求延迟要求？规模？预算？合规？团队规模？（需求驱动架构，别上来选型）
架构CDC/Kafka 摄取 → Flink 实时 + Spark/dbt 离线 → 湖仓(Iceberg) → StarRocks 服务层 → 大屏/BI。
关键实时离线一套口径（指标层）；Kappa 风格减双代码；SCD2 保历史；retract 处理退款。
治理第一天起建血缘/质量门/PII 权限/成本标签；按 MVP→实时→平台化 演进，不过度设计。
权衡什么时候该上实时数仓？什么时候 T+1 离线就够？
上实时决策窗口以秒/分钟计：风控、大促大屏、实时推荐、库存预警。
T+1 够经营月报、留存复购、模型训练等隔天看没问题的分析——实时反而是浪费。
原则实时有显著的复杂度/成本溢价，按业务决策时效需求决定，不为实时而实时。
综合贯穿这套架构的几条"第一性原理"是什么？
少干活列存+谓词下推(D01)、少shuffle(D03)、分区裁剪(D05)、预聚合(D07)——既快又省。
分而治之分片/分区是并行与扩展的根(D01/D03/D06)；本地预聚合减传输。
权衡延迟vs吞吐、准vs快、空间vs时间、一致性vs可用性——处处是取舍，无银弹。
工程纪律口径统一、治理左移、需求驱动、按需演进——技术之外，这些决定成败。
🎓 系列结语：十篇走完，你应该建立起一个完整的心智地图——大数据不是一堆孤立的时髦组件，而是围绕「把低密度的数据价值，用可接受的成本与延迟、可信赖地提炼出来」这一目标，层层咬合的工程体系。组件会更替（明年也许又有新引擎），但这套第一性原理与权衡思维会长久陪伴你。作为 C++/系统老兵，你比多数人更懂"性能与权衡"的底层逻辑——这是你进入数据 / AI Infra 领域最坚实的底气。祝你玩得开心。
