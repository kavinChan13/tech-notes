00

数据摄取全景 · ETL vs ELT

数据从源头流进数据栈的第一公里 · 批 / 流 / CDC 三种姿势

数据不会自己跑进数仓。摄取（Ingestion）是把数据从各种源头（业务库、日志、埋点、第三方 API）搬进数据平台的第一步。它决定了下游能拿到多新、多全、多干净的数据。

ETL vs ELT：两种摄取范式。ETL（Extract-Transform-Load）——先抽取、在中途转换好、再装载进仓，转换发生在入仓前（传统数仓时代）。ELT（Extract-Load-Transform）——先把原始数据原样装进仓/湖，再用仓内强大算力做转换（现代云数仓主流，dbt 即此范式，呼应 D02）。ELT 的好处：原始数据留底可重跑、转换逻辑用 SQL 在仓内统一管理。
三种摄取姿势：批量（定时全/增量同步）、流式（实时事件经 Kafka）、CDC（捕获数据库变更）。Kafka 常作为汇聚一切的中枢。
本篇先讲采集端（本章，数据进 Kafka 前的最后一公里），再讲中枢 Kafka（第 1-3 章）、CDC（第 4 章）、编排（第 5 章）。流式计算引擎本身在 D04（Flink）已讲。

0.1 采集端：数据进 Kafka 之前的「最后一公里」

上面的全景图里，业务库走 CDC、流式事件走 Kafka——但日志和埋点数据是怎么变成 Kafka 里的一条条消息的？答案是采集端（Collection Agent）。它常被初学者忽略，却是真实数据平台里最容易丢数据、最难排查的一环。

如果 Kafka 是中央邮局，采集端就是街角的取件员：他要挨家挨户（每台服务器、每个 App）把信件（日志行 / 埋点事件）收上来，本地先装进背包（缓冲），路上不能弄丢，到邮局后准确投递。取件员干得好不好，直接决定邮局收到的信全不全、及不及时。
两条典型采集路：① 客户端埋点 → 上报服务 → Kafka；② 服务器日志文件 → 采集 Agent → Kafka。Agent 在本地缓冲、批量、断点续传。

0.2 埋点 SDK 与日志采集 Agent

埋点（Tracking / Instrumentation）：在 App / 网页 / 服务代码里主动记录用户行为或业务事件（如「点击了商品」「下单成功」），按统一的事件 schema 上报。分两类：客户端埋点（App/Web 端 SDK 采集，能拿到 UI 行为，但受网络/防丢影响）与服务端埋点（在后端服务里打点，可靠、不易被篡改，但拿不到纯前端行为）。埋点数据是用户行为分析、推荐、AB 实验的原料（呼应 pm/data_ab_testing）。
埋点规范是数据质量的源头：事件命名、字段类型、必填项若不统一，下游再多治理也救不回来。成熟团队会用埋点元数据管理 + schema 校验（如上报时校验事件结构，呼应 D06 的 Schema Registry 与 D09 的数据契约），从源头卡住脏数据。这就是「数据质量左移」。

服务器上的日志文件，则由采集 Agent 监听、读取、发往 Kafka。主流 Agent 各有取向：

Agent出身 / 生态特点适用
FilebeatElastic 系轻量、低资源、稳定日志采集进 ES / Kafka，最常见
LogstashElastic 系功能强但重、吃内存需要复杂解析/富化的场景
Fluentd / Fluent BitCNCF云原生、K8s 日志标配（Bit 极轻）容器 / K8s 日志采集
VectorDatadog（Rust）高性能、可观测性统一管道现代新选，logs/metrics 统一
FlumeHadoop 系老牌、与 HDFS/Kafka 集成好传统 Hadoop 体系内
采集端的工程难题，本质都是老兵熟悉的可靠性问题：① 不丢——本地磁盘缓冲 + 断点续传（记录读到文件哪个 offset，重启续读）；② 不重——配合下游幂等，做到至少一次 + 去重（呼应 Kafka 的 Exactly-once）；③ 背压——下游/Kafka 写不动时本地缓冲并降速，别撑爆内存；④ 多行/格式——堆栈日志多行合并、结构化解析。这些在 D04 反压、D06 可靠性里都会再次相遇。
01

Kafka 核心架构

本质是一个分布式、可持久化、可重放的 commit log · 老兵秒懂

Kafka 是大数据世界的「中枢神经」——几乎所有实时数据都先流进 Kafka，再被各路下游消费。别被「消息队列」这个标签局限，它的本质是一个分布式的、持久化的、可重放的提交日志（commit log）。

Kafka 像一份不断追加、永不修改的"流水账本"：生产者只管在账本末尾一行行追加记录（顺序写，极快）；每个读者（消费者）拿一个书签（offset）记自己读到第几行，各读各的、互不干扰；账本按主题分成很多本、每本再撕成几叠（分区）让多人同时记。这就是为什么它又快、又能多方消费、还能"倒回去重读"。

1.1 Topic / Partition / Offset / Consumer Group

Topic 是逻辑主题；它被切成多个 Partition（并行 + 扩展单位）；每条消息在分区内有唯一递增的 Offset；消费者用 offset 记录读到哪。
核心概念一次过：
  • Topic（主题）：一类消息的逻辑分类，如 orders、clicks。
  • Partition（分区）：Topic 的物理切分，是并行与扩展的单位（呼应 D01 分片）。分区内有序，跨分区无序。
  • Offset（偏移量）：消息在分区内的唯一递增编号。消费者提交 offset 记录"读到哪了"，故可重放（把 offset 调回去重读）。
  • Broker：Kafka 服务节点；多个 broker 组成集群，分区副本分散其上。
  • Producer / Consumer：生产者按 key 哈希决定写哪个分区；消费者从分区拉取。
  • Consumer Group（消费者组）：一组协作消费者，组内每个分区只分配给一个消费者（实现并行）；不同组互相独立（各自全量消费）。

02

Kafka 可靠性：副本 / ISR / Exactly-once

数据不能丢 · 这套机制和 D01 的副本、D04 的 exactly-once 一脉相承

2.1 副本、ISR 与 acks

副本（Replica）与 Leader/Follower：每个分区有多个副本分散在不同 broker。其中一个是 Leader（负责所有读写），其余是 Follower（从 Leader 同步数据）。Leader 挂了，从同步副本里选新 Leader——这就是容错。（呼应 D01 副本、共识选主。）
ISR（In-Sync Replicas，同步副本集）：与 Leader保持同步的副本集合。只有 ISR 里的副本才有资格在 Leader 故障时被选为新 Leader——保证选出来的副本数据是全的。落后太多的 follower 会被踢出 ISR。
acks（生产者确认级别）：决定"写入算成功"的标准，是可靠性 vs 吞吐/延迟的权衡：
  • acks=0：发了就算成功，不等确认。最快，可能丢。
  • acks=1：Leader 写入即确认。Leader 挂且未同步则可能丢。
  • acks=all（=-1）：ISR 全部确认才算成功。最可靠，延迟最高。金融/关键数据用它。
# 生产者高可靠配置（不丢数据的典型组合）
    acks=all                      # ISR 全确认
    min.insync.replicas=2         # ISR 至少 2 个才接受写（否则拒绝，宁可不写也不丢）
    enable.idempotence=true       # 幂等生产者：重试不产生重复
    retries=Integer.MAX_VALUE     # 失败无限重试
    delivery.timeout.ms=120000    # 总投递超时，需与重试策略联动设计
    # broker 侧建议：unclean.leader.election.enable=false（避免脏选主导致数据回退）

2.2 顺序保证与 Exactly-once

Kafka 只保证「分区内有序」，不保证「Topic 全局有序」。 所以需要保序的消息（如同一订单的状态变更）必须用同一个 key（如 order_id）——相同 key 哈希到同一分区，自然有序。这是用 Kafka 最容易踩的坑之一。
Kafka 的 Exactly-once（EOS）：通过幂等生产者（去重重复发送）+ 事务（跨分区/Topic 原子写）实现。配合 Flink（D04）的 checkpoint + 两阶段提交，可达端到端精确一次：源（Kafka offset 可重放）→ 计算（Flink 状态快照）→ sink（事务写）。三者缺一不可。这正是 D04 讲的"端到端 exactly-once 需要可重放源 + 事务 sink"的源头。注意：端到端 EOS 不是默认自动获得，只有当 sink 侧支持事务或幂等写，且"处理结果提交"与"消费位点推进"能绑定一致性语义时才成立。
消费语义：取决于何时提交 offset。先处理后提交 = 至少一次（at-least-once）（可能重复，崩在提交前会重处理）；先提交后处理 = 至多一次（at-most-once）（可能丢）；要精确一次需把"处理结果"和"offset 提交"放进同一个事务。

2.3 生产可观测性与告警基线

Kafka 出问题通常不是"挂了"，而是慢慢退化：消费追不上、ISR 抖动、再均衡频繁。上线前应先定义一套可观测性基线：

指标含义告警建议
Consumer Lag消费位点落后最新位点的差值持续增长且超过业务 SLA（例如 15 分钟）即告警
End-to-End Latency从生产到下游可见的总延迟P95/P99 超阈值持续 N 分钟告警
ISR Shrink/Expand同步副本集频繁收缩/恢复短时频繁抖动说明磁盘/网络或负载异常
Rebalance 次数与时长消费者组重分配的频率与停顿时间频繁 rebalance 会带来吞吐抖动与重复消费
Produce/Consume TPS写入与消费速率观察供需差，提前扩容分区/消费实例
DLQ 速率进入死信队列的消息比例突增往往是 schema 变更或下游解析故障
最低限度的监控组合：lag + 端到端延迟 + ISR 抖动 + rebalance + DLQ。少一个维度，排障就容易"盲人摸象"。

2.4 失败处理：重试队列与死信队列（DLQ）

重试策略不是一把梭：先区分错误类型。可重试错误（下游短暂超时、网络抖动）走退避重试；不可重试错误（字段缺失、schema 不兼容、业务校验失败）应直接入 DLQ，避免把主消费线程拖死。
错误类型处理建议备注
临时性故障（5xx/超时）指数退避重试（如 1s/5s/30s）限制最大重试次数，避免雪崩
脏数据/校验失败直接入 DLQ + 告警由离线修复任务二次处理
Schema 不兼容阻断上线并回滚 schema优先在 Registry 做兼容性守门
保序场景（同 key 同分区）中，遇到坏消息不要简单 "skip"。更稳妥做法是：把该 key 暂停、旁路到隔离队列处理，再恢复主链路，避免后续消息乱序或语义破坏。
03

Kafka 生态：Connect · Schema Registry · Pulsar

不只是个队列 · 围绕它长出的一整套数据集成能力

Kafka Connect：Kafka 官方的数据集成框架，用配置（而非写代码）把数据在 Kafka 与外部系统间搬运。Source Connector 把外部数据导入 Kafka（如 Debezium 抓 MySQL binlog）；Sink Connector 把 Kafka 数据导出到外部（如写入 S3 / ES / JDBC）。它解决"又要写一遍数据搬运代码"的重复劳动。
Schema Registry（模式注册中心）：集中管理 Kafka 消息的 schema（常配 Avro/Protobuf）。生产者按注册的 schema 序列化、消费者按 schema 反序列化，并强制schema 兼容性校验（如只允许向后兼容地加字段）。它解决流数据的"schema 演进"难题——上游改了字段不会让下游突然解析失败。呼应 D01 的 Avro（行式、自描述、利于流式写入）。
没有 Schema Registry 的流数据像大家用各自的方言传话，上游改个说法下游就听不懂、整条链路崩。有了它就像全公司统一的术语词典 + 变更审批：改字段要先过兼容性检查，保证老消费者不被突然搞挂。
维度KafkaPulsar
架构存算一体（broker 存数据）存算分离（broker + BookKeeper）
多租户 / 地理复制需额外组件原生支持
生态成熟度最成熟、事实标准较新、生态追赶中
选择绝大多数场景的默认强多租户/分层存储需求时考虑
Kafka 仍是事实标准；Pulsar 的存算分离思想（呼应 D01）在超大规模/多租户场景有优势。了解即可，选型上 Kafka 通常是稳妥默认。

3.1 生产环境安全治理（常被低估）

传输与鉴权

启用 TLS 加密链路，配合 SASL（SCRAM/OAuth 等）做客户端鉴权，避免明文传输与匿名接入。

最小权限 ACL

按 Topic 和 Consumer Group 下发最小权限（读/写/管理分离），禁止一把通配符账号跑全链路。

敏感数据治理

对 PII 字段（手机号/证件号）做脱敏或令牌化；日志与 DLQ 也要遵守同样的数据合规策略。

很多团队只做了吞吐压测，没做安全压测。建议补一次"证书轮换 + ACL 变更 + 凭据过期"演练，确认不会导致大面积消费中断。
04

CDC：变更数据捕获

让数据库的每一次增删改，实时流进数据栈 · 实时数仓的源头活水

业务数据躺在 MySQL / PostgreSQL 里，每天都在增删改。怎么让数仓实时跟上这些变化，而不是 T+1 全表拉一遍？答案是 CDC（Change Data Capture，变更数据捕获）。

CDC（变更数据捕获）：实时捕获数据库的数据变更（INSERT/UPDATE/DELETE）并以事件流输出。主流做法是读数据库的事务日志（MySQL 的 binlog、PostgreSQL 的 WAL）——这是数据库本就为复制/恢复而写的日志（呼应 architect 的 WAL）。读 binlog 几乎不增加业务库负担、且能捕获到每一次变更（包括删除），远优于"定时查全表/查 updated_at"的老办法。
老式同步像每天给整个仓库盘点一次——慢、费力、还抓不到"中途进又出"的货。CDC 像在仓库门口装了监控，每件货的进出立刻记一笔——实时、轻量、一笔不漏（连"删除"都记得到）。这笔"流水"恰恰就是数据库自己写的 binlog。
Debezium 读 MySQL binlog、把每条变更转成结构化事件灌进 Kafka，下游（Flink/湖仓/OLAP）实时消费——OLTP 的每一次写，秒级出现在分析侧。

Debezium

最流行的开源 CDC 工具，以 Kafka Connect Source Connector 形式运行，支持 MySQL/PG/Mongo/Oracle 等。输出标准化的变更事件（before/after 镜像）。

Flink CDC

把 CDC 能力直接集成进 Flink，无需 Kafka 中转即可"数据库 → Flink → 下游"。还能做全量 + 增量无缝衔接（先快照存量、再接 binlog 增量）。

CDC 是实时数仓（D08）的源头活水，也是"把核心业务库的数据安全地拷贝一份去做分析、避免分析查询拖垮线上库"的标准手段。它产出的变更流天然适合 D02 的SCD2 / 拉链表——每条 UPDATE 都是一个新版本。
CDC 落地的坑：① DDL 变更（源表加列/改类型）要能传导到下游，否则解析失败；② 初始全量快照对大表可能很重，要选低峰期或用增量快照；③ 删除事件（tombstone）下游要正确处理，否则数据删不掉。

4.1 DDL 变更与发布顺序（实战版）

变更类型推荐做法风险点
新增可空字段优先新增，保持向后兼容下游默认值逻辑需补齐
字段重命名先双写新旧字段，灰度后再移除旧字段直接改名会导致历史消费者解析失败
类型变更新字段承载新类型，旧字段保留过渡期强改类型最容易引发序列化异常
灰度顺序：先升级下游消费者（兼容新旧 schema）→ 再改上游 DDL → 观察 lag 与 DLQ → 最后清理旧字段。把"向前兼容"放在上线前，而不是事故后补救。
05

工作流编排：Airflow · DolphinScheduler · DataWorks

几百个任务、复杂依赖、定时与回填——靠编排器统一调度

数据平台每天要跑成百上千个任务：抽取、清洗、各层加工、报表生成……它们之间有依赖（B 要等 A 完成）、有时间（每天凌晨跑）、还可能失败重试、补跑历史。工作流编排器就是统一管理这些任务的"总调度室"。

工作流编排（Workflow Orchestration）：用 DAG（有向无环图，又见 D03！）描述任务及其依赖，由调度器按依赖顺序、按时间触发、并处理失败重试/告警/回填。代表：Airflow（Python 定义 DAG，最流行）、DolphinScheduler（国产，可视化拖拽）、Dagster / Prefect（更现代、数据感知）。
编排器像厨房的总厨：他手里有一张"出菜流程图"（DAG）——切配、炖煮、摆盘谁先谁后；该并行的同时开火（洗菜和烧水一起），该等的等着（没炖好不能摆盘）；某道菜搞砸了就重做（重试）；客人要补昨天的菜也能补（回填）。厨师（具体任务）只管做自己那道，总厨负责"什么时候、按什么顺序、谁做"。

5.2 回填、调度与 SLA

回填（Backfill）：重新跑某段历史时间区间的任务。常见于：修了 bug 要重算过去 30 天、新加的指标要补历史。编排器把每个调度周期看作一次独立的 run（按 execution_date 参数化），所以能精确地"重跑 2026-06-01 那天的管道"——这要求任务幂等（重跑结果一致，不会重复累加）。
幂等性（Idempotency）：同一任务重跑多次，结果与跑一次相同。是回填/重试安全的前提。实现常用「先按分区删除再写入（overwrite）」而非「追加（append）」，确保重跑不产生重复数据。

调度触发

Cron 定时（每天 2 点）、或数据/事件驱动（上游数据就绪才触发，data-aware）。

失败处理

自动重试（带退避）、告警、依赖任务阻塞，避免错误数据往下游扩散。

SLA 与监控

设定任务应在何时完成；超时告警。配合血缘定位"谁卡住了整条链"。

编排器是数据平台的"心跳"。它和 D02 的分层是绝配：编排器管何时触发、跨系统依赖、回填与告警，Spark/Hive SQL 管怎么转换。互联网常见 Airflow + dbt；国企/杭州政企项目更常见 DolphinScheduler 或阿里云 DataWorks（见下文 5.3 / 5.4）。

5.3 DolphinScheduler 国企实战（推荐本地练手）

Apache DolphinScheduler 是国产开源分布式调度系统，可视化拖拽 DAG，在江浙沪政企、国企信息化项目中出现频率很高。与 D10 华港集团 Capstone 配套的典型 DAG 如下：

T0→DataX 同步：子公司 ERP/MES → Hive ODS（可并行多个 DataX 任务）

T3→导出报表库：Sqoop/DataX 把 ADS 写入 MySQL/ClickHouse
T4→告警：失败发邮件/钉钉；成功写 SLA 日志

5.4 DataWorks 控制台（杭州上云国企备选）

若项目部署在阿里云（杭州政务云、省属国企上云很常见），编排常用 DataWorks 代替自建 DolphinScheduler。概念与 DS 一一对应，只是界面在云上：

DataWorks 文档入口：阿里云 DataWorks 帮助中心。MaxCompute SQL 与 Hive SQL 语法高度相似，Capstone 里的 SQL 改少量方言即可迁移。

摄取模式选型

批 / 流 / CDC 怎么选 · 一张决策清单

模式怎么做延迟适用
批量全量定时拉整张表T+1小表、维表、对实时无要求
批量增量按 updated_at / 自增 ID 拉新增小时级大表、有可靠增量标记、抓不到删除
流式（Kafka）事件实时进 Kafka秒级埋点、日志、IoT 等天然事件流
CDC读 binlog 捕获变更秒级业务库实时同步（含删除）、实时数仓
Q1→数据源是数据库还是事件？ 数据库→CDC/批增量；事件流→Kafka
Q2→要实时吗？ 要→CDC / 流式；T+1 够用→批量（更简单省钱）
Q3→需要捕获删除、还原完整历史？ 需要→CDC（批增量抓不到删除）
Q4→表很大、全量太重？ → CDC 增量快照 或 增量拉取
实战常混用：埋点/日志走 Kafka 流式，业务库走 CDC 实时同步，少量小维表 T+1 批量。没有银弹，按"延迟要求 + 数据源类型 + 成本"组合。

6.1 容量规划与成本速查

参数调大后的收益副作用/成本
Partition 数并行度上升、单组消费吞吐提升元数据与 rebalance 开销增大，小文件更多
副本因子（RF）容灾能力增强磁盘与网络成本近似按 RF 线性增加
Retention 保留时长回放窗口更长、故障恢复更从容占用磁盘增大，恢复扫描更慢
压缩（lz4/zstd）降低网络与存储成本CPU 消耗上升，需平衡延迟
一个简单估算：日存储量 ≈ 日写入量 × 副本因子 × 保留天数 ÷ 压缩比。先算账再扩容，避免"吞吐够了、磁盘先爆"。

6.2 上线前 10 条检查清单

  1. 是否定义了分区 key，并明确保序边界（按用户、订单或设备）？
  2. 生产端是否启用 acks=all、幂等与合理重试上限？
  3. min.insync.replicas 与副本因子是否匹配（避免单点写入）？
  4. 是否启用 DLQ，并对 DLQ 速率做告警？
  5. 是否监控 lag、端到端延迟、ISR 抖动、rebalance？
  6. 是否做过 broker 故障与消费者重启演练？
  7. Schema Registry 是否开启兼容性校验并通过回归？
  8. CDC 是否验证了 DELETE 与 DDL 变更链路？
  9. 任务是否幂等，支持回填与重跑不重复？
  10. ACL/TLS/凭据轮换是否在预发演练过？

小结 · 面试 · 术语表

收束成判断力 · 并接上 D07

一页纸总结

① Kafka = 分布式 commit log

Topic→Partition→Offset；分区是并行单位、分区内有序、可重放。

② 消费者组

组内每分区只给一个消费者；有效并行 = min(消费者数, 分区数)。

③ 可靠性

副本+ISR+acks=all+min.insync；幂等+事务=EOS。保序靠同 key 同分区。

④ 生态

Connect（配置式集成）+ Schema Registry（schema 演进治理）。

⑤ CDC

读 binlog 实时捕获增删改（含删除），Debezium/Flink CDC，实时数仓源头。

⑥ 编排

DAG 调度依赖+定时+回填+SLA；幂等是回填/重试安全的前提。

面试自测（点击展开）

基础Kafka 为什么快？Topic/Partition/Offset 是什么关系？
快本质是 commit log：生产者顺序追加写（顺序 IO 极快）、零拷贝、批量发送、页缓存。
关系Topic（逻辑主题）→ 切成多个 Partition（并行/扩展单位，分区内有序）→ 每条消息有唯一递增 Offset（消费者记录读到哪、可重放）。
核心消费者数和分区数什么关系？消费者多了会怎样？
规则同一消费者组内，每个分区只分配给一个消费者。有效并行 = min(消费者数, 分区数)。
多了消费者数 > 分区数时，多出来的消费者分不到分区、空闲。要提并行必须先加分区。
高频Kafka 怎么保证不丢数据？怎么保证顺序？
不丢acks=all + min.insync.replicas≥2 + 多副本 + 幂等生产者；ISR 保证选主时数据全。
顺序只保证分区内有序。要保序的消息用同一 key（哈希到同一分区）。全局有序需单分区（牺牲并行）。
实战什么是 CDC？相比定时拉全表/查 updated_at 有什么优势？
CDC读数据库事务日志（binlog/WAL）实时捕获增删改事件。
优势① 实时（秒级）；② 轻量（不查业务表、不加锁）；③ 能抓到删除（查 updated_at 抓不到）；④ 完整变更历史（天然适配 SCD2）。
工程为什么回填要求任务幂等？怎么实现幂等？
原因回填/重试会让同一任务对同一时间分区多次运行，非幂等会重复累加/插入，数据错。
实现「按分区先删后写（overwrite）」而非「追加」；用 execution_date 参数化每次 run；写入带主键去重/MERGE。

术语表速查 · Glossary

本篇核心名词一次收齐：

① Kafka 核心

Topic / Partition / Offset逻辑主题 / 物理切分（并行单位，分区内有序）/ 分区内唯一递增编号（可重放）。
BrokerKafka 服务节点，多个组成集群，分区副本分散其上。
Consumer Group协作消费者组；组内每分区只给一个消费者；有效并行 = min(消费者数, 分区数)。
commit log只追加、不修改的有序日志——Kafka 的本质数据结构，顺序写极快、可重放。
Consumer Lag 消费滞后最新 offset 与消费者已提交 offset 的差，衡量消费跟不跟得上，核心监控指标。

② 可靠性

副本 / Leader / Follower分区多副本；Leader 读写、Follower 同步；Leader 挂从 ISR 选新主。
ISR In-Sync Replicas与 Leader 保持同步的副本集，只有它们有资格被选为新 Leader。
acks生产者确认级别：0（不等）/1（Leader）/all（ISR 全确认，最可靠）。
幂等生产者 / 事务去重重试 + 跨分区原子写，是 Kafka Exactly-once（EOS）的基础。
at-least / at-most / exactly-once消费语义，取决于"处理与提交 offset 的先后"；精确一次需把结果与 offset 放进同一事务。

③ 生态与 CDC

Kafka Connect配置式数据集成框架：Source 导入、Sink 导出，免写搬运代码。
Schema Registry集中管理消息 schema 并强制兼容性校验，治理流数据的 schema 演进。
CDC Change Data Capture读数据库事务日志实时捕获增删改事件，轻量、实时、能抓删除。
binlog / WAL数据库为复制/恢复写的变更日志，CDC 的数据来源。
Debezium / Flink CDC主流 CDC 工具：前者经 Kafka Connect，后者集成进 Flink（全量+增量无缝）。

④ 编排

工作流编排 Orchestration用 DAG 描述任务与依赖，按依赖/时间调度、重试、回填、告警。Airflow/DolphinScheduler。
DAG有向无环图，描述任务依赖；能并行的并行，有依赖的排队（又见 D03）。
回填 Backfill重跑某段历史时间区间的任务（补历史/修 bug）。依赖任务幂等。
幂等性 Idempotency重跑多次结果与一次相同。回填/重试安全的前提，常用先删后写实现。
SLA任务应完成的时限约定，超时告警，保障数据及时性。
