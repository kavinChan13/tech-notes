为什么对象存储上「建表」这么难

廉价灵活的对象存储，缺的恰恰是数据库最基本的几样东西

D01 给出了结论：对象存储（S3 / OSS）+ 列式文件（Parquet）正在成为大数据的统一底座，便宜、近乎无限扩展、存算分离。但如果你直接拿一堆 Parquet 文件当「表」用，很快会撞上一堆数据库里早就解决了、而对象存储上重新冒出来的老问题。

本篇大量复用 D01（对象存储 / Parquet）、D02（数仓分层）、D08（实时数仓 / Paimon）的概念。建议先有这些底子；遇到 ACID、谓词下推、CDC 等词时，本篇只在用到处简述，不重复展开。

把「一堆文件」当表用，会遇到什么

① 没有原子性

一次写入要落 100 个 Parquet 文件，写到第 60 个时作业挂了——读的人会读到半张表。对象存储没有「多文件一起成功或一起失败」的事务。

② 读写互相打架

一边在追加 / 覆盖文件，另一边在读——读到的可能是写了一半的中间状态（脏读）。没有快照隔离。

③ list 又慢又不准

「这张表当前由哪些文件组成？」只能去 LIST 目录。对象存储的 list 既慢（百万文件要翻很久）、历史上还最终一致（刚写的文件可能 list 不到）。

④ 改一行 = 重写一堆

对象不可原地修改。要更新 / 删除一行（GDPR 删除、CDC 同步），只能重写整个文件甚至整个分区，代价巨大且无事务保证。

⑤ Schema 一改就乱

加一列、改类型、改列序——老文件和新文件 schema 对不上，读出来错位或报错。没有受管的 schema 演进。

⑥ 看不到历史

覆盖即丢失。想「查昨天那一版的数据」「误删后回滚」？裸文件做不到。没有版本 / 时间旅行。

裸 Parquet 文件就像一个没有账本、没有目录的杂货仓库：货（数据）确实廉价地堆在那儿，但「现在仓库里到底有哪些货、谁正在搬、上周三的库存是多少、搬一半断电了怎么办」全都没人记。表格式就是给这个仓库配上一本「事务账本 + 目录」：每次进出货都原子地记一笔，读的人永远看到某个一致的版本。仓库（对象存储）没变，变的是上面那层元数据管理。
一句话定义：表格式 = 在对象存储的不可变文件之上，加一层「元数据账本」，用「原子地切换一个指针」来模拟数据库的事务、快照与版本管理。数据文件还是那些 Parquet/ORC，魔法全在元数据层。

01

表格式的核心思想：元数据指针 + 原子换指针

所有表格式都在做同一件事，只是账本长得不一样

三家实现细节不同，但核心机制惊人地一致：数据文件不可变（只增不改），用一层元数据记录「当前这张表由哪些文件组成」；每次写入产生新数据文件 + 新元数据，最后原子地把「当前版本指针」从旧元数据切到新元数据。读的人读到哪个指针，就看到哪个一致的版本。我们以生态最广的 Iceberg 为主线讲透，再点出 Delta / Hudi 的差异。

1.1 Iceberg 的三层元数据树

Iceberg 把元数据组织成一棵三层树，根是一个可原子替换的指针。理解这棵树，就理解了 90% 的表格式原理：

从根指针一路下钻：Catalog 指针 → metadata.json（含快照列表）→ 当前 Snapshot → Manifest List → 若干 Manifest → 真正的 data files。每一层都带统计，供逐层裁剪。
四层文件各管什么：
  • metadata.json：表的「总账」——schema、分区规格（partition spec）、所有历史快照的列表、当前快照 id。每次提交生成一个新版本（v1→v2→v3…）。
  • Snapshot（快照）：某一时刻表的完整视图，指向一个 Manifest List。「时间旅行」就是选不同的 snapshot。
  • Manifest List（清单列表）：列出本快照包含哪些 Manifest，并记录每个 Manifest 覆盖的分区范围 → 可在这层就跳过整批 Manifest。
  • Manifest（清单）：列出一批 data file 的路径，以及每个文件的列级 min/max、行数、null 数 → 谓词下推在这层就能跳过整个文件，根本不用 list 目录。
Delta / Hudi 的「账本」长什么样？ Delta Lake 用一个 _delta_log/ 目录，里面是一串有序的 JSON 事务日志（00000.json, 00001.json…，每条记录 add/remove 了哪些文件），定期 checkpoint 成 Parquet 加速回放——本质是「日志即真相」。Hudi 用 Timeline（一串带时间戳的 action：commit / deltacommit / compaction…）+ 每条记录的主键索引。三者数据结构不同，但「不可变数据 + 元数据记录组成 + 原子提交」的内核完全一致。

1.2 交互演示：一次写入如何「原子地」生效

表格式怎么做到「要么全可见、要么全不可见」？关键在最后一步——只切换一个指针，而这一步是原子的（catalog 上的一次 CAS / 重命名）。点下面的按钮，模拟一次写入提交：

交互演示 · 原子提交（commit）
当前指针：current → S0
点「提交」看新快照如何生成，以及指针如何瞬间切换——读者要么看到旧版本、要么看到新版本，绝不会看到中间态。
关键洞察：写入过程中产生的所有新 data file 和 manifest，对读者完全不可见，因为指针还指着旧快照。直到最后那一次原子指针切换成功，新版本才「一瞬间」整体生效。失败了？指针没动，新写的文件变成孤儿文件（后续 GC 清理），表毫发无损。这就是对象存储上 ACID 的「A（原子性）」。

1.3 读流程：为什么比裸文件快得多

读一张 Iceberg 表（带 WHERE）时，引擎自上而下逐层裁剪，绝大多数文件根本不会被打开：

1→从 catalog 拿到当前 metadata.json → 当前 snapshot（一次定位，不 list 目录）
2→读 Manifest List：按分区范围统计，跳过整批不相关的 manifest
3→读剩下的 Manifest：按每个 data file 的 min/max 统计，跳过整个不可能命中的文件（谓词下推）
4→只对存活下来的少数 Parquet，再做文件内的 Row Group 级裁剪（D01 的下推）
对比裸文件：裸文件必须先 LIST 整个目录（慢、最终一致），再逐个打开 Parquet 读 footer 才知道能不能跳。Iceberg 把「有哪些文件 + 每个文件的统计」预先汇总进 manifest，于是规划查询时连对象存储都很少碰——这就是它在百万文件大表上仍然规划飞快的原因（呼应 D01 的「元数据即性能」）。
02

三大能力：时间旅行 / Schema 演进 / 分区演进

「不可变文件 + 版本化元数据」一旦成立，这些能力几乎是免费的副产品

2.1 时间旅行与快照隔离

既然每次提交都生成新快照、且旧数据文件不被删除，那么「查过去某个版本」就只是选一个旧 snapshot id 来读而已。

-- Iceberg：按快照 ID 或时间戳查历史版本
SELECT * FROM orders VERSION AS OF 3821550127947089987;
SELECT * FROM orders TIMESTAMP AS OF '2026-06-24 10:00:00';

-- 误操作回滚：把当前指针挪回旧快照（秒级、无需重算）
CALL catalog.system.rollback_to_snapshot('db.orders', 3821550127947089987);

审计 / 复现

「报表为什么变了」→ 对比两个快照即可。ML 训练也能锁定某个数据版本保证可复现。

误删救命

删错 / 写错？回滚到上一快照，秒级恢复，不用从备份重灌。

快照隔离

长查询读的是开始时那个快照，期间别人怎么写都不影响它（不会读到半张表）。

代价：旧快照和旧数据文件会持续占空间。必须配套过期清理（expire_snapshots 删掉 N 天前的快照 + GC 孤儿文件），否则存储无限膨胀。保留窗口是「能回溯多久」与「存储成本」的权衡。

2.2 Schema 演进：靠 field-id，而不是列的位置

裸 Parquet 改 schema 容易错位，根因是「按列的位置/名字对齐」。Iceberg 给每一列分配一个永不复用的 field id，读写都按 id 匹配——于是加列、删列、改名、调整列序都安全且无需重写数据。

这就像给每个员工发一个永久工号，而不是靠「坐在第几排第几座」认人。即使重新排座位（调列序）、有人离职（删列）、改了名字（rename），工资照样发对人——因为认的是工号（field id），不是座位。
演进操作安全吗要重写数据吗
加列（add column）安全否（老文件读出为 null）
删列 / 改名 / 调列序安全（按 field-id）否
类型提升（int→long 等）安全（向上兼容）否
类型收窄 / 不兼容改不允许—（会被拒绝，保护数据）
这与 D06 的 Schema Registry / 数据契约 思想一脉相承：用受控的、向后兼容的演进规则，让「表结构变化」不再是一场灾难。三家都支持 schema 演进，Iceberg 的 field-id 机制被认为最干净。

2.3 隐藏分区与分区演进（Iceberg 杀手锏）

传统 Hive 分区有两个老大难：① 用户必须知道分区列并手动写对（忘了带 WHERE dt=... 就全表扫）；② 分区方案一旦定死就改不了（从按天改按小时，得重建整张表）。Iceberg 用隐藏分区和分区演进把这俩都解决了。

隐藏分区（Hidden Partitioning）：你定义分区时写的是「对某列做一个转换」，比如 days(event_time)、bucket(16, user_id)。Iceberg 自己维护「列值 → 分区」的映射。查询时你照常写 WHERE event_time > '2026-06-01'，引擎自动推导出该裁哪些分区——用户完全不需要知道分区是怎么分的，也不可能写错分区谓词。
隐藏分区把「分区怎么分」从用户脑子里搬进引擎，既防写错、又免去冗余分区列，还让分区方案可以后续演进。
分区演进（Partition Evolution）：业务变大了，想从「按天分区」改成「按小时分区」？Iceberg 允许只改未来写入的分区规格，老数据原样不动。一张表里可以新旧分区方案并存，查询时引擎对不同分区用对应的规格裁剪——再也不用为改分区而重建整张表。这是大表运维的巨大解放。
这三大能力本质都来自同一个设计：数据文件不可变 + 元数据版本化、且元数据按 field-id / 转换函数描述。一旦底层是这套，时间旅行、安全演进、隐藏/演进分区就几乎是「白送」的。Delta 也有 schema 演进与（较新的）Liquid Clustering，但隐藏分区 + 分区演进是 Iceberg 最被称道的差异点。
03

写入语义：Copy-on-Write vs Merge-on-Read

「改一行」到底怎么落地 · 写放大与读放大的经典权衡

对象存储不可原地改，那 UPDATE / DELETE 一行到底怎么实现？这里有两种根本不同的策略——COW（写时复制）和 MOR（读时合并），它们是「写快还是读快」的经典取舍，也是 Hudi/Iceberg 表选型时必答的题。这正是老兵在 LSM-Tree、B+Tree 里见过的同一种权衡。

COW：写时把整文件重写干净，读不用合并；MOR：写时只记增量，读时再合并。和 LSM-Tree「写快、靠 compaction 维持读」的取舍同源。

先选策略，再点「更新 3 行」看写代价，再点「查询读」看读代价。每个大文件含 100 万行。

3.2 MERGE INTO / Upsert：CDC 入湖的主力句式

有了 COW/MOR，表格式就能支持数据库式的 UPDATE / DELETE / MERGE。其中 MERGE INTO（存在则更新、不存在则插入 = Upsert）是把 CDC 变更（D06）合并进湖仓表的标准句式：

-- 把 CDC 捕获的订单变更（changes）合并进湖仓事实表
MERGE INTO dwd.orders t
USING cdc_changes s
ON t.order_id = s.order_id
WHEN MATCHED AND s.op = 'DELETE' THEN DELETE
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;
这正是 D08 实时数仓「CDC → 入湖」链路落地的关键能力。Hudi 天生为 Upsert 而设计（自带主键索引，定位「这条 key 在哪个文件」很快）；Iceberg / Delta 也都支持 MERGE，但大规模高频 Upsert 时 Hudi 的索引优势更明显。

3.3 小文件问题与 Compaction

流式 / 高频写入的死敌是小文件：每次提交都产生一批小 Parquet，越攒越多，导致元数据膨胀、读时要打开海量小文件（呼应 D01 的「HDFS 小文件问题」，在湖仓里换了个形态又回来了）。

Compaction（合并 / 压实）：后台定期把大量小文件重写成少量大文件（也叫 file compaction / OPTIMIZE）；对 MOR 表，还要把基线 + 增量合并成新基线（也叫 minor/major compaction）。这是湖仓表的必备运维动作，类似 LSM-Tree 的 compaction、ClickHouse 的 MergeTree merge——都是「为了写得快而欠下的债，靠后台慢慢还」。
-- Iceberg：重写小文件 + 按列排序聚簇
CALL catalog.system.rewrite_data_files(
  table => 'db.orders',
  strategy => 'sort', sort_order => 'event_time');

-- Delta 等价
OPTIMIZE db.orders ZORDER BY (event_time, city);
VACUUM db.orders RETAIN 168 HOURS;   -- 清理过期文件
运维三件套缺一不可：① Compaction（合并小文件）② Expire/Vacuum（清过期快照与孤儿文件）③ 元数据维护（清理过多 manifest）。很多「湖仓上线后越来越慢」的事故，根因就是只管写、不做这三件事。托管平台（Databricks 的 auto-optimize、各云的表服务）会自动做，自建则要排定时任务。
04

数据布局优化：让谓词下推「跳得更狠」

表格式给了统计，但跳得多不多，取决于数据怎么摆

D01 讲过：Parquet 的谓词下推靠 min/max 跳过整段数据。但能不能跳得动，取决于数据的物理排布——如果目标值散落在每个文件里，每个文件的 min/max 都覆盖全域，那一个都跳不掉。布局优化就是把「相关的数据物理聚到一起」，让统计信息真正起作用。

分区（Partitioning）

按列把数据切进不同目录/分区（按天、按地区）。粒度大、最粗的裁剪。配合 Iceberg 隐藏分区免去写错风险。

排序 / 聚簇（Sorting / Clustering）

分区内按某列排序写入，让每个文件的 min/max 范围窄而不重叠 → 谓词下推能精准跳文件。Compaction 时顺带排序。

Z-order / 多维聚簇

单列排序只对一列友好。Z-order（空间填充曲线）把多列的相近值映射到相邻位置，让多个过滤列同时受益。Delta ZORDER BY / Iceberg sort。

Liquid Clustering（新）

Delta 较新的方案：不固定分区列，自适应、可演进的聚簇，避免分区选错和数据倾斜，运维更省心。

Z-order（Z 序曲线）：一种把多维坐标交错编码成一维的「空间填充曲线」。直觉上，它让「(city, event_time) 都接近」的行在物理上也排得近。于是无论按 city 还是按 event_time 过滤，都能跳过大量文件——解决了「单列排序只能优化一列」的痛点。代价是写入/compaction 时要算 Z 值、开销略高。
配合手段还有 Bloom Filter / 字典 / min-max 索引（D01 已讲）用于高基数列的等值过滤。一句话：表格式给你「记账+统计」的能力，布局优化决定这些统计能帮你跳过多少数据。两者叠加，才是湖仓查询快的完整答案。
05

Catalog 之争：谁来保管那个「根指针」

原子提交的前提，是有一个可靠的地方存放「current 指向哪」

回到第 1 章的核心：原子提交 = 原子地切换「current → 哪个 metadata」这个根指针。这个指针存在哪、由谁保证原子切换，就是 Catalog（目录服务）的职责。Catalog 还负责「库.表 → 表的元数据位置」的命名映射、权限、审计。这一层选型，直接决定多引擎能否共享同一张表。

Catalog（目录 / 元存储）：管理「有哪些库、哪些表，每张表当前的元数据指针在哪」，并提供原子的指针更新（commit）。它是湖仓的「户口本 + 公证处」。没有它，多个引擎/作业并发写同一张表就会冲突错乱。
Catalog定位特点 / 适用
Hive Metastore (HMS)老牌事实标准生态最广、几乎所有引擎都认；但是个要单独运维的有状态服务、偏重
AWS Glue CatalogAWS 托管免运维、与 Athena/EMR/Redshift 打通；锁定 AWS
REST CatalogIceberg 开放标准（趋势）标准化 HTTP 接口，引擎/实现解耦，可对接任意后端。云中立的未来方向
Unity CatalogDatabricks（已开源）统一数据 + AI 资产治理、行列级权限、血缘；Databricks 生态核心
PolarisSnowflake 开源Iceberg REST Catalog 的开源实现，主打开放、多引擎共享
Nessie开源给湖仓加「Git 式分支/提交」语义，多分支实验友好
趋势很清晰：从「各家私有元存储」走向 Iceberg REST Catalog 这个开放标准。一旦 catalog 标准化，同一张 Iceberg 表就能被 Spark、Flink、Trino、StarRocks、各家云数仓共享读写——这正是「一份数据、多引擎共用、不锁定厂商」愿景的关键拼图，也是 Snowflake（Polaris）、Databricks（Unity）、各云厂商集体押注 Iceberg 的根本原因。
面试常问：「为什么对象存储自己不能保证原子提交？」因为 S3 的 PUT 是单对象原子，但「把表指针从 v2 换到 v3」需要一个带条件的原子比较交换（CAS）——这正是 catalog 提供的（HMS 用数据库行锁、REST Catalog 用服务端 CAS、早期 Iceberg 在纯文件系统上靠原子 rename）。原子提交能力在 catalog，不在存储。

6.1 Apache XTable：一份数据，三种「读法」

选型最大的顾虑是「选错了要不要重灌全部数据」。XTable 给出的答案是：不用——因为三家的数据文件都是标准 Parquet，只有元数据账本不同。XTable 做的就是「读一种格式的元数据，翻译生成另一种格式的元数据」，数据文件原地不动、零拷贝。

同一批货（Parquet 数据文件）放在仓库里没动，XTable 只是按不同公司的格式各开一份「库存清单」：给 Iceberg 引擎一份 Iceberg 格式的清单、给 Delta 引擎一份 Delta 清单。货是同一批货，谁拿对应的清单都能盘点——不用把货搬来搬去。这就是「元数据互转、数据零拷贝」。
交互演示 · XTable 元数据互转
源格式：
选一个源格式，点「同步元数据」，看 XTable 如何让同一批 Parquet被三种引擎都读到。
XTable 能做什么、不能做什么：它同步的是当前快照的元数据（schema、文件清单、分区、统计），让另一种格式的引擎能读到同一份数据；它不复制数据文件。局限：不是所有特性都能完美互转（如各家独有的高级布局/删除向量），且需要在写入后触发同步（有延迟）。它适合「一份数据要被多个固定用 Iceberg/Delta 的下游消费」的互操作场景，而非替代「统一选一种格式」的治理建议。
更底层的趋势是 统一 Catalog（REST Catalog / Unity / Polaris，见第 5 章）：与其在格式间互转，不如让大家共享同一种开放格式 + 开放 catalog。XTable 是「过渡期的互操作粘合剂」，开放 catalog 是「终局的统一」——两条路都指向同一个目标：一份数据、多引擎共用、不锁厂商。

小结 · 面试 · 术语表

把这一篇收成可带走的判断力

一页纸总结

① 解决什么

给「对象存储 + Parquet」补上 ACID、快照、版本、Upsert、schema/分区演进——湖长出仓的可靠性。

② 核心机制

不可变数据文件 + 版本化元数据 + 原子切换根指针。读到哪个指针就看到哪个一致版本。

③ 三层树

metadata → snapshot → manifest list → manifest → data file，逐层带统计，逐层裁剪，不 list 目录。

④ COW vs MOR

写时合并（读快）vs 读时合并（写快），靠 compaction 还债。CDC/高频写偏 MOR。

⑤ 布局 + Catalog

排序/Z-order 决定能跳多少；Catalog 保管根指针、提供原子提交，REST Catalog 是开放趋势。

⑥ 选型

中立/多引擎 → Iceberg；Databricks 栈 → Delta；高频 Upsert/CDC → Hudi；流式 → Paimon。

面试自测（点击展开）

基础为什么不能直接拿一堆 Parquet 当表用，非要表格式？
无事务多文件写入没有原子性，会读到半张表；读写并发会脏读。
list 慢「当前有哪些文件」只能 list 目录，慢且历史上最终一致。
不能改更新/删除一行要重写整文件、无事务；schema 改易错位；无版本/回滚。
表格式加一层版本化元数据账本 + 原子切指针，把这些一次性补齐。
核心Iceberg 一次写入是怎么做到原子的？
先写后切先把新 data file、manifest、新 metadata.json 都写好（此时对读者不可见）。
原子切指针最后在 catalog 上把 current 指针从旧 metadata 原子地 CAS 到新 metadata——这一步成功，整版本才生效。
失败安全切指针失败 → 表不变，新写的文件成孤儿待 GC。所以要么全可见、要么全不可见。
高频COW 和 MOR 的区别？CDC 实时入湖该用哪个？
COW更新时重写整个文件，写放大大、读最快。适合读多写少的分析表。
MOR更新时只追加增量/删除文件，写最快、读时要合并基线+增量。靠 compaction 还读性能。
CDC 入湖高频小批更新 → 选 MOR（写延迟低），再配后台 compaction 维持读性能。这也是 Hudi 的主场。
进阶Iceberg 的隐藏分区解决了 Hive 分区的什么痛点？
不写错Hive 要用户手动写对分区列谓词，忘了就全表扫；隐藏分区让用户写自然谓词，引擎自动裁。
可演进分区方案（按天→按小时）可只对新数据生效、老数据不动，无需重建整表。
机制分区定义为「列的转换」（days/bucket），引擎维护映射，按 field-id 而非位置。
架构为什么各大云厂商（Snowflake/Databricks）都在拥抱 Iceberg + 开放 Catalog？
客户诉求不想被锁定：一份数据想被多引擎共享读写、想换引擎不搬数据。
REST Catalog标准化的 catalog 接口让「库表指针 + 原子提交」跨厂商通用，是多引擎共享的关键。
本质竞争从「锁住存储格式」转向「在开放数据之上拼计算与体验」。Polaris/Unity 都是这盘棋的落子。

术语表速查 · Glossary

① 核心概念

表格式 Table Format在对象存储的不可变文件之上加一层版本化元数据，提供 ACID、快照、版本、Upsert、schema 演进。Iceberg/Delta/Hudi。
湖仓一体 Lakehouse在数据湖的廉价灵活之上叠加数仓的事务与治理，兼得两者。表格式是其实现基石。
快照 Snapshot某一时刻表的完整一致视图。每次提交生成新快照，是时间旅行与隔离的基础。
原子提交 Atomic Commit先写好新数据与元数据，最后原子切换根指针使整版本一次性生效，失败则表不变。
时间旅行 Time Travel按快照 id / 时间戳查询历史版本，或回滚到旧快照。用于审计、复现、误删恢复。

② Iceberg 元数据

metadata.json表的总账：schema、分区规格、所有快照列表、当前快照。每次提交生成新版本。
Manifest List一个快照包含哪些 manifest，并记录各 manifest 的分区范围，供整批跳过。
Manifest一批 data file 的清单 + 每文件的列级 min/max/行数/null，供谓词下推跳过整个文件、免 list。
field id每列的永久唯一编号。读写按 id 匹配而非位置/名字，使 schema 演进安全。
孤儿文件 Orphan File提交失败或被回收后不再被任何快照引用的文件，需 GC 清理。

③ 写入与维护

COW Copy-on-Write更新时重写整个数据文件。写放大大、读最快，适合读多写少。
MOR Merge-on-Read更新时只追加增量/删除文件，读时合并。写最快、读较慢，适合高频写/CDC。
MERGE INTO / Upsert存在则更新、不存在则插入。CDC 变更合并进湖仓表的标准句式。
Compaction 合并/压实后台把小文件重写成大文件、MOR 基线+增量合并。湖仓必备运维，类 LSM compaction。
小文件问题高频写产生大量小 Parquet，撑大元数据、拖慢读。靠 compaction 治理。
expire / vacuum清理过期快照与不再引用的旧文件，控制存储膨胀。

④ 分区 / 布局 / Catalog

隐藏分区 Hidden Partitioning分区定义为列的转换（days/bucket），引擎自动裁剪，用户写自然谓词、不会写错。Iceberg 招牌。
分区演进 Partition Evolution改分区方案只对未来数据生效、老数据不动，无需重建整表。
Z-order / 聚簇 Clustering用空间填充曲线让多列相近值物理相邻，多个过滤列同时受益于谓词下推。
Liquid ClusteringDelta 较新方案：自适应、可演进的聚簇，免去选错分区与倾斜。
Catalog 目录/元存储保管「库表→元数据指针」并提供原子提交。HMS/Glue/REST/Unity/Polaris/Nessie。
REST CatalogIceberg 的开放 HTTP catalog 标准，使表可被多引擎/多厂商共享，是去厂商锁定的趋势。
Apache XTable 原 OneTable在 Iceberg/Delta/Hudi 之间互转元数据（数据文件零拷贝），让一份 Parquet 被多种格式读到。过渡期互操作粘合剂。
