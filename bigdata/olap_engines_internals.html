OLAP 引擎是什么 · 为什么这么快

不是 Spark 那种批作业 · 是为"即席交互式分析"而生的查询引擎

D03 的 Spark 擅长跑离线批作业（分钟级、吞吐导向）。但 BI 看板、即席查询需要的是「点一下，1 秒出结果」的交互体验。这就是 OLAP 引擎（也叫 MPP 数据库 / 分析型数据库）的主场——ClickHouse、StarRocks、Doris、Druid、Trino 都属此类。

Spark 批作业像叫一辆货车搬一整仓库的货——量大、可靠，但你得等它装车上路卸货，分钟起步。OLAP 引擎像便利店收银——你拿几样东西，扫码、秒结。两者都"处理数据"，但一个为吞吐优化、一个为低延迟交互优化。OLAP 引擎把"扫十亿行做聚合"压到亚秒，靠的全是榨干硬件的极致工程。
OLAP 引擎为什么快？五个支柱（本篇逐一展开）：
  • ① 列存 + 数据跳过：只读相关列、用索引跳过无关数据块（D01 列存的极致应用）。
  • ② 向量化执行：一次处理一批数据而非一行，消除逐行虚函数调用开销。
  • ③ SIMD + cache 友好：用 CPU 向量指令并行算一批，数据布局贴合 cache。
  • ④ MPP 大规模并行：把查询拆到多节点多核并行执行，再汇总。
  • ⑤ CBO + 预计算：成本优化器选最优计划，物化视图/预聚合把重活提前算好。
OLAP 引擎专为「交互式分析」而生：扫海量行、低延迟出聚合结果，介于"离线批"与"在线事务"之间的独特生态位。
01

列存引擎内部：以 ClickHouse MergeTree 为例

D01 讲了"为什么列存"，这里讲"列存引擎具体怎么组织数据、怎么跳过"

D01 我们建立了"列存 + Parquet"的直觉。OLAP 引擎把它做到极致。以业界广泛使用、且在许多单机扫描场景表现突出的 ClickHouse 的核心存储引擎 MergeTree 为例，看数据在引擎内部怎么组织。

1.1 MergeTree：排序 + 分块 + 稀疏索引

MergeTree（ClickHouse 核心引擎）三个关键设计：① 数据按排序键（ORDER BY）物理排序存储；② 每列数据分成固定行数的颗粒（granule，默认 8192 行）压缩存放；③ 维护一个稀疏索引（sparse index）——只为每个 granule 的第一行记一条索引项（而非每行），所以索引极小、能常驻内存。它和 D01 的 LSM-Tree 同源（后台不断 merge 小 part 成大 part，名字由此而来）。
索引"稀疏"在于每 8192 行才记一条——索引小到能放内存。查询先用它定位 granule，再只读那几块的列数据。这是"少读"的极致。

02

向量化执行 vs 火山模型

OLAP 引擎快的"第一性原理" · 也是 C++ 老兵最有共鸣的一章

就算只读了该读的数据，怎么算也决定性能。传统数据库用火山模型（逐行处理），现代 OLAP 引擎几乎都改用向量化执行（按批处理）——这一个改变，常带来数量级的提速。

2.1 火山模型为什么慢

火山模型（Volcano / Iterator Model）：经典的查询执行模型。每个算子实现一个 next() 接口，一次返回一行；上层算子反复调用下层的 next() 把数据一行行"拉"上来。优雅、好组合，但每行每算子都是一次虚函数调用——海量行时，调用开销、分支预测失败、cache miss 把 CPU 时间全吃掉了。
火山模型像流水线上工人每次只递一个零件给下一站，每递一次都要喊一声"接好了"（虚函数调用）、对方回应一次。零件上亿时，光"喊话"的时间就压垮了实际加工。向量化像每次递一整箱（1024 个）零件——喊一次话搬一箱，"沟通成本"摊薄到几乎为零，而且一箱同类零件能用机械臂（SIMD）一起加工。
向量化执行（Vectorized Execution）：算子一次处理一批（batch / vector，通常 1024~8192 行）的一列数据，而非一行。好处：① 函数调用开销摊薄（一批一次调用，不是一行一次）；② 内层是对一段连续同类型数组的紧凑循环，编译器易优化、cache 命中高；③ 这种循环天然能上 SIMD（下一章）。列存 + 向量化是绝配——列存本就把同列数据连续摆好了。

这就是为什么 ClickHouse/StarRocks 比"列存但逐行执行"的老式引擎还要快几倍：列存解决"少读"，向量化解决"快算"，两者缺一不可。把"一行一次虚调用"变成"一批一次紧凑循环"，正是从 OLTP 思维转向 OLAP 思维的关键一跃。
03

SIMD 与 CPU 友好

老兵主场 · 把单核的并行潜力榨到底

SIMD（Single Instruction Multiple Data，单指令多数据）：现代 CPU 的向量指令（SSE/AVX2/AVX-512、ARM NEON），一条指令同时对多个数据做同样运算——比如 AVX-512 一次加 16 个 int32。向量化执行的"一批连续同类型数组"正好喂给 SIMD：求和、过滤比较、哈希都能一次算一打。这是单核内的数据级并行，和多核/多机的并行叠加。
标量计算像用一根筷子一颗一颗夹豆子；SIMD 像用一把宽勺一次铲 16 颗。同样的手速（时钟频率），吞吐却高了一个数量级——前提是豆子得整齐排好（数据连续、对齐），这正是列存 + 向量化创造的条件。

数据连续

列存让同列数据在内存里连成一片数组，SIMD 才能批量加载。这是 array-of-structs vs struct-of-arrays 之争（呼应 cpp/perf）。

cache 友好

紧凑顺序访问 → cache 命中高、预取有效。逐行跳着访问则频繁 cache miss。

无分支 / 可预测

向量化常把过滤写成"算出 bitmap 掩码"而非逐行 if，减少分支预测失败。

老兵的会心一刻：到这里你应该发现——OLAP 引擎的单节点性能优化，和你在 C++ 里做的完全是同一套：数据导向设计（SoA）、cache 局部性、SIMD、避免虚调用与分支。Meta 开源的 Velox、ClickHouse 的执行层，本质都是"把这套 C++ 性能工程做到极致的列式向量化引擎"。这也是C++ 老兵转 OLAP 内核/AI Infra 的最大杠杆点（呼应 cpp/transition_to_ai_infra）。
04

MPP：把单机性能横向放大

单节点已榨干 · 再用大规模并行把吞吐乘上节点数

MPP（Massively Parallel Processing，大规模并行处理）：把一个查询拆分到多个节点、每节点多核同时执行，再把局部结果汇总。每个节点处理自己那份数据分片（shared-nothing 架构，呼应 D01 分片），所以在数据分布均衡且网络/协调不成瓶颈时，吞吐可随节点数接近线性扩展。ClickHouse/StarRocks/Doris/Trino 都是 MPP 引擎。
关键优化：本地预聚合（partial aggregation）——每个节点先把自己的数据聚合成小结果，只把小结果传给协调节点汇总，极大减少网络传输（呼应 D03 reduceByKey 的 map 端预聚合）。
分布式 Join 策略（和 D03 同源）：小表 join 大表用 Broadcast Join（小表广播到各节点本地 join）；大表 join 大表用 Shuffle Join（两表按 join key 重分布到同节点）。Colocate Join（共置 join）是 OLAP 特色——若两表按相同 key 分桶存储，join 时数据天然在同节点，免 shuffle，StarRocks/Doris 重度依赖它做高性能 join。
MPP 引擎通常是常驻进程（不像 Spark 每个作业重新拉起 Executor），省去启动开销，这也是它能做到亚秒延迟的原因之一。代价是资源常占用、并发高时可能互相挤占。

4.1 扩展边界与抖动来源

现象常见原因处置思路
节点越多，提速不明显网络汇聚、协调节点成为瓶颈优先做本地预聚合，减少跨节点传输
尾部任务明显拖慢数据倾斜（热点 key）改分桶键、热点拆分、重写聚合路径
查询延迟抖动内存不足导致 spill 到磁盘控制并发、调大内存、优化中间结果规模
MPP 调优永远是先少搬数据，再谈加节点。跨节点网络流量通常比 CPU 更先成为上限。
05

CBO · 物化视图 · 预聚合

选对计划 + 把重活提前算好 · 查询时少干活

CBO（Cost-Based Optimizer，基于成本的优化器）：和 D03 Catalyst 同源。利用表的统计信息（行数、列基数、数据分布）估算不同执行计划的"成本"，选最优的——尤其是多表 join 的顺序与算法选择（先 join 小的中间结果、选 broadcast 还是 shuffle）。统计信息不准是 CBO 选错计划的常见原因，所以要定期 ANALYZE 收集统计。
物化视图（Materialized View）：把一个查询的结果预先算好并存下来，查询时直接读结果，不再现算。OLAP 引擎的物化视图常能自动改写——你查明细表，引擎发现有匹配的物化视图就自动改用它（你无感、却快了 N 倍）。代价：数据更新时物化视图要同步刷新。
预聚合（Pre-aggregation）：和 D02 的 DWS 层思想一致——把高频的聚合（按天/按品类的销量）提前算好。Druid/Doris 的聚合模型在写入时就边写边聚合；ClickHouse 用 AggregatingMergeTree 在后台 merge 时聚合。查询十亿明细，实际可能只读了几万行预聚合结果。
物化视图/预聚合像餐厅的"预制菜"：把耗时的炖煮提前做好，客人点单时只需加热摆盘，出餐飞快。代价是要预判哪些菜常点（选对要预聚合的维度组合）、且食材更新时预制菜也得重做（刷新）。
OLAP 调优三板斧：① 选对排序键/分桶键（让数据跳过 + colocate join 生效）；② 建物化视图/预聚合（把高频重查提前算）；③ 收集统计信息（让 CBO 选对计划）。和 D03 一样——先想办法少干活，再谈并行与资源。
06

引擎横评与选型

ClickHouse / StarRocks / Doris / Druid / Pinot / Trino / Kylin 各擅胜场

引擎定位 / 强项弱项典型场景
ClickHouse单表扫描/聚合极快，写入吞吐高复杂多表关联需更谨慎建模与索引设计日志/埋点分析、大宽表、监控
StarRocks / Dorisjoin 强、有 CBO、兼容 MySQL 协议版本演进快，需关注升级与兼容策略实时数仓、多表关联报表、统一 OLAP
Druid实时摄取 + 预聚合，时序快不擅长明细/join实时大屏、时序监控指标
Apache Pinot超低延迟、高并发点查，丰富索引（倒排/星树/Bloom）运维组件多、不擅复杂 join面向用户的实时分析（如领英「谁看了我」）、高 QPS 看板
Trino / Presto联邦查询（跨数据源），存算分离自身不存数据、靠下游数据湖即席查询、跨源 join
Apache Kylin预计算 Cube，固定报表亚秒响应维度组合爆炸、灵活性差、Cube 构建重固定维度的 BI 报表（理念偏传统，新项目少选）
三条路线：① 「自带存储」的 MPP 数据库（ClickHouse/StarRocks/Doris）——数据导入它自己的列存、性能极致；② 「实时摄取型 OLAP」（Druid/Pinot）——边摄取边可查、为高并发、低延迟、面向用户的实时分析而生，索引和预聚合做得极致，但不擅长复杂明细 join；③ 「查询引擎」（Trino）——自己不存数据，直接查数据湖（Parquet/Iceberg）、跨源联邦。前两者快但要导数据，后者灵活但受限于底层存储。
实时摄取型 OLAP（Druid / Pinot）凭什么「边写边查」？ 它们把数据按时间切成 Segment，新数据先进可查的内存实时节点（near-real-time），后台再固化成不可变 Segment 落盘并建索引——这与列存 LSM 的「热数据在内存、冷数据在盘」思想一脉相承，只是为「亚秒查询 + 秒级可见」做了极致裁剪。Pinot 的星树索引（Star-Tree）是「按需预聚合」的精巧折中：在空间与查询速度间可调，避免了 Kylin 那种全量 Cube 的维度爆炸。
选型直觉：单大宽表极致扫描 → ClickHouse；复杂多表 join 的实时数仓 → StarRocks/Doris；实时时序大屏/监控 → Druid；面向 C 端用户、高 QPS + 极低延迟的实时分析 → Pinot；直接查数据湖、跨源 → Trino；固定维度老 BI 报表 → Kylin（但新项目多被 StarRocks/Doris 取代）。很多公司多引擎并存，按场景分流。湖仓（见湖仓表格式专篇 / Iceberg）+ Trino/StarRocks 直查是当下热门组合。

6.1 压测口径与稳定性检查

同一套 SQL，压测口径不同，结论可能相反。对比引擎时至少要固定：数据规模、冷热缓存状态、并发度、是否命中物化视图、数据分布（是否倾斜）、压缩率。
检查项为什么重要
冷缓存 vs 热缓存热缓存结果常优于冷启动，混测会误判引擎能力
单并发 vs 多并发单查询快不代表高并发稳定，需看尾延迟 P95/P99
是否命中 MV/预聚合命中与否可带来数量级差异，应单独标注
失败重试与资源隔离生产稳定性不只看平均延迟，还看抖动与恢复能力

小结 · 面试 · 术语表

收束成判断力 · 并接上 D08

一页纸总结

① 定位

OLAP 引擎为"亚秒级交互式聚合"而生，介于 OLTP 与 Spark 批之间。

② 列存 + 数据跳过

MergeTree 排序+颗粒+稀疏索引；选对排序键大量跳过 granule。少读。

③ 向量化

按批取代逐行（火山），摊薄虚调用、紧凑循环。快算。

④ SIMD + cache

一条指令算一批、数据连续 cache 友好。单核数据级并行（老兵主场）。

⑤ MPP

多节点多核并行 + 本地预聚合 + colocate join；常驻进程低延迟。

⑥ CBO + 预计算

CBO 选最优计划；物化视图/预聚合把重活提前算。先少干活再谈并行。

面试自测（点击展开）

基础OLAP 引擎（如 ClickHouse）为什么能比传统数据库快这么多？
少读列存只读相关列 + 稀疏索引/跳数索引跳过无关数据块。
快算向量化执行（按批，摊薄虚调用）+ SIMD + cache 友好布局。
并行MPP 多节点多核并行 + 本地预聚合减少传输。
预计算物化视图/预聚合把高频重查提前算好。
核心火山模型和向量化执行有什么区别？为什么向量化快？
火山每个算子 next() 一次返回一行，逐行拉取——每行每算子一次虚函数调用，开销巨大、cache 不友好。
向量化算子一次处理一批（1024+ 行）的一列——调用开销摊薄、内层是紧凑数组循环、cache 命中高、能上 SIMD。
绝配列存把同列数据连续摆好，正好喂给向量化 + SIMD。
深度SIMD 是什么？OLAP 引擎怎么用上它？
SIMD一条 CPU 指令同时对多个数据做同样运算（AVX-512 一次 16 个 int32），单核数据级并行。
用上向量化执行的"一批连续同类型数组"正好是 SIMD 的输入；求和/过滤比较/哈希都能批量算。
前提数据连续、对齐（列存创造的条件）；这正是 C++ 性能工程那一套。
实战ClickHouse 建表时排序键怎么选？为什么重要？
选把最常用于 WHERE 过滤的列（及其组合，低基数在前）选作排序键。
为何数据按排序键物理有序，稀疏索引才能让查询跳过大量无关 granule，少读=快。
选错过滤列不在排序键上 → 无法跳过、退化成全扫，再多机器也慢。
选型ClickHouse、StarRocks、Trino 分别什么时候用？
ClickHouse单大宽表极致扫描/聚合（日志、埋点、监控），多表 join 弱。
StarRocks/Doris需要强 join + CBO 的实时数仓、多表报表，兼容 MySQL 协议。
Trino不自存数据，直接查数据湖 / 跨源联邦即席查询。

术语表速查 · Glossary

本篇核心名词一次收齐：

① 引擎与存储

OLAP 引擎 / MPP 数据库为交互式分析而生的查询引擎，亚秒扫海量行做聚合。ClickHouse/StarRocks/Doris/Druid/Pinot/Trino/Kylin。
实时摄取型 OLAP Druid / Pinot数据按时间切 Segment，新数据先进可查内存节点、后台固化建索引，做到「边写边查」。为高并发低延迟、面向用户的实时分析而生。
星树索引 Star-TreePinot 的「按需预聚合」索引：在存储空间与查询速度间可调，避免全量 Cube 的维度爆炸。
MergeTreeClickHouse 核心引擎：按排序键物理排序 + 分颗粒压缩 + 稀疏索引；后台 merge（LSM 同源）。
颗粒 Granule列数据的固定行数块（默认 8192 行），读取与索引的最小单位。
稀疏索引 Sparse Index每个 granule 只记一条索引（首行），索引极小可常驻内存，用于定位目标 granule。
排序键 / 跳数索引数据物理排序依据；跳数索引（min-max/bloom/set）进一步在 granule 级跳过无关数据。

② 执行模型

火山模型 Volcano / Iterator算子 next() 逐行拉取。优雅但每行每算子一次虚调用，海量行时开销巨大。
向量化执行 Vectorized算子一次处理一批（1024+ 行）一列。摊薄调用、紧凑循环、cache 友好、能上 SIMD。
SIMD单指令多数据，一条 CPU 指令算多个数据。向量化的天然加速器，单核数据级并行。
代码生成 / Veloxcodegen 把算子编译成紧凑代码（呼应 D03 Tungsten）；Velox 是可复用的 C++ 向量化执行库。

③ 分布式执行

MPP大规模并行处理：查询拆到多节点多核并行，shared-nothing，吞吐随节点近线性扩展。
本地预聚合 Partial Aggregation各节点先本地聚合成小结果再汇总，大减网络传输（呼应 D03 map 端预聚合）。
Broadcast / Shuffle / Colocate Join广播小表 / 按 key 重分布 / 相同分桶天然同节点免 shuffle（OLAP 特色高性能 join）。

④ 优化与预计算

CBO Cost-Based Optimizer用统计信息估算计划成本选最优，尤其 join 顺序/算法。需定期 ANALYZE 收集统计。
物化视图 Materialized View预先算好并存储查询结果，常自动改写命中。数据更新需刷新。
预聚合 Pre-aggregation把高频聚合提前算（写入时或后台 merge 时），查询读少量预聚合结果（呼应 D02 DWS）。
联邦查询 Federated一条 SQL 跨多个数据源（湖/库）查询并 join，Trino 的招牌能力。
