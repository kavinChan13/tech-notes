Spark 是什么 · 为什么取代了 MapReduce

一个分布式计算引擎 · 用内存 + DAG 把批处理提速一个数量级

Apache Spark 是当今最主流的分布式计算引擎：你写一段看似单机的代码（map、filter、join、聚合），它自动把任务切片、分发到一群机器上并行执行、容错、再汇总结果。D01 提过它的由来——MapReduce 太慢，Spark 用内存计算 + DAG 调度把迭代式作业提速 10–100 倍，逐渐成为批处理事实标准。

MapReduce 像每做一道工序就把半成品打包入库、下道工序再拆箱取出——每一步都落磁盘，反复搬运极慢。Spark 像一条流水线，半成品在工位间直接传递（留在内存），只在必要时才入库。同样的活，省掉了大量来回搬运（磁盘 IO），自然快得多。
为什么 Spark 比 MapReduce 快？三个核心原因：① 内存计算——中间结果尽量留内存，不像 MR 每步都落盘；② DAG 执行——把整个作业看成一张有向无环图统一优化、流水线执行，而非一连串割裂的 map/reduce；③ 惰性求值 + 优化器——先攒齐所有操作再整体优化，避免冗余计算。
Driver 是大脑（你的代码 + 调度），Executor 是工人（真正执行 Task、缓存数据）。Cluster Manager 负责申请机器资源。
本篇默认你接触过 Spark 的基本 API。我们聚焦「它内部到底怎么跑」——这正是从「会用」到「能调优、能面试」的分水岭。涉及的分布式基础（容错、调度）可回看 architect/ 板块。
01

RDD 与惰性求值

Spark 的「内存指针」抽象 · 理解它，才懂上层 DataFrame / SQL 为何那样跑

RDD（Resilient Distributed Dataset，弹性分布式数据集）是 Spark 最底层的核心抽象。如今你多半写 DataFrame / SQL，但它们最终都编译成 RDD 执行——理解 RDD，是理解一切的基础。

RDD：一个分布在多台机器上、被切成多个分区（partition）的只读数据集。它有三个关键词：弹性（Resilient）——靠血缘可重算恢复，不怕节点挂；分布式（Distributed）——数据分区散在集群各处并行处理；数据集（Dataset）——一份逻辑上的数据集合。它更像「一份计算的配方 + 数据指针」，而非真正持有数据。

1.1 RDD 的五大特性

① 分区列表（Partitions）

数据被切成多个分区，分区数 = 并行度。一个分区由一个 Task 处理。

② 计算函数（compute）

对每个分区怎么算的函数。Spark 在每个分区上独立应用它。

③ 依赖（Dependencies）

这个 RDD 由哪些父 RDD 算来——即血缘（lineage），是容错和 DAG 的基础。

④ 分区器（Partitioner）

（针对 KV 型 RDD）数据按 key 怎么分到各分区，如 Hash / Range。决定 shuffle 行为。

⑤ 首选位置（Preferred Locations）

每个分区「最好在哪台机器上算」——数据本地性（data locality）。Spark 尽量把计算调度到数据所在节点，避免网络搬运。这与 HDFS「移动计算比移动数据便宜」一脉相承。

1.2 转换（Transformation） vs 动作（Action）——惰性求值

写 RDD 操作像列购物清单：你写「买菜、洗菜、切菜」（transformation），并不会真的开始动手——清单只是记下来。直到你说「开饭！」（action），厨房才一次性按清单把活干完。这就是惰性求值（lazy evaluation）：攒齐所有步骤，到 action 才真正触发计算。
转换 Transformation动作 Action
作用定义「怎么变」，返回新 RDD触发真正计算，返回结果/落地
是否立即执行否（惰性，只记血缘）是（触发整个 DAG）
例子map / filter / groupByKey / joincollect / count / save / take
 
// 这几行不会触发任何计算——只是在攒"配方"（血缘）
val rdd = sc.textFile("logs/*.txt")   // transformation
  .map(line => line.split(","))            // transformation
  .filter(arr => arr(2) == "ERROR")       // transformation

val n = rdd.count()   // ← action！到这里 Spark 才真正读文件、跑整条链
惰性求值不是偷懒，是为了优化：攒齐所有操作后，Spark（尤其上层的 Catalyst）能做谓词下推、合并算子、裁剪列等全局优化——这和编译器「看到整段代码再优化」是同一种智慧。代价：调试时 action 才报错，错误栈常指向 action 那一行而非真正出错的 transformation。

1.3 血缘与容错——「弹性」从哪来

血缘（Lineage）：RDD 记录的「我是从哪些父 RDD、经过哪些转换得来的」这条family tree。容错的核心：某个分区的数据因节点宕机丢了，Spark 不需要回滚整个作业，只需沿血缘把那个分区重算一遍即可。这比 MapReduce「靠落盘 + 重跑」轻量得多，也是「弹性」二字的来源。
血缘太长（如迭代上百次的机器学习）会让重算代价高、DAG 臃肿。此时用 checkpoint() 把 RDD 物化到可靠存储「截断血缘」，或用 cache()/persist() 把热点 RDD 缓存到内存复用——后者正是 Spark 对迭代算法极快的关键。
02

DAG 与宽窄依赖

action 一触发，Spark 把血缘编译成 DAG，再按「依赖类型」切成 Stage · 这是性能的关键分水岭

当 action 触发，Spark 把整条血缘组织成一张 DAG（有向无环图），然后做一件最关键的事：按依赖类型把 DAG 切成若干 Stage（阶段）。切分的依据，就是「窄依赖」还是「宽依赖」。

2.1 窄依赖 vs 宽依赖

窄依赖（Narrow Dependency）：父 RDD 的每个分区，最多被一个子分区使用（1→1 或 N→1 的局部）。如 map / filter。数据不跨分区流动，可在同一台机器上流水线连续执行，无需网络传输。
宽依赖（Wide Dependency / Shuffle Dependency）：父 RDD 的一个分区，会被多个子分区使用（需按 key 重新分发）。如 groupByKey / reduceByKey / join。数据必须跨节点重新洗牌（Shuffle）——这是昂贵操作，且是 Stage 的分界线。
窄依赖的连线是「平行直线」（1→1）；宽依赖是「交叉网状」（N→M，要按 key 重新分发）。这张图是理解 Stage 划分的关键。
Stage（阶段）：Spark 把 DAG 按宽依赖切成的执行单元。一连串窄依赖被合并进同一个 Stage 流水线执行；每遇到一个宽依赖（Shuffle）就切出一个新 Stage。 所以记住这条黄金公式：Stage 数 = Shuffle 次数 + 1。每个 Stage 内部又按分区拆成多个并行的 Task。

2.2 交互演示：Shuffle 如何切分 Stage

选择不同的算子流水线，看 Spark 如何按宽依赖（红色算子）切分 Stage。注意 Stage 边界永远落在 Shuffle 处：

交互演示 · DAG → Stage 划分
绿色算子（map/filter）是窄依赖，能和前后算子挤进同一个 Stage 流水线跑，几乎零额外成本；红色算子（groupByKey/join）是宽依赖，强制 Shuffle、切出新 Stage。优化 Spark 作业，本质就是「想方设法减少 Shuffle」——这正是下一章的主题。
03

Shuffle 内部：性能的头号命门

几乎所有 Spark 慢、OOM、倾斜问题都和 Shuffle 有关 · 老兵尤其要懂它的磁盘 / 网络代价

如果只能记住 Spark 调优的一件事，那就是：Shuffle 很贵，能少则少。它同时吃磁盘 IO、网络、序列化、内存四样资源，是绝大多数性能问题的根源。

Shuffle 像一个班级重新分组：原来按「入学顺序」坐（父分区），现在要求「同省份的坐一起」（按 key 重分区）。于是每个人都得收拾东西（map 端写磁盘），走到新组（网络传输），新组再把来自各方的同省同学聚齐（reduce 端拉取 + 合并）。全班一起搬，乱哄哄又费时——这就是 Shuffle 昂贵的直觉。
Map 端把输出按目标分区切好、写本地磁盘（shuffle write）；Reduce 端跨网络把属于自己的分区都拉过来（shuffle read / fetch）再聚合。中间这一写一读，就是 Shuffle 的全部开销来源。

为什么 Shuffle 这么贵？四笔账

  • 磁盘 IO：map 端输出要落本地盘（shuffle write），reduce 端读取——大量随机/顺序 IO。
  • 网络：reduce 端要跨节点把数据拉过来，数据量大时网络成瓶颈。
  • 序列化：跨网络传输前要序列化、收到后反序列化，吃 CPU。
  • 内存与排序：聚合 / 排序需要内存缓冲，放不下就 spill 到磁盘，进一步拖慢。
少 Shuffle 的实战手段：① 用 reduceByKey 替代 groupByKey（前者 map 端先局部聚合，shuffle 的数据量小得多）；② 小表 join 大表用 Broadcast Join（把小表广播到各节点，完全消除 shuffle）；③ 提前 filter 减少进入 shuffle 的数据；④ 复用已分区的 RDD（相同 partitioner 的 join 可免 shuffle）。
groupByKey vs reduceByKey 是经典面试题：两者都触发 shuffle，但 reduceByKey 在 map 端先做局部聚合（combine），跨网络传的是「已经合并过的小结果」；groupByKey 把所有原始值都搬过网络再聚合，数据量大时极易 OOM。能用 reduceByKey / aggregateByKey 就别用 groupByKey。
04

Catalyst 优化器 & Tungsten 引擎

为什么写 DataFrame / SQL 往往比手写 RDD 还快 · 因为有一个编译器在帮你优化

很多人惊讶：用高层的 DataFrame / SQL，居然常比手写 RDD 更快。原因是 Spark SQL 背后有两套「黑科技」——Catalyst（查询优化器）和 Tungsten（执行引擎）。它们把你的声明式查询，编译成高度优化的物理执行。

4.1 Catalyst：把 SQL 当代码来编译优化

Catalyst：Spark SQL 的查询优化器。它把一条 SQL / DataFrame 操作，经过解析 → 分析 → 逻辑优化 → 物理计划 → 代码生成多个阶段层层优化，最终生成高效的 RDD 执行。这和你熟悉的编译器前端/中端/后端结构几乎一一对应。
和编译器同构：解析→语义分析→中端优化→后端代码生成。正因有这层优化，声明式的 SQL 反而比手写 RDD 更容易跑得快。
  • 谓词下推（Predicate Pushdown）：把 WHERE 尽量推到数据源（如只读 Parquet 中符合条件的 row group）——呼应 D01！
  • 列裁剪（Column Pruning）：只读查询用到的列——又是 D01 列存的红利。
  • 常量折叠、谓词合并、Join 重排：CBO 利用统计信息选更优的 join 顺序与算法。

4.2 Tungsten：榨干 CPU 与内存

Tungsten：Spark 的底层执行引擎，目标是「把硬件性能榨干」。三板斧：① 堆外内存 + 自管理——绕开 JVM 对象开销和 GC，用紧凑的二进制格式存数据；② 缓存感知计算——数据布局对 CPU cache 友好，减少 cache miss；③ 全阶段代码生成（Whole-Stage Codegen）——把一个 Stage 里的多个算子编译成一段紧凑的 Java 代码，消除虚函数调用和中间对象，像手写的 for 循环一样高效。
老兵会心一笑：Tungsten 做的事，正是你在 C++ 里追求的——避免对象开销（紧凑内存布局）、cache 友好、减少分支与虚调用（codegen 把火山模型的 next() 虚调用消成直筒循环）、必要时上 SIMD 向量化。分布式引擎的单节点性能，最终还是单机性能工程那一套。 这也是 D07（OLAP 引擎内核）的核心母题。
DataFrame / Dataset / SQL 共享 Catalyst + Tungsten，所以性能相近且通常优于 RDD；RDD 是更底层的逃生舱（需要细粒度控制、非结构化数据时才直接用）。日常优先 DataFrame / SQL，把优化交给 Catalyst。
05

内存管理与执行模型

Executor 里的内存怎么分 · 一个作业从提交到执行经历了什么

5.1 执行模型：Job → Stage → Task

一个 action 触发一个 Job；Job 按宽依赖切成多个 Stage；每个 Stage 按分区拆成多个 Task（最小执行单元，一个 Task 处理一个分区）。Task 被分发到 Executor 的 CPU core 上并行跑。

Job→一个 action（如 count()）= 一个 Job
Stage→Job 按 Shuffle 边界切成多个 Stage（Stage 数 = shuffle 数 + 1）
Task→Stage 按分区切成 N 个 Task，并行跑在 Executor 的 core 上（并行度 = 分区数）
并行度（Parallelism）：同时能跑多少 Task ≈ 分区数，上限是「Executor 数 × 每个 Executor 的 core 数」。分区太少 → 并行不足、机器闲置；分区太多 → 调度开销大、小文件多。经验值：分区数设为总 core 数的 2–4 倍较均衡。

5.2 统一内存模型

Executor 的内存像一个合租的开间：执行内存（Execution，跑 shuffle/排序/聚合用）和存储内存（Storage，缓存 RDD 用）是两个室友，共用一套空间。Spark 的「统一内存管理」让它们能互相借用：谁暂时不用，另一个可以多占；但执行内存「优先级更高」，需要时能把存储内存挤出去（缓存被驱逐）。
内存不够时数据会 spill 到磁盘（溢写），作业还能跑完但会变慢。频繁 spill / OOM 是内存配置或数据倾斜的信号。
常见 OOM 不一定是「内存小」，更多是单个 Task 的数据太多（分区不均 / 数据倾斜）或 groupByKey 把海量值堆到一处。先排查倾斜（下一章），再谈加内存。
06

调优与数据倾斜

Spark 工程师的日常 · 90% 的「Spark 跑得慢」最后都查到 Shuffle 或倾斜

数据倾斜（Data Skew）是 Spark 最常见、最折磨人的性能问题：大部分 Task 秒级完成，个别 Task 跑几十分钟——整个 Stage 被这几个「拖油瓶」卡住。根因是数据按 key 分区时分布极不均匀（如某个热点 key 占了一半数据）。

掉队者 / 长尾 Task（Straggler）：一个 Stage 必须等它所有 Task 都完成才算结束，所以 Stage 耗时 ≈ 最慢的那个 Task。哪怕 99 个 Task 各 1 秒，只要 1 个 Task 要 100 秒，这个 Stage 就得 100 秒。这就是为什么「平均很快但总体很慢」。

下面每个条代表一个 Task 处理的数据量（≈ 耗时）。倾斜时 P2 是热点、拖垮整个 Stage。点「加盐打散」看如何把热点 key 拆开、让负载均衡：

加盐（Salting）：给热点 key 拼一个随机后缀（如 hotkey_0…hotkey_9），把它「拆成 10 个 key」分散到不同分区先做局部聚合，再去掉后缀做一次全局聚合。热点被打散，长尾 Task 消失。这是处理倾斜最通用的招。

Broadcast Join大表 join 小表的 shuffle小表广播到各节点，彻底消除 shuffle。注意小表别太大（默认阈值 10MB，可调）

AQE（Adaptive Query Execution，自适应查询执行）：Spark 3 的杀手锏。传统优化在「跑之前」基于静态统计做决策，常因估计不准而次优。AQE 让 Spark 在运行中、拿到每个 Stage 真实的中间结果统计后，动态调整后续计划——合并过小的分区、把倾斜的大分区再切分、把原计划的 shuffle join 改成 broadcast join。它把很多手工调优自动化了，是现代 Spark 调优的第一道开关。

6.3 Join 策略与 AQE 动态优化

Join 是最容易出性能问题的算子。Spark 有几种物理 Join 策略，选对策略往往比加资源更管用。理解它们，也是面试高频考点。

Join 策略原理是否 Shuffle适用
Broadcast Hash Join
(BHJ)		把小表广播到每个 Executor，在本地用大表流式探测哈希表无（仅广播小表）大表 join 小表（小表 < 阈值，默认 10MB）。最快
Shuffle Hash Join
(SHJ)		两表按 key shuffle 到同分区，小侧建哈希表、大侧探测是一侧较小但放不下广播、且无需排序时
Sort-Merge Join
(SMJ)		两表按 key shuffle 后各自排序，再归并匹配是（+排序）两个大表 join 的默认选择，稳健但有排序开销
Broadcast Nested Loop
(BNLJ)		嵌套循环 + 广播，O(n×m)视情况兜底（如非等值 join），慢，尽量避免
Spark 怎么选 Join 策略？ 大致优先级：能广播（有一侧足够小）→ BHJ；否则等值 join → SMJ（默认）或 SHJ；非等值/无法 hash → BNLJ 兜底。你可以用 EXPLAIN 看物理计划里到底走了哪种，也能用 broadcast() hint 强制广播。「大表 join 小表慢」十有八九是没走 BHJ——要么小表超了广播阈值，要么统计信息不准让 Spark 误判。
能广播就别 shuffle：BHJ 把小表搬给数据、SMJ 把两个大表都搬到一起。这正是「移动小数据 ≪ 移动大数据」的体现。

AQE 的三大动态优化（运行时才出手）

① 动态合并分区

Coalesce Shuffle Partitions：shuffle 后发现很多分区很小，自动合并成更少的大分区，减少 Task 调度开销与小文件。

② 动态处理倾斜

Skew Join：运行时检测到某分区远大于中位数，自动把它再切成多个子分区并行处理，免去手工加盐。

③ 动态切换 Join

运行时拿到真实大小，发现一侧其实够小 → 把原计划的 SMJ 临时改成 BHJ，省掉一次大 shuffle。

为什么 AQE 有用？ 传统优化在「跑之前」靠静态统计估算，常因数据分布估不准而选错策略。AQE 等到每个 Stage 真实跑完、拿到准确统计再决定下一步——用"运行时真相"纠正"编译期猜测"。这就是为什么它是现代 Spark 调优的第一道开关：spark.sql.adaptive.enabled=true（Spark 3 默认开）。
调优顺序建议：① 先确认开了 AQE；② 看 Spark UI 找最慢的 Stage / Task（是不是倾斜？）；③ 减少 / 避免 Shuffle（broadcast、reduceByKey、提前 filter）；④ 检查 Join 策略是否走了预期的 BHJ；⑤ 最后才是加资源。先定位再动手，别上来就堆内存堆 core。 Spark UI 的 DAG 图和 Task 时间分布是你最好的朋友。
07

Structured Streaming 与作业部署

同一套 DataFrame API 也能写流 · 以及作业到底怎么跑在 YARN / K8s 上

前 6 章讲的都是批。但 Spark 也能做流——而且用的是同一套 DataFrame/SQL API。这一章补齐两块面试与实战的高频盲区：Structured Streaming（结构化流）的心智模型，以及一个作业从 spark-submit 到真正在集群上跑起来，中间经历了什么（YARN / K8s / 动态资源）。

7.1 从 DStream 到 Structured Streaming：把流看成「一张不断长高的表」

Spark 的流处理有两代：老的 Spark Streaming（DStream）是「把流切成一个个小批 RDD」；新的 Structured Streaming（2.0+，现在的事实标准）则换了个更优雅的心智模型——

Structured Streaming 把一条无界数据流，想象成一张永远在底部追加新行的表。你对这张「无界表」写的查询（groupBy、join、agg），和写一张静态表的批查询一模一样；引擎负责在每个微批到来时，增量地把新行并进结果。你写的是「批」的逻辑，跑出来的是「流」的效果。
批与流共用一套 API 的本质：流只是「无界的批」，引擎把你的批查询持续增量地跑在新到的行上。
微批 vs 连续处理（Micro-batch vs Continuous）：Structured Streaming 默认是微批——每隔很短的触发间隔（如 1s）把累积的新数据当一个小批处理，延迟通常百毫秒~秒级。它还有实验性的 Continuous Processing 模式可做到毫秒级，但功能受限。这正是它与 Flink 的根本差异：Spark 是「批优先、流是把批切小」，Flink 是「流优先、批是有界的流」（详见 D04）。
# Structured Streaming：读 Kafka → 开窗聚合 → 写下游，全是 DataFrame API
df = (spark.readStream
        .format("kafka")
        .option("subscribe", "orders")
        .load())

agg = (df.selectExpr("CAST(value AS STRING)")
         .select(from_json("value", schema).alias("o")).select("o.*")
         .withWatermark("event_time", "10 minutes")        # 容忍迟到 10min
         .groupBy(window("event_time", "5 minutes"), "city")
         .sum("amount"))

query = (agg.writeStream
           .outputMode("update")          # 输出模式见 7.2
           .format("console")
           .option("checkpointLocation", "/ckpt/orders")  # 容错的根
           .trigger(processingTime="5 seconds")
           .start())

7.2 Watermark、输出模式与「精确一次」

流处理的三个共性难题，Structured Streaming 都有自己的答法（概念与 D04 Flink 相通，这里只讲 Spark 的形态）：

Watermark 水位线

withWatermark 告诉引擎「事件时间晚于水位这么久的迟到数据就丢弃」，从而能关闭旧窗口、清理状态，否则状态无限膨胀。

三种输出模式

Append（只追加最终行）、Update（只输出有变化的行）、Complete（每次输出整张结果表）。开窗聚合常用 Update/Append。

Exactly-once

靠 Checkpoint + WAL + 可重放 source（Kafka offset）+ 幂等 sink。故障重启从 checkpoint 恢复进度与状态，端到端不丢不重。

Structured Streaming 的状态默认存在 Executor 内存 + HDFS checkpoint，大状态场景（海量 key 的去重/聚合）不如 Flink 的 RocksDB 状态后端从容。选型直觉：已有重 Spark 批生态、延迟要求秒级、想「一套代码批流复用」→ Structured Streaming；超低延迟、复杂事件时间/大状态/CEP → Flink（D04）。

7.3 作业部署：spark-submit 之后发生了什么

写完代码只是第一步。一个 Spark 作业要真正跑起来，得先向集群管理器申请资源、拉起 Driver 与 Executor。这是「能写代码」和「能上生产」之间的分水岭，也是高频面试点。

集群管理器定位适用
StandaloneSpark 自带的简单调度测试 / 小集群，不与其他框架共享资源
YARNHadoop 生态主流传统大数据集群，和 Hive/MR 共用资源池
Kubernetes云原生主流（趋势）容器化、弹性、存算分离、多租户隔离
Mesos早期方案已基本退出历史舞台
client 模式 vs cluster 模式：决定 Driver 跑在哪。① client：Driver 跑在你提交命令的那台机器（如跳板机），适合调试、能直接看到输出，但提交机一关作业就挂。② cluster：Driver 也被调度到集群里某个容器跑，提交机可以下线，生产标准。
无论 YARN 还是 K8s，套路一致：管理器先拉起 Driver，Driver 再申请一批 Executor 容器执行 Task。区别只在「容器是 YARN Container 还是 K8s Pod」。
# 提交到 YARN（cluster 模式，生产常见）
spark-submit --master yarn --deploy-mode cluster \
  --num-executors 20 --executor-cores 4 --executor-memory 8g \
  --conf spark.sql.adaptive.enabled=true \
  app.py

# 提交到 Kubernetes（存算分离 / 云原生）
spark-submit --master k8s://https://<api-server> --deploy-mode cluster \
  --conf spark.kubernetes.container.image=my-spark:3.5 \
  --conf spark.executor.instances=20 \
  --conf spark.dynamicAllocation.enabled=true \   # 动态资源：按需增减 Executor
  local:///opt/app.py
动态资源分配（Dynamic Allocation）：让 Spark 根据待处理 Task 量自动增减 Executor——空闲就释放、积压就申请。配合 K8s 弹性，是云上省成本的关键开关（呼应 D09 FinOps）。on K8s 时还需开 shuffleTracking 或外部 shuffle 服务来安全回收带 shuffle 数据的 Executor。

7.4 DataFrame / PySpark 实用 API 速查

面试写代码、实战上手，这套结构化 API 是肌肉记忆。一页带走最常用的几类操作：

# 读 → 转换 → 聚合 → 写，典型批处理骨架
df = spark.read.parquet("s3://lake/orders/")              # 也可 .format("iceberg").table("db.orders")

res = (df
  .filter(col("status") == "PAID")                     # 谓词（会下推到 Parquet）
  .withColumn("day", to_date("event_time"))            # 派生列
  .groupBy("day", "city")
  .agg(sum("amount").alias("gmv"),
       countDistinct("user_id").alias("uv"))
  .join(broadcast(dim_city), "city")                   # 小表广播 join，免 shuffle
  .orderBy(desc("gmv")))

res.write.mode("overwrite").partitionBy("day").parquet("s3://lake/dws/gmv/")

# 等价 SQL 写法（同样走 Catalyst，二选一看团队习惯）
df.createOrReplaceTempView("orders")
spark.sql("SELECT day, city, sum(amount) gmv FROM orders WHERE status='PAID' GROUP BY day, city")
记住一条工程原则：优先 DataFrame/SQL，而非裸 RDD（呼应第 4 章 Catalyst）。前者能享受谓词下推、列裁剪、AQE、Tungsten codegen；RDD 只在需要细粒度控制（自定义分区器、非结构化数据）时才下沉使用。PySpark 的 DataFrame 算子本质是调用 JVM 端的 Catalyst，没有 Python 序列化开销——但一旦用 Python UDF 就会有，能用内置函数就别写 UDF。

小结 · 面试 · 术语表

收束成判断力 · 并接上 D04 Flink

一页纸总结

① RDD + 惰性

RDD 是分区化的「计算配方」；transformation 惰性、action 触发；血缘支撑容错。

② DAG → Stage

窄依赖流水线进同一 Stage；宽依赖（shuffle）切 Stage。Stage 数 = shuffle 数 + 1。

③ Shuffle 最贵

磁盘+网络+序列化+内存。少 shuffle 是调优主线：broadcast、reduceByKey、提前 filter。

④ Catalyst + Tungsten

SQL 被编译优化（下推/裁剪/CBO）+ codegen/堆外内存/cache 友好。DataFrame 优先。

⑤ 执行 / 内存模型

Job→Stage→Task；并行度≈分区数；执行/存储内存统一可借用，不够则 spill。

⑥ 倾斜与 AQE

长尾 Task 拖垮 Stage；加盐打散热点；先开 AQE 自适应，再看 UI 定位。

面试自测（点击展开）

基础什么是 RDD？transformation 和 action 有什么区别？
RDD弹性分布式数据集——分区化、只读、靠血缘容错的「计算配方 + 数据指针」。
transformation惰性，只记血缘返回新 RDD（map/filter/join），不触发计算。
action触发整条 DAG 真正执行并返回结果（count/collect/save）。
为何惰性攒齐操作后能整体优化（下推、合并算子），和编译器优化同理。
核心窄依赖和宽依赖的区别？Stage 是怎么划分的？
窄依赖父分区→最多一个子分区（map/filter），不跨网络，可流水线。
宽依赖父分区→多个子分区，需按 key 跨节点 shuffle（groupByKey/join）。
划分从 action 回溯 DAG，遇宽依赖就切 Stage——Stage 数 = shuffle 数 + 1。
高频groupByKey 和 reduceByKey 有什么区别？该用哪个？
共同点都是宽依赖、都触发 shuffle。
reduceByKeymap 端先局部聚合（combine），跨网络只传合并后的小结果。
groupByKey把所有原始值都搬过网络再聚合，数据量大时极易 OOM。
结论能用 reduceByKey/aggregateByKey 就别用 groupByKey。
实战线上一个 Spark 作业卡在某个 Stage 很久，怎么排查？
看 UISpark UI 找最慢 Stage，看 Task 时间分布——少数 Task 远慢于其他 → 数据倾斜。
确认看倾斜 Task 的输入数据量是否远大于其他（热点 key）。
解决① 开 AQE 倾斜处理；② 加盐打散热点 key；③ 大表 join 小表改 broadcast；④ 用 reduceByKey 减少 shuffle 量。
原则先定位（UI）再动手，别盲目加资源。
深度为什么 DataFrame/SQL 常比手写 RDD 快？
Catalyst优化器对声明式查询做谓词下推、列裁剪、Join 重排、CBO——手写 RDD 享受不到。
Tungsten全阶段代码生成 + 堆外内存 + cache 友好布局，消除虚调用与对象开销。
本质你把"怎么做"交给引擎，引擎比大多数人手写的 RDD 优化得更好。RDD 留给需要细粒度控制的场景。
高频Spark Structured Streaming 和 Flink 该怎么选？
Spark SS微批模型、延迟秒级；最大优势是和批共用一套 DataFrame/SQL、复用现有 Spark 生态与运维。
Flink真流优先、延迟毫秒级、事件时间/大状态(RocksDB)/CEP 更强。
选型已有重 Spark 批栈、要"一套代码批流复用"、延迟容忍秒级 → Spark SS；超低延迟、复杂事件时间、大状态去重 → Flink。
实战client 模式和 cluster 模式有什么区别？生产用哪个？
clientDriver 跑在提交机（跳板机），能直接看输出、便于调试；但提交机一断，作业就挂。
clusterDriver 被调度进集群容器，提交机可下线，稳定——生产标准。
本质区别只在「Driver 进程跑在哪」。on YARN/K8s 生产作业一律 cluster 模式。

术语表速查 · Glossary

本篇核心名词一次收齐：

① 核心抽象

RDD Resilient Distributed DatasetSpark 最底层抽象：分区化、只读、靠血缘容错的分布式数据集。DataFrame/SQL 最终都编译成 RDD。
分区 PartitionRDD 被切成的数据块，一个分区由一个 Task 处理。分区数 = 并行度。
血缘 LineageRDD 记录的「从哪些父 RDD 经哪些转换得来」。丢分区时沿血缘重算即可，是「弹性」之源。
惰性求值 Lazy Evaluationtransformation 只记血缘不执行，直到 action 才触发整条 DAG。为全局优化创造空间。
转换 / 动作 Transformation / Action转换返回新 RDD（map/filter/join，惰性）；动作触发计算返回结果（count/collect/save）。
DataFrame / Dataset带 schema 的高层结构化 API，享受 Catalyst+Tungsten 优化，性能通常优于裸 RDD。

② 执行与调度

DAG 有向无环图action 触发后，Spark 把血缘组织成 DAG 来调度与优化。
窄依赖 / 宽依赖 Narrow / Wide窄=父分区→最多一子分区（可流水线）；宽=需跨节点按 key shuffle（切 Stage）。
Stage / Task / JobJob（一个 action）→ Stage（按 shuffle 切）→ Task（按分区切，最小执行单元）。
Driver / ExecutorDriver=大脑（建 DAG、调度）；Executor=工人（跑 Task、缓存数据）。
数据本地性 Data Locality尽量把计算调度到数据所在节点，减少网络搬运。「移动计算比移动数据便宜」。
并行度 Parallelism同时可跑的 Task 数 ≈ 分区数，上限 = Executor 数 × 每 Executor core 数。

③ Shuffle 与性能

Shuffle按 key 跨节点重新分发数据（map 端写盘 + reduce 端拉取）。吃磁盘/网络/序列化/内存，最贵操作。
数据倾斜 Data Skew数据按 key 分布极不均，热点分区拖慢整个 Stage。Spark 头号性能杀手。
掉队者 Straggler远慢于其他的长尾 Task。Stage 耗时 ≈ 最慢 Task，所以一个 straggler 能拖垮全局。
加盐 Salting给热点 key 拼随机后缀拆成多 key，先局部聚合再全局聚合，打散倾斜。
Broadcast Join把小表广播到各 Executor，本地完成 join，彻底消除 shuffle。大表 join 小表首选。
spill 溢写内存放不下时把中间数据写到磁盘。能跑完但变慢，频繁 spill 是内存/倾斜信号。

④ 优化引擎

CatalystSpark SQL 查询优化器：解析→分析→逻辑优化→物理计划→codegen，含谓词下推/列裁剪/CBO。
Tungsten底层执行引擎：堆外内存、cache 友好布局、全阶段代码生成，榨干单节点 CPU/内存。
Whole-Stage Codegen把一个 Stage 的多算子编译成一段紧凑代码，消除虚调用与中间对象，像手写循环般高效。
AQE Adaptive Query Execution运行时根据真实统计动态调整计划：合并小分区、拆倾斜分区、改 join 策略。现代调优第一开关。
cache / persist把 RDD/DataFrame 缓存到内存（或磁盘）复用，避免重算血缘。迭代算法提速关键。

⑤ 流处理与部署

Structured StreamingSpark 现代流引擎：把流看成「无界表」，用与批相同的 DataFrame/SQL 表达，默认微批增量执行。
微批 Micro-batch把流按短触发间隔切成小批处理，延迟百毫秒~秒级。与 Flink 的真流（毫秒级）相对。
输出模式 Output ModeAppend（只追加最终行）/ Update（只输出有变化行）/ Complete（输出整张结果表）。
Checkpoint 检查点流作业把进度（offset）与状态持久化到 HDFS/对象存储，故障重启从此恢复，是 Exactly-once 的根。
集群管理器 Cluster Manager为作业分配资源、拉起 Driver/Executor：Standalone / YARN / Kubernetes。
client / cluster 模式决定 Driver 跑在提交机还是集群容器里。生产用 cluster（提交机可下线）。
动态资源分配 Dynamic Allocation按待处理 Task 量自动增减 Executor，空闲释放、积压申请，云上省成本关键开关。
