Flink 流处理深潜

D03 的 Spark 是「批优先」——流只是把批切得很小。本篇的 Flink 是「流优先」——把无界数据流当作一等公民。这套世界观带来一组全新的硬核概念：事件时间与 Watermark（如何处理乱序与迟到）、窗口（在无限流上怎么做聚合）、State 与 Checkpoint（有状态计算如何容错）、Exactly-once（故障重放也不丢不重）、反压。理解它们，你就理解了实时数仓的引擎担当，也理解了「真正的流」与「微批」的本质差异。

00

流优先 vs 批优先：两种世界观

Spark 把流当"小批"，Flink 把批当"有界的流" · 这个起点差异决定了一切

Spark Streaming 的本质是 微批（micro-batch）：把流切成一个个小批次，每隔几百毫秒/几秒跑一次批处理。Flink 则是 真正的流（true streaming）：数据一条来就处理一条，延迟可达毫秒级。「批是有界的流」——这是 Flink 的核心信条。

微批像公交车：你到站后要等下一班车（攒够一个批次窗口）才能走，发车间隔就是延迟下限。真流像自动扶梯：你一踏上去就被持续往上送，几乎不用等。公交吞吐高、调度简单；扶梯延迟低、随到随走。两种都能到目的地，但"等不等"的体验天差地别。
微批把时间切块、整块处理；真流持续不断逐条推进。Spark 后来也有 Continuous Processing，Flink 也支持批，但二者的「出身」差异塑造了各自的设计。
维度Spark（批优先 / 微批）Flink（流优先）
核心模型把流切成小批把批看成有界的流
延迟秒级（批间隔）毫秒级
状态 / 时间语义较弱（后期补强）原生强大（事件时间/Watermark/State）
最擅长大吞吐离线批 + 准实时低延迟实时、复杂事件、实时数仓
不是「谁取代谁」。批处理（离线数仓、训练）Spark 依然强势；低延迟实时（风控、实时大屏、监控）Flink 是标杆。很多公司两者并存：Spark 跑离线，Flink 跑实时（D08 实时数仓会把它们拼起来）。
01

事件时间与 Watermark

流处理最硬核、也最考验理解的概念 · 如何在「数据乱序、迟到」的现实里算对结果

流处理的头号难题是：数据不会按发生顺序乖乖到达。手机离线缓存、网络抖动、分布式乱序——一条「10:00 发生」的事件，可能 10:05 才到。如果按「到达时间」算「10:00 这一分钟的 PV」，结果就错了。Flink 用事件时间 + Watermark这套机制来应对。

1.1 三种时间

事件时间 Event Time

事件实际发生的时间（数据自带的时间戳）。如用户点击的那一刻。唯一能算出"正确"结果的时间。

处理时间 Processing Time

事件被算子处理到的机器时间。最简单、延迟最低，但结果不可复现（取决于到达快慢）。

摄入时间 Ingestion Time

事件进入 Flink的时间。介于两者之间，用得少。

事件 10:00 发生，10:05 才被处理。要算「10:00 这一分钟」的指标，必须用事件时间——但又得等"可能迟到的"数据，这就引出了 Watermark。

1.2 Watermark：流处理的"时间进度条"

Watermark 像考试交卷的"截止广播"：老师说「事件时间到 10:05 为止的卷子应该都交齐了，开始批改 10:00–10:05 这批」。它不是说"绝对没有更晚的"，而是一个务实的判断：等到这个点，迟到的就少到可以接受了，该出结果了。设大了结果更准但更慢，设小了更快但可能漏掉迟到数据——这是延迟 vs 准确性的经典权衡。
Watermark（水位线）：流中穿插的一种特殊标记，声明「事件时间已经推进到 T，比 T 更早的数据应该基本都到齐了」。它由数据源根据已见到的最大事件时间生成（常见：watermark = 当前最大事件时间 − 允许的最大乱序时间）。当 Watermark 越过某个窗口的结束边界，该窗口就触发计算并输出。它是 Flink 在"乱序无界流"上做出"确定性结果"的关键。多输入/多分区场景下，下游算子的有效 Watermark 通常取最小值，因此最慢分区会拖住整体窗口触发（常见现象：watermark stagnation）。
允许迟到（Allowed Lateness）：Watermark 之后才到的数据叫迟到数据（late data）。默认会被丢弃；可设置一段"宽限期"让窗口暂不销毁、迟到数据仍能更新结果；再晚的可以走侧输出流（side output）单独收集，避免直接丢失。

02

窗口 Window：在无限流上做聚合

流是无界的，但「每分钟 PV」需要边界 · 窗口就是把无限流切成有限片来算

流没有尽头，但业务问题总是有界的：「最近 1 分钟的订单量」「每小时 UV」。窗口（Window）就是把无界流按时间（或数量）切成一段段有限的"桶"，在每个桶上做聚合。

2.1 三类窗口

滚动窗口 Tumbling

固定大小、首尾相接不重叠。如「每 5 秒一个窗口」。每条数据只属于一个窗口。最常用。

滑动窗口 Sliding

固定大小，但按更小的步长滑动、会重叠。如「窗口 5 秒、每 2 秒算一次」。一条数据可属于多个窗口。

会话窗口 Session

没有固定大小，按活动间隔切：两条数据间隔超过 gap 就断开成新窗口。适合用户会话分析。

03

State：有状态计算

流处理要"记住过去" · 状态是 Flink 区别于无状态管道的灵魂

很多流计算需要记忆：累计 UV、连续登录天数、上一条事件做差分……这些"到目前为止的中间结果"就是 State（状态）。Flink 把状态做成一等公民——它帮你管理、持久化、容错恢复状态，你只管用。

无状态算子像计算器：算完一个就忘，输入什么输出什么。有状态算子像记账本：每来一笔都记在账上、累计起来。Flink 的厉害之处是这个"账本"跑在内存里飞快，又能定期拍快照存到可靠存储，崩了能从快照恢复——既快又不怕丢。

Keyed State（按 key 的状态）

最常用。数据按 key 分流后，每个 key 维护自己的状态（如每个用户的累计消费）。类型：ValueState、ListState、MapState、Reducing/AggregatingState。

Operator State（算子状态）

与 key 无关，绑定到算子实例。典型用途：Kafka source 记录每个分区读到的 offset。

State Backend（状态后端）：决定状态存哪、怎么存。两种主流：① HashMapStateBackend——状态放 JVM 堆内存，快但受内存限制；② RocksDBStateBackend——状态存本地 RocksDB（可溢写磁盘），能支撑超大状态（TB 级），代价是序列化开销。大状态作业几乎都用 RocksDB——这里又见到了 D01 的 LSM-Tree 老朋友（RocksDB 是 LSM 引擎）。
状态会无限增长吗？会——所以要用 State TTL（过期清理） 给状态设过期时间，否则像 UV 去重这种状态会越积越大，最终 OOM 或撑爆磁盘。状态治理是有状态流作业的运维重点。

3.1 状态原语：选对类型，少踩坑

Keyed State 不是只有一种。选错类型会带来不必要的序列化和性能问题，这是 Flink 进阶面试的细节考点：

状态类型存什么典型用途 / 注意
ValueState<T>单个值计数器、上一条值（做差分）。最常用。
ListState<T>一个列表缓存一批待处理元素。别无限 append（要配 TTL/触发清理）。
MapState<K,V>一个 map去重、按子键聚合。RocksDB 下按需读单个 key，比把大 Map 塞进 ValueState 高效得多。
Reducing/AggregatingState增量聚合结果边来边聚合（sum/avg），只存结果不存原始值，省内存。
高频坑：用 ValueState<Map> 存一个大 Map——每次访问都要反序列化整个 Map，状态一大就慢爆。正解是用 MapState，RocksDB 后端能只读写你访问的那个子 key。这正是 D01「按需读 vs 全量读」直觉在状态层的复现。

3.2 增量 Checkpoint 与状态可伸缩

增量 Checkpoint（Incremental Checkpoint）：RocksDB 后端独有。普通 checkpoint 每次都把全量状态上传到 DFS（对象存储/HDFS），TB 级状态会让 checkpoint 又慢又占带宽。增量 checkpoint 利用 RocksDB 的 LSM 特性——只上传自上次以来新增/变更的 SST 文件，把每次 checkpoint 的数据量从「全量」降到「增量」。大状态生产作业几乎必开。代价：恢复时要合并多个增量、且文件清理更复杂。
Changelog State Backend（Flink 1.15+）：为「让 checkpoint 更频繁、更稳定」而生。它把状态变更先写一份 changelog 日志（持续上传），checkpoint 时只需确认日志位点，从而大幅缩短 checkpoint 时长、削平毛刺。代价是恢复要回放日志、且多一份写放大。延迟敏感、要求 checkpoint 间隔极短时考虑。
状态可伸缩与 Key Group：把作业并行度从 4 改到 8（rescale），每个 key 的状态怎么跟着重新分配？Flink 把 keyspace 预先切成固定数量的 Key Group（键组）（= 最大并行度 maxParallelism），状态按 key group 为单位分配给各 subtask。扩缩容时以 key group 为粒度搬迁，而非逐 key——这就是 Flink 能「带着 TB 状态平滑扩缩容」的关键。所以 maxParallelism 一旦设定就不能改（改了 key group 划分就变、老状态无法对应），上线前要预留好。
把 ch4 的 Checkpoint/Savepoint 连起来看：Checkpoint（自动、增量、容错恢复）+ Savepoint（手动、全量对齐、用于升级/迁移/改并行度）+ Key Group（让状态可重分配）= Flink「有状态作业也能像无状态服务一样运维」的完整拼图。这是它区别于「自己用 Redis 存状态」的根本价值。
04

Checkpoint 与 Exactly-once

有状态 + 长时间运行 = 一定会遇到故障 · Flink 怎么做到"崩了重启也不丢不重"

一个流作业可能跑几个月。机器宕机、网络抖动是常态。Flink 靠 Checkpoint（检查点）定期给整个作业的状态拍"一致性快照"，故障后从最近的快照恢复，配合可重放的数据源（如 Kafka），实现 Exactly-once（精确一次）。

4.1 EOS 的边界条件（别神化）

Exactly-once（精确一次）：即使中途故障重放，每条数据对状态/结果的影响有且只有一次（不丢不重）。注意：这通常指"状态的精确一次"。要做到端到端精确一次（连下游输出也不重复），还需要可重放的源（Kafka offset）+ 支持事务/幂等的 sink（两阶段提交 2PC）+ 正确的提交协议。任何一环缺失，端到端语义通常会退化为 at-least-once。
很多外部系统并不支持真正事务提交（或跨系统事务成本过高），实践中常见折中是at-least-once + 业务幂等去重。因此要先明确你追求的是"状态 EOS"还是"端到端 EOS"。
Chandy-Lamport 异步屏障快照（Asynchronous Barrier Snapshotting）：Flink Checkpoint 的核心算法。做法：往数据流里周期性插入特殊标记 Barrier（屏障），它随数据一起流过每个算子。算子一收到 barrier，就把自己当前的状态异步快照下来，然后把 barrier 传给下游。当 barrier 流过整个拓扑，一次全局一致的快照就完成了——全程几乎不停止数据处理，这是它的精妙之处。
Barrier 像流水线上的"分隔板"，把"屏障前"和"屏障后"的数据隔开。各算子在 barrier 到达时记录自己的状态，无需全局停顿。恢复时所有算子回滚到同一个 checkpoint，源重放 barrier 之后的数据。
Checkpoint vs Savepoint：两者机制相同，用途不同。Checkpoint 是 Flink 自动周期触发、用于故障自动恢复的快照，作业停了通常清掉。Savepoint 是你手动触发、用于运维的快照——升级代码、迁移集群、A/B 切换前打一个，之后从它恢复，状态无损。

4.2 Checkpoint 调参速查（生产常用）

参数作用建议
execution.checkpointing.interval两次 checkpoint 间隔低延迟链路通常 10s~60s，先从 30s 起步
execution.checkpointing.timeout单次 checkpoint 超时设置为稳定耗时的 3~5 倍，避免误杀
execution.checkpointing.min-pause两次 checkpoint 最小间隔防止连续 checkpoint 压垮 IO
execution.checkpointing.max-concurrent-checkpoints并发 checkpoint 数先用 1，确认稳定后再评估并发
execution.checkpointing.unaligned反压场景加速 barrier 对齐高反压链路可考虑开启，注意状态体积上升
线上排障顺序可固定为：先看 checkpoint duration / timeout → 再看 backpressure → 再看状态大小与 RocksDB compaction。这比盲目扩容更高效。
05

反压 · 流式 Join · 维表关联

流系统特有的几个工程问题 · 老兵会觉得似曾相识（流控、背压）

5.1 反压（Backpressure）

反压像高速公路的连环刹车：前方堵了（下游算子处理不过来），后车只能跟着减速，一路往后传，最终连入口（数据源）都被迫限流。这是流系统的自我保护——避免下游被压垮、数据在内存里堆爆。
反压（Backpressure）：当某算子处理速度跟不上上游产出速度时，它通过阻塞/限流把"我忙不过来"的信号逐级往上游传递，迫使上游降速，直到数据源。Flink 基于有界缓冲 + 信用（credit-based）流控实现。排查时看 Flink UI 的反压指标：哪个算子标红/High，瓶颈就在它（或它下游）。常见原因：数据倾斜、某算子逻辑重、外部 IO 慢、资源不足。
老兵视角：这就是 TCP 流控 / 生产者-消费者背压在分布式流引擎里的翻版。理念完全一致——用反向的压力信号匹配生产与消费速率，避免无界堆积。

5.2 流式 Join 与维表关联

批处理里 join 很自然，但流是无界的——你不能"等两条流都到齐"再 join。Flink 提供几种流上 join：

类型做法场景
Window Join在同一个窗口内的两条流做 join同时段的两类事件关联
Interval Join流 A 的事件 join 流 B 在 [-下界, +上界] 时间区间内的事件下单与支付在 N 分钟内配对
维表 Join（Lookup）流去关联一张外部维表（DB/缓存）给订单流补上商品/用户属性
维表关联（Dimension / Lookup Join）：用流里的 key 去查一张相对静态的维度表（如商品表），补全属性——正是 D02 维度建模在实时侧的落地。难点是维表也会变：实现上常用缓存 + 定期刷新，或用 Temporal Table（时态表）关联到"事件发生时刻"的维度版本（呼应 D02 的 SCD2！时间点正确性在流里同样关键）。
06

CEP · Flink SQL · 流批一体

从底层 API 到声明式 SQL · Flink 的上层能力与统一愿景

6.1 CEP：复杂事件处理

CEP（Complex Event Processing，复杂事件处理）：在事件流中匹配特定的模式序列。比如「同一用户 5 分钟内连续 3 次登录失败 → 触发风控告警」「下单后 30 分钟未支付 → 发提醒」。Flink CEP 让你用类正则的方式声明这种"事件模式"，引擎自动在流上做模式匹配。是风控、监控、营销场景的利器。

6.2 Flink SQL：用 SQL 写流

和 Spark 一样，Flink 也提供声明式的 Table API / SQL，让你用 SQL 表达流处理——窗口、join、聚合都能写。底层同样有优化器把 SQL 编译成高效的流算子。

-- Flink SQL：每 5 分钟滚动窗口，按商品统计销量（事件时间 + Watermark）
SELECT
    window_start,
    product_id,
    SUM(amount) AS sales
FROM TABLE(
    TUMBLE(TABLE orders, DESCRIPTOR(event_time), INTERVAL '5' MINUTES))
GROUP BY window_start, product_id;
注意这里的 event_time 和 TUMBLE 窗口——前几章的概念在 SQL 里直接成了语法。声明式让你专注"算什么"，把"怎么高效流式执行"交给引擎。

6.3 流批一体（Unified Batch & Stream）

流批一体：用同一套 API、同一段代码，既能处理无界流（实时），也能处理有界流（批）。Flink 的理念是「批是有界的流」，所以天然统一；Spark 也从批侧努力统一。好处：业务逻辑写一遍，离线回溯和实时计算复用，避免"Lambda 架构"里实时/离线两套代码对不齐的痛（D08 实时数仓会深入 Lambda vs Kappa）。
本篇把 Flink 的"流优先"世界观讲透了。它和 D03 的 Spark 不是对立，而是互补的两种引擎哲学：批吞吐用 Spark，实时低延迟用 Flink。D08 会把 Kafka + Flink + OLAP 拼成一个完整的实时数仓。

一页纸总结

① 流优先

Flink 把"批看成有界流"，逐条处理、毫秒延迟；Spark 是微批。各擅胜场。

② 事件时间 + Watermark

用事件时间算"正确"结果；Watermark 是时间进度条，决定窗口何时触发、迟到怎么处理。

③ 窗口

把无界流切成有限桶：滚动（不重叠）/滑动（重叠）/会话（按间隔）。

④ State

有状态计算的"记账本"；Keyed/Operator State；大状态用 RocksDB backend；记得设 TTL。

⑤ Checkpoint + Exactly-once

Barrier 异步快照（Chandy-Lamport）；配可重放源 + 事务 sink 达端到端精确一次。

⑥ 反压 / Join

反压逐级传导保护系统；流上 join 靠窗口/区间/维表（时态表）。

面试自测（点击展开）

基础事件时间和处理时间有什么区别？为什么流处理偏爱事件时间？
事件时间事件实际发生的时间（数据自带时间戳）。
处理时间事件被算子处理到的机器时间，受网络/排队影响。
为何偏爱只有按事件时间，结果才正确且可复现（重放同一批数据得到同样结果）；按处理时间，结果取决于到达快慢，不可复现。
核心Watermark 是什么？它怎么解决乱序数据问题？
定义流中穿插的标记，声明"事件时间已推进到 T，比 T 早的数据基本到齐了"。
机制常用 WM = 已见最大事件时间 − 允许乱序。WM 越过窗口结束边界 → 窗口触发计算。
迟到WM 之后才到的是迟到数据，默认丢弃；可配 allowed lateness 宽限或侧输出收集。
权衡WM 设大→更准更慢；设小→更快但可能漏。是延迟 vs 准确性的折中。
高频Flink 怎么做到 Exactly-once？是端到端的吗？
机制Checkpoint 用 Chandy-Lamport 屏障快照定期保存一致状态；故障后所有算子回滚到同一 checkpoint。
范围默认是状态的精确一次。
端到端还需可重放的源（Kafka offset）+ 事务/幂等 sink（两阶段提交）才能保证连输出也不重复。
实战Flink 作业延迟变高/吞吐下降，怎么排查反压？
看 UIFlink UI 的 Backpressure 指标，找标红/High 的算子——瓶颈在它或它下游。
常见因数据倾斜（某 key 热点）、算子逻辑重、外部 IO（维表/sink）慢、资源不足。
解决打散热点 key、增加并行度、异步 IO 访问外部、优化序列化、扩资源。
对比Flink 和 Spark Streaming 的本质区别？什么时候选谁？
模型Spark=微批（攒批再算，秒级延迟）；Flink=真流（逐条处理，毫秒延迟）。
时间/状态Flink 原生强大（事件时间/Watermark/State/Exactly-once）；Spark 后期补强。
选型大吞吐离线批 + 准实时→Spark；低延迟实时、复杂事件、实时数仓→Flink。很多团队两者并存。

术语表速查 · Glossary

本篇核心名词一次收齐：

① 模型与时间

真流 vs 微批 True Streaming / Micro-batchFlink 逐条处理（毫秒延迟）；Spark 把流切成小批（秒级延迟）。「批是有界的流」是 Flink 信条。
事件时间 Event Time事件实际发生的时间（数据自带）。唯一能算出正确且可复现结果的时间。
处理时间 Processing Time事件被算子处理到的机器时间。简单、低延迟，但结果不可复现。
Watermark 水位线流中标记，声明"事件时间已推进到 T，更早的数据基本到齐"。越过窗口边界即触发计算。
迟到数据 Late DataWatermark 之后才到的数据。默认丢弃，可配宽限期（allowed lateness）或侧输出收集。

② 窗口

窗口 Window把无界流切成有限片来聚合的机制。
滚动窗口 Tumbling固定大小、首尾相接不重叠，每条数据只属一个窗口。最常用。
滑动窗口 Sliding固定大小、按小步长滑动会重叠，一条数据可属多个窗口。
会话窗口 Session无固定大小，按活动间隔（gap）切分，适合用户会话分析。
触发器 Trigger决定窗口何时输出结果（默认 Watermark 越过窗口末尾时触发）。

③ 状态与容错

State 状态流计算"记住的中间结果"。Keyed State（按 key）/ Operator State（按算子）。
State Backend状态存储方式：HashMap（堆内存，快）/ RocksDB（本地 LSM，可存 TB 级大状态）。
State TTL给状态设过期时间，防止无限增长导致 OOM / 磁盘爆。
Checkpoint 检查点Flink 自动周期触发的一致性状态快照，用于故障自动恢复。
Savepoint 保存点手动触发的快照，用于升级/迁移/回滚等运维操作。
增量 CheckpointRocksDB 后端独有：只上传自上次以来变更的 SST 文件，大状态作业大幅降低 checkpoint 数据量与时长。
Key Group 键组keyspace 预切的固定分片（= maxParallelism），状态按它分配，扩缩容以键组为粒度搬迁。maxParallelism 设定后不可改。
Barrier 屏障插入数据流的标记，流过算子时触发其状态快照（Chandy-Lamport 异步屏障快照）。
Exactly-once 精确一次故障重放后每条数据对结果影响有且仅一次。端到端需可重放源 + 事务 sink（2PC）。

④ 工程与上层

反压 Backpressure下游处理不过来时逐级向上游传递降速信号，保护系统不被压垮。看 UI 定位瓶颈算子。
维表 / 时态表 Join Lookup / Temporal流关联外部维表补属性；时态表关联到"事件发生时刻"的维度版本（呼应 SCD2）。
CEP 复杂事件处理在流中匹配事件模式序列（如 5 分钟内连续 3 次失败），用于风控/监控。
Flink SQL / Table API声明式写流处理，窗口/join/聚合皆可，底层优化器编译成流算子。
流批一体同一套代码处理有界（批）与无界（流）数据，避免实时/离线两套逻辑对不齐。
