大数据开发实战：本地全栈一键跑通

前面 13 篇讲透了原理，但找工作和实战开发，最终要能「跑起来、写代码、排得了错」。这一篇是纯动手篇：用一份 docker-compose 在你自己的笔记本上拉起一套迷你但完整的现代数据栈——Kafka（消息）+ MinIO（对象存储）+ Iceberg（湖仓表，D11）+ Flink/Spark（计算，D03/D04）+ Trino（查询，D07），然后端到端跑通一条「造数据 → 流式入湖 → SQL 查询 → 时间旅行」的链路。把前面所有概念在本地接成一个能动的整体。代码可直接抄。

为什么一定要自己动手搭一套

看懂 ≠ 会用 · 面试官一问「你实际跑过吗」就露馅

大数据最大的学习门槛，不是概念难，而是组件多、要互相连、环境难搭。很多人卡在「装不上 / 连不通」，于是永远停留在「看过博客」的层面。容器化（Docker Compose）彻底改变了这件事：一条命令，几分钟拉起一整套数据栈，让你能在本地反复试验、弄坏了重来。

这就像学做菜——光看菜谱（前 13 篇）永远学不会火候。本篇给你一个配齐了灶台、锅具、半成品食材的迷你厨房（docker-compose），你只管照着步骤把一道「实时数据入湖」的菜做出来、尝到味道。做坏了？docker compose down -v 重置，再来一次，零成本。
前置要求：装好 Docker Desktop / Docker Engine + Docker Compose，给它分配至少 8GB 内存（全栈跑起来吃内存）。本篇命令在 Linux/macOS/WSL2 下验证；纯 Windows PowerShell 把续行符 \ 换成反引号 ` 即可。
这条链路串起了：D06（Kafka）→ D04/D03（Flink/Spark 计算）→ D11（Iceberg 湖仓表 on MinIO）→ D07（Trino 查询）。跑通它，等于把半个系列在本地复现了一遍。
01

一键起本地全栈

一份 compose · 五个角色 · 几分钟就绪

1.1 我们要搭的迷你数据栈

五个角色：Producer 造数据 → Kafka 缓冲 → Flink/Spark 计算入湖 → 数据落 MinIO(S3) + Iceberg 元数据 → Trino 查询。一条最小可用的现代数据栈。

1.2 docker-compose.yml（直接抄）

新建一个目录，放入下面的 docker-compose.yml。它定义了 Kafka、MinIO、Iceberg REST Catalog、Spark（带 Iceberg 包）、Trino 五个服务：

# docker-compose.yml —— 本地迷你数据栈（学习用，非生产配置）
services:
  kafka:                                # 单节点 KRaft 模式，免 ZooKeeper
    image: bitnami/kafka:3.7
    ports: ["9092:9092"]
    environment:
      KAFKA_CFG_NODE_ID: 0
      KAFKA_CFG_PROCESS_ROLES: controller,broker
      KAFKA_CFG_LISTENERS: PLAINTEXT://:9092,CONTROLLER://:9093
      KAFKA_CFG_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092
      KAFKA_CFG_CONTROLLER_QUORUM_VOTERS: 0@kafka:9093
      KAFKA_CFG_CONTROLLER_LISTENER_NAMES: CONTROLLER
      ALLOW_PLAINTEXT_LISTENER: "yes"

  minio:                                # 对象存储（S3 兼容）
    image: minio/minio
    ports: ["9000:9000", "9001:9001"]   # 9001 是控制台
    environment:
      MINIO_ROOT_USER: admin
      MINIO_ROOT_PASSWORD: password
    command: server /data --console-address ":9001"

  mc:                                   # 启动后自动建 bucket
    image: minio/mc
    depends_on: [minio]
    entrypoint: >
      /bin/sh -c "
      until mc alias set m http://minio:9000 admin password; do sleep 2; done;
      mc mb -p m/warehouse; mc anonymous set public m/warehouse; "

  iceberg-rest:                         # Iceberg REST Catalog（保管表指针，D11）
    image: tabulario/iceberg-rest
    ports: ["8181:8181"]
    depends_on: [minio]
    environment:
      CATALOG_WAREHOUSE: s3://warehouse/
      CATALOG_IO__IMPL: org.apache.iceberg.aws.s3.S3FileIO
      CATALOG_S3_ENDPOINT: http://minio:9000
      AWS_ACCESS_KEY_ID: admin
      AWS_SECRET_ACCESS_KEY: password
      AWS_REGION: us-east-1

  spark:                                # Spark + Iceberg，用来跑入湖与查询
    image: tabulario/spark-iceberg
    depends_on: [iceberg-rest, minio]
    ports: ["8888:8888", "8080:8080"]   # 8888 Jupyter, 8080 Spark UI
    environment:
      AWS_ACCESS_KEY_ID: admin
      AWS_SECRET_ACCESS_KEY: password
      AWS_REGION: us-east-1

  trino:                                # 交互式 SQL 查询引擎（D07）
    image: trinodb/trino:latest
    ports: ["8088:8080"]
    depends_on: [iceberg-rest]
# 启动整套栈（首次会拉镜像，耐心等几分钟）
docker compose up -d

# 看各服务状态；都 healthy/running 即可
docker compose ps

# 访问 Web UI 验证：
#   MinIO 控制台  http://localhost:9001   (admin/password)
#   Spark Jupyter http://localhost:8888
#   Spark UI      http://localhost:8080
这套栈刻意用了 Iceberg REST Catalog（D11 讲的开放 catalog 趋势）+ MinIO 当 S3（D01 的对象存储）。它麻雀虽小，五脏俱全——和云上真实的「Iceberg on S3 + Trino/Spark」架构同构，只是缩到了一台机器。学会它，云上那套你也就懂了。
02

① 造数据：源源不断写进 Kafka

模拟业务系统产生订单事件流

先建 topic，再用一个 Python 脚本持续往 Kafka 灌订单事件（模拟 D06 的「流式摄取」）：

# 建 topic（进 kafka 容器执行）
docker compose exec kafka kafka-topics.sh --create \
  --topic orders --bootstrap-server localhost:9092 \
  --partitions 3 --replication-factor 1
# producer.py —— 每秒造几条订单事件丢进 Kafka
import json, time, random, datetime
from kafka import KafkaProducer   # pip install kafka-python

p = KafkaProducer(bootstrap_servers="localhost:9092",
                  value_serializer=lambda v: json.dumps(v).encode())
cities = ["BJ", "SH", "GZ", "SZ"]
oid = 0
while True:
    oid += 1
    evt = {
        "order_id": oid,
        "city": random.choice(cities),
        "amount": round(random.uniform(10, 500), 2),
        "event_time": datetime.datetime.now().isoformat(),
    }
    p.send("orders", evt)
    print("sent", evt)
    time.sleep(0.3)
# 跑起来；另开一个终端用控制台消费者验证有数据进来
python producer.py
docker compose exec kafka kafka-console-consumer.sh \
  --topic orders --bootstrap-server localhost:9092 --from-beginning
能在 console-consumer 里看到 JSON 滚动，就说明采集/摄取这一段通了（呼应 D06）。Kafka 此刻就是那个「分布式可重放 commit log」——你随时能 --from-beginning 倒回去重读，这正是它解耦生产与消费的价值。
03

② 流式入湖：Kafka → Iceberg 表

两种写法任选 · Spark Structured Streaming 或 Flink SQL

现在把 Kafka 的订单流，持续写进一张 Iceberg 表（数据文件落在 MinIO）。给两种主流写法，体现 D03/D04 的差异。

写法 A：Spark Structured Streaming（D03 增强章）

进 Spark 容器开 spark-sql 或 PySpark，catalog 指向我们的 REST Catalog：

# 在 spark 容器里启动（已预置 Iceberg 包与 REST catalog 配置）
docker compose exec spark spark-sql
-- 建一张 Iceberg 表（隐藏分区按天，D11）
CREATE TABLE demo.db.orders (
  order_id BIGINT, city STRING, amount DOUBLE, event_time TIMESTAMP)
USING iceberg
PARTITIONED BY (days(event_time));
# PySpark：从 Kafka 读流 → 解析 JSON → 追加写 Iceberg（微批）
from pyspark.sql.functions import from_json, col
from pyspark.sql.types import *

schema = "order_id LONG, city STRING, amount DOUBLE, event_time TIMESTAMP"
df = (spark.readStream.format("kafka")
        .option("kafka.bootstrap.servers", "kafka:9092")
        .option("subscribe", "orders")
        .load()
        .selectExpr("CAST(value AS STRING) AS j")
        .select(from_json("j", schema).alias("o")).select("o.*"))

(df.writeStream
   .format("iceberg")
   .outputMode("append")
   .option("checkpointLocation", "s3://warehouse/ckpt/orders")
   .toTable("demo.db.orders"))   # 持续把订单追加进 Iceberg 表

写法 B：Flink SQL（D04 风格，纯 SQL 也能干）

-- Flink SQL Client 里：定义 Kafka 源表 + Iceberg 目标表，一条 INSERT 起流作业
CREATE TABLE k_orders (
  order_id BIGINT, city STRING, amount DOUBLE, event_time TIMESTAMP(3))
WITH ('connector'='kafka', 'topic'='orders',
      'properties.bootstrap.servers'='kafka:9092',
      'format'='json', 'scan.startup.mode'='earliest-offset');

INSERT INTO iceberg_catalog.db.orders SELECT * FROM k_orders;  -- 流式入湖
checkpointLocation 是流作业的命根子（呼应 D04 Checkpoint）：它记录「Kafka 读到哪个 offset、写了哪些文件」。作业重启从这里恢复，保证 exactly-once、不重不漏。删了它，重启就会重复消费或丢数据。本地试验时如果想重头来，记得连 checkpoint 一起清。
观察现象：作业跑起来后，去 MinIO 控制台（:9001）翻 warehouse/ 桶，你会看到 Iceberg 写出的 data/（Parquet 数据文件） 和 metadata/（manifest、metadata.json）——这正是 D11 讲的三层元数据树落在真实存储上的样子。每攒一个微批就提交一个新快照。
04

③ 查询、聚合与时间旅行

用 Trino / Spark SQL 把湖仓表当数仓查

数据进湖了，用 Trino（D07 的查询引擎）做交互式查询——它直接读 Iceberg 表：

# 进 Trino CLI
docker compose exec trino trino
-- 实时看各城市 GMV（数据还在持续流入，多查几次数字在涨）
SELECT city, count(*) AS orders, round(sum(amount), 2) AS gmv
FROM iceberg.db.orders
GROUP BY city ORDER BY gmv DESC;
-- 时间旅行：表有哪些快照？（D11 的杀手锏在本地复现）
SELECT * FROM iceberg.db."orders$snapshots" ORDER BY committed_at;

-- 查 5 分钟前那一版数据（按时间旅行）
SELECT count(*) FROM iceberg.db.orders
FOR TIMESTAMP AS OF (current_timestamp - interval '5' minute);
-- 维护动作：合并小文件（D11 的 compaction，流式写最需要）
ALTER TABLE iceberg.db.orders EXECUTE optimize;
交互演示 · 端到端链路跟随
点「下一步」，跟随一条订单从产生到被查询的全过程。
跑到这一步，你已经在本地独立复现了一条「实时数仓」的最小骨架（D08）：采集(Kafka) → 计算入湖(Flink/Spark) → 湖仓表(Iceberg/MinIO) → 查询(Trino) → 时间旅行。这套链路的每个环节，都对应着前面某一篇的原理。
05

常见坑与排查清单

本地搭栈最容易卡住的几处 · 也是排错思维的练习

现象多半原因怎么查 / 解
容器起不来 / 一直重启内存不够、端口被占docker compose logs <svc> 看报错；给 Docker 加内存；换端口
连不上 Kafka容器内外地址不同容器间用 kafka:9092，宿主机脚本用 localhost:9092（注意 advertised.listeners）
写 Iceberg 报 S3 403/连不上MinIO 凭证/endpoint 没配对核对 AWS_* 与 S3_ENDPOINT=http://minio:9000；bucket 是否已建
Trino 查不到表catalog 没指向同一个 REST确认 Trino 的 iceberg catalog 配的是 http://iceberg-rest:8181
越跑越慢 / 小文件爆炸流式写没做 compaction定期 EXECUTE optimize + expire_snapshots（D11 运维三件套）
重启后重复消费checkpoint 丢了/没配确认 checkpointLocation 持久且唯一
排错通用心法（呼应 perf-debug/ 板块）：① 先看日志（logs）定位是哪一环；② 分段验证（Kafka 有数据吗？MinIO 有文件吗？catalog 有表吗？）；③ 一次只改一个变量。能独立排掉这些坑，比「跑通一次」更值钱——面试时讲一个你踩过并解决的坑，远胜背概念。
06

小结 · 进阶 · 写进简历

把这套动手经历转化成能力与谈资

一页纸总结

① 一键全栈

docker-compose 拉起 Kafka+MinIO+Iceberg+Spark/Flink+Trino，本地复现现代数据栈。

② 端到端链路

造数据→Kafka→流式入湖(Iceberg/MinIO)→Trino 查询→时间旅行，每环对应一篇原理。

③ 两种入湖写法

Spark Structured Streaming（微批）与 Flink SQL（流），按场景任选。

④ 排错心法

看日志、分段验证、一次改一个变量。能排坑比能跑通更值钱。

进阶练习（自己加戏）

  • 加 CDC：起一个 MySQL + Debezium/Flink CDC，把库变更入湖（练 D06）。
  • 加 OLAP：把 StarRocks/Doris 接进来，对比 Trino 直查湖 vs 导入 MPP 的性能（练 D07）。
  • 做 Upsert：用 MERGE INTO 把 CDC 变更合并进 Iceberg 表，观察 COW/MOR（练 D11）。
  • 加 编排：用 Airflow 把「入湖→compaction→聚合」串成定时 DAG（练 D06）。
  • 接 服务层：把聚合结果写进 Redis，模拟实时大屏读取（练 D12）。
怎么写进简历 / 面试讲：与其写「了解 Kafka/Flink/Iceberg」，不如写「独立搭建并跑通基于 Kafka + Flink + Iceberg(on S3) + Trino 的实时入湖链路，实现流式摄取、湖仓表存储、时间旅行查询与小文件治理」。再准备 1–2 个你踩过的坑（如小文件、checkpoint、热点）讲排查过程——这就是把「学习」变成「经历」，面试含金量天差地别。
延伸：本篇是把 D01/D04/D06/D07/D08/D11 的概念在本地接成整体的「实操总集」。理解每一环背后的原理，请回到对应专篇；想做端到端架构设计，看 D10 Capstone 与系统设计面试专题。
