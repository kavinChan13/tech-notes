为什么要「建模」——从能存到好用

存得下 ≠ 查得快、用得对、说得清 · 建模是数据工程的「秩序」

D01 我们把数据可靠地存了下来（HDFS / 对象存储 / Parquet）。但「能存」离「好用」还差得远——原始数据往往杂乱、重复、口径不一。数据建模就是给数据建立秩序：定义有哪些表、表之间什么关系、每张表什么粒度、指标怎么算。建模做得好，分析师写 SQL 又快又不会算错；做得烂，同一个「销售额」十个人能算出十个数。

数据建模像开一家超市怎么布局货架：仓库里堆着货（原始数据）不等于顾客好买。你得分区（生鲜 / 日用 / 冷冻）、立招牌、定动线，让人「一眼找到、不会拿错」。建模就是给数据设计这套「货架与动线」——它不增加数据本身，却决定了所有人用数据的效率与正确性。
数据建模（Data Modeling）：用一套结构化方法，把杂乱的业务数据组织成「易理解、易查询、口径一致」的表结构。在分析型（OLAP）场景，最主流的方法是 Kimball 提出的维度建模（Dimensional Modeling）——本篇的主角。

01

维度建模核心：事实表 & 维度表

把业务拆成「发生了什么（事实）」+「从哪些角度看（维度）」 · Kimball 的全部精髓

维度建模的世界观极其简洁：任何业务分析，都可以拆成两类东西——事实（Fact）：业务过程中「发生了、且可度量」的事件；维度（Dimension）：你想从哪些「角度」去观察这些事件。掌握这一刀切分，数仓建模就入门了。

1.1 事实表 vs 维度表

想象超市收银：每张收银小票记录了「谁、在哪家店、哪天、买了什么、多少钱」——这就是事实表，又长又多（每笔交易一行）。而「商品目录、门店名录、日历、会员名册」是用来解释小票里那些代号的——这就是维度表，相对又宽又稳定。事实表回答「多少」，维度表回答「谁 / 何时 / 何地 / 什么」。

事实表 Fact Table

  • 存度量值（measure）：金额、数量、时长等可加总的数字
  • 存一堆外键指向各维度表
  • 形态：窄而长（列少、行极多，可达十亿级）
  • 随业务持续增长，是数仓里最大的表

维度表 Dimension Table

  • 存描述性属性（attribute）：名称、类别、地区、等级等
  • 提供分析的「切入角度」与「过滤 / 分组」依据
  • 形态：宽而短（列多、行相对少）
  • 变化慢，但会变（→ 第 3 章 SCD）
事实表居中、维度表环绕，连线就是「外键→主键」。形状像星星，故名「星型模型（Star Schema）」。

1.2 粒度、度量、代理键——三个必懂的词

粒度（Grain）：事实表里「一行代表什么」。是「一笔订单」还是「订单里的一个商品行」？粒度是建模的第一决策，且越细越好（细粒度可以汇总成粗粒度，反之不行）。一张事实表必须保持单一、明确的粒度，混粒度是新手最常见的事故。
度量（Measure）：事实表里那些可以做聚合（SUM/AVG/COUNT）的数值，如销售额、数量。按可加性分三种：可加（金额，任意维度都能加）、半可加（库存余额，可跨商品加但不能跨时间加）、不可加（比率 / 单价，只能重算不能直接加）。
代理键（Surrogate Key）：维度表的主键，用一个无业务含义的自增整数（如 1,2,3）来当，而不直接用业务的自然键（Natural Key）（如身份证号、SKU 编码）。好处：① join 用小整数更快；② 业务键变了也不影响；③ 支持 SCD 留存历史版本（同一客户的不同版本用不同代理键）——第 3 章会看到它的威力。

1.3 三类事实表

类型一行代表典型场景特点
事务事实表
Transaction		一个业务事件发生的瞬间每一笔下单 / 支付 / 点击最常见、最细粒度、只追加

02

星型 vs 雪花 vs 宽表

同一份事实 + 维度，三种组织方式 · 本质是「冗余 / 空间」与「查询性能 / 复杂度」的权衡

把事实表和维度表连起来，有三种典型风格。它们的差异，老兵会非常熟悉——这就是范式化 vs 反范式化之争，用空间冗余换查询性能，和你在数据库 / 缓存里做的取舍同源。

星型把维度「摊平」（如 product 里直接带 category/brand 名）；雪花把维度再拆成 category、brand 子表（规范化）。
风格做法查询性能空间 / 冗余适用
星型 Star维度反范式、直接连事实快（join 少）有冗余绝大多数数仓首选
雪花 Snowflake维度再规范化拆多层较慢（join 多）省空间维度极大 / 强一致需求
宽表 OBT事实 + 所有维度摊成一张大表最快（零 join）冗余最大固定报表 / 列存 OLAP 引擎
老兵视角：星型 = 适度反范式，是「读多写少 + 列存压缩能吃掉冗余」下的理性选择。宽表（One Big Table）在 ClickHouse / StarRocks 这类列存引擎里尤其流行——反正列存压缩强、又没有 join 开销，干脆一张大宽表打天下（D07 会再遇到）。这正是「用存储换计算」在数仓层的体现。
不要无脑追求「规范化好看」。数仓的第一目标是查询快、易理解，不是消除冗余。雪花模型在实践中常因 join 过多、分析师难懂而被劝退——除非维度真的大到冗余不可接受。
03

缓慢变化维 SCD 与拉链表

客户搬了家、产品改了名——维度会变，历史怎么办？这是建模里最经典的一道题

维度属性不是一成不变的：客户会搬家、商品会调类目、员工会升职。这些变化慢、但一变就影响历史分析的维度，叫缓慢变化维（Slowly Changing Dimension, SCD）。怎么处理它，直接决定了你的报表「能不能正确还原历史」。

客户「张三」2024 年住北京、2025 年搬到上海。现在要统计「2024 年北京区的销售额」——如果维度表里张三的城市被直接改成了上海，那他 2024 年在北京的消费就被错误地算进上海了。历史被篡改了。 SCD 要解决的就是：变化发生后，过去的事实还能不能挂回它当时正确的维度版本。

3.1 六种 SCD 类型

类型做法历史说明
Type 0不变（保留原值）—如「开户日期」这种本就不该变的属性
Type 1直接覆盖旧值丢失简单，但历史被改写。适合纠错类变更
Type 2新增一行版本，加生效/失效时间完整保留⭐ 最常用。能精确还原任意历史时点
Type 3加一列存「上一个值」只留前一版只关心「当前 vs 上次」时用
Type 4当前表 + 单独的历史表保留（分表）当前查询走主表，历史走附表
Type 61+2+3 混合保留 + 当前值冗余既留版本、又在每行带「当前值」便于查询
实战里 Type 1（覆盖） 和 Type 2（加版本） 覆盖了 90% 的需求。关键判断：这个属性的历史值，对分析有没有意义？ 有 → Type 2；没有（或就是纠错）→ Type 1。

3.3 拉链表——SCD2 的中国式实现

拉链表（Zipper Table）：国内数仓对 SCD Type 2 的常见叫法与实现。核心就是给每行加 start_date / end_date 两个字段，记录该版本的「有效区间」，当前有效行的 end_date 用一个极大值（如 9999-12-31）。一行的「生命周期」像拉链齿一样首尾相接，故名。它能用最省空间的方式（只在变化时新增行）保存全量历史，查询某天的状态只需 WHERE start_date <= 某天 AND end_date > 某天。
-- 查询「2024-06-01 那天」每个客户的城市（拉链表的典型用法）
SELECT id, name, city
FROM dim_customer_zipper
WHERE start_date <= '2024-06-01'
  AND end_date > '2024-06-01';   -- 命中当天有效的那个版本
拉链表是离线数仓的「时间机器」，每天的 ETL 会比对当日数据与昨日拉链：没变的不动、变了的关闭旧链 + 开新链。它和 D06 湖仓表格式（Iceberg/Delta）的 Time Travel 解决的是同一类问题——保留历史、可查任意时点——只是一个是手工建模、一个是存储层原生能力。
04

三大方法论之争：Inmon / Kimball / Data Vault

数仓该「自上而下」还是「自下而上」？理解流派，才能解释为什么不同公司数仓长得不一样

数仓建模有三种主流方法论，背后是不同的哲学。你不必站队，但要知道它们各自的取舍，才能在面试和实战中讲清「我们为什么这么建」。

Inmon · 自上而下

先建一个企业级、规范化（3NF）的中央数仓（CIF），再从中派生面向部门的数据集市。强调一致、严谨、单一事实源。前期重、见效慢，但根基稳。

Kimball · 自下而上

直接面向业务过程建维度模型（星型），用一致性维度把各数据集市串成「总线」。落地快、业务友好、最流行——本篇主要讲它。

Data Vault · 可审计

用 Hub（业务键）/ Link（关系）/ Satellite（属性历史） 三类表建模。极强的可扩展与可审计性，适合强合规、源系统多变的场景。建模偏重，查询前常需转成星型。

维度InmonKimballData Vault
方向自上而下自下而上分层（Raw Vault → Star）
核心结构3NF 规范化星型维度模型Hub / Link / Satellite
落地速度慢快中
业务友好度中高低（需转换层）
最擅长企业级一致性快速交付分析审计 / 多变源 / 合规
三者像盖楼的三种思路：Inmon 先把整栋楼的钢筋骨架（企业模型）打牢再分房间；Kimball 按一个个「能立刻入住的样板间」（业务主题）快速交付，再用统一标准（一致性维度）保证它们能拼成一栋楼；Data Vault 则像搭乐高——每块都可追溯、可拆装，适合需求老变、要留全痕迹的场景。
现实中绝大多数互联网数仓走 Kimball 维度建模 + 分层（下一章），因为它交付快、业务方看得懂。Inmon 思想体现在「企业一致性」要求里，Data Vault 多见于金融 / 强合规行业。
05

数仓分层：ODS → DWD → DWS → ADS

每天工程上真正在用的东西 · 把「原始数据 → 看板」拆成几级流水线

5.1 为什么要分层？

如果让每张报表都直接从原始数据现算，结果就是：逻辑到处复制、口径各不相同、改一处崩一片、出了问题没法定位。分层就是给数据加工建一条标准流水线，每层只干一件事：

① 解耦与隔离

上游源系统变了，只需改最底层，上层不受影响；某层出问题，能快速定位是哪一级。

② 复用

中间层的明细 / 汇总表被多个报表共享，避免「同样的清洗逻辑写一百遍」。

③ 口径一致

指标在统一的层里算一次，所有人引用同一个结果——杜绝「十个人十个销售额」。

④ 可维护 / 可追溯

每层职责单一，新人能看懂；血缘清晰，方便排查与回溯。

5.2 经典四层 + Medallion 对照

数据自左向右逐级加工：越往右越「熟」、越贴近业务消费。右下角是 Databricks 推广的 Medallion（铜/银/金）说法，与国内 ODS/DWD/DWS 大致对应。
层全称职责Medallion
ODSOperational Data Store贴源层：原样落地源数据，几乎不加工，留作「数据备份 + 可重跑」🥉 Bronze
DWDData Warehouse Detail明细层：清洗、去重、统一编码、维度退化，建事实明细表🥈 Silver
DWSData Warehouse Summary汇总层：按主题做轻度聚合（如「每天每商品的销量」），形成可复用的宽表/指标🥇 Gold
ADSApplication Data Store应用层：面向具体报表/看板/接口的最终结果，高度定制Gold+
DIMDimension维度层：横跨各层、被反复复用的维度表（一致性维度）—
不同公司层级命名略有差异（有的把 DWD/DWS 合称 DW，有的多一层 DM 数据集市），但「贴源 → 明细 → 汇总 → 应用」的递进思想是通用的。Medallion（bronze/silver/gold）是同一思想在湖仓语境下的别名。
06

一致性维度 · 指标体系 · dbt

让多个主题「对得上账」的粘合剂 · 以及现代数据栈的转换工具

6.1 一致性维度与总线矩阵

一致性维度（Conformed Dimension）：一份在多个事实表 / 主题之间通用、口径完全一致的维度表。比如「日期维度」「商品维度」全公司只有一份，销售、库存、物流主题都用它。这样「按商品看销售」和「按商品看库存」才能正确地关联到一起。它是 Kimball 把一个个数据集市拼成统一数仓的关键粘合剂。
总线矩阵（Bus Matrix）：一张「业务过程 × 维度」的二维表，行是各业务过程（下单/支付/退货…），列是各维度（日期/商品/客户/门店…），打勾表示「该过程用到该维度」。它是数仓的顶层规划蓝图，一眼看清哪些维度被共享（应做成一致性维度）。
维度退化（Degenerate Dimension）：有些「维度」其实没有属性、只是个标识号（如订单号 order_id）。没必要单独建维度表，直接把它留在事实表里当一个字段即可。这就是「退化」。

6.3 指标体系与 dbt

指标（Metric）：业务关心的可量化结果。分原子指标（不可再拆，如「支付金额」）和派生指标（原子指标 + 维度/限定/时间，如「华北区近 7 天支付金额」）。建「指标体系」的核心，是让每个指标的口径（怎么算）唯一且被治理，避免同名不同义。
dbt（data build tool）：现代数据栈里最流行的数据转换工具。核心理念是「用 SQL + 软件工程实践管理数仓」：每个模型是一段 SELECT，dbt 负责按依赖顺序物化成表/视图，并自带测试（非空/唯一/枚举）、文档、血缘图、版本管理。它不搬数据（那是 EL 工具的事），只做 T（Transform），是 ELT 范式的代表。
-- dbt 模型示例：models/dws/dws_sales_daily.sql
-- {{ ref() }} 声明依赖，dbt 自动排好构建顺序、生成血缘
SELECT
    d.date_day,
    p.category,
    SUM(f.amount)  AS gmv,
    COUNT(DISTINCT f.order_id) AS order_cnt
FROM {{ ref('dwd_fact_order_item') }} f
JOIN {{ ref('dim_date') }}    d ON f.date_key = d.date_key
JOIN {{ ref('dim_product') }} p ON f.product_key = p.product_key
GROUP BY 1, 2
dbt 把 D02 的「分层 + 建模」从「一堆零散的调度脚本」变成了「可测试、可复用、有血缘的工程项目」——这正是把你熟悉的软件工程纪律（版本控制、测试、模块化）带进了数据建模。D09 治理篇会再展开血缘与质量。

指标语义层（Metrics Layer / Headless BI）

即使分了层、用了 dbt，仍有一个老大难：同一个指标在不同 BI 报表/SQL 里被各自重写，口径慢慢飘了——「活跃用户」在 A 报表是登录过、在 B 报表是下单过。指标语义层就是为根治这个问题而生。

指标语义层像公司唯一的"度量衡标准器"：所有人量"一米"都来对这把尺，而不是各家自带一把。指标的定义（口径、聚合方式、关联维度）只在这里写一次，BI 工具、SQL、API 全都来这里取——从此"同名同义、人人一致"。
指标语义层（Metrics Layer / Semantic Layer）：在数仓表之上、BI 之下的一层统一指标定义中枢。你只声明"指标 = 哪个度量 + 怎么聚合 + 可按哪些维度切"，它在查询时动态生成正确 SQL。因为它不带自己的可视化界面、只供上层调用，又叫 Headless BI（无头 BI）。代表：dbt Semantic Layer（MetricFlow）、Cube、LookML（Looker）、AtScale。
原子指标 vs 派生指标 vs 复合指标：原子指标=不可拆的基础度量（支付金额）；派生指标=原子指标 + 时间周期 + 修饰词/维度限定（近 7 天 + 华北 + 支付金额）；复合指标=多个指标运算而来（客单价 = 支付金额 / 订单数、支付转化率）。指标平台的核心，就是用原子指标 + 维度 + 修饰词"拼"出成百上千个派生指标，而口径只维护一份。
没有语义层有语义层（Metrics Layer）
每个报表各写一段 SQL 算指标指标定义集中一处，查询时自动生成 SQL
口径易飘、同名不同义全公司同名同义，单一事实源
改口径要改 N 个报表改一处，处处生效
BI / SQL / API 各算各的BI / SQL / API 共用同一指标服务
分层 + 指标语义层是"口径治理"的两道防线：DWS 层把明细聚合沉淀成可复用宽表（物理层面少算），语义层把指标定义统一成单一来源（逻辑层面不飘）。两者配合，才能真正终结"十个人十个 GMV"。这也是现代数据团队从"会建表"走向"会治理"的标志。

阿里 OneData：国内大厂的「数据中台」方法论

上面的语义层是「西方现代数据栈」的叫法。国内大厂（尤其阿里系）把同一套思想沉淀成了一套更体系化的方法论——OneData，这是国内大数据/数仓岗位面试的高频名词，值得专门认全。

OneData 三件套：阿里数据中台的核心方法论，目标是「全局一致、避免重复建设」。
  • OneModel（统一数据模型）：全公司一套规范的维度建模 + 分层（ODS/DWD/DWS/ADS），一致性维度 + 指标只建一次，杜绝烟囱式重复开发。就是本篇前面所有内容的工程化落地。
  • OneID（统一实体识别）：把同一个用户/商品在各业务线散落的标识（手机号、设备号、账号）打通成一个全局唯一 ID。没有 OneID，跨业务的用户画像和指标根本对不齐。
  • OneService（统一数据服务）：数据以统一的服务接口对外提供（而非各自捞表），屏蔽底层存储差异，主题式逻辑表对外（对应 D12 服务层 + 指标 API）。
指标中台 / 指标平台：把「指标语义层 + OneData 的指标治理」产品化的国内常见叫法。核心动作和前面讲的一致——原子指标 + 维度 + 修饰词拼派生指标、口径单一来源、指标血缘与负责人、变更审批。详见 interview/bigdata_system_design.html 的「设计指标平台」一题（I09 ③）。
如果说维度建模（前几章）是「怎么盖好一栋楼」，那 OneData 就是「整个城市的统一规划」：统一的图纸标准（OneModel）、统一的门牌编号系统（OneID）、统一的市政服务窗口（OneService）。它解决的不是「单张表建得好不好」，而是「几百个团队各建各的，最后整个公司数据对不齐、重复造轮子」这个组织级难题。
面试常被问「数据中台是什么 / OneData 三个 One 是什么」。一句话答法：OneModel 解决「模型与口径一致」、OneID 解决「实体打通」、OneService 解决「服务统一」，合起来根治「重复建设 + 数据不一致」。它和西方的「语义层 / 数据网格」殊途同归，只是组织语境不同。数据中台近年也有争议（重平台 vs 轻治理），但其方法论内核仍是国内数据团队的通用语言。
07

实战走查：电商订单端到端建模

把前六章串成一条线 · 从一句业务需求，走到分层落地的表

假设需求是：「我要分析电商的销售——能按日期、商品类目、客户等级、地区，灵活看 GMV、订单量、客单价。」 我们用 Kimball 四步法走一遍。

第 1 步 · 选业务过程
下单（Place Order）
先锁定一个清晰的业务过程。这里是「下单」。一个过程对应一组事实表。
第 2 步 · 定粒度
一行 = 一个订单中的一个商品行（order item）
选最细粒度。一个订单可能买多件商品，拆到「商品行」最灵活，能汇总出订单级、也能下钻到商品级。
第 3 步 · 选维度
日期 / 商品 / 客户 / 门店（地区）
围绕「从哪些角度看」选维度表，全部用一致性维度。订单号无属性 → 做成退化维度留在事实表。
第 4 步 · 选度量
数量 / 销售额 / 折扣额（皆可加）
客单价 = 销售额 / 订单量是不可加派生指标，不存事实表，查询时再算。
注意 dim_customer 标了 SCD2——客户等级/城市会变且影响历史分析，必须留版本。order_id 作为退化维度留在事实表。

落到分层里：一条数据的旅程

ODS→业务库订单表原样同步进来（ods_order、ods_order_item），不加工，留底可重跑
DWD→清洗去重、关联维度代理键、退化 order_id，生成明细事实表 dwd_fact_order_item
DWS→按「天 × 商品类目」预聚合出 dws_sales_daily（GMV、订单量），供多报表复用
ADS→面向「销售大屏」拼出 ads_gmv_dashboard，直接喂前端 / API
-- DWD：把贴源订单明细，关联各维度代理键，落成事实明细表
INSERT INTO dwd_fact_order_item
SELECT
    dd.date_key,
    dp.product_key,
    dc.cust_key,           -- 关联的是「下单当时」有效的客户版本（SCD2）
    ds.store_key,
    o.order_id,            -- 退化维度，直接留字段
    oi.qty,
    oi.qty * oi.price AS amount,
    oi.discount
FROM ods_order_item oi
JOIN ods_order o   ON oi.order_id = o.order_id
JOIN dim_date dd   ON to_date(o.created_at) = dd.date
JOIN dim_product dp ON oi.sku = dp.sku
JOIN dim_store ds  ON o.store_id = ds.store_id
JOIN dim_customer dc
     ON o.cust_id = dc.cust_id
    AND o.created_at >= dc.valid_from   -- 拉链表：挂到下单当时的客户版本
    AND o.created_at <  dc.valid_to;
注意那段 created_at BETWEEN valid_from AND valid_to——这就是第 3 章 SCD2 / 拉链表在实战里的落点：让每笔订单挂到「下单那一刻」正确的客户版本，历史分析才不会错。建模的各个概念，最终都在这条流水线里咬合到一起。
08

小结 · 选型 · 面试 · 术语表

收束成判断力 · 并接上 D03

一页纸总结

① 事实 + 维度

把业务拆成「可度量的事实」+「观察角度的维度」，是维度建模的全部起点。

② 先定粒度

一行代表什么？越细越好；一张事实表保持单一粒度。

③ 星型优先

反范式换查询性能；列存时代宽表更香。雪花仅在维度极大时用。

④ SCD2 留历史

维度会变。用代理键 + 生效/失效时间（拉链表）保住历史正确性。

⑤ 分层流水线

ODS→DWD→DWS→ADS，解耦、复用、口径一致、可维护。

⑥ 一致性维度 + dbt

一致性维度粘合多主题；dbt 把建模工程化（测试/血缘/版本）。

面试自测（点击展开）

基础事实表和维度表的区别是什么？怎么判断一个字段该放哪？
事实可度量、会聚合的数值（金额/数量）+ 指向维度的外键，又窄又长。
维度描述性属性（名称/类别/地区），用来过滤和分组，又宽又短。
判断问「这个字段是用来算的（SUM/AVG）还是用来切的（group by/where）」——算→事实度量；切→维度属性。
核心什么是缓慢变化维？SCD Type 1 和 Type 2 怎么选？
定义维度属性会随时间缓慢变化（客户搬家、商品改类目），需要决定历史值如何处理。
Type1直接覆盖，历史丢失。适合「纠错」或「历史值无分析意义」。
Type2新增版本行 + 代理键 + 生效/失效时间（拉链表），完整保留历史。适合「历史值对分析重要」。
关键问「这个属性的历史值，分析时要不要还原？」要→Type2。
辨析为什么数仓常用星型（反范式）而不是规范化的雪花？
目标数仓是读多写少的 OLAP，第一目标是查询快 + 易理解，不是消除冗余。
星型维度反范式 → join 少、分析师好懂；冗余被列存压缩吃掉，代价小。
雪花规范化省了点空间，但 join 暴增、模型难懂，通常得不偿失，仅维度极大时才考虑。
工程数仓为什么要分层（ODS/DWD/DWS/ADS）？不分层会怎样？
收益解耦隔离（源变只改底层）、复用（中间层被多报表共享）、口径一致（指标算一次）、可维护可追溯。
不分层每张报表直连原始数据现算 → 逻辑到处复制、口径各异、改一处崩一片、问题无法定位。
延伸Medallion（bronze/silver/gold）是同一思想的湖仓版别名。
综合"同一个销售额，不同报表算出来不一样"，从建模角度怎么治理？
根因口径分散：每张报表各自从明细现算，过滤/聚合规则不一致。
治理① 指标在 DWS 层统一算一次，报表只引用结果；② 建一致性维度保证关联口径相同；③ 建指标体系明确每个指标唯一口径；④ 用 dbt 做测试 + 血缘锁住一致性。

术语表速查 · Glossary

本篇核心名词一次收齐，当字典用：

① 维度建模基础

维度建模 Dimensional ModelingKimball 提出的分析型建模法：把业务拆成事实表 + 维度表（星型）。OLAP 主流方法。
事实表 Fact Table存度量值 + 维度外键的表，又窄又长（行极多）。回答「多少」。
维度表 Dimension Table存描述性属性的表，又宽又短。提供分析的切入角度与过滤分组依据。
粒度 Grain事实表一行代表什么。建模第一决策，越细越好，且必须单一明确。
度量 Measure事实表里可聚合的数值。分可加 / 半可加 / 不可加三类。
代理键 / 自然键 Surrogate / Natural Key代理键=无业务含义的自增整数主键（join 快、支持 SCD）；自然键=业务自带的键（身份证/SKU）。
退化维度 Degenerate Dim.没有属性、仅是标识号的「维度」（如 order_id），直接留在事实表里当字段。

② Schema 与变化

星型模型 Star Schema维度反范式、直接连事实。join 少、查询快，数仓首选。
雪花模型 Snowflake维度再规范化拆成多层子表。省空间但 join 多，仅维度极大时用。
宽表 One Big Table事实 + 所有维度摊成一张大表。零 join 最快、冗余最大，列存引擎里流行。
缓慢变化维 SCD变化慢但影响历史分析的维度。Type1 覆盖（丢史）、Type2 加版本（留史）最常用。
拉链表 Zipper TableSCD2 的国内实现：每行带 start/end_date 记录有效区间，省空间地保全量历史。

③ 方法论与分层

Inmon / Kimball / Data Vault三大数仓方法论：企业级 3NF 自上而下 / 维度建模自下而上 / Hub-Link-Satellite 可审计。
ODS / DWD / DWS / ADS经典数仓分层：贴源 / 明细 / 汇总 / 应用。对应 Medallion 的 bronze/silver/gold。
数据集市 Data Mart面向某个部门 / 主题的小型数仓子集。Kimball 用一致性维度把多个集市拼成统一数仓。
一致性维度 Conformed Dim.跨多个事实 / 主题通用、口径一致的维度表（全公司一份）。串联各主题的粘合剂。
总线矩阵 Bus Matrix「业务过程 × 维度」的规划蓝图，一眼看清哪些维度被共享。

④ 指标与工具

指标 Metric可量化的业务结果。原子指标（不可拆）+ 派生指标（原子 + 维度/时间/限定）。核心是口径唯一。
上卷 / 下钻 Roll-up / Drill-down上卷=减少维度看更宏观（月→年）；下钻=增加维度看更细（地区→城市）。维度建模的核心能力。
dbt data build tool用 SQL + 软件工程实践管理数仓转换的工具：模型化、测试、血缘、文档、版本。ELT 范式代表。
ETL / ELTETL=先转换再入仓；ELT=先把原始数据装进仓，再用仓内算力转换（现代云数仓主流，dbt 属此）。
指标语义层 Metrics Layer数仓表之上、BI 之下的统一指标定义中枢，查询时动态生成 SQL。又叫 Headless BI。dbt SL/Cube/LookML。
OneData阿里数据中台方法论：OneModel（统一模型口径）+ OneID（实体打通唯一 ID）+ OneService（统一数据服务），根治重复建设与数据不一致。
指标中台 / 数据中台把指标治理 + OneData 产品化的国内叫法：原子指标拼派生指标、口径单一来源、指标血缘与审批。
