NoSQL 与在线服务层

前面的篇章一路把数据存好（D01/D11）、建好模（D02）、算好（D03/D04）、查好（D07）——但分析的结果、宽表、画像最终要低延迟、高并发地服务给线上系统（App 点查用户画像、风控毫秒级查特征、推荐拉取候选）。这是大数据链路最后、也最容易被忽略的一环：在线服务层（Serving Layer）。本篇系统讲清四大类 NoSQL——宽列（HBase/Cassandra）、KV/缓存（Redis）、文档、搜索（Elasticsearch）——它们各自的内核（LSM、RowKey、倒排索引）、一致性取舍，以及在大数据架构里何时该用谁。老兵会发现：处处是 LSM-Tree、CAP、缓存局部性的影子。

为什么需要「在线服务层」

OLAP 引擎扫得快，但扛不住 C 端的高并发点查

D07 的 OLAP 引擎能亚秒扫十亿行，但它面向的是低并发、大查询（分析师、看板）。而线上系统的需求恰恰相反——极高并发（每秒几万~几十万）、极低延迟（毫秒）、每次只取一两个 key（查这个用户的画像、这个商品的库存）。拿 OLAP 引擎扛 C 端点查，就像用货运卡车送外卖：能拉，但又慢又贵还堵。

数仓/OLAP 像图书馆的总书库——藏书海量、适合做大规模检索研究，但不能让全城读者每秒几万次冲进去翻书。服务层就是书库门口的「热门书速取柜」：把高频要的结果（画像、特征、候选集）预先算好、按 key 摆好，读者刷卡（点查）即取，毫秒拿到。它不替代书库，而是书库的「对外服务窗口」。

在线服务层（Serving Layer）：把离线/实时计算产出的结果（用户画像、推荐候选、实时特征、聚合指标）写入支持高并发低延迟点查的存储，供线上业务实时读取。这正是 Lambda/Kappa 架构（D08）里「批/流层算完 → 写进 serving 层 → 应用查 serving 层」的那一层。
维度OLAP 引擎（D07）在线服务层（本篇）
查询形态大范围扫描 + 聚合按 key 点查 / 小范围
并发低（几~几十）极高（万~十万 QPS）
延迟亚秒~秒毫秒~亚毫秒
典型存储ClickHouse/StarRocksHBase/Cassandra/Redis/ES

01

NoSQL 四大家族总览

不是「不用 SQL」，而是「不止关系模型」· 按数据模型分四类

NoSQL（Not Only SQL）不是反对 SQL，而是在关系型（OLTP 行存、强事务、固定 schema）之外，提供针对特定访问模式优化的数据模型。它们普遍放松一致性 / schema 约束，换取水平扩展与高并发性能——又是一次工程权衡。

① 键值 KV（Redis / RocksDB）

最简单：key → value。读写极快，适合缓存、会话、计数、特征。无复杂查询。

② 宽列 Wide-Column（HBase / Cassandra）

BigTable 血统：按 rowkey 存超大稀疏表，列可动态、按列族存。海量、可扩展、按 key 快读写。

③ 文档 Document（MongoDB）

存 JSON 文档，schema 灵活、可嵌套。适合内容/配置/半结构化数据，查询比 KV 丰富。

④ 搜索 / 倒排（Elasticsearch）

倒排索引：全文检索、多维过滤、日志分析。近实时、聚合强，但非强一致事务库。

还有图数据库（Neo4j）（关系遍历）、时序数据库（InfluxDB/TDengine）（监控指标）等专用品类。本篇聚焦大数据服务层最常见的宽列、KV/缓存、搜索三类——它们在数据平台里出场率最高。
选型第一性原理：NoSQL 是「按查询模式设计数据模型」——先想清「线上要怎么查」（按什么 key、要不要全文搜、要不要范围扫），再反推该用哪类、rowkey/索引怎么设计。这与关系型「先规范化建模、再随意查」的思路相反。
02

宽列存储：HBase 与 Cassandra

BigTable 的两个开源回声 · 海量数据的按 key 读写主力

宽列存储是 D01 提到的 Google BigTable 的开源实现，专为「千亿行、按 rowkey 快速读写、可线性扩展」而生。它们是用户画像、实时特征、消息/订单海量明细等场景的常客。理解它们，关键是看清底层的 LSM-Tree 和 RowKey 设计。

2.1 LSM-Tree：为什么写这么快

HBase/Cassandra（以及 RocksDB、ClickHouse MergeTree）底层都是 LSM-Tree（Log-Structured Merge Tree）。它把「随机写」变成「顺序写」——正是老兵熟悉的「顺序 IO ≫ 随机 IO」的极致应用。

LSM 写路径：WAL 保证不丢 + MemTable 攒批 + 顺序刷成 SSTable + 后台 compaction。读靠 Bloom Filter 跳过无关文件。与 D11 的湖仓 compaction、D07 的 MergeTree 同源。
LSM-Tree：写先进内存的 MemTable（同时写 WAL 防丢），满了顺序刷成不可变的 SSTable 文件；后台 compaction 把多个 SSTable 合并去重。把随机写变顺序写，写吞吐极高——代价是读要查多层（读放大）、compaction 占资源。与 B+Tree（读快、随机写慢）正好相反，所以写密集选 LSM、读密集/点查多选 B+Tree。

2.2 RowKey 设计：宽列存储的「命门」

宽列存储里，RowKey 设计几乎决定一切——它既是唯一索引、又决定数据物理分布（按 rowkey 排序分片到不同 Region/节点）。设计不好，海量写入会全砸在一个节点上（热点），其他节点闲着——这是宽列存储头号事故。

交互演示 · RowKey 设计与写入热点
选 RowKey 策略，点「灌入写入」，看 8 个 Region（分片）的负载分布。

反模式：单调递增 key

用时间戳/自增 ID 当 rowkey，新写入永远落在「最大」那个 Region → 写入热点，单点被打爆，扩容也救不了。

解法：打散

加盐（salt 前缀）、哈希、反转（如手机号反转）把 key 均匀散到各 Region。代价：牺牲了「按时间范围顺序扫」的能力——又是取舍。

RowKey 设计要同时满足两个常冲突的目标：① 写入均匀（防热点）② 读取高效（相关数据聚在一起、能范围扫）。比如「最近订单」想范围扫就要时间在前，但时间在前又会热点。常见折中：散列前缀 + 业务维度 + 时间（如 hash(user)_user_ts），让同一用户数据聚簇、又整体打散。没有通用最优 rowkey，只有匹配查询模式的设计。

2.3 HBase vs Cassandra：CP 还是 AP

维度HBaseCassandra
架构主从（依赖 HDFS + ZK）去中心化对等（无主）
CAP 取向CP（强一致，分区时可能不可用）AP（高可用，可调一致性）
一致性强一致（单行）可调（QUORUM / ONE …）
运维重（要 HDFS+ZK 整套）相对独立、易扩
适合强一致、与 Hadoop 生态集成、随机读写海量数据多写、跨机房、高可用、写密集
可调一致性（Tunable Consistency）：Cassandra 每次读写都能指定要几个副本确认（ONE / QUORUM / ALL）。W + R > N 即可保证读到最新（N 副本数）。这让你在「延迟/可用」与「一致性」之间逐请求地权衡——把 CAP 的取舍变成了一个旋钮。呼应 architect/ 的 CAP/Quorum。
国内还常见 HBase + Phoenix（加 SQL 层）、以及把宽列能力融进 OLAP 的 OpenTSDB（时序，建在 HBase 上）。云上则有 AWS DynamoDB（托管 KV/宽列）、阿里云 Tablestore（OTS）等托管对等物，思想一致、免运维。
03

KV 与缓存：Redis

服务层最快的一档 · 也是面试缓存题的主场

Redis 是内存 KV 存储的事实标准——亚毫秒延迟、单机十万级 QPS。在数据平台里它既是缓存（挡在数仓/服务层前面），也是实时特征/计数/排行榜的服务存储。

3.1 不只是字符串：丰富的数据结构

String / Hash

缓存对象、用户画像字段（Hash 存一个对象的多字段）。

ZSet（有序集合）

排行榜、Top-N、按分数范围取——实时榜单神器。

HyperLogLog / Bitmap

极省内存地估算 UV、签到/活跃统计——大数据基数统计常用。

# 典型：缓存旁路（Cache-Aside）—— 服务层最常见的读模式
v = redis.get(key)
if v is None:                 # 未命中
    v = db.query(key)            # 回源到数仓/服务层
    redis.setex(key, ttl, v)     # 写回缓存 + 设过期
return v

3.2 缓存三大经典问题（高频面试）

缓存穿透

查根本不存在的 key，每次都击穿到 DB。解：缓存空值 + Bloom Filter 先挡掉一定不存在的 key（呼应 D01）。

缓存击穿

某热点 key 过期瞬间，大量请求同时回源压垮 DB。解：互斥锁/单飞只让一个请求回源 + 热点 key 不过期/逻辑过期。

缓存雪崩

大量 key 同时过期或 Redis 宕机，请求全砸 DB。解：过期时间加随机抖动 + 高可用集群 + 限流降级。

缓存还有一致性难题：DB 更新了、缓存怎么同步？常用「更新 DB + 删除缓存（Cache-Aside）」而非更新缓存，并接受短暂不一致（最终一致）。强一致缓存极难且贵——这又回到 CAP 的本质取舍。
04

搜索与日志分析：Elasticsearch

倒排索引 · 全文检索 / 多维过滤 / 日志分析的主力

当需求是「按任意字段全文搜、多条件过滤、近实时聚合」——比如商品搜索、日志检索（ELK）、可观测性——关系库和宽列都不擅长，这是 Elasticsearch（基于 Lucene）的主场。它的核心是倒排索引。

倒排索引把「文档含哪些词」翻转为「每个词出现在哪些文档」，于是按词检索是 O(1) 取列表 + 求交，而非全表扫描。
ES 与 OLAP 引擎的区别：ES 强在全文检索、灵活多维过滤、近实时；OLAP（ClickHouse 等）强在大规模数值聚合扫描。日志分析两者都能做，但：要「按关键词搜日志、看上下文」→ ES；要「对海量日志做大规模指标聚合」→ ClickHouse 往往更快更省。很多公司日志检索用 ES、指标聚合用 ClickHouse 并存。
ELK / EFK 栈：Elasticsearch（存储检索）+ Logstash/Fluentd（采集，呼应 D06）+ Kibana（可视化），是日志与可观测性的经典组合。ES 也常作为「搜索服务层」：把数仓产出的可搜索数据（商品、内容）同步进 ES 供线上搜索。
ES 不是事务数据库：近实时（默认 1s refresh 才可见）、不擅高频更新、强一致弱。别拿它当主存储或 OLTP 库，它是「检索/分析的副本视图」。
05

选型与一致性：该用谁

把访问模式映射到存储 · 一张决策清单

Q1→要极致低延迟、临时/可重建（缓存、计数、排行榜、实时特征）？ → Redis
Q2→要海量数据按 key 持久化读写、可线性扩展（画像、明细、消息）？ → HBase（强一致/Hadoop 生态）或 Cassandra（高可用/多写）
Q3→要全文搜索 / 多维过滤 / 日志检索？ → Elasticsearch
Q4→要灵活 schema 的文档 / 内容？ → MongoDB
Q5→要大规模分析聚合（不是点查）？ → 回到 D07 OLAP 引擎，不要硬塞给服务层
用 CAP 给它们定位（呼应 architect 板块）：网络分区时只能 CP 或 AP。HBase 偏 CP（要强一致，分区时宁可拒绝服务）；Cassandra / Redis 集群偏 AP（要高可用，容忍最终一致）。选型即回答业务：「读到旧数据可接受吗？」可接受 → AP 拿可用性；不可接受（如余额）→ CP 或干脆上事务库。
架构常识：服务层数据通常是计算层产出的「派生视图」，可重建。所以它更看重读性能与可用性，一致性可放松（坏了从数仓重灌）。这与「真相之源」（数仓/湖仓 + CDC）分工明确——真相在湖仓，服务层是为线上读优化的缓存/视图。这正是 Lambda/Kappa（D08）serving 层的定位。
06

小结 · 面试 · 术语表

收成判断力

一页纸总结

① 服务层定位

为线上高并发低延迟点查而生，是计算结果的对外窗口；真相仍在湖仓。

② 四大家族

KV/缓存(Redis)、宽列(HBase/Cassandra)、文档(Mongo)、搜索(ES)，按访问模式选。

③ LSM 内核

宽列/KV 多基于 LSM：顺序写换吞吐，靠 compaction + Bloom 维持读。写密集首选。

④ RowKey 命门

决定分布与热点。单调递增=热点；散列打散防热点但失范围扫，需折中。

⑤ 缓存三问题

穿透(Bloom/空值)、击穿(互斥/不过期)、雪崩(随机TTL/高可用)。

⑥ CAP 定位

HBase 偏 CP、Cassandra/Redis 偏 AP；可调一致性把取舍变旋钮。

面试自测（点击展开）

基础什么场景该用 NoSQL 而不是 MySQL？
规模数据量超单库、需水平线性扩展（宽列）。
并发/延迟极高并发点查、毫秒延迟（Redis/宽列服务层）。
模式schema 灵活（文档）或要全文搜索（ES）。
代价放松强一致/事务/复杂 join。要事务一致仍用关系库。
核心HBase 为什么写很快？读为什么可能慢？
写快LSM：先写 WAL+MemTable（内存顺序），攒批顺序刷 SSTable，没有随机写。
读慢一个 key 可能分散在 MemTable + 多层 SSTable，要查多处（读放大），靠 Bloom Filter 跳过无关文件缓解。
还债后台 compaction 合并 SSTable 减少层数，恢复读性能——和 D11 湖仓、D07 MergeTree 同源。
高频RowKey 用时间戳有什么问题？怎么改？
热点单调递增 → 新写入永远落最后一个 Region，单点被打爆，扩容无效。
打散加盐前缀/哈希/反转把 key 散到各 Region。
折中散列前缀 + 业务键 + 时间，兼顾防热点与同维度聚簇/范围扫。按查询模式定。
实战缓存穿透、击穿、雪崩怎么区分和应对？
穿透查不存在的 key 直击 DB → 缓存空值 + Bloom Filter。
击穿单个热点 key 过期瞬间高并发回源 → 互斥锁/单飞 + 热点不过期。
雪崩大量 key 同时过期/Redis 宕 → TTL 加随机抖动 + 高可用 + 限流降级。
架构实时数仓算好的指标，怎么服务给前端大屏/App？
写服务层Flink/批算好结果写入 serving 层（Redis 存热点聚合、HBase 存明细/画像、ES 存可搜索数据）。
应用读前端只查 serving 层，毫秒返回，不直连数仓。
分工真相在湖仓（可重建），服务层是为线上读优化的派生视图——这就是 Lambda/Kappa 的 serving 层（D08）。

术语表速查 · Glossary

① 总览

NoSQL Not Only SQL关系型之外、针对特定访问模式优化的数据模型：KV/宽列/文档/搜索/图/时序。普遍放松一致性换扩展与性能。
在线服务层 Serving Layer把计算结果写入支持高并发低延迟点查的存储，供线上实时读取。Lambda/Kappa 的对外窗口。
点查 vs 扫描点查=按 key 取一两条（服务层）；扫描=大范围读聚合（OLAP）。两者优化方向相反。

② 宽列与 LSM

宽列 / BigTable按 rowkey 存超大稀疏表、列可动态、按列族存。HBase/Cassandra 是其开源实现。
LSM-Tree写进内存 MemTable+WAL，顺序刷 SSTable，后台 compaction。把随机写变顺序写，写吞吐高、读放大。
SSTable不可变的有序键值文件，LSM 的磁盘存储单元。
RowKey宽列存储的主键兼物理排序键，决定数据分布。设计不当致写入热点。
热点 Hotspot读写集中在少数分片/节点。单调递增 rowkey 是典型成因，靠加盐/哈希打散。
可调一致性Cassandra 每请求指定确认副本数（ONE/QUORUM/ALL），W+R>N 保证读最新。

③ 缓存与搜索

Cache-Aside 缓存旁路读先查缓存、未命中回源并写回；写时更新 DB + 删缓存。最常见缓存模式。
穿透/击穿/雪崩穿透=查不存在 key；击穿=热点 key 过期瞬间；雪崩=大量 key 同时失效。各有对策。
倒排索引 Inverted Index从「文档→词」翻转为「词→文档列表」，使按词检索 O(1) 取列表，是搜索引擎的根。
近实时 Near Real-TimeES 写入后需 refresh（默认 1s）才可被搜到。不是强一致即时可见。
ELK / EFKElasticsearch + Logstash/Fluentd + Kibana，日志与可观测性经典栈。
