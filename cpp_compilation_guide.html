总览 — 编译流水线

源代码.cpp / .hpp
→
① 预处理.i / .ii
→
② 编译.s
→
③ 汇编.o / .obj
→
④ 链接可执行文件

对应 GCC / Clang 分步命令：

# 完整过程一步完成
g++ main.cpp -o main

# 分步执行，观察每个阶段产物
g++ -E  main.cpp -o main.i     # 阶段1: 预处理
g++ -S  main.i   -o main.s     # 阶段2: 编译
g++ -c  main.s   -o main.o     # 阶段3: 汇编
g++     main.o   -o main       # 阶段4: 链接

① 阶段 1：预处理（Preprocessing）

工具：预处理器（cpp / gcc -E）
输入：.cpp + .hpp
输出：.i（C）/ .ii（C++）

预处理器执行的操作

操作说明示例
头文件展开 递归替换 #include 为头文件实际内容 #include "PmRpcAdapter.hpp" → 展开整个头文件
宏展开 替换所有 #define 宏 #define MAX 100 → 所有 MAX 替换为 100
条件编译 根据条件保留/删除代码块 #ifdef DEBUG ... #endif
删除注释 移除所有 // 和 /* */ 注释
行号标记 插入 #line 指令用于调试定位 # 1 "main.cpp"
字符串拼接 合并相邻字符串字面量 "hello" " world" → "hello world"

以 PmRpcAdapter.cpp 为例

// 预处理前
#include "PmRpcAdapter.hpp"
#include <algorithm>
#include <NetconfAgent/Adapter/PerformanceMonitoring/EpeStats.hpp>
// ...

// 预处理后（.ii 文件可能膨胀到数万行）
// PmRpcAdapter.hpp 的全部内容被插入这里
// algorithm 标准库的全部内容被插入这里
// EpeStats.hpp 的全部内容被插入这里
// ...然后才是 PmRpcAdapter.cpp 自身的代码
关键概念：翻译单元（Translation Unit）
  • 一个 .cpp 文件经过预处理后的完整产物称为一个翻译单元
  • 每个翻译单元独立编译，互不可见
  • 这就是为什么头文件需要 include guard（#pragma once 或 #ifndef）

② 阶段 2：编译（Compilation）

工具：编译器前端+后端（cc1plus / clang）
输入：.i / .ii 预处理后的文件
输出：.s 汇编文件

这是最复杂的阶段，内部又分为 6 个子阶段：

1 词法分析 源码 → Token 流
2 语法分析 Token → AST
3 语义分析 类型检查 / 模板实例化
4 IR 生成 AST → 中间表示
5 优化 -O0 / -O2 / -O3
6 代码生成 IR → 汇编 .s

2.1 词法分析（Lexical Analysis / Tokenization）

将源代码字符流分解为 Token（词法单元）：

// 源代码
int count = 42;

// Token 流
[关键字:int] [标识符:count] [运算符:=] [整数字面量:42] [分号:;]

Token 类型包括：关键字、标识符、字面量、运算符、分隔符等。

2.2 语法分析（Syntax Analysis / Parsing）

将 Token 流构建为 AST（抽象语法树）：

         声明语句
        /    |    \
     类型  标识符  初始化表达式
      |      |         |
     int   count      42

对于模板代码：

template <typename TMeasurementObject>
void addMeasurementObjects(std::vector<TMeasurementObject> const& measurementObjects, ...)

AST 会记录这是一个函数模板，TMeasurementObject 是模板类型参数。

2.3 语义分析（Semantic Analysis）

检查项说明示例
类型检查确保操作数类型兼容int + string → 编译错误
名称查找解析标识符指向的声明yangAdapter_->setWithXml()
重载决议选择最佳匹配的重载函数addMeasurementObject() 有 9 个重载
模板实例化用具体类型替换模板参数TMeasurementObject → EMeasurementObject_RxWindow
隐式转换插入必要的类型转换bool → "true"/"false" 字符串
访问控制检查 private/protected/public
const 正确性确保 const 对象不被修改

模板实例化细节（与 PmRpcAdapter.cpp 直接相关）

1. 模板定义 — 编译器记录模板但不生成代码

template <typename TMeasurementObject>
EOranPmHandleResult PmRpcAdapter::stopMonitoring(
    std::vector<TMeasurementObject> const& measurementObjects,
    std::string const& objectUnit)
{ ... }

2. 显式实例化 — 告诉编译器"为这些类型生成代码"

template EOranPmHandleResult PmRpcAdapter::stopMonitoring(
    std::vector<EMeasurementObject_RxWindow> const&, std::string const&);
template EOranPmHandleResult PmRpcAdapter::stopMonitoring(
    std::vector<EMeasurementObject_Transceiver> const&, std::string const&);
// ...共 8 个显式实例化

3. 全特化 — 为特定类型提供完全不同的实现

template <>
EOranPmHandleResult PmRpcAdapter::startMonitoring(int interval,
    std::vector<EMeasurementObject_RxWindow> const& measurementObjects, ...)
{ /* RxWindow 专用的实现 */ }

编译器选择优先级：

全特化 > 偏特化 > 通用模板

2.4 中间代码生成（IR Generation）

将 AST 转换为 中间表示（Intermediate Representation）：

; LLVM IR 示例（简化）
define i32 @stopMonitoring_RxWindow(%class.PmRpcAdapter* %this,
                                     %vector* %measurementObjects,
                                     %string* %objectUnit) {
entry:
  ; ... 中间指令
  ret i32 0
}
IR 的特点
  • 与平台无关（不绑定 x86/ARM）
  • 保留了足够的类型信息用于优化
  • SSA 形式（Static Single Assignment，每个变量只赋值一次）

2.5 优化（Optimization）

优化级别说明
-O0不优化，编译最快，便于调试
-O1基本优化（死代码消除、常量折叠）
-O2标准优化（内联、循环优化、指令调度）最常用
-O3激进优化（向量化、函数克隆）可能增大代码体积
-Os优化代码大小
-Ofast-O3 + 不严格遵循 IEEE 浮点标准

常见优化手段：

常量折叠:    int x = 3 + 5;         → int x = 8;
死代码消除:  if (false) { ... }      → 删除
函数内联:    小函数调用               → 直接展开函数体
循环展开:    for(i=0;i<4;i++) f()   → f();f();f();f();
尾调用优化:  递归调用                 → 循环

2.6 目标代码生成（Code Generation）

将优化后的 IR 转为目标平台的 汇编代码：

; x86-64 汇编（简化示例）
_ZN13netconfagent7adapter13PmRpcAdapter14stopMonitoringINS...
    push    rbp
    mov     rbp, rsp
    sub     rsp, 64
    ; ... 函数体
    leave
    ret
Name Mangling（名称修饰）

函数名 _ZN13netconfagent7adapter13PmRpcAdapter14stopMonitoring... 编码了命名空间、类名、参数类型等信息，使得 C++ 的函数重载、命名空间等特性能够在链接阶段正确区分。

使用 c++filt 命令可以还原可读名称。

③ 阶段 3：汇编（Assembly）

工具：汇编器（as / nasm）
输入：.s 汇编文件
输出：.o / .obj 目标文件

汇编器做两件事：

  1. 将汇编助记符翻译为 机器码（二进制指令）
  2. 生成 目标文件，包含多个段（Section）

目标文件的段结构

段名内容示例
.text可执行机器码函数体
.data已初始化的全局/静态变量int g = 42;
.bss未初始化的全局/静态变量（不占文件空间）int g;
.rodata只读常量数据字符串 "performance-measurement-objects"
.symtab符号表函数名、变量名及地址
.rel.text重定位表记录需要链接器填充的地址

目标文件中的符号类型

U Undefined — 引用了但未定义（需要链接器从其他 .o 找）
T Text — 在本文件中定义的函数
D Data — 在本文件中定义的已初始化全局变量
B BSS — 在本文件中定义的未初始化全局变量

对于 PmRpcAdapter.o：

T PmRpcAdapter::stopMonitoring<EMeasurementObject_RxWindow>(...) — 已定义
U YangAdapter::setWithXml(...) — 未定义，需从 YangAdapter.o 获取

④ 阶段 4：链接（Linking）

工具：链接器（ld / lld / gold）
输入：多个 .o + 库文件
输出：可执行文件或共享库

4.1 符号解析（Symbol Resolution）

将所有目标文件的符号表合并，为每个未定义符号（U）找到其定义：

PmRpcAdapter.o:  需要 YangAdapter::setWithXml()  ← 未定义
YangAdapter.o:   定义了 YangAdapter::setWithXml()  ← 找到！
常见链接错误
  • undefined reference — 找不到符号定义（忘了链接某个 .o 或库）
  • multiple definition — 同一符号在多处定义（违反 ODR 规则）

4.2 重定位（Relocation）

编译阶段每个 .o 的代码地址从 0 开始，链接器分配最终地址：

PmRpcAdapter.o 中:  call 0x00000000  (占位地址)
链接后:              call 0x00401234  (YangAdapter::setWithXml 的真实地址)

4.3 段合并（Section Merging）

最终可执行文件:
  .text   = main.o(.text) + PmRpcAdapter.o(.text) + Controller.o(.text) + ...
  .data   = main.o(.data) + PmRpcAdapter.o(.data) + ...
  .rodata = 所有字符串常量合并

4.4 库链接

类型后缀特点
静态库 .a / .lib 编译时将用到的 .o 提取并合并到可执行文件。体积大，但不依赖外部库文件
动态库 .so / .dll 可执行文件只记录依赖，运行时由动态链接器加载。体积小，可共享

CMake 构建流程

add_executable(netconf_agent
    src/main.cpp
    src/NetconfAgent/Adapter/PerformanceMonitoring/PmRpcAdapter.cpp

    # ... 其他源文件
)

target_link_libraries(netconf_agent

)

实际编译命令（CMake 生成）

# 每个 .cpp 独立编译为 .o（可并行 -j8）
g++ -c -std=c++17 -O2 -I./src src/main.cpp \
    -o build/main.o
g++ -c -std=c++17 -O2 -I./src \
    src/NetconfAgent/Adapter/PerformanceMonitoring/PmRpcAdapter.cpp \
    -o build/PmRpcAdapter.o

    -o build/Controller.o
# ...

# 最后链接所有 .o 和库
g++ build/main.o build/PmRpcAdapter.o build/Controller.o ... \
    -lsysrepo -lyang -lboost_system -lpthread \
    -o build/netconf_agent

面试深度专题 ⭐ — 编译链接进阶

站在高级 C++ 工程师视角，编译/链接的深度考点集中在：ODR、LTO、内联/模板的链接行为、动/静态库对比、Name Mangling 与 ABI 稳定性、常见链接错误诊断。

1. ODR — One Definition Rule（单一定义规则）

C++ 最基础也最易违反的规则：

  • 一个翻译单元内：每个变量/函数/类/模板/枚举 定义 只能出现一次
  • 整个程序：非 inline 函数和全局变量 定义 只能出现一次
  • 多个 TU 中：类、inline 函数、模板可重复定义，但必须逐 token 相同

违反 ODR 的典型场景（UB，链接器不一定报错）

// foo.hpp
struct Foo { int x; };

// a.cpp
#define DEBUG
#include "foo.hpp"   // 假设 hpp 里有 #ifdef DEBUG 改变布局

// b.cpp
#include "foo.hpp"   // 未定义 DEBUG —— Foo 在 a/b 中布局不同！UB

隐形 ODR 违反：-D 宏不一致、不同版本头文件、匿名命名空间 vs static、static library 版本混用。

检测：gold --detect-odr-violations、clang -fsanitize=odr、ThinLTO 偶发报告。

2. LTO / ThinLTO — 链接时优化

传统编译以单个翻译单元为优化边界：跨 .cpp 的函数无法内联、无法常量传播、无法死代码消除。

.cpp × N
→
编译器发 LLVM IR / GIMPLE-flto
→
链接器全程序优化跨 TU 内联 / DCE
选项作用代价
-flto (Full LTO)全程序合并成一个大 IR 再优化链接时间/内存暴涨，大项目不可用
-flto=thin (ThinLTO)每个 TU 独立 IR + 轻量跨 TU 导入增量/并行友好，Clang/LLD 推荐
-fuse-linker-pluginGCC LTO 需配合 gold/lld—

典型收益：纯 C++ 后台服务 5-15% 加速，包含大量模板/小函数的代码收益更高（跨 TU 内联）。

3. inline / constexpr / 模板 的链接行为

声明定义可在头文件？符号类型说明
void f();❌ 头只声明external定义必须在 .cpp，否则链接报 multiple def
inline void f(){}✅弱符号 (COMDAT)链接器按签名合并，任选一份保留
constexpr f(){}✅隐式 inline等同 inline
template<> T f(){}✅COMDAT 弱符号每个实例化按需合并
static void f(){}✅internal linkage每 TU 独立副本（代码膨胀）
namespace { void f(); }✅internal匿名命名空间等同 static（但支持模板）
C++17 inline 变量✅COMDAT全局单例不再需要 .cpp 放定义

关键点：inline 的链接语义比展开优化更重要——它告诉链接器允许多定义并合并。没有 inline 的函数放头文件会 multiple definition。

4. 静态库 vs 动态库 深度对比

维度静态库 (.a)动态库 (.so)
链接时机链接时复制进可执行文件运行时 ld.so 加载
符号解析按归档成员按需抽取（未引用不链入）整个 so 全载入
符号顺序⚠️ 依赖顺序敏感：-lA -lB，A 引用 B不敏感
可执行体积大（打包所有依赖）小（共享）
ABI 升级重编译全程序替换 .so 即可（若 SONAME 兼容）
符号冲突链接期报错按加载顺序解析，可能静默覆盖（LD_PRELOAD 原理）
初始化/析构随进程启停__attribute__((constructor))/dlopen 时机

动态库的隐形坑：

  • SONAME vs filename：libfoo.so.1 是 SONAME（ABI 版本），libfoo.so.1.2.3 是实际文件，libfoo.so 是 devel 软链
  • 全局符号污染：默认所有 extern 符号可见，应显式 -fvisibility=hidden + __attribute__((visibility("default")))
  • C++ 静态对象析构顺序跨 so 不保证，易 crash

5. Name Mangling 与 ABI 稳定性

C++ 函数重载、模板、命名空间都通过 name mangling 编码进符号名。GCC/Clang 使用 Itanium C++ ABI。

// 源代码
namespace ns { int foo(int, const std::string&); }

// mangled
_ZN2ns3fooEiRKSs            // GCC < 5, old ABI
_ZN2ns3fooEiRKNSt7__cxx1112basic_stringIcSt11char_traitsIcESaIcEEE  // GCC ≥ 5, C++11 ABI

// 还原
c++filt _ZN2ns3fooEiRKSs   // → ns::foo(int, std::string const&)

GCC 5+ Dual ABI — 经典坑

GCC 5 为了支持 C++11 的 std::string/std::list 标准（SSO、O(1) size），引入 双 ABI：

  • _GLIBCXX_USE_CXX11_ABI=1（默认）：新 ABI，命名空间 std::__cxx11
  • _GLIBCXX_USE_CXX11_ABI=0：旧 ABI，兼容 GCC 4

两种 ABI 编译的 .o 链接会 undefined reference——因为 mangled 名不同。解决：统一宏；或 extern "C" 接口隔离。

为什么 C 没有这些问题：C 无重载、无模板，函数名直接就是符号名，ABI 极其稳定（几十年不变）。

6. 常见链接错误诊断手册

错误原因排查
undefined reference to 'foo'声明了但没定义 / 缺库 / 库顺序错 / ABI 不一致nm -u a.out | c++filt；检查 -l 顺序；对比 _GLIBCXX_USE_CXX11_ABI
multiple definition of 'foo'非 inline 函数/全局变量定义在头文件且被多次 include加 inline 或移到 .cpp
undefined reference to vtable for X类的第一个非纯虚函数没定义（vtable 跟它放一起）确认所有虚函数都有定义，或用 =default/=delete
undefined reference to typeinfo for X同上，RTTI 信息跟 vtable 一起同上

relocation R_X86_64_PC32 ... recompile with -fPIC静态库想被 so 链接，但没用 PIC 编译静态库用 -fPIC 编译

7. 高频面试题 Q&A

Q1: #pragma once vs #ifndef 头文件保护？

功能等价。#pragma once 更简洁、避免 macro 重名，但依赖编译器识别同一物理文件（符号链接/同一文件不同路径可能失败）。#ifndef 标准但 macro 可能冲突。现代项目优先 #pragma once，所有主流编译器都支持。

Q2: 为什么模板的定义必须放在头文件？

模板不是代码，是代码生成器。编译器在实例化点才知道具体类型，需要看到完整定义。若模板定义在 a.cpp，b.cpp 想用 Foo<int>，b.cpp 的编译器根本看不到定义，链接器也不会生成——会 undefined reference。

例外：显式实例化 template class Foo<int>; 放 a.cpp，此时 a.o 会生成该符号，b.cpp 声明 extern template class Foo<int>; 避免重复实例化。

Q3: extern "C" 的真正作用？

告诉 C++ 编译器对这块声明禁用 name mangling，使用 C 风格符号名。用途：

  • C++ 写库给 C 调用（.so 导出符号）
  • C++ 调 C 库的头文件（extern "C" { #include <pthread.h> }，C 头通常自带 #ifdef __cplusplus 包裹）
  • 绕过 ABI 不稳问题（C ABI 超级稳定）

限制：extern "C" 函数不能重载（同名 mangling 冲突）、不能是成员函数、不能是模板。

Q4: 静态库里的全局对象构造函数不被调用？

经典坑。链接器对静态库按需抽取——某 .o 里的符号如果主程序没引用，该 .o 整个不被链入，连带它的全局对象构造也消失。

场景：注册模式（全局对象构造时调 Factory::Register(...)），静态库里没人直接 new 这个类 → 被优化掉。

解决：

# 强制链入整个归档
g++ main.o -Wl,--whole-archive -lplugin -Wl,--no-whole-archive
# 或 clang
clang++ main.o -Wl,-force_load,libplugin.a

Q5: 为什么要用 -fvisibility=hidden？

Linux .so 默认导出所有 extern 符号——产生三大问题：

  1. 符号冲突：多个 .so 里同名 inline 函数，加载时首个 so 的版本赢（ODR 幽灵违反）
  2. 加载慢：ld.so 要解析巨量符号
  3. 二进制体积：symbol table 膨胀

最佳实践：

g++ -fvisibility=hidden -fvisibility-inlines-hidden ...

// 需要导出的 API 显式标记
#define API_EXPORT __attribute__((visibility("default")))
API_EXPORT void publicFunc();

Q6: 什么是弱符号 / strong symbol？

ELF 符号有强度：

  • 强符号：普通函数、已初始化的全局变量
  • 弱符号：inline 函数、模板实例（COMDAT 组）、未初始化全局变量（C 老语义）、__attribute__((weak))

规则：多强符号冲突 → 报错；强弱冲突 → 强胜；多弱冲突 → 任选。这是 inline 函数可以出现在多个 TU 的底层机制。

Q7: RTTI 与 exception 跨 so 有什么坑？

RTTI 的 typeid 和 dynamic_cast 跨 so 比较的是 type_info 的指针（旧）或 mangled name 字符串（新）。若：

  • 两个 so 都静态链 libstdc++ → 各自有独立 type_info 拷贝 → dynamic_cast 失败
  • 异常抛出后跨 so 边界 catch → 若 type_info 不同，catch 不到

解决：动态链 libstdc++（默认），或导出关键类型的 RTTI 符号（-fvisibility=default for typeinfo）。

Q8: 为什么 precompiled header (PCH) 能加速编译？

C++ 编译慢的主因是头文件重复解析。典型项目：<vector>、<string>、Boost、自家公共头——每个 .cpp 都要重新解析一遍，展开数十万行。

PCH 把这些公共头预先编译成二进制 AST，每个 .cpp 编译时直接加载，跳过词法/语法/语义分析前端。大型项目可加速 20-50%。

配合技巧：

  • CMake 3.16+：target_precompile_headers(tgt PRIVATE <string> <vector>)
  • Unity Build（把多个 .cpp 合并编译）进一步加速，但影响调试和增量编译
  • ccache：编译结果缓存，CI 环境命中率可达 90%+

维度Debug (-O0 -g)Release (-O2/-O3)

Q10: 如何大幅加速一个百万行 C++ 项目的编译？

  1. 分布式编译：distcc / icecc / goma

现代链接器与符号管理 进阶 ★ Senior

lld / mold / gold · visibility · weak / strong · gc-sections · 全局构造顺序 — 大型 C++ 项目链接阶段的真知识

L1. lld / mold / gold 横向对比

BFD ld 是 GNU 的"老古董"链接器，单线程、慢。现代项目应当根据场景选择：

链接器开发方典型加速关键特性何时选
ld (BFD)GNU1×（基线）兼容性最好，单线程极少新项目
goldGoogle (2008)2-5×多线程、ICF (相同函数折叠)、增量链接遗留项目过渡（已停止维护）
lldLLVM5-10×跨平台 (ELF/COFF/MachO/Wasm)、ThinLTO 友好clang 工具链、跨平台
moldRui Ueyama (lld 作者)10-50×极致并行、 大型 C++ 项目压倒性快Chromium/LLVM/Rust 等巨型项目
# 切换链接器（GCC 12+ / Clang）
g++ -fuse-ld=lld  ...      # 用 lld
g++ -fuse-ld=mold ...      # 用 mold
# 或者临时
mold -run cmake --build .

# CMake 风格
add_link_options(-fuse-ld=mold)

# 看真实生效
ld.lld --version
readelf -p .comment myapp

实测数据感觉（来自社区基准）

  • Chromium 全量链接：BFD ld 53s → lld 12s → mold 2s
  • Clang 自身链接：BFD ld 32s → mold 1.6s
  • Linux kernel：mold 还不够稳，主线仍用 BFD（kernel 重 incremental）

选型经验：用户态 C++ 项目无脑选 mold；底层库 / 嵌入式 / 老编译器选 lld；只有 BFD ld 才支持某些 linker script 极端用法时才回到 BFD。

L2. Symbol Visibility —— 四档完整规则

visibility本 so 内可见外部 so 可见可被覆盖典型用途
default是是（导出）是（先到先得）公共 API（默认）
protected是是（导出但本 so 内调用不走 PLT）否性能敏感的公共 API
hidden是否否所有内部实现细节
internal是否否架构特定，极少用

实战范式：默认 hidden + 显式 API_EXPORT

// CMake
set_target_properties(mylib PROPERTIES
    CXX_VISIBILITY_PRESET hidden
    VISIBILITY_INLINES_HIDDEN ON
    C_VISIBILITY_PRESET hidden)

// api_export.h —— 跨平台导出宏
#if defined(_WIN32)
  #ifdef MYLIB_BUILDING
    #define MYLIB_API __declspec(dllexport)
  #else
    #define MYLIB_API __declspec(dllimport)
  #endif
#elif defined(__GNUC__)
  #define MYLIB_API __attribute__((visibility("default")))
#else
  #define MYLIB_API
#endif

// 公开头文件
MYLIB_API void publicFunc();
MYLIB_API class Connection { ... };       // 整个类（含 vtable / RTTI）都导出

为什么大型 C++ 项目必须开 hidden？

  1. 减少符号冲突：两个 so 里同名 inline 函数 / 模板实例 → ld.so 加载时第一个赢。结果：A 调用自家的 fn() 实际跑的是 B 的实现，且 ABI 不同→崩溃。这是"so 顺序敏感"bug 的根源。
  2. 启动加速：ld.so 不再做巨量符号解析。大型 KDE / Qt 应用启动可加速 30%。
  3. 二进制瘦身：典型项目 .dynsym 缩小 60-90%。
  4. 优化机会：编译器对 hidden 符号知道"外部不可能 override"→ 内联更激进。

历史教训：libstdc++ 因为历史原因没全 hidden，Google 内部所有用 -fvisibility=hidden 的 C++ 项目都遇到过 std::ostream 全局对象在不同 so 里被复制初始化的诡异 bug。

Version Script —— 更精细的导出控制

// myapi.ver
LIBMYAPI_1.0 {
    global:
        myapi_init;
        myapi_call;
        _ZN7Connection*;      // 通配 mangled 名
    local:
        *;                    // 其他全部隐藏
};

# 链接时
g++ -shared -Wl,--version-script=myapi.ver -o libmyapi.so *.o

Version script 比 -fvisibility 更强：可以做版本化导出，新增符号属于 LIBMYAPI_2.0，老程序还能正常链接老符号。glibc、OpenSSL 都用这个机制保持几十年 ABI 兼容。

L3. Weak Symbol 的真实威力

弱符号不只是"可以重复定义"——它是 C/C++ 实现可选依赖、运行时插桩、默认实现可被覆盖的核心机制。

用法 ①：声明可选依赖（库可能不存在）

// 应用可能链接也可能不链接 libprofiler.so
extern "C" void ProfilerStart(const char*) __attribute__((weak));
extern "C" void ProfilerStop() __attribute__((weak));

int main() {
    if (ProfilerStart) {              // 检查符号是否解析成功
        ProfilerStart("my.prof");
        run_workload();
        ProfilerStop();
    } else {
        run_workload();                // 没装 profiler，正常跑
    }
}

用法 ②：提供默认实现，允许用户覆盖

// libmyapi 内部 —— 默认 logger 实现，weak
__attribute__((weak)) void myapi_log(const char* msg) {
    fputs(msg, stderr);
}

// 用户代码 —— 提供 strong 版本
extern "C" void myapi_log(const char* msg) {
    spdlog::info(msg);              // 走自家日志
}
// 链接器规则：strong 永远胜 weak，无需任何配置

用法 ③：Function interposition / 系统调用插桩

// glibc 内部很多函数都是 weak，可被用户覆盖。例如 malloc：
// libc.so.6 内: __attribute__((weak)) void* malloc(size_t);

// jemalloc / tcmalloc 的核心机制就是提供 strong 的 malloc/free，
// 链接时优先匹配（或 LD_PRELOAD 运行时插入）。

COMDAT —— inline 函数 / 模板的"自动 weak"

C++ 模板和 inline 函数在多个 TU 都会生成定义，链接器靠 COMDAT 组实现"任选一份保留，剩下丢弃"：

L5. 全局对象的初始化顺序（深坑专区）

同一 TU 内

按声明顺序构造，逆序析构。简单可靠。

跨 TU —— Static Initialization Order Fiasco (SIOF)

// a.cpp
Foo g_foo = Bar::instance().getX();   // 用了 b.cpp 的 g_bar

// b.cpp
Bar g_bar(42);                            // 谁先谁后？标准未规定！

解决方案三选一：

  1. Construct on First Use idiom：把全局对象藏进函数局部 static，第一次调用才构造，顺序由调用关系决定。
  2. Meyers Singleton（C++11 起 magic static 线程安全）： 
    inline Bar& getBar() {
    static Bar instance(42);   // C++11 保证线程安全初始化
    return instance;
}
  3. __attribute__((init_priority(N)))（GCC 特有，101-65535，越小越早）：精确控制 .init_array 中的顺序。仅适用于同一 so 内，跨 so 仍不可控。

底层机制：.init_array / .ctors

# 查看一个 .o / .so 的全局构造函数列表
readelf -d libfoo.so | grep INIT_ARRAY
objdump -j .init_array -d libfoo.so

# 调试：让 gdb 在每个全局构造前停下
gdb ./myapp
(gdb) catch syscall                       # 或
(gdb) b _GLOBAL__sub_I_*                  # 全局构造的 mangled 名

真实事故：so 加载顺序导致 logger 构造前被调用

项目结构：libcore.so 提供 Logger 单例（全局 ostream），libplugin.so 的某全局对象 Foo 在构造里写日志。理想顺序：libcore 先初始化。

但因为 g++ main.o -lplugin -lcore 这个顺序，ld.so 反向解析依赖，libplugin 先初始化。Foo 构造时调 Logger，Logger 自身还未构造 → ostream 是 zero-initialized 状态 → 段错误。

修复：把 Logger 改 Meyers Singleton（按需构造），或调整链接顺序，或用 __attribute__((init_priority(1000))) 强制 Logger 早构造（同 so 内有效）。这种 bug 在 release 才出现—— debug build 因为某些优化不同，构造顺序刚好正确，骗你过测试。

L6. 跨 .so 边界的 C++ 类型坑（ABI 入门）

跨 so 传递是否安全原因
int / POD struct（C ABI 兼容）✅C ABI 是稳定的
std::string by value⚠️两边必须用同版本 libstdc++ + 同 dual ABI 设置（_GLIBCXX_USE_CXX11_ABI）
std::vector<T> by ref/ptr⚠️同上 + T 的 ABI 也要一致
std::shared_ptr<T>⚠️控制块布局必须一致，跨 libstdc++/libc++ 边界会崩
抛异常跨 so⚠️type_info 必须共享（动态链 libstdc++）+ -fvisibility=default for typeinfo
虚函数跨 so 调用⚠️派生类的 vtable 由派生方 so 提供，基类的 layout 必须 ABI 稳定（不能加成员变量）

三个 ABI 安全设计准则

  1. so 之间只暴露 C ABI 头文件（POD struct + 函数指针），C++ 类型只在 so 内部使用。
  2. 必须暴露 C++ 类时用 PIMPL：公开头只放指针 + 一两个稳定 inline 接口，所有真实成员藏在 impl 里。这样添加成员/改实现不破坏 ABI。
  3. 所有交付物锁住 toolchain：编译器版本、libstdc++ 版本、_GLIBCXX_USE_CXX11_ABI 三件套，整条链路必须一致。这是为什么大型 C++ 产品要么 monorepo + 同 toolchain 编译，要么 vendoring libstdc++。
