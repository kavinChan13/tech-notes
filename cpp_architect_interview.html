🏛️架构方法论10 题

M01
架构定义🟢
什么是软件架构？
架构 = 难改的关键决策的总和（Fowler）。包括：宏观结构、质量属性承诺、约束。判断标准：回滚成本 > 1 个月才算架构决策。
M02
C4 模型🟢
C4 模型的四个层级是什么？
L1 Context（系统+用户+外部系统）→ L2 Container（独立部署单元）→ L3 Component（单 Container 内部模块）→ L4 Code（类/函数，通常 IDE 生成不画）。不同层级给不同受众看。
M03
ATAM🟡
ATAM 评估法的四类输出是什么？
🔴 Risk（无法达到关键场景）/ 🟡 Sensitivity Point（决策显著影响某属性）/ 🟠 Tradeoff Point（决策同时影响多属性方向相反）/ 🟢 Non-Risk（评估后无问题）。Tradeoff Point 必须写 ADR。
M04
质量属性🟡
什么是"刺激-响应-度量"三段式？
质量属性必须用 Source + Stimulus + Environment + Response + Measure 表达。例：错误："系统要快"。正确："正常负载下 10k QPS GET 请求到 API 网关，P99 ≤ 50ms"。不可测量的需求不是工程需求。
M05
CAP🟡
CAP 定理常见误读？
三大误读：(1) CAP 是永恒三选二——错，只在分区时才二选一；(2) CAP 的 C = ACID 的 C——错，是 Linearizability；(3) 关系数据库是 CA 系统——错，没有 CA 系统（分布式必有 P）。
M06
架构风格🟡
Hexagonal、Onion、Clean Architecture 的区别？
本质相同，都是依赖倒置 + 业务核心独立。区别只在视觉化：Hexagonal 是两层（核心+外部），Onion 是同心圆多层，Clean 是 4 层 + 明确 Dependency Rule。面试时讲清楚"本质相同"就能加分。
M07
微服务🟡
什么时候应该用微服务？
三个信号：(1) 团队 > 50 人，多团队并行阻塞；(2) 模块扩展需求差异巨大；(3) 不同模块发布节奏差异大。Fowler "Monolith First"：先单体，撑不住再拆。30 人以下团队上微服务是过度设计。
M08
ADR🟢
ADR 的核心三段是什么？
Context（背景和约束）+ Decision（决定做什么）+ Consequences（正负面后果）。ADR 一旦 Accepted 永不删除——历史是资产；改变决策写新 ADR 标记"Superseded by"。
M09
演进式架构🔴
Fitness Function 是什么？给一个例子。
把架构原则写成自动化测试 / CI 检查。例：CI 脚本 grep -r "infra/" src/domain/，找到立即 fail——防止 Domain 层意外依赖 Infrastructure。让机器守护架构是防止腐化的唯一可靠手段。
M10
重构🔴
Strangler Fig 模式怎么运作？
渐进式替换遗留系统：(1) 建 Facade 拦截所有调用；(2) 用 Feature Flag 控制流量到新/旧实现；(3) 影子流量对账；(4) 灰度切流；(5) 必须设 decommission deadline，到期删旧代码。不删 = 双倍维护负担。

📐DDD 与设计模式8 题

D01
DDD 战略🟢
Bounded Context 和微服务是什么关系？
不是 1:1。BC 是语言学边界（同一名词含义一致），微服务是物理部署边界。一个 BC 可以实现为模块、共享库、独立服务、产品线。是否拆服务取决于团队规模、变更节奏、质量属性差异。
D02
DDD 战术🟡
Aggregate 设计的 5 条规则？
(1) 根是唯一入口；(2) 一个事务只改一个 Aggregate（跨用 Domain Event）；(3) 跨 Aggregate 只引用 ID；(4) Aggregate 要小（能一次性 load+save）；(5) 最终一致性优于强一致性。大 Aggregate 是新人最常犯错。
D03
DDD 战术🟡
Value Object 和 Entity 区别？
VO：用值区分（两个相同属性 = 同一对象，如金额、坐标）、不可变、零身份。Entity：用 ID 区分（两个同名的人是不同的人）、可变状态、ID 不变。Entity 只用 ID 判等。
D04
DDD 战略🔴
什么是防腐层（ACL）？什么时候用？
对接糟糕的外部系统（遗留系统、第三方 API、混乱的供应商）时，建一层翻译层，把外部模型翻译成自己的领域模型。防止外部混乱概念污染核心域。例：对接 SAP 时不在 Order 实体里出现 KUNNR/MATNR 等 SAP 字段。
D05
DDD 战术🔴
Outbox 模式解决什么问题？
解决"业务操作 + 事件发布"的原子性：业务写 DB 后直接发消息，如果消息发送失败 → 数据不一致。Outbox 做法：在同一事务把事件写入 outbox 表，后台 poller / CDC 异步发布。最常用的可靠消息发布方案。
D06
CQRS🔴
CQRS 和 Event Sourcing 必须一起用吗？
不必。CQRS 是"读写模型分离"，可以单独用（写关系库、读 Elasticsearch）。Event Sourcing 是"存事件不存状态"，可以单独用（小聚合）。两者结合威力最大，但复杂度也最高——慎用，绝不是 default。
D07
事件风暴🟡
Event Storming 三种粒度？
Big Picture（业务全景，2-4h）→ 发现 BC 边界；Process Level（聚焦一个流程，1 天）→ Command-Event-Policy 链；Design Level（聚焦 Aggregate，0.5-1 天）→ 边界和不变量。红色 sticky（hotspot）是认知冲突点，最有价值。
D08
DDD 战略🟡
Domain 三分（Core/Supporting/Generic）有什么用？
决定资源投入策略。Core（核心竞争力）→ 自研、最强工程师、长期投入。Supporting（必须有但不是竞争力）→ 简化自研或外包。Generic（通用能力如认证/邮件）→ 买现成的（Auth0/SendGrid），不要自己写。

⚙️C++ 工程化9 题

C01
ABI🟡
API break 和 ABI break 的区别？
API = 源码约定，破坏 → 编译报错（立刻可知）。ABI = 二进制约定（符号名、调用约定、对象布局、vtable），破坏 → 运行时崩溃 / 静默数据损坏（最难找）。改类的私有成员不破坏 API 但破坏 ABI。
C02
ABI🟡
Pimpl 模式的核心收益？
公开头只有 unique_ptr<Impl> 指针（8B），sizeof 永远不变。Impl 类在 .cpp 里可随意改成员，ABI 不变。代价：每次方法调用多一层间接 + 一次堆分配。跨 SO 边界的库必备。
C03
CMake🟢
CMake PUBLIC / PRIVATE / INTERFACE 区别？
影响"自己"还是"消费方"：PRIVATE = 只影响自己；PUBLIC = 影响自己+传播给消费方；INTERFACE = 不影响自己，只传播。判断依据：看头文件——头文件 #include 的依赖是 PUBLIC，只在 .cpp 用的是 PRIVATE。
C04
跨编译器🔴
跨编译器（GCC + MSVC）共用 C++ 库怎么做？
必须用 C ABI（extern "C" + 纯 C 函数表 + Opaque Handle）。原因：GCC 和 MSVC 的 name mangling、vtable、异常机制完全不兼容。陷阱：extern "C" 只让符号名稳定，不能传 std::string（C++ 类型本身布局不同）。
C05
Modules🟡
C++20 Modules 解决了 #include 的什么痛点？
四大顽疾：(1) 重复解析（同头文件被解析 N 次）；(2) 宏污染（windows.h 的 min/max 等）；(3) 符号泄漏（间接 include 意外可见）；(4) 顺序依赖。Modules 用语义导入而非文本替换，编译速度提升 40-80%。2026 年生态仍未完全成熟。
C06
插件化🔴
dlopen 用 RTLD_LOCAL 还是 RTLD_GLOBAL？
几乎所有场景用 RTLD_LOCAL。RTLD_GLOBAL 会让插件符号进入全局命名空间，两个插件定义同名函数时静默互相覆盖（违反 ODR 但不会报错）。RTLD_LOCAL 让插件符号隔离，主程序通过 PluginContext 注入服务。
C07
跨平台🟢
long 在不同平台多少位？
Linux 64 位 = 8 字节；Windows 64 位 = 4 字节；Linux/Windows 32 位 = 4 字节。这是 LLP64 vs LP64 模型差异。跨平台代码必须用 int64_t / uint32_t 这类定宽类型。
C08
构建性能🟡
大型 C++ 项目编译慢，按 ROI 怎么优化？
优先级：(1) Ninja（替代 Make，30-50%）；(2) ccache（命中时 90%+）；(3) sccache（CI 共享）；(4) Include 优化（前向声明 + IWYU）；(5) PCH（预编译头）。先 profile 再优化：clang -ftime-trace + chrome://tracing。
C09
ABI 工具🔴
怎么自动检测 ABI break？
两个工具：abi-compliance-checker（生成 HTML 报告）和 libabigail 的 abidiff（直接比较两个 .so）。集成到 CI：每个 PR 把 baseline.dump 和 pr.dump 比较，任何 break 必须显式提升 MAJOR 版本号才能合入。

🌐分布式系统10 题

N01
理论🔴
FLP 不可能定理是什么？
在异步系统中，即使只有一个节点可能故障，不存在确定性的共识算法。实践含义：所有实际共识算法（Raft/Paxos）都必须在某些条件下牺牲 liveness 或引入随机化（如随机选举超时）。
N02
PACELC🟡
PACELC 是什么？比 CAP 强在哪？
If Partition, A or C; Else, Latency or Consistency. CAP 只描述分区时；PACELC 加上"正常时也有 Latency vs Consistency 权衡"——强一致需要更多同步开销 → 更高延迟。DynamoDB 是 PA/EL（可用+低延迟），etcd 是 PC/EC（一致+一致）。
N03
一致性🟡
Linearizability、Sequential、Causal 区别？
由强到弱：Linearizable（每次读到最新写，全局实时顺序）> Sequential（全局顺序但不要求实时）> Causal（只保证因果关系，并发可乱序）。Linearizable 最强但开销大；Causal 是 AP 系统能给的最强保证。
N04
Raft🔴
Raft 的"日志最新"投票规则为什么必要？
保证已提交日志永远不丢。已提交 = 多数派写入；新 Leader 也来自多数派，两者交集至少有一个节点拥有所有已提交日志。投票规则要求 candidate 日志 ≥ 投票者 → 当选 Leader 一定包含所有已提交日志 → 不会回滚。
N05
Raft🔴
Raft 选举超时为什么随机化？
避免split vote：多个 Follower 同时超时同时发起选举，互相不投，没人获得多数票。随机化让一个节点通常比其他节点早超时，能先获得多数票。典型范围 150-300ms。
N06
分布式事务🟡
2PC 为什么不适合微服务？
三大问题：(1) Coordinator 单点，Phase 2 中崩溃导致事务悬挂；(2) 阻塞性，Prepared 后必须持锁等 Phase 2；(3) 2× RTT 延迟，性能差。替代：用 Saga / TCC / Outbox 实现最终一致性。
N07
Saga🔴
Saga 的"脏读"问题怎么处理？
Saga 执行中其他事务可能读到中间态数据。三种应对：(1) 语义锁：状态字段标记"处理中"；(2) 乐观并发：版本号；(3) 补偿幂等：补偿可以多次执行不出错。完全的 ACID 隔离 Saga 给不了。
N08
Kafka🟡
Kafka 的 ISR 和 HW 是什么？
ISR（In-Sync Replicas）= 与 Leader 足够同步的副本集合（最近 30s 内拉过数据且不落后超阈值）。HW（High Watermark）= ISR 成员里最小的 LEO。消费者只能读 HW 以下的数据，保证所有可见数据都已复制到 ISR。
N09
Kafka🔴
Kafka 怎么实现 Exactly-Once？
两个机制叠加：(1) 幂等 Producer（PID + Sequence Number，单分区不重复）；(2) 事务 API（跨分区原子写）。Consumer 设 isolation.level=read_committed。注意：EOS 只在 Kafka 内部生效，下游系统需自己保证幂等。
N10
分布式锁🔴
Redis 分布式锁的 fencing token 是干什么的？
解决GC 暂停导致双写：客户端 A 获锁后 GC 30s，锁过期被 B 获取。fencing token 是单调递增的版本号，每次操作资源带 token，资源系统拒绝旧 token。即使两个客户端都"以为"自己持锁，资源只接受新 token 的写入。

🗄️存储与缓存9 题

S01
B+ Tree🟢
为什么 MySQL 用 B+ Tree 不用 B Tree？
B+ Tree 三大优势：(1) 只在叶节点存数据，内部节点更小，树更矮（同样数据 3-4 层就能装下亿条）；(2) 叶节点双向链表，范围查询极快（找到起点后顺链表扫）；(3) 单次 IO 16KB 页能装更多键，减少 IO 次数。
S02
索引🟡
为什么主键不要用随机 UUID？
InnoDB 主键是聚簇索引（数据按主键顺序物理存储）。随机 UUID 插入位置随机 → 页分裂、随机 IO、碎片化（典型 50%+ 碎片率）。改用 AUTO_INCREMENT 或 UUID v7（时间前缀，趋势递增）→ 顺序插入到末尾，零页分裂。
S03
LSM🟡
LSM-Tree 的三大放大因子是什么？
写放大（WA） = 实际写 IO / 用户写量（LSM 高，因 Compaction）；读放大（RA） = 实际读 IO / 用户读量（LSM 中，因多层查找）；空间放大（SA） = 存储 / 用户数据（LSM 中，因旧版本暂存）。三者不可同时优，选型就是选哪个最重要。
S04
MVCC🔴
InnoDB 的 RR 能完全防幻读吗？
不完全。快照读（普通 SELECT）用 MVCC 防幻读；当前读（SELECT FOR UPDATE）用 Next-Key Lock 防幻读。混用（先快照读再当前读）可能看到幻象——快照里没的，当前读看到别的事务新插入的行。完全防需 Serializable。
S05
缓存🟢
缓存更新顺序：先更 DB 还是先删缓存？
先更 DB，再删缓存（Cache-Aside 模式）。删除比更新更安全——下次读 miss 后从 DB 加载新值。反例：先删缓存再更 DB，期间另一线程读到 DB 旧值并写回缓存 → 缓存还是旧值。最佳实践：CDC 监听 DB binlog 自动失效。
S06
缓存🟢
缓存穿透 / 击穿 / 雪崩区别？
穿透 = key 不存在于缓存和 DB，每次都查 DB（恶意攻击）→ 缓存空值/布隆过滤器；击穿 = 单个热点 key 过期，瞬间大量并发涌入 → 互斥锁/逻辑过期；雪崩 = 大量 key 同时过期/Redis 宕机 → 随机化 TTL + Redis HA。
S07
Redis🟡
Redis 为什么用跳表实现 ZSet？
Antirez 解释过：(1) 实现简单（~200 行 vs 红黑树 500+）；(2) 范围查询友好（L1 链表天然支持 ZRANGE）；(3) 缓存友好（内存访问模式线性）；(4) 更易扩展（层数可调）。性能上跳表和红黑树相同 O(log N)。
S08
Redis Cluster🟡
Redis Cluster 跨槽多 key 操作怎么办？
Redis Cluster 不支持跨槽事务/Lua。解决：用 {hashtag} 强制多个 key 到同一槽位。例：{user:1001}:profile 和 {user:1001}:orders 因 hashtag 部分相同，hash 落在同一 slot。
S09
索引🟡
深分页 LIMIT 1000000, 10 怎么优化？
两种方案：(1) 游标分页（推荐）：WHERE id > last_id LIMIT 10，O(1) 跳过；(2) 覆盖索引 + JOIN：子查询只扫覆盖索引（无回表）找 pk，主查询 10 次点查。禁止直接 LIMIT 1000000, 10 扫前 100 万行丢弃。

🔭可观测性6 题

O01
概念🟢
监控（Monitoring）和可观测性（Observability）有什么区别？
监控：预先知道会出什么问题，去看那些指标（known unknowns）。可观测性：无论出什么问题都能通过系统输出找到原因（unknown unknowns）。源自控制理论——系统是 observable 当且仅当能从外部输出推断内部状态。
O02
三支柱🟢
可观测性四支柱是什么？
Traces（发生了什么——请求路径）/ Metrics（多严重——数值聚合）/ Logs（细节——事件流）/ Profiles（慢在哪——代码热点）。OpenTelemetry 统一了前三个的数据模型和 SDK。
O03
Metric🟡
Histogram 和 Summary 哪个适合多副本？
Histogram。原因：Summary 在客户端预计算分位数，多副本的 P99 不可数学聚合（各实例 P99 = 50/60/70，合起来 ≠ 60 也 ≠ 70）。Histogram 存 bucket 计数（可加），用 histogram_quantile() 服务端聚合估算整体分位数。
O04
采样🟡
Tail 采样的核心优势？
在 Trace 完成后再决定是否采样，能100% 保留有价值的 Trace（错误 / 慢请求），其他均匀采样 1%。Head 采样在请求开始就决定，无法知道这条 Trace 是否"有趣"。代价：Tail 采样需缓存全部 Trace，需要更多内存。
O05
Tracing🟡
W3C TraceContext 的 traceparent 格式？
traceparent: 00-{traceId 32 hex}-{spanId 16 hex}-{flags 2 hex}。例：00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01。version=00；flags=01 表示 sampled。2020 年 W3C Recommendation，跨服务传播的事实标准。
O06
告警🔴
告警阈值怎么定？SLO 和 Error Budget 是什么？
不要凭感觉定 P99 阈值——基于 SLO（如可用性 99.9%）反推。Error Budget = 1 - SLO（如 0.1% 允许失败）。当 Error Budget 消耗速率超预期（如 7 天用了 30%）触发告警。这比硬阈值更合理，避免告警疲劳。

💡架构师软技能7 题

F01
角色定位🟢
架构师和 Tech Lead 的区别？
用决策影响半径 + 时间尺度区分：Tech Lead 影响一个服务/产品功能，时间尺度 1 季度；架构师影响多个服务/产品线，时间尺度 1-3 年。都需要技术深度，但架构师额外需要沟通和战略能力。
F02
决策框架🟡
面试官问"讲一个你做过的架构决策"，怎么答？
用 STAR + ADR 结构：(1) Situation：当时业务背景；(2) Task：要解决的问题（量化质量属性）；(3) Action：列出 2-3 个候选方案+取舍；(4) Result：选了哪个+实际效果。关键：讲清为什么"否决"其他选项。
F03
沟通🔴
怎么向业务方解释技术决策？
翻译成业务语言：(1) 用业务影响（用户体验、营收、风险）；(2) 用类比（"分布式锁就像图书馆的借阅卡"）；(3) 用数字（"P99 从 200ms 降到 50ms = 转化率提升 X%"）。避免：技术黑话、过度细节、emotional debate。
F04
技术选型🟡
Build vs Buy 怎么决策？
看四个维度：(1) 核心竞争力？（Core Domain → 自研）；(2) 团队能力？（不擅长 → 买）；(3) 成本？（人力 + 维护 vs 订阅）；(4) 定制需求？（标准化 → 买；定制重 → 自研）。陷阱：低估自研的长期维护成本（通常是开发成本的 3-5 倍）。
F05
技术债🔴
怎么向产品/老板争取偿还技术债？
把技术债翻译成业务影响：(1) 量化（"这块代码每次改动 5 人天，重构后 1 人天"）；(2) 风险化（"这个 bug 每月发生 1 次，重构后归零"）；(3) 速度化（"修了之后新功能开发提速 30%"）。20% 法则：每个 sprint 预留 20% 容量做技术债。
F06
演进策略🔴
遗留系统怎么演进？大重写 vs 渐进？
几乎总是渐进（Strangler Fig）。Joel Spolsky 名言："Rewriting from scratch is the single worst strategic mistake."。大重写的真实风险：(1) 期间业务停滞；(2) 新系统重复老 bug；(3) 三年才能赶上现有功能；(4) 团队大幅流失。例外：技术栈已死、必须迁移（如 Flash → HTML5）。
F07
影响力🔴
怎么让团队接受你的架构决策？
三个层次：(1) 权威（凭借头衔，最弱）；(2) 论证（用数据、ADR、ATAM 矩阵——架构师常用）；(3) 共创（让团队参与设计，最强）。Staff Engineer（Will Larson）：好架构师是"影响力放大器"，不是"决策机器"。
