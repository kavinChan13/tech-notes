01是什么 / 为什么

Google SRE 的复盘哲学："假设所有相关人员都做了在当时所知信息下的最佳决策"。

复盘的目的是找 "系统让人做错的机制"，而不是找一个倒霉蛋背锅。

有责怪 = 下次出事没人敢说真话 = 风险升级。
3 个必须从模板剔除的措辞
  • ❌ "XXX 应该 / 没有 / 失败了" → 改成 "流程允许 / 系统未阻止"
  • ❌ "如果 XXX 当时小心一点" → 改成 "工具 / checklist 应该把它变成不可能犯的错"
  • ❌ "人为失误" → 永远改成 "系统缺陷"，再追问 "是什么让人犯错"

02字段说明

段说明
Incident summary1 句话：何时 / 影响什么 / 持续多久
Impact客户 / 业务 / 收入 / SLA 量化
Timeline分钟级时间线，标 Detection / Mitigation / Resolution 三个 milestone
Root cause (5 Whys)连问 5 个为什么挖到机制层 — 不是"工程师没注意"
Contributing factors不是单一根因，是多个 contributing factor叠加
What went well不要只看错误 — detection 快不快 / response 顺不顺也要复盘
What went wrong对事不对人
Action items必须有 Owner + Due + Priority — 区分 4 类：Prevention / Detection / Response / Process
Lessons learned未来防范的"机制层"教训

📝填好示例

下面是一份完全虚构的示例（电商 SaaS 后台、为模板演示用），不基于任何真实事件、客户、同事。请按照自己团队的真实情境替换。

Postmortem (示例): Checkout API 5xx 错误率飙升 47 分钟

Incident ID: INC-EXAMPLE-031 · Severity: P1 · Status: Final
Author: Manager · Reviewers: Tech Lead, On-call Engineer, Director · Date: example date

1. Summary

示例日期 10:23-11:10（共 47 分钟），在一次例行配置推送后，Checkout API 错误率从基线 0.1% 上升到 27%，影响一部分用户的下单流程。后端数据库本身正常，但 Checkout 服务的配置层无法处理新配置，导致返回 500。

2. Impact

  • Customer impact: 约 12,000 个下单请求失败 / 用户重试后多数成功
  • Service impact: Checkout 写路径不稳定；读路径正常
  • Duration: 9 分钟（从故障到响应） · 总计 47 分钟
  • Business impact: 没有违反 SLA（容忍 15 分钟内的部分降级）；客户支持收到约 200 个工单
  • Detected by: 客户支持团队反馈，监控告警漏判

3. Timeline (UTC)

TimeEvent
10:18Engineer A 通过 config service 推送了一次例行 feature flag 更新
10:23Checkout API 5xx 比例开始上升
10:24内部 "5xx surge" 告警触发（默认配置为 P3 —— 第一个问题）
10:32客户支持收到大量工单，电话联系工程团队
10:35Engineer A 和 on-call EM 加入事故桥
10:47初步假设：新配置在 commit 阶段触发了 schema 校验异常
10:58Mitigation：在 1 个分片上回滚配置，确认能恢复
11:05Mitigation 推广到所有分片
11:10所有分片错误率回到基线
11:30解除事故状态，关闭事故频道

4. Root cause (5 Whys)

  1. 为什么 Checkout 大量 5xx？→ 一次配置更新在 commit 时触发了配置服务的未处理异常。
  2. 为什么配置更新会触发异常？→ 新增的一个 feature flag 与默认值处理逻辑冲突，抛出 NullPointerException。
  3. 为什么 NPE 会传播到 API 层？→ 异常在 commit 代码里被捕获，但在 pre-commit 的 dry-run 校验里没有被捕获；commit 代码又把异常重新抛出。
  4. 为什么只有一条路径捕获了异常？→ dry-run 校验和 commit 这两条代码路径上，防御性编程模式不一致。
  5. 为什么这个不一致没被发现？→ 代码层面没有任何检查能确保校验与 commit 两条路径的异常处理对称；CI 也没有测试 "dry-run 通过但 commit 失败" 这个场景。

5. Contributing factors

  • "5xx 突增 ≥ 5×" 这个告警被错误配置成 P3（应该是 P1）
  • 客户支持比我们自己的告警升级流程快了 9 分钟联系到我们
  • 这次配置改动所在的分支测试覆盖率很低（3 个单元测试，没有 e2e）

6. What went well

  • 一旦加入事故桥，12 分钟内定位到根因
  • Mitigation（配置回滚）在所有分片上执行得非常顺畅
  • 客户支持和工程的协作很顺（平时关系打底有用）
  • 读路径完全没受影响 — 写读分离的设计起作用了

7. What went wrong

  • 告警等级配错成 P3 让事故被忽略了 9 分钟
  • 我们是从客户支持那里知道事故的，不是从自己的监控
  • 这个配置改动通过了 pre-commit 检查，本不该通过
  • "Checkout 大面积 5xx" 这种场景没有任何 runbook

8. Action items

#ActionTypeOwnerDuePriStatus
1把 "5xx 突增 ≥ 5×" 告警重新归为 P1（自动 page）DetectionSRE Lead+1 周P0Done
2新增 e2e CI 测试："dry-run 通过 + commit 失败" 这条路径PreventionTech Lead+3 周P0In progress
3统一 validation / commit 两条路径上的异常处理模式PreventionTech Lead+4 周P0In progress
4编写 runbook "Checkout 大面积 5xx"ResponseEngineer B+2 周P1Done
5建立 "故障超过 30 分钟 = 触发 SEV review" 机制ProcessManager+3 周P1Open
6配置改动 checklist 必须包含 "validation/commit 对称性" 这一项ProcessTech Lead + PM+2 周P2Open

9. Lessons learned

  • 客户比我们自己的监控更快发现故障 — 这是最大的教训。需要一个不依赖告警等级的 "事故经过时长" 兜底机制。
  • "看起来像在处理错误" 的代码，在不同代码路径上的实际处理可能完全不一致。需要靠 Linter / CI 来强制对称性。
  • 读写隔离 / 配置层与数据层隔离在这次事故里完全起作用 — 这是后续架构 review 时要继续保留的设计原则。

10. Open questions

  • 配置服务是否应该有 circuit breaker，在连续 N 个失败 commit 后自动回退到上一份已知良好配置？（由 Architect 在下季度评估）

⚖️DO / DON'T

DO

  • 对事不对人 — 永远用"流程/工具/系统"，不用"某某没做"
  • 必有 Action Items + Owner + Due + Priority，下次 retro 先 review
  • 区分 Prevention / Detection / Response / Process 4 类 action
  • 包含 "What went well" — 鼓励团队复制好做法
  • Public 给全公司 — 失败的传播比成功更值钱
  • 5 Why 挖到机制层，不要停在"工程师没注意"

DON'T

  • "张三推了错配置导致..." — 致命 blameful 措辞
  • "以后大家小心点" — 0 行 Action Item，废纸
  • 把 P0 复盘藏起来不让其他团队看 — 失去最大学习价值
  • 用 "人为失误" 当根因 — 它永远是表象，不是根因
  • Action items 写 "continue monitoring" — 0 动作
  • 复盘开完不跟进 action — 下次同样事故 100% 重演
