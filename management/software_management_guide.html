关键洞察： 在大型平台软件公司，EM = TL + SM + 部分 TechL + 资源 / 招聘 / 绩效。 这意味着你作为 SM 想要转 EM，最大的缺口往往不是"管理"本身，而是技术决策权威性和人员发展能力。 如果你在外企，"Line Manager" 通常约等于 EM；在国内大厂，"研发经理 / 技术经理" 也基本指 EM。

1.1 九大管理岗位详解

下面分别从核心职责 · 日常工作 · KPI · 能力要求 · 与开发/PM 的区别 · 大厂定位 · 晋升路径展开。

① Team Lead（团队负责人）

核心职责
PEOPLE
通常带 5~10 人 的一线小组，对小组的日常产出、节奏、氛围负责。 一手抓任务分配，一手抓 1on1 与新人成长，是"最贴近工程师"的管理岗。
日常工作
  • 每日 standup、任务排期、blocker 跟踪
  • Code review、设计评审参与
  • 1on1（双周/每周一次）、绩效初评
  • 新员工 onboarding、Mentor 配对
KPI / OKR
  • 小组 Sprint Velocity 与 commitment 命中率
  • Bug Escape Rate（漏到 SIT/客户的缺陷比）
  • Team eNPS / 离职率（< 8%）
  • 新人转正成功率、晋升通过率
能力要求
仍需扎实技术功底（约 60% 时间能动手）+ 基本的 people skills。 与开发的最大区别：你要为"小组的产出"而非"你个人的产出"负责。 与 PM 的区别：你管"人和代码"，PM 管"计划和里程碑"。

② Tech Lead（技术负责人）

核心职责
TECH
对技术方案、架构演进、质量、稳定性负总责。不一定带人（不一定是 Line Manager），但对所有人的代码与设计有否决权。
日常工作
  • 架构评审、关键模块设计、技术选型
  • 主导难题攻坚（性能、稳定性、跨模块联调）
  • 制定 coding guideline、CI/CD 策略
  • 对外的技术接口人
KPI
  • 架构演进的关键里程碑达成
  • 系统 SLI / 稳定性指标（P99、可用性）
  • 技术债务清理量、规范落地度
  • 关键人才（架构师/Senior）培养数
能力要求
通常是团队里技术最强者之一。在大型软件公司体系下，Tech Lead 经常对应"Architect" 或 "Principal Engineer"。 与 EM 的区别：TL 对"技术"负责，EM 对"团队和交付"负责；同一个产品线常常二者并存（"双 Lead" 模式）。

③ Scrum Master（你目前的位置）

核心职责
AGILE
守护团队的敏捷流程与协作健康度。是 servant leader，不直接给任务、不打绩效； 但负责移除 blocker、推动 retrospective、保护团队不被随意打断。
日常工作
  • 主持 Planning / Standup / Review / Retro
  • 维护 Sprint board、Burndown chart
  • 跨团队同步、PO 协调
  • 识别 impediment、推动 process improvement
KPI
  • Sprint commitment 命中率（>= 85%）
  • Cycle time / Lead time 缩短趋势
  • Retro action 闭环率
  • 团队成熟度（Shu-Ha-Ri 阶段提升）
与 EM 的最大区别 ⚠
SM 没有人事权（不给绩效、不签字招聘、不调薪）。 正因为没有 authority，所以 SM 优秀的 影响力 / 协调 / 流程设计 能力，是转 EM 时极宝贵的资产。 但若仅停留在"主持会议 + 维护 board"，则会被认为"管理深度不够"。

④ Engineering Manager（你要瞄准的核心目标）

核心职责
PEOPLE + DELIVERY + TECH
对一个工程团队（10~30 人）的所有事情兜底：交付、质量、人员、技术方向、招聘、绩效、跨团队合作。 是"Line Manager"，下属向你直线汇报。
日常工作
  • 50% — 1on1 / 绩效 / 招聘 / 冲突
  • 25% — 跨团队协调、向上汇报、规划
  • 15% — 技术决策、架构 review
  • 10% — 流程/工具改进
KPI
  • 交付：季度/版本目标达成率
  • 质量：线上事故数、SLA 达成
  • 人员：主动离职率、晋升数、eNPS
  • 组织：招聘漏斗、岗位填充率
  • 技术：架构里程碑、技术债下降
能力要求
必须同时具备：① 技术判断力（不必最强，但要能 challenge 设计）、 ② 人员发展能力（识人、用人、育人、退人）、 ③ 交付能力（项目管理、风险预警）、 ④ 跨团队影响力（向上 / 向横）。 可以理解为 EM = 0.4 × TL + 0.3 × SM + 0.2 × TechL + 0.1 × PM。

⑤ Delivery Manager（交付经理）

核心职责
DELIVERY
对"按时、按质、按预算把产品交到客户/下游"负责。 在 To-B 大型软件厂商里，Delivery Manager 常对接客户项目（如某重点客户的部署），是客户视角的项目主。
日常工作
  • 客户/PO 沟通、需求确认、变更评估
  • 多团队进度集成、依赖管理
  • 风险登记、escalation
  • 验收、试点、版本发布协调
与 EM/PM 的区别
EM 管"内"（团队），DM 管"外"（客户与跨团队交付）。 PM 管"单个项目"，DM 管"持续向客户交付的产品 / 服务"，往往持续多个版本。 DM 不一定有 Line Manager 权限，但需要有非常强的 stakeholder management 能力。
SM 转 DM 是较容易的路径 ✓
如果你暂时拿不到 EM offer，Delivery Manager 是 SM 转管理的最自然桥梁。 原因：你已具备多团队协调、风险跟踪、跨职能沟通经验，只需补充客户视角和商业敏感度即可。

⑥⑦⑧ PM / Program Manager / Agile Coach

Project Manager
PROJECT
管单个项目的范围、时间、成本、质量（PMBOK 经典四要素）。 国内大厂偏甲方/集成商，外企更常作为 PMO 角色。 典型 KPI：里程碑达成、预算偏差、风险闭环。
Program Manager
PROGRAM
管多个相关项目的项目群（如"平台核心组件 2026 版本"包含 10+ 个子项目）。 在 Google/Meta 称 TPM（Technical Program Manager），技术要求更高，对架构有 say。
Agile Coach
COACH
"教练角色"，跨多个团队 / 部门推动敏捷转型。 通常服务于 10+ 团队，关注组织级敏捷（SAFe / LeSS / Scrum@Scale）。 是 SM 的"放大版"，但不直接带兵。

⑨ R&D Manager（研发总监 / 部门经理）

核心职责
STRATEGY
管多个 EM，团队规模 50~200 人。重点不再是单个项目交付， 而是组织设计、人才梯队、技术战略、预算、与产品/销售层面对接。
关键转变
  • 从"管事"变为"管管理者"
  • 从"季度 KPI"变为"年度 / 多年战略"
  • 不再直接接触代码，但要看得懂架构图与风险
  • 开始要懂一点财务（HC / 预算 / 折旧）

2.1 人员管理（People Management）

"你的成果 = 你团队的成果。"
从 IC/SM 转管理后，最大的认知反转是：你的产出不再来自你的双手，而来自你能让多少人发挥多大价值。 所以 people management 不是"软技能"，而是 EM 的核心硬技能。

2.1.1 1on1：管理者最被低估的工具

什么是好的 1on1
  • 员工的会议，不是你的：议题由下属设
  • 固定节奏：双周 30~45 min，不轻易取消
  • 不谈具体任务进度（去 standup 谈）
  • 聊 career / blocker / feedback / wellbeing
  • 建立"私下吐槽通道"，避免问题烂在心里
1on1 经典问题清单
  1. "最近一周你最 frustrated 的事是什么？"
  2. "如果你是 manager，你会改变什么？"
  3. "我能为你移除什么 blocker？"
  4. "你目前的工作和你的 career 目标对齐吗？"
  5. "对我（manager）的 feedback 是什么？"
大型平台研发场景实操： 通信行业团队往往技术导向、内向居多，沉默不代表没问题。 建议在 1on1 第一次就明说："这个会不汇报进度，是你的会。" 并用 Skip-level 1on1（隔级 1on1，每季度一次）形成双通道，避免直属经理过滤信息。

2.1.2 绩效管理：从主观感觉到客观证据

绩效管理失败的两大根源：① 全年只评一次（没有 continuous feedback）、② 评价依赖 manager 的"印象"而非证据。

2.1.8 PIP & 退人：管理者最难的一课

正确做法

2.2 项目管理（Project Management）

项目管理的本质是"在不确定性下做承诺"。 作为 SM，你已经很懂"过程级"的项目管理；下面我们扩展到"结果级"的项目管理 —— 这是 EM/DM 与 SM 的关键差距。

2.2.1 项目方法论选型

方法适用场景优势陷阱
Scrum需求变化中等、特性团队节奏稳定、可预测仪式过多、对突发响应慢
Kanban运维 / SRE / 高变化运营WIP 控制、流动性缺少节奏、估时困难
Waterfall合规 / 硬件 / 协议栈认证文档完整、可追溯变更成本极高
Hybrid Agile大型 To-B 项目 ✓外层 Waterfall 节奏 + 内层 Scrum需要明确 sync 节点
SAFe / LeSS多团队（>50 人）协同规模化、对齐 PI Planning过程开销巨大
大型 To-B 场景：绝大多数项目实际跑的是 Hybrid Agile —— 外面是客户合同的瀑布里程碑（GA、RFA、RFC），里面每个团队跑 Scrum。EM 的关键是把"外瀑布"翻译成"内 Sprint"，并管理两层之间的 buffer。

2.2.2 估时（Estimation）的本质

大多数估时不准，不是因为开发员"不会估"，而是因为问的是错误的问题："还要多久？" —— 这个问题预设了"未来可知"。

  • 相对估时 vs 绝对估时：Story Point 是"相对复杂度 + 不确定性 + 工作量"的综合，不是"小时"
  • 三点估时：Optimistic + 4×Most Likely + Pessimistic / 6 (PERT 公式)
  • 历史 Velocity：用过去 3~6 个 Sprint 的均值预测，比"问一遍开发"准 3 倍
  • "按团队估，不按个人估"：避免依赖某个超人
  • Buffer 不可耻：对外承诺时加 20~30% buffer 是 professional，不是不专业
Hofstadter 定律："It always takes longer than you expect, even when you take into account Hofstadter's Law." 永远会比你以为的久。EM 的工作不是消灭这种不确定，而是在估时基础上设计安全垫。

风险管理 4 步法（Risk Register）：

  1. 识别 Identify：每个 Sprint planning 抽 15 min 集体 brainstorm
  2. 评估 Assess：标注概率（H/M/L）× 影响（H/M/L）

2.2.4 项目失控：为什么 & 如何干预

失控的根因不是"出问题"，而是"问题被隐藏"。 团队不上报，往往是因为 manager 之前对坏消息的反应不好（指责、加班、推责任），形成了信息黑洞。

项目失控的 5 个早期信号（EM 应像看仪表盘一样监控）：

① Velocity 连续两 Sprint 下降 > 20%
不是个例失败，而是系统性问题（依赖/技术债/士气）。
② Burndown 形状异常
Sprint 中后期才"突然完成"= 缺 daily merge；前期高、后期停 = 范围 underestimate。
③ Bug 增速 > 修复增速
缺陷在堆积 → 一定会有"集成阶段大坑"。
④ Standup 越来越短 / 越来越长
变短：大家不愿暴露；变长：blocker 在堆积无人 own。

EM 干预 SOP：

  1. 第一步：不打断节奏。先和 SM/Tech Lead 私下对齐数据
  2. 第二步：开 Get Well Plan 会议（不是 retro，目标是"如何恢复绿灯"）
  3. 第三步：决定动作 —— 减范围 / 加 buffer / 补人手 / 升级 escalation
  4. 第四步：向上汇报，主动暴露问题（不要等被发现）
  5. 第五步：固定 cadence 跟踪 recovery，直到恢复绿灯

2.2.5 延期处理：Brooks 定律与正确响应

Brooks's Law："Adding manpower to a late software project makes it later." 给晚期项目加人只会更晚 —— 因为：① 新人 onboarding 占老人时间 ② 沟通成本指数增加 ③ 任务难以再切分。

正确的延期响应优先级（从优到劣）：

  1. 减范围 Cut Scope（80% 场景适用）
  2. 分阶段发布 Phased Release
  3. 调质量标准（仅限非关键路径）
  4. 加时间 Move Deadline（先承认现实）
  5. 加人 Add People（最后选项，且新人做边缘任务）

2.3 技术管理（Technical Management）

作为 EM，你不需要是最强的 coder，但必须能听懂、能 challenge、能拍板。 以下技术管理主题，是 EM 面试和实战中最容易暴露"是否真懂研发"的地方。

2.3.1 技术债：一个被滥用、却几乎没人真正管理的概念

"技术债"原本（Ward Cunningham）指"为了快速交付而有意识做的次优选择，并明确知道未来要还"。 现在被滥用为"我不喜欢的代码"。EM 必须区分：

真技术债 ✓
  • 有意识、有记录
  • 有偿还计划
  • 影响可量化（性能、稳定性、扩展性）
"伪技术债" ✗
  • "代码风格我不喜欢"
  • "用了过时框架但还能跑"
  • "想重写一遍"

技术债配额制：建议每个 Sprint 固定 15~20% 容量用于技术债 / 重构 / 工具改进。 写进 capacity planning，从 PO 那里"买走"这部分容量。

2.3.3 Code Review：文化指标

Code Review 的本质不是"找 bug"，而是知识共享 + 团队代码风格统一 + 质量第二道关。EM 应该关注的 CR 指标：

  • 平均 review 响应时间 < 4 小时（工作时间内）
  • PR size 中位数 < 400 行（超大 PR 是 review 失效的信号）
  • Review 通过率（一次通过 vs 多次返工）
  • 谁在 review 谁 —— 避免"只有 senior review junior"的单向流动

DORA 四大指标是 EM 必须知道的"工程效率仪表盘"：

2.3.5 质量体系 & 稳定性治理

从"被动救火"到"主动可靠"：

  • SLI / SLO / SLA：把"稳定性"量化为可承诺的指标
  • Error Budget："允许多少不稳定"，给特性开发和稳定性投入提供平衡杠杆
  • 无责 Postmortem：每个 P0/P1 事故必有 RCA + Action，禁止 finger-pointing
  • Chaos Engineering：主动注入故障验证韧性（如模拟客户侧网络抖动）

2.3.6 工程效率（Developer Productivity Engineering）

近几年，DPE 成为大厂 EM 的新关注点。核心思想：把"工程师的时间"当作最贵的资源来优化。

  • 本地编译 / CI 跑通时间
  • 新人到第一个 PR 合并的时间（< 5 天为优秀）
  • 会议占工程师时间比例（< 25%）
  • 工具链流畅度（构建、调试、可观测）

2.4 敏捷管理深度解析

这是你最熟的领域，但作为未来 EM，需要从"流程执行者"升级到"敏捷哲学的解读者"。 以下是面试官最爱深挖的几个反直觉问题。

2.4.1 Scrum 真正核心：经验主义而非仪式

Scrum 的本质是 Empirical Process Control（经验过程控制）： 透明（Transparency） · 检查（Inspection） · 适应（Adaptation）。
所有 Scrum 仪式都只是手段，不是目的。如果你的 standup 没有让任何"检查与适应"发生，那它就是浪费。

2.4.2 为什么 80% 的 Scrum 团队失败

失败不是因为"Scrum 不好"，而是因为只学了形式没学精神。Mike Cohn 把这种情况叫 "Cargo Cult Scrum"。

典型失败模式
  • Daily 变汇报会（向 manager 汇报，不是团队 sync）
  • Retro 只吐槽，没 action（或 action 无人 own）
  • PO 不参与 Sprint，事后才抱怨
  • Story Point 被"翻译"成小时考核个人
  • Sprint 中途老板加需求，被迫接
  • "敏捷转型"自上而下硬推，团队抵触
SM 升级路径（你需要做的）
  • 从"主持仪式"到"教练对话 (powerful questions)"
  • 从"维护 board"到"诊断团队成熟度"（Shu-Ha-Ri）
  • 从"protect 团队"到"赋能团队 self-management"
  • 从"单团队"到"组织级敏捷设计"
  • 开始量化你的影响（Velocity、CFR、Lead Time）

2.4.3 Story Point 的本质：相对复杂度

常见错误：把 1SP = 4 小时这种"汇率"建立起来。一旦如此，SP 就退化成小时，且 manager 会用它考核个人产出 → 必然 sandbagging。

正确理解：SP 是相对的复杂度估算，包含工作量、不确定性、风险。它是团队级的预测工具，不是个人考核工具。

2.4.4 Retrospective：从仪式到改进引擎

Retro 失败的根因："问题被讨论但没改变"。下面是 6 步深度 Retro 框架：

Step 1 · 5 min
Set the Stage

用一句话开场，明确本次 retro 焦点（不是泛泛而谈）。例如："本次只聚焦上 Sprint 的延期"。

Step 2 · 10 min
Gather Data

Timeline / Mad-Sad-Glad / Start-Stop-Continue。重点是事实，不是判断。

Step 3 · 15 min
Generate Insights

用 5 Whys 或 Fishbone 分析根因。"为什么延期？" → "因为依赖晚到" → "为什么晚到？" ...

Step 4 · 10 min
Decide What to Do

每个 action 必须有 owner + deadline + 验收标准，否则不算 action。

Step 5 · 5 min
Close

ROTI（Return on Time Invested）评分。低分要 retro 这个 retro。

Step 6 · 持续
Follow-up

下一次 retro 第一件事：review 上次 action。这是闭环的关键，90% 团队漏掉这一步。

2.4.5 Velocity 提升：可持续节奏 vs 短期冲刺

红线：Velocity 不能作为绩效 KPI。一旦如此，团队会立刻通胀 SP，数字好看但实际更慢。 正确做法：监控 Velocity 的稳定性（变异系数），而不是绝对值。

真正能提升 Velocity 的杠杆：

  • 减少在制品 (WIP) → 减少 context switch
  • 提升测试自动化覆盖 → 减少手工回归时间
  • 明确 Definition of Ready / Done → 减少返工
  • 砍掉无价值会议 → 释放工程师 deep work 时间
  • 优化 CI 反馈循环 → 减少等待

2.4.6 Sprint Planning 的两层结构

Part 1 · What
PO-LED

SENIOR
"看到大家没看到的雷"。需要主动建立 risk register，每周 review。 通信行业特别重要：客户合同 SLA 违约风险、互通测试风险、合规风险。
⑧ Ownership / 担责
CORE
"This is mine." 即使不是你直接的错，但只要在你的范围里，就要扛起来。 判别：出了事第一反应是"how do we fix it"还是"who did it"？
⑨ 目标拆解
SENIOR
把"2026 年要发布平台核心组件 GA 版本"这种大目标，逐层拆为：年度 OKR → 季度 KR → Epic → Story → Task。 每一层都可验证、可追溯。
⑩ 汇报 / 写作
SENIOR
"If you can't write it, you don't understand it." Amazon 6-pager 文化、Stripe writing culture 都说明：高层 manager 的影响力来自书面表达。 练习：每周一份团队 weekly，每季度一份业务回顾。
⑪ PPT / 演讲
CORE
重要不在于"画得好看"，而在于结构化思考 → 视觉化呈现。 最低标准：能用 5 页 PPT 讲清楚一个季度的 progress / risk / ask。
⑫ 会议管理
CORE
每个会都有：明确 agenda + 必到人员 + 时长 + outcome。没有 outcome 的会都该砍掉。 EM 升级版：能识别哪些会是"真正需要同步的"，哪些可以用文档异步解决。
⑬ 冲突处理
CORE
Thomas-Kilmann 5 种模式 + 非暴力沟通 (NVC) 四步法：观察 - 感受 - 需要 - 请求。 高阶：能让冲突双方自己达成共识，而不是你裁判。
⑭ 战略思维
DIRECTOR
从"这个季度交付什么"上升到"未来 2 年我们的护城河是什么"。 EM 起步阶段不要求，但向 Senior EM / Director 晋升的核心分水岭。

3.2 "优秀 Manager" vs "普通 Manager" 的真实差距

这一节是我观察过 50+ 个 manager 后总结的"分水岭行为"。能力分数都差不多，但行为习惯天差地别。

优秀 Manager 的 10 个习惯
  1. 把坏消息当礼物：team 越早暴露，自己越早干预
  2. 开会前必有 written agenda（30 秒看完）
  3. 1on1 雷打不动，从不取消
  4. 每周固定时间 deep work（不被会议吞噬）
  5. 给反馈具体到行为 + 影响 + 期望（SBI）
  6. 夸奖在公开场合，批评在私下
  7. 会议结束前 1 min 复述 action（"所以我们达成的是…"）
  8. 主动管理向上沟通 cadence（不是被问才说）
  9. 承认自己不知道（"我不确定，我去查一下再回你"）
  10. 对低绩效 fast & humane（不是慢慢拖）
普通 Manager 的 10 个陷阱
  1. 当"团队消防员"，自己亲自下场救火，团队不成长
  2. 开会无 agenda，靠"看大家想聊啥"
  3. 1on1 经常因为"急事"被取消
  4. 日程被会议填满，没有思考时间
  5. 反馈含糊（"你最近不太行"）
  6. 当众批评，被批评者从此 disengage
  7. 会议结束后大家各自理解不同
  8. 等到老板问才上报问题
  9. 不懂装懂，被工程师识破
  10. 对低绩效"再观察看看"，半年没动作
核心心智： 从 SM 转 EM 最难的认知跳跃，不是"学新技能"，而是"承认自己要为别人的产出负责"。 当一个团队成员表现不好，普通 manager 想的是"他怎么这么差"，优秀 manager 想的是"我没招对人 / 没给对反馈 / 没移除 blocker"。 这种 ownership 心态，是面试官最在意的"manager material"。

4.1 STAR 答题模板（Master Template）

S — Situation
15%
背景 1~2 句话。不超过 30 秒。说清楚：什么团队、什么阶段、什么挑战。
T — Task
10%
你要解决的具体任务。突出"你"的角色，而不是团队的。
A — Action
60%
最重要的部分。"我"做了什么。包含：选项分析、决策依据、推进步骤、过程中的调整。
R — Result
15%
用数字！"Velocity 提升 30%"、"延期从 2 周降到 3 天"、"线上事故下降 50%"。加上 lessons learned 加分。
3 个最容易踩的坑：
  1. "我们团队 ..." 听起来谦虚，但面试官想听"我"。要区分"我做的"和"团队做的"。
  2. Action 讲成"流程"而非"决策"。重点是"为什么选 A 而非 B"。
  3. 没有 Result，或 Result 只有形容词没有数字。EM 面试官只信数字。

4.2 30+ 高频面试题（含高分/低分对照）

A. 人员管理类
Q1. 你如何处理低绩效员工？
面试官真正想看的
  • 你有没有区分能力问题 vs 意愿问题
  • 有没有及时、具体的反馈
  • 是否会用文档化的 improvement plan
  • 必要时能不能 fast & humane 地 let go
✓ 高分回答（STAR）

"在我接手某团队 2 个月时，发现 A 同学连续 3 个 Sprint commit 偏低、PR 多次返工。 我先和他 1on1 区分原因：是能力跟不上还是缺乏支持？ 通过 2 周观察 + skip-level 1on1，我判断主因是缺乏明确目标与 mentor，而非态度问题。 于是：① 立即给他配一个 senior 做 mentor，每周 30 min 技术辅导； ② 我和他共同定 30/60/90 plan，每周对齐； ③ 同步 HR 备案，做好 improvement plan 文档化。 60 天后，他的 PR 通过率从 40% 升到 85%，主动提出了 OPS 工具改进。 这件事让我意识到：低绩效往往是系统问题，而非个人问题。"

✗ 低分回答

"我会先和他谈谈，然后给他一些时间，如果还不行就反映给 HR。" — 太流程化、没有诊断、没有数据、没有 ownership。

Q2. 如何处理高绩效员工 / 防止他离职？
面试官真正想看的
  • 你识别高潜的能力
  • 是否真心关注下属的 career growth（而不是只想"用人"）
  • 能否给挑战、给空间、给认可
✓ 高分回答

"高绩效员工最怕的不是工作累，是无聊和被忽视。我会做三件事：
① 给他够大的项目：让他 own 一个 cross-team 的方向，而不是只交付 story；
② 主动 sponsor 他的可见度：在向上汇报和跨团队会议上给他发言机会；
③ 定期讨论 career path：不要等他主动提辞职。 在我上一个团队，我们有一位 P5 同学，我推他做了系统的 SRE 项目并在 R&D 全员大会上分享，6 个月后他晋升 P6。"

Q3. 团队两个 senior 工程师在技术方案上严重分歧，你怎么处理？
✓ 高分回答（关键是不替他们决策）

"我会分四步：
① 分开 1on1：先理解每个人真实立场（往往技术分歧背后是对客户场景理解不同）；
② 组织 design review：要求双方各自写 1 页对比文档，列出假设、权衡、退出条件；
③ 引入第三方：找另一个团队的架构师做 reviewer，让大家 focus 在数据上而非个人；
④ 如仍无共识，由 Tech Lead 拍板，但要明确"基于什么假设拍的"+ "什么条件下会重新评估"。 最重要的是：让双方都觉得自己被认真听了，即使最终方案没采纳。"

✗ 低分回答

"我会听双方意见然后决定一个方案。" — 完全没体现影响力和过程管理。

Q4. 团队氛围不好、士气低，你怎么诊断？
✓ 高分回答

"我用三个层次诊断：
① 现象层：Velocity / 离职率 / 1on1 反馈 / retro action 完成率；
② 互动层：standup 沉默时长、PR 评论质量、谁不再主动提议；
③ 根因层：信任缺失 (Lencioni Level 1)？目标不清？感觉不公平？老板"消失"？
诊断完后，不要急着上动作，先和团队 share 你看到了什么，让他们参与定义问题——这本身就是修复信任的开始。"

Q5. 你是怎么做 1on1 的？多久一次？
✓ 高分要点
  • 固定 cadence：每人每 2 周一次，30~45 min，不轻易取消
  • 员工议题为主，不是进度汇报
  • 必谈三件事：Wellbeing / Blockers / Career
  • 每季度一次 career conversation（> 1 小时，专门聊成长）
  • 记录 follow-up，下次 1on1 复盘
  • 对 skip-level 也做 1on1（每季度一次）
Q6. 怎么招到合适的人？你如何 set hiring bar？
✓ 高分要点

"我把 hiring bar 定义为：这个人加入后，让团队整体水平上升而不是稀释。具体三条：
① 结构化面试：固定题目矩阵 + scorecard，避免个人偏好；
② 多人独立打分 + debrief：每个面试官 written first，避免 anchoring；
③ 明确 must-have vs nice-to-have：宁缺勿滥。 我面试中一定会问的：'告诉我一个你失败的项目'—— 能反思失败的人，才能在 manager 反馈中成长。"

B. 项目交付类
Q7. 项目要延期了，你怎么向老板汇报？
面试官想看的
  • 你不会等到最后一刻才说
  • 你不是简单 escalate 问题，而是带着方案来
  • 你知道老板的"真正问题"是什么（客户、合规、收入？）
✓ 高分回答

"我用一个 'Bad News Memo' 框架：
1️⃣ What changed：原计划 vs 现在的判断，用具体日期/数字；
2️⃣ Why：根因（不是借口）；
3️⃣ Impact：客户 / 财务 / 团队影响；
4️⃣ Options：至少给 3 个方案 + 推荐项（带 trade-off）；
5️⃣ Ask：明确需要老板做什么决策。
关键是：越早越好。让老板感到自己被尊重（提前知道），而不是被 surprise。"

Q8. 跨团队依赖卡住了你的项目，怎么推动？
✓ 高分回答

"影响力 without authority 的经典场景。我的 SOP：
① 先 understand：去对方团队 standup 旁听一周，理解他们的优先级和压力；
② 找共同利益：把我们的需求包装为对他们也有价值（如：你帮我，我帮你抢一个 share resource）；
③ 正式化承诺：写到双方共同 Jira / Confluence，有 owner、deadline；
④ 建立 weekly sync，主动 follow-up，不靠"我催他"；
⑤ 升级机制透明：如果一周没进展，提前说"下周我会 escalate 给 Director"，给对方留面子。"

Q9. 团队估时永远偏低，每个 Sprint 都做不完，怎么办？
✓ 高分回答

"先诊断根因，常见 4 种：
① SP 在通胀/缩水：用历史 Velocity 验证；
② 需求不清：DoR 不严，开始做了才发现复杂；
③ 外部干扰多：被支持/救火/会议挤占；
④ 团队怕老板：不敢拒绝，硬塞 story。
不同根因不同药方：
① → 重新校准 SP（找 3 个 reference story 锚定）；
② → 强化 DoR：必须有 AC + Mock + 接口约定；
③ → 把 unplanned work 单独 track，让 PO 看到真实容量；
④ → 这是 manager 的责任：和 PO 重新约定 commitment 文化。"

Q10. 你如何做优先级管理？
✓ 高分要点

"我用两层框架：
战略层：和 PO/老板对齐三个北极星指标（business / quality / team），季度 review；
战术层：每个 Sprint 用 RICE = Reach × Impact × Confidence / Effort 排序。
关键原则："不是所有事都重要"。我会强迫团队做 'Stop Doing List'—— 砍掉的比加进来的更重要。"

Q11. 上线后出了 P0 事故，你作为 manager 第一时间做什么？
✓ 高分回答

"四个阶段：
① 0~30 分钟 — Recover：先恢复，不追责；指定 IC（Incident Commander），其他人执行；
② 30 min ~ 4 小时 — Communicate：客户/老板/相关团队同步状态，固定 cadence；
③ 事后 24 小时 — Postmortem：5 Whys，无责，专注根因和 action；
④ 一周内 — Follow-through：所有 action 写进 Sprint，公开 tracking。
EM 不下场写代码，而是确保：决策清晰 + 信息透明 + 团队不崩溃。"

C. 跨团队 / 向上类
Q12. 你怎么向老板汇报？多久一次？
✓ 高分要点
  • 三个 cadence：每周 written update（4~6 项 bullet）+ 双周 1on1（30 min）+ 季度 business review（1h+）
  • 三层信息：Progress（已完成）/ Next（即将开始）/ Risks & Asks（需要他什么）
  • 不要 surprise：坏消息要提前 heads-up，不要让老板从别处听说
  • 写在 doc 里，不是只在会上口头说 — 老板可以转给他老板
Q13. 你如何 say no？
✓ 高分回答

"我从来不直接说 No，而是说 'Yes, and here's the trade-off'：
① 把请求转译成机会成本：'如果做 X，意味着 Y 要推后 2 周'；
② 把决策权交回给请求方：'请你帮我们 prioritize'；
③ 把'我'变成'我们'：'我们的容量目前已满，要一起决定怎么平衡'。
这样一来，每个 Yes 都是有意识的选择，而不是无脑接活。"

Q14. 你如何推动流程改进 / 引入新工具？
✓ 高分要点
  • 先找痛点，不是先找方案：和团队一起定义"什么不爽"
  • 小范围试点：1 个 sprint，1 个 team，metric 明确
  • 团队 own 改进，不是 manager 强推
  • 试点成功后讲故事给其他团队，让他们自己来要
  • 预留回滚机制：流程不是金科玉律
D. 技术与业务平衡
Q15. PO 一直要新特性，团队想还技术债，你怎么平衡？
✓ 高分回答

"我不会让团队和 PO 对立。三个动作：
① 量化技术债的业务影响：把'代码烂'翻译成'每次发版多花 3 天 + bug 率 +20%'，PO 听得懂；
② capacity 配额制：和 PO 谈判每 Sprint 固定 15~20% 容量做技术债 / 工具改进；
③ 把大债拆为小债，混在特性 story 里逐步还，不要做大爆炸式重构。
一年下来，我们的 deploy lead time 从 5 天降到 1 天，PO 也认可了这种平衡。"

Q16. 你团队的 senior 工程师建议大规模重构，你怎么决策？
✓ 高分回答

"我会先 challenge 他三件事，不是反对，而是帮他想清楚：
① 目标：重构后我们能做什么是现在做不到的？能否用 metric 衡量？
② 风险：重构期间是否影响交付？回滚成本？
③ 替代方案：能不能 strangler pattern 逐步替换，而不是 big bang？
如果三个问题答得过去，我会支持，但要求每 2 周一个 milestone + go/no-go 检查。否则就缓推。"

Q17. 客户要一个不合理的需求，你如何应对？
✓ 高分要点

"'不合理' 往往是从工程师视角看，从客户角度可能完全合理。我会：
① 先理解：不合理的背后是什么真实痛点？
② 提供替代：能否用 80% 成本解决他 90% 痛点？
③ 透明 trade-off：用数据说明做了这个意味着什么；
④ 如果客户坚持，我会和销售一起评估商业价值 vs 工程成本，必要时升级到 Director。"

E. 自我认知与反思类
Q18. 你最大的 weakness 是什么？
面试官想看的不是 weakness 本身，而是自我认知 + 改进行动。
✓ 高分回答模板

"我曾经的最大短板是【具体 + 真实】，过去【时间】我意识到它带来的代价是【影响】， 所以我【具体动作】，现在已经【可衡量的改善】。当然，这是一个持续过程，我目前在做的是【下一步】。"

— 例："我之前做 SM 时，过度 protect 团队，自己冲在前面解决一切 blocker。后来一次 retro，团队反馈他们感觉不被信任。我开始有意识地 'sit back'，把问题先抛给团队解，2 个月后团队 ownership 明显上升。但我仍在学如何更长时间放手。"

Q19. 你过去最大的失败是什么？学到了什么？
✓ 高分要点
  • 选真正失败的（不是"我太完美主义"这种伪失败）
  • 主语是"我"，不是"团队"或"环境"
  • 要有具体后果（数字、人员、客户）
  • 最重要：这次失败后你做了什么不同？（行为改变是核心）
Q20. 为什么想从 Scrum Master 转 Engineering Manager？
"因为 SM 没有人事权"、"因为 EM 工资高" — 都是炸弹答案。
✓ 高分回答框架

"作为 SM 这 X 年，我能影响团队的节奏和健康度，但我越来越觉得：
① 我看得到团队真正的 blocker 不只是 process，更是人员配置、技术方向、组织结构；
② 我想能对这些做出决定，并对结果负责，而不只是 facilitate；
③ 我相信我已经具备的影响力、协调、流程意识，加上我正在补的技术判断与人员发展，可以让我胜任 EM。
这不是'升级'，是责任范围的转变，我准备好了。"

F. 情境题 / 假设题
Q21. 假设你刚接手一个 15 人的新团队，前 30/60/90 天你会做什么？
✓ 高分回答（这是 EM 面试经典题）

0~30 天 · LISTEN（绝不动）

  • 每人 1on1，问 4 个固定问题（"团队最好的是？最差的是？如果你是我会改什么？我能为你做什么？"）
  • 旁听 standup / planning / retro，不发表意见
  • 和 stakeholders（PO、上下游 manager、老板）各做 1 次访谈
  • 看交付数据：3 个月 velocity / bug / cycle time

30~60 天 · DIAGNOSE

  • 形成 "我看到的 3 个 strength + 3 个 gap" 文档
  • 和老板对齐 priority
  • 开始小动作（如改 1on1 cadence、修 retro 模板）

60~90 天 · ACT

  • 共享 90-day 观察给团队，让他们参与定 north star
  • 启动 1~2 个明确改进（不要超过 2 个，会失焦）
  • 定 quarterly OKR

核心心法："前 30 天不做改变" 是最难也最重要的纪律。

Q22. 你的团队里有个明星工程师，但他对同事很 toxic，你怎么办？
✓ 高分回答

"No brilliant jerks —— 这是 Netflix 文化原则，也是我的原则。
① 直接、私下、具体地反馈：'你在 X 会议上的 Y 行为，让 Z 同事感到 Q，下次能不能 ...'（SBI 框架）；
② 给他 3 个月明确的改善标准（含同事 360 反馈）；
③ 如果不改 → 即使产出再高也要 let go。因为一个 brilliant jerk 会让 3~5 个好同事离职，组织净损失。
我曾经遇到过这种情况，初期我犹豫了 2 个月，结果团队 2 人离职。后来我下定决心处理，团队氛围 1 个月就明显好转。"

Q23. 你的老板让你裁掉团队 20% 的人，你怎么做？
✓ 高分要点
  1. 先问 why：是 cost / re-org / 战略转向？理解上下文才能做对决策
  2. 不要拒绝：但可以协商时间、范围、方式
  3. 用绩效证据，不要凭印象
  4. 有尊严地沟通：每个被影响者都值得提前 1on1 解释、不被群体邮件通知
  5. 照顾留下的人：survivor guilt 是真实的；要重新沟通愿景
  6. 自己也要扛压：不可避免会有情绪，自己也要找 mentor 倾诉
Q24. 你的团队有人公开质疑你的决策，你怎么办？
✓ 高分回答

"欢迎质疑，但区分场合：
① 如果在会上 — 当场感谢，复述他的观点，承诺会后深入讨论。绝不当场翻脸；
② 会后 1on1 — 听完整他的逻辑，可能他对，可能我错；
③ 如果决策不变 — 给清晰的 reasoning，让他知道他被听到了；
④ 'Disagree and commit'（Bezos 原则）：可以不同意，但定了就要全力执行。
有质疑的团队是健康的；没人质疑你才该担心。"

Q25. 反向提问 — 你会问面试官什么？
✓ 高分问题清单（按效果排序）
  1. "这个团队目前最大的挑战是什么？为什么之前的 manager 没能解决？"（看真实环境）
  2. "这个 role 6 个月后，什么样算 successful？1 年后呢？"（看预期）
  3. "团队最近的 attrition 情况怎样？主要原因是？"（看健康度）
  4. "这个 EM 的 manager（director）是什么风格？多久 1on1？"（看你未来老板）
  5. "组织对 manager 的成长有什么投入？mentor / training / community？"（看公司是否真的 invest in managers）
  6. "过去 12 个月这个团队最大的成功 / 失败分别是什么？"（看 culture）

4.3 Scrum Master 转管理岗的"面试雷区"

为什么很多 SM 面 EM 都挂在 onsite？面试官在担心 5 件事：

① 你能不能"做技术决策"？
High Risk
SM 通常被 PO / Tech Lead 视为"不写代码的人"。面试官会怀疑你能否 challenge 设计、识别风险。
破解：面试中主动讲 1~2 个你影响过技术决策的故事（即使是通过提问、组织 review）。
② 你能不能"扛交付结果"？
High Risk
SM 的传统 KPI 是过程指标，不是业务结果。
破解：把你的故事改写为"因为我做了 X，团队按时交付了 Y，业务获得 Z"。把过程指标和结果指标连起来。
③ 你敢不敢"做艰难决定"？
High Risk
SM 角色 servant-leader 倾向温和、协作。面试官担心你不敢 PIP、不敢拒绝、不敢说 No。
破解：准备至少 1 个"我曾经做出让一些人不开心但正确的决定"的故事。
④ 你能不能"管 senior"？
Medium
EM 经常要管比自己资深的工程师，靠 authority 不够。
破解：讲一个你影响过 senior 工程师的故事（design review push back、tech lead 协调）。
⑤ 你是不是"过度依赖流程"？
Medium
过度敏捷主义者会让面试官担心你是"工具人"。
破解：主动说"Scrum 是工具不是宗教"，分享 1 个你"反 Scrum"的决策（如停掉一个仪式因为没价值）。
⑥ 你是否有"ownership"？
High Risk
SM 容易说"不是我们团队的问题"。EM 必须说"这是我的"。
破解：所有故事都用"I owned ... I decided ... I was accountable for ..."，避免"we / the team / they"。
面试黄金 5 法则（SM 转 EM 专用）：
  1. 用数字：所有故事必有 metric
  2. 用"我"：明确区分个人 contribution 和团队 contribution
  3. 讲决策：不只是 process，要有 "选 A 而非 B" 的判断
  4. 讲艰难：至少 2 个"不容易的 trade-off"故事
  5. 讲影响半径：从"我的团队"扩展到"跨团队 / 组织级"

6.1 案例一 · 项目延期 2 个月，客户开始升级投诉

背景
CRISIS
某平台核心组件版本（约 4 个 Scrum team 共 25 人）目标 6 月底 RFA（Ready For Acceptance），目前 5 月底， 经 EM 评估实际进度约 60%。客户已二次 escalate，销售方面施压"必须按时"。 团队连续两个 Sprint commitment 命中率 50%，Lead 工程师开始抱怨"需求一直在变"。

第一反应：错误的应对

普通 Manager 反射性动作
  • 立刻召集"战时会议"，每天加 1 次 sync
  • 从其他团队抽调 3 人过来支援（Brooks Law!）
  • 要求团队周末加班赶进度
  • 向上承诺"我们一定按时"
  • 下场亲自 review 所有 PR、参与所有设计

结果：2 周后 Velocity 进一步下降 30%，2 个核心成员请病假，质量事故反弹。

有经验 EM 的诊断式响应
  • 第一天 — 只做诊断，不做动作
  • 调出 Velocity / Burndown / Bug Trend / Cycle Time 数据
  • 和每个 Tech Lead + SM 做 1on1 听真实情况
  • 看最近 4 周新增 / 变更的需求量
  • 判断根因后再决定动作

推进：4 个动作的并行执行

  1. 对外：Director 出面与客户沟通"负责任的 phased delivery"，并主动提供折让
  2. 对内：EM 在全员会上坦诚说明决策，"不是 team 的错，是计划的错"
  3. 过程：每周 cross-team sync，公开 risk register；冻结 4 周 scope，超过 4 周后才能 review 新需求
  4. 支持：HR fast-track 2 个开放 HC；为离职 Senior 做 knowledge transfer 项目（4 周）

3 个关键 lesson：
  1. "项目失控不是从延期那天开始的，是从'不敢说真话'那天开始的" — Velocity 第一次连降时就该 raise。
  2. Bad news memo 是 EM 最重要的 artifact 之一。 一页纸，含 ① 现状 ② 根因 ③ 选项 ④ 推荐 ⑤ ask。
  3. 对客户：phased delivery 比 perfect delivery 更可信任。 客户最怕的不是延期，是突然延期。

6.2 案例二 · 团队失控：高级工程师集体抗议新 manager

背景
PEOPLE
新 EM 上任 6 周后，团队 4 位 Senior 中 2 位向 Director 发邮件抗议："新 EM 不懂技术，频繁干涉设计，把团队'敏捷化'过度"。 Director 把球踢回给 EM："你自己处理"。

EM 的应对：先理解，再行动

Day 1 · 自我反思
不要立刻去找抗议的 senior 谈

那会让他们觉得你"被告状了所以来找他"。先独立反思：过去 6 周我做了哪些决策？哪些可能让 senior 觉得被冒犯？

Day 2~3 · 数据收集
Skip-level 1on1

找团队中 中立的中级工程师 做 skip-level 1on1（你的下属的下属），问 3 个问题："最近团队最大的变化是什么？大家在 hallway 里都在讨论什么？如果你是我，你会怎么做？" 这比直接找抗议者更能得到真相。

Day 4~5 · 与抗议者 1on1
承认 + 提问 + 邀请

"我听说你最近对一些决策有强烈意见 — 我想听你完整说说。我可能犯了错，也可能有上下文你没看到，我们一起搞清楚。"
关键技巧：不防御、不解释、不打断。只问、只听、只记。

Day 6 · 真实根因
诊断真相

真实根因往往是 3 件具体小事：① 一次设计 review 上 EM 否决了 Senior 的方案没解释清楚； ② EM 引入"所有设计必须先有 RFC"被理解为"不信任 senior"；③ EM 的 standup 风格被认为"变成汇报会"。
不是"敌意"，是"沟通失败 + 节奏不匹配"。

Day 7~14 · 修复行动
三件事
  • ① 道歉具体行为（不是"对不起让你不舒服"，而是"关于 X 决策我应该先和你讨论再宣布"）
  • ② 邀请共同设计新流程（RFC 模板由 senior 起草，不是 EM 强加）
  • ③ 调整 standup 节奏，恢复 self-management
30 天后 · 修复关系
验证

Senior 主动来找 EM 讨论一个跨团队问题。这就是关系修复的信号。

核心心法： 新 manager 进团队，第一个月的"形象"决定了未来一年的关系。 如果出了问题：认错的速度 = 信任恢复的速度。死扛 = 永远修不好。

6.3 案例三 · 架构腐化：5 年老系统每发版都引入新 bug

背景
TECH
基带控制平面模块（约 80 万行 C++ 代码），5 年内 4 任 owner。 近 6 个版本 change failure rate 从 12% 升至 38%，集成测试通过率从 90% 跌至 65%。 Tech Lead 提议"大规模重构"，预计需 6 个月专投入。

EM 的两难

选项 A · 大爆炸式重构（Tech Lead 提议）
  • 6 个月停业务，组 8 人专项
  • 风险：新系统未完成时旧系统继续腐烂
  • Joel Spolsky 著名警告："Things You Should Never Do, Part I — 重写"
  • 历史数据：90% 的重写项目失败或延期 3x
选项 B · Strangler Fig Pattern
  • 识别 5 个最腐烂模块（用 SonarQube + Bug 热点图）
  • 每 Sprint 替换 1 个模块（"strangle" 旧代码）
  • 用 feature flag 双跑、灰度切换
  • 预计 9~12 个月完成，但每周都有可发版状态

EM 的决策与推进

  1. 数据先行：组织 1 周的 tech debt audit，用 metrics 排序"腐烂热点"（hotspot analysis = git log frequency × cyclomatic complexity × bug density）
  2. 说服 PO：用业务语言翻译技术债："每 sprint 投入 20% 容量做架构治理，6 个月后我们的新特性 lead time 会缩短 40%"
  3. 建立 dashboard：DORA 4 指标 + bug rate + test coverage 公开 visible 给 director
  4. 不止技术：同时启动 mentor 配对、引入新 senior、改 code review 规则
  5. 每 quarter review：根据数据决定是否调整节奏

复盘的关键洞察

当多个问题叠加时，新 EM 最大的诱惑是"全打"。但这是错的。 正确做法："序列化处理" —— 先止血（防止离职扩散），再处理人际（冲突），再处理低绩效（PIP），最后修复文化。 每一步都需要 1~2 周稳定后再开下一步。同时打 = 全部失败。

A.2 常用管理模型速记

SBI 反馈模型
Situation · Behavior · Impact。
避免主观评判，专注行为与影响。
BICEPS 情绪触发
Belonging / Improvement / Choice / Equality / Predictability / Significance。Lara Hogan 框架，用于识别下属情绪根因。
RACI 矩阵
Responsible / Accountable / Consulted / Informed。跨团队项目必备。
RICE 排序
Reach × Impact × Confidence / Effort。Intercom 的优先级框架。
Eisenhower 矩阵
紧急 × 重要。EM 应把 60% 时间花在"重要但不紧急"象限（人员发展、战略思考）。
Radical Candor 2×2
Challenge Directly × Care Personally。两轴都要高才是"Radical Candor"。
Tuckman 团队发展
Forming → Storming → Norming → Performing → Adjourning。判断团队处于哪一阶段。
Drexler-Sibbet 模型
7 阶段：Orientation → Trust → Goal → Commit → Implement → High Performance → Renewal。比 Tuckman 更适合现代团队。
