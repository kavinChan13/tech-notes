W1
Week 1 · 接手第一周
不发表观点 · 不做决策 · 只收信息 + 建立信任

1.1 · 5 个必开的 1:1（按优先级）

① 你的 ML 技术骨干 / Tech Lead（120 min）第一天必做

  • 问：现在系统最大 3 个技术风险 / 你最担心的 1 件事 / 团队最缺的 1 类能力 / 你需要我做的 3 件事
  • 不问：你的薪资 / 上一任 EM 怎么样 / 团队谁强谁弱（早问会显得不信任）
  • 送：明确表达你的角色 — "我不是来当二号 IC，是来帮你 unblock 工程 + offload 管理负担"

② 你的产品经理 / PM（90 min）Day 2-3

  • 问：业务最看重的 3 个 metric / 短中长期路标 / 跟其它团队的依赖 / 最近 3 个被推迟的需求
  • 关键：摸清 PM 对"AI quality vs 速度"的偏好 — 这决定后续 eval 投入

③ SRE / 平台对接人（60 min）Day 3-4

  • 问：你们的 GPU pool / 配额 / 当前 utilization / 最近 3 个 incident / on-call 是怎么轮的
  • 关键：搞清楚 你团队的 GPU cost 在公司里排第几 — 这是后续谈预算的筹码

④ 数据 / 标注 / Eval 团队（60 min）Day 4-5

  • 问：你们给我团队的 eval set 多大 / 怎么更新 / 标注延迟 / 数据访问权限
  • 关键：很多 AI 团队踩坑 = eval set 太小或 stale。要在 Week 1 摸清

⑤ 上一任 EM（如果还在）/ 你的老板（60 min）Day 5

  • 问：团队没解决的 3 个老问题 / 你期望我前 3 个月做出什么 / 团队 promotion / PIP 在途 case
  • 关键：拿到老板的明确"成功标准"，避免 3 个月后才发现期望错位

1.2 · 5 类必收的信息（Week 1 末交出）

信息类具体收什么用来干嘛
Eval 现状eval set 大小 / 上次跑日期 / 主要 metric / 最近变化趋势判断 Month 1 是否要重建 eval
Model usage用了哪些模型（API + 自部署）/ 调用量 / cost / latency P50/P99判断是否要做模型整合 / 切换
GPU cost每月 GPU 账单 / 团队占公司总 GPU 比例 / 利用率 / 浪费点谈下季度预算 / 找 quick win
Safety incidents过去 3 个月所有 incident / 已解决 / 待解决 / 是否有合规 audit 在途识别 hidden time bomb
Hiring pipeline当前 open headcount / pipeline 候选人 / 团队 retention 风险评估是否要紧急招 / 留人

1.3 · 识别 3 个最大坑（Week 1 末口袋里要有）

每个 AI 团队都有 3 个"它不爆你也忍着，但爆了就是灾难"的坑。Week 1 必须识别出来 — 不一定 Week 1 解决，但要进 Month 1 决策矩阵。常见 3 类：

🕳 Eval 黑洞

"团队从不跑 eval" 或 "eval set 半年没更新"。下次模型升级回归 = 必爆。

🕳 单点依赖

"某个模型 / API / 关键工程师离职 = 整条线挂"。识别后立刻准备替代方案。

🕳 GPU cost 失控

"每月超预算 30% + 老板没怎么管" — 哪天 CFO 一看账单就会发火。提前找 quick win 砍 20%。

Week 1 通关
  • 5 个 1:1 全部完成（写有结构的笔记，存到自己的 1:1 dashboard）
  • 5 类信息表填齐（哪怕"未知"也要标记"未知 + 谁问"）
  • 3 个最大坑识别（不一定解决，但记在你的 issue list）
  • 团队 retro：第一周给团队发一份 "我在 Week 1 听到的 + 还没听到的" 短 memo，建立信任
  • 跟老板第一次 1:1：报告你看到的 3 个坑 + 你 Month 1 优先级 draft
M1
Month 1 · 5 项关键决策
开始做小决策 · 验证团队信任 · 锁定 Q1 优先级

2.1 · 决策 1：人员评估（Day 8-15）

评估 4 维度

  • 技术深度：每人主要"专长" + "能 hold 住"的复杂度等级
  • 团队互补：是否过度集中（如全是 modeling，没人懂 infra）
  • retention 风险：薪资 / 职业进展 / 跟同行对比 / 跳槽迹象
  • 潜力：3-6 个月能否担起新职责（带 1 个项目 / mentor junior）

输出：每人 1 张 1-page 评估（不分享给本人，是你自己的 mental model）

2.2 · 决策 2：项目优先级（Day 10-20）

3 类项目要排清：

项目类排序原则
① 保维持类（现有线上 AI feature 维护 + 持续 eval）不能停 · 占 30-40% capacity
② 下个季度承诺类（PM 已答应业务的 feature）按业务价值排，砍 1-2 个 lower-impact 的
③ 技术债 / infra 投入（eval 体系 / observability / 量化等）至少留 20% capacity

2.3 · 决策 3：eval 体系评估（Day 12-22）

3 问问出真相：

  • "上周如果改了 prompt，团队怎么知道效果好坏？" 答不上 → eval 体系不完整
  • "如果换 base model（GPT-4o → Claude），决策怎么做？" 答 "感觉" → 没有 model selection eval
  • "online metric 跟 offline eval 一致吗？" 答 "不知道" → 没有 alignment

3 个问题任 1 个不及格 → Month 1 末必须启动 eval 体系重建，参考 X12 AI 评测体系。

2.4 · 决策 4：cost 摸底 + 找 quick win（Day 15-25）

典型 3 类 quick win：

🔧 模型 routing

简单 query 用便宜模型（Haiku/4o-mini）+ 复杂 query 才上 GPT-4。典型省 40-60% cost。

🔧 prompt 优化

砍掉冗余 system prompt / 用 prefix caching。典型省 10-20%。

🔧 量化 / 自部署

如果有自部署能力，INT4 量化 + vLLM 可砍 50-70%（参考 Y04 + Y02）。

2.5 · 决策 5：安全合规审查（Day 20-30）

5 条红线检查（任 1 个未通过 → 立刻 escalate）：

  1. PII 流转：用户数据是否合规进入 LLM API / training set
  2. 第三方 model 数据 retention：OpenAI/Anthropic 的 zero-retention 是否开
  3. Prompt injection 防护：用户输入是否有 sanitization
  4. Output safety：是否有 content filter / moderation
  5. 监管备案：国内生成式 AI 备案 / EU AI Act 适用性
Month 1 通关
  • 5 项决策都有明确结论（哪怕是 "保持现状 + 30 天后重审"）
  • 1 张 Month 2-3 优先级矩阵（business / tech / hire 三轴）
  • 1 份 cost quick win 清单（至少 2 个能开干的）
  • 1 次 all-hands / team retro：分享你的 Month 1 观察 + Q1 方向
  • 跟老板第二次正式 review：报告 Month 1 + 提前打 Q1 期望
Q1
Quarter 1 · 3 份必交付
把 Month 1 决策落成产物 · 季度末向上汇报有据可依

3.1 · 必交付 1：AI Roadmap v1（含 Model Risk Register）

结构（推荐 6 段）

  • ① Business context：业务目标 + 团队定位 (1 page)
  • ② 现状 baseline：当前 metric / cost / quality 雷达图 (1 page)
  • ③ Q1-Q2 milestones：3-5 个 deliverable + DRI + 时间 (2 pages)
  • ④ 资源 ask：headcount / GPU / 跨团队依赖 (1 page)
  • ⑤ Model Risk Register：5-10 条已知 model risk + mitigation (2 pages) · 这是 AI 团队特有的
  • ⑥ Success metric：怎么算 Q1 / Q2 成功 (1 page)

Model Risk Register 示例条目：

风险概率影响Mitigation
GPT-4o → 4.5 升级，回归高中regression eval set + canary deployment
vendor 模型涨价 30%中高Q2 启动开源模型 POC + 量化
用户 prompt injection 攻击低高Q1 加 input sanitization + 红队测试
关键 ML 工程师离职中高Q1 招 backup + 关键代码 review pair
EU AI Act 强制合规中中Q2 跟法务 align + 评估影响

3.2 · 必交付 2：AI Hiring Scorecard 改造

跟通用 EM Hiring Scorecard（参考 EM 模板 08）的区别 — AI 团队多 3 维评估：

+ Eval Mindset

候选人是否懂"AI 没有 bug-free，只有 metric trade-off"。问"你怎么知道你的模型变好了" — 答不上判 Junior。

+ Cost Awareness

候选人是否懂 "1 个 GPT-4 调用值 $X"。给 1 个简单算账题（"100k 用户 / day，per query 2k tokens，估月 cost"）。

+ Real-world Limits

候选人是否知道 LLM 的"失败模式"（幻觉 / 长 context 退化 / 推理慢 / cost 不可控）。面试问 3 个 limit。

3.3 · 必交付 3：AI Cost Model + Q2 预算

3 张表必须有：

表内容
① 当前 cost 拆解API cost / GPU cost / 人力 cost / 第三方服务 cost
② per-feature unit economics每个 AI feature 的 per-request cost · gross margin
③ Q2 预算 ask基于业务增长 + cost optimization 的 net forecast
CFO 最喜欢的 AI EM 是 "能用财务语言谈 AI" 的。如果你能拿出 unit economics 表，预算谈判成功率 +50%。不要等 CFO 来问，主动准备。
Quarter 1 通关
  • AI Roadmap v1 文档（含 Model Risk Register）已发布给 stakeholder review
  • Hiring Scorecard 改造 + 至少 5 场面试已用新 rubric 进行
  • AI Cost Model + Q2 预算已跟老板 + CFO 对齐
  • 团队 retro：Q1 哪些 deliverable 达成 / 哪些没达成 / 为什么
  • 你自己的 90 天复盘 doc（你做对了什么 / 错过什么 / Q2 调整）
04

5 个危机场景 · 应对剧本

遇到时不要从零思考 · 直接套这 5 个剧本

AI EM 这个岗位的"危机种类"比传统 EM 多 — 不只是工程 incident，还有 model regression / safety / cost / 团队焦虑 / 老板焦虑。下面 5 个场景按出现频率排，每个剧本你 Q1 内大概率会遇到至少 2 个。

场景 1 · 模型升级后回归
触发
vendor 升 model（GPT-4o → 4.5） 或 团队改了 prompt → 用户开始投诉"答案变差"
第 1 小时
不要 rollback（除非现象明显严重）。先拉 eval：用 regression set 跑新旧 model 对比。要看数据再决策。
第 1 天
出 5 类 case 对比表：① 用户投诉 case ② 你的核心 eval set ③ 真实流量随机抽样 ④ 长尾 case ⑤ 边界 case。不要只看 1 个数。
第 3 天
3 选 1：A) 确认回归 → rollback + 写 RCA；B) 部分回归 → A/B 灰度；C) 误判（用户偏好变化）→ 跟 PM 沟通调 prompt
教训
长期：必须建 auto regression eval（每次 prompt / model 改动自动跑）。这是为什么 Month 1 决策 3 要审 eval 体系。
场景 2 · 用户投诉幻觉 / 错误答案上头条
触发
某用户在 X / 知乎晒"AI 给我的离谱答案" → 转发 1000+ → PR 团队找你
第 1 小时
3 件事：① 截图存证 ② 复现 case ③ 给 PR/法务一句话："我们正在复现 + 4 小时内给响应方案"。不要立刻道歉（也别狡辩）。
第 4 小时
出 1-page incident summary：root cause 假设 / 影响范围（多少用户可能遇到）/ 短期 mitigation（如加 disclaimer）/ 长期 fix
第 3 天
RCA + 跟 PR/法务 align 对外回应口径。承认"AI 会出错 + 我们如何降低 + 用户可以 X 反馈"，不要承诺"永远不再发生"
教训
长期：必须建 user feedback loop（in-product feedback button + auto triage）+ red team（季度对抗测试）。
场景 3 · GPU 预算超支 30% · CFO 找上门
触发
月末 finance review · 你团队 GPU 账单超预算 30% · CFO 邮件 cc 老板
第 1 天
不要急着回邮件解释。先拉数据：① 哪个 feature / 哪个时间段超 ② 是流量涨还是 cost-per-request 涨 ③ 跟 forecast 偏差原因
第 2 天
回 CFO 邮件 3 段：① 数据归因（"80% 来自 X 业务增长 + 20% 是 cost-per-request 涨"）② 短期 mitigation（具体 2-3 个能立刻做的）③ 长期 plan（model routing / 量化 / 自部署 timeline）
第 1 周
启动 2-3 个 quick win（typically model routing / prompt 优化 / cache）+ 给 CFO 1 张 Q2 cost trajectory 图
教训
长期：每月主动给 CFO 发 1-page cost dashboard + forecast。不要等他找你 — 主动报告时你是 owner，被动回应时你是 victim。
场景 4 · 团队焦虑 · "AI 会不会取代我们"
触发
团队某成员 1:1 时说 "我们做的事 GPT 自己就能做了，我的价值在哪" 或 multiple 成员低气压
第 1 周
不要否认（"放心你不会被取代"是 BS）。承认现实："AI 确实在改变工程，但改的是工程方式不是工程师价值"。给 1 张图：哪些事 AI 做得更好 / 哪些事人做得更好 / 哪些事需要人 + AI 配合
第 2 周
个人层：跟每个焦虑的人重新 align "未来 12 个月你的能力建设方向"。把"被取代"焦虑转化为"成长方向"
第 1 月
团队层：办 1 次 "AI tooling demo day" 让每个人分享"我用 AI 加速了什么"，重塑团队对 AI 的关系（从对手到工具）
教训
长期：把 "AI Leverage Ratio"（每人借 AI 完成的工作量）做成正向指标，鼓励而非惩罚 AI 使用。
场景 5 · 老板每周问 "我们怎么用 AI / 别人都在做 X 我们怎么不做"
触发
老板看完某个 X 公司 demo / 某篇文章后，每周给你转发 "我们怎么不做这个" · 持续 1+ 月
第 1 次
不要直接说"不做"。先问 3 个问题：① 你看重的是技术 demo 还是商业价值？② 我们的优先级排序是什么 vs 这个？③ 如果做，要砍现有哪个项目？让他自己排序。
连续 2-3 次
建立"AI Watch List"机制：每周你主动给他发 1-page "本周值得关注的 AI 进展 + 我们的评估"。抢主动权，从被动变主动。
长期
每季度跟老板 align "我们的 AI 战略" — 让他参与决策而非旁观，他会少问 "我们怎么不做 X"，多问 "X 跟我们战略怎么配"
教训
老板的 AI 焦虑 = AI EM 必须管理的事。这是岗位的隐性任务。Q1 内必须建立向上沟通的节奏 + 内容。
05

AI EM 的 5 个职业资本积累方向

3-5 年看 · 让你跟传统 EM 拉开差距

同样做 EM 3 年，AI EM 比传统 EM 有 5 个独特的"职业资本"机会。Q1 之后就要有意识投入。

① Eval 体系设计 · 跨公司可迁移

每家 AI 公司都缺懂 eval 的 EM。你在团队建过的 eval pipeline + 学到的 trade-off是高度可移植的能力，下一份 offer 直接 sell。

② Model Risk 管理 · 监管时代的稀缺技能

2026 后 AI 监管收紧（中美欧）。能 own Model Risk Register + 跟法务沟通的 EM 会是大公司 strategic hire。

③ AI Cost 经济学 · 跟 CFO 对话

"能用 unit economics 谈 AI" 的 EM 会被 promote 到 Sr.EM / Director 路径。每月主动给 finance 发 cost dashboard 是关键习惯。

④ 跨学科招聘 · 你建的 hiring scorecard

AI 团队招聘比传统团队难（ML + Infra + App + Data 多 persona）。你做的 hiring scorecard / interview rubric 是个人品牌资产。

⑤ Public visibility · OSS / talk / 公众号

AI 圈热度高，你管的项目 + 你写的 EM 内容更容易被看到。坚持每季度 1 篇深度文章 / 1 次 talk，2 年后你就是这个领域有名字的 EM。

⚠️ 反模式：只做"AI 翻译官"

很多 AI EM 沦为"传话筒"（业务 ↔ ML 工程师）。3 年后没有独特技术或战略价值，下一份 offer 难谈。必须主动积累上面 5 个之一。

AI EM 跟传统 EM 的根本不同：传统 EM 的资本是"经验 + 人脉"，AI EM 多了一层 "AI-specific judgment"（model risk / cost / eval / safety / hiring）。有意识积累这层，未来 10 年红利。
06

8 个 FAQ

第一次当 AI EM 最常见的疑虑
Q1 · 我没 ML 背景，能管 AI 团队吗？
能 — 但 Day 1-30 必须补 4 类概念：① LLM/RAG/Embedding 基本概念 ② Eval / Hallucination / Guardrails ③ GPU cost 数量级 ④ 主要模型对比（GPT/Claude/Qwen/DeepSeek 价格 + 能力）。
不需要会推 backprop / 不需要会 fine-tune。EM 的价值不是"比工程师懂得多"，是"决策框架 + 跨部门沟通 + 风险管理"。
推荐第一周读：EM01 第 1-2 章 + X06 Ch1-3 + X12 Ch1-2。
Q2 · 团队有 ML PhD，我怎么 establish credibility？
不要在技术上跟他们 compete。Establish credibility 靠 3 件事：
① 问对问题 — 不需要给答案，问"如果换 model 你怎么决策" / "这个 metric 我们 confidence 多大" 就够了 — 显示你懂 framework
② unblock — 帮他们砍掉无关 meeting / 谈下 GPU 配额 / 摆平跨部门依赖
③ 讲 business — 把他们的工作翻译成 business value，给老板 / PM 听
PhD 工程师最讨厌"装懂 ML"的 EM。"我不懂 X 但我能帮你做成 Y" 比"我也懂 X"更有说服力。
Q3 · 团队总说 "再给 3 个月就能做出来"，但永远做不出来
AI 项目本质高 uncertainty，"3 个月"经常是工程师的乐观估计。3 招破：
① 要 milestone 拆解 — 不接受"3 个月" 的整体承诺，要"Week 4 / Week 8 / Week 12 各看到什么"
② 要 success criteria — 提前定"什么数算成 / 算败"，避免"最后才发现还差很远"
③ 设 kill switch — 提前约定"如果 Week 8 没达 X，我们重新评估方向" — 给团队心理预期，不让"沉没成本"绑架决策
关键：EM 的工作是 "管理不确定性"，不是 "消除不确定性"。
Q4 · GPU 预算我谈不下来，怎么办？
AI EM 不会谈 GPU 预算 = 一年内必爆。3 招：
① 带数据谈 — 拿出 unit economics 表，证明"每张 GPU 创造 $X 业务价值"（不能给数 = 不会谈）
② 给 alternative — 不是"要 100 张"，是"100 张 vs 50 张 + 量化优化 vs 50 张 + 砍 X 项目"
③ 建立月度 cost dashboard — 主动给老板 + CFO 看，你是 cost owner 而不是 cost 申请者，谈判地位完全不同
没谈下来 ≠ "公司没钱"。多半是你没把 ROI story 讲清。
Q5 · 团队选 model 总在内耗（GPT vs Claude vs 开源）
这是因为没有 model selection eval framework。3 步落地：
① 建 model selection eval set（200-500 个 case 覆盖主要 use case + 边界）
② 定 4 维 trade-off matrix：quality / cost / latency / 合规
③ 每季度重跑 一次（市场变化太快）
有了数据 + framework，决策 5 分钟搞定。没有 framework，争论无解。EM 的责任是建 framework，不是参与争论。
Q6 · 团队不想跑 eval，觉得"凭感觉就行"
这是 AI 团队最大文化问题之一。"凭感觉" 工程师不是懒，是觉得 eval cost > 收益。3 招建 eval 文化：
① 第一次自己跑给团队看 — 选 1 个最近争议的决策（如换 prompt），跑 eval 出数据，让大家看到"原来差 X%"
② 建 lightweight eval pipeline — 30 秒内跑完 50 个 case，eval cost 低到团队没理由不跑
③ 把 eval 纳入 review process — PR 改 prompt / model 必须附 eval diff，否则不 merge
3 个月后团队会上瘾 — 因为 eval 让他们晚上能睡好觉。
Q7 · 我是从传统 backend EM 转的，团队不信任我怎么办？
信任靠时间 + 行动，不是靠 talk。前 30 天 3 件事建信任：
① 承认你 don't know — "我从 backend 来，AI 这块还在学，请你们帮我 ramp up"。诚实比装懂赢更多尊重
② 用 backend 经验帮他们解决 1 个具体问题 — 比如 deployment / observability / on-call 流程（AI 团队这块通常薄弱）
③ 第 30 天给团队发 "我学到了什么 + 我看到团队 3 个机会" — 展示你在认真学 + 有 outside-in 视角
90 天后团队会发现"我们之前缺的就是有工程文化的 EM"。
Q8 · 我 Q1 干得好，Q2 该怎么 scale up？
Q2 从 "做事" 转 "建系统"。3 个 scale-up 优先级：
① 把 Q1 一次性决策变成机制（eval review / cost review / model selection 进周/月会议）
② 培养 1 个能 backup 你的 IC / TL（不是接班，是关键决策能 2 个人扛）
③ 开始投资跨团队影响力（其它团队的 AI eval / cost 你能不能提供 template）
这 3 个做好，你从"团队 EM" 升级为 "组织 AI 能力 builder"，promotion path 打开。
最后忠告：AI EM 是 2024-2030 最容易"做出 outsized impact" 的 EM 岗位。因为：行业新 + 没有 best practice + 老板都在摸索。你的 Q1 决策可能就成了公司的 AI engineering 标准。认真做，回报 5x。
