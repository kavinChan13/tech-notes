01

为什么组织设计是 Director+ 核心技能

组织设计影响什么 · 新晋 Director 3 大幻觉

Line EM 时代你修人（hire / coach / fire）；Director+ 你修组织—— 通过结构 / 流程 / 权责设计，让组织自身能 scale。

组织设计影响什么

  • Velocity—— 团队之间依赖少 → 各自快
  • Quality—— ownership 明确 → 没人 own = 没人 care
  • Hiring—— 团队结构决定能 hire 啥 senior
  • Retention—— 不合理结构 → senior 流失
  • Politics—— 重叠职责 → 内耗
  • Strategic execution—— 战略改但组织没改 → 战略失败

新晋 Director 3 大幻觉

  1. "我能 micromanage 100 人"—— 不能，必须 through manager
  2. "完美 org design 存在"—— 不存在，是 trade-off + iterative
  3. "reorg 解决所有问题"—— 多数问题是 culture / process / 战略，不是 org
02

Conway's Law + 反向 Conway

反向 Conway Maneuver · 真实 case · 常见 anti-pattern

Melvin Conway 1968 提出（"How Do Committees Invent?"）：

📐 Conway's Law

"Any organization that designs a system... will inevitably produce a design whose structure is a copy of the organization's communication structure."

组织结构决定系统架构。3 个 team 设计的系统必然有 3 个 module + 团队间的 communication 模式。

反向 Conway Maneuver

Software Architects 反向用：先设计目标系统架构，再调整组织 match 它。

目标系统架构对应组织
单体1 大团队 + functional sub-team
微服务多 small team，每 team own 1-3 service
Platform + productPlatform team + product team
Data meshDomain team own data + platform team

真实 case

Spotify 早期：定 squad / tribe / chapter / guild 结构后，软件架构自然变 microservice + 各 squad own 1-N service。不是 Spotify 先设计 microservice 再设计组织，是反向。

常见 anti-pattern

  • Cross-team 重 dependency—— 共享 monolithic codebase 但分 5 个 team，每个 PR 涉 5 个 team review
  • Inverse Conway—— 团队按职能分（frontend / backend / mobile）但 product 切 vertical → 每个 feature 跨多 team coordinate
  • Distributed Monolith—— 名义微服务但 deploy 必须一起 → worst of both
03

Team Topologies 4 类

大 team 类型 · 种交互模式 · 核心原则 · Cognitive Load 限制

Matthew Skelton & Manuel Pais 2019 同名书。2026 已成 industry de facto framework。

4 大 team 类型

类型职责例
Stream-aligned对齐业务流，端到端 own product"用户登录 team" / "支付 team"
Enabling赋能 stream-aligned team（短期帮 build capability）"Agile coach" / "Cloud migration help team"
Complicated-subsystem处理需要深度专业的子系统"ML platform" / "Encryption / Crypto team"
Platform提供 internal product / 工具，让 stream-aligned 更快"DevOps platform" / "Data platform"

3 种交互模式

模式含义持续时间
Collaboration两 team 紧密合作 explore短期
X-as-a-ServiceProvider team 提供 service，consumer 调用长期默认
Facilitating1 team 帮另一 team 提升 capability短期

核心原则

  • Stream-aligned 是 60-70% 的 team—— 它们直接 deliver business value
  • Platform team 让 stream-aligned faster—— 不是 ivory tower
  • Enabling team 是临时—— 目标是 transfer skill 后 disband / re-purpose
  • Complicated-subsystem 仅在真有 deep expertise need—— 不要乱给 team 这 label
  • Cognitive load < 团队能 handle—— 太复杂 / 太多 service 拆开

Cognitive Load 限制

Team Topologies 强调 team 的 cognitive load 三层：

  • Intrinsic—— task 本身的难度
  • Extraneous—— 环境带来的负担（工具难用 / 文档烂）
  • Germane—— 学习 / 提升的"好"负担

team 总 load < capacity 才能持续。过载 → burnout / mistakes / 流失。

Span of Control

典型数字 · 影响 span 的因素 · 太小 / 太大 的症状 · Skip-level 的 span

"1 个 manager 直接管多少人合适？"

典型数字

层级典型 span解释
First-line EM5-9 人Sweet spot；< 5 underutilized，> 9 不够 1on1 时间
Manager of Managers (Director)4-7 manager每个 manager 自己有 5-9 直属
VP5-8 director跨多 functional / product line
C-level7-12 直接含 VPs + 关键 advisor

影响 span 的因素

  • Team maturity—— senior 多 → span 可大；junior 多 → span 必小
  • Task variety—— task 类型多 → span 小；同质 → span 可大
  • Geographic distribution—— 多时区 → span 小
  • Manager 的 IC time—— 仍写代码 → span 必小 (≤ 5)

太小 / 太大 的症状

Span症状
< 4Manager underutilized / micromanagement / "谁 own 啥" 不清
5-9Sweet spot
10-141on1 时间紧 / 部分员工感觉被忽略
> 14Manager burnout / coaching 缺失 / Senior 流失

Skip-level 的 span

Skip-level（"EM 的 EM"，即 Director-to-IC）也有 span 概念—— Director 不应该跟 30+ IC 1on1（不现实），但应该跟每个 IC quarterly skip-level。

主流 org model 对比

Spotify / Amazon 2-pizza / Functional / Matrix

5.1 Functional

按职能分（Engineering / Product / Design / Marketing 等），每职能下再分团队。

  • ✅ Expertise pool / career path 清晰
  • ❌ Cross-function coordination overhead 大
  • 典型用：小公司 / 大 enterprise functional silos

5.2 Product / Business Unit

按产品 / business unit 分，每 unit 内有自己的 eng / product / design 等。

  • ✅ End-to-end ownership / fast decision
  • ❌ Skill 重复投入（每 unit 都有自己的 frontend / backend）
  • 典型用：大公司 multi-product portfolio

5.3 Matrix

双 reporting line —— 1 个 functional manager + 1 个 product manager。

  • ✅ 同时获 functional expertise + product focus
  • ❌ 双 boss 冲突 / 决策慢 / 员工 confused
  • 典型用：大企业 ad-hoc 项目；少数公司 long-term matrix

5.4 Spotify Model (Squad / Tribe / Chapter / Guild)

Spotify 2012 paper 提出。2026 已知 Spotify 自己也不再这么用，但作为 model 仍有 reference 价值。

层定义
Squad5-9 人 cross-functional team，own 1 个 mission
Tribe多个相关 squad（typically < 150 人）
Chapter跨 squad 同 discipline 的人（如所有 frontend 工程师）
Guild跨 tribe 兴趣 group（如 React / Security guild）
  • ✅ 自治 + 跨职能 + 知识共享
  • ❌ Squad alignment 难 / Chapter lead 角色 ambiguous
  • 典型用：2010s 多家公司 cargo-cult 抄；2020s 退潮，回到 Team Topologies-style 设计

5.5 Amazon Two-Pizza Team

Bezos 名言："If a team can't be fed by two pizzas, it's too big" (typically 6-10 人)。

  • ✅ 严格 size limit → 强制 modular 架构 + 快 decision
  • ✅ 配 microservice + DevOps 完美
  • ❌ 服务 sprawl（成百上千 service / team）
  • 典型用：Amazon / AWS / 互联网 microservice 公司

5.6 Holacracy / Self-Management

Zappos 等尝试。大多失败—— 完全 flat 不 scalable，决策瘫痪 / accountability 缺失。Critical mass > ~50 人时崩。

选型决策

公司阶段推荐
Pre-seed（< 10 人）无 formal model，cross-function chaos OK
Seed-A（10-50）Functional + dual-track（eng / product）
Series B-C（50-200）Stream-aligned squad / Two-pizza
Growth（200-1000）Team Topologies / BU per product line
Enterprise（1000+）Business unit + platform / shared service

Talent Density vs Headcount

Talent Density 计算 · 权衡 · Reed Hastings (Netflix) · Director 决策

Netflix 公开倡导 "Talent Density"—— 少 senior 比多 mid 更好。

Talent Density 计算

非正式定义：team 中"能独立做 staff-level work"的比例。

Talent Density团队特征
> 60%极高，能 attempt 复杂战略项目（Anthropic / OpenAI 早期 / 顶级 unicorn）
40-60%很高（FAANG core engineering）
20-40%正常 health（大多数 well-run startup）
10-20%低，依靠少数 senior carry
< 10%危险，无法 attempt 复杂 problem

权衡

  • High talent density: 单人产出高 + 决策快 + 但 hire 难 + comp 高
  • Low talent density + 多 mid: 容易 scale headcount + 但管理 overhead 大 + 跑不动复杂项目

Reed Hastings (Netflix)

"少 mediocre 比加更多 mediocre 让 team 更强"—— 不留 underperformer，给 senior 工资 + 自由（"keeper test"）。

不是普世答案—— Netflix-style "minimum bar very high" 文化适合 specific industry / stage，不适合所有公司。

Director 决策

预算 1 个 staff（$500k）vs 3 个 mid（$500k 总）：

  • 探索性 / 高 leverage 项目 → 1 staff
  • 已知 + scale execution → 3 mid
  • 需要 mentor 新人 → 1 staff + 2 mid 混合
07

团队成长

从 5 到 50 到 500

关键阶段

5 → 15 人 ("一个 team")

  • 仍是 1 个 EM 直接管所有
  • 无 sub-team，决策 informal
  • 关键事件: 第一次需要拆 sub-team 或 hire 第二个 EM

15 → 30 人 ("一个部门")

  • 2-3 个 sub-team（EM 各一）
  • 需要 Director / Lead EM 协调
  • 关键事件: 团队 lead 转 manager-of-manager，多数人这转型挂掉

30 → 100 人 ("一个业务单元")

  • 多 Director，1 VP
  • 需要正式 process（OKR / 季度 planning / promotion calibration）
  • 关键事件: Conway's Law 开始 dominate，必须 reverse Conway maneuver

100 → 500 人 ("一个 org")

  • 多 VP + 1 CTO
  • Platform team / shared service emerge
  • 关键事件: 文化稀释（早期 culture 不再自动传播）+ 政治开始

每阶段坑

阶段典型坑
5 → 15founder 不愿放权 / 招的人都 like 自己 (homogeneous)
15 → 30第一次 fire / 拆 team 没经验
30 → 100process > outcome / 创业 culture 稀释
100 → 500政治 / silos / 决策慢 / senior 流失

Reorg 实战 + 反模式

什么时候真需要 reorg · 什么时候不该 reorg · Reorg 5 步流程 · 反模式 · Reorg 后必做

Reorg（重组）是大杀器—— 解决问题 same time 制造问题。Director+ 必须懂何时 / 怎么 reorg。

什么时候真需要 reorg

  • 战略大变（pivot / 新 product line / IPO 后扩张）
  • 持续 cross-team 冲突（Conway's Law 跟 product 错配）
  • Senior 集中流失（结构问题表面化）
  • Quality / velocity 持续下滑（> 2 个 quarter，明显结构问题）
  • 新 leader 上任（"clean slate" 时机）

什么时候不该 reorg

  • 问题是 individual（某员工 underperform → fire 或 transfer，不是 reorg）
  • 问题是 culture / process（process fix 而非结构 fix）
  • 问题是 strategy 不清（先 fix strategy 再 reorg）
  • 距上次 reorg < 1 年（员工还在适应）
  • 关键 deadline 前 3 个月（reorg 必然降 velocity）

Reorg 5 步流程

  1. Pre-work (4-8 weeks): 跟 senior IC / EM / VP 1on1 收集 input，draft 多版本 design
  2. Decision (1 week): 关 doors with leadership team，pick a design + commit
  3. Communication (1 week):
    • 提前一天告诉受影响 EM
    • All-hands announce
    • 每个 sub-team 1on1 with new EM
    • Internal FAQ doc
  4. Transition (1-2 month): 老 project handover / 新 1on1 cadence 建立 / OKR re-align
  5. Retrospective (3 month): 评估 reorg 效果，调整

反模式

#反模式后果
1泄漏的 reorg —— 员工提前听到 rumorpanic / trust 崩
2没 prep 直接 announce员工 confused / 大流失
3每年 reorg 多次"reorg fatigue" + 大流失
4新 EM 没 calibration 时间perf review 不公
5不动 leader 只换标签cosmetic + 无效
6没 explicit 战略 rationale员工不 buy in
7合并 team 但留 multiple leader实际无变化 + 冲突
8FAQ 不答硬问题员工不 trust leadership

Reorg 后必做

  1. 1on1 every individual within 1 week with new manager
  2. Re-set OKR / 优先级 (老 OKR 可能不适用)
  3. 新 team identity 建（kickoff / off-site / 名字 / Slack channel）
  4. Quarterly retrospective: reorg 效果如何，是否需要调整
  5. 给 6 月时间—— 早期 turbulence 正常

Layoff 的 org 视角

Layoff 4 种类型 · Director 角色 · Layoff 后 90 天 · Survivor 关键 risks

2022-2025 tech layoff 周期让 layoff 成 "新常态"。Director+ 必懂。

Layoff 4 种类型

类型触发策略
Performance-basedbottom 5-10% perf真因业绩 fire，正常公司每年 5-10%
Reorg / strategy shift砍特定 business line / product影响整个 sub-org
Cost reduction财务压力 / 经济下行cross-team 砍 % headcount
Acquisition redundancyM&A 后重复职位砍 acquired company 重复职能

Director 角色

  • Pre-decision: 跟 leadership 提供 input —— 谁是 high-impact / who 不可或缺
  • Decision: 必须公平 + 可辩护（perf-based / role-based / 不能 demographic biased）
  • Communication: 准备 talking points / answer 难问题
  • Execute: 通知 + severance + benefits + reference
  • Aftermath: 跟 survivor 1on1（"survivor guilt" 真存在）
  • Rebuild: 重新 set expectation / OKR / vision for remaining team

Layoff 后 90 天

时间focus
Week 1跟所有 survivor 1on1，多 listen，不 over-promise
Week 2-4Re-prioritize OKR，砍 nice-to-have，focus core
Month 2新 cadence 建（如果 team size 变小，1on1 / planning / 跨 team 都得调）
Month 3Retrospective + 公开 commit "下一步是 X"，让 team 感受 forward motion

Survivor 关键 risks

  • "下次会不会是我"—— anxiety
  • Workload doubled—— 同 task 少人做
  • Trust 损失—— "公司不再 care me"
  • Best people 主动跳（不等被 layoff，先找新 job）—— 这是最大 risk

怎么 measure 组织健康

层 metrics · Annual Survey 标准 questions · Red flag 信号

"If you can't measure it, you can't improve it"—— 但 org metrics 不像 system metrics 直接。

3 层 metrics

Layer 1: Output（结果）

  • Quarterly OKR achievement
  • Customer outcome（NPS / churn / revenue）
  • Quality（P0 incident / SLO miss / customer complaints）
  • Velocity（PR merged / cycle time / deploy frequency — DORA metrics）

Layer 2: Talent

  • Attrition rate（quarterly / 全公司平均 vs benchmark）
  • Regrettable attrition rate（high-performer 流失，不是 underperformer）
  • Internal mobility（员工跨 team 调动率）
  • Promotion rate（每季度 % promoted）
  • Hiring funnel quality（offer accept rate / 6-month retention）

Layer 3: Engagement / Culture

  • Engagement survey（quarterly / anonymous）
  • Manager NPS / eNPS（员工 recommend 公司给朋友吗）
  • Skip-level feedback（director quarterly 跟 IC 跨 manager 聊）
  • Glassdoor / Blind 监控（虚假信息也 reveal 真实趋势）
  • 1on1 cadence health（manager 不取消 1on1 比例）

Annual Survey 标准 questions

Questionsignal
"我会推荐这里给朋友"eNPS
"我清楚公司战略"communication health
"我相信我的 manager"manager quality
"我的工作让我成长"career growth
"我有 psychological safety"culture health
"我有合适工具完成工作"operational excellence
"我经常 burnout"workload sustainability
"我感受 diversity 跟 inclusion"D&I health

Red flag 信号

  • Regrettable attrition > 8% / 年（行业平均 3-5%）
  • 1 个 manager 下属 attrition > 团队平均 2×
  • Skip-level meeting 频繁 surface "我跟 manager 沟通有问题"
  • Engagement survey 跨 quarter 持续下降
  • 1on1 cadence 一致被取消
  • Internal mobility 极低（< 5% / 年）→ 团队 stagnant
🎯 关键 take-aways
  • Director+ 不修人，修组织—— 通过结构 / 流程 / 权责 scale
  • Conway's Law: 组织结构决定系统架构；反向 Conway = 先架构再 reorg 组织 match it
  • Team Topologies 4 类: Stream-aligned (主) / Enabling / Complicated-subsystem / Platform + 3 种交互
  • Span of Control: EM 5-9 / Director 4-7 manager / VP 5-8 director
  • 5 大 org model: Functional / Product / Matrix / Spotify / Amazon two-pizza —— 按 stage 选
  • Talent Density > headcount —— 少 senior 比多 mid 强（但有 trade-off）
  • 团队成长 5 → 15 → 30 → 100 → 500 各阶段独特挑战
  • Reorg 是大杀器 —— 5 步流程 + 8 大反模式；不到 1 年内不 re-reorg
  • Layoff 后关键是 survivor —— 1on1 / re-prioritize / forward momentum
  • 3 层 metrics: Output / Talent / Engagement —— red flag 早识别比修复重要
