EM 的两个反直觉发现

① AI 提升个人产出 ≠ 提升团队产出。DORA 2024 报告显示，过度依赖 AI 的团队，change failure rate 反而上升。原因：individual leverage ↑ + system controls ↓ = 整体不稳。

② AI 拉大了 Senior 和 Junior 的差距而不是缩小。Senior 用 AI 加速判断；Junior 用 AI 跳过学习。半年后 Senior 越来越强，Junior 出现 "skill atrophy"（技能萎缩）。这是 EM 必须正视的问题。

研发流程重塑 · 6 个具体环节

代码生成 · Copilot / Cursor / Claude Code 在团队的真实 ROI · AI-augmented Code Review · Spec / PRD 生成 · 测试生成与 Eval · Deb…

EM 不需要懂模型训练，但必须懂 AI 在研发流程"哪些点"产生 ROI、哪些点产生 risk。

1.1 代码生成 · Copilot / Cursor / Claude Code 在团队的真实 ROI

不假思索全开 Copilot 的团队

  • 个人感觉"快了 30%"，但 lead time for change 没怎么改善
  • Code review 时间反而变长（review 大段不熟悉代码）
  • Change failure rate 上升 — AI 会"自信地"写出错代码
  • Junior 工程师"假装"会但不真懂；技术债隐性堆积

系统化 AI 团队

  • 明确"AI 高 ROI 任务"和"AI 低 ROI 任务" 的分类
  • Review 标准升级 — 强制要求"为什么这样"的 PR 描述
  • 对 Junior 增加 "讲解时间" — 防止 skill atrophy
  • 专门跟踪 AI-related defect rate（独立指标）

AI ROI 分类法 · 高/中/低 ROI 任务速查

不同任务，AI 加速效果差异巨大。EM 应该 explicit 引导团队往高 ROI 区做

高 ROI 区 (3~10x)
  • 样板代码 / glue code：REST endpoint、CRUD、proto buffer 转换
  • 测试代码：unit test、mock 生成
  • 语言/框架翻译：Python ↔ Go、React Class → Hooks
  • 文档生成：API doc、Changelog、README、Runbook
  • 日志 / 错误 triage：解析 stack trace、给出可能原因
中 ROI 区 (1.5~3x)
  • 新功能 implementation（小到中等）
  • Refactoring（已知意图的）
  • Code review 辅助（找 bug、安全 hot spot）
低 ROI / 反 ROI 区
  • 架构决策：AI 给的方案常 trick，但缺 trade-off 推理
  • 性能优化：需要深度上下文 + 测量 — AI 经常"看起来 fix 了" 但 regression
  • 安全敏感代码：crypto、auth — 一个 hallucination 就是事故
  • 并发 / 分布式：subtle bug，AI 难以正确处理
  • 大型重构：跨多文件、多模块 — AI 上下文不够

1.2 AI-augmented Code Review

两层 Review · 新标准

  1. 第 1 层 (AI 做)：linting、style、common bug、安全 hot spot、test coverage gap — Copilot Review / CodeRabbit / Codacy
  2. 第 2 层 (人做)：架构、命名一致性、trade-off、长期可维护 — 人类只看这一层

效果：人的 review 时间从 30 min/PR → 10 min/PR，但深度上升 — 因为只看"AI review 不了的"。

PR 描述模板（强制）

## What changed
[1~3 句]

## Why this approach (not others)
[trade-off / alternatives considered]

## AI assistance
[ ] AI generated > 50% of new code
[ ] Tests AI-generated (verify coverage)
[ ] Architecture decision by human

## Risks
[breaking change / perf / security]

这个模板逼工程师 explicit 标注"哪部分是 AI 写的"，给 review 者 calibration。

1.3 Spec / PRD 生成

1

PRD → 自动展开为 Tech Spec

PM 给 PRD，AI 帮工程师"展开" 5 个章节：API、数据模型、错误处理、迁移、测试 plan。工程师审查并补充。把 spec 从 4 小时 → 1 小时。

Claude / GPT-4 long context Internal RAG over existing services
2

Spec → API 草稿 + Mock Server

Tech spec 写好后，AI 生成 OpenAPI schema + 一个 fake mock server，前端/QA 可以并行启动。把"前后端等待时间" 砍掉。

Cursor in repo Prism mock
3

Spec → 自动生成测试用例骨架

根据 spec 里的"约束 / SLA / 错误码" 自动生成 test scaffolding（不只是 unit，包括 contract test）。

1.4 测试生成与 Eval

AI 时代的"测试金字塔" 升级

底层不变：Unit / Integration / E2E 还是按金字塔。

新增第 4 层 · "Eval Set"：

  • 如果你团队产品含 AI 功能（任何调用 LLM 的 feature），需要建一个"Eval set" — 一个评估集 + 评分函数 + 人审样本。
  • 跟 unit test 一样跑在 CI 里 — 但更慢、更贵、需要 token budget。
  • 关键指标：accuracy / hallucination rate / format compliance / latency / cost per 1k requests

💡 没有 Eval set 的 AI 功能就是"风险炸弹"。EM 必须把"上 AI 功能"和"上 Eval set" 绑成同一个 epic。

1.5 Debug & Log Triage

故障响应流程的 AI 改造

步骤Pre-AIAI-Native
P1 告警触发on-call 接到 page同上
初步定位 (T+5 min)人翻 dashboard、log、codeAI agent 已经聚合 dashboard 异常、grep 相关 log、给出 top 3 suspect

1.6 RAG-based Onboarding · 新人 30 天 → 7 天

Internal Knowledge RAG · 把"团队知识"喂给 LLM

EM 投资 ROI 最高的 AI 项目之一 — 在新人 onboarding 上立刻见效

索引内容
  • Confluence / Wiki 全量文档
  • code repo + commit message + PR description
  • 过去 1 年的 Slack #eng-channel + 工单系统
  • 架构图 / ADR / Decision Log
  • Runbook + Postmortem 归档
效果
  • 新人提问 "我们这个服务为什么用 Kafka 不用 NATS" — RAG 答 "见 ADR-2023-12，原因是 X，但已有 issue #4567 在重评估"
  • 新人 30 天 → 7 天能开始独立做 PR
  • Senior 工程师的"被问问题" 时间从 8 h/week → 2 h/week
EM 风险点
  • RAG 答案有 hallucination — 必须显示"来源链接" 让人验证
  • 权限边界 — 不能让 RAG 回答跨权限内容（薪资 / 评审 / 法务）
  • 过期内容 — 每月 evict + 重新 index

2.2 AI 时代 · 5 个必加新指标

① AI Leverage Ratio · AI 杠杆比

公式：(AI 生成代码行数 / 总新增行数) 或 (AI 接受率 acceptance rate)

看法：太低 (< 20%) → 团队没真用 AI；太高 (> 70%) → 警惕"skill atrophy" / 质量下滑

健康区：30~55%

② AI-related Defect Rate

公式：(标记为 AI 主写的 PR 引入的 bug / 总 AI 主写 PR)

看法：跟 non-AI 对照 baseline。如果 AI 引入的 bug 率显著高 — review 流程需加强

③ Eval Pass Rate (针对 AI 功能)

对每个 AI-powered feature，建 eval set 跑在 CI 里。目标：accuracy > X%, hallucination < Y%

没有 eval 的 AI feature 不允许上线 — 这是新的"上线 quality gate"。

④ Token / Cost Budget Tracking

如果团队产品调 LLM，$ per 1k requests 是新的 SLA 指标。

跟传统 "p99 latency" 并列，进 dashboard。EM 必须盯，不然账单失控。

⑤ Junior Skill Health Index

非标准指标，但 EM 自己要建：通过 design review / mentor feedback / 主动 quiz，定期评估 junior 是否"靠 AI 拐杖" 而没真正建立基本功。

很难量化但必须主观追踪 — 否则 12~18 个月后整个团队 senior 出现断层。

⑥ Knowledge Diffusion (Optional)

跨 team 知识共享次数 / RAG query 命中率。检测"团队是不是被 AI 拐成孤岛"。

2.3 度量陷阱 · 千万别用这些

陷阱 1：用代码行数 / commit 数衡量个人产出

AI 让任何人"产出代码行数" 翻 3 倍 — 但根本不代表 impact。用这个度量 = 鼓励 AI bloat。

✓ 用 sprint 内"明确价值交付" (feature, KPI move, bug fix) 衡量

陷阱 2：用 Copilot 接受率作为唯一的 AI 健康指标

接受率高，可能是开发者"接受了一切"，没好好审。接受率低，可能是 AI 没帮到 / 配置错。单看一个数没用。

✓ 把接受率和 defect rate 一起看，互为校验

陷阱 3：把 AI 加速当 capacity gain — 砍 headcount

"AI 让我们快 30%，所以可以减 30% 人" — 这是 2024 一些 CFO 的算法，但 DORA 数据反驳了它。AI 加速不等价于团队 capacity 翻倍，反而可能因 review/quality overhead 抵消大部分收益。

✓ EM 必须用数据反推 CFO：拿出 lead time / CFR / quality 数据证明"砍人 = 净亏"

招聘标准的重构

JD 关键词演变 · 面试题怎么改 · Bar 怎么升 / 怎么降

3.1 JD 关键词演变

Pre-AI JD（2022 风）

  • 5+ years C++
  • 熟悉 STL / 多线程 / 网络编程
  • 有大流量系统经验
  • 解决复杂问题

AI-Native JD（2026 风）

  • 5+ years C++ + 1+ year 用 AI 工具高效开发的实践
  • 熟悉 STL / 多线程 / 能用 AI 加速但能识别 AI 错误建议
  • 大流量系统经验 + 对 GenAI 在系统中的能力边界有判断
  • 解决复杂问题 + 能 frame 问题让 AI 帮你拆解
  • (加分) 有 prompt engineering / RAG / eval set 经验

3.2 面试题怎么改

3 类新题型

① "With-AI" Coding 题

给候选人一个真实任务（半小时），允许用 Copilot / Cursor / ChatGPT。考的是：

  • 问题分解能力（要不要把任务拆给 AI 做？哪些自己做？）
  • 验证能力（AI 给的代码能不能看出问题？）
  • 调试能力（AI 跑出 bug 怎么办？）
  • "拒绝 AI" 的能力 — 真正 senior 知道哪些任务 AI 帮不上忙

② 系统设计题增加 "AI 模块"

不再是"设计一个 Twitter"，而是"设计一个 GenAI-powered Customer Support 系统"，必考：

  • LLM provider 选型 + 备份策略
  • RAG vs Fine-tune 的判断
  • Token cost budget + rate limiting
  • Hallucination 控制（Guardrails / Eval / Human-in-the-loop）
  • Latency + caching 策略
  • 合规 / PII 处理

③ 行为题增加 "AI 决策" 题

  • "讲一次你团队过度依赖 AI 出了问题的故事？"
  • "你怎么帮 junior 开发者用 AI 但不丧失基本功？"
  • "你怎么评估一个 AI feature 是否应该上线？"

3.3 Bar 怎么升 / 怎么降

能力变化方向原因
语法 / 框架熟练度↓ 降低AI 能补，不是 differentiator
算法手撕↓ 适度降低AI 现场能给最优 — 考"为什么这种解" 更重要
System Design↑ 上升AI 暂时帮不上 — 抽象 / trade-off 仍是人的强项
判断力 / Trade-off↑↑ 大幅上升AI 给"看起来对" 的代码，但 trade-off 判断只能人做
沟通 / 写作↑ 上升spec / review / mentor 更重要 — AI 替代不了
领域知识深度↑ 上升AI 在通用任务上 ok，但 domain expert 仍稀缺
对 AI 工具的判断力↑↑↑ 新维度"知道何时用 / 不用 AI" 比"会用 AI" 重要

团队系统 vs 个人英雄 · EM 容易掉的最大坑

个人杠杆 ≠ 团队杠杆 · 团队 AI Workflow 设计 · 5 个支柱 · 培养 AI-positive 文化（不是 AI-mandatory）

4.1 个人杠杆 ≠ 团队杠杆

常见 EM 陷阱：自己用 Cursor 很爽，就以为团队也都很爽

个人现实：你用 Cursor 写代码、prompt 写 doc、Claude 做 brain dump — 主观感受"快了一倍"。

团队现实：

  • 团队 12 个人，每人用 AI 的姿势不同 — 有人深度用、有人不会用
  • Review 时间没减少 — 因为别人不知道你是 AI 写的
  • Onboarding、文档、知识共享 没自动改善 — 还是手动
  • 没有团队级 prompt 库 / pattern 库 — 重复造轮子
✓ EM 的关键任务：把"个人 AI 杠杆" 沉淀为"团队 AI workflow"

4.2 团队 AI Workflow 设计 · 5 个支柱

① 团队 Prompt Library

把团队成员发现的"好用 prompt" 收集到一个 repo（每周 1 次添加）：

  • Code review prompt
  • Spec 展开 prompt
  • Postmortem timeline prompt
  • Refactor 建议 prompt

每月 retro 时讲 "本月最好用的新 prompt"。

② RAG Knowledge Base

索引内部文档 + code，团队成员都可以查询，权限分级。

③ 团队级 Eval Pipeline

所有 AI feature 共享一个 Eval framework — 不要每个 feature 自建。

④ AI Cost Dashboard

token 用量 + cost per request + 各 feature 占比 — 每周 review。

⑤ AI Policy 文档

明确"哪些数据不能传给公网模型"、"哪些 AI 工具批准使用" — 见 模块 5。

4.3 培养 AI-positive 文化（不是 AI-mandatory）

4 条文化原则

  1. "AI as a teammate, not a tool"：把 AI 当 pair programmer 看待 — 它会犯错，你要 review；它有强项，你借力。
  2. "Disagree with AI freely"：鼓励对 AI 输出 push back。Junior 工程师最大风险是"被 AI 说服"。
  3. "AI use is transparent"：在 PR / spec 里 explicit 标注 AI 写了多少。不是惩罚，是 calibration。
  4. "Skill 不能因 AI 萎缩"：每周给 junior 安排 1 个"不用 AI" 的小练习 — 保持基本功。
05

风险 & 合规 · EM 必须有的 4 条红线

数据 / 代码泄漏 (Confidentiality) · IP & 许可证 · Hallucination 责任归属 · 团队 AI Policy · 1 页模板

5.1 数据 / 代码泄漏 (Confidentiality)

真实案例（2023 Samsung）

三星员工把内部 source code 粘进 ChatGPT 让它帮 debug — 这些代码被 OpenAI 用于训练。三星紧急禁用公网 AI 工具。

✓ EM 必须：① 明确公司哪些代码 / 数据不能传公网模型 ② 提供合规替代（On-prem LLM / Enterprise tier 不训练你的数据）③ 在 onboarding 第一周讲清楚

分级管理（参考）

数据等级可用 AI 工具
Public / Open Source任意
Internal · 一般业务代码Enterprise 版（不训练你数据）+ Copilot Business
Internal · 敏感（客户数据 / 安全 / 财务）仅 On-prem / Private 部署 LLM
Regulated · PII / 医疗 / 金融仅特定合规通过的部署，需 Legal 批

5.2 IP & 许可证

3 个最常被忽视的问题

  • 训练数据归属：Copilot 早期被 GPL 代码训练 — 输出的代码可能"实质相似" 触发 GPL。
  • 客户合同条款：很多企业客户合同里明确"不允许 AI 生成代码进入交付物" — 你必须知道你 BU 的客户合同写了什么。
  • 开源贡献：你的开源 project 接受 AI 生成的 PR 吗？大部分 OSS 项目（Linux Kernel、Postgres）已经明确拒绝。

💡 在 To-B / 强客户合同的行业，AI Policy 必须经过 legal + sales 共同确认。EM 不能自己拍。

5.3 Hallucination 责任归属

Human-in-the-loop · AI 决策的责任拆解

谁对 AI 输出错误负责？永远是用了它的人。EM 必须建立机制保证有"人"在路径上

3 类决策
  • AI 单独决策 (Autonomous)：高风险 — 几乎所有生产系统都不应该走这条
  • AI 推荐 + 人批准 (Suggested-then-Approved)：90% 的研发场景应走这条
  • AI 辅助 + 人主导 (Assisted)：debugging / brainstorming — 最安全
案例 · AI 自动 merge PR 的红线有些团队让 AI bot 看到 "trivial change" 就自动 merge — 但只要这个 PR 触及核心模块就必须人 review。规则要 explicit 写在 CI policy 里。

📄 模板：团队 AI Policy v1

# Team AI Policy (v1, owner: K. Chan, last review: 2026 Q2)

## 1. 批准的工具
- 编码：Copilot Business（Enterprise tier，不训练我们数据）
- 对话/文档：内部部署 ChatGPT Enterprise / Claude for Work
- 禁用：ChatGPT 个人版（数据泄漏风险）、公开 codeium 免费版

## 2. 数据红线 · 绝对不传公网模型

## 3. PR 标注
- 所有使用 AI 主写（>50%）的 PR，必须在描述里勾选 [AI-Assisted]
- AI 主写的 production code，至少一个 senior reviewer

## 4. AI Feature 上线 Gate
- 必须有 Eval Set 进入 CI

- 必须 legal review (如果 customer-facing)

## 5. Skill Health
- Junior 每月至少 1 次 "no-AI" coding 练习（在 1on1 里 explicit 安排）

## 6. 触发 review 条件
- 团队规模翻倍
- 出现 AI-related incident
- 新的 LLM / 工具上市
- 每 6 个月强制 review

面试题集 · 10 道高频题 + 高分答案

这是 2025~2026 EM 面试里出现频率暴涨的题型

这是 2025~2026 EM 面试里出现频率暴涨的题型。背熟回答骨架，能 cover 80%。

Q1AI 改变了你团队的工作方式吗？具体怎么改？
低分答法

"我们都开了 Copilot，写代码快多了。" — 太浅。

高分结构 · 5 个具体环节
  1. 代码生成：Copilot Business 全员开通，AI Leverage Ratio 维持在 ~40% 健康区。
  2. Code Review：CodeRabbit 做第 1 层（style / common bug），人做第 2 层（架构 / trade-off）。Review 时间 -40%，深度反而上升。
  3. RAG Onboarding：内部 wiki + code 索引，新人 30 天 → 10 天可独立 PR。
  4. CI Log Triage：AI 自动归类 CI 失败，节省 senior 工程师 8 h/week。
  5. Postmortem：AI 起草 timeline，人补 root cause。

同时我也 explicit 提到 trade-off："但我也看到风险 — DORA Change Failure Rate 上升 5%，所以我加强了 second reviewer 机制。AI 不是免费的。"

加分

提到具体数字 + 提到 trade-off + 提到机制化应对 = 立刻和"只会用 Copilot" 的候选人拉开档次。

Q2你怎么衡量 AI 工具给团队带来的 ROI？
3 层指标
  1. Input 层：AI tool 成本（Copilot license + token usage + on-prem GPU cost）
  2. Output 层：DORA 4 指标 + 自定义"AI Leverage Ratio" + "AI-related Defect Rate"
  3. Outcome 层：Lead time / time-to-market 改善 / 新人 onboarding 时间 / Senior 工程师"被打扰" 时间

关键 caveat："我从不孤立看一个指标"。比如 Copilot 接受率 60% — 如果同时 defect rate 上升，说明大家"无脑接受"，需要加强 review。

加分

"我也跟 finance 一起算过 — 我们 BU 每月 AI cost 是 $X，对应节省的工程师时间换算成 fully-loaded cost 大约 $Y，ROI ≈ 1:5。这帮我跟老板要 budget 时有数据。"

Q3讲一次 AI 给你团队带来麻烦的故事。
STAR

S: 团队 junior 工程师用 Copilot 自动生成了一段错误处理代码，把一个 retry 写成了无限循环。Code review 没看出来（因为代码看起来"很合理"）。Production deploy 后引发短暂 cascade failure。

T: 既要修这个具体问题，也要从机制上预防。

A:

  1. Hotfix（5 min）
  2. Postmortem（AI 草稿 + 人补 root cause）
  3. 识别 root cause 不是 AI — 是 review 流程没适应 AI 时代。AI 生成的代码"语法正确、命名漂亮"，反而让 reviewer 放松警惕。
  4. 引入新机制：① PR 必须标注 AI-assisted (>50%) ② AI-assisted PR 必须 2 个 reviewer ③ retry / loop / 资源释放 类代码必须有 unit test 覆盖 corner case ④ 团队培训 "AI 输出的危险信号"
  5. 6 个月后 AI-related defect 显著下降，但 AI 使用率没下降。

R: AI-related defect 从月均 4 个 → 1 个。团队对 AI 的态度从"无脑用" 到"系统用"。

加分

"我学到的 lesson 是 — AI 不是问题，缺乏适应 AI 的 process 是问题。EM 的工作不是禁 AI，是把 review 流程升级到匹配 AI 速度。"

Q4Junior 工程师过度依赖 AI 怎么办？
3 个具体做法
  1. 1on1 里 explicit 提：不批评，但讲清楚 "skill atrophy" 的 5 年风险。"AI 在 P5 阶段加你 50%，在 P7 阶段加你 20%，在 P8 阶段几乎加不了 — 因为 P8 干的事 AI 不会。所以 P5 时跳过基本功，3 年后你卡死。"
  2. 设计 "no-AI" 练习：每月 1 次 design review、1 次 debugging 演习、1 次 architecture 讨论 — 不允许用 AI。强迫纯思考。
  3. 讲解任务（teach-back）：AI 帮他写了一段 — 让他给我讲"为什么这样写"。讲不清楚 = 他没真懂。这强迫他不能只 copy-paste。
为什么不直接禁 AI

禁 AI 不现实（他会偷偷用）+ 不公平（其他 senior 都用）+ 短期看输出会下降。关键是 "用 AI 但保持成长"。

Q5你的团队产品要加一个 GenAI 功能，你怎么决定 ship-or-not？
5 个 Quality Gate
  1. Eval set + accuracy threshold：必须有，且 baseline 跑过
  2. Hallucination 控制：guardrails + fallback path + human-in-the-loop（适用场景）
  3. Token / Cost budget：peak cost 不超过 $X / 1k requests
  4. Latency P99：要 ship 的场景能接受这个延迟
  5. Compliance review：legal + security 签字（特别是 customer-facing）

缺一个我都不让上线。我会让 PM 知道："这不是技术 push back，这是 risk control" — 5 个 gate 都过了之后再讲 'go to market'。

加分

"我也会加 'rollback plan' — 如果上线后 hallucination rate 超过阈值，能不能 1 小时内 fallback 到非 AI 路径？没有 rollback plan 的 AI feature 我不让 ship。"

Q6你怎么 hire 一个 "AI 时代" 的 senior 工程师？
3 个新维度
  1. "With-AI" Coding 题：给他真实任务，允许用 AI。考察 — 任务分解、AI 输出的批判性审视、知道何时不该用 AI
  2. System Design 加 AI 模块：必考 RAG / Eval / Token Budget / Guardrails
  3. 行为题："讲一次你团队过度依赖 AI 出问题的故事" — 没踩过坑的人是没真用过 AI 的
不变的维度（不要被 AI 噪音误导）

① 沟通和写作（AI 替代不了）② 判断力和 trade-off（AI 给不出）③ 领域知识深度（垂直行业 AI 训练数据不深）④ 跨人合作能力

加分

"我会问候选人 — '你最近什么时候故意不用 AI？为什么？' — 答得出来的，是真懂 AI 的。"

Q7CFO 说 "AI 让团队效率 +30%，今年砍 30% headcount"，你怎么 push back？
数据 + Frame 转换
  1. 挑战前提：用 DORA 2024 报告反驳 — 高 AI 采用团队的 lead time 改善仅约 15%，而非 30%；CFR 还上升 7%。
  2. 切换 Frame：AI 不是"capacity 增加"，是"capacity 重分配"。砍人 = 同样的工作但 capacity 反而下降，因为 review/test/coordination 不是 AI 能 cover 的。
  3. 给替代方案：与其砍人，不如砍 OPEX 别处（云、SaaS、外包）。或者扩 scope — 同样 headcount 做更多事。
  4. 底线："如果一定要砍，我可以承诺承担 -10% headcount，但不能再多 — 否则我对 SLA / customer commitment 不能 own。"
加分

"我会主动跟 CFO 一起做 ROI 分析 — 显示我们 BU 的 AI cost 投入和效率 gain 的真实 ratio，让数字说话。"

Q8你怎么处理 "AI 工具公司说不训练你数据，但客户合同要求我们 0 风险" 这种矛盾？
分层策略
  1. 读懂合同：找 legal 解读 — "0 风险" 是字面 0 还是"商业 reasonable best effort"？
  2. 分级数据：客户数据 → 严格 (on-prem only)；内部代码 → 中等 (enterprise tier ok)；通用学习 → 宽松。
  3. 给 alternative：客户相关的 AI 用 on-prem (Azure OpenAI VPC / 内部 LLM)；其他用 enterprise public tools。
  4. 对客户透明：写 1-pager 给客户讲我们的 AI Policy；让他们看见 control 在我们手上。
  5. 不可避免妥协时：跟 sales / legal 一起跟客户重谈合同 — 如果客户彻底拒绝任何 AI，要让 finance 明白这是 "代价"。
Q93 年后 AI 让 EM 工作发生什么变化？你怎么准备？
3 个预测 + 3 个准备

预测：

  1. 团队会变小。同样产出 8 人 → 5 人。但 5 人都是 senior。
  2. EM 用更多时间做"系统设计 + 决策"。1on1 时间不变，跨域协调和 AI Policy 时间变多。
  3. "AI 能做的部分"会从代码 → review → architecture → strategy 不断进阶。EM 自己的"差异化"必须前移。

我的准备：

  1. 更深的商业 / 战略素养。AI 不会做的事 — 在政治环境里判断、跨组协调、解雇决定。
  2. 更系统化的团队 workflow。把"我个人 AI 能力" 沉淀成"团队级 AI workflow"。
  3. 持续 hands-on。我每周 4~6 小时碰代码 / 工具，确保我对 AI 演进有 first-hand 感知 — 不能成"听汇报的 manager"。
Q10给我讲一个你主动拒绝 AI 的决策。
为什么是好题

这题专门筛掉"AI 狂热者"。Senior EM 知道 — AI 不是万能。

高分答法

S: 某客户提议我们用 LLM 自动审 5G 网络配置文件 — 听起来很酷。

分析：① 配置错误 = 网络中断 = 5 个 9 SLA 违约 ② LLM 在协议细节上有 hallucination 史 ③ 即使准确率 99%，1% 错误率 × 数千节点 = 灾难。

决策："我拒绝 'AI 自主审配置'。但我同意 'AI 作为第一层 review 提建议，人最终 approve'。这样既享受 AI 速度，又保留 5 个 9 的 control。"

结果：客户接受了这个 frame，项目按"AI-assisted but human-final" 上线，0 incident。

为什么这答案高分

① 显示对 AI 能力边界的判断 ② 给出 nuanced 方案而不是 yes/no ③ 把 risk 用 SLA 量化 ④ Senior 才能做这种"拒绝 hype" 的决策
