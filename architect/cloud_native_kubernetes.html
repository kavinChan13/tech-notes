01

开篇：Kubernetes 是什么

一句话——它是「容器编排系统」，帮你自动管理成千上万个容器

如果只用一句话概括：Kubernetes（简称 K8s）是一个「容器编排系统」——你把应用打包成容器交给它，它负责在一堆机器上自动地部署、调度、扩缩容、故障自愈、滚动升级和联网。你不再关心「这个容器该跑在哪台机器、挂了谁来重启、流量怎么分发」，这些统统交给它。

名词解释 · Kubernetes / K8s / 容器编排

Kubernetes 源自希腊语「舵手 / 领航员」（掌舵集装箱货轮的人，正好呼应「容器」）。K8s 是它的缩写——首尾 K、s 之间有 8 个字母。容器编排（Container Orchestration）指对大量容器进行统一的调度、编排与治理，就像乐队指挥协调众多乐手。

为什么会有 K8s？它解决了什么问题

容器（Docker）解决了「单个应用怎么打包、怎么一致地跑起来」。但真到生产环境，你面对的是成百上千个容器、几十上百台机器，一连串新问题冒出来——这些正是 K8s 要解决的：

只有容器时的难题K8s 怎么解决
这个容器该放到哪台机器上跑？调度器按资源/亲和性自动选最合适的节点
容器崩了 / 机器宕了谁来救？自愈：自动重启、重新调度到健康节点
流量高峰要加机器，低谷要省钱？弹性伸缩：按指标自动增减副本（HPA）
发新版本怎么不停机、出问题怎么退回？滚动升级 / 一键回滚
成百上千容器 IP 乱变，怎么互相找到？Service + DNS 提供稳定入口与负载均衡
密码、配置怎么安全下发？ConfigMap / Secret 与镜像解耦
生活类比：集装箱货运 + 港口调度中心

容器（Container）就是标准化的集装箱：不管里面装什么，尺寸接口统一，哪条船、哪个港口都能装卸。但光有集装箱不够——一个大港口每天有成千上万个箱子进出，需要一个调度中心安排「哪个箱子上哪条船、堆到哪个泊位、哪台吊机来搬、船坏了货怎么改道」。Kubernetes 就是这个港口调度中心（舵手）：你只管把集装箱交过去并说明「我要 3 个这样的箱子一直在线」，剩下的调度、搬运、补货、改道全自动完成。

映射：集装箱=容器 · 货轮/泊位=机器节点 · 调度中心=Kubernetes 控制平面 · 「保持 3 个在线」=声明式期望状态
K8s 的血统：它脱胎于 Google 内部运行了十多年的集群管理系统 Borg，2014 年开源，2018 年成为 CNCF 首个「毕业」项目。如今几乎所有云厂商都提供托管 K8s（EKS/GKE/AKS/ACK），它已是云原生时代事实上的「分布式操作系统」——向下屏蔽机器差异，向上提供统一的应用运行接口。

02

什么时候该用 K8s / 适用场景

K8s 的价值只有在「容器多 × 机器多 × 变更频繁」时才盖过它的复杂度

K8s 很强，但绝不是「万物皆可 K8s」。它的本质是容器编排——只有当你要管理的容器和机器多到人工运维扛不住、且真正需要自愈/弹性/独立发布/服务发现这些能力时，它的收益才盖过它陡峭的学习与运维成本。下面把「该用」「别用」和「怎么判断」讲清楚。

✅ 典型适用场景

🧩 微服务架构

几十上百个服务、各自独立部署与扩缩、服务间要互相发现调用——K8s 最核心的战场。Service+DNS 做发现、Deployment 做独立发布、网关做统一入口（见 §7–§10）。

📈 弹性伸缩的流量型业务

电商大促、直播、在线教育、票务这类波峰波谷明显的系统。HPA 按指标自动扩缩副本、Autoscaler 自动加减节点：高峰扛得住、低谷省成本。

🛡️ 高可用 / 要自愈的关键服务

要求「机器宕、进程崩自动恢复，尽量不影响用户」。控制循环天然提供重启、重新调度、滚动升级与一键回滚。

👥 多环境 / 多团队共享底座

用 namespace + RBAC + ResourceQuota 隔离 dev/test/prod 或多团队在同一集群，统一治理与 CI/CD、GitOps 发布流水线。

☁️ 混合云 / 多云 / 避免锁定

K8s 屏蔽机器与云厂商差异，同一套 YAML 可在 AWS/GCP/Azure/私有机房跑。追求可迁移性、多活容灾时很合适。

🏗️ 平台工程 / 内部 PaaS

把 K8s 当底座，用 CRD + Operator 把「一键装一套 Kafka/数据库/AI 任务」做成自助平台（见 §11 Operator）。

🔬 批处理 / 大数据 / AI

Job/CronJob 跑定时与批处理；Spark/Flink on K8s、GPU 调度做模型训练与推理服务，日益主流。

📡 电信云原生（CNF）

5G 核心网、容器化网络功能大量跑在 K8s 上，用 Operator 管有状态网元生命周期，用 FCAPS（CM/PM/FM/SWM）对齐运维（见 §11）。

❌ 不太需要（甚至别上）K8s 的场景

场景为什么不必上 K8s更合适的选择
单体小应用 / 个人项目 / 流量平稳一两台机器就够，K8s 是「杀鸡用牛刀」Docker Compose / systemd / 单机部署
团队没有运维 / 平台能力学习曲线陡、运维复杂，没人 hold 得住反成负担Serverless 容器（Cloud Run / Fargate）
纯静态站点 / 简单 API无需编排，托管服务更省心CDN / 函数计算 FaaS / 托管 PaaS
极致低延迟 / 强本地状态的特殊负载多一层抽象反而带来开销与不确定性裸机 / 专用调度 / 就近部署
生活类比：要不要请一支物流车队

你在家门口摆个小摊，进货自己骑电动车就够了（单机部署）；但你要经营一家全国连锁、每天成千上万件货物在几十个仓库间调拨，就必须有一套调度系统 + 车队（K8s）。规模不到，硬请车队只是徒增成本与管理负担——工具要匹配规模。

一句话判断信号：当「要管的容器数 × 机器数 × 变更频率」大到人工运维扛不住，且你确实需要自愈、弹性、独立发布、服务发现时，才该上 K8s。规模不到临界点，它带来的复杂度会超过收益——这和微服务里「Monolith First」是同一种克制。
两道门槛都过（规模到位 + 确实需要编排能力）才上 K8s，否则更轻量的方案更划算
03

云原生：不是「上云」，而是四大原则

Cloud Native 是一套构建与运行方式，Kubernetes 是它的操作系统

「云原生（Cloud Native）」不是「把应用搬到云上跑」，而是一套为弹性、动态、分布式环境而生的构建与运行方式。CNCF 的定义落到四条可执行的工程原则上，它们互相咬合，缺一不可。Kubernetes 之所以成为事实标准，正是因为它把这四条原则做成了平台能力。

名词解释 · CNCF

Cloud Native Computing Foundation，云原生计算基金会。托管 Kubernetes、Prometheus、Envoy、etcd 等项目，是云原生技术栈的中立治理组织。

📦 ① 容器化（Containerization）

把应用及其全部依赖打进一个不可变镜像，用 Linux namespace（隔离视图）+ cgroup（限制资源）实现进程级轻量隔离。「一次构建，处处运行」，消除环境差异。

底层原理见 容器底层原理。

🧩 ② 微服务架构（Microservices）

把大应用拆成一组围绕业务能力、独立部署、独立数据、通过网络通信的小服务，让团队各自演进。拆分边界、Saga、网关见 微服务架构。

🧊 ③ 不可变基础设施（Immutable Infrastructure）

服务器/实例一旦部署不再原地修改——要变更就用新镜像替换整个实例，旧的销毁。杜绝「配置漂移」，让部署可重复、可回滚、可审计（「牛群，不是宠物」）。

📜 ④ 声明式 API（Declarative API）

你只描述「我要什么」（期望状态 desired state），而不是「怎么一步步做」（命令式）。系统持续把现状拉向期望状态。这是 K8s 全部行为的内核，见 §4。

生活类比：点外卖 vs 自己下厨

命令式像自己下厨：你要亲手完成「洗菜→切→炒→装盘」每一步，中途锅糊了得自己救。声明式像点外卖：你只说「我要一份宫保鸡丁」（期望状态），平台负责接单、做、配送、出问题重做——你不关心过程。K8s 就是那个永不下班的外卖平台：你交一份 YAML（订单），它 7×24 盯着，少了就补、坏了就换。

映射：订单=期望状态 YAML · 外卖平台=控制平面 · 骑手重送=自愈 · 永不下班=控制循环
原则解决的问题在 K8s 里的体现
容器化环境不一致、部署重、隔离弱Pod / 容器运行时（containerd）/ 镜像
微服务单体耦合、无法独立扩缩每个服务一组 Deployment + Service
不可变基础设施配置漂移、回滚困难镜像 tag 滚动更新 / 重建而非修改

深入理解 · 为什么这样设计
基础「声明式」到底好在哪？命令式不行吗？
是什么
命令式 = 你写一串「怎么做」的步骤（先启动 3 个进程，挂了你自己去重启）；声明式 = 你只写「最终要什么」（replicas: 3），系统自己想办法达成并维持。
为什么
命令式脚本只在「执行那一刻」有效，执行完就没人管了——机器重启、进程崩溃、有人手动改动之后，现实和你的意图就偏离了，而且没人知道。声明式把「意图」变成一份持续存在的目标，系统 7×24 拿现实和目标对比、自动纠偏。
解决了什么
① 自愈：Pod 挂了自动补齐；② 幂等：同一份 YAML 应用一次和一百次结果一样，不会重复创建；③ 可审计/可回滚：意图是一份可以进 Git 的文件，谁改了什么一目了然。
基础「配置漂移」是什么？不可变基础设施怎么根治它？
是什么
配置漂移（Configuration Drift）指：同一批本应一模一样的服务器，因为长期被手动打补丁、改配置、装依赖，慢慢变得「各不相同」。于是出现经典甩锅——「在我机器上是好的」。
为什么
只要允许「原地修改」，随着时间推移人为改动必然累积，没人能说清某台机器现在到底是什么状态。不可变基础设施禁止原地改：要变更，就用新镜像整体替换旧实例，旧的直接销毁。这样每台实例的状态 100% 由镜像决定，可精确复现。
解决了什么
环境一致性、可重复部署、秒级回滚（回滚 = 换回旧镜像）。业界叫「牛群，不是宠物」——实例是可随时替换的牛，不是需要精心照料、坏了要抢救的宠物。
04

Kubernetes 架构：声明式的控制循环

理解这一张图，就理解了 K8s 90% 的行为

Kubernetes 是一个分布式状态机：所有期望状态存在 etcd，所有变更都经过 api-server，无数个 Controller（控制器）各自跑一个「观察现状 → 对比期望 → 采取行动」的控制循环（Control Loop / Reconcile Loop），把系统持续拉向你声明的样子。这就是声明式 API 的落地机制。

控制平面负责「大脑决策」，工作节点上的 kubelet 负责「手脚执行」；一切经 api-server，状态存 etcd
组件角色职责
kube-apiserver唯一入口所有读写的 REST 入口；认证、鉴权、准入校验；无状态可水平扩展
etcd真相之源Raft 强一致 KV，存全部期望状态与集群状态；只有 api-server 直接访问它
scheduler调度器把未分配的 Pod 按资源/亲和/污点等打分，绑定到最合适的 Node
controller-manager控制循环集Deployment/ReplicaSet/Node 等控制器，持续 reconcile 现状=期望
kubelet节点代理接管本节点 Pod 生命周期，调容器运行时拉起容器，上报状态与健康探针
kube-proxy网络转发在每个节点维护 Service 的转发规则（iptables/IPVS），见 §9
containerd容器运行时通过 CRI 接口真正拉起/销毁容器（替代早期 dockershim）
控制循环的心智模型：Watch（监听变化）→ Diff（对比期望与现状）→ Act（调用 API 修正），无限循环。所以删掉一个 Pod 它会自己长回来——不是「魔法」，是 ReplicaSet 控制器发现「现状 2 < 期望 3」后补了一个。
生活类比：恒温空调

你设定「26℃」（期望状态，声明式），空调温度传感器持续测当前温度（现状），高了就制冷、低了就制热（reconcile），直到逼近 26℃。你从不需要告诉它「压缩机转 5 分钟」（命令式）。K8s 就是给整个集群装的「恒温器」。

深入理解 · 为什么这样设计
进阶etcd 是什么？为什么非它不可、还非要用 Raft？
是什么
etcd 是一个分布式的键值数据库，K8s 把「集群里的一切」（有哪些 Deployment、每个的期望副本数、每个 Pod 现在在哪台机器……）全存在里面。它是整个集群唯一的「真相之源（source of truth）」。
为什么
集群大脑（控制平面）为了高可用要跑多个副本，多个副本必须对「当前期望状态」达成完全一致——否则一个副本以为要 3 个 Pod、另一个以为要 5 个，就打架了。Raft 共识算法保证：即使少数节点宕机，多数派也能就「每一次写入」达成唯一一致的结论。这就是为什么用 etcd（内建 Raft）而不是随便一个数据库。
解决了什么
控制平面的高可用 + 强一致。也因此 etcd 是集群最关键、最需要备份的部件——etcd 数据没了，集群的「记忆」就全没了。共识原理详见 Raft / Paxos。
进阶为什么所有请求都必须经过 api-server，不能直连 etcd？
是什么
api-server 是集群唯一的读写入口：kubectl、各个 Controller、kubelet，任何组件想读或改状态，都得通过它，由它去操作 etcd。
为什么
如果谁都能直连 etcd，就没有统一的认证（你是谁）、鉴权（你能不能改）、准入校验（这份 YAML 合不合法），也没法给其他组件推送变更通知（watch）。把这些横切能力收敛到一个「守门人」，etcd 才安全、集群行为才可控。
解决了什么
安全边界统一、状态变更可被审计与订阅（Controller 正是靠 watch api-server 才知道「有东西变了，我该干活了」）。这也是一种典型的「网关/门面」设计。
基础「控制循环 reconcile」到底在循环什么？举个例子
是什么
一段永不停止的逻辑：看现状 → 比期望 → 做修正，然后再来一遍。每种资源都有对应的控制器（Deployment 控制器、Node 控制器……）。
举例
你声明 replicas: 3。某个 Pod 所在节点断电挂了 → 控制器下一轮循环发现「现状只剩 2 个，期望 3 个」→ 立刻创建 1 个新 Pod 补上 → 现状回到 3 → 循环稳定。你手动 kubectl delete pod 删一个，它照样长回来——不是 bug，是控制器在忠实履职。
解决了什么
把「维持稳态」这件需要人 7×24 盯着的事，变成了系统的自动行为，这就是自愈的本质。
05

工作负载对象：从 Pod 到 Deployment

你声明的「积木」，控制器负责搭起来

K8s 用一层层对象把「跑一个容器」抽象成「跑一个高可用、可扩缩、可滚动升级的服务」。理解它们的包含关系是关键。

名词解释 · Pod

K8s 最小调度单元。一个 Pod 内可含一个或多个共享网络与存储的容器，它们同生共死、同机部署（见 §7 的「Pod 内部通信」）。Pod 是临时的（ephemeral）——会被销毁重建，IP 会变，所以永远不要直连 Pod IP。

对象作用典型场景
Pod最小运行单元基本不直接创建，由上层控制器管理
ReplicaSet保证 N 个 Pod 副本存活由 Deployment 自动管理，很少手写
Deployment无状态服务 + 滚动升级/回滚Web/API/微服务（最常用）
StatefulSet有稳定网络标识+持久存储的有状态服务数据库、Kafka、有状态中间件
DaemonSet每个节点各跑一个日志采集、监控 agent、CNI（见 §12）
Job / CronJob一次性/定时批处理数据迁移、定时任务

最常见的 Deployment YAML——注意它是纯声明式的「我要什么」：

# deployment.yaml —— 声明「我要 3 个 web 副本」
apiVersion: apps/v1
kind: Deployment
metadata: { name: web }
spec:
  replicas: 3                 # 期望副本数：控制器盯着它
  selector:
    matchLabels: { app: web } # 用 label 关联它管的 Pod
  template:                   # Pod 模板
    metadata:
      labels: { app: web }
    spec:
      containers:
      - name: web
        image: myregistry/web:1.4.2   # 不可变：升级 = 换 tag
        ports: [ { containerPort: 8080 } ]
        resources:                     # PM 关键：见 §11
          requests: { cpu: "250m", memory: "256Mi" }
          limits:   { cpu: "500m", memory: "512Mi" }

包含关系：Deployment → 管理 → ReplicaSet → 管理 → 一堆 Pod。升级镜像时，Deployment 会新建一个 ReplicaSet 并滚动把流量从旧 RS 迁到新 RS（见 §11 SWM）。

深入理解 · 为什么这样设计
基础为什么要有 Pod 这一层？直接调度「容器」不行吗？
是什么
Pod 是「一个或多个总要一起部署的容器」的打包单位，是 K8s 调度的最小单位（调度的是 Pod，不是单个容器）。
为什么
现实里常有几个进程必须同机、共享网络与磁盘才能协作：主业务容器 + 一个负责收集它日志的 Sidecar，或 + 一个做流量代理的 Envoy（见 §10）。如果调度单位是「单容器」，就无法保证它们被放到同一台机、共享 localhost。Pod 这层抽象正是为「紧耦合的一组容器」而生。
解决了什么
让「协作型多容器」有了统一的生命周期（同生共死）、统一的网络与存储命名空间（§7 会讲）。绝大多数 Pod 其实只有一个容器，多容器是为 Sidecar 模式准备的能力。
进阶Deployment / ReplicaSet / Pod 为什么要分成三层？
是什么
三层各管一件事：Pod 管「跑起来」、ReplicaSet 管「保持 N 个副本」、Deployment 管「版本与滚动升级/回滚」。
为什么
滚动升级的本质是「同时存在新旧两个版本，逐步把副本从旧换到新」。要实现这一点，就需要两个 ReplicaSet（旧版一个、新版一个），Deployment 在它们之上做「旧的缩容、新的扩容」的编排。职责一分层，回滚就只是「把流量切回旧 ReplicaSet」，非常干净。
解决了什么
用「单一职责 + 分层组合」实现了零停机发布与一键回滚——这正是不可变基础设施在应用层的落地。
基础label 和 selector 是什么？为什么不用「名字」直接关联？
是什么
label（标签）是贴在对象上的键值对（如 app: web）；selector（选择器）是「筛选条件」（如 app=web）。Service/ReplicaSet 用 selector 圈定「我要管哪些 Pod」。
为什么
Pod 是临时的、会不断被创建销毁、名字（含随机后缀）时刻在变。如果用「名字」硬绑，Pod 一重建关联就断了。用 label 这种松耦合的「标签匹配」，无论底下 Pod 换了多少茬，只要贴着 app=web 就自动被选中。
解决了什么
把「谁提供服务」和「具体是哪几个 Pod」解耦，是 Service 能对着一群动态变化的 Pod 稳定工作的根基（见 §7、§8）。
06

安全：身份、通信加密与最小权限

api-server ⇄ kubelet TLS · ServiceAccount · Secret · RBAC

K8s 安全的核心是回答三个问题：你是谁（认证 AuthN）、你能做什么（鉴权 AuthZ）、通信是否被窃听/篡改（传输加密）。

① 组件间 TLS：api-server ⇄ kubelet 双向认证

控制平面与节点之间的每一跳都走 mTLS（双向 TLS）。集群内置一套 CA，为每个组件签发证书：

  • api-server → kubelet：api-server 用客户端证书向 kubelet 证明身份（用于 kubectl logs/exec 等），kubelet 也校验 api-server；否则任何人都能连 kubelet 执行命令。
  • kubelet → api-server：kubelet 持节点证书上报状态，api-server 校验它只能改自己节点的对象（Node Authorizer + NodeRestriction）。
  • etcd：只允许 api-server 用证书访问，且数据可开启静态加密（encryption at rest）。
生产必须开启 kubelet 的认证鉴权（--authorization-mode=Webhook）与 api-server 到 kubelet 的证书校验。历史上大量入侵源于「匿名可访问的 kubelet 10250 端口」。

② ServiceAccount：Pod 的「工牌」

名词解释 · ServiceAccount（SA）

给Pod 内进程用的身份（区别于给「人」用的 User）。Pod 启动时会自动挂载一个 SA 的 JWT Token（投射卷，短期自动轮换），进程用它调 api-server 时表明「我是哪个服务」。

# 给 Pod 指定专属身份，而不是用 default SA
apiVersion: v1
kind: ServiceAccount
metadata: { name: web-sa, namespace: prod }
---
kind: Deployment
spec:
  template:
    spec:
      serviceAccountName: web-sa            # Pod 用这个身份
      automountServiceAccountToken: true    # 不需要就设 false，减小攻击面

③ Secret：敏感数据的载体

数据库密码、TLS 私钥、API Key 不能写死在镜像或 ConfigMap（见 §11 CM）里，用 Secret 承载，以卷或环境变量注入 Pod。

  • Secret 默认只是 base64 编码（不是加密！），必须配合 etcd 静态加密 + RBAC 限制读取。
  • 生产更推荐 外部密钥管理：External Secrets Operator / Vault / 云 KMS + CSI Secret Store，实现集中管理与轮换。

④ RBAC：最小权限鉴权

名词解释 · RBAC

Role-Based Access Control，基于角色的访问控制。Role（命名空间内）/ClusterRole（集群级）定义「能对哪些资源做哪些动作」，再用 RoleBinding/ClusterRoleBinding 把角色绑给某个 User / Group / ServiceAccount。

# 只允许读 prod 命名空间的 Pod —— 最小权限
kind: Role
metadata: { namespace: prod, name: pod-reader }
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get","list","watch"]     # 没有 create/delete
---
kind: RoleBinding
metadata: { namespace: prod, name: bind-web }
subjects: [ { kind: ServiceAccount, name: web-sa, namespace: prod } ]
roleRef: { kind: Role, name: pod-reader, apiGroup: rbac.authorization.k8s.io }
🔐 安全纵深清单（Defense in Depth）
  • 身份：每个服务独立 SA，禁用不必要的 token 自动挂载。
  • 鉴权：RBAC 最小权限，绝不给 cluster-admin 图省事。
  • 网络：默认全通 → 用 NetworkPolicy 做 Pod 间微隔离（零信任，见 §7）。
  • 运行时：非 root、只读根文件系统、drop capabilities、Pod Security Admission。
  • 供应链：镜像扫描 + 签名验签（Sigstore/cosign），准入控制器拦截高危镜像。
深入理解 · 为什么这样设计
进阶为什么组件间要用 mTLS，普通 HTTPS（单向 TLS）不够吗？
是什么
普通 TLS（你上网那种）只有客户端验证服务端身份；mTLS（双向 TLS）是双方互相验证对方的证书。
为什么
单向 TLS 能防窃听，但服务端不知道「连进来的到底是不是合法组件」。集群内部是机器对机器调用，没有「用户输密码」这一环，唯一能证明「我是合法的 kubelet / api-server」的就是证书。所以必须双向校验，否则任何能连到 kubelet 端口的人都能冒充 api-server 下发指令。
解决了什么
在零信任的网络里建立机器身份互信：既防窃听篡改，又防冒充。这也是 §10 服务网格自动给业务间通信加 mTLS 的同一思路。
基础ServiceAccount 和 User 有什么区别？为什么要分开？
是什么
User 是给「人」用的身份（你 kubectl 时用的），K8s 本身不存储 User，由外部（证书/OIDC）提供；ServiceAccount 是给「Pod 内程序」用的身份，是 K8s 里的一等对象。
为什么
人和程序的诉求完全不同：程序数量多、生命周期跟着 Pod、需要凭证自动挂载和自动轮换、且要能被 RBAC 精确授权到「某个服务只能读某类资源」。用统一身份既不安全也不好管，所以拆成两类。
解决了什么
让「每个服务有自己的最小权限身份」变得可行——某个 Pod 被攻破，攻击者拿到的也只是它那点权限，而不是管理员权限（权限爆炸半径最小化）。
基础既然 Secret「只是 base64、不是加密」，那用它还有什么意义？
是什么
Secret 是专门存放密码/密钥/证书的对象，默认存进 etcd 时只是 base64 编码（可轻易还原，≠ 加密）。
为什么
它的价值不在「编码」，而在于把敏感数据变成一个可被 RBAC 单独管控、可与镜像/代码解耦、可统一轮换的对象——你能精确控制「只有 X 服务能读这个 Secret」，还能开 etcd 静态加密、接 Vault/KMS。而写死在镜像或 ConfigMap 里则做不到这些。
解决了什么
把敏感配置从「散落在代码/镜像里」收敛成「集中、受控、可轮换」的资源。生产务必再叠加 etcd 加密 + 外部 KMS，才算真正安全。
基础为什么强调 RBAC「最小权限」，图省事全给 admin 会怎样？
是什么
RBAC 让你精确定义「谁（User/SA）能对哪些资源做哪些动作」。最小权限 = 只给完成工作所必需的那一点点权限。
为什么
权限一旦泄露（Token 被偷、Pod 被攻破），攻击者能造成的破坏 = 这个身份的权限。给了 cluster-admin，等于把整个集群的钥匙交出去，一处失守就全盘沦陷；只给「读 prod 的 Pod」，攻击者也就只能看看。
解决了什么
把安全事故的爆炸半径（blast radius）限制到最小，这是纵深防御最基础也最有效的一环。
07

服务通信：Pod 内 / Pod 间 / Service / DNS

从共享内存到集群域名，一层层「怎么找到对方」

① Pod 内部通信：共享网络与存储

同一个 Pod 里的多个容器共享同一个 network namespace 与 IPC namespace，因此它们：

  • 网络共享：共用一个 IP 与端口空间，彼此可用 localhost:port 直连（像同一台机器上的两个进程）。
  • 内存/IPC 共享：可通过共享内存（SHM）、信号量等 IPC 机制通信；还可挂同一个 emptyDir 卷共享文件。

这正是 Sidecar 模式的基础：业务容器 + 日志/代理 Sidecar 同 Pod，走 localhost 零拷贝协作（服务网格的 Envoy 就是这么注入的，见 §10）。

② Pod 间通信：扁平网络与 ClusterIP

K8s 的网络模型要求：每个 Pod 有独立 IP，任意 Pod 之间无需 NAT 即可直达（由 CNI 插件如 Calico/Cilium 实现）。但 Pod IP 是临时的——扩缩容/重启就变。所以直连 Pod IP 是反模式，要用 Service 提供的稳定虚拟 IP（ClusterIP）。

名词解释 · Service（ClusterIP）

一组 Pod 的稳定访问入口。Service 通过 label selector 选中一组 Pod，分配一个生命周期内不变的虚拟 IP（ClusterIP）与 DNS 名，把请求负载均衡到后端健康 Pod 上（转发机制见 §9）。

# Deployment（后端） + Service（稳定入口）
kind: Service
metadata: { name: order, namespace: prod }
spec:
  selector: { app: order }        # 选中所有 app=order 的 Pod
  ports:
  - port: 80                       # Service 暴露端口（ClusterIP:80）
    targetPort: 8080              # 转发到 Pod 的 8080

③ 服务域名：DNS 才是正确的调用方式

不要记 ClusterIP，直接用 DNS 名调用。集群内置 DNS（CoreDNS）为每个 Service 生成一条固定域名：

<service-name>.<namespace>.svc.cluster.local
  • 同命名空间：可简写为 http://order（DNS 搜索域自动补全）。
  • 跨命名空间：必须带命名空间，如 http://order.prod 或全写 order.prod.svc.cluster.local。
service-A 先向 CoreDNS 解析 service-B.prod.svc.cluster.local 得到 ClusterIP，连接后由 kube-proxy 均衡到某个健康的 B Pod
生活类比：打客服总机，而不是记员工手机号

Pod IP 像员工的私人手机号——人一离职（Pod 重建）号就废了。Service DNS 名像公司客服总机 400 电话：号码永远不变，你拨过去，总机（kube-proxy）自动把你转给当前在岗的任意一个客服（健康 Pod）。你从不需要知道具体是谁接的。

映射：私人手机=Pod IP · 400 总机=Service ClusterIP/DNS · 转接=负载均衡 · 在岗=就绪探针通过
零信任网络：默认所有 Pod 互通。生产用 NetworkPolicy（由 CNI 实施）声明「只有 service-A 能访问 service-B 的 8080」，把东西向流量收敛到白名单，这是 §6 安全的网络层延伸。
深入理解 · 为什么这样设计
基础为什么不能直接用 Pod IP 通信？
是什么
每个 Pod 确实有独立 IP，理论上能直连。但 Pod IP 是临时的——扩缩容、崩溃重启、被驱逐到别的节点，都会换新 IP。
为什么
你在代码里写死 10.1.2.3，那个 Pod 一重建 IP 就作废，你的调用全部失败；而且一个服务有多个副本，你也不知道该连哪一个、某个挂了怎么自动跳过。直连 Pod IP 等于和「随时会变、随时会死」的东西硬绑。
解决了什么
Service 给出一个生命周期内不变的稳定入口（ClusterIP + DNS 名），把「后面到底是哪些 Pod、谁健康」这件事对调用方彻底隐藏。
进阶ClusterIP 是「虚拟」IP，它到底存在于哪里？谁让它能通？
是什么
ClusterIP 不绑定在任何一张网卡上，你在任何机器上 ping 它、抓包都找不到一个「拥有」这个 IP 的实体。它是一个纯粹的逻辑地址。
为什么
让它「可用」的是每个节点上的 kube-proxy：它在内核里装了转发规则（iptables/IPVS，见 §9），意思是「凡是发往这个 ClusterIP 的包，就改写目标地址、随机送到某个健康的后端 Pod」。所以数据包其实从没到过一个叫 ClusterIP 的机器，而是在内核里被「翻译」并转走了。
解决了什么
用一个稳定的虚拟地址 + 内核级转发，实现了「稳定入口 + 负载均衡 + 自动剔除坏 Pod」，且对应用完全透明、几乎零性能损耗。
基础服务域名为什么要带 namespace（<svc>.<ns>.svc.cluster.local）？
是什么
namespace（命名空间）是集群内的逻辑分区，用来隔离不同团队/环境（如 dev / prod）。同名 Service 可以在不同 namespace 各存在一份。
为什么
正因为 prod 和 dev 里可能都有一个叫 order 的服务，光靠名字无法区分「你要哪个 order」。域名里带上 namespace 就唯一确定了目标。同命名空间内因 DNS 搜索域会自动补全，所以可简写为 order；跨命名空间必须写全。
解决了什么
多环境/多团队共用一个集群时的命名隔离与寻址唯一性，也让「同一套 YAML 在 dev/prod 用同一个服务名」成为可能。
08

服务发现：实例动态变化，怎么总能找到活的

Service → Endpoints/EndpointSlice → DNS 的联动

服务发现要解决：Pod 不断扩缩、重启、迁移，「有哪些实例、哪些是健康的」时刻在变，调用方怎么总能拿到一份最新的可用清单？K8s 把它做成了平台内置能力，无需业务集成注册中心 SDK。

①→你创建 Service（带 label selector）。
②→Endpoints 控制器持续 watch 匹配该 selector 的 Pod，把通过就绪探针（readinessProbe）的 Pod IP:Port 写进 EndpointSlice。
③→Pod 挂了/未就绪 → 自动从 EndpointSlice 移除；新 Pod 就绪 → 自动加入。清单永远是最新的健康集合。
④→CoreDNS 提供 Service 名→ClusterIP 的解析；kube-proxy 依据 EndpointSlice 把 ClusterIP 的流量转到某个健康 Pod（见 §9）。
模式机制对比传统注册中心
K8s 内置Service + EndpointSlice + CoreDNS + kube-proxy零侵入，业务只需用 DNS 名；平台负责健康剔除与均衡
客户端发现调用方拿实例列表自己做 LB（如 gRPC + headless Service）更细粒度 LB / 会话保持，但逻辑进业务侧
Consul/Nacos/Eureka服务注册到独立注册中心 + SDK 查询K8s 外/混合环境用；纯 K8s 内通常不必再引入
名词解释 · Headless Service

clusterIP: None 的 Service。DNS 解析不返回一个虚拟 IP，而是直接返回所有后端 Pod 的 IP 列表。用于需要感知每个实例的场景：StatefulSet（每个 Pod 有稳定 DNS 名 pod-0.svc...）、gRPC 客户端侧负载均衡、数据库集群。

就绪探针（readinessProbe）是服务发现的「开关」：探针没过，Pod 就不进 EndpointSlice，流量永远不会打到「还没准备好」的实例上——这是滚动升级零中断的关键（见 §11 FM）。
深入理解 · 为什么这样设计
进阶有了 CoreDNS，为什么还需要 EndpointSlice？
是什么
DNS 只回答「服务名 → ClusterIP」这一层（一个稳定入口）；EndpointSlice 回答的是「这个服务背后当前有哪些健康的 Pod IP」这一层，它随 Pod 增删/健康变化实时更新。
为什么
ClusterIP 只是个门牌号，真正把请求送到某个活着的 Pod，需要一份「实时在岗名单」。这份名单如果放进 DNS，会受 DNS 缓存 TTL 影响而更新不及时（Pod 都挂了客户端还在用旧 IP）。所以 K8s 把「稳定入口」交给 DNS、把「易变的实例清单」交给 EndpointSlice + kube-proxy，各司其职。
解决了什么
兼顾「入口稳定」与「后端实时」，避免了传统 DNS 负载均衡「实例挂了还被解析到」的老问题。EndpointSlice 相对老的 Endpoints 对象，还解决了超大服务下单对象过大的性能瓶颈。
基础readiness 探针和服务发现有什么关系？
是什么
readiness（就绪）探针周期性检查「这个 Pod 现在能不能对外提供服务」（比如它是否加载完配置、连上了数据库）。
为什么
一个 Pod「进程启动了」不等于「能干活了」——可能还在预热、还在建连接池。如果这时就把流量打过来，用户就会撞到错误。K8s 让 readiness 探针直接决定「是否把这个 Pod 加进 EndpointSlice」：没就绪 = 不进名单 = 收不到流量；就绪了才加入。
解决了什么
「优雅上线/优雅摘流量」：新 Pod 准备好了才接客，滚动升级时旧 Pod 先被摘出名单再关闭，从而做到零中断（见 §11）。
09

负载均衡与网关入口

集群内均衡（kube-proxy）+ 集群外入口（LoadBalancer / Ingress / Gateway）

① 集群内负载均衡：kube-proxy 怎么工作

ClusterIP 是个虚拟 IP，不存在于任何网卡上。真正让它「可访问」的是每个节点上的 kube-proxy——它把 EndpointSlice 翻译成内核转发规则：

模式原理规模表现
iptables为每个 Service 装一串 DNAT 规则，随机选一个后端规则线性增长，服务上万时更新/匹配变慢
IPVS内核 L4 负载均衡器（哈希表），支持 rr/lc/sh 等算法大规模首选，O(1) 查找，更多 LB 策略
eBPF（Cilium）绕过 kube-proxy，用 eBPF 在内核直接转发最高性能，可观测性强（见 eBPF）

② 集群外入口：四种 Service 类型 + Ingress

类型暴露范围用途
ClusterIP仅集群内默认，东西向服务间调用
NodePort每个节点的固定端口简单外部访问/自建 LB 前，端口范围有限
LoadBalancer云厂商外部 LB + 公网 IP对外 L4 入口（每个服务一个 LB，成本高）
ExternalNameCNAME 到外部域名把外部服务映射成集群内名字

③ 网关：Ingress 与 Gateway API

为每个服务开一个 LoadBalancer 又贵又难管。网关（Gateway）用一个入口 + 七层路由统一对外，把认证、TLS 终止、限流、灰度等横切关注点集中处理（呼应 微服务 的 API 网关思想）。

  • Ingress：K8s 早期的 L7 入口标准，按 Host/Path 路由到不同 Service，由 Ingress Controller（Nginx/Traefik/Envoy）实现。表达能力有限（主要 HTTP）。
  • Gateway API：新一代标准，角色分离（平台管 Gateway、应用团队管 HTTPRoute），原生支持流量拆分/灰度/多协议，正在取代 Ingress。
一个网关入口按 Path/Host 七层路由到不同 Service，Service 再经 kube-proxy 负载均衡到 Pod
两层负载均衡要分清：L7 网关做「哪个服务」的路由（按 URL/Host/Header），L4 kube-proxy做「服务内哪个 Pod」的均衡。云上通常还有一层外部 LB 把公网流量打到网关。
深入理解 · 为什么这样设计
基础LoadBalancer 也能对外，为什么还要 Ingress/网关？
是什么
LoadBalancer 类型每暴露一个服务，就向云厂商申请一个独立的外部负载均衡器 + 一个公网 IP（工作在 L4）。Ingress/网关则是一个入口按七层规则（域名/路径）路由到很多服务。
为什么
有 50 个对外服务，用 LoadBalancer 就是 50 个 LB、50 个公网 IP、50 份账单，且没法统一做 TLS 证书、认证、限流。网关把这些横切关注点集中在一个入口处理，按 api.x.com/order、api.x.com/user 分流到不同服务。
解决了什么
成本（一个入口 vs N 个 LB）、统一治理（TLS/认证/限流一处配置）、以及七层能力（按路径/域名/Header 路由，L4 的 LoadBalancer 做不到）。
进阶为什么需要「两层」负载均衡，一层不够吗？
是什么
第一层是网关（L7）：决定「这个请求属于哪个服务」；第二层是 kube-proxy（L4）：决定「这个服务的请求发给哪个 Pod」。
为什么
两层解决的是两个不同维度的问题，且工作层次不同。网关要理解 HTTP（看 URL/Header）才能路由到正确服务，代价较高；kube-proxy 只需在内核做 L4 转发，极快，负责把服务内的流量均摊到副本。硬把两件事塞一层，要么慢、要么不灵活。
解决了什么
「服务间寻址」与「服务内均衡」职责分离：网关专注七层路由与治理，kube-proxy 专注高性能的四层分发，各自可独立演进和优化。

流控与服务网格

限流 / 熔断 / 灰度 / mTLS —— 从入口治理到东西向治理

「流控」在云原生里有两层含义：入口流控（保护系统不被外部打垮）和服务间流控（治理东西向调用的可靠性与流量走向）。

① 入口与应用层流控

  • 限流（Rate Limiting）：网关按 IP/用户/API 令牌桶限速，超出返回 429，保护后端（算法与熔断三态详见 限流·熔断·降级·隔离）。
  • 熔断/重试/超时：下游异常时快速失败，防级联雪崩。
  • 并发与队列：结合 §11 的 HPA 弹性扩容一起削峰。

② 服务网格：把治理下沉到 Sidecar

名词解释 · Service Mesh

把重试、超时、熔断、mTLS、流量灰度、指标采集等治理逻辑，从各语言的业务 SDK 中下沉到与业务容器并列的 Sidecar 代理（Envoy），实现零侵入、跨语言统一的服务治理。代表：Istio、Linkerd、Cilium Mesh。

  • 数据面（Data Plane）：每个 Pod 注入一个 Envoy Sidecar（走 §7 的 Pod 内 localhost），拦截所有进出流量做治理。
  • 控制面（Control Plane）：istiod 统一下发路由/策略/证书。
  • 核心能力：自动 mTLS（东西向加密，接续 §6）、金丝雀/蓝绿/按比例流量拆分、全链路指标与追踪、故障注入。
流量灰度方式做法适用
金丝雀（Canary）先切 5% 流量到新版本，观察指标再逐步放量降低发布风险，最常用
蓝绿（Blue-Green）新旧两套并存，一次性切换，可秒级回滚要求可快速回退、资源翻倍
按标记（A/B）按 Header/用户特征把特定流量导向新版本特性灰度、精准验证
Service Mesh 不是免费的：每个 Pod 多一跳延迟、多一份内存/CPU、多一层运维复杂度。中小规模用网关 + 语言库通常够了；服务数量大、多语言、需精细流量治理与统一 mTLS 时才上 Mesh。
深入理解 · 为什么这样设计
进阶Sidecar 为什么能做到「零侵入」治理？
是什么
Sidecar 是和业务容器同 Pod 的一个代理容器（Envoy）。注入后，Pod 内所有进出网络流量会被透明地劫持先经过它。
为什么
因为同 Pod 共享网络命名空间（§7），可以用 iptables/eBPF 把流量重定向到 Sidecar，业务代码完全不用改一行，甚至不知道自己被代理了。重试、超时、熔断、mTLS 加解密、指标采集全在 Sidecar 里完成——这些逻辑就从「每种语言各写一遍 SDK」变成了「统一由代理做」。
解决了什么
治理逻辑与业务代码解耦、跨语言统一。以前 Java/Go/Python 服务要各自实现一套一致的熔断重试，现在下沉到 Sidecar 一处，升级治理策略不用改业务、不用重新发布。代价是每 Pod 多一跳与资源开销。
基础到底什么时候该上 Service Mesh？
判断信号
服务数量多（几十上百）、多语言、需要统一 mTLS、需要精细流量治理（按比例灰度、故障注入、全链路指标）——满足多条时收益才盖过成本。
别过早上
只有十来个服务、单一语言，用网关 + 一个成熟的语言库（如 Resilience4j / gRPC 内置重试）通常就够了。盲目上 Mesh 会引入不必要的延迟、资源和运维复杂度。
本质
Mesh 是「用运维复杂度换治理统一性」的交易，规模不到临界点时这笔交易不划算。这与 微服务 里「Monolith First」是同一种克制。

云原生 O&M：FCAPS 映射（CM / PM / FM / SWM）

用电信网管的经典模型，理解 K8s 的运维闭环

电信网管有一套成熟的运维分类 FCAPS（Fault / Configuration / Accounting / Performance / Security），电信云原生（CNF：容器化网络功能）常在此之上再加 SWM（软件管理）。这套语言恰好能把 K8s 的运维能力串成一个闭环——下面逐一映射到平台机制。

名词解释 · CM / PM / FM / SWM

CM Configuration Management 配置管理 · PM Performance Management 性能管理 · FM Fault Management 故障管理 · SWM Software Management 软件（生命周期）管理。源自电信网管 FCAPS，是「运营一个系统」的标准维度划分。

CM · 配置管理

声明式配置 + 版本化

  • ConfigMap：非敏感配置（环境变量/配置文件），与镜像解耦。
  • Secret：敏感配置（见 §6），配 KMS/Vault。
  • CRD + Operator：把领域配置声明化（如「我要一个 3 节点 Kafka」），Operator 负责 reconcile。
  • GitOps（Argo CD/Flux）：配置全量入 Git，集群状态自动向 Git 收敛——不可变 + 可审计 + 可回滚。
PM · 性能管理

指标 + 弹性伸缩

  • 资源模型：requests（调度依据/保底）+ limits（上限/超限被限流或 OOMKill）。
  • Prometheus：拉取 CPU/内存/QPS/延迟等指标，配 Grafana 看板。
  • HPA：按指标自动横向扩缩副本；VPA 调 requests；Cluster Autoscaler 扩节点。
  • SLO/SLI + RED（Rate/Error/Duration）度量服务质量。
FM · 故障管理

探针 + 自愈 + 告警

  • 三种探针：liveness（挂了重启）、readiness（未就绪摘流量，见 §8）、startup（慢启动保护）。
  • 自愈：Pod 崩溃自动重启、节点故障 Pod 重新调度（控制循环兜底）。
  • Events + Alertmanager：异常事件→告警路由到值班（对应网管的 FM 告警）。
  • PDB（PodDisruptionBudget）：限制自愿中断，保证运维/升级时最小可用副本。
SWM · 软件管理

发布 + 生命周期

  • 滚动升级/回滚：kubectl rollout 逐批换 Pod，maxSurge/maxUnavailable 控节奏，一键回滚。
  • Helm：应用打包与版本化发布（Chart + values）。
  • Operator：把有状态应用的安装/升级/备份/扩容做成自动化控制器。
  • 配合 §10 金丝雀/蓝绿降低发布风险。

滚动升级正是「不可变基础设施 + 声明式」的合体——你只改镜像 tag，控制器负责安全地把现状迁到期望：

# SWM：滚动升级与回滚
kubectl set image deploy/web web=web:1.4.3   # 改期望镜像
kubectl rollout status deploy/web            # 观察滚动进度
kubectl rollout undo   deploy/web            # 出问题一键回滚
FCAPS 视角的价值：它逼你在上线前就想清楚每个维度——配置怎么下发（CM）、指标怎么看/怎么扩（PM）、坏了怎么发现自愈告警（FM）、怎么安全发布回滚（SWM）。缺任何一项，系统在生产环境都会「裸奔」。
深入理解 · 为什么这样设计
基础requests 和 limits 有什么区别？超了会怎样？
是什么
requests = 这个容器「保底需要」的资源，调度器据此决定把它放到哪台机（保证机器有这么多空闲）；limits = 「最多能用」的资源上限。
为什么
两者分开是为了兼顾调度公平与弹性利用：requests 保证每个容器都拿得到承诺的资源（不会被别人饿死），limits 允许它在空闲时短时冲高，同时防止某个容器失控吃光整机。
超限后果
CPU 超 limit：不会被杀，只是被限流（throttle）变慢；内存超 limit：内存不可压缩，会被内核 OOMKill（直接杀掉容器）然后重启。所以内存 limit 设太小会导致反复 OOM 重启。
进阶liveness 和 readiness 探针到底差在哪？（最常见的混淆）
liveness（存活）
回答「这个容器是不是死了/卡死了」。失败 → K8s 重启这个容器。用于自我修复卡死进程。
readiness（就绪）
回答「现在能不能接流量」。失败 → 把 Pod 从 Service 名单摘掉（不重启），等它好了再放回来。
为什么要分开
「暂时不能接客」和「已经死了」是两码事。数据库短暂抖动时，Pod 只是暂时 not ready（该摘流量，别打过来），但它没死、不该被重启——重启反而更慢恢复。混用会导致：本该只摘流量的抖动，被 liveness 当成「死了」不停重启，越滚越糟。
常见坑
liveness 探针别去查「依赖的下游是否可用」——下游一挂，你所有 Pod 的 liveness 全失败被重启，把局部故障放大成集群雪崩。依赖检查放 readiness。
深入CRD 和 Operator 是什么？解决了什么？
是什么
CRD（自定义资源定义）让你给 K8s 新增一种自己的对象类型（如 kind: KafkaCluster）；Operator 是配套的自定义控制器，为这种新对象实现 §4 的控制循环。
为什么
K8s 内置对象（Deployment 等）只懂「无状态服务」。但装一个 Kafka 集群涉及「按序启动、选主、扩容时搬数据、故障时恢复」这些领域专属运维知识，内置对象表达不了。CRD + Operator 让你把「一个资深 DBA/运维的操作手册」编码成软件，用声明式的方式管理复杂有状态应用。
解决了什么
把「专家才会的复杂运维」自动化、声明化：你只写 KafkaCluster: {replicas: 3}，Operator 像那位专家一样，帮你完成安装、升级、扩缩、备份、故障恢复。这是 K8s 可扩展性的精华，也是电信 CNF 大量采用的模式。

日志与可观测性

Metrics / Logs / Traces 三支柱，日志是排障的最后一公里

可观测性三支柱：Metrics（指标，§11 PM）看趋势、Traces（链路）定位是哪一跳慢、Logs（日志）看具体发生了什么。三者用统一的 trace_id 串起来，才能在跨十几个服务的请求里快速定位问题。

K8s 日志采集范式

  • 容器只写 stdout/stderr：不要在容器内写日志文件（违反不可变原则，Pod 一销毁就丢）。运行时把 stdout 落到节点。
  • 节点级 DaemonSet 采集：每个节点跑一个 Fluent Bit / Vector（DaemonSet，见 §5），采集所有容器日志，附加 Pod/Namespace/Label 元数据后转发。
  • 集中存储与检索：汇聚到 Elasticsearch（EFK）或 Loki（PLG 栈，成本更低），Grafana/Kibana 统一查询。
应用只管把日志打到 stdout，采集/富化/存储/查询由平台的日志流水线负责
支柱回答的问题典型工具
Metrics系统整体健康吗？要不要扩容？Prometheus + Grafana
Traces这个慢请求卡在哪一跳？OpenTelemetry + Jaeger/Tempo
Logs那一刻究竟发生了什么？Fluent Bit + Loki/ES

更完整的三支柱工程实践见 可观测性工程 与 eBPF 观测。

深入理解 · 为什么这样设计
基础为什么要求容器只写 stdout，不能自己写日志文件？
是什么
约定：应用把日志直接打到标准输出/错误（stdout/stderr），由平台负责收集，而不是往容器内的 /var/log/app.log 写文件。
为什么
容器是临时且不可变的（§3）——Pod 一销毁，容器内的文件全没了，你恰恰在排查它为什么挂时却丢了它的日志。而且容器可能跑在任意节点、随时被重建，「进容器捞文件」既不现实也不可扩展。让日志走 stdout，运行时统一接管、由节点级采集器（DaemonSet）转走，才符合云原生的运行方式。
解决了什么
日志与容器生命周期解耦：容器随便销毁重建，日志早已被采集汇聚到集中存储，永不丢失，且能跨所有实例统一检索。这是「不可变基础设施」在日志层面的必然要求。
进阶为什么要 Metrics / Traces / Logs 三样都要，一样不够吗？
各司其职
Metrics 告诉你「出事了没 / 要不要扩容」（趋势、告警，成本低可长期存）；Traces 告诉你「这个慢请求卡在跨服务链路的哪一跳」；Logs 告诉你「那一刻那一跳具体发生了什么」（最细，成本高）。
为什么
它们是「由粗到细」的排障漏斗：先用 Metrics 发现异常 → 用 Trace 定位到具体服务/调用 → 用该请求的 trace_id 去 Log 里看细节。少了任何一层，你要么发现不了问题，要么发现了却定位不到，要么定位到了看不清原因。
解决了什么
用三者 + 统一 trace_id 关联，把「一个请求穿越十几个微服务」的黑盒，变成可逐层下钻的白盒，这就是可观测性（Observability）的核心价值。

调试与故障排查

在「声明式 + 临时 Pod」的世界里，怎么定位问题

K8s 里调试和传统「登上服务器 tail 日志」很不一样：Pod 是临时的、可能已经重启、也可能根本没起来。排障要靠一套「看状态 → 看事件 → 看日志 → 进容器」的组合拳。

① 排障四板斧：先问 K8s「发生了什么」

命令回答的问题什么时候用
kubectl get pod -o widePod 现在是什么状态、在哪个节点第一步，看整体状态与 STATUS 列
kubectl describe pod <p>调度、拉镜像、探针、Events 全过程Pod 起不来、Pending、重启时首选
kubectl logs <p> [-c <容器>]容器打到 stdout 的日志进程起来了但行为异常

kubectl exec -it <p> -- sh进容器里看文件/连通性怀疑配置/网络问题
kubectl get events --sort-by=.lastTimestamp命名空间内按时间排序的事件流看「刚刚集群里发生了什么」
口诀：Pod 起不来先 describe 看 Events，起来了行为怪先 logs，崩溃循环加 --previous。90% 的问题在 Events 和日志里就能看出根因。

③ 进阶手段

🩺 临时调试容器 Ephemeral Container

生产镜像常是无 shell 的精简镜像（distroless），没法 exec 进去。用 kubectl debug -it <pod> --image=busybox --target=<容器> 往运行中的 Pod 临时注入一个带工具的容器，共享目标的进程/网络命名空间来诊断，且不改动原 Pod。

🔌 端口转发 port-forward

kubectl port-forward svc/web 8080:80 把集群内服务映射到本地端口，绕过网关直接连后端，快速验证「是服务本身的问题还是入口/网络的问题」。

🧪 节点级调试 node debug

kubectl debug node/<node> -it --image=... 起一个挂载了节点根文件系统的 Pod，排查节点层面问题（磁盘、内核、kubelet）。

📋 声明式先行：dry-run 与 diff

kubectl apply --dry-run=server 和 kubectl diff 在真正下发前，预览「这次变更会改什么」，把很多问题挡在应用之前。

深入理解 · 为什么这样设计
基础为什么不能像以前那样「SSH 上机器看日志」？
是什么
传统运维登录固定服务器 tail 文件；K8s 里你面对的是调度到任意节点、随时被销毁重建的临时 Pod。
为什么
你根本不知道 Pod 现在在哪台机、也可能它刚崩溃重建过（旧容器没了）。所以调试入口从「机器」上移到了「K8s API」——通过 kubectl logs/describe/exec 让平台帮你定位并连到正确的容器，这也是为什么日志必须走 stdout 被集中采集（§12）。
解决了什么
无论 Pod 漂到哪、重启过几次，你都有统一、稳定的方式拿到它的状态与历史，不依赖「知道它在哪台机」。
进阶CrashLoopBackOff 里的「BackOff」是什么意思？
是什么
容器崩溃后 K8s 会重启它，但重启间隔逐次翻倍（10s→20s→40s…最多 5 分钟），这个「越等越久再重试」的退避策略就叫 BackOff。
为什么
如果一崩就立刻无脑重启，遇到「必然失败」的 bug（如配置错）会疯狂重启、烧 CPU、刷爆日志。指数退避给系统喘息、也给你时间介入，是分布式重试的通用智慧。
解决了什么
在「尽量自愈」和「不因无效重试拖垮系统」之间取得平衡。看到它就知道：容器在反复崩，赶紧 logs --previous 查崩溃原因。

资源管理与受限环境

requests/limits · QoS · 配额 · 驱逐 · 边缘轻量 K8s

K8s 的核心职责之一就是在有限的机器资源上，公平又高效地塞下尽可能多的容器。理解它的资源模型，既是做性能/成本优化（§11 PM）的基础，也是在边缘、IoT、车载等受限环境用好 K8s 的前提。

① requests / limits 与超卖

回顾 §11：requests 是「保底 + 调度依据」，limits 是「上限」。两者可以不相等，这就带来「超卖（overcommit）」——节点上所有 Pod 的 limits 之和可以大于节点实际资源，因为大家不会同时打满。

资源可压缩?超 limit 时
CPU可压缩被限流（throttle）变慢，不会被杀
内存不可压缩触发 OOMKilled，容器被杀后重启
这就是为什么内存 limit 要留足余量：CPU 超了只是慢，内存超了直接死。生产常见事故就是内存 limit 拍脑袋设太小导致 CrashLoopBackOff(OOMKilled)。

② QoS 三级：资源紧张时先牺牲谁

当节点资源不足要驱逐（evict）Pod 腾空间时，K8s 按 QoS（服务质量）等级决定先赶谁走。QoS 由你怎么设 requests/limits 自动决定：

Guaranteed（最高）

每个容器 requests == limits（都设了且相等）。最后才被驱逐，适合核心服务/数据库。

Burstable（中）

设了 requests 但 < limits。资源紧张时次优先被驱逐。最常见的一档。

BestEffort（最低）

啥都没设。资源一紧张第一个被赶走，只适合可随时丢的任务。

想让关键服务「稳如老狗」，就把它设成 Guaranteed（requests=limits）；批处理/离线任务可以 BestEffort 去捡便宜资源。这就是用 QoS 表达「谁重要」。

③ 多租户资源治理：Quota 与 LimitRange

  • ResourceQuota：限制一个 namespace 总共能用多少 CPU/内存/Pod 数——防止某个团队/环境吃光整个集群。
  • LimitRange：给单个 Pod/容器设默认值与上下限——防止有人不写 requests（变 BestEffort 到处捣乱）或写个超大 limit。
  • PriorityClass：给 Pod 设优先级，高优先级 Pod 可抢占（preempt）低优先级 Pod 的资源。

④ 受限 / 边缘环境：轻量化 K8s

标准 K8s 控制平面（etcd + 多组件）对边缘盒子、IoT 网关、车载、单板这类内存/CPU 有限的设备太重。为此有一系列轻量发行版：

方案特点适用
K3s单二进制 <100MB，用 SQLite 替代 etcd，去掉冗余组件边缘/IoT/开发测试，Rancher 出品最主流
MicroK8sCanonical 出品，snap 一键装，模块化开关边缘、单机、CI
KubeEdge / OpenYurt云边协同：云端管控、边缘自治（断网仍运行）大规模边缘节点管理（电信 MEC 常用）
k0s零依赖单二进制，内置常用组件轻量生产、边缘
受限环境的取舍：轻量发行版通过换掉 etcd（用 SQLite）、砍掉高可用组件、精简默认插件换取低资源占用，代价是牺牲一部分规模与冗余能力。呼应 资源受限下的 C++ 的同一命题——在有限资源下做减法。
深入理解 · 为什么这样设计
基础为什么要区分 requests 和 limits，设成一样不行吗？
是什么
requests 决定「调度到哪、保底多少」，limits 决定「最多用多少」。设成相等 = Guaranteed，一定程度上更稳。
为什么允许不等
因为大多数服务平时用得少、偶尔冲高。若强制 requests=limits，就得按「峰值」预留资源，机器利用率极低、极浪费。允许 requests<limits 让节点能超卖：按平均值预留、峰值时借用空闲，集群塞得下更多服务、成本更低。
解决了什么
在「稳定性」和「资源利用率/成本」之间给你一个可调的旋钮：核心服务设成相等求稳，普通服务留弹性求省。
进阶节点资源不够时，K8s 凭什么决定驱逐哪个 Pod？
是什么
节点内存/磁盘告急时，kubelet 会主动驱逐一些 Pod 来自保。顺序主要看 QoS 等级和「超用 requests 的程度」。
为什么
BestEffort（没设 requests）最先被赶——因为它没做任何资源承诺，系统对它没有保障义务；Burstable 次之；Guaranteed（requests=limits）最后，因为它「用多少就申请了多少」，是最守规矩、最该被保护的。这是一套「谁做了承诺就保护谁」的公平规则。
解决了什么
让资源紧张时的取舍可预测、可控制：你通过设置 requests/limits，就等于提前声明了自己 Pod 的「被牺牲优先级」。

关键 take-aways

🎯 一页速记
  • 云原生四大原则：容器化 · 微服务 · 不可变基础设施 · 声明式 API——K8s 是它们的落地平台。
  • K8s 内核是声明式 + 控制循环：你声明期望状态存 etcd，Controller 持续 reconcile 把现状拉向期望（删了会自愈）。
  • 安全：组件间 mTLS（api-server⇄kubelet）、Pod 身份用 ServiceAccount、敏感数据用 Secret（配 KMS）、权限用 RBAC 最小化，网络用 NetworkPolicy 微隔离。
  • 服务通信：Pod 内走 localhost/共享内存；Pod 间别连 IP，用 Service ClusterIP + DNS 名 <svc>.<ns>.svc.cluster.local。
  • 服务发现：Service → EndpointSlice → CoreDNS/kube-proxy 平台内置，就绪探针决定是否进转发清单。
  • 负载均衡：集群内 kube-proxy（iptables/IPVS/eBPF）；对外用 Ingress/Gateway API 做 L7 网关统一入口。
  • 流控：入口限流熔断 + 按需 Service Mesh（Sidecar 做 mTLS/灰度/治理），别为小规模强上。
  • 运维闭环用 FCAPS 对齐：CM=ConfigMap/CRD/GitOps · PM=Prometheus/HPA · FM=探针/自愈/告警 · SWM=滚动升级/Helm/Operator。
  • 日志：容器只写 stdout，DaemonSet 采集汇聚，与 Metrics/Traces 用 trace_id 串成可观测性三支柱。
  • 调试：Pod 起不来先 describe 看 Events，行为怪先 logs，崩溃循环加 --previous；精简镜像用 kubectl debug。
  • 资源：requests 保底/调度、limits 上限；CPU 超只限流、内存超被 OOMKilled；用 QoS 表达「谁重要」，Quota/LimitRange 做多租户治理；受限/边缘用 K3s/KubeEdge 轻量化。
