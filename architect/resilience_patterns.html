01

为什么会雪崩

局部故障如何演变成全站崩溃

分布式系统里，一个慢服务能拖垮整条链路：A 调 B，B 变慢 → A 的线程/连接被 B 的请求占满等待 → A 也变慢 → 调 A 的 C 也被拖垮……故障沿调用链逐级放大，这就是级联失败（cascading failure）/ 雪崩。弹性设计的目标不是「永不失败」，而是让失败被控制在局部、快速失败、优雅退化。

生活类比：电路保险丝 + 泄洪闸

家里电流过大时，保险丝会先熔断，牺牲一路电保住整栋楼不着火（熔断）。水库水位过高时，泄洪闸主动放掉一部分水，避免大坝整体溃堤（限流 + 负载脱落）。系统的弹性组件就是这些「主动牺牲局部换取整体存活」的机制——宁可拒绝部分请求，也不要全盘崩溃。

映射：保险丝=熔断 · 泄洪=限流/脱落 · 分区断电=隔离 · 保住整栋楼=系统可用
🧩 四类核心手段
  • 限流：控制入口流量，超出容量的请求拒绝/排队——防被打垮。
  • 熔断：下游故障时快速失败、停止调用，给它恢复时间——防被拖垮。
  • 降级：核心不可用时返回兜底结果，保证基本可用——保体验。
  • 隔离：把资源分池，一个依赖出问题不波及其它——限爆炸半径。
02

限流：控制入口水位

令牌桶 / 漏桶 / 滑动窗口

限流保护系统不被超出其容量的流量打垮。四种经典算法：

算法原理特点
固定窗口计数每窗口计数，超阈值拒绝简单；临界点问题（窗口交界可 2 倍突发）
滑动窗口细分小窗口滑动统计平滑临界问题，精度高，开销略大
漏桶（Leaky Bucket）请求入桶，恒定速率流出强制平滑输出，不允许突发；多余请求溢出
令牌桶（Token Bucket）匀速产令牌，取到令牌才放行允许突发（桶里攒的令牌），最常用
漏桶 vs 令牌桶

漏桶像一个底部有小孔的水桶：不管你倒水多快，出水永远匀速——适合需要平滑限速（如保护下游第三方）。令牌桶像停车场匀速发号，攒着的号可以一次放一批车进——允许合理突发（如用户偶尔的操作峰值）。大多数业务限流用令牌桶，因为完全不允许突发的体验太差。

映射：漏桶=恒定输出 · 令牌桶=允许攒突发 · 令牌数=突发上限
🏗️ 单机 vs 分布式限流
  • 单机限流：本地令牌桶（如 Guava RateLimiter / Sentinel），无网络开销、无单点，但只管本实例。
  • 分布式限流：Redis + Lua 脚本原子计数，做全局配额；代价是每次请求一次 Redis 往返 + 单点风险。
  • 两级方案：本地粗粒度挡大流量 + 中心细粒度控全局配额，兼顾性能与准确。
  • 维度：可按 全局 / 接口 / 用户 / 租户 / IP 分层限流。

算法与集中式实现细节见面试题 分布式限流器设计。

03

熔断：给故障下游喘息

三态状态机

当下游持续失败/超时，继续调用只会浪费资源、拖慢自己、还妨碍下游恢复。熔断器像保险丝：错误率超阈值就「跳闸」，一段时间内快速失败不再真正调用，给下游恢复窗口。

🟢 Closed（关闭）

正常放行，统计失败率。失败率/慢调用超阈值 → 转 Open。

🔴 Open（打开）

直接快速失败（走降级），不调下游。等待冷却时间 → 转 Half-Open。

🟡 Half-Open（半开）

放少量探测请求：成功够多 → 回 Closed；仍失败 → 退回 Open。

🎛️ 关键参数与要点
  • 触发条件：错误率阈值（如 50%）+ 最小请求数（样本太少不熔断，避免误判）+ 慢调用比例。
  • 冷却时间：Open 持续多久后试探（如 5s）。太短下游没缓过来，太长影响可用。
  • 半开探测：只放几个请求验证，避免恢复瞬间又被打垮。
  • 熔断必须配降级：跳闸后返回什么？见下一节。Hystrix（已停更）/ Resilience4j / Sentinel 是常见实现。
04

降级：优雅退化

保住核心，牺牲非核心

降级是「系统压力大或依赖不可用时，主动降低服务质量以保证核心可用」。它是熔断跳闸后的兜底，也是大促预案的核心。

降级方式示例
返回默认值 / 缓存推荐服务挂了，返回热门榜兜底；实时价挂了用缓存价。
关闭非核心功能大促时关掉「猜你喜欢」「评论」，保住下单支付主链路。
异步化 / 延迟处理写操作先入队，稍后处理，返回「处理中」。
降低精度用近似统计（HyperLogLog）代替精确去重。
🎯 降级设计原则
  • 先分级：明确哪些是核心链路（下单/支付）绝不能降，哪些可降。
  • 自动 + 手动：熔断触发自动降级；大促用降级开关手动一键切。
  • 兜底要有意义：返回的默认值要业务可接受，别返回空导致前端崩。
  • 降级也要可观测 + 可恢复：降级期间打点告警，压力过后自动/手动恢复。
05

隔离：限制爆炸半径

舱壁模式（Bulkhead）

隔离把资源（线程池、连接、实例）分舱，让一个依赖的故障不耗尽全局资源、不波及其它依赖。

生活类比：轮船的水密隔舱

大船船体分成多个水密隔舱：即使一个舱破洞进水，水也被封在这个舱里，其它舱照常浮起，船不沉。系统给每个下游依赖分配独立的线程池/连接池：即使「支付服务」变慢占满它自己的池，「商品服务」的池不受影响，整体不被一个慢依赖拖垮。

映射：隔舱=独立资源池 · 一舱进水=一个依赖故障 · 船不沉=系统整体存活
🧱 隔离粒度
  • 线程池隔离：每个下游依赖一个线程池（Hystrix 经典做法），强隔离但线程开销大。
  • 信号量隔离：用计数器限并发，轻量无额外线程，但不能隔离超时（调用方线程仍会等）。
  • 实例/集群隔离：核心业务与非核心部署到不同集群，故障域彻底分开。
  • 租户隔离：大客户与普通客户资源分池，防单租户打爆全局（noisy neighbor）。
06

超时与重试：双刃剑

用不好会放大故障

超时是所有弹性手段的地基——没有超时，一个卡住的调用会永久占用资源。重试能提高成功率，但用不好会把下游打死（重试风暴）。

⏱️ 超时
  • 每个远程调用必设超时，且要合理（P99 + buffer，不是拍脑袋 30s）。
  • 超时预算传播：A(1s) 调 B，B 调 C 的超时不能大于剩余预算（deadline 沿链路递减，gRPC 原生支持）。
🔁 重试（谨慎）
  • 只重试幂等操作——非幂等写重试会重复下单/扣款（配合幂等键）。
  • 指数退避 + 抖动（jitter）：避免所有客户端同时重试形成尖峰（惊群）。
  • 限制重试次数 + 重试预算：全局限制重试占比（如 ≤10%），防重试风暴放大故障。
  • 只对可重试错误（超时、503）重试；4xx/业务错误不重试。
反模式：重试风暴

下游抖动一下 → 所有客户端立刻重试 → 请求量瞬间翻几倍 → 本来能自愈的下游被重试流量彻底打死 → 更多超时 → 更多重试……重试在故障时会火上浇油。所以要退避 + 抖动 + 预算，并配合熔断（跳闸后连重试都停）。

07

负载脱落与自适应限流

从静态阈值到动态自愈

静态限流阈值很难设准：设低了浪费容量，设高了保护不住。现代做法是自适应——根据系统实时状态动态调节。

  • 负载脱落（Load Shedding）：当 CPU / 队列积压 / RT 超阈值时，主动丢弃部分请求（优先丢低优先级），保住系统不崩。Google SRE 的核心实践。
  • 自适应限流：以延迟/成功率为反馈信号动态调整放行量（类似 TCP 拥塞控制 / Netflix concurrency-limits / Sentinel BBR），无需人工调阈值。
  • 优先级 + 公平：过载时按请求优先级脱落（保交易、丢日志上报）；防单用户独占（加权公平）。
📊 前提：可观测

所有弹性策略都依赖度量：QPS、RT（P99）、错误率、饱和度（CPU/队列）。没有监控就没有自适应，也无法验证策略是否生效。参见 可观测性工程。

08

组合成一条防线

它们不是替代，而是叠加

这些手段各管一段，实际系统把它们串成纵深防御：

🛡️ 一次请求的防护链
  • ① 入口限流（网关）：挡住超容量流量，返回 429。
  • ② 隔离：请求进入该依赖的独立资源池，不影响其它。
  • ③ 超时：调用下游设合理 deadline，不无限等。
  • ④ 熔断：下游持续失败则跳闸，快速失败。
  • ⑤ 重试（幂等 + 退避 + 预算）：处理瞬时抖动。
  • ⑥ 降级：以上都失败时返回兜底结果。
  • ⑦ 负载脱落：系统整体过载时主动丢低优先级请求。

记忆口诀：限流挡在门外，隔离分而治之，超时不无限等，熔断快速失败，重试小心翼翼，降级兜底保命，脱落断尾求生。

09

关键 take-aways

🎯 一页速记
  • 弹性的目标不是「不失败」，而是把失败控制在局部、快速失败、优雅退化。
  • 限流：令牌桶允许突发（最常用）；分布式用 Redis+Lua 或两级方案。
  • 熔断：Closed → Open → Half-Open 三态，配错误率阈值 + 冷却 + 探测，必配降级。
  • 降级：先分级，核心不降；自动（熔断触发）+ 手动（大促开关）。
  • 隔离：舱壁模式限爆炸半径，线程池/信号量/集群/租户隔离。
  • 超时必设、重试谨慎（幂等 + 退避抖动 + 预算），否则重试风暴放大故障。
  • 进阶：负载脱落 + 自适应限流，依赖可观测做动态自愈。
