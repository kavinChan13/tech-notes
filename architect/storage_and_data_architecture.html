存储引擎选型地图

读写特征分类 · 存储引擎全景选型矩阵 · 选型决策关键问题

没有最好的存储，只有最适合工作负载的存储。本章先给出选型全景，帮你快速定位；后续各章深入每种的内部原理。

1.1 读写特征分类

OLTP（Online Transaction Processing）

特征：高并发小事务；读写混合；强 ACID；点查询为主；延迟敏感（毫秒级）

典型工作：下单、支付、登录、库存变更

存储选型：PostgreSQL、MySQL（InnoDB）、CockroachDB

OLAP（Online Analytical Processing）

特征：低并发大查询；以读为主；聚合扫描；列存友好；吞吐敏感

典型工作：报表、BI、全量分析、机器学习特征

存储选型：ClickHouse、DuckDB、BigQuery、Snowflake（云仓）

Key-Value

特征：极简读写接口；延迟极低；可大规模扩展

存储选型：Redis（内存）、DynamoDB（云）、RocksDB（嵌入式）、TiKV

时序（Time-Series）

特征：按时间追加写；读以时间范围查询为主；高压缩比

存储选型：InfluxDB、VictoriaMetrics、TimescaleDB、QuestDB

B+ Tree：InnoDB 存储引擎内部

B+ Tree 结构 · InnoDB 的聚簇索引 · 页（Page）：InnoDB 的 IO 单元 · InnoDB 的三大关键文件

B+ Tree 是关系数据库的基石。理解它的内部结构是"为什么这个查询会走索引 / 为什么这个查询走全表"等问题的答案。

2.1 B+ Tree 结构

B+ Tree vs B-Tree 的关键区别

  • B+ Tree：只有叶节点存数据，内部节点只存键（索引）；叶节点间有链表。范围查询非常高效（找到起点后顺链表扫）。
  • B-Tree：每个节点都存数据。读单个值可能更快（无需遍历到叶），但范围查询要回溯。数据库几乎都用 B+ Tree。

2.2 InnoDB 的聚簇索引

聚簇索引：叶节点直接存行数据（不是指针）。InnoDB 的主键就是聚簇索引，整张表按主键顺序存储。

🌳 聚簇索引（PRIMARY KEY）

pk_value
col1 | col2 | col3 | …完整行数据

SELECT * WHERE id = 42 → 1 次 B+Tree 查找即得完整行。

🔗 二级索引（普通索引）

index_value
primary_key只存主键

叶节点只存主键，取完整行还需回表。

🔁 回表查询（额外一次 IO）

WHERE email='x@y.com' → ① 二级索引找到
primary_key = 42 → ② 用 pk=42 回聚簇索引
取完整行

主键选型影响

❌ 随机主键（UUID v4）的代价

每次插入都是随机位置 → 页分裂、随机 IO、碎片化 → 性能差、空间浪费（50%+ 碎片率）

-- UUID v4 是随机的，每次插入位置随机
id = '9b2f3a7c-...'  -- 可能插入到任何叶页

✅ 单调递增主键的优势

每次插入都在末尾 → 顺序 IO、无页分裂 → 写入速度快、碎片少

-- AUTO_INCREMENT 或 UUID v7（时间前缀）
id = 1001, 1002, ...  -- 总是追加到末尾
-- 或 UUID v7（毫秒时间戳前缀）

2.3 页（Page）：InnoDB 的 IO 单元

📦 页（Page）= IO 单元

InnoDB 默认页大小 16KB，所有 IO 都以页为单位——不是行，也不是字节。

  • 读一行 → 读入整个 16KB 页到 Buffer Pool
  • 写一行 → 修改 Buffer Pool 里的页（脏页），后台再刷盘

🧠 Buffer Pool = 内存缓冲区

  • 热数据缓存（类似 OS 的 Page Cache）
  • 通常设为物理内存的 60–80%（innodb_buffer_pool_size）
  • LRU 淘汰（有 Young / Old 分区优化）
Buffer Pool 命中率是 MySQL 最重要的性能指标之一： 
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool%';
命中率 = (buffer_pool_read_requests - buffer_pool_reads) / buffer_pool_read_requests 应该 > 99%。如果 < 95%，考虑增大 innodb_buffer_pool_size。

3.2 SSTable（Sorted String Table）

每个 SSTable 是一个不可变的有序文件：

Bloom Filter：加速点查

Bloom Filter 是一个概率数据结构：

  • 如果 Bloom Filter 说"不存在" → 一定不存在（False Negative 率为 0）
  • 如果 Bloom Filter 说"可能存在" → 还需要真正查数据块（可能有 False Positive）
  • False Positive 率通常控制在 1% 以内

效果：点查时先查 Bloom Filter，如果不存在直接跳过这个 SSTable，大幅减少无效 IO。

3.3 三大放大因子

LSM-Tree 的性能权衡用三个放大因子量化：

放大因子定义B+ TreeLSM-Tree
写放大（WA） 实际写 IO / 用户写数据量 中（索引维护） 高（Compaction 重复写）
读放大（RA） 实际读 IO / 用户读数据量 低（1-3 次 IO） 中（多层查找）
空间放大（SA） 实际存储 / 用户数据量 中（碎片） 中（旧版本暂存）
选型核心依据：
  • 写多读少（SSD 写吞吐优先） → LSM-Tree（RocksDB）。例：写性能 10x B+ Tree
  • 读多写少（随机读优先） → B+ Tree（InnoDB）。例：点查 3x 快于 LSM
  • 混合负载 → 看具体业务，OLTP 一般选 B+ Tree

3.4 Compaction 策略

策略特点适用
Leveled Compaction（默认） L1+ 层内文件按 key 排序无重叠；读放大低；写放大高 读密集型
Tiered Compaction 同层文件可重叠，满了整层 merge；写放大低；读放大高 写密集型
FIFO Compaction 按时间顺序，删旧数据；无写放大 时序 / TTL 场景

WAL 与崩溃恢复

为什么需要 WAL · WAL 的基本结构 · 崩溃恢复流程（ARIES 算法简化版） · Checkpoint 机制 · Double Write Buffer（InnoDB 特有）

Write-Ahead Log 是所有持久化存储引擎的核心机制——在修改数据之前，先把"我要做什么"写到日志，确保即使崩溃也能恢复。

4.1 为什么需要 WAL

存储引擎在写数据时面临的困境：

  • B+ Tree 的页更新可能涉及多个页（分裂、合并）
  • 这些操作不能原子地写入磁盘（磁盘 IO 的最小原子单位是扇区 512B，而一个页 16KB）
  • 写到一半崩溃 → 数据损坏

WAL 的解法：先把整个操作的意图顺序写入日志（顺序写速度极快），然后再修改数据页（随机写）。崩溃后重放日志即可恢复。

4.2 WAL 的基本结构

Type1B

  • LSN：单调递增，是整个系统的"时间戳"
  • 每个页记录 Page LSN（最后一次被哪个 WAL record 修改）

4.3 崩溃恢复流程（ARIES 算法简化版）

Phase 1 · Analysis → Phase 2 · Redo → Phase 3 · Undo

Analysis（分析）

  • 从上一个 Checkpoint 开始扫描 WAL
  • 找出崩溃时哪些事务已提交、哪些未提交
  • 找出哪些脏页可能没刷到磁盘

Redo（重做）

  • 从 earliest_dirty_page_LSN 重放 WAL
  • 把所有操作（含未提交事务）重做一遍
  • 若 page_LSN >= WAL_record_LSN 则跳过（已刷盘）

Undo（回滚）

  • 回滚所有未提交事务
  • 读取 Undo Log，逆向操作

4.4 Checkpoint 机制

WAL 不能无限增长，需要定期 Checkpoint：

📌 Checkpoint 时做三件事

1 把所有脏页（修改过但还在 Buffer Pool 的页）刷盘
2 记录 Checkpoint LSN = 当前已刷新的最大 LSN
3 Checkpoint LSN 之前的 WAL 可以被覆盖重用 ♻️

🚀 崩溃恢复只从上次 Checkpoint 开始重放

WAL 起点可能是几个月前 ··· 上次 Checkpoint恢复起点 → 崩溃点

只需重放绿色区间，恢复时间大幅缩短。

⚖️ Checkpoint 频率的权衡

太频繁 ⏱️⏱️⏱️

  • 大量随机 IO（不断刷脏页）
  • 抢占磁盘带宽，影响业务性能

太少 ⏳

  • 崩溃恢复重放时间长
  • WAL 迟迟无法回收，占用磁盘多

4.5 Double Write Buffer（InnoDB 特有）

问题：WAL 的 redo log 是逻辑日志（"把页 N 的偏移量 X 改为 Y"），但如果写一个 16KB 页时崩溃（写了 8KB），页本身损坏了，redo 无法恢复。

Double Write：写页之前先把页顺序写到 Double Write Buffer（磁盘上的连续区域），确认后再写到真实位置。崩溃时用 DWB 里的备份恢复完整页，然后 redo log 正常重放。

MVCC 与隔离级别

MVCC 的基本思想 · InnoDB 的 MVCC 实现 · 四大隔离级别与读现象 · Undo Log 的两个用途

MVCC（Multi-Version Concurrency Control）是现代数据库实现高并发的核心技术——读不阻塞写，写不阻塞读。本章从 MVCC 原理到四大隔离级别，澄清常见误解。

5.1 MVCC 的基本思想

每次写操作不覆盖旧数据，而是创建新版本。读操作通过版本号找到自己该看的版本。

假设 id=1 的行被更新了 3 次，形成一条版本链（新 → 旧）：

v3 · "Alice III"tx_id=200 · 最新 → v2 · "Alice II"tx_id=150 → v1 · "Alice I"tx_id=100 · 最旧

👁️ 事务 trx_id=180 读取

规则：找最大 tx_id ≤ 180 的版本 → 命中 v2（tx_id=150）。

v3 是 tx_id=200 写的，比 180 大 → 不可见。

5.2 InnoDB 的 MVCC 实现

每行的隐藏列

InnoDB 每行都藏着两个隐藏列，正是 MVCC 的基石：

DB_TRX_ID 6B

最后修改这行的事务 ID。

DB_ROLL_PTR 7B

指向 Undo Log 中的上一个版本。

读操作沿 DB_ROLL_PTR 链逆向回溯，找到对自己可见的版本：

当前行 → Undo Log v2 → Undo Log v1

Read View（快照）

每个事务开始时（或每次 SELECT 时，取决于隔离级别），创建一个 Read View：
Read View = {
    m_ids: 创建时所有活跃事务的 ID 集合
    min_trx_id: 最小活跃事务 ID
    max_trx_id: 下一个将分配的事务 ID
}

判断某个版本是否可见：
if (trx_id < min_trx_id):  可见（已提交的旧事务）
elif (trx_id >= max_trx_id): 不可见（比我晚开始的事务）
elif (trx_id in m_ids): 不可见（创建时还活跃，未提交）
else: 可见（已提交）

5.3 四大隔离级别与读现象

隔离级别脏读不可重复读幻读实现
Read Uncommitted ✗ 有 ✗ 有 ✗ 有 直接读最新版本
Read Committed ✓ 无 ✗ 有 ✗ 有 每次 SELECT 创建新 Read View
Repeatable Read（InnoDB 默认） ✓ 无 ✓ 无 部分有 事务开始时创建 Read View，持续到结束
Serializable ✓ 无 ✓ 无 ✓ 无 加读锁（或 SSI），低并发
常见误解：RR 不能防幻读？ InnoDB 的 Repeatable Read 通过 Next-Key Lock（行锁 + Gap Lock）部分防止幻读，但不是完全防止：
  • 快照读（普通 SELECT）：MVCC 防幻读
  • 当前读（SELECT ... FOR UPDATE / UPDATE / DELETE）：需要 Next-Key Lock 才能防幻读
  • 如果混用快照读和当前读，仍然可能看到幻象
要完全防幻读，用 Serializable 或确保不混用读类型。

5.4 Undo Log 的两个用途

① 事务回滚

记录"如何撤销"这次修改：

  • INSERT → Undo: DELETE
  • DELETE → Undo: INSERT
  • UPDATE → Undo: UPDATE 回旧值

事务 ROLLBACK 时，按 Undo Log 逆向执行。

② MVCC 读历史版本

通过版本链（DB_ROLL_PTR）回溯到旧版本。

清理：当没有任何活跃事务需要看这个旧版本时，Purge 线程清理它。

如果有长事务存在，Undo Log 不能被清理 → Undo Log 暴增 → 磁盘占满。

长事务是 InnoDB 的公害：任何一个活跃事务存在，它能看到的所有历史版本就不能被清理。一个 10 小时不提交的事务 → Undo Log 积累 10 小时的所有写入版本 → 磁盘 / ibdata1 暴涨。监控 information_schema.innodb_trx，禁止超过 60 秒的事务。

索引设计深入

索引选择性（Selectivity） · 联合索引的最左前缀原则 · 覆盖索引（Covering Index） · 索引失效的常见场景 · 分页查询优化

6.1 索引选择性（Selectivity）

选择性 = 不重复值数 / 总行数。越接近 1，索引效果越好；接近 0（如 boolean 列），索引几乎无用。 
-- 查看选择性
SELECT COUNT(DISTINCT email) / COUNT(*) AS selectivity FROM users;
-- 结果 ≈ 1.0 → email 选择性极高，非常适合建索引

SELECT COUNT(DISTINCT status) / COUNT(*) AS selectivity FROM orders;
-- 结果 ≈ 0.003 → status 只有几个值，选择性极低，不适合单独建索引
-- 可以考虑和 user_id 组合成联合索引

6.2 联合索引的最左前缀原则

-- 假设有联合索引 INDEX(a, b, c)
-- 以下查询能用到索引（最左前缀）：
WHERE a = 1                    -- ✓ 用索引，匹配 a
WHERE a = 1 AND b = 2          -- ✓ 用索引，匹配 a,b
WHERE a = 1 AND b = 2 AND c=3  -- ✓ 用索引，匹配 a,b,c
WHERE a = 1 AND b > 2          -- ✓ 部分用索引：a 等值，b 范围

-- 以下查询不能充分用索引：
WHERE b = 2                    -- ✗ 跳过了 a，不能用
WHERE a = 1 AND c = 3          -- ✗ 跳过了 b，只能用 a 的部分
WHERE a > 1 AND b = 2          -- ✗ a 是范围，b 就不能用索引了

6.3 覆盖索引（Covering Index）

-- 所有需要的列都在索引里 → 不需要回表！
-- 极大减少 IO

-- 普通查询（需要回表）：
SELECT name FROM users WHERE email = 'alice@example.com';
-- 如果只有 email 的索引：
-- ① 二级索引找到 pk
-- ② 回聚簇索引找 name → 额外一次 IO

-- 覆盖索引（不回表）：
-- 改成 INDEX(email, name)
-- ① 二级索引叶节点里就有 name → 直接返回，无回表

多级缓存体系

缓存的五种模式 · 缓存一致性策略 · 进程内缓存（L1） · 多级缓存分层架构

缓存是性能优化最重要的手段。本章建立完整的多级缓存架构视图，统一按请求路径从外到内编号：L0 CDN/浏览器 → L1 进程内缓存 → L2 本地 Redis → L3 分布式 Redis → DB（回源）。并深入讨论各层的适用场景和一致性策略。

编号约定：L0–L3 指应用架构师可设计、可失效的缓存层，数字越小越靠近用户。CPU 的 L1/L2/L3 Cache 属于硬件层、由 CPU 自动管理，应用无法直接读写或失效，不在本章 L0–L3 编号内——下文凡写「L1」均指进程内缓存。

7.1 缓存的五种模式

Cache-Aside（旁路缓存，最常用）

// 读：先查缓存
result = cache.get(key)
if result is None:
    result = db.get(key)
    cache.set(key, result, TTL)
return result

// 写：更新 DB，删除缓存
db.update(key, value)
cache.delete(key)  // 删除而不是更新（防竞态）

优点：简单；应用层控制。缺点：缓存 miss 后 DB 压力大；有短暂不一致（删缓存前后）。

Write-Through（写穿）

// 写：同时写缓存和 DB（同步）
cache.set(key, value)
db.update(key, value)
return ok

// 读：直接读缓存

优点：读永远命中缓存。缺点：写延迟高（等两次写）；冷数据也占缓存。

Write-Behind / Write-Back（写回）

// 写：只写缓存，异步写 DB
cache.set(key, value)
async_queue.push(db_write(key, value))
return ok

优点：写极快。缺点：数据丢失风险（缓存宕机）；不一致窗口大。适用：点赞数、访问计数等可接受少量丢失的场景。

Read-Through

缓存层负责从 DB 填充自己（不是应用层），应用层只和缓存打交道。减少应用层重复代码，但缓存需要知道 DB 的访问方式。

Refresh-Ahead（预刷新）

// 热点 key 在 TTL 到期前主动异步刷新
result, meta = cache.get_with_meta(key)
if meta.ttl_left < REFRESH_THRESHOLD:  // 快过期了
    async_refresh(key)   // 后台异步重载，不阻塞本次读
return result       // 本次仍返回旧值（未过期）

优点：热点数据几乎永不 miss，避免过期瞬间大量请求击穿到 DB。缺点：需预测热点；非热点数据白白刷新浪费资源。适用：访问集中、可预测的热点（首页、榜单、热门商品）。

7.2 缓存一致性策略

❌ 先更新缓存，再更新 DB

A 写缓存 → A 挂了 → DB 未更新 → 缓存和 DB 不一致。绝对不要用。

❌ 先更新 DB，再更新缓存

A: DB→update; B: DB→read，cache→update; A: cache→update（A 覆盖了 B 的更新）→ 缓存存了旧值。高并发下有竞态，不推荐。

✅ 先更新 DB，再删除缓存（Cache-Aside）

最常用。最坏情况：删缓存失败 → 缓存里是旧值，直到 TTL 过期。

优化：删缓存失败时重试，或用消息队列保证最终删除（Canal + MQ）。

✅ 延迟双删

更新 DB 前后各删一次缓存，中间 sleep 一小段时间（给进行中的读操作完成的机会）。减少不一致窗口。

7.3 进程内缓存（L1）

// 应用层最快的可控缓存：本进程内存（硬件 CPU Cache 更快，但对应用透明）
// 适合：热点不变数据（配置、字典、用户权限）
// 不适合：需要多实例一致的数据

// C++ 进程内缓存实现（LRU + TTL）
template<typename K, typename V>
class LRUCache {
    size_t capacity_;
    std::list<std::pair<K,V>> list_;          // LRU 链表
    std::unordered_map<K, typename decltype(list_)::iterator> map_;
    std::chrono::seconds ttl_;

public:
    std::optional<V> get(const K& key) {
        auto it = map_.find(key);
        if (it == map_.end()) return std::nullopt;
        list_.splice(list_.begin(), list_, it->second);  // 移到头部
        return it->second->second;
    }
    void put(const K& k, V v) {
        if (map_.count(k)) { list_.erase(map_[k]); map_.erase(k); }
        list_.push_front({k, std::move(v)});
        map_[k] = list_.begin();
        if (list_.size() > capacity_) {
            map_.erase(list_.back().first);
            list_.pop_back();
        }
    }
};

7.4 多级缓存分层架构

把上面的模式落到真实系统里，一次请求按从用户到数据源的方向依次穿过各层（L0 → L1 → L2 → L3），每层未命中才回退（fallback）到下一层，最终才到 DB。编号越小越靠近用户，越大越靠近数据源。
用户请求 → L0 · CDN / 浏览器 → L1 · 进程内缓存 → L2 · 本地 Redis → L3 · 分布式 Redis → DB（回源）
层级典型实现延迟量级容量一致性 / 失效适用场景
L0 · CDN / 浏览器缓存 CDN 边缘节点、HTTP Cache-Control / ETag ~1–50 ms（就近） 海量（边缘分布） TTL + 主动刷新 / 版本化 URL（cache-busting） 静态资源、图片、JS/CSS、可缓存的 GET 接口
L1 · 进程内缓存 本地 LRU/Caffeine、C++ 内存 Map ~10–100 ns 最小（单机内存） 最难保证一致：多实例各存一份，靠短 TTL / 广播失效 配置、字典、权限等热点不变数据
L2 · 本地 Redis 同机房 / sidecar Redis ~0.1–1 ms 中（单机/主从内存） Cache-Aside + 删除；同机房低延迟 会话、单机房热点、二级缓存挡在分布式 Redis 前
L3 · 分布式 Redis Redis Cluster / 跨机房集群 ~1–5 ms 大（可水平扩展） 全局共享，一致性最好保证；DB 的第一道挡板 全局共享数据、跨实例一致的缓存
DB（回源） MySQL / PostgreSQL 等 ~5–50 ms+ 全量持久 唯一真相源（source of truth） 所有缓存未命中时的兜底

硬件层补充：进程内缓存（L1）之下还有 CPU L1/L2/L3 Cache，延迟约 1–10 ns，由硬件和 OS 自动管理，应用无法直接失效——性能调优时关注内存局部性即可，不纳入 L0–L3 架构设计。

命中率的乘法效应

以应用层 L1–L3 为例（L0 CDN 通常单独承担静态资源，此处略过）：假设 L1 命中 80%、L2 对剩余命中 80%、L3 对再剩余命中 90%，则真正打到 DB 的请求 ≈ (1-0.8)×(1-0.8)×(1-0.9) = 0.4%。每加一层，回源压力指数级下降——这正是多级缓存的意义。

多级失效的难点：一致性传播

写操作要让所有层的旧值失效。L3 分布式 Redis 好删，但 L1 进程内缓存散落在每个实例，无法直接删。常用方案：

  • L0（CDN）通过版本化 URL 或主动 purge 刷新；
  • L1 用很短的 TTL（秒级），容忍短暂不一致；
  • 用 L3 Redis Pub/Sub 或 MQ 广播失效消息，各实例收到后清 L1 本地缓存。

⚠️ 多级缓存的三大坑

  • 缓存穿透：查不存在的 key，每次都击穿到 DB。→ 缓存空值（含短 TTL）或布隆过滤器。
  • 缓存击穿：单个热点 key 过期瞬间，大量请求同时回源。→ 互斥锁重建 / 逻辑过期 + Refresh-Ahead。
  • 缓存雪崩：大量 key 同一时刻集中过期。→ TTL 加随机抖动、多级缓存分摊、限流降级。

缓存三大问题与对策

缓存穿透（Cache Penetration） · 缓存击穿（Cache Breakdown） · 缓存雪崩（Cache Avalanche） · 一致性哈希（Consistent Hashing）

9.1 缓存穿透（Cache Penetration）

查询的 key 在缓存和 DB 里都不存在（如 id=-1 的恶意查询），每次都打到 DB。

问题场景

攻击者用大量不存在的 key 发请求，绕过缓存直接打 DB → DB 被打垮。

解决方案

  1. 缓存空值：查 DB 结果为空时，缓存 "NULL"（短 TTL，如 60s）
  2. 布隆过滤器：启动时把所有合法 key 载入 Bloom Filter；请求先过 Filter，不在 Filter 里直接拒绝
  3. 参数校验：在接入层过滤明显非法参数（id < 0 直接拒绝）

9.2 缓存击穿（Cache Breakdown）

某个热点 key 的缓存刚好过期，同时大量请求涌入 → 全打到 DB。

问题场景

秒杀商品缓存在 12:00:00 整点过期，10 万 QPS 同时 miss → DB 被 10 万并发打死。

解决方案

  1. 互斥锁（Mutex）：cache miss 时先加分布式锁，只有获锁者去查 DB，其他人等待 / 返回旧值
  2. 逻辑过期：缓存不设真正的 TTL，而是在 value 里存"过期时间"；过期时异步更新，读到旧值也返回（牺牲一致性换可用性）
  3. 永不过期：热点 key 永久缓存，变更时主动推送更新

9.3 缓存雪崩（Cache Avalanche）

大量 key 在同一时间过期（或 Redis 整体宕机），所有请求同时打到 DB → DB 崩溃。

问题场景

每小时批量预热 10 万条数据，都设 TTL=3600s → 1 小时后同时过期 → DB 被 10 万并发打死。

解决方案

  1. 随机化 TTL：基础 TTL + 随机偏差（如 TTL = 3600 + random(0, 300)s），让过期时间分散
  2. Redis 高可用：主从 + Sentinel / Cluster，单点故障自动切换
  3. 熔断降级：Redis 完全不可用时，直接返回默认值或服务降级
  4. 预热：上线前提前加载热点数据，分批次、分时间段

嵌入式边缘存储方案

车端、工控、IoT 设备的存储面临特殊约束：Flash 寿命、断电安全、有限 RAM、无文件系统（某些 MCU）。本章给出嵌入式存储的完整方案谱系。

11.1 嵌入式存储的特殊约束

Flash 寿命限制

NAND Flash 每个块有擦写次数上限（MLC：3000-10000 次；TLC：1000-3000 次）。频繁写同一个地址 → Flash 早死。

对策：磨损均衡（Wear Leveling）、日志结构存储（LSM / JFFS2 / YAFFS）

断电安全

突然断电 → 写到一半的数据损坏 → 系统无法启动。

对策：WAL、原子更新（A/B 分区）、EEPROM 原子写

有限 RAM

某些 MCU 只有 256KB RAM，无法运行 SQLite 这类数据库（需要 64KB+ heap）。

对策：使用超轻量 KV（LittleDB、EEPROM 直接寻址）

无操作系统文件系统

裸机 MCU 没有文件系统，存储需要直接面向 Flash 地址。

对策：Flash 地址映射层、wear leveling 库

11.4 车端遥测数据存储架构

典型的车端数据存储按时效性分三层，越往上越快、越往下越久：

⚡ L1 · 内存环形缓冲（实时）

  • 容量：100–1000 ms 的信号数据
  • 满了覆盖写，不持久化
  • 实现：lock_free_ring_buffer<SignalSample, 8192>
↓ 触发事件（急刹 / 碰撞）时转存

💾 L2 · 本地 Flash 存储（短期）

  • 格式：二进制时序文件（自定义头 + 压缩数据块）
  • 容量：1–4 GB，保留最近 7 天
  • 按小时分文件，每天凌晨删过期文件
↓ 选择性上传

☁️ L3 · 云端（长期）

  • 只传事件相关数据 + 统计聚合
  • MQTT / HTTPS 传输，断线续传
  • 云端用 InfluxDB / S3 存储
车端存储架构的关键设计决策（必须写 ADR 的项目）：
  1. Flash 文件系统选型：LittleFS（断电安全，MCU）vs ext4（成熟，SoC）vs f2fs（为 Flash 优化）
  2. 数据格式：自研二进制（最高效，最难解析）vs Protobuf（中等）vs CSV（最差）
  3. 持久化触发策略：事件触发 vs 定时 vs 全量—— 影响 Flash 寿命
  4. 上传策略：实时流 vs 批量上传 vs 只上传触发事件；带宽 / 流量成本
  5. 加密：遥测数据是否敏感？是否需要 AES-256 加密存储？
  6. OTA 期间数据保护：升级失败时存储数据是否完整？

12.2 索引设计速查

  • 高选择性列 → 优先建索引（email、phone、order_no）
  • 联合索引 → 最左前缀原则；把等值条件列放前面，范围列放后面

  • 主键 → 单调递增（AutoIncrement / UUID v7），避免随机 UUID
  • 函数调用 → 列上加函数会让索引失效，用函数索引或改写查询
  • 深分页 → 用游标分页替代 OFFSET

  • 过期策略：TTL 加随机偏差防雪崩

  • 内存：maxmemory + allkeys-lru 淘汰策略
  • Pipeline：批量命令用 Pipeline，减少网络往返
  • Lua Script：需要原子性的多步操作用 Lua（在 Server 端原子执行）

12.4 MVCC 与隔离级别速查

  • RR（Repeatable Read）防不可重复读，不完全防幻读
  • 快照读（SELECT）用 MVCC；当前读（SELECT FOR UPDATE）用锁
  • 长事务是 InnoDB 的公害：Undo Log 积累、行锁长持有
  • MVCC 实现：每行有 DB_TRX_ID + DB_ROLL_PTR；Read View 快照

Q1 B+ Tree 和 LSM-Tree 各自的适用场景？为什么 RocksDB 用 LSM？
P0主题：存储引擎难度：⭐⭐⭐
核心答案：读写权衡
维度B+ Tree（InnoDB）LSM-Tree（RocksDB）
写性能中（随机 IO）优（顺序写）
读性能优（点查 1-3 次 IO）中（多层查找 + Bloom）
空间中（碎片）高（压缩 4:1）
写放大低高（Compaction）
读放大低中
RocksDB 选 LSM 的原因
  • SSD 顺序写比随机写快 10x，LSM 把所有写变成顺序写
  • 典型 KV 场景：写多读少（如时序数据、日志）
  • 需要高压缩率（Snappy 4:1）
追问：什么是三大放大因子？
  • 写放大（WA） = 实际写 IO / 用户写量。LSM 高（Compaction 重写）
  • 读放大（RA） = 实际读 IO / 用户读量。LSM 中（多层）
  • 空间放大（SA） = 存储 / 用户数据。LSM 中（旧版本暂存）

三者不可能同时优，选型就是选哪个最重要。

Q2 InnoDB 的 RR 隔离级别能不能完全防幻读？解释 MVCC 怎么实现的。
P0主题：MVCC + 隔离级别难度：⭐⭐⭐⭐必考
核心答案：不完全防

InnoDB 的 RR 通过两种机制防幻读，但有边界条件：

  • 快照读（普通 SELECT）：MVCC 保证看到的是事务开始时的快照，能防幻读
  • 当前读（SELECT FOR UPDATE / UPDATE / DELETE）：用 Next-Key Lock（行锁 + Gap Lock）防幻读
  • 混用：先快照读，再当前读，可能看到幻象（快照里没有，但当前读看到了别的事务插入的行）
MVCC 实现机制
  1. 每行有两个隐藏列：DB_TRX_ID（最后修改事务 ID）+ DB_ROLL_PTR（指向 Undo Log 旧版本）
  2. 更新操作不覆盖，而是把旧版本写入 Undo Log，新版本写到当前位置，形成版本链
  3. 事务开始时创建 Read View = {m_ids（活跃事务集合）, min_trx_id, max_trx_id}
  4. 读时遍历版本链，找到第一个对当前 Read View "可见"的版本
可见性规则
if (trx_id < min_trx_id):    可见（已提交的旧事务）
elif (trx_id >= max_trx_id): 不可见（比我晚的事务）
elif (trx_id in m_ids):     不可见（创建时还活跃，未提交）
else:                       可见（已提交）
陷阱：长事务的杀伤力

有任何活跃事务存在，它能看到的所有历史版本就不能被 Purge 线程清理。一个 10 小时不提交的事务 → Undo Log 积累 10 小时的所有写入 → ibdata1 暴涨、磁盘满。必须监控 information_schema.innodb_trx，禁止 > 60s 的事务。

Q3 缓存更新策略：先更新 DB 还是先删缓存？为什么不是先更新缓存？
P0主题：缓存一致性难度：⭐⭐⭐高频
核心答案：先更 DB，再删缓存

这是 Cache-Aside 模式的标准做法。原因是"删除"比"更新"更安全——删除后下一次读会 miss 然后从 DB 加载新值；更新缓存则有竞态：

// 反例：先更新 DB 再更新缓存的竞态
T1: UPDATE DB to V2
T2: UPDATE DB to V3 (T2 比 T1 晚)
T2: SET cache to V3
T1: SET cache to V2 ← T1 较慢，覆盖了 T2
// 结果：缓存里是 V2（旧），DB 里是 V3 → 不一致
为什么不是"先删缓存再更 DB"
T1: DELETE cache
T2: GET cache (miss) → 从 DB 读到旧值 V1 → SET cache = V1
T1: UPDATE DB to V2
// 结果：缓存 V1（旧），DB V2 → 不一致

所以正确顺序是：先更 DB，再删缓存。最坏情况是删缓存失败，缓存里有旧值，直到 TTL 过期或下次更新——是有界的不一致。

面试加分：延迟双删

"我们用延迟双删：更新 DB 前删一次缓存（防止读到旧值后写回），sleep 一段时间（典型 100-500ms，覆盖未完成的读操作），再删一次。这能进一步缩短不一致窗口。"

追问：如果删缓存失败怎么办？

常见方案：(1) 重试机制，失败后入消息队列重试；(2) 用 CDC（Debezium）监听 DB binlog，自动触发缓存删除——这是最可靠的方案，业务代码完全无感。

Q4 Redis 为什么用跳表实现 ZSet，而不是红黑树？
P0主题：Redis 内部难度：⭐⭐⭐
核心原因（Antirez 解释过）
  1. 实现简单：跳表不需要复杂的旋转/平衡操作，~200 行代码搞定；红黑树要 500+ 行且容易出 bug
  2. 范围查询友好：跳表的 L1 链表天然支持有序遍历（ZRANGE 性能极高），红黑树需要中序遍历
  3. 缓存友好：跳表的内存访问模式更线性，CPU 缓存命中率高
  4. 更易扩展：跳表的层数可调，可在内存和速度间权衡
跳表的时间复杂度
  • 查找/插入/删除：O(log N) 平均（与红黑树相同）
  • 范围查询：O(log N + M)，M 是结果数
追问：ZSet 内部小数据怎么存？

Redis 7+ 用 listpack（小数据：< 128 元素且 member < 64B），紧凑连续内存，省内存；超过阈值才转为 skiplist + hashtable（双索引）。

面试加分：跳表的级数随机化

跳表的每个节点层数是随机决定的（概率 p = 0.5），不需要重平衡——这是它实现简单的核心。期望层数 = log_{1/p}(N)，对 10^9 节点也只有 ~30 层。

Q5 索引失效的常见场景有哪些？深分页 LIMIT 10000, 10 怎么优化？
P1主题：索引优化难度：⭐⭐⭐
索引失效六大场景
  1. 列上有函数/计算：WHERE YEAR(created_at) = 2024 → 改为 BETWEEN '2024-01-01' AND '2024-12-31'
  2. LIKE 前缀通配符：LIKE '%alice%' → 全文索引 / Elasticsearch
  3. 隐式类型转换：WHERE phone = 18888888888（phone 是 varchar，触发转换）
  4. OR 连接不同列：优化器放弃索引 → 改 UNION
  5. 跳过联合索引最左前缀：INDEX(a,b,c) 用 WHERE b=2 不走
  6. 不等号 / NOT IN：通常走全表
深分页优化（核心）

问题：SELECT * FROM t ORDER BY id LIMIT 1000000, 10 扫描 1000010 行丢弃前 1M。

// 方案 1：游标分页（推荐）
SELECT * FROM t WHERE id > last_id ORDER BY id LIMIT 10;
// 必须记住"上一页的最后一个 id"，前端配合

// 方案 2：覆盖索引 + JOIN 主键
SELECT * FROM t o
JOIN (SELECT id FROM t ORDER BY id LIMIT 1000000, 10) p ON o.id=p.id;
// 子查询只扫覆盖索引（无回表），主查询只 10 次点查
面试加分：覆盖索引

"我会优先考虑覆盖索引——把 SELECT 列都放进索引，EXPLAIN 显示 Using index，完全避免回表，性能提升 5-10x。"

面试加分：布隆过滤器实战

"对穿透问题，我会用 Redis 的 BloomFilter 模块，启动时把所有合法 key 加载进去。查询时先过 Filter，不在的直接拒绝，连 Redis 缓存都不查。False Positive 率可调（典型 1%），用 1MB 内存能撑亿级 key。"

追问：互斥锁怎么实现？

缓存 miss 时先 SETNX 加分布式锁，获锁者去查 DB 并回填缓存，其他请求等待 50-100ms 重试（避免轮询 spinning）。等待者第二次 GET 缓存通常命中。
