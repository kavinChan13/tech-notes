01软件架构思维：架构师真正的工作是什么

本章解决三个问题：(1) 什么是软件架构（不同流派的定义）；(2) 架构师 / Tech Lead / 高级开发的真实边界；(3) 架构师每天该做什么、不该做什么。这一章是后续所有方法论的"心法"。

1.1 软件架构的多视角定义

"软件架构"这个词被滥用得很严重。不同流派定义不同，但都指向同一个本质：那些一旦做错就极难改的决策。

Bass / SEI 学派

软件架构是系统的一组结构，每个结构由元素、元素间关系，以及两者的属性构成。 — Software Architecture in Practice (3rd ed.)

关键词：结构（structures）。强调多视图建模。

Fowler 学派

架构就是那些"重要"的东西——而所谓"重要"，就是后期很难改的东西。 — Martin Fowler

关键词：不可逆性（irreversibility）。强调决策成本。

Ford / 演进式架构学派

架构是一组关键决策的总和，这些决策塑造了系统的形态，并约束了它的演进路径。 — Building Evolutionary Architectures (Ford, Parsons, Kua)

关键词：约束（constraints）。强调架构定义了"允许做什么"。

实用工程视角（推荐）

架构 = 关键决策 + 质量属性的权衡 + 对演进路径的约束。

当你需要回答"为什么我们必须这样做"而不仅是"我们怎么做"的时候，你就在做架构。

工作定义：软件架构是指那些塑造系统宏观结构、影响关键质量属性、且修改成本远高于普通代码的决策集合。架构师的工作就是识别、做出、记录、传达、维护这些决策。

1.2 架构师 vs Tech Lead vs 高级开发：边界在哪里

这是个被反复讨论但永远没标准答案的问题。我给一个能落地的实用划分，按照"决策的影响半径"区分：

角色 决策影响半径 时间尺度 关键产出 失败后果
高级开发 (Senior IC) 一个模块 / 一个文件 当前 sprint 高质量代码 + 单元测试 重构这个模块，1-2 周
Tech Lead 一个服务 / 一个产品功能 当前季度 模块设计文档 + 代码评审标准 重写这个服务，2-3 个月
架构师 (System / Solution) 多个服务 / 一条产品线 1-3 年 架构图 + ADR + 质量属性场景 重构整个系统，1-2 年，组织级影响
首席架构师 (Chief / Principal) 整个公司技术栈 / 平台 3-5 年 技术战略 + 平台决策 + 标准规范 战略级失败，公司级损失
判断你是否真的在"做架构"的简易测试：把你今天做的决策回滚需要多久？
  • < 1 天 → 你在写代码
  • 1 周 ~ 1 月 → 你在做模块设计
  • 3 月 ~ 1 年 → 你在做架构
  • > 1 年 → 你在做战略

1.3 架构师的日常活动：6 件事

很多人以为架构师就是"画图 + 评审"。真实的架构师每天在做这 6 件事：

① 识别架构关键点

从需求里识别出"哪些是架构层面要解决的"、"哪些只是业务实现"。

例：客户说"系统要快"——你要追问"99 分位 50ms？吞吐 10k QPS？冷启动时间？"

② 做权衡决策

在质量属性互斥时做权衡：性能 vs 可维护性、一致性 vs 可用性、灵活性 vs 简单性。

例：要不要引入消息队列？带来异步解耦，也带来一致性和运维复杂度。

③ 记录决策

用 ADR 把"为什么这样选"写下来，未来人 (包括三个月后的你自己) 能理解上下文。

没记录 = 没做。口头决策两个月就被新人质疑。

④ 沟通与对齐

跨团队对齐架构假设、跨层级（向上说价值、向下讲约束）。

架构师 50% 的时间在"翻译"——把业务翻译成技术，把技术翻译成业务。

⑤ 守护演进

在 code review、设计评审、技术债盘点中守护架构原则不被无声蚀骨。

架构腐化不是某一天突然发生的，是 200 个"就这一次"积累的。

⑥ 持续学习与雷达

跟踪技术演进，维护技术雷达（Adopt / Trial / Assess / Hold）。

架构师必须比团队提前 6-12 个月看到方向，否则你只是"资深开发"。

1.4 架构师的反模式（你一定见过）

❌ 象牙塔架构师

只画 PPT 不写代码，三个月不进 codebase。导致：架构图和真实代码完全脱节，团队私下骂"不接地气"。

解药：每周至少做一次有代码深度的事——code review、原型 PoC、性能 profile、调一个真实 bug。

❌ 框架收藏家

每两个月引入一个新框架/新语言/新模式。导致：技术栈碎片化，团队学习负担爆炸，运维不可控。

解药："加法决策"必须配"减法计划"——引入新东西时同步规划淘汰什么。

❌ 模式狂热者

到处套用 DDD / 微服务 / Event Sourcing。导致：1000 行能解决的问题写成 1 万行，团队战斗力下降 50%。

解药：奥卡姆剃刀——能用 monolith 解决就不上微服务。复杂度不应来自架构，应来自业务。

❌ 决策瓶颈

所有决策都要架构师拍板。导致：团队成长被压制，架构师成为单点故障。

解药："原则下放权"——架构师定原则和约束，团队在约束内自由决策。

❌ 永恒"重构中"

常年告诉团队"等我们重构完就好了"。导致：业务方失去信任，团队进入永恒救火模式。

解药：架构演进必须切片成可发布的小步骤，每一步都带来可见价值。

❌ 拒绝写文档

觉得"代码即文档"。导致：新人 onboarding 三个月才能改第一行代码，所有"为什么"只活在前辈脑里。

解药：ADR + C4 图 + 服务说明书是架构师的硬性 KPI，不是可选项。

1.5 架构能力模型

架构师的能力可以拆成 5 层金字塔。缺任何一层都不是合格架构师——很多"高级架构师"其实只是高级开发，因为他们没有上面三层。

L5 · 战略层
技术战略、技术雷达、Build vs Buy、组织能力建设
 影响 3-5 年
L4 · 沟通层
向上沟通价值、向下解释约束、跨团队协调、写让人读懂的 ADR
 决定 80% 影响力
L3 · 设计层
C4 建模、DDD、风格选型、ATAM 权衡、可演进性设计
 本书核心
L2 · 实现层
编码、调试、性能调优、并发、内存模型——你已有的 28 篇笔记
 必要基础
L1 · 基础层
CS 基础：算法、OS、网络、数据库、编译原理
 入场券
对嵌入式 / 安全关键系统的架构师，L1 还要加上：实时系统原理、控制理论基础、功能安全（IEC 61508 / ISO 26262）、信息安全（ISO 21434）。这些是"硬约束"，不懂就做不出合规的架构。

1.6 本章小结

  • 架构是"难改的决策"的总和，架构师的工作是识别、决策、记录、沟通、守护这些决策。
  • 架构师 vs Tech Lead 的边界以"决策影响半径 + 时间尺度"区分，而不是 title。
  • 架构能力是 5 层金字塔，L3-L5（设计 / 沟通 / 战略）才是架构师独有的能力。
  • 常见反模式：象牙塔、框架收藏家、模式狂热者、决策瓶颈、永恒重构、拒绝写文档。
读到这里，你可以问自己：今天工作里，我做的事 80% 在哪一层？如果都在 L1-L2，那我离架构师还有距离——不是技术差距，是位置和习惯没切换过来。

02架构视图与建模：4+1 视图、C4 模型、UML 取舍

架构必须能"被看见"——通过图、模型、文档传达给团队和未来。本章对比 4+1 视图、C4 模型、UML、ArchiMate，给出"什么时候用什么"的实用建议，并提供 C++ 项目里 C4 四层图的真实示例。

2.1 为什么需要多视图？"盲人摸象"的工程版

一个系统对不同人有不同的"长相"：

  • 业务方关心：能不能解决他的问题、覆盖哪些场景
  • 产品经理关心：用户流程、功能边界
  • 开发关心：模块怎么组织、谁调谁、代码在哪
  • 运维 / SRE关心：部署在哪、依赖什么、出问题怎么定位
  • 安全关心：信任边界、数据流向、攻击面
  • 架构师关心：质量属性、扩展点、演进路径

不可能用一张图同时回答所有人的问题——这是多视图建模存在的根本原因。

2.2 Kruchten 的 4+1 视图（经典参考）

Philippe Kruchten 1995 年提出的 4+1 视图模型是架构建模的鼻祖。虽然在今天显得稍重，但其分类思想至今有效。

视图关注表达受众
Logical View 逻辑视图 功能性需求：系统提供什么能力 类图、对象图、模块图 最终用户、分析师
Process View 进程视图 并发性、同步、性能、可扩展性 活动图、序列图 性能工程师、集成方
Development View 开发视图 软件模块、子系统、库依赖 包图、组件图 开发者
Physical View 物理视图 部署、硬件、网络拓扑 部署图 SRE、运维、硬件
+1 Scenarios 场景视图 关键用例驱动验证其他四视图 用例图、场景 所有人（驱动一致性）
4+1 的价值不在于强制使用全部 5 个视图，而在于提醒你别只画一张"什么都有的大图"——这种图既不能给业务方看，也不能给运维看。

2.3 C4 模型：现代实用主义之选（强烈推荐）

Simon Brown 提出的 C4 模型是目前最流行、最实用的架构建模方法。核心思想：用"地图缩放"的思路画 4 个层级的图。

C4 的核心理念：缩放隐喻

就像 Google Maps，从国家 → 城市 → 街区 → 房间逐层放大：

  • L1 Context：一张图能看到"我们的系统是什么、跟谁打交道"。给非技术人员看。
  • L2 Container：放大到"系统由哪些可独立部署的单元构成"。给架构师、SRE、跨团队看。"Container" 不是 Docker，是"可独立运行的进程或数据存储"。
  • L3 Component：放大到一个 Container 内部的组件结构。给本应用开发看。
  • L4 Code：放大到类/函数级别。通常不画——IDE 能即时生成。
C4 黄金法则：每个层级有自己的受众和抽象层次，不要在一张图上混杂层次。L1 图不能出现"PostgreSQL"——那是 L2 的事。

C4 三类补充图

补充图用途什么时候画
System Landscape Diagram 多个系统的全景（比 Context 更宏观） 组织内有多个系统协作时
Dynamic Diagram 关键流程的运行时序（基于 Container 或 Component） 有跨组件的关键流程需要解释
Deployment Diagram Container 映射到物理 / 虚拟 / 容器节点 SRE / 运维需要理解部署拓扑

2.4 C++ 项目的 C4 实战：以"车端数据采集系统"为例

下面用一个嵌入式架构师常见的案例——车端数据采集与云端遥测系统——演示 C4 各级图怎么画。

Level 1 · System Context

L1 图能回答：这个系统是什么？谁用？跟外面谁交互？——能跟 PM、业务方、安全 review、合规审计沟通。不出现 CAN 协议、DBC 文件、SQLite 这些技术细节。

Level 2 · Container

L2 图揭示：

  • 系统由 4 个独立进程 + 2 块存储构成，进程间通过无锁共享内存环形缓冲通信（架构决策点：为什么不用 DBus / SOME/IP？需要 ADR 记录）
  • "为什么 Health Daemon 是独立进程"——隔离要求：监控不能跟着业务一起挂
  • "为什么 Local Store 和 Rule Cache 用不同技术"——访问模式不同：前者写多读少（时序），后者读多写少（配置）
L2 图就是架构师"出活"的主要交付物。看一个架构师水平高低，看他的 L2 图能不能让一个从未见过这个系统的工程师在 10 分钟内理解全貌、并能猜对 80% 的关键决策原因。

L3 图给本应用的开发者看，告诉他们："这个进程内部有哪些主要模块、谁调谁、各自的职责"。

以一个OpenStandard RPC 管理代理为例，看 C4 如何应用在真实电信软件里。

这个系统运行在 边缘节点（Distributed Unit）上，通过 config schema 协议管理多个 remote-unit 无线单元。

L1 Context——能在一张图上说清楚：

  • 系统（platform agent）：代理 DU 上的 OAM 框架，管理多个 remote-unit
  • 外部系统 1：Upstream Controller（通过 Mgmt 信息模型数据库给下指令）
  • 外部系统 2：remote-unit 无线单元（每个是一个 RPC Server，SSH 连接）
  • 外部系统 3：告警/性能管理服务（Fault Management、PM Agent）

L2 Container——5 个关键 Container 立刻定义了部署单元：

System Boundary: platform agent [单可执行文件，OAM 进程]
  ├── Controller    [协调层] — 管理所有 remote-unit 的注册/启停；Mgmt 事件路由
  ├── PlatformAgent    [每 节点一个实例] — 封装单个 节点的完整生命周期
  │     ├── Dispatcher  [每 RU 独立线程] — Scenario 串行队列
  │     ├── NetconfService — 与 remote-unit 的 RPC 会话
  │     └── Manager/*   — 故障/PM/账户/软件升级等长生命周期服务
  └── Core Services（FmService / PmAgent / AccountService）
External: Mgmt DB, remote-unit [RPC Server], Platform OAM Framework

这张 L2 图揭示了哪些架构决策，一个从没看过代码的工程师 10 分钟内能猜到：

  1. 每 RU 独立 Dispatcher 线程：remote-unit 的操作是有时序约束的 RPC 序列，不能并行 → 每 RU 串行是必然选择
  2. Controller 与 PlatformAgent 分离：多节点 管理的协调逻辑不该放进 RU 自己的实体 → 分层明确
  3. Core Services 共享：告警服务是全局的，不应每 RU 单独实例化 → 共享依赖注入

L3 Component 图进一步把 PlatformAgent 的内部拆开：Adapter（schema/XML 翻译）/ Manager（长生命周期服务）/ Scenario（可编排业务步骤）/ Dispatcher（执行引擎）—— 每一层有明确的职责边界，可以独立演进。

2.5 用代码生成 C4 图：PlantUML 与 Structurizr

不要用 Visio / Draw.io 画 C4 图——架构图必须可版本化、可 diff、可 review。两个推荐方案：

方案 1：PlantUML + C4-PlantUML

@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Container.puml

Person(driver, "驾驶员")
System_Boundary(c1, "车端遥测系统") {
  Container(adapter, "Signal Adapter", "C++", "CAN/SOME-IP 解析")
  Container(engine, "Collection Engine", "C++", "规则匹配 + 采样")
  Container(up, "Uploader", "C++", "MQTT 上传")
  ContainerDb(store, "Local Store", "RocksDB", "时序数据")
}
System_Ext(cloud, "云端数据平台")

Rel(driver, adapter, "车辆事件")
Rel(adapter, engine, "信号流", "shm-ring")
Rel(engine, store, "写入")
Rel(engine, up, "事件", "shm-ring")
Rel(up, cloud, "MQTT/TLS")
@enduml

优势：纯文本、Git 友好、IDE 插件多、CI 可自动渲染 PNG。

方案 2：Structurizr DSL（更专业）

workspace {
  model {
    driver = person "驾驶员"
    tel = softwareSystem "车端遥测系统" {
      adapter = container "Signal Adapter" "C++"
      engine  = container "Collection Engine" "C++"
      up      = container "Uploader" "C++"
      store   = container "Local Store" "RocksDB"
    }
    cloud = softwareSystem "云端" "External"

    driver -> tel "车辆事件"
    adapter -> engine
    engine -> store
    engine -> up
    up -> cloud "MQTT/TLS"
  }
  views {
    systemContext tel { include * }
    container tel { include * }
  }
}

优势：一次定义，自动渲染 L1/L2/L3 多视图，避免视图间不一致。

工程实践：把架构图源文件放到代码仓库的 docs/architecture/ 目录，CI 自动渲染 PNG/SVG 发布到内部 wiki。每次架构变更必须同时改图，否则 PR 不许合入——这是把架构图变成"活文档"的唯一办法。

2.6 UML 在今天还有用吗？

UML 14 种图你不需要全会，但下面这 4 种在架构师工作里依然不可替代：

UML 图架构师场景替代品
类图 领域模型（DDD 战术）、关键类的继承/组合关系 难替代
序列图 关键流程的跨组件时序、异步消息 C4 Dynamic Diagram、PlantUML sequence
状态机图 协议状态机（TCP、SOME/IP）、嵌入式 ECU 状态 难替代
部署图 物理 / 虚拟节点的拓扑 C4 Deployment Diagram
不要用 UML 类图画"业务流程"或"系统全景"——这是 90% 的 UML 误用。类图只用来表达"代码结构"。

2.7 ArchiMate 简介（企业架构师选修）

ArchiMate 是 The Open Group 的企业架构建模语言，覆盖业务层 / 应用层 / 技术层三层。适用场景：你已经从软件架构师升级到企业架构师（EA），需要画"业务能力 → 应用 → 技术基础设施"的全景图。日常 C++ 架构师工作里 80% 用不到，但作为名词需要认识。

2.8 选型决策矩阵

想跟 PM/业务方对齐 C4 Level 1 (System Context) — 不超过 1 张图
想跟跨团队对齐技术架构 C4 Level 2 (Container) + 部署图
想给本团队解释内部模块 C4 Level 3 (Component) + 关键流程的序列图
想表达领域模型 UML 类图（精简版，只画 Aggregate 边界）
想表达协议或状态机 UML 状态机图
想表达运行时跨组件流程 序列图 / C4 Dynamic Diagram
想做企业级架构治理 ArchiMate
对嵌入式系统：除了 C4 之外，状态机图（UML）和数据流图（DFD）是日常必需。特别是车规/航电系统，DFD 配合 ASIL 等级标注是功能安全分析的标准输入（HARA）。AUTOSAR 还有自己的 AML（ARXML）元模型，本质是受限的 UML。

2.9 本章小结

  • 多视图是架构图的基本原则——不同受众看不同图。
  • C4 模型是 2026 年最实用的方案，用"地图缩放"思路画 L1-L4。
  • 架构图必须 用 PlantUML / Structurizr 文本化，进入 Git，跟代码一起 review。
  • UML 不要全用——只在领域模型、状态机、序列、部署这 4 个场景下用。
  • 嵌入式 / 安全关键系统额外需要：状态机图 + DFD（用于 HARA / FMEA）。

03质量属性与权衡：ISO 25010 与 ATAM 评估法

功能性需求决定"系统能做什么"，质量属性决定"系统做得好不好"。本章给出 ISO 25010 的完整质量模型、7 大核心质量属性的可测量定义、互斥关系矩阵，以及 ATAM 评估法的完整流程。架构师 90% 的决策本质都在做质量属性的权衡。

3.1 什么是"质量属性"，为什么它是架构的核心

很多需求文档里只写"系统必须快速、稳定、安全"——这是无法落地的废话。架构师的第一个职责是把它们翻译成可测量、可验证、可权衡的质量属性场景。

质量属性（Quality Attribute）：系统在某个非功能维度上的可测量特性。每个质量属性必须能用"刺激-响应-度量"三段式表达，否则就不是工程概念，只是市场宣传语。

"质量属性场景"模板（来自 SEI / Bass）

[Source]    谁/什么触发？           （客户端、定时器、攻击者、内部组件）
[Stimulus]  触发什么事件？         （请求、故障、攻击、数据洪泛）
[Artifact]  作用在系统哪部分？     （整个系统、某个服务、数据存储）
[Environment] 在什么环境下？      （正常运行、降级、过载、维护窗口）
[Response]  系统怎么响应？        （处理、拒绝、降级、告警）
[Measure]   怎么衡量这个响应？     （时间、数量、概率）

❌ 不可工程化的需求

  • "系统要快"
  • "系统要高可用"
  • "系统要安全"
  • "代码要易维护"

这些既不能验收、不能权衡、也不能写测试。

✅ 可工程化的场景

  • "正常负载下，10k QPS 的 GET 请求到达 API 网关，99 分位响应时间 ≤ 50ms"
  • "单个 AZ 完全失效时，系统在 30 秒内自动切到备用 AZ，数据丢失 0，可用率从 99.95% 退化不超过 30 分钟"
  • "未认证用户尝试访问 /admin，系统 100% 拒绝，5 秒内写入审计日志，触发 3 次/分钟则告警"

3.2 ISO/IEC 25010 质量模型（行业标准）

2011 年发布的 ISO 25010 是国际公认的软件质量模型，包含 8 大特性 + 31 个子特性。架构师不必背全部，但必须知道整张地图，免得漏掉关键维度。

3.3 7 大核心质量属性深度

① Performance（性能）

性能不是单一指标，是一组互相牵制的指标：

  • 延迟（Latency）：单次请求的响应时间。必须用百分位数——P50 是中位数，P99/P999 才反映尾延迟（tail latency）。平均值误导性极强，必须禁用。
  • 吞吐（Throughput）：单位时间处理的请求数。QPS、TPS、Mbps。
  • 容量（Capacity）：在保持其他指标达标的前提下，能承载的最大负载。
  • 资源效率：CPU%、内存 MB、IOPS、带宽。
P99 是骗局之一：在 10k QPS 下，P99 意味着每秒有 100 个用户体验到尾延迟——这不是"罕见"。高要求系统必须看 P99.9 甚至 P99.99。Google 的 Tail at Scale (2013) 论文是这一思想的经典。

② Availability（可用性）

用"几个 9"表达，但要小心定义：

SLA年停机月停机典型场景
99% (两个 9)3.65 天7.2 小时内部工具
99.9% (三个 9)8.76 小时43 分钟普通业务
99.95%4.38 小时21.6 分钟商业级
99.99% (四个 9)52.6 分钟4.3 分钟金融、关键基础设施
99.999% (五个 9)5.26 分钟26 秒电信交换、生命安全
99.999% 是架构难度的指数级跨越——单服务器 + 单数据中心做不到，必须多 AZ + 多 region + 自动故障切换。每多一个 9，成本通常翻倍。架构师必须帮业务方算清楚"每一个 9 值多少钱"。

③ Scalability（可扩展性）

  • Vertical Scaling（垂直）：换更强的机器。简单，但有物理上限和成本悬崖。
  • Horizontal Scaling（水平）：加更多机器。要求服务无状态或状态可分片。
  • Elastic Scaling：根据负载自动扩缩容。需要良好的指标 + autoscaler + 启动时间足够短。
"无状态"是真的"无状态"吗？通常只是把状态外推到了别人那里（数据库、缓存、消息队列）。这就把扩展性问题转移成了状态服务的扩展性问题——往往是真正的瓶颈所在。

④ Reliability / Resilience（可靠性 / 韧性）

  • MTBF（Mean Time Between Failures）：故障间隔
  • MTTR（Mean Time To Recovery）：恢复时间
  • RTO（Recovery Time Objective）：灾难发生后多快恢复
  • RPO（Recovery Point Objective）：能容忍多久的数据丢失

韧性 = 故障预防 + 故障隔离 + 故障检测 + 故障恢复。第 4 章会展开 Bulkhead、Circuit Breaker 等韧性模式。

⑤ Security（安全性）

三要素 CIA：

  • Confidentiality（机密性）：未授权方不可见
  • Integrity（完整性）：未授权方不可改
  • Availability（可用性）：合法方可访问

扩展为 AAA：Authentication（你是谁）、Authorization（你能做什么）、Audit（你做了什么）。

⑥ Maintainability（可维护性）

这是 C++ 项目最常踩坑的属性——因为它无法直接测量。代理指标：

  • 圈复杂度（cyclomatic complexity）
  • 耦合度（afferent / efferent coupling）
  • 模块化深度（depth of inheritance）
  • 修改影响范围（git log 显示一次提交平均修改多少文件）
  • 新人上手时间（onboarding to first commit）

⑦ Evolvability（可演进性）

这是架构师视角下最重要的质量属性。一个系统能否在 5 年内适应业务变化，取决于：

  • 模块边界是否对齐业务边界（DDD Bounded Context）
  • 接口是否抽象到位（依赖倒置）
  • 有没有架构适应性函数（fitness function）防止架构腐化
Building Evolutionary Architectures（Ford 2017）的核心观点：把架构原则写成自动化测试。例如"业务模块不能依赖 UI 模块"——用 ArchUnit / clang-tidy 自定义规则在 CI 里强制。否则人工 review 必定逐渐放水。

3.4 质量属性的"零和博弈"：互斥矩阵

质量属性互相牵制，几乎任何一对都有冲突。下表是常见的互斥关系（红 = 强冲突、黄 = 弱冲突）：

PerfAvailScaleSecMaintCost
Perf — 弱 弱 强 强 强
Availability 弱 — 协同 弱 弱 强
Scalability 弱 协同 — 弱 强 强
Security 强 弱 弱 — 弱 强
Maintainability 强 弱 强 弱 — 弱
Cost 强 强 强 强 弱 —

典型冲突案例

Perf vs Maintainability

极致优化的代码（SIMD、内联汇编、手动展开循环、cache-line padding）几乎不可读。

架构师的取舍：把性能关键路径（hot path）划成独立模块，剩下的 95% 代码追求可读性。不要在整个 codebase 上追求极致性能。

Perf vs Security

TLS 加密、ASLR、stack canary、bounds check 都有性能开销（5-30%）。

取舍：内部网络 + 高性能需求场景可以选 mTLS + 长连接复用；外部 + 中等性能可以全 TLS 1.3。

Availability vs Consistency

CAP 定理的根本冲突。强一致需要协调（共识、锁），降低可用性；高可用需要松一致（最终一致、可读旧值）。

取舍：按数据分类——账户余额必须强一致；用户头像可以最终一致。

Scalability vs Maintainability

微服务化提升扩展性，但显著增加运维、调试、链路追踪、版本管理的复杂度。

取舍：从 modular monolith 开始，等到真的扩展瓶颈出现再拆服务。"Monolith first" — Fowler。

3.5 ATAM：架构权衡分析法

ATAM（Architecture Tradeoff Analysis Method）是 SEI（CMU 软件工程研究所）开发的架构评估方法。简化版是架构师必须掌握的工作方法。

ATAM 完整流程（9 步）

步骤活动产出
1介绍 ATAM 流程所有干系人理解流程
2介绍业务驱动核心业务目标清单
3介绍当前架构架构现状（C4 图）
4识别架构方法（patterns/styles）架构决策清单
5构建效用树质量属性优先级 + 场景
6分析架构方法对效用树叶子的影响每个决策对每个场景的影响
7头脑风暴新场景并优先级排序补充场景
8分析架构方法（重复步骤 6）完整影响矩阵
9呈现结果：识别 风险点、敏感点、权衡点、非风险点评估报告

效用树（Utility Tree）—— ATAM 的核心工具

效用树把"质量"这个抽象概念分解为可优先级排序、可测量的叶子节点。每个叶子是一个质量属性场景。

系统效用 (Utility)
├─ Performance
│  ├─ Latency
│  │  └─ [H,H] 正常负载下 P99 延迟 ≤ 50ms      ← 叶子=场景
│  │  └─ [M,L] 峰值负载下 P999 延迟 ≤ 200ms
│  └─ Throughput
│     └─ [H,M] 单实例支持 5000 QPS
├─ Availability
│  ├─ Failover
│  │  └─ [H,M] 单 AZ 故障 30s 内切换
│  └─ Recovery
│     └─ [H,L] RPO=0, RTO ≤ 5min
├─ Modifiability
│  └─ [M,L] 增加新协议类型 ≤ 3 人天
└─ Security
   └─ [H,M] 所有外部接口必须经过身份认证

每个叶子标注两个维度：

  • 对业务的重要性：H（高） / M（中） / L（低）
  • 实现的难度：H / M / L

[H,H] 的场景就是"架构关键场景"——必须深入分析，必须 ADR 记录，必须有架构层面的应对方案。

ATAM 的 4 类输出

🔴 Risk（风险点）

某个架构决策不能达到某个关键质量属性场景。

例：用单实例 SQLite 存遥测数据，无法支撑 99.99% 可用性的关键场景。

动作：必须有缓解计划。

🟡 Sensitivity Point（敏感点）

某个决策显著影响某一个质量属性。

例：选 Reactor 单线程模型，决定了延迟下限（敏感于线程模型）。

动作：在 ADR 中标注，未来变更要慎重。

🟠 Tradeoff Point（权衡点）

某个决策同时影响多个质量属性，且方向相反。

例：加 TLS → 安全 ↑ 但延迟 ↑、CPU ↑。

动作：必须有显式的权衡决策（ADR 记录"为什么这样选"）。

🟢 Non-Risk（非风险）

评估后确认不构成问题。

例：评估后确认现有缓存策略对 P99 延迟没影响。

动作：记录评估结论，避免未来反复讨论。

3.6 一个完整的 ATAM 小例：决定"上不上消息队列"

背景：车端遥测系统的 Collection Engine → Uploader 当前是同步直接调用。团队提议引入消息队列（NATS / ZeroMQ / 自研 ring buffer）。架构师做 ATAM 评估。

Step 1: 列出关键场景（从效用树挑 [H,H]）

  1. S1: 网络波动时，不丢失已采集事件（数据完整性）
  2. S2: 突发 10x 数据洪泛，不压垮下游（韧性）
  3. S3: P99 端到端延迟（从信号到云端） ≤ 5 秒（性能）
  4. S4: 单进程 OOM 不影响其他组件（隔离性）
  5. S5: 团队能在 6 个月内维护这套系统（可维护性）

Step 2: 候选架构方案

方案简述
A. 同步直接调用Engine 同步调 Uploader API，简单
B. 内存 ring buffer本进程内 lock-free ring buffer + 异步线程
C. 持久化 MQ（如 NATS JetStream）独立 broker 进程 + 磁盘持久化
D. 共享内存队列（boost::interprocess）跨进程共享内存，无外部依赖

Step 3: 影响矩阵

场景A 同步B 内存环C MQD 共享内存
S1 不丢数据 ❌ 网络挂=丢 ⚠ 重启=丢 ✅ 持久化 ⚠ 重启=丢
S2 抗洪泛 ❌ 反压传染 ✅ 缓冲 ✅ 缓冲+持久 ✅ 缓冲
S3 P99 ≤ 5s ✅ 直接 ✅ 微秒级 ⚠ broker 跳数 ✅ 微秒级
S4 隔离 ❌ 同进程 ❌ 同进程 ✅ 独立 broker ✅ 跨进程
S5 可维护 ✅ 最简单 ✅ 自研代码少 ❌ 新依赖 ⚠ 自研难度

Step 4: 标注 4 类点

  • 风险点（A）：同步调用在 S1/S2/S4 上都失败，被淘汰。
  • 权衡点（C vs D）：C 在 S1/S2/S4 优于 D，但在 S3/S5 劣于 D。
  • 敏感点：S1（不丢数据）是否真的"必须"——如果业务可接受 0.1% 丢失率，可以选 D。
  • 非风险点：所有方案都能满足 S3 的 5s 延迟要求（B/D 微秒级，C 毫秒级，都远低于 5s）。

Step 5: 决策（输出 ADR）

决策：选 D（共享内存队列）。理由：(1) S3 性能要求微秒级，C 的 broker 跳数风险大；(2) 嵌入式资源约束，多一个 broker 进程是大开销；(3) 业务方接受"重启丢失 30 秒内未上传数据"的妥协；(4) 自研难度可控，团队已有 lock-free 经验。风险缓解：(1) 重启时持久化未上传数据到本地（解决 S1）；(2) 加 Health Daemon 监控队列水位（缓解隐藏的反压）。
ATAM 不必"严格按 9 步走"。架构师日常的"轻量版 ATAM"就是：(1) 列 3-5 个关键场景；(2) 列 2-4 个候选方案；(3) 画影响矩阵；(4) 标 4 类点；(5) 写 ADR。整个过程 1-2 个会议就能完成，远比"拍脑袋决策 + 写一段废话理由"专业。

3.7 CBAM 简介：把质量决策货币化

CBAM（Cost-Benefit Analysis Method）是 ATAM 的延伸：把每个架构决策的收益和成本都换算成钱，做 ROI 分析。

核心思想：

  • 每个质量属性场景的"效用"是关于响应度量值的效用函数（不是线性！比如延迟从 100ms 降到 50ms 价值很大，从 50ms 降到 25ms 价值变小）
  • 每个方案的"收益" = ∑ (效用提升 × 场景权重)
  • ROI = 收益 / 成本
CBAM 在 C++ 项目里完整做的不多，但"效用函数非线性"这个思想极其重要。架构师必须能回答："这个优化值得吗？我们离边际效用 0 还有多远？"——这是停止过度优化的依据。

3.8 嵌入式 / 安全关键系统的额外质量属性

传统软件不太强调，但安全关键系统必须有的质量属性：
属性含义标准
Functional Safety 功能安全 故障时进入安全状态，不伤人 IEC 61508、ISO 26262（汽车）、DO-178C（航电）、IEC 62304（医疗）
Determinism 确定性 同样输入产生同样输出，时序可预测 RTOS、Worst-Case Execution Time (WCET) 分析
Real-time 实时性 响应时间有硬上限（hard deadline） POSIX.1b、Xenomai、PREEMPT_RT
Cybersecurity 信息安全 抗篡改、抗注入、安全升级 ISO/SAE 21434（汽车）、UNECE R155/R156
Maintainability over decades 系统要服役 15-20 年 编译器、库、HW 长期可用性

这些属性的特点：

  • 不能事后补救——必须从架构层面设计
  • 合规约束 = 硬约束——不能 trade-off，只能满足或不发布
  • 影响整个生命周期——从需求到验证到部署到维护

3.9 本章小结

  • 质量属性必须用"刺激-响应-度量"三段式表达，否则不可工程化。
  • ISO 25010 提供完整 8+31 模型，架构师从中挑出 5-7 个"关键质量属性"作为非功能基线。
  • 质量属性互相牵制，几乎任何决策都是 trade-off，不存在"完美架构"。
  • ATAM 是工程化的权衡评估法，核心工具是效用树 + 4 类输出（风险/敏感/权衡/非风险）。
  • 架构师的日常思维就是 "轻量 ATAM"：列场景 → 列方案 → 影响矩阵 → 标点 → 写 ADR。
  • 安全关键系统额外有 Functional Safety / Determinism / Real-time / Cybersecurity 等硬约束。
小练习：拿你当前在做的一个系统，列出它的 5 个 [H,H] 质量属性场景。如果列不出 3 个，说明你这个系统的非功能需求基线还没建立——架构师的第一件事就是把这个建起来。

04六大经典架构风格：从单体到事件驱动

"架构风格"（architecture style）是被反复验证的整体组织方式。本章深入六种最常用的风格：分层、六边形（Hexagonal）、洋葱、Clean Architecture、CQRS+Event Sourcing、微服务、事件驱动（EDA）。每种都给出 C++ 落地的目录结构 + CMake 模块化方案，以及"什么时候用、什么时候不用"的判断准则。

4.1 风格选型的"奥卡姆原则"

风格选型第一原则：默认从最简单的风格开始（单体 / 分层），只有出现明确的痛点才升级到更复杂的风格。

风格越复杂 → 学习成本越高、运维成本越高、出错点越多。"为了使用 DDD 而 DDD" 是架构师最常见的失败模式之一。

Fowler 的著名忠告："Monolith First"——先做好单体，等单体真的撑不住了再拆。微软、Shopify、Stack Overflow 都坚持单体到非常大的规模。

4.2 风格 ① · Layered Architecture（分层架构）

最经典、最基础的风格。把代码按"职责层"水平切分，上层只能依赖下层。

分层架构的两个变体

✅ Closed Layered（严格分层）

每层只能调用紧邻下一层。Presentation 不能直接调 Infrastructure。

优势：耦合最小，每层可独立替换。

代价：很多"穿透层"代码（pass-through），开发繁琐。

⚠ Open Layered（开放分层）

每层可跨层调用下方任意层。

优势：减少穿透代码。

代价：耦合增加，分层逐渐模糊。大型项目慎用。

分层架构的致命弱点

分层架构的核心问题：Domain 层依赖了 Infrastructure 层——Order 实体保存到数据库的逻辑写在 OrderRepository（infrastructure）里，但 Order 实体经常需要"知道"自己怎么被持久化（比如 ORM 注解）。结果：业务逻辑跟技术细节耦合，无法独立测试业务规则。这就是六边形架构要解决的问题。

什么时候用分层架构？

  • ✅ 小到中型项目（< 10 万行代码）
  • ✅ 团队 < 10 人，没有专门架构投入
  • ✅ 业务规则相对简单，主要是 CRUD + 编排
  • ❌ 业务规则复杂（需要 DDD）
  • ❌ 多个不同 UI / 不同存储后端要切换
  • ❌ 需要业务逻辑独立单元测试（不依赖 DB / 网络）

4.3 风格 ② · Hexagonal Architecture（六边形 / Ports & Adapters）

Alistair Cockburn 2005 年提出。核心思想：把业务核心放在六边形中央，所有外部依赖（UI、DB、第三方）都通过"端口"接入，端口的实现叫"适配器"。

核心概念

概念含义C++ 表达
Driving Port 外部世界调用核心的接口 核心定义的 use case 类（如 PlaceOrderUseCase）
Driving Adapter 实现外部入口，把 HTTP/CLI 翻译成对 use case 的调用 HTTP controller、gRPC service
Driven Port 核心定义的"我需要什么"接口 核心定义的纯虚类（如 IOrderRepository）
Driven Adapter 实现 Driven Port，封装外部依赖 PostgresOrderRepository、KafkaEventPublisher

C++ 代码示例：依赖倒置的核心

// === domain/order.hpp（核心，零依赖）===
struct Order {
  OrderId id;
  CustomerId customer;
  std::vector<OrderLine> lines;
  Money total() const;
};

// === application/ports/i_order_repository.hpp（端口=接口，核心定义）===
class IOrderRepository {
public:
  virtual ~IOrderRepository() = default;
  virtual void save(const Order&) = 0;
  virtual std::optional<Order> find_by_id(OrderId) = 0;
};

// === application/place_order_use_case.hpp（核心 use case）===
class PlaceOrderUseCase {
  IOrderRepository& repo_;
  IEventPublisher& events_;
public:
  PlaceOrderUseCase(IOrderRepository& r, IEventPublisher& e)
    : repo_(r), events_(e) {}

  OrderId execute(const PlaceOrderCommand& cmd) {
    Order o{generate_id(), cmd.customer, cmd.lines};
    repo_.save(o);              // 通过端口
    events_.publish(OrderPlaced{o.id});
    return o.id;
  }
};

// === infrastructure/postgres_order_repository.hpp（适配器，依赖核心）===
class PostgresOrderRepository : public IOrderRepository {
  pqxx::connection& conn_;
public:
  void save(const Order& o) override { /* INSERT INTO orders ... */ }
  std::optional<Order> find_by_id(OrderId id) override { /* SELECT ... */ }
};

// === presentation/http_order_controller.cpp（适配器，依赖核心）===
void handle_post_order(const HttpRequest& req, HttpResponse& res, PlaceOrderUseCase& uc) {
  auto cmd = parse_json_to_command(req.body);
  auto id = uc.execute(cmd);
  res.status = 201;
  res.body = R"({"order_id":")" + to_string(id) + R"("})";
}

六边形的核心收益

✅ 业务核心 100% 可独立测试

测试 PlaceOrderUseCase 时，传入 InMemoryOrderRepository mock。不需要 DB、不需要 HTTP、不需要 Kafka。单元测试毫秒级。

✅ 技术细节可替换

从 PostgreSQL 换 MongoDB？只改 infrastructure/。核心和 controller 都不动。从 REST 换 gRPC？只改 presentation/。

六边形架构最大的意义：它把"业务核心"从"技术框架"里解放出来。10 年后框架都换了 3 代，但核心业务代码还能用——这才是真正的长生命周期架构。

4.4 风格 ③ · Onion Architecture（洋葱架构）

Jeffrey Palermo 2008 年提出。本质上是六边形的"同心圆可视化版本"，强调依赖只能由外向内。

洋葱 vs 六边形：差异

维度HexagonalOnion
可视化六边形 + 端口/适配器同心圆
层数2 层（核心 vs 外部）多层（Domain Model / Domain Services / App Services / Infrastructure）
强调"端口与适配器"的解耦"层数 + 依赖方向"的纯粹
本质完全相同——依赖倒置 + 业务核心独立

4.5 风格 ④ · Clean Architecture（Uncle Bob 版）

Robert C. Martin 2012 年集大成。本质上还是六边形 / 洋葱，但提供了更精细的层次划分和命名。

四层的精确定义

Layer 1 · Entities

企业级业务规则。最稳定，几乎不变。

例：Order 有什么字段、Order 的总价怎么算（业务规则）。

这一层对应 DDD 的 Entity + Value Object + Aggregate。

Layer 2 · Use Cases

应用业务规则。回答"我们能用 Entity 做什么"。

例：PlaceOrderUseCase、CancelOrderUseCase。每个用例一个类，单一职责。

Layer 3 · Interface Adapters

数据格式转换。在 Use Case 友好格式 ↔ 外部格式之间翻译。

例：Controller（HTTP → Command）、Presenter（Result → ViewModel）、Gateway（Domain → SQL）。

Layer 4 · Frameworks & Drivers

技术细节。所有外部框架、库、设备。

例：Spring/Express/Boost.Beast、PostgreSQL driver、文件系统、设备驱动。

The Dependency Rule（依赖规则）—— Clean Arch 的灵魂

Source code dependencies must point only inward, toward higher-level policies.
源代码依赖只能向内，指向更高层的策略。

这意味着：

  • Entity 不能 #include 任何 Use Case 的头
  • Use Case 不能 #include 任何 Adapter 的头
  • Use Case 需要数据库时，定义一个抽象接口（在 Use Case 层），让 Adapter 实现

4.6 风格 ⑤ · CQRS + Event Sourcing

CQRS（Command Query Responsibility Segregation）

"读"和"写"用不同的模型。Greg Young 2010 年提出，源自 Bertrand Meyer 的 CQS 原则。

什么时候用 CQRS？

✅ 适用场景

  • 读写不平衡：读 QPS 远大于写（如电商商品页）
  • 读写模型差异大：写要规范化（事务），读要反规范化（join 太多）
  • 读端有多种视图：搜索、推荐、列表、详情都用不同优化
  • 需要任意历史时点查询（搭配 Event Sourcing）

❌ 不适用场景

  • 简单 CRUD（90% 的内部系统）
  • 读写量级相当
  • 团队不熟悉异步思维（读端延迟会让用户体验出问题）
  • 对最终一致性接受度低（金融账户余额慎用）

Event Sourcing（事件溯源）

核心思想：不存"状态"，存"事件序列"。当前状态 = 重放所有历史事件的结果。

// 传统：存当前状态
TABLE orders (id, status, total, ...) -- 只知道"现在"

// Event Sourcing：存所有事件
TABLE events (
  aggregate_id,       -- 哪个 Order
  event_id,           -- 全局递增
  event_type,         -- OrderPlaced / ItemAdded / OrderPaid / OrderCancelled
  event_data,         -- JSON / protobuf
  timestamp
)
-- 重放 aggregate_id=X 的所有事件得到当前 Order 状态
Event Sourcing 的魔力：任意时点回放。
  • "昨天下午 3 点这个订单是什么状态？" → 重放到时刻 T
  • "如果我们当时用了 20% 折扣，今天总收入会是多少？" → 替换某段事件，重放
  • "BUG 修了，重新计算所有历史" → 改投影逻辑，全量重放
  • "完整审计日志，零额外成本" → 事件本身就是审计日志
代价：复杂度极高，事件 schema 演进、快照、版本兼容、重放性能都是大坑。

Event Sourcing 的 C++ 简化示例

struct OrderEvent {
  OrderId aggregate_id;
  uint64_t seq;
  std::variant<OrderPlaced, ItemAdded, OrderPaid, OrderCancelled> payload;
  Timestamp ts;
};

class Order { // Aggregate
  OrderId id_;
  OrderStatus status_ = OrderStatus::None;
  std::vector<OrderLine> lines_;
  Money paid_amount_;

public:
  // 重放：把状态构建出来
  void apply(const OrderEvent& e) {
    std::visit([this](auto&& ev) { apply_impl(ev); }, e.payload);
  }
  void apply_impl(const OrderPlaced& e)   { id_ = e.id; status_ = OrderStatus::Placed; }
  void apply_impl(const ItemAdded& e)     { lines_.push_back(e.line); }
  void apply_impl(const OrderPaid& e)      { paid_amount_ += e.amount; status_ = OrderStatus::Paid; }
  void apply_impl(const OrderCancelled& e) { status_ = OrderStatus::Cancelled; }

  // 命令：产生新事件
  std::vector<OrderEvent> place(const PlaceOrderCommand& cmd) {
    if (status_ != OrderStatus::None) throw InvalidStateError{};
    OrderEvent ev{cmd.id, 0, OrderPlaced{cmd.id, cmd.customer}, now()};
    apply(ev);
    return {ev};
  }
};

// 仓储：load = 拉所有事件 + 重放
class EventSourcedOrderRepository : public IOrderRepository {
  IEventStore& store_;
public:
  Order load(OrderId id) {
    Order o;
    for (const auto& ev : store_.load_events(id)) o.apply(ev);
    return o;
  }
  void save(const std::vector<OrderEvent>& events) { store_.append(events); }
};

4.7 风格 ⑥ · 微服务架构 (Microservices)

把单体拆成多个独立部署的小服务，每个服务对应一个业务能力，独立技术栈、独立数据库、通过网络通信。

微服务的核心原则（Sam Newman / Chris Richardson）

  1. 围绕业务能力组织（不是技术分层）
  2. 每个服务有独立数据库（"Database per Service"）
  3. 智能终端 + 哑管道（业务逻辑在服务里，通信尽量薄）
  4. 去中心化治理 + 数据管理
  5. 故障设计（必须假设依赖会挂）
  6. 演进式设计（服务边界会变化）

微服务带来的复杂度（真实成本）

新增复杂度说明
分布式系统网络延迟、超时、重试、幂等、补偿——全是新工程
分布式事务跨服务的一致性几乎只能用 Saga / Outbox / TCC
服务发现需要 Consul / etcd / K8s Service
配置管理100 个服务 × N 个环境 = 配置爆炸
可观测性必须有分布式 tracing，否则定位一次故障要几小时
部署复杂度需要 CI/CD、容器编排、灰度发布
测试难度端到端测试昂贵，需要 contract testing
团队成本团队组织必须与服务边界对齐（Conway's Law）
微服务不是免费午餐。Martin Fowler 在 *Microservice Premium* 一文里明确指出：团队规模 < 30 人、用户 < 100 万的产品，微服务带来的开销大概率超过收益。先做好模块化单体（modular monolith），等真正撑不住再拆。

什么时候用微服务？

✅ 微服务收益大于成本的信号

  • 团队 > 50 人，多团队并行开发互相阻塞
  • 不同模块的扩展需求差异巨大（一个模块要 100x scale，其他不变）
  • 不同模块的发布节奏差异大（一个天天发，一个一年发一次）
  • 需要不同技术栈（一个 C++ 高性能、一个 Python 算法）
  • 故障隔离要求高（一个挂不能影响其他）

❌ 微服务伪需求的信号

  • "看起来更现代"
  • "听说大厂都这么做"
  • 团队 < 10 人却要拆 20 个服务
  • 没有 K8s / 没有 tracing / 没有 CI/CD
  • 分库分表都没做过就想拆服务

4.8 风格 ⑦ · Event-Driven Architecture (EDA)

组件之间通过异步事件解耦，发布者不知道有谁在订阅。两个主流变体：

Broker Topology（Broker 拓扑）

所有事件流过中央 broker（Kafka / NATS）。生产者、消费者完全解耦。

优势：极致解耦，水平扩展容易。

代价：broker 是单点关键基础设施。

Mediator Topology（Mediator 拓扑）

有一个中央协调器（mediator）编排事件流程，类似 BPMN 工作流引擎。

优势：流程清晰、可视化。

代价：mediator 成为新的耦合点。

嵌入式 / 车端系统里，"事件驱动" 是默认架构而不是选项——CAN/Ethernet 信号本身就是事件流，AUTOSAR Adaptive 的 ara::com 提供完整的事件订阅模型，ROS2 的 Topic 也是发布-订阅。架构师要关心的是事件流的拓扑（中心化 vs P2P）、QoS 保证（at-least-once vs exactly-once vs best-effort）、背压策略。

4.9 风格选型决策矩阵

10 万行以下，团队 5 人 分层架构 (Layered) + Modular Monolith
业务规则复杂，需要长期演进 Hexagonal / Clean Architecture + DDD
读写不平衡，多视图 CQRS（先不上 Event Sourcing）
需要完整审计 / 时点重放 CQRS + Event Sourcing
团队 50+ 人，模块边界清晰 Microservices（每个服务内部用 Hexagonal）

嵌入式 / 车端 EDA（信号驱动）+ Layered（控制流）混合

4.10 本章小结

  • 奥卡姆剃刀：默认从最简单风格开始，痛点驱动升级。
  • Layered 是入门基础，Hexagonal / Onion / Clean 本质相同——都是依赖倒置 + 业务核心独立。
  • CQRS / Event Sourcing 是高级模式，复杂度极高，慎用。
  • 微服务不是更现代的单体，是组织级架构，团队规模和业务节奏不匹配时反而是负担。
  • EDA 在嵌入式 / 高吞吐场景下是默认选择。
  • C++ 落地核心：用 CMake 的 target 边界 + INTERFACE / PRIVATE 控制依赖方向，让架构原则在编译期就被强制。
小练习：拿你现在的项目，画一个简单的"模块依赖图"。如果出现循环依赖，或者"业务模块依赖了 UI/DB 模块"——你已经偏离了任何一种健康架构风格。这是第一个要修的架构问题。

05DDD 战略设计：Bounded Context、Ubiquitous Language、Context Map

DDD（Domain-Driven Design）不是一种代码风格，而是一种"业务建模 + 架构边界"的方法论。本章只讲战略部分——这是架构师必修；战术部分（Entity / VO / Aggregate）放在下一章。如果你只能学 DDD 的一半，请学战略——它能直接决定"系统应该拆成几个、按什么边界拆"。

5.1 DDD 解决的根本问题：为什么这么多系统会"烂掉"？

大型系统腐化的根因不是技术烂，而是业务模型和代码模型脱节：

  • 产品和开发说着两套"方言"——开同一个会，理解不一致
  • 同一个业务概念（"订单"）在不同模块里含义完全不同，但都叫 Order
  • 业务规则散落在 service / controller / SQL / 前端校验里，没有"唯一真理"
  • 模块边界按"技术分层"切（service / dao / model），不是按业务能力切——导致一个业务变更牵动 N 个模块
DDD 的核心信念：软件应该模型化业务，业务模型应该由业务专家和开发共同建立、共同维护，并直接体现在代码里。

5.2 Ubiquitous Language（统一语言）

这是 DDD 第一块基石，也是最容易被忽略的一块。

Ubiquitous Language：业务专家 + 产品 + 开发 + QA 共同使用的同一套词汇，同一个词在所有地方（需求文档、会议、代码、数据库表名、日志）含义一致。

例子：大型企业客户的"订单"歧义

说话人"订单"指什么
销售客户签的合同
计费每月生成的账单条目
开通团队装机/开通的工单
客服客户报修的工单

如果都叫 Order，代码就成了一锅粥。DDD 的做法是：

  • 在每个上下文里给它独立的名字：SalesContract、BillingItem、ProvisioningTicket、ServiceRequest
  • 每个名字在它自己的上下文内含义唯一
  • 跨上下文交互时显式翻译，不假装它们是同一个东西
"统一语言"的检验标准：把会议录音转成文字，关键名词是否都能在代码里找到同名类/方法？ 如果业务讲"客户解约"，代码里却找不到 cancel_subscription 而是 delete_record，那语言已经分裂了。

5.3 Domain（领域）的三层划分

不是所有业务能力都同等重要。DDD 把领域分成 3 类，投入策略完全不同：

Core Domain · 核心域

公司的核心竞争力。决定生死的能力。

例：电商的搜索/推荐、银行的风控、电信的计费、自动驾驶的感知决策。

策略：自研、最资深的工程师、DDD 战术深度建模、长期投入。

Supporting Domain · 支撑域

必须有，但不是竞争优势。

例：用户管理、权限系统、消息通知。

策略：可以自研但简化、可以外包、可以用开源。

Generic Domain · 通用域

所有公司都需要的通用能力。

例：身份认证（OAuth）、邮件发送、文件存储、日历。

策略：买现成的（Auth0、SendGrid、S3），不要自己写。

架构师的关键洞察：识别"核心域"是战略问题，不是技术问题。把团队最强工程师放到通用域写邮件服务，是公司最大的浪费。

5.4 Bounded Context（限界上下文）—— DDD 战略的核心

Bounded Context：一个明确的边界，在其内部，模型和语言是统一一致的。跨越边界，同一个词可以有不同含义。

Bounded Context 不是模块、不是服务、不是层

它是一个语言学概念：在这个边界内，"客户"这个词只有一个含义。出了这个边界，"客户"可能就是另一回事了。

边界形式什么时候用
同一进程内的模块（namespace + 目录）单体应用，团队 < 20 人
独立服务（独立部署）团队大、不同节奏、不同扩展需求
独立产品 / 业务线组织级别的隔离
常见误解："Bounded Context = 微服务"。错。一个 Bounded Context 可以实现为单体内的模块、共享库、独立服务——具体形式取决于团队结构和质量属性需求。DDD 不预设具体的物理实现。

如何识别 Bounded Context？

四个信号告诉你"这里应该切一刀"：

信号 ① · 语言变化

同一个词在不同场景含义不同，或者出现了"我们这里叫 X，你们那里叫 Y"。

这是最强信号——语言边界 = 上下文边界。

信号 ② · 团队边界

不同团队负责不同业务，沟通成本高。

Conway's Law：系统结构最终会和组织结构对齐——主动设计这种对齐。

信号 ③ · 变更节奏

两块业务变更频率差异很大——一个天天改，一个一年改一次。

不切开，慢的会拖累快的，快的会让慢的不稳定。

信号 ④ · 质量属性差异

一块要 5 个 9 可用、毫秒级延迟，另一块只要分钟级响应。

不同的质量需求 → 不同的技术栈、不同的运维投入。

Bounded Context 实战：电商系统

注意上图中："Product" 这个词在 4 个上下文里含义完全不同——这就是每个 Bounded Context 拥有自己的语言的真实例子。千万不要造一个"统一 Product"对象给所有人共用——那就是经典的"贫血上帝模型"。
OpenStandard RPC 管理代理的 Bounded Context 是如何划分的？

这个系统有 30+ 个业务域（Scenario 目录），但用 DDD 战略视角来看，可以清晰归为 6 个 Bounded Context：

Bounded Context类型语言特征典型操作
载波管理Core"Carrier" = 频谱资源单元，有状态机创建/删除/激活/休眠载波
告警管理Core"Alert" = 已规范化的告警条目，含厂商扩展规则告警映射、规则引擎、转发
软件管理Supporting"Package" = 软件包版本，有安装状态机下载/安装/激活/回滚
账户管理Supporting"Account" = SSH/TLS 凭据增删改查 SSH 公钥/TLS 证书
性能监控Supporting"Counter" = PM 计数器，有采样周期配置采集、上报
schema 协议栈Generic（遵从）"Session" / "RPC" = rpc-lib2 的概念edit-config / get / notification

为什么这种划分比"按技术层分"更好？

  • 告警团队和载波团队可以独立迭代，互不阻塞（两者在代码里是完全独立的 Scenario 子目录）
  • "Fault"这个词在告警上下文里只有一个含义，不会和载波的"故障状态"混淆
  • schema 协议栈是 Generic Domain——对接 rpc-lib2 / libschema，不自研，作为 Conformist 直接遵从它的数据模型

代码印证示例：系统里有 VendorSpecificAlertRules（告警 BC 专属的规则引擎）、SessionScenario（会话 BC 专属的编排逻辑），两者完全不共享业务对象——正是 Bounded Context 在代码层面的体现。

5.5 Context Map（上下文映射图）—— 战略设计的全景

Context Map 描述多个 Bounded Context 之间的关系。Eric Evans 定义了 9 种关系模式，下面是最重要的 7 种：

关系含义典型场景
Partnership 合作 两个上下文必须同步演进，互相影响对方设计 两个相互依赖的团队，必须一起 plan
Shared Kernel 共享内核 两个上下文共享一小段领域模型代码 团队都信任的核心定义，但变更必须双方同意
Customer / Supplier 客户/供应商 下游是上游的客户，上游优先考虑下游需求 核心域被多个支撑域消费
Conformist 跟随者 下游完全采用上游的模型（不翻译） 上游强势（如第三方 API），下游无议价能力
Anticorruption Layer (ACL) 防腐层 下游建一层"翻译层"，把上游模型翻译成自己的 对接遗留系统、第三方 API；保护核心域不被污染
Open Host Service (OHS) 开放主机服务 上游提供一个公开的、标准化的协议，所有下游用同一套接口 平台服务（如 RESTful API），下游多且多样
Published Language 公开语言 OHS 配套的标准数据格式（JSON schema、protobuf、AsyncAPI） OHS + 明确的数据契约

架构师最重要的两个关系：ACL 与 OHS

Anticorruption Layer（防腐层）

当你必须对接糟糕的外部系统（遗留系统、第三方 API、设计混乱的供应商）时，不要让它的模型渗透进你的核心域。

// 不要这样做：核心域里出现 SAP 的字段
class Order {
  std::string KUNNR;  // SAP 的客户号字段名
  std::string MATNR;  // SAP 的物料号
};

// 正确：用 ACL 翻译
class SapCustomerGateway {  // 防腐层
  public:
  CustomerId find_by_sap_code(const std::string& kunnr) {
    auto sap_resp = sap_client_.query("BAPI_...", kunnr);
    return CustomerId{translate(sap_resp.id_field)};
  }
};

核心域的代码看不到 SAP 的任何痕迹。

Open Host Service（开放主机服务）

当你是上游，被多个下游消费时，提供一个稳定的、自包含的接口契约，不要为每个消费者单独写适配。

  • 定义清晰的 API：REST + OpenAPI、gRPC + protobuf
  • 提供清晰的 Published Language（数据 schema）
  • 版本化（v1、v2 并存）
  • 变更通过 deprecation 流程

把"对接成本"从 O(N) 降到 O(1)。

Context Map 实战示例：车端遥测系统

Context Map 揭示的架构决策：

  • 对接车辆信号 → 必须用 ACL（车型多、HW vendor 杂、信号编码各异）
  • 对外暴露给云端 → 提供 OHS（protobuf schema 作为 Published Language）
  • 跟 OTA 是 Customer/Supplier 关系 → 优先考虑 OTA 团队的接口需求
  • 跟 Prometheus 是 Conformist → 直接用它的数据模型，不翻译

5.6 Event Storming：发现 Bounded Context 的工作坊技术

Event Storming（事件风暴）是 Alberto Brandolini 2013 年提出的协作建模方法，是识别 Bounded Context 的最有效工具。

三种粒度

Big Picture Event Storming

整个业务的事件全景。2-4 小时。

产出：业务事件地图、找到边界裂缝、识别热点（hotspot）。

Process Level

聚焦一个流程的深度建模。1 天。

产出：完整流程的 Command-Event-Policy 链。

Design Level

聚焦一个 Aggregate 的设计。0.5-1 天。

产出：Aggregate 边界、不变量、操作。

Event Storming 的标准符号

颜色含义语法
🟧 橙色 StickyDomain Event过去时："订单已下"
🟦 蓝色 StickyCommand动词原形："下单"
🟨 黄色 StickyActor / Person触发 command 的人
🟪 紫色 StickyPolicy / Reaction"当 X 发生时，Y 自动发生"
🟩 绿色 StickyRead Model / ViewUI 看的数据
🟥 红色 StickyHotspot / Problem"这里有问题/疑问/争议"
🟫 棕色 StickyExternal System第三方系统
Event Storming 的最大价值：红色 sticky（hotspot）。它们标记了团队认知不一致的地方——往往就是 Bounded Context 边界的位置，或者业务流程里真正复杂的部分。架构师应该主动找红色，而不是回避它。

5.7 战略设计的产出物

一次完整的战略设计活动后，架构师应该带走这些产出物：

  1. Bounded Context 清单（名字 + 一句话职责 + 业务负责人）
  2. 每个 Context 的 Domain 分类（Core / Supporting / Generic）+ 投入策略
  3. Context Map（含 ACL / OHS 等关系标注）
  4. Ubiquitous Language 词典（每个 Context 一份）
  5. 团队 - Context 映射（谁拥有哪个 Context）
  6. Event Storming 照片（保留原始讨论痕迹）
  7. 关键 ADR：为什么这样划分、为什么核心域是 X

5.8 嵌入式 / 安全关键系统的 DDD 适配

DDD 在嵌入式领域的适配：
  • Bounded Context 通常对应"功能簇"：在车载系统里，"动力总成"、"车身控制"、"信息娱乐"、"ADAS" 通常就是天然的 Bounded Context。
  • Context Map 通常体现在通信矩阵（CAN matrix / SOME/IP service interface）：信号 / 服务跨域调用就是上下文交互。
  • "统一语言"在嵌入式领域有现成的：DBC 文件、ARXML、IDL。架构师要保证这些文件就是"唯一真理"，代码生成、文档、测试都从它派生。
  • Core / Supporting 分类对应 ASIL 等级：ASIL-D 的功能（如刹车控制）就是核心域，必须投入最多资源；QM 等级的（如座椅记忆）是支撑域。
  • ACL 在嵌入式至关重要：对接不同 vendor 的 ECU、HW 驱动、AUTOSAR Classic ↔ Adaptive 之间的过渡，都需要明确的防腐层。

5.9 本章小结

  • DDD 战略设计的核心是给业务边界画线，让代码模型跟着画。
  • Ubiquitous Language：业务和代码用同一套词汇，是 DDD 第一基石。
  • Domain 三分（Core / Supporting / Generic）决定资源投入策略。
  • Bounded Context：语言一致的边界，不预设物理实现（模块 / 服务 / 产品都可以）。
  • Context Map用 ACL / OHS / Conformist 等关系描述上下文间的协作模式。
  • Event Storming是发现 Bounded Context 的最有效工具，特别关注红色 hotspot。
  • 嵌入式领域 DDD 适配：Bounded Context 对应功能簇，统一语言对应 DBC/ARXML/IDL，ACL 对接异构 vendor。
小练习：拿你公司一个最熟悉的业务，能不能在一张纸上画出它的 Bounded Context 划分？每个 Context 能不能用一句话说清职责？如果不能，团队的业务认知本身就不统一——这是任何架构工作开始前必须先解决的问题。

06DDD 战术设计：Entity / VO / Aggregate / Repository（C++ 实现）

战术设计回答："在一个 Bounded Context 内部，代码到底怎么组织？"本章深入 Entity、Value Object、Aggregate、Repository、Domain Service、Domain Event、Factory 七大构建块，每个都给出地道的现代 C++ 实现。这是 DDD 与 C++ 类型系统结合最紧密、收益最大的部分。

6.1 Value Object（值对象）—— 最被低估的构建块

Value Object：用值来区分的对象。两个 VO 的所有属性相等 ⇒ 它们就是同一个值。不可变（immutable）、无身份（no identity）、可自由替换。

判断"是 Entity 还是 VO"的关键问题

问自己："两个长得一样的东西，是不是同一个？"

  • 两张 100 元人民币 → 同一个值（VO）
  • 两个叫"张三"的人 → 不同的人（Entity）
  • 两个 (经度=116.4, 纬度=39.9) 的坐标 → 同一个位置（VO）
  • 两个序列号都是 SN12345 的设备 → 同一台设备（Entity）

为什么 VO 重要？

❌ 没有 VO 的代码（原始类型病）

void transfer(
    long from_account,
    long to_account,
    double amount,
    std::string currency,
    long timestamp_ms);

// 调用：参数极易写错
transfer(1001, 1002, 100.50, "USD", 1700000000000);
// 写错参数顺序编译器不会报错
transfer(1002, 1001, 1700000000000, "USD", 100.50);
// double 表示金额？精度问题！

✅ 用 VO 的代码（类型即文档）

void transfer(
    AccountId from,
    AccountId to,
    Money amount,
    Timestamp at);

// 调用：编译器帮你校验
transfer(AccountId{1001}, AccountId{1002},
         Money::of(100, 50, Currency::USD),
         Timestamp::now());

// 写错顺序 → 编译错误
// 用 double 表示金额 → 编译错误（Money 不接受 double 构造）

C++ 实现 VO 的三个层次

层次 1：strong typedef（最轻量）
// 防止"两个都是 long，传错没人知道"
template<typename Tag>
struct StrongId {
    int64_t value;
    auto operator<=>(const StrongId&) const = default;
};

using AccountId  = StrongId<struct AccountIdTag>;
using CustomerId = StrongId<struct CustomerIdTag>;
using OrderId    = StrongId<struct OrderIdTag>;

void demo() {
    AccountId  a{1001};
    CustomerId c{1001};   // 同样是 1001
    // a == c;   // 编译错误！不同类型
    auto a2 = AccountId{a.value};  // OK
    if (a == a2) { /* OK */ }
}
层次 2：带不变量的 VO（推荐）
class Money {
    int64_t minor_units_;   // 分
    Currency currency_;
public:
    // 工厂方法，强制走校验
    static Money of(int64_t major, int minor, Currency c) {
        if (minor < 0 || minor >= 100)
            throw std::invalid_argument{"minor [0,100)"};
        return Money{major * 100 + minor, c};
    }

    int64_t minor_units() const noexcept { return minor_units_; }
    Currency currency() const noexcept { return currency_; }

    // 不可变 + 业务方法
    Money operator+(const Money& o) const {
        if (currency_ != o.currency_)
            throw CurrencyMismatchError{};
        return Money{minor_units_ + o.minor_units_, currency_};
    }
    Money operator*(int n) const { return Money{minor_units_ * n, currency_}; }

    // 默认 ==、!= （C++20 spaceship 可以自动生成）
    auto operator<=>(const Money&) const = default;

private:
    // 私有构造，强制走工厂
    Money(int64_t m, Currency c) : minor_units_(m), currency_(c) {}
};
关键 C++ 实践：
  • 构造函数私有，工厂方法校验不变量——保证不存在"非法状态的 Money"
  • 所有方法 const，返回新对象（不可变）
  • 用 C++20 operator<=> 让所有比较自动生成
  • 金额绝对不用 float/double——用分 / 厘等最小单位的整数
层次 3：带验证的复杂 VO
class EmailAddress {
    std::string value_;

"Make illegal states unrepresentable"（让非法状态不可表达）—— Yaron Minsky 的 OCaml 名言，在 C++ 同样适用。如果一个 EmailAddress 对象存在，它编译期就保证是合法格式。下游代码不需要再校验，也不可能拿到非法值。这是静态类型系统的真正力量。

6.2 Entity（实体）—— 有身份的对象

Entity：用身份（identity）区分的对象。两个 Entity 的所有属性相等也不一定是同一个。生命周期内属性可变，但身份不变。

典型 Entity 实现

class Customer {
    // 身份：不可变，构造时确定
    const CustomerId id_;

    // 可变状态
    EmailAddress email_;
    PersonName   name_;
    CustomerStatus status_ = CustomerStatus::Active;
    Money credit_limit_ = Money::zero(Currency::USD);

public:
    Customer(CustomerId id, EmailAddress e, PersonName n)
        : id_(id), email_(std::move(e)), name_(std::move(n)) {}

    // 标识访问（只读）
    CustomerId id() const noexcept { return id_; }

    // 业务方法（修改自身状态）
    void change_email(EmailAddress new_email) { email_ = std::move(new_email); }

    void deactivate() {
        if (status_ == CustomerStatus::Banned)
            throw InvalidTransitionError{"banned -> deactivated"};
        status_ = CustomerStatus::Inactive;
    }

    // Entity 的相等：只看身份
    bool operator==(const Customer& o) const noexcept {
        return id_ == o.id_;
    }
};
两个常见错误：
  1. Entity 用所有字段判等——会和 VO 混淆，且会让"改了 email 还是不是同一个客户"变成奇怪问题。Entity 只用 ID 判等。
  2. Entity 内部用 setter 暴露所有字段——退化为"贫血模型"（anemic model），业务规则散落在 service 里。Entity 应该有业务方法，setter 是反模式。

6.3 Aggregate（聚合）—— DDD 战术设计的核心

Aggregate：一组紧密关联的 Entity + VO 的集合，必须作为一个整体被保持一致。Aggregate 有一个根（Aggregate Root），对外只能通过根访问。

为什么需要 Aggregate？事务一致性边界

考虑一个 Order 业务规则："订单总金额必须等于所有 OrderLine 金额之和"。

  • 如果允许外部直接修改 OrderLine（绕过 Order） → 这个不变量随时可能被破坏
  • 如果所有对 OrderLine 的修改都必须通过 Order → Order 可以在每次修改后维护不变量

Aggregate 就是不变量的保护边界。

Aggregate 实战：Order

// === OrderLine：Order Aggregate 内部的 Entity（局部 ID）===
class OrderLine {
    LineId   id_;          // 在 Order 内唯一即可
    ProductId product_;
    int      quantity_;
    Money    unit_price_;
public:
    OrderLine(LineId id, ProductId p, int q, Money price)
        : id_(id), product_(p), quantity_(q), unit_price_(price) {
        if (q <= 0) throw std::invalid_argument{"qty>0"};
    }
    LineId   id() const { return id_; }
    ProductId product() const { return product_; }
    int      quantity() const { return quantity_; }
    Money    line_total() const { return unit_price_ * quantity_; }

    // 只能由 Order 调用
    void change_quantity(int q) {
        if (q <= 0) throw std::invalid_argument{"qty>0"};
        quantity_ = q;
    }
};

// === Order：Aggregate Root ===
class Order {
    OrderId          id_;
    CustomerId       customer_;
    OrderStatus      status_ = OrderStatus::Draft;
    std::vector<OrderLine> lines_;
    Money            total_ = Money::zero(Currency::USD);

    // 关键：保护不变量
    void recalculate_total() {
        total_ = std::accumulate(lines_.begin(), lines_.end(),
            Money::zero(Currency::USD),
            [](Money acc, const auto& l) { return acc + l.line_total(); });
    }

public:
    Order(OrderId id, CustomerId c) : id_(id), customer_(c) {}

    OrderId id() const { return id_; }
    OrderStatus status() const { return status_; }
    Money total() const { return total_; }

    // 不要返回可修改的引用！
    const std::vector<OrderLine>& lines() const { return lines_; }

    // === 业务操作（自动维护不变量）===
    void add_line(ProductId p, int qty, Money price) {
        if (status_ != OrderStatus::Draft)
            throw InvalidOperationError{"only draft order accepts new lines"};
        lines_.emplace_back(LineId::next(), p, qty, price);
        recalculate_total();
    }

    void change_line_qty(LineId line_id, int new_qty) {
        if (status_ != OrderStatus::Draft) throw InvalidOperationError{};
        auto it = std::find_if(lines_.begin(), lines_.end(),
            [&](const auto& l) { return l.id() == line_id; });
        if (it == lines_.end()) throw NotFoundError{};
        it->change_quantity(new_qty);
        recalculate_total();
    }

    void submit() {
        if (lines_.empty()) throw InvalidStateError{"empty order"};
        if (status_ != OrderStatus::Draft) throw InvalidTransitionError{};
        status_ = OrderStatus::Submitted;
    }
};

Aggregate 设计的 5 条规则

① 根是唯一入口

外部代码只持有 Aggregate Root 的引用，不能直接持有内部 Entity（如 OrderLine）。访问内部都通过 Root 的方法。

② 一个事务只修改一个 Aggregate

需要修改多个？用 Domain Event 异步协调。强行跨 Aggregate 加事务 = 性能瓶颈 + 死锁。

③ 跨 Aggregate 只引用 ID

Order 引用 Customer 时存 CustomerId，不存 Customer*。这避免了 Aggregate 边界的模糊。

④ Aggregate 要小

越小越好。大 Aggregate = 并发冲突 + 慢加载。能拆就拆。

⑤ 最终一致性优于强一致性

跨 Aggregate 的一致性默认就是最终一致，通过 Domain Event 实现。

额外：保护不变量

Aggregate 内部任何状态变化后，不变量必须满足。不能存在"中间不一致状态"对外可见。

新人最常犯的错：设计了一个超大 Aggregate（如 Customer 聚合包含所有 Order、Address、PaymentMethod）。看起来"对象关系完整"，实际是性能灾难——加载一个 Customer 要从数据库拉几万行。正确做法：Customer 是一个 Aggregate，Order 是另一个 Aggregate，互相引用 ID。

6.4 Repository（仓储）—— 业务核心和持久化的边界

Repository：以 Aggregate 为单位的"集合幻觉"。让 Domain 层用"找 / 存 / 删"的方式操作 Aggregate，不需要关心数据库存在。

关键设计原则

  • 每个 Aggregate 类型有且仅有一个 Repository
  • Repository 的接口定义在 Domain 层（依赖倒置！）
  • Repository 的实现在 Infrastructure 层
  • 查询方法返回完整的 Aggregate，不返回字段集合

接口定义（Domain 层）

// === domain/repositories/i_order_repository.hpp ===
class IOrderRepository {
public:
    virtual ~IOrderRepository() = default;
    virtual std::optional<Order> find_by_id(OrderId id) = 0;
    virtual std::vector<Order> find_by_customer(CustomerId c) = 0;
    virtual void save(const Order& order) = 0;
    virtual void remove(OrderId id) = 0;
};

实现（Infrastructure 层）

// === infrastructure/postgres_order_repository.cpp ===
class PostgresOrderRepository : public IOrderRepository {
    pqxx::connection& conn_;

    Order map_to_aggregate(pqxx::row order_row, pqxx::result line_rows) {
        Order o{OrderId{order_row["id"].as<int64_t>()},
                CustomerId{order_row["customer_id"].as<int64_t>()}};
        for (const auto& lr : line_rows) {
            o.add_line(ProductId{lr["product_id"].as<int64_t>()},
                       lr["qty"].as<int>(),
                       Money::of(lr["price_minor"].as<int64_t>() / 100,
                                 lr["price_minor"].as<int64_t>() % 100,
                                 Currency{lr["currency"].as<std::string>()}));
        }
        return o;
    }

public:
    explicit PostgresOrderRepository(pqxx::connection& c) : conn_(c) {}

    std::optional<Order> find_by_id(OrderId id) override {
        pqxx::work tx{conn_};
        auto order_rows = tx.exec_params(
            "SELECT * FROM orders WHERE id=$1", id.value);
        if (order_rows.empty()) return std::nullopt;
        auto line_rows = tx.exec_params(
            "SELECT * FROM order_lines WHERE order_id=$1", id.value);
        return map_to_aggregate(order_rows[0], line_rows);
    }

    void save(const Order& o) override {
        pqxx::work tx{conn_};
        tx.exec_params("INSERT INTO orders ... ON CONFLICT (id) DO UPDATE SET ...",
                       o.id().value, /* ... */);
        tx.exec_params("DELETE FROM order_lines WHERE order_id=$1", o.id().value);
        for (const auto& l : o.lines()) {
            tx.exec_params("INSERT INTO order_lines ...", /* ... */);
        }
        tx.commit();
    }
    // ... 其他方法
};
Repository 的常见反模式：暴露 find_all() : vector<Order> 类的"取所有"方法——加载千万行直接 OOM。Repository 应该只暴露业务真正需要的查询，分页、按条件过滤、统计等不属于 Repository（属于 CQRS 的 Read Side）。

6.5 Domain Service（领域服务）

Domain Service：当某个业务逻辑不自然地属于任何单一 Entity 或 VO，但又是核心业务规则时，把它放进 Domain Service。

判断标准

当你发现一段逻辑：

  • 涉及多个 Aggregate（跨实体业务）
  • 是核心业务规则（不是基础设施操作）
  • 放进任何一个 Entity 都"不合适"（不属于它）

→ 它就是 Domain Service。

示例：转账

// "转账" 是 Account 之间的业务规则，不属于任何单个 Account
class TransferService {
    IAccountRepository& accounts_;
    IFeePolicy&         fees_;
public:
    TransferService(IAccountRepository& a, IFeePolicy& f)
        : accounts_(a), fees_(f) {}

    void transfer(AccountId from_id, AccountId to_id, Money amount) {
        auto from = accounts_.find_by_id(from_id).value();
        auto to   = accounts_.find_by_id(to_id).value();
        auto fee  = fees_.calculate(amount, from.country(), to.country());

        from.withdraw(amount + fee);   // Account 的业务方法
        to.deposit(amount);

        accounts_.save(from);
        accounts_.save(to);
    }
};
Domain Service vs Application Service 的区分：
  • Domain Service：纯业务逻辑，不关心事务、不关心日志
  • Application Service（也叫 Use Case）：编排事务、调用 Domain Service、发事件、处理异常
新手区分困难，可以先合并；到一定规模再拆。

6.6 Domain Event（领域事件）

Domain Event：领域中发生的、值得被记录和广播的事情。用过去时命名：OrderPlaced, PaymentReceived, CustomerDeactivated。

为什么需要 Domain Event？

  • 跨 Aggregate 解耦：Order 不需要直接调用 Inventory 减库存，发个 OrderPlaced 事件，Inventory 自己订阅
  • 跨 Bounded Context 通信：销售上下文发 OrderPlaced，物流上下文订阅
  • 审计 / 历史：事件序列是最完整的审计日志
  • 读模型构建：CQRS 的 Read Side 用事件构建

C++ 实现

// === domain/events/order_events.hpp ===
struct OrderPlaced {
    OrderId order_id;
    CustomerId customer_id;
    Money total;
    Timestamp at;
};
struct OrderPaid {
    OrderId order_id;
    Money amount;
    PaymentMethod method;
};
struct OrderCancelled {
    OrderId order_id;
    std::string reason;
};

using OrderEvent = std::variant<OrderPlaced, OrderPaid, OrderCancelled>;

// === domain/order.hpp（修改后）===
class Order {
    // ... 之前的字段 ...
    std::vector<OrderEvent> pending_events_;   // 待发布事件

public:
    void submit() {
        if (status_ != OrderStatus::Draft) throw InvalidTransitionError{};
        status_ = OrderStatus::Submitted;
        pending_events_.push_back(OrderPlaced{id_, customer_, total_, Timestamp::now()});
    }

    std::vector<OrderEvent> collect_events() {
        std::vector<OrderEvent> out;
        std::swap(out, pending_events_);
        return out;
    }
};

// === application/place_order_use_case.cpp ===
void PlaceOrderUseCase::execute(const PlaceOrderCommand& cmd) {
    Order o{OrderId::next(), cmd.customer};
    for (const auto& l : cmd.lines) o.add_line(l.product, l.qty, l.price);
    o.submit();

    // 一个事务里：保存 Aggregate + 发布事件（Outbox 模式见第 7 章）
    repo_.save(o);
    for (const auto& e : o.collect_events()) events_.publish(e);
}
Outbox 模式：为保证"Aggregate 保存"和"事件发布"的原子性，把事件先写入同一个数据库的 outbox 表（同一事务），再由后台进程异步推送到 broker。这样即使发布失败，事件也不会丢。这是事件驱动架构的标配。

6.7 Factory（工厂）

当 Aggregate 的创建逻辑复杂（需要校验、需要查询、需要多步组装），用 Factory 封装。

class OrderFactory {
    IProductCatalog& catalog_;
    IPricingService& pricing_;
public:
    Order create(CustomerId customer, const std::vector<CartItem>& items) {
        Order o{OrderId::next(), customer};
        for (const auto& item : items) {
            auto p = catalog_.find(item.product_id).value();
            if (!p.is_available()) throw ProductUnavailableError{};
            auto price = pricing_.calculate_for(customer, p, item.qty);
            o.add_line(p.id(), item.qty, price);
        }
        return o;
    }
};

简单创建（构造函数能直接搞定）不需要 Factory。强行用 Factory 包装是过度设计。

6.9 嵌入式 / 安全关键系统中的 DDD 战术适配

DDD 战术构件在嵌入式领域的映射与变体：
  • Value Object 天然契合 → 物理量（Velocity, Temperature, Voltage）必须用 VO 而不是裸 double——避免单位错误（如把摄氏度当华氏度）。用 strong typedef + 编译期单位检查（如 mp-units 库）。
  • Entity 通常对应物理设备 → ECU、Sensor、Actuator 都是 Entity（用 serial number / VIN 作为 ID）。
  • Aggregate 通常对应控制循环 → 一个 Aggregate = 一个独立的实时控制环 + 它的传感器输入 / 执行器输出。事务一致性边界 = 控制周期。
  • Repository 通常退化为 RAM-only → 嵌入式系统状态主要在内存，"持久化"是断电保存（EEPROM / Flash）+ 启动恢复。Repository 接口仍有价值——隔离 NVM 实现细节。
  • Domain Event 必须考虑 QoS → 不是所有事件都能"最终一致"。安全相关事件（如"刹车踩下"）必须 hard real-time，不能进消息队列；统计事件可以走异步。
  • 禁用 std::optional / std::variant 等 → MISRA C++ 通常禁用动态分配相关、限制异常，DDD 实现要做相应调整（用 etl::optional、etl::variant 这些 ETL 库）。
代码示例：用 mp-units 库防单位错误 
using namespace mp_units;
using namespace mp_units::si::unit_symbols;

quantity speed = 60 * km / h;
quantity time  = 2 * h;
quantity dist  = speed * time;   // 120 km，编译期保证单位

void set_brake_pressure(quantity<isq::pressure[bar]> p);
set_brake_pressure(5 * bar);    // OK
// set_brake_pressure(60 * km/h); // 编译错误：单位不匹配

6.10 本章小结

  • Value Object：用值区分、不可变、零身份。"让非法状态不可表达"是 VO 的最大威力。
  • Entity：用身份区分，只用 ID 判等。避免贫血模型。
  • Aggregate：一致性边界。5 条规则：根入口、单事务单 Aggregate、跨 Aggregate 引用 ID、小、最终一致。
  • Repository：以 Aggregate 为单位，接口在 Domain 层、实现在 Infrastructure 层。
  • Domain Service：跨 Entity 的纯业务规则。
  • Domain Event：解耦 Aggregate / Context，配合 Outbox 模式保证可靠发布。
  • C++ 落地核心：用 CMake target 边界强制依赖方向，让架构原则在编译期就被守护。
  • 嵌入式适配：物理量必用 VO + 单位库；Aggregate = 控制循环；事件 QoS 必须分级。
小练习：把你最熟悉的一个业务模块，按 Aggregate + VO + Domain Service 重构第一个类。你会发现：
  • 很多 "double / int / string" 都该升级成 VO
  • 很多 service 里的业务逻辑该搬回 Entity
  • 很多 "if-else 链" 该拆成 Policy / Strategy
重构一个模块的时间 < 重新理解一遍业务的时间。这就是 DDD 的真正价值——代码本身就是业务文档。

07ADR 架构决策记录：让决策可追溯、可演进

没记录的决策 = 没做过的决策。ADR (Architecture Decision Record) 是架构师必备的基础工具，把每个重要决策的"背景、选项、决定、后果"以轻量化文档的方式记录下来。本章给出 Nygard 与 MADR 两种主流模板、ADR 流程实操、真实完整 ADR 范例，以及 ADR 在团队和 C++ 项目中的落地方式。

7.1 为什么必须写 ADR

不写 ADR 的真实代价

  • 6 个月后，新人问"为什么用 protobuf 不用 JSON？"——没人答得出
  • 1 年后，团队推翻这个决策，发现是有原因的，已经晚了
  • 2 年后，同一个讨论又开了 5 次会，每次都重新发明历史
  • 架构腐化的最大原因不是技术差，是决策上下文丢失

写 ADR 的收益

  • 决策可追溯——任何时候能回答"我们为什么这样做"
  • 新人 onboarding 提速——读 ADR 比读代码快 10 倍
  • 决策可演进——明确"什么条件下应该重新评估"
  • 跨团队对齐——把口头共识固化为白纸黑字
  • 架构评审有据可依，避免拍脑袋
"Architecture is about the important stuff. Whatever that is." — Ralph Johnson
而 ADR 就是回答"什么是重要的、为什么重要"的载体。

7.3 ADR 完整真实范例

ADR-012: 在车端 Collection Engine 中引入异步规则引擎

Status: Accepted · 2026-03-22 · Decided by: 平台架构组、采集团队

Context

当前车端 Collection Engine 使用同步规则匹配：每个信号到达时，遍历所有规则做表达式求值。随着规则数从 50 增加到 500，P99 单信号处理延迟从 5µs 上升到 80µs，已经接近实时性下限（100µs）。

profiler 显示 70% 时间花在规则的字符串表达式解析上（每次重新解析）。规则下发频率低（小时级），但匹配频率高（kHz 级）。

Decision Drivers
  • P99 单信号处理延迟必须 ≤ 30µs（硬要求）
  • 支持规则热更新（不能重启进程）
  • 团队对 LLVM JIT、libcurl、AST 解析的熟悉度
  • 二进制体积约束（嵌入式 Flash 限制 200MB）
Considered Options
  1. 简单缓存解析后 AST：解析一次，每次匹配复用 AST
  2. 规则编译为字节码 + 解释器：自研一个简单字节码 VM
  3. LLVM JIT 编译规则：每条规则 JIT 成 native code
  4. 规则用 Lua 写 + LuaJIT：业务侧写 Lua
Decision Outcome

选项 1（AST 缓存）+ 后续可演进到选项 2。

Justification
选项性能复杂度体积热更新
1. AST 缓存预计 ~10µs ✅低 ✅+0 ✅容易 ✅
2. 自研字节码~5µs ✅中+200KB容易 ✅
3. LLVM JIT~1µs ✅高 ❌+50MB ❌复杂
4. LuaJIT~5µs ✅中+500KB容易 ✅

选项 3 的二进制体积超出 Flash 约束；选项 4 引入新语言增加业务方学习成本；选项 1 完全满足性能要求，且实现简单；如果未来规则增长到 5000+，可演进到选项 2。

Positive Consequences
  • P99 处理延迟预计降到 8-12µs，满足 hard real-time 要求
  • 实现工作量小：1 人 1 周可完成
  • 不引入新依赖，二进制体积不变
Negative Consequences
  • AST 缓存有内存开销：500 条规则约 ~2MB；可接受
  • 规则更新时需要清空 / 重建缓存（小延迟尖峰）
  • 规则数量再增长 10× 时需要重新评估（触发 ADR）
Follow-ups
  • 规则数 > 2000 时触发性能 retest，可能升级到选项 2
  • 规则更新延迟 > 50ms 时触发分析
Links
  • Related: ADR-009 (规则下发 channel)
  • Benchmark report: docs/perf/rule-engine-bench.md
  • Supersedes: 无
来自RPC 管理代理的真实 ADR 重建——每 remote-unit 独立 Dispatcher 线程

这是一个在真实电信项目中可以（也应该）写下来的 ADR。它记录了一个影响整个并发模型的关键决策。

ADR-T01

为每个 remote-unit 分配独立的 Dispatcher 串行线程

Accepted
状态：隐式决策，补充记录 领域：应用 OAM / 网元管理
Context · 上下文

platform agent 需要同时管理 N 个 remote-unit（典型值 1–8 个），每个 remote-unit 接收来自 Upstream Controller 的 Mgmt Task（创建载波、配置参数、升级软件等）。这些 Task 需要转化为按顺序执行的 RPC 序列发往 remote-unit。

关键约束：remote-unit 是一个顺序状态机，同一时刻只能处理一个 RPC 操作；RPC 序列有严格的时序依赖（必须先创建 Processing Element，再创建 ArrayCarrier）。

Considered Options · 候选方案
  1. 全局共享线程池（N 个线程服务所有 RU）
  2. 每个 RU 独立 Dispatcher 线程（串行队列）
  3. 单线程事件循环（类 Reactor，异步回调）
Decision · 决策
选择方案 2：每个 remote-unit 独立分配一个 Dispatcher 线程，内部串行执行 Scenario 队列。
Justification · 论证（ATAM 影响矩阵）
质量属性方案 1（线程池）方案 2（独立线程，选）方案 3（事件循环）
时序正确性需要复杂锁天然串行 ✅需要状态机
RU 间隔离共享状态风险完全隔离 ✅隔离 ✅
资源占用低（共享）中（N 线程）低
代码复杂度高（竞态处理）低（无锁业务代码）✅高（回调嵌套）
优先级支持需要特殊队列优先级队列天然支持 ✅需要特殊队列

典型 RU 数量 ≤ 8，N 个线程的资源成本完全可接受（每线程约 256KB 栈）。方案 2 让告警等紧急操作可以通过 addPriorityScenario() 插队，而不需要复杂的优先级调度逻辑。

Positive Consequences · 正面后果
  • 业务代码（Scenario 类）完全不需要加锁——串行执行天然保证了 RU 状态的一致性
  • 一个 RU 的慢操作（如软件升级）不阻塞其他 RU 的正常操作
  • 优先级告警可以通过 addPriorityScenario() 插到队首，实现简单
Negative Consequences · 负面后果
  • 需要 EDispatcherState 枚举维护线程状态（Running / Stopped / Idle），且此状态应该是 std::atomic——当前实现有潜在竞态
  • 线程 detach() 模式在 Dispatcher 销毁时有 UAF（Use-After-Free）风险，应改为 join()
Follow-ups / Tech Debt · 后续行动与技术债
  • 将 EDispatcherState 改为 std::atomic<EDispatcherState>
  • 将 thread_.detach() 改为受控的 join()，配合 stop token

注意这条 ADR 的价值：它明确指出了两个已知的技术债（state 非原子、detach UAF）并写进 Follow-ups——这是"谨慎 + 故意"类技术债的标准处理方式：承认它、记录它、计划偿还它，而不是假装不存在。

ADR 状态机

               ▼
          ┌──────────┐
          │ Accepted │  <-- 团队同意，正式生效

关键原则：ADR 一旦 Accepted，就永不删除。如果决策变了，写一个新 ADR，把旧的标记为 "Superseded by ADR-XX"。历史是资产——未来人需要理解"为什么我们曾经这样做"。

ADR 在团队中的流程

  1. 识别决策：任何"影响半径 > 1 个模块、回退成本 > 1 周"的决策都该写 ADR
  2. 起草：决策提出者起草 ADR（Status: Proposed），放在 docs/adr/ 目录
  3. 评审：提 PR，相关干系人 review，讨论沉淀到 PR 评论里
  4. 决策：达成共识，Status 改为 Accepted；如果有分歧，决策者裁定
  5. 合入：PR 合入，ADR 编号锁定（不再变动 ID）
  6. 实施：相关代码 commit 引用 ADR 编号
  7. 回顾：每季度回顾，看哪些 ADR 应该升级或废弃

7.5 哪些决策值得写 ADR？

✅ 必须写

  • 架构风格选型（分层 / 六边形 / 微服务）
  • 数据库选型（PostgreSQL vs MongoDB）
  • 消息队列选型（Kafka vs NATS）
  • 通信协议（gRPC vs REST vs Thrift）
  • 构建系统（CMake vs Bazel）
  • 依赖管理（Conan vs vcpkg）
  • 关键的 C++ 特性使用决策（要不要用异常？要不要用 RTTI？）
  • 测试策略（金字塔 / 钻石）
  • API 版本策略
  • 错误处理范式（异常 vs Result<T,E>）

⚠ 不需要写

  • 具体函数实现细节
  • 临时性的代码风格调整
  • 纯局部的算法选择（除非影响性能 KPI）
  • 已经被组织标准强制的（如公司用 GitLab，不用为这写 ADR）

原则："如果一个新人 6 个月后看到这段代码会问'为什么'，就值得写 ADR"。

7.6 反 ADR 模式

反模式描述解药
"ADR 大坟场" 写了 100 个 ADR，没人读 把核心 ADR 放新人 onboarding 必读清单；季度回顾
"事后补" 决策做完三个月才补 ADR，没人记得当时考虑了什么 把 ADR 列为决策的 DoD（Definition of Done）
"水货 ADR" 只写"我们决定用 X"，没有 Context 和 Consequences PR review 拒绝缺少 Context 的 ADR
"删除 ADR" 觉得旧 ADR 没用就删了 ADR 只 Supersede，不删除
"过度细化" 每个小决策都写 ADR，团队烦透了 设定门槛：影响半径 + 回退成本

7.7 嵌入式 / 安全关键系统中的 ADR

安全关键系统的 ADR 有额外要求：
  • 合规追踪：每个 ADR 必须标注是否影响 ASIL 等级、是否涉及安全机制；这些 ADR 是安全审计的输入
  • Tool Qualification：选用编译器、静态分析工具的 ADR 必须包含 tool qualification 证据（ISO 26262-8 Clause 11）
  • 变更影响分析：每个 Superseded 的 ADR 必须做 Change Impact Analysis (CIA)，证明安全性不受影响
  • 追溯到需求：ADR 应链接到 Safety Requirements（如 SaR-0042：刹车系统必须冗余）
  • 语言选择：用 C++ 子集（如 MISRA C++ 2008 / AUTOSAR C++14）必须有 ADR 明确，并约束工具链
汽车行业的 ADR 模板通常包含额外字段：Safety Impact, ASIL Level, Linked Hazards, Verification Method。

7.8 本章小结

  • ADR 是架构师的基础工具，没有 ADR 的架构师不是合格架构师
  • 选 Nygard 模板（5 分钟轻量）作为日常默认，MADR 模板用于重大决策
  • ADR 只 Supersede，不删除——历史是资产
  • ADR 必须包含 Context / Decision / Consequences 三段；前者解释"为什么"，后者预测"会有什么影响"
  • 放在 docs/adr/，提 PR review，跟代码一起版本化
  • 识别"该写 ADR"的标准：影响半径 + 回退成本
  • 安全关键系统的 ADR 必须额外承载合规追溯信息

08API 设计哲学：稳定、可演进、不强迫

架构师 50% 的工作是在边界上做决策——而 API 就是最重要的边界。一个好 API 让 10 年后的人还能用、不痛苦；一个坏 API 让团队 3 个月就想推翻重写。本章给出 API 设计的七大原则、错误处理的三种范式对比（异常 vs Result vs 返回码）、同步 vs 异步选型、C++ 专属 API 设计专题（Pimpl / Builder / 不强迫继承）以及API 版本管理策略。

8.1 七大设计原则（按重要性排序）

① 难误用 (Hard to misuse)

API 应该让正确使用容易，让错误使用编译失败或显著困难。

Joshua Bloch："好的 API 应该像下楼梯——你自然就走对了。"

反例：参数顺序 transfer(double amount, long from, long to) ——很容易把 to/from 搞反。

正例：transfer(AccountId from, AccountId to, Money amount) ——强类型阻止错误。

② 最小完备 (Minimal yet complete)

用最少的方法覆盖所有用例。每多一个方法 = 多一份维护负担。

反例：vector 提供 50 个方法。

正例：std::span 只有十几个方法但覆盖了"只读视图"几乎所有需求。

③ 正交性 (Orthogonality)

不同方法做不同事，没有功能重叠。

反例：既有 add() 又有 insert(end)，做的是同一件事。

正例：STL 的 push_back vs insert 各有明确职责。

④ 一致性 (Consistency)

同一概念，同一名字，同一参数顺序。整套 API 应该感觉是"同一个人写的"。

反例：get_user(id) / fetchPost(id) / loadComment(commentId) ——三个不同动词、不同参数名做同一件事。

⑤ 稳定性 (Stability)

公开 API 一旦发布就难以改。比起加错，宁可晚加。

"It's easier to add than to remove."

⑥ 可演进性 (Evolvability)

预留扩展点：版本号、option struct、tag dispatch、变参模板。

新增字段不应破坏旧客户端（向后兼容）。

⑦ 自描述 (Self-documenting)

API 看名字就知道做什么。不需要查文档。

反例：setFlag(2)；正例：setRetryPolicy(RetryPolicy::ExponentialBackoff)。

额外（C++ 特有）

不强迫继承（prefer composition）、不强迫异常（提供 noexcept 版本）、不强迫动态分配（提供 allocator 钩子）。

8.2 错误处理的三种范式

C++ 架构师必须在项目级别决策一个错误处理范式，不能混用（混用 = 调用方崩溃）。

① 异常 (Exception)

Order place_order(const Cmd& c) {
  if (!valid) throw InvalidArg{};
  return Order{...};
}

优点：

  • 主流程代码干净
  • 不会被忽略
  • 跨多层自动传播

缺点：

  • 有性能开销（cold path 几乎零，hot path 不可控）
  • 嵌入式 / 实时系统通常禁用
  • 跨 ABI / 跨语言不能用

适用：业务系统、用户级代码、cold path

② Result<T, E>

tl::expected<Order, Error>
place_order(const Cmd& c) {
  if (!valid)
    return tl::unexpected{Err{InvalidArg}};
  return Order{...};
}

优点：

  • 显式：编译期强制处理错误
  • 零成本：没有异常机制开销
  • 跨 ABI 安全
  • C++23 已有 std::expected

缺点：

  • 主流程被错误处理打断
  • 跨多层需要逐层 propagate（可用 monadic 链式缓解）

适用：库 API、跨进程边界、需要确定性的代码

③ 返回码 + Out 参数

int place_order(const Cmd& c, Order* out) {
  if (!valid) return ERR_INVALID;
  *out = Order{...};
  return OK;
}

优点：

  • C 兼容、跨语言友好
  • 最简单

缺点：

  • 容易忘记检查返回值
  • 主流程嘈杂
  • 错误信息少（只有错误码）

适用：C API、OS 接口、跨语言 ABI

项目级决策建议：
  • 纯 C++ 业务系统：用异常。主流程清晰最重要。
  • 跨 ABI 边界（库 / 插件）：用 Result<T,E>（C++23 std::expected 或 tl::expected）。
  • 嵌入式 / 实时：用 Result<T,E>（异常被禁用）。
  • C ABI 暴露：用返回码 + out 参数。
一个项目内只选一种，写进 ADR。

8.3 同步 vs 异步 API

方式形态优点缺点
同步阻塞 Result do_x() 简单、易理解 占用调用线程
同步 + Callback void do_x(callback) 无阻塞 回调地狱
std::future future<T> do_x() 标准、可组合 future 表现力弱（无 then）
协程 (C++20) task<T> do_x() 同步写法，异步执行 学习成本、库生态待成熟
消息 / Event 发个 Event，等订阅 松耦合 无返回值、不易追踪
2026 年的现代 C++ API 推荐：同步阻塞为默认；如果调用方需要异步，用协程包一层（co_await sync_call()）。不要为每个方法都提供同步 + 异步两套——API 翻倍，bug 翻倍。

8.4 C++ 库 API 设计专题

① Pimpl 模式（隔离实现）

问题：库的头文件暴露了所有私有成员，导致：

  • 用户每次编译都要包含一堆实现头
  • 改私有成员就破坏 ABI
  • 编译时间爆炸

解药：Pimpl (Pointer to Implementation)。

// === order.hpp（公开头，干净）===
class Order {
public:
    Order(OrderId, CustomerId);
    ~Order();                          // 必须 user-defined（因为 unique_ptr<Impl> 的析构）
    Order(Order&&) noexcept;
    Order& operator=(Order&&) noexcept;

    OrderId id() const noexcept;
    void add_line(ProductId, int, Money);
    Money total() const;
private:
    class Impl;
    std::unique_ptr<Impl> pimpl_;
};

// === order.cpp（实现头，私有）===
class Order::Impl {
public:
    OrderId id;
    CustomerId customer;
    std::vector<OrderLine> lines;
    Money total;
    // 想加什么字段就加什么，ABI 不变
};

Order::Order(OrderId i, CustomerId c)
    : pimpl_(std::make_unique<Impl>()) {
    pimpl_->id = i;
    pimpl_->customer = c;
}
Order::~Order() = default;                 // 必须在 .cpp 里定义
Order::Order(Order&&) noexcept = default;
Order& Order::operator=(Order&&) noexcept = default;

OrderId Order::id() const noexcept { return pimpl_->id; }
// ...
Pimpl 的代价：
  • 每次方法调用多一层间接（性能 hot path 慎用）
  • 必须堆分配（嵌入式可用 PMR allocator 缓解）
  • 调试时多看一层
适用场景：跨 SO/DLL 边界的库 API，编译时间敏感的大型项目。

② Builder 模式（参数过多时）

当配置选项 > 4 个，构造函数变成参数地狱。用 Builder。

// === 反例：参数地狱 ===
HttpClient(std::string url, std::chrono::milliseconds timeout,
           int retries, bool verify_tls,
           std::optional<Proxy> proxy, size_t max_redirects,
           bool follow_redirects, std::string user_agent);

// 使用：参数顺序错了不易发现
HttpClient c{"https://api", 5000ms, 3, true, std::nullopt, 5, true, "agent"};

// === 正例：Builder ===
auto client = HttpClient::Builder{}
    .url("https://api")
    .timeout(5s)
    .retries(3)
    .verify_tls(true)
    .max_redirects(5)
    .build();

更现代的替代：Designated Initializers (C++20)

struct HttpClientConfig {
    std::string url;
    std::chrono::milliseconds timeout = 30s;
    int retries = 3;
    bool verify_tls = true;
    size_t max_redirects = 5;
};

HttpClient c{HttpClientConfig{
    .url = "https://api",
    .timeout = 5s,
    .retries = 3,
}};   // 其他用默认值

③ 不强迫继承（Type Erasure / std::function）

反模式：要求用户继承基类才能使用 API。

// === 不好：强制继承 ===
class IEventHandler {
public:
    virtual ~IEventHandler() = default;
    virtual void on_event(const Event&) = 0;
};
void register_handler(IEventHandler*);   // 用户必须建一个类继承

// === 好：函数对象 ===
using EventHandler = std::function<void(const Event&)>;
void register_handler(EventHandler);

// 用户：
register_handler([](const Event& e) { log(e); });   // 一行搞定
现代 C++ 倾向：用 std::function / 模板 / Concepts 替代抽象基类。除非：
  • 真的需要类有多态状态（不止一个方法）
  • 跨 SO/DLL 边界（virtual table 是稳定 ABI 之一）
  • 对象生命周期复杂，需要继承体系管理

④ Span / View / Range（零拷贝传参）

// === 不好：强迫拷贝、强迫特定容器 ===
void process(const std::vector<int>& data);
// 调用方有 array？必须先转 vector

// === 好：用 span（C++20）===
void process(std::span<const int> data);
// 调用方可传 vector / array / 裸指针+长度

// === 字符串：用 string_view ===
void log(std::string_view msg);   // 不拷贝

8.5 API 版本管理

SemVer (Semantic Versioning)

版本号 MAJOR.MINOR.PATCH：

MAJOR不兼容的 API 变更（破坏性）客户必须改代码
MINOR新增功能、向后兼容客户无需改代码
PATCHbug 修复、向后兼容客户无需改代码

向后兼容性的具体保证

✅ MINOR 可以做

  • 新增函数
  • 新增类
  • 给函数加有默认值的新参数
  • 放宽前置条件（接受更多输入）
  • 加强后置条件（保证更多）

❌ MINOR 不能做（需要 MAJOR）

  • 删除函数 / 类
  • 改函数签名（参数顺序、类型）
  • 改返回类型
  • 抛新的异常类型
  • 收紧前置条件（拒绝以前接受的输入）
  • 放宽后置条件（保证变少）

Deprecation 流程

// v1.5：标记为 deprecated（仍可用）
[[deprecated("Use place_order_v2 instead")]]
void place_order(const OldCommand&);

void place_order_v2(const NewCommand&);

// v2.0：删除老的（MAJOR 版本变更）
void place_order_v2(const NewCommand&);
Deprecation 的两个原则：
  1. 给客户至少一个完整 MAJOR 版本周期的过渡时间
  2. 提供清晰的迁移路径："Use X instead, with Y migration guide"
直接删除 = 失去客户信任。

8.6 API 设计的反模式

反模式例子问题
布尔参数地狱 save(data, true, false, true) 调用时完全不知道是什么意思 → 用 enum
输出参数 void parse(string s, Result* out) 容易忘记初始化 → 用返回值
静默失败 失败时返回 null / 默认值 用户不知道出错 → 用 optional / Result
过度暴露内部 返回 vector& 让调用方乱改 破坏不变量 → 返回 const& 或 span
God Object 一个类 100 个方法 难学难用 → 拆成多个职责类
命名不一致 getUser / fetch_post / loadComment 违反一致性 → 团队约定单一动词
魔法字符串 set("policy", "round_robin") 不可发现 / 拼写错 → 用 enum / 强类型

8.7 嵌入式 / 安全关键系统 API 设计

嵌入式系统 API 的额外约束：
  • 所有 API 必须 noexcept（异常被禁用） → 错误处理必须用 Result<T,E> 或 status code
  • 禁止动态分配（new/malloc） → 提供 PMR allocator 或预分配 buffer 版本
  • 避免 RTTI 和 dynamic_cast → 用 std::variant / type tag
  • 避免 virtual（vtable 占用 + 间接调用） → 用 CRTP / templates
  • 必须 deterministic → 不依赖任何"可能慢"的标准库（如 std::regex、std::function）
  • 必须可静态分析 → 符合 MISRA / AUTOSAR C++14 规则，便于 LDRA / Coverity 分析
  • 线程安全文档化 → 每个方法必须明确："thread-safe" / "single-threaded" / "main loop only"
  • 实时性文档化 → 每个方法必须明确 WCET（Worst-Case Execution Time）
例如，AUTOSAR Adaptive 的 API 文档每个方法都有： 
/// @brief Subscribe to event
/// @threadsafe yes
/// @reentrant yes
/// @synchronous no  (returns immediately, callback async)
/// @realtime soft (typical < 100us)

8.8 本章小结

  • API 设计七大原则：难误用 / 最小完备 / 正交 / 一致 / 稳定 / 可演进 / 自描述
  • 错误处理三选一：异常 / Result<T,E> / 返回码，在 ADR 里明确
  • 同步默认，异步用协程包一层；不要为每个方法做双版本
  • C++ 专题：Pimpl（隔离实现）、Builder / Designated Init（参数过多）、不强迫继承（用 function/concept）、span/view（零拷贝）
  • 版本管理用 SemVer，MINOR 只能加，MAJOR 才能改
  • Deprecation 给至少一个 MAJOR 周期 + 清晰迁移路径
  • 嵌入式 API 额外约束：noexcept、无动态分配、无 virtual、deterministic、文档化 WCET 和线程安全
小练习：找你正在维护的一个 C++ 库的主要 public API，对照七大原则评分（每条 0-2 分）。低于 8/14 → 这个 API 需要重新设计。大多数 C++ 库 API 的真实得分 < 6/14——这就是 C++ 库总被人吐槽难用的根因。

09架构演进与技术债：演进式架构、Strangler Fig、债务管理

没有任何架构能"一次设计、永远不变"。架构师真正的价值不是画好一张图，而是守护架构在 5 年内的健康演进。本章覆盖：演进式架构的核心理念、Fitness Function（架构适应性函数）、Strangler Fig 重构模式、技术债的识别 / 量化 / 偿还节奏、防止架构腐化的工程手段。

9.1 演进式架构的核心理念

"An evolutionary architecture supports guided, incremental change across multiple dimensions." — Building Evolutionary Architectures (Ford, Parsons, Kua, 2017)

三个关键词：

  • Guided（被引导）：变更不是随机漂移，有目标和约束
  • Incremental（增量）：小步快跑，不是几年一次大重写
  • Multiple dimensions（多维度）：技术、业务、运维、组织都能演进

9.2 Fitness Function（架构适应性函数）—— 把架构原则代码化

Fitness Function：把架构原则写成自动化测试 / CI 检查，每次提交都验证架构特性是否保持。

为什么必须自动化？人工守护必然失败——三个月后没人记得"业务模块不能依赖 UI 模块"这条原则，新人没人告诉，架构就这样被无声地侵蚀。

四类 Fitness Function

类别检查C++ 实现
依赖方向 "Domain 不能依赖 Infrastructure" CMake target 边界 + cpp-dependencies / include-what-you-use
性能基线 "关键路径 P99 ≤ 50ms" Google Benchmark + CI 阈值检查
API 稳定性 "public API 不能破坏 ABI" abi-compliance-checker
代码质量 "圈复杂度 < 15、文件 < 500 行" clang-tidy / SonarQube / lizard

实战：用 CMake 强制依赖方向

# CMakeLists.txt
add_library(domain INTERFACE)
target_include_directories(domain INTERFACE src/domain)
# domain 不允许 target_link_libraries

add_library(application STATIC src/application/...)
target_link_libraries(application PUBLIC domain)

add_library(infrastructure STATIC src/infrastructure/...)
target_link_libraries(infrastructure PUBLIC application PRIVATE pqxx)

# Fitness function: 在 CI 里跑这个脚本
# scripts/check_arch_deps.sh
#!/bin/bash
set -e

# 1. 检查 domain 不包含任何外部头
if grep -r "#include <" src/domain --include=*.hpp \
   | grep -v "<cstdint>\|<optional>\|<string>\|<vector>"; then
    echo "❌ domain layer includes forbidden headers"
    exit 1
fi

# 2. 检查 application 不包含 infrastructure
if grep -r "#include "infrastructure/"" src/application; then
    echo "❌ application depends on infrastructure"
    exit 1
fi

echo "✅ Architecture dependency rules satisfied"

9.3 Strangler Fig 模式：渐进式重构遗留系统

Fowler 2004 年提出的"绞杀者无花果"模式。来自一种植物：长在大树上，缓慢包裹宿主树，最后宿主死去而新树代之。

Strangler Fig 的完整步骤

  1. Identify Seam（识别接缝）：找到遗留系统中"可以被替换的最小独立单元"
  2. Build Facade（建立外观）：所有调用先经过一个 router/gateway，对调用方透明
  3. Implement New（实现新版本）：在新系统里重写这个单元的功能
  4. Switch by Feature Flag（按开关切流量）：先内测、灰度、再扩到 100%
  5. Verify（对账验证）：双写或影子流量，比对新老系统输出
  6. Decommission（下线遗留代码）：流量稳定切完后真的删除老代码，不要留着
  7. Repeat：选下一个 seam，重复
失败的典型：第 6 步没做——老代码留着"以防万一"。结果两套代码并存，所有维护成本翻倍，没人敢碰老代码（怕回滚），新代码又不敢全切（怕老代码功能没覆盖完）。必须设定 "decommission deadline"，到时间就删，不删就回退到上一步。

Strangler Fig 在 C++ 项目中的落地

// === Façade：路由到新老实现 ===
class OrderServiceFacade : public IOrderService {
    LegacyOrderService& legacy_;
    NewOrderService& new_impl_;
    IFeatureFlag& flags_;
public:
    Order place(const Command& cmd) override {
        if (flags_.is_enabled("new_order_service", cmd.customer_id)) {
            auto result = new_impl_.place(cmd);

            // 影子模式：异步用 legacy 算一遍，对账
            if (flags_.is_enabled("shadow_mode")) {
                async_compare(cmd, result, [this](auto& legacy_result) {
                    if (!matches(new_result, legacy_result)) {
                        log.error("shadow mismatch", cmd, new_result, legacy_result);
                        metrics.increment("strangler_mismatch");
                    }
                });
            }
            return result;
        }
        return legacy_.place(cmd);
    }
};

9.4 技术债的识别与管理

Ward Cunningham 的金融类比（1992）

"Shipping first-time code is like going into debt. A little debt speeds development so long as it is paid back promptly... The danger occurs when the debt is not repaid."

四种技术债（Martin Fowler 矩阵）

故意 (Deliberate)无意 (Inadvertent)
谨慎 (Prudent) "我们必须先发货，下季度重构这个"
有意识，有还款计划		 "原来应该用 Repository 模式"
事后才意识到
鲁莽 (Reckless) "没时间做设计，先写"
明知有更好做法但偷懒		 "什么是 SOLID？"
没意识到问题
架构师的责任：把"鲁莽 + 无意"象限清零（通过培训、code review、ADR），把"鲁莽 + 故意"变成"谨慎 + 故意"（带还款计划）。剩下的两类是可控的。

技术债的量化方法

SQALE 方法

SonarQube 用的标准。把每个 issue 转换为"修复需要的时间"，得到技术债总额（小时 / 天）。

例：圈复杂度过高 = 30 min；模块过大 = 1 day；重复代码 10 处 = 2 hr

计算债务率：debt / 总开发时间。> 5% 红灯。

痛苦指数

更直观：让团队每个 sprint 投票"哪段代码最痛苦"。用"修改频率"× "每次修改的痛苦"得分。

这是 Adam Tornhill 在 *Your Code as a Crime Scene* 中倡导的方法——结合 git 历史 + 复杂度，找出"经常改且又复杂"的代码，优先重构。

偿还节奏：20% 法则

工程实践共识：每个 sprint / 季度，预留 20% 容量做技术债偿还和架构演进。
  • 少于 20% → 债务持续增长，3 年后系统瘫痪
  • 多于 20% → 业务方不爽，架构师变成"挡路者"
  • 20% 是经验值，可根据债务严重度调整
怎么让业务方同意 20%？把技术债的影响翻译成业务语言："这块代码每次改动需要 5 人天，重构后只需 1 人天，10 次改动就回本"。

9.5 防止架构腐化的工程手段

① 强制依赖检查

CMake 边界 + CI 脚本 + clang-tidy 自定义规则。架构原则必须由机器守护。

② Code Review 清单

把架构原则做成 review checklist，每个 PR 必走。不依赖 reviewer 记忆。

③ 架构 Fitness Function

把架构特性写成自动化测试，CI 跑。失败 = 阻塞 merge。

④ 定期架构评审

每季度回顾架构状态：ADR 累计了多少？哪些 superseded？技术债趋势？

⑤ 架构 Onboarding

新人第一周必读：核心 ADR + C4 图 + Coding Standard。否则新人就是腐化的来源。

⑥ 重构作为一等公民

重构 task 跟 feature task 同等优先级，进 backlog、有 estimate、有 demo。

9.6 大型重构的"安全网"

当你必须做 > 1 个月的重构时，需要建立这些安全保障：

  1. Baseline 测试：重构前必须有充分的 regression test。如果没有 → 先补测试再重构
  2. 小步提交：每次提交都编译通过 + 测试通过。永远不要有"半重构"的 PR
  3. Feature Flag 隔离：新老代码并存，开关切换
  4. 影子流量 / 双写对账：生产环境验证新版本，对比输出
  5. Canary 发布：1% → 10% → 50% → 100%
  6. Rollback 演练：必须能 5 分钟内回滚，定期演练
  7. Decommission Deadline：明确"老代码什么时候彻底删除"，不留模糊地带

9.7 嵌入式 / 安全关键系统的演进特殊性

车载 / 航空 / 医疗等安全关键系统的演进面临特殊挑战：
  • 必须保证 ASIL 等级不退化：架构演进每一步都要做 Impact Analysis，证明安全性不变
  • OTA 升级的回滚能力：必须有 A/B 分区、原子升级；任何升级失败必须能自动回到上一版本
  • 长寿命 product line：汽车产品线 10-15 年；架构师要做"未来兼容性"决策——10 年后的工具链还能编译这份代码吗？
  • Variant / Configuration Management：同一份 codebase 要支持几十款车型、几百个软件 variant；架构必须支持编译期或运行期变体管理（feature toggle、preprocessor、AUTOSAR variant handling）
  • 认证再评估：架构变更可能触发全套 ISO 26262 认证流程——每次大重构成本极高，所以 Strangler Fig 这种"小步快跑"在嵌入式领域更有价值
  • 硬件演进同步：HW 平台每 3-5 年大换代（如从 Renesas RH850 转 Infineon AURIX），架构必须做硬件抽象层（HAL），保护业务代码
嵌入式架构演进的两个"必备 ADR"：HAL 边界 ADR（什么是硬件抽象层、谁不能跨边界）、Variant Management ADR（feature / configuration / version 怎么管理）。

9.8 本章小结

  • 没有架构能"一次设计永远不变"，架构师真正的工作是守护演进
  • Fitness Function：把架构原则写成自动化测试，让机器守护架构
  • Strangler Fig：处理遗留系统的安全重构模式——小步替换，永远可回滚，最后必须 decommission
  • 技术债四象限：把"鲁莽+无意"清零，剩下的债务都可控
  • 偿还节奏：每周期预留 20% 容量偿还债务
  • 防腐化六手段：依赖检查 / Review 清单 / Fitness Function / 季度评审 / 新人 Onboarding / 重构一等公民
  • 大型重构必须建立安全网：测试 baseline、小步提交、Feature Flag、影子对账、Canary、Rollback、Decommission Deadline
  • 嵌入式系统额外考虑：ASIL 不退化、OTA 回滚、长寿命产品线、HAL 抽象、认证再评估成本
小练习：写下你当前项目里3 个最大的技术债，分类（故意/无意，谨慎/鲁莽），估算每个的偿还成本（人天），以及"不偿还的话每月业务损失多少"。如果三个数都写不出来——技术债没被识别 / 没被量化，永远不会被偿还。

10嵌入式 / 安全关键系统的架构特殊性

前面 9 章是通用方法论。本章专门讨论嵌入式 / 车规 / 航电 / 医疗 / 工控等"安全关键系统"的架构特殊性。这是你（E + G 双方向）选择的领域，方法论的应用方式与互联网软件有本质差异——资源约束、确定性、合规要求、长寿命，每一个都改变架构决策的权重。

ASIL 等级对架构决策的影响（以汽车为例）

QM / ASIL-A（低风险）

例：座椅记忆、车窗控制

  • 常规 C++ 工程实践即可
  • 可用 STL、智能指针、异常（如果允许）
  • 普通单元测试 + 集成测试

ASIL-B（中风险）

例：自动大灯、车道偏离告警

  • MISRA / AUTOSAR C++14 子集
  • 无动态分配（启动后）
  • WCET 分析
  • 静态分析工具强制

ASIL-C（高风险）

例：自适应巡航、自动泊车

  • 更严格的子集
  • 必须有 fault detection 机制
  • Lock-step CPU（双核同步比对）
  • 完整 FMEA / FTA

ASIL-D（最高风险）

例：刹车控制、转向、ABS

  • 冗余架构（独立硬件 + 独立软件）
  • Tool Qualification（编译器、静态分析都要认证）
  • 覆盖率 MC/DC 100%
  • 形式化方法（部分场景）
  • 故障率 < 10⁻⁸ /h

"Freedom From Interference" — 等级隔离原则

FFI（Freedom From Interference）：低 ASIL 等级的组件不能干扰高 ASIL 等级的组件。互相必须隔离。

典型架构落地：

  • 内存隔离：用 MMU/MPU 把 ASIL-D 进程内存隔离开
  • 时间隔离：用分区调度（ARINC 653 / AUTOSAR Adaptive 的 Resource Group），保证 ASIL-D 任务的 CPU 配额
  • 通信隔离：跨等级通信走显式接口 + 数据校验，不共享内存
  • Watchdog 监控：ASIL-D 任务有独立 hardware watchdog

10.3 安全架构模式

① Redundancy（冗余）

2oo2 (2 out of 2)

两路必须都同意，否则进入安全状态。偏向安全。

例：刹车的两个独立通道，任一通道说"不能刹"就立即报错。

2oo3 (2 out of 3)

三路投票，多数胜。偏向可用。

例：飞控的三冗余。一路坏了系统继续工作。

② Diverse Redundancy（异质冗余）

用不同实现互相检查，避免共因失效（common-cause failure）：

  • 主路径用 C++，监控路径用不同算法 / 不同语言（C / Rust）
  • 主芯片用 ARM Cortex-A，监控芯片用 Cortex-M 或 PowerPC
  • 主算法用神经网络，监控算法用传统规则法

③ Lock-Step（步进式同步）

两个相同核同步执行同一指令，硬件比对结果。任何不一致 → 立刻报错。AURIX TC397、Cortex-R5 等都支持。

④ Safe State（安全状态）

系统检测到无法继续安全工作时，进入预定义的"安全状态"。例：

  • 自动驾驶：交还驾驶员 / 紧急停车（Minimum Risk Maneuver）
  • 工业机械臂：所有关节减速到 0、刹车锁死
  • 医疗设备：停止输液、维持维生信号

⑤ Graceful Degradation（优雅降级）

非关键功能可以降级，关键功能保持。例：

  • 娱乐系统挂了 → 不影响驾驶
  • 巡航系统挂了 → 退化为手动
  • 毫米波雷达挂了 → 用摄像头继续工作（降级 ASIL 等级）

10.5 ROS2：自动驾驶 / 机器人事实标准

ROS2 架构亮点（相对 ROS1）

  • DDS 替代自研中间件：可插拔的 DDS 实现（FastDDS、CycloneDDS、RTI），QoS 完整
  • 无 master 节点：去中心化，单点故障消除
  • 实时支持：可在 PREEMPT_RT 上做软实时
  • Lifecycle 节点：明确的生命周期状态机（unconfigured / inactive / active / finalized），便于编排
  • 多 OS 支持：Linux / Windows / QNX / macOS

架构师视角的 ROS2 关键决策

DDS Vendor 选型FastDDS（开源）vs CycloneDDS（性能）vs RTI Connext（商业支持）
Topic / Service / Action 选用持续数据流 → Topic；同步调用 → Service；长任务 → Action
QoS 策略Best-Effort vs Reliable；Volatile vs Transient Local；Deadline；Lifespan
节点粒度"一节点一进程" vs Component Container（多节点共享进程，零拷贝）
Executor 选择SingleThreaded / MultiThreaded / StaticSingleThreaded（实时性能差异巨大）
实时性保证预分配内存 + 锁页 + CPU 隔离 + PREEMPT_RT
ROS2 不是 "导入即可用"。架构师必须做大量约束设计才能在车规级应用：限制 std 库使用、自定义内存策略、绕过 ROS2 默认线程模型、严格控制 callback 复杂度。大厂常用做法是"ROS2 仅作为中间件，业务代码自己写"，避免 ROS2 框架带来的不确定性。

10.6 嵌入式 C++ 架构师的 8 条"硬约束"

  1. 禁动态分配（启动后）：所有内存必须启动时分配，运行时无 new/malloc。用 PMR allocator + 预分配池。
  2. 禁异常（多数情况）：用 std::expected / 自定义 Result<T,E>。
  3. 禁 RTTI：用 std::variant + visitor、type tag。
  4. 限制 virtual：vtable 占用 Flash + 间接调用延迟；用 CRTP / templates。
  5. 禁递归（多数情况）：栈深度不可控；用循环 + 显式栈。
  6. 禁 std::regex / std::function：不确定的性能 / 动态分配。
  7. 显式线程亲和性：所有线程必须绑定 CPU，不能让 OS 自由调度。
  8. 所有 API 文档化 WCET：必须给出最坏执行时间。

10.7 信息安全（Cybersecurity）—— 新热点

2024 年起，汽车软件必须符合 ISO/SAE 21434（行业）+ UNECE R155/R156（法规）。架构师必须考虑：

威胁建模（TARA）

Threat Analysis and Risk Assessment。对每个组件做：

  • 识别威胁场景
  • 评估影响（safety / financial / operational / privacy）
  • 评估攻击可行性（attack feasibility rating）
  • 计算风险
  • 选择对应措施

关键架构机制

  • Secure Boot：启动时校验固件签名
  • Secure OTA：升级包签名 + 完整性校验
  • HSM（Hardware Security Module）：密钥存储
  • Network Segregation：CAN / Ethernet 分域隔离
  • Intrusion Detection：CAN bus 异常报文检测
  • Secure Logging：审计日志防篡改

10.8 嵌入式架构师工作中的 6 个 ADR 必写场景

  1. HAL 边界：硬件抽象层在哪、谁能 / 不能跨
  2. C++ 子集：用 MISRA C++ 2008 还是 AUTOSAR C++14？哪些规则放行 / 强制？
  3. 错误处理范式：Result<T,E> 还是 status code？跨 SIL 等级如何传播？
  4. 线程模型：每个 ASIL-D 任务的优先级、CPU 绑定、调度策略
  5. 通信中间件：自研 IPC？AUTOSAR Adaptive ara::com？ROS2 DDS？SOME/IP？
  6. Variant Management：feature toggle 用编译期 (preprocessor / cmake) 还是运行期？

10.9 本章小结

  • 嵌入式架构师与互联网架构师的本质差异：核心质量属性、资源约束、合规、产品寿命都不一样
  • 功能安全是嵌入式架构的硬约束，ASIL/SIL 等级直接决定架构决策（冗余、隔离、Lock-step、Safe State、Graceful Degradation）
  • Freedom From Interference：不同等级组件必须做内存 / 时间 / 通信隔离
  • AUTOSAR Adaptive是高级 ECU 的事实标准；架构师需理解 ara::com、Execution Management、State Management
  • ROS2是机器人 / 自动驾驶的事实标准；车规应用需要严格约束设计
  • 嵌入式 C++ 8 条硬约束：禁动态分配、禁异常、禁 RTTI、限 virtual、禁递归、禁 regex/function、显式线程亲和、文档化 WCET
  • Cybersecurity 是 2024+ 的强制要求：威胁建模 + Secure Boot/OTA/HSM/网络分域/IDS
  • 嵌入式架构 6 个必写 ADR：HAL / C++ 子集 / 错误处理 / 线程模型 / 通信中间件 / Variant 管理

高频架构面试题 · 深度解析

从一线公司架构师/Principal/Staff 岗位真实面试中抽取的高频题，附深度解答与追问框架。
Q1 什么是软件架构？它和详细设计、代码风格有什么本质区别？
P0主题：架构定义难度：⭐⭐常见公司：FAANG/BAT 架构师面试
核心答案（标准定义）

架构 = 那些后期很难改的关键决策的总和（Fowler）。它包括：(1) 系统的宏观结构（模块划分）；(2) 对关键质量属性的承诺（性能 / 可用性 / 可演进性）；(3) 约束（允许做什么、不允许做什么）。

和详细设计的区别（用"决策回滚成本"区分）
  • 架构决策：回滚成本 1 年以上（如选关系数据库 vs NoSQL、选共享内存 vs 消息队列）
  • 详细设计：回滚成本 1-4 周（如某个函数的接口、某个类的内部数据结构）
  • 代码风格：回滚成本 1 天以内（变量命名、缩进格式）
追问 1：你怎么判断一个决策属于哪一层？

问自己三个问题：(1) 影响多少模块？(2) 影响多长时间？(3) 修改它需要多少团队参与？三个问题答案都很大 → 架构决策。

面试加分：举出最近做过的架构决策（STAR）

用真实经历回答：当时遇到什么 (Situation)，背景约束是什么 (Task)，你考虑了哪些选项 (Action)，结果如何 (Result)。关键是讲清楚为什么"否决"了其他选项——这才是架构师的核心动作。

常见陷阱

❌ 把"用了某某模式"当成架构（这只是局部设计）。
❌ 把"用了某某框架"当成架构（这是技术选型，架构是更高维度的决策）。

Q2 什么时候选六边形（Hexagonal）架构，什么时候用普通分层架构就够了？
P0主题：架构风格选型难度：⭐⭐⭐
核心答案

看两个信号：(1) 业务规则复杂度；(2) 业务核心是否需要长期独立演进。

  • 简单 CRUD + 单一前端 + 单一数据源 → 分层架构足够（10 万行以下、团队 < 10 人）
  • 业务规则复杂 + 多种 UI / 多种存储 + 需要业务逻辑单元测试 → Hexagonal / Clean
  • 核心商业逻辑必须 10 年后还能用 → 必须 Hexagonal，把核心和框架解耦
Hexagonal 的核心收益（不会就是普通开发）
  • 业务核心 100% 可独立测试（无 DB / 无 HTTP / 无 Kafka）
  • 技术细节可替换（PostgreSQL → MongoDB 只改 infrastructure/）
  • 核心代码寿命远长于框架代码（Spring/Boost.Beast 都会被淘汰，业务核心不会）
追问：Hexagonal 的代价是什么？

(1) 开发慢 20-30%（每个外部依赖要写两个接口：Port 和 Adapter）；(2) 简单功能也要走全套依赖注入；(3) 团队 onboarding 成本高。所以小项目"过度设计"是真实痛点。

追问：Onion / Clean / Hexagonal 有什么区别？

本质相同，都是依赖倒置 + 核心独立。区别只在视觉化和层数：Hexagonal 是 2 层（核心 vs 外部），Onion 是同心圆多层，Clean 是 4 层 + 明确的 Dependency Rule。面试时讲清楚"它们本质相同"就能加分。

Q3 DDD 的 Bounded Context 和微服务是什么关系？一个 Bounded Context = 一个微服务吗？
P0主题：DDD + 微服务难度：⭐⭐⭐⭐
核心答案（这题考概念辨析）

不是。Bounded Context 是语言学/业务边界（语言一致的范围）；微服务是物理部署边界（独立部署的单元）。一个 BC 可以实现为：

  • 单体应用里的一个模块（namespace + 目录）
  • 共享库（library）
  • 独立服务（微服务）
  • 独立产品线

选哪种实现取决于：团队规模、变更节奏、质量属性差异、运维能力。

什么时候 BC 应该升级为微服务
  1. 团队大到沟通成本超过协作收益（> 50 人）
  2. 这个 BC 的变更节奏和其他 BC 差异巨大
  3. 这个 BC 需要独立扩展（如 10x scale）
  4. 这个 BC 有不同的可用性/安全等级要求
常见陷阱：把 DDD = 微服务

很多人一上来就把所有 BC 拆成微服务，结果是"分布式单体"——所有服务都得一起部署，跨服务调用一团乱。正确做法：先做 modular monolith（按 BC 切模块但单一进程），等真正需要再拆。

面试加分：Fowler 的 "Monolith First" 观点

引用 Fowler 的话："Almost all the successful microservice stories have started with a monolith that got too big and was broken up." 直接上微服务的项目几乎都失败了。

Q4 什么是 Aggregate？它的边界怎么划？讲一下 5 条经典规则。
P0主题：DDD 战术难度：⭐⭐⭐
核心答案

Aggregate = 一组紧密关联的对象，必须作为一个整体保持一致。它是事务一致性的边界——一个事务最多只能修改一个 Aggregate。

5 条经典规则（Vaughn Vernon）
  1. 根是唯一入口：外部只持有 Aggregate Root 引用
  2. 一个事务只修改一个 Aggregate：跨 Aggregate 用 Domain Event 异步协调
  3. 跨 Aggregate 只引用 ID：不引用对象本身
  4. Aggregate 要小：能在一个事务内 load + save
  5. 最终一致性优于强一致性：跨 Aggregate 的一致性默认是最终一致
追问：如果违反"一个事务只修改一个 Aggregate"会怎样？

(1) 性能问题：需要分布式锁或大事务，并发吞吐下降；(2) 死锁风险：多个 Aggregate 加锁顺序不同会死锁；(3) 测试困难：业务规则跨 Aggregate 分散，难以理解。

常见陷阱：大 Aggregate

新人最常错：把 Customer 设计成包含所有 Order、Address、PaymentMethod 的"超大聚合"，加载一个 Customer 要拉几万行数据库 → 性能灾难。正确做法：Customer 和 Order 是两个 Aggregate，互相只引用 ID。

Q5 为什么要写 ADR？没写 ADR 会怎样？讲一个你写过的 ADR。
P0主题：架构治理难度：⭐⭐
核心答案

ADR（Architecture Decision Record）记录每个重要决策的"Context（背景）/ Decision（决定）/ Consequences（后果）"。

没写 ADR 的代价：

  • 6 个月后新人问"为什么这样做"——没人答得出
  • 1 年后团队推翻这个决策，实施后才发现是有原因的
  • 同一个讨论反复开会 5 次，每次都重新发明历史
  • 架构腐化的最大原因不是技术差，是决策上下文丢失
哪些决策值得写 ADR

判断标准：影响半径 > 1 个模块，回退成本 > 1 周。典型场景：数据库选型、消息队列选型、通信协议、构建系统、错误处理范式、API 版本策略。

面试加分：讲一个真实 ADR（用 STAR）

例子："在 XX 项目里，我们要选 IPC 方案。背景是 P99 延迟必须 ≤10ms（来自质量属性场景）；候选是共享内存、ZeroMQ、SOME/IP；决定用共享内存（因为延迟最低、无 broker 依赖）；后果是性能达标但需要更严格的并发测试；follow-up是上线后加入 TSan 持续检测。"

讲完后强调："这条 ADR 后来帮了我们大忙——一年后有同事质疑这个方案，我们直接拿出 ADR，5 分钟讲清前因后果，避免了无谓的重写。"

Q6 如何防止架构腐化？Fitness Function 是什么？给一个具体例子。
P1主题：演进式架构难度：⭐⭐⭐⭐
核心答案

架构腐化的根因是"架构原则只在架构师脑子里"——人会忘记、新人不知道、PR review 会放水。

Fitness Function：把架构原则写成自动化测试 / CI 检查，每次提交都验证。这是"让机器守护架构"的唯一可靠方式。

四类 Fitness Function
  • 依赖方向：CMake target 边界 + CI grep 检查 "Domain 不能 include Infra"
  • 性能基线：Google Benchmark + CI 阈值检查（P99 不能退化）
  • API 稳定性：abi-compliance-checker（公开 API 不能破坏 ABI）
  • 代码质量：clang-tidy / SonarQube（圈复杂度、文件大小）
面试加分：具体的 Fitness Function 实现
# CI 脚本：检查 domain 层不依赖 infrastructure
if grep -r "#include "infrastructure/"" src/domain/; then
    echo "❌ domain depends on infrastructure"
    exit 1
fi

这是把 Clean Architecture 的"Dependency Rule"代码化——一行命令防止架构腐化。
