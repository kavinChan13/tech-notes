02

Paxos 完整深入

Single-decree / Multi / Fast / Flexible

Leslie Lamport 1989 paper（1998 published）。共识算法的祖先。Raft 是后辈，但 Paxos 本身有 variant 适合不同场景。

Basic Paxos (Single-decree)

目标：N 个 node 对一个值达成一致，容忍 ⌊N/2⌋ 失败。

3 个 role

  • Proposer: 提出 value
  • Acceptor: 投票（majority quorum）
  • Learner: 学习被 chosen 的 value

实际 system 中一 node 可担多 role。

2 阶段协议

  • Phase 1：Proposer 广播 Prepare(n)；Acceptor 若 n > highest_promised 则返回 Promise(n, accepted_value?)
  • Phase 2：收到多数 Promise 后，用最高 accepted_n 对应的 value（或自己的 value）广播 Accept(n, v)
  • 多数 Accepted → value chosen

关键 invariant

  • Proposer 选下次 proposal number > 见过的所有 n
  • Acceptor 不接受小于 promised 的 n
  • Phase 2 必须用 Phase 1 看到的最高 accepted value（不是自己的）→ 保 safety

为什么 Paxos 难理解

Lamport 自己也承认 Paxos "notoriously hard"。原因：

  • 抽象层级高（用 "chosen value" 而不是 log entries）
  • 每 instance 只能定一个 value
  • 实际系统要 chain Paxos instance → Multi-Paxos
  • 论文中 corner case 散布在多处

Multi-Paxos

实际用：sequence of Paxos instances 对 log 每个 entry 达成共识。

优化

  • Stable leader—— 跳 Phase 1（稳定 leader 可直接 Phase 2）
  • Batching—— 多个 client request 合一 Paxos instance
  • Pipelining—— 同时跑多个 instance 不等前一个完成

Fast Paxos

Lamport 2006 提出。允许 client 直接发给 Acceptor（跳过 Proposer），减一个 RTT。但需要更大 quorum（typically 3/4 而非 1/2）。实战很少用—— 复杂度增加 vs 节省微薄。

Flexible Paxos (FPaxos)

Howard 2016。关键 insight：Phase 1 quorum (Q1) + Phase 2 quorum (Q2) 不必都 majority，只需Q1 ∩ Q2 ≠ ∅。

实战 implication：

  • 读多写少：Q1 = N (all), Q2 = 1 → 读极快但每次 Q1 慢
  • 写多读少：Q1 = 1, Q2 = N → 写快
  • Vertical Paxos / EPaxos 等变种

EPaxos (Egalitarian Paxos)

Moraru 2013。无 leader，每个 node 可同时处理 request。冲突时降到 slow path。适合地理分布（cross-region），avoid leader 远 client 的 latency。

03

Raft 实战细节

Leader election / 日志 / Snapshot / 配置变更

Ongaro & Ousterhout 2014 USENIX paper，"Search for an Understandable Consensus Algorithm"。Raft 是2026 工业界共识算法首选。

关键设计差别（vs Paxos）

  • 显式 leader-based—— 简化逻辑
  • Log 作一等公民—— 不抽象成 "chosen value"
  • 强 leader—— 所有 client 走 leader，log 严格 leader → follower 复制
  • Election 用 randomized timeout—— 避免 split vote

Raft 4 个核心子问题

3.1 Leader Election

  • 3 个 state: Follower / Candidate / Leader
  • 每个 server 维护currentTerm（单调递增）
  • Follower 在 election timeout（150-300ms random）内没听到 leader heartbeat → 成为 Candidate
  • Candidate 发 RequestVote 给所有 server，得 majority → 成为 Leader
  • Random timeout 避免 split vote（理论可能但实际罕见）

3.2 Log Replication

  • Leader 收到 client request → append 自己 log → 通过 AppendEntries 复制到 follower
  • 当某 entry 在 majority replicated → commit
  • Commit 后 apply 到 state machine + 返回 client
  • Log matching property: 如果两 log 在 index N 有 same term，则 [0, N] 全 identical

3.3 Safety Property

Property含义
Election Safety1 term 最多 1 leader
Leader Append-OnlyLeader 不删 / 不改自己 log，只 append
Log Matching同 index + term → 全 prefix 同
Leader Completeness当 entry committed → 所有 future leader 必含此 entry
State Machine Safety所有 server 在同 index 的 state machine apply 同 entry

3.4 Membership Change

Raft 最 tricky 部分。Naive 同时加 / 删 server 可能出现两个 majority。

解：Joint Consensus—— 中间 transition state 同时使用 old + new config：

  1. Leader 收到 reconfig 请求
  2. Append C_old,new log entry → 双 config 期
  3. 双 config 期所有决议需 old majority + new majority
  4. Append C_new entry → 单 new config 期
  5. 不在 new config 的 server 退休

etcd / TiKV 实际用single-server change（一次加 / 删 1 个），简化但需要满足: |old majority| + |new majority| > N。

Raft 实战陷阱

  • Pre-Vote—— 候选人先问"我能赢吗"再增 term，避免网络分区后回归 node 干扰
  • CheckQuorum—— Leader 定期检查自己仍是 majority，否则 step down
  • Log compaction (snapshot)—— log 大了 install snapshot 给 lag 太多的 follower
  • Read-only optimization—— Leader read 时也要 check leader lease 或 quorum (确保仍 leader)
  • Lease-based read—— Leader 保 lease（election timeout 内），lease 内直接读不走 quorum

Paxos vs Raft

选型 + Byzantine 扩展

维度PaxosRaft
可理解性难易（设计目标）
实现复杂度高中等
Leader 模式可选（Multi-Paxos 通常有）强 leader（设计基础）
Membership change需特别设计内置 joint consensus
性能同（理论 + 实测）同
容错F = ⌊N/2⌋（crash fault）同
典型应用Google Chubby / Spanner / Paxos-based DBetcd / Consul / TiKV / CockroachDB

2026 实战Raft 占主流—— 易实现 + 易理解 + 生态强。Paxos 在 Google / 老牌 DB（Cassandra Lightweight Tx）等仍用。

Byzantine Fault Tolerance (BFT)

Paxos / Raft 假设crash fault（node 死但不撒谎）。BFT 容忍恶意 node（撒谎 / 发不一致 message）。

BFT 算法容错用
PBFT (Castro & Liskov 1999)F < N/3HyperLedger Fabric / 部分 permissioned blockchain
HotStuff (2018)F < N/3Diem / Aptos / Sui
TendermintF < N/3Cosmos / 多个 PoS chain
Nakamoto (BTC)F < N/2 (assuming honest majority hash)Bitcoin / PoW chain

实战：

  • 普通 backend / cloud → crash-fault BFT 够，用 Raft
  • 多组织协作（Permissioned blockchain）→ PBFT / HotStuff
  • 开放区块链 → Nakamoto / PoS variant
05

时间与因果

Lamport / Vector / Hybrid / TrueTime

"2 个 event 谁先发生？" 在分布式系统不 trivial。

5.1 Physical Clock (NTP)

  • NTP 同步 typically 10-100ms 精度（network jitter + clock drift）
  • PTP (IEEE 1588) 可达 µs（专门硬件 + 网络）
  • 物理 clock不可信—— 时钟回退 / 跳 / drift / leap second

5.2 Lamport Clock (1978)

逻辑时间 —— 不需物理时钟，纯协议：

local event → clock++
send message → clock++ → attach clock
receive msg → clock = max(clock, msg.clock) + 1

性质

  • 若 event a → event b（happens-before），则 L(a) < L(b)
  • 反向不成立—— L(a) < L(b) 不能推出 a → b（可能并发）

5.3 Vector Clock

每 process i 维护 N-元 vector V_i。

local @ i → V_i[i]++
send @ i → V_i[i]++ → attach V_i
recv @ i → V_i[i]++ → V_i = max(V_i, msg.V_j) element-wise

性质

  • 能精确判断 a → b（VC(a) < VC(b)，每元 ≤ 且至少一个 <）
  • 不能定的就是真正并发（concurrent）
  • 开销：N-元 vector，N 大时贵

5.4 Hybrid Logical Clock (HLC)

Kulkarni 2014。结合物理 + 逻辑：

  • 每 timestamp 是 (physical, logical)
  • 跟物理时钟接近（NTP），但 monotonic 保证
  • vs Lamport: 接近 wall clock，可跨 service compare
  • vs Vector: 只 1 个 (pt, lt) pair，不随 process 数量增长
  • 用：CockroachDB / YugaByte / 部分 distributed DB

5.5 Google TrueTime

Spanner 用，特殊在硬件支持：

  • 每个 datacenter 部署 GPS + 原子钟
  • API: TT.now() → TTInterval{earliest, latest}—— 显式 uncertainty
  • 典型 uncertainty 7 ms
  • 等待"commit wait" 让 uncertainty 过去 → globally consistent timestamp
  • Spanner 用这个做 globally distributed strong consistency

其他云厂商类似：

  • AWS TimeSync (PHC + PTP)
  • Azure / GCP 各有 service

一致性模型谱系

Linearizable → Causal → Eventual

"强一致" vs "最终一致"是粗划分，实际谱系细：

模型保证例
Linearizable所有 op 像在 single point in time atomic 执行etcd / ZK / Spanner
Sequential Consistency所有 process 看到同样 order，但不一定是 real-time order少数 DB
Causal Consistency因果相关的 op 全 process 同 order；不相关的 op 各自可不同COPS / Bayou
PRAM (Pipeline RAM)同 process 的 write 别人看到 same order；不同 process 的 write 可不同 order部分 NoSQL
Read-Your-Writes自己看得到自己写过的多数现代 system
Monotonic Reads同 reader 不会"退回去"读到更老数据多数现代 system
Eventual无 update 时最终一致，无 ordering 保证DNS / Cassandra default

Session Guarantees

实际系统经常给"session-level"保证（per user / connection）：

  • Read-your-writes
  • Monotonic reads
  • Monotonic writes
  • Writes follow reads

组合起来在per-user 看是 strong，但跨 user 是 eventual—— 对实际应用够用且 cheaper。

Linearizable vs Serializable

常被混淆：

  • Linearizable—— single object，real-time order
  • Serializable—— multiple object transaction，equivalent to some serial order（不一定 real-time）
  • Strict Serializable—— Linearizable + Serializable

CRDT

两大类 · 常用 CRDT · 真实用 · CRDT vs Raft 选型

Conflict-free Replicated Data Type—— 数据结构 mathematically guarantee 多 replica无协调也能 converge。

两大类

State-based (CvRDT)

每 replica 周期性 broadcast 自己的全量 state。Receiver 用commutative + associative + idempotent merge function 合并。

Example: G-Counter（grow-only counter）:

  • 每 replica 存 vector V[N]（N = replica 数）
  • Inc on replica i: V[i]++
  • Merge: 各 element 取 max
  • Value: sum(V)

Operation-based (CmRDT)

每 op 通过causal broadcast 传到所有 replica。Op 必须 commutative。

常用 CRDT

类型用
G-Counter (grow-only)统计 / 点赞
PN-Counter (pos + neg)账户 balance
G-Set (grow-only set)用户 follow / 收藏
2P-Set (two-phase set)add + remove
OR-Set (Observed-Remove)主流 set CRDT
LWW-Register / LWW-Maplast-write-wins 简单冲突 resolve
RGA / Treedoc / Logootcollaborative text editing
YATA / Yjs主流协同编辑 (Figma / Linear)
AutomergeJSON-like CRDT framework

真实用

  • Figma—— 实时 collaborative design
  • Linear—— 任务管理同步
  • Riak—— DB-level CRDT support
  • Redis—— Redis CRDT (Enterprise) / RedisGears
  • Notion / Obsidian / Logseq—— offline-first sync
  • iCloud / Local-first—— Apple 部分 sync

CRDT vs Raft 选型

08

FLP 不可能性 + 实际系统

"异步" 关键 · 那 Raft / Paxos 怎么 work？ · 分布式 3 大 trade-off

FLP 不可能定理（Fischer-Lynch-Paterson 1985）：

📐 FLP 定理

在异步分布式系统中，即使仅 1 个 process crash，也不存在能确保 termination 的 deterministic consensus 算法。

"异步" 关键

  • 消息可延迟任意时间
  • 无法区分"node 死了" vs "消息延迟"
  • 无可信全局时钟

那 Raft / Paxos 怎么 work？

"违反 FLP"？不，它们 trade safety 跟 liveness：

  • Raft / Paxos 始终 safe（不会给出错误共识）
  • 但可能不 terminate（如持续 election 不出 leader）
  • 实际用 random timeout + "partially synchronous" assumption 让 termination 概率 → 1

分布式 3 大 trade-off

定理说
FLP异步 + 1 crash → 不能保 termination
CAP网络分区下 C vs A 二选一
PACELC有分区 P→ A vs C；无分区 E→ L (latency) vs C

形式化验证

TLA+ / P / Verdi

分布式协议 bug 是噩梦——分支组合爆炸，单测 catch 不到。形式化验证可数学证明协议正确性。

TLA+ (Lamport)

Lamport 自己设计的形式化规约语言。用法：

  1. 用 TLA+ 写规约（spec）—— 系统 state + transition
  2. 用 TLC（model checker）穷举有限 state space
  3. 验证 invariant + temporal property
  4. 发现违反时给反例

真实用

  • AWS—— S3 / DynamoDB / EBS 等核心 service 用 TLA+ 验证（Newcombe 等 2015 paper）
  • Microsoft Azure—— Cosmos DB 等
  • MongoDB—— replication protocol
  • Tendermint / Diem—— consensus protocol
  • etcd—— 部分

实战 ROI

AWS 2015 paper 报告：

  • TLA+ 验证后发现多个 production-blocking bug（手工 review + 测试 miss）
  • 1 个 senior engineer × 几周写规约
  • ROI: $$ saved on later P0 incidents

其他工具

工具定位
TLA+ / PlusCal主流，Lamport 推

VerdiCoq 中 Raft 完整证明

生产事故 case study

AWS DynamoDB 2015 outage · Cloudflare 2020 6-hour outage · GitHub 2018 24-hour outage · etcd 2020 issue

AWS DynamoDB 2015 outage

  • "metadata service" 时间漂移导致 partition assignment 错乱
  • 事故 4+ hours US-East
  • Root cause: time-based assumption + 无形式化验证 metadata service
  • 后续 AWS 大规模引入 TLA+ verify metadata service

Cloudflare 2020 6-hour outage

  • BGP 配置错误 + Raft cluster 分裂
  • 主因: 路由配置 + 无 graceful network partition handling
  • Lesson: 网络分区 + 配置 push 双重 failure 才 surface

GitHub 2018 24-hour outage

  • MySQL fail-over + cross-region 主从切换
  • 10s 网络分区 → 两 region 都认自己是 primary
  • 2 region 接受写 → 数据 diverge
  • 恢复花 24h（需要数据对比 + 手工 reconcile）
  • Lesson: 跨 region failover 必有quorum + fencing token，否则 split brain

etcd 2020 issue

  • etcd 在large lease + auth token revalidation 组合下出现 deadlock
  • Lesson: 实际系统的 corner case 多于 paper 描述
🎯 关键 take-aways
  • Paxos 是共识祖先，Raft 是易理解的现代版—— 2026 工业界首选 Raft
  • Multi-Paxos / Fast Paxos / EPaxos / Flexible Paxos 各为特定场景
  • Raft 4 子问题: leader election / log replication / safety / membership change（joint consensus）
  • BFT (PBFT / HotStuff) 容忍 malicious node，permissioned blockchain 用
  • 时间: Lamport / Vector / HLC / TrueTime —— 不同精度 + 不同开销
  • 一致性谱系: Linearizable → Sequential → Causal → Eventual；session guarantees 实用
  • CRDT 无协调 eventually consistent —— collaborative editing / offline-first 必备
  • FLP 不可能性—— 异步 + 1 crash 不能保 termination；Raft/Paxos trade liveness for safety
  • TLA+ 等形式化验证在 AWS / Microsoft / Azure 是关键服务 standard practice
  • 真实事故: AWS / Cloudflare / GitHub 教训 —— 网络分区 + 时间漂移 + 跨 region 是最难
