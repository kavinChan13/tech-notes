01

系统背景与业务目标

背景 · 利益相关者（Stakeholders）地图 · 核心约束

本 Capstone 以车端分布式数据采集与遥测系统为案例。这个系统同时涵盖：嵌入式 + Linux SoC + 分布式 + 实时性 + 安全关键——是练习前 5 篇方法论最好的综合载体。

1.1 背景

某汽车 OEM 正在开发下一代 ADAS（高级驾驶辅助系统）平台。需要一套车端数据采集与遥测系统，满足：

  • 从 CAN / CAN-FD / Automotive Ethernet 采集传感器信号（摄像头、雷达、激光雷达）
  • 按配置规则触发式记录数据（如：TTC < 2s 时记录前后 30s 数据）
  • 本地 Flash 缓冲，联网时选择性上传到云端
  • 云端平台负责数据分析 + 模型训练 + OTA 规则下发
  • 系统必须满足ISO 26262 ASIL-B（不直接控制车辆，但与 ADAS 共存）

1.2 利益相关者（Stakeholders）地图

Stakeholder关心什么关键 QA
ADAS 算法团队数据完整性、触发准确性、采集频率可靠性、时序精度
平台架构师（我）模块边界、ABI 稳定、长期可演进可维护性、可演进性
嵌入式 BSP 团队驱动接口、资源占用、WCET 可预测实时性、确定性
云端平台团队数据格式、上传协议、数据量互操作性
功能安全团队ASIL-B 合规、FFI、故障隔离功能安全、故障检测
数据保护团队哪些数据被记录、数据加密、GDPR信息安全、隐私
验证团队可观测性、可调试、HIL 测试接入可测试性、可分析性

1.3 核心约束

硬约束（必须满足）

  • 目标 SoC：NVIDIA Orin（ARM Cortex-A78 + 16GB RAM）
  • OS：QNX Hypervisor + Linux Guest（ADAS）+ QNX Guest（Safety）
  • 存储：1TB NVMe（边缘本地），每天最多写 50GB
  • C++ 版本：C++17（C++20 等工具链稳定后迁移）
  • ASIL-B 合规：必须通过 ISO 26262 Part 6 audit
  • OTA 可更新：规则配置 + 软件本身均支持 OTA

软约束（需要权衡）

  • 团队规模：6 人 C++ + 2 人 DevOps + 1 人安全
  • 交付节奏：季度里程碑，1.5 年上线
  • 云端带宽：4G/LTE，平均 10 Mbps，费用敏感
  • Flash 寿命：50GB/day × 5 年 = 91.25 TB 写入（NVMe TBW 需匹配）

质量属性场景（ATAM 效用树）

效用树（Utility Tree） · 架构关键点识别

首先用 ATAM 效用树把"系统要好"这个废话转化成可测量的质量属性场景。每个 [优先级, 难度] 的 [H,H] 场景是架构的关键驱动力。

2.1 效用树（Utility Tree）

质量属性场景度量优先级难度
Performance 信号采集延迟：传感器信号到采集引擎处理 P99 ≤ 10ms HH
触发响应：TTC 事件发生到录制开始 ≤ 50ms HM
上传吞吐：上传带宽利用率 ≥ 80% 可用带宽 ML
Reliability 触发事件的数据不丢失（重启后可恢复） RPO = 0（触发后数据） HH
单进程故障不影响其他进程 故障隔离 > 99.9% HM
Safety 遥测系统异常不干扰 ADAS 计算路径 FFI 验证通过 HH
ASIL-B 合规 audit 通过 ISO 26262 Part 6 HH
Modifiability 新增采集协议（如新型传感器） ≤ 5 人天，不影响其他模块 HM
修改采集规则不需要重新发布整个系统 OTA 规则更新 ≤ 5 分钟 HM
Security 本地录制数据加密，防止物理提取 AES-256 加密，密钥在 HSM HM
Testability 在 HIL 环境可完整模拟信号源 可注入模拟信号替换真实硬件 MM

2.2 架构关键点识别

标记为 [H, H] 的场景是架构关键场景，每一个都需要 ADR：

🔴 Risk（风险点）

  • 信号采集 P99 ≤ 10ms：同步 IPC 架构无法满足 → 需要无锁共享内存
  • RPO = 0：进程重启会丢数据 → 需要 WAL / 原子持久化
  • FFI：遥测进程可能污染 ADAS 内存 → 需要进程隔离 + QNX 内存保护

🟠 Tradeoff（权衡点）

  • 低延迟 IPC vs 进程隔离：共享内存延迟低但隔离弱；消息队列隔离强但延迟高 → ADR-001
  • 数据加密 vs 写入速度：AES-256 加密有 CPU 开销 → 选 NVMe 加密（硬件加速）
  • 规则热更新 vs 系统确定性：动态加载规则引入不确定性 → ADR-006

ADR-001：进程间通信选型

Signal Adapter → Rule Engine → Recorder 之间的信号流需要 IPC 方案

ADR-001

选择无锁共享内存环形缓冲区作为信号流 IPC

Accepted
日期：2025-09-15 决策人：平台架构师 + 嵌入式负责人
Context · 上下文

Signal Adapter → Rule Engine → Recorder 之间的信号流需要 IPC 方案。核心需求：P99 延迟 ≤ 10ms（信号采集到规则评估）、进程隔离（一个挂不影响另一个）。系统资源：4 个 CPU 核，8GB 内存可用于遥测系统。

Decision Drivers · 决策驱动因素
  • P99 ≤ 10ms（来自 [H,H] 质量属性场景）
  • 进程崩溃隔离
  • < 5MB IPC 内存占用
  • 零第三方 broker 进程
Considered Options · 候选方案
  1. POSIX 共享内存 + 无锁环形缓冲（自研）
  2. SOME/IP ara::com（AUTOSAR Adaptive）
  3. ZeroMQ inproc/ipc
  4. NATS local broker
Decision · 决策
选择方案 1：POSIX 共享内存 + 无锁环形缓冲（自研）。
Justification · 论证
方案P99 延迟隔离资源成熟度
1. 共享内存（选）~5µs ✅进程隔离 ✅<1MB ✅自研
2. ara::com~100µs完整 ✅10MB+重 ❌
3. ZeroMQ~50µs进程隔离5MBOK
4. NATS~1msbroker 独立15MB ❌OK

SOME/IP/ara::com 资源占用过大；NATS 增加 broker 进程，违反资源约束；ZeroMQ 延迟可接受但比方案 1 高 10x；方案 1 虽然自研，但团队已有 lock-free 经验，代码量 < 500 行。

Positive Consequences · 正面后果
  • 信号延迟 P99 ~5µs，远低于 10ms 要求
  • 零额外进程，资源占用最小
  • 自研代码，对自己有完整控制权
Negative Consequences · 负面后果
  • 自研：需要严格的并发测试（TSan + sanitizers）
  • 进程崩溃可能导致缓冲区状态不一致 → 启动时做 sanity check，重置环形缓冲
  • 无内建背压 → 写满时覆盖旧数据（对实时系统可接受）
Follow-ups · 后续行动
  • 信号总线压力 > 80% 触发自动告警
  • 下一版本（v2）评估是否切换到 POSIX MQ（支持优先级）

ADR-002：数据存储选型

本地存储需要：写密集（触发时最高 500MB/s）、按时间范围读取、支持断电安全、NVMe TBW 寿命约束

ADR-002

使用 RocksDB 作为本地时序数据存储

Accepted
日期：2025-09-20
Context · 上下文

本地存储需要：写密集（触发时最高 500MB/s）、按时间范围读取、支持断电安全、NVMe TBW 寿命约束。

Considered Options · 候选方案
  1. RocksDB（LSM-Tree，Facebook）
  2. SQLite WAL 模式
  3. InfluxDB 本地部署
  4. 自研时序文件（二进制追加）
Decision · 决策
选择方案 1：RocksDB + FIFO Compaction + Snappy 压缩。
Justification · 论证
方案写性能空间效率成熟度复杂度
RocksDB（选）优（LSM）高（Snappy）成熟 ✅中
SQLite中（B-Tree）中成熟 ✅低
InfluxDB优（TSM）高重 ❌高
自研文件优自定义—高（维护）
  • RocksDB LSM-Tree 对写密集场景性能最佳
  • FIFO Compaction 模式天然适合时序数据（按时间删旧数据，无写放大）
  • Snappy 压缩率约 4:1，显著减少 NVMe 写入量（延长寿命）
  • SQLite 在写密集场景性能不足；InfluxDB 资源占用过大
RocksDB 关键配置
compaction_style                   = kFIFO
max_total_wal_size                 = 1 GB
compression                        = kSnappyCompression
write_buffer_size                  = 64 MB
level0_file_num_compaction_trigger = 4
Positive Consequences · 正面后果
  • 写入性能满足触发时峰值需求
  • FIFO 模式自动按时间清理，实现 7 天数据保留
  • WAL 保证断电安全（RPO = 0）
Negative Consequences · 负面后果
  • RocksDB Compaction 在后台持续 IO，需要限制 compaction 带宽防止影响前台写
  • 需要额外配置 HW 加密（RocksDB 本身不加密，由 NVMe 加密层处理）
  • 跨平台移植需要验证（确认目标 SoC 工具链支持 RocksDB）

ADR-003：ABI 边界设计

信号适配器需要支持多种车辆协议（CAN-FD、SOME-IP、ROS2 Topic），这些适配器来自不同供应商，用不同编译器和版本编译

ADR-003

插件接口使用 C ABI + Opaque Handle 模式

Accepted
日期：2025-10-01
Context · 上下文

信号适配器需要支持多种车辆协议（CAN-FD、SOME-IP、ROS2 Topic），这些适配器来自不同供应商，用不同编译器和版本编译。核心系统需要在不重启的情况下加载新适配器。

Decision · 决策
适配器插件接口使用 C ABI（extern "C" + 纯 C 函数表），内部实现用 C++ + Pimpl。
接口定义
// signal_adapter_api.h — 这是唯一的 ABI 合约
typedef struct SignalAdapterDescriptor {
    uint32_t    api_version;    // Must == SIGNAL_ADAPTER_API_VERSION
    const char* name;           // e.g., "can-fd-adapter"
    const char* version;        // semver
    void (*destroy)(struct SignalAdapterDescriptor*);
    int  (*init)(struct SignalAdapterDescriptor*, const char* config_json);
    int  (*subscribe)(struct SignalAdapterDescriptor*,
                      const char** signal_ids, size_t count,
                      SignalCallback callback, void* userdata);
    void (*unsubscribe)(struct SignalAdapterDescriptor*, const char** ids, size_t count);
} SignalAdapterDescriptor;

typedef SignalAdapterDescriptor* (*AdapterCreateFn)(void);
Positive Consequences · 正面后果
  • 跨编译器（GCC / Clang / vendor），无 ABI break 风险
  • 供应商只需实现 C ABI，内部可用任意语言（C / C++ / Rust）
  • 适配器可以独立 OTA 更新，无需升级整个遥测系统
Negative Consequences · 负面后果
  • C ABI 无 RAII；每个适配器需要显式 init/destroy → 封装 RAII wrapper
  • 错误处理只能用返回码（无异常）→ 用 error code + error message out-param
  • 调试时多一层间接（符号名都是 C 的）

ADR-004：可观测性方案

ADR-004

可观测性方案 — DLT（车内）+ OTel（云端）双轨

Accepted
日期：2025-10-10
Decision · 决策
车内可观测性使用 DLT（Diagnostic Log and Trace，AUTOSAR 标准）；
云端可观测性使用 OpenTelemetry（OTLP → Grafana Stack）；
两者通过 Health Monitor 桥接。
分层设计
层级组件与协议
Layer 1
（in-vehicle）
  • DLT for logs（二进制，ECU ID + Context ID 分类）
  • Atomic counters for metrics（无 lock 开销）
  • W3C Trace Context 在 DLT message 头传递（简化版 span）
Layer 2
（cloud gateway）
  • Health Monitor 定期把聚合 Metric 转换成 OTel OTLP 发送给云端 Collector
  • DLT → JSON → Loki（只上报关键事件，不全量）
为什么不在车内直接用 OTel
  • OTel C++ SDK 动态分配，不符合 ASIL-B 约束
  • OTLP gRPC 在网络波动时有不确定延迟
  • 资源：OTel SDK + Collector 需要额外 ~50MB RAM
Positive Consequences · 正面后果
  • 车内轻量（DLT 标准，工具生态完整）
  • 云端统一 Grafana 可视化（所有服务用同一平台）
  • 实现渐进：先 DLT，稳定后再考虑 OTel 迁移
Negative Consequences · 负面后果
  • 双轨维护成本
  • DLT 和 OTel 的 Trace 模型不完全等价，跨域追踪有信息损失

ADR-005：C++ 子集与工具链

ADR-005

采用 AUTOSAR C++14 子集 + C++17 非违规扩展

Accepted
日期：2025-10-15
Decision · 决策
  • 基准：AUTOSAR C++14 Guidelines（基于 MISRA C++）
  • 扩展：C++17 的 std::optional / std::variant / if constexpr（经过审查，无违规）
  • 禁止：动态分配（runtime new/delete）/ 异常 / RTTI / 递归（除静态 constexpr）
  • 工具链：GCC 12.3（Linaro cross）+ Clang 16（代码分析）
  • 静态分析：Coverity（合规检查）+ clang-tidy（代码质量）
内存管理策略
阶段策略
启动阶段允许动态分配（初始化资源池、规则缓存）
运行阶段零动态分配（所有结构从预分配池获取）
自定义分配器etl::pool<T,N>（Embedded Template Library）
Positive Consequences · 正面后果
  • 满足 ASIL-B Clause 8（Software Unit Implementation）
  • 零动态分配 → WCET 可计算
  • Coverity + AUTOSAR checker 覆盖 95% 的合规规则
Negative Consequences · 负面后果
  • std::vector / std::map 无法使用（改用 etl::vector / etl::map）
  • 异常禁用 → 所有 API 返回 etl::expected<T,E>
  • 学习成本：团队需要熟悉 ETL 库

ADR-006：插件化与规则热更新

OTA 更新采集规则时，系统必须继续工作（不允许服务中断）

ADR-006

规则配置 OTA 更新采用原子替换 + 双缓冲策略

Accepted
日期：2025-11-01
Context · 上下文

OTA 更新采集规则时，系统必须继续工作（不允许服务中断）。规则文件是 Protobuf 序列化，版本化。

Decision · 决策
使用原子文件替换 + 双缓冲 Rule Cache。
  1. OTA 下载新规则到 /data/rules/rules_new.pb
  2. 校验签名 + 语法检查（沙盒解析）
  3. 原子 rename：rules_new.pb → rules.pb（POSIX rename 是原子操作）
  4. Rule Engine 监听文件变化（inotify），触发异步重新加载
  5. 双缓冲：加载新规则到 Buffer B；加载完成后原子替换 Buffer A（rule pointer swap）
  6. 旧 Buffer A 延迟释放（等待所有正在执行的 match 完成）
为什么不用 dlopen 动态加载
  • 规则是配置数据（Protobuf），不是代码 → 不需要 dlopen
  • dlopen 引入代码执行不确定性，ASIL-B 下难以验证安全性
Positive Consequences · 正面后果
  • OTA 期间系统不中断（零停机规则更新）
  • 原子替换：断电安全（rename 是原子的）
  • 新规则加载失败 → 保持旧规则不变（graceful degradation）
Negative Consequences · 负面后果
  • 双缓冲内存开销（两份规则同时在内存）
  • inotify 有延迟（约 100-500ms），规则更新有短暂延迟
  • 需要验证"指针原子交换"在目标 CPU 上的内存序（用 std::atomic<RuleSet*>）

11.1 Signal Adapter 接口（C++ 内部）

// 信号采集上下文的 Port 接口（依赖倒置）
// domain/ports/i_signal_source.hpp
class ISignalSource {
public:
    virtual ~ISignalSource() = default;
    virtual etl::expected<void, SignalError>
        subscribe(const etl::string_view& signal_id,
                  std::function<void(const SignalSample&)>) = 0;
    virtual etl::expected<void, SignalError>
        unsubscribe(const etl::string_view& signal_id) = 0;
    virtual etl::optional<SignalSample>
        get_latest(const etl::string_view& signal_id) const noexcept = 0;
};

// 核心领域对象（值对象）
struct SignalSample {
    SignalId  id;           // strong typedef
    Timestamp timestamp;   // 高精度，纳秒级
    SignalValue value;     // std::variant<bool, int32_t, float, double, etl::string<64>>
    Quality quality;       // GOOD / DEGRADED / BAD (故障诊断用)
};

// 适配器包装（ACL：防腐层）
class CanFdSignalSource : public ISignalSource {
    const SignalAdapterDescriptor* adapter_;  // C ABI 插件
    DbcDecoder dbc_decoder_;                 // DBC 文件解码器（ACL）
public:
    etl::expected<void, SignalError>
        subscribe(const etl::string_view& signal_id, ...) override {
        // 把我们的 domain signal_id 翻译成 CAN message ID + bit offset
        auto can_sig = dbc_decoder_.resolve(signal_id);
        if (!can_sig) return etl::unexpected{SignalError::NOT_FOUND};
        // 调 C ABI 适配器
        auto rc = adapter_->subscribe(adapter_, can_sig->ids(), ...);
        return rc == 0 ? etl::expected<void, SignalError>{}
                       : etl::unexpected{SignalError::ADAPTER_ERROR};
    }
};

// RecordingSession：核心聚合（触发一次录制的生命周期）
class RecordingSession {  // Aggregate Root
    SessionId        id_;
    TriggerEventId   trigger_;      // 触发这次录制的事件 ID（跨 Aggregate 只存 ID）
    Timestamp        start_;        // 录制开始（pre-buffer 时间）
    Timestamp        end_;          // 录制结束（触发后 N 秒）
    RecordingStatus  status_ = RecordingStatus::Recording;
    etl::vector<SignalId, 256> channels_;  // 录制的信号通道
    StorageLocation  location_;     // RocksDB 里的 key prefix

    // 不变量：Recording 状态下 end_ > start_
    // 不变量：channels_ 不能为空

public:
    // 工厂方法（用 TriggerRule 决定参数）
    static etl::expected<RecordingSession, DomainError>
        create(TriggerEventId trigger_id, const TriggerDecision& decision,
               SessionId id = SessionId::generate());

    // 业务方法
    etl::expected<void, DomainError> finalize(Timestamp now) noexcept {
        if (status_ != RecordingStatus::Recording)
            return etl::unexpected{DomainError::INVALID_TRANSITION};
        end_ = now;
        status_ = RecordingStatus::Completed;
        return {};
    }
    bool is_active() const noexcept { return status_ == RecordingStatus::Recording; }
};

14.1 依赖方向检查

#!/bin/bash
# scripts/check_arch_deps.sh
# Fitness Function #1: Domain 层不能依赖任何非标准库

# 检查 domain 目录下的 .hpp 不包含 infrastructure / RocksDB / spdlog 等
forbidden=(rocksdb spdlog boost mosquitto DLT)

for lib in "${forbidden[@]}"; do
    if grep -r "#include.*${lib}" src/domain/; then
        echo "❌ FAIL: domain layer includes ${lib}"
        exit 1
    fi
done

echo "✅ PASS: Domain dependency rules"
# Fitness Function #2: ABI 兼容性检查（每次改 signal_adapter_api.h）
# .github/workflows/abi-check.yml
- name: Check Signal Adapter ABI
  run: |
    abi-dumper build/libtelemetry.so -output current.dump
    abi-compliance-checker -lib signal-adapter \
        -old baseline.dump -new current.dump
    # 失败 = ABI break = CI 不通过
# Fitness Function #3: 性能基线（P99 ≤ 10ms）
# tests/perf/signal_flow_bench.cpp
# 每次 PR 运行，延迟超过阈值 fail
BENCHMARK(SignalFlowE2E) {
    // 注入 10000 个 SignalSample，测量到 Rule Engine 处理完的延迟
    auto t0 = now();
    inject_signal(sample);
    auto t1 = rule_engine_.wait_for_match(sample.id);
    auto latency = t1 - t0;

    ASSERT_LT(latency.p99_ns(), 10 * 1000 * 1000);  // 10ms in ns
}
# Fitness Function #4: 禁止 dynamic allocation 在 run phase
# 用 custom allocator 替换 new，运行时检测
namespace {
  std::atomic<bool> run_phase{false};
}

void* operator new(std::size_t n) {
    if (run_phase.load()) {
        spdlog::critical("FATAL: dynamic allocation in run phase! size={}", n);
        std::abort();   // 测试模式下 abort，生产模式下 DLT 告警
    }
    return std::malloc(n);
}

真实项目对比：车端遥测 vs 分布式 RPC 代理

两个系统的核心对比 · 相同方法论，不同决策——关键对比 · 电信项目里的架构反模式——真实技术债 · 从真实项目学到的"方法论无法教你的东西"

同一套架构方法论在不同领域的真实系统上落地，决策过程和权衡结果会有很大差异。本章用OpenStandard RPC 管理代理作为对比案例，帮你理解"方法论是通用的，但应用时必须根据领域约束做调整"。

16.1 两个系统的核心对比

维度车端遥测系统（Capstone）RPC 管理代理（真实）
领域 汽车 ADAS / 车联网 网元管理（OAM）
安全等级 ASIL-B（间接影响驾驶安全） 无 safety 要求（OAM 层，非实时控制）
核心协议 CAN-FD / SOME-IP（信号采集） RPC/YANG over SSH（配置管理）
并发模型 无锁共享内存环形缓冲（信号流） 每 remote-unit 独立 Dispatcher 线程（Scenario 串行）
存储 RocksDB（本地时序数据） Mgmt 信息模型数据库（内存 MO 树，无持久化）
异常使用 禁用（ASIL-B + AUTOSAR 子集） 生产禁用（-fno-exceptions），UT 开启
依赖管理 Conan 2.x（目标） pkg-config + 内部 SDK（公司内部生态）
测试规模 HIL 为主（硬件在环） 377+ UT + 20 MT + 777+ SCT（Python）
代码规模 估计 50K 行（MVP） ~1300 源文件（生产级）

16.2 相同方法论，不同决策——关键对比

C4 建模：两者都适用，受众不同

车端遥测（Capstone）：L1 受众是 OEM 业务方和 ADAS 算法团队；L2 揭示的是"为什么 4 个进程"的隔离决策。

RPC 代理：L1 受众是网元 OAM 架构师；L2 的关键点是"每 remote-unit 独立 PlatformAgent 实例"——这是按业务能力而不是技术层分割的直接体现。

共同结论：L2 图的价值在于让人 10 分钟内理解最重要的架构决策，而不是画出所有技术细节。

DDD Bounded Context：边界的依据不同

车端遥测：按数据生命周期划分——信号采集 / 规则匹配 / 存储 / 上传各自独立。

RPC 代理：按OpenStandard 功能域划分——载波管理 / 告警管理 / 软件管理 / 账户管理。每个 Scenario 子目录就是一个隐式 Bounded Context。

共同结论：划分依据是"语言变化"和"变更节奏"，不是代码行数或技术分层。

Scenario 模式：事件驱动的状态机

RPC 代理的核心并发模式是Scenario 引擎——每个业务操作（创建载波、软件升级）是一个 BaseScenario 派生类，把步骤拆成 deque<function<Status()>> 逐步执行，支持 Waiting 状态等待异步响应。

这和 Capstone 的 Saga 编排模式本质相同——把长事务拆成步骤，每步可重试/回滚。两者都避免了阻塞线程等待远端响应。

区别：Scenario 是进程内单线程串行；Saga 通常是跨服务异步。但核心思想相同。

ADR：决策的动机不同，但写法相同

车端遥测 ADR-001：选共享内存 IPC → 核心驱动是 P99 延迟 ≤ 10ms。

RPC 代理 ADR-T01：选每 RU 独立 Dispatcher 线程 → 核心驱动是 remote-unit 的时序约束（RPC 操作必须串行）。

共同结论：ADR 的价值不在于格式，而在于把驱动决策的质量属性场景（[H,H]）写清楚，让未来人能理解"为什么不能换成另一种方式"。

下面是从这个真实电信项目里识别出的架构层技术债，对照 Capstone 的 Fitness Function 概念，每一条都可以写成自动化检查：
问题类别对应原则可自动检查？
GLOB_RECURSE 收集所有 .cpp 构建工程化 CMake Anti-Pattern ✅ CI 检查 CMake 语法
set(CMAKE_CXX_FLAGS ...) 全局设置 构建工程化 现代 CMake Target-Centric ✅ cmake-lint 检查

16.4 从真实项目学到的"方法论无法教你的东西"

学了所有方法论之后，真实项目里还有几件事是书本无法完全传授的：

遗留系统的"约束继承"

RPC 代理使用 pkg-config 而不是 Conan，因为它依赖内部 SDK（无法放进公共包仓库）。这是一个组织约束，架构师无法"设计出来"，只能适应并在约束内做最优选择。

Capstone 里的 Conan 方案在纯绿地项目里是理想选择，但在企业内网封闭环境下，pkg-config + 内部 Artifactory 是更现实的方案。

协议的业务侵入

RPC/YANG 协议的数据模型直接影响了 Bounded Context 的划分——OpenStandard-fm.yang、OpenStandard-swm.yang 每个 YANG 模块基本对应一个业务域。这是标准协议驱动的 BC 划分，而不是从业务分析推导出来的。

这告诉我们：在有强标准的行业（分布式、汽车 AUTOSAR、医疗 HL7），标准本身就是最重要的 Bounded Context 输入，要先理解标准再做 DDD 分析。

总结：方法论的价值是"提供语言和框架"，而不是"替代你对领域的理解"。 C4、DDD、ADR、ATAM——这些工具让你能把脑子里的想法清晰地表达出来、让别人理解、让团队对齐。但真正决定架构质量的，是你对业务领域（功能安全 / 实时信号处理）、技术约束（内部 SDK / 硬件寿命）、组织现实（团队规模 / 工期压力）的综合判断。 两个系统的共同经验：
  1. 好架构不是一开始就完美——而是有清晰的技术债清单和偿还计划
  2. 每个"反常识"的决策背后都有一个未被记录的约束——写 ADR 就是把这个约束永久化
  3. Fitness Function（自动化架构检查）是防止腐化的唯一可持续手段

FRAMEWORK 系统设计题的标准 5 步框架（先记住这个，再做题）
速记所有系统设计题通用
STEP 1
需求澄清（5-7 分钟）

不要立刻开始画图！先问清楚：

  • 功能需求：核心用例是什么？读多还是写多？
  • 规模：用户量？QPS？数据量？保留时间？
  • 质量属性：P99 延迟？SLA？一致性要求？
  • 约束：地域？预算？团队规模？
STEP 2
容量估算（3-5 分钟）

用BOTE（Back-of-the-envelope）估算：QPS 高峰 = DAU × 平均请求次数 / 86400 × 峰值倍数；存储 = QPS × 单条大小 × 保留秒数。面试官想看你能用数字思考，而不是一上来就微服务。

STEP 3
高层设计（10-15 分钟）

画 C4 L2 Container 图：客户端 → API Gateway → Service → DB / Cache / MQ。从最简单的方案开始（先单体再分布式），逐步演进。

STEP 4
深入设计（10-15 分钟）

面试官会挑一个组件让你深入："这个服务怎么扩展？" / "数据库怎么分片？" / "缓存怎么一致？"。这里展示你对 #A01-#A05 知识的运用。

STEP 5
权衡与演进（5-10 分钟）

主动指出：(1) 瓶颈在哪；(2) 未来怎么演进（从 1k QPS 到 100k QPS）；(3) 什么时候要重新设计。展示架构师的远见。

MOCK-1 设计一个分布式限流系统（Distributed Rate Limiter），支持 1M QPS、跨数据中心、精确到用户维度
P0主题：分布式系统常见公司：Cloudflare/Stripe/AWS
STEP 1 · 澄清问题
  • 限什么？(1) 用户级（user_id）；(2) IP 级；(3) API 级
  • 限流算法？(1) 固定窗口；(2) 滑动窗口；(3) Token Bucket；(4) Leaky Bucket
  • 容忍误差？（精确 vs 近似）
  • 跨数据中心一致性要求？
STEP 2 · 容量估算

1M QPS，每次限流检查 1ms → 需要 1000 个工作单元；用户量 100M，活跃 10M → 状态存储 ~1GB

STEP 3 · 高层设计

核心方案：Token Bucket 算法 + Redis 集群 + 本地缓存

Client → API Gateway → Rate Limit Service → Redis Cluster
                              ↓
                     Local LRU Cache (减少 Redis 调用)
                              ↓
                     Decision: ALLOW / REJECT
STEP 4 · 关键设计点
  1. 算法选 Token Bucket：支持突发流量，比固定窗口更精确
  2. Redis 用 Lua 脚本 原子地"取 token + 扣减 + 返回剩余"，避免竞态
  3. 分片：按 user_id hash 分布到 Redis 集群，单点 QPS 受限于 Redis 上限（10W QPS / 节点 × N 节点）
  4. 本地缓存：每个网关节点本地 LRU 缓存最近用户的 token 余量，异步同步到 Redis——牺牲精度换性能
  5. 跨数据中心：每个 DC 有独立 Redis Cluster + 局部限流；全局限流通过 异步聚合 实现（接受秒级延迟）
STEP 5 · 权衡
  • 瓶颈：Redis QPS、网络延迟（每次 1ms × 1M = 1000 CPU 秒）
  • 精度 vs 性能：本地缓存会让用户多用 1-2% 配额，但 QPS 翻 10x
  • 演进：> 10M QPS 时考虑 cell-based 架构（按用户分租户，每个租户独立 Redis）
面试加分

提到 Cloudflare 的"Sliding Window Counter"——结合固定窗口的简单性和滑动窗口的精度，用加权平均近似。这是工业界常用的实用算法。

MOCK-2 设计一个类似 Kafka 的分布式消息队列：百万 QPS、至少一次语义、可重放
P0主题：分布式 + 存储常见公司：LinkedIn/Confluent/字节跳动
STEP 1 · 澄清
  • 消息保留多久？（影响存储设计）
  • 需要顺序保证吗？（影响分区设计）
  • Exactly-Once 还是 At-Least-Once？
  • Consumer 类型？（Push vs Pull）
STEP 3 · 高层设计（Kafka 思路）
Producer → Broker (Leader Partition) ──→ Followers (副本)
                    ↓                          ↑
              Local Log (append-only)    Replication
                    ↑
            Consumer (Pull) - Offset
            Consumer Group - Auto Rebalance
STEP 4 · 关键设计点
  1. Topic 分区：每个 Topic 多个 Partition；写入按 key hash 路由到 Partition；同 key 顺序保证，不同 key 并行
  2. 顺序写日志：每个 Partition 是 append-only 日志，顺序 IO 让单节点支撑 1GB/s 写入（SSD 顺序写极快）
  3. 副本机制：每个 Partition 有 N 个副本，Leader 负责读写，Follower 拉取复制；ISR（in-sync replicas）管理同步副本集
  4. At-Least-Once：Producer acks=all 等所有 ISR 写入；Consumer 提交 offset 在处理之后（消息可能重处理）
  5. 可重放：消息按时间保留（不删除），Consumer 可以从任意 offset 重读
  6. 水平扩展：加 Broker → 新建 Partition / 迁移 Partition
STEP 5 · 权衡
  • Partition 数量：太少 → 并行度不够；太多 → 元数据爆炸 + Rebalance 时间长。典型 100-1000 个
  • 副本数：3 副本是平衡点（容错 1 节点失败 + 不浪费）
  • 磁盘 vs 内存：消息持久化到磁盘，利用 OS Page Cache 加速读取（不用应用层缓存）
面试加分

提到 Pulsar 的分层存储（热数据 BookKeeper + 冷数据 S3）——比 Kafka 更现代的设计。或者讨论 Kafka KRaft 模式（去 ZooKeeper）的演进。

MOCK-3 设计一个高频交易系统的订单匹配引擎：P99 < 100µs、确定性、高可靠
P0主题：低延迟 C++ 系统常见公司：HFT 公司、券商技术、Citadel
STEP 1 · 澄清
  • 多少符号（symbols）？每秒订单数？
  • 需要支持哪些订单类型？（限价/市价/止损）
  • 是否需要 wire-to-wire 延迟保证？
STEP 3 · 高层设计
Network NIC (DPDK / Solarflare) → Decoder
                                       ↓
                              Order Book (per symbol)
                                       ↓
                                Matching Engine
                                       ↓
                              Trade Output / WAL
                                       ↓
                                  Risk Check
                                       ↓
                              Market Data Publisher
STEP 4 · 关键设计点（低延迟 C++ 核心）
  1. Kernel bypass：DPDK / Solarflare OpenOnload，绕过内核 TCP/IP 栈，从 NIC 直接到应用，延迟从 50µs 降到 2µs
  2. Single-threaded matching engine per symbol：避免锁，单线程跑完整个匹配；多个符号并行多线程
  3. Lock-free 数据结构：SPSC ring buffer 在 IO 线程和 matching 线程之间传递订单
  4. Order book 数据结构：通常用跳表或自研价格层级数组，根据典型订单簿深度优化
  5. CPU pinning + isolation：matching 线程绑定特定 CPU 核（isolcpus），关中断，关 NUMA balancing
  6. 预分配内存：避免运行时 malloc；用对象池
  7. WAL 写日志：每笔成交先写到磁盘的内存映射文件（mmap），崩溃后可恢复
STEP 5 · 权衡
  • 延迟 vs 可靠性：完全的 WAL fsync 增加延迟；HFT 通常用异步 WAL + 主备热备方案
  • 性能 vs 可维护性：极致优化代码极难维护，必须有专门团队
  • FPGA 加速：> 10M orders/s 时，把市场数据解码下沉到 FPGA
常见陷阱

不要说"用微服务"或"用 Kafka"——HFT 系统的设计哲学是"单进程极致优化 + 简单冗余"，加任何网络跳数都不可接受。

面试加分

能引用具体的延迟分布：典型 wire-to-wire 2-10µs（kernel bypass），50ns（FPGA）。能讨论 LMAX Disruptor 的设计思想（环形缓冲 + 单写者模型）。
