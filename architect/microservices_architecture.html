01

单体 vs 微服务：不是越微越好

微服务是组织与规模问题的解，不是技术时髦

微服务把一个大应用拆成一组围绕业务能力、独立部署、独立数据、通过网络通信的小服务。它的核心收益是团队自治与独立演进——不同团队能各自开发、部署、扩容而互不阻塞。但它把「函数调用」换成了「网络调用」，引入了分布式系统的全部复杂性。

维度单体（Monolith）微服务（Microservices）
开发/部署简单，一个仓库一次部署独立部署，团队自治，但需 CI/CD 基建
调用进程内函数调用（快、可靠、事务简单）网络调用（有延迟、会失败、需分布式事务）
扩展整体扩（浪费）按服务独立扩（精准）
技术栈统一可异构（也可能失控）
故障隔离一处崩全崩可隔离（若设计得当）
调试/排障简单（一处日志）难（跨服务链路追踪）
🧭 什么时候该上微服务
  • 团队规模：一个团队维护不动一个大单体、部署互相阻塞时（康威定律：架构映射组织）。
  • 业务复杂度：不同模块有截然不同的扩展/迭代节奏。
  • 反过来：初创/小团队/需求不清时，先单体（Modular Monolith），边界稳定后再拆——「Monolith First」是行业共识。
生活类比：大厨房 vs 美食广场

单体像一个大厨房：所有菜一个团队做，沟通快、协调易，但人一多就挤、一个灶台坏影响出所有菜。微服务像美食广场：每个档口（服务）独立经营、各自扩张、互不干扰，但顾客要凑齐一顿饭得跑好几个窗口（跨服务调用），且需要统一的收银/找位（网关/服务发现）。档口分得太细，顾客就累了——这就是过度拆分。

映射：档口=服务 · 跑多个窗口=跨服务调用 · 统一收银=API 网关 · 分太细=过度拆分
02

拆分边界：最难也最关键

DDD 限界上下文

微服务成败 80% 取决于边界划得对不对。划错了会得到「分布式单体」——服务间高度耦合、改一个要动一串、还多了网络开销，比单体更糟。领域驱动设计（DDD）的限界上下文（Bounded Context）是划分边界的主流方法论。

📐 划分原则
  • 按业务能力，不按技术分层：拆成「订单服务/库存服务/支付服务」，而不是「DAO 服务/逻辑服务」。
  • 高内聚低耦合：一个业务变更尽量只落在一个服务内（避免跨服务改动）。
  • 限界上下文：同一个词在不同上下文含义不同（「商品」在售卖上下文 vs 仓储上下文关注点完全不同）——沿这些语义边界切。
  • 数据自治：服务应拥有自己的数据，不共享库（见 §4）。
  • 合适的粒度：不是越小越好；一个服务应能被一个小团队完整拥有（two-pizza team）。
怎么找边界

用 Event Storming（事件风暴）：把业务里发生的领域事件（「订单已创建」「已支付」「已发货」）贴满墙，观察哪些事件、命令、数据天然聚在一起——聚成团的就是一个限界上下文，团与团之间的稀疏连接就是服务边界。边界应落在业务耦合最弱的地方。

更完整的领域建模与战略/战术设计见 架构方法论。

03

服务间通信：同步还是异步

通信方式决定耦合度
维度同步（REST / gRPC）异步（消息 / 事件）
模式请求-响应，调用方等结果发消息即返回，解耦时间
耦合时间耦合（下游必须在线）时间解耦（下游可稍后处理）
一致性易做强一致但易级联失败最终一致，天然削峰抗峰
复杂度低，易调试高（顺序/重复/顺序性/可观测）
适用需即时结果的查询/命令事件通知、解耦、削峰、跨服务数据同步
🔗 实践建议
  • 查询/需即时反馈：同步（内部 gRPC，对外 REST，详见 API 设计）。
  • 状态变更通知/解耦：异步事件（Kafka/RabbitMQ）——「订单已创建」事件让库存、积分、通知服务各自订阅处理。
  • 少用同步链式调用：A→B→C→D 同步串联，延迟累加、任一挂全挂。能异步/并行就别串。
  • 事件驱动：发布领域事件而非直接命令下游，降低耦合（发布者不需知道谁在消费）。
同步像打电话，异步像发短信

同步调用像打电话：对方必须此刻接听，你得一直等；对方不在你就办不成事。异步消息像发短信：发出去就能干别的，对方有空再处理、也能积压慢慢消化。前者及时但脆弱耦合，后者解耦抗峰但要接受「稍后才处理」。

04

数据管理：每服务一库

Database per Service

微服务最重要也最反直觉的原则之一：每个服务独占自己的数据库，其它服务不能直接访问。共享数据库会让服务在数据层紧耦合——改个表结构影响一片，等于没拆。

🗄️ 原则与代价
  • 数据私有：只能通过对方的 API/事件访问其数据，不能直连它的库。
  • 好处：服务可独立选型（订单用 MySQL、搜索用 ES）、独立演进 schema、独立扩展。
  • 代价：没有跨库 JOIN 和跨库事务了——这是微服务数据的核心难题。
难题解法
跨服务查询（原来一个 JOIN）API 组合（网关/BFF 聚合多服务）或 CQRS（构建只读物化视图）
数据同步/冗余订阅领域事件维护本地副本（如订单服务冗余商品名快照）
跨服务一致性Saga / 事件驱动最终一致（见 §5）
报表/分析CDC 汇聚到数仓统一分析，不在线上跨库查
05

分布式事务：Saga 与最终一致

放弃 2PC，拥抱补偿

「下单」要跨订单、库存、支付三个服务改数据——没有跨库事务怎么保证一致？传统 2PC（两阶段提交）强一致但会阻塞、可用性差、协调者是单点，互联网基本不用。主流是Saga：把一个大事务拆成一串本地事务 + 失败时的补偿操作，换取最终一致。

方案一致性特点
2PC / XA强一致阻塞、协调者单点、可用性差——不推荐
TCC（Try-Confirm-Cancel）较强业务侵入大（每步写三个方法），资金类可用
Saga最终一致本地事务 + 补偿；编排（Orchestration）/ 协同（Choreography）两种风格
本地消息表 / 事务消息最终一致保证「本地事务 + 发消息」原子，配 Saga 用
🎭 Saga 两种风格
  • 编排式（Orchestration）：一个中心协调器按步骤调各服务，失败则反向调补偿。逻辑集中、易理解易监控，但协调器是重点。
  • 协同式（Choreography）：各服务订阅事件、自主反应，无中心。松耦合，但流程分散、难追踪（易变「事件意大利面」）。
  • 补偿要幂等：补偿可能重试，须幂等；且要处理「补偿也失败」（告警 + 人工兜底）。
  • 注意 Saga 没有隔离性：中间态对外可见，需业务上处理（如订单「处理中」状态）。

分布式一致性、共识与故障模型的底层原理见 分布式基础。

06

API 网关：统一入口

把横切关注点从服务里抽出来

客户端不该直接调几十个后端服务（要处理服务地址、认证、协议差异）。API 网关作为统一入口，把横切关注点集中处理。

🚪 网关职责
  • 路由：把请求转发到对应后端服务。
  • 认证鉴权：统一校验身份，后端服务信任网关。
  • 限流熔断：入口过载保护（见 限流熔断降级）。
  • 协议转换：对外 REST/GraphQL，对内 gRPC。
  • 聚合：一次请求聚合多服务结果（BFF）。
  • 其它：日志、监控埋点、灰度、缓存、请求整形。
BFF：给每类前端定制网关

BFF（Backend For Frontend）是网关的一种：为 Web / App / 第三方各配一个专属聚合层，各自按需裁剪与聚合数据。就像酒店的前台——客人不用直接找客房部、餐饮部、保洁部，前台统一接待并协调各部门，还能针对散客/团客/VIP 提供不同服务。

映射：前台=网关 · 各部门=后端服务 · VIP 专属接待=BFF

⚠️ 注意别把网关做成新的「单体」——业务逻辑不该塞进网关，它只做编排与横切。

07

服务发现与服务网格

服务多了怎么找、怎么治理

服务实例会动态扩缩、重启、迁移，IP 不固定。服务发现解决「A 怎么找到 B 的实例」：

  • 注册中心：服务启动时注册到 etcd / Consul / Nacos / Eureka，调用方查询可用实例 + 健康检查剔除坏节点。
  • 客户端发现 vs 服务端发现：前者调用方自己拿列表 + 负载均衡；后者经 LB/网关转发。
  • K8s：内置 Service + DNS + kube-proxy 做服务发现与负载均衡，是云原生事实标准。
🕸️ 服务网格（Service Mesh）

当服务治理逻辑（重试、熔断、mTLS、流量灰度、追踪）散落在各语言的 SDK 里难以统一时，Service Mesh（如 Istio/Linkerd）把这些下沉到Sidecar 代理（与业务容器并列的 Envoy）：

  • 数据面：Sidecar 拦截所有进出流量，做负载均衡、mTLS、重试、熔断、指标。
  • 控制面：统一下发策略与配置。
  • 收益：治理与业务解耦、跨语言统一、零侵入。代价：每个 Pod 多一跳延迟 + 资源开销 + 运维复杂度。
  • 权衡：中小规模用 SDK/网关够了；服务数量大、多语言、需精细流量治理时上 Mesh。
08

可观测与弹性：微服务的生存基建

拆了就必须能看清、扛得住

单体的一个 bug 看一处日志就行；微服务一个请求跨 5 个服务，没有可观测就是盲人摸象。这些是微服务的必备基建，不是可选项：

  • 分布式追踪：全链路 trace_id 串起跨服务调用（OpenTelemetry / Jaeger），定位是哪一跳慢/错。
  • 集中日志：各服务日志汇聚 + trace_id 关联检索。
  • 指标监控：RED（Rate/Error/Duration）+ 饱和度，每服务 + 全链路。
  • 弹性：超时、重试、熔断、降级、隔离——防级联失败（详见 限流熔断降级）。

完整的可观测三支柱（Trace/Log/Metric）与实践见 可观测性工程。

09

反模式与迁移策略

别掉进这些坑
🚫 常见反模式
  • 分布式单体：拆了但服务间强耦合、必须一起部署——最糟结果，边界划错的标志。
  • 共享数据库：多服务连同一个库，数据层耦合，等于没拆。
  • 过度拆分（纳米服务）：拆太细，网络开销 + 运维成本压垮收益。
  • 同步长链：A→B→C→D→E 串行同步调用，延迟累加、脆弱。
  • 缺基建就上微服务：没有 CI/CD、可观测、自动化，微服务运维会失控。
🪓 迁移：绞杀者模式（Strangler Fig）

把单体改造成微服务不要「大爆炸重写」（风险极高）。用绞杀者模式：在单体前放一层路由，逐个把功能剥离成新服务、把流量慢慢切过去，老单体功能逐步「被绞杀」直至下线。每步可回滚、风险可控。

10

关键 take-aways

🎯 一页速记
  • 微服务解决的是团队自治与规模问题，代价是分布式复杂性——Monolith First。
  • 成败看边界：按业务能力 + DDD 限界上下文划，别按技术分层，别划出分布式单体。
  • 通信：查询用同步（gRPC/REST），状态变更用异步事件解耦；少用同步长链。
  • 每服务一库，不共享数据库；跨服务查询用 API 组合/CQRS。
  • 分布式事务用 Saga + 补偿（最终一致），别用 2PC。
  • 网关集中认证/限流/路由/聚合；别把它做成新单体。
  • 服务发现（注册中心/K8s）+ 按需 Service Mesh；可观测与弹性是必备基建。
  • 迁移用绞杀者模式，别大爆炸重写。
