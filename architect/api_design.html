01

API 是契约，不是实现细节

一旦发布就很难改——设计时就要考虑演进

API（Application Programming Interface）是服务对外暴露的契约：调用方依赖它编程，一旦有人在用，你就不能随意破坏它。好的 API 设计追求四个目标：易用、一致、可演进、难以误用。坏 API 的代价是长期的——它会渗透进所有调用方的代码，改一个字段可能要协调几十个团队。

生活类比：餐厅菜单

API 就像餐厅的菜单：顾客（调用方）照着菜单点菜，不关心后厨（实现）怎么做。菜单要命名清晰（「宫保鸡丁」而非「菜品 A」）、分类合理、价格明确（返回结构稳定）。后厨可以换厨师、改工艺（重构实现），但只要菜单不变，顾客就无感知。一旦你把「宫保鸡丁」下架或改成别的味道，所有习惯点它的老顾客都会投诉——这就是破坏性变更。

映射：菜单=接口契约 · 后厨=实现 · 换厨师=重构 · 下架菜品=breaking change
🧭 设计原则（贯穿全文）
  • 面向调用方：以业务语义而非数据库表结构组织接口。
  • 一致性：命名、分页、错误格式全局统一，降低学习成本。
  • 最小惊讶：行为符合直觉，GET 不产生副作用，DELETE 幂等。
  • 为演进而设计：预留扩展、只做加法、破坏性变更走版本。
02

风格选型：REST / gRPC / GraphQL

没有银弹，看场景

三种主流风格解决不同问题。选型的第一问是：谁在调用？浏览器/第三方 → REST 或 GraphQL；内部服务间高性能调用 → gRPC。

维度REST (HTTP+JSON)gRPC (HTTP/2+Protobuf)GraphQL
数据形态资源为中心方法/RPC 为中心查询为中心
传输/编码HTTP/1.1 + 文本 JSONHTTP/2 + 二进制 Protobuf（紧凑快）HTTP + JSON
性能一般高（二进制 + 多路复用 + 流式）一般（解析复杂）
浏览器友好✓ 原生✗ 需 gRPC-Web✓
取数灵活易过度/不足获取固定按需取字段，解决 over/under-fetch
典型场景公开 API / CRUD内部微服务、低延迟聚合多源、前端驱动
⚖️ 决策要点
  • 对外公开 API：REST 生态最成熟、调试最方便、缓存友好（走 HTTP 语义）。
  • 内部服务间：gRPC——强类型契约（.proto）、代码生成、高性能、支持流式。
  • 前端聚合多后端：GraphQL——一次请求取全所需、避免 N 次往返；代价是缓存与限流更难、有 N+1 查询陷阱。
  • 三者可共存：内部 gRPC + 对外 REST/GraphQL 网关转换。
03

RESTful 设计规范

资源 + 动词 + 状态码

REST 的核心是用 URL 表达资源（名词），用 HTTP 方法表达动作（动词）。把动词塞进 URL（/getUser?id=1）是最常见的反模式。

操作坏例 ❌好例 ✅
查用户GET /getUser?id=1GET /users/1
建用户POST /createUserPOST /users
改用户POST /updateUserPUT /users/1（全量）/ PATCH /users/1（部分）
删用户GET /deleteUser?id=1DELETE /users/1
子资源GET /getUserOrders?uid=1GET /users/1/orders
📐 方法语义（安全性 / 幂等性）
  • GET：只读，安全（无副作用）+ 幂等，可缓存。
  • PUT：全量替换，幂等（重复调用结果一致）。
  • DELETE：删除，幂等（删已删的返回 204/404，不报错重复删）。
  • POST：创建/触发，非幂等（重复会创建多条）——见下一节幂等设计。
  • PATCH：部分更新，是否幂等取决于语义（如 increment 就不幂等）。

命名约定：资源用复数名词（/users）、小写 + 连字符、层级不超过 2-3 层；非 CRUD 的动作型接口可用子资源表达（POST /orders/1/refund）。

04

错误处理与状态码

让调用方能程序化处理错误

正确使用 HTTP 状态码 + 结构化错误体，是专业 API 的分水岭。不要 200 里塞 {"code": 500}——让 HTTP 层就能区分成功失败。

类别码含义
2xx 成功200 / 201 / 204OK / 已创建 / 无内容
4xx 客户端错400 / 401 / 403 / 404 / 409 / 422 / 429参数错 / 未认证 / 无权限 / 不存在 / 冲突 / 语义校验失败 / 限流
5xx 服务端错500 / 502 / 503 / 504内部错 / 网关错 / 不可用 / 超时

结构化错误体应包含机器可读的 code + 人类可读的 message + 可选的字段级细节 + trace_id（便于排障）：

{
  "error": {
    "code": "ORDER_ALREADY_PAID",
    "message": "订单已支付，无法重复支付",
    "details": [{ "field": "order_id", "issue": "state conflict" }],
    "trace_id": "a1b2c3d4"
  }
}
⚠️ 常见坑
  • 4xx vs 5xx 混淆：客户端传错参数是 400，别返回 500（会误触发告警/重试）。
  • 错误码要稳定：code 是契约的一部分，调用方按它分支；message 可随时改文案。
  • 不泄露内部信息：5xx 别把堆栈/SQL 返回给外部。
05

幂等性设计

网络会重试，你的写接口扛得住吗

分布式环境下，请求超时不等于失败——可能服务端已处理，只是响应丢了。客户端重试就会导致重复下单、重复扣款。幂等性保证「同一操作执行一次和多次效果相同」。

生活类比：电梯按钮

电梯的楼层按钮是幂等的：你焦急地按 10 次「5 楼」，电梯也只去一次 5 楼。而「加 1」这种操作不幂等——按 10 次就加了 10。写 API 要设计成电梯按钮，而不是计数器。

映射：按 5 楼=幂等写 · 按 10 次仍去一次=重试安全
🔑 实现方式
  • 幂等键（Idempotency-Key）：客户端为每次业务操作生成唯一 key 放请求头；服务端用「唯一约束 / 去重表」记录已处理的 key，重复请求直接返回首次结果。Stripe 就是这么做的。
  • 业务唯一约束：如订单号唯一索引，重复插入直接冲突返回既有订单。
  • 状态机：只允许合法状态流转（待支付→已支付），重复支付命中「已支付」直接返回成功。
  • 乐观锁 / 版本号：UPDATE ... WHERE version = N，防并发覆盖。

补充：GET/PUT/DELETE 天然幂等，重点是给 POST 加幂等键。幂等键要有过期时间，并注意「同 key 不同参数」应报冲突而非返回旧结果。

06

分页 / 过滤 / 排序

列表接口的三个必答题

任何返回列表的接口都必须分页——否则数据一多就拖垮服务与网络。分页有两种主流方案：

方案Offset 分页游标（Cursor / Keyset）分页
形式?page=3&size=20 / ?offset=40&limit=20?cursor=xxx&limit=20
实现LIMIT 20 OFFSET 40WHERE id > last_id LIMIT 20
深翻性能✗ 差（OFFSET 越大越慢，要扫过前面所有行）✓ 稳定（走索引定位）
数据漂移✗ 会重复/漏（翻页间有增删）✓ 稳定
能否跳页✓ 可跳到第 N 页✗ 只能上下翻
📌 选型
  • 后台管理/需要跳页：Offset（数据量可控时）。
  • 信息流/大数据量/无限滚动：游标分页（性能与一致性都更好）——大厂列表接口标配。
  • 过滤/排序：白名单允许的字段（防注入 + 防扫全表），如 ?status=paid&sort=-created_at。
07

版本演进与向后兼容

改 API 的艺术：只做加法

API 会持续演进，但你不能随时打断存量调用方。核心原则：兼容变更直接改，破坏变更走新版本。

兼容变更（可直接做）✅破坏变更（需新版本）❌
新增可选字段 / 新增接口删除或重命名字段
新增枚举值（调用方需容忍未知值）改字段类型 / 语义
放宽输入校验收紧校验 / 新增必填参数
新增可选响应改错误码 / 状态码含义
🔢 版本化策略
  • URL 版本：/v1/users → /v2/users。最直观、缓存/路由友好，最常用。
  • Header 版本：Accept: application/vnd.api.v2+json。URL 干净但调试不直观。
  • 灰度与下线：新旧版本并行 → 引导迁移（Deprecation 头 + 文档）→ 监控旧版流量归零 → 下线。切忌硬切。
Protobuf 的兼容智慧

gRPC 用 Protobuf，天生为演进设计：字段靠编号（field number）而非名字识别。只要不复用已删字段的编号、不改类型，加字段、删字段（保留编号 reserved）都向后兼容。这是「只做加法」原则的工程化体现。

映射：field number=稳定标识 · reserved=防编号复用 · 新增字段=安全
08

认证 / 授权 / 限流

安全与稳定的入口防线
  • 认证（你是谁）：API Key（简单，服务间）、OAuth2 / JWT（用户态，无状态可扩展）、mTLS（服务间强身份）。
  • 授权（你能做什么）：RBAC（角色）/ ABAC（属性）/ scope（OAuth 细粒度）。在网关或服务入口统一校验。
  • 限流：按 key/用户/IP 限速，返回 429 + Retry-After + X-RateLimit-* 头，保护后端。详见 限流熔断降级。
  • 其它：全站 HTTPS、输入校验防注入、CORS、请求大小限制、幂等键防重放。
🧱 分层拦截

典型链路：API 网关做统一认证 + 限流 + 路由 → 服务做业务授权 + 参数校验。把横切关注点（认证/限流/日志/熔断）下沉到网关，业务服务保持轻量。

09

gRPC 深入

内部微服务的高性能选择

gRPC = Protobuf（IDL + 序列化）+ HTTP/2（传输）+ 代码生成（多语言 stub）。为什么内部服务间它比 REST 强：

  • 强契约：.proto 定义即契约，编译期生成客户端/服务端代码，避免手写解析与字段拼错。
  • 性能：二进制编码比 JSON 小且快；HTTP/2 多路复用（一连接跑多请求，避免队头阻塞）+ 头压缩。
  • 四种调用：一元（Unary）、服务端流、客户端流、双向流——原生支持流式（如实时推送、大文件分块）。
  • 拦截器：统一做认证、日志、重试、超时（deadline 传播）。
代价 / 权衡说明
浏览器支持需 gRPC-Web + 代理转换，不如 REST 直接。
可调试性二进制不可肉眼读，需 grpcurl / 反射；REST 用 curl 即可。
基础设施负载均衡需 L7（HTTP/2 感知），传统 L4 LB 会把多路复用打到单实例。

实践：内部 gRPC + 边缘 REST/GraphQL 网关兼得性能与开放性——这是主流微服务栈的标准姿势。参见 微服务架构。

10

关键 take-aways

🎯 一页速记
  • API 是契约——为演进而设计，破坏变更代价极高。
  • 选型：对外 REST、内部 gRPC、前端聚合 GraphQL；可共存。
  • REST：资源做名词、方法做动词、状态码要用对、错误体结构化 + 稳定 code。
  • 写接口必做幂等（幂等键 / 唯一约束 / 状态机），因为网络会重试。
  • 列表必分页——大数据量用游标分页。
  • 演进「只做加法」，破坏变更走版本；Protobuf 用字段编号天然兼容。
  • 认证/授权/限流下沉网关，业务服务保持轻量。
