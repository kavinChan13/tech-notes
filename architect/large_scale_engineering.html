什么是 ABI，为什么比 API 难十倍

API vs ABI：层次对比 · ABI 的六个组成部分 · 为什么 C++ ABI 问题比 C 严重 10 倍 · ABI 问题在工程中的真实场景

API（Application Programming Interface）是代码层面的约定；ABI（Application Binary Interface）是二进制层面的约定。同一份 API 可以有无数种 ABI——这就是 C++ 标准不规定 ABI 却让工程师吃无数苦头的根因。

1.1 API vs ABI：层次对比

API：源码层约定

  • 函数签名（名字 + 参数类型 + 返回类型）
  • 类的公开成员
  • 语义（文档约定的行为）

破坏 API → 编译报错（立刻可知）

// 改了参数类型 → 编译错误
void foo(int x);    // v1
void foo(long x);   // v2 — API break

ABI：二进制层约定

  • 符号名（name mangling 规则）
  • 函数调用约定（参数在寄存器/栈的位置）
  • 对象内存布局（成员偏移、虚函数表结构）
  • 异常处理机制（LSDA / personality routine）
  • 对象生命周期（谁 new 谁 delete）

破坏 ABI → 运行时崩溃 / 静默数据损坏（最难找）

// API 没变，但加了私有成员 → ABI break
class Foo {
public:
    void bar();
private:
    int x_;        // v1: sizeof(Foo)=4
    int x_, y_;    // v2: sizeof(Foo)=8 → 旧代码崩溃
};

1.3 为什么 C++ ABI 问题比 C 严重 10 倍

C 语言的 ABI 相对简单——函数签名 + struct 布局。C++ 在此基础上叠加了：

Name Mangling 复杂

void foo(int) 在 GCC 变成 _Z3fooi；在 MSVC 变成 ?foo@@YAXH@Z。两者不兼容——所以 C++ 动态库跨编译器调用必须走 C 接口。

vtable 复杂

多重继承、虚继承的 vtable 布局极其复杂，各编译器实现不同，版本间也可能不同。

Exception 复杂

LSDA 格式、unwind 机制都是编译器私有实现。异常从一个 SO 飞到另一个，两边格式必须兼容。

模板实例化

std::vector<int> 在两个编译单元都实例化，内存布局必须完全一致；否则跨边界传递就是 UB。

std::string / std::vector 布局变化

GCC 5 改变了 std::string 的内部实现（COW → SSO）。这是一次巨大的 ABI break，影响了所有 Linux 发行版。

inline 函数内联边界

头文件里的 inline 函数由调用方编译——如果库升级了 inline 函数实现，老的调用方拿到的是新 .so 但执行的是旧代码。

1.4 ABI 问题在工程中的真实场景

场景 1：升级了 .so，不重新编译应用

库 v2 在一个类里加了新私有成员（sizeof 变大），应用按 v1 的 sizeof 分配内存，v2 的构造函数写超界 → 堆损坏。

现象：随机崩溃，valgrind 显示 heap corruption，错误在完全无关的位置。

根因：ABI break（data layout）。

场景 2：插件加载后 std::string 跨边界

主程序用 GCC 4.9（old string ABI），插件用 GCC 8（new string ABI）。主程序把 std::string 传给插件 → 内部 refcount / SSO buffer 解析出错 → 崩溃 / 乱码。

现象：只在特定字符串长度触发（short string optimization 的边界）。

根因：ABI break（std::string 布局不同）。

场景 3：virtual 函数跨 SO 调用

基类在主程序，派生类在插件。主程序调用虚函数，vtable 指针来自插件，但 vtable 偏移量按主程序版本计算 → 调用错误的函数或崩溃。

现象：调用基类方法却执行了派生类里完全不同的方法。

场景 4：内存在模块 A 分配，在模块 B 释放

A 和 B 各自链接了不同版本的 CRT（如 MSVC 上 Debug vs Release）。B delete 了 A new 的内存 → heap 损坏。

规则：new/delete 必须在同一个模块里配对，或者模块边界提供显式的 destroy() 方法。

ABI 问题的最危险特征：它让程序在一种情况下正常工作，在稍微不同的条件下崩溃——改了环境变量、换了 .so 路径顺序、加了一个无关依赖……这就是为什么"在我机器上能跑"这句话在 C++ 库开发中如此危险。架构师的责任是在设计阶段就画好边界，避免 ABI 问题发生，而不是事后救火。

tanium C++ ABI 详解：Name Mangling、vtable、Exception

Name Mangling：函数名如何变成符号 · 虚函数表（vtable）的真实内存布局 · 对象内存布局与 ABI · 异常 ABI：LSDA 与 personality routine · Itanium ABI …

Itanium C++ ABI 是 GCC、Clang 在 Linux/macOS 上共同遵守的标准。理解它是理解"为什么改了这里会破坏 ABI"的前提。本章重点讲 Name Mangling 规则、vtable 结构、异常 ABI，以及 MSVC ABI 的主要差异。

2.1 Name Mangling：函数名如何变成符号

C++ 允许函数重载、模板、命名空间——这些特性导致符号名不能像 C 那样直接用函数名。编译器必须把完整的函数签名编码进符号名。

Itanium ABI Mangling 规则（简化版）

// 规则：_Z + 编码后的名字

foo()                       → _Z3foov         // 3=名字长度, v=void
foo(int)                    → _Z3fooi         // i=int
foo(int, double)           → _Z3fooid        // d=double
ns::foo(int)              → _ZN2ns3fooEi    // N...E=namespace
Foo::bar(int)             → _ZN3Foo3barEi   // 类方法
template<> foo<int>()      → _Z3fooIiEv     // I...E=模板参数

// 用工具还原
$ echo _ZN3Foo3barEi | c++filt
Foo::bar(int)

$ nm -D libmylib.so | c++filt | grep bar

MSVC 的 mangling 完全不同

// GCC/Clang (Itanium ABI):
Foo::bar(int)  →  _ZN3Foo3barEi

// MSVC:
Foo::bar(int)  →  ?bar@Foo@@QAEXH@Z

// 结论：GCC 编译的 .so 不能被 MSVC 直链（符号找不到）
// 唯一跨编译器安全的边界：extern "C" 接口

2.2 虚函数表（vtable）的真实内存布局

Itanium ABI 规定了 vtable 的精确结构：

┌─────────────────────────────────────────────────────────────┐
│                    vtable for Derived                       │
├──────────────────────┬──────────────────────────────────────┤
│ offset to top        │ 0 (指向对象起始的偏移，多继承时非0)    │
│ RTTI pointer         │ &typeinfo for Derived                │
│ virtual func slot[0] │ &Derived::foo()     ← 第一个虚函数    │
│ virtual func slot[1] │ &Derived::bar()                     │
│ virtual func slot[2] │ &Derived::~Derived()  (析构)         │
└──────────────────────┴──────────────────────────────────────┘
        ↑
每个对象实例的头 8 字节（64bit）是指向 vtable 的指针
                            │
                            └── 类对象内存布局：
                                [vptr | field1 | field2 | ...]

ABI Break：改变虚函数顺序

vtable slot 顺序改变 → ABI Break
// v1
class Base {
  virtual void foo();   // slot[0]
  virtual void bar();   // slot[1]
};
// v2 — 加了一个新虚函数在中间！
class Base {
  virtual void foo();     // slot[0]
  virtual void baz();     // slot[1] ← 新增！
  virtual void bar();     // slot[2] ← 偏移变了！
};
// 旧调用方用 slot[1] 调 bar() → 实际执行 baz() → 崩溃或静默错误
正确做法：只在末尾追加虚函数，绝对不在已有虚函数之间插入。删除虚函数同样是 ABI break——必须先 deprecate，等一个 MAJOR 版本周期再删。

2.3 对象内存布局与 ABI

// 以下任何改变都可能破坏 ABI：

class Widget {
public:
    void draw();
private:
    int  x_, y_;            // offset 0, 4
    float opacity_;         // offset 8
    std::string name_;      // offset 16 (通常 32 字节 SSO string)
};

// sizeof(Widget) = ?
// → 取决于 alignment、编译器、string 实现
// → v1 是 48 字节，v2 加了一个字段是 56 字节 → ABI break

2.4 异常 ABI：LSDA 与 personality routine

C++ 异常处理是 ABI 的核心组件，涉及三张表：

LSDA

(Language-Specific Data Area)。每个函数有一份，记录：哪些代码范围内有 try block、catch 能抓什么类型、cleanup 怎么做。

Personality Routine

Itanium ABI 规定是 __gxx_personality_v0。每帧展开时调用，决定是否接受异常、是否 cleanup。

Unwind Tables

记录每个函数如何从栈帧恢复寄存器状态（CFI）。每个帧展开的入口。

当异常跨越 SO 边界时，两侧的 personality routine 必须来自同一个 libstdc++/libc++。如果主程序和插件各自静态链接了不同版本的运行时，异常无法正确传播——异常会在边界处被 terminate() 杀死，或者更糟，在错误的帧展开导致栈损坏。

03

ABI 兼容性：如何守护、如何检测、如何发布

防线一：设计时——ABI 稳定的设计原则 · 防线二：检测工具 · 防线三：版本管理策略

守护 ABI 兼容性需要三道防线：① 设计时避免 ABI 不稳定的设计；② 开发时有工具自动检测 ABI break；③ 发布时有版本管理策略。本章给出三道防线的具体落地方法。

3.1 防线一：设计时——ABI 稳定的设计原则

黄金规则：只通过以下方式扩展公开 API

✅ ABI 安全的变更

  • 加非虚方法（末尾或任意位置）
  • 加 static 成员函数
  • 加 static 数据成员（非内联）
  • 在末尾加虚函数（单一继承时）
  • 加新的重载（不影响现有签名）
  • 加新的枚举值（末尾）
  • 加新的类（不影响现有类布局）
  • 改函数实现（不改签名）
  • 修复 bug（不改签名和语义）

✗ ABI 破坏性变更

  • 改类的任何数据成员（加/删/改类型）
  • 改类的大小（sizeof）
  • 改成员对齐
  • 改虚函数的顺序 / 删除虚函数
  • 加/删/改基类
  • 改函数签名（参数类型/返回类型）
  • 改 inline 函数的行为
  • 改现有枚举值的数值
  • 改 #pragma pack
  • 改异常规格（noexcept vs 有 throw）

例外规则：inline 函数的陷阱

// === 头文件里的 inline 函数 ===
class Widget {
public:
    int width() const { return width_; }  // inline！
private:
    int width_;
};

看起来无害，但如果之后加了成员 int padding_ 导致 width_ 偏移从 0 变成 4，这个 inline 函数在旧调用方里读的还是 offset 0——ABI Break。

对策：公开头文件里尽量不 inline 访问成员的函数，放到 .cpp 里。或者用 Pimpl（见第 4 章）把实现完全隐藏。

3.2 防线二：检测工具

① abi-compliance-checker（Linux，最强大）

# 安装
sudo apt install abi-compliance-checker

# 生成 v1 的 ABI dump
abi-dumper libfoo.so.1 -output abi_v1.dump -lver 1.0

# 生成 v2 的 ABI dump
abi-dumper libfoo.so.2 -output abi_v2.dump -lver 2.0

# 比较
abi-compliance-checker -lib foo \
    -old abi_v1.dump -new abi_v2.dump

# 输出 HTML 报告：哪些函数/类/成员发生了 ABI break

② libabigail（GCC 官方工具链）

# abidiff：直接比较两个二进制
abidiff libfoo.so.1 libfoo.so.2

# abicheck：检查是否符合已记录的 ABI
abicheck --abi-file foo.abi libfoo.so.2

# abipkgdiff：比较两个 RPM/DEB 包的 ABI
abipkgdiff libfoo-1.0.rpm libfoo-2.0.rpm

③ 把 ABI 检测加入 CI

# .github/workflows/abi-check.yml
name: ABI Compatibility Check
on: [pull_request]

jobs:
  abi-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 0

      - name: Build base (main branch)
        run: |
          git checkout main
          cmake -B build_base -DCMAKE_BUILD_TYPE=Release
          cmake --build build_base
          abi-dumper build_base/libfoo.so -output base.dump

      - name: Build PR
        run: |
          git checkout ${{ github.sha }}
          cmake -B build_pr -DCMAKE_BUILD_TYPE=Release
          cmake --build build_pr
          abi-dumper build_pr/libfoo.so -output pr.dump

      - name: Compare ABI
        run: |
          abi-compliance-checker -lib foo -old base.dump -new pr.dump
          # 如果有 break，工具以非零 exit code 退出，CI 失败
工程实践：把 ABI 检查设为 PR 的必过项（required check）。任何 ABI break 必须在 PR 描述里说明：是意外的（修复它）还是故意的（提 MAJOR 版本）。这个 Fitness Function 确保 ABI 破坏永远不会悄悄溜进库。

3.3 防线三：版本管理策略

Linux .so 的三级版本命名

libfoo.so        → 开发时链接用的符号链接
libfoo.so.1      → SONAME，dlopen 时用
libfoo.so.1.2.3  → 真实文件（MAJOR.MINOR.PATCH）

# 关系：
libfoo.so    → libfoo.so.1      （符号链接）
libfoo.so.1  → libfoo.so.1.2.3  （符号链接）
变更类型版本号变化SONAME 变化？
bug 修复，ABI 兼容1.2.3 → 1.2.4否
新功能，ABI 兼容（只加）1.2.3 → 1.3.0否
ABI break（改/删）1.x.x → 2.0.0是（libfoo.so.2）

SONAME 变化时，旧的 libfoo.so.1 继续存在——两个 major 版本可以共存。这给下游提供迁移时间。

CMake 里设置正确的版本号

add_library(foo SHARED src/foo.cpp)

# VERSION：真实文件名 (MAJOR.MINOR.PATCH)
# SOVERSION：SONAME 中的版本号 (通常只是 MAJOR)
set_target_properties(foo PROPERTIES
    VERSION   1.2.3
    SOVERSION 1
)

Windows DLL 的版本管理

// 导出宏
#ifdef FOO_BUILDING_DLL
#  define FOO_API __declspec(dllexport)
#else
#  define FOO_API __declspec(dllimport)
#endif

FOO_API void foo_do_something();

// .def 文件：显式控制导出符号（可防意外导出内部符号）
// LIBRARY foo
// EXPORTS
//   foo_do_something
//   foo_create_widget
//   foo_destroy_widget

用 version script 精确控制导出符号（Linux）

# foo.map
FOO_1.0 {
  global:
    foo_do_something;
    foo_create_widget;
    foo_destroy_widget;
  local:
    *;              # 其他所有符号不导出
};

FOO_1.1 {
  global:
    foo_new_feature;  # v1.1 新增的符号
} FOO_1.0;            # 继承 v1.0 的集合

# CMake 里使用
target_link_options(foo PRIVATE
    -Wl,--version-script=${CMAKE_CURRENT_SOURCE_DIR}/foo.map)
Version Script 是精确控制 ABI 表面积的最强手段：只有白名单里的符号才导出，内部实现的改动对外完全不可见。同时，版本标签（FOO_1.0）让你知道每个符号是哪个版本引入的。所有专业级 C++ 库（glibc、libstdc++、Qt）都在用这个机制。
04

impl 与稳定 ABI 的设计模式

Pimpl 惯用法（Pointer to Implementation） · Opaque Handle（不透明句柄）模式 · 抽象接口 + 工厂（virtual 稳定接口） · 三种模式对比选型

除了被动守护，还有主动设计——通过特定的设计模式，把实现细节从公开头文件里彻底隐藏，让 ABI 表面积最小化，实现变化不传播到 ABI。

4.1 Pimpl 惯用法（Pointer to Implementation）

核心思想：公开头文件里只有前向声明的 Impl 类指针，所有实现细节都在 .cpp 里。

// === widget.hpp（公开头，ABI 表面积极小）===
class Widget {
public:
    Widget(int width, int height);
    ~Widget();                            // 必须 user-defined

    // 移动语义（不能 copy Pimpl，除非 Impl 可 copy）
    Widget(Widget&&) noexcept;
    Widget& operator=(Widget&&) noexcept;

    // 公开 API
    void draw();
    void resize(int w, int h);
    int width() const;
    int height() const;

private:
    class Impl;                           // 前向声明，不完整类型
    std::unique_ptr<Impl> pimpl_;         // 只有指针（8 字节，永不变）
};
// sizeof(Widget) = 8，永远！无论 Impl 怎么变。
// === widget.cpp（实现，私有）===
#include "widget.hpp"
#include <SDL2/SDL.h>        // 私有依赖，不泄漏到公开头
#include <vector>
#include "internal_renderer.hpp"

class Widget::Impl {
public:
    int width, height;
    SDL_Renderer* renderer = nullptr;
    std::vector<SDL_Texture*> textures;
    InternalRenderer internal;
    // 加任何成员都不影响 Widget 的 ABI！

    void draw_impl() {
        SDL_RenderClear(renderer);
        // ...
    }
};

// 必须在看到完整 Impl 定义的地方定义 ctor/dtor/move
Widget::Widget(int w, int h)
    : pimpl_(std::make_unique<Impl>()) {
    pimpl_->width = w;
    pimpl_->height = h;
    pimpl_->renderer = SDL_CreateRenderer(/* ... */);
}

Widget::~Widget() = default;          // unique_ptr 自动 delete Impl
Widget::Widget(Widget&&) noexcept = default;
Widget& Widget::operator=(Widget&&) noexcept = default;

void  Widget::draw()             { pimpl_->draw_impl(); }
void  Widget::resize(int w, int h) { pimpl_->width=w; pimpl_->height=h; }
int   Widget::width()  const     { return pimpl_->width; }
int   Widget::height() const     { return pimpl_->height; }

Pimpl 的 ABI 稳定性分析

✅ 以下变化 ABI 不变

  • 给 Impl 加任何成员
  • 给 Impl 加任何方法
  • 改变私有依赖（SDL → Vulkan）
  • 给 Widget 加非 virtual 公开方法
  • 改任何 .cpp 里的实现

因为：Widget 的 sizeof 永远是 8（一个指针），公开头不包含任何 Impl 的定义。

✗ 仍然会破坏 ABI 的变化

  • 给 Widget 加/删数据成员（在 .hpp 里）
  • 给 Widget 加/改 virtual 方法
  • 改 Widget 公开方法签名
  • 改 Widget 的基类

因为：这些都影响 Widget 的公开 ABI 表面积。

Pimpl 的性能优化：fixed_size buffer

// 缺点：Pimpl 每次构造都需要堆分配
// 嵌入式 / 高频创建场景的对策：
class Widget {
    // ...
private:
    class Impl;
    // 对齐的固定大小缓冲区（预先知道 Impl 的最大 sizeof）
    alignas(8) std::byte storage_[64];  // 大小必须 >= sizeof(Impl)
    Impl* pimpl() { return reinterpret_cast<Impl*>(storage_); }
    // 此时 sizeof(Widget) = 64，固定
    // 构造时用 placement new，析构时手动调用 ~Impl()
};
固定 buffer 的代价：如果 Impl 大小超过预分配的 buffer，编译时必须失败。用 static_assert 保证： 
Widget::Widget(...) {
    static_assert(sizeof(Impl) <= sizeof(storage_), "Impl too large");
    new(storage_) Impl{...};
}

4.2 Opaque Handle（不透明句柄）模式

比 Pimpl 更彻底：对外不暴露任何 C++ 类，只暴露一个整数 ID 或 void*。

// === widget.h（纯 C 头，任何语言都能用）===
#ifdef __cplusplus
extern "C" {
#endif

typedef struct Widget_opaque* WidgetHandle;

WidgetHandle widget_create(int width, int height);
void         widget_destroy(WidgetHandle h);
void         widget_draw(WidgetHandle h);
void         widget_resize(WidgetHandle h, int w, int h_size);
int          widget_width(WidgetHandle h);

#ifdef __cplusplus
}
#endif
// === widget.cpp（实现，可用任意 C++ 特性）===
struct Widget_opaque {
    // 完整 C++ 实现，任意修改
    std::unique_ptr<InternalRenderer> renderer;
    std::vector<Layer> layers;
    Widget_opaque(int w, int h) : renderer(make_renderer(w, h)) {}
};

WidgetHandle widget_create(int w, int h) {
    return new Widget_opaque{w, h};
}
void widget_destroy(WidgetHandle h) { delete h; }
void widget_draw(WidgetHandle h) { h->renderer->draw(); }

优势：

  • C ABI，跨语言（Python/Java/Rust/Go 都能 FFI 调用）
  • 跨编译器（MSVC / GCC / Clang）
  • 实现可以随意用 exception / RTTI / 任意 C++ 特性
  • 内部实现任意变化，ABI 永远稳定（只有函数签名是 ABI）

代价：

  • 没有 RAII（调用方必须手动 destroy）→ 可以提供 C++ wrapper 封一层
  • 没有类型安全（void* 或 opaque struct*）
  • 没有 operator overloading

4.3 抽象接口 + 工厂（virtual 稳定接口）

// === public/widget.hpp ===
struct IWidget {
    virtual ~IWidget() = default;
    virtual void draw() = 0;
    virtual void resize(int w, int h) = 0;
    virtual int  width() const = 0;
    // 新功能只在末尾追加虚函数
};

using WidgetPtr = std::unique_ptr<IWidget>;
WidgetPtr create_widget(int w, int h);   // 工厂，跨 SO 安全
// === impl/widget_impl.cpp ===
class WidgetImpl : public IWidget {
    // 完整实现，可随意加私有成员
    std::unique_ptr<InternalRenderer> r_;
public:
    void draw() override   { r_->draw(); }
    void resize(int w, int h) override { r_->resize(w,h); }
    int  width() const override { return r_->width(); }
};

WidgetPtr create_widget(int w, int h) {
    return std::make_unique<WidgetImpl>(w, h);
}
抽象接口模式的 ABI 稳定性规则：
  • 只在 IWidget 末尾追加 virtual 方法
  • WidgetImpl 随意改，对外不可见
  • 工厂函数 create_widget 签名不变
  • 析构必须是 virtual（确保 delete 通过接口也能正确析构 Impl）

4.4 三种模式对比选型

模式ABI 稳定性跨语言性能C++ 易用性适用场景
Pimpl 高 仅 C++ 一次堆分配 好 同编译器、同语言的库
Opaque Handle (C ABI) 最高 所有语言 一次堆分配 需要手动管理 跨编译器、跨语言 FFI
抽象接口 + 工厂 中高（vtable 稳定） 仅 C++ virtual dispatch 好 插件系统、策略替换
嵌入式系统的选型：
  • AUTOSAR Adaptive 推荐 Pimpl：隐藏内部实现、稳定 ABI；但注意堆分配——如果用 PMR allocator 可控
  • 跨域（Classic ↔ Adaptive）推荐 Opaque Handle + C ABI：跨越 AUTOSAR Classic 的纯 C 世界和 Adaptive 的 C++ 世界
  • 禁 virtual 的 ASIL-D 场景：用 CRTP + type tag 替代抽象接口

++20 Modules 深入：原理、迁移策略与工程实战

C++20 Modules 是 C++ 历史上最大的工程化变革——解决了头文件的四大顽疾：重复 include、宏污染、符号泄漏、编译时间爆炸。本章从 Modules 的工作原理入手，给出从 #include 体系渐进式迁移的实战策略，以及在 CMake / GCC / Clang / MSVC 中的当前状态。

5.1 #include 的四大顽疾

顽疾 ① 重复解析

每个 .cpp include 的头文件都被从头解析一遍。100 个 .cpp 都 include <string>，就解析 100 次。

PCH（预编译头）是绕路，不是根本解决。

顽疾 ② 宏污染

include 是文本替换。windows.h 的 min/max、linux/types.h 的 likely 等宏，随着 include 传播污染每个翻译单元。

顽疾 ③ 符号泄漏

include 了 A，A 内部 include 了 B，意外得到了 B 的所有符号。你删了直接 include B 的语句，可能因为 A 还在 include 它而不报错——直到某天 A 改了 include 顺序，就开始报错。

顽疾 ④ 顺序依赖

#include 顺序不同，宏定义不同，行为可能不同——这是不可重现 bug 的温床。

5.2 Modules 的工作原理

// === math.cppm（模块接口单元）===
export module math;           // 声明模块名

import <cmath>;               // import 标准库（不传染调用方）

export namespace math {
    double sqrt(double x) {
        return std::sqrt(x);
    }
    double pow(double base, double exp) {
        return std::pow(base, exp);
    }
    // 没有 export 的：不对调用方可见
    double internal_helper(double x) { return x * 2; }
}

// 宏不会从 module 传出（这是 Modules 最重要的特性之一）
#define INTERNAL_MACRO 42   // 只在这个翻译单元可见
// === main.cpp（使用方）===
import math;                // 不是文本替换，是语义导入

int main() {
    auto r = math::sqrt(2.0);
    // math::internal_helper 不可见（未 export）
    // INTERNAL_MACRO 不可见（宏不传出 module）
    // std::sqrt 也不可见（import cmath 在 module 内部）
    return 0;
}

BMI（Binary Module Interface）

编译器把 .cppm 编译成 BMI（GCC 叫 .gcm，Clang 叫 .pcm，MSVC 叫 .ifc）。消费方 import 时读取 BMI，不需要重新解析源码。

5.3 Module 的四种文件类型

文件类型作用扩展名约定
Primary Module Interface Unit export module name; 定义模块，导出符号 .cppm / .ixx (MSVC)
Module Implementation Unit module name; 实现接口，不导出 .cpp
Module Partition Interface export module name:part; 大模块分片 .cppm
Module Partition Implementation module name:part; 分片的实现 .cpp

大模块用分片（partition）管理

// === geometry.cppm（主接口，聚合分片）===
export module geometry;
export import :point;   // 从分片 re-export
export import :circle;
export import :polygon;

// === geometry-point.cppm（分片接口）===
export module geometry:point;
export struct Point { double x, y; };

// === geometry-circle.cppm（分片接口）===
export module geometry:circle;
import :point;            // 用同一模块的 point 分片
export struct Circle { Point center; double r; };

5.4 渐进迁移策略：Header Units 和 Shim 层

大型代码库不可能一步迁移到 Modules，需要渐进式策略。

策略 1：import + Header Unit（最保守，最快落地）

// 不修改任何现有头文件
// 只改调用方

// 旧写法：
#include <vector>
#include <string>
#include "mylib/widget.hpp"

// 新写法（使用 Header Unit）：
import <vector>;           // 标准库 header unit
import <string>;
#include "mylib/widget.hpp"  // 还没迁移的头文件保留 #include

策略 2：Module Shim（给老头文件包一层 module）

// === shims/widget_module.cppm ===
export module mylib.widget;

// 把老头文件包进来，导出所有内容
export {
#include "mylib/widget.hpp"
}

// 调用方：
import mylib.widget;   // 享受 module 语义，但不需要改 widget.hpp

策略 3：完整迁移路线图

阶段工作内容所需时间估计
Phase 1 切换编译器到支持 Modules 的版本（GCC 14+ / Clang 16+ / MSVC 17.4+） 1-2 周
Phase 2 新增代码全用 Modules；老代码用 Header Unit import 持续推进
Phase 3 把内部 utility 库迁移成 Module（收益最大） 按库大小
Phase 4 把公开 API 头文件迁移成 Module Interface 需要 ABI 规划
Phase 5 全部迁移完成，删除所有 #include 长期目标

"现代 CMake"（2.8 之后，尤其 3.x+）的核心转变是：从"变量驱动"转向"target + property 驱动"。本章涵盖大型项目的完整 CMake 结构、target 属性模型、依赖传播控制、测试 / install / package 集成、以及最常见的 CMake 坑。

6.1 现代 CMake 的核心哲学

❌ 古老 CMake（Avoid）

include_directories(include/)
link_directories(lib/)
add_definitions(-DFOO_ENABLE)
set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -Wall")
add_executable(app src/main.cpp)
target_link_libraries(app foo bar)

全局变量污染所有 target，include 路径乱传播，没法控制依赖范围。

✅ 现代 CMake（Prefer）

add_library(foo STATIC src/foo.cpp)
target_include_directories(foo
    PUBLIC  include/      # 消费方继承
    PRIVATE src/internal/ # 只有 foo.cpp 能看到
)
target_compile_options(foo PRIVATE -Wall -Wextra)
target_compile_definitions(foo PUBLIC FOO_ENABLE)
target_link_libraries(foo PUBLIC bar PRIVATE baz)

所有设置归属于 target，通过 PUBLIC/PRIVATE/INTERFACE 精确控制传播范围。

6.2 PUBLIC / PRIVATE / INTERFACE 传播规则

这是现代 CMake 最重要的概念，必须烂熟于心：

关键字影响当前 target？传播给消费方？使用场景
PRIVATE ✓ 是 ✗ 否 实现细节（内部依赖、编译选项、私有 include）
PUBLIC ✓ 是 ✓ 是 API 依赖（消费方也需要的头文件、链接库）
INTERFACE ✗ 否 ✓ 是 Header-only 库；消费方需要但自己不直接用

典型例子：多层依赖

# domain 层：纯头文件接口库
add_library(domain INTERFACE)
target_include_directories(domain INTERFACE include/)
target_compile_features(domain INTERFACE cxx_std_20)

# application 层：使用 domain，内部使用 spdlog
add_library(application STATIC src/application/...)
target_link_libraries(application
    PUBLIC  domain      # 消费方（presentation）也需要 domain 的头
    PRIVATE spdlog::spdlog  # spdlog 是实现细节，不暴露
)

# infrastructure 层：实现 domain 的 interface
add_library(infrastructure STATIC src/infrastructure/...)
target_link_libraries(infrastructure
    PUBLIC  application     # 需要 application ports 的接口
    PRIVATE pqxx::pqxx      # PostgreSQL driver，私有
    PRIVATE Boost::asio     # 网络，私有
)

# 可执行文件：只需链接 infrastructure（PUBLIC 传递会自动带上 application, domain）
add_executable(server src/main.cpp)
target_link_libraries(server PRIVATE infrastructure)
PUBLIC/PRIVATE 的正确判断方法：看头文件 include——如果你的 .hpp 里有 #include "dependency/foo.hpp"，这个依赖就是 PUBLIC（消费方包含你的头文件时也会间接需要它）。如果只在 .cpp 里 include，就是 PRIVATE。

坑表现解决

# ❌ 坑 1：用 set(CMAKE_CXX_FLAGS ...) 全局设置编译选项

# ❌ 坑 3：OBJECT 库拼接 + 最终 executable 人工管理 OBJECTS
add_executable(${PROJECT_NAME}
    $<TARGET_OBJECTS:RadParams>
    $<TARGET_OBJECTS:${PROJECT_NAME}-main>
    $<TARGET_OBJECTS:${PROJECT_NAME}-Core>
    $<TARGET_OBJECTS:${PROJECT_NAME}-Controller>
    $<TARGET_OBJECTS:${PROJECT_NAME}-PlatformAgent>
)
# 这在 CMake 2.x 时代是常见做法；现代 CMake 推荐 STATIC 或 INTERFACE 库
#  配合 target_link_libraries 自动传递，而不是手动拼 OBJECTS

# ❌ 坑 4：依赖通过 pkg-config 全局暴露，缺乏 PUBLIC/PRIVATE 控制
pkg_check_modules(rpc-lib2 rpc-lib2)
include_directories(${rpc-lib2_INCLUDE_DIRS})    # 全局污染！
target_link_libraries(RPC ${rpc-lib2_LIBRARIES})  # 没有 PUBLIC/PRIVATE

# ✅ 正确：每个 target 独立管理编译选项（使用 generator expressions 区分生产/测试）
add_library(RpcCore STATIC)
target_sources(RpcCore PRIVATE           # 显式列出，不用 GLOB
    src/Core/MainComponent.cpp
    src/Core/LimServiceMoFilter.cpp
    # ... 其余源文件
)
target_compile_options(RpcCore PRIVATE
    -Wall -Wextra -Os -fPIC
    $<$<NOT:$<BOOL:${BUILD_TESTS}>>:-fno-exceptions>  # 生产才禁异常
    $<$<BOOL:${BUILD_TESTS}>:-fexceptions>             # 测试允许异常
)

# ✅ 正确：依赖用 IMPORTED target + PUBLIC/PRIVATE 控制
find_package(PkgConfig REQUIRED)
pkg_check_modules(rpc-lib2 REQUIRED IMPORTED_TARGET rpc-lib2)

target_link_libraries(RpcCore
    PRIVATE PkgConfig::rpc-lib2    # rpc-lib2 是实现细节，不传播
    PUBLIC  rpc-domain            # domain 接口传播给消费方
)

这个真实案例的核心教训：

  • GLOB + OBJECT 库 + 全局 FLAGS 是老项目的典型遗留债务，在小规模时看不出问题，但阻碍了 CI 并行、增量编译、target 级配置差异化
  • 电信项目里 -fno-exceptions 在生产开启、测试关闭 的需求非常真实——如果用全局 FLAGS，这个差异就需要 hack 实现；用现代 CMake 的 generator expressions 则是一行搞定
  • 这些问题完全可以作为 Strangler Fig 渐进迁移的对象：先把 GLOB 改成显式文件列表，再逐模块改 OBJECT 库为 STATIC 库，最后统一 compile_options

插件化架构设计：接口、隔离、生命周期

插件系统的四大设计目标 · 插件系统的核心接口设计 · 插件端的实现 · 版本兼容性策略 · 错误隔离：进程隔离 vs 线程隔离

插件化架构允许在不修改主程序的情况下动态扩展功能。设计难点在于：ABI 边界隔离、版本兼容、错误隔离、生命周期管理。本章给出完整的插件系统设计方案。

8.1 插件系统的四大设计目标

① ABI 隔离

插件和主程序可以用不同版本的编译器编译。插件的内部变化不影响主程序。

→ 必须用 C ABI 边界 或 稳定 vtable 接口。

② 版本兼容

主程序 v2 要能加载 v1 的插件（向后兼容）；理想情况下 v1 主程序也能加载 v2 插件（向前兼容）。

→ 需要版本握手协议。

③ 错误隔离

插件崩溃不能让主程序崩溃（进程隔离），或者至少能检测到并优雅降级（异常隔离）。

④ 生命周期管理

插件的 init/cleanup、在正确时机 unload、内存在正确模块释放。

8.2 插件系统的核心接口设计

// === plugin_api.h（公开 API，永久稳定的 C 接口）===
#pragma once
#ifdef __cplusplus
extern "C" {
#endif

#define PLUGIN_API_VERSION 2

// 所有插件必须导出这个函数（入口点）
typedef struct PluginDescriptor* (*PluginCreateFn)(void);

typedef struct PluginDescriptor {
    unsigned int api_version;     // 必须是 PLUGIN_API_VERSION
    const char*  name;
    const char*  version;         // 插件自己的版本
    const char*  description;

    // 函数表（C 语言版 vtable）
    void (*destroy)(PluginDescriptor* self);
    int  (*process)(PluginDescriptor* self, const void* input, size_t len,
                    void** output, size_t* out_len);
    void (*free_output)(void* output);    // 在插件模块内 free，避免跨模块 free
} PluginDescriptor;

#ifdef __cplusplus
}
#endif
// === 主程序：插件加载器 ===
class PluginLoader {
struct LoadedPlugin {
    void* handle;
    PluginDescriptor* desc;
};
    std::unordered_map<std::string, LoadedPlugin> plugins_;

public:
    bool load(const std::string& path) {
        void* handle = dlopen(path.c_str(), RTLD_NOW | RTLD_LOCAL);
        // RTLD_LOCAL 防止插件符号污染全局命名空间
        if (!handle) {
            spdlog::error("dlopen failed: {}", dlerror());
            return false;
        }

        auto* create_fn = reinterpret_cast<PluginCreateFn>(
            dlsym(handle, "plugin_create"));
        if (!create_fn) {
            dlclose(handle);
            spdlog::error("No plugin_create symbol in {}", path);
            return false;
        }

        PluginDescriptor* desc = create_fn();
        if (!desc || desc->api_version != PLUGIN_API_VERSION) {
            // 版本握手失败
            if (desc) desc->destroy(desc);
            dlclose(handle);
            spdlog::error("Plugin API version mismatch in {}", path);
            return false;
        }

        plugins_[desc->name] = {handle, desc};
        spdlog::info("Loaded plugin: {} v{}", desc->name, desc->version);
        return true;
    }

    void unload(const std::string& name) {
        auto it = plugins_.find(name);
        if (it == plugins_.end()) return;
        it->second.desc->destroy(it->second.desc); // 先调 destroy
        dlclose(it->second.handle);                 // 再 unload so
        plugins_.erase(it);
        // 注意：destroy 必须在 dlclose 之前调用！
        // dlclose 之后这段内存里的代码就不能再执行了
    }
};

    int state = 0;
};

    delete static_cast<MyPlugin*>(self);
}

                       void** output, size_t* out_len) {
    auto* p = static_cast<MyPlugin*>(self);
    // 处理输入...
    *output = malloc(len);              // 在本模块分配
    memcpy(*output, input, len);

    free(output);  // 在本模块 free（和 malloc 配对）

// 导出的工厂函数

    p->name         = "my_plugin";
    p->version      = "1.0.0";
    p->description  = "Example plugin";
    p->destroy      = my_destroy;
    p->process      = my_process;
    p->free_output  = my_free_output;

8.4 版本兼容性策略

向后兼容（新主程序加载旧插件）

  • 只在 PluginDescriptor 末尾追加新字段
  • 新字段有默认值或可为 null
  • 主程序检查 api_version，旧插件没有新字段时用默认行为
if (desc->api_version >= 2) {
    desc->new_feature(); // 只在 v2+ 调用
}

向前兼容（旧主程序加载新插件）

  • 新插件必须能在 api_version = 1 模式下工作
  • 插件检查主程序的能力（通过 capability flags）
  • 通常更难——通常只保证向后兼容

8.5 错误隔离：进程隔离 vs 线程隔离

方式隔离程度性能适用
同进程 dlopen 低（崩溃=主程序崩溃） 最高 可信插件、追求性能
独立进程 + IPC 高（崩溃=子进程） 中（IPC 开销） 不可信插件、安全性要求高
沙盒进程（seccomp） 最高（系统调用受限） 低（沙盒 + IPC） 第三方不可信代码
嵌入式插件化注意事项：
  • 传统嵌入式（MCU）没有 MMU → 不能动态加载 .so，用静态插件注册表（函数指针数组，编译期确定）
  • AUTOSAR Adaptive 支持动态部署，但受 Execution Management 管控，不是传统 dlopen
  • Linux 嵌入式（高端 SoC）可以用 dlopen，但注意对 MMU/flash 的约束
  • 静态注册表示例：REGISTER_PLUGIN(my_plugin, &create_fn) 宏展开成 linker section 里的指针

动态加载深入：dlopen / LoadLibrary 的坑与最佳实践

dlopen 的关键标志 · RTLD_LOCAL 与 self-contained 插件 · static 变量的 unload 陷阱 · Windows 上的 LoadLibrary 对应关系

9.1 dlopen 的关键标志

void* handle = dlopen("libplugin.so", flags);

// 常用 flags：
RTLD_NOW    // 立即解析所有符号（推荐：找到符号错误立刻失败）
RTLD_LAZY   // 延迟解析（第一次调用时才解析）
RTLD_LOCAL  // 符号不进入全局表（推荐：防止插件符号污染）
RTLD_GLOBAL // 符号进入全局表（插件需要 dlopen 别的库时用）
RTLD_NODELETE // unload 时不真正删除（保留 static 变量）
RTLD_GLOBAL 的危险：如果插件 A 和插件 B 都定义了 void foo()，用 RTLD_GLOBAL 加载后，后加载的会覆盖先加载的（One Definition Rule 在动态链接时静默违反）。几乎所有场景都应该用 RTLD_LOCAL。

9.2 RTLD_LOCAL 与 self-contained 插件

用 RTLD_LOCAL 加载插件后，插件里的全局函数对外不可见。但如果插件需要用主程序提供的函数（比如 logging API），怎么办？

// 方案：通过 PluginContext 把主程序服务注入给插件
typedef struct PluginContext {
    void (*log)(int level, const char* msg);
    void* (*alloc)(size_t bytes);
    void  (*dealloc)(void* ptr);
} PluginContext;

typedef PluginDescriptor* (*PluginCreateFn)(const PluginContext* ctx);

// 插件端：把 ctx 保存下来，用于调用主程序服务
static const PluginContext* g_ctx = nullptr;

extern "C" PluginDescriptor* plugin_create(const PluginContext* ctx) {
    g_ctx = ctx;
    g_ctx->log(1, "plugin_create called");
    // ...
}

9.3 static 变量的 unload 陷阱

// 插件里的 static 变量（单例、全局缓存）
static std::string g_cache;   // 有析构函数！

// 当 dlclose() 被调用：
// 1. .so 的析构函数们会被调用（包括 std::string g_cache 的析构）
// 2. 如果此时还有其他线程持有 .so 的地址 → 崩溃
// 3. std::string 的析构可能调用 libstdc++ 里的函数 → .so 已卸载但还在调用 → UB

// 对策：
// 1. 确保 dlclose 之前所有使用此 .so 的线程都完成工作
// 2. 用 RTLD_NODELETE 彻底避免卸载（泄漏 .so 内存，但安全）
// 3. 避免 .so 里有 static 对象（用 PluginContext 的 alloc/dealloc 管理生命周期）

10.1 编译器宏检测

// 标准做法：按编译器/版本分支
#if defined(__GNUC__) && !defined(__clang__)
  // GCC 专属
  #define LIKELY(x)   __builtin_expect(!!(x), 1)
  #define UNLIKELY(x) __builtin_expect(!!(x), 0)
#elif defined(__clang__)
  // Clang 专属
  #define LIKELY(x)   __builtin_expect(!!(x), 1)
  #define UNLIKELY(x) __builtin_expect(!!(x), 0)
#elif defined(_MSC_VER)
  // MSVC：没有 __builtin_expect，用空宏
  #define LIKELY(x)   (x)
  #define UNLIKELY(x) (x)
#endif

// 更现代：C++20 [[likely]] / [[unlikely]] 属性（三大编译器都支持）
if (condition) [[likely]] {
    fast_path();
} else [[unlikely]] {
    slow_path();
}

// === pal/linux/thread.cpp（Linux 实现）===
#include "pal/thread.hpp"
#include <pthread.h>
#include <sched.h>

struct Thread::Impl {
    pthread_t tid;
    Func func;
};

Thread::Thread(Func f, std::string name) : impl_(std::make_unique<Impl>()) {
    impl_->func = std::move(f);
    pthread_attr_t attr;
    pthread_attr_init(&attr);
    pthread_create(&impl_->tid, &attr,
        [](void* arg) -> void* {
            static_cast<Impl*>(arg)->func();
            return nullptr;
        }, impl_.get());
    pthread_attr_destroy(&attr);
    if (!name.empty()) pthread_setname_np(impl_->tid, name.c_str());
}
void Thread::set_cpu_affinity(int cpu) {
    cpu_set_t set; CPU_ZERO(&set); CPU_SET(cpu, &set);
    pthread_setaffinity_np(impl_->tid, sizeof(set), &set);
}

11.3 PCH（预编译头）

# CMake 3.16+ 的 PCH 支持
target_precompile_headers(mylib PRIVATE
    <vector>
    <string>
    <memory>
    <unordered_map>
    "src/common_includes.hpp"   # 项目内稳定的头
)

# 跨 target 共享 PCH（减少重复编译）
target_precompile_headers(myapp REUSE_FROM mylib)
PCH 的坑：
  • PCH 包含的头文件任何变化都要重新生成 PCH——所以只放几乎不变的头（std 库、第三方库）
  • 不要把项目内业务头放进 PCH——它们会变化，触发全量重编
  • PCH 是per-target 的，不同 target 的 PCH 不能直接共享（除非 REUSE_FROM）

12.1 ABI 稳定性检查清单

每次发布库的新版本前，逐项检查：

  1. 所有公开类的 sizeof 是否变化？（static_assert(sizeof(Foo) == N)）
  2. 是否在已有虚函数之间插入了新虚函数？
  3. 是否改变了任何公开 struct/class 的成员变量？
  4. 是否改变了函数签名（参数类型、返回类型）？
  5. 是否改变了inline 函数的行为（而不只是修复 bug）？
  6. 是否改变了枚举值的数值？
  7. 是否改变了基类（加/删/改）？
  8. abi-compliance-checker 或 abidiff 是否通过？
  9. 如果有 break → 是否提升了 MAJOR 版本？

高频面试题 · 大型 C++ 工程化

C++ 架构师专属题——ABI、Modules、CMake、插件化的核心考点。
Q1 为什么改了一个类的私有成员会导致 ABI break？它和 API break 有什么本质区别？
P0主题：ABI 基础难度：⭐⭐⭐
核心答案

因为类的私有成员影响 sizeof 和成员偏移——这些数值被编码进了已编译的调用方代码。新版本库的对象布局变了，旧代码按旧偏移读写，访问到错误地址 / 写超界，触发 heap corruption。

API vs ABI 的本质区别：

  • API = 源码层约定（函数签名、类的接口）。破坏 API → 编译报错（立刻可知）。
  • ABI = 二进制层约定（符号名、调用约定、内存布局、vtable、异常表）。破坏 ABI → 运行时崩溃或静默数据损坏（最难找）。
追问：如何检测 ABI break？

用 abi-compliance-checker 或 libabigail 的 abidiff。比较两个 .so，生成 HTML 报告。CI 里把它作为 PR 必过项——任何 ABI break 都必须显式提升 MAJOR 版本号。

面试加分

提到 GCC 5 的 std::string ABI 切换（COW → SSO）——这是历史上最大规模的一次 ABI break，影响了所有 Linux 发行版。可以引出 _GLIBCXX_USE_CXX11_ABI 编译开关的话题。

Q2 设计一个跨编译器（GCC + MSVC）共用的 C++ 插件系统，你的接口怎么定义？
P0主题：插件化 + ABI难度：⭐⭐⭐⭐
核心答案

必须用 C ABI，因为 GCC 和 MSVC 的 C++ name mangling、虚表布局、异常机制完全不兼容。具体做法：

  1. 头文件 extern "C" 包裹接口
  2. 结构体只用 C 类型（不能放 std::string / std::vector）
  3. 对象用 opaque handle（typedef struct Foo* FooHandle），所有操作走函数指针表
  4. 内存分配/释放必须在同一模块内配对（提供 free_xxx 函数）
  5. 错误处理用返回码（不抛异常，异常无法跨 ABI 边界）
  6. 第一字段是 api_version，加载时握手校验
陷阱：在边界传 std::string

"我用了 extern "C" 就跨 ABI 安全了？"——错。如果函数签名里有 std::string 参数，std::string 本身就是 C++ 类型，跨编译器布局不同（COW vs SSO）。extern "C" 不能让 C++ 对象跨 ABI 安全，只能让函数符号名稳定。

追问：你怎么实现版本兼容（v2 主程序加载 v1 插件）？

在 PluginDescriptor 结构体的末尾追加新字段。主程序先检查 api_version，如果是 v1 就不访问新字段。绝对不能在中间插入字段，那会破坏整个 ABI。

Q3 现代 CMake 里 PUBLIC、PRIVATE、INTERFACE 的区别是什么？给个具体例子。
P0主题：CMake难度：⭐⭐
核心答案（一句话记法）

影响"自己"还是"消费方"：

  • PRIVATE：只影响自己（消费方不传播）— 实现细节
  • PUBLIC：影响自己 + 传播给消费方 — API 依赖
  • INTERFACE：不影响自己，只传播给消费方 — header-only 库
具体例子：判断依据是看头文件
target_link_libraries(my_lib
    PUBLIC  boost::asio    # my_lib.hpp 里 #include <boost/asio.hpp>
                           # → 消费方编译时也需要看到 asio 头文件
    PRIVATE spdlog::spdlog # 只在 my_lib.cpp 里 #include <spdlog/spdlog.h>
                           # → 消费方不需要 spdlog
)
追问：为什么 header-only 库要用 INTERFACE？

因为 header-only 库没有可编译的 .cpp 文件——它本身不需要任何编译选项，但消费方在使用时需要 include 路径和编译特性。INTERFACE 表达的正是这个意思：" 我自己什么都不做，但消费方需要这些"。

陷阱：不写关键字

CMake 老风格 target_link_libraries(my_lib boost) 不带关键字，CMake 3.x 默认按 PUBLIC 处理，会意外传播。规则：永远写关键字。

Q4 C++20 Modules 解决了什么 #include 的痛点？2026 年应该开始用 Modules 吗？
P1主题：Modules难度：⭐⭐⭐
核心答案：#include 的四大顽疾
  1. 重复解析：100 个 .cpp 都 #include <string>，解析 100 次
  2. 宏污染：windows.h 的 min/max 宏污染整个翻译单元
  3. 符号泄漏：A include 了 B 内部 include 的 C，意外能用 C
  4. 顺序依赖：include 顺序不同，宏定义不同，行为可能不同

Modules 用语义导入而非文本替换，彻底解决这些问题。BMI（编译产物）可被多个 .cpp 共享，编译速度提升 40-80%。

2026 年用 Modules 的判断
  • 新绿地项目：可以试，但要做 POC
  • 生产大型项目：暂缓，等工具链更稳定（CMake 3.28+、GCC 14+ / Clang 16+ / MSVC 17.4+ 才可用）
  • 嵌入式/车规：禁用——嵌入式编译器（IAR、Green Hills）大多还在 C++14/17
渐进迁移策略

Phase 1: 切到支持 Modules 的编译器 → Phase 2: 新代码用 Modules，老代码用 Header Unit (import <vector>;) → Phase 3: 内部 utility 库改 Module → Phase 4: 公开 API 改 Module。不要一次性切。
