RAG Query 改写与查询理解

RAG 的效果，一半输在「检索前」。用户的原始问题往往又短、又模糊、又口语、还带指代——直接拿去检索，向量对不上、关键词命不中。Query 改写（Query Rewriting / Transformation）就是在检索之前，把「人话」翻译成「更适合被检索到」的一个或多个查询。本篇系统讲透 Multi-Query、HyDE、Step-back、子问题分解、会话式改写五大主力技术，配一个可交互的改写演示，帮你建立「什么场景该用哪一招」的判断力。

原始 query 为什么不够用

「问得烂」是 RAG 检索失败最被低估的原因

rag_deep_dive.html 讲了 chunking、embedding、hybrid、rerank——那些都在优化「怎么把库里的东西找出来」。但还有个前提被忽视：你拿去找的那句 query 本身好不好。真实用户的提问，和知识库里文档的写法，常常「说的不是一种话」。检索是拿 query 去匹配文档，两边对不上，后面 rerank 再强也白搭——召回不到的东西，永远排不出来。

原始 query 直接检索，像你用大白话问图书管理员：「那个……讲赚钱的书？」管理员一脸茫然。会问的人（Query 改写）会先把它翻译成检索友好的说法：「个人理财 / 投资入门 / 财务规划」几个方向分别去书架找，再把结果汇总。你问题问得好，书架命中率就高——改写做的就是「替用户把问题问好」。
Query 改写 / 查询变换（Query Rewriting / Query Transformation）：在检索之前，用 LLM（或规则）把用户的原始查询，改写、扩展或分解成一个或多个「更利于检索命中」的查询，再送去检索。它是 RAG 里成本最低、收益却常被低估的一环——因为它治的是「召回」这个上游，而召回是整条链路的天花板。
原始 query 的毛病后果对应的改写招式
太短 / 太宽泛（"性能怎么优化？"）召回一堆泛泛而谈Step-back / Multi-Query
用词和文档不一致（口语 vs 术语）向量对不上、关键词命不中Multi-Query / HyDE
一句话里含多个子问题检索被"平均"、每个都答不全子问题分解
多轮对话里的指代（"它贵吗？"）脱离上下文根本没法检索会话式改写 / 指代消解
问法抽象、缺乏可匹配的实体词向量漂移、召回噪声大HyDE / Step-back
改写不是「越多越好」。每多一次改写就多一次 LLM 调用（延迟 + 成本），多召回一批也会引入噪声。本篇最后一章会给出「什么场景值得上、怎么权衡」的决策表。先把五大招式逐个讲清。
01

查询理解全景：改写在流水线的哪一步

先看地图，再逐个拆招

1.1 改写在 RAG 流水线里的位置

改写属于「① 查询理解」。它和 Rerank（③）是两个互补的杠杆：改写提升召回上限（把对的文档捞进候选池），Rerank 提升精度（把候选池里对的排到前面）。

1.2 先分诊：意图识别与查询路由

并不是每个 query 都需要改写，甚至不是每个都需要检索。一个成熟的查询理解层，往往先做一次分诊（路由）：

要不要检索？

"你好""帮我总结上一段"这类，不需要走知识库，直接让 LLM 回答，省一次检索。

查哪个库 / 哪种检索？

问产品文档 vs 问代码 vs 问表格数据，路由到不同索引 / 不同检索方式（向量 / SQL / 关键词）。

是简单还是复杂问题？

简单事实题一次检索即可；复杂 / 多跳题才启用分解或 Multi-Query，避免无脑加成本。

要不要消解上下文？

多轮对话里带指代（它/那个/上面说的），先补全成独立可检索的问题（见第 6 章）。

查询路由（Query Routing）：根据 query 的意图 / 类型，把它分派到最合适的处理路径（是否检索、查哪个数据源、用哪种改写策略）。它让「改写」变得按需触发而非一刀切，是控制成本与延迟的关键闸门。
工程上，路由和改写常合并成一次 LLM 调用：给模型一个 prompt，让它同时输出「意图分类 + 是否需检索 + 改写后的查询」，用结构化 JSON 返回。一次调用拿到全部决策，省延迟。
02

Multi-Query：一问变多问

同一个意图，换几种说法分别检索，再合并去重

最直觉的一招：让 LLM 把原始 query 改写成多个语义等价、但用词/角度不同的查询，分别去检索，再把结果合并去重。它对付的是「用户用词和文档用词对不上」——多个说法总有一个能命中。

Multi-Query 像问路问三个人：同样想去火车站，你分别问「火车站怎么走」「高铁站在哪」「坐车去市中心站」，三个人给的路线合起来，比只问一个人更全、更不容易被单个人的口误带偏。多个查询版本 = 多个「问路的说法」，合并召回 = 汇总大家的答案。
扇出（fan-out）→ 并行检索 → 合并去重。合并常用 RRF（Reciprocal Rank Fusion，倒数排名融合）把多路结果排名融合成一份，再交给 rerank。
RRF（Reciprocal Rank Fusion）：把多路检索结果按「1/(k+排名)」累加分数后重排的融合方法。它只看排名不看原始分数，天然抹平不同检索通道的分数量纲差异，是合并 Multi-Query / Hybrid 多路结果的常用手段（rag_deep_dive.html 的 Hybrid 章有细讲）。
Multi-Query 的代价：① 多一次 LLM 调用生成变体；② 检索次数翻几倍；③ 召回并集变大 → 更依赖后面的 rerank 去噪。适合召回率是瓶颈的场景，不适合对延迟极敏感的实时问答。
03

HyDE：先让模型「瞎编」一个答案

Hypothetical Document Embeddings · 用假设答案去检索，而不是用问题

HyDE 的洞察很妙：问题和答案，往往不在同一个向量空间的位置。 用户的问题很短（"RAG 幻觉怎么办"），而知识库里的答案是一整段陈述文字。直接拿「问题」去和「答案段落」比相似度，天然有落差。HyDE 的做法是：先让 LLM 凭空生成一个"假设的答案"（哪怕内容是编的、可能不准），再用这个假设答案的向量去检索——因为「假答案」和「真答案文档」在措辞、结构上更像，命中率更高。

找一本书，你只记得大概讲什么。HyDE 像你先自己写一段"这本书大概会这么说"的摘要，再拿这段摘要去书架比对——比起只用「书名关键词」找，用「一整段像样的内容」去找，明显更容易撞上真正那本书。假答案哪怕细节是错的也没关系，它的"文体和用词"和真文档更接近，这才是检索要的。
HyDE = Hypothetical Document Embeddings。用假设文档而非原始问题去嵌入检索，缩小「问题↔文档」的向量鸿沟。
什么时候 HyDE 有用：问题短、抽象、和文档措辞差距大（尤其专业领域）。什么时候慎用：如果 LLM 对该领域一无所知，编出的假答案会跑偏，反而把检索带歪；此时可结合原始 query 一起检索（HyDE + 原问题双路召回）兜底。
HyDE 每次要多一次生成调用（延迟↑），且假答案质量受模型知识影响。它是「用生成换召回」，务必用 eval 验证在你的数据上确实带来净收益，别当银弹无脑上。
04

Step-back：先退一步问更本质的问题

Step-back Prompting · 用「上位概念」召回背景知识，再回答具体问题

有些问题太具体，直接检索反而捞不到有用的背景。Step-back 的思路是：先让 LLM 把问题「后退一步」抽象成一个更宏观、更本质的问题，用它去检索背景/原理知识，再结合原始具体问题作答。它对付的是「只见树木不见森林」——先把森林（原理）找到，再看这棵树（细节）。

学生问："2008 年 9 月雷曼倒闭那周，某某债券为什么暴跌？" 直接查这个日期太细，可能查不到。好老师会先退一步："先搞清楚——金融危机中信用债为什么会集体抛售？" 把这个原理弄懂了，再回头看那一周的具体表现就顺理成章。Step-back 就是让模型自己扮演这位「先讲原理再讲细节」的老师。
Step-back 生成一个更抽象的「后退问题」召回原理/背景，与原始具体问题的检索结果合并后一起送给 LLM 作答。
Step-back 与 Multi-Query 的区别：Multi-Query 是「同一层次的换说法」（横向扩展），Step-back 是「抽象层次上移」（纵向后退）。前者补召回的广度，后者补召回的背景深度。复杂推理题两者可叠加。
05

子问题分解：多跳问题拆成几步

Query Decomposition · 一个复杂问题拆成若干可独立检索的子问题

有些问题一句话里其实藏着多个需要分别查证的子问题，或者需要「先查 A、拿 A 的结果再查 B」的多跳（multi-hop）推理。整句直接检索会「四不像」：每个子问题都只召回一点点，哪个都答不全。分解就是把它拆开，各自检索，再汇总作答。

问："我们团队用的框架和竞品比，哪个更省内存、社区哪个更活跃？" 这其实是两个独立调查拼在一起。像做尽调：你不会拿整句去搜，而是拆成「① 两个框架的内存占用对比」「② 两个框架的社区活跃度」分别查资料，最后写成一份综合报告。分解 = 把「大问题」拆成「几张独立的调查任务卡」。
子问题分解（Query Decomposition）：让 LLM 把一个复合/多跳问题拆成若干个更简单、可独立检索的子问题，分别检索作答后再汇总。分两类：并行分解（子问题彼此独立，可同时查）与串行/多跳分解（后一步依赖前一步的答案，须顺序执行）。

并行分解

"A 和 B 各自的优缺点？" → 拆成"A 的优缺点" + "B 的优缺点"，同时检索，最后对比汇总。子问题间无依赖。

串行 / 多跳分解

"张三的经理审批了几个项目？" → 先查"张三的经理是谁"，拿到答案（李四）再查"李四审批的项目数"。后一跳依赖前一跳。

多跳问题必须串行：跳 1 的答案（李四）代入跳 2。这类"检索→推理→再检索"的循环，正是 Agentic RAG 的雏形。
串行多跳本质上是「检索 + 推理交替」，和 Agent 的 ReAct 循环是同一回事——当分解需要动态决定「下一步查什么」时，RAG 就演化成了 Agentic RAG（见 agent_architecture.html）。分解是从「静态 RAG」通向「Agent」的桥。
06

会话式改写：把「它」补全成能检索的问题

多轮对话里的指代消解与上下文补全 · 最实用、最容易被漏掉

前面几招是「把好问题变得更好」，这一招是「把根本没法检索的问题变得能检索」。多轮对话里，用户第二句往往省略主语、带指代："那它多少钱？""换成大号呢？"——单独拿这句去检索，向量和关键词都无从下手。会话式改写就是结合前几轮对话，把它补全成一个自包含（standalone）的独立问题。

这就是客服接电话的基本功。客户说「那这个能退吗？」，新手客服一脸懵；老客服脑子里会自动补全成「客户上一句问的那款蓝色 XL 卫衣，能退货吗？」——因为他记得刚才聊了什么。会话式改写就是给检索系统装上这份「记得刚才聊了什么」的能力，把口语里的省略和指代还原成完整问题。
会话式改写 / 指代消解（Conversational Query Rewriting / Coreference Resolution）：结合对话历史，把当前这轮带省略、指代（它/那个/上面）的追问，改写成一个不依赖上下文、可独立检索的完整查询（standalone query）。它是多轮 RAG / 聊天机器人的必备前置步骤，缺了它多轮对话的检索几乎必然崩。
对话历史用户新追问改写后（standalone）
问：ThinkPad X1 多重？
答：约 1.1kg。		"它续航呢？""ThinkPad X1 的电池续航是多久？"
在聊「年假政策」"那病假呢？""公司的病假政策是怎样的？"
问：如何创建索引？
答：用 CREATE INDEX…		"删除呢？""如何删除数据库索引？"
实现要点：不要把整段对话历史一股脑塞进检索（噪声大、还超长）。正确做法是先用一次轻量 LLM 调用做「改写成 standalone query」，只把这一句干净的查询送去检索。历史只用于「改写」，不用于「检索」。
别过度改写：如果新问题其实已经自包含（换了全新话题），强行拼历史反而会把旧话题的词带进来污染检索。改写 prompt 里要允许模型判断「本轮是否延续上文」，独立话题就原样透传。

7.2 怎么选：一张决策表 + 成本权衡

场景 / 症状首选策略额外成本
多轮对话、带指代追问会话式改写（几乎必做）+1 次轻量 LLM
用户用词和文档差异大Multi-Query 或 HyDE+1 生成 & N 倍检索
问题短、抽象、专业领域HyDE（模型懂该领域时）+1 次生成
问题太宏观 / 缺背景Step-back+1 生成 & 2 路检索

改写策略像做菜的火候：不是每道菜都要猛火爆炒。简单的凉拌（事实题）直接上；难处理的硬菜（复杂/多跳）才值得多几道工序（分解 + 多路召回）。无脑给所有 query 都上全套改写 = 每盘凉菜都拿去爆炒——又慢又贵还未必好吃。
工程落地三原则：① 用路由按需触发改写，别一刀切（第 1 章）；② 改写 + 意图分类合并成一次结构化 LLM 调用省延迟；③ 上线前用 eval（ai_eval.html）量化「改写前后召回率 / 端到端答案质量」的净收益——改写不是免费的，要用数据证明它值。
改写（提召回）+ Rerank（提精度）+ 幻觉治理（提可信，见 rag_hallucination.html）是 RAG 三个正交的质量杠杆。本篇管住了"检索前"，rerank 管"检索后排序"，幻觉治理管"生成时别编"——三者合起来才是生产级 RAG。
08

小结 · 面试 · 术语表

收束成判断力

一页纸总结

① 改写治召回上游

召回不到的，rerank 再强也救不了。改写是整条链的天花板，成本低收益常被低估。

② 五招各治一病

Multi-Query（换说法）、HyDE（用假答案）、Step-back（退一步）、分解（拆多跳）、会话改写（补指代）。

③ 按需触发

用路由分诊，简单题不改写，复杂题才上重招。改写不免费，要用 eval 证明净收益。

面试自测（点击展开）

基础为什么 RAG 需要 Query 改写？它和 Rerank 有什么分工？
动因用户 query 短/模糊/口语/带指代，与文档措辞对不上，直接检索召回差。
分工改写在检索前，提升召回上限（把对的文档捞进候选池）；Rerank 在检索后，提升精度（把候选池里对的排前面）。
关键召回不到的文档，Rerank 永远排不出来——所以改写是天花板。
核心HyDE 的原理是什么？为什么用「假答案」检索反而更准？
原理先让 LLM 生成一个假设性答案，再用假答案的向量去检索真文档。
为什么问题（短）和答案文档（长陈述）在向量空间位置有落差；而"假答案↔真答案"在措辞/文体上更同构，相似度更高。
风险模型不懂该领域时假答案跑偏会带歪检索；可与原始 query 双路召回兜底，并用 eval 验证。
辨析Multi-Query、Step-back、子问题分解，三者有何不同？
Multi-Query同层次换说法（横向扩展），补召回广度。
Step-back抽象层次上移（纵向后退），补召回背景/原理。
分解把复合/多跳问题拆成可独立检索的子问题，处理复杂度；串行分解即"检索+推理"交替，是 Agentic RAG 雏形。
工程多轮对话的 RAG，为什么必须做会话式改写？怎么做？
必须追问常带指代/省略（"它续航呢"），单独检索无从下手，多轮 RAG 会崩。
做法用一次轻量 LLM 调用，结合历史把追问改写成 standalone 独立问题，只把这句干净查询送检索；历史只用于改写不用于检索。
陷阱新话题别强拼历史词污染检索；prompt 要允许模型判断是否延续上文。
综合改写不是免费的，你怎么决定「哪个 query 该用哪种改写」？
路由分诊先判断：要不要检索、查哪个库、简单还是复杂、要不要消解指代——按需触发对应策略。
合并调用路由 + 意图分类 + 改写合并成一次结构化 LLM 调用，省延迟。
数据说话用 eval 量化改写前后的召回率与端到端答案质量，只保留净收益为正的策略；简单题/延迟敏感场景直接不改写。

术语表速查 · Glossary

① 查询理解

Query 改写 Query Rewriting/Transformation检索前把原始查询改写/扩展/分解成更利于命中的查询。治"召回"上游，是 RAG 链路天花板。
查询路由 Query Routing按意图/类型把 query 分派到不同处理路径（是否检索、查哪个库、用哪种改写），实现按需触发。
意图识别 Intent Detection判断 query 属于哪类（闲聊/事实/复杂/多跳），决定后续检索与改写策略。

② 五大改写技术

Multi-Query把一问改写成多个等价变体分别检索再合并去重，补召回广度。常用 RRF 融合。
HyDE Hypothetical Doc Embeddings先让 LLM 生成假设答案，用假答案向量检索真文档，缩小"问题↔文档"向量鸿沟。
Step-back Prompting把问题后退成更抽象的"后退问题"召回背景/原理，再结合原问题作答。补召回深度。
子问题分解 Decomposition把复合/多跳问题拆成可独立检索的子问题；串行分解即"检索+推理"交替（Agentic RAG 雏形）。
会话式改写 / 指代消解结合对话历史把带指代/省略的追问补全成 standalone 独立问题。多轮 RAG 必备。

③ 相关概念

RRF Reciprocal Rank Fusion按 1/(k+排名) 融合多路检索结果，抹平不同通道分数量纲。合并 Multi-Query/Hybrid 常用。
Standalone Query不依赖上下文、可独立检索的完整问题。会话式改写的产物。
Agentic RAG检索与推理动态交替（决定下一步查什么）的 RAG，是分解/多跳的进阶形态，趋近 Agent。
召回率 / 精度 Recall / Precision召回率=该找到的找到了多少（改写提升它）；精度=找到的里对的占比（Rerank 提升它）。
