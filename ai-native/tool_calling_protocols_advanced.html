AI Engineering · 工具调用协议进阶

工具调用协议进阶：A2A · 流式传输 · LLM 网关

MCP 那篇解决了「一个 Agent 怎么标准化地接工具」。但真实生产系统里还剩三块拼图：多个 Agent 之间怎么互相委托（A2A）、token 怎么实时流到用户屏幕（SSE / WebSocket / WebRTC）、几十个模型和 key 怎么统一治理（LLM 网关）。这三者共同决定了一个 Demo 能不能变成扛得住流量、控得住成本、挂了能自愈的产品。本篇用图、表和两个可交互演示把它们讲透。

MCP 之后，还缺什么？

工具接上了 ≠ 系统跑得起来 · 生产化还差三块拼图

在 agent_protocols_and_skills.html 里我们讲清了 Function Calling → MCP 这条线：让一个 Agent 能用统一协议调用外部工具（查数据库、读文件、调 API）。但当你把一个 Demo 推向生产，会立刻撞上三个 MCP 管不到的问题。

MCP 像给一个员工配好了各种工具（电钻、扳手）——他一个人干活没问题。可一旦要开公司，你还得解决三件事：员工之间怎么派活协作（A2A）、怎么把进度实时汇报给客户（流式传输）、公司的水电煤（各家模型 API）怎么统一走一个总闸、超支了自动限流、断了自动切备用（LLM 网关）。工具齐全只是起点，协作、实时、治理才让它成为一家能运转的公司。
MCP 管「向下接工具」；本篇补齐另外三个方向：向右接别的 Agent（A2A）、向上把结果实时流给用户、向左把众多模型统一收口（网关）。
这三块与 MCP 是正交互补的，不是替代关系。一个成熟系统往往同时用到：网关统一收口模型 → Agent 用 MCP 调工具 → 需要专业能力时用 A2A 委托子 Agent → 全程用流式协议把中间结果推给前端。
01

A2A：Agent 与 Agent 之间的协议

Google 2025 年提出的 Agent2Agent · 让 Agent 能把活儿「外包」给另一个 Agent

1.1 为什么需要 A2A

MCP 解决的是「Agent → 工具」。但工具是被动、无状态的：你调 search(q)，它算完就返回。而另一个 Agent 是主动、有状态、会思考的：你把「帮我订一趟满足这些约束的差旅」交给差旅 Agent，它会自己规划、追问、多步执行，最后交付。把「Agent 当工具调」很别扭——这正是 A2A 要标准化的场景。

MCP 像你用一件工具：拿起计算器按一下出结果。A2A 像你委托一个同事：「这个报表你帮我做下」——他会自己拆解任务、中途可能问你要点信息、干几分钟甚至几小时、最后把成品给你。工具是「调用」，同事是「委托」。当参与方从「工具」升级为「另一个会自主干活的 Agent」，你就需要 A2A 这套「同事之间怎么派活、怎么对进度、怎么交付」的协议。
A2A（Agent2Agent Protocol）：由 Google 于 2025 年发起、后捐赠给 Linux 基金会的开放协议，用于让不同厂商、不同框架构建的 Agent 之间互相发现、委派任务、交换结果。它基于 HTTP + JSON-RPC + SSE，核心抽象是：用 Agent Card 描述「我是谁、我会什么」，用 Task 承载一次有生命周期的委托。

1.2 三个核心概念：Agent Card / Task / Message

Agent Card

一份放在 /.well-known/agent.json 的「名片」：声明这个 Agent 的名称、能力（skills）、支持的输入输出模态、认证方式、端点。别的 Agent 靠它发现并判断「能不能把某类活交给你」。

Task

一次委托的有状态载体，有明确生命周期：submitted → working → input-required → completed / failed。长任务可持续几秒到几小时，客户端靠它跟踪进度。

Message / Artifact

Message 是双方来回的对话轮（可含文本、文件、结构化数据）；Artifact 是任务产出的成品（报告、图片、代码）。中间进度通过 SSE 以事件流推送。

注意 ④：Task 可以中途进入 input-required 反问客户端——这是「工具调用」做不到的，也是 Agent 作为「会思考的同事」的关键特征。

一句话「给 Agent 装工具」「让 Agent 互相外包」
它们是搭档，不是对手。 Google 官方的说法：MCP 让 Agent 接工具，A2A 让 Agent 接 Agent。一个多智能体系统里，主 Agent 用 MCP 调本地工具，遇到需要专业能力（法务审查、差旅规划）时，用 A2A 把子任务委托给专门的远端 Agent——而那个远端 Agent 内部又用自己的 MCP 工具去干活。
A2A 还年轻（2025 起步），生态和标准仍在演进。当前多数「多 Agent」系统其实是同进程内的 sub-agent 编排（见 agent_architecture.html 的多 Agent 章节），并不走 A2A。A2A 的价值主要在跨组织、跨厂商的 Agent 互操作——这也是它最像「HTTP 之于网站」的地方：真正的红利要等生态起来。
02

流式传输三选一：SSE / WebSocket / WebRTC

让 token「打字机式」地流到用户眼前 · 三种传输的取舍

2.1 为什么 LLM 应用必须「流式」

LLM 是逐 token 生成的，一段 500 字的回答可能要生成好几秒。如果等全部生成完再一次性返回，用户会盯着转圈圈干等；而边生成边推送（生成一个 token 就发一个），用户几百毫秒内就看到文字开始「往外冒」，体感天差地别。这就是所有 ChatGPT 类产品都用流式的原因：首 token 延迟（TTFT）才是体感关键，而不是总时长。

非流式像点了外卖等全部做好一起送：40 分钟内你什么都吃不到。流式像自助餐/边做边上：一道好了先端一道，你立刻开动。菜的总量没变、总时间也差不多，但「多久能吃到第一口」完全不同——这就是 TTFT 决定体感。

2.2 三种传输：单向流够用吗？

把 token 从服务端推到浏览器，主流有三种传输层技术，复杂度和能力递增：

箭头方向 = 数据能流的方向。SSE 只能「服务端 → 浏览器」单向（够用于纯文字流）；WebSocket 双向；WebRTC 走 UDP、点对点，为音视频而生。

底层HTTP（长响应）TCP（ws://）UDP（+ 信令/穿透）

选型口诀：能用 SSE 就别上 WebSocket，能用 WebSocket 就别上 WebRTC。 纯文字问答的 token 流式，SSE 足矣（这也是 OpenAI / Anthropic 文本 API 的默认流式方式）；需要「用户说话的同时模型也在回」的双向低延迟（如实时语音），才上 WebSocket 甚至 WebRTC。复杂度和运维成本是阶梯式上升的。
注意区分「模型 API 的流式」和「你的前端到你的后端的流式」。OpenAI 的 stream=true 用的是 SSE；你的后端收到这些 chunk 后，再决定用 SSE / WebSocket 转发给自己的前端。两段可以是不同传输：后端↔模型走 SSE，前端↔后端也常走 SSE。

03

LLM 网关：几十个模型 / key 的总闸

LLM Gateway / Router · 把「直连各家 API」升级成「统一收口 + 治理」

3.1 网关解决什么问题

Demo 阶段你直接 openai.chat(...) 就够了。可一旦上生产、接多个模型（GPT / Claude / Gemini / 国产 / 自托管），代码里会散落一堆 if-else、各家 SDK、各种 key、各种重试逻辑，还没法统一算成本、限流、监控。LLM 网关就是在你的应用和各家模型之间加一层统一代理，把这些横切关注点收口到一处。

LLM 网关就像公司的总机 + 财务 + 保安合体：员工（你的代码）不用记每个供应商的电话（各家 API 细节），只打给总机（网关）说「要个 GPT-4 级别的回答」；总机负责转接哪家最合适（路由）、这家占线就转备用（降级 fallback）、记账（成本归集）、不许超预算（限流）、常见问题直接给存档答案（缓存）。你的业务代码从此只跟一个「总机」说话。
应用侧只认「一个统一的 OpenAI 兼容接口」；网关在中间做全部治理，再把请求翻译成各家协议分发出去。换模型 = 改网关配置，业务代码零改动。

3.2 网关的七大能力

① 统一接口 / 多模型路由

对外一套 OpenAI 兼容 API，内部适配各家。可按「便宜优先 / 质量优先 / 就近 / A-B 灰度」把请求路由到不同模型。

② 降级 Fallback / 重试

主模型超时/限流/报错时，自动切换到备用模型或备用 key，用户无感。是可用性的关键。

③ 限流 / 配额

按用户 / 团队 / API-key 设 RPM、TPM、每日预算，防止某个调用方打爆账单或挤占额度。

④ 语义缓存

相同/相似请求直接返存档结果，省钱又提速（尤其高频重复问题）。命中判定可用精确匹配或向量相似。

⑤ 成本归集 / 观测

统一统计每次调用的 token、费用、延迟、错误率，按业务线出账单，接 tracing（见 LLMOps 观测章）。

⑥ 密钥集中管理

各家 key 只存网关，业务侧拿的是网关签发的虚拟 key，可随时吊销、按 key 限权限/预算。

⑦ Guardrail / 合规

在入口/出口统一挂敏感词、PII 脱敏、Prompt 注入检测（见 ai_safety.html），一处生效、全线受控。

额外：可观测 + 审计

全量请求日志可回放、可审计，出问题能定位是哪个模型/哪次调用，是排障与合规的底座。

3.4 选型：LiteLLM / Portkey / OneAPI …

方案形态特点适合
LiteLLM开源库 + 独立 Proxy100+ 模型统一成 OpenAI 格式，路由/降级/预算/缓存齐全，社区最活跃自建、想开源可控
PortkeySaaS + 开源 Gateway主打可观测 + Guardrail + 语义缓存，控制台完善要开箱即用的托管
OneAPI / NewAPI开源自托管国内流行，聚合国内外模型 + 计费/分发，中文生态好国内多模型聚合分发
云厂商网关托管（Bedrock/Vertex 等）与云生态深度集成，省运维但绑定较强已重度用某朵云
Kong/APISIX + 插件通用 API 网关扩展复用现有 API 网关基建，AI 能力靠插件已有成熟 API 网关团队
多数团队从 LiteLLM Proxy 起步：一个进程就把「统一接口 + 路由 + 降级 + 预算 + 缓存 + 观测」全给了，且完全兼容 OpenAI SDK——业务几乎零改造。等规模上来、对 Guardrail/审计要求变高，再评估 Portkey 这类更重的方案。
网关是关键路径上的单点：所有 LLM 流量都过它。务必给它做高可用（多副本 + 健康检查）、别让「治理层」自己成了故障源。缓存也要小心：带用户隐私/时效性的内容误命中缓存会串答案，需按场景设 TTL 与命中键。

小结 · 选型 · 面试 · 术语表

把三块拼图收束成判断力

一页纸总结

① A2A = Agent 接 Agent

MCP 接工具（被动），A2A 接另一个会思考的 Agent（有状态 Task、能反问）。跨厂商互操作场景才真正需要。

② 流式：能低就低

文字流用 SSE（原生+自动重连），双向实时用 WebSocket，音视频才上 WebRTC。TTFT 才是体感关键。

③ 网关 = 模型总闸

统一接口 + 路由/降级/限流/缓存/成本/密钥/Guardrail。LiteLLM 起步，业务零改造。

面试自测（点击展开）

基础A2A 和 MCP 有什么区别？分别解决什么问题？
MCPAgent ↔ 工具/数据源。对方被动、无状态，请求-响应，调完即返。「给 Agent 装工具」。
A2AAgent ↔ 另一个 Agent。对方主动、有状态，走有生命周期的 Task，能中途反问（input-required）。「让 Agent 互相外包」。
关系互补而非替代：主 Agent 用 MCP 调本地工具，用 A2A 把专业子任务委托给远端 Agent，后者内部又用自己的 MCP。
核心做一个聊天产品的流式输出，SSE / WebSocket / WebRTC 怎么选？
SSE纯文字 token 流式首选：基于 HTTP、浏览器 EventSource 原生、自动重连、基建简单。ChatGPT 网页就是它。
WebSocket需要「双向、低延迟、边说边回」时用，如实时语音转写、协作。代价是要自己管重连/心跳。
WebRTC真正的实时音视频通话 Agent 才用，走 UDP + STUN/TURN，最重。
口诀能用 SSE 别上 WS，能用 WS 别上 WebRTC——复杂度阶梯上升。
辨析流式输出让模型「更快」了吗？TTFT 是什么？
没更快流式不改变总生成时间，模型该生成多久还是多久。
TTFTTime To First Token，首个 token 到达用户的延迟。流式把它压到几百 ms，用户马上有反馈。
附加值流式还能让用户提前中断跑偏的生成，省 token；并把「干等」变成「边看边等」，体感天差地别。
工程为什么生产系统要引入 LLM 网关？直连各家 API 不行吗？
直连痛点代码散落各家 SDK/key/重试逻辑，无法统一算成本、限流、监控、降级；换模型要改业务代码。
网关收益统一接口 + 路由/降级/限流/语义缓存/成本归集/密钥集中/Guardrail，横切关注点收口一处；换模型改配置即可。
注意网关是关键路径单点，须高可用；缓存要按场景设 TTL/命中键，避免隐私串答。
综合设计一个多 Agent 客服系统，这三样怎么协同？
网关所有 Agent 的 LLM 调用统一走网关：按成本路由（简单意图走小模型）、主模型故障自动降级、按租户限流、常见问答走语义缓存。
MCP主客服 Agent 用 MCP 接订单库、知识库、工单系统等工具。
A2A遇到专业问题（退税、法务）委托给专门的远端 Agent，用 Task 跟踪、必要时反问补充信息。
流式全程用 SSE 把「正在查询…/正在生成…/最终答案」增量推到用户界面，TTFT 低、可中断。

术语表速查 · Glossary

本篇核心名词一次收齐，当字典用：

① A2A / Agent 协作

A2A Agent2Agent ProtocolGoogle 发起、捐给 Linux 基金会的开放协议，让不同厂商/框架的 Agent 互相发现、委派任务、交换结果。
Agent Card放在 /.well-known/agent.json 的能力名片：声明名称、技能、模态、认证、端点，供他人发现与判断。
Task（A2A）一次委托的有状态载体，生命周期 submitted→working→input-required→completed/failed，可长时运行。
ArtifactTask 产出的成品（报告/图片/代码）；Message 是双方来回的对话轮。
MCP Model Context Protocol让 Agent 标准化接工具/数据源的协议（被动、无状态）。与 A2A 互补：接工具 vs 接 Agent。

② 流式传输

SSE Server-Sent Events基于 HTTP 的服务端单向推送，浏览器 EventSource 原生、自动重连。文字流式首选。
WebSocketTCP 上的全双工长连接，双向低延迟。需自己管重连/心跳。用于双向实时（语音转写/协作）。
WebRTC基于 UDP 的点对点实时通信，可传音视频，需信令 + STUN/TURN 穿透。最重，用于实时语音 Agent。
TTFT Time To First Token首个 token 到达用户的延迟。流式的核心价值就是压低 TTFT，改善体感（总时长不变）。
TPS / 吞吐每秒生成 token 数。与 TTFT 一起构成流式体验的两个关键指标（见 inference_serving）。

③ LLM 网关

LLM 网关 LLM Gateway / Router应用与各家模型间的统一代理层，收口路由/降级/限流/缓存/成本/密钥/Guardrail 等横切能力。
Fallback 降级主模型超时/限流/报错时自动切换备用模型或备用 key，保障可用性。
语义缓存 Semantic Cache对相同/相似请求返回存档结果，省钱提速。命中可用精确匹配或向量相似度。
限流 / 配额 Rate Limit / Quota按用户/团队/key 设 RPM、TPM、预算上限，防打爆账单或挤占额度。
LiteLLM / Portkey / OneAPI主流 LLM 网关方案：LiteLLM（开源库+Proxy，最流行）、Portkey（SaaS+观测/Guardrail）、OneAPI（国内自托管聚合）。
