6 AI 招聘市场全景

本文从我作为 AI 高级开发 + AI 高级招聘员 双视角写

本文从我作为 AI 高级开发 + AI 高级招聘员 双视角写。先说三个让候选人最不舒服但必须先内化的事实：

3 个反直觉事实

  • "会调 API + 会 prompt" 不是 AI 工程师。这是 2023 年的红利期产物，2026 已经是入门下限。
  • Junior AI 岗在收缩，Senior+ 严重短缺。LLM 抹平了"做出 demo"的门槛，但拉高了"做出 production"的门槛。
  • "会 PyTorch" 不再是壁垒。真正的壁垒是：在线上跑出 99.9% SLO 的推理服务 / 跑通千卡训练不炸 / 把 RAG 准确率从 60% 调到 92%。

3 个真实需求信号

  • AI Infra 极度缺人。国内字节/阿里推理团队、智驾感知规控、AI 创业公司基建团队 — C++/CUDA + LLM 复合背景 月薪可顶 30%+。
  • "懂业务 + 能落地" 的 LLM App 工程师溢价。能从"业务痛点 → RAG/Agent 方案 → eval → 上线 → 持续优化"端到端走通的，国内大厂 P7/P8 抢手。
  • AI EM / TL 全市场缺。能管 GPU 预算 + 模型选型 + 团队 onboarding + 跟 PM/法务对齐的人，几乎是空缺。
给本文读者的建议读法：先看 §2 6 类岗位画像 → 自己对号入座（或选目标）→ 看 §3 能力矩阵自评差距 → 看 §4 薪资是否符合预期 → 看 §5 JD 解码看招聘原文怎么读 → 看 §9 自己的背景该走哪条路。最后读 §6-§8 简历 + 红旗 + STAR — 这是冲刺阶段才会用的。
02

类 AI 岗位画像（招聘官眼里的真实分类）

① LLM Application Engineer — 应用工程师 · ② AI Infra / Inference Engineer — 基建工程师 ⭐ 你的杠杆方向 · ③ ML Engineer / Founda…

市面上 AI JD 题目五花八门，但招聘官内部其实就分这 6 类。你要先知道自己投的是哪类，再去匹配能力。

① LLM Application Engineer — 应用工程师

📝 一句话定义
用 LLM API + 框架（LangChain / LlamaIndex / 自研）+ RAG / Agent 模式，把业务问题做成 production 产品。
  • 典型职责：搭建 RAG 系统 / 设计 Agent workflow / Prompt 工程 / 模型选型 + cost 控制 / 上线 + eval / 业务集成。
  • 典型 stack：Python（占 95%）/ TypeScript / FastAPI / LangChain / LlamaIndex / DSPy / Mirascope（2026 新）/ Qdrant / Milvus / Redis / OpenAI/Claude/Qwen/Kimi/DeepSeek API。
  • 典型公司：所有有"AI 产品"的公司都缺。国内：字节豆包/扣子、阿里通义/夸克 AI、腾讯元宝、百度文心、美团 AI 助手、小米小爱、京东言犀、AI 创业公司全部。智驾：座舱 AI 助手团队。
  • 招聘难度：Junior 易，Senior 中等，Staff 难（要能讲清"为什么这个 RAG 准确率从 60% 提到 92%"的方法论）。
  • 转岗友好度：⭐⭐⭐⭐⭐（后端 / 全栈最容易转）

② AI Infra / Inference Engineer — 基建工程师 ⭐ 你的杠杆方向

📝 一句话定义
让大模型跑得起来 + 跑得快 + 跑得便宜。从 GPU kernel 到推理引擎到分布式训练，写 production 级 C++/CUDA/Triton。
  • 典型职责：vLLM/SGLang/TRT-LLM 二次开发 / 写 CUDA kernel / 模型量化 (INT4/FP8) / 推理服务性能优化 / 分布式训练 framework / KV Cache 调度 / Multi-LoRA serving。
  • 典型 stack：C++ / CUDA / Triton / Python / PyTorch internals / NCCL / TensorRT-LLM / vLLM / SGLang / DeepSpeed / Megatron-LM / FlashAttention / FasterTransformer。
  • 典型公司：国内 字节 ByteBrain / Doubao Infra、阿里通义 Infra、DeepSeek Infra、Moonshot Infra、智谱 Infra；智驾 小鹏/蔚来/理想/华为/Momenta 感知推理团队；创业 SiliconFlow / Hyperbolic / NetMind 等推理服务平台。
  • 招聘难度：Senior+ 极度短缺。能写 CUDA kernel 的中国大陆候选人估计 < 5000 人，能在 vLLM/SGLang 提 PR 的 < 500 人。
  • 转岗友好度：⭐⭐⭐⭐⭐（C++ 系统编程 / 性能优化老兵最佳路径）。你已有的 C++/内存模型/并发/低延迟 60% 直接复用，缺的是 CUDA + LLM 特定知识（vLLM 内部 / PagedAttention / 量化 / Triton lang）。
C++ 老兵转 AI Infra，是 2026 国内市场 性价比最高 + 竞争最少 + 薪资增幅最大 的路径。详见 §9 "转岗路径 · 路径 A"。

③ ML Engineer / Foundation Model Engineer — 训练工程师

📝 一句话定义
让模型变好。Pretrain 数据 curation → SFT → RM → RLHF/DPO，让模型能力 + 5% 是日常 KPI。
  • 典型职责：数据 pipeline / 数据合成 / SFT / DPO / RLHF / 评测 / model card / 长上下文 / 多模态对齐 / 模型 release。
  • 典型 stack：PyTorch / DeepSpeed / FSDP / Megatron / TRL / vLLM (eval 用) / wandb / 大量 SQL+Spark+Ray 做数据。
  • 典型公司：DeepSeek / Moonshot / 阿里通义 Qwen team / 字节豆包 model team / 智谱 ChatGLM team / 百度文心 / 阶跃 / 百川。这是最 "fancy" 但门槛最高 的方向。
  • 招聘门槛：硕士+，国内顶校 (清北复交 + CMU/Stanford/MIT 等)，有可量化的模型贡献（参与某 release 的某能力提升 X%）。Junior 几乎只招实习转正。
  • 转岗友好度：⭐⭐（最难的方向 — 需要长期数据+训练经验沉淀）

④ AI Research Engineer — 研究工程师

📝 一句话定义
0 → 1 探索。论文复现 / 新架构 / 新训练方法 / 长上下文 / 多模态融合，未必上线，但需要可发表 / 可专利 / 可成为下个模型的核心。
  • 典型职责：读 + 写 paper / 复现 SOTA / 提新 idea + 实验 / 优化 attention 变体 / 训练算法创新。
  • 典型 stack：PyTorch / JAX / 数学推导深 / 大量实验记录。
  • 典型公司：DeepSeek / 月之暗面 / OpenAI / Anthropic / Google DeepMind / Meta FAIR / 智源 / 上海 AI Lab / 阿里达摩 / 字节 SEED。
  • 招聘门槛：PhD / 顶会一作（NeurIPS / ICML / ICLR / CVPR / ACL），有突破性工作（如 DeepSeek-V3 MLA 设计、Qwen3 MoE 架构、FlashAttention）。极少招本科+硕士。
  • 转岗友好度：⭐（PhD 路径，工业界转过来基本只能曲线救国）

⑤ AI Tech Lead / AI EM — 技术领导

📝 一句话定义
让一个 AI 团队 + 一个 AI 产品 同时跑起来。模型选型 + GPU 预算 + 团队招聘 + 跟 PM/法务/SRE 对齐 + 上线节奏。
  • 典型职责：定义技术路线（自研 vs API vs 微调）/ 跟 leadership 谈 GPU 预算 / Eval + Safety 体系 / 招聘 + 团队建设 / Cost 治理 / Incident handling / 给 VP 写 status update。
  • 典型背景：5+ 年工程 + 1+ 年带过 AI 项目 + 有完整上线经验。技术 / 商业 / 人 三轴都要打开。
  • 典型公司：所有有 AI 团队的公司都缺。国内 大厂 P8/P9 级别。
  • 招聘难度：极度短缺。市场上 80% 的 "AI EM" 其实是 "管了一个 AI 子方向的传统 EM"，真正端到端做过 AI 产品 + 团队的 EM < 1%。
  • 转岗友好度：⭐⭐⭐⭐（传统后端/系统 EM 1-2 年深度参与 AI 项目即可）

⑥ AI PM — AI 产品经理

📝 一句话定义
用 AI 做出用户真的会用的产品。AI 产品 ≠ 传统产品 + LLM 调用；要懂 eval / 概率思维 / 数据驱动 / cost / safety tradeoff。
  • 典型职责：定义 AI 产品愿景 / 设计 eval（不是只看 thumbs up）/ 跟模型团队/工程团队对齐 / 用户调研 / 失败 case 分析 / cost vs quality trade-off。
  • 典型背景：PM 经验 + 强数据敏感性 + 至少跑通一个 LLM 产品 + 能跟工程聊技术（不需要写代码，但要懂"为什么这个 case 失败"）。
  • 典型公司：字节豆包/扣子、阿里通义/夸克、Moonshot Kimi、智谱、Anthropic Claude PM、OpenAI ChatGPT PM。
  • 招聘难度：中等。"懂 LLM 的 PM" 比 "懂 PM 的 ML 人" 多，所以PM 转 AI PM 比 ML 转 PM 容易。
  • 转岗友好度：⭐⭐⭐⭐（传统 PM 自己做几个 GPT 项目就能转）

给候选人的 takeaway：
  • AI Infra Senior 是性价比之王 — 同级别比 LLM App 高 20-30%，比传统 backend 高 50%+，竞争人数还少。
  • 智驾 / 具身大模型是 2026 新增量 — 总包带宽接近 AI 独角兽，且对 C++/CUDA/系统能力的需求和你已有 skills 高度匹配。

  • 研究岗是头部游戏 — 没顶会一作 + PhD 不建议投，错位竞争只会受伤。

JD 关键词解码 — 招聘官写 JD 时真正在想什么

任职要求段 · 解码表 · 工作内容段 · 看出岗位真实定位

JD 是双向密码。你读 JD 时要做 3 件事：① 区分 真要求 vs wishlist ② 看出岗位真实定位（是 App 还是 Infra）③ 推断对方 pain point。下面是常见 JD 段落的招聘官内心 OS。

5.1 任职要求段 · 解码表

JD 原文招聘官真实意思你的对策
"熟悉 Transformer 架构 / 主流 LLM" 必考。会问 KV-Cache、attention 复杂度、为什么 GQA。不是 wishlist。 读完本系列 X06
"有 vLLM / SGLang / TRT-LLM 等推理框架经验" AI Infra 岗。会问到内部实现（PagedAttention、Continuous Batching 调度）。"用过" 不够。 读完 Y02，最好提个 PR
"熟悉 RAG 系统设计与优化" LLM App 岗。必考你的失败 case + 优化数字。"用过 LangChain" 是减分项。 读完 X08，准备 1-2 个真实数字案例
"有 CUDA / GPU kernel 开发经验" AI Infra 硬核岗。会让你白板写 / 改 kernel。"了解 CUDA" 不够。 读完 Y01 + 写过 1-2 个 kernel + 知道 Tensor Core
"熟悉分布式训练（DDP / FSDP / Megatron）" ML Eng / Infra 岗。会问 ZeRO 各阶段差异 + 训练崩了怎么 debug。 读完 Y03 + 跑过千卡训练（或讲清楚为什么没机会跑过）
"了解 RLHF / DPO / Reward Model" ML Eng 岗。会推公式 + 讨论 reward hacking。 读完 X07，能讲 DPO loss 推导
"有 LangChain / LlamaIndex / DSPy 经验" 初/中级 LLM App 岗。2026 招聘官眼中 "LangChain" 已经偏负面（认为是 over-engineered）。 讲 "用过但不推荐"，或讲自研 framework
"熟悉 prompt engineering" 初级岗专属。Senior+ JD 出现这个 = 这个团队水平不高。 判断是否值得投
"了解 Agent / Multi-Agent / Function Calling" LLM App 中高级。会问 MCP / Skills / Tool 选择策略。 读完 X09 + X13
"有 Eval / benchmark / 自动评测体系经验" ML Eng / AI PM。会问 "为什么 MMLU 不能用作你产品的 metric"。 读完 X12
"了解 model serving 性能优化" AI Infra。必考 throughput / latency / cost 三角 trade-off。 能讲 batch size × seq len 对 latency 的影响
"熟悉量化 (INT4 / INT8 / FP8) / 蒸馏" AI Infra。会问 GPTQ vs AWQ vs SmoothQuant 算法差异 + 精度损失原因。 读完 Y04
"有顶会论文 / 开源贡献优先" 不是 wishlist。Research 岗几乎是硬门槛；其它岗有的话直接 +1 档。 没有则别投 Research；其它岗主推 production 数字
"3 年以上相关经验" 实际 1.5-5 年都看。更看 "做过什么"，不严卡年限。 项目密度优先于年限
"良好的英文读写能力" 读 paper + 提 issue + 跟海外 OSS / 外籍同事沟通。不需要口语 fluent（除非外企）。 能读 paper 就够

5.2 工作内容段 · 看出岗位真实定位

✅ 这是 Infra 岗的信号

  • "优化推理 / 训练 throughput"、"降低 latency"、"GPU 利用率"
  • "开发 / 优化 CUDA kernel"
  • "参与 vLLM / SGLang / TRT-LLM 二次开发"
  • "分布式训练框架"、"NCCL"、"3D parallel"
  • "模型量化 / 压缩"、"INT4 / FP8"

✅ 这是 App 岗的信号

  • "业务场景"、"用户体验"、"AI 助手"、"对话产品"
  • "RAG"、"知识库"、"Agent"、"workflow"
  • "prompt 设计"、"模型选型"、"cost 控制"
  • "集成 OpenAI / Claude / Qwen API"
  • "提升答题准确率 / 用户满意度"

⚠️ 这是 ML Eng 岗的信号

  • "参与模型预训练 / SFT / RLHF"
  • "数据 pipeline"、"数据合成"、"数据清洗"
  • "Reward Model 设计"、"偏好对收集"
  • "模型能力评估 + 持续改进"
  • "参与某 release"（XXX 模型 v2 → v3）

🚩 这是要打折扣的信号

  • "AI 应用全栈" + "LangChain" + "1-3 年" → 初级
  • JD 列了 20+ 技术词 → 没明确方向，团队也没想清楚
  • "AI 工程师 / 算法工程师" 不分 → 公司新设岗位

❌ 招聘官不喜欢的写法

✅ 招聘官想看的写法（STAR-C 结构）

- 主导公司客服 RAG 系统从 POC → 生产（DAU 12k → 80k）
- 挑战：长尾问题准确率只有 47%，幻觉率 18%
- 方案：① 上线 Hybrid Search (BM25+bge-large) ② 接入
  Cross-Encoder rerank ③ 引入 self-RAG 自评 + 兜底降级
- 结果：准确率 47% → 89%；幻觉 18% → 3.2%；
  P95 latency 8.4s → 1.6s；月调用成本 -42%（GPT-4 → Qwen-72B）
- 额外：方案沉淀为团队 RAG 模板，赋能 3 个兄弟产品

为什么好：有规模（DAU）/ 有挑战（具体数字）/ 有方案（带技术名词）/ 有结果（4 个数字）/ 有影响力（赋能他人）。招聘官打分 +1 档。

6.3 简历必备 5 段

  1. Headline：1 行（"5 年 C++ + 2 年 LLM Infra，主攻推理引擎 + 量化，vLLM 提过 3 个 PR"）

  3. 工作经历：最近 1-2 份每段 4-6 个 bullet，用上面的 STAR-C 结构
  4. 项目 / 开源 / 论文：单独一段，这是 AI 岗的差异化关键

STAR 故事库 — AI 岗专属

必备 7 类 STAR 故事（建议每类准备 1-2 个） · 范例：一个 Latency Win 故事（90 秒版）

STAR (Situation / Task / Action / Result) 是行为面试的标准结构。AI 岗特有的故事类型：

8.1 必备 7 类 STAR 故事（建议每类准备 1-2 个）

故事类型典型问题必含元素
Quality win（提升质量） "讲一次你显著提升某模型/产品质量的经历" 明确的 metric（accuracy / hallucination / 用户满意度）+ before/after 数字 + 关键技术决策
Cost win（降低成本） "如何在不降质的前提下降低 cost" GPU 时长 / token cost / API call 数 / 月成本，before/after
Latency win（性能优化） "讲一次推理 / 训练性能优化" P50/P95/P99 latency 或 throughput before/after，关键 bottleneck 分析
Eval win（建评测体系） "你怎么知道模型/产品变好了" 从只看 thumbs up → 建立 offline + online + LLM judge 多维 eval 的过程
Failure / Incident（失败/事故） "讲一次 AI 产品上线后出问题" 具体事故描述 + 你的角色 + 应急方案 + 长期 mitigation + 学到的
Cross-functional（跨职能） "跟 PM/法务/SRE 怎么对齐" 具体冲突 + 你的处理 + 双方让步 + 最终结果
Tech bet（技术选型） "为什么选 X 不选 Y" 明确的选项 + 你的评估维度 + 决策 + 1 年后的复盘

8.2 范例：一个 Latency Win 故事（90 秒版）

🎤 范例答题 — 90 秒版（适合电话筛）

S · 我们公司的客服 RAG 系统上线 6 个月后，P95 latency 是 8.4 秒，用户大量流失。CEO 拍下来"必须 2 秒以内"。

T · 我作为 tech lead，要在 6 周内把 P95 降到 2s，不能降准确率（当时 89%）。

A · 我做了 3 个事情。① 用 perf trace 拆解 latency，发现 65% 时间在 LLM generation（GPT-4-turbo）。② 评估替代：开源 Qwen-72B + vLLM 自部署，测了 8 个开源模型 + 5 个量化方案，最终选 Qwen-72B + FP8 量化 + vLLM continuous batching。③ Prefill / Decode 分离 + 用 GPT-4-mini 做兜底质量门。

R · 6 周后 P95 = 1.6s，准确率 89% → 91%（rerank 升级带来），月成本 GPT-4 → Qwen 自部署省 42%（含 GPU 折旧）。CEO 直接给我升了一档。

+ · 沉淀经验：写了一篇内部技术博客，被复用到 3 个兄弟团队；用这个 case 跟开源社区 vLLM 提了 1 个 PR。

STAR 黄金法则：
  1. S+T 不要超过 25%（很多人花 70% 讲背景，A+R 草草收尾）
  2. R 必须有 ≥ 3 个数字（quality / cost / time 三个维度尽量都出）
  3. 每段最多 90 秒，准备 30s / 90s / 3min 三个版本
  4. 用 "我"，不要用 "我们"（用 "我们" > 30% 直接判 reject 信号）
