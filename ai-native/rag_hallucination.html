AI Engineering · RAG 进阶

RAG 幻觉治理：让模型「有据才说，无据就认」

很多人以为「接了 RAG 就不会幻觉」——错。即使检索到了正确文档，模型仍可能忽略上下文、过度外推、把检索内容和自己的先验混着编。RAG 幻觉是生产事故的头号来源，也是「Demo 惊艳、上线翻车」的分水岭。本篇建立一套事前防御 → 事中约束 → 事后校验 → 评测兜底的四道防线体系，配一个可交互的 Faithfulness 逐句检测演示，讲清「怎么让答案有据可依、无据时老实认怂」。

接了 RAG，为什么还会「一本正经地胡说」

检索 ≠ 遵从 · RAG 降低了幻觉，但远没消灭它

RAG 的初衷正是治幻觉：给模型喂真实资料，让它「照着资料答」而不是「凭记忆编」。它确实大幅降低了幻觉率——但没有清零。因为「把资料放进上下文」和「模型真的遵从资料」是两码事。模型可能：检索给了对的它没看、检索给了错的它照单全收、检索没提到的它自己脑补。幻觉从"模型编"升级成了"模型没老老实实用检索结果"。

RAG 像开卷考试：允许你翻书答题，作弊（凭空捏造）的概率大降。但开卷≠满分——有的学生没翻到对的那页（检索失败）、有的翻到了却看漏了（忽略上下文）、有的书上没写就自己瞎补（过度外推）、还有的把书上的和自己记的混着写，结果串了（先验冲突）。幻觉治理，就是把这些"开卷考试还能答错"的漏洞逐个堵上。
RAG 幻觉（Hallucination in RAG）：RAG 系统生成的、无法由检索到的上下文支撑的内容。注意它不等于「事实错误」——一个内容可能事实正确但检索里没有（也算幻觉：无据），也可能检索里写了但模型答反了。RAG 语境下衡量的核心是 Faithfulness（忠实度 / 是否忠于给定上下文），而非笼统的"对不对"。
为什么必须专门治理：在客服、医疗、金融、法律等场景，一句"听起来很专业但其实是编的"回答，比"我不知道"危害大得多——用户会信。RAG 幻觉是这些领域能否上线的一票否决项。
01

幻觉的成因链路：先分类，才能对症

不同成因要用不同防线，无差别"加 prompt"是治不好的

1.1 三类 RAG 幻觉

① 检索失败型

压根没召回到相关文档（召回差 / 库里就没有），模型只能凭先验编。根在检索，不在生成。

② 忽略上下文型

检索对了，但模型没用它——被自身先验带跑，或长上下文里"看漏了"（迷失在中间）。

③ 过度外推型

上下文只说了一部分，模型自作主张"补全""推断"出上下文没有的结论，听起来很顺但无据。

先分诊再开方：检索失败型 → 去优化检索/改写（见 rag_deep_dive.html / rag_query_rewriting.html）；忽略上下文型 → 事前 grounding prompt + 事中引用约束；过度外推型 → 事后 faithfulness 校验 + 弃权兜底。无脑往 prompt 里堆"请不要编"，只对第 ② 类有点用，治不了另外两类。

1.2 成因归因图：一条从提问到幻觉的链路

幻觉不是单点问题：切分、检索、拼装、生成每一环都可能引入。所以治理也不是"一个 prompt"，而是覆盖全链路的纵深防御。
迷失在中间（Lost in the Middle）：模型对长上下文开头和结尾的信息利用最好，中间的容易被忽略。这是"忽略上下文型"幻觉的常见诱因——把最相关的片段放在上下文中部，模型可能视而不见。对策：rerank 后把最相关的放到首尾，控制上下文长度。
02

四道防线总览

纵深防御 · 事前 / 事中 / 事后 / 评测兜底，逐层拦截

没有单一"银弹"能消灭幻觉。工程上靠的是纵深防御（defense in depth）：在生成前、生成中、生成后、以及长期评测四个环节各设一道关，让幻觉即使漏过前一道，也会被后一道拦下。下面先看全景，后面各章逐一展开。

① 事前防御
生成之前。用 grounding prompt 约束"只依据给定资料回答"、提供检索质量兜底（召回太差就别硬答）、允许拒答。成本最低、性价比最高的第一关。
② 事中约束
生成过程中。要求逐句标注引用（citation）、输出结构化格式（先证据后结论）、约束"没有出处的话不要写"。让答案天然可追溯。
③ 事后校验
答案生成后。做 faithfulness 校验（逐句判断是否被上下文支撑）、引用溯源核对、self-check（模型自查/二次核对），发现不忠实就重生成或标注。
④ 评测兜底
长期与线上。用 faithfulness/groundedness 指标持续评测、监控幻觉率；触发阈值就弃权（abstain）、转人工或降级。把治理从"一次性"变成"闭环"。
这就像食品安全的多重把关：进货前查供应商资质（事前）、加工时全程留样标签（事中）、出厂前抽检化验（事后）、上市后持续抽查+出问题召回（评测兜底）。任何一关都不敢保证 100%，但四关叠加，漏网的概率被压到可接受。幻觉治理也是同一个哲学——不追求单点完美，追求纵深可靠。
四道防线的成本与延迟递增：事前几乎零成本（改 prompt），事中略增（要求引用），事后最贵（额外的校验调用）。工程上按场景取舍：高风险场景（医疗/金融）四道全上；低风险内部工具，事前+事中可能就够。
03

事前防御：Grounding 与「敢于拒答」

最便宜的一关 · 用 prompt 把模型"锚"在给定资料上

第一道防线在生成之前，核心是两件事：把模型锚定在检索资料上（grounding），以及给模型"我不知道"的许可（拒答）。很多幻觉的根源，是模型被"必须给个答案"的隐性压力逼着编——明确告诉它"没有依据就说没有"，能拦下一大批。

Grounding（接地 / 锚定）：通过 prompt 明确要求模型「只依据提供的上下文回答，不要使用你自己的知识」，把生成"锚"在给定资料上。它把模型从"开放问答"切换到"阅读理解"模式，是降低幻觉最简单有效的一招。
// 一个 grounding + 拒答的 system prompt 骨架
你是一个严格基于资料回答的助手。规则：
1. 只能使用下面【资料】中的信息回答，禁止使用你自己的知识补充。
2. 如果【资料】不足以回答，直接说「根据现有资料无法回答」，不要猜测。
3. 每个结论后用 [1][2] 标注它来自哪段资料。
4. 资料之间若有冲突，指出冲突而不是自行选一个。

【资料】
[1] {chunk_1}
[2] {chunk_2}
...
【问题】{query}

给"拒答"的许可

明确写"无据时说不知道"。人类客服敢说"这个我需要确认下"，模型也要被允许如此。缺了这句，模型倾向硬编。

检索质量门槛

若 top-1 相似度低于阈值 / 召回为空，根本不进生成，直接返回"没有找到相关资料"。从源头掐断"检索失败型"幻觉。

冲突显式化

资料互相矛盾时，要求模型"指出冲突"而非默默挑一个。避免它"编一个和稀泥的答案"。

约束回答范围

"只回答资料覆盖的部分，未覆盖的明确说明"。防止模型把 80% 有据 + 20% 脑补缝合成一个看不出破绽的整体。

Grounding prompt 不是万能：它主要治"忽略上下文型"，对"检索失败型"（库里根本没有）只能靠拒答 + 检索质量门槛兜住。而且过强的 grounding 会让模型变得"死板"（明明能合理推断也拒答），需要按场景调松紧。
04

事中约束：逐句引用 + 结构化输出

让答案「自带出处」，把可追溯变成硬约束

第二道防线在生成过程中，核心是逼模型边答边给出处。当每句话都必须标注"这来自资料[几]"时，模型很难再凭空编——因为编的话没法给出处。引用不仅方便用户核查，还反向约束了生成：要引用，就得真的从资料里找依据。

引用 / 归因（Citation / Attribution）：要求模型在答案中逐句或逐结论标注其依据来自哪段检索资料（如 [1][3]）。它让答案可被逐条核查，也在生成时约束模型"无出处不下笔"，是可信 RAG 的标配（如 Perplexity、各类 AI 搜索的角标引用）。
引用把"答案"和"证据"一一挂钩。事后校验（下一章）正是沿着这些 [n] 反查——引用是可核查性的基础设施。

结构化输出

让模型输出 JSON：{claim, evidence_id, quote}。先证据后结论的结构，天然抑制"先下结论再找理由"式的脑补，也便于程序化校验。

强制引用 / 无引用不输出

规则："每个事实性陈述必须带 [n]；给不出出处的内容，删掉或标为不确定。" 把"可追溯"从建议变成硬约束。

抽取式优先

高风险场景可要求"尽量引用原文片段"而非改写，抽取式（extractive）比生成式更难跑偏——代价是不够自然。

区分"资料内 vs 常识"

若允许模型用常识补充，要求它明确标注哪些来自资料、哪些是它的推断，让用户能分辨可信度。

引用要"真"：模型有时会编出"看起来有 [2] 但 [2] 根本没这内容"的假引用。所以引用本身也要被事后校验（下一章）——不能只看"有没有角标"，要核"角标指的那段到底支不支持这句话"。
05

事后校验：Faithfulness 逐句核对 + 自检

答案生成后，再过一遍"这句话，资料撑得起吗"

5.1 Faithfulness 校验：把答案拆成句，逐句判定

最硬核的一道关：把生成的答案拆成一条条原子陈述（claim），对每条判断"检索上下文是否支撑它"（supported / not-supported）。支撑的留，不支撑的就是幻觉——重生成、删除、或标注为"不确定"。支撑比例就是 Faithfulness 分数。

Faithfulness（忠实度） / Groundedness（有据性）：衡量答案中的陈述有多少能被检索上下文支撑。计算方式：拆出 N 条 claim，判定其中 M 条被上下文 entail（蕴含），Faithfulness = M / N。它是 RAG 幻觉的核心度量——不看答案"对不对"，只看"忠不忠于给的资料"。
NLI / 蕴含判定（Entailment）：判断"上下文是否蕴含某条陈述"的技术。可用轻量 NLI 模型，也可用 LLM-as-judge（让另一个 LLM 判"这句话资料支不支持"）。是 faithfulness 校验和 self-check 的底层判定器。

拆句 → 判定

答案 → 原子 claim 列表 → 每条与上下文做蕴含判定 → 算支撑比例。RAGAS 等库把这套自动化了。

Self-Check / 二次核对

让模型"复查自己的答案：逐句核对资料，把没依据的标出来"。或采样多次答案看一致性（SelfCheckGPT 思路）。

不达标怎么办

低于阈值：① 删掉不支撑句；② 带着"请仅用资料"重生成；③ 降级为弃权/转人工（见下一章）。

06

评测闭环与兜底：把治理变成长期机制

离线评测 + 线上监控 + 弃权/人审，闭环收尾

前三道防线作用于"单次请求"。第四道把视角拉到长期：用指标持续量化幻觉、上线后监控、并在系统"没把握"时优雅地弃权而不是硬答。没有评测闭环，你甚至不知道自己的幻觉率是升是降。

离线评测

用带标注的 eval set 跑 Faithfulness、Answer Relevance、Context Precision/Recall 等指标（RAGAS / TruLens）。每次改动前后对比，防止"改好了 A 却退化了 B"。详见 ai_eval.html。

线上监控

对线上流量抽样跑 faithfulness 校验，把幻觉率做成看板 + 告警。用户点踩、"这不对"反馈也是重要信号。

弃权 Abstain

检索质量差 / faithfulness 低 / 置信度低时，主动说"无法确定"或转人工，而不是硬编。"我不知道"是特性，不是 bug。

降级与人审

高风险问题走人工兜底；或降级为"给出相关资料链接让用户自读"，把最终判断交还给人。

RAGAS / TruLens：主流 RAG 评测框架。用 LLM-as-judge 自动算 Faithfulness、Answer Relevancy、Context Precision/Recall 等指标，无需大量人工标注，是把幻觉治理"量化 + 闭环"的常用工具。
弃权就是老中医的"这个我看不准，建议你去做个检查"：一个敢说"我不确定"的系统，比一个"什么都敢答"的系统更值得信任。产品设计上要把"无法确定"做成体面的一等公民（给替代路径、给人工入口），而不是让它显得像故障。
治理优先级：如果预算有限，先做"事前 grounding + 拒答"（几乎免费、收益最大）和"离线 faithfulness 评测"（让你看得见问题）。行有余力再上"逐句事后校验 + 线上监控"这类较贵的重武器。
07

小结 · 面试 · 术语表

收束成判断力

一页纸总结

① 检索 ≠ 遵从

RAG 降幻觉但不清零：检索失败、忽略上下文、过度外推三类，各有各的成因。

② 纵深四道防线

事前 grounding+拒答 → 事中引用+结构化 → 事后 faithfulness 校验 → 评测闭环+弃权。

③ Faithfulness 是核心度量

不看"对不对"，看"忠不忠于给的资料"。拆句判定支撑比例，低了就弃权。

面试自测（点击展开）

基础接了 RAG 为什么还会幻觉？有哪几类？
前提"把资料放进上下文" ≠ "模型遵从资料"。
三类① 检索失败型（没召回到，只能编）；② 忽略上下文型（召回对了却没用/看漏）；③ 过度外推型（上下文没说的自己补）。
要点不同类根因不同：①在检索，②③在生成——要分诊对症，不能一个 prompt 打天下。
核心什么是 Faithfulness？怎么算？和"答案正确性"有何区别？
定义答案中有多少陈述能被检索上下文支撑。拆出 N 条 claim，M 条被上下文蕴含，Faithfulness = M/N。
区别它衡量"忠于给定资料"，不是笼统的"对不对"——一句事实正确但资料没提的话，faithfulness 角度也算无据。
实现判定用 NLI 模型或 LLM-as-judge；RAGAS/TruLens 把它自动化。
辨析说说治理 RAG 幻觉的四道防线，成本各如何？
事前grounding prompt + 拒答 + 检索质量门槛。几乎零成本，性价比最高。
事中逐句引用 + 结构化输出。略增，让答案自带出处、可追溯。
事后faithfulness 逐句校验 + self-check。最贵（额外调用），拦漏网的。
兜底离线评测 + 线上监控 + 弃权/人审。把治理变闭环。按风险取舍层数。
工程模型给了引用 [2]，就一定可信吗？
不一定模型会编"假引用"：标了 [2]，但 [2] 那段其实没这内容。
对策引用本身要被事后校验——核对"[2] 指的那段到底支不支持这句话"，而不只是看有没有角标。
延伸这正是 faithfulness 校验在做的事：把引用当作"待验证的声明"，逐条回查。
综合设计一个医疗问答 RAG，如何把幻觉风险压到可上线？
事前强 grounding + 强制拒答 + 高检索质量门槛（召回不够好直接不答）。
事中抽取式为主 + 逐句强制引用；区分"资料内"与"一般建议"。
事后faithfulness 逐句校验，低于高阈值则不出答案；假引用一律拦截。
兜底高风险问题弃权 + 引导就医/转人工；线上持续监控幻觉率 + 明确免责与人工入口。
原则宁可多说"请咨询医生"，绝不编一个像模像样的错误答案。

术语表速查 · Glossary

① 幻觉与成因

RAG 幻觉 Hallucination生成的、无法被检索上下文支撑的内容。分检索失败型/忽略上下文型/过度外推型三类。
迷失在中间 Lost in the Middle模型对长上下文首尾利用好、中间易忽略。对策：rerank 后把最相关片段放首尾、控上下文长度。
先验冲突模型自身记忆（先验）与检索资料矛盾时，可能用先验覆盖资料，导致答案偏离上下文。

② 度量与判定

Faithfulness / Groundedness答案陈述被上下文支撑的比例（M/N）。RAG 幻觉的核心度量：看"忠于资料"而非"对不对"。
NLI / 蕴含判定 Entailment判断"上下文是否蕴含某陈述"。可用 NLI 模型或 LLM-as-judge，是校验的底层判定器。
Claim 拆分把答案拆成原子陈述，逐条做支撑判定，是 faithfulness 计算的前置步骤。
Answer Relevancy / Context Precision·RecallRAG 评测的配套指标：答案切题度、上下文精度/召回，与 faithfulness 一起构成质量画像。

③ 治理手段

Grounding / 锚定prompt 约束"只依据给定资料回答"，切换到阅读理解模式。事前防线核心。
引用 / 归因 Citation逐句标注结论来自哪段资料，让答案可追溯并反向约束生成。需警惕"假引用"。
Self-Check / SelfCheckGPT让模型自查答案是否有据，或采样多次看一致性来发现幻觉。
弃权 Abstain系统没把握时主动说"无法确定"/转人工，而非硬答。是可信系统的特性。
RAGAS / TruLens主流 RAG 评测框架，用 LLM-as-judge 自动算 faithfulness 等指标，实现量化闭环。
