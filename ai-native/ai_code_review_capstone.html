Coding Agent 的难不在"找出问题"（LLM 很会找）而在"少误报 + 让工程师信任"。一个 50% false positive 的 agent，3 周内就会被团队 disable 掉。整本 case study 的核心是怎么把 precision 调到工程师能接受的水位。
01

业务场景 · PR Review 为什么是 AI 的好 use case

从工程团队"瓶颈分析"看为什么需要自动化

1.1 现状：PR review 的 3 个工程团队痛点

🎬生活类比 · 编辑团队审稿
想象一个出版社：作者们每天交稿，必须先经过编辑审完才能出版。每个编辑要：① 检查错别字 ② 看事实是否有错 ③ 评判风格 ④ 提改进建议。

问题：① 编辑成了瓶颈（作者等审稿天数）② 编辑被简单错别字消耗精力（没时间看深度问题）③ 不同编辑标准不一致 ④ 编辑跳槽 → 团队 review 质量震荡。

PR review 一模一样 — 工程师就是 reviewer，PR 就是稿件。AI Code Review Agent = 给团队配一个 24/7 不累的"初级编辑"，专门处理重复工作，让 senior 工程师专注架构 / 设计层面的 review。

1.2 团队层面 3 个量化痛点

  • PR cycle time 长：典型 100+ 人团队 PR 平均 18-36 hr 才被 review · 大头是"等人有空" 而非"看代码"
  • Senior 工程师"被淹没"：每周 review 20-40 个 PR · 平均每个 30-90 min · 80% 时间花在 trivial 问题（命名 / null check / 格式）
  • Review 质量参差：紧急时 reviewer 直接 approve 走形式 · 后期 bug 翻车率 +30-50%

1.3 3 个 persona（典型用户）

👨‍💻
Junior 工程师 (开 PR 一方)
提了 PR · 等 reviewer 24 hr 后告诉你"命名不好/缺单测/null check"等基础问题 · 改完再等
PR 提交 5 min 内拿到基础 review · 修完再 ask human reviewer 看架构层问题
👨‍🏫
Senior 工程师 (review 一方)
每天被 PR 淹没 · 80% 是 trivial · 没精力看真正影响架构的 hunk · 自己代码反而 commit 不动
AI 先把 trivial 问题处理 · 我只看 "high-impact" hunks 和 architecture concern
📊
Engineering Manager
看不到团队 review quality · 不知道哪类问题最频繁 · 没法系统改进 code health
月度 dashboard：找出 top reject 模式 · 进 onboarding 培训 / linter 规则

1.4 为什么这类问题适合 AI

  • 结构化输入：PR diff / file / line 都有明确 schema · 跟 ticket 这种 free-form 比好处理多
  • 分类 + 生成混合：bug / style / security / perf 是分类 · 建议代码是生成 · LLM 两者都行
  • 评判可 verify：跑测试 / 跑 linter / 看历史 · 大部分 AI 判断有 ground truth 可以验（不像 chat agent 主观）
  • 容错友好：错了用户可以 dismiss · 不像 ticket triage 错了误导 OAM 工程师 30 min
  • 持续学习信号好：工程师 accept / reject / modify 每条 comment 都是monotonic 学习信号
02

需求 + ATAM · 质量属性 trade-off

2.1 用户故事

  • US-1：作为 Junior 工程师，开 PR 后 5 分钟内能看到 AI 的初步 review
  • US-2：作为 Senior 工程师，AI comments 必须有 severity 等级（critical / mid / nit），让我能快速跳过 nit 看 critical
  • US-3：作为 EM，月度看 dashboard：top reject 模式 / agent precision / 处理 PR 数 / cost
  • US-4：作为 OSS 维护者，agent 不能产生 spam 评论（每个 PR < 10 个 comment · 否则关闭）
  • US-5：作为 Security 工程师，agent 必须把 OWASP top-10 类问题标 critical · 不能跟 nit 混

2.2 ATAM 质量属性矩阵

属性scenario响应度量优先级
PrecisionAI 提的 high-severity comment 中误报 / 总数< 10%（高），< 20%（中）P0
Recall真实 bug 中 agent 标出的 / 全部> 60% (高), > 40% (中)P1
LatencyPR 提交 → AI 完整 review 出现P50 < 60s, P95 < 5 minP0
Cost每个 PR 平均 AI cost< $0.20 / PR (target $0.05)P1
Trust工程师 mute / 反馈机制能 mute / 反 / acceptP0
Modifiability加新检查类型（如 perf rule）耗时1 天内可上线（用 Skills 不需要改 code）P1
Privacycode 是否进 vendor LLM必须配置 zero-retention / 或自部署P0
Auditability每条 comment 可追溯到 LLM call每 comment 有 trace_id + reasoningP1

2.3 关键 trade-off

  • Precision vs Recall：宁要少漏报多，不要误报多（工程师对误报 0 容忍）→ ADR-5 处理
  • Latency vs Cost：并行 5 个 subagent 提速但 cost +3-5x · 取舍：核心 PR 并行，小 PR 串行 → ADR-4 处理
  • "全自动" vs "工程师可控"：完全自动 = 工程师失控 · 加 mute / accept-modify-reject 反馈 → ADR-5
  • "用 frontier 模型" vs "用 mini 模型"：好质量但贵 · 路由 → ADR-3
03

5 个关键 ADR · 第一人称决策记录

用 ADR 模板记录推理过程 · 模拟"系统设计面试题 + 答案"
ADR-1
触发模式：PR diff 钩子 vs 主动扫 vs commit 增量

Context

团队希望 AI 在用户体验最好的时机给 review。太早（commit 增量）打扰开发流程；太晚（merge 后）失去价值。

Options

  • A · PR 打开/更新时触发：webhook 收到 pull_request opened / synchronize → 跑完整 review
  • B · 每次 commit 都跑：实时反馈但 noise 大 · cost 高 (commit 数 ≫ PR 数)
  • C · 用户 @bot 主动调用：精准但用户要主动 · 利用率低
  • D · 主动扫描 main：找历史 commit 中的隐藏问题 · 对 PR review 无直接价值

权衡

A 最贴合工程习惯（PR 是 git review 的标准 unit）· B cost 太高且 noise (push -f 重写历史会重跑) · C 错过用户没主动叫的 PR · D 是 nice-to-have 不是 must-have。A 兼顾价值 + cost。

采用 A · PR opened / synchronize 触发 · 同时支持 user @-mention 强制重跑 (C 作为补充)。不支持 commit 级触发 (B)。后续若 OSS 项目反馈想要"持续监控"，再加 D 作为独立 cron。
ADR-2
Comment 粒度：per-file vs per-line vs per-hunk

Context

AI review 输出怎么落到 GitHub PR · 不同粒度影响阅读体验 + 信息密度。

Options

  • A · 一个 PR-level summary comment：1 个大 comment 列出所有发现 · 简洁但难定位
  • B · per-file comment：每个改动文件 1 个 comment · 介于粗细之间
  • C · per-line review comment：精确到行 · 用户上下文最好但容易刷屏
  • D · 混合：top-level summary + per-line for high-severity：综合两者优点

权衡

A 信息密度低但不刷屏 · C 信息密度最高但 OSS 项目超过 15 个 line-comment 会被维护者烦 · B 折中但 boring。D 是各成熟 review bot (CodeRabbit / GitHub Copilot review) 的共同选择。

采用 D · 1 个 PR top-level summary (列出 5 个 critical + 5 个 mid + count of nit) + per-line comment 仅对 critical 和 mid (绝不对 nit 发 per-line)。每个 PR comment 数硬上限 15 个 (防 spam · OSS 友好)。
ADR-3
Model 选型 + Routing：单一 frontier vs 多模型分流

Context

用 Claude 3.5 Sonnet 单一模型 vs 用 mini 模型当 first-pass + Sonnet 升级处理复杂 case。cost 差 6-10x · quality 差 ~10%。

Options

  • A · 纯 Claude 3.5 Sonnet：质量上限 · $3/M input + $15/M output · PR ~5K input 15K output → $0.24/PR · 月预算翻车
  • B · 纯 GPT-4o-mini：cost 低 ($0.0036/PR) 但 critical issue 漏报率 +20%
  • C · DeepSeek-V3 自部署：极便宜但需要 AI Infra 团队 · 团队规模不够
  • D · Two-tier routing：先用 mini 跑全部 file → 高 ambiguity 文件升级 Sonnet · 平均 cost $0.04 · quality 接近 A

权衡

A 钱烧不起 · B 质量降太多 · C 还没到自部署规模 · D 是工程平衡。routing 判断条件：① 文件 LOC > 500 ② 改动 hunk > 50 行 ③ 包含 SQL / auth / crypto / payment 关键字 → 升级 Sonnet · 其它走 mini。

采用 D · two-tier routing。注意：每个 subagent (bug/security/style/perf/doc) 各自做 routing 决策 · security 一律走 Sonnet 不省 cost · style 一律走 mini 不浪费 cost。Skills 用 mini · Aggregator 用 Sonnet（综合 reasoning 难度大）。
ADR-4
5 Subagent + 1 Aggregator vs 单 god-prompt agent

Context

给 LLM 一个"全能 prompt" 让它一次 review 所有 aspect (bug + security + style + perf + doc)，vs 5 个 专门 subagent + 1 个 aggregator 综合。

Options

  • A · 单 god-prompt：1 个 prompt 列出"你是 reviewer，找 bug / security / style ..." · LLM 一次输出 N 条 comment
  • B · 5 个 specialized subagent：每个独立 prompt + 独立工具集 · 并行跑 · 结果给 aggregator 聚合
  • C · CrewAI 多 agent 协作 (大模型互相 prompt)：agent 之间发消息 · cost / latency 极高

权衡

A 简单 cost 低但 quality 差 (LLM 注意力分散，看 security 时漏 perf · 看 perf 时漏 bug) · C 学术 fancy 但 production 不实用。B 是 production agent (Cursor / Devin / Claude Code) 主流，每个 subagent 单一 focus 质量更高 + 可独立 eval / 独立 disable / 独立 model routing。

采用 B · 5 个 subagent (BugHunter / SecurityReviewer / StyleCritic / PerformanceAuditor / DocReviewer) + 1 个 Aggregator (去重 + 排序 + 综合 narrative)。每个 subagent 独立的 prompt + tools + model routing。subagent 之间不直接通信 (用 handoff schema 给 aggregator)。
ADR-5
False Positive 处理：静默 vs 标记 vs 学习

Context

Coding agent 必然有 false positive (FP)。工程师对 FP 的容忍度极低 · ~15% FP rate 就开始抱怨。处理 FP 直接决定产品生死。

Options

  • A · 静默忽略：FP 出现工程师 thumb-down · agent 无反应 · 下次还犯同样错
  • B · 标记问题但不学习：记 FP 到 log · 人工定期看 · cost 高 + 慢
  • C · 自动 learn → 调 prompt：每次 FP 自动更新 system prompt · 漂移风险
  • D · Skills 反馈循环 + reviewer-style allowlist：FP → 生成"skill 不再举报这种 case" + 项目级 allowlist (如 ".aireview/allow.yaml") + 月度 audit

权衡

A 用户体验崩 · B 操作 cost 高 · C prompt drift 风险大（一个 user wrong feedback 影响所有人）· D 平衡——allowlist 是结构化的、可 review 的、跟项目相关的。

采用 D。具体：① 每条 comment 带 👍 / 👎 / 🤔 (need more info) 按钮 ② 👎 触发"添加到项目 allowlist" 提示 ③ 工程师可 commit ".aireview/allow.yaml" 到 repo (review 一次后 agent 不再举报) ④ 同时EM dashboard 看 top-15 FP pattern，每月 review 决定是否加入 system skill (跨项目)。不自动改 prompt (避免 drift)。

系统架构 (C4 简化版)

4.1 Context 层 · 跟谁打交道

  • 上游：GitHub / GitLab webhook (PR opened / synchronize 事件)
  • 下游：GitHub PR comment API (post review) + GitHub status API (check run) + 团队 dashboard
  • 外部依赖：① OpenAI / Anthropic API (LLM) ② Snyk / Semgrep (静态分析 SaaS) ③ pgvector (codebase embedding 存储) ④ S3 / GCS (artifact: agent trace / repo snapshot)

4.3 端到端数据流

1
Webhook 收 PR opened · 鉴权 · 入 queue (job_id = PR id)
2
Orchestrator 拉 diff + 准备 context：调 Codebase Index 拉相关函数 / git history / test coverage
3
5 个 subagent 并行：BugHunter / SecurityReviewer / StyleCritic / PerfAuditor / DocReviewer · 每个独立调 LLM + tools · 输出按 schema 的 finding 列表
4

关键架构决策：subagent 之间独立 + 并行 + 不互相通信。这是跟 X16 的 hypothesis flow 最大差异 — X16 是串行依赖 (假设生成 → 验证), X17 是并行独立 (5 个 aspect 同时评)。不同 agent 范式对应不同业务。
05

Codebase Index 子系统 · 4 层

让 agent "理解" repo · 不只看 PR diff

PR diff 只有改动的 +/- 行。但要判断"这个改动是否引入 bug" agent 必须知道函数的 caller / callee / 历史改动 / 已有测试。4 层 codebase index 给 agent "穿越" 整个 repo 的能力。

L1语义
函数 / 类级别 embedding 搜
"找跟这个 PR 改动函数语义相似的函数" · 用于发现"同类 bug 在其它地方有没有"
pgvector + bge-code-large
每函数 1024 维 embedding
repo-level / cross-repo
L2符号
精确符号搜：caller / callee / 引用
"这个函数被谁调用 · 调了谁 · 这个变量哪里赋值" · 用于评估改动的 blast radius
ctags / tree-sitter
LSP server 集成
per-language 精确
L3git
Git history 检索
"这文件最近 3 个月谁改的 · 改动趋势 · 是否曾经回滚过 · commit message 提示" · 用于找历史 bug 模式
git blame + log
cached 增量更新
Tantivy 全文索引 commit msg
L4test
Test coverage 数据
"这个函数有没有 test 覆盖 · 哪些 test 文件相关 · 改了之后 test 还过吗 (CI 数据)" · 用于判断 changes 的 test gap
Codecov / coverage.py
CI artifact 拉取
Map test → source line

5.1 4 层 index 的调用模式

不同 subagent 用不同的 layer · 不全员 query 全部。Aggregator 也不直接 query (信赖 subagent 结果)。

Subagent主用 layer典型 query
BugHunterL1 + L4"找类似函数 · 看历史 bug pattern" · "这个改动有 test 吗"
SecurityReviewerL2 + L1"这个 user-input 流到哪些函数 · 有 sanitize 吗"
StyleCriticL1 (sparse)"项目主流 naming convention · 类似函数怎么写"
PerformanceAuditorL2 + L4"这个函数在 hot path 吗 · benchmark 改前后"
DocReviewerL2 + L3"PR 函数被多少个文件 import · 历史是否更新 doc"

5.2 Index 更新策略

  • L1 (embedding)：每次 commit 增量更新（只 re-embed 改动函数）· batch dispatch · 不阻塞 PR review
  • L2 (symbols)：ctags 整 repo 重建 ~5 min · 缓存 5 min TTL · stale OK 因为 PR review 容忍 5 min 延迟
  • L3 (git history)：实时 git log · cached commit msg full-text index 每天 incremental
  • L4 (test coverage)：CI 后 webhook 推送 · pull-based fallback
06

5 Subagent + 1 Aggregator

每个 subagent 独立 prompt / tools / model · 并行执行
🐛 BugHunter
Claude 3.5 Sonnet (升级) · LLM cost ~$0.04/PR
职责：找潜在 logic bug · null deref · off-by-one · race condition · 错误处理缺失 · type confusion。
用的 index layer：L1 (类似函数找 bug pattern) + L4 (test gap 检测)。
关键 prompt 设计：先 list "5 类常见 bug pattern"，让模型 explicitly 看 diff 是否中招 (避免 prompt 抽象)。severity 评分：null deref 在 production code 是 critical · 在 test code 是 nit。
输出 schema：{file, line, severity, category, message, suggested_fix?, confidence}
🔒 SecurityReviewer
Claude 3.5 Sonnet (一律不省 cost) · ~$0.06/PR
职责：OWASP top-10 类风险 · SQL injection · XSS · 密钥硬编码 · auth check 缺失 · CSRF · 不安全反序列化。
用的 index layer：L2 (污点分析 user input 流向) + L1 (找相似已知漏洞)。
合并工具结果：先跑 semgrep --config p/security-audit 拿规则 hit · LLM 再次 review rule hit 是 true positive 还是 false (减少 semgrep 自身的 noise)。
不省 cost 理由：security false negative 比 false positive 后果严重 100x。这是唯一一律 frontier 的 subagent。
🎨 StyleCritic
GPT-4o-mini (一律便宜) · ~$0.005/PR
职责：变量命名 · 代码结构 · 项目 convention 一致性 · 函数过长 · 复杂度高。
用的 index layer：L1 sparse (找项目同类函数命名风格)。
分级：所有 style 问题最多 mid severity，绝不 critical。项目已有 linter 覆盖的不再举报 (跟 ESLint / black / clippy 输出 dedupe)。
反模式：不抱怨"驼峰 vs 下划线"这种格式 — 那是 linter 的事，不是 LLM 的事。
⚡ PerformanceAuditor
Two-tier routing · ~$0.02/PR avg
职责：N+1 query · 不必要循环 · 大对象拷贝 · 同步 IO 在 async context · 漏 cache。
用的 index layer：L2 (查 caller / 看是否 hot path) + L4 (benchmark 改前后)。
启发式：① 函数被 caller 数 > 5 → 看 perf ② 函数有 @benchmark 注解 / 在 benchmark 目录 → 看 perf ③ 否则不深入分析 perf (节省 cost)。
upgrade 触发：如果 mini 报"疑似" + 改动涉及 SQL → 升级 Sonnet 复核。
📚 DocReviewer
GPT-4o-mini · ~$0.003/PR
职责：public function 注释 · param doc · 行为变化时 README 更新 · API 文档同步。
用的 index layer：L2 (函数 visibility) + L3 (是否曾经更新 doc)。
策略：function 是 internal → 不强求 doc · public API → 必须有 docstring · 历史从未有 doc 的项目跳过这个 subagent (用户没 buy in)。
🎯 Aggregator
Claude 3.5 Sonnet · ~$0.05/PR
职责：5 subagent 输出 → 1 个统一 PR review。包括 ① 去重（同 line 多 subagent 报，merge 成 1 个 comment）② 排序（按 severity × confidence × subagent 优先级）③ 裁剪（限 15 个 comment 上限）④ narrative（生成 top-level summary "X 个 critical, Y 个 mid, AI 建议优先看 file:line"）。
关键：aggregator 不做 review（不引入新 finding）· 只综合 subagent 结果 · 这样保持分层 责任。
用 frontier 不省 cost：聚合 reasoning 复杂 · narrative 质量直接影响用户感知。

6.1 Handoff Schema 设计原则

5 个 subagent → Aggregator 的接口必须 stable schema · 否则加 / 改 subagent 时 aggregator 也要改。

interface SubagentFinding {
  subagent: "bug" | "security" | "style" | "perf" | "doc";
  file: string;
  line_start: number;
  line_end: number;
  severity: "critical" | "mid" | "nit";
  category: string;       // e.g. "null-deref" / "sql-injection"
  message: string;        // 1-2 句话解释
  suggested_fix?: string; // 可选 code patch
  confidence: number;    // 0-1
  trace_id: string;       // 关联 LLM call log，eval 用
}

4 个 schema 原则：① 不传 free-form reasoning (太长 + 不结构化) ② 必带 trace_id (eval / audit 必需) ③ severity / category 用 enum (aggregator 才能排序 / 去重) ④ line range 而非单点 (block-level 问题需要)。

8 个 Tools · 按用途分类

read_diff(pr_id, file)
读 PR 改动 hunk · 含 before/after context · subagent 默认输入
read_file(path, lines?)
读完整文件或片段 · 用于看 caller / 依赖 / 周边代码
search_codebase(query, layer?)
L1 语义搜或 L2 符号搜 · 找类似函数 / caller / 引用
run_static_analysis(file, tool)
触发 semgrep / Snyk / ESLint / clippy · 返回 rule hit
query_git_history(file, since)
最近 N 个月谁改 · commit msg · 是否曾经 revert
query_test_coverage(file, lines?)
看 codecov · 改动行有 test 吗 · CI 跑测试结果
query_runtime_data(symbol)
profiler / 调用频率 / hot path 标记 (可选 · 项目接入 APM 才有)
post_comment(pr_id, file, line, body, sev)
仅 Comment Poster 用 · subagent 不直接调（隔离 side-effect）

7.1 Tool 设计 4 个原则

  • ① Side-effect 隔离：post_comment 只在最后阶段调用 · subagent 不能直接 post (避免错误发布)
  • ② 输入有 schema · 输出有 schema：所有 tool 严格 JSON schema · LLM 不能"乱传"
  • ③ 限制 query 频率：每 subagent 每个 PR 最多 5 次 search_codebase (防止 LLM 死循环 query)
  • ④ Idempotent：相同输入应返回相同结果 (用于 retry · debug)

7.2 跟 MCP 范式的关系

这 8 个 tool 本质都是可以包成 MCP server 的。早期实现可以 in-process Python function · production 长期可以拆成独立 MCP server (代码搜 / git history / static analysis 等)。这样工具能复用给其它 agent 系统 (X16 OAM agent / 内部 chatbot 等)。详见 X09 MCP / Skills。

Memory + Skills · 持续学习机制

8.1 3 层 Memory

① Short-term (PR 级)

单次 PR 处理过程的 working memory · 包括 subagent 输出 / tool result · TTL = PR 处理完 · ~30 min

② Project-level (repo 级)

每个 repo 的 metadata · 主语言 / coding convention / allowlist (FP) · history 最常见 reject pattern

③ Cross-project Skills (公司级)

从多 repo FP 反馈中沉淀的 system skills · 如"不要在 test code 报 null deref" · 全公司复用

8.2 Skills 学习机制 (跟 X09 配套)

从 user feedback → skill 的 3 步流程：

  • Step 1 · 收集 FP：每个 thumb-down comment 写 user_feedback table (comment_id / pattern / FP reason "test code 误报")
  • Step 2 · 月度 audit：EM dashboard 看 top-15 FP pattern · 决定哪些可以加进 system skill
  • Step 3 · Skill 发布：Skill 用 YAML 定义 (e.g. "if file in test/ AND category == null-deref AND severity == nit THEN suppress")· 写入 skill registry · subagent 加载时 inject 进 system prompt

注意：skill 不是 LLM 自动改 prompt — 是结构化的 rule，可 review / 可回滚 / 可解释。这避免了 ADR-5 提的 "prompt drift" 风险。

8.3 项目级 allowlist (.aireview/allow.yaml)

# .aireview/allow.yaml · 项目级 FP suppression (commit 到 repo)
version: 1
rules:
  - id: legacy-getter-null
    paths: ["legacy/**/*.js"]
    category: null-deref
    reason: "legacy code 用 truthy 判断不用 strict null check (review by @alice 2026-Q1)"
  - id: test-code-no-style
    paths: ["**/*.test.ts", "**/__tests__/**"]
    subagent: style
    action: skip
    reason: "test 代码可以糙不需要 style polish"

Eval 体系 · 怎么知道 Agent 在变好

9.1 三层 Eval (跟 X12 配套 + AT05 Eval Plan 落地)

层什么频率谁负责
L1 · Offline regression set200 个 hand-curated PR + 标注 (ground truth bug 列表) · 跑 RAGAS 类自动 evalCI 每次 prompt 改ML eng
L2 · Shadow eval (staging)shadow new agent 跑 1% production PR · 不发 comment · 人工 sample audit每次大版本前ML + Reviewer
L3 · Online user feedback实时 thumb-up/down · accept / dismiss rate · 是否被 mute持续 (dashboard)EM + PM

9.2 关键 metric 公式

  • Precision = (thumb-up + 真的修了) / 全部 comment · P0 metric · 目标 > 80%
  • Recall (offline) = agent 标出的 bug / ground truth bug 总数 · 目标 > 60%
  • Critical recall (security): > 90% · 不能漏关键安全问题
  • FP rate per severity：critical FP < 5% · mid FP < 15% · nit FP < 30% (nit FP 高 OK 因为容易 mute)
  • Comment-to-Action ratio = comments accepted+fixed / total comments · > 35% 是健康
  • Mute rate：< 5% 的 repo 显式禁用 agent (overall trust 指标)
  • Cost per PR：mean / p99 · 目标 mean < $0.10

9.3 Aggregator 排序公式

5 个 subagent 给出 N 个 finding · 排序公式：

priority_score = (
  SEVERITY_WEIGHT[finding.severity]   # critical=10, mid=5, nit=1
  * finding.confidence                  # 0-1
  * SUBAGENT_WEIGHT[finding.subagent]  # security=1.5, bug=1.3, perf=1.0, style=0.8, doc=0.6
  * HISTORY_FACTOR(finding.category)   # 历史 high accept rate → boost
)

# 然后 sort desc · 取 top-15

HISTORY_FACTOR 是 dynamic — 上月 "null-deref" comment accept rate 高 → 这月 null-deref 提升优先级。这是 online feedback 反馈到排序的关键。

9.4 Eval dashboard 关键看什么

  • 本月 Precision / Recall（vs 上月趋势）
  • Top-15 FP pattern（哪些 category 误报最多）
  • Top-15 missed bug pattern（被 senior reviewer 加但 agent 没标的）
  • 每个 subagent 单独 health（StyleCritic 精度可能高，PerfAuditor 可能低）
  • Cost trajectory（cost / PR 的月度趋势 · routing 优化 ROI）
  • Repo-level adoption（哪些 repo 没启用 · 哪些 mute · why）

2 个 hypothetical 案例

案例 1 · 一个 SQL injection PR 被拦截

背景：Junior 工程师改了 user search API · 拼接 SQL string 用 user input。

[T+0s] PR opened webhook → orchestrator [T+2s] Codebase Index 拉取相关 caller / 历史 SQL 处理代码 [T+3s] 5 个 subagent 并行: - SecurityReviewer (Claude Sonnet): → run_static_analysis (semgrep p/security-audit) → hit 1: user_input → SQL string concat → LLM verify: confirmed (not a FP, this is direct injection) → emit finding: severity=critical, category=sql-injection, confidence=0.95 - BugHunter: ↻ 无 finding (没看出 null deref 等) - StyleCritic (mini): emit nit: "consider extract query builder" - PerformanceAuditor: 无 (函数被 caller < 5) - DocReviewer: 无 [T+12s] Aggregator (Sonnet): - dedup, sort by priority_score - top finding: SQL injection (critical) - narrative: "⚠️ 1 critical SQL injection in user_search.py:42" [T+13s] Comment Poster: → post per-line comment with suggested fix (use parameterized query):
SecurityReviewer · src/api/user_search.py:42 · CRITICAL
src/api/user_search.py:42 - query = f"SELECT * FROM users WHERE name = '{user_input}'" + query = "SELECT * FROM users WHERE name = $1" + rows = await db.fetch(query, user_input) ⚠ SQL Injection 风险: 用户输入直接 f-string 拼接到 SQL · 攻击者可注入 `' OR '1'='1` 建议: 用 parameterized query (上面 patch) · postgres asyncpg 用 $1/$2 风格 placeholder ─────────────────────────────────── trace_id: trc_abc123 · cost: $0.04 · 👍 👎 🤔

结果：Junior 工程师 5 min 内看到 + 改 + 重 push · merge 时已 fix · 避免一次生产 SQL injection (典型 cost: 10+ hr remediation + audit + 可能数据泄漏)。

案例 2 · False Positive 处理 · skill 学习闭环

背景：某项目大量 legacy callback 风格代码 (Node.js) · agent 频繁报 "missing await"。

[Week 1] 5 个 PR 中 agent 报 12 个 missing-await · Senior 看后: → 这些其实是故意的 fire-and-forget 调用 · 不是 bug → 全部 👎 thumb-down + 评论 "legacy callback style, not awaiting on purpose" [Week 1 end] Dashboard 显示: - top FP pattern: missing-await in legacy/* · 12 cases this week - 自动 suggest 加 allowlist? [Week 2] Senior 创建 .aireview/allow.yaml: - id: legacy-callback-no-await paths: ["legacy/**/*.js"] category: missing-await reason: "legacy callback style" → commit 进 repo · 下次 PR 不再报 [Month 1 end] EM 月度 audit dashboard: - 5 个 repo 都有相同 missing-await-in-legacy FP pattern → 决策: promote 到 system skill (全公司) → 加 system skill: "skip missing-await if file path contains legacy/ AND project flag legacy_callback=true" [Month 2] 跨 repo FP 数从 47 / month → 3 / month · -94%

关键：没有自动改 prompt (避免 drift) · 是结构化的 allowlist + skill (可 review / 可 git diff / 可 rollback)。这是 ADR-5 决策的落地。

7 个工程教训

教训 1
Precision > Recall · 工程师对 FP 0 容忍
上线第 2 周 agent precision 跌到 65% (FP 35%)，团队开始 mute · 第 3 周 5 个 repo 关 agent。不是 recall 低杀死它的，是 FP 多杀死它的。
修复方向：① 提高 confidence 门槛（confidence < 0.7 不发 critical 而发 mid）② 加 allowlist 机制 ③ severity 重新分级（很多原来标 critical 的降到 mid）。
教训 2
Subagent 隔离 vs God Prompt · 维护性是核心
早期试过 "god prompt" 一次性 review 所有 aspect · prompt 长 5000 tokens · LLM 注意力分散 · 漏报严重。更要命的是改一个 aspect 要担心整个 prompt 回归。
5 个 subagent 后：每个独立 prompt 一两百行 · 单独 eval / 单独 model routing / 单独 disable · 可维护性 +10x。Production agent 几乎都是这条路（Cursor / Devin / Claude Code 都是 multi-agent）。
教训 3
Tool 设计 · Side-effect 隔离是底线
早期 SecurityReviewer 直接调 post_comment · 一次 LLM 幻觉发了 8 个错误 comment · 用户怒删。
重构后：所有 subagent 只输出 schema-typed finding · 只有 Comment Poster 一个角色能 post · 加 "comment count quota" 二次 check。Side effect 必须 funnel 到一个 chokepoint。
教训 4
Cost routing 早做 · 否则月底翻车
第一个月用纯 Claude Sonnet · 月底 cost $4200 (远超预算 $800) · 团队差点 OK 项目。
紧急加 two-tier routing (ADR-3) · 第二月降到 $580 · quality 跌 ~3% (acceptable)。教训：第一周就要建 cost dashboard · 不要等月底。详见 AT06 Cost Model。
教训 5
Eval set 必须涵盖"应该 silence 的 case"
初版 eval set 全是 "真 bug PR" · 测出来 recall 92% 觉得很好 · 上线 FP 35% 灾难。
后来加 100 个 "clean PR" 到 eval set (没 bug 的 PR · agent 应该 silent) · 测 silence rate (clean PR 没多余 comment 比例) · 这才暴露 FP 问题。Eval set 必须 balance · 不能只测 positive case。
教训 6
用户反馈循环不是 "nice-to-have" 是核心组件
初版没 thumb-up/down · 没法量化 quality · 没法学习 FP · 3 个月后团队失去 trust。
后来加 in-product feedback (3 个 button) + .aireview/allow.yaml + skill registry · FP rate 月度持续下降 · trust 回升。反馈机制 = agent 长期生命力。
教训 7
Skills 不是"自动改 prompt" · 是结构化 rule
短暂试过让 LLM 看 FP 反馈 → 自动改 system prompt。结果：一次 wrong feedback 让 agent 对整个公司"忽略 null deref" · 漏了真 bug 2 天后才发现。
回退到结构化 skill (YAML) · 必须 EM review + commit 进 registry · 慢但安全。"AI 自学习"听起来酷 · production 必须 human-in-the-loop。详见 X09 Skills 章。
