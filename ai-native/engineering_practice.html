① AI-Native 是什么 · 不是什么

是什么

  • 把 LLM 当成额外的 team member—— 不是工具，而是协作者。你要给它 context、review 它的输出、跟它讨论。
  • 从"打字 → 写代码"转成"表达意图 → 收 diff → 决策"。每个工程师的 throughput 不再被打字速度限制。
  • 知识资产显式化—— 团队内隐性知识（"这个 module 历史是这样"）必须写下来，否则 LLM 帮不上忙。
  • 测试 / 文档 / 重构 ROI 翻倍—— 之前因人力成本高被砍的工作，现在 LLM 可以快速完成。

不是什么

  • 不是"IDE 装个 Copilot 就完事"—— 那只是 ① 的入门版，停在这层会被淘汰。
  • 不是"AI 替代工程师"—— 是 "AI 增强工程师"，初级工程师反而比 senior 更受冲击（senior 的判断 / 架构 / debug 仍 irreplaceable）。
  • 不是"vibe coding 万岁"—— 不读 diff / 不测试 / 不理解逻辑就 ship 的"AI slop" 是新形态技术债，长期吃 team velocity。
  • 不是 silver bullet—— 仍需 SE 基础（类型系统 / 数据结构 / 调试 / 系统设计 / Cmunication）。LLM 放大你的 baseline，baseline 太低 LLM 也救不了。
🎯 一句话定义
AI-Native = 团队工作流 + 个体 throughput + 知识管理 都基于 "LLM 是常驻 collaborator" 这个假设来设计；不是局部 plugin，是系统重构。

③ AI Pair Programming 4 模式

不是所有任务都该一种用法。4 模式光谱—— 由 AI 自主性从低到高：

① Pilot

你写主体，AI 补全句子级 / 行级。

AI 自主度 ★☆☆☆☆

② Co-pilot

你给 prompt，AI 写一段；你 review + 改。

AI 自主度 ★★☆☆☆

③ Wingman

你给目标，AI 跨文件改 + 自跑测试；你 review diff。

AI 自主度 ★★★★☆

④ Autonomous

你给一个 ticket，AI 端到端完成（含 PR + 自测）。

AI 自主度 ★★★★★

各模式典型场景

模式典型场景风险关键技巧
Pilotboilerplate / 已知 pattern / 单行重复低 —— 你看得见每个 tokentab 接受时仍要扫一眼语义
Co-pilot新功能 / refactor 一个 function / 单文件改中 —— 容易"看着对实际错"必须读完整 diff 再 commit；让 AI 解释关键选择
Wingman跨多文件 refactor / 加测试 / migrate API高 —— diff 可能 100+ 行① 限定 scope 明确；② 每步 review；③ 测试 + lint 必须过
Autonomouschore（bump deps / format / 删 dead code） / well-defined ticket极高 —— 错误不易追溯① 用 PR + CI 守门；② "回滚成本低"才用；③ 不要 blind merge
选错模式的代价：在 Wingman / Autonomous 模式下跑一个 architectural 决策（"换 DB engine"）—— 你以为 "AI 帮我 migrate 完事"，实际暴露 50 个 edge case + 你不懂为什么这么做。archi 决策永远 Pilot / Co-pilot 模式，AI 给草稿你拍板。

④ 10 大反模式

  1. Vibe Coding（不读 diff 就 commit）
    • "AI 写完测试也跑过我就 ship" → 测试可能本身就被 AI 改坏 / cover 不全；逻辑可能 "看似 work 其实暗藏 bug"
    • 对策：① 永远 review 完整 diff；② 关键路径手写测试再让 AI 实现；③ 给 AI 输出做 senior-eyes pass
  2. Context 污染（chat history 越长越糟）
    • 同一 chat session 跨多个 task → 上下文混乱，AI 引用错误 file / pattern
    • 对策：每个新 task 起新 chat / context；用 @file / @symbol 精确给 context
  3. 测试盲信（"AI 加的测试都过了"）
    • AI 写测试时倾向于让测试 pass —— assertion 可能写错方向，可能 mock 掉关键逻辑
    • 对策：① 测试反 review（看 assertion 是否真测核心）；② mutation testing 验证测试敏感性
  4. Over-trust（"AI 比我懂"）
    • 对自己不熟领域的 AI 输出全盘接受 → 学不到 + 错误难发现
    • 对策：陌生领域用 AI 时故意问"为什么这么写"+ 读官方 doc 验证
  5. Prompt 蹩脚（"give me code"）
    • 给一行 prompt → AI 猜你意图 → 错的概率 80%
    • 对策：明确 ① context / ② constraint / ③ 输出格式 / ④ 测试标准
  6. Hallucination 不验证
    • AI 编 API / library / 函数名 / 版本号 / 性能数字
    • 对策：① 关键 fact 必查文档；② 写代码前先 sanity check API 真存在
  7. Knowledge regression（"团队靠 AI 不再讨论"）
    • 每人独自跟 AI 聊，团队共识 / debate 消失 → 系统理解碎片化
    • 对策：保留 design review / 重要决策仍要人类同步讨论 + 写 ADR
  8. Junior 跳过 fundamentals
    • 新人靠 AI 完成 task 但不学底层（指针 / 数据结构 / 网络协议）→ 5 年后没法 debug 复杂问题
    • 对策：① junior 强制定期 no-AI sprint；② mentor review 加问 "为什么这么写"
  9. 性能 / 安全盲点
    • AI 生成代码默认不考虑 hot path / SQL injection / XSS / 内存 / 并发 race
    • 对策：① 性能 / 安全敏感代码必须人 review；② AI 输出加 sanitizer / lint pass
  10. "silver bullet" 心态
    • 遇到任何问题先问 AI → 失去自己思考 / debug 的 muscle
    • 对策：复杂 bug 给自己 30 min 先独自调；解决不了再带已经做过的 hypothesis问 AI

⑤ 团队 workflow

5.1 .cursorrules / AGENTS.md

Cursor 等 AI IDE 支持仓库根目录放 rules / instructions 文件，让 AI 自动遵守团队约定。

# 示例 .cursorrules（精简版）

You are a senior C++ engineer working on a network management agent project.

Code style:
- Use C++20 features when possible (concepts, ranges, designated init).
- Prefer std::expected over exceptions for predictable errors.
- All new files must have copyright header (see existing files).
- Tests use GoogleTest. Place under test/ mirror structure.

Conventions:
- Lifetimes: prefer unique_ptr over shared_ptr; raw pointer only for non-owning.
- Threading: use jthread + stop_token (C++20); avoid raw std::thread.
- Logging: spdlog at file level; no printf / iostream.
- Memory order: justify any non-default memory_order in a comment.

When uncertain:
- Ask before changing > 3 files at once.
- Ask before adding new dependencies.
- Ask before touching the public API (include/agent/).

更复杂场景用 AGENTS.md（标准化 agent context 格式，跨工具兼容）—— 描述项目结构、build 命令、测试方法、常见 task 模板。

5.2 PR 规范变化

非 AI-NativeAI-Native
PR 大小"尽量小，< 400 行""仍要小，AI 让人想偷懒一次塞 2000 行 → reviewer 反弹"
Commit message"what + why""+ 哪些是 AI 生成 + 你怎么验证的"
Description"功能描述 + test plan""+ 列出 AI 帮做了什么 / 你 review 了什么"
测试"加测试覆盖新代码""+ 测试也要 review（AI 测试常 weak）"

5.3 Shared context（团队共享 AI 输入）

  • 把 onboarding 文档 / 架构图 / ADR / postmortem 放仓库 + 让 AI 都能 index
  • 关键决策写成 ADR（每个 ~ 1 页），方便新 LLM session 也能 onboarding
  • 避免 "口口相传的 tribal knowledge"—— AI 用不上 + 新人 ramp 慢

⑥ AI 做代码 Review

不是替代人 review，是分层—— AI 先扫一遍 + 人深度 review 复杂部分。

AI 适合的

  • 风格 / 格式 / lint（其实更适合 linter，AI 是 fallback）
  • 简单 bug 模式（null deref / use after move / 循环边界）
  • 明显的安全 issue（hardcoded secret / SQL injection / path traversal）
  • 测试覆盖建议（哪些 case 没测）
  • 文档完整性（new public API 有没 doc）
  • PR description 改进建议（漏了 test plan / 没说 why）

AI 不适合的

  • 架构 / 设计决策（"这个改动符合我们模块边界吗"—— 需要 tribal knowledge）
  • 跨模块 ripple effect（"这改动会影响 X 团队的 service 吗"）
  • 权衡判断（performance vs maintainability vs deadline）
  • 政治 / soft skill（"这位 author 最近 burnout，我应该 push 还是 ease？"）
  • 历史决策的尊重（"3 年前我们决定不这么做因为 X，AI 不知道 X"）

实战 setup

  • GitHub Action / GitLab CI 跑 AI review（如 llm-code-review / aider-review / 自建脚本调 Claude）
  • 输出作为 PR comment，人 reviewer 再 deep dive
  • 关键：人 reviewer 仍是 accountable，AI 只是 catch lowhanging fruit

⑦ 知识管理 + RAG

为什么需要

LLM 的训练数据 cutoff + 不知你的 codebase = 默认能力有限。靠 ① 喂 context；② RAG（Retrieval Augmented Generation）能极大提升相关性。

常见架构

方案原理适合不适合
IDE 自带索引（Cursor / Copilot）本地嵌入 + 增量更新个人开发，无配置团队级
Sourcegraph CodySaaS 仓库索引大型团队 / 多仓库 / 跨仓库搜索privacy 严格
自建 RAGpgvector / Pinecone / Qdrant + embeddings有 ML 团队 / 特殊需求资源有限团队
纯文档 + @reference把关键 doc 放 repo / wiki，prompt 时 @小团队 / 起步阶段大型 codebase

Context pollution 是个真问题

  • 把整个 codebase 都喂给 LLM → context window 满了反而效果差（needle in haystack）
  • 原则：相关 context > 完整 context
  • 实战：让 RAG 系统返回 top 5 chunks，不是 top 100

知识结构化

  • 团队 wiki / README / ADR 越结构化（小标题 + 关键词）→ embedding 越准 → AI 抓的越准
  • 避免 "一篇 50 页大文档"，拆成 10 篇 5 页主题文档更好 retrieve
  • 关键 fact 用 显眼 tag 标（如 [ARCHITECTURE-DECISION] / [CRITICAL-PATH]）方便 retrieve weighting

⑧ AI-Native onboarding

新人 ramp up 是 EM 长期痛点。传统 3 周才能 productive，AI-Native team 能压到 3 天。

Day 0 准备

  • 仓库根目录有清晰的 AGENTS.md / CONTRIBUTING.md / README.md，让 AI 一开始就给新人正确 context
  • 关键路径 ADR（5-10 个最重要决策）整理好
  • onboarding checklist：clone / build / run / 第一个 trivial PR 流程

Day 1-3 workflow

  1. 新人: "我要做 X 功能。这是 ticket。"
  2. AI（Cursor 等）: 根据 codebase + ADR + AGENTS.md 给出 implementation plan
  3. 新人: review plan + 提问 + 让 AI 实现
  4. 新人: review diff，跑测试，提 PR
  5. Mentor: review 时关注新人对 "为什么" 的理解，不只 diff 本身
风险：AI-onboarding 让新人 first-week productivity 高，但fundamental 不扎实 → 3-6 月后复杂 task 卡壳。对策：
  • 每周至少 1 个 "no-AI" 任务，强迫思考
  • 每月 1 次 mentor 1on1 深挖 "X 是怎么 work 的"
  • 给新人推荐基础书 / 课程（不只 codebase）

⑨ AI-Native debugging / incident response

哪些 debug 任务 AI 加速大

  • Stack trace 分析：贴 stack + 关键 source → AI 高概率指对方向
  • log pattern 找异常：贴 1000 行 log → AI 帮你找 outlier
  • 已知 framework 的常见 bug：React / Node / Python 等社区 well-covered
  • 正则 / SQL / 复杂查询编写：经典强项

哪些不行

  • 偶发 race / heisenbug：AI 没运行环境，无法复现
  • 系统级 / kernel level：data 太少，hallucination 多
  • 跨 service / 跨 team：context 太分散，AI 无法 stitching
  • "我也不知道哪坏了" 的 P0 incident：需要 explore → human intuition 仍主导

Incident workflow

  1. 事故发生 → 人 IC（Incident Commander）启动
  2. 分头 collect data：metrics / logs / dashboards
  3. 关键 data 一部分喂 AI 求 hypothesis（AI 适合 pattern matching）
  4. 人 commander 综合判断 + 决策（rollback / patch / monitor）
  5. Postmortem 写时 AI 帮 draft，人 finalize（特别 cultural / blameless tone）

⑩ 2027+ 展望 + 学习路径

2027 预测（个人观察 ≠ 共识）

  • "Vibe coding"被反弹：2026 末已经有 codebase 因 AI slop 难维护，2027 大公司开始有"AI-generated code 质量标准"
  • Autonomous agent 普及：well-defined task（bump deps / format / 简单 bugfix）AI 自动 PR，开始进入主流 CI
  • Coding interview 形态变化：白板手写在 senior+ 招聘进一步消失；改成 "AI-assisted 但你解释 + debug + 优化" 形式
  • Junior 就业市场紧：传统 junior task 被 AI 替代，新人需要"更高 baseline"才能拿 entry-level offer
  • EM 时间分配变：less code review more product / architecture / coaching / hiring

学习路径（针对个人 IC / EM）

阶段时间内容
入门2-4 周① Cursor / Copilot 装上每天用；② 学 prompt engineering 基础（OpenAI cookbook）；③ 跟 1 个工作中真任务实操，体会 4 模式（pilot/copilot/wingman/autonomous）
进阶2-4 月① 团队级 workflow（.cursorrules / AGENTS.md / PR 规范）；② 写自己的 "AI 套件"（常用 prompt template）；③ 学 RAG 基础 + 给团队搭一个
高级持续① 关注 SOTA 模型迭代（每月 1h 读 release notes）；② 跟踪开源 agent 框架（LangChain / DSPy / Aider）；③ 自己写一个 specific-domain agent
