Embedding 本质 · 把万物变向量

🎬生活类比 · 把每首歌画一个"坐标"
想象世界上每首歌都被映射到 768 维空间里的一个"坐标"。风格相似的歌坐标接近（如周杰伦的所有歌聚成一片）。

这样找"跟这首歌相似的"就是找空间里距离最近的 K 个坐标。

Embedding 就是这个映射函数：训练好的神经网络把任意输入（文本/图/音频）映射到 N 维向量，让"语义相近"对应"距离近"。

1.1 为什么要 embedding

问题传统方法Embedding 方法
搜"跟这段相似的"关键字匹配（漏掉同义词）向量近邻（捕获语义）
跨语言查询需要翻译同一向量空间天然跨语言
"模糊查询"正则 / Levenshtein向量距离自然容忍模糊
个性化规则 / 协同过滤用户兴趣 = 向量

1.2 距离度量

  • Cosine（最常用）：关心方向不关心长度。Embedding 主流。
  • Dot Product：考虑长度。某些模型需要（如 ColBERT）。
  • Euclidean (L2)：欧氏距离。CV 偶尔用。

ANN 算法 · 怎么在亿级向量里快速找近邻

HNSW / IVF / PQ 三大主流
🎬生活类比 · 在 1 亿本书里找相似的
1 亿本书，每本被映射到 768 维向量。给你一个 query，找最相似的 10 本。

暴力法：跟每本算距离，1 亿次。要几分钟。
ANN（Approximate Nearest Neighbor）：用索引结构提前组织数据，查询时只看少数候选 — 速度 100-1000x，但牺牲一点点准确（99% recall）。

3.1 HNSW · 分层图导航

HNSW 思路：建一个分层小世界图。上层稀疏（远距跳跃），底层密集（精确）。查询时从顶层开始贪心导航到目标。类似导航软件先看高速路再看小路。
  • 优势：查询极快（亿级 1-10ms），高 recall（99%+）
  • 劣势：内存占用大（每向量 ~1KB 图结构），更新慢
  • 主流库：FAISS HNSW / Qdrant 默认 / Milvus 支持

3.2 IVF · 倒排索引

  • 原理：把所有向量 K-means 聚成 N 个 cluster（如 1024 个）。查询时先找最近 cluster，只在 cluster 内搜。
  • 优势：内存小，更新快
  • 劣势：边界 case 漏召（向量恰好在 cluster 边界附近）
  • 调参：nlist（cluster 数）和 nprobe（查多少个 cluster）trade-off speed vs recall

3.3 PQ · Product Quantization 量化压缩

把 768 维向量切成 M 段（每段 96 维），每段用 256 个 codebook 中的 ID 表示。1 个向量从 768×4=3072 字节压到 8 字节，压缩 384x。常跟 IVF 组合用：IVF-PQ。

3.4 选型决策

规模推荐
< 1M 向量HNSW（内存能装下，速度快 recall 高）
1M-100MHNSW 或 IVF-PQ（看预算）
100M-1BIVF-PQ 必须（HNSW 内存炸）
1B+分布式 IVF-PQ（多机分片）

05

Hybrid Search 工程实战

详见 X08 Ch5。这里补充工程实践。

5.1 部署架构

  1. Dense：Qdrant cluster（3 副本）
  2. Sparse / BM25：OpenSearch / Elasticsearch（同样 3 副本）
  3. Coordinator：API server 并发查 2 路 + RRF 融合
  4. Rerank：Cross-Encoder service（独立部署，GPU）
  5. Cache：Redis 缓存热门 query embedding + 结果

5.2 性能优化 tip

  • Embedding 服务批处理：collect 100 query 一起算（throughput +20x）
  • 预计算 BM25 倒排：写入时建索引避免实时计算
  • Cache 命中策略：热门 query 缓存结果 24h；用户独享 query 缓存 1h
  • Rerank 限 top-K：rerank 50 个就够，再多 latency 涨

5.3 增量更新策略

批量重建（夜间）

每天凌晨重新算所有 embedding 重建索引。简单但延迟高（D+1 才能搜到新内容）。

流式增量

新文档实时 embedding + insert。HNSW 支持但慢，IVF 需要 periodic rebuild。

10 道高频面试题

基础Q1. Embedding 是什么？为什么 RAG 需要？
把文本/图/音频映射到 N 维向量，语义相近对应距离近。RAG 用它做"语义搜索"，超越关键字匹配。
基础Q2. Cosine vs Dot Product vs L2 怎么选？
Cosine 主流（embedding 长度不重要，只看方向）。Dot Product 某些 ColBERT 类模型用。L2 偶尔 CV 用。
进阶Q3. HNSW 算法原理？
分层图：上层稀疏（远距跳跃）下层密集（精确）。查询时从顶层贪心导航。类似导航软件先看高速再看小路。优势：查询极快+高 recall。劣势：内存大、更新慢。
进阶Q4. IVF-PQ 是什么？跟 HNSW 怎么选？
IVF
K-means 聚 cluster，查询时先找最近 cluster 只在内搜。
PQ
向量分段量化，768×4 字节 → 8 字节，压缩 384x。常跟 IVF 组合。
选型
< 100M 用 HNSW（recall 高）；> 100M 用 IVF-PQ（内存装得下）。
进阶Q5. Embedding 模型怎么选？BGE vs OpenAI vs Qwen3？
看场景：中文 → Qwen3 / BGE-zh；多语言 → BGE-M3；英文 → NV-Embed-v2；商业 → OpenAI/Cohere。实测在自己 eval set 上才数，MTEB 排名跟业务未必匹配。
基础Q6. Matryoshka Embedding 是什么？解决什么问题？
训练时同时学多个维度的好 embedding（512/1024/4096）。用户根据 cost/quality 截取前 N 维。不用为每个 dim 训一个模型。OpenAI text-embedding-3 / Qwen3-Embedding 支持。
进阶Q7. 1 亿向量做向量搜索 latency 目标 < 50ms，怎么做？
① IVF-PQ 索引（HNSW 1 亿内存爆）② 分片到 5-10 个 node 并行 ③ nprobe 调到 64（recall ≥ 95%）④ embedding 服务批量化预 cache 热 query。
基础Q8. Qdrant / Milvus / pgvector 怎么选？
小规模（< 10M）+ 已有 Postgres → pgvector。中等规模 (10M-100M) → Qdrant（开源/Rust/性能好）。大规模/云原生 → Milvus（国内常用）。
进阶Q9. 向量库的 Metadata Filter 为什么重要？
业务总要 "用户 X 的私有文档" / "v2.5 版本" / "上月新增" 这种过滤。filter + vector search 同时做才能生产可用。Qdrant / Milvus 都内置高效 filter 索引。
高频Q10. 为什么纯 Dense 不够要 Hybrid (BM25+Dense)？
Dense 擅长语义但差在精确匹配（版本号 / ID / 专有名词 embedding 太近分不清）。BM25 关键字精确但漏近义词。Hybrid RRF 融合：Recall@10 从 55%/70% → 85%。生产 RAG 标配。
