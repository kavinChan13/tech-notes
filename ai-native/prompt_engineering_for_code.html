① 为什么 prompt 仍然重要（frontier 模型时代）

有人说 "GPT-5 / Claude 5 之后 prompt 不重要了，模型够聪明"。错。

3 个仍然重要的原因

  1. Context window 仍是宝贵资源
    Frontier model context 1M-10M token，但有效 attention 衰减（needle in haystack 现象）。塞太多无关 context → 模型 ignore 关键信息。好 prompt = 精确 select 相关 context。
  2. 意图歧义不可避免
    "给我加个缓存" 模型必须猜：哪层缓存（DB / API / browser）？什么 eviction（LRU / TTL）？什么 key？容量？分布式 or local？好 prompt 把这些 explicit。
  3. Verification cost 不变
    模型越聪明 → 错误越微妙。Hallucinated API 看起来像真的（合理签名 + plausible docstring），但跑不通。Prompt 引导模型给 可验证的输出（reference doc / 跑测试）能避免大半 hallucination。
Mental model 转变：把 prompt 想成"给一个聪明但不了解你 codebase的 contractor 的工单"。你给他越多 context + 越明确 acceptance criteria，他做得越准。

② 好 prompt 的解剖学

任何高质量代码 prompt 都包含 4 个组件，可以缺，但缺少时显式注明 placeholder。

组件问题例
Context你的世界长什么样？"我在写一个 C++17 的网络服务，用 boost.asio。已有 Session class（如下）..."
Goal你想达成什么？"加 reconnect 逻辑，断开后指数退避重连"
Constraint什么不能做？"不能改 Session 的 public API；不能引入新依赖；必须 noexcept move"
Output format怎么返回？"先讲思路 5 句话，再给完整 patch（git diff format），最后列测试 case"

对比

❌ Vague prompt
给我写个 reconnect 逻辑
模型猜 5 个不同假设，可能跟你的 codebase 完全不匹配。
✅ Anatomy 完整
Context: network service agent (C++17, boost.asio).
Existing Session class:
@Session.hpp

Goal: 加 auto-reconnect，断开后指数退避（1s, 2s, 4s, max 60s）

Constraints:
- 不改 Session 公开 API
- 不引入新依赖（boost / std 即可）
- Move ctor / dtor noexcept
- 必须 thread-safe（多 worker 调用）

Output:
1. 5 句话说思路
2. 完整 patch (git diff format)
3. 列 3 个 unit test case

③ 12 个实战模式

Pattern 1 · Spec-Then-Code（先 spec 再实现）

触发：复杂逻辑 / 你自己也没想清楚边界 case

分两步：第 1 turn 让 LLM 列出所有 edge case + behavior spec；第 2 turn 才让它实现。

Turn 1:
"我要给 Session 加 reconnect。
**先不写代码**，列出：
1. 触发 reconnect 的条件（如哪些 error code）
2. 退避策略边界（首次 / 最大间隔 / 重置条件）
3. 5 个 edge case（如 reconnect 中调 send / 多线程并发 reconnect / shutdown 时 reconnect）
4. 每个 edge case 期望行为"

Turn 2 (after reviewing spec):
"按上面 spec 实现，重点处理 edge case 3 + 5"

分两步比一步 "给我加 reconnect" 输出质量高 3-5×—— 模型有结构化 thinking 时间。

Pattern 2 · Test-First（先要测试）

触发：纯函数 / 算法实现 / 重构需要回归保护

"基于现有签名:
  std::vector<Order> matchOrders(const Book& book, Side side);

**先生成 GoogleTest 测试 cases**，覆盖：
- 空 book / 空 side
- 单边市
- 部分 fill
- 价格相同时 FIFO
- 异常输入（负数 quantity / NaN price）

测试 review OK 后再实现"

好处：① 测试 review 比 code review 快；② 实现错时测试立刻 catch；③ 防止 AI 自己写测试时调整测试 fit 错代码的陷阱。

Pattern 3 · Explain-Then-Edit（先解释再改）

触发：legacy code refactoring / 不熟悉的代码区

"先 walk me through @existing_module.cpp：
1. 这个 class 干什么的
2. 关键 invariant 是什么
3. 历史决策（看 comment / git blame 推断）

review OK 后，再 propose 这样改：
[你的改动想法]"

避免 AI 不理解上下文就大改—— 它会先 verify 自己理解，你 review 它的理解是否对，再让它改。

Pattern 4 · Constraint-First（约束优先）

触发：codebase 有强约束（lint / style / lib 选型）

"实现 X. 约束（必读，每条违反扣分）:
- C++17 only（no C++20 features）
- 不用 std::regex（性能差）
- 不用 std::shared_ptr（用 unique_ptr + raw ptr）
- 错误用 std::expected (从 tl::expected lib)
- 测试用 doctest 不用 GoogleTest
- noexcept 优先

如果约束跟需求冲突，先 raise，不要 silently 违反"

让 LLM 知道不能默认按 mainstream 风格写。

Pattern 5 · Chain-of-Edits（链式小改）

触发：大改动（> 200 行）

拒绝单 prompt 出 1000 行 patch。分 5-10 steps，每步 50-100 行，每步 review + 跑测试 + 再 next。

Step 1: "加 ReconnectPolicy class（仅声明 + 测试），暂不集成"
[review + test]
Step 2: "在 Session 内嵌 ReconnectPolicy 成员，但不调用"
[review + test]
Step 3: "Session::onDisconnect 时调用 policy.shouldRetry()"
[review + test]
...

关键: 每 step 独立可 ship；不到 step N 才发现 step 1 的错。

Pattern 6 · Few-Shot Example（给样板）

触发：内部 idiom / 你 team 的 style 跟主流不同

"看 @example_handler.cpp 的 RPC handler 风格。

按同样的风格实现 newRpcMethod：
- 错误处理用 tl::expected
- log 用 spdlog::info / error  with same format
- 参数 validation 用 [[nodiscard]] checker
- 函数结构: validate → execute → log → return"

LLM 模仿 example 比按抽象 description 准 5×。

Pattern 7 · Critique-Then-Refine（自我批评）

触发：模型刚给你一段代码，你不确定够好

(after model gave initial code)
"现在你是个 senior C++ reviewer，列出这段代码的 5 个问题
（性能 / 安全 / 可维护性 / 错误处理 / 测试性）。

然后基于这 5 个问题给修正版"

模型当 reviewer 时找到的问题比当 coder 时多。双 pass 比单 pass 输出质量高。

Pattern 8 · Boundary Exploration（边界探索）

触发：你已有方案但担心边界 case

"我的方案如下：[代码 / 思路]
扮演 adversarial tester，给我 10 个能 break 这段代码的 input。
重点考虑：
- 极端值 (INT_MAX / negative / NaN / 空 string)
- 并发情况
- 资源耗尽（memory / fd）
- Malicious input (oversized / 异常 encoding)"

AI 找 edge case 比人快 + 不会 ego defensive。

Pattern 9 · Cite-Source（要引用）

触发：库 / API 用法不确定 / 防 hallucination

"用 boost::asio 实现 X。
**每个 API 调用旁边 comment 它在 boost 哪个版本起可用 + 引用 official doc 章节**。
如果某 API 你不确定存在，明确说 '不确定 X，请 verify'。"

让 LLM 显式标 confidence。它编 API 时通常仍能跟 cite，但不准的 cite 一查就出，比代码 review 找 bug 容易 10×。

Pattern 10 · Persona Layer（角色叠加）

触发：希望模型 channel 特定风格

"角色叠加：
- 资深 C++ 系统工程师（10 年）
- 强偏好 RAII / unique_ptr / noexcept
- 不喜欢异常作业务流程，错误用 expected
- Code review 严格：每个函数 ≤ 30 行，每个 class ≤ 200 行
- 写代码先写测试

按这个 persona 实现 X"

比 "写得 senior 一点" 更精确控制风格。

Pattern 11 · Diff-Native（diff 风格输入输出）

触发：基于已有代码改

给 AI git diff format 让它清楚哪段已改、哪段不变。

"基于以下 diff 继续：

```diff
@@ -10,3 +10,7 @@
 class Session {
 public:
+  void enableAutoReconnect();
+  void setReconnectPolicy(ReconnectPolicy p);
   void send(Message m);
```

实现 enableAutoReconnect + setReconnectPolicy 在 cpp 文件中"
Pattern 12 · Bounded Search（限定搜索空间）

触发：解决方案多种，要 AI 给最适合的

"解决并发 cache 问题，候选方案：
A. std::mutex + std::unordered_map
B. std::shared_mutex 读写锁
C. concurrent_hash_map (TBB / folly)
D. RCU / hazard pointer
E. 不用 cache，每次重算

我的场景：
- 99% 读 1% 写
- 读 hot path P99 < 100ns
- 写不要求 strict consistency (eventual OK)
- 单机不分布式

按这个场景，A-E 各自评分（performance / 复杂度 / 维护性 1-5），最后 recommend 1 个"

用限定选项 + 显式 trade-off，模型给出的 reasoning 比开放性问题深 3 倍。

④ 10 大反模式 + 修正

#反模式为什么烂修正
1"给我写个 X"无 context / constraint，5 个不同实现用 Anatomy 4 组件
2跨多任务用同一 chatContext 污染，AI 引用错文件每任务新 chat
3不读 diff 直接 commitAI hallucinate API / silent bug逐行 review；至少跑 lint + test
4"简单点" / "聪明点"主观词 AI 无法 calibrate给具体 metric（行数 / 复杂度 / 性能）
5不给 codebase exampleAI 默认 mainstream style，跟 codebase 风格冲突用 Pattern 6 few-shot
6同时改多文件每个文件改动浅，跨文件 inconsistent分文件改 / chain-of-edits
7测试也让 AI 写完不 review测试可能 fit 错代码而 pass测试也要 review；或 test-first 模式
8"修复 bug" 不给 reproduceAI 猜 bug 在哪给 stack trace / minimal repro / 期待行为
9无限轮 retry "再试一次"每次 retry 没新 context，结果可能更糟每次 retry 加新约束（"上次错在 X，避免"）
10对 hallucinated API 没 verification ritual编译过但 runtime 崩API 用 Pattern 9 cite-source；跑 lint + 测试

⑤ 工具特定技巧

Cursor

  • @file / @symbol / @folder—— 精确 select context，不要靠 auto-attach
  • .cursorrules 仓库根目录放团队约定（见 AI-Native 全景 §5）
  • Composer (Cmd+I) for 单文件改 / Chat (Cmd+L) for 多文件 / Agent for 自主任务
  • Apply diff 而不是 copy-paste—— Cursor 自动 reconcile 跟你的本地改动
  • Multi-model routing—— code 用 Claude / debug 复杂 用 GPT / 快速补全 用 Cursor's own model

Claude (Desktop / Web)

  • 长 context (200k+)—— 适合 dump 整个 module 让它 explain / refactor
  • Artifacts—— 让它写成可独立 render 的文档 / mermaid 图
  • Projects 功能—— 保存 system prompt + reference 文档，跨 session 复用
  • Computer Use—— 让 Claude 直接操作浏览器 / 终端做 agent-like 任务
  • 不擅长 实时 multi-file IDE-integrated edit（不如 Cursor）

GitHub Copilot

  • Tab 补全—— 最顺，hot path 写代码时
  • Chat (sidebar)—— question 用，不擅长 multi-file 改
  • @workspace—— 整 workspace context（但精度不如 Cursor）
  • Copilot Workspaces—— 类似 Cursor Agent，但 maturity 2026 仍跟不上
  • 合规优势—— GitHub Enterprise 集成深，部分大公司只能用 Copilot

⑥ 多轮 vs 单次 · 何时切新 chat

单次（one-shot）适合

  • 简单封闭任务（< 100 行 code generation）
  • 无后续 follow-up（不会基于结果迭代）
  • 不需要保留 reasoning history

多轮（multi-turn）适合

  • spec-then-code 模式
  • chain-of-edits
  • debugging（每轮新数据 / hypothesis）
  • 需要 AI "记住" 之前的 trade-off 决策

切新 chat 的信号

  • 任务语义变了（从 reconnect 改去 metrics 实现 → 切）
  • AI 开始引用不存在的之前内容（context 污染）
  • 同样 prompt 输出突然变差（context 太长 attention 衰减）
  • 切换角色 / persona（之前 reviewer，现在要 coder）
  • 对话超过 ~20 轮 / 100k token（多数模型 effective context 限制）
"context 污染" 的实例: 第 1 轮你给了 fileA, 第 5 轮让它改 fileB。AI 可能开始把 fileA 的概念套到 fileB。每个不相关任务起新 chat。

⑦ 验证仪式（不要 vibe ship）

不读 diff 就 commit 是 2024-2026 最大 AI 反模式。建立verification ritual：

分层 verification

Layer检查项成本
L1 自动Linter / formatter / type check pass0（CI 自跑）
L2 测试所有现有测试 + 新加测试 pass分钟级
L3 静态分析clang-tidy / SAST tool clean分钟级
L4 SanitizerASan / TSan / UBSan 通过分钟级
L5 人 review逐行读 diff + 理解每一改30 sec / 行
L6 同事 review另一人 review (常规 PR process)正常 review 时间

不可跳过 L5。Sanitizer 也漏一些 bug（业务逻辑错 / API 用错版本 / 安全 issue）。

"AI-generated label"

PR 标记哪些是 AI 生成 + 你 review 了什么。Future 你 / team / audit 都能查。

## PR description

This PR adds auto-reconnect to Session.

🤖 AI-assisted: ~70% (Claude Sonnet)
- Initial design: AI-suggested, I revised exponential backoff bounds
- Implementation: AI wrote, I reviewed all 200 lines
- Tests: I wrote first, AI implemented to pass
- Verified: ASan + TSan + manual 30-min stress test

Reviewer focus areas:
- ReconnectPolicy thread safety (I added explicit mutex, AI's initial was missing)
- Edge case: reconnect during shutdown (added test)

⑧ 何时放弃 prompt 自己写

不是所有任务都该让 AI 做。如下情况放弃 prompt 自己写：

  • 3 轮 prompt 仍不对—— AI 不理解你的意图，自己写更快
  • 1 行 trivial 改（typo / rename）—— 启动 AI cost > 自己改
  • 核心架构决策—— AI 能 generate options，但选哪个不该 AI 拍板
  • 性能 critical hot path—— AI 输出通常"正确但慢"，hot path 自己写 + benchmark
  • 安全敏感代码—— AI 不能 evaluate 你的 threat model
  • 非常规问题（domain-specific / proprietary protocol）—— AI 训练数据不够
  • 你需要 learn—— 让 AI 写完你也不懂 → 长期 ceiling 低
启发式：写 prompt 之前问"这事我自己写要多久"。如果 < 5 min，多半自己写更快。Prompt 的 setup / verification cost 在小任务上不划算。

⑨ 团队 prompt 库

个人 prompt 技巧 → 团队 amplifier 关键是把好 prompt 标准化 + 共享。

实施方法

  1. 仓库 .prompts/ 目录—— Markdown 文件 1 个 prompt / file 
    .prompts/
├── add-new-handler.md
├── refactor-to-expected.md
├── debug-with-stack.md
├── design-review-checklist.md
└── README.md  ← index
  2. Cursor / Continue 集成—— 把 .prompts 注入 .cursorrules 或当 snippet
  3. 每个 prompt 包含：① 适用场景；② Anatomy 4 组件；③ 1-2 个 example 输出；④ 已知 limitation
  4. 团队 retrospective—— 月度 review "哪些 prompt 高频用" + 哪些 retire
  5. 新人 onboarding—— prompts/ 是 onboarding doc 的一部分

例：refactor-to-expected.md

# Refactor Exception → std::expected

## When
重构 throwable function 到 std::expected pattern.
Codebase 倾向 expected 而非异常（hot path / 跨 ABI / 业务可预期错误）.

## Prompt template

Context: 我们 codebase 用 tl::expected (C++17) for predictable errors.
Reserve exceptions for catastrophic only (OOM / programmer bug).

Goal: 重构 @target_function.cpp 把 throw X 改成 return std::unexpected(...).

Constraints:
- API 签名变 (return type T → expected<T, ErrorCode>)
- ErrorCode 用现有 enum class（看 @errors.hpp）
- Caller 必须更新（找 callsite）
- 所有现有测试需更新断言（expected.has_value() / expected.error()）

Output:
1. 列 callsite 数量
2. 给 patch (function impl + caller updates)
3. 更新测试 (don't write new ones unless missing coverage)

## Example
[link to past PR using this prompt]

## Limitations
- 不适用 ctor (ctor 不能 return)，那种保留异常
- 不适用 RAII dtor (无 return)

⑩ 2027+ 展望

Trends

  • Tool use / function calling—— LLM 不只写代码，能 invoke 工具（编译 / 测试 / git / 数据库）。Prompt 形式从 "describe task" 变 "给 goal + 工具集"
  • Agent autonomy 提升—— 简单 task (bump deps / lint fix) 完全 autonomous; 复杂 task 半自动
  • Prompt 自动化—— prompt-writing-prompt (元 prompt)，让 LLM 帮你 refine prompt
  • 团队 prompt 库变 prompt registry—— versioned, tested, deployed like code
  • Verification 自动化—— AI 自己跑 lint / test / benchmark, only escalate fail case
