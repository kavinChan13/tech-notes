3 种 Coding Agent 形态

🎬生活类比 · 3 种助理
Tab 补全（Copilot 经典模式）= 边写边给你递词，像有人在你旁边轻声提示。
Chat / Edit（Cursor / Continue）= 你告诉它"做 X"，它帮你改文件，像跟同事远程协作。
Autonomous Agent（Devin / Claude Code）= 给一个 high-level 任务，它自己规划+执行，像派一个实习生跑去干一周。
形态典型用户参与任务粒度
Tab 补全GitHub Copilot / Tabnine每秒1-3 行
Chat / EditCursor / Cline / Continue / Aider每分钟10-100 行
AutonomousDevin / OpenAI Codex / Claude Code每小时-每天整 feature / bug fix
2026 趋势：3 种同一产品融合。Cursor 既有 Tab 补全又有 Chat 又有 Agent 模式。用户根据任务复杂度自由切换。
02

核心组件（每个 Coding Agent 都有）

① Codebase Indexing

把整个 repo embedding 进向量库。Cursor 用 ast-grep 拆函数级 + bge-code embedding；本地存（不上传完整代码）。查询时把相关函数拉进 context。

② Context Builder

动态组装 prompt：当前文件 + 光标位置周围 + 相关函数（vector search）+ LSP 信息（type / definition）+ 最近 edit 历史。这是 Cursor 跟 Copilot 准确率差异的关键。

③ Model Router

不同任务路由不同模型：Tab 补全用 1B 快模型，Chat 用 GPT-4 / Claude / DeepSeek，复杂 reasoning 用 o1/R1。Cost 跟 latency 优化的关键。

④ Tools

read/write file, run terminal, git diff, web search, run test, debug。Cursor / Cline / Claude Code 内置一套；Devin 额外有 browser。

⑤ Sandboxing

所有代码改动跑在隔离环境（container / VM），失败可 rollback。Devin 跑独立 VM；Cursor agent mode 用临时 git branch。

⑥ Agent Loop

Plan → Action → Observation → Loop。带 reflection（失败反思）+ termination（成功/失败/超时）。

⑦ Memory

长期记忆（用户偏好 / 之前 session）+ 短期（当前对话 + recent edits）。Cursor 的 .cursorrules / memory feature 就是这个。

⑧ UI / Streaming

streaming token 显示 + 可中断 + diff 预览 + 接受/拒绝按钮。UX 是 coding agent 用户体验的命门。

Cursor 内部拆解

3.1 Cursor 跟 VSCode 的关系

Cursor 是 VSCode fork（直接 fork 源码改），不是插件。这给了它 3 个优势：① 深度集成 UI（chat 嵌入 IDE）② 控制 LSP / 文件系统 / git 集成 ③ 不被 Microsoft 限制 API。

3.2 Cursor Tab（补全）的快速秘密

  • 自研小模型：Cursor 训练了自己的 1-2B 模型用于 Tab 补全，比直接调 OpenAI API 快 10x，cost 低 50x
  • Context-aware：不只看当前行，看光标前后 + recent edits（你刚改了什么）
  • Multi-line：能补全整段，不只一行
  • Speculative：用户停顿前就开始算，显示时已经准备好

3.3 Cursor Composer / Agent 模式

Composer = 多文件 edit。用户描述需求 → Cursor 自动改多个文件 + 显示 diff + 用户接受/拒绝。
Agent mode = 完全 autonomous（planning + tool use + iteration）。

Cursor 价值的本质：不是"用了 GPT-4"（大家都能用），是 context engineering（找到当前任务最相关的 100-1000 行代码塞进 prompt）+ UX engineering（diff 预览 / accept-reject / 流畅切换）+ 自研小模型（Tab 补全的速度 + cost 优势）。
04

Devin / SWE Agent · 完全自主

4.1 Devin 关键架构

  1. 独立 VM：每个任务起一个 isolated VM（不污染本地）
  2. Tool 套件：终端 / 浏览器 / 代码编辑器 / git / web search
  3. 长程 planning：能跑 hours / days 级任务
  4. UI 重构：不是 IDE 插件，是 web 仪表盘，可以放着不管

4.2 真实表现 vs 营销

⚠ 营销 demo 跟现实

Demo 里 "Devin 自动修复 Bug + 提 PR" 看起来很美。实际 SWE-bench 准确率 ~30-50%（2025 数据），即一半以上任务失败。

✓ 真实有用的场景

① 简单的 bump deps 类 chore ② 高度结构化的 lint fix ③ 重复性 boilerplate 生成。复杂 feature 当下 autonomy 还不够。

4.3 同类项目

  • OpenAI Codex CLI：CLI 风 agent，开源
  • SWE-agent（学术）：princeton 出，SWE-bench 测试用
  • OpenHands（前 OpenDevin）：开源 Devin 替代
  • Aider：CLI 多文件 edit，git native
05

Claude Code · Anthropic CLI Agent

Claude Code（2025）是 Anthropic 推出的 CLI coding agent。设计哲学跟 Devin 完全不同：

🎯 设计哲学

  • "打开一个 terminal 就能用"，不需要 IDE
  • 跟你对话式而非 hands-off
  • 每个改动都展示 diff，需要你确认
  • 支持 git / tools / web search
  • 原生支持 Anthropic Skills

📦 跟 Cursor / Devin 互补

  • Cursor: IDE 内 edit，visual
  • Devin: 完全 autonomous，放着跑
  • Claude Code: CLI + 对话，chatops 风格
  • 选择取决于工作流

Skills 实现机制

详见 X09 Ch6。这里拆解Cursor 怎么实现 Skills。

6.1 Cursor Skills 加载流程

  1. 启动时扫描 ~/.cursor/skills 和项目 .cursor/skills
  2. 解析每个 SKILL.md：提取 name / description / 使用条件
  3. 把 skill 描述加入 system prompt（让 LLM 知道有这些可选）
  4. 用户请求时 LLM 自行判断"这个任务适合哪个 skill"
  5. 选定后把对应 SKILL.md 完整内容加入 prompt
  6. 按 SKILL.md 描述的工作流执行（调 MCP tools 等）

6.2 Skills 的工程含义

Skills 把"工作流知识"从模型权重移到可加载文件。优势：① 更新 skill 不用重训模型 ② 团队定制不影响 base 模型 ③ 可 review / version / 共享。是 2026 agent 工程的关键范式。

6.3 写一个好 Skill 的原则

  1. description 写好 — LLM 据此判断何时激活
  2. 工作流明确 — 步骤化（1, 2, 3...）
  3. 声明依赖的 tools
  4. 给 examples（few-shot 极大提升准确率）
  5. 限制 scope（1 skill 1 个明确任务，避免 catch-all）

自己设计一个 Coding Agent · 关键决策

7.1 5 个关键决策

决策选项推荐
载体IDE 插件 / IDE fork / CLI / Web视团队 stack 选；新项目用 CLI 起步
autonomy完全人工 review / 部分 / 全自动从"部分"开始，渐进加 autonomy
模型OpenAI / Claude / 开源多模型 router；Tab 用小模型
Codebase 上传本地 / 云端合规优先 local（用本地 embedding）
EvalSWE-bench / 内部 / 用户反馈3 个都要：bench 找 SOTA / 内部 set 找业务匹配 / 用户反馈是金标准

7.2 Minimum Viable Agent (~500 行代码)

# Pseudo Python
class CodingAgent:
    def __init__(self, llm, tools, codebase_index):
        self.llm = llm
        self.tools = tools  # dict: name → function
        self.codebase = codebase_index  # vector search

    def run(self, task: str, max_iters=15):
        history = [system_prompt()]
        history.append({"user": task})

        for i in range(max_iters):
            # 1. Build context: 检索相关代码
            relevant_code = self.codebase.search(task, k=10)
            # 2. 调 LLM 决定下一步动作
            response = self.llm.complete(history + relevant_code)
            # 3. Parse tool call
            tool_call = parse(response)
            if tool_call.name == "DONE": break
            # 4. Execute
            result = self.tools[tool_call.name](**tool_call.args)
            history.append({"assistant": response,
                            "observation": result})
        return history

500 行就能跑一个基础 coding agent。真正难的是后面的 99% 工程：context 选择 / 错误恢复 / 多轮 streaming / diff UI / sandbox / cost 控制 / eval。

10 道 Coding Agent 面试题

Q1Cursor / Copilot / Devin 3 种形态区别？
Tab 补全（用户每秒互动 / 1-3 行）/ Chat-Edit（每分钟互动 / 10-100 行）/ Autonomous（每小时互动 / 整 feature）。3 种 2026 在 Cursor 等产品里融合。
Q2Cursor 比直接调 GPT-4 API 快 / 准在哪？
① Context engineering：codebase indexing + 智能选相关函数塞 prompt ② 自研小模型 Tab 补全（1-2B，快 10x cost -50x）③ UX 工程（streaming / diff / accept-reject）④ Multi-model routing（小问题用小模型）
Q3Devin 为什么 demo 牛但实际效果一般？
SWE-bench 准确率 ~30-50%（2025），一半以上任务失败。真正有用场景：bump deps / lint fix / boilerplate 生成。复杂 feature 当下 autonomy 不够，需要人工 review。
Q4Codebase Indexing 怎么做？
① 用 ast-grep 按函数/类拆 chunk（不是固定字符）② Embedding（如 bge-code）③ 存 Qdrant / 本地 sqlite-vec ④ 增量更新（监听文件变化）⑤ 查询时用 query embedding 找 top-K 相关函数。
Q5Multi-model routing 怎么决定用哪个模型？
① 任务类型分类器：Tab 补全 → 1B；简单 chat → 7B；复杂 reasoning → o1/R1 ② 用户 tier：免费用户小模型，付费用户大模型 ③ cost budget：超出 quota 降级 ④ fallback：模型 down 切备用
Q6Agent Loop 怎么避免无限循环？
① max_iters（典型 15-30）② cost budget（超 $X 停）③ 无进展检测（连续 3 次相同 action 停）④ time budget（超 N min 停）⑤ 用户中断按钮
Q7Sandbox 怎么设计？为什么必须？
所有 agent 改动跑在隔离环境（container / 临时 git branch / VM）。必须原因：① agent 可能写错代码破坏 main ② 跑测试不污染本地 env ③ 失败时一键 rollback ④ 安全（防 agent 意外删文件）
Q8Coding Agent eval 怎么做？SWE-bench 局限？
3 层 eval：① SWE-bench（学术标准，但 hard，跟生产场景未必匹配）② 内部 eval（自己业务 repo + 自己 task）③ 用户反馈（接受率 / 撤销率 / NPS）
SWE-bench 局限：题集偏 Python OSS / 单 PR fix / 不涉及多服务 / 时间敏感性差
Q9给团队选 Cursor / Cline / Claude Code 怎么选？
① 主力开发用 Cursor（IDE 集成最好）② privacy 严格用 Cline + 本地模型（开源 VSCode 插件）③ CLI 工作流用 Claude Code / Aider（ChatOps）。多个并用是正常的。
Q102026 Coding Agent 趋势？
① 融合：Tab / Chat / Autonomous 3 模式同产品 ② Skills 范式：领域工作流封装 ③ autonomy 提升（但需要 review/sandbox）④ 多 agent 协作（reviewer + coder + tester）⑤ 本地小模型（隐私 + cost + offline）⑥ codebase scale 升级（百万行 repo 也能用）
