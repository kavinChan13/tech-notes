01

为什么是 Transformer

2017 年 Attention is All You Need 之前

2017 年 Attention is All You Need 之前，序列建模主流是 RNN/LSTM。Transformer 把它扫地出门，核心因为它一次性解决了 3 个老问题：

RNN 时代的痛

  • 串行计算 — 无法并行训练
  • 长距离依赖 — 梯度消失
  • 固定隐藏态 — 信息瓶颈

Transformer 答卷

  • 所有位置同时算 — 训练并行
  • Attention 任意距离 O(1)
  • 每层全 token 间通信

但新代价

  • Attention 计算 O(n²)
  • 训练显存 ∝ n²
  • 推理时 KV-Cache 显存爆
2026 视角下的核心命题：现代 LLM 的所有架构创新（GQA / MLA / FlashAttention / RoPE / MoE / Reasoning）—— 本质上都是在跟 Transformer 引入的 3 个新代价斗争：①训练显存②推理 KV-Cache③推理 latency。看每一项创新时都问"它在打哪一个？"，理解会立刻深一层。
02

Attention 数学原理（推到细节）

从一个直觉开始 · Scaled Dot-Product Attention 公式 · 为什么要除 √dk？（高频面试题） · Multi-Head Attention

2.1 从一个直觉开始

给你一个句子"The cat sat on the mat because it was tired"，"it" 指什么？人类看上下文决定。Attention 就是把这个"看上下文"形式化：每个 token 用自己的 query 去问其它 token "我跟你多相关"，相关性高就多拿一点对方的信息。

2.2 Scaled Dot-Product Attention 公式

📐 核心公式（一定要背）
Attention(Q, K, V) = softmax(QKT / √dk) · V
  • Q（query）= 当前 token 的提问，shape [batch, seq, d]
  • K（key） = 每个 token 的"我是谁"标签，shape [batch, seq, d]
  • V（value）= 每个 token 的"实际内容"，shape [batch, seq, d]
  • QKT：相关性 logits，shape [batch, seq, seq]
  • ÷√dk：scaling，防止 d 大时 softmax 梯度饱和
  • softmax：每行归一化为 attention weight
  • ·V：按 weight 加权求和 value
Q → Q WK → K WV → V QKT / √d + Causal Mask (decoder 才用) softmax · V → 输出 QKT: [n, n] 相关性矩阵 · softmax 每行归一 · 与 V 加权求和 Causal Mask: 上三角填 -inf，让 token i 只能看 0..i 复杂度: 计算 O(n²d), 显存 O(n²) (中间矩阵)

2.3 为什么要除 √dk？（高频面试题）

假设 Q, K 元素独立 iid，方差均为 1。则点积 q·k = Σ qiki 是 dk 个独立项相加，方差 = dk，标准差 = √dk。如果 dk=128，点积量级是 ~11，softmax 输入差几个 11 就基本是 one-hot 了 — 这时反向传播梯度几乎为 0，模型学不动。

除以 √dk 把方差归一回 1，softmax 在"温和"区间，梯度健康。这就是为什么不是 d 也不是 d²。

2.4 Multi-Head Attention

把一个 d 维 Q/K/V 切成 h 个 head（每个 head 维度 dh = d/h），每个 head 学到不同的关系模式（语法、语义、指代…），最后 concat 回 d 维做 linear。

def multi_head_attention(x, n_heads):
    Q, K, V = x @ W_q, x @ W_k, x @ W_v        # [B, N, D]
    Q = Q.view(B, N, n_heads, d_h).transpose(1, 2)  # [B, H, N, d_h]
    K = K.view(B, N, n_heads, d_h).transpose(1, 2)
    V = V.view(B, N, n_heads, d_h).transpose(1, 2)
    scores = Q @ K.transpose(-2, -1) / sqrt(d_h)  # [B, H, N, N]
    scores = scores.masked_fill(causal_mask, -inf)
    attn = softmax(scores, dim=-1)
    out = attn @ V                              # [B, H, N, d_h]
    out = out.transpose(1, 2).reshape(B, N, D)
    return out @ W_o
面试可能追问：
  • "如果 head 数 = 1 vs head 数 = 32，模型表达力差别在哪？" → 答：head 切分等价于不同子空间投影，head 多 = 模型能同时关注多种关系，但每个 head 维度更小。GPT-3 175B 用 96 head，每个 dh=128 是经验最优。
  • "为什么不直接用一个 D 维 attention？" → 答：实证上 multi-head 显著优于 single head，因为 softmax 限制了每个 head 的 attention 分布只能"集中于少数 token"，多 head 才能并行关注多个模式。
03

KV-Cache

3.1 为什么需要 KV-Cache · 3.2 KV-Cache 算法（伪代码） · 3.3 KV-Cache 显存计算（必考） · 3.4 Prefill vs Decode 两个阶段

3.1 为什么需要 KV-Cache

LLM 推理是 自回归 — 一次出一个 token。生成第 t 个 token 时要算 attention(Qt, K0..t, V0..t)。如果每步都重新算 K0..t-1 和 V0..t-1，总复杂度 O(n³)，根本跑不动。

关键观察：Ki 和 Vi 一旦算出来，后续所有 step 都不会变（因为它们只依赖 token i 自己和它的 position）。所以缓存起来。

3.2 KV-Cache 算法（伪代码）

# 第一次：处理 prompt（prefill 阶段，并行算）
K_cache = K(prompt_tokens)   # [B, H, n_prompt, d_h]
V_cache = V(prompt_tokens)

while not_done:
    # Decode 阶段：每步只算 1 个新 token
    q_t = Q(new_token)            # [B, H, 1, d_h]
    k_t, v_t = K(new_token), V(new_token)
    K_cache = concat(K_cache, k_t, dim=-2)
    V_cache = concat(V_cache, v_t, dim=-2)
    attn = softmax(q_t @ K_cache.T / sqrt(d_h)) @ V_cache
    next_token = sample(attn @ W_out)
    new_token = next_token

3.3 KV-Cache 显存计算（必考）

💾 单 batch 的 KV-Cache 显存 = 2 × n × L × H × dh × bytes
  • 2 = K 和 V 各一份
  • n = 序列长度（含已生成）
  • L = transformer 层数
  • H = head 数（MHA）或 GQA 的 KV head 数
  • dh = 每 head 维度
  • bytes = FP16 是 2，BF16 是 2，FP8 是 1

实例：Llama 3 70B (MHA, L=80, H=64, dh=128, FP16) 在 32k 上下文：

2 × 32k × 80 × 64 × 128 × 2 字节 ≈ 84 GB

对，单个用户 32k 上下文的 KV-Cache 就要 84 GB。H100 80GB 装不下。这就是为什么有了 GQA / MLA / PagedAttention — 都是为了砍这个数字。

3.4 Prefill vs Decode 两个阶段

Prefill 阶段

  • 输入：整个 prompt（n 个 token）
  • 输出：1 个 token + KV-Cache (n 个位置)
  • compute-bound（GEMM 大矩阵 × 大矩阵）
  • GPU 利用率高，throughput-friendly
  • FLOPs ≈ 2 × n × P (参数量)

Decode 阶段

  • 输入：1 个新 token + 已有 KV-Cache
  • 输出：1 个新 token + cache 长度 +1
  • memory-bound（每步要读整个 KV-Cache）
  • GPU 利用率低，latency-dominated
  • FLOPs ≈ 2 × P，但要读 GB 级 KV-Cache
记住这个 trade-off：Prefill 卡算力，Decode 卡显存带宽。所有推理优化要么针对前者（FlashAttention / Quantization 算力），要么针对后者（GQA / MLA / KV 量化 / PagedAttention）。在 Y02 推理引擎深潜里会详细讲。

位置编码

从 Sinusoidal 到 RoPE 到 YaRN

4.1 为什么需要位置编码

Attention 是排列不变的 — 把 token 顺序打乱，输出不变。但语言显然依赖顺序（"狗咬人" vs "人咬狗"）。所以需要给模型"位置信息"。

4.2 演进路线

方法典型模型原理外推能力
Absolute (learned) BERT / GPT-1/2 每个 position 一个可学习 embedding，跟 token embedding 相加 ❌ 训了几位置只能用几位置
Sinusoidal 原 Transformer 用不同频率正余弦函数生成 position vector ⚠️ 理论可外推，实际效果差
RoPE Llama / Qwen / DeepSeek 全部 把 Q/K 在二维平面上旋转 θ·m 角度 ✅ 2026 事实标准
ALiBi BLOOM / MPT 对 attention score 减一个跟距离成正比的 bias ✅ 外推好，但效果稍不如 RoPE
YaRN 所有长上下文模型 RoPE 频率重新缩放，把短训练扩到长推理 ⭐ 长上下文必备

4.3 RoPE · Rotary Position Embedding（必懂）

RoPE 的核心 insight：用旋转编码相对位置。把 Q 和 K 的每相邻两个维度看成一个 2D 平面，按位置 m 旋转 θ·m 角度。计算 qm·kn 时，结果 只依赖相对位置 m-n，绝对位置自动消掉。

📐 RoPE 数学（简化版）
# 把 q 的 d 维拆成 d/2 对，每对做 2D 旋转
for i in range(d/2):
    θ_i = 1 / 10000^(2i/d)
    angle = m * θ_i
    [q_2i, q_2i+1] = rotation_matrix(angle) @ [q_2i, q_2i+1]
# 对 k 做同样操作（用它自己的位置 n）
# 最后 q_m · k_n 的结果 = 只依赖 (m-n) 的函数

4.4 YaRN · 长上下文外推

问题：模型在 8k 上下文训练，能不能直接推到 128k？不能。RoPE 在没见过的位置（θ·m 旋转角度极大）会失效。

YaRN 的做法：对不同频率分量做不同 scaling。高频（旋转快的）保持，低频（旋转慢的）缩小，让原本训过的低频信号继续 work，高频信号通过 fine-tune 微调适应。

  • NTK-aware scaling：高频 vs 低频差异对待
  • YaRN：在 NTK 基础上加 temperature scaling + attention 修正
  • Llama 3.1 用类似技术从 8k 扩到 128k
  • Qwen2.5-1M 把上下文推到 1M tokens
记住这个事实：所有"长上下文模型"的本质都是 "短上下文训练 + 位置编码外推 + 长上下文微调"。从 0 训一个 1M 上下文的模型，attention O(n²) 让你直接破产。
05

Attention 变体

MHA → MQA → GQA → MLA

都是在打"KV-Cache 太大 / Decode 带宽太慢"这个问题。

5.1 GQA · Grouped-Query Attention（业界事实标准）

Q 还是 H 个 head，但 K/V 共享 g 组（典型 g=8）。Llama 3 70B 用 H=64, g=8（每 8 个 Q head 共享一组 K/V）。

  • KV-Cache 降低 H/g 倍（Llama 70B 降 8x）
  • 效果几乎不掉（仔细调参，比 MHA 差 < 0.5%）
  • 2026 几乎所有新模型都用 GQA（Llama 3 / Qwen 2/3 / Mistral / Gemma / Yi）

5.2 MLA · DeepSeek 的创新（超越 GQA）

DeepSeek-V2/V3 提出。核心 insight：不存储 K/V 本身，存一个低维 latent representation，需要算 attention 时用一个矩阵 W 投影出 K 和 V。

  • 原理：K = WK · c, V = WV · c，c ∈ Rr（r 远小于 H × dh）
  • 结果：KV-Cache 只存 c（比 GQA 还小 1.5-2x），但因为 c 用全维信息压缩，效果比 GQA 略好
  • 代价：每次 attention 多一次矩阵乘（WK/WV），prefill 稍慢，但 decode 大幅加速
  • 巧妙之处：在 attention 计算时做 operation fusion，能把 WK 跟 Q 的 weight 合并，避免显式还原 K
面试加分：能讲 "MLA 的 KV-Cache 大小 ≈ GQA / 2，但准确率持平甚至更好。代价是 prefill 多 ~15% FLOPs。DeepSeek-V3 把它做到极致，吸引了大量 Infra 团队跟进研究。"

FlashAttention 1/2/3

6.1 为什么标准 attention 慢 · 6.2 FlashAttention 核心 trick · Tiling + online softmax · 6.3 Online Softmax…

不是新的 attention 公式，是 同样的公式，更快地算。核心思想：避免显式实例化 n×n 的 attention 矩阵到 HBM。

6.1 为什么标准 attention 慢

原始实现 softmax(QKT) · V 需要：

  1. 算 QKT → 写到 HBM（n×n 矩阵）
  2. 读 QKT → softmax → 写回 HBM
  3. 读 softmax 输出 + V → matmul → 写到 HBM

瓶颈在 HBM ↔ SRAM 的数据搬运（GPU 上 HBM ~3 TB/s, SRAM ~19 TB/s）。中间 n×n 矩阵在 n=8k 时是 256MB，反复读写直接卡死。

6.2 FlashAttention 核心 trick · Tiling + online softmax

把 Q, K, V 分块（tile）加载到 SRAM，每次只算一个 Q-tile 跟所有 K-tile 的乘积，用 online softmax 算法增量更新结果，永远不实例化完整 n×n 矩阵。

  • FlashAttention v1 (2022)：解决 HBM ↔ SRAM 数据搬运。速度 2-4x, 显存从 O(n²) 降到 O(n)
  • FlashAttention v2 (2023)：进一步并行化（per warp）+ 减少非 matmul 算子。再 +2x
  • FlashAttention v3 (2024)：H100 Tensor Core + WGMMA + warp-specialization + FP8。再 +1.5-2x

6.3 Online Softmax 是核心

普通 softmax 要先看到全部 logits 才能算 max + sum。Tiling 时只看到一个 chunk —— online softmax 用增量更新：维护当前 max 和 累计 sum，下一个 chunk 来时 rescale。数学上等价但避免了存全部 logits。

# Online softmax (简化)
m_old, l_old, o_old = -inf, 0, 0
for chunk_idx in K_tiles:
    s = q @ K[chunk_idx].T          # scores for this chunk
    m_new = max(m_old, s.max())
    α = exp(m_old - m_new)
    β = exp(s - m_new)
    l_new = α * l_old + β.sum()
    o_new = (α * o_old + β @ V[chunk_idx]) / l_new
    m_old, l_old, o_old = m_new, l_new, o_new
return o_new
这是 AI Infra 的硬通货：理解 FlashAttention 是 Y 系列 (尤其 Y01 CUDA + Y02 vLLM 推理) 的前提。面试时一定会问"为什么 FlashAttention 快"——答 "tile + online softmax 避免实例化 n×n + HBM↔SRAM 搬运" 就稳。
07

现代 LLM 的其它关键模块

RMSNorm（替代 LayerNorm） · SwiGLU（替代 ReLU/GELU） · MoE · Mixture of Experts（DeepSeek-V3 / Qwen3 / Mixtral 关键）

7.1 RMSNorm（替代 LayerNorm）

LayerNorm 要算均值 + 方差。RMSNorm 只算均方根：RMS(x) = sqrt(mean(x²))，output = x / RMS(x) · g（g 是可学习 gain）。

  • 没有减均值项 — 假设特征已经 0-centered（实证 OK）
  • 计算 -20% ~ -30%，效果不掉
  • Llama / Qwen / DeepSeek 全部用 RMSNorm

7.2 SwiGLU（替代 ReLU/GELU）

FFN 块的激活函数。原始 Transformer: FFN(x) = max(0, xW₁) W₂（ReLU）。SwiGLU: FFN(x) = (Swish(xW₁) ⊙ xW₃) W₂，多一个 gating 分支。

  • Swish(x) = x · sigmoid(βx)，比 ReLU 平滑
  • ⊙ gating 让 FFN 有了"选通"能力，实证 +1-2% 准确率
  • 2026 几乎所有新模型用 SwiGLU。代价：FFN 多 50% 参数（3 个矩阵 vs 2 个）

7.3 MoE · Mixture of Experts（DeepSeek-V3 / Qwen3 / Mixtral 关键）

把一个大 FFN 拆成 N 个小 expert，每个 token 只激活 top-k（典型 k=2）个 expert。等参数 vs 等计算的 trade-off：参数多 N/k 倍，FLOPs 不变。

🔀 MoE 核心组件
  • Router：一个小 linear，给输入 x 输出 N 个 expert 的得分，softmax + top-k 选 expert
  • Experts：N 个独立的 FFN（典型 N=8 到 256）
  • Load balancing loss：防止 router 总选少数 expert 导致其它 dead
  • Shared experts（DeepSeek-V3）：少数 expert 总是激活，提供通用能力

MoE 的训练/推理工程含义：
  • 训练：参数全部要存 + 全部要训 — 需要 expert parallelism (EP)，跨 GPU 切 expert，详见 Y03
  • 推理：激活少，但 router 路由不可预测，对 batch 调度极不友好（不同 token 走不同 expert）— vLLM 0.5+ / SGLang 才支持，详见 Y02
  • 面试：能讲清"为什么 MoE 参数大但推理快" + "load balancing 的实现" 就很加分

8.1 DeepSeek-V3 为什么"出圈"

2024 年底发布的 DeepSeek-V3 是 2026 招聘官最常拿来问的模型。它的工程价值：

  1. MLA 把 KV-Cache 压到极致（比 GQA 还小，效果不掉）

长上下文工程

长上下文 3 大挑战 · 长上下文工程武器库

为什么 2026 模型上下文从 4k 推到 1M？应用场景：法律 / 医学全文档分析、长视频理解、Agent 长 horizon planning、Codebase 全库索引。

9.1 长上下文 3 大挑战

① Attention 计算 O(n²)

1M 上下文 = 1T 对 attention 计算。FlashAttention + Sparse Attention 也救不了。

② KV-Cache 显存爆炸

Llama 3 70B 在 1M 上下文 KV-Cache ~2.6 TB。再 GQA 也是几百 GB。

③ "中间遗忘" (Lost in the Middle)

模型对位置中段的 token attention 信号弱。位置编码外推后更明显。

KV-Cache 压缩（H2O / StreamingLLM）丢掉不重要的 K/V
RAG（外部检索）不用真的塞进上下文，按需取
2026 业界 consensus：纯长上下文（128k+）很贵，大多数场景 "RAG + 中等上下文" 比纯长上下文更经济。但 Agent 长 horizon、代码库分析这种是真长上下文场景，没法 RAG。
10

Reasoning Models

10.1 核心思想 · 10.2 工程含义 · 10.3 面试常问

2024 年底 OpenAI o1，2025 年 DeepSeek-R1 + QwQ-32B + Qwen3-Reasoning，开启了 Test-Time Scaling 时代 — 推理时让模型"想得更久"换来更好答案。

10.1 核心思想

传统 LLM：1 个 prompt → 1 个直接答案。
Reasoning LLM：1 个 prompt → 长长的 "thinking" → 最终答案。thinking 长度从几百到几万 tokens 不等，且 thinking 越长，难题准确率越高。

🧠 训练范式（DeepSeek-R1 paper 公开）
  1. R1-Zero：纯 RL（GRPO），不需要 SFT，奖励只用"答案对/错"，模型自发学会长 thinking
  2. R1：加少量高质量 SFT cold-start，再 RL，再蒸馏 → 更稳定 + 可读
  3. Reward：rule-based（数学题答案是否正确）+ language reward（中英不混）

10.2 工程含义

  • 推理延迟成倍增加：reasoning model 输出长 10-100x，latency 直接乘倍
  • Cost 飙升：1k token thinking + 100 token answer，价格按 1.1k 算
  • UI/UX 改变：用户看不到 thinking（hide）或看到一部分（streaming with summary），跟传统 chat 完全不同
  • Eval 方法重新设计：reasoning model 在数学 / code / 长任务上猛涨，但日常对话可能反而变啰嗦 — 新评测体系
  • 非 reasoning 也在融合：Qwen3 同一个模型可切换 "thinking" / "non-thinking" 模式

10.3 面试常问

"Reasoning model 跟 Chain-of-Thought (CoT) prompting 有什么本质区别？"

答案要点：CoT prompting 是提示工程层面让模型一步步想，但模型本身没专门训练；reasoning model 是训练层面用 RL 让模型 internalize "想清楚再答"的策略，不需要 prompt 提示，且 thinking 质量 + 长度自适应难度。

Prefill vs Decode

11.1 Roofline 模型 · 11.2 工程含义

前面 §3 讲过基本概念，这里从 Infra 工程视角再展开 — 这是 AI Infra 面试的硬通货。

11.1 Roofline 模型

一个 GPU 的"算力上限" = 算力（FLOPS）vs 带宽（B/s）的 roof。一个 kernel 是 compute-bound 还是 memory-bound，看 arithmetic intensity（每读 1 字节做多少 FLOPs）。

  • Prefill：高 arithmetic intensity（大 GEMM）→ compute-bound
  • Decode：低 arithmetic intensity（每读整个 KV-Cache 只做 1 个新 token 的少量 op）→ memory-bound

11.2 工程含义

优化对 Prefill 影响对 Decode 影响
FlashAttention✅ 大幅加速✅ 加速（少 HBM 读）
Quantization (INT4)✅ 大幅加速✅✅ 极大加速（KV-Cache 也量化）
GQA / MLA⚠️ 中性（FLOPs 不变）✅✅✅ 大幅加速（KV 小）
Continuous Batching✅ 提升 throughput✅✅ 提升 throughput
Speculative Decoding—✅✅ 加速 decode
Prefill-Decode 分离专属 GPU专属 GPU
顶级面试回答模板："首先识别瓶颈 — prefill 是 compute-bound 看 FLOPs，decode 是 memory-bound 看带宽。如果客户是 batch size 大的离线任务，prefill 主导，重点是 throughput 和 attention 加速；如果是在线 chat，decode 主导，重点是 KV-Cache 砍 + speculative decoding 加速 + continuous batching 提高 GPU 利用率。"

高频面试题（含答案要点）

iid 假设下点积方差 = dk，量级 = √dk

Q1. Attention 公式里为什么要除以 √dk？

iid 假设下点积方差 = dk，量级 = √dk。如果不归一化，dk 大时 softmax 输入差距大，输出近 one-hot，梯度饱和模型学不动。除 √dk 把方差归一回 1。

Q2. KV-Cache 的显存怎么计算？为什么需要 GQA / MLA？

显存 = 2 (K+V) × n (seq) × L (层) × H (head) × dh × bytes。Llama-70B 32k 上下文 ~84GB，单 H100 装不下。GQA 把 H 减到 H/g，MLA 用 latent 进一步压缩 — 都是为了让显存 fit + 提高 batch size。

Q3. RoPE 为什么比 Sinusoidal / Absolute 好？

RoPE 编码相对位置（qm·kn 只依赖 m-n），而前两者编码绝对位置。相对位置更符合语言本质（"前一个词"无论在哪都重要），且对未见过的长度外推更稳。

Q4. FlashAttention 为什么快？跟 Sparse Attention 区别？

FlashAttention 用 tiling + online softmax 避免实例化 n×n 矩阵到 HBM，靠减少 HBM↔SRAM 数据搬运提速 — 数学完全等价标准 attention。Sparse Attention 是数学上不同（只算部分 token 对），是 approximation。

Q5. MoE 模型推理时为什么"激活参数" 远小于"总参数"？工程上的挑战？

每个 token 经 router 只选 top-k expert（k 远小于 N），所以单步只激活 k/N 参数。工程挑战：① 所有 expert 都要装显存 ② 不同 token 路由到不同 expert，导致 batch 内"算 expert 时部分 GPU 空闲"，需要 expert parallelism + 调度优化 ③ Load balancing 防止 dead expert。

Q6. 为什么 Prefill 是 compute-bound 而 Decode 是 memory-bound？怎么优化？

Prefill 处理 n 个 token 同时，attention 是 n×n GEMM — 算力密集。Decode 每步只算 1 token 的 attention，但要读整个 KV-Cache（GB 级），arithmetic intensity 极低。Prefill 优化方向：FlashAttention / 量化 / 减少 FLOPs；Decode 优化方向：GQA / MLA 砍 KV / 量化 KV / Continuous Batching / Speculative Decoding。

Q7. RMSNorm 跟 LayerNorm 区别？为什么改用？

RMSNorm 不算均值，只用 RMS 归一。计算少 -25%，效果实证持平。其它简化（参数也少一个 bias）让现代 LLM 都换了。

Q8. SwiGLU 比 ReLU FFN 好在哪？代价？

SwiGLU 引入 gating 分支 (Swish(xW₁) ⊙ xW₃) W₂，"选通" 能力更强，效果 +1-2%。代价：3 个权重矩阵 vs 2 个，FFN 参数和计算 +50%（实际为保持参数量，中间维度会缩到 8/3）。

Q9. DeepSeek-V3 跟 Llama 3 70B 架构上几个核心差异？

① MLA (DeepSeek) vs GQA (Llama) ② MoE 671B/37B vs Dense 70B ③ MTP 训练目标 (DeepSeek) vs next-token ④ FP8 mixed precision 训练 vs BF16 ⑤ 训练数据中推理样本占比 (DeepSeek 高)。

Q10. 长上下文（128k+）的核心工程难点？业界 work-around？

难点：① Attention O(n²) ② KV-Cache TB 级 ③ "lost in the middle"。Work-around：YaRN 外推 + GQA/MLA + KV 量化 + Sliding Window + Ring Attention + 大场景下用 RAG 替代纯长上下文。

Q11. Reasoning Model (o1 / R1) 跟传统 LLM + CoT prompting 本质区别？

CoT prompting 在提示层引导，模型未经训练；Reasoning model 在训练层用 RL 让模型学会自适应长度的 thinking，不需要 prompt 提示，且能根据题难度调节 thinking 长度。代价：latency 和 cost 飙升。

Q12. 给一个 13B 模型，估算它在单 A100 (80GB) 推理的最大 batch size（设上下文 4k, GQA H=32, g=8, FP16）

解：模型参数 ~26GB (13B × 2 bytes)。KV-Cache 每 batch ≈ 2 × 4k × L (~40) × (32/8) × 128 × 2 ≈ 0.33 GB/seq。可用显存 80 - 26 - 4 (activations) = 50 GB，max batch ≈ 150。面试加分：讨论 PagedAttention 能进一步提高有效 batch size（碎片化少）。
