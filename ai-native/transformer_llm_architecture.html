01

为什么是 Transformer · 它解决了什么

10 年技术演进的一句话总结 · 现代 LLM 一切创新的起点
🎬生活类比 · 翻译公司的进化
想象一家翻译公司。

RNN 时代（2014-2017）：只有 1 个翻译员。他必须从第 1 个字读到最后 1 个字，每读完一个字，把"已读内容的总结"写在小本子上交给下一秒的自己。问题：① 慢，串行 ② 翻到第 100 个字时，第 1 个字早忘了 ③ 永远不能并行。

Transformer 时代（2017-）：换成 100 个翻译员，所有人同时看完整段。每个翻译员负责一个字，但翻译时可以随意瞄旁边任何人的位置来理解上下文。结果：① 快，全并行 ② 任意距离的字都能直接联系 ③ 训练效率暴涨。

代价：100 个翻译员之间互相"看一眼"的次数是 100×100 = 10000 次。如果文档变成 10 万字，要互相看 100 亿次 — 这就是 Transformer 著名的 O(n²) 问题。
类比 → 术语
翻译公司 → LLM  ·  每个翻译员 → token  ·  "瞄旁边" → Attention  ·  O(n²) → attention 复杂度
Transformer 的核心创新不是"更好的算法"，而是"用并行换串行 + 用 O(n²) 换全局可见"。后续所有架构创新（GQA / MLA / FlashAttention / MoE / Reasoning）都是在跟这个 O(n²) 和它带来的显存/带宽问题做斗争。

10 年技术演进一张表

🪨 RNN/LSTM 时代

  • 串行计算 — 训练慢得离谱
  • 梯度消失 — 长依赖记不住
  • 固定隐藏态 — 信息瓶颈
  • 无法 GPU 充分利用

🚀 Transformer 答卷

  • 所有 token 同时算 — 训练快 10-100x
  • Attention 任意距离 — O(1) 跳跃
  • 每层全 token 间通信 — 信息密集
  • 完美适配 GPU 大矩阵乘

⚠ 但带来新代价

  • Attention 计算 O(n²)
  • 训练显存 ∝ n²
  • 推理 KV-Cache 显存爆炸
  • 推理 Decode 阶段带宽瓶颈
把上面这张表记在脑子里 — 后面每讲一个新技术，都会回头说 "它在打哪一条代价"。GQA 打 KV-Cache 大、FlashAttention 打训练显存、MoE 打参数瓶颈、MLA 同时打两个。这条"创新都是为了打补丁"的线索贯穿全文。

Attention · 图书馆查资料的数学化

Q/K/V 三件套 · 为什么除 √d · 多头机制

2.1 直觉来源 · 你在图书馆找资料

🎬生活类比 · 图书馆的智能助手
周末你去图书馆写论文，主题是"机器学习历史"。书架上有 1000 本书。你怎么办？

步骤 1（Query）：你心里有一个问题："我想找机器学习起源相关的内容"。这就是 Q（Query, 提问）。
步骤 2（Key）：每本书背面有一个标签，告诉你"我讲的是什么"。比如《深度学习》的标签是"神经网络/BP/AlexNet"。这就是 K（Key, 索引）。
步骤 3（匹配）：你拿你的问题跟每本书的标签对比相似度，越像分越高。这就是 Q · K 点积。
步骤 4（取内容）：每本书还有真正的内容（叫 V, Value）。你按分数权重，把内容糅在一起变成你的论文素材。

句子里的 attention 完全一样：每个词都同时扮演"提问者"+"索引"+"内容"3 个角色，互相查询、互相加权融合。
图书馆类比 → Transformer 术语
你的问题 → Query (Q)  ·  书的标签 → Key (K)  ·  书的内容 → Value (V)  ·  打分 → Q·KT  ·  权重融合 → softmax · V
Attention = 每个 token 在"我跟句子里所有人有多相关"这个问题上做加权投票。这就是为什么"The cat sat because it was tired"的 it 能准确知道指 cat — 因为 cat 的 K 跟 it 的 Q 匹配度最高。

2.2 Q / K / V 三件套 · 公式拆解

每个 token 的 embedding（向量 x）通过 3 个矩阵 WQ/WK/WV 投影出 q/k/v 三个向量：

# 对每个 token x，算出它的三件套
q = x @ W_Q   # 你的"问题" 
k = x @ W_K   # 你的"标签"，给别人看的
v = x @ W_V   # 你的"内容"，别人会拿走融合的

整段 attention 就是一行公式：

Attention(Q, K, V) = softmax(QKT / √dk) · V

具体每一步在干什么：

① QKT 算"相关性矩阵"

把所有 Q 跟所有 K 两两点积，得到一个 [n, n] 的矩阵。第 [i, j] 个值 = token i 对 token j 的关注度（未归一化）。

② ÷ √dk · 防数值爆炸

如果 d 大，点积量级很大，softmax 直接饱和（一个 1，其它都 0），梯度断掉模型学不动。除 √d 让方差归一。

③ + Causal Mask · Decoder 必须

语言模型只能看历史，不能看未来。把矩阵上三角填 -∞，softmax 后那些位置变 0。

④ softmax + 乘 V · 加权融合

每行归一为概率分布（和为 1），然后用这个权重去 V 里加权求和，得到该 token 的"上下文表示"。

2.3 为什么要除 √dk？高频面试

Q 和 K 各有 d 个维度，点积 = d 个独立项相加。方差是 d。如果不归一化，d 越大点积越大，softmax 输入差几个量级，输出几乎是 one-hot（一个 1 其余 0），梯度直接死掉。除 √d 把方差归一回 1，softmax 在"温和"区间，梯度健康。
数学为什么是 √d 不是 d 或 d²？
推导
假设 Q, K 元素独立 iid，方差均为 1，均值 0。则点积 q·k = Σ qiki 的方差 = Σ Var(qi·ki) = d × 1 = d。标准差 = √d。除以标准差就让 softmax 的输入方差归一回 1。
如果是 d
除以 d 会让方差变成 1/d，分布过于平坦，所有 token 权重几乎相等，attention 失去"选择"能力。
如果是 d²
除以 d² 会让方差变成 1/d²，softmax 完全均匀，attention 等于平均池化，完全废掉。

2.4 Multi-Head Attention · 多个翻译员各看一个方面

🎬生活类比 · 多个翻译员各看一个方面
一个翻译员可能只能关注"语法结构"。但如果你雇 8 个翻译员，每人专注一个角度（语法 / 指代 / 情感 / 命名实体 / 时态…），最后把他们的结果拼起来 — 这就是 Multi-Head Attention。

关键：8 个翻译员看同一句话，但每人用不同的 Q/K/V 投影矩阵，所以学到的"关注点"不一样。最后 concat 拼回去再过一次 linear 融合。 
def multi_head_attention(x, n_heads):
    Q, K, V = x @ W_q, x @ W_k, x @ W_v        # [B, N, D]
    Q = Q.view(B, N, n_heads, d_h).transpose(1, 2)  # [B, H, N, d_h]
    K = K.view(B, N, n_heads, d_h).transpose(1, 2)
    V = V.view(B, N, n_heads, d_h).transpose(1, 2)
    scores = Q @ K.transpose(-2, -1) / sqrt(d_h)  # [B, H, N, N]
    scores = scores.masked_fill(causal_mask, -inf)
    attn = softmax(scores, dim=-1)
    out = attn @ V                              # [B, H, N, d_h]
    out = out.transpose(1, 2).reshape(B, N, D)
    return out @ W_o
设计为什么不直接用一个 D 维 attention，而要分 H 个 head？
本质
因为 softmax 让每个 attention 分布倾向于"集中在少数 token 上"。如果只有 1 个 head，模型只能学一种关注模式。多 head 让每个 head 专注一种模式（语法/指代/情感…），表达力大幅提升。
数据
GPT-3 175B 用 96 head × d_h=128 是经验最优；head 太多每个维度太小学不出东西，head 太少表达力受限。

KV-Cache · 做卡片放手边

推理性能的命门 · 现代 LLM 推理优化的核心战场

3.1 为什么需要 KV-Cache

🎬生活类比 · 学生写论文做引用卡片
你写一篇 1 万字论文，边写边引用前面提到过的内容。

笨办法：每写一个新句子，都要重新读一遍前面所有内容才知道该引用什么。写到第 5000 字时，每个新句子都要回看 5000 字。
聪明办法：每读完一段，就做一张引用卡片（标题 + 关键词 + 内容摘要），放在桌子右边。新句子来了，扫一眼桌上的卡片就行，不用回头读全文。

LLM 推理一模一样：每个 token 的 K 和 V 一旦算出来就永远不变（只跟自己和位置有关）。所以每次只算新 token 的 K/V，跟桌上的 KV-Cache 一起做 attention，就够了。
引用卡片 → KV-Cache  ·  桌子大小限制 → 显存限制  ·  卡片越多越慢扫 → 长上下文带宽瓶颈
没有 KV-Cache 的推理：复杂度 O(n³)，1000 token 要算 10 亿次，根本跑不起来。有 KV-Cache：复杂度降到 O(n²)，但代价是显存占用线性增长，跟 batch size × 上下文长度成正比。

3.2 显存计算 · KV-Cache 究竟有多大高频面试

🎮 交互演示 · KV-Cache 增长过程

点"生成下一个 token"看 KV-Cache 怎么每步线性增长。10 步后看显存占用。

已生成: 0 tokens
已用显存: 0 MB
显存上限: 80,000 MB (H100 80GB)
还可生成: ∞ tokens

按 Llama-70B 70B 模型 (L=80, H=64, d_h=128, FP16, MHA) 配置估算

显存公式（必背）：

KV-Cache 大小 = 2 × n × L × H × dh × bytes

符号含义典型值
2K 和 V 各一份—
n序列长度（prompt + 已生成）4k / 32k / 128k
LTransformer 层数32 (7B) / 80 (70B)
Hhead 数（MHA）或 KV head 数（GQA）32 / 8
dh每 head 维度128 / 256
bytes精度FP16=2, FP8=1, INT4=0.5

典型实例：Llama 3 70B 在 32k 上下文

纯 MHA 配置（Llama 1/2 7B 风格）

2 × 32k × 80 × 64 × 128 × 2 字节
= 84 GB per sequence

单用户单 sequence 就要 84 GB — H100 80GB 装不下！

这就是为什么必须 GQA / MLA

GQA 配置（Llama 3 70B 实际）

2 × 32k × 80 × 8 × 128 × 2 字节
= 10.5 GB per sequence

同样的模型，GQA 把 KV-Cache 减到 1/8，能跑多用户 batch。

这是 GQA 在 2026 成为事实标准的原因

3.3 推理两阶段 · Prefill vs Decode

LLM 推理不是一个"过程"，是两个完全不同的阶段，工程优化思路截然相反。

🎬生活类比 · 餐厅服务流程
Prefill 阶段 = "上菜前的准备"：你点了 10 道菜（=输入 prompt 1000 字），厨房一次性并行准备所有菜。10 个厨师齐上手，效率极高，瓶颈在"火候"（算力）。

Decode 阶段 = "一道道吃菜"：你不能一口吃完，要一道道慢慢吃（=一次生成一个 token）。每吃一口都要回想"我之前吃了什么"（=读整个 KV-Cache）。这时瓶颈是"嘴和喉咙"（显存带宽），不是火候。

🔥 Prefill 阶段

  • 输入：完整 prompt（n 个 token）
  • 输出：1 个 token + 长度 n 的 KV-Cache
  • 计算特性：n×n 大矩阵乘 — compute-bound（算力瓶颈）
  • GPU 利用率高，throughput 友好
  • FLOPs ≈ 2 × n × P（P=参数量）
  • 优化方向：FlashAttention / Tensor Core / FP8

🐌 Decode 阶段

  • 输入：1 个新 token + KV-Cache
  • 输出：1 个新 token + cache 长度 +1
  • 计算特性：要读整个 KV-Cache（GB 级）— memory-bound（带宽瓶颈）
  • GPU 利用率低，latency-dominated
  • FLOPs ≈ 2 × P，但要读几 GB
  • 优化方向：GQA/MLA 砍 KV / 量化 / Continuous Batching
面试必背：所有现代推理优化都可以按"打 prefill 还是 decode"分类。FlashAttention 两个都打，但更利 prefill；GQA/MLA 主要打 decode；Continuous Batching 提高整体利用率；Speculative Decoding 专打 decode。Y02 vLLM 深潜全展开。

位置编码 · 钟表指针的旋转

从 Absolute 到 RoPE 到 YaRN · 让模型知道"谁在前谁在后"

4.1 为什么需要位置编码

🎬生活类比 · 打乱字条的问题
假设你把"狗咬人"3 个字写在 3 张纸条上，打乱顺序给模型。Attention 不在乎顺序（数学上 attention 是排列不变的），所以模型看到的是 {狗, 咬, 人} 这个"集合"，无法区分"狗咬人" vs "人咬狗"。

所以必须人为给每个 token 加一个"位置标识"，让 Q/K 不仅编码"我是什么词"，还编码"我在第几位"。

4.2 演进路线一张表

方法代表模型怎么编码外推到长上下文2026 地位
Absolute (learned) BERT / GPT-1/2 每个位置一个可学习 embedding，跟 token 加起来 ❌ 训了几位置只能用几位置 淘汰
Sinusoidal 原始 Transformer 用不同频率的正余弦 ⚠️ 理论可外推，实际不灵 过时
RoPE Llama / Qwen / DeepSeek 全部 把 Q/K 在 2D 平面"旋转" ✅ 配 YaRN 外推到 1M 事实标准
ALiBi BLOOM / MPT 给 attention score 减跟距离成正比的 bias ✅ 外推好 少数派
YaRN 所有长上下文模型 RoPE 频率重新缩放 ⭐ 长上下文必备 2026 必备

4.3 RoPE · 钟表指针类比

🎬生活类比 · 钟表指针的位置编码
想象每个 token 的 Q 和 K 是一个钟表指针。

第 0 个 token：指针指向 12 点（0°）
第 1 个 token：指针顺时针转 30°，指向 1 点
第 2 个 token：转 60°，指向 2 点
...
第 m 个 token：转 m × θ 度

神奇之处：当你算 qm 跟 kn 的点积时，结果只跟两个指针的"相对角度差"(m-n)有关，跟它们各自指向哪里无关。这就是"相对位置编码" — 模型不在乎"绝对第几位"，只在乎"我跟你差多少位"。

这才符合语言本质：你说"刚才那个"，模型只需要知道"在前面不远"，而不是"在第 142 位"。
RoPE 把 Q/K 每相邻两个维度看成 2D 平面上的向量，按位置 m 旋转 θ·m 角度。巧妙之处：qm·kn 的结果数学上等于 cos((m-n)θ) — 只依赖相对位置。所有的"前文/后文/n 步前"的语义都自动 baked in。 
# RoPE 简化版（实际用 d 对维度并行旋转）
def apply_rope(q, position):
    for i in range(d // 2):
        theta_i = 1 / 10000 ** (2*i / d)   # 第 i 对维度的旋转频率
        angle = position * theta_i          # 位置 m 转 m·θ 角
        # 把 (q[2i], q[2i+1]) 这一对维度旋转 angle 角
        q[2i], q[2i+1] = rotate_2d(q[2i], q[2i+1], angle)
    return q

4.4 YaRN · 让模型"看到没见过的远方"

🎬生活类比 · 钟表加长指针的危险
假设模型只在"8000 个 token 内"训练过 — 钟表指针只在 [0°, 8000θ] 这个范围内见过。

现在推理时给它 128000 个 token，指针要转到 [0°, 128000θ] — 大部分角度模型从没见过，效果会崩。

YaRN 的做法：把高频维度（旋转快的指针）保留原速，低频维度（旋转慢的指针）按比例放慢。这样模型见过的"角度范围"被压缩到原来的训练区间，用熟悉的频率去表达更长的距离。
  • NTK-aware scaling：高频维度 vs 低频维度区别对待
  • YaRN：在 NTK 基础上加 temperature scaling + attention 修正
  • Llama 3.1：用类似技术从 8k 扩到 128k
  • Qwen2.5-1M：把上下文推到 1M tokens
几乎所有"宣称支持百万上下文"的模型，本质都是 "短上下文训练 + RoPE 外推 + 少量长上下文微调"，不是从零训练百万上下文。从 0 训百万上下文，attention O(n²) 让你直接破产。

Attention 变体 · 翻译员共用查词员

MHA → MQA → GQA → MLA · 每代都在打 KV-Cache 太大这个问题

5.1 MHA / MQA / GQA 演进

🎬生活类比 · 翻译公司的精简方案
翻译公司有 32 个翻译员（=32 个 attention head），每人配 1 个独立的词典助手（=K/V）。每个翻译员有自己的提问角度，但都要带着一本厚字典出差 — 字典就是 KV-Cache，32 本字典加起来超重。

MHA（原始）：32 翻译员 × 32 独立字典助手 = 32 本厚字典。最准，但最重。
MQA（极端方案）：32 翻译员共用同 1 个字典助手。轻多了，但翻译员可能"撞车"（共享信息少），效果掉。
GQA（折中方案 ⭐）：32 翻译员分成 8 组，每组 4 个翻译员共用 1 个字典助手 — 共 8 本字典。效果接近 MHA，重量只有 1/4。

2026 几乎所有新模型都用 GQA。

5.2 MLA · DeepSeek 的杀手锏2026 新

🎬生活类比 · 带索引而不是带整本字典
GQA 让 8 个翻译员共用 1 本字典 — 还是要带字典出差。
MLA 的革命想法：不带字典本，只带一个超小的"索引册"（压缩的潜表示 latent c）。需要查词时，用一个公式把索引册临时还原成字典（WK·c, WV·c）。

带索引比带字典轻 5-10 倍（KV-Cache 极小），还原的字典质量 ≥ 原字典（因为是用全维度信息压缩出来的）。代价：每次查词多一步还原计算（prefill 慢 ~15%，但 decode 大幅加速）。

巧妙之处：DeepSeek 用了一个数学技巧，把 WK 跟 Q 的权重合并，实际计算时根本不用真的还原 K — 这就是 "MLA 比 GQA 快还效果好"的真相。

FlashAttention · 一摞摞搬书

不改算法，只改"算的时候怎么搬数据" · 2-5x 提速的工程奇迹
🎬生活类比 · 你为什么不一次搬完整个书架
想象你在图书馆找资料，书架在大厅（远 = HBM 显存），你的桌子在小阅览室（近 = SRAM 显存）。

标准 Attention：把 1000 本书全搬到桌上，对照查询、写笔记、再把笔记放回书架。光是搬书来回就累死。
FlashAttention：一摞一摞地搬（典型 64 本一摞）。每搬来一摞就当场算完跟你问题的相关性，把结果累积到桌角的"小本子"（online softmax），算完就搬走。从不一次性把全部书堆桌上。

魔法：数学上跟标准 attention 完全等价（不是 approximation），但避免了"把超大中间矩阵实例化到 HBM"这个最慢的操作。
书架（远） → HBM 显存（3 TB/s）  ·  桌子（近） → SRAM（19 TB/s）  ·  搬书 → tiling 分块加载  ·  小本子累积 → online softmax

6.1 为什么标准 Attention 这么慢

看一下标准 attention 的"数据搬运清单"：

1.
→
算 QKT，写到 HBM（一个 n×n 的大矩阵，n=8k 时是 256 MB）
2.
→
读 QKT，做 softmax，写回 HBM（再写 256 MB）
3.
→
读 softmax 输出 + V，乘起来，写最终结果到 HBM
GPU 上 HBM 带宽 3 TB/s，SRAM 带宽 19 TB/s。瓶颈是 HBM ↔ SRAM 的数据搬运，不是计算本身。Attention 是典型的 memory-bound 操作 — 算力没用够，时间都耗在搬数据。

6.2 FlashAttention 核心 trick · Tiling + Online Softmax

把 Q、K、V 切成小块（tile），每次只把 1 块加载到 SRAM。关键算法：

# Online softmax: 不需要看到全部 logits 就能算"等价 softmax"
m_old, l_old, o_old = -inf, 0, 0
for chunk_idx in K_tiles:                  # 一摞摞搬 K
    s = q @ K[chunk_idx].T                  # 当前摞的相关性
    m_new = max(m_old, s.max())            # 更新当前最大值
    alpha = exp(m_old - m_new)              # 旧结果的 rescale 系数
    beta  = exp(s - m_new)                  # 新结果
    l_new = alpha * l_old + beta.sum()       # 累计归一化分母
    o_new = (alpha * o_old + beta @ V[chunk_idx]) / l_new
    m_old, l_old, o_old = m_new, l_new, o_new
return o_new

关键洞察：QKT 矩阵 256 MB 在标准 Attention 必须实例化到 HBM（n=8k 时 = 8k × 8k × 4 byte），读 + 写 = 512 MB 的纯搬运开销。FlashAttention 让这个矩阵从未被实例化，所有计算在 SRAM 内 tile-by-tile 完成。这就是 2-5x 加速的来源。

6.3 FlashAttention 1/2/3 演进

FA v1 (2022)

第一次系统化 tiling + online softmax。速度 2-4x，显存从 O(n²) 降到 O(n)。把长上下文从奢侈品变成可行选项。

FA v2 (2023)

进一步 per warp 并行化 + 减少非 matmul 算子。再 +2x。Llama 2/3 训练标配。

FA v3 (2024)

H100 Tensor Core + WGMMA + warp-specialization + FP8。再 +1.5-2x。Llama 3 405B 训练标配。

面试稳过的答法："FlashAttention 数学等价标准 attention，但用 tiling + online softmax 避免实例化 n×n 矩阵到 HBM，减少 HBM↔SRAM 搬运 — 这才是 attention 真正的瓶颈。不是近似，是工程优化。"

现代 LLM 的其它关键模块

RMSNorm / SwiGLU / MoE · 让现代 LLM 比 GPT-2 强 1000 倍的"小改进"

7.1 RMSNorm · 砍一半计算的 Norm

🎬生活类比 · 期末考试归一
老师批 100 个学生作文。LayerNorm 的做法：算全班均分 + 算标准差，每个学生分数减均值再除标准差。RMSNorm：只算标准差（其实是 RMS），不减均值。

实验发现：减均值这步在深度网络里没什么用（特征自然 0-centered），省一半计算，效果不掉。 
# LayerNorm (原始)
mean = x.mean(dim=-1)
var = x.var(dim=-1)
output = (x - mean) / sqrt(var + eps) * gamma + beta

# RMSNorm (Llama / Qwen / DeepSeek 都用)
rms = sqrt((x ** 2).mean(dim=-1) + eps)
output = x / rms * gain                  # 没有减均值，没有 beta
  • 计算减少 25-30%（少一次 sum + 少一次 sub）
  • 参数减少 1 个（没有 bias）
  • 效果几乎完全持平甚至略好
  • 2026 几乎所有新模型都用 RMSNorm

7.2 SwiGLU · 给 FFN 加"门"

🎬生活类比 · 一道菜需要两个厨师
原始 FFN：1 个厨师炒菜（matmul + ReLU），出菜。
SwiGLU：2 个厨师，一个炒菜（matmul1），另一个掌握火候开关（matmul2 + Swish → gating）。出菜前要乘起来 — 火候没开的部分直接被关掉。

这就是"门控"机制 — 模型能动态决定 FFN 的哪些维度起作用，比 ReLU 更灵活，准确率 +1-2%。代价：FFN 计算和参数 ×1.5（需要 3 个矩阵）。 
# 原始 FFN (ReLU)
output = max(0, x @ W_1) @ W_2

# SwiGLU (Llama / Qwen / DeepSeek 都用)
gate = swish(x @ W_gate)              # 门控分支 (sigmoid 平滑版)
up   = x @ W_up                       # 主分支
output = (gate * up) @ W_down         # 乘起来再降维

7.3 MoE · 256 个专家选 8 个2026 新

🎬生活类比 · 大公司请专家咨询
公司业务太多元，养 1 个全能型咨询师不现实（学不深）。所以养 256 个各领域专家，每个问题来时，智能门卫（Router）看一眼问题，挑出 8 个最相关的专家，让他们各自给意见，最后融合。

结果：① 总人数（参数）很多 ② 但每个问题只动用 8/256 = 3% 的人（计算量小） ③ 因为有专业分工，整体水平比 1 个全能型高得多。

这就是 MoE — 用稀疏激活换"等同 dense 大模型"的能力。

MoE 的 4 个核心组件

① Router · 智能门卫

一个小 linear layer。输入 x，输出 N 个专家的分数，softmax + top-k 选出 k 个专家。

② Experts · 专家池

N 个独立的 FFN（典型 N=8 到 256），结构跟普通 FFN 一样，但权重独立。

③ Load Balancing Loss · 防"独宠"

防止 router 总选少数几个专家（其它 dead）。在训练 loss 里加一项"专家被选次数应该均衡"。

④ Shared Experts · 通用基座（DeepSeek）

DeepSeek-V3 创新：少数专家每次都激活（提供通用能力），其余 routed expert 提供专业能力。

面试经常追问的 trade-off：
  • 训练：所有参数都要存 + 都要训 — 需要 expert parallelism (EP) 跨 GPU 切 expert。Y03 分布式训练 展开。
  • 推理：激活少但 router 不可预测 — 对 batch 调度极不友好（不同 token 走不同 expert）。Y02 推理 展开。
  • 能力：MoE 模型在"知识密度"任务（多领域 QA）上明显胜出，在"逻辑深度"任务上跟 dense 差距小。

长上下文 · 图书馆扩建的代价

为什么 2026 模型从 4k 推到 1M · 3 大挑战 · 工程武器库
🎬生活类比 · 图书馆从 100 本扩到 100 万本
原来你的图书馆 100 本书，你每次问问题，要跟 100 本书的标签对比（n²=1 万次匹配），很快。
现在扩到 100 万本，每次匹配 100 万 × 100 万 = 1 万亿次 — 直接破产。

更糟：每本书都得放手边的"做卡片"（KV-Cache），100 万张卡片 ~~TB 级显存，没机器装得下。
再糟：你心理学也累了，第 50 万张卡片基本看不见（lost in the middle）。

3 大挑战

① Attention O(n²) 爆炸

1M 上下文 = 1T 次 attention 计算。FlashAttention + Sparse 也救不了。

② KV-Cache TB 级

Llama 3 70B 在 1M 上下文 KV-Cache ~2.6 TB。即使 GQA 也是几百 GB。

③ "中间遗忘"

模型对位置中段的 token attention 信号弱。"Lost in the Middle" 论文实证。

工程武器库

技术解决什么用在
YaRN / NTK-aware scalingRoPE 外推到长上下文Llama 3.1, Qwen2.5
FlashAttention减少 attention 显存到 O(n)所有现代模型
Sliding Window Attention只看局部窗口（牺牲全局）Mistral, Gemma
Ring Attention跨 GPU 切 sequence，环状通信训练长上下文
Sparse Attention只算重要 token pairLongformer, BigBird
KV-Cache 量化 (INT8/INT4)把 KV-Cache 显存砍 2-4xvLLM, TGI 都支持
KV-Cache 压缩 (H2O / StreamingLLM)丢掉不重要的 K/V实验阶段
RAG（外部检索）不真塞进上下文，按需取X08 RAG 深潜
2026 业界 consensus：纯长上下文（128k+）很贵，大多数业务场景 "RAG + 中等上下文" 比纯长上下文更经济。Agent 长 horizon、代码库分析这类是真长上下文场景。

Reasoning Models · 草稿纸推理2026 新范式

o1 / DeepSeek-R1 / QwQ / Qwen3-Reasoning · 推理时让模型"想得更久"
🎬生活类比 · 学霸做题的草稿纸
普通 LLM 像普通学生：看到题目，凭直觉脱口而出答案。简单题对，难题瞎答。
Reasoning LLM 像学霸：看到题目，先在草稿纸上推一长串（"先用乘法分配律展开... 再设 x=... 检查边界..."），想几分钟后才写最终答案。

神奇之处：草稿纸的长度自适应题目难度。简单题草稿很短，难题草稿写满 1 页。整体准确率显著提升，尤其是数学/代码/逻辑。

代价：response 长度变成原来的 10-100 倍，每次调用 cost + latency 都飙升。

训练范式 · DeepSeek-R1 paper 公开

3 步训练

  1. R1-Zero：纯 RL（GRPO），不需要 SFT，奖励只用"答案对/错"，模型自发学会长 thinking（涌现现象）
  2. R1：加少量高质量 SFT cold-start，再 RL，再蒸馏 — 让 thinking 更稳定 + 可读
  3. Reward 设计：rule-based（数学题答案 / code 测试通过）+ language reward（中英不混）

工程含义

  • 推理延迟成倍增加：reasoning model 输出长 10-100x，latency 直接乘倍
  • Cost 飙升：1k token thinking + 100 token answer，按 1.1k 计费
  • UI/UX 改变：用户看不到 thinking（hide）或看部分（streaming with summary），跟传统 chat 不同
  • Eval 方法重新设计：reasoning model 在数学/code/长任务上猛涨，但日常对话可能变啰嗦 — 新评测体系
  • 融合方向：Qwen3 同一个模型可切换 "thinking" / "non-thinking" 模式
高频面试Reasoning Model 跟 CoT prompting 本质区别？
CoT
是提示工程层面 — 用 "let's think step by step" 这种 prompt 引导模型一步步想，但模型本身没专门训练。每次都需要你提示。
Reasoning
是训练层面用 RL 让模型 internalize "想清楚再答"的策略。不需要 prompt 提示，自动激活，且 thinking 质量+长度自适应难度。
类比
CoT = 教学生答题时"先想再写"；Reasoning = 学生从骨子里就养成了先打草稿的习惯，不需要再提醒。

Prefill vs Decode · 工程师视角再回顾

Roofline 模型 · 推理优化的"选哪一招"决策表

11.1 Roofline 模型 · 谁卡你

🎬生活类比 · 餐厅的两个瓶颈
一家餐厅性能上限取决于两个指标的较小者：
① 厨房做菜能力（菜/小时） = GPU 算力 FLOPS
② 服务员上菜速度（盘/小时） = 显存带宽 GB/s

大宴会（prefill）：一次上 100 桌，厨房忙得团团转 — 厨房瓶颈（compute-bound）
独自就餐（decode）：每次只点 1 道菜，但每道都要服务员从仓库（KV-Cache）翻箱倒柜搬调料 — 服务员瓶颈（memory-bound）

不同瓶颈要用不同优化招式。给厨房加帮厨（FlashAttention / FP8 量化）对独自就餐没用；给服务员加助理（GQA / MLA 减少 KV）对大宴会作用很小。

11.2 优化决策表

优化对 Prefill 影响对 Decode 影响原因
FlashAttention ✅ 大幅加速 ✅ 加速 减 HBM↔SRAM 搬运
Quantization INT4/FP8 ✅ 大幅加速 ✅✅ 极大加速 Decode 时 KV-Cache 也量化
GQA / MLA ⚠️ 中性 ✅✅✅ 大幅加速 KV 小，decode 读得少
Continuous Batching ✅ 提 throughput ✅✅ 提 throughput 填满 GPU 利用率
Speculative Decoding — ✅✅ 加速 用小模型猜，大模型批量验
Prefill-Decode 分离 专属 GPU 专属 GPU 各自优化最优
面试加分答法："优化前先识别瓶颈 — prefill 是 compute-bound 看 FLOPs，decode 是 memory-bound 看带宽。客户离线 batch 任务 → prefill 主导 → throughput + 算力加速；在线 chat → decode 主导 → KV-Cache 砍 + speculative + continuous batching。"

12 高频面试题 · 含完整答案 + 加分点

折叠卡片 · 建议先想再展开 · 招聘官点名考的题
基础Q1. Attention 公式里为什么要除以 √dk？
核心
iid 假设下点积方差 = dk，量级 = √dk。如果不归一化，dk 大时 softmax 输入差距大，输出近 one-hot，梯度饱和模型学不动。除 √dk 把方差归一回 1。
加分
能讲"除 d 会让分布过平→ attention 失去选择性；除 d² 会让 softmax 完全均匀→ 等于平均池化"。
基础Q2. KV-Cache 的显存怎么计算？为什么需要 GQA / MLA？
公式
显存 = 2 × n × L × H × d_h × bytes。Llama-70B 32k 上下文 MHA 配置 ~84GB，单 H100 装不下。GQA 把 H 减到 H/g，MLA 用 latent 进一步压缩 — 都是为了让显存 fit + 提高 batch size。
基础Q3. RoPE 为什么比 Sinusoidal / Absolute 好？
核心
RoPE 编码相对位置（qm·kn 只依赖 m-n），而前两者编码绝对位置。相对位置更符合语言本质，且对未见过的长度外推更稳（配合 YaRN）。
进阶Q4. FlashAttention 为什么快？跟 Sparse Attention 区别？
FA
用 tiling + online softmax 避免实例化 n×n 矩阵到 HBM，靠减少 HBM↔SRAM 数据搬运提速 — 数学完全等价标准 attention。
Sparse
数学上不同（只算部分 token pair），是 approximation，可能掉精度。
进阶Q5. MoE 模型推理时为什么"激活参数" 远小于"总参数"？工程上的挑战？
原理
每个 token 经 router 只选 top-k expert（k 远小于 N），所以单步只激活 k/N 参数。
挑战
① 所有 expert 都要装显存 ② 不同 token 路由到不同 expert，batch 内部分 GPU 空闲，需要 expert parallelism + 调度优化 ③ Load balancing 防止 dead expert。
高频Q6. 为什么 Prefill 是 compute-bound 而 Decode 是 memory-bound？怎么优化？
Prefill
处理 n 个 token 同时，attention 是 n×n GEMM — 算力密集。
Decode
每步只算 1 token attention，但要读整个 KV-Cache（GB 级），arithmetic intensity 极低。
优化方向
Prefill: FlashAttention / 量化 / 减 FLOPs。Decode: GQA/MLA 砍 KV / 量化 KV / Continuous Batching / Speculative Decoding。
基础Q7. RMSNorm 跟 LayerNorm 区别？为什么改用？
区别
RMSNorm 不算均值，只用 RMS 归一。计算少 -25%，参数少 1 个 bias，效果实证持平。
进阶Q8. SwiGLU 比 ReLU FFN 好在哪？代价？
优势
引入 gating 分支 (Swish(xW₁) ⊙ xW₃) W₂，"选通" 能力更强，效果 +1-2%。
代价
3 个权重矩阵 vs 2 个，FFN 参数和计算 +50%（实际为保持参数量，中间维度会缩到 8/3）。
高频Q9. DeepSeek-V3 跟 Llama 3 70B 架构上几个核心差异？
5 差异
① MLA (DeepSeek) vs GQA (Llama) ② MoE 671B/37B vs Dense 70B ③ MTP 训练目标 vs next-token ④ FP8 mixed precision 训练 vs BF16 ⑤ 训练数据中推理样本占比 (DeepSeek 高)。
进阶Q10. 长上下文（128k+）的核心工程难点？业界 work-around？
难点
① Attention O(n²) ② KV-Cache TB 级 ③ "lost in the middle"。
Work-around
YaRN 外推 + GQA/MLA + KV 量化 + Sliding Window + Ring Attention + 大场景下用 RAG 替代纯长上下文。
高频Q11. Reasoning Model (o1 / R1) 跟传统 LLM + CoT prompting 本质区别？
CoT
在提示层引导，模型未经训练；每次都需要提示。
Reasoning
在训练层用 RL 让模型学会自适应长度的 thinking，不需要 prompt 提示，能根据题难度调节 thinking 长度。代价：latency 和 cost 飙升。
实战Q12. 给一个 13B 模型，估算它在单 A100 (80GB) 推理的最大 batch size（设上下文 4k, GQA H=32 g=8, FP16）
求解
模型参数 ~26GB (13B × 2 bytes)。
KV-Cache 每 batch ≈ 2 × 4k × L(~40) × (32/8) × 128 × 2 ≈ 0.33 GB/seq。
可用显存 80 - 26 - 4 (activations) = 50 GB，max batch ≈ 150。
加分
讨论 PagedAttention 能进一步提高有效 batch size（碎片化少）+ 提到 INT8 KV-Cache 量化可再翻倍。
