① 什么是 Agent

2024-2026 年 "agent" 这个词被 industry 滥用。我们用严格定义：

📐 Agent 定义

一个 Agent = LLM（推理引擎）+ Tools（执行能力）+ Memory（状态保持）+ Control Loop（自主决定下一步），能根据 goal 自主决定多步行动。

Chatbot vs Agent

维度ChatbotAgent
输入1 个 prompt1 个 goal
步骤数1 (input → output)N (动态决定)
状态无（或仅 chat history）显式 state machine
能力纯文本生成调工具 / 读写外部世界
错误恢复需用户重试自主 retry / 切方案
例ChatGPT 默认对话Cursor Agent / Claude Code / Devin

Spectrum of Autonomy

Agent 不是 binary——是 spectrum。Anthropic 提的 5 级自主度：

  1. Tool-use chatbot：人决定每步调啥工具
  2. Augmented agent：agent 自主调工具 sequence，但每步人 approve
  3. Workflow agent：agent 跑 predefined workflow，分支由 LLM 决定
  4. Open-ended agent：agent 自主决定 plan + 工具 + 终止条件（Devin / AutoGPT）
  5. Self-improving agent：agent 修改自己的 prompt / 工具集（experimental）

2026 主流生产 agent 在level 2-3。Level 4 仍 unreliable（success rate 30-60% on real-world tasks）。

③ Tool Use 设计

Tool 是 agent 跟外部世界的桥梁。Tool API design 是 agent 成败的关键。

Tool 规范（OpenAI / Anthropic 通用）

现代 LLM 用 JSON Schema 描述 tool：

{
  "name": "read_file",
  "description": "Read content of a file at given path. Returns file content as string. Errors: file not found, permission denied.",
  "parameters": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "Absolute or relative path to file"
      },
      "max_lines": {
        "type": "integer",
        "description": "Max number of lines to read (default 500)",
        "default": 500
      }
    },
    "required": ["path"]
  }
}

10 条 Tool Design Rule

  1. Description 长 + 具体—— LLM 看 description 决定何时用。3-5 句话 > 1 句话。
  2. 显式列错误模式—— "Errors: X, Y, Z"。LLM 知道 X 时怎么 retry。
  3. 返回 structured—— JSON / 严格 schema > raw text。LLM 解析准确率高。
  4. 幂等优先—— 同 input 同 output。LLM retry 不破坏状态。
  5. 限制副作用—— destructive op (delete / format) 需要 confirmation step。
  6. Output 紧凑—— 一个 tool call 返回 > 1k token 浪费 context。pagination / truncation。
  7. 错误信息 actionable—— "File not found" 不够，应该 "File X not found. Did you mean Y or Z? (suggestions from fuzzy match)"
  8. Tool 命名一致—— read_file / write_file 不要 read_file / save
  9. 少而精—— 30 个 tool 比 100 个好（LLM 决策 paralysis）
  10. 组合 over 嵌套—— list_files + read_file 比 list_and_read_files 灵活

Tool 错误处理 pattern

# Bad: 模糊错误
def read_file(path):
    try:
        return open(path).read()
    except Exception as e:
        return {"error": str(e)}

# Good: structured + actionable
def read_file(path: str, max_lines: int = 500) -> dict:
    p = Path(path)
    if not p.exists():
        return {
            "error": "file_not_found",
            "message": f"{path} does not exist",
            "suggestions": fuzzy_match_existing_files(path, top_k=3),
            "recovery": "Try one of the suggestions, or list_files() to explore."
        }
    if not p.is_file():
        return {"error": "not_a_file", "recovery": "Use list_files() for directories"}
    content = p.read_text().splitlines()
    truncated = len(content) > max_lines
    return {
        "content": "\n".join(content[:max_lines]),
        "total_lines": len(content),
        "truncated": truncated,
        "next_action": f"To read more, call again with offset" if truncated else None
    }

④ 5 种 Planning 模式

4.1 ReAct (Reason + Act)

Yao et al. 2022 提出。Agent 输出交错的 thought + action：

Thought: I need to find the file that defines `Session` class
Action: search_code(query="class Session")
Observation: Found in src/network/session.hpp:15

Thought: Now I should read it to understand the API
Action: read_file(path="src/network/session.hpp")
Observation: [class content]

Thought: I see Session has a connect() method. Let me find its impl.
Action: search_code(query="Session::connect")
Observation: ...

ReAct 把推理过程显式化（"thought"）让 LLM 不要直接跳 action。Cursor / Claude Code 等都基于 ReAct。

4.2 Chain-of-Thought (CoT)

单纯让 LLM step-by-step reasoning，没显式 tool 调用。适合纯 reasoning task。

User: 10 + 23 × 4 = ?

Thought: Let me follow order of operations.
First multiplication: 23 × 4 = 92
Then addition: 10 + 92 = 102
Answer: 102

CoT 是 ReAct 的 reasoning 部分。简单任务 CoT 够；复杂 task 用 ReAct 加 tool。

4.3 Plan-and-Execute

Agent 先整体 plan（一次 LLM call），再逐步执行（每步 LLM call）。区别于 ReAct 边想边做。

Step 1 (planning):
User: refactor module X to use std::expected

Plan:
1. Read X's public API (api.hpp)
2. Identify all throw sites
3. For each: convert throw to return std::unexpected
4. Update callers
5. Update tests
6. Run lint + tests

Step 2 (execution):
[follow plan step-by-step, allowing replanning at each step]

优势：① 早期发现 plan 不可行；② human review plan 在 execute 前；③ 复杂任务步骤多时更结构化。

劣势：plan 不准时整个 execution off-track。关键: 每步后让 agent re-plan if needed。

4.4 Tree-of-Thought (ToT)

Yao et al. 2023。把 reasoning 当 tree —— 每步生成多个候选 thought，BFS / DFS / beam search 探索：

Step 1: 3 candidate plans
  Plan A: ...
  Plan B: ...
  Plan C: ...

Evaluate each (LLM scores 1-10)

Step 2: 选 top-2 (A, B) 继续

Step 1+1: A 的 3 个 next thought
Step 1+1: B 的 3 个 next thought

Evaluate, prune, continue.

类似 chess engine 的 minimax + pruning。开销大（每节点 LLM call），适合高 stake 决策（如 medical diagnosis），不适合日常 coding。

4.5 Reflection（自我批评）

Agent 完成 task 后自评，找问题，重试。

1. Agent: 实现 function X
2. Agent: 自检：是否满足 spec? 测试 pass? edge case 处理?
3. Agent: 发现 edge case 没处理
4. Agent: 修正
5. Loop 2-4 until satisfied or max iterations

Reflection 通常 boost 任务成功率 10-30%（多个 paper 验证）。代价：2-3× LLM calls。

⑤ Memory 架构

Agent 的 memory 不是单一 concept —— 分多层：

类型时间尺度载体例
Working memory单步当前 prompt context当前 tool call result
Short-term会话内 (minutes - hours)Chat history / scratchpad之前看过的文件 / 决策
Episodicsession 间 (days - months)持久化 DB"上次跟这用户聊过 X"
Semantic / Long-termpermanentVector DB / Knowledge graph领域知识 / 文档库 (RAG)
ProceduralpermanentSystem prompt / fine-tuned model"怎么做" 的 skill

Context window 管理

LLM context window 1M-10M token 看起来大，但 effective attention 衰减。3 个 management 策略：

  1. Summarization—— 老 chat history → 1-2 段 summary
  2. Retrieval—— 只 include 跟当前任务相关的 history（RAG-style）
  3. Hierarchical—— 短期 full / 中期 summary / 远期 archive

Vector DB / RAG 选型

选项定位

FAISS (Meta)纯 library，无 server，自管
Vespa / Elasticsearch + dense vectors已用 search infra 顺便加

⑥ 控制循环模式

6.1 Single-shot

response = llm(prompt + tools)
if response.tool_calls:
    for call in response.tool_calls:
        execute(call)
return response.text

简单，但不能 retry / refine。适合非 agentic chatbot。

6.2 Iterative (ReAct-style)

messages = [system, user_goal]
for step in range(max_steps):
    response = llm(messages, tools)
    messages.append(response)
    if response.is_final:
        return response.text
    for call in response.tool_calls:
        result = execute(call)
        messages.append({role: "tool", content: result})
return "max steps reached"

主流 agent 模式。关键参数：

  • max_steps—— 防 infinite loop。Typical 10-50
  • 终止条件—— LLM 输出特殊 token / final answer / "I'm done" 标志
  • 错误处理—— tool 失败时 retry / 让 LLM 决定 fallback

6.3 Hierarchical (Supervisor + Workers)

             ┌─ Supervisor agent ─┐
             │   (高层 planning)   │
             └────┬───────┬───────┘
                  │       │
         ┌────────┴───┐ ┌─┴────────┐
         │  Worker 1  │ │ Worker 2 │
         │ (specific)  │ │ (specific)│
         └─────────────┘ └───────────┘

Supervisor 把 task 拆给 worker agents，每个 worker 用更专的 prompt + tool subset。例：

  • Cursor Composer: supervisor 决定改哪些文件，workers 改单文件
  • Devin: supervisor 协调 plan / code / debug / deploy sub-agents

3 大 pattern

什么时候用 multi-agent

  • ✅ 任务需要多种 specialized expertise（如"系统设计 → 编码 → 测试 → 部署"4 段）
  • ✅ 单 agent context window 不够（split context per agent）
  • ✅ 需要多角度 review（adversarial）
  • ❌ 任务简单时 multi-agent 反而慢 + 易 miscommunication
  • ❌ Tool 集很小时 multi-agent 没必要
2026 现实: Multi-agent 是热门 research 话题，但生产 multi-agent 系统仍少。OpenAI / Anthropic 内部多用 hierarchical supervisor（看起来 multi 但本质是 1 supervisor + worker），真正对等多 agent reliability 仍不高。

⑧ 怎么评估 Agent

Agent 评估比 chatbot 难 10×—— 多步 / 工具 / state 都要测。

3 层 evaluation

Layer测什么方法
Unit单个 tool / prompt 工作standard unit test
Trajectory多步 reasoning + tool 序列Snapshot test of trace / step-by-step assertion
End-to-end整 task 是否完成Output-based eval（LLM-as-judge 或人）

Benchmark

Benchmark覆盖
SWE-bench / SWE-bench Verified真实 GitHub issue → fix。2026 最严肃 coding agent benchmark
WebArena / VisualWebArenaWeb 操作（浏览器 agent）

AgentBench (清华)多任务综合（OS / DB / KG / Web）

LLM-as-judge

用另一个 LLM 评估 agent 输出。便宜 + scalable，但有偏（喜欢长答案 / 自己模型偏好）。

判断 prompt:
"""
Task: {original_task}
Agent's solution: {agent_output}
Ground truth: {expected_output}

Evaluate on 1-5 scale:
- Correctness
- Completeness  
- Code quality
Output JSON with rationale.
"""

Trace replay testing

记录 agent 完整执行 trace（LLM calls + tool calls + outputs），保存为 fixture。CI 时 replay + assert：

  • Final answer 跟 expected match
  • Tool calls 数 ≤ threshold（不浪费）
  • 没调禁用 tool（如 production 不能 web_search）
  • Cost < budget
  • Latency < SLO

⑨ 生产部署

4 个核心考虑

9.1 Cost

  • Agent 通常 10-100× chatbot cost（多步 LLM call + tool result 在 context 累积）
  • 典型 agent task: $0.10-$5 per task
  • 优化：① 小 task 用 small model (Haiku / GPT-4o-mini)；② Cache prompt prefix；③ Limit max_steps

9.2 Latency

  • 每个 step ≈ 1-3 seconds LLM + tool latency
  • 20 step task ≈ 30-60s end-to-end
  • 用户预期跟 chatbot 不同——agent task is "fire and forget"
  • 关键：streaming intermediate progress—— 让用户知道在做什么

9.3 Safety

风险缓解
调危险 tool (rm -rf / drop table)destructive tool 需 user confirmation + dry-run
Prompt injectionUntrusted input 隔离；不让外部 content 改 agent behavior
Infinite loop / runawayMax steps + cost budget + timeout
Data leak (sensitive in context)PII filtering before send to LLM；audit log
Unauthorized actionTool 权限模型 + scoped credentials

9.4 Observability

Agent 是distributed system on steroids—— observability essential：

  • Trace: 每个 step 完整 LLM call + tool call (LangFuse / Arize / Helicone / OTel)
  • Metrics: success rate / cost / latency / tool error rate per task type
  • Replay: 失败 task 能 reproduce + debug
  • Human review: 抽样 5-10% trace 人工 review
