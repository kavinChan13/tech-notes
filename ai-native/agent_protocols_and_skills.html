读前地图 · 一年内最火的 2 个新概念

MCP 和 Skills 究竟解决了什么

2024 年底 Anthropic 推出 MCP（Model Context Protocol），2025 同步发酵 Agent Skills 范式。这两个东西在你的 2026 简历里如果能说"我用过 + 我设计过"，是绝对的差异化加分项。本文一次性把两个讲透。

MCP vs Skills · 互补不互斥

🔌 MCP · 让 Agent 接外部世界

通用协议。让 LLM 能调用外部工具（读文件 / 查数据库 / 发邮件 / 调 API）。解决"连接"问题。

类比：USB-C — 标准化"工具接口"，让任何 agent 能用任何 tool。

📦 Skills · 让 Agent 装上领域专家

封装格式。把专家工作流 + 知识 + 工具调用打包成可加载的"skill"包。解决"能力扩展"问题。

类比：APP — 通用手机装上专业 APP，秒变专业设备。

01

为什么需要协议 · USB-C 之前的混乱

每个工具都自己定义接口的代价
🎬生活类比 · USB-C 之前的充电线噩梦
2015 年之前每个手机品牌都用自己的充电接口：苹果 Lightning / 三星 Micro-USB / 华为 mini-USB。出门要带 5 根线。

2018 年 USB-C 普及，一根线走天下。手机、笔记本、显示器、耳机全部一根 USB-C。

2024 年的 Agent 生态像 2015 年的充电线：每个 agent 框架都自己定义工具接口（LangChain Tools、LlamaIndex ToolSpec、Anthropic tool_use、OpenAI function、Cursor command...）。开发一个工具要写 5 套适配代码。

MCP = Agent 生态的 USB-C。统一协议，1 个 MCP server 可以被所有支持 MCP 的 agent 调用。
USB-C → MCP 协议  ·  手机 / 笔记本 → 各 Agent (Claude / Cursor / Cline)  ·  移动电源 / 显示器 → MCP server (工具)
MCP 不是"新功能"，是"标准化"。它能做的事 function calling 早就能做了。它的革命价值在于：1 次开发，全平台可用。这是 OSS 生态爆发的常见模式（HTTP / OAuth / OpenAPI 都是同样的逻辑）。

2.1 每一代解决了什么

Function Calling
2023 · OpenAI 首发
解决：让 LLM 输出"结构化函数调用"而不是只输出文本。

限制：每个 SDK 定义不同，OpenAI / Claude / Gemini 不通用。需要为每个模型写适配。
MCP
2024.11 · Anthropic 提出
解决：1 次写 server，所有支持 MCP 的 agent 都能用。包括 Claude Desktop / Cursor / Cline / Zed。

定位：协议层，类似 HTTP。
Agent Skills
2025 · 多家并行
解决：把专家工作流（不只是工具调用）打包成可加载单元。包含 prompt + tools + 知识。

定位：应用层，比 MCP 高一层。
3 层关系：
  • Function Calling = 单次工具调用（最底层 "API"）
  • MCP = 标准化工具协议（"网络协议"层）
  • Skills = 工作流封装（"应用"层）
三者互补，不是替代关系。生产 agent 可能同时用：MCP server 提供工具，Skills 定义工作流，Function Calling 是底层执行机制。
03

MCP 协议拆解 · 3 个核心概念

从协议设计到手写 server

3.1 MCP 为什么 4 个月就席卷了 AI 圈

🚀 加分 1 · 真正的标准化

Anthropic 不抢主导权，开源 MIT License，spec 公开，欢迎所有 agent 厂商接入。这跟 OpenAI 的"我的协议你们都用"姿态完全不同。

🔌 加分 2 · 简单到能 1 小时上手

协议本身就是 JSON-RPC 2.0 over stdio / SSE。没有奇怪的二进制协议、没有专属 SDK。Python / TypeScript SDK 是社区写的轻量包。

📦 加分 3 · 生态爆发

官方提供 reference MCP servers（filesystem / git / sqlite / brave-search 等），4 个月内社区贡献 300+ 个第三方 server。插一个 npm install 就能用。

🎯 加分 4 · 解决真痛点

每个团队都在写"agent 接公司内部工具"的脏代码。MCP 一次写完所有 agent 复用。Cursor / Claude Desktop 等大平台立刻接入提供分发渠道。

3.2 MCP 3 个核心概念

🎬生活类比 · 餐厅厨房 + 调料柜 + 菜单
MCP 把 agent 跟外部世界的交互拆成 3 类：

Tools（工具） = 厨师能用的厨具：刀、锅、烤箱。主动调用，会修改状态。例：发送邮件 / 写文件 / 调用 API。

Resources（资源） = 厨房里的食材柜：你能查"有什么"和"具体是什么"。被动读取，不修改。例：读文件 / 查数据库表 schema / 看日历。

Prompts（提示模板） = 餐厅的菜单：列出"标准菜式"+"配方"。用户选一个，agent 按模板执行。例：标准 PR review 流程 / 标准 bug 报告格式。
概念谁触发能改世界吗典型例子
ToolsAgent 自主调用✅ 能send_email / write_file / run_sql / api_call
ResourcesAgent 选择性读取❌ 不能list_files / read_file / get_schema / get_calendar
Prompts用户选择 + Agent 填充—"code_review" / "bug_report" 等模板

3.3 MCP 通信流程

# MCP server 通过 stdio 或 SSE 跟 client 通信
# 1. Client 启动 server 子进程 (stdio) 或连 SSE endpoint

# 2. Initialize 握手
client → server: { "method": "initialize", "params": { "protocolVersion": ... } }
server → client: { "result": { "serverInfo": {...}, "capabilities": {...} } }

# 3. List tools / resources / prompts
client → server: { "method": "tools/list" }
server → client: { "result": { "tools": [
  { "name": "send_email", "description": ..., "inputSchema": {...} },
  { "name": "read_calendar", ... }
] } }

# 4. Call tool
client → server: { "method": "tools/call", "params": {
  "name": "send_email", "arguments": { "to": ..., "body": ... }
} }
server → client: { "result": { "content": [...] } }

3.4 手写一个最简 MCP server（Python）

# pip install mcp
from mcp.server import Server
from mcp.types import Tool, TextContent

app = Server("weather-server")

@app.list_tools()
async def list_tools():
    return [
        Tool(
            name="get_weather",
            description="获取城市当前天气",
            inputSchema={
                "type": "object",
                "properties": { "city": { "type": "string" } },
                "required": ["city"]
            }
        )
    ]

@app.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "get_weather":
        city = arguments["city"]
        weather = await fetch_weather_api(city)
        return [TextContent(type="text", text=f"{city}: {weather}")]

# 跑起来 (stdio mode)
if __name__ == "__main__":
    import mcp.server.stdio
    import asyncio
    asyncio.run(mcp.server.stdio.stdio_server(app))

在 Claude Desktop / Cursor 配置文件加：

{
  "mcpServers": {
    "weather": {
      "command": "python",
      "args": ["/path/to/weather_server.py"]
    }
  }
}

重启 Claude/Cursor，weather tool 自动出现。你问 "北京天气怎样" 时 Agent 会调用这个 tool。就这么简单。

Tool Use · 5 种使用模式

从单工具到 multi-step planning

Agent 跟工具交互有 5 种 pattern，按复杂度递增：

① Single Call

最简单。用户问 "北京天气" → agent 调 1 次 get_weather → 给答案。

实现：1 次 LLM 调用 + 1 次 tool 调用 + 1 次 LLM 总结

② Parallel Tool Calls

用户问 "北京和上海天气对比" → agent 同时调 2 次 get_weather → 总结。

实现：LLM 一次返回多个 tool_use call，并发执行

③ Sequential (ReAct)

"查北京天气，如果下雨就帮我订酒店"。第二个 tool 依赖第一个结果。

实现：ReAct 循环 — Reason → Act → Observe → Reason ...

④ Plan-Execute

"组织部门聚餐" — agent 先列计划（查日历 / 问偏好 / 找餐厅 / 订位 / 通知），再串行执行。

实现：先调"规划 prompt" 输出 plan steps，再每步执行

⑤ Agentic Loop with Reflection

"修这个 bug" — agent 写代码 → 跑测试 → 失败 → 反思 → 改代码 → 跑测试 → 通过。失败后自动调整。

实现：循环 + 反思 prompt + 终止条件（成功 / 达到 max iterations）。Cursor agent / Devin 都是这模式。

面试常问："为什么 ReAct 和 Plan-Execute 不一样？" 答：ReAct 是 step-by-step 短期反应（动一步看一步），Plan-Execute 是先全局规划再执行。简单任务 ReAct 更灵活，复杂任务 Plan-Execute 更稳定（避免迷路）。2026 主流 agent 框架两者都支持。
05

Computer Use · GUI Agent 的诞生

Anthropic 2024.10 引爆新方向
🎬生活类比 · 给 Agent 装上"眼睛 + 鼠标"
普通 agent 通过API 调用跟世界交互（必须有 API）。问题：很多 SaaS / 内部系统没有 API，只有 GUI。

Computer Use：让 LLM 接收"屏幕截图"作为视觉输入 + 输出"鼠标坐标 / 键盘按键"作为动作。跟真人一样使用任何带 GUI 的软件。

Anthropic Claude 3.5 Sonnet 第一个支持。一推出 demo：让 Claude 自主打开浏览器 → 搜索机票 → 比价 → 填表 → 下单。整个过程像看真人操作。

5.1 Computer Use 的能力 + 限制

✅ 能做

  • 操作任何带 GUI 的软件
  • 处理无 API 的遗留系统
  • 自动化重复 GUI 任务（数据录入 / 报表）
  • 跨应用工作流（从 Excel 拷数据到网页）

❌ 当前限制

  • 慢：每步要截图 + LLM 推理 + 操作，~2-5 秒/步
  • 贵：每个动作都消耗 vision token
  • 易错：CAPTCHA / 复杂动画 / 弹窗会卡住
  • 安全风险：能干任何事，需要严格隔离（沙箱 VM）
Computer Use 是API 不可达场景的"最后兜底"。2026 共识：有 API 用 MCP，没 API 才用 Computer Use。后者贵 10-100x，速度慢 5-20x，错误率高。

Agent Skills 范式 · 把工作流变成可加载单元

Anthropic / Cursor / OpenAI 三家方案对比
🎬生活类比 · 通用手机装上专业 APP
通用 LLM 像裸机手机：能打电话能上网，但做专业事情（剪视频/做 PPT/查股票）笨拙。

Skills 范式：给 LLM 装上"APP"（=skill 包）。每个 skill 包含：
① 专家 prompt（你是 PR review 专家）
② 专属工具（read_diff / post_comment 的 MCP server）
③ 领域知识（review 标准 + 公司 coding guideline）
④ 工作流（多步执行顺序）

用户在 agent 里说"做 PR review"，agent 自动加载 review skill，秒变专家。
手机 → 通用 LLM  ·  APP → Skill 包  ·  APP 用手机 API → Skill 用 MCP tools  ·  APP Store → Skill 市场

7.1 团队工程化最佳实践

📁 1 · Skills 放 git 仓库

团队共享 .cursor/skills/ 或 anthropic-skills/。每个 skill PR 走 code review，跟代码同等对待。

🔌 2 · MCP servers 容器化

企业 MCP server（接公司内部系统）部署成 docker，团队成员一键 docker run。避免每个人本地配置。

🔐 3 · 严格权限隔离

MCP server 接生产数据库 → 只给只读 role。Agent 能 write 的 server → 走 sandbox + audit log。

📊 4 · 监控 + Eval

Skill 使用频率 / 失败率 / 用户满意度。定期下线没人用的 skill，迭代高频 skill。

7.2 反模式（要避免）

  1. 每个开发者各搞各的 skill — 没 review 没标准，质量参差
  2. Skill 过于细分 — 200 个 skill 每个只解决 1 个小问题，agent 选择困难
  3. 给 agent 直接 production write 权限 — 没 sandbox 没 audit，事故风险极大
  4. Skill 写死特定 LLM 行为 — 模型升级后失效
  5. 没有 eval 就上线 — 不知道 skill 是变好还是变差

实战 · 30 分钟手写一个 Skill

基于 Cursor + Claude · 包含 MCP server

8.1 目标 · "Daily Standup 自动生成器"

需求：用户说"生成今天的 standup"，agent 自动：

  1. 从 git log 拉今天的 commits
  2. 从 Jira 拉今天的 ticket 变更
  3. 归纳成 "Yesterday / Today / Blockers" 三段式 standup

8.2 SKILL.md

---
name: daily-standup
description: 自动从 git + Jira 拉今日工作, 生成 daily standup
tools: [git-mcp, jira-mcp]
---

# Daily Standup Generator

## 工作流

1. 调 git-mcp 的 `get_today_commits` (作者 = 当前用户)
2. 调 jira-mcp 的 `get_today_ticket_changes` (assignee = 当前用户)
3. 按以下格式输出:

```
## Yesterday
- [按 commit + ticket 归纳]

## Today
- [基于未完成 ticket + 进行中 commit 推测]

## Blockers
- [如果有 ticket status = blocked]
```

## 风格要求
- 每段 3-5 条
- 用过去时讲昨天, 现在时讲今天
- 不要复制原 commit message, 要归纳

8.3 测试

在 Cursor 里说："生成今天的 daily standup"。agent 自动识别这个 skill，按工作流执行，输出 standup。

10 道 Agent / MCP / Skills 高频面试题

折叠卡片 · 先想再展开
基础Q1. MCP 跟 Function Calling 区别？
Function Calling
各 LLM SDK 内置（OpenAI / Claude / Gemini 各家不同），需要为每个模型写适配代码。属于"SDK API"层。
MCP
跨模型的标准协议（JSON-RPC 2.0 over stdio/SSE）。1 个 MCP server 所有支持 MCP 的 agent 都能用。属于"网络协议"层。
基础Q2. MCP 的 3 个核心概念是什么？
3 概念
① Tools：agent 主动调用，会改世界（write_file / send_email）② Resources：agent 选择性读取，不改世界（list_files / read_file）③ Prompts：标准模板，用户选 agent 填（code_review / bug_report）
进阶Q3. MCP server 用 stdio 跟 SSE 怎么选？
stdio
本地子进程，简单 + 安全（无网络暴露）。大部分本地工具用这个（filesystem / git / sqlite）。
SSE
HTTP Server-Sent Events，支持远程。适合接公司内部服务、跨机器 agent。需要鉴权。
进阶Q4. ReAct 跟 Plan-Execute 区别？什么时候用哪个？
ReAct
Reason → Act → Observe → Reason ... 循环。每步根据观察决定下一步。灵活但易迷路。
Plan-Execute
先全局规划所有步骤，再按计划执行。稳定但不灵活（中途有意外难调整）。
选型
简单任务 ReAct（探索 + 反应）；复杂多步任务 Plan-Execute（避免迷路）；最高级用 Plan + Reflection 混合。
高频Q5. Agent Skills 是什么？跟 MCP 关系？
Skills
把专家工作流 + prompt + tools + 知识打包成可加载单元。让通用 LLM"装上 APP"变专家。Anthropic / Cursor / OpenAI 都有自己范式。
关系
互补不互斥。Skills 定义"做什么 + 怎么做"，MCP 提供"用什么工具"。Skills 是应用层，MCP 是协议层。
进阶Q6. Computer Use 跟 MCP 比，什么场景用 Computer Use？
规则
有 API → MCP。无 API（只有 GUI）→ Computer Use。
代价
Computer Use 比 MCP 慢 5-20x，贵 10-100x，错误率高。是兜底方案。
基础Q7. 给团队做 agent 工具化，工程化 4 步是？
4 步
① Skills 放 git 仓库走 code review ② MCP server 容器化 + docker compose 一键 up ③ 权限严格隔离（read-only / sandbox / audit log）④ Skill 使用监控 + eval 持续迭代
高频Q8. MCP 为什么 4 个月就席卷 AI 圈？
4 个原因
① 真正标准化（Anthropic 不抢主导，spec 开放）② 简单（JSON-RPC over stdio，1 小时上手）③ 生态爆发（4 个月 300+ 社区 server）④ 解决真痛点（每团队都在写脏代码接公司工具）
大平台采纳
Cursor / Claude Desktop / Cline / Zed 立刻接入提供分发，正向循环。
进阶Q9. 简历上写"我设计了团队的 Skill 库"，可能被追问什么？
追问 5 个
① 你的 skill 库有多少 skills，覆盖什么场景？② 怎么决定哪些工作流值得封装成 skill？③ Skills 怎么 review + 测试 + 监控？④ 团队 adoption 率多少 / 节省了多少时间？⑤ 你自己写过哪个 skill 是 team 最高频使用的？
加分
能讲具体数字 + 失败 case + 怎么 sunset 没人用的 skill。
实战Q10. 你的 MCP server 给 agent 操作生产数据库，怎么保证安全？
5 道防线
① 权限最小化：只给 read-only role，不给 DELETE/UPDATE
② 查询白名单：MCP server 内部限制只能跑 SELECT，禁 DDL
③ 速率限制：避免 agent 失控大量查询拖垮 DB
④ 审计日志：每条 query 记录 who/what/when，可追溯
⑤ 影响范围隔离：用 read replica 而不是 master DB
