读前地图 · Eval 是 AI 工程的隐藏护城河

这本"招聘官最爱问的话题"

面试时招聘官最爱问 "你怎么知道你的 AI 系统变好了？" 这一题。答不出来 = 立刻判 Junior。答得好不仅说明你懂 eval，还说明你是"产品 + 工程"双能力工程师。本文 1-2 小时帮你过这道关。

01

没 eval 体系 = 凭感觉调

这是 90% AI 团队的现状
🎬生活类比 · 减肥不称体重
你立志减肥，但不买体重秤。每天凭"感觉"判断："今天好像瘦了 / 今天有点胖"。3 个月后照镜子发现完全没变。

因为没有客观 metric，所有改进都是错觉。换 RAG 框架感觉变好了？换模型感觉更智能了？没数据支撑就是错觉。

更糟：没 eval 你不知道什么时候模型在悄悄退化（提示词改动 / 新数据进来 / API 升级），用户开始抱怨时已经晚了。
体重秤 → Eval set  ·  每天称 → regression test  ·  BMI 公式 → metric 定义

1.1 没 eval 的 5 个典型症状

症状 1 · 凭"感觉"做决策

"换 GPT-4 感觉更好了" / "新 prompt 用户更满意了"。没数据。问怎么知道，回 "用户没投诉"。

症状 2 · 改一个地方动一个 bug

改 prompt 解决了 case A，结果 case B 退化。因为没 regression 测试。

症状 3 · 老板问"准确率多少"答不上

老板问 "我们 AI 客服准确率多少？" 你说 "感觉还不错"。立刻显得不专业。

症状 4 · 上线后悄悄退化

3 个月前 90% 准确率，现在 70% — 不监控你不知道。等用户投诉就晚了。

症状 5 · 团队互相争吵

"我觉得 prompt A 好" vs "我觉得 B 好" — 没 eval 永远吵不完。有 eval 一跑数字结案。

Senior AI 工程师跟 Junior 最大的差别：Senior 第一个动作是建 eval set，第二个动作才是写 RAG / 训模型。Junior 直接动手，等做完才发现"我没法证明它好"。这是面试一眼能区分的信号。
02

评测金字塔 · 4 层结构

从快但粗糙到慢但精确
🎬生活类比 · 学生评估的 4 个层次
评一个学生有 4 种方法：
① 自动判分题（=单选 / 算术，秒判）
② 老师批改（=作文 / 简答，几分钟）
③ 考试 + 等级（=高考统一标准，几小时）
④ 多年职业发展（=最终用人单位的反馈）

AI eval 一模一样 4 层 — 各有用，缺一不可。
Online · 生产监控 真实用户反馈 / NPS / A/B test · 最慢但最真
Human Eval 人工 5 分制打分 / 偏好对 · 1 题几分钟
LLM-as-Judge 用 GPT-4 当裁判 · 1 题几秒
Rule-based 代码执行 / 关键字 / 正则 · 毫秒级
层速度成本覆盖范围主要用途
Rule-based毫秒~0简单CI 跑 regression / 数学题对错 / code 通过率
LLM-Judge~秒$0.001-0.01/条中等主观题大规模 eval / 上线前回归测试
Human~分钟$1-50/条主观/复杂金标准 / 模型最终 release 评估
Online~天-周—真实业务A/B 实验 / 长期监控
实战策略：4 层都要有，但比例不同。Rule-based 占 70%（快速 CI 回归）+ LLM-Judge 占 25%（主观题大规模 eval）+ Human 占 5%（关键 release 把关）+ Online 兜底（生产持续监控）。
03

Offline 评测 · CI 跑就能拦住退化

最基础也最重要的一层

所有 AI 项目第一个该建的就是 offline eval set + CI 集成。简单 1 天搞定，可以拦住 80% 的退化。

3.1 Eval Set 怎么建

  1. 从生产日志抽 100-500 题（典型用户问题 + 长尾问题混合）
  2. 人工标注标准答案 + 评分要点（耗时 1-3 天，价值最高）
  3. 分类打标签：简单事实 / 复杂推理 / 多文档综合 / 边界 case / 已修复 bug
  4. 放 git 仓库（跟代码一起 versioning）
  5. 每次 PR 自动跑 + 报告 diff 给 reviewer

3.2 Rule-based 评测的 4 种范式

① 完全匹配（Exact Match）

数学题答案 / 多选 ABCD / SQL 返回的数字。答案是 42 就是 42。简单但严格。

② 模糊匹配（Fuzzy / Substring）

关键词出现 / 正则匹配 / Levenshtein 距离。"答案包含'退货流程'就算对"。比 exact 灵活。

③ 代码执行（Code Eval）

生成的代码能不能跑通 + 通过测试。HumanEval / SWE-bench 都用这个。最客观。

④ NLP 经典 metric

BLEU / ROUGE / BERTScore — 算 N-gram 重合或 embedding 相似度。2026 已过时，被 LLM-Judge 取代。

04

LLM-as-Judge · 用 AI 当裁判

2024+ eval 主流 · 但要懂偏差
🎬生活类比 · 让特级教师当主考
人工评分太贵（请 100 个人评 1000 题，几万美元）。用 GPT-4 当裁判评分，1 题几分钱，1000 题几十块。

问题：GPT-4 有偏好（长答案 / 第一个 / 风格相近），跟人类一致率约 80%。但作为"持续监控趋势"足够 — 你只关心 "是不是变好/变差"，不在乎单题分数绝对值。

用法：① 上线前快速过 1000 题 ② 改 prompt 后看 trend ③ 不要单独决策 release（关键 release 要 human 把关）。

4.1 LLM-Judge 的 2 种范式

方式 A · Direct Scoring

给 LLM 一道 Q + 一个 A，让它打分（1-5）。简单但 noisy（同题打分波动大）。

Prompt:
Question: {Q}
Answer: {A}
Reference: {ref}

按下列标准打 1-5 分:
- Accuracy
- Completeness
- Relevance
- ...

方式 B · Pairwise Comparison（推荐）

给 LLM 一道 Q + 两个 A，让它选哪个更好。更稳定。Chatbot Arena 用这个。

Prompt:
Question: {Q}
Answer A: {A1}
Answer B: {A2}

按下列标准选哪个更好:
[A is better / B is better / Tie]

4.2 LLM-Judge 的 5 种已知偏差

① Position Bias · 位置偏差

同一对 (A, B)，把 A 放第一位 vs 第二位，LLM 倾向选第一位（5-10% 偏差）。解：每对随机两次顺序，平均结果。

② Length Bias · 长度偏差

LLM-Judge 偏好长答案（认为更全面）。但用户可能更喜欢简洁。解：prompt 明确"简洁 vs 啰嗦不影响打分"，或加 length penalty。

③ Self-Preference Bias · 自我偏好

用 GPT-4 当 judge，它倾向于给 GPT-4 生成的答案打高分（相比 Claude / Qwen）。解：用不同模型当 judge 取平均，或避免用同模型评自己。

④ Style Bias · 风格偏差

LLM-Judge 偏好结构化输出（带 bullet / 标题 / 表情）。但你的产品可能要简洁对话。解：明确 "evaluate based on content not style"。

⑤ Verbosity-Cascade Bias · 啰嗦传染

LLM-Judge 给详细 reasoning 时更倾向打高分。解：让 judge 先输出 score 再 reasoning（而不是反过来）。

🎮 交互演示 · Position Bias + Length Bias mock 实验

模拟 LLM-Judge 在"实力相当"的 2 个答案上的 100 次评分。如果 judge 无偏，胜率应该 ~50/50。点按钮看实际偏差有多大（数据基于公开 paper 实验结果）。

点上方按钮开始 mock 实验
提示：每次 100 个 trial 用 random + 已知 bias rate 模拟

4.3 LLM-Judge 最佳实践

  1. 给 judge 明确 rubric：不要只说"打分"，要列出 4-5 个维度（准确性 / 完整性 / 清晰度 / 安全性）
  2. 用 pairwise 比 direct scoring 更稳
  3. 多 judge ensemble：用 GPT-4 + Claude + Qwen 各评一遍，多数派
  4. 定期跟 human 对齐：每 1000 题抽 50 让人评，算 agreement，跌破 70% 要重新调 judge
  5. 不要单独决定 release：LLM-Judge 是 monitor，不是 gatekeeper
反模式：用同一个 GPT-4 既生成答案又当 judge。它会过度肯定自己。永远用不同模型当 judge。
05

Online 评测 · 生产环境的真实反馈

最慢但最真

5.1 4 类 Online 信号

① Implicit Feedback（隐式）

用户行为：点赞 / 复制答案 / 追问数 / 会话时长 / 跳出率。每个用户都贡献，不需要主动反馈。

② Explicit Feedback（显式）

👍 / 👎 按钮、星级评分、文字反馈。采样率低 (1-5%) 但信号强。

③ Business Metric（业务）

客服 RAG → 转人工率 / 客单转化率 / 退订率。跟产品 KPI 直接挂钩。

④ A/B Test

把流量 50/50 分给新旧模型，看上面 metric 是否显著差异。因果证明的金标准。

5.2 A/B Test 在 AI 场景的特殊性

跟传统 web A/B test 几个关键差异：

  1. 需要更长时间：AI 系统 latency 高，单次实验数据少，2-4 周才有足够样本
  2. quality vs cost trade-off 难量化：新模型贵 2x 但准确率 +3%，怎么决策？业务/财务对齐难
  3. "长尾效应"明显：90% 用户感受不到差异，10% 长尾用户感受巨大 — 平均数据骗人
  4. 反馈延迟：用户当下没投诉不代表满意，可能下周就流失
06

主流 Benchmark · 解读 + 局限

为什么 "MMLU 90 分" 不代表"你的产品好"

每次新模型发布，必然炫耀 MMLU / HumanEval / GSM8K / Chatbot Arena 各种分数。但作为工程师你要懂：这些分数跟你产品场景的相关性可能很弱。

MMLU
2020 · 57 学科多选题
看模型的百科知识广度。覆盖 STEM / 法律 / 医学 / 历史 / 文学。
① 已严重污染（训练数据里有 MMLU 题）② 选择题不反映生成能力 ③ 跟实际业务 task 相关性弱
HumanEval
2021 · 164 道 Python 编程题
看代码生成基础能力。能跑通 + 通过测试。
① 题太简单（10 行解决）② 跟真实工程差距大 ③ 严重污染
GSM8K / MATH
2021/2022 · 数学题
看数学推理能力。GSM8K 小学奥数 / MATH 高中竞赛。
已饱和（顶级模型 95%+），区分度低
MT-Bench
2023 · 多轮对话
看多轮对话质量。80 个 prompt 涵盖 8 类。
① 用 GPT-4 当 judge 有偏差 ② 题量少（80 题）方差大
Chatbot Arena
2023+ · 真实用户对战
看真实人类偏好。两个模型答同题，用户盲选。Elo 评分。
① 用户偏好 ≠ 业务价值 ② 短答案 / 风格化答案占便宜 ③ 高端用户少
SWE-bench
2024 · 真实 GitHub bug fix
看 real-world coding agent 能力。给 issue 和 repo，能不能 fix。
还很难（顶级 ~70%），但已经是2026 最接近真实场景的 benchmark
AIME / ARC-AGI
2024+ · 推理模型主战场
看 reasoning 能力上限。AIME 数学竞赛 / ARC 抽象推理。
普通模型分数极低，o1 / R1 / DeepSeek-R1 时代才有意义
面试常问："为什么 MMLU 90 分的模型在你的业务上效果差？" 答：① 你的业务任务跟 MMLU 题型不匹配（业务可能是多轮对话 / agent / 工具调用） ② benchmark 已污染 ③ 业务有特定数据 distribution（如方言 / 行话） ④ 业务 metric 不只准确率（latency / 安全性）。这就是为什么必须建自己的 eval set。
07

业务 Metric 设计 · 不只看准确率

把"AI 好"翻译成业务能算账的指标
🎬生活类比 · 评价餐厅不只看菜好吃
一家餐厅，评 "好不好" 不只是"菜好吃"。要看：
① 菜好吃（=accuracy）
② 上菜快（=latency）
③ 价格合理（=cost）
④ 不吃坏肚子（=safety）
⑤ 让人想再来（=retention）

AI 产品一样 — 5 维度都要测。只优化 accuracy 经常翻车（用户满意度反而下降）。

7.1 5 维度业务 metric

Accuracy
质量
答得对 / 准确率 / faithfulness
Latency
速度
P50 / P95 / P99 / TTFT (first token)
Cost
成本
$ / query, $ / DAU, token usage
Safety
安全
幻觉率 / 越狱率 / PII 泄露

7.2 Goodhart's Law · 别变成度量优化的奴隶

Goodhart's Law："When a measure becomes a target, it ceases to be a good measure."

把"准确率"作为唯一 KPI → LLM 学会"答得长 / 含糊"来骗高分。把"用户满意度"作为唯一 KPI → AI 学会过度恭维客户。

所以必须 multi-objective + 持续校准。每季度审视 metric 跟业务真实价值还匹配吗。

Eval Pipeline 工程化

从一次性脚本到持续运行的系统

8.1 成熟 Eval Pipeline 5 个组件

  1. Eval Set 管理：版本化 / 分类标签 / 持续扩充

5 个评测翻车真实案例

已脱敏 · 学他人血泪

事故 1 · MMLU 92 分模型上线后效果差

某团队选了 MMLU 92 分的模型上线，结果实际业务（客服 RAG）准确率只有 47%。

根因：MMLU 是百科知识题，跟客服领域知识无关。必须用自己业务 eval set 测。

事故 2 · LLM-Judge 给 GPT-4 自己打高分

用 GPT-4 评 (GPT-4 vs Claude 3.5) 谁好，GPT-4 系统性偏向自己 5-15%。决策错误，选了贵的。

根因：Self-preference bias。修复：用 Qwen / Gemini 当 judge 替代。

事故 3 · Online metric 都涨但用户流失

A/B test 显示新模型 accuracy +5% / 答案更详尽，决策上线。1 个月后用户流失 12%。

根因：新模型答案啰嗦（800 字 vs 200 字），用户烦。没监控简洁度 / 阅读时间。

事故 4 · Eval set 过拟合

团队 6 个月一直用同一个 200 题 eval set。最后模型 99% 通过率，但生产准确率只有 73%。

根因：迭代过程中 prompt + RAG 配置不知不觉过拟合到了 eval set。必须定期扩充 eval set + 留 holdout。

事故 5 · 没有 regression test，新功能弄坏旧功能

PR 加了 "VIP 客户优先" 功能，结果 normal 客户的回答质量崩了。3 天后用户投诉发现。

根因：没 CI eval。修复：每个 PR 必须跑 eval set，metric 退化 > 2% 阻止 merge。

10 道 AI Eval 高频面试题

这一章是面试当场提分最快的
高频Q1. 你怎么知道你的 AI 系统变好了？
3 层答案
① Offline：自建 100-500 题 eval set + Rule-based + LLM-Judge 持续 CI ② Pre-release：human eval 关键 release 把关 ③ Online：A/B test + 业务 metric 跟踪。缺一不可。
加分
能讲具体数字（"准确率从 60% → 89%"）+ 提到 multi-objective（accuracy / latency / cost / safety / retention 都要看）。
基础Q2. 评测金字塔 4 层是？
4 层
① Rule-based（最快最便宜 / CI 用） ② LLM-Judge（快 / 主观题大规模） ③ Human eval（最贵最准 / 关键 release） ④ Online（生产真实反馈）。速度 / 成本 / 真实性 trade-off。
进阶Q3. LLM-as-Judge 的 5 种偏差？怎么 mitigate？
5 偏差
Position / Length / Self-Preference / Style / Verbosity-Cascade
Mitigate
① 随机两次顺序平均（fix position） ② prompt 明确"长度不影响打分"+ length penalty（fix length） ③ 用别家模型当 judge（fix self-pref） ④ rubric 明确"看内容不看风格" ⑤ score 在 reasoning 前输出（fix verbosity-cascade）
高频Q4. 为什么 MMLU 高分模型在你的业务上效果差？
4 原因
① 业务任务跟 MMLU 题型不匹配 ② Benchmark 已污染（训练数据里有题） ③ 业务有 specific data distribution（方言 / 行话 / 领域知识） ④ 业务 metric 不只准确率（latency / cost / safety）。所以必须自建 eval set。
进阶Q5. Goodhart's Law 在 AI eval 里怎么体现？怎么避免？
体现
"准确率"作为唯一 KPI → LLM 学会"答得长 / 含糊"骗高分。"用户满意度"作为 KPI → AI 过度恭维。
避免
① multi-objective metric（不靠一个） ② 持续校准 metric 跟业务价值的关联 ③ 定期 audit eval set，看 metric 是否被 hack
基础Q6. RAGAS 的 4 个核心 metric 是？怎么算 Faithfulness？
4 metric
Context Precision / Context Recall / Faithfulness / Answer Relevance
Faithfulness
"答案里每个事实是否能在 context 中找到支撑"。实现：用 LLM 把答案拆成 atomic statements，每个判断是否被 context 支持，支持数 / 总数 = Faithfulness。反映"反幻觉"程度。
进阶Q7. Pairwise vs Direct Scoring 哪个更好？
Pairwise (推荐)
"哪个更好"比"打 1-5 分"更稳。人类擅长比较，不擅长绝对评分。LLM 也是。Chatbot Arena 用这个。
Direct 局限
同题不同次打分波动 ±1 分常见；没有 reference 时绝对分数意义弱。
高频Q8. 给一个 A/B test 新模型 accuracy +5% 但用户流失 +12%，怎么排查？
排查 5 步
① 检查实际答案：长度 / 风格 / 结构变了？
② 检查 latency：accuracy 涨了但变慢了？
③ 检查具体 case：抽 100 个用户流失的会话看到底什么变了
④ 检查 metric 定义："accuracy" 跟用户真正关心的一致吗？
⑤ 分群分析：哪类用户流失最多（VIP / 新用户）？
进阶Q9. Eval set 怎么避免过拟合？
4 步
① 留 holdout：80% 用于迭代 / 20% 留作 final eval（不要每次都看） ② 定期扩充：每月从生产新增 50-100 题 ③ 多样性：覆盖简单 / 复杂 / 长尾 / 边界 ④ 定期换：6 个月做一次大改
实战Q10. 从 0 给一个 RAG 产品建 eval 体系，1 周时间，怎么做？
5 天计划
Day 1：从生产日志抽 100 个真实问题，分类（简单事实 / 复杂推理 / 边界 case）
Day 2：人工标注标准答案 + 关键评分要点（最累但最有价值）
Day 3：用 RAGAS 实现 4 metric 自动评测 + 写 runner 脚本
Day 4：跑 baseline 数据 + 集成到 CI（PR 必跑）
Day 5：建 simple dashboard（Streamlit / metabase） + 设告警（关键 metric 退化 > 5% 通知）
加分
提到"先 100 题 baseline，看 1 个月数据再决定是否 scale up"，不上来就追求 1000 题。
