01

业务场景 · 电信网元 OAM 工单分析

这是个行业通用问题 — 任何做电信网元 / 基站设备的厂商都遇到

1.1 一句话讲清 OAM 工单分析

🎬生活类比 · 物业管理员的工单
想象你是一个大型小区的物业管理员。每天收到几十张用户报修工单：电梯坏了、车库进水了、消防告警响了。

每张工单都要：① 看现场记录（监控/日志） ② 查跟过去类似工单（"上次电梯坏成这样是什么原因") ③ 看相关规章（消防规范怎么处理） ④ 定位是哪个工程队的事（电气 / 给排水 / 弱电）⑤ 给出初步处理建议。

这件事每张工单平均花 ~3 小时。一个小区每天 4-8 张工单，2 个物业全职做这事都做不完。

电信网元 OAM 工单完全一样的模式 — 只是工单不是"电梯坏了"，是"某基站某射频单元失联"，规章不是"消防规范"，是"协议规范 + 产品规范"。

1.2 行业普遍问题

电信厂商（基站设备 / 核心网设备 / 传输设备等）都有 OAM 团队（Operations / Administration / Maintenance），其日常工作很大比例是"处理客户报障工单"。每张工单典型包含：

  • 客户描述的症状（"某机房 X 个站点失联"）
  • 触发的 alarm/fault ID 列表（5-6 位整数）
  • 日志归档（从现场设备抓的状态/协议 trace/系统日志，几十 MB ZIP）
  • 客户提供的上下文（机房 / 时间 / 涉及版本）

1.3 3 个 persona（典型用户）

👨‍💻
OAM 工程师
每天 3-5 工单排队。每张平均 3 小时定位（看日志 + 翻规范 + 找历史工单 + 写 root cause）。重复劳动多。
能不能"扫一眼"AI 给的初步分析，15 分钟内决定是不是我的域，是的话能不能给"3 个最可能 root cause"。
👔
Tech Lead
团队工单积压。senior 时间被简单工单占住做不了架构改进。新人 onboarding 慢。
让 simple 工单自动化分析 + handoff。让 senior 只看 deep analysis。新人有"AI 教练"参考。
🎯
邻域接收方
经常收到"路由错"的工单 — 客户说"是 A 域问题"，实际是 B 域。互相踢皮球浪费时间。
明确告诉我"这真不是 A 域问题，是 B 域，建议 handoff 到 X 团队"。

1.4 为什么这类问题适合 AI

✅ 适合 AI 的特征

  • 大量历史工单（千级）— 天然 RAG 训练语料
  • 结构化 + 非结构化混合 — fault ID 是结构化、日志/描述是自然语言
  • 有正确答案（已修复工单有 root cause + fix）— 可建 eval set
  • 专家知识可形式化（规范 + 协议 + 域规则）
  • 重复模式高（80% 工单落在已知 ~50 种 root cause 模板）

⚠ AI 不能 100% 解决的

  • 长尾（新硬件 / 新协议版本）AI 没见过
  • 需要现场配合的（如要打流测试）
  • 跨域复杂故障（4-5 个组件交互失败）
  • 极端 incident（critical sev 要 senior 手动）

→ 这就是为什么是 "AI 辅助" 而非 "AI 替代"。系统 80% 自动 + 20% 升级给人。

商业价值估算示意：假设团队 8 工程师 × 每天 4 工单 × 3 hr/工单 = 96 工时/天。AI 把"首次定位"从 3hr → 15min，每天省 ~30 工时。等价于多招 4 个工程师的输出。投入：1-2 人月开发 + ~$2k/月 LLM API。ROI 几周回本。
02

需求 + ATAM · 质量属性 trade-off

用架构师方法论框定"做成什么样"

2.1 用户故事

  1. P0 · 工程师：作为 OAM 工程师，我把工单 ID 给 AI，15 分钟内拿到 ① 是不是我的域 ② 如果是，3-5 个可能 root cause ③ 每个的证据链 ④ 建议 fix 位置
  2. P0 · 路由：如果 AI 判断不是本域，给出 handoff 建议 + 判断依据
  3. P1 · 批量：作为 Tech Lead，每晚跑前一天所有工单，第二天看汇总
  4. P1 · 可解释：每个结论必须有 artifact 引用（哪条日志 / 哪个 ID / 哪条规范），不能黑盒
  5. P2 · 学习：工程师 review 后能反馈 "这个 hypothesis 不对，正确是 X"，系统下次学

2.2 ATAM 质量属性矩阵

2.3 关键 trade-off

⚖️ 准确率 vs Latency

每多一个 evidence 来源 / 每多一轮 cross-validation → 准确率 +5% 但 latency +30s。选择："hypothesis-driven" 范式 — 只 deep-analyze 高 confidence hypothesis，不全量扫所有 evidence source。

⚖️ Cost vs 可解释

让 LLM "写出完整证据链" 会用 2-3x token。选择：分阶段 — fault_analyzer 快速分类（少 token），deep_analyzer 才写完整证据链（多 token，但只对 confirmed hypothesis）。

03

5 个关键 ADR · 第一人称决策记录

为什么这么设计 — 写给未来 6 个月后忘记的自己

下面是这类系统设计中 5 个最关键的架构决策。每个 ADR 用"问题 / 选项 / 决策 / 后果"四段结构。这些决策是 AI 系统设计面试最爱问的。

ADR-1
为什么把 11 个 subagent 砍到 6 个

问题

v1 设计时按"理论最优" — 一个 subagent 对应一个责任，共 11 个：工单元数据抽取 / 日志解析 / fault 抽取 / 域分类 / 证据归一 / fault catalog 查询 / 历史检索 / hypothesis 生成 / 设备状态分析 / 协议分析 / 规范分析。运行后发现 latency 是预期的 2.5 倍。

根因（来自 trace）

  • 每个 subagent 调用都有 ~3-5s 启动开销（context 加载 + LLM 推理）
  • 每个 subagent 之间 handoff 要 LLM 总结成 schema，这步本身花 1-2s
  • 很多 subagent 处理的 evidence 高度重叠
  • 11 个 subagent 累积启动开销 ~40s，相当于整个分析 30% 时间花在"切换"上

选项

  • A：保留 11 个 + cache 复用 evidence → 复杂度炸
  • B：合并到 6 个，按"phase 而非 source"分组（normalize / hypothesize / verify / conclude）
  • C：变成 1 个 monolithic agent → 失去 checkpoint 能力
选 B。合并后 6 个：explorer / log_parser / fault_analyzer / historical_matcher / deep_analyzer / summary。fault_analyzer 吸收了 fault_extract + domain_classifier + normalizer + catalog_lookup（4 合 1）。deep_analyzer 吸收了 device/protocol/spec/code 多个 analyzer（统一为 "hypothesis-driven 验证"）。

后果

  • ✅ Latency 降 ~40%（11 → 6 subagent 启动开销少 5 次）
  • ✅ Handoff schema 复杂度大幅降低（少 5 个中间 schema）
  • ✅ 每个 subagent 职责更"phase-aligned"
  • ⚠️ 单个 subagent prompt 变长（fault_analyzer 从 ~1500 → ~3500 token），需要更好的 prompt engineering
  • ⚠️ 调试某一个 sub-phase 难度增加
ADR-2
为什么用 hypothesis-driven 而非 fixed pipeline

问题

v1 deep_analyzer 用固定 5-phase 流水线：扫设备状态 → 扫协议 trace → 扫规范 → 扫代码 → 综合。每个工单都跑完整 5 phase。问题：

  • 大部分工单用不上 5 phase 全部（如纯 alarm 触发的工单不需要扫代码）
  • "先扫一遍再下判断"违背专家工程师的真实思维方式（专家是"先有假设再针对性验证"）
  • token 浪费严重

选项

  • A · 智能路由 fixed pipeline：按工单类型 dispatch → 维护成本高
  • B · Hypothesis-driven：先生成 3-5 个 hypothesis，按需 deep-analyze → 灵活但难调
  • C · 全 LLM agent loop：让 LLM 自己决定下一步 → 失控风险高
选 B。historical_matcher 同时担任 "search + hypothesis generator" 双责。deep_analyzer 不再跑固定 5 phase，而是"载入 hypothesis → 按 verification_plan 收集证据 → cross-validate"。

Hypothesis 工作流（核心）

1
Normalize（fault_analyzer）：把所有 evidence 整理成结构化 fault_context（symptom_signature / temporal_sequence / involved_modules / protocol_observations / anomalies）
2
Hypothesize（historical_matcher）：从 3 个 source 生成 3-5 个 hypothesis — ① 历史 RAG 匹配 ② fault_context 推理 ③ 域知识规则
3
Verify（deep_analyzer）：对每个 hypothesis 按其 verification_plan 收集证据（设备状态 / 协议 / 规范 / 代码 中选）
4
Cross-validate：每个 hypothesis 算分 = initial_confidence × 0.2 + verification_score × 0.6 + anomaly_coverage × 0.2。≥ 0.7 + 无矛盾证据 → 确认
5
Conclude（summary）：报告全部 hypothesis（包括被否定的）+ 证据链 + fix 建议

后果

  • ✅ 平均 latency 再降 ~25%（不再扫不必要的 source）
  • ✅ Token cost 降 ~30-40%
  • ✅ 准确率反而提升（targeted verification 比"全扫"质量更高）
  • ✅ 可解释性飞跃：报告能看到"曾考虑 hypothesis X，但被证据 Y 否定"——工程师信任度提升
  • ⚠️ Hypothesis 质量决定一切。如果生成阶段漏了真因 hypothesis，整个方向就错
ADR-3
为什么 4 层 RAG（不是 1 个 dense 搞定）

问题

v1 想"简单点" — 把所有 evidence source（历史工单 + 产品规范 + 接口规范 + 协议文档 + 代码）都嵌入一个 vector DB，dense search 全打。结果：

  • 历史工单搜不准 — 业务术语 + 模块名 + fault ID 这些 "精确 token" dense embedding 区分不出（"FaultXX01" vs "FaultXX10" embedding 极近）
  • 规范文档失精 — 规范有强结构字段（component / managed_object / fault_id），dense 忽略
  • 代码完全错 — 代码语义不适合 text embedding，"找定义 / 找引用" 必须 AST / ripgrep
按 source 特性4 层独立 RAG，每层用最适合的检索技术。不要追求"统一架构"，承认不同 source 需要不同 retrieval。

4 层架构（详细 §5 拆解）

L1历史
历史工单语义检索"找跟当前工单症状相似的过去工单" — 给 hypothesis 生成提供模式
PostgreSQL + pgvector
text-embedding-3-small (1536d)
ivfflat 索引 + GIN 元数据 filter
L2规范
产品规范全文搜"找跟某 fault / 某 component 相关的规范段落" — 给 hypothesis 验证提供"应该是怎样"基线
Tantivy（Rust 全文索引）
结构化字段 + BM25
由独立 indexer 工具构建
L3协议
协议文档检索"行业标准协议规范中找具体 message / state machine 描述"
Tantivy（独立 index）
BM25 + 结构 (chapter / section)
L4代码
多域代码搜索"找具体函数定义 / 引用" — 给 fix 建议定位代码位置
ripgrep + ctags
按域分仓
精确字符串 / regex / 符号

后果

  • ✅ 历史工单召回率从 ~50% → ~87%（加 GIN 精确 filter）
  • ✅ 规范段落召回率从 ~40% → ~80%（按结构字段 boost + BM25）
  • ✅ 代码定位准确（ripgrep + ctags 接近 100%）
  • ⚠️ 维护 4 套独立 retrieval 系统，初期工作量大
  • ⚠️ Indexer pipeline 需要单独运维
ADR-4
Agent SDK 选型 · 商业 SDK 还是自研

问题 + 选项

评估了 5 个 agent SDK 选项：

  • A · LangChain / LangGraph：生态最广。问题：抽象层多、performance 损失、breaking change 频繁
  • B · 自研 agent runner：完全可控。问题：6 个月开发投入
  • C · GitHub Copilot SDK：GitHub 官方，subagent / task / handoff API 模型合适
  • D · OpenAI Assistants API：简单但锁 OpenAI
  • E · LlamaIndex AgentRunner：偏 RAG 中立但 agent 抽象不够
选 C · GitHub Copilot SDK。3 个原因：① 跟企业 GitHub 集成好 ② subagent + task + read_agent API 模型恰好匹配 hypothesis-driven 工作流 ③ Handlebars 模板风格的 prompt 文件友好（vs YAML / Python decorator）。

后果

  • ✅ 开发周期短（POC 2 周内跑通端到端）
  • ✅ 跟企业 SSO / 认证 / 日志合规一站式集成
  • ✅ Subagent + handoff 模式直接对应 hypothesis-driven 范式
  • ⚠️ Vendor lock-in 风险（虽然 prompt 模板是 portable）
  • ⚠️ 高级特性（如 LangGraph 的可视化）缺失，自己加
ADR-5
为什么 TypeScript 而非 Python

问题

"AI 项目 = Python" 是默认认知。但此项目选了 TypeScript + Node.js + ESM。原因：

  • Agent SDK 是 TS 一等公民（Python binding 滞后）
  • 团队既有 OAM 工具链很多是 TS / Node（vscode extensions / web UI）复用人才
  • 大量 tool 是 "API 编排" 类（HTTP API / 数据库 / 文件） — TS async/await + 强类型适合
  • 大模型推理本身不在我们代码里跑，是调外部 API — Python 数值优势用不上
全部 TS + ESM + strict mode。嵌入 / 索引等"非 ML 核心"工作完全够用。需要 ML 重计算的部分（如规范 index 构建用 Tantivy）用 Rust 独立工具。

后果

  • ✅ 强类型让 schema 跟实际数据流一致（IDE 报错代替运行时 bug）
  • ✅ async/await 处理多 tool 并行调用清晰
  • ✅ 团队既有人才直接上手
  • ⚠️ 万一未来想加"本地小模型推理"或"自定义 embedding"会很别扭
  • ⚠️ AI 圈学习资源 80% 是 Python，团队成员要做"反向翻译"

C4 系统视图

Context → Container → Component 三层

4.1 Context 层（系统跟谁打交道）

系统外部

  • OAM 工程师（user · 输入工单 ID / 接收 Final_Analysis.md）
  • 工单系统（外部 DB · 提供工单元数据）
  • 日志归档（外部存储 · 日志 ZIP）
  • Fault Catalog 服务（外部 API · fault 定义查询）
  • 代码 review 系统（外部 · 历史 fix commits）
  • LLM 服务（外部 · embedding + 推理 API）
  • 规范文档 git 仓库（外部 · 通过 indexer 工具构建索引）
  • 历史工单知识库（PostgreSQL + pgvector · 历史已修复工单沉淀）

4.2 Container 层（系统内部）

  1. Agent Runner（TS / Agent SDK）— 编排 6 subagent，管理 checkpoint，处理 handoff
  2. 6 个 Subagents（prompt 模板 + 工具组合）— 见 §6
  3. Tools 层（10 个 TS 函数）— 见 §7
  4. RAG 4 层（pgvector + 3 个 Tantivy 进程 + ripgrep）— 见 §5
  5. Memory 3 层（session_state / pattern_store / checkpoint）— 见 §8
  6. Spec Indexer（独立 Rust 工具，离线运行，每周一次）
  7. Historical Indexer（TS pipeline，每天增量从工单 + commit 系统抽取已修复工单，embedding 后写入 pgvector）

RAG 4 层深潜 · 每层的设计决策

联动 X08 RAG 深潜 + X11 Embedding & Vector Search

5.1 L1 · 历史工单语义检索（pgvector）

表结构（示意简化版）

CREATE TABLE historical_tickets (
    id SERIAL PRIMARY KEY,
    ticket_id VARCHAR(20) UNIQUE NOT NULL,
    title TEXT,
    description TEXT,
    root_cause_summary TEXT,
    fix_domain VARCHAR(50),
    fix_description TEXT,
    fix_commits TEXT[],
    fault_ids INTEGER[],            -- 精确 filter 用
    protocols TEXT[],
    affected_domains TEXT[],
    severity VARCHAR(20),
    description_embedding vector(1536),    -- 4 个独立 embedding
    root_cause_embedding vector(1536),
    fix_description_embedding vector(1536),
    combined_embedding vector(1536),       -- 主查询用
    created_at TIMESTAMP,
    fixed_at TIMESTAMP
);

关键决策

① 4 个 embedding 字段而非 1 个

不同 hypothesis 生成场景需要不同侧重：找"相似症状"用 description embedding；找"相似 root cause"用 root_cause embedding。combined_embedding 是默认。

② ivfflat 索引 + GIN 双重

combined_embedding 用 ivfflat 做 ANN（lists=100）。fault_ids / protocols / affected_domains 各建 GIN 索引 做精确 array 包含 filter。不是只用 dense。

③ 召回 4x topK 再 rerank

ANN 先召回 topK × 4，再用 rerank 二次评分（semantic + 0.1×fault_overlap + 0.05×domain_overlap + 0.05×protocol_overlap），取 topK。这是 X08 §6 Rerank 思想的落地，但用规则评分代替 Cross-Encoder（避免额外 LLM 调用）。

④ applicability 三档

最终给 hypothesis_generator 的不是裸 similarity，而是 high/medium/low 三档（≥0.85 / ≥0.65 / 其它）+ 文字理由。让 LLM 不被假高分误导。

SQL 查询（关键 trick）

SELECT id, ticket_id, title, root_cause_summary, fix_description, fix_commits,
       fault_ids, protocols, affected_domains,
       1 - (combined_embedding <=> $1::vector) AS similarity
FROM historical_tickets
WHERE ($2::int[] IS NULL OR fault_ids && $2)       -- GIN array overlap
  AND ($3::text[] IS NULL OR affected_domains && $3)
  AND ($4::text[] IS NULL OR protocols && $4)
ORDER BY combined_embedding <=> $1::vector   -- 余弦距离
LIMIT $5;

关键：&& 是 PostgreSQL array overlap 操作符，跟 GIN 索引天然适配，O(log n) 而不是全表扫。

5.2 L2 · 产品规范全文搜（Tantivy）

产品规范文档（功能规格 / 接口规格）一般写在 RST / Markdown 文件里，存在 git 仓库。由独立 Rust 工具构建 Tantivy 索引。

Indexer 设计要点

  • git-native：直接读 git branch 不需要 working tree（用 libgit2）
  • Rayon 并行：所有规范文件多核处理
  • 结构化抽取：每个文件抽 12 个字段（spec_type / component / fault_ids / managed_objects / participants / ...）
  • Tantivy BM25：text 字段 + 结构字段联合查询

检索 query 模式

  • "找 fault X 相关 spec" → fault_ids:FaultXXXXX
  • "找 component A 的 alarm 处理" → component:A AND description:alarm
  • "找 ModuleX_L managed object" → managed_objects:ModuleX_L

5.3 L3 · 协议文档检索（独立 Tantivy index）

行业标准协议文档（如 fronthaul 接口协议、配置协议规范）建独立 index 是因为：

  • 查询模式不同（产品规范按 component / fault_ids，协议按 message / state）
  • 需要保留章节 / 段落结构
  • 独立维护方便更新（协议版本变了不影响产品规范 index）

5.4 L4 · 代码搜索（ripgrep + ctags）

代码 RAG 故意不用 embedding。原因：

Hypothesis verification 里"找代码"的 query 都是精确符号（函数名 / 变量名 / 类名）— 跟 dense embedding 完全不匹配。"找 handleFaultXXX 函数" 这种 query，ripgrep 1ms 精确命中，embedding 0.7 相似度都给不出。所以 L4 反主流地用传统工具。

按域分仓：不同子域有自己的 code repo path，code_search tool 接收 domain 参数路由到对应 path 下 grep。

5.5 4 层 RAG 的调用模式

Subagent主用层偶尔用
historical_matcherL1 (历史工单)—
deep_analyzer (verify)L2 (spec), L3 (protocol), L4 (code)L1 (回查相似 fix)
fault_analyzerFault Catalog API（非 RAG）L2 (查 fault 定义)

  • matches[] — 召回的相似历史工单 + applicability 评级

  • recommended_verification_order

责任：对每个 hypothesis 按其 verification_plan 收集证据并 cross-validate
Tools：device_state_parser / protocol_search / spec_search / code_search / commit_search
不跑固定 5-phase：只查 hypothesis 真正需要的证据

责任：拼接所有 artifact 出最终报告。
关键：必须列出所有 hypothesis（包括被否定的 + 否定理由）。这是工程师建立对系统信任的关键 — "我能看到 AI 考虑过哪些可能性，不是黑盒猜"

6.5 Handoff Schema 设计原则

✅ Do

  • handoff 只传 artifact path + 几个关键 metadata + 1 行 rationale
  • 所有大 JSON 落地到 reports/，handoff 引用而非复制
  • 每个 subagent 必须先 verify 上游完成才能消费 artifact
  • compact JSON > 长 markdown

❌ Don't

  • 把大 JSON 整段贴进下游 prompt（context 爆炸）
  • handoff 5 句话以上（"过于详细" = 没真的归纳）
  • 跳过 checkpoint 直接看下一步（中断恢复会乱）
  • 同一个 evidence 在多个 subagent 重复抽（用 cache 复用）

7.2 Tool 设计 4 个原则

  1. 每 tool 只做一件事 — 不要"瑞士军刀"。例：spec_search 只做产品规范检索，不兼任 protocol_search
  2. 输入 / 输出 schema 严格 — TS strict mode + explicit interface。LLM 调用错参数就立刻 fail（vs 让 LLM 解释失败）
  3. 幂等 + cacheable — 同样输入永远同样输出。给 checkpoint 恢复带来巨大便利
  4. 降级方案 — Tantivy 挂了 spec_search 用 CSV fallback；API 超时返回 cached 版本（带过期警告）

7.3 跟 MCP 范式的关系

本案例的 10 tools 是项目内 TS 函数，不是 MCP server。原因：项目开发时 MCP 还不流行，且这些 tool 都是项目专属。

如果今天重做：会把"通用 tool"（commit_search / spec_search）改写成 MCP server，跟其它 agent 复用；项目专属 tool（如 device_state_parser）仍保持内置。这就是 X09 §7 讲的 "Skills × MCP 组合工程化" 的真实演化路径。

3 层 Memory + 持续学习

L1 · Working Memory

session 内的 artifact 和 handoff context。文件落地 + Map in-memory。
生命周期：单工单处理

L2 · Historical Memory

pattern_store — 持久化的 "症状 → root cause" 模式。从已修复工单累积。
生命周期：永久

L3 · Domain Knowledge Cache

spec_search / protocol_search 的结果缓存（去重）。
生命周期：半持久（按版本失效）

8.1 Pattern Store 学习机制

  1. 工程师在 Final_Analysis.md 上 review 后给 feedback（"hypothesis 2 是真因" / "都不对，真因 X"）
  2. pattern_store 把这条 feedback 存为新 pattern
  3. 下次 historical_matcher 生成 hypothesis 时多一个 source: "pattern_store 匹配"
  4. 慢慢积累成"团队领域知识"，初期靠 historical 检索，后期 pattern 命中率高于历史检索

这是简化版 online learning，不重训模型，只丰富 retrieval pool。

Eval 体系 · 怎么知道 Agent 变好

联动 X12 AI Eval

9.1 评测金字塔（X12 思想落地）

  • L1 Rule-based 回测：用 100 个已修复工单做 eval set。预测 root cause 跟实际 fix domain 是否匹配、fix_commit 文件是否在 hypothesis verification 提到的代码路径里
  • L2 LLM-as-Judge：用 GPT-4 评 Final_Analysis.md 的 4 个维度（accuracy / completeness / explainability / actionability）
  • L3 工程师 review：每周抽 20 工单让 senior 评 thumbs up/down + 文字 feedback
  • L4 上线指标：实际工单的"首次定位时间" trend + "工程师采纳 AI 建议" 比例

9.2 Hypothesis Scoring 公式

score = 0.2 × initial_confidence + 0.6 × verification_score + 0.2 × anomaly_coverage

  • initial_confidence（0-1）：historical_matcher 生成时给的初始信心
  • verification_score（0-1）：deep_analyzer 按 verification_plan 收集证据后的支持率（支持 evidence / 总查询 evidence）
  • anomaly_coverage（0-1）：本 hypothesis 能解释多少 fault_context 中的 anomaly
  • 权重 0.2/0.6/0.2 是 tune 出来的 — verification_score 权重最大因为最 ground truth

9.3 confirmation 规则

不只看分数，还要满足：

  • score ≥ 0.7
  • 无强矛盾证据（任何 contradict_evidence 让 hypothesis 进入 disputed 状态）
  • 所有 anomaly 至少被一个 confirmed hypothesis 覆盖（否则标记 unexplained_anomalies）

9.4 metric 演进示意

版本L1 准确率L2 LLM judge 平均L3 工程师采纳L4 首次定位时间 median
v1 (11 subagent fixed pipeline)~42%~3.1/5~38%~52 min
v2 (6 subagent fixed pipeline)~56%~3.4/5~51%~28 min
v3 (hypothesis-driven)~71%~4.0/5~67%~17 min
v3.5 (+ pattern_store 6 月后)~78%~4.2/5~74%~14 min

注：所有 metric 为示意值，用于展示"hypothesis-driven 相对 fixed pipeline 的相对改善幅度"。

案例 1 · 经典 fronthaul 链路故障

背景设定：客户报告某局点 2 个基站同时丢失 1 个射频单元连接。工单标题 "RU comm lost, FaultXX01 + FaultXX02 raised"。

[a explorer] ticket metadata: severity=Major / 涉及 fault: FaultXX01, FaultXX02 ticket_handoff: "RU comm loss in 2 sites, possibly fronthaul" [b log_parser] 下载 zip, 解压 → 提取 按域 logs: domain-A.log, domain-F.log Protocol traces: fronthaul_trace_*.bin Device state files: device_state_*.json [c fault_analyzer] ├─ device_state_parser: RU 进入 NOT_INITIALIZED, 1 hr 前 last seen OK ├─ fault_catalog_lookup: FaultXX01 = "Fronthaul LOS detected" │ FaultXX02 = "RU keepalive timeout" ├─ domain_classifier: is_in_scope=true, primary_domain=domain-A, protocol=fronthaul-type-1 → fault_context: { symptom: "fronthaul link down both sites", temporal_sequence: ["link OK", "spurious LOS", "keepalive miss x3", "RU declared lost"], anomalies: ["both sites simultaneously (rare)", "no preceding RU restart event"] } [d historical_matcher] ├─ L1 RAG: 找到 3 个 high-applicability 历史工单 ├─ 生成 4 个 hypothesis: │ H1 "RU 光模块异常" (init_conf=0.7, plan: device_state SFP 状态) │ H2 "fronthaul fiber 退化" (init_conf=0.6, plan: protocol trace + spec) │ H3 "基带 driver bug" (init_conf=0.5, plan: code_search + commit) │ H4 "时钟同步漂移" (init_conf=0.3, plan: device_state PTP) → recommended order: H1, H2, H3, H4 [e deep_analyzer] H1: device_state SFP_status=OK throughout. verification_score=0.1 (refuted) H2: protocol trace 显示 8B/10B disparity errors 之前 5 min 暴增 spec_search 命中"fronthaul LOS recovery"章节 verification_score=0.85 (supported) anomaly_coverage=0.9 (能解释"两站同时"=同一光纤段) H3: code_search "handleFronthaulLOS" + commit 近 30 天无相关 fix verification_score=0.2 (no evidence) H4: device_state PTP 正常, score=0.1 winner: H2 (final score = 0.6×0.2 + 0.85×0.6 + 0.9×0.2 ≈ 0.81) [f summary] 生成 Final_Analysis.md: Root Cause: 跟 H2 完全一致, "fronthaul fiber 段退化" Fix proposal: "通知现场更换 fiber 段, 临时降速 line rate 缓解" Hypothesis chain 完整 (H1-H4 + refute reasons) 耗时 ~12 分钟 (vs 工程师手动 ~3 hr)

价值：① 12 分钟出可行 fix 建议 ② 工程师能看到"为什么排除了 SFP 异常和 driver bug"（建立信任） ③ 现场可立刻派人换 fiber

案例 2 · 非本域工单路由（is_in_scope=false）

背景设定：工单标题 "fronthaul cell setup failed"。看似 fronthaul 问题，工程师初看也以为是。

[a explorer] ticket metadata, fault 涉及 cell setup [b log_parser] 下载 logs, 含调度器 logs 和配置变更 log [c fault_analyzer] ├─ fault_catalog_lookup: Fault = "Cell setup procedure timeout" ├─ domain_classifier 规则: │ - 协议看到配置协议, 不是 fronthaul 协议 │ - involved_modules 是 SchedulerMgr 不是 RU/BBU │ - 时间序列显示 cell setup 在上层调度未就绪时被触发 → is_in_scope = false → primary_domain = "upper_layer_scheduler" (out of fronthaul scope) [d, e 跳过] [f summary] 生成 non-scope handoff 报告: "本工单不属于 fronthaul 域. 主因是上层调度器在 cell setup 触发时 未完成 X precondition 配置. 建议路由到 调度团队." 附 evidence chain: - 涉及模块: SchedulerMgr / CellConfigMgr (非 fronthaul) - 协议: 配置协议而非 fronthaul 链路 - 时序: scheduler_not_ready 先于 cell_setup_attempt 30s

7 个工程教训

Lesson 1
Subagent 数量是 trade-off 不是越多越好
11 → 6 砍掉 5 个 subagent 反而提升所有 metric。理论上"职责单一"是 SOLID 原则，但 LLM agent 系统里每多一次 subagent 切换 = 启动开销 + handoff schema 设计 + token cost。砍到 6 是 sweet spot — 按 phase 分组（normalize / hypothesize / verify / conclude）而非按 source 分组。
Lesson 2
Hypothesis-driven 是 LLM agent 设计的核心范式
Fixed pipeline 是过去 ML 系统的设计方式（数据流确定）。但 LLM agent 不一样 — 它真的能 generate 假设。让它先 hypothesize 再 verify 比"扫描全部"更准、更快、更可解释。"探索 vs 利用"的最佳平衡。任何复杂 agent 工程都该问 "我能不能用 hypothesis-driven 替代 fixed pipeline"。
Lesson 3
"1 个 dense embedding 搞定一切" 是骗局
v1 想"统一架构" — 所有 source 嵌入到一个 vector DB。结果 30% 准确率。不同 source 的检索特性根本不同：历史工单适合 dense + 元数据 filter / 规范文档适合 BM25 + 结构字段 / 代码必须用 ripgrep + ctags。承认异构，用 4 层独立 RAG，准确率到 ~87%。X08 讲的"hybrid search"在这里就是跨 source 而非跨技术。
Lesson 4
Checkpoint 不是 nice-to-have 是 must-have
长程 agent 调用任何一步可能挂（API timeout / 下载 fail / token quota）。没有 checkpoint = 每次失败重头跑，cost 翻倍 + latency 翻倍。每个 subagent 完成必须落地 subagent_execution_state.json，重启时从最后一个 completed checkpoint 续。这个机制让生产可用性从 ~60% → 95%+。
Lesson 5
"可解释" 是工程师采纳的关键 — 不是技术优势
v2 准确率 ~56% 工程师采纳率却只有 ~51%。v3 准确率 ~71% 采纳率 ~67%。采纳率涨幅小于准确率涨幅。原因：v2 给的是"这是 root cause"，工程师不信黑盒。v3 给的是"考虑了 H1/H2/H3/H4，最终选 H2 因为 X 证据，H1 被 Y 证据否定" — 可解释让工程师能"快速 sanity check"。所以 summary 必须列出 refuted hypotheses，不是为了完整，是为了信任。
Lesson 6
不要让 LLM 决定 "是不是本域"
早期 domain_classifier 用 LLM zero-shot 判断 is_in_scope。准确率波动很大（60-90%） — LLM 受 prompt / case 长度影响误差大。改成规则 + 启发式（基于协议特征 + module 名 + fault_ids 范围），准确率稳定到 95%+。结构化决策不要交给 LLM，留给规则。LLM 只做开放性推理（hypothesize / explain）。
Lesson 7
Pattern Store 是简化版"持续学习"，但有效
不重训模型，只让工程师反馈丰富 retrieval pool — 这是低成本的"模型适应业务"机制。半年后 pattern_store 命中率 ~30%，每命中一次相当于"曾经的同事知识传给了 AI"。给我的启发：很多团队想"自训领域 LLM"，但其实 RAG + persistent feedback loop 已经能拿到 80% 价值，cost 是训练的 1/100。

12.2 3 个最重要的 takeaways

  1. 真实 AI 系统的"难"不在算法，在领域理解 + 工程纪律 + 跟用户建立信任。本案例所有创新点（hypothesis-driven / 4 层 RAG / pattern_store）都来自具体业务洞察而非"AI 最新论文"
  2. 架构决策必须留 ADR 记录。本文 5 个 ADR 写下来后，6 个月后回看团队还能复盘"为什么当时这么选"，避免重复犯错
  3. Hypothesis-driven 是 LLM Agent 的核心范式。任何复杂 agent 系统都应该问 "我能不能用 hypothesis-driven 替代 fixed pipeline"。这是 LLM 时代的"控制论"
