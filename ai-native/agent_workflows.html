Agent Workflow 范式：先用 Workflow，再谈 Agent

"要不要上 Agent"是当下最容易做错的架构决策。Anthropic 在《Building Effective Agents》里给了一个冷静的框架：把 LLM 系统分成 Workflow（预定义流程编排） 和 Agent（模型自主决策循环） 两类，并强调——绝大多数生产场景，一个结构清晰的 Workflow 就够了，别急着上 Agent。本篇讲透 Workflow 与 Agent 的分界，逐一拆解五种可组合的 Workflow 范式（链式 / 路由 / 并行 / 编排 / 评估-优化），配一个可切换五种范式看数据流的交互演示。

Workflow vs Agent：一条关键分界线

谁在做决策？流程是「写死的」还是「模型自己定的」

大家把一切 LLM 应用都叫"Agent"，但 Anthropic 提出一个更有用的区分。关键问题只有一个：流程的走向，是由代码预先定义的，还是由模型在运行时自己决定的？

Workflow（工作流）

LLM 和工具按预先写好的代码路径编排。每一步做什么、下一步去哪，都是你定的。模型只负责"填空"，不决定流程。可预测、可控、好调试。

Agent（智能体）

模型自己决定下一步做什么、用哪个工具、何时停止，在一个循环里自主推进。流程运行时动态生成。灵活但更难预测、更贵、更难调试。

Workflow 像菜谱：步骤固定——先切菜、再热锅、后下料，厨师（LLM）只在每步发挥手艺，但不会自作主张改流程。Agent 像给厨师一个目标"做一桌菜"，让他自己决定买什么、先做哪道、火候如何——灵活应变，但也可能跑偏、超时、买贵了。你越清楚该怎么做，就越该写菜谱（Workflow）；越是开放、难以预定义，才越需要放权给厨师（Agent）。
Workflow vs Agent（Anthropic 定义）：Workflow 是"LLM 与工具通过预定义代码路径编排"的系统；Agent 是"LLM 动态决定自身流程与工具使用、自主完成任务"的系统。两者不是优劣关系，而是控制权归属不同：流程写死在代码里 = Workflow，流程交给模型即时决定 = Agent。
还有一个更基础的起点叫 增强型 LLM（Augmented LLM）：单个 LLM + 检索 + 工具 + 记忆。它是构成 Workflow 和 Agent 的"积木"。很多需求一个增强型 LLM 调用就解决了，连 Workflow 都不用编排。
01

何时用 Workflow，何时才上 Agent

Anthropic 的核心忠告：从最简单的开始，能不上 Agent 就不上

这是全篇最重要的一条工程判断。业界的教训是：过早上 Agent 是常见的过度设计——换来的是难以预测的成本、延迟和 bug，而收益往往还不如一个清晰的 Workflow。

维度Workflow 更合适Agent 更合适
任务结构步骤清晰、可预先拆解开放、步数/路径难以预定义
可预测性要求高（可控、可复现）可接受一定不确定性
成本/延迟敏感（步数固定可控）能容忍（循环次数不定）
调试/审计易（路径确定）难（每次路径可能不同）
典型例子内容审核流水线、固定格式抽取、分类后处理开放式编码任务、复杂研究、需多步试错的问题
三条黄金原则（Anthropic）：① 能简单就别复杂——先试单次增强型 LLM，不够再上 Workflow，还不够才上 Agent；② 只在收益明确时增加复杂度——Agent 的灵活性是用可预测性、成本、延迟换的；③ 把框架当理解工具，别当拐杖——先想清数据流，很多时候几十行原生代码比重框架更清晰。
"上了 Agent 才显得高级"是最贵的错觉。一个把 90% 确定性流程写成 Workflow、只在真正需要动态决策处才放权给 Agent 的系统，几乎总是比"一个大 Agent 包打天下"更快、更省、更稳。
02

五种 Workflow 范式

可单用、可组合 · 覆盖绝大多数「非 Agent」需求

Anthropic 总结了五种经过验证的 Workflow 模式。它们是积木：可以单独用，也可以嵌套组合成复杂流程。理解这五种，你就有了"不上 Agent 也能搞定复杂任务"的工具箱。

① 链式 · Prompt Chaining

把任务拆成固定的顺序步骤，前一步的输出是后一步的输入。可在步骤间加"闸门（gate）"做程序化检查，不合格就停或回退。

例：先"生成大纲"→ 检查大纲合规 → 再"按大纲写正文"→ 最后"润色"。适合能清晰拆成固定步骤、且拆开后每步更准的任务。

② 路由 · Routing

先用一个 LLM（或分类器）判断输入属于哪一类，再分派到专门的后续处理。让每条分支能用更专、更省的处理方式。

例：客服先分类问题类型，简单的走小模型省钱、复杂的走大模型、敏感的转人工。分类准确时，各分支都能优化到最优。

③ 并行 · Parallelization

把任务同时跑多路，再聚合。两种形态：分片（Sectioning）——把任务切成独立子任务并行做；投票（Voting）——同一任务跑多次取多数/择优，提升可靠性。

分片例：一篇长文分段并行摘要再合并（快）。投票例：让模型独立判 3 次"这段代码有无漏洞"，多数通过才放行（准）。

④ 编排者-执行者 · Orchestrator-Workers

一个编排者 LLM 动态拆解任务、分派给多个执行者 LLM，再汇总结果。与"并行分片"的区别：子任务不是预先定死的，而是编排者按输入现场决定的——这一步已带一点 Agent 味道。

例：改一个功能需要改哪几个文件，事先并不知道——编排者读需求后动态决定拆成哪些子任务、派给哪些执行者。适合子任务无法预定义的场景。

⑤ 评估者-优化者 · Evaluator-Optimizer

一个 LLM 生成，另一个 LLM 评估并给出改进反馈，循环迭代直到达标。像"写作 + 编辑"的反复打磨。

例：翻译 → 评估者挑出不地道之处 → 重译 → 再评…直到通过。适合"有明确评价标准、且迭代确实能改进"的任务（如代码、翻译、文案）。

04

Agent：把控制权交给模型的自主循环

当流程无法预定义时，才轮到它出场

当任务开放到"连步骤都没法预先拆"时，就需要 Agent：给它目标和工具，让它在一个循环里自己决定"下一步做什么 → 调工具 → 看结果 → 再决定"，直到判断任务完成或触发停止条件。

这就是 ReAct（Reason+Act）循环——本篇不展开，详见 agent_architecture.html。关键在于：循环次数、路径、工具选择全由模型即时决定，这是它区别于 Workflow 的本质。
Agent 的代价必须正视：成本/延迟不可预测（循环几次不定）、可能失控（跑偏、死循环、烧 token），所以生产 Agent 必须配停止条件、步数上限、预算护栏、人类介入点、充分的可观测（见 agent_architecture.html 的生产部署章）。这些"护栏"本身，往往又是用 Workflow 思路实现的。
05

选型与落地

组合优于单选 · 从简单开始逐步加复杂度

先增强型 LLM

一个 LLM+检索+工具能解决就别编排。这是复杂度的起点。

再组合 Workflow

流程清晰就用链式/路由/并行；子任务要动态拆才上编排；有明确评价标准且能迭代改进才上评估-优化。范式可嵌套。

最后才 Agent

只在"流程无法预定义、且灵活性的收益 > 不可预测的代价"时上 Agent，并配足护栏。

框架是理解工具

LangGraph / DSPy / 各家 SDK 能加速，但先想清数据流。很多时候直接写代码比套框架更透明、更好调试。

这套选型像交通工具选择：走两步（增强型 LLM）就别开车；固定通勤路线（Workflow）坐地铁又快又准时；只有去没有路线覆盖、随时改目的地的地方（开放任务），才值得自己开车（Agent）——但开车要油钱、会堵、可能迷路，得配导航和油量表（护栏）。没人会为了下楼买瓶水而开车。
06

小结 · 面试 · 术语表

收束成判断力

一页纸总结

① 分界=控制权

流程写死在代码=Workflow；流程由模型运行时决定=Agent。不是优劣，是控制权归属。

② 五种 Workflow

链式/路由/并行/编排/评估-优化，可单用可组合，覆盖绝大多数需求。

③ 能不上 Agent 就不上

从简单开始，只在收益明确时加复杂度；Agent 灵活但拿可预测性/成本/延迟换。

基础Workflow 和 Agent 的本质区别是什么？
WorkflowLLM+工具按预定义代码路径编排，流程由你写死，可预测、可控、好调试。
Agent模型在循环里自主决定下一步/用什么工具/何时停，流程运行时动态生成，灵活但不可预测、贵、难调。
一句话区别在"控制权归属"：代码定流程 vs 模型定流程。二者是连续谱。
核心说出五种 Workflow 范式及各自适用场景。
链式固定顺序步骤，前一步喂后一步，可加闸门。任务能清晰拆步时用。
路由先分类再分派到专门处理。不同输入需不同处理、分类可靠时用。
并行分片(切子任务并行)或投票(多次取优)。求快或求稳时用。
编排-执行编排者动态拆分派活再汇总。子任务无法预定义时用。
评估-优化生成+评估反馈循环迭代。有明确评价标准且迭代能改进时用。
辨析"并行分片"和"编排者-执行者"看着都在拆任务并行，区别在哪？
并行分片子任务是预先定死的（代码知道要拆成哪几块），流程可预测=Workflow。
编排子任务由编排者 LLM 看输入现场决定，事先不知道拆成什么，已带 Agent 味道。
关键"谁决定拆成什么"——代码 vs 模型，决定了它偏 Workflow 还是偏 Agent。
工程团队想"给产品加个 Agent"，你怎么判断该不该上、怎么落地？
先问这个任务的流程能不能预先拆清楚？能→用 Workflow，别上 Agent。
阶梯先试增强型 LLM→不够上 Workflow(五范式组合)→仍不够、且流程确实无法预定义、灵活性收益大于代价，才上 Agent。
护栏上 Agent 必配步数上限/预算/停止条件/人类介入/可观测，否则成本与失控风险不可控。
心态"上 Agent 显得高级"是最贵的错觉；多数场景清晰的 Workflow 更快更省更稳。

① 总体框架

Agent模型在循环里自主决定流程与工具、完成任务的系统。灵活但不可预测、贵、难调。
