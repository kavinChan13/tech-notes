GraphRAG 深入：把知识连成图再检索

普通向量 RAG 擅长「找到最像的几段」，但面对需要跨多篇文档串联关系的问题（"整个项目里谁和谁有协作"、"这份年报的整体主题是什么"）就力不从心——因为它只会挑相似片段，不理解片段之间的关系与全局结构。GraphRAG 先把语料抽取成知识图谱（实体+关系），再对图做社区检测与分层摘要，检索时能同时支持「精确的局部实体查询」与「宏观的全局主题归纳」。本篇讲透它的构建、检索与选型，配一个 Local vs Global 检索的可交互演示。

向量 RAG 的天花板：它不懂「关系」和「全局」

相似片段 ≠ 结构化知识 · 有两类问题它天生答不好

普通（向量）RAG 的世界观是：把语料切块 → 嵌入 → 查询时找最相似的 top-k 块 → 塞给 LLM。它对「答案就写在某几段里」的问题很强。但有两类问题，它会系统性地失败——不是召回不准，而是这种架构根本不适合。

① 多跳关系推理

"A 的合作者，又参与了哪些 B 项目？" 答案分散在多篇、需要顺着关系链跳。向量检索一次只捞相似块，串不起这条链。

② 全局 / 归纳型问题

"这 100 篇报告整体在讲什么趋势？" 答案不在任何单一块里，而在全局分布中。top-k 只给你几段，永远看不到全貌。

向量 RAG 像用关键词在一摞资料里抽出最相关的几页——问具体的事很好用。但你要是问「这整摞资料讲的是什么故事、谁和谁是什么关系」，光抽几页没用，你需要先有人把它读完、画成一张人物关系图 + 章节大纲。GraphRAG 干的就是这件事：先把散落的资料整理成「关系图 + 分层摘要」，再回答问题。
GraphRAG：微软 2024 年提出的检索增强范式。先用 LLM 把语料抽取成知识图谱（实体为节点、关系为边），对图做社区检测并为每个社区生成摘要；检索时根据问题类型走「局部实体检索（Local）」或「全局社区摘要归纳（Global）」。核心价值：补上向量 RAG 缺失的关系推理与全局归纳能力。
01

GraphRAG 是什么：向量库 → 知识图谱

把「一堆孤立的块」升级成「相互连接的知识网络」

向量 RAG 的知识表示是一堆互不相连的向量点；GraphRAG 的知识表示是一张图：实体是节点、关系是带标签的边。这个表示差异，决定了它能回答的问题类型完全不同。

同样的语料，向量 RAG 存成孤立向量点（只能算相似度），GraphRAG 存成带关系的图（能沿边做多跳推理、能按连通性归纳全局）。
GraphRAG 不是要取代向量 RAG——它更重、更贵（构建图要大量 LLM 调用）。它是在「关系推理 / 全局归纳」这两类向量 RAG 的死角上补位。实践中常与向量检索混合使用（见第 5 章）。
02

构建：从文本抽取实体与关系

索引阶段的重头戏 · 也是 GraphRAG 成本的大头

GraphRAG 的图不是现成的，要在索引阶段用 LLM 从文本里一点点「读」出来。这一步是它比向量 RAG 贵得多的根源——每个文本块都要过一遍 LLM 做抽取。

构建管线：切块 → LLM 抽取实体/关系 → 合并同名实体 → 建成带权图 → 社区检测（下一章）。
实体与关系抽取（Entity & Relation Extraction）：用 LLM（配抽取 prompt）从每个文本块里识别出实体（人/组织/概念…）、它们的类型与描述，以及实体之间的关系（谁负责谁、谁属于谁）。同一实体在多块中出现会被合并（entity resolution）成一个节点，关系被多次提及则边的权重更高。
抽取质量决定一切：抽错/抽漏实体，图就歪了，后面全错。且这一步是离线一次性大成本（大语料可能几千上万次 LLM 调用），更新语料还要增量维护图。这是 GraphRAG 落地最大的门槛。
03

社区检测与分层摘要

把大图切成「主题社区」，为每个社区写摘要 · 这是 Global 检索的基础

图建好后，节点可能成千上万。直接拿整张图没法回答「全局讲什么」。GraphRAG 借用图算法做社区检测：把联系紧密的节点聚成一个个「社区」（相当于自动发现的主题簇），再为每个社区生成一段摘要。社区还能逐层聚合成更大的社区，形成一棵「主题层级树」。

这就像给一本厚书自动生成多级目录：先把内容聚成章（社区），每章写个概要；章再聚成篇，篇写个更宏观的概要。等你问"这本书大概讲啥"，我不用重读全书，直接把顶层几篇的概要拼给你就行。社区摘要就是这套「预先写好的分层概要」——让全局性问题变得可答且便宜。
社区检测（Community Detection）：用图聚类算法（GraphRAG 默认用 Leiden 算法）把图划分成内部连接紧密、彼此连接稀疏的节点群（社区）。每个社区近似一个"主题"。社区可分层：小社区聚合成大社区，形成层级。
社区摘要（Community Summary）：索引阶段就用 LLM 为每个社区（及其层级）生成的自然语言摘要。Global 检索时，系统把相关层级的社区摘要作为"预制的全局知识"来归纳作答，无需在查询时重扫全部原文——这是 GraphRAG 能高效回答全局问题的关键。
Leiden 把图切成社区，LLM 为每个社区（及上层聚合）预写摘要。回答"全局问题"时直接归纳这些摘要，快且全。
04

检索：Local search vs Global search

两种问题，两条检索路径 · GraphRAG 的精髓

4.1 两种检索模式

Local Search（局部检索）

面向具体实体的问题（"张三负责什么？"）。定位到实体节点，沿邻居扩展（相关实体、关系、原文块），把这一片"局部子图 + 证据"喂给 LLM 作答。=向量 RAG 的关系增强版。

Global Search（全局检索）

面向宏观归纳的问题（"整体有哪些主题？"）。不查具体节点，而是遍历社区摘要：对每个相关社区摘要生成局部答案（map），再汇总成最终答案（reduce）——map-reduce 式归纳。

Local vs Global 的选择：问题里有明确实体/指向具体细节 → Local；问题是"总体/趋势/有哪些/归纳" → Global。成熟系统会先做一次意图判断自动路由（呼应 rag_query_rewriting.html 的查询路由）。Global 走 map-reduce 因此更慢更贵，别拿它答简单细节题。

05

对比与选型：向量 / Graph / 混合

GraphRAG 不是升级版，是补位版 · 多数场景是混合

维度向量 RAGGraphRAG混合 Hybrid
擅长问题具体事实、语义相似关系推理、全局归纳两者兼顾
索引成本低（嵌入即可）高（大量 LLM 抽取）中高
更新维护加块即可要维护图/重算社区较复杂
多跳/全局弱强强
延迟（Global）—较高(map-reduce)视路由
选型建议：绝大多数场景先上向量 RAG（便宜、够用）。只有当你确认业务里有大量「关系推理 / 全局归纳」型问题、且这些问题价值高，才引入 GraphRAG——通常是混合：向量负责具体事实题，Graph 负责关系与全局题，用查询路由分派。别为了"听起来高级"无脑上 GraphRAG，它的构建与维护成本很现实。
向量 RAG 是搜索引擎，GraphRAG 是专门整理好的百科全书 + 人物关系图谱。查个具体信息，搜索引擎又快又省；但要理解"整个领域的脉络和人物关系"，你会想要那本精心编纂的百科。代价是：百科编起来又慢又贵，内容一变还得重新修订。
06

落地与工具

从原型到生产要面对的现实问题

微软 GraphRAG

开源参考实现，把抽取→社区(Leiden)→摘要→Local/Global 检索一条龙做好。研究/原型首选，也是理解范式的最佳读物。

Neo4j + LLM

用成熟图数据库存图，配 Cypher 查询 + 向量索引。生产级图存储/查询能力强，适合已有图数据库基建的团队。

LlamaIndex / LangChain

提供 Knowledge Graph Index / GraphRAG 集成，快速搭原型，与现有 RAG 管线衔接。

成本与更新

抽取是大头开销：可用小模型做抽取降本、只对高价值语料建图；语料更新走增量抽取 + 局部重算社区，别全量重建。

上生产前务必用 eval（ai_eval.html）对比「向量 vs GraphRAG vs 混合」在你的真实问题分布上的效果与成本。GraphRAG 在"关系/全局"题上收益明显，但若你的问题 90% 是简单事实题，它的高成本就不划算。

小结 · 面试 · 术语表

收束成判断力

一页纸总结

① 补关系与全局

向量 RAG 只会挑相似块，答不好多跳关系与全局归纳。GraphRAG 把知识连成图来补这两块。

② 构建三步

LLM 抽实体/关系建图 → Leiden 社区检测 → 预写社区摘要。抽取是成本大头。

③ Local vs Global

具体实体走 Local（子图+证据），全局归纳走 Global（社区摘要 map-reduce）。多数场景混合。

面试自测（点击展开）

基础GraphRAG 相比向量 RAG 解决了什么问题？
向量短板只挑相似 top-k 块，块之间无关系，答不好①多跳关系推理②全局/归纳型问题。
GraphRAG把语料抽成知识图谱（实体+关系）+ 社区摘要，支持沿关系链推理与全局归纳。
代价索引要大量 LLM 抽取，构建/更新贵——是补位而非替代。
核心Local search 和 Global search 分别怎么工作？何时用哪个？
Local定位实体节点→沿邻居扩展子图+原文证据→作答。用于具体实体/细节问题。
Global遍历社区摘要，map（每社区出局部答案）-reduce（汇总）。用于全局/归纳/趋势问题。
路由先判意图自动选模式；Global 较慢较贵，别用它答简单细节题。
辨析社区检测和社区摘要在 GraphRAG 里起什么作用？
社区检测用 Leiden 把图聚成主题簇，可分层。相当于自动发现"章节主题"。
社区摘要索引期预写每个社区的摘要，让 Global 检索直接归纳摘要而非重扫全文。
类比相当于给语料自动生成多级目录 + 各级概要，全局问题因此可答且便宜。
工程什么情况下你会（不会）在生产里上 GraphRAG？
会上业务里高价值问题大量属于"关系推理/全局归纳"，且语料相对稳定、值得一次性建图。
不上问题 90% 是简单事实题、语料频繁更新、预算紧——向量 RAG 更划算。
折中混合架构 + 查询路由：向量答细节、Graph 答关系/全局。用 eval 在真实问题分布上验证收益 vs 成本。

术语表速查 · Glossary

① 核心概念

GraphRAG先把语料抽成知识图谱 + 社区摘要，再做 Local/Global 检索的 RAG 范式。补关系推理与全局归纳。
知识图谱 Knowledge Graph实体为节点、关系为带标签边的结构化知识表示。GraphRAG 的核心数据结构。
实体/关系抽取 Entity/Relation Extraction用 LLM 从文本抽出实体、类型、描述与关系；同名实体合并成节点。索引成本大头。

② 图处理

社区检测 Community Detection图聚类（GraphRAG 用 Leiden）把节点分成主题簇，可分层。近似"自动发现主题"。
Leiden 算法高质量社区划分算法（Louvain 的改进），GraphRAG 默认使用。
社区摘要 Community Summary索引期为每个社区(及层级)预写的摘要，Global 检索直接归纳它，无需重扫全文。

③ 检索模式

Local Search定位实体→扩展邻居子图+证据作答。用于具体实体/细节问题，是向量 RAG 的关系增强版。
Global Search遍历社区摘要做 map-reduce 归纳。用于全局/趋势/归纳问题，较慢较贵。
混合 RAG Hybrid向量答细节、Graph 答关系/全局，用查询路由分派。生产常见形态。
