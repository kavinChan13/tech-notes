一、简单对象的内存布局

C++ 中最基本的对象就是一个无虚函数、无继承的 struct/class。编译器按声明顺序在内存中铺放各数据成员。

struct Point {
    int    x;     // offset 0,  4 bytes
    int    y;     // offset 4,  4 bytes
    double z;     // offset 8,  8 bytes
};  // sizeof(Point) = 16 (无 padding)
成员函数不占用对象内存——它们存在于代码段 (.text)。只有数据成员影响 sizeof。

二、内存对齐与 Padding

CPU 要求数据按自然对齐边界存放。编译器会在成员之间和结构体尾部插入填充字节 (padding)。

对齐规则

  • 每个成员的起始地址必须是其 alignof 的倍数
  • 结构体总大小必须是最大成员 alignof 的倍数

未优化 vs 优化布局
struct Bad { char a; // 1B double d; // 8B int i; // 4B char b; // 1B }; // = 24B struct Good (16 bytes) — 零浪费 d double 8B i int 4B a b 2B struct Good { double d; // 8B int i; // 4B char a; // 1B char b; // 1B }; // = 16B 优化技巧：按成员大小从大到小排列，节省 33% 内存！

常见类型的对齐要求

类型大小对齐 (alignof)
char11
short22
int / float44
double / long long / pointer88
long double (x86)12 / 164 / 16

三、this 指针与成员访问

成员函数被编译器隐式传递一个 this 指针，指向调用对象的起始地址。

S* this) { return this->a + this->b; } 内存中的成员访问 = 基址 + 偏移量: this->a ⟹ *(int*)((char*)this + 0) // offset 0 this->b ⟹ *(int*)((char*)this + 4) // offset 4
this 指针是编译器自动传递的"隐藏参数"。在 x86-64 (System V ABI) 上通过 rdi 寄存器传递；Windows x64 上通过 rcx。

四、虚函数表 (vtable) 与 vptr

当类含有 virtual 函数时，编译器在每个对象中插入一个虚表指针 (vptr)，指向该类的虚函数表 (vtable)。

class Animal {
public:
    virtual void speak() { cout << "..."; }
    virtual void eat()   { cout << "nom"; }
    int age;
};

class Dog : public Animal {
public:
    void speak() override { cout << "Woof!"; }  // 覆盖
    virtual void fetch() { cout << "ball!"; }     // 新增虚函数
    int tricks;
};
一个类只有一张 vtable（存在 .rodata），所有该类实例共享。vptr 通常在构造函数的初始化列表之后、构造体之前被设置。

五、单继承对象模型

class Base {
    int x;
    virtual void f();
};

class Derived : public Base {
    int y;
    void f() override;
    virtual void g();
};

六、多重继承对象模型

class A { virtual void fa(); int a; };
class B { virtual void fb(); int b; };
class C : public A, public B {
    void fa() override;
    void fb() override;
    int c;
};
多重继承中 static_cast 向上/向下转型会自动调整指针值。但 reinterpret_cast 不会——用它在多继承层次中转型是未定义行为。

七、虚继承与菱形继承

菱形继承问题

class Animal  { int weight; virtual void breathe(); };
class Mammal  : public Animal  { int fur; };
class WingedAnimal : public Animal { int wingspan; };
class Bat : public Mammal, public WingedAnimal { int sonar; };
// 问题：Bat 中有两份 Animal！weight 是哪个？

// 解决方案：虚继承
class Mammal       : virtual public Animal { int fur; };
class WingedAnimal : virtual public Animal { int wingspan; };
class Bat : public Mammal, public WingedAnimal { int sonar; };
// 现在 Bat 只有一份 Animal 子对象，通过 vbase offset 间接访问
虚继承的代价：每个虚继承路径增加一个 vbase offset（存在 vtable 或 VTT 中），访问虚基类成员需要额外的间接寻址。性能和空间都比普通继承差。

八、RTTI 与 type_info

RTTI (Run-Time Type Information) 使程序在运行时可以查询对象的真实类型。它依赖 vtable。

type_info 通常存储在 vtable 的负偏移处（slot [-1]）。编译选项 -fno-rtti 可禁用 RTTI 以减小二进制体积。

九、构造与析构顺序

构造顺序（从内到外）

在构造函数/析构函数中调用虚函数不会触发多态！此时 vptr 指向当前正在构造/析构的类的 vtable，而非最终派生类。

十、对象切片 (Object Slicing)

将派生类对象按值赋给基类变量时，派生部分被"切掉"。

Dog dog;
Animal animal = dog;   // ✗ 切片！调用 Animal 的拷贝构造
animal.speak();         // 输出 "..." 而非 "Woof!" — 多态丧失

Animal& ref = dog;     // ✓ 引用，无切片
ref.speak();            // 输出 "Woof!" — 多态正常

Animal* ptr = &dog;    // ✓ 指针，无切片
ptr->speak();           // 输出 "Woof!" — 多态正常

十一、空基类优化 (EBO)

C++ 标准要求每个对象至少占 1 字节（以保证不同对象有不同地址）。但空基类可以被优化掉。

标准库大量使用 EBO，例如 std::vector 继承 allocator（通常是空类），std::unique_ptr 继承 deleter。C++20 的 [[no_unique_address]] 提供了更优雅的方案。

C++11 将旧的 POD 概念拆分为两个正交属性：Trivial 和 Standard Layout。

14.1 Itanium C++ ABI — vtable 的真实布局

GCC / Clang 遵循 Itanium C++ ABI，vtable 实际上是一个结构数组：

// vtable 布局 (单继承简单情况)
vtable[-2] = offset_to_top        // 本 vptr 到完整对象起始的偏移（用于 dynamic_cast）
vtable[-1] = type_info*            // RTTI 信息 (typeid)
vtable[ 0] = &virt_func1            // 虚函数指针
vtable[ 1] = &virt_func2
vtable[ 2] = &virt_func3
...

对象首 8 字节存 vptr（指向 vtable[0]，不是 vtable 开头）。因此：

  • typeid(obj) = *(type_info**)(vptr - 1) — O(1)
  • dynamic_cast<Derived*>(base_ptr) — 需要 offset_to_top 和 type_info，通过 __dynamic_cast 运行时查找，可能是 O(继承深度)
  • 多继承时，每个非 primary base 对应一个独立 subobject 的 vptr

用 GDB 查看真实 vtable

(gdb) p /x obj
(gdb) info vtbl obj
vtable for 'Derived' @ 0x400b20 (subobject @ 0x7ffe...)
[0]: 0x400a00 <Derived::foo()>
[1]: 0x400a30 <Derived::bar()>
[2]: 0x400a60 <Derived::~Derived()>
[3]: 0x400a90 <Derived::~Derived()>   # 两个析构：complete & deleting

14.2 多继承与 vcall Thunk（this 指针调整）

struct A { virtual void f(); int a; };   // 8B vptr + 4B + pad = 16B
struct B { virtual void g(); int b; };   // 16B
struct C : A, B {                         // C 布局: [A subobj][B subobj]
    void g() override;                    // 通过 B 的接口调用
};

C c;
B* pb = &c;              // pb = &c + 16  (指针调整)
pb->g();                  // 通过 B 的 vtable 调到 C::g，但 C::g 需要完整 C 的 this

当通过 B* 调用已被 C override 的 g() 时，C 的 vtable 中对应 B 接口的那一项指向一个 vcall thunk：

// 自动生成的 thunk（伪汇编）
C_override_g_thunk:
    sub  rdi, 16        # 调整 this: B subobj → C 起始
    jmp  C::g          # 跳到真正实现

因此：多继承的虚调用有一次额外的指针调整，热路径应避免 diamond/多继承。

14.3 空基类优化 (EBO) 与 [[no_unique_address]]

C++ 要求每个对象有独立地址 → 空类 sizeof == 1。但作为基类时可被优化为 0：

情况sizeof说明
struct Empty{};1保证 &a != &b
struct S{ Empty e; int i; };8e 占 1B + 3B pad
struct S: Empty { int i; };4✅ EBO — 基类 Empty 占 0B
C++20 [[no_unique_address]] Empty e;4✅ 成员版 EBO
// 经典用法：allocator、compare、deleter 常是空类
template<class T, class Compare = std::less<T>>
class Set {
    // C++17 及以前：Compare 用 EBO 私有继承
    struct Node : Compare { T value; };

    // C++20：更优雅
    [[no_unique_address]] Compare cmp;
    T* root;
};
// std::tuple、std::function、std::unique_ptr 都大量使用 EBO

MSVC 坑：默认不启用 EBO，需要 __declspec(empty_bases)。

14.4 Trivially Copyable 与 memcpy 合法性

只有 trivially copyable 类型，用 memcpy 复制对象才是合法的（等价于拷贝构造）。

类别要求可 memcpy可 std::memset(0)
Trivial所有特殊成员函数都 trivial + 默认构造 trivial✅✅
TriviallyCopyable拷贝/移动构造、赋值、析构都 trivial（默认构造可非 trivial）✅取决于值
StandardLayout所有成员同访问级别、无虚函数、无虚基类、首成员非基类等❌（除非同时 TriviallyCopyable）❌
POD (C++20 deprecated)Trivial + StandardLayout✅✅
// 编译期检查，推荐放 static_assert
static_assert(std::is_trivially_copyable_v<MyStruct>);
static_assert(std::is_standard_layout_v<MyStruct>);   // 可 offsetof / C 互操作

// 非 trivially copyable 的类用 memcpy 是 UB！例如：
struct Bad { std::string s; };              // has user-defined copy
Bad a, b;
std::memcpy(&b, &a, sizeof(Bad));         // ❌ UB: s 共享内存，析构时 double free

14.5 SBO / SSO — 标准库的小对象优化

很多标准库类型内部有"小对象缓冲"，小值直接存对象内部，避免堆分配：

类型优化阈值（libstdc++/MSVC）
std::stringSSO: 小字符串存对象内libstdc++ 15B, libc++ 22B, MSVC 15B
std::functionSBO: 小可调用对象存对象内通常 16-24B，大则堆分配 + 虚调用
std::anySBO: 小值存对象内通常 < 1-2 pointer
std::variant按最大 alternative 大小分配（无堆）—
// std::string 的典型布局 (libstdc++ cow 前 / libc++)
class string {
    union {
        struct { char* ptr; size_t size; size_t cap; } heap;
        struct { char buf[22]; char size_bit; }           sso;
    };
    // 最低位标志：0=SSO, 1=heap
};
// 结论：短字符串零堆分配；> 22 字节才走 malloc

性能影响：std::function<void()> f = someLambda;——如果 lambda 捕获过多导致超过 SBO，每次赋值/拷贝都是一次 new。高频路径首选 function_ref 或 inline template。

14.6 位域 (Bit-field) 与 Flexible Array Member

位域

struct PacketHeader {
    uint32_t version : 4;    // 4 bits
    uint32_t ihl     : 4;    // 4 bits
    uint32_t tos     : 8;
    uint32_t length  : 16;
};  // sizeof == 4，完美描述 IP 头

注意：位域排列顺序 implementation-defined——GCC 大端机 vs 小端机顺序相反！协议解析推荐用显式位运算。

Flexible Array Member (C99 → C++)

struct Msg {
    uint32_t len;
    char     data[];     // C99 柔性数组；C++ 需用 [0] 扩展（非标准）
};
Msg* m = (Msg*)malloc(sizeof(Msg) + n);
m->len = n;
// 一次分配，header + payload 紧挨。内核/网络库常用。
// C++ 标准做法: 用 std::unique_ptr<char[]> + placement new 的 struct 布局

14.7 高频面试 Q&A

Q1: sizeof 空类为什么是 1 不是 0？

C++ 要求不同对象有不同地址。如果 sizeof(Empty) == 0，数组 Empty a[10] 里所有元素地址相同、指针算术失效。作为基类时不受此限制，故有 EBO。

Q2: vptr 放对象首部还是尾部？为什么？

放首部（Itanium ABI）。原因：

  • 单继承时 Base* 和 Derived* 指同一地址——指针零调整，性能好
  • 派生类增加成员只影响尾部，不改 vptr 位置
  • 但多继承下，非 primary base 的 vptr 必然在对象中间

Q3: 为什么构造函数中调虚函数不触发多态？

每层构造函数执行时，vptr 正被设置为该层自己的 vtable。若基类构造中调虚函数，此时派生类对象的派生部分还没构造，vptr 指向基类 vtable，调用的是基类版本。析构同理（反方向）。
推论：永远不要在构造/析构中调虚函数，改用两阶段初始化。

Q4: 虚函数有多少运行时开销？现代 CPU 能优化吗？

  • 每次调用：一次内存读（vptr）+ 一次内存读（vtable 项）+ 一次间接跳转
  • 间接跳转对 CPU 分支预测不友好，但现代 CPU 的 indirect branch predictor 对稳定类型能预测准确
  • 编译器的去虚化 (devirtualization) 可消除：final 类/方法、LTO 能看到唯一派生、constexpr 等场景

实测：热路径 virtual 调用比直接调用慢 10-30%，冷路径几乎看不出。

Q5: final 关键字对性能有影响吗？

✅ 有。final 告诉编译器该类/方法不会再被继承/覆盖，编译器可以静态解析虚调用为直接调用（devirtualization），避免 vtable 查找。LTO 下效果更好。

class Impl final : public Interface { ... };
Interface* p = new Impl;
p->foo();   // 编译器可直接调 Impl::foo，无 vtable

Q6: 为什么基类析构函数一般要 virtual？什么时候可以不 virtual？

通过基类指针 delete 派生对象时，若 ~Base 非 virtual 则只调 Base 析构 → 派生部分资源泄漏。
可以不 virtual 的情况：

  • 基类设计为永不被 delete by pointer（如 CRTP 静态多态的基类）
  • 通过 protected 析构 + 禁止派生类 delete 的方式控制
  • 标准库的 std::enable_shared_from_this 等

Q7: struct 和 class 有区别吗？

语法层面仅 2 点：① 默认访问权限（struct=public, class=private）；② 默认继承权限同上。
内存布局、虚函数、模板参数推导等完全相同。惯例：纯数据用 struct，有不变式 / 封装的用 class。

Q8: 如何查看一个类的真实内存布局？

  • Clang: clang -Xclang -fdump-record-layouts -c foo.cpp — 最友好
  • GCC: g++ -fdump-lang-class foo.cpp 产出 .class 文件
  • MSVC: /d1 reportSingleClassLayoutClassName
  • pahole（依赖 DWARF）：pahole -C MyClass a.out，直接显示 offset、hole、padding
  • GDB：ptype /o MyClass（新版支持）或 print &obj.member - (char*)&obj

Q9: 为什么 std::vector<bool> 被视为设计失误？

标准规定它为节省空间特化为位打包——每个 bool 占 1 bit。但带来：

  • operator[] 返回代理对象而非 bool&，auto& b = v[0]; 不是你想的
  • 不能取 &v[0] 作为 bool*
  • 非 thread-safe：并发修改相邻 bool 会冲突写同一 byte（word tearing）
  • 违反 Container concept

替代：std::vector<char>、std::bitset、boost::dynamic_bitset。

Q10: 什么是 layout-compatible？有什么用？

两个 struct 如果成员序列的类型一一对应兼容，它们是 layout-compatible。可以通过 union 共同访问或 reinterpret_cast 合法转换——C++ 标准特别允许此时读共同初始序列不是 UB。

struct Base   { int tag; };
struct Msg1   { int tag; int payload; };
struct Msg2   { int tag; double payload; };
// 通过读 tag 决定用哪个类型 — C 风格 tagged union
union U { Msg1 m1; Msg2 m2; } u;
if (u.m1.tag == 0) ...   // 合法：Msg1 和 Msg2 的初始公共序列是 { int tag; }
