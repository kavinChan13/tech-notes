⚖️ 三个高频权衡

• 短期交付 vs 长期架构
• 技术理想 vs 业务节奏

📈 晋升的关键判据

不是"你做了什么"，而是 "因为有你，这件事才发生 / 才做成" — 也就是不可替代性。

02 背景差距分析 · 从 SM 到技术负责人

Scrum Master 的优势是流程、协调、沟通，劣势是技术权威、技术决策、技术判断。面试官对 SM 转型最容易质疑的就是这一点。

🧭 你当前的坐标（SM + C++）

  • 熟悉敏捷流程、站会、回顾、迭代
  • 擅长跨角色协调与冲突化解
  • 有 C++ 编码能力（个人贡献者级）
  • 项目进度、风险跟踪经验
  • 已接触团队管理与氛围建设
➜

🚀 技术负责人要求的坐标

  • 有"主导"而非"协调"的项目经历
  • 能当众做技术选型与架构决策
  • 对业务指标/成本/性能有数据化判断
  • 带过人、招过人、挂过绩效
  • 有对外/对上的技术影响力

03 能力提升路径（可执行）

每一项能力都对应做事 + 总结 + 输出三个动作。"没做过"可以造场景，"做过"必须能量化。

① 技术决策能力

1主动接手 1 个 有歧义的需求，自己做 3~5 页架构设计文档（含方案对比表、trade-off、风险）
2每个选型都写出"我拒绝了什么"：拒绝的方案、拒绝理由、拒绝后的风险兜底
3刻意练习 容量估算 / QPS / 成本：每个方案至少配 1 组数字
4读 1~2 本：《DDIA》《架构整洁之道》《凤凰项目》

② 团队管理能力

1主动申请带 1~2 个新人，做结构化 Mentor（月度 1-on-1 + 成长地图）
2参与至少 5 场技术面试（哪怕是陪面），积累"招聘标准"语言
3试着主导一次绩效对齐会议（对每个成员用数据说话，而非印象）
4读：《Manager's Path》《高效能经理人》《Radical Candor》

③ 项目推进能力

1从 SM 的"跟进者"升级为"owner"：自己定目标、拆 WBS、盯关键路径
2每个项目输出 3 份文档：Kick-off PRD / Tech Design / 结项复盘
3刻意练习 风险识别表：开工前列出 top10 风险，每周更新
4量化：人力 / 工时 / 延期率 / 返工率，养成"用数字讲项目"的习惯

④ 跨团队协作能力

1主动找 至少 2 个上下游团队做接口规范对齐（平台 / 测试 / 运维）
2学会开 "决策会议"：3 分钟背景 + 2 个方案 + 1 个推荐 + 1 个结论
3冲突场景训练：遇到"对方不配合"时，先给对方拿结果，再提要求
4建立 1 个跨团队虚拟小组（技术委员会 / SIG），积累影响力

⑤ 业务理解能力

1每周花 2h 读 产品日报 / 业务指标看板，能说出 3 个北极星指标
2每个需求问自己 3 个问题：业务价值是什么？会带来多少收益？成本是多少？
3尝试反向提议：给 PM 提 1~2 个技术驱动业务的点子（如架构优化带来的成本下降）
4面试时务必能说出 你负责业务的规模（DAU/QPS/GMV）

⑥ 技术影响力

1每季度做 1 次部门级技术分享（不是产品分享）
2维护 1 份团队 技术规范 / 最佳实践 文档
3有条件写博客 / 投稿技术大会（哪怕内部技术周）
4主导 1 次故障复盘宣讲，把事故变成沉淀

🗓️ 6 个月提升路径（时间线）

M1 — 技术复权

重新定位"我是有技术判断力的人"

复习你熟悉的 C++ 项目，重写 1 个模块；主动在 Code Review 中提出架构性建议；准备 1 篇深度技术文。

M2 — 主导一个 0→1 项目

从 SM 转 owner，积累架构 case

自荐或争取一个小项目全流程主导。即使只是子模块，也完成"设计评审 → 拆分 → 交付 → 复盘"闭环。

M3 — 建立人员影响

从"我一个人"到"我这一小组"

明确带 2~3 个人；做一份团队能力地图；和每个人完成 2 次 1-on-1；推动 1 次面试。

M4 — 跨团队突破

走出自己团队

主导一次跨团队架构对齐；有 1 份被其他团队引用的设计文档；推动 1 个"非你不可"的协同项目。

M5 — 业务化表达

从 tech speak → business speak

用业务指标重新包装你过去的所有项目；准备面试用的 3 个黄金故事（每个覆盖多个能力维度）。

M6 — 模拟面试 & 投递

系统演练，拿 offer

找 2~3 个 P7/P8 朋友做模拟面；根据反馈打磨"故事版本 v2"；开始精准投递；每家公司复盘总结。

04 面试表达 · STAR 法则示例（基于 NETCONF Agent 项目）

STAR = Situation 情境 / Task 任务 / Action 行动 / Result 结果。面试官听的不是故事本身，而是 "你这个人在事里的角色 + 决策 + 影响"。下面 6 个故事全部基于真实的 NETCONF Agent 项目（5G DU ↔ O-RU M-plane），每个故事包含 STAR 主叙述 + 技术深挖 + 追问答法 + 金句。

📡 项目背景 · 面试开场 1 分钟自我介绍（务必背熟）

"我目前主导的 NETCONF Agent 运行在 5G 基站 DU 侧，作为 OAM 进程通过 NETCONF/YANG 协议（over SSH/TLS）管理 O-RU 设备。上接 BTS Controller（通过 LIM 数据库 IPC），下连 O-RU（支持 O-RAN WG4 标准）。核心代码 C++ / ~500 源文件 / 集成 sysrepo、libnetconf2、libyang。 我负责的模块涉及 Controller → RadioAgent → Dispatcher → Scenario → NetconfService 这条主链路，核心技术点包括：Scenario 协程式编排引擎、50+ FaultHandler 规则引擎、Per-RU 多线程并发模型、O-RAN YANG 多版本兼容（@2024-12-23 / @2025-04-14 / @2025-08-11 并存）。"

🎯 开场核心要素：业务域（5G/O-RAN）+ 技术栈（NETCONF/YANG/C++）+ 规模（500 文件 / 50+ Handler）+ 你的位置（主导主链路）。避免讲成"我们做了个网管 Agent"——要具体。
🧠 能力 1 · 技术决策 — "Scenario 编排引擎的选型" Dispatcher / BaseScenario
Situation 我们的 NETCONF Agent 在单个 O-RU 生命周期中需要按严格时序执行多个 NETCONF 操作：Carrier 创建就有 ProcessingElement → ArrayCarrier → LowLevelTxEndpoint → UplaneConf → tearDownOk 等 6~10 个步骤；中间任何一步 O-RU 回复 rpc-error 或超时，都需要回滚或 recheck。项目早期用"一个任务一个函数串着跑"的写法，代码里到处是 if (step1_ok) step2(); else retry()，状态散落在十几个成员变量里，新增场景（FaultHandler / SwInstall / AntennaCalibration）就得复制一套样板代码。当时已经有 30+ Scenario，再加新功能基本上是地狱。
Task 我作为 NETCONF Agent 技术负责人，需要设计一个可扩展的场景编排引擎，让后续新增场景从"改十几个文件"变成"新增一个子类 + 注册"，同时保持单 RU 操作时序严格串行（并行会出现配置冲突），且要能被 Notification Thread（来自 O-RU 的异步告警）以优先级插队。
Action 我组织了一次架构评审，对比了 3 个方案：
① Thread-per-Scenario：每个场景一个线程，用 mutex 串行——被我否了，原因是单 RU 会产生几十个 short-lived 线程，线程创建/销毁 overhead 在嵌入式 DU 上不可接受（我们 RU 密度 1 进程管 10~20 个 RU）。
② 状态机（State Pattern）：每个场景一个显式状态机——被我否了，原因是 Carrier 创建这种"6~10 步线性任务"写成状态机反而膨胀，维护成本高。
③ 协程式 Task Queue（我最终选的）：BaseScenario 持有 std::deque<std::function> tasks_，每个 step 是一个 lambda，handle() 被 Dispatcher 反复调用逐步 pop + 执行；遇到要等 O-RU 回包时 return Status::Waiting 让出控制权，Dispatcher 去跑别的；回包到达后 recheck() 重新调度。这其实是"非抢占式协程"，但不依赖 C++20 coroutine（我们 DU 编译器只到 C++17）。

我做了 trade-off 表对齐团队：让步时序（自然串行）、调度成本（单线程 O(1) 出队）、可扩展性（新场景只 override initialize() 把 tasks 压进 deque）、可测性（每个 lambda 可独立 mock）。

我还额外设计了：优先级队列 prioScenarios_ 给 Notification 线程插队 FaultHandler；state_ 机制避免 tearDown 阶段接收新任务；Chrome Tracing 格式的性能埋点方便排查慢步骤。
Result 这套引擎落地后成为整个 NETCONF Agent 最核心的抽象，目前承载 50+ Scenario 类（Carrier / FaultHandler / SwInstall / Supervision / BeamForming 等）。新增一个场景的平均代码量从旧方式的 ~600 行降到 ~150 行；引擎上线 2 年 无架构级重构；后续新业务（O-RAN 共享小区、天线校准）全部复用同一套编排层；Chrome Tracing 埋点帮团队在 3 次线上性能问题中快速定位到单步耗时瓶颈。我写的设计文档被沉淀为部门"嵌入式场景编排"模板。
🔬 技术深挖（被追问时你要 hold 得住的细节）
  • 为什么不用 C++20 coroutine？——嵌入式 DU 侧编译器只支持 C++17；且自研 task-queue 模型编译产物小，符合 -Os + --gc-sections 的二进制瘦身要求。
  • tasks_ 是 deque 而不是 vector？——因为 recheck 场景下需要在队首插入重试任务（push_front），vector 做头部操作 O(n)。
  • 怎么支持优先级插队？——Dispatcher 每次调度前先检查 prioScenarios_（non-empty 时立刻处理），FaultHandler 通过 addPriorityScenario() 入队；同时在当前 Scenario 处于 Waiting 状态时才允许插队，避免切断串行操作。
  • 怎么保证线程安全？——Dispatcher 自己是单线程，Datastore 只由 Dispatcher 线程访问（线程亲和约定）；队列用 mutex + condition_variable；state_ 我最初没加 atomic，后来 Coverity 扫出 memory visibility 风险，改成了 std::atomic<EDispatcherState>（下面故障处理故事会讲到）。
🎯 面试官可能追问的 3 个问题 + 回答模板
"如果让你重做一次，你会怎么改？"
"如果现在重做，在 C++20 环境下我会直接用 std::coroutine + co_await，代码可读性会更好；但核心抽象（单 Dispatcher 线程 + 优先级插队 + 线程亲和数据）不会变——这个分层是对的。这也是我给下一代 Agent 的建议。"
"为什么单线程 Dispatcher 不是瓶颈？"
"热点路径 Profiling 显示真正的瓶颈是 NETCONF RPC 的网络往返（单次 20~200ms），单 Dispatcher CPU 占用只有 3% 左右。并行化也没意义——同一个 O-RU 对多并发 RPC 不友好，我们测过会产生 configuration lock conflict。"
"你怎么说服团队接受这个设计？"
"我给了团队里对状态机方案有执念的资深同事一个'挑战点'——让他用状态机实现我们最复杂的 CreateCarriersScenario，我同时用 task-queue 实现。两版代码行数对比是 800 vs 180；他自己就服了。我从不靠权威压方案，我靠事实。"
💎 金句
"我选 task-queue 不是因为它最炫，而是因为它最匹配我们'单 RU 串行 + 异步回调'的真实执行模型。好架构是贴着业务的形状长出来的。"
"trade-off 表里我写得最用心的是'拒绝什么'那一列——体现判断力的不是你选了什么，是你明确拒绝了什么。"
👥 能力 2 · 团队管理 — "带新人从 0 到独立 owner FaultHandler" 50+ FaultHandler / Nokia Rules
Situation NETCONF Agent 的 FaultManagement 模块是重灾区：50+ FaultHandler 对接 Nokia 自研的告警规则引擎（每条告警根据 faultId 查规则表做字段组装、合并、去抖、升级降级），入门门槛高——既要懂 O-RAN 告警语义（o-ran-fm.yang 里的 alarm-notif），又要懂内部 FaultMO 结构，还要懂 LIM 数据库。团队新入职 2 位同学（1 个社招 C++ 但没做过电信、1 个校招），前 6 周每周都在返工，PR 平均 3~5 轮 review 才能合入，团队整体节奏被拖慢。
Task 作为 Tech Lead，我要在 3 个月内让两个新人都能独立 owner 至少一类 FaultHandler；同时不占用团队核心成员过多带教时间（我们当时还在交付 O-RU 告警升级的硬需求）。
Action 我做了 4 件事：
① 切分成长地图（90 天 3 阶段）：M1 只让他们读代码 + 写 3 个"最简单的单告警透传 Handler"（不碰规则引擎）；M2 做 pair coding 一起开发一个中等复杂度的合并告警 Handler；M3 独立 owner 一类（如 Transceiver 类告警）。
② 提炼"FaultHandler 十问 Checklist"——我把过去 2 年踩过的坑归纳成 10 个自查问题（例：告警清除时有没有 check matchingCriteria？有没有处理 RU reset 时的孤儿告警？LIM 写入失败时有没有 retry？），每次 PR 他们先自己打勾，不满足不给 review。这把 PR 轮次从 3~5 轮降到 1~2 轮。
③ 每周 1-on-1 结构化：不是闲聊，是"3 题固定 + 1 题自选"：这周最卡的点 / 下周最想突破的点 / 对团队的反馈 / 自选话题。我用 Notion 记录每人的成长线。
④ 让他们反向 teach：M3 开始，每人给团队做 1 次 "我这类 Handler 的踩坑分享"——这倒逼他们把碎片知识系统化，也让其他同事看到他们的成长。
Result 3 个月后 2 人都独立 owner 了 FaultHandler 的一大类：社招同学负责 SFP/Transceiver 相关 8 个 Handler，校招同学负责 Antenna/Calibration 相关 6 个。半年考核社招同学绩效 A（团队 TOP20%），校招第二年晋升到 Senior Engineer。"FaultHandler 十问 Checklist" 被部门所有 FM 相关团队复用，作为新人 onboarding 的标配；我的 1-on-1 模板也被 2 位同级 Tech Lead 借鉴。

🚨 能力 6 · 故障处理 — "Dispatcher 线程安全 P1 事故复盘" Dispatcher::state_ / onRuDeleted detached thread
Situation 现网某站点每隔 3~7 天出现一次 NETCONF Agent 进程崩溃，影响 15~30 个 O-RU 同时断管理面；coredump 堆栈指向 Dispatcher::isRunning()，但读的是一个早已被回收的 Dispatcher 对象。故障时间不固定，现网难复现，已经开了 3 个 IR（Incident Report）且都被 workaround 掉。我接手时已经升级到 P1 级，客户投诉进了总监周报。
Task 48 小时内给出根因 + 止血方案；7 天内出永久修复 + 全系统同类问题扫描。
Action ① 止血（2 小时内）：先给现网下一个配置开关，禁用 RU 动态删除（onRuDeleted 路径）——这能避免触发崩溃，代价是要整机重启才能清理。现场运维接受。
② 根因定位（48 小时内）：我带着 2 个核心成员搭了复现环境，通过日志 diff 发现崩溃都在 RU deletion 后 1 秒内发生。深挖代码发现两个互相放大的问题：
    问题 A：Controller::onRuDeleted() 用了 std::thread(...).detach() 异步释放 RadioAgent，进程退出时没人 join，detached 线程还在跑就被回收 → UAF。
    问题 B：Dispatcher::state_ 不是 atomic，isRunning() 被另一个线程无锁读，x86 上没事但某些 ARM 核上内存可见性丢失 → 读到了旧值 → 以为还在 running → 继续访问已释放成员。
Coverity 其实早扫出来过 Problem B，但当时被 // coverity[SUPPRESS] 压了——这是最打脸的一段。
③ 永久修复（5 天）：
    a) Controller 持有 std::vector<std::jthread>（C++20）统一管理销毁线程，进程退出时等待 join；
    b) Dispatcher::state_ 改为 std::atomic<EDispatcherState>，移除所有 SUPPRESS；
    c) 新增 线程亲和运行时断言：Datastore 的所有写入点加 assert(std::this_thread::get_id() == dispatcherThreadId_)，让类似违规在开发期就暴露。
④ 复盘 + 机制化（24 小时内）：我主持了全组线上复盘，不追人只追机制。产出 3 条规则 + 1 个工具：
    • 规则 1：禁止在 Code Review 里使用 // coverity[SUPPRESS]，必须 Tech Lead 签字 + 写明为何安全。
    • 规则 2：所有 detach() 必须走 ThreadManager 统一管理，不再允许裸 detach。
    • 规则 3：跨线程共享的状态字段必须显式标 atomic 或 guarded_by(mutex)（用 clang thread-safety annotation）。
    • 工具：写了一个 grep-based CI check 自动扫描新增的 .detach() / SUPPRESS，违反就 block merge。
Result 48 小时给出根因，7 天永久修复上线，现网 14 天无同类问题，后续半年零复发；复盘报告被部门当作"线程安全事故教材"在其他 3 个团队做了宣讲；我设计的 CI 扫描工具扩展后扫出另外 2 个潜在同类 bug，都是在爆发前修掉；我主持的"不追人追机制"复盘文化被部门采纳，之后复盘再也没出现过 blame 文化。
🔬 故障定位的关键工具链
  • Core dump + gdb：info threads 找到崩溃线程；thread apply all bt 看全线程栈；关键是看崩溃线程不是 Dispatcher 自己——是另一个 detached 线程碰了它的成员。
  • 日志 diff：对比崩溃前 5 秒 vs 正常前 5 秒日志，锁定 onRuDeleted 时间窗口。
  • ThreadSanitizer（TSan）：在开发环境开 -fsanitize=thread 复现，立刻报出数据竞争位置。TSan 只有开发环境能开（性能开销 5~10 倍），但是最靠谱的证据。
  • ARM vs x86 行为差异：内存模型弱序的 ARM 上更容易暴露，这也是为什么实验室 x86 测不出来但现网 ARM 基站上爆。

05 高频面试问题 · 回答思路

以下题目覆盖一线大厂 P7/P8 管理岗 80% 高频考点。"回答思路"给出骨架，"加分点"给出超出平均线的抓手。

Q1. 你最近主导的一个技术项目是什么？给我讲讲。
思路
  • 选 最近 1 年 + 你是主导 + 有量化结果 的项目（不要讲你只是成员的）
  • 结构：业务背景 → 技术挑战 → 方案选型（含 trade-off） → 执行 → 量化结果 → 沉淀
  • 控制 3~4 分钟，留出被追问的空间
加分点 ✦
  • 主动说出"我当时纠结过 XX 方案，最终没选，因为…"
  • 讲出"这个项目让团队/公司沉淀了什么"
  • 主动补一句"复盘来看我还可以做得更好的是…"
Q2. 你的技术判断力体现在哪？举一个你拍板但团队有反对意见的例子。
思路
  • 承认有分歧（否则显得假），明确冲突点
  • 说出你怎么"听"的：数据 / POC / 业界案例
  • 说出你怎么"拍"的：给出明确结论和时间窗口（"先按我说的做 2 周，不行就改"）
  • 承担后果：结果好 → 说结果；结果不好 → 说你学到了什么
加分点 ✦
  • 强调"我没有压制反对意见，而是把它作为风险兜底的来源"
  • 提一句"决策不是一次性的，我设置了 checkpoint 如果数据不对会回滚"
Q3. 你怎么给团队定绩效？你最差的一次绩效谈话是什么样？
思路
  • 先讲框架：OKR/KPI + 行为准则 + 同事反馈 三维度
  • 再讲过程：日常记录、中期对齐、季末评定不能突袭
  • 讲一个"给低绩效"或"解雇"的案例：事实 → 反馈 → 改进期 → 结果
  • 体现"对事不对人，且帮他看到下一步"
加分点 ✦
  • 提到"我给差绩效前会自问：是不是我作为 leader 没给够支持"
  • 有 PIP (Performance Improvement Plan) 的具体经验
Q4. 你从 Scrum Master 转技术负责人，为什么？有什么差距？
思路
  • 不要否定 SM 的价值：SM 给了我流程、协作、结构化思考
  • 讲主动性："我发现我更想对结果负责，而不只是流程"
  • 差距坦诚：技术决策深度 + 架构全局视角 — 但我在做什么来补
加分点 ✦
  • 有具体的"我从 SM 时代就开始做的技术决策"案例 — 证明你没停过技术
  • 说出"我比纯技术出身的 TL 多的一项能力是：跨角色协同和节奏感"
Q5. 你招人的标准是什么？讲一个你招错的案例。
思路
  • 维度：硬技能 + 学习能力 + 文化 fit + 动机
  • 讲一个错招的例子（技术牛但不 fit 团队），讲你怎么发现、怎么处理
  • 讲你后续怎么优化面试流程
加分点 ✦
  • 点明"招人不是招最强的，而是招和团队当前阶段最匹配的"
  • 讲到"行为面试" / "系统设计面试"的具体题目设计
Q6. 线上出 P0 事故你会怎么做？
思路 — 务必按这个顺序讲
  • ① 止血（优先级永远是恢复 > 定位）
  • ② 同步（拉 war-room / 向上汇报 / 客户沟通）
  • ③ 定位（日志 / 监控 / diff）
  • ④ 复盘（5 Why + 机制化 + 自动化扫描）
  • ⑤ 闭环（action items + owner + deadline）
加分点 ✦
  • 强调"复盘不追人，追机制 — 出事是因为体系没兜住"
  • 能讲出事故分级标准（SLA、MTTR、MTTD）
Q7. 你怎么和产品经理冲突？
思路
  • 先假设 PM 是对的，问清楚业务诉求
  • 用数据/原型表达技术风险，不要用"这个做不了"
  • 给出 Plan A / Plan B，让 PM 来选
  • 实在冲突 → 向上对齐，不要把人变成敌人
加分点 ✦
  • "我从不说'这个做不了'，我说'这个要多少成本，我们要不要付'"
  • "冲突其实是信息差，我 90% 的冲突通过补齐信息就消失了"
Q8. 你怎么做团队的技术规划？（P8 级高频）
思路（三层递进）
  • ① 业务对齐：公司 OKR → BU OKR → 团队 OKR
  • ② 现状盘点：技术债清单、架构成熟度、团队能力画像
  • ③ 规划输出：1 年节奏（季度里程碑）+ 关键指标（成本/稳定性/效率）+ 人员梯队
加分点 ✦
  • 讲到"技术规划不是技术愿望清单，是有 ROI 的投资组合"
  • 有"做什么 / 不做什么"的明确拒绝清单
Q9. 你怎么看 AI / 大模型对你们团队的影响？
思路
  • 避免喊口号，讲你已经在做什么（哪怕只是 Copilot / CodeReview / 文档总结）
  • 给出衡量："我们用 AI 做 X，效率提升了 Y%"
  • 讲风险判断：哪些能 AI 化、哪些不能
加分点 ✦
  • 有实际落地 case（MCP / Agent / RAG 工具），哪怕小
  • 能把"AI 提效"换算成人力/金额
Q10. 你未来 3 年的职业规划？
思路
  • 短期 (1 年)：在当前岗位做出 2~3 个标杆项目
  • 中期 (2~3 年)：带更大团队 / 跨 BU 项目 / 架构委员会
  • 长期：想清楚自己是走技术路线 (架构师) 还是管理路线 (经理/总监)
  • 承诺：对应届公司的具体价值输出
加分点 ✦
  • 不要说"我想进步"，要说"我想在 X 领域成为被认可的专家"
  • 把规划和应聘公司的业务挂钩（做足功课）

06 避坑指南

管理岗面试"挂得冤"的人，80% 死在以下这些坑里 — 尤其对于 SM 转型，"只会流程不懂技术"是最致命的印象。

❌ 坑 1：答题全是"我们"，没有"我"

"我们团队做了…我们改进了…" — 面试官根本听不出你贡献了什么。

✓ 修正：Action 部分必须用第一人称，明确"我主导/我决定/我推动"。

❌ 坑 2：只讲流程，不讲技术细节

"我组织了 review，开了对齐会" — 听起来你只是 coordinator，不是 tech lead。

✓ 修正：每个管理动作背后都补一个技术决策 + 技术数字（选型/量化）。

❌ 坑 3：没量化结果

"项目顺利上线了，效果不错" — 面试官心里打了一个大问号。

✓ 修正：延迟/QPS/成本/人力/满意度 — 至少给 1 个数字，哪怕是估算。

❌ 坑 4：规避冲突，显得软弱

"我们团队没什么冲突" — 没冲突 = 没在做难事，面试官直接判你没有 ownership。

✓ 修正：坦诚讲一个真实冲突 + 你怎么处理 + 学到什么。

❌ 坑 5：炫技术，忘业务

技术方案讲得很嗨，但面试官问"这个业务价值是什么"你答不上来。

✓ 修正：每个技术项目必须能用 1 句业务话 收尾（省了多少钱/多少人/多少时间）。

❌ 坑 6：把"打分"当绩效管理

"我给他打了 B" — 然后呢？面试官想听的是你怎么让他变成 A。

✓ 修正：绩效 = 目标对齐 + 过程反馈 + 结果评定 + 成长牵引，讲完整的闭环。

❌ 坑 7：SM 经历讲成"排会议的人"

"我组织站会、我协调资源" — 把自己讲成了行政角色。

✓ 修正：把 SM 经验重新包装为 "风险管理 / 团队效率 / 跨角色协同" 能力。

❌ 坑 8：不敢承认不会

被问到不懂的问题硬编，立刻暴露 — 面试官不反感"不会"，反感"假装会"。

✓ 修正："这块我还没深度实践过，但我的理解是…，如果真遇到我会从 X 入手。"

❌ 坑 9：没有"拒绝过什么"的故事

只讲"做了什么"的人，面试官会怀疑你没有判断力 — P7 的核心是取舍。

✓ 修正：每个黄金故事准备 1 个"我拒绝了什么" — 拒绝的人/方案/需求。

❌ 坑 10：问不出好问题

面试最后"你有什么问题"，答"没有"或问薪资假期 — 瞬间掉档。

✓ 修正：准备 3 个问题：团队当前最大挑战 / TL 岗位半年内期望 / 晋升路径。

🎭 如何避免"只会流程不懂技术"的致命印象（SM 转型专项）

常见陷阱面试官看到的你破解动作
简历写"组织 retrospective" 你是敏捷教练而不是 TL 改成"主导 X 项目复盘，输出 Y 条改进项，使返工率下降 Z%"
答题反复出现"协调/对齐" 你只是搬砖传话 换成"我决定/我拍板/我做了技术评审"
无法讲清项目技术细节 你没真正碰过代码 每个项目务必能讲到架构图/核心数据流/关键代码逻辑
没有技术决策 case 你没有技术判断力 主动在简历里挑出 1~2 个"我做了 X 选型"的 bullet
答不出系统设计题 不是真正的 TL 刷经典系统设计（短链、Feed、IM、监控、限流），确保能画出架构图

🧭 一句话总结

  1. 技术负责人 ≠ 懂技术的管理者，而是既扛技术决策又扛团队结果的双重角色。
