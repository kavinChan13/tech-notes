本 PPT 涵盖的模式

本 PPT 与 GoF 23 个模式的关系

  • 覆盖 GoF 中真正在 C++ 工程中常用的模式
  • 剔除 GoF 中已被语言特性取代的（如 Iterator → STL；Strategy → lambda）

Singleton — 4 种实现演化

从经典到现代，一个模式问题反映了 C++ 30 年的并发演进。

① 朴素实现（不线程安全）

class Logger {
    static Logger* instance_;
    Logger() {}
public:
    static Logger* get() {
        if (!instance_) instance_ = new Logger();    // ❌ 多线程下竞争
        return instance_;
    }
};
Logger* Logger::instance_ = nullptr;

② 双重检查锁（DCLP）— 历史方案

static std::atomic<Logger*> instance_;
static std::mutex mu_;

static Logger* get() {
    Logger* p = instance_.load(std::memory_order_acquire);
    if (!p) {
        std::lock_guard lk{mu_};
        p = instance_.load(std::memory_order_relaxed);
        if (!p) {
            p = new Logger();
            instance_.store(p, std::memory_order_release);
        }
    }
    return p;
}
// ⚠️ DCLP 的难点: C++11 之前无 atomic + memory order, 极易写错

③ std::call_once（C++11，安全且通用）

static std::once_flag flag_;
static std::unique_ptr<Logger> instance_;

static Logger& get() {
    std::call_once(flag_, []{ instance_ = std::make_unique<Logger>(); });
    return *instance_;
}

④ Meyers Singleton（C++11，最简最优 ⭐⭐⭐）

class Logger {
    Logger() {}
public:
    Logger(const Logger&)            = delete;
    Logger& operator=(const Logger&) = delete;

    static Logger& instance() {
        static Logger inst;            // C++11 保证局部 static 线程安全初始化 (magic static)
        return inst;
    }
};

// 使用
Logger::instance().log("hello");
结论：现代 C++ 单例 = Meyers Singleton。① 自动线程安全；② 不需要显式删除指针；③ 析构顺序符合"逆构造序"；④ 代码最少。
能讲出 "Magic Static" 是 C++11 标准 [stmt.dcl]/4 强制保证的，编译器会插入类似 call_once 的同步代码 → 满分回答。
不能 讲出"DCLP 历史问题" + "为什么 atomic + memory order 是 C++11 才能正确实现 DCLP" → 加分。
反模式警告：单例方便但滥用会让代码难测试、隐式依赖。优先依赖注入 (DI)，只有"全局唯一" + "无状态依赖" 才用单例（Logger / MetricsRegistry）。

四种实现一眼对比

方案线程安全代码量析构评价
① 朴素裸指针❌ 竞争少泄漏仅单线程玩具
② DCLP⚠️ 需 atomic+内存序多手动历史方案，易写错
③ call_once✅中自动通用、可控初始化
④ Meyers✅ magic static最少逆序自动首选 ⭐

本质：多个调用方共享同一个实例

所有调用方拿到的是同一个引用 —— 这也正是"难以替换/测试"的根源

到底该不该用单例？

✅ 可以用
  • 进程内天然唯一且无业务状态：日志器、指标注册表、配置只读快照
  • 替代"到处传同一个对象"的样板，且不影响测试
🚫 改用依赖注入
  • 持有可变业务状态 / 外部资源（连接、设备会话）
  • 单测需要替换或重置 → 单例的隐藏全局态会让用例互相污染
  • 只是图方便少写一个参数 → 隐式依赖，后患无穷

Factory — 工厂方法 + 注册表

简单工厂方法 · 注册表式工厂（可插拔）

核心思想

把对象的构造和使用解耦。调用方只通过抽象接口/工厂得到对象，不关心具体类型。

什么时候用

  • 对象类型由配置/运行时数据决定
  • 可插拔的模块系统
  • 测试需要替换实现

简单工厂方法

class Codec {
public:
    virtual ~Codec() = default;
    virtual void encode(const Message&, Buffer&) = 0;
};

std::unique_ptr<Codec> make_codec(const std::string& type) {
    if (type == "json") return std::make_unique<JsonCodec>();
    if (type == "pb")   return std::make_unique<PbCodec>();
    throw std::invalid_argument(type);
}

注册表式工厂（可插拔）

class CodecRegistry {
    using Creator = std::function<std::unique_ptr<Codec>()>;
    std::unordered_map<std::string, Creator> map_;
public:
    static CodecRegistry& instance() { static CodecRegistry r; return r; }
    void register_codec(const std::string& name, Creator c) { map_[name] = std::move(c); }
    std::unique_ptr<Codec> create(const std::string& name) {
        auto it = map_.find(name);
        return it == map_.end() ? nullptr : it->second();
    }
};

// 自动注册 — 在每个 codec 实现的 .cpp 中
struct JsonCodecRegister {
    JsonCodecRegister() {
        CodecRegistry::instance().register_codec("json",
            []{ return std::make_unique<JsonCodec>(); });
    }
};
static JsonCodecRegister _reg;             // 静态对象在 main 之前自动注册
工程价值：插件 / Codec / Driver / Strategy 等机制都基于注册表 — 新增类型时不需要改主代码，遵循开闭原则。

什么时候真的需要 Factory？（先想清楚再写）

✅ 适合
  • 构造哪种对象由运行时数据（配置 / 报文 / 节点标签）决定
  • 类型集合会持续增长，希望新增时不改调用方
  • 构造逻辑本身有分支 / 校验 / 默认值，不只是 new
  • 测试需要替换具体实现（配合接口）
🚫 过度设计
  • 只有一两种类型且基本不变 → 直接 if / make_unique 即可
  • 为「将来可能扩展」预留的抽象工厂 → YAGNI
  • 能用枚举 + 表驱动解决，却套了三层抽象类
选型顺序：① 一两种 → 自由函数 make_xxx()；② 多种且固定 → 表驱动 / dict 分发；③ 需要第三方插件注册 → 注册表式工厂。从简单往复杂走，不要一上来就抽象工厂层次。

CASE工具项目：按「节点标签」分发的工厂（取材自真实校验器）

配置校验工具链解析 XML 时，同一个 create() 入口按节点的 tag 选择不同构造分支，返回对应领域对象 —— 这正是「表驱动工厂」的典型形态。下面在保留结构的前提下改写了名称、去掉了内部常量与产品细节。

# 形态取自真实 PropertiesFactory：静态 create + tag→builder 映射
class PropertyFactory:
    @staticmethod
    def create(node):
        builder = {
            "paragraph": PropertyFactory._make_paragraph,
            "list":      PropertyFactory._make_list,
        }.get(node.tag)
        if builder is None:
            raise ValueError(f"unsupported node: {node.tag}")
        return builder(node)

    @staticmethod
    def _make_paragraph(node):
        return ParagraphProperty(_name(node), _value(node))

    @staticmethod
    def _make_list(node):
        items = [PropertyFactory.create(c) for c in node.children]
        return ListProperty(_name(node), items)
为什么这样写而不是一堆 if/elif？ ① 映射表一眼看全支持哪些类型；② 新增标签只加一行；③ create 递归处理嵌套结构（list 里再含 paragraph），结构清晰。

CASE代理项目（C++）：同一问题的另一种语法

C++ 设备代理侧更常见的不是「字符串分发」，而是接口 + 在 .cpp 里 make_unique 具体实现，把构造点收敛到一处、对调用方只暴露接口：

// 调用方只看见 ISwmRpcAdapter；具体类型在 .cpp 决定，便于替换 / 打桩
class SwInventoryScenario {
    std::unique_ptr<adapter::ISwmRpcAdapter> rpc_;
public:
    explicit SwInventoryScenario(RadioContextPtr ctx)
        : rpc_(std::make_unique<SwmRpcAdapter>(ctx)) {}   // ← 构造点收敛于此
};
两种写法同一个目的：把「选哪种实现」与「怎么用」分开。Python 侧因鸭子类型用 dict 分发更轻；C++ 侧靠接口 + 智能指针获得可替换性与 RAII。

Builder — 流式构造大型对象

典型 Builder · C++20 现代写法（聚合 + 指定初始化）

解决什么

构造函数参数太多（10+）+ 大多有默认值 → 调用方写得痛苦。

用 Builder 链式调用，可读性提升 10×。

C++20 替代方案：指定初始化

Foo{.a=1, .b=2} 大多数 builder 场景可以被取代（适用于聚合类型）。

典型 Builder

class HttpRequest { /* 大量字段 */ };

class HttpRequestBuilder {
    HttpRequest req_;
public:
    HttpRequestBuilder& method(std::string m)        { req_.method = std::move(m); return *this; }
    HttpRequestBuilder& url(std::string u)           { req_.url    = std::move(u); return *this; }
    HttpRequestBuilder& header(std::string k, std::string v) {
        req_.headers.emplace(std::move(k), std::move(v)); return *this;
    }
    HttpRequestBuilder& body(std::string b)          { req_.body = std::move(b); return *this; }
    HttpRequestBuilder& timeout(std::chrono::milliseconds t) { req_.timeout = t; return *this; }

    HttpRequest build() && { return std::move(req_); }     // rvalue ref qualifier
};

// 使用
auto req = HttpRequestBuilder{}
    .method("POST")
    .url("https://api.example.com/v1/users")
    .header("Authorization", "Bearer ...")
    .header("Content-Type", "application/json")
    .body(R"({"name":"alice"})")
    .timeout(5s)
    .build();

C++20 现代写法（聚合 + 指定初始化）

struct HttpRequest {
    std::string method = "GET";
    std::string url;
    std::map<std::string, std::string> headers;
    std::string body;
    std::chrono::milliseconds timeout{2000};
};

auto req = HttpRequest{
    .method = "POST",
    .url    = "https://...",
    .body   = R"(...)",
    .timeout = 5s,
};
复杂构造（含校验/分支）仍用 Builder；纯字段集合用 C++20 designated initializer。

链式构造：每一步都返回自己

每个 setter return *this → 可读性强；build() && 一次性 move 出成品

Builder 还是 C++20 指定初始化？

✅ 用 Builder
  • 构造含校验 / 分支 / 计算（如互斥字段、默认推导）
  • 需要"半成品不可见"（只有 build() 后才拿到合法对象）
  • 字段非常多、且常常只设其中几项
🚫 直接聚合初始化
  • 纯数据结构、字段都有默认值 → Foo{.a=1,.b=2} 更短更直观
  • 没有不变量需要在构造时强制 → Builder 是多余样板

Object Pool — 对象池

实现 — 通用 Object Pool · CASE：连接池 vs 惰性缓存

解决问题

  • 对象构造 / 析构开销大（含系统调用 / 大块内存）
  • 高 QPS 场景频繁创建 → GC / fragmentation 压力
  • 典型：连接池 / 线程池 / 大 Buffer 池

关键设计

  • 用 RAII guard 自动归还（不依赖调用方调用 release）
  • 线程安全 (mutex 或 lock-free)
  • 高水位 / 超时回收

实现 — 通用 Object Pool

template<class T>
class ObjectPool : public std::enable_shared_from_this<ObjectPool<T>> {
    std::vector<std::unique_ptr<T>> pool_;
    std::mutex mu_;
public:
    // 自定义 deleter — 归还到池而不是 delete
    using Handle = std::unique_ptr<T, std::function<void(T*)>>;

    Handle acquire() {
        std::unique_lock lk{mu_};
        std::unique_ptr<T> obj;
        if (!pool_.empty()) {
            obj = std::move(pool_.back());
            pool_.pop_back();
        }
        lk.unlock();
        if (!obj) obj = std::make_unique<T>();
        auto self = this->shared_from_this();
        return Handle(obj.release(), [self](T* p){
            std::lock_guard lk{self->mu_};
            self->pool_.emplace_back(p);
        });
    }
};

// 使用 — 离开作用域自动归还
auto pool = std::make_shared<ObjectPool<Buffer>>();
{
    auto buf = pool->acquire();
    do_work(*buf);
}    // buf 析构 → deleter 把对象放回池

何时该上对象池？（多数情况答案是「不用」）

✅ 值得池化
  • 构造成本真的高：建连接、握手、分配大块内存、加载模型
  • 对象可安全复用：能 reset 到干净状态
  • 高频获取 / 释放，且并发争用可控
🚫 别池化
  • 对象很轻（POD / 小 buffer）→ 直接构造更快也更安全
  • 复用状态难清理 → 容易泄漏脏数据，得不偿失
  • 用池只是为了"省 new" → 现代分配器已很快，先 profile
先问自己：构造开销是否进入了热路径？没量化过就别引入池 —— 它带来线程安全、生命周期、容量回收三类新复杂度。

CASE连接池 + 「昂贵初始化」对照

左：经典数据库连接池

与 C++ 代理项目中「长生命周期、系统调用密集」对象同一类动机 — 教学代码。

class DbConnectionPool {
    std::queue<std::unique_ptr<DbConn>> idle_;
    std::condition_variable cv_;
    std::mutex mu_;
    size_t max_size_;
    size_t active_ = 0;
public:
    std::shared_ptr<DbConn> acquire(std::chrono::milliseconds timeout) {
        std::unique_lock lk{mu_};
        if (!cv_.wait_for(lk, timeout, [this]{
            return !idle_.empty() || active_ < max_size_;
        })) throw std::runtime_error{"timeout"};
        std::unique_ptr<DbConn> conn;
        if (!idle_.empty()) { conn = std::move(idle_.front()); idle_.pop(); }
        else conn = std::make_unique<DbConn>();
        ++active_;
        return std::shared_ptr<DbConn>(conn.release(),
            [this](DbConn* p){ release(p); });
    }
    void release(DbConn* p) {
        std::lock_guard lk{mu_};
        idle_.emplace(p);
        --active_;
        cv_.notify_one();
    }
};

右：工具项目 — 惰性解析缓存（同族动机）

校验器对大配置文档只 objectify 解析一次，后续入口直接复用 —— 不是严格对象池，但共享「昂贵初始化只做一次」的核心动机（取材自真实的模块级缓存，名称已改写）。

# 形态取自真实的「模块级缓存全局量」
_doc_cache = None

def get_doc():
    global _doc_cache
    if _doc_cache is None:                 # 第一次才真正解析
        _doc_cache = objectify.fromstring(_read_bytes())
    return _doc_cache
这种「全局缓存」本质是隐藏单例：单测必须留reset钩子，否则用例间状态互相污染。

PIMPL — Pointer to Implementation 编译墙

C++ 经典编译墙 / 二进制兼容惯用法

解决什么

  • 编译时间：头文件不暴露成员 → 改实现不用重编译用户代码
  • 二进制兼容 (ABI)：库的内部数据成员可以增删，sizeof 不变
  • 编译墙：头文件不含第三方库 #include → 用户不需要那些依赖
  • 隐藏实现：私有成员真的对用户不可见

代价

  • 多一次堆分配（对象 + impl）
  • 多一次指针跳转
  • 不能 inline 实现
  • 需要写转发函数（机械工作）

典型实现

// foo.h - 头文件 (干净)
#pragma once
#include <memory>

class Foo {
public:
    Foo();
    ~Foo();
    Foo(Foo&&) noexcept;            // 必须显式声明 (因为 unique_ptr<Impl> 中 Impl 不全)
    Foo& operator=(Foo&&) noexcept;

    void do_something(int);
    std::string get_state() const;
private:
    struct Impl;                       // 前置声明
    std::unique_ptr<Impl> pImpl;
};
// foo.cpp - 实现
#include "foo.h"
#include <heavy_third_party.h>     // 用户看不到

struct Foo::Impl {
    HeavyThing data;
    Cache cache;
    Logger* log;
};

Foo::Foo() : pImpl(std::make_unique<Impl>()) {}
Foo::~Foo() = default;                          // ⚠️ 必须在 .cpp 定义 (Impl 此时已知)
Foo::Foo(Foo&&) noexcept = default;
Foo& Foo::operator=(Foo&&) noexcept = default;

void Foo::do_something(int x) { pImpl->cache.put(x); }
std::string Foo::get_state() const { return pImpl->data.to_string(); }
坑：① 析构必须在 .cpp（否则 unique_ptr 删除不完整类型 → 编译错或 UB）；② 移动构造也要在 .cpp 默认。这两条踩过的人不在少数。
"PIMPL 主要解决什么问题?" — 答：① 编译时间（头文件改动不影响用户）；② ABI 稳定性（动态库版本演进）；③ 隐藏实现细节。不是为了"封装"（C++ 已有 private）。

编译墙：改实现，用户不用重编

头文件不暴露成员 / 不含重依赖 → 改 Impl 时用户的 .o 无需重新编译；动态库 sizeof 不变保 ABI

什么时候值得 PIMPL？

✅ 用
  • 对外发布的库，要保持 ABI / 头文件稳定
  • 头文件被大量 TU 包含，编译时间是痛点
  • 实现依赖一堆重型第三方头，不想传染给用户
🚫 不必
  • 纯内部类、热路径小对象 → 多一次堆分配 + 指针跳转不值
  • 只是想"隐藏私有成员" → C++ 的 private 已经够了
  • 需要 inline / constexpr 性能 → PIMPL 挡住了内联

CRTP — 静态多态

静态多态示例 · 注入比较运算符

核心思想

派生类把自己作为模板参数传给基类：class Derived : public Base<Derived> {}

基类编译期知道派生类类型 → 可以静态调用派生类方法 → 无虚函数 / 无 vptr / 无运行时开销。

典型用途

  • 静态多态 (静态接口)
  • 给派生类注入通用功能（如比较运算符、计数）
  • Mixin 模式
  • std::enable_shared_from_this 就是 CRTP

静态多态示例

template<class Derived>
class Shape {
public:
    double area() const {
        return static_cast<const Derived*>(this)->area_impl();   // 静态分发
    }
    void print() const { std::cout << "area=" << area(); }
};

class Circle : public Shape<Circle> {
    double r_;
public:
    Circle(double r) : r_(r) {}
    double area_impl() const { return 3.14159 * r_ * r_; }
};

Circle{1.0}.print();    // area=3.14 — 编译器内联，零开销

注入比较运算符

template<class Derived>
class Comparable {
    friend bool operator!=(const Derived& a, const Derived& b) { return !(a == b); }
    friend bool operator>(const Derived& a, const Derived& b)  { return b < a; }
    friend bool operator<=(const Derived& a, const Derived& b) { return !(b < a); }
    friend bool operator>=(const Derived& a, const Derived& b) { return !(a < b); }
};

class Money : public Comparable<Money> {
    int cents_;
public:
    friend bool operator==(const Money& a, const Money& b) { return a.cents_ == b.cents_; }
    friend bool operator<(const Money& a, const Money& b)  { return a.cents_ < b.cents_; }
};

// (C++20 起 spaceship 运算符让这种 mixin 大多没必要了)
性能优势：CRTP 编译期分发 → 编译器可以完全 inline，零运行时开销。虚函数有：① vptr 跳转；② 阻碍 inline 优化。
用法："编译期就知道派生类" → CRTP；运行时不知道 → 虚函数。

静态分发 vs 动态分发

同样是"基类调派生实现"，CRTP 把这步从运行期搬到编译期

该用 CRTP 还是虚函数？

✅ 用 CRTP
  • 类型在编译期已知、追求零开销（热路径、数值库）
  • 给一批类注入通用能力（比较运算符、计数、shared_from_this）
  • Mixin / 静态接口约束
🚫 用虚函数
  • 需要运行时把不同子类装进同一容器（异构集合）
  • 跨动态库边界、插件式扩展
  • 代码可读性 > 极致性能，且不在热路径
C++20 起 operator<=>（spaceship）已能自动生成比较运算符 → "注入比较" 这类 CRTP mixin 大多不再需要。

Type Erasure — std::function/std::any 风格接口

核心套路（手写一个简化 std::function） · REALstd::function 内部结构示意

核心思想

把"任意符合接口的类型"统一存入同一个容器/变量，但不需要继承公共基类。

典型代表：std::function、std::any、std::shared_ptr<void>。

价值

  • 解耦使用方与具体类型（无需共同基类）
  • 第三方类型也能"接入"（不能改基类）
  • API 接受任意 callable 而无需模板感染

核心套路（手写一个简化 std::function）

class AnyCallable {
    // 内部抽象基类
    struct Concept {
        virtual ~Concept() = default;
        virtual void invoke() = 0;
        virtual std::unique_ptr<Concept> clone() const = 0;
    };

    // 模板化派生 — 包装具体类型
    template<class F>
    struct Model : Concept {
        F f;
        Model(F f) : f(std::move(f)) {}
        void invoke() override { f(); }
        std::unique_ptr<Concept> clone() const override { return std::make_unique<Model>(f); }
    };

    std::unique_ptr<Concept> pImpl_;

public:
    template<class F>
    AnyCallable(F f) : pImpl_(std::make_unique<Model<F>>(std::move(f))) {}

    void operator()() { pImpl_->invoke(); }
};

// 使用 — 任意 callable 都能装
AnyCallable a = []{ std::cout << "lambda\n"; };
struct F { void operator()() { std::cout << "functor\n"; } };
AnyCallable b = F{};
void f() {} AnyCallable c = f;

✅ 用类型擦除

  • API 想接受"任意 callable / 任意可打印"而不让模板感染调用方
  • 第三方类型无法改继承关系，但符合接口

  • 类型集合编译期已知 → std::variant 更省（无堆、无虚调用）
  • 本就有公共基类 → 直接用虚函数更简单
  • 极致性能热路径 → 一次间接调用 + 可能堆分配的代价要掂量

Observer / Signal-Slot — 事件订阅

现代 C++ 实现 — 用 std::function 替代 IObserver 接口

template<class... Args>
class Signal {
    using Slot = std::function<void(Args...)>;
    std::vector<std::pair<size_t, Slot>> slots_;
    size_t next_id_ = 0;
    std::shared_mutex mu_;
public:
    size_t connect(Slot s) {
        std::unique_lock lk{mu_};
        auto id = next_id_++;
        slots_.emplace_back(id, std::move(s));
        return id;
    }
    void disconnect(size_t id) {
        std::unique_lock lk{mu_};
        std::erase_if(slots_, [id](auto& p){ return p.first == id; });
    }
    void emit(Args... args) {
        std::shared_lock lk{mu_};
        for (auto& [_, s] : slots_) s(args...);
    }
};

// 使用
Signal<int, std::string> on_login;
auto id = on_login.connect([](int uid, const std::string& n){
    LOG("user {} ({}) logged in", uid, n);
});
on_login.emit(42, "alice");

    // 禁拷贝, 允许 move

Observer 何时用，要当心什么

✅ 用

  • 发射方不该知道、也不想知道有谁在听
  • 订阅关系运行时动态增减
⚠️ 当心
  • 生命周期：订阅者先析构却没退订 → 悬空回调（用 RAII Connection）

State — 状态机模式

简单实现 — enum + switch

enum class ConnState : uint8_t { Init, Connecting, Established, Closing, Closed };

class Connection {
    std::atomic<ConnState> state_{ConnState::Init};
public:
    void on_event(Event e) {
        switch (state_.load()) {
            case ConnState::Connecting:  handle_connecting(e); break;
            case ConnState::Established: handle_established(e); break;
            // ...
        }
    }
    bool try_transition(ConnState from, ConnState to) {
        return state_.compare_exchange_strong(from, to);
    }
};

用 std::variant 实现强类型状态机

struct Init {};
struct Connecting { std::chrono::steady_clock::time_point start; };
struct Established { std::string peer; int fd; };
struct Closed { int reason; };

using ConnState = std::variant<Init, Connecting, Established, Closed>;

class Connection {
    ConnState state_{Init{}};
public:
    void on_event(Event e) {
        std::visit([this, &e](auto& st) {
            using T = std::decay_t<decltype(st)>;
            if constexpr (std::is_same_v<T, Init>) {
                if (e.type == EventType::Connect) state_ = Connecting{now()};
            } else if constexpr (std::is_same_v<T, Connecting>) {
                if (e.type == EventType::Done)
                    state_ = Established{e.peer, e.fd};
            } // ...
        }, state_);
    }
};
每个状态携带各自需要的数据（peer/fd 只在 Established 才有）→ 类型上禁止访问无关字段，比 enum + 类成员清晰得多。

Template Method — 固定流程骨架

基类定下「步骤顺序」，子类只填空 —— 取材自真实场景基类

核心思想

基类把流程顺序写死（如：准备 → 执行 → 成功收尾 / 失败收尾），把其中会变的步骤留成虚函数让子类实现。

何时用

  • 一批操作骨架相同、细节不同（典型：各种"场景/事务"）
  • 想强制所有子类都走同样的收尾 / 错误处理
  • 注意：继承耦合强，步骤少时用「传函数」更轻

CASE代理项目：所有「场景」共享同一生命周期

设备代理里几十个操作都建模成 Scenario：基类 BaseScenario 固化了执行与收尾的骨架，并暴露 tearDownOk() / tearDownError() 两个钩子；每个具体场景只 override 自己的收尾逻辑。下面方法名取自真实基类，去掉了内部类型。

// 基类：固化"成功收尾 / 失败收尾"两个扩展点
class BaseScenario {
public:
    virtual ~BaseScenario() = default;
protected:
    virtual void tearDownOk() = 0;       // 子类填空
    virtual void tearDownError() = 0;    // 子类填空

UML statechart / Miro Samek 的核心思想 — 解决"扁平状态机状态爆炸"问题

动机：扁平状态机的痛

普通 FSM (上一张) 有 N 个状态、M 个事件 → 最坏情况 N×M 个转移。一个电话机 HSM 的状态可能：

  • 挂机 / 忙 / 振铃 / 拨号 / 通话中 / 来电中 / 静音 / 免提 ...
  • "按音量键"在每一个非挂机状态下都要处理一遍 → 重复代码

HSM 把状态组织成树：子状态继承父状态的事件处理。父状态处理"按音量键"一次即可。

扁平 FSM — 状态爆炸

// 简化 HSM 实现 — 用 std::variant + 静态父子映射
enum class St { Active, Idle, OffHook, Dialing, Talking, Ringing };

// 父表 — 编译期常量
constexpr St parent_of[] = {
    St::Active,    // Active 自身是根
    St::Active,    // Idle → Active
    St::Active,    // OffHook → Active
    St::OffHook,   // Dialing → OffHook
    St::OffHook,   // Talking → OffHook
    St::Active,    // Ringing → Active
};

class Phone {
    St state_ = St::Idle;
public:
    void dispatch(Event e) {
        St s = state_;
        while (s != St::Active) {                  // 子→父冒泡
            if (handle(s, e)) return;             // 当前层吃了事件
            s = parent_of[static_cast<int>(s)];   // 向上传
        }
        handle(St::Active, e);                  // 根层兜底
    }

    void transit(St target) {
        // 1. 计算 LCA (最近公共祖先)
        // 2. 从 state_ 一路 exit 到 LCA
        // 3. 从 LCA 一路 entry 到 target
        exit_path(state_, lca(state_, target));
        entry_path(lca(state_, target), target);
        state_ = target;
    }
};

Visitor — std::variant + std::visit 现代实现

更优雅：Overloaded helper

GoF 旧写法 — 双重分派

class Visitor {
public:
    virtual void visit(Circle&)   = 0;
    virtual void visit(Square&)   = 0;
    virtual void visit(Triangle&) = 0;
};
class Shape {
public:
    virtual void accept(Visitor&) = 0;
};
class Circle : public Shape {
public:
    void accept(Visitor& v) override { v.visit(*this); }

问题：① 加新形状必须改所有 Visitor；② 类层次膨胀；③ 调用复杂。

C++17 现代写法

struct Circle   { double r; };
struct Square   { double a; };
struct Triangle { double b, h; };

using Shape = std::variant<Circle, Square, Triangle>;

double area(const Shape& s) {
    return std::visit([](auto&& x) -> double {
        using T = std::decay_t<decltype(x)>;
        if constexpr (std::is_same_v<T, Circle>)
            return 3.14 * x.r * x.r;
        else if constexpr (std::is_same_v<T, Square>)
            return x.a * x.a;
        else
            return 0.5 * x.b * x.h;
    }, s);
}

更优雅：Overloaded helper

template<class... Fs> struct Overloaded : Fs... { using Fs::operator()...; };
template<class... Fs> Overloaded(Fs...) -> Overloaded<Fs...>;       // CTAD

double area(const Shape& s) {
    return std::visit(Overloaded{
        [](const Circle& c)   { return 3.14 * c.r * c.r; },
        [](const Square& sq)  { return sq.a * sq.a; },
        [](const Triangle& t){ return 0.5 * t.b * t.h; }
    }, s);
}

✅ variant + visit

  • 常加新操作而很少加新类型 → 写个新 visitor 即可
  • 想要值语义、无堆分配、编译期穷尽检查
🚫 用虚函数 / 类层次
  • 类型由第三方 / 插件开放扩展（编译期不知道全集）
  • 常加新类型而操作稳定 → 继承体系更顺
  • 需要跨动态库边界多态

Iterator — 自定义容器的迭代器

STL 的 Iterator 是核心抽象

STL 的 Iterator 是核心抽象。给自定义容器加 begin/end/iterator → 自动获得 range-for / 算法支持。

class RingBuffer {
    std::vector<int> data_;
    size_t head_ = 0, size_ = 0, cap_;
public:
    class iterator {
        RingBuffer* rb_;
        size_t i_;
    public:
        using iterator_category = std::forward_iterator_tag;
        using value_type = int;
        using difference_type = std::ptrdiff_t;
        using reference = int&;
        using pointer = int*;

        iterator(RingBuffer* rb, size_t i) : rb_(rb), i_(i) {}
        int& operator*() const { return rb_->data_[(rb_->head_ + i_) % rb_->cap_]; }
        iterator& operator++() { ++i_; return *this; }
        bool operator==(const iterator& o) const { return i_ == o.i_; }
        bool operator!=(const iterator& o) const { return i_ != o.i_; }
    };
    iterator begin() { return {this, 0}; }
    iterator end()   { return {this, size_}; }
};

Producer-Consumer — 经典并发模型

用

用有界队列 + mutex + cv 解耦生产者和消费者速率。

template<class T>
class BoundedQueue {
    std::queue<T> q_;
    std::mutex mu_;
    std::condition_variable cv_not_full_, cv_not_empty_;
    size_t cap_;
    bool closed_ = false;
public:
    explicit BoundedQueue(size_t cap) : cap_(cap) {}

    bool push(T v) {
        std::unique_lock lk{mu_};
        cv_not_full_.wait(lk, [this]{ return closed_ || q_.size() < cap_; });
        if (closed_) return false;
        q_.push(std::move(v));
        cv_not_empty_.notify_one();
        return true;
    }

    std::optional<T> pop() {
        std::unique_lock lk{mu_};
        cv_not_empty_.wait(lk, [this]{ return closed_ || !q_.empty(); });
        if (q_.empty()) return std::nullopt;            // closed + drained
        T v = std::move(q_.front()); q_.pop();
        cv_not_full_.notify_one();
        return v;
    }

    void close() {
        std::lock_guard lk{mu_};
        closed_ = true;
        cv_not_empty_.notify_all();
        cv_not_full_.notify_all();
    }
};

// 使用
BoundedQueue<Job> q{1024};
std::thread producer{[&]{ while (more) q.push(make_job()); }};
std::thread consumer{[&]{
    while (auto j = q.pop()) process(*j);
}};

有界队列：用"满/空"两个条件变量解耦速率

队列把两端的速率波动吸收掉：快的一方被反压、慢的一方不空转
高性能场景考虑 lock-free MPMC queue（如 moodycamel::ConcurrentQueue）。

Active Object — 异步消息驱动对象

把对象的每个公开方法变成"投递到队列的消息"

把对象的每个公开方法变成"投递到队列的消息"，由内部专属线程顺序执行 → 无需锁就能保证状态一致。

class ActiveSession {
    std::queue<std::function<void()>> q_;
    std::mutex mu_;
    std::condition_variable cv_;
    std::jthread worker_;
    bool stop_ = false;

    // 内部状态 - 只在 worker 线程访问，不需要锁
    int state_ = 0;

public:
    ActiveSession() : worker_([this](std::stop_token st){ loop(st); }) {}

    // 同步接口 → 投递为异步消息 + future 等待
    std::future<int> get_state() {
        auto p = std::make_shared<std::promise<int>>();
        auto f = p->get_future();
        enqueue([this, p]{ p->set_value(state_); });
        return f;
    }
    void increment() { enqueue([this]{ ++state_; }); }

private:
    void enqueue(std::function<void()> f) {
        { std::lock_guard lk{mu_}; q_.push(std::move(f)); }
        cv_.notify_one();
    }
    void loop(std::stop_token st) {
        std::stop_callback cb(st, [this]{ cv_.notify_all(); });
        while (!st.stop_requested()) {
            std::unique_lock lk{mu_};
            cv_.wait(lk, [this, &st]{ return !q_.empty() || st.stop_requested(); });
            if (st.stop_requested() && q_.empty()) return;
            auto task = std::move(q_.front()); q_.pop();
            lk.unlock();
            task();
        }
    }
};

调用 = 投信；专属线程顺序处理邮箱

状态只被那一个 worker 线程访问 → 无需加锁；同步结果通过 future 取回
类似 Erlang Actor / Akka — 每个 Active Object 一个邮箱 + 一个线程 + 自有状态。没有锁，但有 future 延迟开销。

// Scope Guard — 通用清理
template<class F>
class ScopeExit {
    F f_;
    bool active_ = true;
public:
    explicit ScopeExit(F f) : f_(std::move(f)) {}
    ~ScopeExit() noexcept { if (active_) try { f_(); } catch (...) {} }
    void dismiss() noexcept { active_ = false; }
    ScopeExit(const ScopeExit&) = delete;
};
template<class F> ScopeExit(F) -> ScopeExit<F>;

// 使用

// ... 任何抛点都能保证 close ...

    rollback.dismiss();              // 成功 → 取消回滚

copy-and-swap 惯用法

class String {
    char* data_ = nullptr;
    size_t size_ = 0;
public:
    // 1. 写好 swap (noexcept!)
    void swap(String& o) noexcept {
        using std::swap;
        swap(data_, o.data_); swap(size_, o.size_);
    }

    // 2. 拷贝构造 (异常时不影响 *this)
    String(const String& o) : data_(new char[o.size_]), size_(o.size_) {
        std::memcpy(data_, o.data_, size_);
    }

    // 3. 唯一的赋值 — 同时实现拷贝赋值 + 移动赋值
    String& operator=(String rhs) noexcept {       // ⭐ 按值收
        swap(rhs);
        return *this;
    }
};

// 调用方:
String a, b;
a = b;                   // 拷贝构造 rhs (可能抛) → swap (noexcept) → rhs 析构 (旧资源)
a = std::move(b);        // 移动构造 rhs (noexcept) → swap → rhs 析构

4 个好处：

  • ① 强异常安全保证（任何抛都不影响 *this）
  • ② 一份代码搞定拷贝赋值 + 移动赋值（重载消除）
  • ③ 自动处理自赋值 (a = a)
  • ④ 简洁、不易写错

    // User(std::string&& n) : name_(std::move(n)) {}

什么时候用

  • ✅ 参数最终要存入成员（"sink" 参数）
  • ✅ 类型有高效的移动构造（vector / string / unique_ptr）
  • ❌ 仅"读用"的参数 → 用 const T&
  • ❌ 类型移动开销 = 拷贝开销（int / 小 POD） → 直接按值传

Q1 · 实现一个线程安全的 Singleton (4 种)

// 1. Meyers Singleton — C++11 后首选
Singleton& get() {
    static Singleton inst;                  // 标准保证 thread-safe 且仅一次
    return inst;
}

// 2. std::call_once
static std::once_flag flag;
static std::unique_ptr<Singleton> inst;
Singleton& get() {
    std::call_once(flag, []{ inst.reset(new Singleton); });
    return *inst;
}

// 3. DCLP (Double-Checked Locking) + atomic — C++11 起合法
static std::atomic<Singleton*> inst{nullptr};
static std::mutex mu;
Singleton* get() {
    Singleton* p = inst.load(std::memory_order_acquire);
    if (!p) {
        std::lock_guard lk(mu);
        p = inst.load(std::memory_order_relaxed);
        if (!p) { p = new Singleton; inst.store(p, std::memory_order_release); }
    }
    return p;
}

// 4. 老 DCLP（C++03 无 atomic，编译器/CPU 会重排）—— 仅作反例，不要写

加分：几乎永远选 1。call_once 跨 DLL 边界更稳；DCLP 仅在性能极敏感场合且需自己管析构时考虑。能讲出 SIOF (Static Init Order Fiasco) 与 Meyers Singleton 的关系再加分。

Q2 · 什么是 RAII? 举 3 个例子

答：Resource Acquisition Is Initialization — 把资源生命周期绑定到栈对象生命周期：构造函数获取资源，析构函数释放资源。任何路径（含异常）下析构都会被调用，保证不泄漏。

  • std::unique_ptr / shared_ptr — 管理堆内存
  • std::lock_guard / scoped_lock — 管理 mutex
  • std::fstream / jthread — 管理文件 / 线程 join

加分：能说出"RAII 是 C++ 优于 GC 语言的根本之一 — 资源释放点确定 + 异常安全免费"。Java try-with-resources / Python with、Rust Drop 都是模仿品。

Q3 · PIMPL 解决什么问题? 代价?

答：Pointer to IMPLementation — 把私有成员藏到 .cpp 里的 Impl 类，头文件只持有 std::unique_ptr<Impl>。

解决：① 编译防火墙（改私有成员不触发用户重编译）；② 二进制兼容 ABI（私有数据升级不破 ABI）；③ 隐藏依赖（Impl 用到的 third-party 头不暴露）。

代价：① 每次访问多一次堆间接 + cache miss；② 额外堆分配；③ 析构必须放 .cpp（unique_ptr 需 complete type），构造、拷贝、move 也常要外置 — 写起来啰嗦。

加分：Qt 全家桶用 PIMPL（Q_D / Q_Q 宏自动生成）；Windows COM 接口本质也是 PIMPL 思想。

Q4 · CRTP 是什么? 用一个例子说明

答：Curiously Recurring Template Pattern — 派生类把自己作为模板参数传给基类：Derived : Base<Derived>。基类编译期就知道派生类型 → 用 static_cast<Derived*>(this) 静态分发，无 vptr / 无堆 / 零开销。

template<class D>
struct Shape {
    double area() const {
        return static_cast<const D*>(this)->area_impl();   // 编译期分发
    }
};
struct Circle : Shape<Circle> {
    double r;
    double area_impl() const { return 3.14 * r * r; }
};

典型用途：① 静态多态（性能敏感场景）；② 给派生类注入功能 (mixin)；③ 标准库 std::enable_shared_from_this。

Q5 · Strategy 模式在现代 C++ 怎么实现?

答：4 种方案，按现代化程度排序：

方案开销切换时机适用
虚函数 + 抽象基类（GoF 原版）vptr + 可能堆运行时经典 OOP，类型多
std::function 成员type erasure + 可能堆运行时单一回调，写得最简洁
模板参数（policy-based design）零开销（编译期内联）编译期性能敏感（STL sort 的 cmp）

加分：标准库 std::sort(it, it, cmp) 中的 cmp 就是 Strategy（模板参数版）；STL 容器的 Allocator 也是 policy-based Strategy。能讲出"用 lambda + std::function 替代 90% 的 GoF Strategy 模板代码"。

Q6 · 为什么 Meyers Singleton 在 C++11 后是首选?

答：C++11 标准 [stmt.dcl]/4 正式规定：函数内的 static 局部变量初始化是线程安全的（其他线程必须等待初始化完成）—— 编译器用类似 once_flag 的机制实现。

  • 零代码 —— 一行 static Singleton inst; 完事
  • 零开销 —— 初始化后访问就是普通函数调用（编译器一次 acquire load 即可）
  • 延迟初始化 —— 第一次调用 get() 才构造
  • 析构有序 —— 程序退出时按构造逆序析构

加分：知道 C++03 时代 Meyers Singleton 不是 thread-safe，需 DCLP 兜底；能讲 SIOF（跨翻译单元的全局 static 构造顺序不定）—— Meyers Singleton 用函数局部 static 自动规避。

Q7 · std::function 怎么实现的? (type erasure)

答：核心三件套：

  1. 抽象基类：struct Base { virtual R call(A...) = 0; virtual ~Base(); };
  2. 模板派生包装：template<class F> struct Impl : Base { F f; R call(A... a) override { return f(a...); } };
  3. 父指针擦除：std::unique_ptr<Base> p_; 通过 base 调 virtual

优化：SBO (Small Buffer Optimization) —— callable ≤ 16-24 字节直接存内部 buffer 不堆分配；高性能实现用 manual vtable（函数指针数组）代替 virtual 便于内联。

加分：指出 std::function 拷贝可能抛 bad_alloc（大 lambda 时）；C++23 move_only_function / function_ref 是更轻量替代（不要求拷贝）。

Q8 · copy-and-swap 4 个好处?

Widget& operator=(Widget o) {   // 值传参 → 已拷贝/移动
    swap(o);                    // noexcept
    return *this;
}
  1. 代码少 — 一份 copy ctor 自动派生 copy assign
  2. 强异常安全 — 拷贝在参数构造时若抛，*this 状态未变；swap 是 noexcept 不再抛
  3. 自赋值天然正确 — x = x 先拷贝再 swap，无需 if (&o == this)
  4. 统一处理 copy/move — 值传参，左值走 copy、右值走 move，一份 operator= 同时支持两种

缺点：每次赋值强制构造临时对象 → move-only 类型或赋值极热路径应用经典 if-self-check 写法。

Q9 · variant + visit vs 虚函数 怎么选?

virtualvariant + visit
类型集合开放（外部可继承）封闭（编译期固定）

Expression Problem 法则：类型频繁扩、操作稳定 → virtual；操作频繁扩、类型稳定 → variant + visit。AST / 状态机 / 消息处理是 variant 天然场景。

Q10 · 设计一个 LRU Cache (list + unordered_map)

template<class K, class V>
class LRU {
    size_t cap_;
    std::list<std::pair<K, V>> lst_;                  // 头=最新, 尾=最旧
    std::unordered_map<K, typename decltype(lst_)::iterator> map_;
public:
    explicit LRU(size_t cap) : cap_(cap) {}

    std::optional<V> get(const K& k) {
        auto it = map_.find(k);
        if (it == map_.end()) return std::nullopt;
        lst_.splice(lst_.begin(), lst_, it->second);    // 移到头 O(1)
        return it->second->second;
    }
    void put(K k, V v) {
        if (auto it = map_.find(k); it != map_.end()) {
            it->second->second = std::move(v);
            lst_.splice(lst_.begin(), lst_, it->second);
            return;
        }
        if (lst_.size() == cap_) {
            map_.erase(lst_.back().first);            // 淘汰尾
            lst_.pop_back();
        }
        lst_.emplace_front(k, std::move(v));
        map_[std::move(k)] = lst_.begin();
    }
};

关键点：① get/put 都 O(1)；② splice 不让 map 里的 iterator 失效（list 杀手锏，vector 做不到）；③ 不要用 erase+push_front — 那会导致 iterator 失效，必须重新插入到 map。

Q11 · 简化版 Object Pool (含自动归还)

template<class T>
class Pool {
    std::vector<std::unique_ptr<T>> idle_;
    std::mutex mu_;
public:
    using Handle = std::unique_ptr<T, std::function<void(T*)>>;

    Handle acquire() {
        std::lock_guard lk(mu_);
        std::unique_ptr<T> p = idle_.empty()
            ? std::make_unique<T>() : std::move(idle_.back());
        if (!idle_.empty()) idle_.pop_back();
        return Handle(p.release(), [this](T* raw){      // 自定义 deleter = 归还
            std::lock_guard lk(mu_);
            idle_.emplace_back(raw);
        });
    }
};

加分：① deleter 闭包让 unique_ptr 出作用域自动归还，使用者无感；② 生产环境用 shared_ptr<T>(raw, deleter) 避免 function 类型擦除开销；③ Pool 必须长于 Handle，工业级用 weak_ptr 检查 Pool 还在。

Q12 · BoundedQueue (Producer-Consumer)

template<class T>
class BoundedQueue {
    std::queue<T> q_;  size_t cap_;
    std::mutex mu_;
    std::condition_variable not_full_, not_empty_;
public:
    void push(T x) {
        std::unique_lock lk(mu_);
        not_full_.wait(lk, [&]{ return q_.size() < cap_; });
        q_.push(std::move(x));
        lk.unlock();
        not_empty_.notify_one();
    }
    T pop() {
        std::unique_lock lk(mu_);
        not_empty_.wait(lk, [&]{ return !q_.empty(); });
        T v = std::move(q_.front()); q_.pop();
        lk.unlock();  not_full_.notify_one();
        return v;
    }
};

要点：① 两个 condvar（分别管空/满），避免"消费者唤醒消费者"无效轮询；② 谓词版 wait 自动处理 spurious wake；③ notify 在 unlock 之后，避免被唤醒线程立刻又陷入锁等待；④ 进阶：std::stop_token + wait_until 支持优雅 shutdown。

Q13 · Active Object 异步消息驱动

答：把方法调用变成消息塞进队列，由对象自己的线程串行执行 → 调用者无锁、对象内部无锁。

class ActiveObject {
    BoundedQueue<std::function<void()>> q_;
    std::jthread worker_;
public:
    ActiveObject() : q_(1024), worker_([this](std::stop_token st){
        while (!st.stop_requested()) q_.pop()();    // 阻塞取 + 执行
    }) {}

    template<class F>
    auto submit(F&& f) {
        auto t = std::make_shared<std::packaged_task<decltype(f())()>>(std::forward<F>(f));
        auto fut = t->get_future();
        q_.push([t]{ (*t)(); });
        return fut;
    }
};

工程对应：Erlang Actor / Java AKKA / C# Dispatcher / ROS callback queue。优势：对象状态天然线程安全（单线程串行）；劣势：单线程瓶颈，工业级需 sharding 多个 ActiveObject 分流。

Q14 · type erasure 简化版 std::function

template<class Sig> class Func;
template<class R, class... A>
class Func<R(A...)> {
    struct Base { virtual R call(A...) = 0; virtual ~Base() = default; };
    template<class F> struct Impl : Base {
        F f;
        Impl(F x) : f(std::move(x)) {}
        R call(A... a) override { return f(std::forward<A>(a)...); }
    };
    std::unique_ptr<Base> p_;
public:
    template<class F>
    Func(F f) : p_(std::make_unique<Impl<F>>(std::move(f))) {}
    R operator()(A... a) { return p_->call(std::forward<A>(a)...); }
};

加分：① 三要素 = 抽象基类 / 模板派生包装 / 父指针擦除；② 加 bool empty() 状态；③ SBO：union{ aligned_storage; Base*; } + flag，< 24 字节内联；④ 拷贝需 virtual unique_ptr<Base> clone() —— 这正是 std::function 比 move_only_function 慢的原因。

Q15 · CRTP vs 虚函数 — 什么时候用哪个?

维度CRTP（静态多态）virtual（动态多态）
派生类型决定时机编译期运行期
调用开销零（可完全内联）vtable lookup（间接调用）
异构容器存多种❌ 模板实例化是不同类型✅ 同一基类指针
ABI 稳定 / 库边界❌ 模板每次实例化✅ 二进制兼容好
编译时间 / 错误信息慢 / 长 / 难懂快 / 清晰

CRTP 优于虚函数：性能极敏感（STL 算法策略、Eigen 表达式模板）、静态多态足够、需要给派生类注入功能（mixin，如 enable_shared_from_this）。

虚函数优于 CRTP：需要异构容器（如 vector<Shape*>）、类型由配置/运行时决定（plugin / 工厂）、库 ABI 稳定优先。

加分："Static Polymorphism vs Dynamic Polymorphism 是 C++ 独有的二元设计选择，其他大多数 OOP 语言只有后者"。Eigen / Boost.Asio executor / Ranges adaptor 全部基于 CRTP 静态多态。
