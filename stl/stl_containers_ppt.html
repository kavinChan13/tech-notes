STL 容器全景与分类

先建立分类心智模型，后续每张 slide 才有锚点。

总览

STL 容器全景图

先建立分类心智模型，后续每张 slide 才有锚点。

① 序列容器 (Sequence)

  • vector — 动态数组 (push_back O(1) 摊销)
  • array (C++11) — 编译期固定大小
  • deque — 双端队列 (分段连续)
  • list — 双向链表
  • forward_list (C++11) — 单向链表
  • string — 实质上是 vector<char> 的特化（含 SSO）

② 关联容器 (Associative · 红黑树)

  • set / multiset — 元素本身有序
  • map / multimap — key→value 有序
  • 底层：红黑树（自平衡 BST）
  • 所有操作 O(log n)，迭代有序

③ 无序关联容器 (Unordered · 哈希表) C++11

  • unordered_set / unordered_multiset
  • unordered_map / unordered_multimap
  • 底层：拉链法哈希表（vector<list<...>>）
  • 平均 O(1)，最坏 O(n)；不保序

④ 容器适配器 (Adapter)

  • stack — 默认底层 deque（LIFO）
  • queue — 默认底层 deque（FIFO）
  • priority_queue — 底层 vector + heap 算法
  • 不提供 begin/end 迭代器！
面试常问 "vector 和 list 怎么选?" 这是最基础的考察点 — 不要只答「随机访问 vs 插入快」，要说缓存友好性 + 链表节点开销 + 现代 CPU 上 vector 几乎全胜。

vector — 三指针模型与扩容

内存布局示意

序列

vector ① — 三指针模型 + 扩容策略

内部结构（libstdc++ / libc++ 一致）

template<class T>
class vector {
    T* _begin;       // 数据起始
    T* _end;         // 已用元素之后 (size 终点)
    T* _end_cap;     // 已分配容量之后
};
// size()     = _end - _begin
// capacity() = _end_cap - _begin
// sizeof(vector<int>) = 24 bytes (64-bit, 三指针)

扩容（关键面试点）

  • 容量不够 → 分配新一段 → 把旧元素移动/拷贝过去 → 释放旧的
  • 增长因子：libstdc++ = 2x，libc++ / MSVC = 1.5x
  • 2x 简单但内存利用率差；1.5x 让新空间可能复用旧释放的内存（黄金分割理论）
  • 复杂度：单次最坏 O(n)，摊销 O(1)（几何级数）

内存布局示意

关键：resize(n) 改变 size（构造或析构元素）；reserve(n) 只改变 capacity（仅分配，不构造）。capacity 永远只增不减，除非调用 shrink_to_fit()（非强制）。
面试问"为什么 1.5 优于 2"：
2x 时新分配 = 之前所有累计容量之和 + 1，无法复用之前释放的内存（连续分配的内存地址递增）；1.5x 在 N 次扩容后新需要的容量 ≤ 之前累计释放的容量，allocator 可能把它们合并复用 → 内存利用率更好。

vector — 操作复杂度 + emplace vs push_back

操作复杂度 · emplace_back vs push_back（高频面试题）

序列

vector ② — 操作复杂度 + emplace 真相

操作复杂度

操作复杂度会引起扩容/失效?注意
operator[] / atO(1)否at 越界抛 out_of_range
push_back / emplace_backO(1) 摊销可能cap 满才扩容
pop_backO(1)仅尾后迭代器失效不释放容量
insert / emplace (中部)O(n)是移动后续元素
eraseO(n)当前及之后使用 erase-remove 惯用法
clearO(n)所有迭代器失效不释放容量
resize(n)O(|n - size|)可能 (n > cap)构造/析构元素
reserve(n)O(n) 若扩容是 (n > cap)不构造元素
shrink_to_fitO(n)是非强制；通常会重新分配

emplace_back vs push_back（高频面试题）

区别本质

push_back(x)：要求 x 已经是 T；可能拷贝或移动。

emplace_back(args...)：在原地转发构造 — 直接在 vector 内存上 placement new T(args...)，避免临时对象。

什么时候真的有差别

  • 构造一个新对象时 → emplace 省一次临时对象 + 移动
  • 已经有现成对象时 → 几乎无差别
  • 注意：emplace_back 会绕过explicit 构造保护，可能被滥用
struct Conn { Conn(int fd, std::string ip); };

std::vector<Conn> v;
v.push_back(Conn{fd, ip});           // ① 临时 Conn ② 移动到 vector ③ 析构临时
v.emplace_back(fd, ip);                // 直接在 vector 内构造，省 ①②③

std::vector<Conn> v2;
Conn c{fd, ip};
v2.push_back(std::move(c));            // 移动构造 1 次
v2.emplace_back(std::move(c));         // 也是移动构造 1 次（无差别）
面试加分：能说出"reserve 后 emplace_back 才是性能巅峰" — 因为 reserve 消除扩容拷贝，emplace 消除临时对象，两者结合 = 理论最优。

vector — 迭代器失效完全规则 ★

典型 BUG 与正确写法

序列⭐ 面试必考

vector ③ — 迭代器失效完全规则 ★

操作失效范围原因
push_back 未触发扩容仅 end()新元素加在末尾后
push_back 触发扩容全部迭代器、指针、引用底层重新分配，旧地址失效
insert(pos, x) 未扩容pos 及之后所有迭代器后续元素被向后挪
insert(pos, x) 扩容全部同上 + 重新分配
erase(pos)pos 及之后所有迭代器后续元素向前挪
clear()全部（end 仍可用）所有元素析构，size=0
reserve(n) n > cap全部重新分配
resize(n) n > cap全部同上
shrink_to_fit()全部（可能）重新分配到刚好 size
swap(other)不失效（指向被交换前的元素）仅交换三个指针

典型 BUG 与正确写法

// ❌ BUG: erase 之后 it 已失效
for (auto it = v.begin(); it != v.end(); ++it) {
    if (*it < 0) v.erase(it);          // UB: 之后 ++it 越界
}

// ✅ 正确：用 erase 返回的下一个迭代器
for (auto it = v.begin(); it != v.end(); ) {
    if (*it < 0) it = v.erase(it);
    else ++it;
}

// ✅ 更现代：erase-remove (C++20 起 std::erase_if 一行)
v.erase(std::remove_if(v.begin(), v.end(), [](int x){ return x < 0; }), v.end());
std::erase_if(v, [](int x){ return x < 0; });           // C++20
记忆口诀：vector 一旦容量变化 → 全部失效；位置变化（insert/erase）→ 改动点之后失效。

string — SSO 小字符串优化

实现示意 (libc++ 风格) · 对工程的实际影响

序列

std::string — SSO 小字符串优化

核心概念

SSO (Small String Optimization)：短字符串直接存在 string 对象内部 buffer，不分配堆内存。

避免大量短字符串场景的频繁 malloc/free，性能巨大提升。

历史上：g++ 5 之前 string 是 COW（写时复用），但与 C++11 多线程冲突 → 改为 SSO。

各实现的阈值

  • libstdc++: SSO 阈值 ≤ 15 字节，sizeof(string) = 32
  • libc++: ≤ 22 字节，sizeof(string) = 24（更激进）
  • MSVC: ≤ 15 字节，sizeof(string) = 32

实现示意 (libc++ 风格)

class string {
    union {
        struct __long {
            size_t cap;
            size_t size;
            char*  data;
        };
        struct __short {
            unsigned char size : 7;     // 1-bit 标志位 + 7-bit size
            unsigned char is_long : 1;
            char data[23];                // 直接内嵌
        };
    };
};

对工程的实际影响

SSO 受益的场景

  • HTTP header name (大多 < 22 字节)
  • map 的 key 是短 string
  • 日志 tag、metric 名字
  • 配置项 key

注意事项

  • SSO 字符串移动 = 拷贝（数据在对象内部，没办法只 swap 指针）
  • 故 std::move 短 string 不省事，长 string 才有效
  • 跨 SSO 阈值的扩展会触发一次 malloc + memcpy
// 验证你的实现的 SSO 阈值
std::string s = "hello";
std::cout << (void*)s.data() << " " << (void*)&s << '\n';
// 如果 data() 地址在 &s 附近 → 走 SSO
面试可问"为什么 g++ 放弃了 COW string?" — 答：① C++11 要求 operator[] 不抛异常 + 多线程下 const 操作必须线程安全，COW 的 ref-count 增加复杂度；② 现代多核场景，原子计数开销甚至超过节省的内存。

array / span — 固定大小与视图

REAL实战示例

序列

std::array (C++11) & std::span (C++20)

std::array<T, N>

  • 编译期固定大小，栈上分配
  • 替代 C 数组：有 size()/at()/begin()/end()，可拷贝、可与 std 算法配合
  • 零开销：sizeof(array<int,4>) == sizeof(int[4]) == 16
  • 支持结构化绑定 + CTAD (C++17)

std::span<T>

  • {T*, size_t} 的非拥有视图（"轻引用"）
  • 统一接受：C 数组、std::array、std::vector、子段
  • 替代 C 风格 (T* p, size_t n) 双参
  • 不延长生命周期，调用方负责

REAL实战示例

// 协议字段固定大小 → array
struct EthHeader {
    std::array<uint8_t, 6> dst_mac;
    std::array<uint8_t, 6> src_mac;
    uint16_t ether_type;
};
static_assert(sizeof(EthHeader) == 14);

// 通用解析函数 → span（接收 vector / array / 裸指针）
bool parse(std::span<const uint8_t> data) {
    if (data.size() < 14) return false;
    auto hdr = data.first(14);                      // O(1) 子段
    auto body = data.subspan(14);
    // ...
    return true;
}

// 调用
char buf[1500];
std::vector<uint8_t> v = recv_packet();
std::array<uint8_t, 64> small;

parse({(const uint8_t*)buf, 1500});   parse(v);   parse(small);   // ✅ 全部兼容

deque — 分段连续的双端队列

内部布局 · vector vs deque 选型

序列

std::deque — 分段连续的双端队列

核心结构

不是单一连续内存，而是「中央 map（指针数组）+ 多段固定大小 chunk」。

每个 chunk 通常 4KB（一页），里面元素连续；chunk 之间不连续。

支持O(1) 双端 push/pop；O(1) 随机访问（要算两次跳转）。

但中部 insert/erase 是 O(n)（与 vector 类似）。

什么时候用

  • 需要前后都能 push（vector 不能 push_front）
  • 不需要长时间稳定的指针（push_front/back 不会让已存在元素的指针失效，但迭代器会失效）
  • STL stack / queue 默认底层用 deque

内部布局

vector vs deque 选型

需求选型理由
只在尾部 push/popvector缓存友好，单一内存块
双端 push/popdequevector 不支持 O(1) 头部
追求最高扫描速度vectordeque 跨 chunk 有跳转
大量元素，内存碎片敏感deque不需要单块巨大连续内存
需要稳定指针 (push 不让旧元素地址变)deque (头尾 push)vector 扩容会让所有指针失效
实际工程中 vector 用得远多于 deque。deque 出现的最大场景是「stack/queue 的默认底层」和「需要频繁 push_front 的日志/任务队列」。

list / forward_list — 链表家族

仍然合适的场景（少而具体）

序列

std::list & std::forward_list — 几乎不该用的链表

结构

  • list：双向链表，每节点有 prev/next/T 三部分
  • forward_list(C++11)：单向链表，每节点 next/T
  • 每节点单独 malloc → 内存不连续，缓存极不友好

为什么"几乎不该用"

  • 每个 int 节点开销 = 24 字节（vs vector 的 4 字节）
  • 遍历比 vector 慢 10~50 倍（cache miss）
  • insert/erase O(1) 的优势在已知位置下才生效；查找仍是 O(n)
  • Bjarne Stroustrup 著名实验：1 万元素插入排序，vector 全胜 list

仍然合适的场景（少而具体）

  • 稳定的迭代器/指针：节点不会因为其他元素 insert/erase 而失效（除了被删除的节点本身）
  • splice O(1)：把一个 list 的子段直接接到另一个，不拷贝节点内容
  • 大型对象（移动昂贵）的链表式调度队列（极少）
  • 侵入式链表（intrusive list）— 但那一般是手写
// list 唯一显著优势：splice — O(1) 不拷贝
std::list<Task> ready, runnable;
runnable.splice(runnable.end(), ready, ready.begin());     // 把 ready 第一个节点接到 runnable

// 迭代器稳定性：以下场景 it1 始终有效
std::list<int> l{1,2,3};
auto it1 = std::next(l.begin());           // 指向 2
l.push_front(0); l.push_back(99);             // it1 仍指向 2
面试问"vector 和 list 怎么选?"答：默认 vector，仅当需要稳定迭代器/splice/超大元素的链表式调度时考虑 list。"链表查找 O(1)" 是错觉 — 必须先 O(n) 找到位置才能 O(1) 插。

map / set — 红黑树原理

常用操作 · 迭代器失效（很温和）

关联

std::map & std::set — 红黑树家族

核心概念

底层都是红黑树（Red-Black Tree），一种自平衡 BST。

所有操作 严格 O(log n)，元素按 key 自然有序。

map: key → value 映射；set: 仅 key（去重）。

multimap/multiset：允许相同 key。

红黑树性质（5 条，面试可能让背）

  1. 每个节点是红或黑
  2. 根节点是黑
  3. 叶子（NIL）是黑
  4. 红节点的子节点必须是黑（不能两红相邻）
  5. 从任一节点到其后代叶子的路径上黑节点数相同

→ 树高最大 2 * log2(n+1)

常用操作

std::map<std::string, int> m;
m["a"] = 1;                         // O(log n) 插入或访问，注意：会默认构造不存在的 key

// 安全访问：
if (auto it = m.find("b"); it != m.end()) use(it->second);

// 不存在则插入，存在则跳过 — 用 try_emplace（C++17）
m.try_emplace("a", 999);              // 不会更新已有的 1

// 不存在则插入，存在则更新 — insert_or_assign（C++17）
m.insert_or_assign("a", 999);

// 范围查找（红黑树特有的有序优势）
auto it_low  = m.lower_bound("a");    // 第一个 >= "a"
auto it_high = m.upper_bound("c");    // 第一个 >  "c"
auto [b, e] = m.equal_range("a");    // [lower, upper)

迭代器失效（很温和）

操作失效
insert/emplace不失效任何已存在的迭代器/引用
erase仅被删除元素的迭代器；返回下一个
clear全部
陷阱：m["key"] 在 key 不存在时会插入默认构造的 value，统计场景极易踩坑。只读检查请用 find 或 contains(C++20)。

unordered_map — 哈希表实现

结构示意 · 性能与陷阱

无序⭐ 面试必考

std::unordered_map — 哈希表内部

底层结构

STL 标准要求拉链法（separate chaining）：vector<forward_list> 风格。

每个 bucket 是一个链表，hash 冲突的元素挂在同一个 bucket 链表上。

这意味着：元素地址稳定（rehash 时迭代器失效，但指针/引用仍有效）— 这是与多数语言（如 Java HashMap）不同的关键点。

关键参数

  • load_factor() = size / bucket_count
  • max_load_factor() 默认 1.0；超过即rehash
  • rehash(n)：手动指定至少 n 个 bucket
  • reserve(n)：保证容纳 n 个元素不 rehash

结构示意

性能与陷阱

场景复杂度
find/insert/erase 平均O(1)
极端 hash 冲突 (恶意输入)退化为 O(n)
rehash (load_factor 超阈值)O(n) — 重新分配 bucket，重新 hash 所有元素
面试陷阱：①「unordered_map 比 map 一定快」是错的 — 元素少（< 100）时 map 因为缓存友好可能更快；② string key 的 hash 可能很慢（要遍历全字符串）；③ STL 的 unordered_map 因为稳定地址保证不能用 open addressing，性能不如 absl/flat_hash_map。
高级答法："如果性能敏感，用 absl::flat_hash_map 或 robin-hood-hashing；STL unordered_map 的实现因 ABI 历史包袱（指针稳定性 + bucket 接口）注定无法极致快。"

unordered_map — 自定义 key 和 hasher

正确写法（生产级） · 替代方案：模板参数传 Hasher

无序

unordered_map — 自定义 Key / Hasher

必须满足两点

  1. operator== 或独立 KeyEqual：判等
  2. std::hash<T> 特化 或 自定义 Hasher：算 hash 值

关键约束：a == b ⇒ hash(a) == hash(b)

常见 hash 组合错误

  • 简单 XOR 多个字段：hash(a) ^ hash(b) — 字段相等时退化为 0
  • 正确做法：boost::hash_combine 风格，混入位移

正确写法（生产级）

struct Endpoint {
    std::string ip;
    uint16_t port;
    bool operator==(const Endpoint&) const = default;     // C++20
};

namespace std {
template<> struct hash<Endpoint> {
    size_t operator()(const Endpoint& e) const noexcept {
        size_t h = std::hash<std::string>{}(e.ip);
        // boost::hash_combine 公式（黄金分割数）
        h ^= std::hash<uint16_t>{}(e.port) + 0x9e3779b9 + (h << 6) + (h >> 2);
        return h;
    }
};
}

std::unordered_map<Endpoint, Connection*> conn_table;

替代方案：模板参数传 Hasher

struct EndpointHash {
    size_t operator()(const Endpoint& e) const noexcept { /* ... */ }
};
std::unordered_map<Endpoint, Connection*, EndpointHash> t;
C++14 异构查找：透明 hasher (std::hash<void>) + 透明比较器允许用 string_view 查找 string key 的 unordered_map（C++20 才完整支持，需要 hasher 提供 is_transparent）。

multimap / multiset — 允许重复 key

REAL定时器轮（简化版）— multimap 的经典用法

关联

multimap / multiset — 允许重复 key

与 map/set 的差别

  • 允许多个相同 key
  • 无 operator[]（因为不知该返哪个）
  • 查找用 equal_range 拿到所有匹配
  • count(k) 返回 k 的实际个数
  • 插入按红黑树规则维持有序

典型用途

  • 定时任务表：key=触发时间，value=任务
  • 多对一 mapping：一个 key 多条记录
  • 排行榜：按分数排，相同分数允许多人

REAL定时器轮（简化版）— multimap 的经典用法

class TimerWheel {
    using TimePoint = std::chrono::steady_clock::time_point;
    std::multimap<TimePoint, Task> queue_;
    std::mutex mu_;
public:
    void schedule(TimePoint when, Task t) {
        std::lock_guard lk{mu_};
        queue_.emplace(when, std::move(t));
    }
    void tick() {
        std::lock_guard lk{mu_};
        auto now = std::chrono::steady_clock::now();
        auto end = queue_.upper_bound(now);          // 全部到期任务的边界
        for (auto it = queue_.begin(); it != end; )
            it = queue_.erase(it).first;             // run + erase
    }
};
高性能版本会用 四级时间轮 (hierarchical timing wheel)，把 multimap 替换为环形 bucket 数组。但对中低 QPS 场景，multimap 简单可靠。

priority_queue 的内部实现 (面试高频)

template<class T, class Container = std::vector<T>, class Compare = std::less<T>>
class priority_queue {
    Container c;     // 默认 vector
    Compare comp;
public:
    void push(const T& v) {
        c.push_back(v);
        std::push_heap(c.begin(), c.end(), comp);    // O(log n)
    }
    void pop() {
        std::pop_heap(c.begin(), c.end(), comp);     // O(log n) — 把堆顶移到末尾
        c.pop_back();                                // 移除末尾
    }
    const T& top() const { return c.front(); }
};

常见用法

// 默认大顶堆
std::priority_queue<int> pq;
pq.push(5); pq.push(2); pq.push(8);
pq.top();           // 8

// 小顶堆 — 用 greater
std::priority_queue<int, std::vector<int>, std::greater<int>> min_pq;

// 自定义优先级 (任务调度)
struct Task { int priority; int id; };
auto cmp = [](const Task& a, const Task& b){ return a.priority < b.priority; };  // 大顶
std::priority_queue<Task, std::vector<Task>, decltype(cmp)> tq(cmp);
能讲出"priority_queue 不是新数据结构，是 vector + std::push_heap/pop_heap 包装"= 加分；能写出 用 vector 自己跑 std::heap 算法（更灵活，能批量 build_heap） = 高分。

口诀

记 3 句话

  1. vector / deque（中部）→ 看是否扩容/重新分配
  2. node-based (list/set/map/unordered)：插入不失效已有迭代器
  3. rehash / resize-with-realloc → 全部迭代器失效

「指针 vs 迭代器」失效不同

  • deque 头尾 push → 迭代器失效，指针/引用不失效
  • unordered_* rehash → 迭代器失效，指针/引用不失效（节点内存没动，bucket 索引变了）
典型问："std::vector::push_back 会让所有迭代器失效吗?" — 不一定，取决于是否触发扩容。这种"看似简单实则有陷阱"的问题最容易区分新手和资深。
资深加分：vector::end() 任何修改大小的操作都会失效（因为 end 永远指向 size 的下一个位置）。这是为什么 for (auto it=v.begin(); it!=v.end(); ) 中循环体若改 size 必须重新取 end。

容器与异常安全

实测对比

异常安全⭐

容器与异常安全 — 移动构造为何要 noexcept

STL 的「强异常安全保证」

vector 扩容时承诺：操作要么成功，要么 vector 状态完全不变（强保证）。

问题：扩容要把元素从旧 buffer 转到新 buffer。如果移动构造可能抛异常，转移过程中失败 → 旧 buffer 被破坏，无法回滚。

解决方案

STL 在扩容时检查： noexcept(T(std::move(t))) 是否为 true？

  • true：用移动构造（快）
  • false：用拷贝构造（慢但安全；旧 buffer 可保留作回滚）

实测对比

struct A {
    std::string s;
    A(A&&) /* 没标 noexcept */ = default;
};

struct B {
    std::string s;
    B(B&&) noexcept = default;        // ✅
};

std::vector<A> va; va.reserve(1);
for (int i=0; i<1000000; ++i) va.emplace_back("hello world");
// ↑ 每次扩容都走拷贝路径 → 慢

std::vector<B> vb; vb.reserve(1);
for (int i=0; i<1000000; ++i) vb.emplace_back("hello world");
// ↑ 走移动路径 → 快几个数量级
面试经典问："为什么自定义类的移动构造一定要标 noexcept?" — 答：① STL 容器扩容性能退化成拷贝；② 多线程并发原语（如 std::lock_guard）期待 noexcept；③ 析构函数也应 noexcept（C++11 起默认就是）。

问：std::sort 内部用什么算法？

答：introsort（内省排序）— 开始用 quicksort，递归深度超过 2*log2(n) 时切换到 heapsort 防止最坏 O(n²)；分段规模小（<16）用 insertion sort。这样保证最坏 O(n log n) 且常数因子小。
