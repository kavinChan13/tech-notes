01

为什么需要 Sanitizer · 跟 Valgrind 区别

Sanitizer 是 Google 2012-2014 期间开源的编译期插桩 + 运行期检测方案

Sanitizer 是 Google 2012-2014 期间开源的编译期插桩 + 运行期检测方案。在它之前，C++ 内存错误的主要工具是 Valgrind（动态二进制翻译）。

Valgrind 的根本限制

  • 20-50× slowdown — 二进制翻译 + JIT，单 thread + 解释每条指令
  • 不能检测 stack overflow（虚拟化 stack 复杂）
  • Helgrind 慢 50-100× + 假阳性多
  • 无法跑大型 CI / 部分生产工作负载

Sanitizer 的革命

  • 编译期插桩（CTI）— 每条 memory access 前后插指令，runtime 0 翻译
  • 2-5× slowdown（vs Valgrind 20-50×）
  • 开发 / CI / 部分生产可跑
  • 共享 framework：ASan / TSan / MSan / UBSan / LSan / HWASan / GWP-ASan
设计哲学：Sanitizer 是 "慢 5× 但 production-ready" vs Valgrind "慢 50× 但 dev-only"。这个 5× 差距让 sanitizer 进了 CI、canary deploy、甚至部分 production fleet（GWP-ASan / HWASan）。
02

ASan 实现深入

核心机制：Shadow Memory · Heap Buffer Overflow：redzone · Use-After-Free：Quarantine · Stack Use-After-Return · ASan …

AddressSanitizer (ASan) 是 sanitizer 家族的始祖（Konstantin Serebryany 等，USENIX ATC 2012）。检测：

  • Heap / stack / global buffer overflow / underflow
  • Use-after-free / Use-after-return / Use-after-scope
  • Memory leak（程序退出时统计）
  • Double-free / invalid-free
  • Initialization order bugs（global object 跨 TU 初始化顺序）

核心机制：Shadow Memory

关键 trick：用 1 字节 shadow memory 表示 8 字节真实内存的可访问性。

用户内存（每 8 字节一格）：

[0] [1] [2] [3] [4] [5] [6] [7] ← 8 字节

对应 shadow（1 字节）：

0x00 = 整 8 字节都可访问
0x01-0x07 = 前 N 字节可访问，后 (8-N) 字节是 redzone
0xFA = stack-left-redzone (poison)
0xFB = stack-mid-redzone
0xFD = heap-freed
0xFC = heap-left-redzone
0xFF = stack-use-after-return

Shadow memory mapping

// x86_64 Linux 默认 mapping:
shadow_offset = 0x7fff8000;  // 2^46 - 0x8000
shadow_addr   = (user_addr >> 3) + shadow_offset;

// 用户地址空间 [0x10000000000, 0x7fffffffffff] (~128 TB)
// shadow 占 1/8 = 16 TB（虚拟，按需 page 分配）

插桩指令

编译器在每条 memory access 前生成 ~10 行汇编：

// 原始 C++:
*p = 42;

// ASan 插桩后（伪汇编）:
shadow = (p >> 3) + 0x7fff8000;
shadow_val = load_byte(shadow);
if (shadow_val != 0) {
  if (shadow_val < 0) { __asan_report_store1(p); }
  if ((p & 7) >= shadow_val) { __asan_report_store1(p); }
}
*p = 42;

这就是 ASan ~2× slowdown 的来源——每次内存访问多 ~10 条指令 + 1 次 shadow load。

Heap Buffer Overflow：redzone

关键：每个 heap allocation 前后插入 128 字节 redzone，shadow 标 poison。

// 用户调用: new int[10] (40 bytes)
// ASan 实际分配:

[ left redzone 128B ][ user data 40B ][ padding ][ right redzone 128B ]
       0xFA               0x00              0xFA           0xFA
                    (shadow values)

任何访问 [user, user+40) 外的字节立刻 hit poison shadow → ASan 报警 + stack trace。

🛡️ 交互式演示：点一个字节，看 ASan 查 shadow 判生死
这是一块 new int[4]（16 字节用户数据）外加左右 redzone。点任意格子模拟一次内存访问，ASan 会查它的 shadow 字节并裁决。先试试越界、再点「free 这块」后访问，复现 use-after-free。生活化类比：shadow 就像每段内存门口的"红绿灯"，ASan 每次进门前先看灯。

Use-After-Free：Quarantine

free() 不立即归还内存给 allocator。而是：

  1. 把这块内存放入 quarantine queue（FIFO，默认 256 MB）
  2. shadow 整块标 poison (0xFD)
  3. quarantine 满了 → 最老的 chunk 才真正 free 给 OS

结果：UAF 在 quarantine 期间（~几秒）被检测；超过 quarantine size 才漏。

Stack Use-After-Return

这是 ASan 比 Valgrind 强的点。机制：

  1. 函数 entry：在 fake stack (heap-allocated) 分配 frame，局部对象指针指向 fake stack
  2. 函数 exit：fake stack frame 标 poison (0xF5)，不立即释放
  3. 之后访问该 frame → hit poison → 报警
  4. ~10 个函数 call 后才真正释放 fake stack frame
性能代价：use-after-return 默认 disabled，需 ASAN_OPTIONS=detect_stack_use_after_return=1 显式开启，可使 ASan 慢额外 30-50%。

ASan 的局限性与根因

限制根本原因缓解 / 替代
越界 ≥ redzone 距离才漏（典型 arr[i+1000] 落进下一个合法 alloc） Redzone 固定 128B；远距离跳越 redzone 后落入合法但错误的 shadow（0x00），无法区分 调大 ASAN_OPTIONS=redzone=512（代价：内存放大）；或换 HWASan（tag 不连续，远距离也能 catch）
Use-after-free 超过 quarantine 期漏 Quarantine 是 FIFO 有限队列（默认 256 MB），老 chunk 被真实 free 后 shadow 复位 quarantine_size_mb=4096（牺牲峰值内存）
Intra-object overflow 不检测（同一 struct 内 field_a 写越界踩 field_b） Redzone 只放在整个 alloc 的前后，struct 内字段间共享一段合法 shadow (0x00) 编译加 -fsanitize-address-field-padding=1（ABI break，谨慎用）
Memory exhaustion 风险（容器内 ASan 进程被 OOM kill） 3× 内存放大：shadow 1/8 + redzone + quarantine + fake stack K8s limits ≥ 3× 正常；或用 HWASan / GWP-ASan
自定义 allocator (jemalloc/tcmalloc/对象池) 不被监控 ASan 通过拦截 malloc/free插 redzone；用户级 arena 内 sub-alloc 对 ASan 透明 对象池调用 __asan_poison_memory_region() 手动标 poison；或换成 malloc/free
ASan-instrumented 跟非 instrumented 混链危险 未插桩 code 写入 redzone 不报，下次读时假阳/假阴；跨 .so 调用栈帧布局错乱 全 binary 统一 build；动态库要 ASan-built 版本
不检测读取未初始化内存 ASan shadow 只编码"可访问 / 不可访问"，无"已写入"位 这是 MSan 的工作
不检测 data race 没有 vector clock / happens-before 追踪 TSan
对 mmap / 大块 syscall 缓冲弱 Kernel 路径不经 ASan allocator → 无 redzone 对 syscall 边界手动 __asan_poison_memory_region
SIGSEGV handler 行为变了 ASan 自己装 signal handler 报 stack trace，跟应用自有 crash handler 冲突 ASAN_OPTIONS=handle_segv=0 让出
本质局限：ASan 抓的是空间错误（越界 / UAF / 类型错配），靠 shadow 的"每 8 字节是否合法"。对语义错误（初始化遗漏、race、UB 算术）完全无感。这也是为什么 sanitizer 家族会分裂成 5+ 个工具。

TSan 实现深入

核心算法：FastTrack · 同步原语的处理 · TSan 的局限性与根因

ThreadSanitizer (TSan) 检测 data race——两线程访问同一内存且无 happens-before 关系，至少一方是写。

核心算法：FastTrack

TSan 用 FastTrack 算法（Flanagan & Freund, PLDI 2009）——vector clock 的优化版本。朴素 vector clock 100× slowdown，FastTrack 5-10×。

朴素实现的开销

每个内存位置需要存上次访问者的完整 N-元 vector clock。1000 线程 × 1 GB 内存 = 不可接受。

FastTrack 的优化

洞察：读操作不形成 race——多个并发读没问题，只要不跟写冲突。

访问类型朴素FastTrack
Readfull vector clock1 个 epoch（thread_id + 该 thread 计数）
Writefull vector clockfull vector clock

因为多个 read 之间不会 race，所以只需记最新一次 read（最大 epoch）即可。Read 占大多数 access → FastTrack 5-10× faster than naive。

同步原语的处理

原语vector clock 操作
mutex.lock()my_vc = max(my_vc, mutex.release_vc)
mutex.unlock()mutex.release_vc = my_vc; my_vc[me]++
atomic.load(acquire)my_vc = max(my_vc, atomic.release_vc)
atomic.store(release)atomic.release_vc = my_vc; my_vc[me]++
thread.create()新 thread 的 vc = parent vc
thread.join()my_vc = max(my_vc, joined.vc)
cv.wait + cv.signal类似 mutex pair
陷阱：TSan 不识别用户自定义同步（手写 lock-free / 自己的 ring buffer）。需手动 annotate：__tsan_acquire(addr) / __tsan_release(addr)，否则 TSan 会报大量 false positive race。

TSan 的局限性与根因

限制根本原因缓解 / 替代
只能 catch "实际跑到了"的 race（动态检测，非静态证明） FastTrack 只在 access 真正发生时比 vector clock；没跑到的代码路径不分析 结合 fuzzing / stress test 提升覆盖；静态扫用 RacerD / Coverity
对用户态自旋锁 / lock-free 数据结构假阳性 TSan 只识别 pthread / std::mutex / std::atomic 等已知原语；手写 CAS 循环没 happens-before record __tsan_acquire/release annotate；或用 std::atomic 代替手写 CAS
Vector clock 内存占用爆炸（N thread × M 内存位置） 每写一次都要存 N-元 VC；线程多时 shadow 内存放大 5-10× 限制线程数；分片测试
不能跟 ASan / MSan 同跑 都要独占 shadow 内存区域且 layout 不兼容；runtime allocator 冲突 分 build 分别跑
对 signal handler 内 race 检测弱 Signal 异步打断，VC 更新不在 handler 中插桩 慎在 handler 里改 shared state（也是好实践）
动态加载 .so 时漏检 非 TSan-built .so 的 mutex 操作不 update VC，跨 .so 同步看不到 happens-before 所有 .so 统一 TSan-built
memory_order_relaxed 的真 race 可能漏 relaxed 不建立 happens-before，但 TSan 当 sync 点（避免假阳）→ 反而掩盖真 race 仔细审 relaxed 用法；用 acquire/release 表达意图
Fork 后 child 进程 VC 状态不一致 Fork 复制 VC 但 thread 关系断裂，runtime 不重置 TSAN_OPTIONS=die_after_fork=0 + 手动 reset；或 fork 后立即 exec
5-15× slowdown 让 stress test 难现实化 每次 access 都比 VC + shadow 写入；atomic 越多越慢 分片跑：staging 上低 QPS / 长时间
不检测 deadlock（只 catch race） TSan 算法专设计 race，没建 lock graph 用 Clang ThreadSafety 静态分析；或 deadlock detector（Helgrind/glibc）
本质局限：TSan 是动态 happens-before 检测器——只能看到「这次执行真发生的」race。Race 跟时序强相关，单次跑可能 0 race、再跑出 100 个。所以 TSan 必须长时间 stress + 多种 schedule（thread sanitizer + fuzzer 组合）才有意义。

MSan 实现深入

Bit-level Shadow · "延迟检测"哲学 · Origin Tracking · MSan 的局限性与根因

MemorySanitizer (MSan) 检测 uninitialized memory reads——从未写入过就读。

Bit-level Shadow

MSan shadow 是每 bit 1 个 shadow bit（不是 ASan 的 1:8 byte ratio）：

  • 用户 64-bit int → shadow 64 bit
  • shadow bit = 0 → 该位已初始化（有效）
  • shadow bit = 1 → 该位未初始化（poisoned）

"延迟检测"哲学

int a;          // uninit
int b = a;      // 读 uninit — 但不报，shadow 传播
int c = b + 1;  // c 仍 uninit-tainted
if (c > 0) ...  // 报警！uninit value 影响了 control flow

MSan 只在分支 / 系统调用 / 算术结果传给 caller 等 observable 行为时报警。

Origin Tracking

问题：报警时"X 字节未初始化"不够 actionable——哪个 alloc 没初始化？

MSan 可启用 origin tracking（MSAN_OPTIONS=origin_history_size=...）：

  • 每个 4-byte 内存区域记录"哪个 stack trace 让我 poisoned"
  • 跨 propagation 保留：A 没初始化 → B = A * 2，B 的 origin 也指向 A 的 alloc site
  • 开销：内存 +2×，CPU +30%
MSan 的痛点：MSan 比 ASan / TSan 难采用——整个 binary 必须 MSan-instrumented，包括 libstdc++ / glibc / 第三方 lib。Untrumented code 写入的内存 MSan 不知道 → 大量 false positive。需要 MSan-built libcxx + libstdc++（耗时建）。实战部署 rare，多数项目用 ASan + valgrind memcheck 替代。

MSan 的局限性与根因

限制根本原因缓解 / 替代
整链路必须 MSan-built（最大痛点） Shadow 是每 bit 一位，未插桩代码写内存不会清 shadow → MSan 仍认为 poisoned → 假阳 用 libcxx-msan + 重 build 全部依赖；或对未插桩边界手动 __msan_unpoison()
syscall 返回数据需手动 unpoison Kernel 写 userspace buffer 不经 MSan 插桩 MSan runtime 对常见 syscall (read/recv/...) 有 wrapper；冷门 ioctl 自己写
跟 inline asm 不兼容 插桩在 LLVM IR 层；inline asm 是 opaque blob，无法分析其内存读写 asm 后手动 __msan_unpoison(buf, len)
JIT-generated / 自修改代码不监控 插桩在 AOT 编译期；运行期生成的代码无 shadow 更新 对 JIT 输出 buffer 手动 unpoison
"延迟报警"让根因难定位 设计上 propagate-without-report：bug 在 alloc 处，报警在百行外分支处 MSAN_OPTIONS=origin_history_size=... 开 origin tracking
memcpy 把 poisoned bytes 复制到新位置（不报，污染扩散） memcpy bit-wise copy 时 shadow 也 bit-wise copy；不"读"poisoned data 所以不报 By design；触发分支时才报
Struct padding 默认 poisoned → memcpy / 比 struct 假阳 编译器不主动写 padding，MSan 视其为未初始化 显式 memset(0) 整 struct；或 __msan_unpoison()
不检测越界 / UAF / leak Shadow 只编码"已初始化"，不编码"合法访问" 这是 ASan / LSan 的工作；分开 build 跑
不能跟 ASan / TSan 同跑 Shadow layout 互斥 分 build
性能 + 内存放大让 CI 慢 5× 每 bit 1 shadow bit + origin tracking 让 cache miss 翻倍 仅在专门 job 跑
本质局限：MSan 是整链路 taint tracking——一个环节不插桩，整个 chain 失效。这跟 ASan（只看 alloc 边界）/ TSan（只看 sync 原语）不同，让 MSan 在多语言混合项目（C++/Rust/CUDA/汇编）几乎不可用。Google 内部能跑是因为monorepo 全 MSan-built libc++，普通团队代价过高。

UBSan · 低开销 UB 检测

覆盖的 UB 类别 · 开销极低 · UBSan 的局限性与根因

UndefinedBehaviorSanitizer (UBSan) 检测各种显式 UB——不需 shadow memory，直接在 instruction level 插 check。

覆盖的 UB 类别

类别典型例子
signed-integer-overflowINT_MAX + 1
shiftx << 64（width > type width）
nullnull pointer dereference / method call
alignmentmisaligned access
vptr错误 virtual table（dangling C++ object）
boundsarray bounds（部分场景）
float-divide-by-zero1.0 / 0.0
integer-divide-by-zero1 / 0
returnnon-void function 没 return

开销极低

每条 check 是 1-2 条额外指令 + 1 个分支。典型程序 1.05-1.10× slowdown。生产 fleet 可常开（仅 trap-on-error，不写报警）。

# Dev: full report
g++ -fsanitize=undefined ...

# Prod: trap-only (无 report, crash on UB)
g++ -fsanitize=undefined -fsanitize-trap=undefined ...

UBSan 的局限性与根因

限制根本原因缓解 / 替代
只 catch "标准列举"的 UB（约 20 类） UBSan 是 check per-UB-kind 手写实现；C++ UB 上百种（aliasing / lifetime / 比较未关联指针…）很多没插桩 需求驱动：profile 关心的 UB；CFI / shadow call stack 等 sibling sanitizer 补充
Strict aliasing UB 几乎不报 需要类型流追踪，编译期信息已被优化丢弃；插桩成本极高 -fno-strict-aliasing；或 TBAA-aware 静态分析
Lifetime UB（dangling reference）不查 需要 escape analysis + lifetime tracking，等价于半个 borrow checker 静态：clang lifetime attrs；动态：ASan 抓 UAF/UAR
vptr check 假阳常见 -fsanitize=vptr 依赖 typeinfo 比较；动态库跨 RTTI 边界、多继承 ambiguous 时误报 白名单 / 关 vptr 子检查
unsigned-integer-overflow 默认不算 UB（C/C++ 标准定义为 wrap） 非 UB；仅 -fsanitize=unsigned-integer-overflow 显式开时才报（属"unwanted behavior"非"undefined"） 需要时显式打开
signed overflow 在已优化 release 下可能"消失" 编译器假设 signed 不 overflow → 死代码消除把 check 删了 用 -O1 + UBSan；或 -fwrapv
UBSan-trap 模式 crash 无信息 Trap = ud2 指令，runtime library 不链接 → SIGILL 无 stack 解释 用 -fsanitize-minimal-runtime 折中（少量 binary 增大）
不查内存安全 / race / 初始化 UBSan 在 instruction level 插独立 check，不维护 shadow / VC 状态 组合 ASan / TSan / MSan
对模板大量实例化的 binary size 膨胀显著 每个 check site 生成独立 fail handler symbol；C++ template-heavy 项目 +20-30% binary -fsanitize-minimal-runtime 或精选子集
本质局限：UBSan 是"枚举式" UB 检测——标准列了 UB 它才查。所以它性能极低（可生产），但覆盖窄。把它当 production safety net，不是 dev 主力 bug-catcher。

HWASan · ARM 硬件辅助

原理 · 现状 · HWASan / MTE 的局限性与根因

HWASan（Hardware-assisted AddressSanitizer）利用 ARM 的两个硬件特性：

  • Top-Byte-Ignore (TBI)——ARMv8 起，地址的最高字节被忽略（实际地址只用低 56 位）。这一字节可放 metadata。
  • Memory Tagging Extension (MTE)——ARMv8.5 起，硬件级地址 tagging：每 16 字节 memory 有 4-bit tag，访问时 CPU 自动比较 ptr tag 跟 memory tag。

原理

  1. 每个 allocation 分配时设 random 4-bit tag 到该区域
  2. 返回的 pointer 的高字节也包含该 tag
  3. 访问时 CPU（MTE）/ runtime（TBI）比较 ptr tag vs memory tag
  4. 不匹配 → trap / 报警
维度ASanHWASan (TBI)HWASan (MTE)
shadow 内存1/8 用户内存 ratio1/16 ratio0（硬件直接存）
CPU 开销~2×~1.3-1.5×~1.05-1.15×
误报率1/2^8 = 0.4%（redzone fixed）1/16 (4-bit tag)同 TBI
生产可用不勉强（dev/CI）是

2026 现状

  • Pixel 8+ / 9 系列默认开 MTE（kernel + userspace）
  • Android 15+ 部分系统服务跑 MTE
  • Linux 6.x 内核 KASAN-HW（kernel HWASan）
  • Apple 计划 M-series CPU 加类似支持（2026 rumored）
HWASan 的真正意义：让 sanitizer 进生产 fleet。开销 5-15% 意味着可以全量部署，捕获之前 dev 阶段漏掉的 1 in 1000 corner case bug。

HWASan / MTE 的局限性与根因

限制根本原因缓解 / 替代
误报率 ~6.25% (1/16)——bug 有 1/16 概率随机漏 Tag 只有 4 bit；新 alloc tag 跟相邻 free chunk 撞概率 1/16 多次跑提升 cumulative catch；或加宽 tag（需硬件改）
仅 ARMv8.5+ 才有 MTE（Pixel 8+ / Apple M4 等） 需 ISA 扩展支持 LDG/STG tag 指令；x86 无对应 x86 用 ASan / GWP-ASan；TBI-only 模式无 MTE 也能跑（开销 1.3-1.5×）
x86 无 TBI——HWASan 在 x86 几乎无用 x86_64 不忽略高字节（Intel LAM 5-level 之后部分支持，但生态未跟） x86 等 LAM 普及；或继续 ASan
Pointer arithmetic 跨 tag boundary 假阳 不同 alloc tag 不同；正常 ptr 加减跨 alloc 时 tag 不变 → 比 memory tag 失败 避免 inter-object pointer arithmetic（本来也是 UB）
对 stack 越界粒度粗（16 字节 granule） MTE tag 1 个/16 字节，小于此的 overflow 漏 关键 hot 函数 disable HWASan 改 ASan
跟 PAC (pointer authentication) 共享 high bits ARM PAC 也用高字节签名 → 跟 HWASan tag 抢位 编译器协调；MTE + PAC 同开时 PAC 用更少 bit
所有动态库需 HWASan-built 非插桩 .so 设置/读 ptr 时高字节随机 → tag 比对全失败 系统 lib 提供 HWASan 版本（Android NDK）
MTE async 模式延迟报警 Async 模式由 CPU 累积报告（非每条指令），bug site 跟 trap site 不同 Dev 用 sync 模式；prod 用 async（性能 vs 精度权衡）
kernel 支持需特定版本 需 Linux 5.10+ ARM64 MTE patch；userspace 库需 prctl 配合 Android 14+ / Pixel 8+ 默认就绪
本质局限：HWASan 用概率换性能——4-bit tag 决定它不能像 ASan 那样确定性 catch。但在fleet-scale这是优势：100k 设备每个有 15/16 概率 catch，整体 catch 率趋近 1。HWASan 不是 ASan 的替代，是 ASan 的 production sibling。
07

GWP-ASan · 生产采样

原理 · 开销 vs 覆盖 · 生产采用 · GWP-ASan 的局限性与根因

GWP-ASan（"GuardWithProbability"-ASan）是另一思路：采样。

原理

  • 大多数 allocation 走正常 allocator——0 开销
  • 每 ~10000 个 allocation 采样一个放入"guarded" region
  • Guarded region 有 redzone + page-level protection
  • 对该 alloc 的越界 / UAF 立刻 crash + 详细报告

开销 vs 覆盖

  • 开销：~ 1% CPU + ~ 100 MB memory
  • 覆盖：1/10000 allocation 被监控
  • 对于 frequent bug（1 in 1000 access），高概率 catch；rare bug 漏

生产采用

  • Google Chrome 全 fleet 默认开（自 2020）
  • Apple iOS 部分内置 lib 用类似机制
  • Android 14+ 内置 GWP-ASan + GWP-TSan

GWP 风格 sanitizer 是"find rare bugs at scale"的最佳方案——单实例不能保证 catch 你的 bug，但 100k production users 累积起来 catch 罕见 bug 的概率非常高。

GWP-ASan 的局限性与根因

限制根本原因缓解 / 替代
单实例几乎不 catch 任何 bug 采样率 1/10000 → 期望要跑 10000 次 alloc 才碰一个 guarded 仅在 fleet-scale 才有意义；dev 阶段用 ASan
只检 heap，不检 stack / global 原理是替换 malloc 把采样 alloc 放 mmap 守护区；stack 不经 malloc Stack overflow 用 stack canary / shadow stack
只 catch 越界 + UAF，对 intra-object 错配无能 整 page 当 redzone，相对 byte 级 redzone 粒度粗 ASan 做精确检查
~100MB 内存预留 Guarded slot 各占 2 page（前后 guard page）；slot 数固定（~4k） 调小 slot 数（采样率不变，命中率降）
Slot 满后停止采样 Pool 是固定大小；用完不复用直到老 alloc 真 free 调大 slot；或缩短 sample lifetime
Bug repro 需要服务端聚合 单 crash 不一定有 actionable info；需 stack trace 聚合 + 去重 Crashpad / Sentry / Crashlytics 后台收集
对低 QPS 程序无效 采样需要量级；每天几千 alloc 的 daemon 几乎 catch 不到 这类程序 dev 阶段 ASan + fuzzing
跟某些 jemalloc/tcmalloc 实现配合需 patch GWP-ASan 要 hook allocator 入口；tcmalloc 内部直接 mmap path 绕过 tcmalloc 已内置 GWP-ASan 集成（Google fork）
本质局限：GWP-ASan 是概率工程——对单 bug 期望 catch time = (1 / 采样率) × access frequency。它跟 telemetry 后台是一体的，不是孤立工具。Chrome / Android / iOS 这类有 fleet 的产品才能受益。

跨 Sanitizer 通用局限 + 互动陷阱

互斥矩阵——不能同跑 · 跟 Allocator 生态的张力 · 构建复杂度 · 优化级别相互作用 · 真 bug 容易被误当 sanitizer bug · …

除了各 sanitizer 自身的限制（见 §2-§7 各自的"局限性与根因"小节），整个 sanitizer 家族还有共同的结构性限制。

9.1 互斥矩阵——不能同跑

ASanTSanMSanHWASanUBSanLSan
ASan—❌❌❌✅✅ (内置)
TSan❌—❌❌✅⚠️ 部分
MSan❌❌—❌✅⚠️ 部分
HWASan❌❌❌—✅✅
UBSan✅✅✅✅—✅

根因：

  • Shadow 内存 layout 冲突——ASan/MSan/TSan/HWASan 都把虚拟地址空间的特定段独占当 shadow。两两 layout 不兼容。
  • Allocator 冲突——每个 sanitizer 替换 malloc/free 实现自己的 redzone / quarantine / tag。两个同时 hook 一个 malloc 会破坏 ABI。
  • Signal handler 冲突——多 sanitizer 都装 SIGSEGV/SIGBUS handler 处理 shadow trap。
  • UBSan 是例外：它在 instruction level 插 in-place check，不用 shadow / 不替 allocator → 跟所有 sanitizer 同跑。

9.2 跟 Allocator 生态的张力

  • jemalloc / tcmalloc：sanitizer allocator 直接替换 → 失去 jemalloc 的 arena / size class 调优，性能特性不同。Sanitizer build 的 latency profile 不能代表生产。
  • 用户对象池 / arena：sub-allocation 对 sanitizer 不可见。Arena 内部一片 alloc 越界踩另一片，ASan 看到的还是 "整个 arena 合法"。
  • mmap / shm 共享内存：跨进程 sanitizer 状态不共享，跨进程越界检测失效。

9.3 构建复杂度

  • 每种 sanitizer 一份独立 build → CI 资源 ×N
  • 第三方 .so 必须用对应 sanitizer build 否则假阳/假阴（尤其 MSan / HWASan）
  • Cross-language（C++/Rust/Go/CUDA）只有 C/C++/部分 Rust 支持 → 跨边界假阳

9.4 优化级别相互作用

  • -O0 跑 sanitizer：stack 布局 / inline 行为跟生产差异大，可能漏 prod-only bug
  • -O2 跑 sanitizer：编译器把死代码消除（包括"反正 signed 不 overflow"），可能消除掉 sanitizer check 本身
  • 推荐 -O1 -g 平衡

9.5 真 bug 容易被误当 sanitizer bug

  • placement new 没初始化某 field（C++ 故意）→ MSan 报，是真 UB
  • 读已构造对象的某 padding byte → ASan / MSan 报，标准 UB
  • 用自己的 atomic primitive → TSan 报 race，因为没 annotation
  • memcpy(struct) 触发 MSan padding 警告
  • std::vector::data() + 越界 1 byte 写 → ASan 在 capacity 边界报，看似"应该合法"的 bug

9.6 不能查的 bug 类别（全家桶都搞不定）

Bug 类原因替代手段
逻辑 bug（算错答案但不 crash）Sanitizer 只查 UB / memory error，不懂"对的算法"单元测试 / property-based test
死锁TSan 抓 race 不抓 deadlock静态 lock-graph analysis / Helgrind
livelock / 优先级反转非 UB，运行能继续压测 + tracing
性能退化（如假共享）非正确性问题perf / pmu-tools
跨进程数据竞争（共享内存）TSan 单进程视角专用 IPC race detector / 设计审查
Spectre / Meltdown 类 side channel非架构层 UB编译期 mitigations (-mretpoline 等)
Cryptographic timing leak正确执行但泄漏ctgrind / 专用工具
Hardware bug（cache coherence / TLB）软件视角看是正确的架构级 stress + 厂商 erratum
总结哲学：Sanitizer 不是"开了就安全"。它是一组互补的动态检测工具，每个 catch 一类特定 bug，每个有结构性盲区。生产质量靠组合：dev 阶段 ASan+UBSan 抓 80%、CI 加 TSan 抓 race、production 用 HWASan / GWP-ASan 抓长尾 + 静态分析 + fuzzing 补盲区。
10

自己写 Sanitizer · LLVM Pass 入门

个核心组件 · 最小 example：BoundsSan · 更进一步 · 学习路径

Sanitizer 的设计精髓是 compile-time 插桩。理解后你可以写自己的 domain-specific sanitizer。

3 个核心组件

  1. LLVM Pass（compile-time）—— 在 IR 上插桩，每条 memory access 前后加 check call
  2. Runtime library—— 实现 check call 的具体行为（管 shadow memory / 报错）
  3. Driver integration—— -fsanitize=mysan flag 接入 clang driver

最小 example：BoundsSan

// LLVM pass
class BoundsSanPass : public ModulePass {
  bool runOnModule(Module &M) override {
    for (Function &F : M)
      for (BasicBlock &BB : F)
        for (Instruction &I : BB)
          if (auto *Load = dyn_cast<LoadInst>(&I))
            instrumentLoad(Load);
    return true;
  }
  void instrumentLoad(LoadInst *Load) {
    IRBuilder<> B(Load);
    Value *Addr = Load->getPointerOperand();
    Value *Size = ConstantInt::get(B.getInt64Ty(),
      Load->getType()->getPrimitiveSizeInBits() / 8);
    B.CreateCall(checkFunc, {Addr, Size});
  }
};

// Runtime:
extern "C" void __boundssan_check(void *addr, size_t size) {
  if (!isValidRange(addr, size))
    reportError(addr, size, __builtin_return_address(0));
}

更进一步

  • SanitizerCoverage —— LLVM 已有 framework，只 instrument 不 check，给 fuzzer 用
  • DataFlowSanitizer (DFSan) —— taint tracking，比 MSan 更通用
  • RealtimeSanitizer (RTSan) —— C++26 提案，检测 RT context 中调 malloc / blocking call

学习路径

  1. 读 LLVM 源码 llvm/lib/Transforms/Instrumentation/ —— 真实 sanitizer pass 实现
  2. Compiler Sanitizer USENIX ATC 2012 论文（ASan）
  3. PLDI 2009 FastTrack 论文（TSan 算法）
  4. LLVM tutorial: Writing an LLVM Pass
🎯 关键 take-aways
  • Sanitizer = compile-time 插桩 + runtime check，是 Valgrind 模型的 10× faster 演化
  • ASan shadow memory 是 1:8 byte ratio，redzone + quarantine 抓 overflow / UAF
  • TSan FastTrack 算法区分 read epoch / write full vector clock，让 race 检测从 100× → 5-10×
  • UBSan 极低开销（~5%），生产 fleet 可用
  • HWASan + MTE 让 sanitizer 进 production（ARM 上 ~10% 开销）
  • GWP-ASan 用采样让 100k production users 累积 catch rare bug
  • 2026 默认 dev/CI stack：ASan + UBSan + TSan (separate)；production 加 HWASan / GWP-ASan
