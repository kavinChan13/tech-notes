C 风格 cast 为什么"危险"

C++ cast 的 4 大优势 · C 风格 (T)expr 和 函数风格 T(expr) 都是"全能 cast"，编译器按以下顺序尝试：

C 风格 (T)expr 和 函数风格 T(expr) 都是"全能 cast"，编译器按以下顺序尝试：

  1. const_cast
  2. static_cast
  3. static_cast + const_cast
  4. reinterpret_cast
  5. reinterpret_cast + const_cast

问题：你看不出编译器最终用了哪种 — 维护时极易翻车。

// C 风格的隐患
const Base* b = getBase();
Derived* d = (Derived*)b;
// ↑ 实际是 const_cast + static_cast 组合，悄无声息地去掉了 const
// 还做了非检查的下行转换 — 如果 b 不是 Derived 就 UB

// 用 C++ cast 你必须显式分两步，无法不小心做错
Derived* d2 = const_cast<Derived*>(static_cast<const Derived*>(b));

2.1 C++ cast 的 4 大优势

优势说明
意图明确读代码立刻知道做了什么
可被编译器/工具检查错误使用立刻报错
易于 grep/搜索grep "reinterpret_cast" 能找到所有可疑代码
更安全dynamic_cast 运行时检查；static_cast 不会去掉 const
团队规范：禁止 C 风格 cast。clang-tidy 规则 google-readability-casting 自动检查。例外：与纯 C API 交互时偶尔可用 (但仍推荐改为 C++ cast)。

static_cast 完全详解

能做的转换 · 不能做的 · 隐藏用法：触发用户定义转换

编译期转换。最常用，不做运行时检查，错误使用是 UB。

3.1 能做的转换

类型例子
数值类型互转static_cast<int>(3.14)
枚举 ↔ 整型static_cast<int>(Color::Red)
void* → T*static_cast<int*>(malloc(4))
类层次下行转换（不检查）static_cast<Derived*>(base_ptr)
类层次上行转换（安全）static_cast<Base*>(d)
显式调用 用户定义转换static_cast<std::string>("abc")
转 rvalue（实现 std::move）static_cast<T&&>(x)

3.2 不能做的

  • 去掉 const / volatile（要用 const_cast）
  • 无关指针类型互转（要用 reinterpret_cast）
  • 跨不相关类层次（不安全的 down-cast 也能编译但是 UB）
double d = 3.7;
int i = static_cast<int>(d);                  // 3 (truncate)

enum class Color { Red, Green, Blue };
int c = static_cast<int>(Color::Green);         // 1

void* p = malloc(64);
int* arr = static_cast<int*>(p);                // 必须显式

struct Animal { virtual ~Animal() = default; };
struct Dog : Animal {};
Animal* a = new Dog;
Dog* dog = static_cast<Dog*>(a);                // ✅ 你确定它是 Dog

Animal* a2 = new Animal;
Dog* d2 = static_cast<Dog*>(a2);              // ⚠️ 编译通过，访问 d2 是 UB

3.3 隐藏用法：触发用户定义转换

class Money {
    int cents_;
public:
    explicit Money(int cents) : cents_(cents) {}
};

Money m1 = 100;                                  // ❌ explicit 阻止隐式
Money m2(100);                                  // ✅ 直接构造
Money m3 = static_cast<Money>(100);              // ✅ 显式触发
04

dynamic_cast + RTTI

核心语义 · 用法 · 性能开销与禁用 · dynamic_cast 的替代方案

4.1 核心语义

运行时类型检查 + 安全的下行/横向转换。失败时：

  • 转换指针失败 → 返回 nullptr
  • 转换引用失败 → 抛 std::bad_cast

前提：源类型必须是 polymorphic（含至少一个虚函数）— 因为依赖 RTTI（vtable + type_info）。

4.2 用法

struct Animal { virtual ~Animal() = default; };
struct Dog  : Animal { void bark(); };
struct Cat  : Animal { void meow(); };

Animal* a = getAnimal();
if (Dog* d = dynamic_cast<Dog*>(a)) {     // ✅ 安全：失败返回 nullptr
    d->bark();
} else if (Cat* c = dynamic_cast<Cat*>(a)) {
    c->meow();
}

// 引用版本：失败抛异常
try {
    Dog& d = dynamic_cast<Dog&>(*a);
} catch (std::bad_cast&) { /* ... */ }

4.3 性能开销与禁用

开销来源

  • 需要 vtable 内的 type_info（RTTI）
  • 跨继承层次时遍历 type_info 链表/树
  • 典型耗时 100~1000 ns（10~100× 比 static_cast 慢）
  • 多继承场景更慢

禁用 RTTI 的代价

g++ -fno-rtti 可以关闭 RTTI，二进制更小（无 type_info 字符串）。

代价：① dynamic_cast 不能用；② typeid 不能用；③ 异常仍能用（不依赖 RTTI）。

嵌入式 / 游戏引擎常关闭 RTTI 以减少二进制和性能。

4.4 dynamic_cast 的替代方案

  1. 虚函数多态：在基类加 virtual void on_visit(Visitor&) — visitor 模式，零运行时类型查询
  2. std::variant + std::visit：编译期已知所有类型时更快
  3. 枚举 + switch：自定义 type tag
  4. CRTP 静态多态：编译期就知道类型，无运行时开销
// variant 替代 dynamic_cast — 性能更好且类型安全
using Shape = std::variant<Circle, Square, Triangle>;
std::visit([](auto&& s){
    using T = std::decay_t<decltype(s)>;
    if constexpr (std::is_same_v<T, Circle>)   s.draw_circle();
    else if constexpr (std::is_same_v<T, Square>) s.draw_square();
}, shape);
"什么时候不该用 dynamic_cast?" — 答：① 性能敏感的热路径；② 嵌入式 / 资源受限平台；③ 类型集合是封闭的 — 优先 variant。"用 dynamic_cast = 设计有问题" 的极端观点过激，但最佳实践是限制其使用范围。
05

const_cast 真实用法

唯一安全用法 · 一定不安全的用法（UB） · 经典惯用法：const 与非 const 重载共享代码

唯一能添加或去除 cv 限定符（const / volatile）的 cast。

5.1 唯一安全用法

// ✅ 当原对象本来不是 const，只是被 const 引用了 → 去掉 const 安全
void legacy_api(char* msg);            // C 老接口（实际不会改）

void my_log(const std::string& msg) {
    legacy_api(const_cast<char*>(msg.c_str()));   // ✅ 接口承诺不修改
}

5.2 一定不安全的用法（UB）

const int x = 42;                          // 真正的 const 对象
int* p = const_cast<int*>(&x);
*p = 0;                                    // ⚠️ UB! 修改真正的 const 对象
                                           // 编译器可能把 x 放在只读段

5.3 经典惯用法：const 与非 const 重载共享代码

class Container {
    std::vector<int> v_;
public:
    const int& at(size_t i) const {
        if (i >= v_.size()) throw std::out_of_range{"i"};
        return v_[i];
    }

    // 非 const 版本复用 const 版本逻辑
    int& at(size_t i) {
        return const_cast<int&>(static_cast<const Container&>(*this).at(i));
        // 步骤: 先把 *this 看作 const → 调 const at → 把返回的 const int& 去 const
    }
};

C++23 引入 deducing this 让这个惯用法变得更优雅：

// C++23
template<class Self>
auto& at(this Self&& self, size_t i) {
    if (i >= self.v_.size()) throw std::out_of_range{"i"};
    return self.v_[i];
}
规则：const_cast不会真的让对象可写 — 它只能"承诺底层对象本来不是 const"。如果原对象真的 const → UB。
06

reinterpret_cast 深坑

能做的（合法用法） · 严格别名规则（Strict Aliasing）⭐ · 真实工程合法用法

"把这段比特当作另一种类型来看"。不做任何转换，仅是类型重解释。

6.1 能做的（合法用法）

用法是否安全
整数 ↔ 指针：reinterpret_cast<void*>(0x1234)实现定义
函数指针 ↔ void*（POSIX dlsym）POSIX 保证；ISO C++ 不保证
不相关指针互转能转，但解引用通常 UB（违反严格别名）
T* ↔ char* / std::byte*✅ 严格别名规则的例外

6.2 严格别名规则（Strict Aliasing）⭐

C++ 规定：通过类型 A 的指针访问类型 B 的对象是 UB（除了 char/byte/同类型/有共同部分的 union 等几个例外）。

编译器假设不同类型的指针不会指向同一内存 → 优化掉一些读写顺序 → 真在 UB 时结果变态。

// ❌ 经典 UB
float f = 3.14f;
uint32_t bits = *reinterpret_cast<uint32_t*>(&f);  // 严格别名违反 → UB

// ✅ 安全方式 1：memcpy
uint32_t bits2;
std::memcpy(&bits2, &f, sizeof(f));            // 编译器会优化为单条 mov 指令

// ✅ 安全方式 2：std::bit_cast (C++20)
auto bits3 = std::bit_cast<uint32_t>(f);          // 类型安全，零开销

// ✅ 唯一允许的 reinterpret_cast：转 char* 读字节
const char* bytes = reinterpret_cast<const char*>(&f);
for (size_t i = 0; i < sizeof(f); ++i) std::cout << std::hex << (int)bytes[i];

6.3 真实工程合法用法

// ① 网络收发 — 把 socket 缓冲区按字节传给 recv/send
char buf[4096];
ssize_t n = ::recv(fd, reinterpret_cast<void*>(buf), 4096, 0);

// ② 共享内存 / mmap — 把 void* 解释为业务结构
struct SharedHeader { uint32_t magic; uint32_t version; size_t count; };
void* shm = mmap(nullptr, size, ...);
auto* hdr = reinterpret_cast<SharedHeader*>(shm);
// ↑ 这里读 hdr->magic 在多数实现下 OK，但理论上若 shm 没用 placement new 构造过对象，是 UB

// ③ 函数指针互转（POSIX dlsym）
void* sym = dlsym(handle, "my_func");
auto fp = reinterpret_cast<void(*)(int)>(sym);
fp(42);

// ④ Hash 计算 — 把对象按字节看
const auto* p = reinterpret_cast<const uint8_t*>(&obj);
uint64_t h = fnv1a(p, sizeof(obj));            // ✅ 通过 char/byte/uint8_t 安全
团队规范：reinterpret_cast 出现的代码必须配 code review + 注释为什么安全。clang-tidy 的 cppcoreguidelines-pro-type-reinterpret-cast 默认报警。
07

隐式转换链 + 算术转换规则

标准转换序列（4 类，按发生顺序） · 整型提升（Integral Promotion） · 整型转换（Integral Conversion） · 通常算术转换（Usual Arithmetic Conversion）…

7.1 标准转换序列（4 类，按发生顺序）

  1. Lvalue Transformations — Lvalue→Rvalue / 数组→指针 / 函数→指针
  2. Promotion (提升) — char/short → int；float → double
  3. Conversion (转换) — int → long / int → double / 指针 → bool / 派生 → 基类
  4. Qualification — 加 const / volatile

7.2 整型提升（Integral Promotion）

所有比 int 窄的整型（char、short、bool）在算术上下文中自动提升为 int（或 unsigned int）。

char a = 100, b = 100;
auto c = a + b;                   // 类型是 int (200), 不是 char

short s1 = 30000, s2 = 30000;
auto sum = s1 + s2;               // 60000 — int 不溢出；short 会

7.3 整型转换（Integral Conversion）

  • 无符号目标类型：取模 2^N — 完全定义行为，不是 UB
  • 有符号目标类型且值不能表示：实现定义（C++17 起；C++20 起统一二进制补码）
unsigned char uc = 257;            // = 257 % 256 = 1，定义行为
int i = -1;
unsigned int u = i;                // = UINT_MAX (4294967295)，模 2^32

if (i < u) {}                      // ⚠️ false! 比较时 i 被转为 unsigned
// → -1 → 4294967295，4294967295 < 4294967295 是 false

7.4 通常算术转换（Usual Arithmetic Conversion）

二元算术运算符（+ - * / % < > ==）会做以下转换找共同类型：

  1. 任一操作数是 long double → 都转 long double
  2. 任一是 double → 都转 double
  3. 任一是 float → 都转 float
  4. 都做整型提升后：
    • 同符号：转为等级更高的类型（int < long < long long）
    • 无符号等级 ≥ 有符号 → 转无符号
    • 否则 → 转为更高等级的有符号

7.5 经典陷阱集合

// ① unsigned 与 signed 混算
int i = -1;
unsigned u = 1;
if (i < u) std::cout << "yes";     // ⚠️ 输出: 不输出 (i 转为 UINT_MAX)

// ② 整型字面量类型
auto x = 0x80000000;               // C++11+ 是 unsigned int (太大溢出 int)
// 在 32-bit int 平台。表示无负号

// ③ size_t 减法 (常见 bug)
std::vector<int> v;
for (int i = 0; i < v.size() - 1; ++i) {}     // ⚠️ v.size()=0 时 size()-1 = SIZE_MAX

// ④ char 当数字处理
char c = 200;
int i = c;                         // 在 signed char 平台是 -56！

// ⑤ 浮点比较
if (0.1 + 0.2 == 0.3) {}           // false! 浮点精度问题

// ⑥ 整型 / 整型 → 整型截断
double avg = 5 / 2;                // 2.0! 不是 2.5
double avg2 = 5.0 / 2;             // ✅ 2.5

7.6 工具：现代检查

  • -Wall -Wextra — 大量隐式转换警告
  • -Wsign-compare — 有符号/无符号比较
  • -Wsign-conversion — 有符号到无符号转换
  • -Wconversion — 一般类型缩窄
  • -Wfloat-equal — 浮点 == 警告
  • UBSan：运行时检查整型溢出 / 严格别名违反
"auto x = 0u + (-1); x 是什么类型? 值是多少?" — 答：unsigned int 提升 → 取模 2^32 → x 是 unsigned int，值 4294967295（在 32-bit int 平台）。表面简单，迷惑大量初学者。
08

用户定义转换 + explicit

隐式转换链规则 · explicit 关键字（C++11） · explicit 的常见目标 · C++20 conditional explicit: explicit(bool)

用户能自定义两种转换：

  • 转换构造函数：T(U) — 把 U 转为 T
  • 转换运算符：operator U() — 把 T 转为 U

8.1 隐式转换链规则

编译器在重载决议时，最多只允许一次用户定义转换 + 任意次数的标准转换。

struct A {
    A(int);                 // 转换构造函数 (隐式)
};
struct B {
    B(A);                   // 转换构造函数
};

void f(B);
f(42);                       // ❌ 需要 int → A → B 两次用户定义转换
f(A(42));                  // ✅ A(42) 是显式构造，剩下 A → B 一次

8.2 explicit 关键字（C++11）

把构造函数标记为 explicit → 禁止隐式转换，必须显式调用。

class Vec {
    int* data_;
public:
    explicit Vec(size_t n) : data_(new int[n]) {}
};

Vec v1 = 100;                 // ❌ 编译错 (隐式转换)
Vec v2(100);                 // ✅ 直接初始化
Vec v3{100};                 // ✅ 列表初始化
Vec v4 = Vec(100);          // ✅ 显式
规则：除非你确认类型支持隐式转换是合理的（如 string 从 const char*），否则所有单参构造都应该 explicit。clang-tidy 规则 google-explicit-constructor 强制此规范。

8.3 explicit 的常见目标

  • 资源管理类（FileHandle、Buffer）— 防止偶然的"int → 资源"

  • 智能指针 — 防止裸指针隐式包装

explicit operator bool 与 conversion operator

conversion operator（转换运算符） · explicit operator bool — 重要惯用法 · 标准库案例

9.1 conversion operator（转换运算符）

class Money {
    double usd_;
public:
    operator double() const { return usd_; }     // 隐式转换为 double
};

Money m{100};
double d = m;                 // ✅ 隐式
double d2 = m + 1.5;          // ✅ 算术上下文

同样可以加 explicit：

class Money {
public:
    explicit operator double() const { return usd_; }
};
double d = m;                 // ❌ 编译错
double d2 = static_cast<double>(m);  // ✅

9.2 explicit operator bool — 重要惯用法

C++11 引入 explicit operator bool 解决了"safe bool 惯用法"的复杂性。

class FileHandle {
    int fd_ = -1;
public:
    explicit operator bool() const noexcept {
        return fd_ >= 0;
    }
};

FileHandle fh = open_file("x");
if (fh) { /* ... */ }       // ✅ explicit 仍然在「条件上下文」生效
if (!fh) { /* ... */ }      // ✅
while (fh) {}                // ✅

bool b = fh;                 // ❌ 编译错 — explicit 阻止
int n = fh + 1;             // ❌ 编译错 — 不会被算术上下文滥用

背后机制：C++ 标准把以下场景叫做 "contextual conversion to bool"，即使 explicit 也允许：

  • if (x)
  • while (x)
  • for (...; x; ...)
  • !x
  • x && y / x || y
  • x ? a : b

9.3 标准库案例

// std::optional, std::unique_ptr, std::shared_ptr 都是 explicit operator bool
std::optional<int> opt = find_value();
if (opt) use(*opt);                            // ✅

std::unique_ptr<Foo> p = make_foo();
if (p) p->do_something();                       // ✅
"为什么 std::optional 用 explicit operator bool?" — 答：① 防止 opt + 1 这种荒谬代码；② 仍然支持 if (opt) 这种语义自然的判定 — 因为标准把它列为 contextual conversion 例外。

重载决议（Overload Resolution）中的转换

转换序列优先级（从好到差） · 经典题 · 万能引用劫持 · 修复万能引用劫持

10.1 转换序列优先级（从好到差）

  1. Exact match — 完全匹配（含 lvalue→rvalue、const 加 / 减、数组→指针）
  2. Promotion — 整型 / 浮点提升
  3. Standard conversion — 整型/浮点/指针/bool 等转换
  4. User-defined conversion — 转换构造或转换运算符
  5. Ellipsis — ... 可变参数

编译器找最好的匹配；多个一样好 → 二义性错误。

10.2 经典题

void f(int);
void f(double);
void f(long);

f(0);            // f(int) — exact
f(0.0);          // f(double) — exact
f(0L);           // f(long) — exact
f(0u);           // f(int) — unsigned int → int (conversion，比 → double 好)
f('c');          // f(int) — char promotes to int (promotion，比 conversion 好)
f(0.5f);         // f(double) — float promotes to double (promotion)

// ❌ 二义性
void g(long);
void g(double);
g(0);            // 都是 standard conversion，二义性

10.3 万能引用劫持

class Person {
public:
    template<class T>
    Person(T&& n) : name_(std::forward<T>(n)) {}    // 万能引用
    Person(int id) : id_(id) {}
};

short id = 42;
Person p(id);    // ⚠️ 调用万能引用版本! 而不是 int 版本
// 因为 short → int 需要 conversion，万能引用 T = short& 是 exact match

10.4 修复万能引用劫持

class Person {
public:
    template<class T,
             class = std::enable_if_t<!std::is_integral_v<std::decay_t<T>>>>
    Person(T&& n) : name_(std::forward<T>(n)) {}    // 排除整型
    Person(int id) : id_(id) {}
};

// C++20 用 concepts 更优雅
template<class T>
    requires (!std::integral<std::decay_t<T>>)
Person(T&& n);

std::bit_cast (C++20)

背景：传统方法的问题 · std::bit_cast 完美解决 · 约束 · 真实工程用法

类型穿越的现代方式

11.1 背景：传统方法的问题

方法问题
reinterpret_cast违反严格别名规则 → UB
union 重叠C++ 标准技术上是 UB（C 允许 type-punning，C++ 不允许 — 实践中大多 OK）

11.3 约束

  • 源和目标类型必须 sizeof 相同
  • 都必须是 trivially copyable
  • 不构造目标对象 — 只是按字节复制

11.4 真实工程用法

// ① 浮点数判定
bool isNaN(float f) {
    auto bits = std::bit_cast<uint32_t>(f);
    return ((bits >> 23) & 0xFF) == 0xFF && (bits & 0x7FFFFF) != 0;
}

// ② 网络字节序处理
uint32_t read_be32(const uint8_t* p) {
    std::array<uint8_t, 4> bytes{p[0], p[1], p[2], p[3]};
    if constexpr (std::endian::native == std::endian::little)
        std::ranges::reverse(bytes);
    return std::bit_cast<uint32_t>(bytes);
}

// ③ 实现 fast inverse sqrt (经典 Quake 算法)
constexpr float fast_inv_sqrt(float x) {
    auto i = std::bit_cast<uint32_t>(x);
    i = 0x5F3759DF - (i >> 1);
    auto y = std::bit_cast<float>(i);
    return y * (1.5f - 0.5f * x * y * y);
}
规则：C++20 起，所有需要"按位重新解释"的场景都用 std::bit_cast。reinterpret_cast 仅留给：① char* 字节访问；② 函数指针；③ 整数 ↔ 指针。

SFINAE 入门 + concepts

什么是 SFINAE · 经典 SFINAE 写法（C++11/14） · enable_if 风格 · void_t 检查类型成员（C++17） · C++20 concepts —— SFINAE 终结者 · …

12.1 什么是 SFINAE

SFINAE = Substitution Failure Is Not An Error。模板实例化时，如果类型代入失败（不是真错误），编译器把这个候选移出重载集，而不是报错。

这是模板"鸭子类型"的基础 — 让模板根据类型能力选择不同实现。

12.2 经典 SFINAE 写法（C++11/14）

// 检查 T 是否有 .size() 方法
template<class T>
auto size_of(const T& c) -> decltype(c.size()) {
    return c.size();
}

template<class T, size_t N>
size_t size_of(T (&)[N]) {
    return N;
}

// 调用
std::vector<int> v;
size_of(v);                       // ✅ 第一个版本
int arr[10];
size_of(arr);                     // ✅ 第二个版本

12.3 enable_if 风格

// 仅对整型启用
template<class T>
std::enable_if_t<std::is_integral_v<T>, T>
half(T x) { return x / 2; }

// 仅对浮点启用
template<class T>
std::enable_if_t<std::is_floating_point_v<T>, T>
half(T x) { return x * 0.5; }

half(10);        // ✅ 整型版
half(3.14);      // ✅ 浮点版

12.4 void_t 检查类型成员（C++17）

template<class...> using void_t = void;

template<class T, class = void>
struct has_size_method : std::false_type {};

template<class T>
struct has_size_method<T, void_t<decltype(std::declval<T>().size())>> : std::true_type {};

static_assert(has_size_method<std::vector<int>>::value);
static_assert(!has_size_method<int>::value);

12.5 C++20 concepts —— SFINAE 终结者

template<class T>
concept Sizeable = requires(T t) { { t.size() } -> std::convertible_to<size_t>; };

template<Sizeable T>
size_t size_of(const T& c) { return c.size(); }

template<class T, size_t N>
size_t size_of(T (&)[N]) { return N; }

// 编译错误信息变成: "T 不满足 concept 'Sizeable'"，而不是 8 屏 SFINAE 噩梦

12.6 真实工程：Detection Idiom

// 通用检测：T 是否有某个方法/字段/操作符
namespace detail {
template<class T, class = void>
struct has_serialize : std::false_type {};
template<class T>
struct has_serialize<T, std::void_t<decltype(std::declval<T>().serialize(std::declval<Buffer&>()))>> : std::true_type {};
}

template<class T>
void save(const T& obj, Buffer& buf) {
    if constexpr (detail::has_serialize<T>::value) {
        obj.serialize(buf);
    } else if constexpr (std::is_trivially_copyable_v<T>) {
        buf.append_bytes(&obj, sizeof(obj));
    } else {
        static_assert(sizeof(T) == 0, "unsupported type");
    }
}

13.1 协议解析 — bit_cast + memcpy 安全字节操作

// 解析 RPC wire 帧头
struct FrameHeader {
    uint32_t magic;
    uint16_t version;
    uint16_t length;
};
static_assert(sizeof(FrameHeader) == 8);

FrameHeader parse_header(const uint8_t* buf) {
    // 用 memcpy / bit_cast 而不是 reinterpret_cast
    FrameHeader hdr;
    std::memcpy(&hdr, buf, sizeof(hdr));
    hdr.magic   = std::bit_cast<uint32_t>(be32_to_host(hdr.magic));
    hdr.version = be16_to_host(hdr.version);
    hdr.length  = be16_to_host(hdr.length);
    return hdr;
}

class Cache {
    std::unordered_map<Key, Value> m_;
public:
    const Value* find(const Key& k) const {
        auto it = m_.find(k);
        return it == m_.end() ? nullptr : &it->second;
    }

    // 非 const 版本 = const 版本 + 去 const (Item 3 of EC++)
    Value* find(const Key& k) {
        return const_cast<Value*>(
            static_cast<const Cache&>(*this).find(k));
    }
};

易错点速查（10 个高频陷阱）

  1. C 风格 cast 隐藏 const_cast 
    const T* p = ...; T* q = (T*)p;     // ⚠️ 默默去 const
  2. 无符号 / 有符号比较 
    int i = -1; unsigned u = 1; if (i < u) {}    // ⚠️ false!
  3. 整型字面量溢出 → 类型变化 
    auto x = 0x80000000;   // 是 unsigned int 不是 int (32-bit 平台)
  4. 整型相除截断 
    double avg = 5 / 2;    // 2.0 不是 2.5
  5. narrowing conversion (列表初始化以外允许) 
    int n = 3.14;          // ⚠️ 默默截断为 3
int m{3.14};           // ✅ 编译错 (列表初始化禁缩窄)
  6. reinterpret_cast 违反严格别名 
    float f = 1.0f; int i = *reinterpret_cast<int*>(&f);  // UB
  7. dynamic_cast 用于非 polymorphic 类 
    struct A {};      struct B : A {};
A* a; dynamic_cast<B*>(a);     // ❌ 编译错 — A 没有虚函数
  8. 函数指针 ↔ void* 不可移植 
    auto* fp = (void(*)())sym;     // ISO C++ 不保证；POSIX dlsym 例外
  9. 用 0 表示 nullptr 
    void f(int); void f(char*);
f(0);    // ⚠️ 调 f(int) 而不是 f(char*) — 用 nullptr
  10. 万能引用劫持 
    template<class T> Ctor(T&& t);
Ctor(short{42});                // 调用模板版本 而不是 Ctor(int)
编译警告必开：-Wall -Wextra -Wsign-conversion -Wfloat-equal -Wsign-compare -Wshadow -Wconversion -Wold-style-cast。能消除上述 80% 的隐患。

面试题集（20 道分层）

基础（必答） · 中级 · 高级（体现深度） · 速答示范（第 12 题）

点击每道题展开参考答案（先自测，再对照）。

15.1 基础（必答）

基础4 种 C++ cast 各做什么? 哪个有运行时检查?
static_cast
编译期已知安全的转换：数值类型、继承上行、void*↔T*、显式触发转换运算符。无运行时检查。
dynamic_cast
唯一有运行时检查：多态类型间安全下行 / 交叉转换，依赖 RTTI；失败时指针返回 nullptr、引用抛 bad_cast。
const_cast
增删 const / volatile 限定。
reinterpret_cast
底层位模式重解释：指针↔整数、不相关指针类型。
基础为什么不应该用 C 风格 cast?
问题
C cast 会依次尝试 const_cast / static_cast / reinterpret_cast 组合，意图不明、可能悄悄做危险的 reinterpret，且无法用 grep 搜索定位。
对策
C++ 四种 cast 各司其职、显式可搜索、受类型系统约束。
基础static_cast 与 dynamic_cast 的区别?
static
编译期完成、零运行时开销、不校验实际类型；多态下行若类型不符是 UB。
dynamic
运行时查 RTTI，安全但有开销，要求类型是多态的（含虚函数）。
基础const_cast 用在哪些场景?
场景
去 const 以对接老式 C API（确信不改）、在 const 成员函数中复用非 const 实现。
红线
修改一个真正 const 的对象仍是 UB —— const_cast 只能用于"对象本身非 const、只是当前引用带 const"的情况。
基础什么是隐式转换? 什么是显式转换?
隐式
编译器自动插入：整型提升、算术转换、派生→基、单参构造、转换运算符。
显式
程序员用 cast 或 explicit 主动触发。隐式方便但易出错，关键构造 / 运算符应标 explicit。

15.2 中级

中级整型提升与算术转换的关系?
整型提升
比 int 小的类型（char/short/bool/enum）先提升为 int / unsigned int。
算术转换
在提升之后，把两操作数转成共同类型（按 rank + 有无符号规则）。提升是通常算术转换的第一步。
中级什么是严格别名规则? 怎么合法做 type punning?
规则
不能通过不兼容类型的指针 / 引用访问对象（char / std::byte 例外），否则 UB；编译器据此做激进优化。
合法手段
memcpy、C++20 std::bit_cast、std::start_lifetime_as(C++23)；union 读非活动成员在 C++ 中并不保证。
中级explicit 关键字解决了什么问题?
作用
阻止单参构造 / 转换运算符参与隐式转换，避免意外转换（如 void f(MyString); f("x") 悄悄构造）与重载二义性。
中级explicit operator bool 与普通 operator bool 区别?
explicit
只在语境布尔转换（if/while/!/&&/||/?:）中生效，安全。
普通
参与所有隐式转换，危险：int x = obj; 会整型提升、obj1 == obj2 退化成 int 比较。
中级下列调用哪个重载? void f(int); void f(double); f(0u);
答案
0u 是 unsigned int。unsigned int→int 与 unsigned int→double 都属"转换"等级、优先级相同 → 二义性，编译错误。须显式 cast 消歧。

15.3 高级（体现深度）

高级dynamic_cast 的开销来源?
来源
运行时遍历继承层次比对 type_info（RTTI），多重 / 虚继承下需更复杂的运行时算法，外加 vtable 间接。比 static_cast 慢一到两个数量级。
高级std::bit_cast 与 reinterpret_cast 的区别? 什么时候用哪个?
bit_cast
按字节复制构造新对象，类型安全、可 constexpr、不违反别名；要求 sizeof 相等且 trivially copyable。
reinterpret
指针 / 引用类型穿越，解引用常违反严格别名 → UB。
口诀
"按位重解释"用 bit_cast，"指针类型转换"用 reinterpret_cast。完整对比见 §15.4。
高级SFINAE 是什么? 给一个 enable_if 的例子
定义
Substitution Failure Is Not An Error：模板替换失败不报错，仅把该重载移出候选集。
例子
template<class T, std::enable_if_t<std::is_integral_v<T>, int> = 0>
void f(T);   // 仅整型可调用；C++20 用 concepts/requires 更清晰
高级用 SFINAE / concepts 检查类型是否有 .size() 方法
concept
template<class T>
concept HasSize = requires(const T& t) {
    { t.size() } -> std::convertible_to<std::size_t>;
};
C++17
用 std::void_t + 偏特化检测表达式 decltype(std::declval<T>().size()) 是否合法。
高级"声明万能引用构造可能劫持其他构造" 怎么办?
原因
万能引用构造 template<class T> C(T&&) 对非 const 左值比拷贝构造 C(const C&) 更匹配，会被错误选中。
修复
用 requires / enable_if 排除自身类型（!std::same_as<std::decay_t<T>, C>），或提供精确重载，或 tag dispatch。
高级在 const 成员函数中如何重用非 const 版本的代码?（const_cast 惯用法）
惯用法
const T& get() const { /* 真正实现 */ }
T& get() { return const_cast<T&>(std::as_const(*this).get()); }
让 const 版做实现、非 const 版转调，避免重复。
C++23
用 deducing this（auto&& self）一个函数即可搞定。
高级用户定义转换链最多允许几次? 为什么?
答案
一个隐式转换序列里最多 1 次用户定义转换。否则 A→B→C 链会组合爆炸、难以推理，标准禁止；需多步必须显式写。
高级列表初始化 int n{3.14} 为什么编译错而 int n = 3.14 不会?
花括号
列表初始化禁止窄化转换（double→int 丢精度）→ 编译错误。
等号
拷贝初始化允许窄化，只是截断为 3（可能伴随警告）。
高级编译器在重载决议时按什么顺序匹配类型转换?
优先级
精确匹配 > 提升 > 标准转换 > 用户定义转换 > 省略号 ...。同级出现多个最佳 → 二义性。
高级什么是"contextual conversion to bool"? 它与 explicit 的关系?
语境转换
在 if/while/for/!/&&/||/?: 语境中，对象会进行"语境布尔转换"，此时即使 operator bool 是 explicit 也会被调用。
关系
这正是 explicit operator bool 的设计点：只放行这些语境，禁止其它一切隐式 bool 转换。

15.4 速答示范（第 12 题）

问：std::bit_cast 与 reinterpret_cast 的区别? 什么时候用哪个?

答：

维度reinterpret_caststd::bit_cast (C++20)
语义把指针/引用看作另一种类型 (类型穿越)按字节复制并解释为新类型
严格别名解引用通常违反 → UB不依赖指针 → 安全
constexpr不能可以 (C++20)
对象生命周期不构造目标对象构造目标对象 (复制构造)
大小要求无源和目标 sizeof 必须相等
典型用途字节访问 (char*)、函数指针、整数↔指针浮点 ↔ 整数 bit 表示、序列化

规则："按位重解释" 用 bit_cast，"指针类型转换" 用 reinterpret_cast。

16.1 4 种 cast 一表速查

cast用途检查时机能去 const?能改类型?
static_cast已知安全的转换 (数值、上行、enum)编译期否是 (兼容)
dynamic_cast多态层次的安全下行运行时 (RTTI)否是 (基↔派)
const_cast加/去 const / volatile—是否
reinterpret_cast不相关类型重解释—否是 (按 bit)
std::bit_cast类型安全 bit 复制编译期/运行时—是 (按 bit)

16.2 转换序列优先级

  1. Exact match (含 lvalue→rvalue / 加减 const)
  2. Promotion (char/short→int, float→double)
  3. Standard conversion (int↔long / int→double / 派→基)
  4. User-defined conversion (≤ 1 次)
  5. Ellipsis (...)

16.3 explicit 用在哪

  • 所有单参构造（除非确实希望隐式转换，如 string）
  • 资源类、领域类型、智能指针
  • 转换运算符（除非确实希望隐式转换）
  • operator bool 几乎都该 explicit
