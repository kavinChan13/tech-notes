01

编译流水线全景

把 main cpp 变成 a

把 main.cpp 变成 a.out / a.exe 不是"一步到位"的魔法，而是 6 个串联阶段。理解这 6 阶段是后续所有讨论（IR / 优化 / 链接 / Sanitizer）的基础。

① Preprocess（预处理）

  • 处理 #include / #define / #if；展开宏；删除注释
  • 输出 纯 C++ 翻译单元（Translation Unit）
  • 看一眼：g++ -E main.cpp -o main.ii

② Frontend（前端）

  • 词法 → 语法 → 语义 三步：tokens → AST → 类型检查 + 模板实例化 + ODR + 重载决议
  • 耗时最多的阶段——C++ 模板和复杂 overload 让前端慢，是 build time 主因

③ IR（中间表示）

  • AST 翻译成编译器自己的 IR，独立于源语言 + 独立于硬件
  • LLVM: .ll（文本）/ .bc（bitcode）
  • GCC: GIMPLE → RTL（两级 IR）
  • 所有跨语言 / 跨平台优化都基于 IR 做

④ Optimize（优化）

  • 跑一连串 passes：inline / DCE / GVN / LICM / loop vectorize / SROA / ...
  • -O0/-O1/-O2/-O3/-Os/-Oz/-Ofast 决定开哪些 passes（§3）
  • LTO 让优化跨翻译单元（§4）

⑤ Backend（后端 / Codegen）

  • IR → 目标平台汇编（x86-64 / ARM64 / RISC-V / WebAssembly）
  • 关键工作：指令选择 + 寄存器分配 + 调度
  • 输出 .s → 汇编为 .o

⑥ Link（链接）

  • 多个 .o + 静态库 .a + 动态库 .so / .dll 合并
  • 解析未定义符号、重定位、生成最终可执行（§4 详）
实战 tip：gcc -v 让编译器把每阶段的子进程调用都打印出来，能看到 cc1 / cpp / as / collect2 / ld 各自跑了什么。
02

LLVM IR 速览

从 C++ 到 IR · SSA 的核心：phi 节点 · vs GCC GIMPLE

LLVM IR 是 SSA（Static Single Assignment）形式的类汇编语言，但类型化 + 跨平台。理解 IR = 理解编译器优化的"通用语言"。

从 C++ 到 IR

// hello.cpp
int add(int a, int b) {
  return a + b;
}

跑 clang -S -emit-llvm hello.cpp -O1 -o hello.ll 得到：

; hello.ll (简化)
define i32 @_Z3addii(i32 %a, i32 %b) {
entry:
  %0 = add nsw i32 %a, %b
  ret i32 %0
}

关键观察：

  • i32 —— 类型化（传统汇编没有）
  • %a / %b / %0 —— SSA virtual register（无穷多个，每个只赋值一次）
  • nsw —— "no signed wrap"，告诉优化器假设不溢出（开启更激进优化）
  • 函数名 _Z3addii —— Itanium ABI mangled name

SSA 的核心：phi 节点

SSA 要求每个变量只赋值一次。但 if/else 或循环里两条分支都给同一变量赋值怎么办？答案是 phi node —— "根据走的哪条分支挑选合适的值"。

vs GCC GIMPLE

维度LLVM IRGCC GIMPLE / RTL
表示层级单一 IR，所有 phase 共用两级：GIMPLE（高级）+ RTL（接近汇编）
SSA原生 SSAGIMPLE 是 SSA
类型化强类型类型化但不如 LLVM 干净
外部可见性设计为 library，外部工具可读写主要内部用，外部访问难
跨语言Swift / Rust / Julia / 部分 Go 都用 LLVM主要服务 C/C++/Fortran/Ada
可视化godbolt.org 实时；opt -S 看 pass 输出需 -fdump-tree-* 系列开关
godbolt 实战：godbolt.org 选 LLVM IR 输出，左侧 C++，右侧 IR + 汇编。所有 C++ 性能问题都该先去 godbolt 看一眼实际生成代码。
03

优化 Pass 重点

高 ROI 的 10 个 Pass · 看 Pass 在干什么 · -O0 / -O1 / -O2 / -O3 / -Os / -Oz / -Ofast

LLVM 把"优化"组织成 pass 流水线。每个 pass 干一件事（如 inline、消除死代码）。-O2 大致跑 ~150 个 pass，-O3 多 20-30 个。

高 ROI 的 10 个 Pass

Pass干什么典型收益
Inliner把函数体替换 call5-20%（消除 call overhead + 暴露更多优化机会）
DCEDead Code Elimination，删未用代码清理 + 减小 binary size
GVNGlobal Value Numbering，识别等价表达式只算一次常见 hot path 1-5%
LICMLoop Invariant Code Motion，把循环里不变的提出去循环密集代码 10-30%
SROAScalar Replacement of Aggregates，把 struct 字段拆成 SSA消除大量 load/store
Loop Vectorizer把循环改成 SIMD 指令数值 / 数据密集 2-8×
Loop Unrolling循环展开循环开销大时 1.5-3×
InstCombine局部模式重写（如 x*2 → x<<1）持续小幅改善
SimplifyCFG合并 / 删冗余分支清理 + 改善后续优化
Devirtualization已知类型时把虚调用变直接调用面向对象代码 5-15%

看 Pass 在干什么

# 看每个 pass 之前 / 之后的 IR 变化
clang -O2 -mllvm -print-after-all hello.cpp 2> passes.txt

# 只跑某些 pass
opt -S -passes='inline,gvn,licm' hello.ll -o opt.ll

# 看 pass 详细 timing
clang -O2 -ftime-report hello.cpp
🎚️ 交互式演示：同一段代码，不同 -O 级别编译器做了什么
切换优化级别，观察编译器如何"看穿"代码：常量折叠、内联、死代码删除、向量化…… 右侧是近似汇编与指令数。生活化类比：-O0 是逐字照做的实习生，-O3 是会抄近路的老师傅。
源代码（始终不变）
int sum() { int s = 0; for (int i = 0; i < 4; ++i) s += i * 2; return s; // = 0+2+4+6 = 12 }
近似生成的汇编 / IR
指令数：— 关键 pass：—

-O0 / -O1 / -O2 / -O3 / -Os / -Oz / -Ofast

级别含义用途
-O0不优化，debug 友好（变量都在）开发 / 调试
-O1基本优化，编译快极少用
-O2主流生产级99% 项目 release 用这个
-O3更激进（loop unroll / vectorize 更狠）数值密集 / HPC；可能 binary size 涨
-Os优化 size（关闭 size 不友好的 pass）embedded / mobile
-Oz极端 size 优化flash 受限的 IoT / wasm
-OfastO3 + 放宽 IEEE 浮点语义科学计算（接受不严格 IEEE）
-Ofast 警告：会启用 -ffast-math，让编译器假设 无 NaN / Inf / 浮点结合律可重排。对很多正确性敏感代码（比较、NaN 检测）是 UB-诱发—— 别在通用代码默认开。

链接器内部

主流 linker 对比 · 链接速度直观对比（100 万行 C++ 项目） · 符号解析（symbol resolution） · GC sections（死代码段消除） · LTO（Link-Time Optimiz…

链接器把多个 .o + 库合并成可执行 / 共享库。看似简单，实际有几大复杂点：符号解析、重定位、section GC、LTO。

主流 linker 对比

Linker来源特点2026 现状
GNU ldbinutils 原版历史最老，稳定但慢仍是 GCC 默认
goldGoogle 2008C++ heavy 项目快 5-10×维护停滞，逐渐被 lld 取代
lldLLVM极快（多线程），支持 ELF / Mach-O / COFF / Wasm2026 主流，FreeBSD / Android NDK 默认
moldRui Ueyama 2021比 lld 再快 2-5×，并行度极高大型项目（如 Chromium）已迁移
MSVC link.exeMicrosoftWindows COFF / PE 标准仍是 Windows 主流

链接速度直观对比（100 万行 C++ 项目）

切链接器：g++ -fuse-ld=lld 或 -fuse-ld=mold。真实经验：把一个 100 万行 C++ 项目从 ld（45s 链接）→ lld（8s）→ mold（3s），增量构建 dev 体验提升 ×15。

符号解析（symbol resolution）

每个 .o 含定义符号（function / global var 的实现）+ 未定义符号（call 出去 / 引用其他 TU）。链接器扫所有 .o + 库，把每个未定义符号匹配到一个定义。

# 看 .o 文件的符号
nm main.o            # 列符号
nm -C main.o         # C++ demangle
nm -u main.o         # 只看 undefined
nm -D libfoo.so      # 看动态库导出符号

# 看 ELF 详细信息
readelf -Ws main.o   # 完整 symbol table
readelf -d a.out     # 动态段（NEEDED / RPATH / etc）

# 反汇编
objdump -d main.o    # 看 codegen 结果
objdump -t main.o    # 符号表
objdump -r main.o    # 重定位条目

GC sections（死代码段消除）

链接时如果一个函数 / 全局 var 没人调，理论上可以扔掉减小 binary。但 linker 默认不做——因为分不清"没人调"还是"通过函数指针 / dlsym 调"。

# 编译时把每个函数 / 数据放独立 section
g++ -ffunction-sections -fdata-sections ...

# 链接时回收没引用的 section
g++ -Wl,--gc-sections ...

# 看具体回收了哪些
g++ -Wl,--print-gc-sections ...

典型 binary size 减小 20-40%（依赖代码 dead-code 比例）。但要 __attribute__((used)) 保留"看似无引用但运行时反射调"的函数。

LTO（Link-Time Optimization）

核心：编译时不立即生成 native code，而是把 IR 存到 .o 文件；链接时所有 .o 的 IR 一起优化，跨 TU 做 inline / DCE / devirt。

模式原理编译时间性能收益
无 LTO每 TU 各自优化，链接时只 resolve symbolsbaselinebaseline
Full LTO所有 IR 一起放内存，全局优化×5-10+10-15%
ThinLTO分 module 增量优化，带 summary×1.5-2+8-12%（80% of full LTO）
2026 实践：production binary 都开 ThinLTO（性价比最高）。开法：clang -flto=thin ... 或 GCC -flto（GCC 没分 thin / full）。

Sanitizer 实现原理

大 Sanitizer 全家桶 · ASan 工作原理图解 · TSan：vector clock + happens-before · MSan：未初始化内存读 · HWASan：ARM 硬件辅助

Sanitizer 是 Google 2012 后陆续开源的编译期插桩 + runtime 检测方案。比 Valgrind 快 10-50×，能在 CI / 部分生产场景跑。

5 大 Sanitizer 全家桶

Sanitizer抓什么CPU 开销Memory 开销核心机制
ASan越界 / UAF / 栈帧越界×2×3shadow + red zone
TSandata race×5-15×5-10vector clock + happens-before
MSan未初始化内存读×3×2bit-level shadow + origin tracking
UBSan各种 UB×1.05~0trap-on-error 插桩
HWASan (ARM)越界 / UAF (硬件辅助)×1.1×1.1Top-Byte-Ignore / MTE 标签

ASan 工作原理图解

核心机制：shadow memory + red zone。每 8 字节真实内存对应 1 字节 shadow，标记"可访问性"。每个 heap allocation 前后各 ~128 字节 poison 区。

# 编译 + 运行
g++ -fsanitize=address -fno-omit-frame-pointer -g -O1 buggy.cpp
./a.out
# 越界 / UAF 时输出详细 stack trace + 哪次 alloc / free

TSan：vector clock + happens-before

  • 每个内存位置的 shadow 记录"上次访问者 + 那时的 vector clock"
  • 每线程维护自己的 vector clock；同步操作（mutex / atomic / cv）合并双方 clock
  • 访问内存时检查：上次访问的 clock 是否 ≤ 当前 → 否 = 没 happens-before → race
  • 用的是 FastTrack 算法（2009）—— 读用单 epoch，写用 full vector，加速 5-10×
  • 开销 CPU 5-15×，比 ASan 贵；不能跟 ASan 同时开

MSan：未初始化内存读

  • shadow 记录每 bit 是否有效（被写入过）；分支 / 算术 / 系统调用读到未初始化数据时报
  • origin tracking：进一步记录"这块未初始化内存是哪个 alloc 没初始化的" → 报告更易定位
  • 要求整个 binary（含依赖 lib）都用 MSan 编译，否则 false positive 多 → 实战部署难

HWASan：ARM 硬件辅助

  • 用 ARM Top-Byte-Ignore / MTE（Memory Tagging Extension）做 tagging
  • 比 ASan shadow 便宜 5-10×；开销 5-15%，生产可用
  • 2026 年 Pixel / 部分 Android 设备已默认开
📍 想深入：参见 Sanitizer 内部实现 子页 —— shadow map 内存映射、TSan FastTrack 算法、MSan origin tracking、HWASan MTE、GWP-ASan 采样。

ABI 工具栈

核心工具速查 · ABI 对比工具 · 看 mangled 名字

C++ ABI 兼容性是共享库设计的最难部分。配套工具能帮你 detect / debug。

核心工具速查

工具用途典型用法
nm列符号表nm -CD libfoo.so 看导出符号 + demangle
readelf看 ELF 详细信息（section / symbol / dynamic）readelf -Wa main.o
objdump反汇编 + 重定位 + sectionobjdump -d main.o
c++filtdemangle 单个符号名echo _Z3fooi | c++filt → "foo(int)"
ldd列动态依赖ldd a.out
strings列 binary 中字符串查 hardcoded 路径 / 版本号
strip去 debug 信息（减小 binary）release build 用 + 单独保留 debuginfo

ABI 对比工具

# libabigail / abidiff —— 精确比较两版本 ABI
abidiff libfoo-v1.so libfoo-v2.so

# abi-compliance-checker —— 生成 HTML 报告
abi-compliance-checker -l libfoo -old v1.xml -new v2.xml
CI 集成：每 PR 跑 abidiff baseline.so HEAD.so，若 ABI break 必须 explicit bump major 版本号。典型流程：
  1. release 时记下 baseline.so 到工件存储
  2. PR 构建后 abidiff vs baseline
  3. break 触发 CI 失败，开发显式打标签 [ABI-BREAK] + bump major 才能合

看 mangled 名字

# Itanium ABI（GCC/Clang Linux）
$ echo _Z3fooi | c++filt
foo(int)

$ echo _ZN3foo3barEv | c++filt
foo::bar()

# MSVC ABI（完全不同语法）
$ undname ?foo@@YAHH@Z
int __cdecl foo(int)
07

编译时间优化

组合实战：100 万行项目 build time 演进 · 大型 C++ 项目 build time 是 dev productivity 头号杀手…

大型 C++ 项目 build time 是 dev productivity 头号杀手。10 万行 C++ 项目从慢构建（30 min）到快构建（< 5 min）的工具 + 技巧：

Precompiled Header (PCH)

  • 把稳定的头（STL / boost）预编译成 .pch / .gch，每 TU 复用
  • 典型加速 30-50%（看 STL 使用密度）
  • CMake: target_precompile_headers(...)

C++20 Modules

  • 替代 #include：import std;
  • 本质：预解析的二进制接口，不再重新 parse 整个 stdlib
  • 2026 现状：MSVC / Clang 16+ / GCC 14+ 都支持，但生态采用慢

Unity Build（合并源文件）

  • 多个 .cpp 合成一个大 .cpp，减少 TU 数
  • 加速 2-3×，但 incremental build 较差
  • CMake: UNITY_BUILD ON

ccache / sccache

  • ccache：本地缓存（按内容 hash），增量 build 加速 3-10×
  • sccache：支持 S3 / Redis 远程缓存 → 团队共享
  • 用法：export CC="ccache gcc"

distcc / icecream

  • 把 compile job 分发到多机；linker 仍本地
  • 团队规模 > 5 工程师时开始有意义

Ninja + IWYU

  • Ninja：替代 make，快 10-50×。cmake -G Ninja
  • IWYU：删冗余 #include，build time -10-20%

组合实战：100 万行项目 build time 演进

100 万行 C++ 项目（platform agent 历史）从 4 min incremental build → 40 sec，组合手段：Ninja + lld + ccache（团队共享 sccache）+ PCH + IWYU + ThinLTO。团队 productivity 显著上升。
