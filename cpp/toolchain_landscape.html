① 编译流水线全景

把 main.cpp 变成 a.out / a.exe 不是一步到位的"魔法"，而是 6 个串联阶段。理解这 6 阶段是后续所有讨论（IR / 优化 / 链接 / Sanitizer）的基础。

① Preprocess.cpp → .ii
→
② Frontendparse / sema
→
③ IR.ll / .bc
→
④ Optimizepasses
→
⑤ Backend.s / .o
→
⑥ Linka.out

① Preprocess（预处理）

  • 处理 #include / #define / #if 等指令；展开宏；删除注释
  • 输出 纯 C++ 代码（一个独立的 翻译单元 / Translation Unit）
  • 看一眼：g++ -E main.cpp -o main.ii

② Frontend（前端）

  • 词法分析（lexer / tokenizer）→ token 流
  • 语法分析（parser）→ AST（Abstract Syntax Tree）
  • 语义分析（semantic analysis）→ 类型检查、模板实例化、ODR、重载决议
  • 这一步耗时最多—— C++ 模板和复杂 overload 让前端慢，是 build time 的主因

③ IR（Intermediate Representation）

  • AST 翻译成 编译器自己的中间表示，独立于源语言 + 独立于目标硬件
  • LLVM：.ll（文本）/ .bc（bitcode）
  • GCC：GIMPLE → RTL（两级 IR）
  • 所有 跨语言 / 跨平台优化都基于 IR 做

④ Optimize（优化）

  • 跑一连串 passes：inline / DCE / GVN / LICM / loop vectorize / SROA / ...
  • O0 / O1 / O2 / O3 / Os / Oz / Ofast 决定开哪些 passes（参见 §3）
  • LTO（Link-Time Optimization）让优化跨翻译单元（参见 §4）

⑤ Backend（后端 / Codegen）

  • IR → 目标平台汇编（x86-64 / ARM64 / RISC-V / WebAssembly）
  • 关键工作：指令选择 + 寄存器分配 + 调度
  • 输出 .s，汇编后 .o

⑥ Link（链接）

  • 多个 .o + 静态库 .a + 动态库 .so / .dll 合并
  • 解析 未定义符号、重定位、生成最终可执行
  • 详细见 §4
实战 tip：用 -v 参数让 gcc/clang 把每阶段的子进程调用都打印出来，能看到 cc1 / cpp / as / collect2 / ld 各自跑了什么。

② LLVM IR 速览

LLVM IR 是 SSA（Static Single Assignment）形式的类汇编语言，但类型化 + 跨平台。理解 IR = 理解编译器优化的"通用语言"。

从 C++ 到 IR

// hello.cpp
int add(int a, int b) {
  return a + b;
}

跑 clang -S -emit-llvm hello.cpp -O1 -o hello.ll 得到：

; hello.ll (简化)
define i32 @_Z3addii(i32 %a, i32 %b) {
entry:
  %0 = add nsw i32 %a, %b
  ret i32 %0
}

关键观察：

  • i32 —— 类型化（不像传统汇编）
  • %a / %b / %0 —— SSA virtual register（无穷多个，每个只赋值一次）
  • nsw —— "no signed wrap"，告诉优化器假设不溢出（开启更激进优化）
  • 函数名 _Z3addii —— Itanium ABI mangled name

vs GCC GIMPLE

维度LLVM IRGCC GIMPLE / RTL
表示层级单一 IR（LLVM IR），不同优化 phase 都在它上面两级：GIMPLE（高级）+ RTL（接近汇编）
SSA原生 SSAGIMPLE 是 SSA
类型化强类型类型化但表达不如 LLVM 干净
外部可见性设计为 library，外部工具可读写主要内部用，外部访问较难
跨语言Swift / Rust / Julia / 部分 Go 都用 LLVM主要服务 C/C++/Fortran/Ada
调试 / 可视化godbolt.org 实时；opt -S 看 pass 输出需 -fdump-tree-* 系列开关

SSA 入门：phi 节点

SSA 要求每个变量只赋值一次。但 if / else / 循环里两条分支都给同一变量赋值怎么办？答案是 phi node —— "根据走的哪条分支挑选合适的值"。

// C++
int abs(int x) {
  return x < 0 ? -x : x;
}

; LLVM IR
define i32 @abs(i32 %x) {
  %cmp = icmp slt i32 %x, 0
  br i1 %cmp, label %neg, label %pos
neg:
  %1 = sub nsw i32 0, %x
  br label %exit
pos:
  br label %exit
exit:
  %r = phi i32 [ %1, %neg ], [ %x, %pos ]   ; 从哪条边来就取哪个值
  ret i32 %r
}
godbolt 实战：godbolt.org 选 LLVM IR 输出，左侧 C++，右侧 IR + 汇编。所有 C++ 性能问题都该先去 godbolt 看一眼实际生成代码。

③ 优化 Pass 重点

LLVM 把"优化"组织成 pass 流水线。每个 pass 干一件事（如 inline、消除死代码）。-O2 大致跑 ~150 个 pass，-O3 多 20-30 个。

高 ROI 的 10 个 Pass

Pass干什么典型收益
Inliner把函数体替换 call5-20%（消除 call overhead + 暴露更多优化机会）
DCE (Dead Code Elimination)删除从不被用的代码清理 + 减小 binary size
GVN (Global Value Numbering)识别等价表达式只算一次常见 hot path 1-5%
LICM (Loop Invariant Code Motion)把循环里不变的表达式提出循环循环密集代码 10-30%
SROA (Scalar Replacement of Aggregates)把 struct 字段拆成单独 SSA 值消除大量 load/store
Loop Vectorizer把循环改成 SIMD 指令数值 / 数据密集 2-8×
Loop Unrolling循环展开循环开销大时 1.5-3×
InstCombine局部模式重写（如 x * 2 → x << 1）持续小幅改善
SimplifyCFG合并 / 删冗余分支清理 + 改善后续优化
Devirtualization已知类型时把虚调用变直接调用面向对象代码 5-15%

看 Pass 在干什么

# 看每个 pass 之前 / 之后的 IR 变化
clang -O2 -mllvm -print-after-all hello.cpp 2> passes.txt

# 只跑某些 pass
opt -S -passes='inline,gvn,licm' hello.ll -o opt.ll

# 看 pass 详细 timing
clang -O2 -ftime-report hello.cpp

O0 / O1 / O2 / O3 / Os / Oz

级别含义用途
-O0不优化，debug 友好（变量都在）开发 / 调试
-O1基本优化，编译快极少用
-O2主流生产级99% 项目 release 用
-O3更激进（loop unroll / vectorize 更狠）数值密集 / HPC；可能 binary size 涨
-Os优化 size（关闭某些 size 不友好的 pass）embedded / mobile
-Oz极端 size 优化flash 受限的 IoT / wasm
-OfastO3 + 放宽 IEEE 浮点语义科学计算（接受不严格 IEEE）
-Ofast 警告：会启用 -ffast-math，让编译器假设 无 NaN / Inf / 浮点结合律可重排。这对很多正确性敏感代码（比较、NaN 检测）是UB-诱发—— 别在通用代码默认开。

④ 链接器内部

链接器把多个 .o + 库合并成可执行 / 共享库。看似简单，实际有几大复杂点：符号解析、重定位、section GC、LTO。

主流 linker 对比

Linker来源特点2026 现状
GNU ldbinutils 原版历史最老，稳定但慢仍是 GCC 默认
goldGoogle 2008C++ heavy 项目快 5-10×维护停滞，逐渐被 lld 取代
lldLLVM极快（多线程），支持 ELF / Mach-O / COFF / Wasm2026 主流，FreeBSD / Android NDK 默认
moldRui Ueyama 2021比 lld 再快 2-5×，并行度极高大型项目（如 Chromium）已迁移
MSVC link.exeMicrosoftWindows COFF / PE 标准仍是 Windows 主流
切链接器：g++ -fuse-ld=lld 或 -fuse-ld=mold。真实经验：把一个 100 万行 C++ 项目从 ld（45s 链接）→ lld（8s）→ mold（3s），增量构建 dev 体验 ×15。

符号解析（symbol resolution）

每个 .o 含 定义符号（function / global var 的实现）+ 未定义符号（call 出去 / 引用其他 TU）。链接器扫所有 .o + 库，把每个未定义符号匹配到一个定义。

# 看 .o 文件的符号
nm main.o            # 列符号
nm -C main.o         # C++ demangle
nm -u main.o         # 只看 undefined
nm -D libfoo.so      # 看动态库导出符号

# 看 ELF 详细信息
readelf -Ws main.o   # 完整 symbol table
readelf -d a.out     # 动态段（NEEDED / RPATH / etc）

# 反汇编
objdump -d main.o    # 看 codegen 结果
objdump -t main.o    # 符号表
objdump -r main.o    # 重定位条目

GC sections（死代码段消除）

链接时如果一个函数 / 全局 var 没人调，理论上可以扔掉减小 binary。但 linker 默认不做—— 因为它分不清"没人调"还是"通过函数指针 / dlsym 调"。

# 编译时把每个函数 / 数据放独立 section
g++ -ffunction-sections -fdata-sections ...

# 链接时回收没引用的 section
g++ -Wl,--gc-sections ...

# 看具体回收了哪些
g++ -Wl,--print-gc-sections ...

典型 binary size 减小 20-40%（依赖代码 dead-code 比例）。但要 __attribute__((used)) 保留有些"看似无引用但其实运行时反射调"的函数。

LTO（Link-Time Optimization）

核心：编译时不立即生成 native code，而是 把 IR 存到 .o 文件；链接时 所有 .o 的 IR 一起优化，跨 TU 做 inline / DCE / devirt。

模式原理编译时间性能收益
无 LTO每 TU 各自优化，链接时只 resolve symbolsbaselinebaseline
Full LTO所有 IR 一起放内存，全局优化×5-10+10-15%
ThinLTO分 module 增量优化，带 summary×1.5-2+8-12%（80% of full LTO）
2026 实践：production binary 都开 ThinLTO（性价比最高）。开法：clang -flto=thin ... 或 GCC -flto（GCC 没分 thin / full）。

⑤ Sanitizer 实现原理

Sanitizer 是 Google 2012 后陆续开源的 编译期插桩 + runtime 检测方案。比 Valgrind 快 10-50×，能在 CI / 部分生产场景跑。

ASan（AddressSanitizer）—— 抓内存越界 / UAF

核心机制：shadow memory + red zone。

  • Shadow memory：1 字节影子标记 8 字节真实内存的"可访问性"。0 = 全可用；1-7 = 前 N 字节可用；负值 = 被 poison（无效）。
  • Red zone：每个 heap allocation 前后各 ~128 字节 poison 区，越界访问立刻报。
  • UAF：free 后不立即归还堆，进 quarantine 队列 + shadow 标 freed；再访问报警。
  • 插桩开销：每次内存访问前生成 ~10 行汇编查 shadow。CPU ~2×，memory ~3×。
# 编译 + 运行
g++ -fsanitize=address -fno-omit-frame-pointer -g -O1 buggy.cpp
./a.out
# 越界 / UAF 时输出详细 stack trace + 哪次 alloc / free

TSan（ThreadSanitizer）—— 抓 data race

核心：vector clock + happens-before。

  • 每个内存位置的 shadow 记录"上次访问者 + 那时的 vector clock"。
  • 每线程维护自己的 vector clock；同步操作（mutex / atomic / cv）合并双方 clock。
  • 访问内存时检查"上次访问"的 clock 是否 ≤ 当前 → 否 = 没 happens-before → race。
  • 用的是 FastTrack 算法（2009）—— 读用单 epoch，写用 full vector，加速 5-10×。
  • 开销 CPU 5-15×，比 ASan 贵；不能跟 ASan 同时开。

MSan（MemorySanitizer）—— 抓未初始化内存读

  • shadow 记录每 bit 是否 有效（被写入过）；分支 / 算术 / 系统调用读到未初始化数据时报。
  • origin tracking：进一步记录"这块未初始化内存是哪个 alloc 没初始化的" → 报告更易定位
  • 要求 整个 binary（含依赖 lib）都用 MSan 编译，否则 false positive 多 → 实战部署难

UBSan（UndefinedBehaviorSanitizer）—— 抓 UB

  • 插桩检测各种 UB：signed overflow / null deref / shift > width / 浮点除零 / 类型 puning / vptr / alignment 等
  • 开销极低（CPU 1.05×），生产可常开（仅 trap-on-error 模式）
  • 用法：g++ -fsanitize=undefined ...

HWASan（Hardware-assisted ASan，ARM）

  • 用 ARM Top-Byte-Ignore / MTE（Memory Tagging Extension）做 tagging
  • 比 ASan shadow 便宜 5-10×；开销 5-15%，生产可用
  • 2026 年 Pixel / 部分 Android 设备已默认开

⑥ ABI 工具栈

C++ ABI 兼容性是共享库设计的最难部分。配套工具能帮你 detect / debug。

核心工具

工具用途例
nm列符号表nm -CD libfoo.so 看导出符号 + demangle
readelf看 ELF 详细信息（section / symbol / dynamic）readelf -Wa main.o
objdump反汇编 + 重定位 + sectionobjdump -d main.o
c++filtdemangle 单个符号名echo _Z3fooi | c++filt → "foo(int)"
ldd列动态依赖ldd a.out
strings列 binary 中字符串查 hardcoded 路径 / 版本号
strip去 debug 信息（减小 binary）release build 用 + 单独保留 debuginfo

ABI 对比工具

# libabigail / abidiff —— 精确比较两版本 ABI
abidiff libfoo-v1.so libfoo-v2.so

# abi-compliance-checker —— 生成 HTML 报告
abi-compliance-checker -l libfoo -old v1.xml -new v2.xml

CI 集成：每 PR 跑 abidiff baseline.so HEAD.so，若 ABI break 必须 explicit bump major 版本号。典型流程：

  1. release 时记下 baseline.so 到工件存储
  2. PR 构建后 abidiff vs baseline
  3. break 触发 CI 失败，开发显式打标签 [ABI-BREAK] + bump major 才能合

看 mangled 名字

# Itanium ABI（GCC/Clang Linux）
$ echo _Z3fooi | c++filt
foo(int)

$ echo _ZN3foo3barEv | c++filt
foo::bar()

# MSVC ABI（完全不同语法）
$ undname ?foo@@YAHH@Z
int __cdecl foo(int)

⑦ 编译时间优化

大型 C++ 项目 build time 是 dev productivity 头号杀手。10 万行 C++ 项目从慢构建（30 min）到快构建（< 5 min）的工具 + 技巧：

Precompiled Header（PCH）

  • 把稳定的头（STL / boost）预编译成 .pch / .gch，每个 TU 复用
  • 典型加速 30-50%（看 STL 使用密度）
  • CMake：target_precompile_headers(target PRIVATE <string> <vector> ...)

C++20 Modules

  • 替代 #include：import std; / import mymodule;
  • 本质：预解析的二进制接口—— 不再每 TU 重新 parse 整个 stdlib
  • 2026 现状：MSVC / Clang 16+ / GCC 14+ 都支持，但生态采用慢（CMake / Conan / vcpkg 支持仍在 maturing）
  • 新项目可尝试；老项目移植代价高（要重写 #include 风格）

Unity Build（合并源文件）

  • 把多个 .cpp 合成一个大 .cpp，减少 TU 数（少前端解析开销）
  • 典型加速 2-3×，但 incremental build 较差（一个文件改要重编整个 unity）
  • CMake：set_target_properties(target PROPERTIES UNITY_BUILD ON)

ccache / sccache —— 编译结果缓存

  • ccache：本地缓存（按文件内容 hash），增量 build 加速 3-10×
  • sccache：Mozilla 的 ccache 替代，支持 S3 / Redis 远程缓存 → 团队共享
  • 用法：export CC="ccache gcc" 或 CMake set(CMAKE_C_COMPILER_LAUNCHER ccache)

distcc / icecream —— 分布式构建

  • distcc：把 compile job 分发到多机；linker 仍本地
  • icecream：SuSE 改良版 distcc，自动 scheduler + cross-arch
  • 团队规模 > 5 工程师时开始有意义

Ninja（替代 make）

  • 更快的 build executor（make 的~10-50×）
  • CMake 生成器：cmake -G Ninja
  • 2026 默认应该用 Ninja，make 只在传统场景

IWYU（Include What You Use）

  • 静态分析每个 TU 的 真正用了哪些头，删冗余 #include
  • 典型项目能删 30-50% includes → build time -10-20%

组合实战

真实经验：100 万行 C++ 项目（NETCONF agent 历史）build time 演进：

阶段配置full buildincremental
baselinemake + ld + 无 PCH32 min4 min
+ Ninja + lld22 min1.5 min
+ ccache（团队共享 sccache）22 min15 sec（缓存命中）
+ PCHSTL / Boost 预编译14 min1 min
+ IWYU 清理砍 40% includes9 min40 sec
+ ThinLTOrelease only11 min—

总 dev incremental build 从 4 min → 40 sec，团队 productivity 显著上升。
