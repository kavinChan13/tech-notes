为什么是你 · 3 个反直觉事实

从招聘官视角说清楚 — 为什么 C++ 背景是 AI Infra 最贵的画像

大部分 C++ 工程师以为"AI 跟我没关系，我又不懂 ML"。这是错的。AI Infra 这个 2024-2026 的最大缺口岗位，不要 ML PhD，要的就是 C++ + 系统编程 + 性能优化背景。本章帮你看清你的位置。

0.1 三个反直觉事实

事实 1 · AI Infra 比 ML Eng 缺口大 5 倍

所有公司都需要 AI Infra，但只有头部公司养 ML Eng。每个用 vLLM/SGLang/TRT-LLM 的团队都缺懂 CUDA 的 senior C++。供需比 1:5。

事实 2 · ML PhD 反而抢不过你

PhD 强在 modeling，弱在工程。AI Infra 80% 是工程：CUDA / 分布式 / profiling / 编译。你的 lock-free queue / memory pool / cache 优化经验，PhD 学 3 年都学不会。

事实 3 · ML 只需懂"够用"

AI Infra 不需要你会推导 backprop / 不需要会 fine-tune。需要懂的：Transformer 算什么 / KV-Cache 是什么 / attention 是 memory-bound。这些 1 周能学完。

C++ ↔ CUDA · 6 个核心概念映射

你已经懂的 → 你需要补的（学 CUDA 不是从零，是改名字）

CUDA 的"难"很大部分是命名陌生。其实绝大多数概念你都见过 — 只是叫法不同。把这 6 个映射记住，CUDA 上手时间砍 70%。

你已懂（C++）
新名字（CUDA）
差异点 / 学这里就够
std::thread / 线程池
→
CUDA thread / Warp (32 thread)
CUDA 是 32 个 thread 锁步执行（SIMT）。关键差异：分支 divergence 会让一半 thread 空跑 → Y01 §4-5
cache-friendly 内存访问
→
memory coalescing
GPU 上 相邻 thread 必须访问相邻地址，否则带宽降 10x。跟 CPU cache line 同源思想 → Y01 §3
SIMD / AVX-512 intrinsics
→
Tensor Core / WMMA / mma.sync
都是"一条指令算多个数"。Tensor Core 算 4×4 矩阵乘，比 CUDA core 快 8-16x → Y01 §5
lock-free queue / Disruptor
→
Continuous Batching (vLLM scheduler)
"有任务就上，完成就下，不阻塞"的同样思想。你写过的 SPSC/MPMC 队列经验直接 transfer → Y02 §4
memory pool / slab allocator
→
PagedAttention (vLLM KV 池)
"固定 block 大小按需分配，free 回收"。vLLM 把 KV-Cache 按 OS 虚拟页方式管 → Y02 §3
perf / vtune / gperftools
→
Nsight Systems / Nsight Compute / nvprof
同样"采样 → 火焰图 → roofline"流程。命令换但方法论一样 → Y01 §7
看到 CUDA kernel __global__ / __shared__ / threadIdx.x 别慌 — 那只是把"线程 / cache / 队列"重新命名。本质都是"在 1000 个并发执行单元上协调内存访问 + 计算"，跟你写多线程 C++ 一回事。

1.1 一份"反派词典" · 别被这些词唬住

听着唬人的术语它其实就是
Streaming Multiprocessor (SM)= CPU core 的 GPU 版（H100 有 132 个）
Warp Scheduler= OS 的线程调度器（但调度 warp 不调度 thread）
HBM (High Bandwidth Memory)= GPU 的 DRAM（带宽 3 TB/s）
Shared Memory / SMEM= 每个 SM 自己的 L1 cache（你能手动管理）
Kernel Launch Overhead= CUDA 版的 syscall 开销（5-10 µs）
CUDA Graph= 把多个 kernel 调用预编排，省 launch 开销（类似你写过的 batched ioctl）
NCCL= MPI 在 GPU 上的实现 + GPUDirect RDMA 优化
Triton (OpenAI)= Python DSL 写 CUDA kernel，编译器帮你做 tiling（不是 Triton Inference Server）

W1
Week 1 · GPU 速通
从 0 到能改 CUDA kernel · 约 20 hr

3.1 Day 1-2 · 读 Y01 全文（6 hr）

必读：Y01 · GPU/CUDA 给 C++ 工程师 全文 11 章。重点：

  • Ch1-2：GPU 硬件模型（SM / Warp / SIMT）— 跟 CPU 类比就懂
  • Ch3：memory hierarchy（HBM / L2 / SMEM / register）— 跟 L1/L2/L3 cache 同构
  • Ch4：编程模型（grid / block / thread）— 你只需记 "thread 是 worker，block 是 worker 组，grid 是总任务"
  • Ch6：手写 3 个 kernel · 这是 Week 1 的核心实战

3.2 Day 3-4 · 手写 3 个 CUDA kernel（8 hr）

三个 kernel 难度递进：

kernel难度学到什么面试用
vector add (a + b = c)★thread / block / grid 基础语法能口述
naive matrix multiply★★2D 索引 + global memory 访问白板能写
tiled matrix multiply★★★shared memory + __syncthreads + tile 优化这个让面试官眼前一亮

实践技巧：用 Google Colab T4 GPU 免费，不需要本地装 CUDA。装 nvcc4jupyter 直接在 notebook 里跑。

3.3 Day 5-6 · 读 1 个真实 vLLM CUDA kernel（4 hr）

推荐：vLLM 的 paged_attention_v1_kernel（attention_kernels.cu）。这是 vLLM 性能核心，~500 行 CUDA。

读法：

  1. 不要从头读到尾。先看函数签名理解输入输出
  2. 画一张数据流图：输入 query / key / value / block_table → 输出 output
  3. 找出"哪行用了 Tensor Core / shared memory / warp shuffle"
  4. 遇到不懂的 intrinsic 查 CUDA Programming Guide

3.4 Day 7 · 写博客 + 复习（2 hr）

博客标题："5 yoy C++ 工程师 7 天学 CUDA 的笔记 — 哪些是炒概念，哪些是真新东西"

主结构（800-1000 字够了）：

  1. 开场：我的背景 + 为什么 1 周学 CUDA
  2. 5 个"C++ ↔ CUDA 概念映射"（用 Ch1 表）
  3. 我写的 3 个 kernel（贴 GitHub 链接）
  4. 读 vLLM kernel 的 takeaway
  5. 下周计划：读 vLLM scheduler

发布：个人博客 / Medium / 知乎 / 公众号都行。关键不是阅读量，是 "搜得到 + 简历能贴链接"。

4.3 Day 6 · trace 2 个调用路径（3 hr）

"trace 调用路径" = 用纸笔画"一个 HTTP 请求进来，经过哪些函数到 GPU"。

2 个必 trace 的路径：

  1. Prefill 路径：POST /v1/completions → AsyncLLMEngine.generate → Scheduler.add_request → step() → model_runner.prepare_input_tensors → PagedAttentionV1 kernel
  2. Decode 路径：scheduler.schedule（选下一批 token）→ block_manager.allocate（要新 block）→ model_runner forward → 取出新 token → return

项目难度找 issue 入口

5.2 Day 3-5 · 写 PR + 沟通（8 hr）

PR 工作流（标准 OSS）：

  1. Fork → clone → 在 issue 下评论 "I'd like to work on this"（避免重复）
  2. 建 feature branch，本地修改 + 跑 test（README 通常有 dev setup）
  3. 提 PR，title 简短 + body 详细（动机 / 改了什么 / 测了什么 / 是否 breaking change）
  4. Maintainer review 后改，最多 1-2 轮
  5. CI 全绿 → merge → ⭐

  4. code review 中学到的 (从他们 review 反馈学的)
  5. 对 OSS 协作流程的体感

5 个 "C++ 经验 sell 到 AI Infra" STAR 故事范式

把你已有的经验改写成 AI Infra 招聘官想听的故事

行为面 / 技术面追问深度时，你需要 5 个 60-90 秒能讲完的故事，把已有 C++ 经验 sell 到 AI Infra 语境。下面 5 个范式覆盖招聘官最常问的 5 类问题。

范式 1 · "讲一个你做的最难的并发问题"
S
"我们网关一个 lock-free MPMC queue 在压测下出现偶发性 use-after-free，1 万 QPS 跑 6 小时偶发 1 次。"
T
"必须修，但 reproducer 难找（heisenbug）。"
A
"我用 TSAN + 在 hot path 加 epoch counter 复现 → 发现是 ABA 问题 → 改用 hazard pointer → 同时优化 cache line padding 减少 false sharing。"
R
"修复后稳定运行 + 吞吐反而 +12%。这套经验后来我读 vLLM continuous batching scheduler 时直接 transfer — vLLM 的请求队列也用了类似 lock-free 设计。"
范式 2 · "讲一个你做的性能优化案例"
S
"某热路径函数 P99 latency 2.5 ms，目标砍到 1 ms。"
T
"代码已经看不出明显瓶颈。"
A
"用 perf record + flamegraph 发现 60% 时间花在 L3 cache miss → 重排数据布局把热字段集中（cache line aware）+ 用 SoA 替代 AoS。"
R
"P99 降到 0.8 ms（-68%）。同样的方法论我在学 Y01 时发现叫 'memory coalescing' — GPU 版的 cache miss 优化，思路完全一致。"
范式 3 · "讲一个你做的内存管理"
S
"高频交易系统每秒分配 100 万次小对象，malloc 成本占 CPU 30%。"
T
"必须降低 alloc 开销但不能改业务逻辑。"
A
"实现了 slab + thread-local pool — 固定 size class，按 chunk 批量分配，free 回 thread-local，定期 batch 还。"
R
"malloc 成本降到 5%。这就是 vLLM PagedAttention 在 GPU 上做的事 — KV-Cache 按 block 池管理，避免连续分配的碎片化。我读 Y02 时秒懂为什么这样设计。"
范式 4 · "讲一个你做的 profiling 案例"
S
"线上某 service 偶发 GC stall 50 ms（不是 GC 语言但有自管理对象），影响 SLO。"
T
"现象不规律，传统日志不够。"
A
"用 bpftrace + eBPF probe 跟踪 syscall + 加 USDT marker → 定位是某外部库的 mmap 缩页造成。"
R
"换 huge page 预分配解决。同样的"系统级 profiling"思维迁移到 GPU 就是 Nsight Systems — 工具换，方法论一样。"
范式 5 · "为什么你想转 AI Infra"（行为面必问）
S
"我做了 5 年系统编程，核心擅长是极致性能（lock-free / cache aware / profiling）。"
T
"过去 2 年大模型推理成本是所有用 AI 公司的核心 cost driver，但满足岗位要求的工程师极少 — 大多数 ML 出身工程师弱在系统优化。"
A

8 个 FAQ · 转型最常见的疑虑

"我不会 Python 怎么办" / "PhD 优先怎么办" 等高频问题
Q1 · 我不会 Python（或只是会一点），影响大吗？
影响小。AI Infra 70% 是 C++/CUDA，Python 主要是 binding 层。vLLM 80% 性能关键代码是 C++/CUDA，Python 是胶水。
补 Python 的最快路径：1 周学 type hint / async / decorator / context manager 这 4 个概念就够读 vLLM。不需要会写 web 后端 / 不需要会用 numpy / 不需要 PyTorch 模型。
Q2 · 我没碰过 ML，公司 JD 写"ML 背景优先"怎么办？
JD 写"优先"≠必须。AI Infra 团队常贴 ML JD 是因为 HR 模板，但实际招聘 hiring manager 80% 时间在找懂 GPU/CUDA/分布式的工程师。
你只需 Week 1 把 Y01 + X06 §3（KV-Cache 部分）扫一遍 — 面试时能解释 "Transformer 怎么算 / decoder 推理瓶颈在哪 / KV-Cache 怎么算大小"，就足够说服面试官。
Q3 · 转过去会降薪吗？
大概率不降，往往涨。原因：
① AI Infra senior 缺口大，谈薪空间反而比传统 backend C++ 大
② 你已有的 yoy 算数 — 不是从 zero start，而是 senior IC 转方向
③ 大厂 AI 团队预算独立 + 有 RSU 加成
典型范围（一线城市 / 5+ yoy）：传统 C++ 50-80w → AI Infra 70-120w。参考 X14 §4 详细薪资带宽。
Q4 · 4 周真的够吗？是不是过于乐观？
够过 1 关面试（HR + 一面）。后续要持续学，但拿到 offer 不需要"全懂"。
关键不是"4 周学完所有 AI Infra"（不可能），而是4 周建立可验证 evidence：1 个 OSS PR + 2 篇博客 + 1 张架构图 + 简历重写。
剩下的"全套 vLLM 源码理解 / TRT-LLM / SGLang 对比"等深度内容，入职后边干边学。
Q5 · 没有 H100 / 没有 GPU，怎么练？
① Google Colab 免费 T4 — 装 nvcc4jupyter，notebook 里跑 CUDA kernel
② Kaggle 免费 P100/T4 — 每周 30 hr 配额
③ Modal / RunPod / Lambda Cloud — H100 按小时 ~$2-4，做 vLLM 实测每月 ~$50 够
④ 云厂商新人优惠 — 阿里云 / 腾讯云 / 华为云通常有 GPU 试用
Week 1-2 完全可以零成本，Week 3-4 投 ~$50-100 跑真实 vLLM benchmark。
Q6 · 我已经 35+ / 40+，年龄会被歧视吗？
AI Infra senior 反而是年龄红利岗位（vs. 业务 backend）。原因：
① 需要的是多年系统编程肌肉记忆，不是新酷库会用
② Hiring manager 普遍认为"AI Infra 没 8-10 yoy 系统经验做不好"
③ AI 公司初创团队需要资深稳定的人 hold 住底层
反而2-3 yoy 转 AI Infra 才难 — 因为系统功底不够。
Q7 · 公司说"我们只招 ML PhD"怎么办？
① 那就别去那家 — 这种 JD 多半是 ML Research 岗，不是 AI Infra
② 同公司 AI Infra 岗通常不要 PhD，看团队找对人
③ 真正缺 AI Infra senior 的公司从不强制 PhD（vLLM / SGLang 核心贡献者很多不是 PhD）
④ 投 20-30 家时筛掉 5-8 家"必须 PhD"的，剩 15-25 家就是你的 sweet spot
Q8 · 4 周完了如果还没拿到 offer 怎么办？
正常 — 4 周拿到面试比拿到 offer 容易。后续策略：
① 继续投 + 二面冲刺：通常 1-3 个月才有 offer，AI Infra 周期比 backend 长（题目深 + 候选人慎重比对）
② 持续深化：每周读 1 篇 Y 系列 + 持续 OSS contribute
③ 积累 trophy 项目：再做 1 个有数字的小项目（如"我把 Llama 3-8B 推理在 4090 上 throughput 提了 30%"）
④ 不要回头：别因为 4 周没成就放弃 — 这是最有 ROI 的方向，6 个月内 90% 概率拿到
最后忠告：不要等"准备好了"才投。Week 2 末就开始投，把"正在做 4 周冲刺"写进简历当 ongoing 项目 — 面试官反而觉得你有 momentum + 自驱力。很多人卡在 "我还没准备好" 这一步上耗 1 年。
