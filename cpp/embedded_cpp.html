01

又爱又怕：零成本抽象的边界

哪些抽象真零成本，哪些偷偷收费

"嵌入式只能写 C" 是过时的偏见。现代 C++ 的很多特性零运行时开销：类型安全的枚举、模板生成的定制代码、constexpr 编译期计算、RAII 管资源、std::array/std::span 替代裸数组指针。麻烦的是，另一些特性偷偷收费——异常表、RTTI 元数据、隐式堆分配、虚函数间接跳转——在只有几十 KB Flash 的 MCU 上，这些"费用"可能直接压垮固件。用好 C++ 的关键是分清哪些免费、哪些收费，然后把收费的按需关掉或替换。

生活类比：自助餐 vs 单点

桌面/服务器 C++ 像自助餐：异常、RTTI、堆、STL 全摆着，随便拿，反正内存管够。MCU 像预算严格的单点：每道菜（每个特性）都标着价（Flash/RAM/周期），你得看着钱包点。好消息是很多"招牌菜"其实免费（模板、constexpr、RAII、强类型）；真正贵的是异常表、RTTI、隐式 new 这几样——把它们从菜单划掉，照样吃得又好又饱。

映射：钱包=Flash/RAM 预算 · 免费菜=模板/constexpr/RAII · 贵菜=异常/RTTI/堆 · 划掉菜单=编译开关关特性

1.1 特性成本速查

特性运行时成本Flash / RAM 成本MCU 上建议
模板 / 泛型零（编译期展开）可能代码膨胀（见第 6 章）放心用，控制实例化数量
constexpr / consteval零（编译期算完）常量进 Flash，省 RAM大力用，把计算前移
RAII / 析构近零极小大力用，替代手动清理
enum class / 强类型零零大力用
虚函数一次间接跳转每类 vtable + 每对象 vptr谨慎，热路径避免；可用 CRTP
异常正常路径近零，抛出很贵大：展开表 + .eh_frame + libunwind通常 -fno-exceptions
RTTI（dynamic_cast/typeid）运行时查表每多态类的 type_info通常 -fno-rtti
new/delete/std::string/vector堆分配不确定拖入 malloc 实现禁，改静态/池分配
"零成本抽象"的完整表述是 Stroustrup 那句：你不用的东西不付钱；你用的东西，手写也不会更省。嵌入式的挑战在于"不用的东西"里，异常/RTTI 的存在本身就要付表空间的钱——所以要显式关掉，才真正"不付钱"。
02

-fno-exceptions / -fno-rtti / freestanding

关掉之后能用什么、不能用什么

2.1 三个关键开关

arm-none-eabi-g++ \
  -fno-exceptions \      # 禁异常：throw 变 abort，去掉展开表
  -fno-rtti \            # 禁 RTTI：无 dynamic_cast / typeid 元数据
  -ffreestanding \       # 独立环境：不假设有完整 hosted 标准库/main 约定
  -fno-threadsafe-statics \ # 局部 static 初始化不加锁（单核/自己保证）
  -fno-use-cxa-atexit \  # 不注册全局析构（MCU 通常不会正常退出）
  -Os -ffunction-sections -fdata-sections \
  -Wl,--gc-sections ...
名词：freestanding（独立式）vs hosted（宿主式）实现

C++ 标准区分两种实现。hosted：有操作系统、完整标准库、程序从 main 开始正常退出（PC/服务器）。freestanding：可能没有 OS、只保证一小部分"不依赖运行时"的头（<cstddef>、<type_traits>、<array>、<atomic>、<bit> 等），启动/退出由你控制（裸机/MCU）。写 MCU 固件就是 freestanding 世界。

2.2 关掉后的能力边界

✅ 仍然完全可用

  • 类、继承、虚函数（RTTI 关了不影响虚调用）
  • 模板、constexpr、lambda、auto
  • RAII、移动语义、std::array/span/optional
  • <type_traits>、<atomic>、<algorithm>（非分配部分）

❌ 关掉后不可用 / 需替代

  • throw（触发 std::terminate/abort）、try/catch
  • dynamic_cast 向下转、typeid
  • 会抛异常的 STL：vector::at、std::string、std::stoi
  • 隐式堆分配容器（vector/map/string）
开了 -fno-exceptions 后，标准库里任何"本应 throw"的路径会调用 std::terminate（最终 abort）。所以 vector::at() 越界不再抛异常而是直接挂。要么彻底不用这些会抛的 API，要么接受"错误即 fault"的语义并配合看门狗复位。虚函数不受 -fno-rtti 影响——RTTI 只关 dynamic_cast/typeid，多态照常工作。

2.3 异常在 MCU 上到底贵在哪

异常需要展开表（.eh_frame / .gcc_except_table）和 unwinder 代码常驻 Flash；在几十 KB 的固件里这笔固定开销往往不可接受

异常的正常路径确实"零成本"（不抛就不花周期），但启用它就要携带整套展开元数据和 unwinder，这是空间成本而非时间成本。加上抛出时的延迟不可预测（不满足硬实时确定性），所以绝大多数 MCU 工程直接 -fno-exceptions。深入的异常机制见 异常机制。

03

不用异常，错误怎么传

expected / status code / Result 模式

关掉异常后，错误必须显式地随返回值传播。现代 C++ 有几种成熟范式，比裸 errno 安全得多。

3.1 std::expected（C++23）/ tl::expected

// 值或错误，二选一；调用者被类型系统强制处理
enum class SensorErr { Timeout, CrcFail, NotReady };

std::expected<uint16_t, SensorErr> read_adc(uint8_t ch) {
    if (!adc_ready()) return std::unexpected(SensorErr::NotReady);
    uint16_t raw = adc_convert(ch);
    if (!crc_ok(raw)) return std::unexpected(SensorErr::CrcFail);
    return raw;               // 成功：隐式构造 expected
}

// 调用处：无异常、无隐藏控制流、零堆分配
auto r = read_adc(3);
if (r) use(*r);
else switch (r.error()) { /* 分类处理 */ }
std::expected 是 C++23，编译器/标准库支持不齐时用 header-only 的 tl::expected 或自己写一个基于 union 的小型 Result。它零堆分配、可平凡拷贝、无异常，正好契合 MCU。相比 C 的 errno，它把"错误未处理"变成编译期可见（[[nodiscard]]）。

3.2 三种错误范式对比

范式优点缺点 / 注意
返回 bool + 输出参数最省、最 C易忽略返回值；出参丑
enum class Status 错误码确定、可穷举、可 [[nodiscard]]值和错误分离传递
expected<T,E>值/错误一体、类型安全、可链式需 C++23 或第三方；E 别太大
在功能安全（见 ISO 26262）语境里，确定性压倒一切：错误码/expected 的控制流可静态分析、可覆盖率统计，而异常的隐藏跳转和不确定延迟难以论证，这也是 MISRA/AUTOSAR C++ 指南普遍禁异常的原因。
04

无堆编程：静态分配与对象池

禁 new 之后如何组织内存

MCU 常禁用动态分配：堆会碎片化（长时间运行后大块申请失败）、分配时间不确定（违背实时性）、失败处理复杂。策略是把所有内存在编译期/启动期定好。

名词：堆碎片（fragmentation）

反复 malloc/free 不同大小后，空闲内存被切成许多不连续小块。即便总空闲量够，也可能没有一块连续的大内存满足下一次申请 → 分配失败。桌面靠虚拟内存+GC 掩盖，MCU 没有这层保护，碎片是长跑设备宕机的经典元凶。所以宁可静态分配"最坏情况"的固定内存。

4.1 用固定容量容器替代 vector/string

// 编译期定容量，全部落在 .bss / 栈，零堆
template <class T, size_t N>
class StaticVec {
    alignas(T) unsigned char buf_[N * sizeof(T)];  // 未构造的原始存储
    size_t n_ = 0;
public:
    bool push_back(const T& v) {
        if (n_ >= N) return false;              // 满 → 返回错误，不抛不扩容
        new (&buf_[n_ * sizeof(T)]) T(v);       // placement new：就地构造
        ++n_;
        return true;
    }
    T& operator[](size_t i) { return *reinterpret_cast<T*>(&buf_[i * sizeof(T)]); }
    size_t size() const { return n_; }
    // 析构里逐个 ~T()... 省略
};
工程上直接用成熟库：ETL（Embedded Template Library）提供 etl::vector<T,N>、etl::string<N>、etl::map 等固定容量、无堆、无异常（可配）的容器，接口贴近 STL。别自己重造轮子。标准库里 std::array、std::span、std::optional、std::bitset 也都零堆可用。

4.2 对象池：确定性的"动态"分配

需要"运行时申请/释放对象"（如消息、连接）又想避免堆时，用固定大小对象池：预分配 N 个槽 + 空闲链表，分配/释放都是 O(1) 且不碎片。

template <class T, size_t N>
class Pool {
    union Slot { Slot* next; alignas(T) unsigned char raw[sizeof(T)]; };
    Slot slots_[N];
    Slot* free_;
public:
    Pool() { for (size_t i = 0; i < N-1; ++i) slots_[i].next = &slots_[i+1];
             slots_[N-1].next = nullptr; free_ = &slots_[0]; }
    template <class... A> T* alloc(A&&... a) {
        if (!free_) return nullptr;                 // 池满：显式失败
        Slot* s = free_; free_ = s->next;
        return new (s->raw) T(std::forward<A>(a)...);
    }
    void free(T* p) { p->~T(); auto* s = reinterpret_cast<Slot*>(p);
                        s->next = free_; free_ = s; }
};
对象池 = 用固定内存换确定性。它和 低延迟指南 里的 arena / 对象池是同一思想：低延迟服务器为了"零分配抖动"这么做，MCU 为了"零碎片 + 实时"这么做，动机不同、手段一致。
05

编译期换运行时

constexpr 把工作从 MCU 挪到编译机

MCU 的 CPU 慢、周期贵；编译机的 CPU 快、随便算。C++ 最强的嵌入式武器就是把计算前移到编译期：结果作为常量进 Flash，运行时零开销。

5.1 编译期查找表

// 运行时算 sin 太贵；编译期生成 256 点定点查找表进 Flash
constexpr auto make_sin_table() {
    std::array<int16_t, 256> t{};
    for (int i = 0; i < 256; ++i)
        t[i] = static_cast<int16_t>(cexpr_sin(i * 2 * PI / 256) * 32767);
    return t;                          // 全在编译期算完
}
constexpr auto kSin = make_sin_table();   // 落在 .rodata（Flash），运行时零计算

int16_t fast_sin(uint8_t phase) { return kSin[phase]; }  // 一次访存
名词：consteval（C++20，立即函数）

constexpr 是"可以在编译期算"，若上下文不要求也可能运行时算；consteval 是"必须在编译期算"，否则编译报错。做查找表生成、寄存器位掩码计算这类"绝不该在 MCU 上跑"的逻辑，用 consteval 能从编译期就杜绝"不小心放到运行时"。

5.2 编译期校验寄存器配置

// 用类型 + consteval 在编译期校验分频/寄存器合法，运行时零检查
consteval uint32_t baud_to_brr(uint32_t pclk, uint32_t baud) {
    uint32_t div = (pclk + baud/2) / baud;
    if (div < 16 || div > 0xFFFF)
        throw "baud out of range";   // consteval 里 throw = 编译错误！
    return div;
}
constexpr auto BRR = baud_to_brr(72'000'000, 115200);  // 非法参数编译期就挂
解释「用 throw 做编译期断言」。在 consteval / 常量求值上下文里，走到 throw 会让常量表达式求值失败，从而编译报错——即使全局 -fno-exceptions 也成立，因为它根本不会生成运行时异常代码。这是把"配置错误"从"上电才炸"提前到"编译不过"的强力手段，比 static_assert 更灵活（能在复杂计算中间报错）。
嵌入式 C++ 的一条主线：凡是能在编译期确定的，就别留到运行时——查找表、位掩码、状态机跳转表、协议帧长校验。省的是 MCU 宝贵的周期和 RAM，换来的常量安安静静躺在 Flash 里。
06

代码体积治理

模板膨胀、vtable、.map 分析

Flash 常是最紧的资源。C++ 用不好会代码膨胀（code bloat），尤其模板。治理靠"度量 → 定位 → 削减"三步。

6.1 度量：size / nm / .map

# 各段大小：text=Flash 代码 rodata=常量 data/bss=RAM
arm-none-eabi-size -A firmware.elf

# 按符号大小排序，揪出最占地的函数
arm-none-eabi-nm --print-size --size-sort --radix=d firmware.elf | tail -30

# 链接期让链接器输出 .map，看谁把谁拖进来了
-Wl,-Map=firmware.map,--cref

6.2 模板膨胀与去重

❌ 每个类型一份完整实现

template<class T>
class Ring {
  // 全部逻辑都依赖 T
  void push(T); void pop();
  // Ring<int> / Ring<Msg> / Ring<Foo>
  // 各生成一整份代码
};

✅ 抽出与类型无关的基类

class RingBase {           // 只处理下标/容量
  size_t head_, tail_, cap_;
  bool advance();          // 非模板，编一次
};
template<class T>
class Ring : RingBase {   // 薄壳，仅存取 T
  T buf_[N];
};

把模板里不依赖类型参数的逻辑下沉到非模板基类（"thin template / fat base"惯用法），能显著减少每个实例化复制的代码。类似地：优先 std::span<T> 传数组而非各种长度模板；避免在头文件里对超多类型显式实例化。

6.3 削减开关与技巧

手段作用
-Os / -Oz为体积优化（-Oz 更激进，clang）
-ffunction-sections -fdata-sections + -Wl,--gc-sections每函数/数据独立段，链接期删除未引用的，常省 10%+
LTO（-flto）跨翻译单元内联/去死代码
-fno-exceptions -fno-rtti去掉展开表与 type_info（见第 2 章）
newlib-nano（--specs=nano.specs）精简 C 库，尤其 printf 家族
避免拖入 <iostream>它会拉进巨大的 locale/流机制
名词：vtable 与 vptr 的体积成本

每个含虚函数的类有一张 vtable（函数指针数组，进 Flash）；每个多态对象多一个 vptr（指向 vtable，占 RAM，通常 4 字节）。少量多态没问题，但"给每个小对象都上虚函数"会累积。热路径 / 大量小对象的场景，可用 CRTP（静态多态，见 设计模式）把间接跳转和 vptr 都消掉。对象布局细节见 对象模型。

07

MCU 上的并发原语

volatile、atomic 与关中断临界区

MCU 的并发主要来自中断打断主循环（以及多核 SoC 的核间）。这里最容易犯的错，是把 volatile 当同步工具。

7.1 volatile 的真正含义（以及不含义）

✅ volatile 保证的

  • 每次访问都真的读/写内存，不被优化掉
  • 用于 MMIO 寄存器、被 ISR 修改的标志
  • 阻止编译器缓存到寄存器 / 合并访问

❌ volatile 不保证的

  • 不保证原子性（32 位读写在某些核非原子）
  • 不建立内存序 / happens-before
  • 不能替代锁或 std::atomic
经典错误：用 volatile int counter 在主循环和 ISR 间共享并 ++。counter++ 是"读-改-写"三步，主循环读到后被 ISR 打断、ISR 也改，回来一写就丢更新。volatile 只保证"每步都访存"，不保证"三步不可分割"。要原子性必须用 std::atomic 或关中断临界区。

7.2 单核 MCU 上的 std::atomic 与关中断

#include <atomic>
// 单个标志：atomic 足够，且在单核上通常无锁、极廉价
std::atomic<bool> g_data_ready{false};

// 复合操作（改多个变量）→ 关中断临界区最简单可靠
struct IrqGuard {          // RAII 关中断：出作用域自动恢复
    uint32_t primask;
    IrqGuard()  { primask = __get_PRIMASK(); __disable_irq(); }
    ~IrqGuard() { if (!primask) __enable_irq(); }
};

void update_shared() {
    IrqGuard g;               // 临界区：ISR 不会插进来
    ring.head = next; ring.count++;   // 复合状态一致更新
}                             // g 析构 → 恢复中断
法则：单个标量共享用 std::atomic；一组变量要一致更新，用关中断的 RAII 临界区（越短越好）。关中断期间系统对所有中断失聪，临界区拖长会抬高中断延迟——这与 Linux 驱动 里 spin_lock_irqsave 的取舍完全同构。内存序的底层原理见 内存模型。
08

与 C 驱动 / HAL 互操作

extern "C"、ISR、启动代码里的 C++

厂商 HAL（STM32 HAL、Zephyr、ESP-IDF）大多是 C。C++ 固件必须干净地和它们对接。

8.1 extern "C"：跨越名字修饰

C++ 会对函数名做 name mangling（编码参数类型以支持重载）。C 不做。中断向量表、HAL 回调、链接脚本引用的符号都按 C 名字查找，所以必须用 extern "C" 关掉修饰：

// ISR 会被启动文件的向量表按 C 符号名引用
extern "C" void USART1_IRQHandler(void) {   // 名字必须精确匹配向量表
    if (LL_USART_IsActiveFlag_RXNE(USART1)) {
        uint8_t b = LL_USART_ReceiveData8(USART1);
        on_byte(b);          // 可以调用 C++ 代码
    }
}

// 给 C 代码用的接口：头文件里包一层
#ifdef __cplusplus
extern "C" {
#endif
void app_init(void);
void app_tick(void);
#ifdef __cplusplus
}
#endif
ISR 里可以调用 C++，但受"中断上下文"约束：别在 ISR 里做重活、别隐式分配、别调用可能长时间运行的东西。用 std::atomic 置标志、把处理丢回主循环（前后台架构）或 RTOS 任务（见 RTOS 全景）。这和 Linux 驱动"上半部尽快返回"是同一条纪律。

8.2 全局对象的构造顺序陷阱

名词：静态初始化顺序问题 + .init_array

带非平凡构造的全局/静态 C++ 对象，其构造函数由启动代码遍历 .init_array 段调用。若你的 startup 汇编忘了调用 __libc_init_array，这些构造函数根本不跑，对象处于未初始化状态——极隐蔽的 bug。此外跨翻译单元的全局对象构造顺序未定义，一个全局对象在构造时依赖另一个可能尚未构造。MCU 上的稳妥做法：少用带副作用的全局对象，改用显式 init()、或 Meyers 单例（首次使用时构造，配 -fno-threadsafe-statics）。

坏例 / 好例对照

同一需求的 MCU 友好写法

9.1 收一帧串口数据并解析

❌ 桌面直觉（堆 + 异常 + string）

std::string read_line() {
  std::string s;          // 堆，碎片
  char c;
  while ((c = uart_get()) != '\n')
    s += c;                // 反复重分配
  return s;                // 拷贝/移动
}
int v = std::stoi(read_line()); // 会抛异常

✅ MCU 友好（固定缓冲 + expected）

std::expected<int,Err> read_int(
      std::span<char> buf) {   // 调用方给缓冲
  size_t n = 0; char c;
  while (n < buf.size() &&
        (c=uart_get())!='\n') buf[n++]=c;
  int out{};
  auto r = std::from_chars(   // 不抛、不分配
      buf.data(), buf.data()+n, out);
  if (r.ec != std::errc{})
    return std::unexpected(Err::Parse);
  return out;
}

9.2 一组外设的多态管理

❌ 虚函数 + 堆容器

std::vector<
   std::unique_ptr<Sensor>> v;  // 堆 + vtable
v.push_back(
   std::make_unique<Adc>()); // 隐式 new
for (auto& s : v) s->sample(); // 间接跳转

✅ 静态存储 + CRTP / std::variant

// 编译期已知集合 → 无堆、可内联
std::tuple<Adc, Imu, Baro> sensors;
std::apply([](auto&... s){
    (s.sample(), ...);       // 折叠展开，直接调用
}, sensors);
// 需要运行时集合但类型有限 → variant
规律：把"运行时的动态性"能压到编译期就压——已知集合用 tuple/数组、有限类型用 variant（无堆多态）、必须真多态时才上虚函数或 CRTP。堆和异常是最后手段，不是默认选项。
10

高频面试题

嵌入式 C++ 岗常考
嵌入式为什么常关异常？关了之后错误怎么传？

异常的空间成本是关键：启用它就要携带展开表（.eh_frame/.gcc_except_table）和 unwinder 代码常驻 Flash，几十 KB 固件里往往不可接受；且抛出延迟不确定，不满足硬实时确定性。关掉（-fno-exceptions）后用显式返回值传播错误：std::expected<T,E>（值/错误一体、类型安全、零堆）、enum class 错误码 + [[nodiscard]]，或 bool+出参。功能安全语境还要求控制流可静态分析，这也是禁异常的理由。

volatile 能保证原子性和线程/中断安全吗？

不能。volatile 只保证"每次访问真的读写内存、不被编译器优化掉/缓存到寄存器"，用于 MMIO 寄存器和被 ISR 改的标志。它不保证原子性（counter++ 是读-改-写三步，会丢更新），也不建立内存序。要原子性/同步用 std::atomic，或关中断的 RAII 临界区来保护复合操作。

-fno-rtti 会影响虚函数吗？RTTI 关掉后什么不能用？

不影响虚函数——虚调用只需 vptr→vtable 的间接跳转，与 RTTI 无关。-fno-rtti 去掉的是每个多态类的 type_info 元数据，因此 dynamic_cast（向下/交叉转换）和 typeid 不能用。需要"运行时区分类型"时改用带 tag 的基类、std::variant + visit，或 CRTP 静态多态。

MCU 上为什么禁堆？不用 new 怎么管理内存？

堆有三宗罪：碎片化（长跑后大块申请失败）、分配时间不确定（违背实时）、失败处理复杂。替代方案：编译期固定容量容器（std::array、ETL 的 etl::vector<T,N>）、栈/.bss 静态分配、对象池（预分配 N 槽 + 空闲链表，O(1) 且不碎片）、placement new 就地构造。核心是把内存在编译期/启动期按最坏情况定好。

怎么减小 C++ 固件的代码体积？模板膨胀怎么治？

先度量（size -A、nm --size-sort、链接 .map）定位大符号。开关：-Os/-Oz、-ffunction-sections -fdata-sections+--gc-sections、LTO、-fno-exceptions -fno-rtti、newlib-nano、别拖 <iostream>。模板膨胀用"thin template / fat base"——把不依赖类型参数的逻辑下沉到非模板基类只编一次；用 std::span 替代按长度模板；控制显式实例化数量。多态开销大时用 CRTP 去掉 vtable/vptr。

extern "C" 干什么用？ISR 为什么要它？

C++ 有 name mangling（把参数类型编进符号名以支持重载），C 没有。中断向量表、HAL 回调、链接脚本引用的符号都按 C 名字查找，若 ISR 用 C++ 修饰名，向量表就链接不到正确符号。extern "C" 关掉修饰，让符号名和 C 一致。头文件里用 #ifdef __cplusplus extern "C" { 包住供 C/C++ 共用。

11

checklist 与小结

把现代 C++ 安全落到 MCU 的自检清单

  • 编译开关：-fno-exceptions -fno-rtti -ffreestanding -Os -ffunction-sections -fdata-sections -Wl,--gc-sections。
  • 错误传播用 expected<T,E> / enum class 错误码 + [[nodiscard]]，不靠异常。
  • 禁堆：固定容量容器（std::array / ETL）、对象池、placement new；禁 vector/string/make_unique。
  • 能编译期算的绝不留运行时：constexpr/consteval 查找表、位掩码、编译期配置校验。
  • 共享数据：单标量 std::atomic；复合操作用关中断 RAII 临界区（极短）；volatile 只给 MMIO/标志，不当同步用。
  • 多态按需：热路径/大量小对象用 CRTP 静态多态；真多态才上虚函数；无堆多态用 std::variant。
  • 与 C/HAL 对接：ISR 和向量表符号用 extern "C"；ISR 里只置标志、活丢回主循环/任务。
  • 启动代码确认调用 __libc_init_array（否则全局对象不构造）；少用带副作用的全局对象。
  • 持续度量体积：size / nm / .map 纳入 CI，防止 Flash 悄悄超标。
一句话总结：现代 C++ 完全能上 MCU——把异常/RTTI/堆这几样"收费项"关掉或替换，把计算尽量前移到编译期，用类型系统和 RAII 换来安全，你就得到"和手写 C 一样省，却更安全、更表达"的固件。
