为什么 lock-free

Lock-free 不是"更快"——是消除特定 progress problem

Lock-free 不是"更快"——是消除特定 progress problem。多数应用其实用 std::mutex 足够，lock-free 是特定场景的工具。

Lock 的真正问题

  1. Priority inversion — 低优先级 hold lock 阻塞高优先级（实时系统致命）
  2. Convoy effect — 多线程持续 contend lock，排队等导致 throughput 崩
  3. Deadlock — 多 lock 顺序错
  4. 持锁线程死掉 — 永远 hold lock
  5. Cache line bouncing — lock 是 hot cacheline，多核来回
  6. Latency outliers — P50 fast 但 P99/P999 偶尔几 ms

什么时候真用 lock-free

  • 硬实时系统（车端 / HFT / 音频） — Latency 上界严格
  • 读极多写少 + cacheline 竞争激烈
  • 跟外部 signal / interrupt handler 通信
  • 极高 throughput SPSC / MPSC queue
  • kernel / RCU 类核心数据结构

什么时候不该 lock-free

  • 临界区不极短（> 100 ns） — Lock overhead 占比小
  • 性能不是瓶颈 — Mutex 简单可靠
  • 团队 expertise 不够 — Lock-free bug 极难调
Linus Torvalds 名言："Locking is hard. Lock-free is way harder." Lock-free 的 bug 通常看不出来（编译过 + 大多 input pass + 偶发数据 corrupt），一旦发生几乎不可能 reproduce。

Memory Order · 实战 pattern

大 pattern 速查 · C++11 memory order 是 lock-free 编程的语言。6 种：

C++11 memory order 是 lock-free 编程的语言。6 种：

Order含义典型用
relaxed仅原子，无同步计数器 / 累计

5 大 pattern 速查

Pattern 1: Counter（relaxed）

std::atomic<int> counter{0};
counter.fetch_add(1, std::memory_order_relaxed);  // 计数无同步需求

Pattern 2: Publication（release / acquire）

// Producer:
data = compute();                // 先写 data
ready.store(true, std::memory_order_release);

// Consumer:
if (ready.load(std::memory_order_acquire)) {
  use(data);                     // 保证看到 producer 的 data 写
}

Pattern 3: Reference Counting (relaxed inc, acq_rel dec)

void retain() {
  count_.fetch_add(1, std::memory_order_relaxed);     // 计数
}
void release() {
  if (count_.fetch_sub(1, std::memory_order_acq_rel) == 1) {
    // acquire 看到别的线程之前对 object 的写
    // release 保证我本线程之前的写对 deleter 可见
    delete this;
  }
}

Pattern 4: SpinLock (acquire / release)

class SpinLock {
  std::atomic_flag flag_ = ATOMIC_FLAG_INIT;
public:
  void lock() {
    while (flag_.test_and_set(std::memory_order_acquire)) { /* spin */ }
  }
  void unlock() { flag_.clear(std::memory_order_release); }
};

Pattern 5: Seqlock (release + version check)

读者无锁、写者一把锁、读者通过 version 检测中途被改：

class SeqLock<T> {
  std::atomic<uint64_t> seq_{0};
  T data_;
  std::mutex write_mtx_;
public:
  void write(const T& v) {
    std::lock_guard lg(write_mtx_);
    seq_.fetch_add(1, std::memory_order_acquire);   // seq → 奇 (locked)
    data_ = v;
    seq_.fetch_add(1, std::memory_order_release);   // seq → 偶 (unlocked)
  }
  T read() const {
    while (true) {
      uint64_t s1 = seq_.load(std::memory_order_acquire);
      if (s1 & 1) continue;                          // writer 进行中, retry
      T snapshot = data_;
      uint64_t s2 = seq_.load(std::memory_order_acquire);
      if (s1 == s2) return snapshot;                  // 无变化 → 有效
    }
  }
};

适合读极多 + 数据小 + 读端不能 block场景（如内核 timekeeping）。

建议：写新代码默认 seq_cst。Benchmark 后逐个降级到合适等级。x86 上 seq_cst 跟 acq_rel 几乎同性能（store mfence），ARM 上 seq_cst 较贵（stl 指令）。

CAS 标准 idiom

T expected = atomic_var.load();

典型应用：lock-free stack

template<class T>
class LockFreeStack {
  struct Node { T data; Node* next; };
  std::atomic<Node*> head_{nullptr};
public:
  void push(T value) {
    Node* node = new Node{std::move(value), nullptr};
    node->next = head_.load(std::memory_order_relaxed);
    while (!head_.compare_exchange_weak(
        node->next, node,
        std::memory_order_release,
        std::memory_order_relaxed)) {
      // expected (node->next) 自动更新为当前 head
    }
  }
  bool pop(T& out) {
    Node* old_head = head_.load(std::memory_order_acquire);
    while (old_head) {
      if (head_.compare_exchange_weak(
          old_head, old_head->next,
          std::memory_order_acquire,
          std::memory_order_relaxed)) {
        out = std::move(old_head->data);
        delete old_head;    // 危险!! 见 §4 ABA 问题
        return true;
      }
    }
    return false;
  }
};

这个 stack 有 ABA 问题 + use-after-free 风险——见下节。

ABA 问题 + 4 种解法

什么是 ABA · 对比

什么是 ABA

💡 一句话先懂：CAS 只会问「这个值还等于我上次看到的吗？」，它不会问「这中间有没有被人动过？」。如果一个值从 A 变成 B 又变回 A，CAS 看到的还是 A，于是误以为啥都没发生——这就是 ABA。
🅿️
生活化类比：停车位上的"同款红车"
你把车停好，记住隔壁车位停着朋友的红色 Model 3。你去买杯咖啡，回来一看——隔壁还是一辆红色 Model 3，于是你判断"朋友还没走，一切照旧"。
但其实这几分钟里：朋友开走了 → 车位空了 → 来了个陌生人停了辆一模一样的红色 Model 3。车位"看起来"没变（红 Model 3 → 空 → 红 Model 3），可它的真实含义已经完全不同。你基于"还是朋友的车"做的任何决定（比如把东西放他车顶）都会出错。
CAS 就是那个只看"还是不是红 Model 3"的你。在无锁栈里，"车"就是节点指针，而内存分配器复用了同一个地址，让被释放的节点地址又变回原值——CAS 被骗，导致 use-after-free。

更精确地说：线程 T1 读到 head = A，准备用 CAS 把 head 改成 A->next，但还没执行就被调度暂停；同时 T2 把 A 弹出并释放、又压入新节点、最后新节点复用了 A 原来的地址。等 T1 醒来执行 CAS，发现 head 的值确实还等于它记的 A 地址 → CAS 成功，但此刻这个地址背后的对象早已不是当初那个 → 数据错乱 / use-after-free。

4 种解法 — 本质都在回答"这中间被动过吗？"

四种方案思路不同，但都在补上 CAS 缺的那一问。一句话直觉：

  • Tagged Pointer：给指针配一个只增不减的版本号，每次改都 +1。地址能复用，但"地址+版本号"这对组合不会重复 → 一眼看出被动过。
  • Hazard / EBR / RCU：干脆不让地址被复用——只要还有人可能引用旧节点，就推迟回收。地址不回收，就不可能变回来，ABA 从根上消失。

4.1 Tagged Pointer（双字 CAS）

struct TaggedPtr {
  Node* ptr;
  uint64_t tag;
};
std::atomic<TaggedPtr> head_;
  • x86: cmpxchg16b 支持 16 字节 atomic
  • ARM64: casp（ARMv8.1-A）或 LL/SC pair
  • 缺点：std::atomic<T> 对 16-byte 多数平台不 lock-free，需 libatomic

4.2 Hazard Pointer（见 §7）

不让 reclaim 还在被引用的 node → 地址不会 reuse → ABA 不可能。

4.3 Epoch-Based Reclamation (EBR)

所有 reader 进 critical section 标记 epoch；reclaim 等所有 reader 离开当前 epoch 才释放老节点。比 hazard pointer 简单，memory overhead 较高。

4.4 RCU（见 §8）

Read-side 零开销，writer 等 grace period。Linux kernel 经典方案。

对比

方案读端开销写端开销memory overhead实现复杂度

Hazard Pointer+1 atomic store1 CAS + scan hazard list中（per-thread slot）高

SPSC Queue · lock-free 入门

核心代码 · 关键点 · 性能优化

Single Producer Single Consumer 是 lock-free 最简单也最实用的场景。

核心代码

template<class T, size_t N>
class SpscRing {
  static_assert((N & (N-1)) == 0, "N must be power of 2");
  alignas(64) std::atomic<size_t> head_{0};   // consumer 只改
  char pad1_[64];                              // 防 false sharing
  alignas(64) std::atomic<size_t> tail_{0};   // producer 只改
  T data_[N];
public:
  bool push(T value) {
    const size_t t = tail_.load(std::memory_order_relaxed);
    const size_t next = (t + 1) & (N - 1);
    if (next == head_.load(std::memory_order_acquire)) return false;   // full
    data_[t] = std::move(value);
    tail_.store(next, std::memory_order_release);
    return true;
  }
  bool pop(T& out) {
    const size_t h = head_.load(std::memory_order_relaxed);
    if (h == tail_.load(std::memory_order_acquire)) return false; // empty
    out = std::move(data_[h]);
    head_.store((h + 1) & (N - 1), std::memory_order_release);
    return true;
  }
};

关键点

  • N = 2^k——用 &(N-1) 取模，比 % 快
  • head / tail 各 cache line——防 false sharing
  • release/acquire pair——consumer 看到 producer write 必看到 data
  • 本线程改自己端用 relaxed load（不需同步给别人）

性能优化

  1. Cached opposite end：producer 本地 cache last-seen head（不每次都 cross-cacheline load）
  2. Batch push / pop：一次 16-64 个，amortize cache miss
  3. Spinning vs blocking：empty 时 busy-spin（低延迟）/ yield()（节能）/ std::atomic::wait（futex）

性能数字（典型 64-bit Linux x86_64）：

  • 基础 SPSC: ~10 ns per op
  • + cached head/tail: ~7 ns per op
  • + batch 16: ~2 ns per op amortized

MPSC / MPMC Queue

MPMC 关键：Sequence Number · 开源库选择

多生产者 / 多消费者比 SPSC 难 10×。生产用现成库（folly / moodycamel），别自己写除非真懂。

💡 为什么难 10×？ SPSC 里只有一个生产者动 tail、一个消费者动 head，两边各管各的，永不打架。MPMC 里多个生产者都想抢同一个 tail、多个消费者都想抢同一个 head → 必须用 CAS 协调，还要防 ABA、防"半写状态被别人读到"。
🎫
生活化类比：银行的取号机 + 柜台叫号屏
银行大厅里只有一台取号机，每个人来都按一下，拿到一个永远递增的号码（01、02、03…）——这就是 enqueue_pos / dequeue_pos，用 fetch_add 实现，谁都不会拿到重复号，天然不打架。
拿到号后，你走到"号 % 柜台数"对应的那个柜台，但要等柜台屏幕叫到你的号才能办。每个柜台头顶有块屏（这就是每个 slot 的 sequence），它像红绿灯一样告诉你：现在轮到几号来"存"、还是几号来"取"。
于是抢号（全局递增计数）和真正干活（看柜台屏状态）解耦了——这正是 Vyukov 队列高效又正确的秘诀。

MPMC 关键：Sequence Number

Dmitry Vyukov 的 bounded MPMC queue 是经典实现。抢位置用一个全局递增计数（fetch_add，无 CAS、无 ABA）；每个 slot 自带一个 sequence 当"红绿灯"，靠它判断这格该写还是该读：

slot.seq 与我的号 pos 的关系含义我该做什么
seq == pos这格是空的，正等 pos 号生产者来写✅ 生产者写入，然后 seq = pos + 1
seq == pos + 1这格有数据，正等 pos 号消费者来读✅ 消费者读走，然后 seq = pos + N（让下一轮生产者就绪）
seq < pos还没轮到我（队列满 / 对端慢）⏳ 自旋等待 / 返回"满或空"

seq = pos + N 这步很妙：它正好把这格的"下一次可写号"推进了一整圈（N 个），让 ring buffer 循环复用时红绿灯依然对得上。

struct Slot {
  std::atomic<size_t> sequence;
  T data;
};

// Push (producer):
pos = enqueue_pos_.fetch_add(1, std::memory_order_relaxed);
slot = &buffer_[pos & (N-1)];
while (slot->sequence.load(std::memory_order_acquire) != pos) { /* wait */ }
slot->data = value;
slot->sequence.store(pos + 1, std::memory_order_release);   // signal consumer

// Pop (consumer):
pos = dequeue_pos_.fetch_add(1, std::memory_order_relaxed);
slot = &buffer_[pos & (N-1)];
while (slot->sequence.load(std::memory_order_acquire) != pos + 1) { /* wait */ }
out = std::move(slot->data);
slot->sequence.store(pos + N, std::memory_order_release);  // signal future producer

Hazard Pointer

核心思想 · C++26 std::hazard_pointer

Maged Michael 2002 提出（PODC 2002），是 lock-free 数据结构 + safe memory reclamation 的标准方案。

💡 它解决什么：无锁结构里，线程 A 正在读一个节点，线程 B 同时想 delete 它 → A 读到野指针（use-after-free）。Hazard Pointer 的办法是：读之前先公开声明"我正在用这个节点"，想删的人必须先检查有没有人在用，有人用就先别删、记下来过会儿再说。
🔖
生活化类比：健身房储物柜的"使用中"挂牌
健身房清洁工（writer）每晚要清空没人用的储物柜。但他不能看柜子关着就直接撬开扔东西——万一里面有人的私物呢？
规矩是：使用者（reader）用柜子前，先在柜门挂一个写着自己名字的"使用中"牌子（= 把节点指针写进自己的 hazard slot）。清洁工想清理某个柜子时，先扫一遍所有"使用中"挂牌：
· 牌子上有这个柜子 → 先别动，记到"待清理清单"（retired list），下次再来看；
· 没人挂这个柜子 → 安全清空。
使用者用完把牌子摘掉（reset），下一轮清洁工扫描时就能放心回收了。

核心思想

  1. 每个 thread 维护少量"hazard slot"（typically 1-2 个），存它正在引用的 node 指针
  2. 读 node 前：hazard_slot = node_ptr; memory_barrier();
  3. Writer 要 free 一个 node 时：不立即 free，先 push 到 retired list
  4. 定期 scan 所有线程的 hazard slot，retired node 不在任何 slot → 真 free

C++26 std::hazard_pointer

2024 P1121 提案已 accept 到 C++26 working draft：

// C++26
std::hazard_pointer hp;
Node* node = head_.load();
hp.protect(node);                // 注册 hazard
if (head_.load() != node) {       // re-check 防 race
  continue;
}
use(node);
hp.reset();                      // 退出 critical section

// Writer 端 retire:
std::hazard_pointer_obj_base<Node>::retire(old_node);

Pre-C++26 用 folly::hazptr 或自己实现。

极端读端性能 · Userspace RCU

Read-Copy-Update——Linux kernel 1990s 引入，用于 hot path 读优化。

方案读端开销

RCU 适合读 ≫ 写（如 1000:1）+ 数据可拷贝 / 替换场景。

Userspace RCU

liburcu（Mathieu Desnoyers，Linux kernel RCU 的 userspace port）：

#include <urcu.h>

// Reader:
rcu_read_lock();
Node* node = rcu_dereference(head_);
use(node);
rcu_read_unlock();

// Writer:
Node* old = rcu_dereference(head_);
Node* new_node = new Node(*old);   // copy
new_node->data = updated;
rcu_assign_pointer(head_, new_node);
synchronize_rcu();              // 等 grace period
delete old;                     // safe

层次保证例
Wait-free每个线程有限步骤内完成 opfetch_add / atomic 简单 op
Lock-free整系统有进展（至少一个线程进展）CAS-based stack / queue
Obstruction-free线程无冲突时能独立完成少数 STM
Blocking (有锁)无保证（持锁线程死掉可能全卡）mutex / spin lock

大多数 lock-free 算法不是 wait-free——高并发下个别线程可能饿死。真正 wait-free 算法极少且复杂（如 Kogan-Petrank queue）。实战 lock-free 够用 99% 场景；wait-free 仅严格 RT 系统需要。

跑测试为啥不够

并发 bug 是概率性——单元测试跑 100 万次都不出现，production runtime 跑半年才发生 1 次。必须 systematic 工具。

关键 best practices

  • 从用现成库开始——folly / moodycamel / liburcu
  • 不写新 lock-free 算法——用已 published + 同行 review 过的
  • 每条 atomic 操作显式标 memory order——不要默认 seq_cst 的"魔法"
  • cache line 对齐——hot atomic 用 alignas(64)
  • 注释 happens-before invariant——6 月后没人记得 why
  • 跑 TSan 跟 ARM（weakly ordered，x86 上 pass 的可能 ARM 上 fail）
🎯 关键 take-aways
  • Lock-free 不是"更快"是消除特定 progress problem（priority inversion / convoy / deadlock / latency outlier）
  • Memory order 5 大 pattern：counter (relaxed) / publication (release-acquire) / ref count (acq_rel) / lock / seqlock
  • CAS 标准 idiom：do { compute new from expected } while (!CAS(expected, new))
  • ABA 问题 4 解：tagged pointer / hazard pointer / EBR / RCU
  • SPSC ring buffer 是 lock-free 入门 + 实用场景——~10 ns/op，cache-line align + power-of-2 size
  • MPMC 难 10×——用 moodycamel / folly / boost.lockfree
  • Hazard Pointer 适合 lock-free 数据结构 + safe reclamation；C++26 std::hazard_pointer
  • RCU 适合读 ≫ 写（读端零开销），但 grace period + memory overhead 高
  • Wait-free > Lock-free > Obstruction-free > Blocking 严格保证递减
  • 验证：cppmem (设计) + TSan (动态) + Relacy (interleaving) + rr (replay)
