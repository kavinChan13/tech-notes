五种值类别全景（C++11 起）

文氏图：3 个底层 + 2 个聚合 · 三个底层类别的精确定义 · 两个聚合类别（更高层视角） · 实战识别

C++03 只有 lvalue 和 rvalue。C++11 引入移动语义后，把 rvalue 拆开，最终形成 5 个术语 + 2 个上层概念 的体系。

2.1 文氏图：3 个底层 + 2 个聚合

2.2 三个底层类别的精确定义

类别口语含义判定标准典型例子
lvalue
(left-value)		 "有身份的值，且不能被偷" 有名字、可取地址，且不是 xvalue x、arr[0]、*ptr、字符串字面量 "abc"、返回 T& 的函数
prvalue
(pure rvalue)		 "纯临时，没有持久身份" 表达式的结果本身，不和任何对象关联 42、1+2、foo()（按值返回）、this、lambda 表达式
xvalue
(eXpiring value)		 "将亡值，可被偷的有身份对象" 有身份（有地址），但即将销毁，资源可被窃取 std::move(x)、返回 T&& 的函数、static_cast<T&&>(x)

2.3 两个聚合类别（更高层视角）

聚合类别定义用一句话记
glvaluelvalue ∪ xvalue"有身份（地址）的值"
rvaluexvalue ∪ prvalue"可被移动的值"
判断流程（背下来）：拿到一个表达式 →
① 它有名字或能 &e 取地址吗? 能 → glvalue；不能 → prvalue
② 如果是 glvalue：它是不是 std::move(...) 或返回 T&& 的结果? 是 → xvalue；否 → lvalue
🔍 交互式判定器：点一个表达式，走一遍决策树
选一个表达式，下面两道判定题会自动作答并给出值类别。生活化类比：lvalue 是"有户口的房子"（有地址、长住）；prvalue 是"路边临时摊"（没身份、用完即走）；xvalue 是"挂牌待拆的房子"（有地址但资源随时可搬走）。
① 有名字 / 能取地址 &e？ —
② 是 std::move 或返回 T&&（将亡）？ —
← 先选一个表达式

2.4 实战识别

int  x = 42;
int& getRef();           // returns lvalue ref
int&& getRRef();        // returns rvalue ref (xvalue)
int  getVal();           // returns prvalue

x;                       // lvalue  (有名)
42;                      // prvalue (字面量)
x + 1;                   // prvalue
&x;                      // prvalue (取地址表达式本身)
*ptr;                    // lvalue  (解引用 = 给临时一个名字)
arr[0];                  // lvalue
getRef();                // lvalue  (返回 T&)
getRRef();               // xvalue  (返回 T&&)
getVal();                // prvalue (按值返回)
std::move(x);            // xvalue  (move 是 cast 到 T&&)
static_cast<int&&>(x);   // xvalue  (等价于 std::move)
"abc";                   // lvalue! 字符串字面量是 const char[N]，有地址
true;                    // prvalue
this;                    // prvalue (this 是临时指针值)
[]{};                    // prvalue (lambda 表达式)
易错点：字符串字面量 "hello" 是 lvalue（类型 const char[6]），不是 prvalue！这是与其他字面量（int / true / nullptr）唯一的不同点 — 因为它们有静态存储，能取地址。
回答"什么是 xvalue?"时不要只说"std::move 的结果" — 标准定义是有身份且即将销毁、资源可被窃取的值。"将亡值" 是中文意译，要能讲出英文 eXpiring 的含义。
03

引用类型与「引用折叠」规则

三种引用 · 万能引用 (Universal / Forwarding Reference) · 引用折叠规则（Reference Collapsing）

3.1 三种引用

引用能绑哪些用途
T&仅 lvalue（非 const）修改原对象
const T&所有（lvalue / xvalue / prvalue）只读不复制（万能只读参数）
T&& (具体类型)仅 rvalue（xvalue / prvalue）移动构造、移动赋值
T&& (模板参数)万能引用，看实参完美转发
int x = 42;
int&       a = x;            // ✅ lvalue → T&
int&       b = 42;           // ❌ rvalue 不能绑非 const T&
const int& c = 42;           // ✅ const T& 万能只读
int&&      d = 42;           // ✅ rvalue → T&&
int&&      e = x;            // ❌ lvalue 不能绑 T&&
int&&      f = std::move(x); // ✅ xvalue → T&&
一旦 T&& d = 42; 成立后，d 本身是 lvalue（因为它有名字、可取地址）！
"右值引用变量是 lvalue" — 这是绕过最多人的一个点。

3.2 万能引用 (Universal / Forwarding Reference)

仅当 T&& 出现在模板参数推导上下文，T 是要推导的模板参数，且不带 const/volatile 修饰时，它才是万能引用，能绑任何值类别。

template<class T>
void universal(T&& arg);    // ✅ 万能引用

template<class T>
void notUniversal(const T&& arg);  // ❌ 有 const，是普通右值引用

template<class T>
void notUniversal2(std::vector<T>&& arg);  // ❌ T 不是直接参数

auto&& x = expr;             // ✅ auto&& 也是万能引用

3.3 引用折叠规则（Reference Collapsing）

当模板推导出 T = int&，那 T&& 等于什么？引用的引用语法上不允许，所以 C++ 定义了折叠规则：

原始折叠后
T& &T&
T& &&T&
T&& &T&
T&& &&T&&

口诀："一个 lvalue ref 进来，结果就是 lvalue ref；两个 rvalue ref 才是 rvalue ref"。

template<class T>
void f(T&& arg);

int x = 1;
f(x);     // 实参是 lvalue → T 推导为 int& → T&& = int& && = int&
f(42);    // 实参是 prvalue → T 推导为 int → T&& = int&&
f(std::move(x));  // xvalue → T 推导为 int → int&&

这就是为什么万能引用能同时绑左值和右值 — 通过推导规则 + 折叠规则联合实现。

std::move / std::forward 真相

std::move 不移动任何东西 · std::forward 完美转发 · 易错对比

4.1 std::move 不移动任何东西

它只是个 cast：把表达式无条件转换为右值引用（xvalue）。是否真正发生移动，由接收方决定。

// 标准库实现（简化）
template<class T>
constexpr std::remove_reference_t<T>&& move(T&& arg) noexcept {
    return static_cast<std::remove_reference_t<T>&&>(arg);
}

作用：把 arg（可能是 lvalue 或 rvalue）强制转为 xvalue → 之后调用 T(...) 时优先匹配移动构造。

误区：「std::move(x) 之后 x 就被销毁了」— 错。x 仍然存在，但状态变成「合法但未指定」。你可以给 x 重新赋值或析构，但不应该读它的值（除非你看了具体类型的文档）。 
std::string s = "hello";
std::string s2 = std::move(s);    // s2 拿走 s 的字符

s.size();    // 合法（不是 UB），但值未指定（通常是 0）
s = "world";  // 完全合法 — 重新赋值
// 但不要 assert(s == "hello") — 行为取决于实现

4.2 std::forward 完美转发

它根据模板参数 T 是否带引用，决定到底"做 move 还是不做"。专门给万能引用场景用。

// 标准库简化
template<class T>
constexpr T&& forward(std::remove_reference_t<T>& arg) noexcept {
    return static_cast<T&&>(arg);
}

关键：依赖引用折叠。如果 T 是 int&，T&& 折叠为 int&（不 move）；如果 T 是 int，T&& 是 int&&（move）。

template<class T>
void wrapper(T&& arg) {
    backend(std::forward<T>(arg));     // ✅ 完美转发
}

int x = 1;
wrapper(x);             // T = int& → forward 返回 int& → backend 收到 lvalue
wrapper(42);            // T = int → forward 返回 int&& → backend 收到 rvalue (能 move)
记忆口诀：
· std::move = 我确定要移动 → 无条件转 rvalue
· std::forward = 别人怎么传给我，我就怎么传出去 → 条件转

4.3 易错对比

template<class T>
void bad1(T&& arg) {
    backend(arg);                // ❌ arg 是 lvalue（有名），右值属性丢失
}

template<class T>
void bad2(T&& arg) {
    backend(std::move(arg));     // ❌ 即便实参是 lvalue，也强制 move → 破坏调用方变量
}

template<class T>
void good(T&& arg) {
    backend(std::forward<T>(arg));    // ✅ 按照实参类型转发
}

06

Rule of 0 / 3 / 5（最重要的设计准则）

Rule of 0 的好处（举例） · Rule of 5 完整模板

Rule of 0（首选）

"不写任何特殊成员，让编译器全部默认生成"。

前提：所有成员都是已经管好资源的类型（vector、string、unique_ptr、shared_ptr）— 称为 RAII 友好。

这是现代 C++ 的金标准。

Rule of 3（C++03 时代）

如果你需要写任意一个： 析构、拷贝构造、拷贝赋值，三个都要写。

原因：之所以需要写其中之一，说明类有独占资源（裸 new/file fd）。析构要释放，拷贝要深复制，三者必须一致。

Rule of 5（C++11 起）

有了移动语义，三个变成五个：析构、拷贝构造、拷贝赋值、移动构造、移动赋值。

不一致就是 bug 的温床。

Rule of 5 的工程现实

真实代码里写 5 个成员是很麻烦的。策略：把"独占资源"封装进 RAII 类（如 unique_ptr 自定义 deleter），外层类回归 Rule of 0。

6.1 Rule of 0 的好处（举例）

class User {
    std::string name;
    std::vector<Order> orders;
    std::shared_ptr<Profile> profile;
    // 编译器自动生成：默认构造 / 析构 / 拷贝(2) / 移动(2) — 都正确！
};
// User 的所有 6 个成员都是默认生成且行为正确

6.2 Rule of 5 完整模板

class Buffer {
    char* data_ = nullptr;
    size_t size_ = 0;
public:
    // 1. 默认构造
    Buffer() = default;

    // 自定义构造
    explicit Buffer(size_t n)
        : data_(new char[n]), size_(n) {}

    // 2. 析构
    ~Buffer() { delete[] data_; }

    // 3. 拷贝构造
    Buffer(const Buffer& other)
        : data_(new char[other.size_]), size_(other.size_) {
        std::memcpy(data_, other.data_, size_);
    }

    // 4. 拷贝赋值（copy-and-swap 惯用法 — 异常安全）
    Buffer& operator=(Buffer other) noexcept {     // ✅ 按值收（拷贝或移动）
        swap(other);
        return *this;
    }
    // ↑ 一个赋值搞定 拷贝赋值 + 移动赋值（编译器选择实参传递方式）

    // 5. 移动构造
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;
        other.size_ = 0;
    }

    // 友元 swap
    void swap(Buffer& other) noexcept {
        using std::swap;
        swap(data_, other.data_);
        swap(size_, other.size_);
    }
};
copy-and-swap 惯用法 = 一个 operator=(Buffer)（按值收）+ swap = 同时实现拷贝赋值和移动赋值，且强异常安全保证。

编译器自动生成规则（必背）

自动生成规则总表 · 实战陷阱举例 · 隐式删除场景

编译器何时自动生成、何时= delete by default、何时不生成？这是面试爱挖的细节。

7.1 自动生成规则总表

声明了什么默认构造析构拷贝构造拷贝赋值移动构造移动赋值
什么都没声明✅ 生成✅ 生成✅ 生成✅ 生成✅ 生成✅ 生成
声明任意构造函数❌ 不生成✅✅✅✅✅
声明析构✅—✅✅❌ 不生成❌ 不生成
声明任一拷贝✅✅——❌ 不生成❌ 不生成
声明任一移动✅✅= delete= delete——
关键陷阱：声明析构 → 移动构造/赋值不再生成（仍可拷贝，但拷贝可能性能差）。这是把 Rule of 3 升级到 Rule of 5 时，新人最常踩的坑。

7.2 实战陷阱举例

struct A {
    std::vector<int> data;
    ~A() { LOG("destroying"); }       // 仅声明析构
};

A a1, a2;
a2 = std::move(a1);                  // ⚠️ 走拷贝赋值！而不是移动 — vector 被深复制
// 因为 ~A 声明导致 移动操作不再自动生成

// 修复：显式 = default
struct B {
    std::vector<int> data;
    ~B() { LOG("destroying"); }
    B() = default;
    B(const B&) = default;
    B& operator=(const B&) = default;
    B(B&&) noexcept = default;       // ✅ 显式恢复
    B& operator=(B&&) noexcept = default;
};
"声明了析构函数还会自动生成移动构造吗?" → 不会。这是 C++ 标准设计师的「危险预警」机制：你写析构 = 表达「这个类有特殊资源」 → 自动生成的拷贝/移动可能不安全 → 编译器宁可保守。

7.3 隐式删除场景

下列情况下，相应特殊成员会被= delete（隐式禁用）：

  • 类含引用成员或 const 成员 → 拷贝赋值/移动赋值 = delete（无法重新绑定）
  • 类含不可拷贝/移动的成员（如 std::mutex、std::unique_ptr）→ 对应特殊成员 = delete
  • 基类的对应特殊成员 = delete → 派生类也 = delete
struct C {
    std::mutex mu;          // mutex 不可拷贝/移动
};
C c1, c2;
C c3 = c1;              // ❌ 编译错：拷贝构造被隐式删除
C c4 = std::move(c1);   // ❌ 同样
08

= default / = delete 进阶用法

显式声明 = default 的价值 · = delete 的高级用法

8.1 显式声明 = default 的价值

  1. 恢复被你的其它声明抑制掉的默认行为
  2. 表达意图："我希望编译器生成默认实现"（vs 留空体）
  3. 可以单独控制访问级别（public 默认构造 + private 拷贝构造）
  4. = default 生成的是合成版本（如逐字段拷贝），通常和你手写的等价但更高效
class Point {
public:
    int x, y;
    Point() = default;                      // public
    Point(int a, int b) : x(a), y(b) {}
private:
    Point(const Point&) = default;          // private 拷贝
};

8.2 = delete 的高级用法

① 禁用某个特殊成员

class UniqueResource {
public:
    UniqueResource(const UniqueResource&)            = delete;
    UniqueResource& operator=(const UniqueResource&) = delete;
    UniqueResource(UniqueResource&&) noexcept            = default;
    UniqueResource& operator=(UniqueResource&&) noexcept = default;
};

② 禁用危险重载

void log(const std::string& msg);
void log(int) = delete;     // 防止 int 隐式转换为 const char*

log("hello");          // ✅
log(42);               // ❌ 编译错（指出 = deleted）

③ 禁用所有非指定类型

class Handler {
public:
    void on(const std::string&);
    template<class T>
    void on(T&&) = delete;       // 任何其它类型都拒绝
};
旧 C++03 禁拷贝的写法："声明 private + 不实现"。新代码一律用 = delete（编译期报错而非链接期）。

RVO / NRVO / Copy Elision

三个术语梳理 · RVO（强制） · NRVO（编译器自由选择） · Copy Elision 的精确条件 · Copy Elision vs Move 对比

9.1 三个术语梳理

术语含义C++17 强制?
RVO (Return Value Optimization)函数返回临时对象（prvalue）时，省略拷贝/移动✅ 强制（C++17 起）
NRVO (Named RVO)返回具名局部对象时省略拷贝/移动❌ 仍是允许但不强制
Copy Elision统称：编译器允许在某些场合省略拷贝/移动构造—

9.2 RVO（强制）

Buffer make_buffer(size_t n) {
    return Buffer(n);            // 临时 prvalue
}

Buffer b = make_buffer(1024);
// C++17 起：保证零拷贝。Buffer 直接在 b 的地址上构造。
// 这意味着即便 Buffer 不可拷贝/移动也能这么写。

9.3 NRVO（编译器自由选择）

Buffer make_buffer(size_t n) {
    Buffer b(n);                 // 具名
    b.init();
    return b;                    // NRVO 可能优化掉拷贝/移动
}
// 主流编译器都会优化，但标准不强制
千万不要写 return std::move(b);
这会阻止 NRVO（因为返回类型变了：本应 prvalue → 现在是 xvalue）。手动 move 让你失去零拷贝，反而强制做移动构造，比编译器优化差。

9.4 Copy Elision 的精确条件

必须满足下列全部条件之一，编译器才能 elide：

  1. return 语句返回与函数返回类型同类型（去 cv）的非 volatile 非函数参数的局部对象 → NRVO
  2. return 语句返回与函数返回类型同类型的临时对象（prvalue） → RVO（C++17 强制）
  3. throw 表达式抛出与 catch 类型同的局部对象 → 类似 NRVO
  4. 临时对象传给函数的by-value 参数（C++17 强制）

9.5 Copy Elision vs Move 对比

场景无优化有 RVO/NRVO
Buffer b = make_buffer();构造 + 拷贝/移动 + 析构 临时仅一次构造 — 直接在 b 地址
v.push_back(make_buffer())临时 + 移动到 vector直接在 vector 内构造（等价 emplace）
规则：
① 返回局部对象 → 直接 return obj;，不要 std::move
② 返回函数参数 → 必须 std::move（参数有具名→不能 NRVO）
③ 返回成员 → 必须 std::move（成员是 lvalue，不会被自动当 rvalue） 
struct Foo {
    Buffer buf;
    Buffer extract() {
        return std::move(buf);     // ✅ 成员是 lvalue，必须 move 才能移动构造返回
    }
};

Buffer forward_arg(Buffer b) {
    return std::move(b);          // ✅ 参数有名字，NRVO 不适用，需要 move
}

noexcept 与移动构造的关系

STL 容器的「强异常安全保证」 · 解决方案：std::move_if_noexcept · 实测对比 · noexcept 的影响范围 · 例外：何时不能 noexcept

10.1 STL 容器的「强异常安全保证」

std::vector::push_back 在扩容时承诺：操作要么成功，要么 vector 状态完全不变（强保证）。

问题：扩容要把元素从旧 buffer 转到新 buffer。如果 T 的移动构造可能抛异常，转移过程中失败 → 旧 buffer 数据已经被破坏 → 无法回滚。

10.2 解决方案：std::move_if_noexcept

STL 在扩容时检查：noexcept(T(std::move(t))) 是否为 true？

  • true：用移动构造（快）
  • false：退化为拷贝构造（慢但安全；旧 buffer 可保留作回滚）

10.3 实测对比

struct A {
    std::string s;
    A(A&&) /* 没标 noexcept */ = default;
};
struct B {
    std::string s;
    B(B&&) noexcept = default;        // ✅
};

std::vector<A> va;
std::vector<B> vb;
for (int i = 0; i < 10000; ++i) {
    va.emplace_back("hello world");
    vb.emplace_back("hello world");
}
// va 每次扩容走拷贝路径 → 可能慢 10× 以上
// vb 走移动路径 → 快
"为什么自定义类的移动构造一定要标 noexcept?" — 标准答案：① STL 容器扩容性能退化成拷贝；② STL 算法在判断「能否安全移动」时也用 noexcept；③ 析构函数默认 noexcept，移动构造对称起来。
= default 生成的移动构造，若所有成员的移动都 noexcept，则自动是 noexcept。

10.4 noexcept 的影响范围

  • vector::push_back/emplace_back/insert — 扩容路径
  • vector::resize
  • std::move_if_noexcept 类工具
  • 异常安全保证：noexcept 函数承诺不抛 → 程序员可以更激进优化
  • 编译器对 noexcept 函数的优化也更激进（不需要为异常生成展开代码）

10.5 例外：何时不能 noexcept

class Buffer {
    std::pmr::vector<char> v_;        // 用了多态 allocator
public:
    // pmr 容器的移动构造在跨 allocator 时可能分配 → 可能抛
    Buffer(Buffer&&) /* 不能 noexcept */ = default;
};

C++23 deducing this

10b.1 老问题 — *this 的值类别要写 4 个重载 · 10b.2 C++23 写法 — 一个函数搞定 · 10b.3 语法细节 · 10b…

*this 的值类别终于显式可推导 ⭐

P0847R7 引入"explicit object parameter"，允许成员函数把 this 写成第一个显式参数，从而：① 通过模板推导 *this 的值类别（const / & / &&）；② 一个函数代替原来 4 个重载；③ CRTP 不再需要 derived cast；④ Lambda 可以递归调用自身。这是 C++23 最影响日常代码风格的特性之一。

10b.1 老问题 — *this 的值类别要写 4 个重载

// 这就是 std::optional 的 value() 在标准里的真实重载数
template<class T>
class optional {
    T storage_;
public:
    T&                value() &       { return storage_; }                          // lvalue
    const T&          value() const & { return storage_; }                          // const lvalue
    T&&               value() &&      { return std::move(storage_); }              // rvalue
    const T&&         value() const &&{ return std::move(storage_); }              // const rvalue
};
四个函数体几乎一样，只是返回类型和 *this 值类别不同。维护成本高、易写错、模板化时还要写一堆 const_cast / static_cast。

10b.2 C++23 写法 — 一个函数搞定

template<class T>
class optional {
    T storage_;
public:
    template<class Self>
    auto&& value(this Self&& self) {              // ★ this 是第一个显式参数
        return std::forward<Self>(self).storage_;     // 自动 forward *this
    }
};

// 调用
optional<Big> o = ...;
auto&        a = o.value();                // Self = optional&        → 返回 Big&
auto&&       b = std::move(o).value();     // Self = optional&&       → 返回 Big&& (可移动!)
const auto& c = std::as_const(o).value();  // Self = const optional& → 返回 const Big&
核心收益：① 1 个模板代替 4 个重载；② 通过引用折叠 + perfect forwarding 自动给出正确的值类别；③ 维护成本骤降。

10b.3 语法细节

形式含义类比传统重载
void f(this Self&& self)forwarding reference 模板，Self 推导出 value category四组重载 + 引用折叠
void f(this Self self)by-value：拷贝/移动当前对象（罕用，Builder 模式有用）—
void f(this const C& self)固定类型，仅 const lvalue 调用等价 void f() const &
void f(this C&& self)固定类型，仅 rvalue 调用等价 void f() &&
规则：① 函数体内不能再用 this 指针，必须通过 self；② 不能与同名的非 explicit-object-parameter 成员函数共存；③ 不能用于虚函数、构造/析构函数。

10b.4 杀手级应用 ① — CRTP 不再需要 static_cast

// 旧 CRTP — 派生类调用要 static_cast 取回真实类型
template<class D>
struct Counted {
    int id() const {
        return static_cast<const D&>(*this).compute_id();
    }
};
struct User : Counted<User> {
    int compute_id() const { return 42; }
};

// 新写法 — 完全消除 CRTP 模板基类
struct User {
    int compute_id() const { return 42; }
    template<class Self>
    int id(this const Self& self) {
        return self.compute_id();         // Self 已是 User，无需 cast
    }
};

影响：大量库（CRTP 风格的 expression templates、流式 builder、smart pointer 基类）可以彻底重写。cpp-coro / Eigen / Boost.Hana 都开始迁移。

10b.5 杀手级应用 ② — 递归 lambda（不再需要 std::function 或 Y-combinator）

// C++20 及之前 — 递归 lambda 是地狱
auto fact = [](int n) {
    return n <= 1 ? 1 : n * fact(n - 1);   // ❌ fact 在自己内部不可见
};
// 老的 workaround: std::function 类型擦除（性能差），或写 Y-combinator

// C++23 — 通过 deducing this 自然递归
auto fact = [](this auto& self, int n) -> int {
    return n <= 1 ? 1 : n * self(n - 1);   // ✅ self 就是 lambda 闭包对象
};

std::cout << fact(10);   // 3628800

编译器对 lambda 自动调用 operator() 时把 lambda 闭包对象作为 self 传入。零开销—— 没有 std::function 虚调用、没有堆分配。

10b.6 杀手级应用 ③ — 流式 Builder（链式调用自动 forward）

class RequestBuilder {
    std::string url_;
    std::map<std::string, std::string> headers_;
    std::string body_;
public:
    template<class Self>
    auto&& set_url(this Self&& self, std::string u) {
        std::forward<Self>(self).url_ = std::move(u);
        return std::forward<Self>(self);          // ★ 保持值类别链式传递
    }
    template<class Self>
    auto&& add_header(this Self&& self, std::string k, std::string v) {
        std::forward<Self>(self).headers_[std::move(k)] = std::move(v);
        return std::forward<Self>(self);
    }
};

// 调用
Request r = RequestBuilder{}                       // rvalue 链
              .set_url("/api")                       // 每一步保持 rvalue
              .add_header("Auth", token)
              .build();                              // rvalue → 触发 move into Request

RequestBuilder b;
b.set_url("/api").add_header("Auth", t);   // lvalue 链 — 无意外 move

过去写 Builder 要么把每个 setter 写两遍（& 版 和 && 版），要么牺牲值类别正确性。deducing this 一次解决。

10b.7 杀手级应用 ④ — 容器 / 视图的统一访问

// 你的自定义 Matrix 想要：
//   m(i, j)              → const T& / T&
//   std::move(m)(i, j)   → T&& (允许移动)

class Matrix {
    std::vector<double> data_;
    size_t cols_;
public:
    template<class Self>
    auto&& operator()(this Self&& self, size_t i, size_t j) {
        return std::forward<Self>(self).data_[i * self.cols_ + j];
    }
};

10b.8 性能与开销

  • 零运行时开销 —— 等价于显式重载的代码（编译器直接特化）
  • 二进制可能膨胀：模板特化为不同值类别会产生多个实例（与手写 4 个重载相同），但通过 extern template / type erasure 可控
  • 对调试稍微不友好：栈帧里看到的是 foo<optional&&>，pretty-printer 需要适配

10b.10 与值类别五兄弟（§2）的关系

本章前面讲过 *this 是 lvalue、std::move(*this) 是 xvalue。deducing this 让调用方的 *this 值类别能被推导到模板参数 Self，再通过引用折叠 + perfect forwarding 完整保留：

调用形式Self 推导为std::forward<Self>(self) 是
obj.f()T&lvalue
std::as_const(obj).f()const T&const lvalue
std::move(obj).f()Txvalue
std::move(std::as_const(obj)).f()const Tconst xvalue

这是值类别 + 引用折叠 + 模板推导 + perfect forwarding四个机制的集大成应用。理解了它，前面 §2-§4 的知识就活了。

11.1 FileHandle — POSIX 文件描述符 RAII

class FileHandle {
    int fd_ = -1;
public:
    FileHandle() noexcept = default;
    explicit FileHandle(int fd) noexcept : fd_(fd) {}

    ~FileHandle() noexcept {
        if (fd_ >= 0) ::close(fd_);
    }

    // 禁拷贝（fd 是独占资源）
    FileHandle(const FileHandle&)            = delete;
    FileHandle& operator=(const FileHandle&) = delete;

    // 移动 = 转移 fd
    FileHandle(FileHandle&& o) noexcept : fd_(o.fd_) {
        o.fd_ = -1;
    }
    FileHandle& operator=(FileHandle&& o) noexcept {
        if (this != &o) {
            if (fd_ >= 0) ::close(fd_);    // 释放原资源
            fd_ = o.fd_;
            o.fd_ = -1;
        }
        return *this;
    }

    int get() const noexcept { return fd_; }
    int release() noexcept { int tmp = fd_; fd_ = -1; return tmp; }
    explicit operator bool() const noexcept { return fd_ >= 0; }
};

11.2 Buffer — 用 unique_ptr 实现 Rule of 0

class Buffer {
    std::unique_ptr<char[]> data_;
    size_t size_ = 0;
public:
    Buffer() = default;
    explicit Buffer(size_t n) : data_(std::make_unique<char[]>(n)), size_(n) {}

    // 不写析构、不写移动 — unique_ptr 已经管好了
    // 默认拷贝是 = delete (unique_ptr 不可拷贝) → 显式 = delete 表达意图
    Buffer(const Buffer&)            = delete;
    Buffer& operator=(const Buffer&) = delete;

    // 移动自动生成（unique_ptr 提供）
    Buffer(Buffer&&) noexcept            = default;
    Buffer& operator=(Buffer&&) noexcept = default;
};

11.3 Connection — 持有 fd + Buffer + 状态

class Connection {
    FileHandle sock_;
    Buffer in_, out_;
    std::string peer_;
    std::atomic<ConnState> state_{ConnState::Init};

public:
    Connection(FileHandle s, std::string peer)
        : sock_(std::move(s)), peer_(std::move(peer)) {}

    // atomic 不可拷贝 / 不可移动 → 默认拷贝/移动 都是 = delete
    // 如果想要 movable，需要重新设计 state_ 字段
    Connection(const Connection&) = delete;
    Connection& operator=(const Connection&) = delete;
    // 通常这类对象用 unique_ptr<Connection> 管理（不直接 move 容器）
};

11.4 Session — 含锁，不可移动的"门面"

class Session {
    std::mutex mu_;                        // 不可拷贝/移动
    std::deque<Frame> out_q_;

public:
    // mutex → 默认 都是 = delete

面试题集（20 道分层）

13.1 基础（必答）

  1. 什么是 lvalue / rvalue / xvalue / prvalue?

    用「有无身份(identity，可取地址) × 能否移动(movable)」两维度划分：

    • lvalue：有身份、不可被移动。有名变量、*p、返回 T& 的函数。
    • prvalue：无身份、可移动。字面量 42、表达式 a+b、返回 T（按值）的函数。
    • xvalue（将亡值）：有身份、可移动。std::move(x)、返回 T&& 的函数。
    • glvalue = lvalue ∪ xvalue（都有身份）；rvalue = prvalue ∪ xvalue（都可移动）。
  2. std::move 做了什么? 它真的"移动"吗?

    它不移动任何东西，只是一个 static_cast<remove_reference_t<T>&&>，把表达式无条件转换成 xvalue（右值）。

    • 编译后零运行时开销，只改变值类别，使重载决议命中移动构造 / 移动赋值。
    • 真正的资源转移发生在移动构造 / 移动赋值的函数体里，而非 std::move 本身。
  3. 什么是 Rule of 3 / 5 / 0?
    • Rule of 3：若需自定义「析构 / 拷贝构造 / 拷贝赋值」之一，通常三者都要（管理资源时一致性）。
    • Rule of 5：C++11 起再加「移动构造 / 移动赋值」共五个。
    • Rule of 0：不写任何特殊成员，用 RAII 成员（智能指针、容器）让编译器全自动生成 —— 最推荐。
  4. 移动构造与拷贝构造的关键区别?
    • 拷贝构造 T(const T&)：深拷贝资源，源对象不变，开销 O(n)。
    • 移动构造 T(T&&)：窃取源对象的资源指针并把源置空，O(1)；源对象之后仍需保持可析构、可赋值的有效状态。
  5. 什么时候编译器自动生成移动构造?

    当类没有声明任何「拷贝构造 / 拷贝赋值 / 移动赋值 / 析构」，且所有非静态成员与基类都可移动时，编译器隐式生成移动构造。

    反之：声明了析构或任一拷贝操作 → 移动构造 / 移动赋值不再自动生成（见 §14.3 "X 抑制 Y" 表）。

  6. 为什么 std::vector 的元素需要可移动?

    扩容（reserve / push_back 触发 reallocation）时，需要把旧缓冲区元素搬到新内存：

    • 可移动 → O(1) 转移资源；否则只能逐个深拷贝。
    • 实际用 std::move_if_noexcept：移动构造非 noexcept 时退化为拷贝，以保证强异常安全。

13.2 中级（区分熟练度）

  1. 引用折叠规则? 用一个例子说明万能引用怎么工作。

    折叠规则：只要有一个左值引用，结果就是左值引用 —— & &→&，& &&→&，&& &→&，&& &&→&&。

    万能引用 T&&（T 为待推导模板参数）：

    template<class T> void f(T&& x);
int i = 1;
f(i);   // 传左值 → T 推成 int&，int& && 折叠为 int&
f(42);  // 传右值 → T 推成 int，int && 即 int&&

    于是同一个 T&& 既能接左值也能接右值，再配合 std::forward 保持原值类别。

  2. std::forward 与 std::move 区别? 各用什么场景?
    • std::move：无条件转右值。用于"我确定不再用这个对象，可窃取它"，如具名右值、转移成员。
    • std::forward：有条件转换 —— 左值仍转为左值、右值仍转为右值，保持原值类别。只用于万能引用 T&& 的完美转发。

    口诀：拿到具名右值用 move；转发模板形参用 forward。

  3. RVO / NRVO 区别? C++17 强制了哪个?
    • RVO：返回无名临时（return T(...);）→ C++17 起强制省略，prvalue 直接在目标位置构造，连可拷贝 / 可移动都不要求。
    • NRVO：返回具名局部变量（return obj;）→ 编译器可选优化，C++17 也未强制。
  4. 为什么写 return std::move(local) 是反模式?
    • 它把返回表达式变成 xvalue，阻止了 NRVO（具名变量本可被 copy elision），强行多走一次移动构造，反而更慢。
    • 对 return local; 编译器已会把局部变量当右值处理（隐式移动），所以正确写法就是 return local;。
  5. 为什么自定义类的移动构造要标 noexcept?

    容器（如 std::vector）扩容时用 std::move_if_noexcept：

    • 移动构造标了 noexcept → 扩容用移动（高效）。
    • 未标 noexcept（可能抛）→ 退回拷贝以保证强异常安全 → 性能损失。
  6. const T& 与 T&& 哪个更通用?
    • const T& 更通用：能绑左值、右值、const、临时，几乎一切；但只读、无法移动。
    • T&&（具体类型）只绑右值。
    • 若要"既通用又能转移"，用万能引用 T&&（模板）+ forward；纯只读则 const T&。

13.3 高级（体现深度）

  1. 给一个含 std::mutex 的类，编译器为什么生成不出拷贝构造?

    std::mutex 的拷贝构造被 = delete（且不可移动）。编译器隐式生成的拷贝构造要逐成员拷贝，遇到不可拷贝的 mutex 成员 → 整个拷贝构造被定义为 deleted。

    解决：自己实现拷贝控制，不拷贝 mutex 本身，只在持锁状态下拷贝被保护的数据。

    T(const T& o) {
    std::lock_guard lk(o.mtx_);   // 锁源，不拷 mtx_
    data_ = o.data_;
}
  2. copy-and-swap 惯用法的 4 个好处?
    1. 一份代码两用：按值传参的 operator=(T rhs) 同时实现拷贝赋值与移动赋值。
    2. 强异常安全：先构造副本（可能抛在此处，但 *this 未动），再 swap（通常 noexcept）。
    3. 天然自赋值安全：操作的是独立副本，无需手写 if(this != &rhs)。
    4. 少重复：复用已有的拷贝 / 移动构造与析构逻辑。
  3. std::move_if_noexcept 在 vector::push_back 中如何工作?

    扩容需把旧元素转移到新内存：

    • 元素移动构造为 noexcept → 返回右值引用 → 移动（高效）。
    • 移动可能抛 → 返回 const 左值引用 → 拷贝。因为移动到一半抛异常会让旧缓冲区元素已被破坏、无法回滚，违反强异常安全；拷贝则旧数据完好。
  4. 给一个万能引用 + 普通引用重载的二义性例子，并修复。

    典型坑：万能引用构造抢占了拷贝构造。

    struct Widget {
    template<class T> Widget(T&& x);     // (A) 万能引用
    Widget(const Widget&);              // (B) 拷贝构造
};
Widget w1;
Widget w2 = w1;   // ⚠️ w1 是非 const 左值，(A) T=Widget& 比 (B) const& 更匹配 → 调到 (A)!

    修复：用约束排除自身类型。

    template<class T>
    requires (!std::same_as<std::decay_t<T>, Widget>)
Widget(T&& x);   // C++20 concept；C++17 用 enable_if
  5. 下列代码有几次构造、几次拷贝、几次移动?
    std::vector<std::string> v; v.reserve(10); v.push_back("hello");

    1 次构造 + 1 次移动构造（无拷贝）：

    • "hello"（const char[6]）先构造一个临时 std::string → 1 次构造。
    • 临时是右值，命中 push_back(string&&) → 1 次移动构造进 vector。
    • 已 reserve(10) → 不扩容，无额外搬移。
    • 优化：v.emplace_back("hello") → 仅 1 次原地构造，连临时都省。

    详细分解见 §13.4 速答示范。

  6. 解释为什么 auto&& 是万能引用但 const auto&& 不是。
    • 万能引用的前提是：一个裸的、待推导的 T&& / auto&&，依赖引用折叠才能在接左值时折叠成左值引用。
    • const auto&& 加了 const 限定，破坏了"推导成左值引用"的可能 —— 它只能绑 const 右值，是一个普通右值引用，不再万能。
  7. 函数模板 template<class T> void f(std::vector<T>&& v); — 这是不是万能引用?

    不是。万能引用要求形参恰好是裸的 T&&（T 被直接推导）。这里外层类型是确定的 std::vector<T>&&，T 仅在 vector 内部被推导，所以它是一个普通右值引用，只接受 vector 的右值实参。

  8. 为什么字符串字面量 "hello" 是 lvalue 而不是 prvalue?
    • "hello" 的类型是 const char[6]，是一个存放在静态存储区、有地址的数组对象，可以 &"hello" 取址 → 有身份 → lvalue。
    • 对比：42、true 等内置标量字面量没有持久存储，是 prvalue。数组字面量因为需要实际存储而成为 lvalue。

13.4 速答示范（第 17 题）

问：std::vector<std::string> v; v.reserve(10); v.push_back("hello"); 几次构造、拷贝、移动? 
// 步骤分解：
// 1. "hello" 是 const char[6]，先转换为临时 std::string  → 1 次构造
// 2. push_back 接收 const string& 重载 → 把临时拷贝到 vector  → 1 次拷贝构造
// 3. 也存在 push_back(string&&) 重载 → 临时是 rvalue，会绑过去 → 1 次移动构造
// 实际：1 次构造 + 1 次移动构造（重载选择 string&&）
// 优化：v.emplace_back("hello") → 仅 1 次原地构造
要拿满分，必须分清"有 reserve" → 不会扩容 → 只看 push_back 本身的构造代价。如果没 reserve，扩容时还可能移动之前的元素。

template<class F, class... Args>
auto wrapper(F&& f, Args&&... args)

}
