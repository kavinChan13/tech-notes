为什么 CMake 难用 · "Modern CMake" 是什么

"Modern CMake" 定义 · CMake 版本要求

CMake 是 C++ 事实标准 build system（2026 占 ~80% 主流项目），但大多数人用错了。原因：

  • 1999 年诞生，bug fix + feature 持续 20+ 年，多种 paradigm 并存——老 doc / Stack Overflow 答案误导性极强
  • add_library / include_directories / link_libraries 这些 "directory-scoped" 老命令仍存在但是 anti-pattern
  • 新人学 CMake 通常 copy-paste 老代码，把 anti-pattern 复制下去

"Modern CMake" 定义

2014 Daniel Pfeifer 在 CppCon 演讲 "Effective CMake" 定义了 modern CMake。核心：

📐 Modern CMake 4 原则
  1. Everything is a target. 不再 global include_directories / link_libraries——一切属性挂在 target 上
  2. PUBLIC / PRIVATE / INTERFACE 明确传递性——属性自动随依赖图传播
  3. 不操控 CMAKE_CXX_FLAGS——用 target_compile_options + generator expression
  4. 不写 directory-scoped 命令——include_directories / link_libraries 都是 legacy

CMake 版本要求

2026 推荐 CMake ≥ 3.25（2022 release）。理由：

  • 3.20+ 完整 generator expression 支持
  • 3.21+ FILE_SET for headers
  • 3.23+ FILE_SET HEADERS 用 import-able
  • 3.28+ C++20 Modules 完整支持
cmake_minimum_required(VERSION 3.25)
project(MyProject VERSION 1.0.0 LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)        # 关 GNU extension

Target-Centric 心智模型

种 target · PUBLIC / PRIVATE / INTERFACE 详解 · 判断 PUBLIC vs PRIVATE 的简单规则 · Generator Expression

Modern CMake 的核心是 target。一切属性都挂在 target 上，依赖关系沿着 target graph 传播。

🧪 沙盒：拨动 PUBLIC / PRIVATE / INTERFACE，看属性怎么传给下游
改变 foo 链接两个依赖的可见性，实时观察消费者 myapp 能"看到/链到"什么。生活化类比：PUBLIC = 把名片也转交给客户；PRIVATE = 名片只留自己用；INTERFACE = 自己不用、只转交。
foo 链接fmt::fmt
foo 链接spdlog::spdlog
📦 foo 自己 (编译 foo.cpp 时)
fmt::fmt
spdlog::spdlog
→
🚀 myapp (target_link_libraries(myapp PRIVATE foo))
fmt::fmt
spdlog::spdlog

3 种 target

类型命令产物
Executableadd_executable(myapp ...)可执行文件
Library (static / shared / object)add_library(mylib STATIC|SHARED|OBJECT ...).a / .so / .o
Interface library (header-only)add_library(mylib INTERFACE)无 binary，只属性传递

PUBLIC / PRIVATE / INTERFACE 详解

📐 属性传递规则
  • PRIVATE——只本 target用
  • INTERFACE——只依赖 target 的 target用（本 target 不用）
  • PUBLIC——两者都用（PRIVATE + INTERFACE）

典型 example

# Library libfoo.so
add_library(foo SHARED src/foo.cpp)

target_include_directories(foo
  PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
    $<INSTALL_INTERFACE:include>
  PRIVATE
    ${CMAKE_CURRENT_SOURCE_DIR}/src       # 内部 helper headers
)

target_link_libraries(foo
  PUBLIC  fmt::fmt        # 出现在 foo 的 public header → 用 foo 的都要
  PRIVATE spdlog::spdlog  # 只 foo.cpp 用 → 用 foo 的不需要
)

target_compile_features(foo PUBLIC cxx_std_20)

# Executable myapp 用 foo
add_executable(myapp main.cpp)
target_link_libraries(myapp PRIVATE foo)

# 自动生效:
# - myapp 看到 foo 的 PUBLIC includes (include/)
# - myapp 不看 src/ (PRIVATE)
# - myapp 链 libfmt (PUBLIC link)
# - myapp 不链 libspdlog (PRIVATE link of foo)

判断 PUBLIC vs PRIVATE 的简单规则

  1. 该 dep 是否在我的 public header 里 #include？是 → PUBLIC；否 → PRIVATE
  2. 该 dep 的 symbol 是否出现在我的 public API（函数参数 / 返回值 / 类成员）？是 → PUBLIC
  3. 该 dep 是否只在 .cpp 里用？→ PRIVATE

Generator Expression

Generator expression $<...> 让属性按 build mode / platform / compiler 动态变：

# Debug 时多个 define
target_compile_definitions(foo PRIVATE
  $<$<CONFIG:Debug>:FOO_DEBUG_MODE>
  $<$<CONFIG:Release>:FOO_NDEBUG>
)

# GCC / Clang 加 -Wall, MSVC 加 /W4
target_compile_options(foo PRIVATE
  $<$<CXX_COMPILER_ID:GNU,Clang>:-Wall -Wextra -Wpedantic>
  $<$<CXX_COMPILER_ID:MSVC>:/W4>
)

# BUILD_INTERFACE vs INSTALL_INTERFACE
target_include_directories(foo PUBLIC
  $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
  $<INSTALL_INTERFACE:include>
)

依赖管理 4 方案

find_package（system / 大库） · FetchContent（小依赖 / 源代码 embed） · Conan · vcpkg · 选型决策

2026 主流依赖方案 4 种，按场景选：

3.1 find_package（system / 大库）

find_package(Boost 1.83 REQUIRED COMPONENTS system thread)
find_package(OpenSSL REQUIRED)

target_link_libraries(myapp PRIVATE
  Boost::system Boost::thread
  OpenSSL::SSL OpenSSL::Crypto
)
  • 适合：大库 / 系统库（OpenSSL / Boost / Qt / CUDA / ROS）
  • Imported target（如 Boost::system）是 modern style；老 CMake 用 ${Boost_LIBRARIES} variable 是 anti-pattern

3.2 FetchContent（小依赖 / 源代码 embed）

include(FetchContent)
FetchContent_Declare(
  fmt
  GIT_REPOSITORY https://github.com/fmtlib/fmt.git
  GIT_TAG        10.2.1
  GIT_SHALLOW    TRUE
)
FetchContent_MakeAvailable(fmt)

target_link_libraries(myapp PRIVATE fmt::fmt)
  • 适合：header-only / 小 lib（fmt / json / catch2 / spdlog）
  • 下载 + 编译都在 build 时；跨平台 reproducible
  • 缺点：首次配置时 clone 慢；不适合大型 lib

3.3 Conan

# conanfile.txt
[requires]
fmt/10.2.1
spdlog/1.14.1
boost/1.83.0

[generators]
CMakeDeps
CMakeToolchain

# 构建
conan install . --build=missing -of build/
cmake -S . -B build -DCMAKE_TOOLCHAIN_FILE=build/conan_toolchain.cmake
cmake --build build
  • 适合：企业团队 / 多项目 / 严格版本控制
  • 预编译 binary cache（公司内部 Artifactory）
  • 2024 起 Conan 2.0 主流，clean 设计

3.4 vcpkg

# vcpkg.json
{
  "name": "myproject",
  "version": "1.0.0",
  "dependencies": [
    "fmt",
    "spdlog",
    {"name": "boost", "features": ["thread"]}
  ]
}

选型决策

场景推荐
小项目 + 少依赖（< 10）FetchContent
开源项目，让用户能轻松 buildFetchContent + system fallback (find_package)
企业内部，binary cache + 版本严格Conan 2
Windows / Microsoft 生态vcpkg
Linux 发行版用户find_package（system lib）
嵌入式 / yoctofind_package（cross-compile setup）

CMakePresets.json

典型 CMakePresets.json · 使用 · 团队优势

CMake 3.19+ 支持的声明式配置——替代手写 build script + 分享 dev 配置。

典型 CMakePresets.json

{
  "version": 6,
  "cmakeMinimumRequired": { "major": 3, "minor": 25 },
  "configurePresets": [
    {
      "name": "default",
      "binaryDir": "${sourceDir}/build/${presetName}",
      "generator": "Ninja",
      "cacheVariables": {
        "CMAKE_BUILD_TYPE": "Release",
        "CMAKE_EXPORT_COMPILE_COMMANDS": "ON"
      }
    },
    {
      "name": "dev",
      "inherits": "default",
      "cacheVariables": {
        "CMAKE_BUILD_TYPE": "Debug",
        "ENABLE_SANITIZERS": "ON"
      }
    },
    {
      "name": "asan",
      "inherits": "dev",
      "cacheVariables": {
        "CMAKE_CXX_FLAGS": "-fsanitize=address -fno-omit-frame-pointer"
      }
    }
  ],
  "buildPresets": [
    { "name": "default", "configurePreset": "default", "jobs": 8 }
  ]
}

使用

cmake --preset dev          # 配置
cmake --build --preset dev  # 构建
ctest --preset default      # 测试
cmake --list-presets        # 列所有 preset

团队优势

  • IDE 直接读——VSCode / CLion / Visual Studio 都识别 CMakePresets.json
  • 新人 onboarding 无需口口相传 build 命令
  • CI 也用 same presets → 开发跟 CI 100% 一致
  • 用户自定义：个人放 CMakeUserPresets.json（gitignore）

C++20 Modules 支持

CMake 3.28+ (2023) 完整支持 C++20 modules

CMake 3.28+ (2023) 完整支持 C++20 modules。配套 Ninja 1.11+。

cmake_minimum_required(VERSION 3.28)
project(MyApp CXX)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_SCAN_FOR_MODULES ON)

add_library(mymodule)
target_sources(mymodule
  PUBLIC
    FILE_SET CXX_MODULES
    FILES
      math.cppm        # module interface unit
)
target_sources(mymodule PRIVATE
  math_impl.cpp        # module implementation unit
)

add_executable(myapp main.cpp)
target_link_libraries(myapp PRIVATE mymodule)
2026 现状：C++20 modules CMake 支持 working but rough。
✅ GCC 14+ / Clang 17+ / MSVC 19.30+ + CMake 3.28+ + Ninja 1.11+
❌ 跨 lib import 仍有 corner case；标准 lib（import std;）部分支持
❌ Conan / vcpkg 还在 catch up

建议：新项目可试，大项目等 1-2 年更稳。

原生跨平台

if(CMAKE_SYSTEM_NAME STREQUAL "Linux")
  target_link_libraries(myapp PRIVATE dl rt)
elseif(CMAKE_SYSTEM_NAME STREQUAL "Darwin")
  target_link_libraries(myapp PRIVATE "-framework Foundation")
elseif(WIN32)
  target_compile_definitions(myapp PRIVATE WIN32_LEAN_AND_MEAN NOMINMAX)

endif()

Sanitizer + LTO + PGO 集成

统一 sanitizer module · LTO (Link-Time Optimization) · PGO (Profile-Guided Optimization)

统一 sanitizer module

# cmake/sanitizers.cmake
function(enable_sanitizers target)
  if(NOT CMAKE_CXX_COMPILER_ID MATCHES "GNU|Clang")
    return()
  endif()

  option(ENABLE_ASAN  "Enable AddressSanitizer"    OFF)
  option(ENABLE_TSAN  "Enable ThreadSanitizer"     OFF)
  option(ENABLE_UBSAN "Enable UB Sanitizer"        OFF)

  if(ENABLE_ASAN AND ENABLE_TSAN)
    message(FATAL_ERROR "ASan and TSan are mutually exclusive")
  endif()

  if(ENABLE_ASAN)
    target_compile_options(${target} PRIVATE -fsanitize=address -fno-omit-frame-pointer)
    target_link_options(${target} PRIVATE -fsanitize=address)
  endif()
  if(ENABLE_TSAN)
    target_compile_options(${target} PRIVATE -fsanitize=thread)
    target_link_options(${target} PRIVATE -fsanitize=thread)
  endif()
endfunction()

LTO (Link-Time Optimization)

include(CheckIPOSupported)
check_ipo_supported(RESULT ipo_supported OUTPUT ipo_error)
if(ipo_supported AND CMAKE_BUILD_TYPE STREQUAL "Release")
  set_target_properties(myapp PROPERTIES
    INTERPROCEDURAL_OPTIMIZATION TRUE)
endif()

PGO (Profile-Guided Optimization)

# Step 1: 编译 instrumented binary
cmake -S . -B build-pgo-gen -DCMAKE_CXX_FLAGS="-fprofile-generate=/tmp/pgo-data"
cmake --build build-pgo-gen

# Step 2: 跑代表性 workload
./build-pgo-gen/myapp < representative_input.txt

# Step 3: 用 profile 重编
cmake -S . -B build-pgo-use -DCMAKE_CXX_FLAGS="-fprofile-use=/tmp/pgo-data"
cmake --build build-pgo-use

CI / install / package

安装规则 · CPack — 生成发布包 · GitHub Actions example

安装规则

include(GNUInstallDirs)

install(TARGETS foo
  EXPORT FooTargets
  ARCHIVE  DESTINATION ${CMAKE_INSTALL_LIBDIR}
  LIBRARY  DESTINATION ${CMAKE_INSTALL_LIBDIR}
  RUNTIME  DESTINATION ${CMAKE_INSTALL_BINDIR}
  INCLUDES DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}
)

# 生成 FooConfig.cmake 让用户 find_package(Foo)
install(EXPORT FooTargets
  FILE FooTargets.cmake
  NAMESPACE Foo::
  DESTINATION ${CMAKE_INSTALL_LIBDIR}/cmake/Foo
)

用户 install 后能 find_package(Foo 1.0) + target_link_libraries(... Foo::foo)。

CPack — 生成发布包

set(CPACK_GENERATOR "TGZ;DEB;RPM")
set(CPACK_PACKAGE_NAME "foo")
set(CPACK_PACKAGE_VERSION ${PROJECT_VERSION})
include(CPack)

# 用
cmake --build build --target package

GitHub Actions example

# .github/workflows/build.yml
name: CI
on: [push, pull_request]
jobs:
  build:
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        preset: [default, dev, asan]
        exclude:
          - {os: windows-latest, preset: asan}
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: lukka/get-cmake@latest
      - run: cmake --preset ${{ matrix.preset }}
      - run: cmake --build --preset ${{ matrix.preset }}
      - run: ctest --preset default --output-on-failure

大反模式 + 修正

对比 example

#反模式问题修正
1include_directories(...)全 directory 生效 → 污染target_include_directories(target ...)
2link_libraries(...)同上target_link_libraries(target ...)
3改 CMAKE_CXX_FLAGS 全局全 target 受影响target_compile_options(target PRIVATE ...)
4file(GLOB ...) 收 source新增文件 CMake 不知道显式列 source files
5没 PUBLIC/PRIVATE/INTERFACE属性传递不对每个 target_xxx 都显式标
6${Boost_LIBRARIES} variable老接口imported target Boost::system
7硬编 build dir / path不 reproducible用 CMAKE_CURRENT_* + generator expression
8无 version requirement老 CMake 跑新代码失败cmake_minimum_required(VERSION X.Y)
9add_definitions(-DXXX)全 directorytarget_compile_definitions(target PRIVATE XXX)
10config-mode header in src treeinstall 后路径不对BUILD_INTERFACE / INSTALL_INTERFACE

对比 example

❌ Anti-pattern（老 CMake）
include_directories(include)
link_directories(/opt/lib)
set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -Wall")
add_definitions(-DFOO=1)
file(GLOB SRC src/*.cpp)
add_library(foo ${SRC})
target_link_libraries(foo boost_system)
✅ Modern CMake
add_library(foo
  src/foo.cpp
  src/bar.cpp
)
target_include_directories(foo
  PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
    $<INSTALL_INTERFACE:include>
)
target_compile_options(foo PRIVATE -Wall)
target_compile_definitions(foo PRIVATE FOO=1)
target_link_libraries(foo PRIVATE Boost::system)
🎯 关键 take-aways
  • Modern CMake = Target-Centric——一切属性挂 target，不再 global / directory scope
  • PUBLIC / PRIVATE / INTERFACE 决定属性传递性——public header 用 → PUBLIC，否则 PRIVATE
  • 2026 推荐 CMake ≥ 3.25，C++20 modules 用 3.28+
  • 依赖管理 4 选 1：FetchContent / find_package / Conan / vcpkg
  • CMakePresets.json 替代手写 build script + 开发跟 CI 100% 一致
  • Generator expression 让属性按 config / platform / compiler 动态变
  • Sanitizer / LTO / PGO 全部 modular 集成，cmake option 开关
  • Ninja + ccache 是必备 perf 工具，5-50× 快于 make
  • 10 大反模式：file(GLOB) / include_directories / 全局 flag——全部弃用
