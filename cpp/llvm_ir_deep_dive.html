① 为什么是 LLVM IR

LLVM IR（Intermediate Representation）是编译器内部表示—— 介于高级语言和机器代码之间的类汇编 + 类型化 + SSA 语言。它的设计目标：

  1. 跨语言—— C / C++ / Swift / Rust / Julia / Zig / Crystal 等都编到同一 IR
  2. 跨平台—— x86 / ARM / RISC-V / WebAssembly / NVPTX (GPU) / AMDGPU 等 backend 共享一个 IR
  3. 跨用途—— 同一 IR 既用于 AOT 编译，也用于 JIT、static analysis、obfuscation、sanitizer 插桩
  4. 外部可读写—— 不像 GCC GIMPLE 那么内部，外部工具能 emit / parse / transform IR

三种形态

形态扩展名用途
Textual IR.ll人读 + godbolt + 调试
Bitcode.bc二进制，紧凑，LTO 时存到 .o 里
In-memory—llvm::Module C++ 对象，跑 pass 时

三者语义完全等价，可互转：llvm-dis (bc → ll) / llvm-as (ll → bc)。

② IR 解剖学

IR 是嵌套结构：

Module          ; 一个 .cpp 翻译单元 → 一个 Module
└── Function    ; 每个 C++ function → 一个 Function
    └── BasicBlock     ; 顺序执行的指令序列 (基本块)
        └── Instruction   ; load / store / add / br / call / ret 等

最小完整 example

// hello.cpp
int add(int a, int b) {
  if (a > b) return a;
  return b;
}

编译 clang -S -emit-llvm -O0 hello.cpp -o hello.ll（O0 保留所有细节）：

; ModuleID = 'hello.cpp'
source_filename = "hello.cpp"
target datalayout = "e-m:e-p270:32:32-..."
target triple = "x86_64-pc-linux-gnu"

; Function Attrs: noinline nounwind optnone uwtable
define dso_local noundef i32 @_Z3addii(i32 noundef %0, i32 noundef %1) #0 {
  %3 = alloca i32, align 4              ; 局部 var a (栈)
  %4 = alloca i32, align 4              ; 局部 var b
  %5 = alloca i32, align 4              ; 返回值 slot
  store i32 %0, ptr %3, align 4         ; a = arg0
  store i32 %1, ptr %4, align 4         ; b = arg1
  %6 = load i32, ptr %3, align 4
  %7 = load i32, ptr %4, align 4
  %8 = icmp sgt i32 %6, %7              ; a > b
  br i1 %8, label %9, label %11         ; 条件分支

9:                                       ; preds = %2
  %10 = load i32, ptr %3, align 4
  store i32 %10, ptr %5, align 4
  br label %13                          ; 无条件跳转到 exit

11:                                      ; preds = %2
  %12 = load i32, ptr %4, align 4
  store i32 %12, ptr %5, align 4
  br label %13

13:                                      ; preds = %11, %9
  %14 = load i32, ptr %5, align 4
  ret i32 %14
}

跑 -O1 优化后 IR 截然不同：

define noundef i32 @_Z3addii(i32 noundef %a, i32 noundef %b) {
entry:
  %cmp = icmp sgt i32 %a, %b
  %r = select i1 %cmp, i32 %a, i32 %b     ; select 替代 branch
  ret i32 %r
}

Optimizer 把 5 个 basic block + 多次 load/store 压缩成 3 行 + 1 个 select。这就是SSA 优化的威力—— 见下节。

CFG（控制流图）

每个 Function 的 basic block 之间通过 br 指令连成 CFG（Control Flow Graph）：

entry │ ▼ ┌───────┐ │ %2 │ (icmp + br) └─┬───┬─┘ │true │false ▼ ▼ ┌───┐ ┌───┐ │ %9│ │%11│ └─┬─┘ └─┬─┘ │ │ └──┬──┘ ▼ %13 (ret)

大多数优化 pass 都在CFG 上工作—— 分析 reachability / dominator / loop / def-use。

③ SSA 与 Phi 节点

SSA (Static Single Assignment)—— 每个变量只赋值一次。听起来限制大，但实际让 def-use 分析 trivial—— 每个 use 只能对应一个 def。

问题：if-else 怎么 SSA？

int x;
if (cond) x = 1;
else    x = 2;
use(x);    // x 有 2 个 def，违反 SSA

解：phi 节点—— "根据走的哪条分支挑选合适的 def"：

entry:
  %cond = ...
  br i1 %cond, label %if_true, label %if_false

if_true:
  %x1 = i32 1
  br label %merge

if_false:
  %x2 = i32 2
  br label %merge

merge:
  %x = phi i32 [ %x1, %if_true ], [ %x2, %if_false ]
  call void @use(i32 %x)

phi 节点在 basic block 顶部，根据"从哪条 incoming edge 来"选不同 def。这是 SSA 的核心创新。

mem2reg —— 把栈变量提升到 SSA

看上面 -O0 IR：所有 local var 用 alloca + load/store 操作（即"内存 form"）。-O1 后这些 alloca 消失，变量直接在 register（SSA value）—— 这就是 mem2reg pass 做的。

O0 (内存 form)O1 (SSA form)
%a = alloca i32
store i32 %0, ptr %a
%t = load i32, ptr %a
%t2 = add i32 %t, 1
store i32 %t2, ptr %a		%a1 = add i32 %0, 1

(no load/store!)

mem2reg 让所有后续 pass（GVN / inline / LICM 等）能直接看 def-use chain，效率 ×100。没 mem2reg 就没现代 LLVM 优化。

SSA 不能完全表达什么

  • 逃逸到内存的对象—— 还是 alloca + load/store（不能 mem2reg）
  • 非局部跨函数 def—— global 变量 / heap 对象
  • 循环归纳变量—— 需要 phi 节点配 induction variable analysis

所以 SSA 是函数内 的，跨函数仍需 inter-procedural analysis。

④ 类型系统

LLVM IR 是强类型—— 不像传统汇编。每个 value 都有 type，operation 必须 type-compatible。

基础类型

类型含义例
i1, i8, i16, i32, i64, i128固定 width 整数（无符号性由 op 决定）i32 42
iN（任意 N）任意 bit width（如 i7）偶尔用于 bit-packed 数据
half, float, double, fp128浮点double 3.14
void无返回值（仅函数返回 / 无 phi）ret void
labelbasic block 标签label %entry
token不透明（用于 coroutine / exception）token none

聚合类型

; struct (literal)
{ i32, double, ptr }

; array
[10 x i32]              ; 10 个 int

; named struct
%struct.MyClass = type { i32, ptr, [4 x i8] }

; vector (SIMD)
<4 x float>             ; 4-lane SSE
<8 x i32>               ; 8-lane AVX

; function type
i32 (i32, ptr)          ; takes (i32, ptr), returns i32

Opaque Pointers (LLVM 16+)

2022 年起 LLVM 全部 pointer 类型 typeless—— 之前是 i32* / %struct.X* / ptr，现在统一只 ptr。原因：

  • 历史 typed pointer 是 design mistake（type 跟 pointer 强绑导致 bitcast 满天飞）
  • 实际指针的 "type" 应该在 load/store/GEP 指令上显式标
  • 简化 IR + 减少冗余信息
; 老式 (LLVM < 15):
%1 = load i32, i32* %ptr_to_int
%2 = load %struct.X, %struct.X* %ptr_to_struct

; Opaque pointers (LLVM 16+):
%1 = load i32, ptr %ptr_any
%2 = load %struct.X, ptr %ptr_any
                  // ↑ 类型信息在 load 指令本身

影响：写 LLVM pass 不能再依赖 pointer's "pointee type"，必须看 use site。

⑤ Metadata + Attribute

除了 instruction 本身，LLVM IR 还有大量 metadata：

常见 metadata

Metadata用途
!dbg !NDebug info（源代码 line / column / file）
!tbaa !NType-Based Alias Analysis（指针不会 alias 不同 type）
!range !N值范围（如"this load 一定 ≥ 0"）
!llvm.loop !N循环属性（vectorize / unroll hint）
!nonnullload 出来的指针非 null
!nontemporalnontemporal load/store (绕过 cache)
!alignload 出来的指针对齐保证

Attribute

函数 / 参数级的属性：

Attribute含义
noinline / alwaysinlineinline 控制
readnone / readonly / writeonlymemory effect（让 optimizer 知道函数是不是纯）
nounwind不抛异常（C++ noexcept 翻译来）
willreturn保证 return（不 infinite loop / abort）
noundef参数 / 返回值不能是 undef / poison
nonnull指针参数非 null（caller / callee 保证）
noalias指针参数不 alias 其他参数（C 的 restrict 翻译）

这些属性让 optimizer 更激进。如 readonly 函数的多次调用可以被 CSE 合并；noalias 让 loop vectorizer 可以并行化原本被 alias 阻止的 loop。

⑥ Pass 流水线深入

LLVM 的所有优化 / 分析 / transform 都组织成 Pass。一个 pass 干一件事。-O2 跑约 150 个 pass，-O3 多 20-30 个。

两大类

类型做什么例
Analysis Pass不修改 IR，只产出信息DominatorTree / LoopInfo / AliasAnalysis
Transform Pass修改 IRInstCombine / GVN / LoopUnroll

Transform Pass 通常 依赖 Analysis Pass 的结果。如 LICM (Loop Invariant Code Motion) 依赖 LoopInfo + AliasAnalysis。

Pass 的粒度

粒度作用范围例
Module Pass整个 Module（如跨函数）Inliner / GlobalOpt
Function Pass单个 FunctionInstCombine / GVN
Loop Pass单个 LoopLoopUnroll / LICM
CGSCC PassCall Graph Strongly-Connected ComponentInliner (新版)

New PassManager (NewPM)

LLVM 13+ 默认用 NewPM（新 pass manager），替代了 2003 年的 legacy PM。差异：

  • NewPM 编译时多态（CRTP），legacy 是 runtime virtual call —— NewPM 快 ~20%
  • NewPM 允许 nested pipelines—— Module → Function → Loop 嵌套调度更灵活
  • NewPM 写法不同（继承 PassInfoMixin<MyPass> + 实现 run()）
  • Legacy PM 仍可用但 deprecated

⑦ 7 个关键优化 Pass 内部

7.1 Inliner —— 函数体替代 call

  • 核心算法：cost-benefit。Cost = inlined 后的代码 size 增长；Benefit = 消除 call overhead + 暴露后续优化机会
  • 调用图自底向上—— 先 inline leaf function，再 inline 调用它们的
  • 启发式：函数小（≤ ~ 15 IR instr）/ caller 内的 constant arg / hot call site → inline
  • 典型收益: 5-20%。也是所有后续优化的前提—— 跨函数边界不能 inline 就不能 CSE / GVN

7.2 mem2reg —— 见 §3

把 alloca/load/store 提升到 SSA value。每个现代优化的前置 pass。

7.3 GVN (Global Value Numbering) —— 识别等价表达式

; Before GVN:
%a = add i32 %x, %y
%b = add i32 %x, %y      ; 等价表达式
%c = mul i32 %a, %b

; After GVN:
%a = add i32 %x, %y
%c = mul i32 %a, %a      ; %b 被消除，直接用 %a
  • 不仅看结构等价（同样 op + 同样 operand），还做 partial redundancy elimination
  • 结合 MemorySSA + AliasAnalysis 做 load elimination（"已经 load 过的值不用重 load"）
  • 收益：典型 hot path 1-5%

7.4 LICM (Loop Invariant Code Motion)

; Before:
for (i = 0; i < n; i++) {
  x = a * b;          // invariant in loop
  arr[i] = x + i;
}

; After LICM:
x = a * b;            // hoisted out of loop
for (i = 0; i < n; i++) {
  arr[i] = x + i;
}
  • "循环不变量" 移出 loop —— 只算一次
  • 需要 LoopInfo（哪些 BB 在 loop 里）+ AliasAnalysis（move 不影响 memory state）
  • 收益：循环密集代码 10-30%

7.5 SROA (Scalar Replacement of Aggregates)

; Before SROA:
%pair = alloca { i32, i32 }
%p1 = getelementptr { i32, i32 }, ptr %pair, i32 0, i32 0
store i32 %x, ptr %p1
%p2 = getelementptr { i32, i32 }, ptr %pair, i32 0, i32 1
store i32 %y, ptr %p2

; After SROA:
; (struct 被拆，没有 alloca，每个 field 是独立 SSA value)
; %x 和 %y 直接用

把聚合（struct/array）拆成单独 scalar。配合 mem2reg 让 struct field 也能进 register。极重要 pass —— 没 SROA 现代 C++ 性能崩。

7.6 Loop Vectorizer —— SIMD 自动并行化

; Before:
for (i = 0; i < 1000; i++)
  c[i] = a[i] + b[i];

; After vectorize (AVX2):
for (i = 0; i < 1000; i += 8) {           // 8 个一组
  <8 x float> va = load <8 x float>, a+i
  <8 x float> vb = load <8 x float>, b+i
  <8 x float> vc = fadd va, vb
  store <8 x float> vc, c+i
}
  • 需要 iteration count 已知 / 无 dependency between iterations / 对齐 OK
  • noalias 让 vectorize 更激进（之前 alias 阻止 vectorize 的 loop 现在可以）
  • 收益：数据密集 loop 2-8×

7.7 InstCombine —— 局部模式重写

大量小规则把"显式" pattern 改成"更优" pattern：

%t = mul i32 %x, 2          ; → %t = shl i32 %x, 1
%t = mul i32 %x, 8          ; → %t = shl i32 %x, 3
%t = mul i32 %x, 7          ; → 也可能用 shift + sub
%t = and i32 %x, 0xffff     ; bit mask 优化
%t = sub i32 %x, %x         ; → 0
%t = add i32 %x, 0          ; → %x
%t = sext i8 %x to i32      ; sign-ext patterns

InstCombine 跑很多次（每个 transform pass 之后），持续清理 IR。累计收益不大但 ubiquitous。

⑧ 写自己的 Pass

动手写一个 LLVM pass 是真正理解 LLVM 的 fastest way。下面是完整的 "统计函数中的 call 指令数量" pass。

项目结构

my-llvm-pass/
├── CMakeLists.txt
├── MyPass.cpp
└── test.cpp

CMakeLists.txt

cmake_minimum_required(VERSION 3.20)
project(MyLLVMPass)

find_package(LLVM REQUIRED CONFIG)
include_directories(${LLVM_INCLUDE_DIRS})
add_definitions(${LLVM_DEFINITIONS})

add_library(MyPass SHARED MyPass.cpp)
target_compile_features(MyPass PRIVATE cxx_std_17)
target_link_libraries(MyPass PRIVATE
  LLVMCore LLVMSupport LLVMPasses)

MyPass.cpp

#include "llvm/IR/Function.h"
#include "llvm/IR/Instructions.h"
#include "llvm/IR/PassManager.h"
#include "llvm/Passes/PassBuilder.h"
#include "llvm/Passes/PassPlugin.h"
#include "llvm/Support/raw_ostream.h"

using namespace llvm;

// 1. 实现 pass
namespace {
struct CallCounterPass : public PassInfoMixin<CallCounterPass> {
  PreservedAnalyses run(Function &F, FunctionAnalysisManager &) {
    unsigned count = 0;
    for (BasicBlock &BB : F)
      for (Instruction &I : BB)
        if (isa<CallInst>(&I))
          ++count;

    errs() << "Function " << F.getName()
           << " has " << count << " call instructions\n";

    return PreservedAnalyses::all();  // 没修改 IR
  }
};
}  // namespace

// 2. 注册 pass to PassBuilder
extern "C" ::llvm::PassPluginLibraryInfo LLVM_ATTRIBUTE_WEAK
llvmGetPassPluginInfo() {
  return {
    LLVM_PLUGIN_API_VERSION, "CallCounter", "1.0",
    [](PassBuilder &PB) {
      PB.registerPipelineParsingCallback(
        [](StringRef Name, FunctionPassManager &FPM,
           ArrayRef<PassBuilder::PipelineElement>) {
          if (Name == "call-counter") {
            FPM.addPass(CallCounterPass());
            return true;
          }
          return false;
        });
    }
  };
}

编译 + 运行

# Build
cmake -B build -G Ninja
cmake --build build

# Run 通过 opt
opt -load-pass-plugin=./build/libMyPass.so \
    -passes="call-counter" \
    -disable-output \
    test.ll

# 输出:
# Function main has 3 call instructions
# Function helper has 1 call instructions

插桩（modifying IR）的 pass

上面是 read-only。如果要插桩（如在每个 call 前加 log）：

PreservedAnalyses run(Function &F, FunctionAnalysisManager &) {
  auto *M = F.getParent();
  auto &Ctx = F.getContext();

  // 声明 extern void __my_log(const char*);
  FunctionCallee LogFn = M->getOrInsertFunction(
    "__my_log",
    FunctionType::get(Type::getVoidTy(Ctx),
                       {PointerType::get(Ctx, 0)}, false));

  for (auto &BB : F)
    for (auto &I : BB)
      if (auto *CI = dyn_cast<CallInst>(&I)) {
        IRBuilder<> B(CI);
        Value *Name = B.CreateGlobalStringPtr(
          CI->getCalledFunction()->getName());
        B.CreateCall(LogFn, {Name});  // log 在 call 之前
      }

  return PreservedAnalyses::none();  // 修改了 IR
}

这就是 sanitizer / coverage tool / profiler 的核心机制—— 在 LLVM IR 上插桩。

⑨ 调试 Pass

看每个 pass 前后的 IR

# 看所有 pass 输出（巨多，建议 stderr 重定向）:
opt -O2 -print-after-all input.ll -o /dev/null 2> passes.txt

# 只看特定 pass 后:
opt -O2 -print-after=gvn input.ll -o /dev/null 2> gvn.txt

# 看 pass 之前的 IR:
opt -O2 -print-before=loop-vectorize input.ll -o /dev/null

# 时间：哪个 pass 慢:
opt -O2 -time-passes input.ll -o /dev/null 2> timing.txt

opt-bisect（二分定位 bug pass）

怀疑某个优化 pass 引入了 bug：

# 列所有 pass 的编号:
opt -O2 -opt-bisect-limit=-1 input.ll -o /dev/null

# 输出:
# BISECT: running pass (1) ForceFunctionAttrsPass on [module]
# BISECT: running pass (2) PassManager<Module> on [module]
# BISECT: running pass (3) Annotation2MetadataPass on [module]
# ...

# 二分：跑前 N 个 pass
opt -O2 -opt-bisect-limit=50 input.ll -o out.ll
# 编译 + 测：bug 还在？降 limit，bug 没了？升 limit。
# 直到锁定 bug pass。

FileCheck（pass 单测）

LLVM 自带 llvm-lit + FileCheck 测试 framework。你写一个 .ll，跑 pass 后用 pattern 检查输出：

; RUN: opt -passes=instcombine -S %s | FileCheck %s

define i32 @test(i32 %x) {
; CHECK-LABEL: @test
; CHECK-NEXT: %r = mul i32 %x, 4
; CHECK-NEXT: ret i32 %r
  %a = add i32 %x, %x
  %r = add i32 %a, %a
  ret i32 %r
}
; 这测的是 instcombine 把 (x+x)+(x+x) 改成 x*4

⑩ Contribute LLVM · 从 First Patch 到 Commit Access

LLVM 是世界上最大开源 C++ 项目之一（~10M LOC，1500+ active contributors）。Contribute 流程：

第 1 步：Clone + Build

git clone https://github.com/llvm/llvm-project.git
cd llvm-project
cmake -S llvm -B build -G Ninja \
  -DCMAKE_BUILD_TYPE=Release \
  -DLLVM_ENABLE_PROJECTS="clang;lld" \
  -DLLVM_TARGETS_TO_BUILD="X86;AArch64"
cmake --build build -j
# 首次 build 30-60 min on modern hardware

第 2 步：找 first issue

  • GitHub label "good first issue"—— 适合新人
  • typo / 文档—— 最简单的入门 patch
  • 测试 case 缺失—— 写一个 .ll regression test
  • InstCombine 缺失模式—— 加一个 simple peephole 优化

第 3 步：写 patch + 测试

  • 跟着 LLVM Style Guide（基于 Google C++ Style）
  • 每个 functional change 必须配 test case（llvm/test/ 下加 .ll + FileCheck）
  • 跑 ninja check-llvm 确保没 regress

第 4 步：Submit via GitHub PR

2024 起 LLVM 从 Phabricator 完全迁到 GitHub。流程：

  1. Fork llvm/llvm-project to your GitHub
  2. Push branch + open PR
  3. Reviewers ping 通常 1-3 天回复
  4. Review iterations（typical 2-5 轮）
  5. Approved 后 merge

Commit access

  • ~3-5 个 merged PRs 后可申请 commit access（直 push 权限）
  • Apply 给 community（mailing list / Discord）
  • Commit access ≠ 不需 review—— 仍需 PR + approve，只是 self-merge 可

  • LLVM IR 是类汇编 + 类型化 + SSA—— 跨语言 / 跨平台 / 跨用途的统一中间表示
  • SSA + phi node 是现代优化的基础——每个 use 单一 def 让 def-use 分析 trivial
  • mem2reg 把 alloca/load/store 提升到 SSA value——前置 pass，没它没现代优化
  • Opaque pointers (LLVM 16+) 让 pointer 不带 pointee type，类型信息在 instruction 上
  • Metadata + Attribute 提供优化 hint（TBAA / noalias / readonly），让 optimizer 更激进
  • NewPM（Pass Manager）默认，比 legacy 快 ~20% + 支持 nested pipelines
  • 7 个关键 pass: inline / mem2reg / GVN / LICM / SROA / loop vectorize / instcombine 提供 80% 优化收益
  • 写自己的 Pass ~80 行代码 + plugin 模式注册，用于 sanitizer / profiler / domain-specific 优化
  • opt-bisect 是 LLVM 调试 pass 的核武器——二分定位 bug pass
