2.3 选型决策树（背下来）

选 unique_ptr 当...

  • 资源独占（fd、连接、锁）
  • 工厂返回
  • 所有权可移动转移
  • PIMPL 实现

选 shared_ptr 当...

  • 多个所有者共享同一对象
  • 缓存 + 异步回调持有
  • 观察者注册中的"持有"端
  • 容器存任意可被多处引用的对象

选 weak_ptr 当...

  • 打破 shared_ptr 循环引用
  • 缓存："如果还活着就用，死了就重建"
  • 观察者注册中的"被观察"端
  • 异步回调里持有 this 但不延长生命周期
优先级铁律：能用 unique_ptr 绝不用 shared_ptr（零开销 + 语义更清晰）；能用栈对象绝不用智能指针。shared_ptr 只在真的需要共享所有权时用。

unique_ptr 完全剖析

模板签名与典型大小 · EBO 让"空 deleter 不占空间" · 自定义 deleter 的 3 种正确写法 · 数组特化 unique_ptr<T[]> · release / reset / swap —— 三…

3.1 模板签名与典型大小

template<class T, class Deleter = std::default_delete<T>>
class unique_ptr;
类型sizeof解释
std::unique_ptr<int>8（64-bit）仅一个原始指针；默认 deleter 是空类，EBO 后不占空间
std::unique_ptr<int, MyEmpty>8MyEmpty 是空类 → EBO
std::unique_ptr<int, void(*)(int*)>16函数指针 8 字节 → 直接放
std::unique_ptr<int, std::function<void(int*)>>40+std::function 内置 SBO（典型 32B）
第一个易错点：自定义 deleter 的类型是模板参数，会直接影响 sizeof(unique_ptr)。不要用 std::function 当 deleter（除非必须类型擦除），用 lambda 类型 / 空 struct 才能享受 EBO。

3.2 EBO 让"空 deleter 不占空间"

// libstdc++ unique_ptr 的近似布局
template<class T, class D>
class unique_ptr {
    // __tuple_impl 用偏特化 + EBO，空类 D 占 0 字节
    std::tuple<T*, D> ptr_;
public:
    T* get() const noexcept { return std::get<0>(ptr_); }
};

3.3 自定义 deleter 的 3 种正确写法

// ① lambda 类型（推荐，零开销）
auto fclose_del = [](FILE* f) { if (f) fclose(f); };
std::unique_ptr<FILE, decltype(fclose_del)> file{fopen("x", "r"), fclose_del};
// sizeof = 8（lambda 无捕获 → 空类 → EBO）

// ② 命名空类（最干净）
struct FileCloser {
    void operator()(FILE* f) const noexcept { if (f) fclose(f); }
};
std::unique_ptr<FILE, FileCloser> file{fopen("x", "r")};
// sizeof = 8，且可作为函数参数 / 返回值的稳定类型

// ③ 函数指针（次选，sizeof 翻倍）
std::unique_ptr<FILE, decltype(&fclose)> file{fopen("x", "r"), &fclose};
// sizeof = 16

3.4 数组特化 unique_ptr<T[]>

// 区别：用 delete[] 而非 delete，提供 operator[]
std::unique_ptr<int[]> arr{new int[100]};
arr[5] = 42;             // operator[] 可用
// arr.get() 仍可用，但 *arr 无法解引用（标准禁止）

// C++20 起：std::make_unique_for_overwrite<T[]> 不做值初始化（更快）
auto buf = std::make_unique_for_overwrite<char[]>(4096);
// 内容是 indeterminate，立刻自己写满

3.5 release / reset / swap —— 三件套语义

std::unique_ptr<Big> p{new Big{}};

T* raw = p.release();          // 放弃所有权，返回裸指针，p 变空
                              // 调用方必须自己 delete raw

p.reset(new Big{});           // 释放原对象 + 接管新对象
p.reset();                    // 释放原对象，p 变空

std::unique_ptr<Big> q;
p.swap(q);                    // 交换所有权（也可 std::swap(p, q)）

3.6 从 unique 到 shared 的"单向升级"

std::unique_ptr<Big> u{new Big{}};
std::shared_ptr<Big> s = std::move(u);        // ✅ 隐式转换 + 移动
// 反向：shared → unique 不允许（shared 是共享，无法单独转移）
设计模式：工厂函数返回 unique_ptr，调用方按需升级成 shared。这样默认零开销，需要共享时才付控制块的代价。

3.7 unique_ptr 与 PIMPL

// 头文件 widget.h
class Widget {
    struct Impl;                       // 前向声明
    std::unique_ptr<Impl> pImpl;
public:
    Widget();
    ~Widget();                          // 必须在 .cpp 定义！
    Widget(Widget&&) noexcept;
    Widget& operator=(Widget&&) noexcept;
    void use();
};

// widget.cpp
struct Widget::Impl { /* 完整定义 */ };
Widget::Widget() : pImpl(std::make_unique<Impl>()) {}
Widget::~Widget() = default;                // 这里 Impl 才完整，析构能编译
Widget::Widget(Widget&&) noexcept = default;
Widget& Widget::operator=(Widget&&) noexcept = default;
经典坑：用 unique_ptr 做 PIMPL 时，析构 / 移动函数必须在 .cpp 定义（即使是 = default）。否则在头文件实例化时 Impl 还不完整，default_delete<Impl> 报错 "deleting incomplete type"。

4.1 控制块的 4 个核心字段

4.2 控制块字段解释

4.3 为什么 shared_ptr<T> 不像 unique_ptr 那样把 deleter 放类型里？

这是 shared_ptr 设计的一个关键决定：

  • shared_ptr 要支持动态多态共享：shared_ptr<Base> sp = make_shared<Derived>()。如果 deleter 在类型里，Base* 和 Derived* 的 shared_ptr 类型就不一样，无法转换
  • shared_ptr 要能从裸指针 + 任意 deleter 构造：shared_ptr<FILE> p{fopen("x","r"), fclose}; —— 但 shared_ptr<FILE> 类型不含 deleter
  • 所以 deleter 必须类型擦除存到控制块（运行时虚调用）
这就是 shared_ptr 每次调用 deleter 有虚函数开销的根源——也是它"昂贵"的原因之一。unique_ptr 没这个开销，因为 deleter 类型已知。

4.4 控制块的 3 种实现策略

策略触发场景内存布局
① 分离式shared_ptr<T>(new T) / shared_ptr<T>(p, deleter)对象一次 alloc，控制块另一次 alloc（共 2 次）
② 一体化std::make_shared<T>(args...)控制块 + 对象 一次 alloc 在一起
③ enable_shared_from_this 内嵌类继承 enable_shared_from_this对象内嵌一个 weak_ptr 指向自己的控制块

4.5 看真实 libstdc++ 源码（简化）

// _Sp_counted_base —— 控制块基类，所有变种共享
class _Sp_counted_base {
    _Atomic_word  _M_use_count = 1;       // 32-bit atomic（不是 long!）
    _Atomic_word  _M_weak_count = 1;      // 初值 1 = "use_count != 0 的虚拟引用"
public:
    virtual void _M_dispose() = 0;      // 析构 T
    virtual void _M_destroy() = 0;      // 释放控制块本身
};

// 几个派生类（每个对应一种构造场景）
template<class T, class D, class A>
class _Sp_counted_deleter : public _Sp_counted_base { /* T*, D, A */ };

template<class T, class A>
class _Sp_counted_ptr_inplace : public _Sp_counted_base {
    alignas(T) char _M_storage[sizeof(T)];   // ★ make_shared 用的，对象内嵌！
    A _M_alloc;
};

4.6 weak_count 的初值"1"的玄机

注意 libstdc++ _M_weak_count 初值是 1，不是 0。这个 1 代表"use_count != 0 这一事实本身"。规则：

  • use_count 降到 0 → 析构 T，同时 weak_count -= 1
  • weak_count 降到 0 → 释放控制块本身
// 简化的引用计数变化
// 起始: use=1, weak=1
shared_ptr<T> sp1{new T};        // use=1, weak=1
{
    shared_ptr<T> sp2 = sp1;          // use=2, weak=1
    weak_ptr<T> wp = sp1;             // use=2, weak=2
}                                     // sp2 走, wp 走: use=1, weak=1

sp1.reset();                       // use=0 → 析构 T, weak-=1 → weak=0 → 释放控制块
这个设计避免了一个看似没用但很烦的边界：当只剩 weak_ptr 时（use=0），对象已析构但控制块还活着，weak_ptr 的 lock() / expired() 才能正确判断。

make_shared vs shared_ptr(new T)

两种构造方式的内存布局对比 · 性能基准（典型机器，对象 32B） · make_shared 的"陷阱" —— 内存不立刻还给 OS · 异常安全 —— make_shared 的另一胜场 · make_shared …

性能与陷阱

5.1 两种构造方式的内存布局对比

① shared_ptr<T>{new T} — 2 次分配

// 步骤:
T* p = new T;             // alloc 1: T 对象
shared_ptr<T> sp(p);     // alloc 2: 控制块
  • 2 次 malloc，2 次 free
  • 对象与控制块地址分散 → cache miss
  • 对象 32B 时控制块也占 32B+，开销 50-100%
  • 异常风险：见 5.4

② make_shared<T>(args) — 1 次分配

// 一次性 alloc 一个大块
// [控制块字段 | T 对象]
shared_ptr<T> sp = make_shared<T>();
  • 1 次 malloc，对象与控制块紧邻
  • cache 友好（同一 line 多半装得下）
  • 性能可快 1.5-2×
  • 不再有异常安全坑

5.2 性能基准（典型机器，对象 32B）

构造方式平均耗时 / 次分配次数
shared_ptr<T>(new T)~120 ns2
make_shared<T>()~50 ns1
allocate_shared<T>(pmr_alloc)~20 ns（命中 pool）1（pool）

5.3 make_shared 的"陷阱" —— 内存不立刻还给 OS

因为对象 + 控制块共享一次分配，所以：

  • shared_ptr 全部失效（use_count = 0）→ 对象析构
  • 但只要还有 weak_ptr 存在 → 整个内存块还在
  • 对象的内存"延后到最后一个 weak_ptr 失效才释放"
典型场景：缓存大对象 + weak_ptr 观察。若用 make_shared 创建，缓存清空后 weak_ptr 还活着 → 大对象的内存仍占着（虽然对象已析构）。对大对象（> KB）或大量 weak_ptr 持有的场景，宁可用 shared_ptr<T>(new T) 双分配，确保对象死时内存立即归还。

5.4 异常安全 —— make_shared 的另一胜场

// ❌ 容易写出有异常风险的代码
f(std::shared_ptr<A>(new A), g());
// 参数求值顺序未规定（C++14 前），可能：
//   1. new A         OK
//   2. g() 抛异常    ★
//   3. shared_ptr 还没构造 → A 泄漏！

// ✅ 用 make_shared 一行原子完成
f(std::make_shared<A>(), g());
C++17 起修正了求值顺序规则，上面的 bug 不再发生，但 make_shared 仍然更好（一次分配 + cache 友好）。

5.5 make_shared 不能用的场景

  • 需要自定义 deleter：make_shared 强制用 operator delete，无法定制 → 用 shared_ptr(p, deleter)
  • 类的构造函数是 private/protected：make_shared 在自己作用域里调构造，无法访问 → 用 shared_ptr(new T) 或友元 + 静态工厂
  • 需要"对象死后立刻释放内存"的大对象场景（见 5.3）

5.6 自定义 allocator —— allocate_shared

// 用 PMR allocator 共享 monotonic pool
std::pmr::monotonic_buffer_resource pool{4096};
std::pmr::polymorphic_allocator<T> pa{&pool};

auto sp = std::allocate_shared<T>(pa, args...);
// 控制块 + T 对象都从 pool 分配
// pool 析构时一次性释放（无需 individual delete）

低延时服务的标准做法：每个请求用 monotonic pool，allocate_shared 创建所有临时对象，请求结束 pool 整体清空。

weak_ptr 与循环引用

6.1 循环引用导致的内存泄漏

struct Node {
    std::shared_ptr<Node> next;     // ❌ 错！
    std::shared_ptr<Node> prev;     // ❌ 错！
};

auto a = std::make_shared<Node>();
auto b = std::make_shared<Node>();
a->next = b;   // b.use_count = 2
b->prev = a;   // a.use_count = 2

// 离开作用域 a/b 各自 -1, 都还是 1，永远不归 0 → 泄漏

6.2 用 weak_ptr 打破循环

struct Node {
    std::shared_ptr<Node> next;     // 强引用
    std::weak_ptr<Node>   prev;     // ✅ 弱引用
};
设计规则：图 / 树 / 双向链表中，"父→子"用 shared_ptr，"子→父"用 weak_ptr。或更通用的方向："主要所有权"用 shared，"反向 / 观察"用 weak。

6.3 weak_ptr 的核心 API

std::weak_ptr<T> wp = sp;        // 从 shared 构造

if (auto sp2 = wp.lock()) {     // 尝试提升 — 原子操作
    use(*sp2);                   // 对象保证活着
}
// wp.lock() 是 atomic CAS，比每次拿 raw 指针多一次 atomic

bool dead = wp.expired();        // 看 use_count == 0；竞态不安全
                                  // 真要用还是 lock() 判返回值

6.4 lock() 的内部原子操作

// 简化的 lock() 实现 — 关键是 use_count 的原子 CAS
std::shared_ptr<T> lock() const {
    long uc = ctrl_->use_count.load(std::memory_order_relaxed);
    do {
        if (uc == 0) return {};                  // 对象已死
    } while (!ctrl_->use_count.compare_exchange_weak(
                uc, uc + 1,
                std::memory_order_acq_rel,
                std::memory_order_relaxed));
    // 成功 +1, 构造 shared_ptr 返回
}

6.5 weak_ptr 的典型用法 — 观察者模式

class EventBus {
    std::vector<std::weak_ptr<Observer>> observers_;   // ✅ 弱持
public:
    void register_obs(std::shared_ptr<Observer> o) {
        observers_.push_back(o);
    }

    void fire(const Event& e) {
        auto it = observers_.begin();
        while (it != observers_.end()) {
            if (auto sp = it->lock()) {
                sp->on_event(e);
                ++it;
            } else {
                it = observers_.erase(it);          // 自动清理死引用
            }
        }
    }
};

6.6 weak_ptr 与缓存模式

class ImageCache {
    std::unordered_map<std::string, std::weak_ptr<Image>> cache_;
public:
    std::shared_ptr<Image> get(const std::string& key) {
        auto it = cache_.find(key);
        if (it != cache_.end()) {
            if (auto sp = it->second.lock()) {
                return sp;                              // 命中 — 还活着
            }
            // 死了 — 顺手清理
            cache_.erase(it);
        }
        auto sp = std::make_shared<Image>(load_from_disk(key));
        cache_[key] = sp;                                // 存 weak_ptr
        return sp;
    }
};

这就是"如果还有人用就复用，没人用就让它走"的缓存模式 —— 缓存不延长对象生命周期。

Aliasing Constructor

签名与语义 · 经典用例：指向 vector 内某个元素，但持有 vector 所有权 · 经典用例：返回内部子对象…

指向成员但持有整体所有权

这是 shared_ptr 最有用却最少人知道的高级特性。

7.1 签名与语义

template<class Y, class U>
shared_ptr(const shared_ptr<U>& other, T* ptr) noexcept;
// ↑ 我和 other 共享同一个控制块，但我的 .get() 返回 ptr，与 other 的对象指针无关

7.2 经典用例：指向 vector 内某个元素，但持有 vector 所有权

auto vec = std::make_shared<std::vector<int>>(std::vector<int>{1,2,3,4,5});

// 我想得到 "指向 vec->at(2)" 但 vec 在我手上不会被销毁
std::shared_ptr<int> elem3{vec, &vec->at(2)};

// 此时:
//   elem3.get() == &vec->at(2)
//   elem3 与 vec 共享控制块 → use_count == 2
//   vec 离开作用域时不释放 vector（因为 elem3 还持有控制块）
//   elem3 也走才真正释放整个 vector

7.3 经典用例：返回内部子对象，但延长父对象生命周期

class Parent {
    Child child_;
public:
    static std::shared_ptr<Child> getChild(std::shared_ptr<Parent> self) {
        // 返回的 shared_ptr 指向 child_，但持有 Parent 整体所有权
        return std::shared_ptr<Child>{self, &self->child_};
    }
};

std::shared_ptr<Child> c;
{
    auto p = std::make_shared<Parent>();
    c = Parent::getChild(p);
}                                // p 走了, 但 Parent 不死 — c 还持有控制块
c->do_stuff();                  // ✅ Child 还活着, 因为 Parent 整体还活着

7.4 C++17 与 aliasing 的 move 版本

// C++17 起也支持 rvalue 形式
template<class Y, class U>
shared_ptr(shared_ptr<U>&& other, T* ptr) noexcept;
// 把控制块 "偷" 过来，少一次原子 +1 / -1

7.5 真实使用：CRTP 风格的子视图、Protobuf 子消息

// Protobuf 解析后获得子消息的 shared 视图
std::shared_ptr<Message> msg = std::make_shared<Message>();
msg->ParseFromString(...);

std::shared_ptr<Header> h{msg, msg->mutable_header()};
// h 单独使用时也安全 — 整个 msg 不会被释放
queue_to_async_worker(h);
Aliasing constructor 的本质：把"引用计数"与"当前指针"解耦。让一个 shared_ptr 可以"指向 A，但保活 B"。99% 的开发者一辈子用不到，但 1% 的场景下没有它就只能复制整对象。

enable_shared_from_this

解决什么问题 · 用法 · 内部机制 —— 内嵌一个 weak_this · 关键问题：weak_this 何时被赋值？ · C++17 加了 weak_from_this() · …

从对象里拿自己的 shared_ptr

8.1 解决什么问题

class Session {
public:
    void async_read() {
        socket.async_recv(buf, [this](Error e) {
            // ★ 如果 Session 已被销毁, this 是悬空指针!
            handle(e);
        });
    }
};

异步回调里需要"把自己活到回调执行"，但 this 是裸指针不延长生命周期。如果我们能拿到自己的 shared_ptr 捕获到 lambda 就完美。

8.2 用法

class Session : public std::enable_shared_from_this<Session> {
public:
    void async_read() {
        auto self = shared_from_this();         // ★ 拿到自己的 shared_ptr
        socket.async_recv(buf, [this, self](Error e) {
            handle(e);                            // 安全 — self 保活
        });
    }
};

// 必须通过 shared_ptr 创建!
auto s = std::make_shared<Session>();          // ✅
s->async_read();

// ❌ 栈对象 / unique_ptr → shared_from_this() 抛 bad_weak_ptr
Session s2;
s2.async_read();   // 运行时崩

8.3 内部机制 —— 内嵌一个 weak_this

// 简化的 enable_shared_from_this 实现
template<class T>
class enable_shared_from_this {
protected:
    mutable std::weak_ptr<T> weak_this_;            // ★ 关键字段
public:
    std::shared_ptr<T> shared_from_this() {
        std::shared_ptr<T> sp = weak_this_.lock();
        if (!sp) throw std::bad_weak_ptr{};
        return sp;
    }
};

8.4 关键问题：weak_this 何时被赋值？

shared_ptr 的构造函数会检测 T 是否继承自 enable_shared_from_this：

// shared_ptr 构造（简化）
template<class T>
shared_ptr<T>::shared_ptr(T* ptr) {
    ctrl_ = new control_block{...};
    ptr_  = ptr;
    // SFINAE 检测：T 继承自 enable_shared_from_this 吗？
    if constexpr (std::is_base_of_v<enable_shared_from_this<T>, T>) {
        ptr->weak_this_ = *this;                // ★ 注入！
    }
}
因此：① 必须通过 shared_ptr 创建对象，weak_this 才被填充；② 构造函数内部不能调 shared_from_this（此时 weak_this 还没被注入）；③ 一个对象不能同时归两个独立 shared_ptr 管理（两个控制块 → weak_this 会被覆盖）。

8.5 C++17 加了 weak_from_this()

auto wp = obj.weak_from_this();   // 直接拿 weak_ptr, 不抛异常
// 如果对象未被 shared_ptr 管理, 返回 expired 的 weak_ptr

8.6 真实使用范式 —— Asio / Beast 的 strand-based session

class Connection : public std::enable_shared_from_this<Connection> {
    asio::ip::tcp::socket socket_;
    std::array<char, 8192> buf_;
public:
    void start() { do_read(); }

    void do_read() {
        socket_.async_read_some(asio::buffer(buf_),
            [self = shared_from_this()](auto ec, auto n) {  // 捕获 self 保活
                if (!ec) {
                    self->do_process(self->buf_.data(), n);
                    self->do_read();                          // 链式异步
                }
            });
    }
};
每次启动一段异步链都用 auto self = shared_from_this(); 捕获，确保 callback 跑完前 Session 不死。Boost.Asio / Beast 的所有 examples 都是这个模式。

std::atomic<std::shared_ptr<T>> (C++20)

问题：shared_ptr 变量本身的并发修改是 race · C++11/17 的临时方案 — 自由函数 atomic_load / atomic_store · C++20 正式入标：std::atomic<std:…

9.1 问题：shared_ptr 变量本身的并发修改是 race

std::shared_ptr<Config> g_cfg = std::make_shared<Config>();

// 线程 A — 读取
auto c = g_cfg;                // 读 g_cfg 的指针
use(*c);

// 线程 B — 替换 (热更新场景)
g_cfg = std::make_shared<Config>(load_new());   // 写 g_cfg 的指针

// ★ 上面两个线程同时操作 g_cfg 是 DATA RACE — UB!
// 控制块的 use_count 是原子的, 但 shared_ptr 对象的 ptr_/ctrl_ 两个字段不是原子整体写

9.2 C++11/17 的临时方案 — 自由函数 atomic_load / atomic_store

// C++11 ~ C++17, C++20 弃用, C++26 移除
std::shared_ptr<Config> c = std::atomic_load(&g_cfg);
std::atomic_store(&g_cfg, new_value);

// 库实现多用全局锁池 → 性能差

9.3 C++20 正式入标：std::atomic<std::shared_ptr<T>>

#include <memory>

std::atomic<std::shared_ptr<Config>> g_cfg;            // ★ 现在 OK

// 读 — 原子拿当前 shared_ptr
std::shared_ptr<Config> c = g_cfg.load();
use(*c);

// 写 — 原子替换
g_cfg.store(std::make_shared<Config>(load_new()));

// 也支持 CAS
std::shared_ptr<Config> expected = g_cfg.load();
std::shared_ptr<Config> new_cfg  = std::make_shared<Config>(tweak(*expected));
g_cfg.compare_exchange_strong(expected, new_cfg);

9.4 实现内部 — 不是真的"无锁"

注意 std::atomic<std::shared_ptr<T>> 的 is_always_lock_free 不是 true。主流实现是：

  • libstdc++ 早期版本：内部用 mutex 池，每次 load/store 都过锁
  • 新版本 / libc++：内部用 "split lock" 或 hazard pointer 类技术，热路径接近无锁但仍有原子 spin
  • Folly atomic_shared_ptr：基于"tag bits + 内部 hazard pointer" 实现真正的 lock-free 读

9.5 性能基准

选型：① 简单 / 不热路径 → std::atomic<std::shared_ptr>；② 读极重 / 热路径 → folly 或自己用 hazard_pointer / RCU；③ 读频次远高于写 → 改成 RCU 模式更香。

多线程下的 shared_ptr 真相

三档"安全"

"shared_ptr 线程安全吗？"这是 C++ 面试最常被答错的题。正确答案：分情况。

10.1 三档安全等级

场景线程安全原因
① 不同 shared_ptr 实例指向同对象，多线程 各自读 / 拷贝 / 析构✅ 安全每个 shared_ptr 对自身字段有独立读写；控制块原子计数
② 多线程同时读同一 shared_ptr 实例（仅 const 操作）✅ 安全读没修改 ptr_ / ctrl_
③ 多线程并发写同一 shared_ptr 实例（=, reset, swap）❌ DATA RACEshared_ptr 的两个指针字段不是原子整体写
多线程修改 shared_ptr 指向的对象❌ 用户责任shared_ptr 不管对象本身的并发

10.2 反例与正例

// ❌ 反例: 两个线程并发写同一个 sp 变量
std::shared_ptr<T> sp = std::make_shared<T>();
std::thread([&]{ sp = std::make_shared<T>(); }).detach();
std::thread([&]{ sp.reset(); }).detach();
// 不同线程对 sp 的 ptr_/ctrl_ 字段非原子写 → UB

// ✅ 正例 1: 每线程拷贝，操作自己的副本
auto sp_copy = sp;                       // 单线程拷贝 → ref+1，安全
std::thread([sp_copy]{ sp_copy->use(); }).detach();

// ✅ 正例 2: 用 atomic shared_ptr (C++20)
std::atomic<std::shared_ptr<T>> asp;
asp.store(std::make_shared<T>());
std::shared_ptr<T> cur = asp.load();

10.3 引用计数的内存序

// libstdc++ 引用计数操作的实际 memory_order
void add_ref() {
    use_count_.fetch_add(1, std::memory_order_relaxed);
    // 增加不需要同步 — 我已经有引用了, 不会变 0
}

void release() {
    if (use_count_.fetch_sub(1, std::memory_order_release) == 1) {
        std::atomic_thread_fence(std::memory_order_acquire);   // ★
        dispose();                                      // 我是最后一个 — 析构
    }
}
面试金句："shared_ptr 引用计数 +1 用 relaxed（增加不需要同步），-1 用 release（让其他线程的写操作 happens-before 析构）；最后一个 release 后加 acquire fence 确保读到所有线程的最终修改。" 这就是内存模型在标准库里的实战应用。

10.4 跨 .so 边界的陷阱

shared_ptr 跨 .so 传递必须保证两边用同一份控制块定义：

  • 两个 so 都动态链 libstdc++（默认）→ 共享一份 _Sp_counted_base 定义 → OK
  • 有 so 静态链 libstdc++ → 各自有控制块基类副本 → use_count 错位 → 崩
  • 跨 libstdc++ / libc++ 边界 → ABI 完全不兼容 → 崩
  • 跨 Windows DLL boundary → 需统一 MSVC 版本 + Runtime（/MD）
设计准则：公共库 API 想暴露智能指针时，三选一：① 强制所有依赖方用同一 toolchain；② 在 .so 边界改成 C ABI（POD）传递；③ 用 boost::intrusive_ptr 让计数嵌入对象本身（见 §11），避免控制块 ABI 问题。

11.1 intrusive_ptr 是什么

"intrusive" 引用计数 = 把计数放对象自身里，而不是单独的控制块。代价：对象类型必须配合（提供 add_ref / release）。

struct MyObj {
    std::atomic<int> ref_count{0};
    // 业务字段...

    friend void intrusive_ptr_add_ref(MyObj* p) {
        p->ref_count.fetch_add(1, std::memory_order_relaxed);
    }
    friend void intrusive_ptr_release(MyObj* p) {
        if (p->ref_count.fetch_sub(1, std::memory_order_acq_rel) == 1) {
            delete p;
        }
    }
};

// 用 boost::intrusive_ptr
boost::intrusive_ptr<MyObj> p{new MyObj};

class Subject {

struct PacketBuffer {
    std::atomic<int> refs{0};
    char data[2048];

    friend void intrusive_ptr_add_ref(PacketBuffer* p) {
        p->refs.fetch_add(1, std::memory_order_relaxed);
    }
    friend void intrusive_ptr_release(PacketBuffer* p) {
        if (p->refs.fetch_sub(1, std::memory_order_acq_rel) == 1) {
            return_to_pool(p);              // 不 delete, 回池
        }
    }
};
// 比 shared_ptr 节省 50% 内存 (无控制块) + 零分配 (从池取)
// DPDK / 行情广播服务的常用模式

易错点速查（15 个常见 BUG）

shared_ptr<Base> sp{new Derived}

  1. 同一裸指针给两个 shared_ptr 接管 → 两个控制块 → 双重释放 
    T* p = new T;
shared_ptr<T> sp1(p);     // 控制块 A
shared_ptr<T> sp2(p);     // ❌ 控制块 B, 各自要 delete p → 崩
// ✅ 用 sp2(sp1) 或 make_shared
  2. 用 raw this 创建 shared_ptr（不继承 enable_shared_from_this）→ 双控制块 
    void Foo::register() {
    bus.add(shared_ptr<Foo>(this));   // ❌ 新控制块, 与外面的 shared_ptr 冲突
}
// ✅ Foo 继承 enable_shared_from_this, 用 shared_from_this()
  3. 构造函数内调 shared_from_this → bad_weak_ptr 
    Foo::Foo() {
    bus.add(shared_from_this());     // ❌ weak_this 还未被注入
}
// ✅ 用工厂函数: shared_ptr<Foo> make_foo() { auto p = make_shared<Foo>(); bus.add(p); return p; }
  4. 循环引用（§6.1）→ 内存泄漏
  5. shared_ptr 跨线程并发 = / reset → race（§10.1）
  6. 函数参数用 shared_ptr<T> const& vs by value 的不当选择 
    // 一般规则:
//   - 函数会"接管"或保存 → by value
//   - 函数仅观察, 调用方手上有所有权 → 传裸指针或 T&
//   - 真要共享读 → const shared_ptr<T>&
void f(shared_ptr<T> sp);          // ✅ 接管
void g(const T& t);                // ✅ 只看不持有
  7. weak_ptr.expired() 后立刻 lock() — 中间窗口期对象可能死了/活了 
    if (!wp.expired()) { auto sp = wp.lock(); ... }   // ❌ TOCTOU
// ✅ 直接 lock 然后判返回值
if (auto sp = wp.lock()) ...;
  8. 把 unique_ptr 当 shared 用：返回类型应该是 unique_ptr 的反而用 shared_ptr（一开始就浪费控制块开销）
  9. PIMPL 忘记在 .cpp 定义析构 / 移动（§3.7）
  10. shared_ptr 持有数组但用 delete 而非 delete[] 
    shared_ptr<int> p{new int[10]};        // ❌ 默认 deleter 用 delete (UB)
shared_ptr<int[]> p2{new int[10]};     // ✅ C++17, 用 delete[]
  11. 从基类 shared_ptr 析构派生类对象但基类析构未 virtual 
    shared_ptr<Base> sp{new Derived};      // ✅ shared_ptr 记住了原始 Derived 类型, 安全
unique_ptr<Base> up{new Derived};      // ❌ 必须 ~Base() virtual, 否则 slicing

    这是 shared_ptr 比 unique_ptr 强的少数地方 — 因 deleter 类型擦除记住了真实类型。

  12. 过度使用 shared_ptr 当"GC 替代品" → 性能下降 + 循环引用风险
  13. 跨 .so 边界传 shared_ptr而双方 ABI 不一致 → 崩（§10.4）
  14. 把 shared_ptr 放进容器后改控制块状态：vector 扩容可能拷贝大量 shared_ptr → 原子计数惊群
  15. 用 make_shared 创建大对象 + weak_ptr 持有 → 内存无法及时归还（§5.3）

面试题集（20 道分层）

基础（必答） · 中级 · 高级（体现深度） · 速答示范（第 8 题：shared_ptr 拷贝的 memory_order）

点击每道题展开参考答案（先自测，再对照）。

14.1 基础（必答）

基础unique_ptr / shared_ptr / weak_ptr 各自语义？
unique
独占所有权，不可拷贝、可移动，零开销（等价裸指针）。
shared
共享所有权，原子引用计数，最后一个析构时释放对象。
weak
不增加强引用的观察者，用于打破循环；通过 lock() 临时提升为 shared_ptr。
基础sizeof(unique_ptr) 多少？sizeof(shared_ptr) 多少？
unique
默认 1 个指针大小（8B），无状态 deleter 经 EBO 不占空间；带状态 deleter 会变大。
shared
2 个指针（16B）：对象指针 + 控制块指针。
基础make_shared 比 shared_ptr(new T) 好在哪？至少 3 点
① 单次分配
控制块 + 对象合并成一块，减少一次 malloc + 提升缓存局部性。
② 异常安全
避免 f(shared_ptr(new T), g()) 求值顺序导致的泄漏。
③ 简洁
不写裸 new，代码更短。
代价
只要还有 weak_ptr 存活，整块内存（含大对象空间）不释放。
基础shared_ptr 的引用计数线程安全吗？对象本身呢？
计数
控制块引用计数是原子的、线程安全：多个 shared_ptr 各自拷贝 / 析构安全。
对象
不安全：并发改同一对象需自行加锁；多线程读写同一个 shared_ptr 实例需 atomic<shared_ptr>。
基础循环引用怎么产生？怎么解？
产生
两对象互相用 shared_ptr 持有 → 计数永不归零 → 内存泄漏。
解
把其中一方改 weak_ptr（典型：父→子用 shared，子→父用 weak）。

14.2 中级

中级控制块里有哪些字段？为什么 weak_count 初值是 1？
字段
strong count、weak count、删除器、分配器，以及（make_shared 时）内联的对象存储。
weak=1
所有 strong 引用合起来算"一个 weak 持有者"。strong 归零时减掉这个 1，使控制块在最后一个 weak 也消失时才释放 —— 保证 expired() 等查询期间控制块仍有效。
中级enable_shared_from_this 内部怎么工作？为什么必须先用 shared_ptr 创建对象？
机制
基类内藏一个 weak_ptr；shared_ptr 构造时检测到该基类并回填这个 weak。shared_from_this() 即 weak.lock()。
前提
若对象不是被 shared_ptr 管理（栈对象 / 裸 new），weak 未初始化 → C++17 起抛 bad_weak_ptr（旧标准 UB）。
中级shared_ptr 拷贝时引用计数 +1 用什么 memory_order？为什么？
+1
relaxed。增加引用不改变对象可见性 —— 调用前我已持有合法引用，仅让计数变大，无需 happens-before。
-1
acq_rel（或 release + 末次 acquire fence），保证其他线程对对象的所有写 happens-before 析构。详见 §14.4。
中级weak_ptr.lock() 内部做了什么？
实现
原子检查 strong count 是否 > 0，是则用 CAS 把 strong+1 并返回 shared_ptr，否则返回空。本质是一个无锁 CAS 循环。
中级aliasing constructor 是什么？给一个真实使用场景
定义
shared_ptr<T>(other_sp, raw_ptr)：共享 other 的控制块（计数），但 get() 返回另一个指针。
场景
持有结构体某个成员的 shared_ptr，同时让整个结构体保持存活（计数挂在父对象上）。
中级shared_ptr<Base>{new Derived} 在 Base 析构非 virtual 时为何安全？
原因
shared_ptr 构造时把删除器类型擦除为 delete (Derived*) 并存进控制块，析构时调用记录的删除器删真实类型 Derived，不依赖 Base 虚析构。
对比
unique_ptr 无此保护 —— 多态删除仍需虚析构。
中级shared_ptr 和 std::function 都做类型擦除，开销分别在哪？
shared_ptr
擦除"如何删除"（删除器），开销在控制块一次分配 + 原子计数。
function
擦除"如何调用"，开销在可能的堆分配（超过 SBO 时）+ 一次间接调用。

14.3 高级（体现深度）

高级std::atomic<std::shared_ptr<T>> 内部是真无锁吗？
不保证
is_lock_free() 多为 false。多数实现用 spinlock / 锁分段保护控制块的原子换出 —— 逻辑上无锁语义，物理上常有锁。
高级make_shared 创建大对象 + weak_ptr 长期持有，会有什么问题？
问题
对象与控制块同一块内存；strong 归零对象析构，但只要 weak 还在，整块内存（含大对象空间）不释放 → 内存浪费。
对策
大对象 + 长 weak 改用 shared_ptr(new T) 分离分配。
高级shared_ptr 跨 .so 边界传递有什么 ABI 坑？怎么规避？
坑
不同编译器 / STL 版本控制块布局不同；删除器与分配器跨边界；可能跨不同堆。
规避
边界用 C 接口，或保证两侧同一套 ABI / 运行时；删除器在创建侧定义。
高级boost::intrusive_ptr 比 std::shared_ptr 强在哪？弱在哪？
强
计数嵌入对象内 → 单次分配、sizeof=1 指针、可从裸指针重建。
弱
侵入式需改对象、无内建 weak、需手写 add_ref / release。
高级设计一个"对象池版" shared_ptr（释放时回池，不 delete），怎么做？
做法
用自定义删除器归还池而非 delete： 
std::shared_ptr<T> p(pool.alloc(),
    [&pool](T* q){ pool.recycle(q); });
注意
池的生命周期必须 > 所有指针；删除器捕获的引用 / 指针不能悬空。
高级shared_ptr 持有 const T 与持有 mutable T 的语义区别？
const T
只读共享：不能改对象，但能 reset 指针本身。多读者天然安全。
mutable T
允许改对象，多写者需自行同步。
高级从一个 shared_ptr<Derived> 转成 shared_ptr<Base> 是什么成本？反过来呢？
上行
static_pointer_cast，共享控制块，原子 +1，O(1) 无检查。
下行
dynamic_pointer_cast，含 RTTI 运行时检查，更贵且可能失败返回空。
高级设计一个 LRU cache，key→value 是 weak_ptr，怎么避免 cache 自己延长对象生命周期？
做法
value 存 weak_ptr，get 时 lock()：命中且 lock 成功才返回；失败说明对象已被外部释放 → 当作 miss 并清理该 entry。
效果
cache 不持 strong 引用、不延长寿命；可配合定期清理过期 entry。

14.4 速答示范（第 8 题：shared_ptr 拷贝的 memory_order）

答：拷贝时 fetch_add(1, memory_order_relaxed)。原因是"增加引用"这个动作不会让对象的可见性变化 —— 我执行这条之前已经持有合法引用，增加只是让计数变大，无需任何 happens-before 关系。
析构时 fetch_sub(1, memory_order_release)，最后一个释放后插入 acquire fence，然后才析构对象。这是为了让其他线程在该对象上的所有写都 happens-before 析构 —— 否则可能读到不完整的状态。
这一题是检验"你是真懂 memory_order 还是背了 acquire/release"的金标准。

15.1 三大智能指针决策

"需要共享所有权?"
   ├─ 否 → unique_ptr
   └─ 是 → shared_ptr
          ├─ 有循环引用风险? → 一边用 weak_ptr 打破

          └─ 多线程并发重新指向? → atomic<shared_ptr> 或 RCU

15.2 性能开销速查

动作开销级
unique_ptr 任何操作~0（与裸指针等价）
shared_ptr 拷贝1 次原子 +1
shared_ptr 析构（不是最后一个）1 次原子 -1
shared_ptr 析构（最后一个）原子 -1 + 释放对象 + 释放控制块
weak_ptr 拷贝1 次原子 +1 (weak_count)
weak_ptr.lock()CAS 循环（典型成功 1 次）
make_shared1 次 malloc（控制块 + 对象一体）

15.3 自定义 deleter 写法

// unique_ptr — 类型参数，影响 sizeof
struct FCloser { void operator()(FILE* f) { fclose(f); } };
std::unique_ptr<FILE, FCloser> f{fopen(...)};

// shared_ptr — 运行时存控制块，不影响类型
std::shared_ptr<FILE> f{fopen(...), [](FILE* p){ fclose(p); }};

15.4 异步回调三模板

// ① 强持 — 回调期间对象一定活
io.post([self = shared_from_this()]() { self->work(); });

// ② 弱持 — 对象可能已死, 检查
io.post([weak = weak_from_this()]() {
    if (auto self = weak.lock()) self->work();
});

// ③ 裸 — 调用方保证对象活
io.post([this]() { work(); });
