0. 先解决一个困惑：我日常几乎不写 try/catch，这章跟我有关系吗？

太有关系了，而且恰恰相反——优秀的 C++ 代码里 try/catch 本来就很少。 "不写 try/catch" ≠ "不用管异常"。真相是：异常的处理集中在极少数地方，但异常的安全是每一行代码的事。

火警类比：异常机制就像大楼的火警系统

所以你"日常没怎么写 try/catch"是对的、健康的。问题在于——你可能错把"撤离时随手关门"也忽略了，也就是没意识到自己的代码要做到"异常安全"。这正是本章 §6（异常安全）和 §7（RAII）的核心。

那到底什么时候才"真的需要"写 try/catch？

你可能以为要 try/catch 的场景实际更好的做法
函数里 new 了内存 / 加了锁，怕异常时漏掉清理❌ 不用 try/catch。用 RAII（unique_ptr / lock_guard），异常时自动释放（见 §7）
调用一个可能失败的函数，想往上层传递错误❌ 不用 try/catch。啥都不写，让异常自己往上"冒泡"，路过你的函数时 RAII 自动收尾
中途失败了想撤销前面做的操作❌ 多数不用 try/catch。用 scope guard（见 §7.3）或 copy-and-swap 把它变成"事务"
✅ 这些场景才真正需要 try/catch为什么
main() / 线程函数 / 协程的最外层异常不能"逃出"线程，否则 std::terminate 杀进程。必须在这里兜底（见 §13.5）
RPC / HTTP / 消息处理器的入口一个请求挂了不能拖垮整个服务，要 catch 住转成错误响应
库 / 模块的对外 API 边界不让内部异常类型泄漏给调用方（尤其跨 DLL / 跨语言 C 接口）
你想"换一种错误表达"：异常 → 错误码 / std::expected热路径或 C 接口处，把异常 catch 下来转成返回值（见 §10、§11）

throw / try / catch 基础

三段式语法 · catch 匹配规则 · 重新抛出 · 嵌套异常 (C++11)

2.1 三段式语法

try {
    if (denom == 0)
        throw std::runtime_error{"division by zero"};
    result = num / denom;
}
catch (const std::runtime_error& e) {
    LOG_ERROR("runtime: {}", e.what());
}
catch (const std::exception& e) {
    LOG_ERROR("std: {}", e.what());
}
catch (...) {                       // 兜底，匹配任意
    LOG_ERROR("unknown");
}

2.2 catch 匹配规则

  • 从上到下找第一个匹配（不像虚函数找最具体）— 所以派生类放上面
  • 匹配方式：完全相同类型、引用类型、基类（含基类引用）、...
  • 匹配时不会做用户定义转换（不像函数调用）
  • catch (T&) 与 catch (T) 都可以；引用更好（避免拷贝、保留多态）

2.3 重新抛出

try {
    do_something();
} catch (const std::exception& e) {
    LOG_ERROR("caught: {}", e.what());
    throw;             // ✅ 重新抛出原异常 (保留类型)
    // 不能写 throw e; — 那会触发切片 (slicing)，丢失派生类信息
}

2.4 嵌套异常 (C++11)

try {
    parse_config();
} catch (...) {
    std::throw_with_nested(std::runtime_error{"failed to parse config"});
    // 把当前异常包装成 nested exception 抛出
}

try {
    init_app();
} catch (const std::exception& e) {
    print_exception(e, 0);
}

void print_exception(const std::exception& e, int depth) {
    std::cerr << std::string(depth*2, ' ') << e.what() << '\n';
    try { std::rethrow_if_nested(e); }
    catch (const std::exception& inner) { print_exception(inner, depth+1); }
}
三个语法陷阱：① catch (T) 按值会拷贝 → 优先 catch (const T&)；② throw e 会切片 → 优先 throw 单独使用；③ catch 顺序错误（基类放前面）会让派生 catch 永远不触发，编译器会警告。

栈展开 (Stack Unwinding) 机制详解 ⭐

概念 · 完整流程图 · 关键点：异常对象在哪里？ · ABI 层面：Itanium C++ ABI · throw + 栈展开的简化伪代码

4.1 概念

抛出异常后，运行时沿调用栈向上回退，直到找到匹配的 catch。回退过程中：

  • 每一帧上的所有局部对象按构造逆序析构（包括 catch 块外面的 try 块内对象）
  • RAII 类的析构函数因此被调用 → 资源释放
  • 这是 C++ 异常安全的基石
这就解释了 §1 的"为什么不用写 try/catch 也安全"。 异常会自己逐层往上"冒泡"找 catch；它路过的每一个函数（哪怕没有 try/catch），C++ 都会强制把这一层的局部对象按相反顺序析构掉。 只要你的资源是用 RAII 对象（lock_guard、unique_ptr…）持有的，析构=自动清理，于是你"什么都不写"也不会漏锁、漏内存。
生活类比：异常像一个人从高楼往下逃生，每路过一层，这层的灯会自动关、门会自动锁（=析构）——他本人不用操心，楼层的"自动装置"（RAII）替他做了。下面的动画就是这个过程。
反过来也成立：如果你没用 RAII（裸 new、手动 mu.lock()），栈展开时 C++ 只会析构"对象"，它不知道你那个裸指针/裸锁需要清理 → 异常一来就泄漏。这就是 §7 反复强调"一切资源都交给 RAII"的原因。

4.3 关键点：异常对象在哪里？

不是普通栈对象 — 否则栈展开就把它销毁了。标准只规定行为，实现常见做法：

  • libstdc++ / libc++：在线程局部的异常区（动态分配，紧凑布局）
  • MSVC：异常对象传递时拷贝构造到 catch 形参旁的临时区
  • 这就是为什么异常类的拷贝构造必须 noexcept — 否则在分配后再失败 → terminate

4.4 ABI 层面：Itanium C++ ABI

主流 Linux/macOS 实现遵循 Itanium C++ ABI。核心组件：

组件作用
__cxa_throw抛出异常 — 分配异常对象、调用 unwind
_Unwind_RaiseException (libgcc/libunwind)实际栈展开
.eh_frame 段每个函数的 unwind table — DWARF CFI 信息
.gcc_except_table (LSDA)每个函数的「哪段 PC 范围 → 哪个 catch / 析构」
__cxa_begin_catch / __cxa_end_catch进入/离开 catch 块时调用

4.5 throw + 栈展开的简化伪代码

// __cxa_throw 大致逻辑
void __cxa_throw(void* obj, std::type_info* tinfo, void (*dtor)(void*)) {
    // 1. 把异常对象包成 __cxa_exception (有 header + obj)
    auto* hdr = __cxa_allocate_exception(...);
    hdr->exceptionType = tinfo;
    hdr->exceptionDestructor = dtor;
    // 2. 启动 unwind
    _Unwind_RaiseException(&hdr->unwindHeader);
    // 3. 走到这里说明没 catch → terminate
    std::terminate();
}
"栈展开时如何调用析构?" — 答：每个函数的 .eh_frame + .gcc_except_table 描述了"该函数在哪段 PC 内有哪些局部对象需要析构、有哪些 catch 块"。运行时根据当前 PC 找到 LSDA 表，逐个调用析构。这不需要在每个函数入口/出口埋点 → "零开销" 的关键。
05

异常对象的生命周期

时间轴 · std::current_exception / rethrow_exception

5.1 时间轴

  1. throw 时构造（异常区域，由实现管理）
  2. 栈展开期间一直存活（不像普通局部对象会析构）
  3. catch 块开始：异常对象绑到 catch 形参（按引用零拷贝；按值拷贝）
  4. catch 块结束：异常对象析构
  5. 如果在 catch 块内 throw; 重抛 → 异常对象继续存活

5.2 std::current_exception / rethrow_exception

C++11 引入，让你能把异常对象保存下来跨线程或者延迟处理。

// 工作线程把异常带到主线程
std::exception_ptr g_eptr;

void worker() {
    try {
        do_work();
    } catch (...) {
        g_eptr = std::current_exception();    // 抓住异常
    }
}

void main_check() {
    if (g_eptr) {
        try { std::rethrow_exception(g_eptr); }
        catch (const std::exception& e) {
            LOG_ERROR("worker failed: {}", e.what());
        }
    }
}

// std::async / std::future 内部就是这种实现
auto fut = std::async([]{ throw std::runtime_error{"x"}; });
try { fut.get(); }                          // 重新抛出
catch (const std::exception& e) { /* ... */ }
std::exception_ptr 是引用计数的 → 拷贝便宜，多个线程可共享。是跨线程异常传递的标准方式。

异常安全 3 级保证 ⭐⭐⭐ (面试必答)

basic guarantee 示例 · strong guarantee 示例 — copy-and-swap 惯用法 · nothrow guarantee — swap / 移动构造的责任 · 经典案例对比

函数 / 操作可以提供四档异常安全级别（从弱到强）：

级别承诺典型例子
无保证 (no guarantee)抛异常后状态未知 — 避免裸 new + 未释放
basic guarantee 基本保证不泄漏资源 + 对象处于合法但未指定状态std::vector::push_back 普通情况
strong guarantee 强保证"事务性"：要么成功，要么状态完全不变copy-and-swap 实现的 operator=
nothrow guarantee 不抛保证承诺不抛任何异常 (noexcept)swap、所有析构、移动构造（应该）
一句话理解：异常安全说的不是"会不会抛异常"，而是"万一中途抛了异常，我的对象/数据会被留在什么状态"。 就像银行转账转一半断电了——钱有没有变少？账目还对不对得上？这才是"异常安全"关心的事。

用"网购下单"理解四个级别

6.2 strong guarantee 示例 — copy-and-swap 惯用法

class Buffer {
    char* data_;  size_t size_;
public:
    void swap(Buffer& o) noexcept {
        using std::swap;
        swap(data_, o.data_); swap(size_, o.size_);
    }

    // strong guarantee: copy-and-swap
    Buffer& operator=(Buffer rhs) {        // ① 按值收 — 拷贝在参数构造时发生
        swap(rhs);                              // ② 不抛 — 真正"提交"
        return *this;
        // ③ rhs 出作用域析构旧的资源
    }
};
// ✅ 如果 ① 抛异常 → *this 完全不变
// ✅ ② noexcept → 一旦走到这里，操作必然成功

6.3 nothrow guarantee — swap / 移动构造的责任

// 容器扩容期望元素的移动构造是 noexcept
struct A { std::string s; A(A&&) noexcept = default; };

// 否则: vector 在扩容时退化为拷贝构造（性能损失）

6.4 经典案例对比

实现一个简单的 operator+=：

// ❌ basic guarantee — 异常时部分修改
String& String::operator+=(const String& o) {
    auto new_size = size_ + o.size_;
    auto* new_data = new char[new_size + 1];   // 可能抛
    std::memcpy(new_data, data_, size_);
    std::memcpy(new_data + size_, o.data_, o.size_);
    new_data[new_size] = 0;
    delete[] data_;          // ← 走到这里才能保证 *this 一致
    data_ = new_data; size_ = new_size;
    return *this;
}
// ✅ strong guarantee — 因为 delete 在 new 之后, 所有抛点都在修改前
面试推荐答法：能给一段代码标注异常安全级别 + 解释如何把 basic 提升到 strong（用 copy-and-swap / 临时对象）。这是高级 C++ 工程师标志性能力。

RAII —— 异常安全的基石

RAII 三件套 · 经典对比 · 自定义 RAII guard（scope guard 模式）

Resource Acquisition Is Initialization：把资源 (内存/文件/锁) 的生命周期绑到对象的生命周期 → 析构函数自动释放 → 即便发生异常也安全。

7.1 RAII 三件套

资源RAII 类
动态内存std::unique_ptr / shared_ptr
互斥锁std::lock_guard / unique_lock / scoped_lock
文件std::fstream / std::ofstream
网络 fd / 自定义自己写包装类

7.2 经典对比

// ❌ 不用 RAII — 异常会泄漏锁 + 内存
void bad() {
    mu.lock();
    auto* p = new Foo;
    do_something(p);    // 抛异常 → mu 没解锁 + p 没 delete
    delete p;
    mu.unlock();
}

// ✅ RAII — 异常自动释放
void good() {
    std::lock_guard lk{mu};
    auto p = std::make_unique<Foo>();
    do_something(p.get());
    // 离开作用域 (正常或异常): p 析构 → delete; lk 析构 → unlock
}

7.3 自定义 RAII guard（scope guard 模式）

template<class F>
class ScopeExit {
    F f_;
    bool active_ = true;
public:
    explicit ScopeExit(F f) : f_(std::move(f)) {}
    ~ScopeExit() noexcept { if (active_) try { f_(); } catch (...) {} }
    void dismiss() noexcept { active_ = false; }
};
template<class F> ScopeExit(F) -> ScopeExit<F>;     // CTAD

// 使用
int fd = ::open(...);
ScopeExit guard{[fd]{ ::close(fd); }};

// 实现"事务"语义
void transfer(Account& from, Account& to, double amt) {
    from.withdraw(amt);
    ScopeExit rollback{[&]{ from.deposit(amt); }};   // 失败时回滚
    to.deposit(amt);                                  // 可能抛
    rollback.dismiss();                                // 成功 → 取消回滚
}
规则：所有需要在作用域结束前做某件事的场景（解锁、close、回滚、撤销操作）都用 RAII。手动 try/finally 是反模式（C++ 也没有 finally）。

noexcept 关键字

两个含义 · 哪些函数应该是 noexcept · 哪些场景受益 · 抛了 noexcept 函数会怎样 · noexcept 与函数指针

8.1 两个含义

  1. noexcept 说明符：函数承诺不抛异常 → 抛了直接 std::terminate()
  2. noexcept 运算符：编译期查询表达式是否声明为 noexcept
void f() noexcept;                  // 说明符: 我承诺不抛
void g() noexcept(true);            // 等价
void h() noexcept(false);           // 等价于不写 noexcept

template<class T>
void swap(T& a, T& b) noexcept(noexcept(T(std::move(a))) && noexcept(a = std::move(b)));
// 条件 noexcept: 仅当 T 的移动构造和移动赋值都 noexcept 时才 noexcept

8.2 哪些函数应该是 noexcept

  • 析构函数：默认就 noexcept (C++11 起)，几乎都应该不抛
  • 移动构造 / 移动赋值：强烈推荐 noexcept (STL 性能)
  • swap 函数：必须 noexcept (强异常安全的基石)
  • 简单 getter / setter：自然 noexcept
  • 基本类型操作：算术、比较

8.3 哪些场景受益

① 容器扩容：vector 调用 std::move_if_noexcept — noexcept 移动构造 → 用移动；否则用拷贝。

为什么 vector 扩容这么"纠结"？一句话：它怕"搬一半翻车"。
vector 满了要扩容时，会申请一块更大的新内存，再把老元素一个个"搬"过去，最后释放旧内存。 搬家有两种搬法：移动（快，直接把内脏掏过去）和拷贝（慢，照着做一份新的，旧的留着）。 vector 当然想用"移动"，但它必须先保证strong guarantee（搬家失败也不能丢东西）。
问题来了：如果移动构造可能抛异常，那搬到第 5 个元素时抛了——前 4 个已经被"掏空"了，旧内存里是残骸，新内存也没搬完，两边都是废的，数据全毁，没法回滚。 所以 vector 的策略是：只有当移动构造是 noexcept（保证搬家不会翻车）时，才敢用移动；否则宁可慢，用拷贝（拷贝时旧的一直完好，失败了直接扔掉新内存即可回滚）。
▶ 交互式演示：noexcept 如何决定 vector 扩容是"移动"还是"拷贝"
vector 里有 4 个元素，push_back 第 5 个触发扩容。打开开关把元素的移动构造标成 noexcept，看 vector 的搬运策略如何改变；再观察"搬到一半抛异常"时两种策略的生死差别。
元素的移动构造 noexcept —— 当前：否（move 可能抛）
旧内存（容量 4，已满）
⬇ 搬运到新内存…
新内存（容量 8）
步骤 0
面试金句：被问"为什么移动构造要写 noexcept？"——答："因为 std::vector 扩容时用 std::move_if_noexcept：只有移动构造是 noexcept 时它才敢用移动，否则为了维持 strong 异常安全只能退化成拷贝。一个 noexcept 关键字就能让 vector<T> 的扩容从 O(n) 次拷贝变成 O(n) 次移动，性能差距巨大。"

② 编译器优化：noexcept 函数不需要生成异常展开表 → 二进制更小、调用更快。

③ 接口契约：使用方可以放心使用，不需要 try/catch。

8.4 抛了 noexcept 函数会怎样

void f() noexcept {
    throw std::runtime_error{"oops"};      // 编译通过！但运行时…
}
f();   // → std::terminate() → 默认 abort() → core dump
不要滥用 noexcept：仅在真的不可能抛或必须不抛（移动构造）时才标。否则你"承诺"了无法兑现的事，运行时直接挂程序。

8.5 noexcept 与函数指针

构造 / 析构中的异常

构造函数抛异常 · function-try-block — 构造函数初始化列表的异常 · 析构函数中的异常 — 几乎一律不抛 · 正确的析构错误处理

9.1 构造函数抛异常

规则：构造函数抛异常 → 对象没有构造完成 → 析构不会被调用。
但已构造的成员和基类会按逆序析构。

struct A { A() { LOG("A"); } ~A() { LOG("~A"); } };
struct B { B() { LOG("B"); throw 1; } ~B() { LOG("~B"); } };
struct C { C() { LOG("C"); } ~C() { LOG("~C"); } };

struct X {
    A a;
    B b;        // 抛
    C c;
};

// X x;
// 输出: A B ~A — c 没构造，X 析构没执行

9.2 function-try-block — 构造函数初始化列表的异常

class X {
    A a_;
    B b_;
public:
    // 普通 try 抓不到初始化列表里的异常
    X(int v) try : a_(v), b_(v) {           // function-try-block
        // 主体
    } catch (...) {
        LOG("failed in init list");
        // 此时已构造的成员/基类已经析构了，你只能记日志/转换异常
        throw;          // 必须重抛 (隐式行为) — 不能"吞掉"异常
    }
};
function-try-block 里 不能吞掉异常（不重抛会自动重抛）— 因为对象本身没构造完，外层不能假设它能用。

9.3 析构函数中的异常 — 几乎一律不抛

C++11 起，析构函数默认 noexcept。如果一定要抛，需要显式 noexcept(false)。

class DangerousLog {
public:
    ~DangerousLog() noexcept(false) {       // 显式允许
        if (!flushed_) throw std::runtime_error{"unflushed"};
    }
};

问题：栈展开时另一个析构如果再抛 → 已经在处理一个异常 + 又一个新异常 → std::terminate。

9.4 正确的析构错误处理

class FileWriter {
    FILE* fp_;
    bool flushed_ = false;
public:
    void flush() {
        if (::fflush(fp_) != 0) throw std::system_error{errno, std::system_category()};
        flushed_ = true;
    }
    ~FileWriter() noexcept {                // 不抛
        if (!flushed_) {
            int r = ::fflush(fp_);
            if (r != 0) LOG_ERROR("flush failed in dtor: errno={}", errno);
        }
        ::fclose(fp_);
    }
};

// 调用方负责显式 flush() 来获得错误信息
// 析构只负责"尽力而为" + 记日志
规则：所有析构函数 noexcept (默认)。如果资源释放可能失败：① 显式 flush/close 方法供用户调用；② 析构仅做兜底 + 记日志。

怎么选

10.1 三种风格对比

10.2 经验法则 (Google C++ Style 的做法很激进，C++ 社区分歧)

用异常

  • 真正的异常情况（罕见 + 严重）— 内存不足 / 配置错 / 网络致命错
  • 构造函数失败（构造没法返错误码）
  • 跨多层调用栈传递错误
  • 错误是少数路径，主路径希望干净

用错误码 / expected

  • 可预期的失败（"缓存 miss" "key 不存在"）— 正常控制流
  • 性能敏感的热路径
  • 嵌入式 / 关闭异常的环境 (-fno-exceptions)
  • 与 C API 边界
  • 调用方必然要做错误处理

10.3 反模式：用异常做控制流

// ❌ 把异常当 break/continue 用
void find_in_tree(Node* root, int v) {
    try {
        walk(root, [&](Node* n){
            if (n->val == v) throw n;     // 找到 → throw 跳出
        });
    } catch (Node* n) { return n; }
    return nullptr;
}
// 慢 + 难读 + 滥用

// ✅ 用普通返回值
Node* find_in_tree(Node* root, int v) {
    if (!root) return nullptr;
    if (root->val == v) return root;
    if (auto* l = find_in_tree(root->left, v)) return l;
    return find_in_tree(root->right, v);
}
回答"为什么 Google C++ 禁用异常?" — 答：① Google 大量历史代码不是异常安全 → 一开就崩；② 二进制大小敏感（移动端/嵌入式）；③ 早期编译器异常实现差异大。C++ 社区主流不接受这条规则，但确实是 Google 工程现实选择。

类型化错误处理

受 Rust 的 Result<T, E> 启发。表达"要么有值 T，要么有错误 E"。

11.1 基础用法

#include <expected>

enum class ParseError { Empty, BadFormat, OutOfRange };

std::expected<int, ParseError> parse_port(std::string_view s) {
    if (s.empty()) return std::unexpected(ParseError::Empty);
    int n;
    auto [p, ec] = std::from_chars(s.data(), s.data() + s.size(), n);
    if (ec != std::errc{}) return std::unexpected(ParseError::BadFormat);
    if (n <= 0 || n > 65535) return std::unexpected(ParseError::OutOfRange);
    return n;
}

// 调用
auto r = parse_port("8080");
if (r) {                          // 有值
    LOG("port = {}", *r);
} else {
    LOG("err = {}", (int)r.error());
}

11.2 链式操作 (Monadic 接口)

auto result = parse_port(input)
    .and_then([](int p)  { return connect(p); })       // 仅当 OK 才执行
    .transform([](Conn c){ return c.stats(); })        // 类似 map
    .or_else([](auto e) { LOG("err {}", e); return std::expected<Stats, ParseError>{}; });

11b.1 为什么需要 std::error_code？

异常和 std::expected 是错误传递机制，但"错误本身"用什么类型表达？过去几大派系混战：

  • C 风格 int errno —— 全局变量、多线程地狱、不携带描述
  • POSIX 错误号 errno 常量（EACCES、ENOENT...）
  • Win32 GetLastError()、HRESULT
  • OpenSSL ERR_get_error()、CURL CURLcode ...

问题：每个库一套错误码 → 跨库传递就要"转换地狱"。C++11 引入 std::error_code + std::error_category 解决：一个统一的、可扩展、零开销的错误编码体系。

11b.2 核心三件套

类型含义
std::error_code具体错误：(value, category 指针) 两字段，可拷贝、可比较
std::error_category抽象基类，每个错误源（POSIX / Win32 / 自家库）一个 singleton 实例

11b.4 给自家库定义 error_category（生产模板）

// my_error.hpp
enum class MyError {
    Ok = 0,
    InvalidArg,
    ConnectionLost,
    Timeout,
    QuotaExceeded,
};

class MyErrorCategory : public std::error_category {
public:
    const char* name() const noexcept override { return "myapi"; }

    std::string message(int ev) const override {
        switch (static_cast<MyError>(ev)) {
            case MyError::Ok:              return "ok";
            case MyError::InvalidArg:      return "invalid argument";
            case MyError::ConnectionLost:  return "connection lost";
            case MyError::Timeout:         return "operation timed out";
            case MyError::QuotaExceeded:   return "quota exceeded";
        }
        return "unknown";
    }

    // 可选：映射到 std::errc 让跨库比较生效
    std::error_condition default_error_condition(int ev) const noexcept override {
        switch (static_cast<MyError>(ev)) {
            case MyError::Timeout:
                return std::errc::timed_out;
            case MyError::ConnectionLost:
                return std::errc::connection_aborted;
            default:
                return {ev, *this};
        }
    }
};

inline const std::error_category& my_category() {
    static MyErrorCategory cat;            // magic static 线程安全
    return cat;
}

// 关键：注册到 std::error_code 框架 — 必须在 std 命名空间特化
namespace std {
    template<> struct is_error_code_enum<MyError> : true_type {};
}

// 配合 ADL 的隐式构造
inline std::error_code make_error_code(MyError e) noexcept {
    return {static_cast<int>(e), my_category()};
}
// 使用 — 自然像内置错误码一样工作
std::error_code ec = MyError::Timeout;     // ✅ 隐式转换
if (ec == MyError::Timeout) { ... }         // ✅ 直接比较
if (ec == std::errc::timed_out) { ... }     // ✅ 跨库语义比较
std::println("err: {}", ec.message());     // "operation timed out"
这是所有 C++ 库的标准错误传递方式。boost.asio、boost.beast、std::filesystem、std::regex、std::charconv 全部使用 error_code 作为 noexcept 出参。

11b.8 高级工程师的两个观点

① 把 error_code 看作 C++ 错误处理的"通用 IR"。各种新机制（expected / system_error / Outcome / LEAF）都是包装它。库设计者只要定义自家 enum + error_category，所有上层范式自动适配。
② "异常 vs 错误码"是假二分法。真实大型项目几乎都是分层的：低层（系统调用 / 协议解析）用 error_code / expected；中层（业务逻辑）混用；高层（请求边界 / RPC 入口）才统一转换。一个 codebase 内强行 100% noexcept 或 100% 异常往往两边都不讨好。

异常的性能开销

两种实现策略 · 真正的开销在哪 · 实测对照（粗略） · -fno-exceptions

12.1 两种实现策略

策略正常路径异常路径用在哪
SJLJ (setjmp/longjmp) 每个函数入口/出口都有开销 setjmp/longjmp 跳转 古老编译器、某些嵌入式 ABI
Table-based / DWARF (zero-cost) 0 cycles（不抛时无开销） 高（栈展开 + 表查找） 现代 GCC/Clang/MSVC (主流)

"零开销异常"指的是不抛时没有运行时代价 — 编译器把异常处理信息存在 .eh_frame + .gcc_except_table 等只读段，正常代码路径不读它们。

12.2 真正的开销在哪

  1. 二进制体积：每个函数的 unwind 表，开销 ~5-10% 的 .text/.eh_frame 段
  2. 编译时间：编译器要分析每个可能抛点
  3. 抛出时：动态分配异常对象 + 栈展开 + LSDA 表查找 → ~ 几千 ns 到 µs 级
  4. 禁止某些优化：编译器假设几乎任何调用都可能抛 → 寄存器分配/重排受限

12.3 实测对照（粗略）

操作典型耗时
函数调用 (普通)~ 1 ns
正常 return~ 1 ns
检查错误码 + return~ 2 ns
throw + 走 1 层 catch~ 1-5 µs
throw + 走 100 层 catch~ 10-100 µs

结论：异常成功路径几乎免费；失败路径很贵 — 仅在「失败少」场景使用。

12.4 -fno-exceptions

g++ -fno-exceptions ...
  • 禁用 throw/try/catch 语法（编译错）
  • 标准库不能用 (vector::at 等抛异常的功能不能用)
  • 二进制体积减少 5-15%
  • 嵌入式 / 内核模块 / WebAssembly 常用
"异常的零开销是什么意思?" — 答：正常路径零开销（不抛时无运行时代价），异常路径很贵（µs 级）。这与 SJLJ 实现相反 — 后者每次进入 try 都有 setjmp 开销。

class Database {
    std::unique_ptr<sqlite3, decltype(&sqlite3_close)> db_;

public:
    explicit Database(const std::string& path)
        : db_(nullptr, &sqlite3_close)
    {
        sqlite3* raw;
        if (sqlite3_open(path.c_str(), &raw) != SQLITE_OK) {
            std::string msg = sqlite3_errmsg(raw);
            sqlite3_close(raw);                        // 手动释放半构造资源
            throw std::runtime_error{"open: " + msg};
        }
        db_.reset(raw);                                // 之后由 unique_ptr 管
    }

    void transaction(std::function<void()> work) {
        execute("BEGIN");
        ScopeExit rollback{[this]{ execute("ROLLBACK"); }};
        work();                                          // 可能抛
        execute("COMMIT");
        rollback.dismiss();                              // 成功 → 不回滚
    }
};

13.5 跨线程异常传递

class WorkerPool {
    std::vector<std::future<void>> futures_;
public:
    template<class F>
    void submit(F f) {
        futures_.push_back(std::async(std::launch::async, std::move(f)));
    }

    void wait_all() {
        std::vector<std::exception_ptr> errors;
        for (auto& f : futures_) {
            try { f.get(); }                            // future::get 重抛 worker 的异常
            catch (...) { errors.push_back(std::current_exception()); }
        }
        if (errors.size() == 1) std::rethrow_exception(errors[0]);
        if (errors.size() > 1) throw AggregateError{std::move(errors)};
    }
};

易错点速查（10 个高频陷阱）

  1. throw e 而不是 throw — 切片 
    catch (const std::exception& e) { throw e; }      // ⚠️ 切片为基类
catch (const std::exception& e) { throw; }       // ✅ 保留派生类
  2. catch by value 拷贝 
    catch (std::exception e) {}              // ⚠️ 拷贝 + 切片
catch (const std::exception& e) {}       // ✅
  3. catch 顺序 
    catch (const std::exception& e) {}       // 基类放前面
catch (const RpcError& e) {}              // ⚠️ 永远到不了
  4. 析构函数抛异常 
    ~DAO() { if (!flushed_) throw ...; }    // ⚠️ 栈展开时再抛 → terminate
  5. noexcept 函数中抛 
    void f() noexcept { throw ...; }        // ⚠️ 直接 terminate
  6. 构造异常 + 半构造对象资源泄漏
    → 用智能指针 / make_unique / RAII 包装。
  7. 异常用作控制流
    → 用普通返回 / optional / expected 替代。
  8. 移动构造没标 noexcept
    → vector 扩容退化为拷贝。
  9. function-try-block 吞异常 
    X() try : a_(...) {} catch (...) {
    // 不写 throw → 也会自动重抛 (规则)
}
  10. 用 catch (...) 不重抛
    → 吞掉所有错误，对调试是噩梦。

15.1 基础

基础throw / try / catch 工作流程?
流程
throw 构造异常对象并开始栈展开：从抛出点向外逐层析构已构造的局部对象，沿调用栈寻找类型匹配的 catch；找到则执行处理器，全程无匹配则 std::terminate。
基础std::exception 类层次? 常用派生类?
基类
std::exception（提供 what()）。
两大支
logic_error（invalid_argument / out_of_range / length_error / domain_error）、runtime_error（range_error / overflow_error / system_error）；另有 bad_alloc / bad_cast 等。
基础什么是栈展开?
定义
异常传播时，从抛出点到捕获点之间已完成构造的自动对象按逆序析构、栈帧释放。这正是 RAII 生效的底层机制。
基础异常对象的生命周期?
周期
在 throw 处构造（存于实现保留的安全区域），传播期间存活，直到对应 catch 块结束（或 rethrow）才销毁。catch (const E&) 引用的就是它本身。
基础throw vs throw e 区别?
throw;
在 catch 内重新抛出当前异常，保留动态类型（多态）。
throw e;
抛出 e 的拷贝，发生切片，丢失派生类型。
基础catch by value vs by reference?
by value
拷贝 + 可能切片，丢多态。
by reference
const E&：无拷贝、保留多态类型，推荐。

15.2 中级

中级异常安全 3 级保证分别是什么? 举例。
basic
即使抛异常也不泄漏资源，对象处于合法但未指定状态（如 vector::push_back 失败）。
strong
事务性：要么成功，要么状态完全不变（copy-and-swap 实现的 operator=）。
nothrow
标 noexcept 承诺不抛（swap、析构、移动构造）。完整示范见 §15.4。
中级什么是 RAII? 与异常安全什么关系?
RAII
资源获取即初始化，析构自动释放。
关系
栈展开会自动调析构 → RAII 是实现异常安全（basic 保证）的根本手段，免去手写 try/catch 清理。
中级noexcept 的两个含义?
说明符
声明函数不抛（违反则 terminate）。
运算符
编译期查询表达式是否 noexcept，用于条件 noexcept(...) 与 move_if_noexcept。
中级构造函数抛异常 → 析构会被调用吗?
自身
该对象自己的析构不调用（未构造完成）。
成员
已构造完成的成员 / 基类会被逆序析构。因此构造中获取的资源应交给成员 RAII 管理。
中级析构函数能抛异常吗? 后果?
后果
析构默认 noexcept；若在栈展开期间再抛 → std::terminate。
规则
析构绝不让异常逸出（内部 try/catch 吞掉或记录）。
中级copy-and-swap 惯用法如何实现强异常安全?
机制
operator=(T rhs) 按值收参（拷贝可能抛，但发生在改 *this 之前），再 noexcept swap。任何抛异常都在原对象被修改前 → 原对象完好 = 强保证，且天然自赋值安全。

15.3 高级

高级异常的"零开销"是什么意思? 真的零开销吗?
含义
指不抛异常时正常路径无运行时开销（table-based 模型，无标志位查询）。
代价
转移到抛出时（解析展开表，很慢）+ 二进制体积（展开表）。所以"正常路径零开销、抛出路径昂贵"。
高级异常是怎么实现的 (ABI 层面)?
Itanium EH
编译器为每个函数生成 LSDA + unwind table（.eh_frame / .gcc_except_table）。
抛出时
libunwind + personality routine 解析表，逐帧执行 landing pad（析构局部对象）并匹配 catch。抛出前无任何运行时栈成本。
高级std::move_if_noexcept 怎么工作?
逻辑
若 T 的移动构造为 noexcept → 返回 T&&（移动）；否则返回 const T&（拷贝）。vector 扩容靠它在"移动可能抛"时退回拷贝以保强异常安全。
高级std::exception_ptr 用在什么场景?
用途
用 current_exception() 捕获并存储 / 转移异常，跨线程、跨边界传播（promise::set_exception / async），延后用 rethrow_exception 重抛。
高级std::expected vs 异常 — 怎么选?
expected
可预期、高频失败（解析、查找）：无展开开销、显式体现在返回类型，适合热路径 / 禁异常环境。
异常
真正异常情形、构造失败、需穿越深调用栈时。
高级function-try-block 与普通 try 区别?
区别
function-try-block 把整个函数体（含构造函数初始化列表）包进 try，能捕获成员 / 基类初始化时抛出的异常 —— 普通 try 覆盖不到初始化列表。
约束
构造函数的 function-try 中必须重抛（不能吞掉）。
高级跨线程怎么传递异常?
做法
工作线程 catch 后用 current_exception() 存入 exception_ptr（或 promise 的共享状态），主线程 rethrow_exception 重抛。std::async / future 自动做这件事。
高级为什么 vector::push_back 强异常安全保证依赖移动构造 noexcept?
原因
扩容需把旧元素搬到新缓冲区。移动若可能抛、搬到一半抛 → 旧缓冲区已被破坏、无法回滚。只有移动 noexcept 才用移动且仍保强保证，否则退化为拷贝（旧数据完好）。

15.4 速答示范（第 7 题）

问：异常安全 3 级保证分别是什么? 举例。

答：

  • basic guarantee (基本保证)：即使抛异常，不泄漏资源，对象处于合法但未指定状态。例如 std::vector::push_back 在 T 拷贝构造抛异常时：vector 长度可能改变，但不崩、不泄漏。
  • strong guarantee (强保证)：操作具事务性 — 要么成功，要么状态完全不变（rollback）。例如copy-and-swap 实现的 operator=：先拷贝构造，再 noexcept swap，任何抛异常都发生在 swap 之前 → 原对象不受影响。
  • nothrow guarantee (不抛保证)：函数标 noexcept，承诺不抛。例如 swap、所有析构、移动构造（应该）。

"无保证" = 反模式，避免。

实战中：所有公开接口至少 basic 保证；关键事务性操作 strong；析构和 swap nothrow。
