1. 引言 — Sanitizer 是 C++ 工程师必备

C++ 三大类 bug：

  • 内存错误：越界 / use-after-free / double-free / 泄漏 → ASan/LSan
  • 未定义行为：有符号溢出 / 空指针解引用 / 类型穿越 → UBSan
  • 并发错误：data race / lock order / 内存序错误 → TSan

这些 bug 在测试期常常不复现（"测试通过 = 没 bug" 是错觉），上线后随机崩溃。Sanitizer 是把它们暴露在测试期的标准工具。

2. Sanitizer 全景与选择

2.1 五大 Sanitizer 速查

名称检测目标开销 (CPU/内存)支持
AddressSanitizer (ASan) 越界 / UAF / double-free / heap/stack/global overflow ~ 2× CPU, 3-4× 内存 gcc 4.8+ / clang 3.1+ / MSVC 16.9+
UndefinedBehaviorSanitizer (UBSan) 有符号溢出 / 空指针 / 越界 / 错误 cast / 严格别名 / 未对齐 / VPTR 错误 ~ 1.2× CPU gcc 4.9+ / clang 3.3+
ThreadSanitizer (TSan) 数据竞争 / 死锁 (部分) / 错误的 mutex 用法 ~ 5-15× CPU, 5-10× 内存 gcc 4.8+ / clang 3.2+
MemorySanitizer (MSan) 未初始化内存读取 ~ 3× CPU, 2-3× 内存 仅 clang 3.3+ (gcc 不支持)
LeakSanitizer (LSan) 内存泄漏 (退出时报告) ~ 0 (内置在 ASan) gcc 4.9+ / clang 3.4+

2.2 何时用哪个

开发期

  • 本地跑测试: ASan + UBSan (一起开)
  • 多线程模块: TSan 单独跑
  • 新项目敏感场景: MSan

CI

  • 每次 PR: ASan + UBSan 必跑
  • 每次 PR: TSan 跑并发测试
  • nightly: 完整测试集 + 各 Sanitizer 单独
  • 发版: LSan 检查无新泄漏

Fuzz Testing

  • libFuzzer + ASan + UBSan + LSan
  • OSS-Fuzz 模式

不能用 Sanitizer 的场景

  • 嵌入式/资源极少
  • Production (开销太大)
  • 确定性要求强的环境 (Sanitizer 改变内存布局)

2.3 通用启用方式

# 编译 + 链接都需要 -fsanitize=...
g++ -O1 -g -fno-omit-frame-pointer -fsanitize=address,undefined \
    myapp.cpp -o myapp_asan

# 直接运行 — 错误时输出报告
./myapp_asan

关键编译选项：

  • -fsanitize=... — 启用 (可逗号分隔多个)
  • -O1 — 不开 O0 (太慢) 也不开 O3 (优化掩盖错误)
  • -g — 含调试符号让报告显示行号
  • -fno-omit-frame-pointer — 准确栈回溯
  • -fno-optimize-sibling-calls — 防 tail-call 让栈不准

3. ASan · AddressSanitizer 实战

3.1 能检测什么

错误类型例子
heap-buffer-overflow越界访问 new 的内存
stack-buffer-overflow越界访问栈上数组
global-buffer-overflow越界访问全局数组
heap-use-after-free访问已释放内存
stack-use-after-return函数返回后访问栈对象
stack-use-after-scope作用域结束后访问局部对象
double-free重复 delete
invalid-freedelete 非 new 的指针
memory-leak(LSan 集成) 退出时未释放

3.2 完整示例 — 一个有 bug 的程序

// bug.cpp
#include <cstdio>
int main() {
    int* arr = new int[10];
    arr[10] = 42;       // ❌ heap-buffer-overflow (off-by-one)
    delete[] arr;
    printf("%d\n", arr[0]);    // ❌ heap-use-after-free
    return 0;
}
$ g++ -O1 -g -fsanitize=address bug.cpp -o bug
$ ./bug

==12345==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x602000000038
WRITE of size 4 at 0x602000000038 thread T0
    #0 0x... in main bug.cpp:5
    #1 0x... in __libc_start_main

0x602000000038 is located 0 bytes to the right of 40-byte region [0x602000000010,0x602000000038)
allocated by thread T0 here:
    #0 0x... in operator new[]
    #1 0x... in main bug.cpp:4

SUMMARY: AddressSanitizer: heap-buffer-overflow bug.cpp:5 in main

报告告诉你：

  • 错误类型 (heap-buffer-overflow)
  • 访问地址 + 越界方向 (右越界 0 字节)
  • 访问栈 (代码 + 行号)
  • 原始内存分配处 (line 4 的 new)

3.3 常用错误的 ASan 报告样例

// use-after-free
int* p = new int(42);
delete p;
*p = 1;
// → AddressSanitizer: heap-use-after-free

// stack-use-after-return
int* bad() {
    int local = 42;
    return &local;       // 返回栈地址
}
*bad() = 1;            // UAR
// 注: 默认 ASan 不抓 UAR (开销大), 需 ASAN_OPTIONS=detect_stack_use_after_return=1

// double-free
int* p = new int(1);
delete p;
delete p;
// → AddressSanitizer: attempting double-free

// global overflow
int arr[10];
int main() { arr[10] = 1; }
// → AddressSanitizer: global-buffer-overflow

3.4 ASan 运行时配置

# 通过环境变量配置
ASAN_OPTIONS="detect_leaks=1:abort_on_error=1:halt_on_error=1:print_stats=1:check_initialization_order=1" ./prog

常用选项：

  • detect_leaks=1 — 启用 LSan (默认 Linux 开)
  • halt_on_error=1 — 第一个错误就退出 (默认)
  • abort_on_error=1 — 错误时 abort() 出 core (调试用)
  • detect_stack_use_after_return=1 — 打开 UAR (慢)
  • strict_string_checks=1 — 严格 string 函数检查
  • verbosity=1 — 详细日志
  • log_path=/tmp/asan — 日志写文件
  • suppressions=asan.supp — 抑制规则文件
实践：开发期默认所有选项. CI 用最严格 (halt_on_error=1, detect_leaks=1)。

4. ASan 内部原理 (shadow memory) ⭐

4.1 基本思路

ASan 通过两个机制工作：

  1. 每次内存访问插桩：编译器在 load/store 前插入检查代码
  2. shadow memory：用 1/8 的内存影子标记每个字节是否"可用"

4.2 Shadow Memory 映射

每 8 字节真实内存对应 1 字节影子内存。地址映射：shadow = (real >> 3) + offset。

影子值的含义：

  • 0 — 8 字节全部可用
  • 1-7 — 前 N 字节可用，后 (8-N) 字节不可用
  • 负值 (各种) — 不同含义的"红区" (heap free / stack 已退出 / 全局右侧)
// 简化的检查代码 (编译器插桩)
void check_load(void* addr, size_t size) {
    uintptr_t shadow_addr = ((uintptr_t)addr >> 3) + SHADOW_OFFSET;
    int8_t shadow = *(int8_t*)shadow_addr;
    if (shadow != 0) {
        // 进一步精确检查
        int8_t last_byte = ((uintptr_t)addr & 7) + size - 1;
        if (last_byte >= shadow) __asan_report_error(addr);
    }
}

4.3 红区 (red zone) 机制

每次 malloc/new 时，ASan 在对象前后留出一段"红区"：

UAF 检测：free 时不立即归还内存，而是放入"隔离队列" (quarantine)，并把整个区域 shadow 标为 "freed" → 后续访问立刻报错。

4.4 性能开销

  • CPU：~ 2× (每次内存访问加 ~7 条插桩指令)
  • 内存：~ 3-4× (shadow + quarantine + redzone)
  • 不能上 production (开销 + 内存 + 行为差异)
"ASan 怎么发现 UAF?" — 答：① free 时不立即释放，把内存放入 quarantine 隔离队列；② 把这块区域的 shadow 字节标为 "已释放" (-1 等负值)；③ 任何后续访问触发 shadow 检查 → 立即报告。

4.5 局限

  • 检测不到所有越界 — 红区有限，跨过红区到下一个对象的越界检测不到
  • 不检测栈上局部变量越界写到栈相邻局部变量（除非启用 fsanitize-address-use-after-scope）
  • 误报极少，但有 (低层 hacks 类代码可能触发)
  • 不能查类型错误 / 未初始化读 (需要 UBSan / MSan)

5. UBSan · UndefinedBehaviorSanitizer

5.1 能检测什么

C++ 标准把"未定义行为" 列了 200+ 条。UBSan 能检测其中常见的 ~30 种：

检查项检测的 UB
signed-integer-overflow有符号整数溢出
shift移位超过位宽 / 移位负数
integer-divide-by-zero整数除 0
float-divide-by-zero浮点除 0 (默认关闭)
null空指针解引用
nonnull-attribute违反 __attribute__((nonnull))
vptr错误的多态对象 cast
bounds数组越界 (固定大小)
alignment未对齐访问
object-size访问超出对象大小
return非 void 函数无 return
unreachable到达 unreachable 路径
vla-boundVLA 大小为 0 或负
function函数指针类型错误
enum无效的 enum 值
bool无效的 bool 值 (非 0/1)

5.2 用法

# 全部检查
g++ -O1 -g -fsanitize=undefined myapp.cpp

# 只开几项
g++ -O1 -g -fsanitize=signed-integer-overflow,null myapp.cpp

# 排除某项
g++ -O1 -g -fsanitize=undefined -fno-sanitize=alignment myapp.cpp

5.3 经典报告

// signed overflow
int x = INT_MAX;
int y = x + 1;     // UB!

// 报告:
runtime error: signed integer overflow:
    2147483647 + 1 cannot be represented in type 'int'
    SUMMARY: UndefinedBehaviorSanitizer: signed-integer-overflow at line N
// null deref
int* p = nullptr;
*p;
// runtime error: load of null pointer of type 'int'

// shift
int x = 1 << 32;
// runtime error: shift exponent 32 is too large for 32-bit type 'int'

// vptr (错误的 dynamic_cast 风险)
struct Base { virtual ~Base(){}; };
struct Derived : Base {};
Base* b = new Base;
Derived* d = static_cast<Derived*>(b);
d->virtual_method();
// runtime error: member call on address ... which does not point to an object of type 'Derived'

5.4 UBSan 默认是"trap"

默认行为：触发 UB 时打印消息后继续执行。可改为：

# 触发即 abort (调试用)
g++ -fsanitize=undefined -fno-sanitize-recover=all myapp.cpp

# 或环境变量
UBSAN_OPTIONS="halt_on_error=1:abort_on_error=1" ./prog

5.5 性能与用法

  • 开销 ~ 1.2× (轻量)
  • 建议默认开 (开发 + CI)
  • 常与 ASan 一起用：-fsanitize=address,undefined
  • 有的项目甚至在 Production 开关键检查 (fsanitize=signed-integer-overflow + trap-on-error)
实战经验：UBSan 是性价比最高的 sanitizer — 开销小、能发现的 bug 严重 (UB 是最难调试的 bug 类型)。默认开。

6. TSan · ThreadSanitizer 数据竞争检测

6.1 能检测什么

  • Data race：两个线程同时访问同一内存，至少一个写，且无 happens-before 关系
  • Lock 顺序异常：可能死锁的 lock 顺序
  • Mutex 误用：unlocked unlock / 在持锁线程之外 unlock
  • 已销毁的同步原语使用

6.2 经典 data race 例子

int g = 0;

void thread1() { g = 1; }       // 写
void thread2() { if (g) {} }   // 读

int main() {
    std::thread t1{thread1}, t2{thread2};
    t1.join(); t2.join();
}
$ g++ -O1 -g -fsanitize=thread race.cpp -o race
$ ./race

WARNING: ThreadSanitizer: data race (pid=12345)
  Write of size 4 at 0x... by thread T1:
    #0 thread1() race.cpp:3
  Previous read of size 4 at 0x... by thread T2:
    #0 thread2() race.cpp:4

  Location is global 'g' of size 4 at 0x...

  Thread T1 (tid=12346) created by main thread at:
    #0 std::thread::thread<...> thread:N

SUMMARY: ThreadSanitizer: data race race.cpp:3 in thread1()

6.3 TSan 抓得到的几类问题

// ① 朴素 race
int g = 0;
void a() { g++; }
void b() { g--; }   // race!

// ② 漏锁
std::mutex mu;
int g;
void a() { std::lock_guard lk{mu}; g = 1; }
void b() { g = 2; }   // race! (没加锁)

// ③ memory order 太弱
std::atomic<int> flag{0};
int data = 0;
void producer() { data = 42; flag.store(1, std::memory_order_relaxed); }
void consumer() { while (!flag.load(std::memory_order_relaxed)); use(data); }
// race on data — relaxed 不建立 happens-before

6.4 TSan 能识别的同步原语

  • std::mutex / shared_mutex / recursive_mutex / atomic
  • std::condition_variable
  • std::thread::join / detach
  • future::get / promise::set_value
  • pthread_mutex_t / pthread_rwlock_t / pthread_cond_t
  • POSIX semaphore

6.5 TSan 不能识别的

  • 自定义 spinlock (用普通 atomic 实现) — 需要 __tsan_acquire / __tsan_release 注解
  • 跨进程同步 (共享内存 + futex)
  • 用户态自旋等待 atomic
  • fence 间接建立的 happens-before (relaxed atomic)

6.6 用法

# 编译
g++ -O1 -g -fsanitize=thread myapp.cpp -o myapp_tsan

# 运行时配置
TSAN_OPTIONS="halt_on_error=1:second_deadlock_stack=1" ./myapp_tsan

常用 TSAN_OPTIONS：

  • halt_on_error=1 — 第一个 race 就退出
  • second_deadlock_stack=1 — 死锁报告显示两条调用栈
  • history_size=7 — 内存访问历史长度 (1-7)
  • flush_memory_ms=N — 周期性清理内存

6.7 性能与限制

  • 开销 ~ 5-15× CPU, 5-10× 内存 (很贵)
  • 不能与 ASan 同时用 (二选一)
  • TSan + 长时间测试可能 OOM (内存不断增长)
  • 误报罕见，但"漏报"常见 — race 不一定每次跑都触发
TSan 不保证能发现所有 race — 它只能报告本次执行实际发生的 race。要更彻底，需要多次运行 + 不同负载 / 调度顺序。

7. TSan 内部原理 (happens-before vector clock) ⭐

7.1 核心思想

TSan 维护每个内存位置的访问历史 + 每个线程的vector clock，用 happens-before 关系判定 race。

7.2 happens-before

(详细见 cpp_memory_model.html § happens-before。)

TSan 在以下操作建立 hb 边：

  • 同一线程内的语句先后
  • mutex.unlock() → 下一个 mutex.lock()
  • thread.start → 子线程的所有操作
  • 子线程结束 → thread.join() 之后
  • condition_variable notify → 下一次 wait 返回
  • atomic release → 对应 acquire
  • future set_value → get

7.3 Race 判定

对每个内存位置维护：

  • "上次写它的线程 + 时刻"
  • "上次读它的线程 + 时刻"

当线程 T1 访问 X 时：

  1. 检查 X 的上次访问者 T2 是否 happens-before T1
  2. 没有 hb → 这是 race，报警
  3. 有 hb → 安全，更新 X 的访问记录

7.4 Shadow memory

类似 ASan 但记录的是访问历史而不是有效性：

  • 每 8 字节真实内存对应 N 个 shadow word (默认 4，可配)
  • 每个 shadow word 记录: 线程 ID + 访问时刻 (vector clock) + 读/写
  • 新访问到来时与 shadow 中的历史比对

7.5 性能爆炸的来源

  • 每次内存访问都要更新 shadow + 检查历史 → ~10× CPU
  • vector clock 随线程数增长 → 每线程 N 个 timestamp
  • 访问历史不能太短 (漏报) 也不能太长 (内存爆) → 默认 4 项
"TSan 怎么知道两个访问之间有没有 happens-before?" — 答：每个线程维护 vector clock (本线程 + 其他线程的"已知最新时刻"). 同步操作 (mutex unlock/lock, atomic release/acquire) 时合并双方 vector clock. 访问内存时检查 shadow 中"上次访问"的 vector clock 是否 ≤ 当前 vector clock — 是则有 hb，安全；否则是 race。

8. MSan · MemorySanitizer 未初始化内存读

8.1 检测什么

读取未初始化的内存。这是 C++ 隐藏 bug 的大类，比越界更阴险（很多时候不崩，但产生错误结果）。

struct S { int x, y; };

void f() {
    S s;
    s.x = 1;            // y 未初始化
    if (s.y > 0) {}     // ❌ 读未初始化值 → MSan 报警
}

8.2 用法

# 仅 clang 支持
clang++ -O1 -g -fsanitize=memory -fno-omit-frame-pointer myapp.cpp -o myapp_msan
./myapp_msan

8.3 报告

WARNING: MemorySanitizer: use-of-uninitialized-value
    #0 0x... in f() bug.cpp:6
    #1 0x... in main bug.cpp:N

  Uninitialized value was created by an allocation of 'a' in the stack frame
    #0 0x... in main bug.cpp:N

SUMMARY: MemorySanitizer: use-of-uninitialized-value bug.cpp:6 in f()

8.4 原理

MSan 也用 shadow memory，但记录的是每个 bit 是否初始化：

  • 1:1 shadow（每个真实 byte 对应 1 个 shadow byte）
  • shadow 0 = 已初始化；非 0 = 未初始化的 bit mask
  • 赋值 / memset / read from initialized region → 标 shadow 为 0
  • 读取时检查 shadow，全 0 就 OK

更巧妙：MSan 实现 "延迟报告" — 读未初始化值不立刻报警，而是污染派生值。直到这个污染值被实际使用 (分支条件 / 系统调用参数) 才报告。这样能定位初始值的源头而不是中间传递处。

8.5 MSan 难用的原因

  1. 只支持 clang（gcc 不支持）
  2. 整个程序所有依赖都要用 MSan 重编，包括 libc / libstdc++ / 第三方库 — 否则误报漫天
  3. 实践上：要用 -stdlib=libc++ -L/path/to/msan/libs 链接 MSan-instrumented 标准库
  4. 大型项目集成成本高

替代方案：用 Valgrind Memcheck（开销大但不需要重编依赖）+ 编译警告 -Wuninitialized -Wmaybe-uninitialized（编译期能抓部分）。

8.6 何时用 MSan

  • 新项目 / 全 clang / 控制全部依赖 — 强烈推荐
  • 关键数据处理代码 (会传递错误值) — 单独构建 MSan 版本
  • Fuzz testing 配合 — OSS-Fuzz 提供 MSan 标准库
  • 常规项目：用编译警告 + Valgrind 替代

9. LSan · LeakSanitizer 内存泄漏

9.1 工作方式

LSan 集成在 ASan 中（默认开），也可独立使用：

# 集成在 ASan (推荐)
g++ -O1 -g -fsanitize=address myapp.cpp -o myapp
./myapp     # 退出时自动检查泄漏

# 仅 LSan (独立, 开销最小)
g++ -O1 -g -fsanitize=leak myapp.cpp -o myapp
./myapp

9.2 报告

$ ./bug
==12345==ERROR: LeakSanitizer: detected memory leaks

Direct leak of 40 byte(s) in 1 object(s) allocated from:
    #0 0x... in operator new[]
    #1 0x... in main bug.cpp:5

Indirect leak of 80 byte(s) in 2 object(s) allocated from:
    #0 0x... in operator new
    #1 0x... in some_helper bug.cpp:8

SUMMARY: AddressSanitizer: 120 byte(s) leaked in 3 allocation(s).

9.3 检测原理

LSan 在程序退出时执行类似 GC 的可达性分析：

  1. STW (停所有线程)
  2. 从根 (栈 + 全局变量 + 寄存器 + thread_local) 出发
  3. 遍历堆，标记所有可达对象
  4. 未被标记的 = 泄漏

分两类：

  • Direct leak：根本无人指向（典型泄漏）
  • Indirect leak：被 direct leak 的对象指向（连锁泄漏）

9.4 抑制误报

有些"泄漏" 是预期的（程序退出时不释放是性能优化）。用 suppression 文件：

# lsan.supp
leak:third_party_cache_init
leak:libfoo.so
LSAN_OPTIONS="suppressions=$PWD/lsan.supp:print_suppressions=1" ./prog

9.5 LSan vs Valgrind 对比

维度LSanValgrind Memcheck
开销~ 0 (退出时检查)20-50× (运行时全部检查)
UAF 检测(配 ASan) 是是

实战建议：① 开发期默认 ASan + LSan；② 旧二进制 / 第三方包用 Valgrind；③ CI 全套测试都跑 ASan + LSan，不允许新增泄漏。

10. Sanitizer 组合与冲突

10.1 兼容性矩阵

ASanUBSanTSanMSanLSan

10.2 推荐组合

  • "主流"组合：-fsanitize=address,undefined — 90% 场景
  • 并发测试：-fsanitize=thread,undefined — 单独构建
  • 极致检查：-fsanitize=memory,undefined (clang only) — 仅特定模块

10.3 与其他工具的关系

工具与 Sanitizer 关系
Valgrind Memcheck替代 ASan / MSan / LSan，但慢得多 (20-50×)，不需要重编

cppcheck静态分析，互补
clang-tidy静态分析 + 部分运行时检查 (含 cppcoreguidelines 类规则)
编译警告 -Wall -Wextra互补，第一道防线

11. 运行时配置 + 抑制规则

11.1 通用环境变量

# 各 Sanitizer 的配置环境变量
ASAN_OPTIONS="..."         # AddressSanitizer
UBSAN_OPTIONS="..."        # UndefinedBehaviorSanitizer
TSAN_OPTIONS="..."         # ThreadSanitizer
MSAN_OPTIONS="..."         # MemorySanitizer
LSAN_OPTIONS="..."         # LeakSanitizer

# 多选项用 : 分隔
ASAN_OPTIONS="halt_on_error=1:abort_on_error=1:log_path=/tmp/asan"

11.2 常用通用选项

选项含义
halt_on_error=1第一个错误就退出 (默认 ASan/TSan 是, UBSan 不是)
abort_on_error=1错误时 abort() 出 core dump
print_stacktrace=1详细栈 (默认开)
log_path=/tmp/x错误日志写文件
verbosity=N详细日志
suppressions=file抑制规则文件
print_suppressions=1打印命中的抑制规则
exitcode=N错误时的退出码 (CI 用)

11.3 抑制规则 (suppression)

第三方库或已知问题不能立刻修，可写抑制：

# asan.supp
interceptor_via_lib:libfoo.so
interceptor_via_fun:third_party_func

# tsan.supp
race:third_party::SomeMethod
mutex:badly_implemented_lock

# lsan.supp
leak:legacy_init
leak:libcurl.so
# 程序里嵌入
extern "C" const char* __asan_default_suppressions() {
    return "interceptor_via_lib:libfoo.so\n"
           "interceptor_via_fun:third_party_func\n";
}
抑制规则是技术债。每条都该有 issue tracker 跟进，不能"修不了就埋"。否则真正的 bug 会被掩盖。

11.4 标注代码不被检测

// 函数级别 — 整个函数不做检查
__attribute__((no_sanitize("address")))
void low_level_alloc() { /* ... */ }

// 编译单元级别
#pragma clang attribute push (__attribute__((no_sanitize("thread"))), apply_to=function)
// ... 整个文件
#pragma clang attribute pop

// 内存区域级别 (高级)
__asan_poison_memory_region(p, size);
__asan_unpoison_memory_region(p, size);

12.1 推荐 CI 构建矩阵

Job编译选项用途

on: [push, pull_request]

jobs:
  sanitizer:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        sanitizer: ["address,undefined", "thread,undefined"]
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: |
          mkdir build && cd build
          cmake -DCMAKE_BUILD_TYPE=Debug \

                -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=${{ matrix.sanitizer }}" \
                ..
          make -j$(nproc)
      - name: Test
        env:
          ASAN_OPTIONS: "halt_on_error=1:abort_on_error=1:detect_leaks=1"
          UBSAN_OPTIONS: "halt_on_error=1:abort_on_error=1:print_stacktrace=1"
          TSAN_OPTIONS: "halt_on_error=1:abort_on_error=1:second_deadlock_stack=1"
        run: cd build && ctest --output-on-failure

12.4 处理已知误报

  1. 把抑制规则放代码仓库 (tools/sanitizer_suppressions/)
  2. 每条抑制配 issue 链接 + 截止日期
  3. CI 自动加载抑制规则
  4. 定期 review 抑制清单 (季度)

13. Fuzz Testing + Sanitizer = 黄金组合

13.1 Fuzz Testing 是什么

用大量随机/变异数据喂给程序 → 找 crash / 异常。
配 Sanitizer → 不仅找崩溃，还找潜在问题 (越界 / UAF / UB 等)。

13.2 libFuzzer 入门

// fuzz_target.cpp
#include <cstdint>
#include <cstddef>

extern "C" int LLVMFuzzerTestOneInput(const uint8_t* data, size_t size) {
    // 你的 parser/解码器
    parse_protocol_message(data, size);
    return 0;
}

// 编译
clang++ -O1 -g -fsanitize=fuzzer,address,undefined fuzz_target.cpp -o fuzz_target
./fuzz_target corpus/      # corpus 目录存种子样本

libFuzzer 自动：

  • 变异输入 (bit-flip / byte-shuffle / dict / etc.)
  • 记录覆盖率反馈 (insn coverage / edge coverage)
  • 保留增加覆盖率的样本
  • 遇到 crash → 保存触发输入 + 报告

13.3 OSS-Fuzz

Google 提供的免费持续 fuzz 平台。已知用户：Linux kernel / OpenSSL / SQLite / curl / FFmpeg / V8 / ...

它们用 OSS-Fuzz 持续找 bug，每年发现成千上万的 CVE。

13.4 实战：协议解析器 fuzz

// 给 NETCONF XML 解析器写个 fuzz target
extern "C" int LLVMFuzzerTestOneInput(const uint8_t* data, size_t size) {
    try {
        std::string_view xml{reinterpret_cast<const char*>(data), size};
        NetconfMessage msg = parse_xml(xml);
        // 触发更深路径
        if (msg.has_filter()) apply_filter(msg);
    } catch (const std::exception&) {
        // 异常是预期的, 不算 fuzz 失败
    }
    return 0;
}
# 编译 + 跑 1 小时
clang++ -O1 -g -fsanitize=fuzzer,address,undefined parse_fuzz.cpp -o parse_fuzz
./parse_fuzz corpus/ -max_total_time=3600

# 找到 crash → 保存到 crash-* 文件
./parse_fuzz crash-abc...        # 重现
任何接受外部输入的解析器 / 反序列化器 / 协议处理都应该写 fuzz target. 24h 跑 fuzz 找出的 bug 数量, 远超人工测试.

14. REAL5 个真实工程案例

14.1 案例 1: ASan 找出 yang 解析器越界

症状：QA 报告偶发 crash，gdb 看 core 是 invalid memory，但 backtrace 不准确。

排查：开 ASan 重跑，第一秒就报 heap-buffer-overflow，定位到 parse_xpath 的 off-by-one。

修复：边界检查改 i <= len → i < len。

教训：早开 ASan 节省 weeks。

14.2 案例 2: TSan 抓出 logger 数据竞争

症状：日志偶尔有错乱（一行被另一行截断）。

排查：TSan 报告 logger 内部 buffer 在两个线程间被并发写入。

修复：把 logger 改成线程安全（per-thread buffer + 异步 flush）。

14.3 案例 3: UBSan 找出整型溢出

症状：高负载下偶尔统计数据为负。

排查：UBSan 报告 signed-integer-overflow，定位到 int total = a * b，a/b 都是 30k 级别。

修复：用 int64_t。

14.4 案例 4: LSan 找到 third-party SDK 泄漏

症状：服务跑 1 周内存涨 100MB。

排查：LSan 报告每次连接后第三方 SDK 的 SDKContext 泄漏。

定位：SDK 文档说要 sdk_destroy_context()。我们的代码没调。

修复：用 RAII 包装 SDK 上下文。

14.5 案例 5: Fuzz + ASan 找出 0day

背景：在 NETCONF XML 解析器上跑 fuzz + ASan，24 小时。

结果：找到 3 个 crash：
① 嵌套 100+ 层标签触发栈溢出；
② 特定 UTF-8 字节序列导致 use-after-free；
③ 超长 attribute 触发 buffer overflow。

修复：① 限制递归深度；② 修正 string_view 生命周期；③ 加输入长度上限。

价值：3 个潜在 CVE，修复成本远小于线上事故。

所有案例的共同模式：测试期发现 = 几小时；线上事故发现 = 几天 + 巨大代价。Sanitizer 的投入产出比极高。

15. 局限与陷阱

15.1 不能 100% 保证

  • ASan：红区有限，能跨过红区的越界检测不到
  • TSan：只能发现实际发生的 race，依赖调度顺序 — 多次跑 + 不同负载
  • MSan：要求所有依赖都 instrumented，否则误报
  • UBSan：只检测约 30 种 UB，C++ 标准列了 200+ 种
  • LSan：进程退出时检测，常驻服务要主动 __lsan_do_recoverable_leak_check()

15.2 行为差异

Sanitizer 改变内存布局 → 程序行为可能不同：

  • 地址变化 → ASLR / 缓存行为变
  • 分配开销大 → 时序敏感的 race 可能不复现
  • 内存增长 → OOM 概率增大
  • 开 ASan 不复现的 bug，可能是 timing / 内存布局相关 (race 例子)

15.3 误报来源

  • 低层 hacks：自实现 allocator / 共享内存 / mmap 上的对象
  • 第三方库：未用 sanitizer 重编
  • JIT / 动态生成代码：CPU 直接执行，绕过 instrumentation
  • signal handler：不是所有 sanitizer 函数 async-signal-safe

15.4 Production 上的 Sanitizer?

大多数公司不在 Production 用 ASan/TSan (开销太大)。例外：

  • Chromium：金丝雀版本开 ASan
  • Android：HWASan (硬件辅助 ASan，开销 2× 内存而非 4×)
  • SafeStack：production 友好，仅 1.1× 开销
  • fsanitize=undefined,trap：触发 UB 直接 trap，零开销，可上 production

15.5 大型项目的实践

  1. 所有依赖统一构建 (vcpkg/conan + sanitizer 选项)
  2. CI 矩阵全跑 (6 个组合至少)
  3. 抑制规则版本化
  4. 关键模块单独 fuzz target + 24/7 持续 fuzz
  5. 线上保留 LSan 的低开销检查

16.4 速答示范（第 4 题）

问：TSan 工作原理是什么? 怎么知道两次访问之间没有 happens-before?

答：

  1. 每个内存位置有 shadow memory，记录"上次访问者 + 时刻 (vector clock)"
  2. 每个线程维护自己的 vector clock，记录"本线程当前时刻 + 已知其他线程的时刻"
  3. 同步操作 (mutex unlock/lock, atomic release/acquire) → 合并双方 vector clock
  4. 每次内存访问时：检查 shadow 里"上次访问"的 vector clock 是否 ≤ 当前线程 vector clock
  5. 如果 ≤ → 上次访问 happens-before 本次 → 安全
  6. 否则 → race

限制：① 只能发现实际发生的 race，不发生的看不到；② 自实现 spinlock 等不被 TSan 识别 → 漏报；③ 性能开销 5-15×。
