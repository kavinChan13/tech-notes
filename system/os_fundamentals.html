1.4 fork() 的 Copy-on-Write 机制

Copy-on-Write (COW)：fork() 后父子进程共享同一份物理页面，标记为只读。只有当某方实际写入时，内核才会复制该页——大大减少 fork 的开销。

虚拟内存是现代操作系统最重要的抽象之一。它让每个进程认为自己拥有一片连续的、巨大的私有地址空间，实际上由操作系统和 MMU（内存管理单元）协作将虚拟地址映射到物理内存。

mmap — 内存映射

4.1 mmap vs 传统 I/O · 4.2 mmap 的函数原型与参数 · 4.3 mmap 的四种常见模式 · 4.4 mmap 在内存布局中的位置 · 4.5 mmap 实战代码示例 · …

mmap() 是 Unix/Linux 中最强大的系统调用之一。它将文件或设备直接映射到进程的虚拟地址空间，让你可以像读写内存一样操作文件，绕过传统的 read()/write() 系统调用。

4.1 mmap vs 传统 I/O

4.2 mmap 的函数原型与参数

#include <sys/mman.h>

void* mmap(
    void*  addr,     // 建议映射地址（通常传 nullptr 让内核选择）
    size_t length,   // 映射区域长度（字节）
    int    prot,     // 保护标志: PROT_READ | PROT_WRITE | PROT_EXEC
    int    flags,    // 映射类型: MAP_SHARED | MAP_PRIVATE | MAP_ANONYMOUS
    int    fd,       // 文件描述符（匿名映射传 -1）
    off_t  offset    // 文件偏移（必须是页大小的整数倍）
);

int munmap(void* addr, size_t length);  // 解除映射
int msync(void* addr, size_t length, int flags);  // 同步到磁盘

4.5 mmap 实战代码示例

示例 1：文件映射 — 读取大文件（零拷贝） 
#include <sys/mman.h>
#include <fcntl.h>
#include <unistd.h>
#include <sys/stat.h>
#include <iostream>

int main() {
    int fd = open("large_file.dat", O_RDONLY);
    if (fd == -1) { perror("open"); return 1; }

    // 获取文件大小
    struct stat sb;
    fstat(fd, &sb);

    // 将文件映射到虚拟地址空间
    void* addr = mmap(nullptr, sb.st_size,
                      PROT_READ,         // 只读
                      MAP_PRIVATE,       // 私有映射
                      fd, 0);           // 从文件头开始

    if (addr == MAP_FAILED) { perror("mmap"); close(fd); return 1; }

    close(fd);  // 映射建立后可以关闭 fd

    // 像操作内存一样读取文件！
    auto* data = static_cast<const char*>(addr);
    std::cout << "First 100 bytes: ";
    std::cout.write(data, std::min((off_t)100, sb.st_size));

    // 解除映射
    munmap(addr, sb.st_size);
    return 0;
}
示例 2：匿名映射 — 分配大块内存 
// 匿名映射: 不关联文件，类似于 malloc 但更底层
constexpr size_t SIZE = 1 << 20;  // 1 MB

void* ptr = mmap(nullptr, SIZE,
                 PROT_READ | PROT_WRITE,
                 MAP_PRIVATE | MAP_ANONYMOUS,  // 匿名 + 私有
                 -1, 0);                      // fd = -1

if (ptr != MAP_FAILED) {
    // 页面初始化为零，可以直接使用
    auto* arr = static_cast<int*>(ptr);
    arr[0] = 42;

    munmap(ptr, SIZE);
}

// 实际上 glibc 的 malloc() 对大于 128KB 的分配
// 底层就是调用 mmap(MAP_ANONYMOUS) 实现的！
示例 3：进程间共享内存 
// 父进程创建共享匿名映射
void* shared = mmap(nullptr, 4096,
                     PROT_READ | PROT_WRITE,
                     MAP_SHARED | MAP_ANONYMOUS,  // 共享!
                     -1, 0);

pid_t pid = fork();

if (pid == 0) {
    // 子进程写入
    auto* data = static_cast<int*>(shared);
    data[0] = 12345;  // 写入共享内存
    _exit(0);
}

waitpid(pid, nullptr, 0);

// 父进程可以看到子进程写入的数据！
auto* data = static_cast<int*>(shared);
printf("Shared value: %d\n", data[0]);  // 输出: 12345

munmap(shared, 4096);

4.6 mmap 优缺点总结

✅ 优点

  • 零拷贝：省去内核缓冲→用户缓冲的数据拷贝
  • 延迟加载：只有实际访问的页才从磁盘读入
  • 自动缓存管理：由内核 Page Cache 统一管理
  • 简化编程：指针操作替代 read/write/lseek
  • 内存共享：高效进程间通信
  • 大文件随机读取：性能远优于 lseek+read

❌ 缺点 / 注意事项

  • 映射粒度为页 (4KB)：小文件浪费内存
  • SIGBUS 风险：文件被截断后访问会收到信号
  • 不适合顺序流式处理：此场景 read() + 缓冲更好
  • 32位系统地址空间限制：无法映射大于 ~3GB 的文件
  • 页表压力：映射非常大的文件会占用大量页表项
  • 非原子性：MAP_SHARED 多进程写入需同步机制

5.1 线程与同步 API 全景

5.2 C++ 线程代码示例

#include <thread>
#include <mutex>
#include <shared_mutex>
#include <iostream>

// 共享数据 + 读写锁（类似项目中的 ReadWriteLockedValue<T>）
std::shared_mutex rwMutex;
int sharedData = 0;

void writer(int id) {
    std::unique_lock lock(rwMutex);        // 独占写锁
    sharedData += id;
    std::cout << "Writer " << id << " wrote " << sharedData << "\n";
}   // lock 析构自动释放

void reader(int id) {
    std::shared_lock lock(rwMutex);        // 共享读锁（多个 reader 可并发）
    std::cout << "Reader " << id << " read " << sharedData << "\n";
}

int main() {
    std::vector<std::thread> threads;

    for (int i = 0; i < 3; ++i)
        threads.emplace_back(writer, i);

    for (int i = 0; i < 5; ++i)
        threads.emplace_back(reader, i);

    for (auto& t : threads)
        t.join();  // 等待所有线程完成
}

7.1 用户态 vs 内核态切换代价

系统调用（如 read、write、mmap）会触发用户态到内核态的切换。代价远低于进程切换，但比函数调用贵得多。

切换类型耗时（量级）主要开销
函数调用~1 ns栈帧建立
用户/内核态切换~50-100 nssyscall 指令、寄存器保存、权限检查
线程上下文切换~1-5 μsTLB、寄存器、调度器
进程上下文切换~3-10 μs上述 + 页表切换 + TLB 全刷新
网络往返 (同机)~50 μs内核协议栈处理

优化手段：批量系统调用（epoll 一次返回多个事件）、共享内存（避免 copy_to_user）、io_uring（提交/完成队列在用户态直接构造）。

7.2 同步原语全景对比

原语场景阻塞方式开销C++ API
Atomic无锁计数、标志位不阻塞（CAS 重试）最低（< 10ns）std::atomic<T>
Spinlock极短临界区、中断环境忙等（占满 CPU）低（但空转耗电）std::atomic_flag 自行实现
Mutex通用互斥让出 CPU，内核调度中（~100ns-1μs）std::mutex
RWLock读多写少读共享、写互斥中高std::shared_mutex (C++17)
Condvar生产者/消费者等待阻塞直到被唤醒中std::condition_variable
Semaphore资源计数（连接池）阻塞或超时中std::counting_semaphore (C++20)
Barrier/Latch批量等待点阻塞到所有就绪中std::barrier/latch (C++20)
Futex无竞争时零系统调用有竞争才进内核极低Linux 内部实现，mutex 基础

选择原则：

  • 临界区 < 1μs 且 CPU 充裕 → spinlock
  • 临界区较长或 CPU 紧张 → mutex
  • 读远多于写（如配置热加载） → shared_mutex
  • 只需原子更新单个变量 → atomic（最推荐）

7.3 C++11 内存模型 & Memory Order 高级难点

现代 CPU 为性能会乱序执行和重排内存操作。C++11 引入内存模型，让程序员精确控制可见性与顺序。

Memory Order语义性能典型用法
memory_order_relaxed无任何同步/顺序约束，仅原子性最快统计计数器
memory_order_consume数据依赖顺序（实践中少用，退化为 acquire）—避免使用
memory_order_acquireload：后续读写不能提前到此 load 之前快读取锁、指针
memory_order_releasestore：之前的读写不能延后到此 store 之后快写入锁、发布对象
memory_order_acq_rel读-改-写操作同时具备 acquire + release中CAS 操作
memory_order_seq_cst全局顺序一致（默认）最慢简单且对
// 经典：双重检查锁定的无锁单例 (正确版本)
std::atomic<Singleton*> s_instance{nullptr};
std::mutex s_mutex;

Singleton* getInstance() {
    Singleton* p = s_instance.load(std::memory_order_acquire); // ① acquire 读
    if (!p) {
        std::lock_guard lk(s_mutex);
        p = s_instance.load(std::memory_order_relaxed);
        if (!p) {
            p = new Singleton();
            s_instance.store(p, std::memory_order_release);   // ② release 写
        }
    }
    return p;  // acquire 保证此时 *p 构造完成可见
}

实际开发中 99% 场景用默认 seq_cst 即可。只有性能热点 + 深度 benchmark 后才考虑放宽，否则极易写错。

7.4 Cache Line & False Sharing（伪共享）

CPU 访问内存以 Cache Line（通常 64 字节）为单位。多个线程修改同一 cache line 上的不同变量时，会触发 cache 一致性协议（MESI），性能灾难式下降。

// C++17: std::hardware_destructive_interference_size
struct Counter {
    alignas(std::hardware_destructive_interference_size) long a;
    alignas(std::hardware_destructive_interference_size) long b;
};
// 大多数 x86_64 = 64，Apple M1 = 128

7.5 僵尸进程 & 孤儿进程

类型定义危害处理
僵尸 (Zombie)子进程退出，父进程未调用 wait() 回收占用 PID，耗尽后无法 fork父进程 waitpid(-1, ..., WNOHANG) 或 signal(SIGCHLD, SIG_IGN)
孤儿 (Orphan)父进程先退出，子进程还在运行无害，自动被 init (PID 1) 领养无需处理（或用 prctl(PR_SET_PDEATHSIG) 跟随父死亡）
// 优雅处理子进程退出
void sigchldHandler(int) {
    // WNOHANG: 非阻塞，循环清理所有已退出子进程
    while (waitpid(-1, nullptr, WNOHANG) > 0) {}
}
struct sigaction sa{};
sa.sa_handler = sigchldHandler;
sa.sa_flags = SA_RESTART | SA_NOCLDSTOP;
sigaction(SIGCHLD, &sa, nullptr);

7.6 信号处理的陷阱

信号处理函数（signal handler）执行时机不可控，必须是 async-signal-safe。

✅ 可在 handler 中安全使用

  • write()（注意：不是 printf！）
  • _exit()（不是 exit！）
  • sig_atomic_t 类型的原子赋值
  • kill(), sigaction()
  • 大部分 syscall 直接调用

❌ 绝对禁止

  • printf/malloc/new（非可重入）
  • std::mutex（可能死锁）
  • STL 容器操作
  • std::cout、流操作
  • 抛异常（UB）

现代方案：signalfd() 把信号转为可 epoll 的 fd，或 self-pipe trick——在 handler 里只 write(pipefd, "x", 1)，主循环用 epoll 处理真正逻辑。

7.7 高频面试题 Q&A

Q1: fork() 之后 malloc 的内存是否被复制？copy-on-write 原理？

fork 采用 写时复制（COW）：子进程初始与父进程共享所有物理页，页表项都标为只读。任一方写入时触发缺页异常，内核才真正复制那一页。

所以 fork 本身很快（只复制页表），但若子进程大量修改内存（或父进程继续修改），COW 代价会集中爆发。这也是为什么 Redis 在持久化时 fork 后，若主进程写入压力大，延迟会尖刺。

Q2: 线程栈多大？能无限创建线程吗？

Linux 默认线程栈 8 MB（ulimit -s 查看），32 位进程 3 GB 虚拟空间最多 ~380 个线程；64 位无此限制但受 RAM 和 /proc/sys/kernel/threads-max 约束。

优化：pthread_attr_setstacksize() 调小到 256 KB~1 MB。协程（C++20 coroutine）栈仅几十字节，可并发数百万。

Q3: volatile 能用于多线程同步吗？

不能！ C++ 的 volatile 只阻止编译器优化（不缓存到寄存器），不提供：

  • 原子性（读-改-写不原子）
  • 顺序性（不阻止 CPU 乱序）
  • 可见性（不刷新 cache）

多线程同步必须用 std::atomic 或 mutex。volatile 在 C++ 中的正确用途只有两个：① 映射到硬件寄存器（内存映射 I/O）；② 信号处理函数中的变量（配合 sig_atomic_t）。

Java/C# 的 volatile 等价于 C++ 的 atomic — 不要混淆！

Q4: 死锁的四个必要条件？如何预防？

四条件（Coffman）：① 互斥 ② 持有并等待 ③ 不可剥夺 ④ 循环等待。

破解方案：

  • 统一加锁顺序（最实用）— 所有代码按锁的地址/ID 升序加锁
  • std::lock(m1, m2) 或 std::scoped_lock（C++17）— 死锁避免算法
  • 用 try_lock + 超时回退
  • 减少锁粒度、用无锁数据结构
  • RAII：lock_guard/unique_lock 保证异常安全释放
Q5: 为什么 malloc 不总是向内核要内存？tcmalloc/jemalloc 优势？

glibc 的 malloc (ptmalloc) 从内核申请大块内存（brk/mmap），然后在用户态管理小块分配。释放的内存也不立即还给内核，放入 free list 复用。

tcmalloc (Google)：线程本地缓存 → 小对象分配几乎零锁竞争，高并发性能提升 2-5 倍。

jemalloc (Facebook)：降低内存碎片、更优的大对象管理，MongoDB/Redis/Rust 默认使用。

换 allocator 的方式：链接时 LD_PRELOAD=libtcmalloc.so ./prog。

Q6: 为什么进程切换比线程切换慢？

线程切换只需保存/恢复：寄存器、栈指针、PC、少量 TLS。

进程切换额外需要：

  • 切换页表（CR3 寄存器） → 所有 TLB 条目失效（除非用 PCID）
  • 切换内存映射、文件描述符表、信号处理表
  • CPU cache 内容大部分失效 → 后续内存访问延迟激增

量级差异：线程切换 1-5μs，进程切换 3-10μs，但因cache 冷却实际可感知的性能损失更严重。

Q7: 协程 vs 线程？C++20 coroutine 原理？

线程：由内核调度，抢占式，栈固定（MB 级），切换涉及系统调用。

协程：用户态调度，协作式（主动 suspend），栈可极小（字节-KB），切换是普通函数跳转。

C++20 的 co_await/co_yield/co_return 是无栈协程：编译器把函数重写成状态机，局部变量放入堆分配的 coroutine frame。一台机器能开几百万协程，用于高并发 I/O（Asio、Seastar、folly::coro）。

手写一个最小容器（不到 60 行 C++）

// minimal-container.cpp —— 演示 namespace + cgroup 隔离
// 编译: g++ -std=c++20 minimal-container.cpp -o mc
// 运行: sudo ./mc /bin/bash
#include <sched.h>           // clone, CLONE_NEW*
#include <sys/mount.h>       // mount, MS_*
#include <sys/wait.h>
#include <unistd.h>
#include <cstdio>
#include <cstring>

static int child_fn(void* arg) {
    char** argv = static_cast<char**>(arg);
    // 隔离 mnt 后必须重新挂 /proc，否则 ps 看到的还是宿主机
    sethostname("container01", 11);
    mount("proc", "/proc", "proc", 0, nullptr);
    execvp(argv[0], argv);                // 把 PID=1 让给用户进程
    return 127;
}

int main(int argc, char** argv) {
    static char stack[1 << 20];          // 子进程栈
    int flags = CLONE_NEWPID | CLONE_NEWNS | CLONE_NEWUTS
              | CLONE_NEWIPC | CLONE_NEWNET | SIGCHLD;
    pid_t pid = clone(child_fn, stack + sizeof(stack),
                       flags, argv + 1);
    waitpid(pid, nullptr, 0);
    // 解除挂载留待父进程，避免污染宿主 /proc
}

跑起来你会看到容器内 ps 只显示自己（PID=1）、hostname 是 container01、网络只有 lo。Docker 干的就是这些事，外加 cgroup 限额 + rootfs + 镜像分层。

8.3 cgroup v2 —— C++ 服务的资源天花板

cgroup v1 多层级混乱（CPU / memory / blkio 分散），v2 统一为单一层级 + controller 显式启用。systemd 默认用 v2，容器运行时 (runc、crun) 全面切到 v2。

关键文件含义设置示例
cpu.maxCPU 配额 / 周期 (μs)echo "200000 100000" → 2 核
cpu.weight相对权重 (1-10000，默认 100)echo 500
memory.max内存硬上限（超 → OOM）echo 4G
memory.high软上限（超 → 收缩 + 节流，不杀）echo 3G
memory.swap.maxswap 限额echo 0 → 禁 swap
io.max块设备 IOPS / 带宽限制echo "8:0 rbps=10M"
pids.max进程 + 线程总数限制echo 200
cpu.stat累计 CPU 用量 + throttled_usec只读，看是否被 throttle

容器内 C++ 服务必须知道的三件事

  1. std::thread::hardware_concurrency() 在容器内会撒谎：它读 sysconf(_SC_NPROCESSORS_ONLN)，返回的是宿主机核数，不是 cpu.max。需要自己解析 /sys/fs/cgroup/cpu.max（JVM 11+ 已自动适配，C++ 没有）。线程池如果按 hardware_concurrency 开 64 线程，结果 cgroup 只给 2 核 → 大量 throttle，P99 飙升。
  2. OOM 杀的可能不是你想的进程：cgroup OOM killer 在容器内挑 oom_score 最高的杀；如果你的 sidecar 内存更高，被杀的是 sidecar 而不是主进程。看 dmesg | grep -i oom 与 memory.events 文件。
  3. memory.high 触发的不是 OOM 而是「页回收 + 进程节流」：表象是 C++ 服务卡顿、syscall 变慢但没崩。memory.pressure (PSI) 文件可以观测压力。

程序里读 cgroup 限额的正确姿势

// 从 cgroup v2 读取本进程 CPU 配额，决定线程池大小
#include <fstream>
#include <sstream>
#include <thread>

unsigned effective_cpu_count() {
    std::ifstream f("/sys/fs/cgroup/cpu.max");
    if (!f) return std::thread::hardware_concurrency();
    std::string quota_s, period_s;
    f >> quota_s >> period_s;
    if (quota_s == "max") // 未限额
        return std::thread::hardware_concurrency();
    double n = std::stod(quota_s) / std::stod(period_s);
    return std::max(1u, static_cast<unsigned>(std::ceil(n)));
}

8.4 capabilities —— 把 root 切成 40 块

传统 Unix root 拥有一切权限，过于危险。Linux 把 root 拆成 40+ 个 CAP_*。容器默认只授予 14 个常用 cap（CAP_NET_BIND_SERVICE、CAP_CHOWN...），CAP_SYS_ADMIN（号称"新 root"）通常被丢弃。

常见 capability能力谁需要
CAP_NET_BIND_SERVICE绑定 <1024 端口HTTP 服务 (80/443)
CAP_NET_ADMIN改路由 / iptables / 网卡CNI 插件 / VPN
CAP_SYS_ADMINmount、namespace 创建（接近 root）容器运行时
CAP_SYS_PTRACEptrace / pidfd_getfddebugger / strace
CAP_SYS_NICE提高调度优先级 / 绑核低延时 C++ 服务
CAP_IPC_LOCKmlock 锁内存防换出数据库 / HFT
CAP_SYS_RAWIO访问 /dev/mem、ioperm裸金属驱动
CAP_BPF (5.8+)装载 BPF 程序eBPF 观测工具
低延时 C++ 服务的 cap 清单：通常需要 CAP_SYS_NICE（绑核 + RT 优先级）、CAP_IPC_LOCK（mlockall 锁页防 swap）、CAP_NET_ADMIN（DPDK / 设置 SO_BUSY_POLL）。Kubernetes 通过 SecurityContext 的 capabilities 字段授予。

8.5 seccomp-bpf —— 拦截 syscall 的最后一道闸

seccomp 用 BPF 程序过滤进程能调的 syscall。Docker 默认策略拦截了 ~44 个危险 syscall（reboot、kexec_load、open_by_handle_at...）。Chrome、Firefox、Android 的渲染进程都用 seccomp 缩到只剩几十个 syscall。

// libseccomp 写法：禁掉 execve 后任何代码都无法执行外部命令
#include <seccomp.h>

void harden() {
    scmp_filter_ctx ctx = seccomp_init(SCMP_ACT_ALLOW); // 白名单基线
    seccomp_rule_add(ctx, SCMP_ACT_KILL_PROCESS,
                     SCMP_SYS(execve), 0);
    seccomp_rule_add(ctx, SCMP_ACT_KILL_PROCESS,
                     SCMP_SYS(execveat), 0);
    seccomp_rule_add(ctx, SCMP_ACT_KILL_PROCESS,
                     SCMP_SYS(ptrace), 0);
    seccomp_load(ctx);
    seccomp_release(ctx);
}
常见生产故障：升级 glibc 后程序莫名 EPERM——根因是新 glibc 用了新 syscall（如 clone3、statx、faccessat2），而旧 Docker 的 seccomp 策略不在白名单。解决：升级 Docker 到 20.10.10+ / runc 1.0.3+，或自定义 seccomp profile 放行新 syscall。

8.6 高级工程师视角：容器内 C++ 服务排障清单

Q1: 服务在容器里 CPU 100% 但 throughput 上不去？

大概率是 CFS throttle。看 cat /sys/fs/cgroup/cpu.stat，throttled_usec 在涨说明被节流。可能原因：

  • 线程池按 hardware_concurrency() 开线程，远超 cpu.max，每周期都被切。修复见 §8.3 的代码。
  • GC / 后台线程被踢出 CPU 周期。低延时服务设置 cpu.cfs_period_us 更短（5ms）减小抖动。
Q2: 容器内 OOM 杀的不是我的进程？

cgroup memory OOM killer 在容器内挑分数最高的杀。看 cat memory.events 的 oom_kill 计数和 dmesg。如果你的服务和 sidecar/agent 共一个 cgroup，又比对方内存大，被杀的可能是别人。解决：每个进程独立 cgroup，或调整 oom_score_adj。

Q3: 程序读到的 /proc/meminfo、/proc/cpuinfo 是宿主机的？

是的——这俩文件没有 cgroup-aware。解决方案：

  • lxcfs：用户态 FUSE 文件系统，重写 /proc 文件给容器看到 cgroup 视图。
  • 自己读 cgroup 文件：cpu.max、memory.max、memory.current。
  • 升级到 cgroup v2 + Linux 5.8+ 后部分 /proc 信息自动隔离。
Q4: mlockall 失败 EPERM？

缺 CAP_IPC_LOCK，或 RLIMIT_MEMLOCK 太小。Kubernetes 加 securityContext.capabilities.add: ["IPC_LOCK"] + 通过 --ulimit memlock=-1 解除限制。

Q5: 容器内调 ptrace / strace 失败？

① Docker 默认丢 CAP_SYS_PTRACE，需 --cap-add=SYS_PTRACE；
② /proc/sys/kernel/yama/ptrace_scope 限制非父进程 ptrace（设 0 允许）；
③ seccomp 策略可能直接拒 ptrace。

Q6: 为什么 K8s 推荐 Pod 内多容器，而不是一个容器多进程？

① 进程隔离更彻底（cgroup / 资源限额独立）；
② 生命周期独立，sidecar 重启不影响主容器；
③ 镜像可独立演进；
④ K8s probes / restart policy 都是容器粒度。
但要注意：同 Pod 共享 network namespace（同 IP/端口空间），所以 sidecar 通常用 UDS / loopback 与主容器通信。
