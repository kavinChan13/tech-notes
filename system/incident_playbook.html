🚨 N08 本文

讲production incident · "出问题 5 分钟内怎么定位"。按问题分类组织 · runbook 风格。

Playbook 的核心价值：incident 发生时 you don't think · you execute。提前把"症状 → 工具 → 步骤"沉淀成肌肉记忆 · 5 分钟定位 vs 30 分钟摸索 · 这是 Senior SRE 的护城河。建议把本文 7 个 playbook 打印贴在桌前 · 1 个月内值班用 3 次就成肌肉记忆。
PB1
🔥 CPU 100% · 服务响应慢 / load avg 飙

典型症状

  • load average 远超 CPU 核数（4 核机器 load 30+）
  • 某个 / 多个进程 CPU% 接近 100% / 多核累加 > 100%
  • 请求 P99 latency 飙 · throughput 反而下降

① 快速分类

$ top                  # 找 top 进程 (按 P 排 CPU 顺序)
$ uptime               # load avg 历史
$ vmstat 1 5           # us/sy/id/wa/st 分布
  • us 高 (user CPU) → 应用代码热 → 走 ② 看应用
  • sy 高 (kernel) → syscall / context switch 多 → 走 ③ 看内核
  • wa 高 (IO wait) → 看似 CPU 高其实 IO 慢 → 跳 PB3
  • st 高 (steal time) → 虚拟机宿主在偷 CPU · 跟云厂商联系

② 应用级深入 (us 高)

$ sudo perf top -p <pid>          # 实时看哪个函数热
$ sudo perf record -F 99 -p <pid> -g -- sleep 30
$ sudo perf script | ./stackcollapse-perf.pl | ./flamegraph.pl > flame.svg
  • 看火焰图 · 平顶函数 = 热点 · 平宽函数 = 调用很多次
  • 典型场景：① 死循环（某 condition 没退出）② 正则 / hash 碰撞 ③ JSON parse 整大文档 ④ GC 风暴（Java）

③ 内核级深入 (sy 高)

$ sudo perf top -e cycles --cpu 0,1,2,3   # 看内核热点
$ pidstat -w 1                            # 进程上下文切换频率
$ sudo bpftrace -e 'tracepoint:syscalls:sys_enter_* { @[probe] = count(); }'   # 哪个 syscall 频繁
  • 典型：① 频繁 read/write（小 buffer + 大循环）② 频繁 futex（lock 抢）③ 内核 spinlock 抢（high core count 严重）

④ 缓解 + 修复模式

  • 立刻：限流（nginx limit_req / 网关）· 切流 · 重启该实例
  • 短期：扩容（K8s replicas + 1）· 升 CPU 配置
  • 修复：定位热点后 ① 算法优化 ② 加 cache ③ 异步化 ④ 退化策略（降级一些 feature）
某服务突然 CPU 100% · perf top 显示 90% 在 regex::match · 火焰图发现一个 user input 走到了 regex backtracking 路径（ReDoS · catastrophic backtracking）· 修复：用 RE2 替代 std::regex / 加 input length 上限 / 加 timeout。
PB2
💥 内存涨 / OOM kill · process 突然消失

典型症状

  • RSS 缓慢上涨 (每天 +X MB) · 不释放 = 泄漏
  • RSS 突然飙到 limit → OOM kill · dmesg | grep OOM 有记录
  • swap 使用率高 · 系统响应慢
  • 容器内 OOM 但 memory.current 看似没满 → cgroup memory 计入了 page cache

① 快速分类

$ free -h                          # 总体看 used / free / available
$ ps aux --sort=-%mem | head -10   # top 10 内存进程
$ cat /proc/<pid>/status | grep VmRSS
$ cat /proc/meminfo                # 细分 anon / cached / slab
$ dmesg | grep -i "killed process"  # OOM 历史

② 区分 4 种内存涨

  • AnonPages 涨（用户态 heap）→ 应用泄漏 · 走 ③
  • Cached / Buffers 涨（page cache）→ 正常 · OS 会自动 reclaim · 不是问题
  • Slab 涨（内核 cache）→ inode / dentry cache 多 · 看 slabtop
  • HugePages reserved 多（预留但没用）→ 计入 used 但应用没用

③ 应用泄漏定位

# Production 友好 (开销 < 5%)
$ sudo memleak-bpfcc -p <pid>            # 抓 outstanding alloc + stack trace

# C++ heap profiling (tcmalloc)
$ HEAPPROFILE=/tmp/myapp.heap LD_PRELOAD=/usr/lib/libtcmalloc.so ./myapp
$ pprof --text /tmp/myapp.heap.0001

# Java heap dump
$ jcmd <pid> GC.heap_dump /tmp/heap.hprof
$ jhat / VisualVM 分析

# Python heap (objgraph / tracemalloc)

④ OOM 调查

$ dmesg | grep -i oom -A 20    # 看被杀进程的 RSS / oom_score

# 常驻 OOM 监控 (推荐!)
$ sudo bpftrace -e '
  kprobe:oom_kill_process {
    printf("[%H:%M:%S] OOM kill: pid=%d comm=%s\n", pid, comm);
  }' | tee -a /var/log/oom.log

⑤ 缓解 + 修复模式

  • 立刻：定期 restart（cron）· 升 memory limit · K8s replicas + 1
  • 修复：① 修代码泄漏（最常见：cache 没 LRU、event listener 没 unregister）② 加 cgroup memory.high 让 throttle 不 OOM ③ 调 oom_score_adj 让关键进程不被杀
  • 预防：上线 memleak-bpfcc 持续监控 + 内存涨 alert（每日 RSS > 1.2x baseline）
某 Node.js 服务 RSS 每天涨 100MB · memleak-bpfcc 跑 24h 抓到 stack trace 在 EventEmitter · 发现某 socket disconnect 没 removeAllListeners() · 修后稳定。详见 N07 §7 cgroup memory 配置。
PB3
🐌 IO wait 高 · 服务慢但 CPU 不高

典型症状

  • vmstat 看 wa 列持续 > 20% (单核 wa > 50% 严重)
  • top 看进程 D 状态多（uninterruptible sleep · 等 IO）
  • 磁盘 util% 接近 100%

① 快速分类

$ vmstat 1 5            # wa 列
$ iostat -x 1 5         # 看 util% / await / svctm / r_await / w_await
$ iotop -o              # 哪个进程 IO 多 (-o 只显示有 IO 的)
$ ps aux | awk '$8 == "D"'   # 哪些进程在 D 状态
  • util% 接近 100% → 磁盘饱和 · 看 await
  • await > 100ms → 磁盘慢（SSD 正常 < 5ms · HDD 正常 < 30ms）
  • r/s + w/s 极高 → 应用产生 IO 太多 · 看 iotop 找进程

② 应用级 IO 模式分析

$ sudo biosnoop-bpfcc           # 实时看每个 IO 详细 (process, size, latency)
$ sudo biolatency-bpfcc 10 6    # IO 延迟直方图
$ sudo bpftrace -e 'tracepoint:block:block_rq_issue {
  @[comm, args->rwbs] = count();
}'   # 哪个进程 + 读/写 量

# 跟踪 syscall 模式 (大量小 IO vs 少量大 IO)
$ sudo strace -p <pid> -e read,write -c
  • 常见反模式：① 大量小 IO（buf 太小 / 没 fwrite buffer）② fsync 太频繁（DB 每条 commit fsync）③ 随机 IO（大文件 seek）

③ 文件系统 / 块层调查

$ cat /sys/block/sda/queue/scheduler    # 看 IO scheduler (mq-deadline / kyber / bfq / none)
$ cat /proc/diskstats                   # 详细统计
$ sudo blktrace -d /dev/sda             # Block 层 trace (高级)

④ 缓解 + 修复模式

  • 立刻：限流 / 关闭非核心 IO 任务
  • 云上：升级到 io2 / 高 IOPS volume · 或 local NVMe
  • 应用：① 合并小 IO（用 buffered IO 或 io_uring batching）② 减 fsync（用 fdatasync / batched commit）③ 大文件用 sequential read + readahead ④ 切换 Direct IO 绕过 cache（DB 类）
  • 系统：调 IO scheduler / readahead size / dirty page ratio · 详见 N01 §11
云服务 P99 偶尔 2 秒 spike · iostat 显示 sda util 偶尔 100% · biolatency 显示 P99 = 1.5s · 查云厂商 EBS volume 共享被限流 · 切 io2 高 IOPS volume + 加 IO batching · spike 消失。
PB4
📡 网络丢包 / 延迟高 / 连接异常

典型症状

  • TCP 重传率高 / ESTABLISHED 状态长 / TIME_WAIT 满
  • 用户感知慢 · ping 正常 · 但 HTTP / gRPC 慢
  • 偶发 connection refused / connect timeout

① 快速分类

$ ss -s                          # socket 总览 (TCP/UDP/SYN_RECV 等数量)
$ ss -tan state established | wc -l     # ESTABLISHED 连接数
$ ss -tan state time-wait | wc -l       # TIME_WAIT 数
$ netstat -s | grep -i retrans          # 重传数 (TCPRetrans / RTO)
$ ip -s link show                       # 网卡 RX/TX dropped + errors
$ ifconfig / ethtool -S <iface>

② 实时观测包级

$ sudo tcpdump -i any -w /tmp/dump.pcap host x.x.x.x and port 8080
$ wireshark /tmp/dump.pcap         # 离线分析

# eBPF 实时看内核 drop 原因 (神器!)
$ sudo tcpdrop-bpfcc

# 看重传 (按 src/dst 聚合)
$ sudo tcpretrans-bpfcc

# 看 TCP 连接建立 / 关闭
$ sudo tcpconnect-bpfcc

③ 常见根因

  • backlog 满（tcpdrop 看到 LISTENOVERFLOWS）→ net.core.somaxconn + listen() backlog 调大
  • TIME_WAIT 满（连接数被卡）→ net.ipv4.tcp_tw_reuse=1（client 侧）/ SO_LINGER
  • conntrack 表满（NAT 场景）→ net.netfilter.nf_conntrack_max 调大
  • 网卡 drop（ip -s 看 RX dropped）→ irq affinity / RPS / RFS 调优
  • buffer 不够（rmem/wmem 默认小）→ net.core.rmem_max + tcp_rmem

④ 缓解 + 修复模式

  • 立刻：限流 · 重启 · 加节点（横向扩容）
  • 修复：① 调 sysctl（参考下表 5 个最常用）② 上 BBR 拥塞控制 ③ SO_REUSEPORT 多 listener load balance
sysctl默认推荐含义
net.core.somaxconn1284096listen backlog 上限
net.ipv4.tcp_max_syn_backlog1288192SYN_RECV 队列
net.ipv4.tcp_tw_reuse01TIME_WAIT 复用 (client 侧)
net.ipv4.tcp_fin_timeout6015FIN_WAIT2 超时
net.ipv4.tcp_congestion_controlcubicbbr (Linux 4.9+)拥塞控制算法 · BBR 长肥管道好
电商大促 · 5xx 错误飙升 · tcpdrop-bpfcc 显示大量 LISTENOVERFLOWS drop · listen backlog 写死 128 满了 · 改成 4096 + 调 somaxconn · 错误立刻消失。详见 N04 网络编程。
PB5
📂 fd 泄漏 · "Too many open files"

典型症状

  • 报错 EMFILE: Too many open files · accept() 失败
  • /proc/<pid>/fd/ 文件越来越多
  • 服务跑几天后 socket / file 操作开始失败

① 快速分类

$ ls /proc/<pid>/fd | wc -l        # 当前 fd 数
$ cat /proc/<pid>/limits | grep "Max open files"
$ lsof -p <pid> | head -50         # 看打开了哪些
$ lsof -p <pid> | awk '{print $5}' | sort | uniq -c | sort -rn   # 按类型聚合

# 系统级
$ cat /proc/sys/fs/file-nr         # 系统 fd 总用量
  • 看 lsof 输出按类型聚合 (REG / SOCKET / pipe / FIFO 等) · 哪类多 = 哪类泄漏

② 确认是否真泄漏 (不是流量自然涨)

# 监控 fd 趋势 (每 30 秒)
$ while true; do
    echo "$(date +%s) $(ls /proc/<pid>/fd | wc -l)"
    sleep 30
  done | tee fd_trend.log
  • 稳定上涨 / 不下降 = 泄漏 · 涨跌平衡 = 正常波动

③ 定位代码源头

# bpftrace 抓 open 没对应 close 的 stack
$ sudo bpftrace -e '
  tracepoint:syscalls:sys_enter_openat /pid == TARGET/ {
    @opens[ustack] = count();
  }'

# strace 简单粗暴
$ sudo strace -p <pid> -e trace=open,openat,close 2>&1 | head -100

④ 缓解 + 修复模式

  • 立刻：临时调 ulimit (prlimit --pid X --nofile=65536) + 重启
  • 修复：① 检查 C++: RAII (智能指针 / unique_fd wrapper) ② Python: with 语句 ③ Java: try-with-resources ④ Go: defer close()
  • 常见原因：socket disconnect 没 close · HTTP client 没复用 (每次创建新 conn) · 临时文件没 unlink
某 Python 服务每周必崩一次 (EMFILE) · lsof 看到 30000 个 TIME_WAIT socket · 代码用 requests 没 Session 复用 · 改用 requests.Session() + connection pool · 问题解决。
PB6
🔒 死锁 / hang · 服务不响应但没崩

典型症状

  • HTTP / gRPC 不响应 · 连接 hang
  • thread 卡在某 lock / read · CPU 几乎 0%
  • kill TERM 没用 · 必须 kill -9

① 快速分类

$ ps -eLf | grep <process> | head    # 看线程数 + 状态
$ ps -L -p <pid> -o tid,state,cmd     # 每个线程状态

# 看哪些线程在等 (state = D 或 S)
$ cat /proc/<pid>/task/<tid>/wchan    # 在哪个内核函数等
$ cat /proc/<pid>/task/<tid>/stack    # 内核调用栈
  • D 状态（uninterruptible）= 等 IO / 等内核锁
  • S 状态（interruptible）+ wchan = futex_wait = 等 mutex / cond var
  • 大量同样 wchan = 怀疑死锁或同时等同一资源

② 获取 user-space 调用栈

# C++ - gdb attach
$ sudo gdb -p <pid>
(gdb) thread apply all bt        # 所有线程 backtrace
(gdb) info threads               # 看哪些线程
(gdb) thread <N> bt              # 看某线程

# Java - jstack
$ jstack -l <pid> > /tmp/jstack.log
# 看 BLOCKED / WAITING / 是否有 "Found one Java-level deadlock"

# Python - py-spy
$ sudo py-spy dump --pid <pid>

# Go - pprof goroutine
$ curl http://localhost:6060/debug/pprof/goroutine?debug=2

③ 死锁模式识别

  • 循环等待：T1 持 A 等 B · T2 持 B 等 A → 经典死锁
  • 同一 mutex 大量等：可能是 contention 严重 · 不是死锁但效果类似
  • IO 死锁：所有线程都在等同一文件 read / write
  • 外部依赖 hang：等下游 DB / Redis / 外部 API 永不返回

④ 缓解 + 修复模式

  • 立刻：保留 stack trace + core dump → 再 kill -9 重启
  • 修复死锁：① 全局加锁顺序约定（lock ordering）② 用 std::scoped_lock 多 mutex 一起锁 ③ 用 timeout 锁 try_lock_for ④ 减少锁粒度（细锁 / 无锁结构）
  • 修复外部 hang：所有外部调用必须有 timeout · 加 circuit breaker
  • 预防：dev 阶段用 ThreadSanitizer (-fsanitize=thread) 找潜在死锁
C++ 服务突然 hang · gdb thread apply all bt 看到 2 个线程互相 waiting on different mutex · classic ABBA 死锁 · 修复：约定全局 lock order（A 永远在 B 前）+ 用 std::scoped_lock。
PB7
GC stall / Latency spike · 偶发 200ms+ 延迟

典型症状

  • P50 / P95 正常 · P99 / P99.9 突然 spike 200-2000ms
  • 无规律 · 应用代码看不出原因
  • JVM / Go / Node.js / 任何 managed runtime 都可能
  • C++ 也可能：malloc fragmentation / page fault / 内核 RT throttle

① 快速分类 — 是 GC 还是别的

# JVM GC log (强烈推荐生产开启)
$ jvm flag: -Xlog:gc*:file=/var/log/gc.log

# Go GC trace
$ GODEBUG=gctrace=1 ./myapp

# 通用：跟应用 latency spike 时序对齐 page fault / context switch
$ sudo bpftrace -e '
  software:page-faults:1 /pid == TARGET/ { @[comm] = count(); }
  interval:s:1 { print(@); clear(@); }'

② 4 类可能元凶

  • GC pause（Java / Go）→ GC log 看 STW 时间
  • Page fault 风暴 → memory pressure / THP collapse · 看 /proc/vmstat thp_collapse
  • 调度延迟（cgroup throttle / CPU 抢）→ runqlat-bpfcc 看 runqueue 等
  • JIT compile spike（JVM / V8）→ JIT 编译期间慢
  • swap（page 被换出再被换入）→ vmstat 看 si/so
  • 外部依赖偶发慢（DB / Redis spike）→ trace 上下游

③ JVM GC 调优速查

  • GC 选型：JDK 17+ ZGC / Shenandoah（low-pause < 10ms）· JDK 11+ G1（balance）
  • 典型反模式：Old gen 用 Parallel GC + 大 heap → Full GC stall 多秒
  • 关键 flag：-Xms = -Xmx（避免动态 resize stall）· 启用 -XX:+AlwaysPreTouch 避免 page fault spike

④ 缓解 + 修复模式

  • 立刻：扩容（更多实例分担流量）· 限流 P99 用户
  • 修复：① 换 GC（ZGC / Shenandoah）② 调小 heap（更频繁但更短的 GC）③ 减分配（对象复用 / pool）④ 关 THP（DB / GC heavy 服务）⑤ mlockall + HugePage 避免 swap / page fault
  • 诊断流程：用 spike 时间戳跟 GC log + vmstat + runqlat 时序对齐 · 找元凶
Java service 偶有 1.5s spike · GC log 显示 ZGC pause 都 < 5ms · 排除 GC · bpftrace page-faults 显示 spike 时thp_collapse_alloc 频率高 · THP 在合并 page 导致 stall · 关 THP 后 spike 消失。详见 N07 §4 THP。

9.3 团队 Runbook 模板

给每个核心 service 建 runbook（wiki 一个 page）· 内容：

  • Service overview：架构图 + 核心依赖 + on-call team
