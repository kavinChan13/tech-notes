读前定位 · 为什么 Observability 是第三支柱

10 年前 SRE 三剑客是 log + metric + alert。2020 后变成 log + metric + trace（OpenTelemetry 之后）。2025+ 的 SRE 必修第四："内核级 observability" — 即 eBPF / perf / bpftrace 这条链路。

0.1 这本文 vs 现有 3 个主线

🐧 N01 Linux IO

教"怎么写"：io_uring / mmap / sendfile API 用法 · 核心概念解析。不教"怎么观测"。

⚡ P01-P0x perf-debug 主线

教C++ 应用级 perf：valgrind / sanitizer / vtune / gperftools / debug。偏 user-space + 单进程。

🔬 本文 N06

教内核级 + 跨进程 + 实时 observability：perf / eBPF / bpftrace / ftrace。看到 kernel 内部，看到所有进程。

0.2 什么场景必须用 N06 (不能用 P 主线)？

  • "哪个 syscall 慢" — bpftrace 一行查 · valgrind 看不到
  • "内核 lock 卡了" — perf record on lock_acquire · 应用级 profiler 不知道
  • "哪些进程在抢同一磁盘" — BCC biolatency-bypid · ps/top 看不出
  • "生产环境不能装 valgrind" — eBPF 无侵入 · valgrind 跑慢 10-100x
  • "延迟 spike 1 秒不知道哪里慢" — perf script + 时序对齐 · 应用 log 不够
eBPF / perf 的核心价值：不改代码 · 不重启服务 · 实时观测内核 + 所有进程。这是 production 时代不可替代的能力。10 年前内核级 trace 要重编 kernel · 现在 30 秒一条命令搞定。

工具栈全景 · 12 个核心工具 cheatsheet

先建立工具地图。下面 12 个工具 cover 95% 生产 observability 场景。按"什么场景 → 选什么"组织：

工具解决什么侵入度难度
top / htop"哪个进程吃 CPU/mem 最多"0入门
iotop"哪个进程 IO 多"0入门
ss / netstat"哪些 socket / 端口 / 连接状态"0入门
vmstat / iostat"系统级 CPU/mem/IO 趋势"0入门
strace"某进程在调啥 syscall"低 (跑慢 5-10x)入门
perf top"哪些函数实时占 CPU"低 (采样)进阶
perf record + FlameGraph"完整火焰图找热点"低进阶
ftrace / trace-cmd"内核函数 / event 流"低进阶
bpftrace"自定义内核探针 / one-liner"低 (eBPF JIT)进阶

1.1 按"症状 → 工具"决策树

  • CPU 100%？ → top 找进程 → perf top 看热点 → perf record + FlameGraph 详细分析
  • IO wait 高？ → iostat -x 1 看是否磁盘瓶颈 → iotop 找进程 → biolatency-bpfcc 看 IO 延迟分布
  • 延迟 spike 找不到原因？ → bpftrace 自定义 trace + 时序对齐 → perf sched 看是否调度延迟
  • 内存涨不停？ → memleak-bpfcc 抓泄漏 · vs valgrind 区别：BCC 跑 production · valgrind 只能 dev
  • 网络丢包？ → tcpdrop-bpfcc + tcpretrans-bpfcc 直接看内核 drop 原因
关键认知：90% 工程师停留在 top / strace · 10% 进到 perf / bpftrace · < 1% 自己写 eBPF。从 top → bpftrace 是 Senior 工程师的分水岭，cost 低（学 1 周）但 value 高（解决团队 90% 的"诊断不出"问题）。
02

perf 速通 · 火焰图工作流

2.1 perf 4 个常用子命令

perf top · 实时看哪些函数占 CPU最常用

$ sudo perf top -p <pid>          # 看某进程
$ sudo perf top                   # 系统级
$ sudo perf top -e cache-misses   # 看 cache miss 热点
  • 实时刷新（类似 top）· 默认按 CPU 周期采样 · 立刻能看出"哪个函数最热"
  • 常用 event：cycles / instructions / cache-misses / branch-misses / page-faults

perf record + report · 离线分析深度

$ sudo perf record -F 99 -p <pid> -g -- sleep 30   # 99Hz 采样 30 秒 · -g 含调用栈
$ sudo perf report                                  # 交互式查看
$ sudo perf report --stdio                          # 文本输出
  • 采样后保存 perf.data · 离线慢慢分析
  • 关键参数：-F 99 (Hz, 99 避免跟系统 timer 共振) · -g (call graph) · -p (pid) · --call-graph dwarf (无 frame pointer 时)

perf script + FlameGraph · 生成火焰图汇报神器

# 1. record
$ sudo perf record -F 99 -p <pid> -g -- sleep 30
# 2. 转 stacktrace
$ sudo perf script > out.perf
# 3. 生成 flamegraph (用 Brendan Gregg 的 FlameGraph repo)
$ ./stackcollapse-perf.pl out.perf | ./flamegraph.pl > flame.svg
  • FlameGraph 是 Senior SRE 必备 · 1 张图比 100 行 log 信息密度高
  • 读图：宽度 = CPU 占比 · 高度 = 调用栈深度 · 平顶 = 热点函数
  • 变种：off-cpu flamegraph（看哪在等 IO / lock） · 内存 flamegraph（看哪在分配内存）

perf stat · 看硬件 counter 全景优化判断

$ sudo perf stat -p <pid> sleep 10        # 看 IPC / cache miss / branch miss 等
$ sudo perf stat -e L1-dcache-load-misses,L1-dcache-loads -p <pid> sleep 10
  • 关键指标：IPC (Instructions Per Cycle, 越高越好) · cache-miss rate · branch-miss rate · context switch / 秒
  • IPC < 1 通常意味着 memory-bound · IPC > 2 是 compute-bound

2.2 一行 perf 命令查 "我的服务 hot path 是什么"

推荐工作流（30 秒拿到火焰图）
sudo perf record -F 99 -p $(pgrep myapp) -g -- sleep 30 \
  && sudo perf script | ~/FlameGraph/stackcollapse-perf.pl \
  | ~/FlameGraph/flamegraph.pl > flame_$(date +%s).svg \
  && echo "Done → flame_*.svg"
把这个写成 alias flamegraph-pid() · 团队人手必备
03

ftrace · 内核函数追踪

ftrace 是内核内置 trace 系统（不需要装额外工具）· 比 eBPF 更老但仍然好用 · 看"内核函数调用图"最快。

3.1 启用 + 基本用法

echo 控制 (传统方式)

$ cd /sys/kernel/debug/tracing
$ echo function > current_tracer       # 选 tracer 类型
$ echo do_sys_open > set_ftrace_filter   # 只跟 do_sys_open
$ echo 1 > tracing_on                    # 开始
$ cat trace                              # 看结果
$ echo 0 > tracing_on                    # 停止

trace-cmd · ftrace 高级 wrapper推荐

$ sudo trace-cmd record -p function_graph -g do_sys_open -- sleep 10
$ sudo trace-cmd report
# 或 kernelshark (GUI 看 trace 时序)
$ sudo kernelshark trace.dat
  • function_graph tracer 显示 函数调用层级 + 耗时 · 比 strace 更细
  • kernelshark 是 GUI trace viewer · 看大量 event 时序对齐神器

3.2 ftrace vs eBPF 怎么选

场景选 ftrace选 eBPF
看内核函数被调用次数 / 耗时✓—
自定义聚合 (按 pid 统计 / 直方图)—✓
不在服务器装新工具✓ (内核内置)需要 bpftrace / BCC
看 user-space 函数—✓ (uprobes)
需要 production 长期跑—✓ (高效 JIT)
04

eBPF 概念 · BPF map / probes / JIT / Verifier

eBPF = "在内核里跑 sandboxed 程序"。你写一段 BPF bytecode（一般用 C / bpftrace DSL），内核 verifier 检查安全后 JIT 成原生码运行。这是 Linux 过去 10 年最大的内核创新。

4.1 eBPF 4 个核心概念

① BPF Programs (probes)

在哪里 hook：kprobes (内核函数入口/退出) / uprobes (user 函数) / tracepoints (内核固定埋点) / USDT (用户态固定埋点) / perf_event / xdp / socket filter ...

② BPF Maps

内核 ↔ 用户态共享数据结构：hash / array / lru / histogram / per-cpu。这是聚合统计的核心 — 没有 map 就是逐条 trace。

③ Verifier

内核安全审核：拒绝无界循环 / 越界访问 / 非法指针。eBPF 不会 crash 内核（vs 内核模块可能 crash）— 这是它能 production 跑的根本原因。

④ JIT

BPF bytecode → native code。开销极低（vs 解释执行 100x 快）· 大部分 BPF program 跑起来比手写 C 慢 ~20% 但比 ftrace 快

4.2 eBPF 的 3 个使用层级

  1. L1 · 用现成 BCC 工具：opensnoop-bpfcc / biolatency-bpfcc · 一行命令 · 不用懂 eBPF (Ch6)
  2. L2 · 写 bpftrace one-liner：bpftrace -e 'kprobe:do_sys_open { @[comm] = count(); }' · 学 1 小时 · 解决 80% 自定义需求 (Ch5)
  3. L3 · 写完整 BCC / libbpf C 程序：复杂场景 + 长期运行 · 学 1 周 · 99% 工程师不需要到这层
05

bpftrace · 10 个一行 script

学会这 10 个 · 解决 80% 生产 observability 需求

bpftrace = "eBPF 的 awk" · DSL 极简 · 从看 bash 历史命令到自定义 trace 的最快路径。安装：apt install bpftrace / yum install bpftrace。

① 数所有进程的 open() 次数 (按 comm 聚合)
sudo bpftrace -e 'tracepoint:syscalls:sys_enter_openat { @[comm] = count(); }'
Ctrl-C 退出时打印聚合结果 · 立刻知道"哪些进程开文件最多"
② 抓所有 TCP 连接 (按 dest IP 统计)
sudo bpftrace -e 'kprobe:tcp_connect { @[ntop(((struct sock *)arg0)->__sk_common.skc_daddr)] = count(); }'
不用 tcpdump · 直接内核侧统计
③ 看 syscall 延迟分布（histogram）
sudo bpftrace -e '
  tracepoint:syscalls:sys_enter_read { @start[tid] = nsecs; }
  tracepoint:syscalls:sys_exit_read /@start[tid]/ {
    @ns = hist(nsecs - @start[tid]); delete(@start[tid]);
  }'
输出 read syscall 的 power-of-2 直方图 · 立刻知道 P99 在哪个数量级
④ 抓"哪个进程在用 100% CPU"（采样 CPU stack）
sudo bpftrace -e 'profile:hz:99 { @[kstack, ustack, comm] = count(); }'
类似 perf record · 但更灵活 (能加自定义过滤 / 聚合)
⑤ 监控 page fault 频率（按进程）
sudo bpftrace -e 'software:page-faults:1 { @[comm] = count(); }'
找 "memory thrashing" 元凶 · 比 ps 看 minor_faults 实时
⑥ 跟踪 OOM kill (谁触发 + 哪个被杀)
sudo bpftrace -e '
  kprobe:oom_kill_process {
    printf("OOM kill: pid=%d comm=%s rss=%dMB\n", pid, comm, ((struct task_struct *)arg0)->mm->total_vm/256);
  }'
production 强烈推荐常驻 · OOM 之后用户才发现的 cost 大
⑦ 跟踪 fd 泄漏（统计 open vs close 差异）
sudo bpftrace -e '
  tracepoint:syscalls:sys_enter_openat { @opens[tid] = count(); }
  tracepoint:syscalls:sys_enter_close { @closes[tid] = count(); }
  interval:s:10 { print(@opens); print(@closes); clear(@opens); clear(@closes); }'
每 10 秒打印开关 fd 计数 · 持续 open > close = 漏
⑧ 跟踪 TCP 重传（按 source / dest 聚合）
sudo bpftrace -e '
  kprobe:tcp_retransmit_skb {
    @[ntop(((struct sock *)arg0)->__sk_common.skc_rcv_saddr),
      ntop(((struct sock *)arg0)->__sk_common.skc_daddr)] = count();
  }'
网络丢包诊断必备 · vs ss / netstat 实时 + 按对端聚合
⑨ 跟踪 fork / clone（看进程创建模式）
sudo bpftrace -e 'tracepoint:sched:sched_process_fork { printf("%s(%d) -> %s(%d)\n", comm, pid, args->child_comm, args->child_pid); }'
debug "为什么 X 服务 process 数飙升"
⑩ 跟踪 disk IO 延迟（按设备 + 操作类型）
sudo bpftrace -e '
  kprobe:blk_account_io_start { @start[arg0] = nsecs; }
  kprobe:blk_account_io_done /@start[arg0]/ {
    @us = hist((nsecs - @start[arg0]) / 1000); delete(@start[arg0]);
  }'
disk IO P99 是多少 · 看 histogram 一目了然
把上面 10 个 one-liner 存成 alias / shell script 放到团队 wiki · 3 个月内你 + 同事 70% 性能问题能 30 秒诊断。这是 Senior SRE 的 leverage。

BCC 工具集 · 必备 12 个

已经写好的 60+ 个 eBPF 工具 · 装 bcc-tools 就有

BCC (BPF Compiler Collection) 是 Brendan Gregg 团队维护的 eBPF 工具集 · 安装 apt install bpfcc-tools · 命令后缀 -bpfcc。

命令解决什么
opensnoop-bpfcc实时看哪个进程在 open 哪些文件
execsnoop-bpfcc实时看哪个进程在 exec 哪些命令（看 cron / 后台脚本）
biolatency-bpfccBlock IO 延迟直方图（按设备 / 进程）
biosnoop-bpfcc每次 Block IO 的详细记录（trace 模式）
tcpconnect-bpfcc实时看 TCP connect 事件（src/dst/port）
tcpaccept-bpfcc实时看 TCP accept 事件
tcpretrans-bpfcc实时看 TCP 重传（诊断丢包神器）
tcpdrop-bpfcc看内核 drop 的 TCP 包 + 原因
runqlat-bpfcc线程 runqueue 等待延迟（找调度问题）
cpudist-bpfcc进程 on-CPU 时长直方图
memleak-bpfcc内存泄漏抓取（production 替代 valgrind）
profile-bpfcc系统级 CPU profile（FlameGraph 友好输出）

6.1 真实用法示例

biolatency-bpfcc · 找 disk IO 瓶颈

$ sudo biolatency-bpfcc 10 6     # 每 10 秒输出 · 共 6 轮
Tracing block device I/O...

      usecs               : count     distribution
        16 -> 31         : 0        |                                        |
        32 -> 63         : 12       |*                                       |
        64 -> 127        : 245      |***************                         |
       128 -> 255        : 632      |****************************************|
       256 -> 511        : 198      |************                            |
       512 -> 1023       : 45       |**                                      |
      1024 -> 2047       : 8        |                                        |

读图：直方图 P50 在 128-255us · P99 偶有 4-8ms · 如果 4ms+ 比例 > 1% 就是 IO 问题

07

USDT / kprobes / uprobes · 应用层 hook

前面讲的都是 内核侧 trace。但要看 用户态应用内部（如 Postgres / Java / 自己的 C++ app），需要 uprobes（动态 hook 函数）或 USDT（静态预埋）。

7.1 4 种 hook 对比

类型hook 什么开销稳定性
tracepoints内核静态埋点 (e.g. sched:sched_switch)低稳定 (内核 API)
kprobes内核函数动态 hook (e.g. do_sys_open)低-中不稳 (内核版本变可能挂)
uprobes用户函数动态 hook (e.g. malloc)中-高不稳 (符号变)
USDT用户预埋静态点 (Postgres / Java JVM 内置)低稳定 (应用 API)

7.2 自己 C++ app 加 USDT 探针

用 systemtap-sdt-dev 的 DTRACE_PROBE 宏

// 1. 头文件
#include <sys/sdt.h>

// 2. 业务代码里埋点
void process_order(int order_id, int amount) {
  DTRACE_PROBE2(myapp, order_start, order_id, amount);   // 静态点
  // ... 业务逻辑 ...
  DTRACE_PROBE2(myapp, order_done, order_id, latency_us);
}

# 3. 编译后 bpftrace 接管
$ sudo bpftrace -e 'usdt:./myapp:order_start { @[arg0] = nsecs; }'
  • 静态埋点 0 开销当 trace 关时 · 比 log printf 友好 100x
  • Postgres / MySQL / JVM / Python / Node.js 都已内置 USDT · tplist -p <pid> 列出可用 probe

5 个真实诊断 case

Case 1 · 服务 P99 延迟 spike 到 2 秒 · 不知道哪慢

现象：业务 P50 = 50ms · P99 = 2000ms · log 看不出 outlier 原因。

[Day 1] bpftrace 抓 P99 case 的 syscall： → 发现 P99 case 都伴随 fsync() 慢 (1.8s) [Day 1] biolatency-bpfcc： → Disk IO P99 = 1.5s (设备 sda1 偶尔 spike) [Day 2] 查云厂商 SLA： → 共享 EBS volume · 偶尔限流 [修复] 切到 io2 高 IOPS volume + 加 fsync batching [结果] P99 从 2000ms → 80ms
应用层 log 永远看不到 syscall 级延迟 · 必须用 bpftrace / BCC 才能定位"哪个 syscall 慢"
Case 2 · 内存涨不停 · 但 valgrind 跑不了 (production)

现象：service RSS 每天涨 200MB · valgrind 跑慢 10x 不能上 production。

[Day 1] memleak-bpfcc 直接 production 跑（开销 < 5%）： → 发现 某个 callback 注册 后没 unregister [Day 1] 看 stack trace 定位到代码行 [修复] RAII guard + unregister [结果] RSS 稳定不再涨
eBPF memleak production 友好（开销低 / 无重启）· 这是 vs valgrind 最大优势
Case 3 · 某服务 CPU 没满但响应慢 · ps 看不出问题

现象：CPU 使用率 60% · 响应慢 · top 看进程都正常。

[Day 1] runqlat-bpfcc： → 线程 runqueue 等 P99 = 50ms (异常高) [Day 1] 看 cgroup： → Service 在 cgroup 里 CPU quota 限到 60% 但没人知道 [修复] 调 cgroup CPU 配额 / 跟 SRE 重新协商 [结果] P99 从 200ms → 50ms
runqueue 等待 = "线程 ready 但 CPU 不给"的隐形瓶颈 · top 看不出 · 只有 runqlat 看得到
Case 4 · 网络丢包但不知道在哪丢的

现象：TCP 重传率高 · 不知道是发送方 / 接收方 / 中间网络丢的。

[Day 1] tcpdrop-bpfcc： → 内核侧 drop 原因 = TCP_BACKLOG_FULL [Day 1] 查 listen backlog： → code 里写死 128 · 业务高峰超了 [修复] backlog 改成 4096 + 调整 net.core.somaxconn [结果] 重传率 -95%
tcpdump 看不到内核 drop · tcpdrop-bpfcc 是这类问题唯一工具
Case 5 · OOM kill 了 service · 凶手不明

现象：dmesg 看到 OOM kill · 但不知道是谁突然涨的 mem 触发。

[第一周] 上线 bpftrace OOM 监控（Ch5 #6 one-liner）： → 常驻 logging · 下次 OOM 立刻知道触发进程 [Day 5] 再次 OOM： → log 显示一个 cron 脚本 创建了大文件加载到 mem [修复] 改 cron 用 mmap (lazy load) + 加 cgroup mem 限制 [结果] 再没 OOM
OOM 监控应该常驻 production · 不是事后才查 · bpftrace OOM script < 10 行 · 跑 24/7 几乎无开销

推荐 workflow + 跟 perf-debug 主线分工

9.1 production 问题诊断 5 步法

1
分类问题：CPU / 内存 / IO / 网络 / 调度 — 用 top / vmstat / iostat / ss 1 分钟定位大类
2
用经典工具初筛：CPU → perf top · IO → iotop + iostat -x · 网络 → ss -i + tcpretrans-bpfcc
3
定位到子系统后用 BCC 工具：IO → biolatency-bpfcc · 调度 → runqlat-bpfcc · 内存 → memleak-bpfcc
4
