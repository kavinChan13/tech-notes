01

为什么需要调度器

几千个线程抢几十个核

现代机器有几十个 CPU 核，但同时有成千上万个可运行的线程。调度器决定「下一刻哪个线程在哪个核上跑、跑多久」。它要同时兼顾几个互相冲突的目标：公平（谁都别饿死）、吞吐（少切换多干活）、响应性（交互任务要快）、确定性（实时任务不能被拖）。这些目标无法同时最优——调度就是一门取舍的艺术。

生活类比：医院分诊台

CPU 核是医生，线程是病人。分诊台（调度器）不能简单先到先看：急诊（实时任务）要插队，普通门诊（普通任务）按公平排，但也不能让谁一直排不上（防饿死）。医生频繁换病人（上下文切换）会浪费时间在「重新了解病情」（刷新 cache / TLB），所以也不能切太勤。调度器就是这个既要公平、又要救急、还要高效的分诊台。

映射：医生=CPU 核 · 病人=线程 · 急诊插队=实时优先级 · 换病人成本=上下文切换
调度器的核心难题是「公平」和「响应性」的矛盾：交互任务（键盘、鼠标）需要立刻响应但只占很少 CPU；批处理任务（编译、渲染）想吃满 CPU。好的调度器让交互任务感觉「秒响应」，同时批任务照样跑满——CFS/EEVDF 就是为此设计的。

02

调度类与优先级

Linux 有多套调度策略叠在一起

Linux 不是只有一个调度算法，而是分层的调度类（scheduling class），按优先级从高到低依次挑选：Deadline → RT → CFS/EEVDF → Idle。高优先级类里有可运行任务，就轮不到低优先级类。

调度类策略用途
DeadlineSCHED_DEADLINEEDF，按截止时间调度，最高优先级，硬实时
Real-TimeSCHED_FIFO / SCHED_RR固定优先级（1-99），实时任务
普通（CFS/EEVDF）SCHED_OTHER / SCHED_BATCH / SCHED_IDLE绝大多数进程，靠 nice 值调权重
Idle—没别的可跑时才跑（空闲任务）

nice 值：普通任务的「谦让度」

  • 范围 -20（最不谦让，占更多 CPU）到 +19（最谦让）。默认 0。
  • nice 不是「百分比」——每差 1 级 CPU 权重约变化 1.25 倍；差 10 级 ≈ 10 倍权重差。
  • 普通用户只能调高 nice（更谦让），降低（抢更多）需要权限。
优先级反转（priority inversion）陷阱：高优先级 RT 任务等一个被低优先级任务持有的锁，而低优先级任务又被中优先级任务抢占永远跑不完——高优先级被间接饿死。解法：优先级继承（PI mutex），让持锁的低优先级任务临时继承高优先级。这是 PREEMPT_RT 的关键机制。
03

CFS：完全公平调度

2007–2023 的默认调度器

CFS（Completely Fair Scheduler）的核心思想：让每个任务获得「理想公平份额」的 CPU 时间。它用 虚拟运行时间（vruntime） 追踪每个任务已经跑了多久（按权重加权），永远挑 vruntime 最小的任务跑——即「最亏欠的先跑」。

核心机制

  • vruntime：任务实际运行时间 ÷ 权重（nice 越低权重越高，vruntime 增长越慢 → 能跑更久）。
  • 红黑树：所有可运行任务按 vruntime 排序，取最左（最小）节点，选取 O(log n)。
  • 无固定时间片：目标是让所有任务在一个「调度周期」内各跑一次；任务越多，每次跑得越短（有下限 sched_min_granularity，防切换过频）。
  • 新任务/唤醒任务：vruntime 设为接近当前最小值，让刚醒来的交互任务能快速抢到 CPU（响应性）。
vruntime 像记账

把 CPU 时间想成公司发的「工时」。CFS 给每人记账：谁领的工时最少（vruntime 最小），下一份活就先给谁。nice 低的是「VIP 员工」，同样干 1 小时只记 0.5 小时账，所以能领到更多活。这样长期看每人按权重公平分配，短期又能让「刚来的人（唤醒任务）」马上有活干。

04

EEVDF：CFS 的继任者

Linux 6.6（2023）起的默认调度器

CFS 在「延迟敏感任务」上有短板：它只保证长期公平，无法很好表达「这个任务需要更低的调度延迟」。EEVDF（Earliest Eligible Virtual Deadline First）用「虚拟截止时间」显式建模延迟需求，2023 年合入内核 6.6 取代 CFS。

EEVDF 关键概念

  • lag（亏欠）：任务应得时间与实得时间的差；lag ≥ 0 才「合格（eligible）」被调度，保证公平。
  • virtual deadline（虚拟截止）：由任务请求的「时间片长度」推出——想要低延迟就请求小时间片，截止时间更早，被优先调度。
  • 选择规则：在所有 eligible 任务里，挑虚拟截止时间最早的。
  • 效果：既保持长期公平（靠 lag），又能让延迟敏感任务（小请求）获得更好响应——比 CFS 更精细。

对应用工程师的影响：多数场景无需改代码，但理解「请求小时间片 = 更低延迟、更多切换」有助于用 sched_setattr 等接口精细调优延迟敏感服务。

05

实时调度与 PREEMPT_RT

当「公平」不够，你要「确定」

普通调度追求平均表现；实时任务追求最坏情况可控（bounded latency）——宁可平均慢一点，也要保证「绝不超过 X 微秒」。这靠实时调度类实现：

策略行为
SCHED_FIFO同优先级先到先跑，跑到主动让出/阻塞/被更高优先级抢占，无时间片
SCHED_RR同优先级轮转（有时间片），否则同 FIFO
SCHED_DEADLINE指定 (runtime, deadline, period)，内核用 EDF 保证准入的任务满足截止

用 RT 调度的坑

  • 饿死风险：一个死循环的 SCHED_FIFO 任务会独占 CPU，饿死所有普通任务甚至内核线程 → 内核有 sched_rt_runtime_us 限制（默认给 RT 留 95%）作兜底。
  • 优先级反转：RT 任务等普通任务持的锁 → 必须用 PI mutex。
  • PREEMPT_RT：主线内核（自 6.12 起主流架构默认可选）把大部分内核临界区变为可抢占、中断线程化、用 PI 锁，把最坏调度延迟从毫秒级压到微秒级。详见 PREEMPT_RT 深入。
06

多核负载均衡

调度器是 per-CPU 的

为避免全局锁争用，Linux 给每个 CPU 一个运行队列（runqueue），各自独立调度。这带来新问题：负载可能不均（一个核挤爆、另一个闲着）。负载均衡器周期性/事件性地在核之间迁移任务。

关键机制

  • 调度域（sched domain）：按硬件拓扑分层（SMT 超线程 → 核 → LLC 共享缓存 → NUMA 节点），均衡优先在「代价小」的近邻层做。
  • 迁移代价：跨核迁移会丢失 cache 热度（cache miss 暴增），跨 NUMA 更贵（远端内存访问）——所以均衡有「阻尼」，不轻易迁移。
  • CPU 亲和性（affinity）：用 taskset / sched_setaffinity 把关键线程绑核，避免被迁移、保住 cache 热度——低延迟服务的常用手段。
  • isolcpus / cpuset：把某些核从通用调度中「隔离」出来专供关键任务，杜绝干扰。

NUMA 感知调度与内存本地性见 NUMA + 高级内存。

07

抢占与上下文切换

切换不是免费的
  • 抢占时机：更高优先级任务唤醒、时间片/调度周期到、系统调用返回、中断返回时，内核检查 need_resched 标志决定是否切换。
  • 内核抢占模型：PREEMPT_NONE（吞吐优先，服务器）/ PREEMPT_VOLUNTARY / PREEMPT（低延迟桌面）/ PREEMPT_RT（实时）。新内核支持动态切换。
  • 上下文切换成本：保存/恢复寄存器（快）+ 刷新 TLB、丢失 cache 热度（慢，间接成本才是大头）。切换太频繁会让「花在切换上的时间」吃掉吞吐。
  • 自愿 vs 非自愿切换：voluntary（任务自己阻塞/让出）正常；nonvoluntary（被抢占）过多说明 CPU 争用激烈——用 /proc/<pid>/status 或 pidstat -w 观察。
协程/异步之所以快，正是因为它在用户态做「切换」（只换栈/寄存器，纳秒级、无内核参与、不丢 TLB），而线程切换要陷入内核。这是 协程 相对线程的核心优势。

调优与观测

怎么看、怎么调
命令作用
nice -n 10 cmd / renice调普通任务 nice 值
chrt -f 50 cmd以 SCHED_FIFO 优先级 50 运行
taskset -c 2,3 cmd绑定到 CPU 2、3
pidstat -w 1看每进程上下文切换次数
perf sched record/latency分析调度延迟、唤醒到运行的间隔
cat /proc/<pid>/sched看单任务 vruntime、迁移、等待统计
cat /proc/schedstat各 CPU 调度统计
cgroup cpu.weight / cpu.maxcgroup v2 CPU 配额与权重（容器限流）

cgroup v2 的 CPU 控制（cpu.max 配额 / cpu.weight 权重）是容器 CPU 限制的底座，详见 容器底层原理。

09

checklist

调度相关排障 / 调优速查

  • 关键线程延迟抖动大 → taskset 绑核 + isolcpus 隔离 + 关闭该核的中断亲和。
  • 需要确定性延迟 → 用 SCHED_FIFO/SCHED_DEADLINE + PI mutex + 考虑 PREEMPT_RT。
  • RT 任务卡住其它任务 → 检查是否死循环独占；确认 sched_rt_runtime_us 兜底。
  • 非自愿上下文切换高（pidstat -w）→ CPU 争用，考虑减少线程数 / 绑核 / 提优先级。
  • 跨 NUMA 性能差 → 绑核 + 内存本地化（见 NUMA 篇）。
  • 容器 CPU 被限速（throttled）→ 检查 cpu.max 配额与 cpu.stat 的 throttled 计数。
一句话记忆：普通任务靠 CFS/EEVDF 求公平，实时任务靠 RT/Deadline 求确定，低延迟靠绑核 + 隔离 + 优先级 + PREEMPT_RT 消抖动。
