fd 的本质：为什么是一个 int

1.1 从用户视角看 fd · 1.2 为什么内核不直接返回指针？

1.1 从用户视角看 fd

在用户空间，fd 就是一个 非负小整数，遵循三条铁律：

  • 0, 1, 2 预留：stdin / stdout / stderr
  • 从 3 开始递增：open/socket/pipe 等返回当前进程最小可用的非负整数（POSIX 强制要求）
  • 进程私有：进程 A 的 fd=5 与进程 B 的 fd=5 毫无关系（除非通过 SCM_RIGHTS 传递）
int fd1 = open("/etc/hosts", O_RDONLY);      // 3 (最小可用)
int fd2 = open("/etc/passwd", O_RDONLY);     // 4
close(fd1);                                 // 3 被释放
int fd3 = open("/etc/group", O_RDONLY);      // 3 (复用！)

1.2 为什么内核不直接返回指针？

✓安全隔离

内核指针暴露给用户态是巨大的安全漏洞（KASLR 就是为了防止这个）。int 是进程局部的抽象句柄，用户无法通过它做任何越权操作。

✓引用计数

多个 fd 可以指向同一个内核对象（dup() / fork()），int 句柄让引用计数、生命周期管理变得简单。

✓ABI 稳定

int 是跨架构/内核版本稳定的 ABI。结构体布局可以演化，但 open() 返回 int 这个契约永远不变。

✓表索引即句柄

fd 本质就是 files_struct::fd_array[] 的数组下标，查找 O(1)。

2.3 为什么是三层而不是两层？

▶ 设计动机

第一层（FD Table）是进程私有的，保证进程隔离；第三层（Inode）是文件实体，必须是系统唯一的（否则硬链接无法共享元数据）。

关键是第二层（Open File Table）的存在意义：

  1. 同一文件可被多次独立打开——每次 open() 都产生新的 struct file，各自有独立的 pos。所以两个进程分别 open 同一文件读取，偏移量互不干扰。
  2. 多个 fd 可共享同一 struct file——dup()、fork() 让多个 fd 指向同一个 struct file，共享偏移量。这是 shell 管道/重定向的基础。
  3. O_APPEND 原子追加——每次 write 都定位到文件末尾，需要 struct file 持有 pos 并持锁。

内核数据结构源码

3.1 task_struct → files_struct · 3.2 files_struct 与 fdtable · 3.3 struct file —— 第二层的核心 · 3.4 struct inode —— 第三层

3.1 task_struct → files_struct

进程控制块 task_struct 通过 files_struct* 指向 FD 表：

// include/linux/sched.h (简化)
struct task_struct {
    pid_t                pid;
    struct mm_struct*   mm;        // 内存描述符
    struct files_struct* files;     // ← FD 表指针
    struct fs_struct*    fs;        // 当前目录 cwd/root
    // ...
};

3.2 files_struct 与 fdtable

// include/linux/fdtable.h (简化)
struct files_struct {
    atomic_t       count;              // 引用计数 (线程共享时增加)
    struct fdtable* fdt;                // 指向当前 fdtable (可动态扩展)
    struct fdtable  fdtab;              // 初始嵌入的 fdtable
    spinlock_t     file_lock;          // 保护 fd 操作
    unsigned int   next_fd;            // 下次分配的 fd 提示
    struct file __rcu* fd_array[NR_OPEN_DEFAULT];  // 初始 64 个槽位
};

struct fdtable {
    unsigned int      max_fds;          // 当前容量
    struct file __rcu** fd;              // fd 数组 (可动态 realloc)
    unsigned long*     close_on_exec;    // FD_CLOEXEC 位图
    unsigned long*     open_fds;         // 已打开 fd 位图
    unsigned long*     full_fds_bits;    // 辅助查找空闲 fd
    struct rcu_head    rcu;
};
▶ 性能细节
  • 初始 64 槽位嵌入：fd_array[NR_OPEN_DEFAULT] 直接嵌入 files_struct，绝大多数进程 fd < 64 时零额外分配
  • 动态扩容：超过 64 时分配新的 fdtable，旧的通过 RCU 延迟释放
  • RCU 保护：读者（fget）无锁，写者（close/dup）持 spinlock
  • 位图加速：open_fds 位图 + full_fds_bits 辅助使分配空闲 fd 是 O(log n)

3.3 struct file —— 第二层的核心

// include/linux/fs.h (简化)
struct file {
    struct path               f_path;        // dentry + vfsmount
    struct inode*             f_inode;       // ← 指向 inode
    const struct file_operations* f_op;     // 函数表: read/write/poll...

    atomic_long_t              f_count;       // 引用计数 (dup/fork 增加)
    unsigned int               f_flags;       // O_RDONLY / O_APPEND / O_NONBLOCK
    fmode_t                    f_mode;        // FMODE_READ / FMODE_WRITE
    struct mutex              f_pos_lock;    // 保护 f_pos 的原子性
    loff_t                     f_pos;         // ← 当前文件偏移量!
    struct fown_struct        f_owner;       // SIGIO 的目标进程

    void*                      private_data;  // 驱动私有数据
    struct address_space*     f_mapping;     // page cache
};

// file_operations 是多态的核心 —— "一切皆文件"就靠它
struct file_operations {
    ssize_t (*read) (struct file*, char __user*, size_t, loff_t*);
    ssize_t (*write)(struct file*, const char __user*, size_t, loff_t*);
    int     (*mmap) (struct file*, struct vm_area_struct*);
    __poll_t (*poll)(struct file*, struct poll_table_struct*);
    long    (*unlocked_ioctl)(struct file*, unsigned, unsigned long);
    int     (*release)(struct inode*, struct file*);    // close
    // ...
};
▶ 这就是 C 语言版的多态

每种文件类型（ext4 / socket / pipe / procfs / tmpfs）都提供自己的 file_operations 实现。sys_read 最终调用的是 file->f_op->read()——一个函数指针。这就是"一切皆文件"的本质：统一的 vtable。

3.4 struct inode —— 第三层

struct inode {
    umode_t                  i_mode;       // 类型 + 权限 (S_IFREG/S_IFDIR...)
    kuid_t                   i_uid;
    kgid_t                   i_gid;
    loff_t                   i_size;       // 文件大小
    struct timespec64       i_atime, i_mtime, i_ctime;
    unsigned long            i_ino;        // inode 号 (ls -i 看到的)
    unsigned int             i_nlink;      // 硬链接计数
    atomic_t                 i_count;      // in-memory 引用计数

};

dup/dup2/dup3 语义

4.1 三个 API 的差异 · 4.2 dup 的三级表变化 · 4.3 dup2 实现 Shell 重定向

4.1 三个 API 的差异

API行为典型用法
dup(oldfd)返回最小空闲 fd，指向同一 struct file简单复制
dup2(oldfd, newfd)newfd 若已打开先 close，然后 newfd → oldfd 的 fileShell 重定向
dup3(oldfd, newfd, flags)同 dup2，但支持 O_CLOEXEC；且 oldfd==newfd 时报错需要原子设置 CLOEXEC

4.2 dup 的三级表变化

// dup 的内核实现 (极简化版)
SYSCALL_DEFINE1(dup, unsigned int, fildes) {
    struct file* file = fget_raw(fildes);          // file->f_count++
    int newfd = get_unused_fd_flags(0);             // 找最小空闲 fd
    fd_install(newfd, file);                         // 填入 FD Table
    return newfd;
}
⚠ 关键点：dup 只改第一层，不改第二层

dup 创建的是 FD Table 的新条目，指向同一个 struct file。所以 pos 共享——两个 fd 写入会累加到同一个位置。这与"两次 open 同一文件"是完全不同的！

4.3 dup2 实现 Shell 重定向

// 实现 `ls > output.txt`
int fd = open("output.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
dup2(fd, STDOUT_FILENO);   // fd=1 (stdout) 现在指向 output.txt
close(fd);                 // 原始 fd 可关闭，stdout 仍指向 file
execvp("ls", argv);       // ls 的 printf 输出就写到文件了

5.2 关键规则

  • fork() 复制 FD Table 本身（子进程有独立的 files_struct）
  • 但 FD Table 里每个指针指向的 struct file 不复制，只是 refcount++
  • 因此父子进程共享 struct file 的偏移量 pos——父进程 write 后文件位置前进，子进程下次 write 从新位置继续
  • 子进程的 close(fd) 只减少 FD Table 条目，refcount--；只有 refcount 归零时 struct file 才真正释放

5.3 验证实验

int fd = open("/tmp/test.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
if (fork() == 0) {
    // 子进程
    write(fd, "child\n", 6);   // pos: 0 → 6
    exit(0);
}
wait(nullptr);
write(fd, "parent\n", 7);       // pos: 6 → 13 (不是从 0 开始！)

// 文件内容:
// child
// parent
✗ 竞争条件：并发 write 到共享 fd

父子进程同时 write 到共享 fd 时，pos 的原子性由内核保证（f_pos_lock），但大块 write 可能被拆分（Linux 下 PIPE_BUF=4096 以内是原子的，普通文件则取决于文件系统）。生产代码应当用 O_APPEND——内核保证 lseek+write 的原子性。

07

exec 与 FD_CLOEXEC

6.1 exec 默认行为：fd 保留 · 6.2 FD_CLOEXEC 标志 · 6.3 close_on_exec 位图在内核的位置

6.1 exec 默认行为：fd 保留

与直觉相反，execve() 默认不会关闭进程的 fd——这是 Shell 重定向能工作的前提（先 fork+dup2 再 exec）。

6.2 FD_CLOEXEC 标志

某些 fd（如密钥、日志）不应该被子进程继承，这时需要设置 close-on-exec：

// 方式 1: fcntl 设置 (非原子)
int fd = open("secret.key", O_RDONLY);
fcntl(fd, F_SETFD, FD_CLOEXEC);   // ← 窗口期有竞争!

// 方式 2: open 时原子设置 (推荐)
int fd = open("secret.key", O_RDONLY | O_CLOEXEC);

// 方式 3: 新 API 都有 O_CLOEXEC 对应
int sock  = socket(AF_INET, SOCK_STREAM | SOCK_CLOEXEC, 0);
int ep    = epoll_create1(EPOLL_CLOEXEC);
int pfd[2]; pipe2(pfd, O_CLOEXEC);
int efd   = eventfd(0, EFD_CLOEXEC);
int dirfd = openat(AT_FDCWD, "path", O_DIRECTORY | O_CLOEXEC);
⚠ 多线程程序必须用 O_CLOEXEC

在线程 A 执行 open + fcntl(F_SETFD) 两步之间，线程 B 可能调用 fork+exec，导致 fd 被泄漏到子进程。只有 O_CLOEXEC（open 时原子设置）能完全避免这个窗口。这也是 C++17 std::filesystem 内部始终使用 O_CLOEXEC 的原因。

6.3 close_on_exec 位图在内核的位置

每个 fdtable 有一个 close_on_exec 位图（每 bit 对应一个 fd）。execve() 的最后阶段调用 do_close_on_exec() 遍历位图关闭对应 fd。

7.3 最小实现

// 实现: ls | grep foo
int pfd[2];
pipe(pfd);

if (fork() == 0) {             // ls 子进程
    close(pfd[0]);            // ls 不读管道
    dup2(pfd[1], STDOUT_FILENO);// stdout → 管道写端
    close(pfd[1]);
    execlp("ls", "ls", NULL);
}
if (fork() == 0) {             // grep 子进程
    close(pfd[1]);            // grep 不写管道
    dup2(pfd[0], STDIN_FILENO); // stdin ← 管道读端
    close(pfd[0]);
    execlp("grep", "grep", "foo", NULL);
}
close(pfd[0]); close(pfd[1]);  // 父进程两端都关，否则 EOF 永远不会到达
wait(nullptr); wait(nullptr);
▶ 为什么必须关闭不用的管道端？

管道的读端在所有写端都被关闭后才返回 EOF。如果父进程或任一子进程忘了关写端，grep 的 read 会永远阻塞——这是 shell/IPC 编程中最经典的 bug 之一。

9.2 经典组合：eventfd + epoll 实现唤醒

// 线程 A 阻塞在 epoll_wait，线程 B 如何唤醒它？答：eventfd
int efd = eventfd(0, EFD_NONBLOCK | EFD_CLOEXEC);
int ep = epoll_create1(EPOLL_CLOEXEC);

epoll_event ev{.events = EPOLLIN, .data.fd = efd};
epoll_ctl(ep, EPOLL_CTL_ADD, efd, &ev);

// 线程 A
epoll_event events[32];
int n = epoll_wait(ep, events, 32, -1);
for (int i = 0; i < n; ++i) {
    if (events[i].data.fd == efd) {
        uint64_t val;
        read(efd, &val, sizeof(val));   // 消费事件
        // 处理外部任务
    }
}

// 线程 B (任意线程，甚至信号处理器)
uint64_t one = 1;
write(efd, &one, sizeof(one));       // 唤醒 epoll_wait

9.3 经典组合：timerfd 取代 setitimer

// 传统 setitimer: 信号处理器有严格限制
// timerfd: 定时到期时 fd 变可读，统一用 epoll 处理
int tfd = timerfd_create(CLOCK_MONOTONIC, TFD_NONBLOCK | TFD_CLOEXEC);
struct itimerspec ts = {
    .it_interval = {.tv_sec = 1, .tv_nsec = 0},    // 每秒触发
    .it_value    = {.tv_sec = 1, .tv_nsec = 0}
};
timerfd_settime(tfd, 0, &ts, NULL);
epoll_ctl(ep, EPOLL_CTL_ADD, tfd, &(epoll_event){.events=EPOLLIN, .data.fd=tfd});
// epoll_wait 会在每秒唤醒，read(tfd) 获取到期次数

9.4 signalfd —— 让信号「排队」并进入 epoll

传统 signal()/sigaction() 的最大痛点：信号处理函数只能调 async-signal-safe 的函数（printf、malloc、几乎所有 STL 都不行）。signalfd 把信号变成一个 fd 上的"可读事件"，主循环用普通函数处理。

// 1. 先 block 想接管的信号（必须！否则默认 handler 仍会跑）
sigset_t mask;
sigemptyset(&mask);
sigaddset(&mask, SIGINT);
sigaddset(&mask, SIGTERM);
sigaddset(&mask, SIGHUP);
pthread_sigmask(SIG_BLOCK, &mask, nullptr);   // 多线程必须用 pthread_sigmask

// 2. 建立 signalfd
int sfd = signalfd(-1, &mask, SFD_NONBLOCK | SFD_CLOEXEC);
epoll_ctl(ep, EPOLL_CTL_ADD, sfd, &(epoll_event){.events=EPOLLIN, .data.fd=sfd});

// 3. 主循环里像普通 fd 一样处理
if (events[i].data.fd == sfd) {
    signalfd_siginfo si;
    read(sfd, &si, sizeof(si));
    switch (si.ssi_signo) {
        case SIGHUP:  reload_config();  break;     // 现在可以放心 malloc / log 了
        case SIGINT:
        case SIGTERM: g_should_exit = true; break;
    }
}
⚠ 三个常见坑
  • 忘记 block：必须 SIG_BLOCK 才能让信号"排队进 signalfd"，否则信号被默认 handler 抢走，signalfd 永远收不到。
  • 多线程用 sigprocmask：在多线程程序里 sigprocmask 行为未定义，必须用 pthread_sigmask，并且要在创建任何线程之前统一 block。
  • SIGKILL / SIGSTOP 不能 catch：signalfd 也救不了，这俩永远走内核直杀路径。

9.5 pidfd —— 进程的「稳定句柄」，告别 PID 重用 race

Linux 5.3 引入 pidfd 解决了一个困扰几十年的问题：PID 会重用。当你 kill(pid, SIGTERM) 时，目标进程可能已经死了，PID 被别的进程占用，你误杀了无辜者。pidfd 通过持有内核 task 引用确保句柄"指向同一个进程"。

✗传统 kill(pid)
pid_t pid = fork();
if (pid == 0) exec(...);
// ... 几秒钟后
kill(pid, SIGTERM);   // pid 可能已被回收并复用！
✓现代 pidfd
int pfd = pidfd_open(pid, 0);
// ... pfd 锁住 task_struct，pid 永不重用
pidfd_send_signal(pfd, SIGTERM, nullptr, 0);
// 即使 pid 被复用，信号也只送给原来的进程

pidfd 三大能力（5.3 / 5.4 / 5.6 逐步完善）

// ① 等待子进程（poll 即可，无需 waitpid 阻塞主线程）
int pfd = pidfd_open(child_pid, 0);
epoll_ctl(ep, EPOLL_CTL_ADD, pfd, &(epoll_event){.events=EPOLLIN, .data.fd=pfd});
// 子进程退出时 pfd 变可读，主循环统一处理

// ② 替代 kill —— 无 race
pidfd_send_signal(pfd, SIGTERM, nullptr, 0);

// ③ 从另一进程偷一个 fd 过来（5.6+，需 CAP_SYS_PTRACE）
int dup_fd = syscall(SYS_pidfd_getfd, pfd, target_fd_in_other_proc, 0);
// 用例：debugger、container runtime、checkpoint/restore
▶ 工程价值
  • systemd：从 v245 起全面用 pidfd 管理 unit，不再用 PID 文件这种脆弱的机制。
  • Docker / containerd / Kubernetes：监控容器内 1 号进程退出，pidfd + epoll 是首选，比 SIGCHLD 信号简洁可靠。
  • 进程管家 / supervisor：自写 supervisord 类组件必备。

9.6 memfd_create —— 匿名共享内存的现代姿势

传统跨进程共享内存有 3 条路：① shm_open（POSIX，在 /dev/shm 留名字）；② shmget（SysV，全局 key 命名空间难管理）；③ mmap(MAP_ANONYMOUS|MAP_SHARED) + fork。三种各有缺陷。memfd_create 是无名字、可传递、可"封印"的新方案。

// 1. 创建匿名 memfd（没有路径，不污染 /dev/shm）
int mfd = memfd_create("shared-ring", MFD_CLOEXEC | MFD_ALLOW_SEALING);
ftruncate(mfd, 4096 * 1024);

// 2. mmap 到自己地址空间
void* buf = mmap(nullptr, 4096 * 1024, PROT_READ|PROT_WRITE, MAP_SHARED, mfd, 0);

// 3. 通过 UDS + SCM_RIGHTS 把 mfd 传给另一进程（见 §10）
send_fd(uds, mfd);

// 4. （可选）封印 —— 锁定大小 / 锁定只读
fcntl(mfd, F_ADD_SEALS, F_SEAL_SHRINK | F_SEAL_GROW | F_SEAL_WRITE);
// 此后任何进程 ftruncate / write 都会 -EPERM，
// 即使是恶意进程拿到 fd 也不能改大小

  • dbus-broker：消息超阈值时用 memfd 替代正文，省内核缓冲。

fd 传递 (SCM_RIGHTS)

Unix Domain Socket 可以在进程间传递 fd——不是复制数字 3

Unix Domain Socket 可以在进程间传递 fd——不是复制数字 3，而是让另一进程获得一个指向同一 struct file 的 fd。这是 systemd socket activation、Chrome 多进程架构、Docker containerd 的基础。

graph LR subgraph A["进程 A"] A1["fd=3 → file X"] end subgraph K["内核 (UDS)"] UDS["Unix Socket\n传递 SCM_RIGHTS"] end subgraph B["进程 B"] B1["fd=7 → file X (recv 后)"] end subgraph FILE["Open File Table"] X["struct file X\nrefcount=2 (fd 传递后)"] end A1 --> X B1 --> X A1 -.->|"sendmsg(SCM_RIGHTS)"| UDS UDS -.->|"recvmsg"| B1 
// 发送端: 把 fd 通过 UDS 传给另一进程
void send_fd(int uds, int fd_to_send) {
    char buf[CMSG_SPACE(sizeof(int))];
    struct msghdr msg = {0};
    msg.msg_control = buf;
    msg.msg_controllen = sizeof(buf);

    struct cmsghdr* cmsg = CMSG_FIRSTHDR(&msg);
    cmsg->cmsg_level = SOL_SOCKET;
    cmsg->cmsg_type  = SCM_RIGHTS;
    cmsg->cmsg_len   = CMSG_LEN(sizeof(int));
    *(int*)CMSG_DATA(cmsg) = fd_to_send;

    char dummy = 'x';
    struct iovec io = {&dummy, 1};
    msg.msg_iov = &io;
    msg.msg_iovlen = 1;
    sendmsg(uds, &msg, 0);
}

// 接收端
int recv_fd(int uds) {
    char buf[CMSG_SPACE(sizeof(int))];
    char dummy;
    struct iovec io = {&dummy, 1};
    struct msghdr msg = {.msg_iov=&io, .msg_iovlen=1,
                          .msg_control=buf, .msg_controllen=sizeof(buf)};
    recvmsg(uds, &msg, 0);
    struct cmsghdr* cmsg = CMSG_FIRSTHDR(&msg);
    return *(int*)CMSG_DATA(cmsg);
}
▶ 典型应用
  • systemd socket activation：systemd 先 bind/listen，把 listen fd 传给 daemon 进程。daemon 崩溃重启不丢连接。
  • Chrome 多进程：Browser 进程打开文件后把 fd 传给沙箱中的 Renderer（Renderer 没有文件系统权限）。
  • Nginx 平滑升级：新 master 启动后从老 master 接管 listen fd，零停机升级。

11.2 fd 泄漏检测

// RAII 包装避免泄漏
class FdGuard {
    int fd_;
public:
    explicit FdGuard(int fd) : fd_(fd) {}
    ~FdGuard() { if (fd_ >= 0) ::close(fd_); }
    FdGuard(FdGuard&) = delete;
    FdGuard(FdGuard&& o) noexcept : fd_(o.fd_) { o.fd_ = -1; }
    int get() const { return fd_; }
    int release() { int f = fd_; fd_ = -1; return f; }
};

// 使用
FdGuard fd{open("/tmp/x", O_RDONLY | O_CLOEXEC)};
if (fd.get() < 0) return -1;
// 任何路径退出都自动 close
✗ 常见泄漏来源
  • 异常路径忘记 close：早期 C 代码常见，用 RAII 解决
  • exec 后遗留：未设 FD_CLOEXEC，fd 被意外继承到子进程
  • popen/fopen 忘配对 close
  • epoll 事件处理中 accept 的 fd 没加入管理
  • 第三方库 bug：如某些 HTTP 客户端连接池泄漏

高频面试题

什么是文件描述符？它在内核中对应什么数据结构？
fd 是进程级的非负小整数，是 files_struct::fdtable::fd[] 的数组下标。它通过三级表定位到实际资源：
  • ① FD Table（进程私有，files_struct）：fd → struct file*
  • ② Open File Table（系统级，struct file）：保存 pos、flags、refcount
  • ③ Inode Table（struct inode）：文件大小、权限、数据块位置
关键点：fd 是索引，不是指针，保证安全隔离；多个 fd 可以指向同一 struct file（dup/fork），共享偏移量。
两次 open 同一个文件，和 dup 同一个 fd，有什么区别？
两次 open：每次都创建独立的 struct file，两个 fd 有独立的 pos。一个 fd 写入不影响另一个 fd 的读位置。
dup(fd)：只在 FD Table 中新增一个指针，指向同一个 struct file，f_count++。两个 fd 共享 pos 和 flags——一个写入会前进共同的偏移量。
工程含义：shell 重定向 2>&1 用的是 dup2，让 stderr 和 stdout 共享偏移，输出不会互相覆盖；而并行读取同一文件的不同区域需要各自 open。
fork() 之后父子进程的 fd 是什么关系？
fork 复制了父进程的 files_struct，但其中每个指针指向的 struct file 不复制，只把 f_count++。所以：
  • 父子进程的 FD Table 是独立的——子进程 close 不影响父进程
  • 但指向的 struct file 是共享的——共享 pos 和 flags
  • 父子 write 同一 fd，偏移量会累加（不会互相覆盖）
  • 只有所有 refcount 归零时，内核才真正释放 struct file 和 inode 的打开状态
这是 shell 管道的基础。vfork 更极端——甚至连 FD Table 都共享。
为什么需要 FD_CLOEXEC？什么场景下必须用它？
execve 默认保留所有 fd（shell 重定向依赖这一点）。但很多场景 fd 不应该被子进程继承：密钥文件、监听 socket、日志 fd、审计记录等。
必须用 O_CLOEXEC（原子设置）的场景：
  • 多线程程序：open + fcntl(FD_CLOEXEC) 两步之间有窗口期，其他线程可能 fork+exec 泄漏 fd
  • 库代码：库不知道调用者是否多线程，默认应用 O_CLOEXEC
  • 安全敏感 fd：防止恶意子进程通过 /proc/self/fd/N 读取本应隔离的资源
为什么 socket、pipe、eventfd 都是 fd？这样设计有什么好处？
Linux 把几乎所有可读写的内核对象都抽象为 fd，统一通过 read/write/close/epoll 操作。技术上靠 struct file::f_op 函数表实现——每种资源提供自己的 file_operations。
好处：
  • 统一接口：同一个 event loop 可以同时处理网络、定时器、信号、文件变更
  • 组合性：timerfd + signalfd + eventfd + socket 全部注册到一个 epoll，单线程驱动一切
  • RAII 友好：所有资源生命周期用同一套 close 管理
  • /proc/fd/ 可视化：ops、SRE 能用同一套工具排查问题
你怎么实现 Shell 的 "ls | grep foo | wc -l"？
关键是两次 pipe + 三次 fork + 正确的 dup2 编排：
  • 创建 pipe1 (ls→grep) 和 pipe2 (grep→wc)
  • fork ls：关闭所有不用的端，dup2(pipe1_write, 1)，exec ls
  • fork grep：dup2(pipe1_read, 0) 作为 stdin，dup2(pipe2_write, 1)，exec grep
  • fork wc：dup2(pipe2_read, 0)，exec wc
  • 父进程关闭所有 pipe fd（否则写端永不 EOF，grep/wc 永不结束）
核心是：任何进程必须关闭它不用的所有管道端——这是内存泄漏/死锁的根源。
C10K 问题为什么不能用一个进程 1024 个 fd 的 select 解决？
三层障碍：
  • fd 上限：select 的 FD_SETSIZE=1024（编译期硬编码），poll/epoll 无此限制
  • 系统调用开销：select 每次调用都要把 fd_set 从用户态拷贝到内核态，内核再 O(n) 遍历所有 fd 检查状态。10000 fd 时 CPU 大量消耗在这
  • 内核通知机制：select/poll 是"内核主动扫描"，epoll 是"数据到达时回调把 fd 加入就绪链表"，后者是 O(1) 获取就绪事件
解决 C10K/C10M 的标准答案是 epoll(ET) + 非阻塞 I/O + 多 Reactor + SO_REUSEPORT；C100K 以上可能需要 io_uring。
一个进程最多能打开多少个 fd？
三层限制依次生效：
  • 进程级软限制（RLIMIT_NOFILE）：ulimit -n，默认 1024，可用 setrlimit 调大到硬限制
  • 进程级硬限制：普通用户不能突破，需 root（通常 4096 或 1048576）
  • 系统级：/proc/sys/fs/file-max，所有进程 fd 总和
  • 物理限制：每个 struct file 约 256 字节，100 万 fd ≈ 256MB 内核内存
生产服务器常见配置：ulimit -n 1048576 + fs.file-max=2097152。systemd 服务额外配置 LimitNOFILE=infinity。
如何实现进程间传递一个已打开的 socket fd？
通过 Unix Domain Socket 的 SCM_RIGHTS 辅助消息（ancillary data）：
  • 发送端构造 msghdr，在 cmsghdr 中设置 cmsg_type=SCM_RIGHTS，data 是要传的 fd 整数
  • 用 sendmsg() 通过 UDS 发送
  • 接收端用 recvmsg()，从 CMSG_DATA 取出 fd
关键点：内核在 recv 时分配新 fd（数字不同但指向同一 struct file），引用计数 +1。
应用：systemd socket activation（daemon 从 systemd 继承 listen fd）；Chrome 沙箱（渲染进程从主进程获取文件 fd）；Nginx 平滑升级（新 master 接管老 master 的 listen fd）。
