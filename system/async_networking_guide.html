Reactor / Proactor / Sender-Receiver 三大范式对比

三种范式 · 三者核心差异表 · Reactor 详细流程图 · Proactor 详细流程图（io_uring） · Sender/Receiver 详细流程

2.1 三种范式

Reactor

"反应"模型：主循环 epoll_wait → IO 就绪事件 → 触发用户 callback → callback 内执行同步 IO。

典型：libevent / libuv / Nginx / Redis / Asio (默认 Linux)。

Proactor

"主动"模型：用户提交 IO 请求 → 内核完成 IO 后通知 → callback 直接拿到数据。

典型：Windows IOCP / Linux io_uring / Asio (Windows)。

Sender/Receiver

"代数化异步"：把 IO 操作描述成 sender，组合成管道；scheduler 决定何时何地跑。结构化并发 + 取消传播。

典型：libunifex / stdexec / mvfst / 未来的 std::execution。

2.2 三者核心差异表

项ReactorProactorSender/Receiver
IO 触发时机就绪（用户再读）完成（用户直接拿数据）取决于底层
buffer 持有方用户用户提交时给内核抽象化
组合复杂操作难（嵌套回调）难（嵌套回调）容易（管道算子）
取消语义手动管理手动管理结构化自动传播

2.3 Reactor 详细流程图

// 主循环
while (running) {
    int n = epoll_wait(ep, events, MAX, timeout_ms);
    for (int i = 0; i < n; ++i) {
        auto* handler = (Handler*)events[i].data.ptr;
        if (events[i].events & EPOLLIN)  handler->on_read_ready();
        if (events[i].events & EPOLLOUT) handler->on_write_ready();
    }
}

// 用户 handler 内：自己 read/write
void on_read_ready() {
    int n = read(fd, buf, BUFSZ);     // 不会阻塞 (因为 ready)
    process(buf, n);
}

2.4 Proactor 详细流程图（io_uring）

// 用户提交 IO 请求 + buffer
auto* sqe = io_uring_get_sqe(&ring);
io_uring_prep_read(sqe, fd, buf, len, offset);
io_uring_sqe_set_data(sqe, ctx);
io_uring_submit(&ring);

// 主循环 wait completion
while (running) {
    io_uring_cqe* cqe;
    io_uring_wait_cqe(&ring, &cqe);
    auto* ctx = (Ctx*)io_uring_cqe_get_data(cqe);
    // cqe->res 是 read 字节数, buf 已被内核填好
    process(ctx, cqe->res);
    io_uring_cqe_seen(&ring, cqe);
}
Proactor 优势：① 一次 syscall 完成"提交+检查"（io_uring 甚至零 syscall）；② 内核直接 DMA 到用户 buffer，省一次 copy；③ batch 操作天然支持。
劣势：① buffer 生命周期管理复杂（提交后到完成期间不能释放）；② 错误处理路径长。

2.5 Sender/Receiver 详细流程

namespace ex = std::execution;

auto work = ex::just()
    | ex::let_value([&] { return async_read(sock, buf); })
    | ex::then(parse_request)
    | ex::let_value([&](Req r) { return async_send(sock, response_of(r)); })
    | ex::upon_error(log_error);

// 提交到 scheduler，自己决定何时跑
ex::start_detached(std::move(work), my_scheduler);

Boost.Asio 核心模型 ⭐

核心概念三件套 · 第一个例子：最小 echo · Completion Token 机制 —— Asio 的"插槽" · io_context 的线程模型 · work_guard / executor_work_gu…

Boost.Asio 是 C++ 异步网络编程的事实标准（被广泛用于 gRPC 内部、AWS SDK、msquic、libtorrent 等）。独立版叫"standalone Asio"，不依赖 Boost。

3.1 核心概念三件套

概念语义
io_context事件循环 + 任务队列 + 调度核心（一个进程通常 1 个，或每 CPU 一个）
executor"在哪跑"的抽象（io_context::executor / thread_pool 都是）
strand序列化执行器 —— 同一 strand 上的 handler 永不并发
awaitable<T>协程返回类型（Asio 自家 Task 类型）
completion token"完成处理方式"的可插拔机制（callback / use_future / use_awaitable / use_sender）

3.2 第一个例子：最小 echo

#include <asio.hpp>     // 或 boost/asio.hpp
using asio::ip::tcp;

asio::io_context io;
tcp::acceptor acceptor{io, tcp::endpoint(tcp::v4(), 8080)};

void do_accept() {
    acceptor.async_accept([](std::error_code ec, tcp::socket sock) {
        if (!ec) do_session(std::move(sock));
        do_accept();          // 继续接
    });
}

int main() {
    do_accept();
    io.run();                 // 阻塞跑事件循环
}

3.3 Completion Token 机制 —— Asio 的"插槽"

Asio 所有 async_* 操作的最后一个参数是 completion token。它决定操作完成时怎么通知你：

// ① 经典 callback 风格
sock.async_read_some(asio::buffer(buf), [&](auto ec, auto n) {
    // callback
});

// ② future 风格
std::future<size_t> f = sock.async_read_some(asio::buffer(buf), asio::use_future);
size_t n = f.get();          // 阻塞等

// ③ 协程风格 (C++20)
size_t n = co_await sock.async_read_some(asio::buffer(buf), asio::use_awaitable);

// ④ 同步风格（仅特定场景）
size_t n = sock.read_some(asio::buffer(buf));

// ⑤ Sender 风格（C++26 / asio 1.30+ experimental）
auto sender = sock.async_read_some(asio::buffer(buf), asio::experimental::use_sender);
这是 Asio 最强大的设计：同一个 async_read_some 接口能适配各种 callback 风格。从 2003 年的回调到 2020 年的协程到 2026 年的 Sender —— Asio 一直是同一套 API。

3.4 io_context 的线程模型

// 模型 1: 单线程 - 最简单
asio::io_context io;
io.run();      // 单线程跑所有 callback，无并发问题

// 模型 2: 多线程共享 io_context — 任意 handler 在任一线程跑
asio::io_context io;
std::vector<std::thread> pool;
for (int i = 0; i < N; ++i)
    pool.emplace_back([&] { io.run(); });

// 模型 3: 每核一个 io_context（推荐高性能）
std::vector<asio::io_context> ios(N);
for (int i = 0; i < N; ++i) {
    pool.emplace_back([&, i] {
        // pin 到 CPU i
        pin_to_cpu(i);
        ios[i].run();
    });
}
// 新连接通过 SO_REUSEPORT 在各 io_context 间分流

3.5 work_guard / executor_work_guard

// io_context::run() 在没有"待办工作"时返回
// 想保持 io_context 一直跑：work_guard
auto work = asio::make_work_guard(io);
io.run();   // 永不返回 (直到 work.reset())

3.6 buffer 与生命周期

// ❌ 常见错误
{
    std::vector<char> buf(1024);
    sock.async_read_some(asio::buffer(buf), handler);
}   // ★ buf 出作用域析构，但 async_read 还在用 → UAF

// ✅ 正确
auto buf = std::make_shared<std::vector<char>>(1024);
sock.async_read_some(asio::buffer(*buf), [buf](auto ec, auto n) {
    // buf 捕获保活
});

完整 echo server — 4 种风格对比

Callback 风格（经典） · 协程风格（推荐） · future 风格（少用，仅展示） · Sender/Receiver 风格 (asio 1.30+ experimental)

4.1 Callback 风格（经典）

class Session : public std::enable_shared_from_this<Session> {
    tcp::socket sock_;
    std::array<char, 8192> buf_;
public:
    Session(tcp::socket s) : sock_(std::move(s)) {}
    void start() { do_read(); }
private:
    void do_read() {
        sock_.async_read_some(asio::buffer(buf_),
            [self = shared_from_this()](auto ec, auto n) {
                if (!ec) self->do_write(n);
            });
    }
    void do_write(size_t n) {
        asio::async_write(sock_, asio::buffer(buf_, n),
            [self = shared_from_this()](auto ec, auto) {
                if (!ec) self->do_read();
            });
    }
};

4.2 协程风格（推荐）

asio::awaitable<void> echo_session(tcp::socket sock) {
    try {
        std::array<char, 8192> buf;
        for (;;) {
            size_t n = co_await sock.async_read_some(
                asio::buffer(buf), asio::use_awaitable);
            co_await asio::async_write(sock,
                asio::buffer(buf, n), asio::use_awaitable);
        }
    } catch (std::system_error& e) { /* 连接断 */ }
}

asio::awaitable<void> listener() {
    auto ex = co_await asio::this_coro::executor;
    tcp::acceptor acc(ex, {tcp::v4(), 8080});
    for (;;) {
        tcp::socket sock = co_await acc.async_accept(asio::use_awaitable);
        asio::co_spawn(ex, echo_session(std::move(sock)), asio::detached);
    }
}

int main() {
    asio::io_context io;
    asio::co_spawn(io, listener(), asio::detached);
    io.run();
}
协程版本更短、更线性、更容易调试。生产代码已经是事实标准。

4.3 future 风格（少用，仅展示）

void session(tcp::socket sock) {
    std::array<char, 8192> buf;
    while (true) {
        auto n = sock.async_read_some(
            asio::buffer(buf), asio::use_future).get();
        asio::async_write(sock, asio::buffer(buf, n), asio::use_future).get();
    }
}
// 每个 session 占用一个 worker 线程（阻塞在 .get）
// 失去异步本意，仅用作教学

4.4 Sender/Receiver 风格 (asio 1.30+ experimental)

auto session = asio::experimental::make_parallel_group(
    sock.async_read_some(asio::buffer(buf), asio::experimental::use_sender),
    timer.async_wait(asio::experimental::use_sender)
).async_wait(asio::experimental::wait_for_one(), asio::experimental::use_sender);

// 与 stdexec 等组合算子互通

5.1 co_spawn / detached / use_awaitable / awaitable<T>

asio::awaitable<int> compute() {

    ...

5.2 超时模式

asio::awaitable<void> read_with_timeout(tcp::socket& sock, std::chrono::milliseconds t) {
    asio::steady_timer timer(co_await asio::this_coro::executor);
    timer.expires_after(t);

    // awaitable_operators 提供 || 和 &&
    using namespace asio::experimental::awaitable_operators;

    auto result = co_await (
        sock.async_read_some(asio::buffer(buf), asio::use_awaitable)
        ||
        timer.async_wait(asio::use_awaitable)
    );

    if (result.index() == 1) {
        throw std::runtime_error("timeout");
    }
    // result 是 variant<size_t, monostate>
}

using namespace asio::experimental::awaitable_operators;

asio::awaitable<void> parallel_fetch() {
    auto [r1, r2, r3] = co_await (
        fetch(url1) && fetch(url2) && fetch(url3)
    );
    merge(r1, r2, r3);
}

5.4 取消 (cancellation slot)

Strand —— Asio 的并发解药

问题：多线程 io_context 上的 handler 可能并发 · Strand：序列化执行器 · 每 Session 一个 strand 模式（推荐） · 协程 + strand

6.1 问题：多线程 io_context 上的 handler 可能并发

// 4 个线程跑同一 io_context，一个 Session 的 handler 可能在不同线程
// → Session 状态需要锁 → 性能下降

6.2 Strand：序列化执行器

asio::strand<asio::io_context::executor_type> strand{io.get_executor()};

// 所有 post 到这个 strand 的 handler 永不并发
asio::post(strand, []{ /* h1 */ });
asio::post(strand, []{ /* h2 */ });
// h1 一定先于 h2 跑完

6.3 每 Session 一个 strand 模式（推荐）

class Session : public std::enable_shared_from_this<Session> {
    tcp::socket sock_;
    asio::strand<asio::any_io_executor> strand_;     // ★ 每 Session 自己一个
public:
    Session(asio::io_context& io, tcp::socket s)
        : sock_(std::move(s))
        , strand_(asio::make_strand(io)) {}

    void start() {
        sock_.async_read_some(asio::buffer(buf_),
            asio::bind_executor(strand_,           // ★ handler 绑到 strand
                [self = shared_from_this()](auto, auto n) {
                    // 此 callback 一定在 strand 上跑 → 单 session 内无并发
                }));
    }
};

实战中是每连接一 strand而不是每线程一锁。锁粒度自动最优。

6.4 协程 + strand

asio::awaitable<void> session(tcp::socket sock) {
    // 协程会"记住"它出生时的 executor，
    // 每次 co_await 后自动回到那个 executor
    ...
}

asio::co_spawn(
    asio::make_strand(io),               // ★ 用 strand 作为 spawn 的 executor
    session(std::move(sock)),
    asio::detached);
// 整个协程的 await 后都在 strand 上 → session 内无并发
"每连接一 strand" 是 Asio 高性能并发的最佳实践。替代了"每连接一锁"模式：① 无锁开销；② 调度天然亲和；③ 多核扩展性极佳。

Boost.Beast — Asio 上的 HTTP / WebSocket

Beast 定位 · 最小 HTTP server · WebSocket 升级 · Beast 的优劣 · 上层框架

7.1 Beast 定位

Boost.Beast 是 Asio 之上的 HTTP/1.1 + WebSocket 协议库 —— 不是框架而是协议解析器 + 异步 IO 整合。需要自己组合 router 等。

7.2 最小 HTTP server

#include <boost/beast.hpp>
namespace beast = boost::beast;
namespace http  = beast::http;

asio::awaitable<void> handle_session(tcp::socket sock) {
    beast::tcp_stream stream{std::move(sock)};
    beast::flat_buffer buf;

    while (true) {
        http::request<http::string_body> req;
        co_await http::async_read(stream, buf, req, asio::use_awaitable);

        http::response<http::string_body> res{http::status::ok, req.version()};
        res.set(http::field::content_type, "text/plain");
        res.body() = "hello, world\n";
        res.prepare_payload();

        co_await http::async_write(stream, res, asio::use_awaitable);

        if (!req.keep_alive()) break;
    }
}

7.3 WebSocket 升级

asio::awaitable<void> ws_session(tcp::socket sock) {
    beast::websocket::stream<beast::tcp_stream> ws{std::move(sock)};
    co_await ws.async_accept(asio::use_awaitable);

    while (true) {
        beast::flat_buffer buf;
        co_await ws.async_read(buf, asio::use_awaitable);
        ws.text(ws.got_text());
        co_await ws.async_write(buf.data(), asio::use_awaitable);
    }
}

Sender/Receiver 网络栈 — 未来的方向

概念示例 — 用 sender 组合网络操作 · Asio 1.30+ 的 sender 适配 · 为什么 Sender 路线是未来

P2300 std::execution（C++26）+ mvfst（Meta QUIC 实现）+ stdexec 共同推动"Sender/Receiver 风格"的网络库。

8.1 概念示例 — 用 sender 组合网络操作

namespace ex = std::execution;

auto request = ex::just(url)
    | ex::let_value(dns_resolve)              // DNS
    | ex::let_value(connect_to_first)         // TCP connect
    | ex::let_value(tls_handshake)            // TLS
    | ex::let_value(send_http_request)
    | ex::let_value(read_http_response)
    | ex::upon_error(log_and_default);

// 整条链结构化取消（外部 stop_token 自动传播到每步）
// 整条链能附加 timeout、retry、circuit-breaker 等"装饰算子"
auto with_timeout = std::move(request) | ex::stop_when(timer(5s));
auto with_retry   = std::move(with_timeout) | ex::retry_when(exp_backoff);

8.2 Asio 1.30+ 的 sender 适配

#include <asio/experimental/use_sender.hpp>

auto sender = sock.async_read_some(asio::buffer(buf),
                                   asio::experimental::use_sender);

// 把它丢进 stdexec 管线
auto work = std::move(sender) | stdexec::then(parse);

8.3 为什么 Sender 路线是未来

  • 结构化并发：取消、超时、错误传播全自动
  • 跨硬件统一：CPU executor / GPU / 网络 IO 同一接口
  • 组合代数：when_all / when_any / let_value 等算子可任意拼装
  • 零开销：编译期编译成状态机，运行时无虚调用
  • 标准化：进 C++26 后所有库都对齐
2026 现状：std::execution 还在落地，Asio 协程是当下事实标准；S/R 是未来 5 年的方向。Tech Lead 应该提前 1-2 年布局，让团队中至少一人能写 S/R 风格代码。

gRPC 完整剖析 ⭐

gRPC 核心架构 · 第一个 gRPC 服务 · gRPC 性能数据 · 同步 vs 异步 API

gRPC 是 Google 开源的跨语言 RPC 框架。基于 HTTP/2 + Protobuf，C++ 实现性能/功能最完整。

9.1 gRPC 核心架构

传输层

HTTP/2 streams（每 RPC 一个 stream），头部 HPACK 压缩，可双工流

序列化

Protobuf 默认；可换 JSON / FlatBuffers / Cap'n Proto

调用模型

① Unary（请求-响应）② Server stream ③ Client stream ④ Bidirectional stream

9.2 第一个 gRPC 服务

// echo.proto
syntax = "proto3";
service EchoService {
  rpc Echo(EchoRequest) returns (EchoResponse);
}
message EchoRequest  { string msg = 1; }
message EchoResponse { string msg = 1; }
// 服务端 (同步 API, 简单但每 RPC 一线程)
class EchoImpl final : public EchoService::Service {
    grpc::Status Echo(grpc::ServerContext* ctx, const EchoRequest* req,
                      EchoResponse* res) override {
        res->set_msg(req->msg());
        return grpc::Status::OK;
    }
};

int main() {
    EchoImpl service;
    grpc::ServerBuilder builder;
    builder.AddListeningPort("0.0.0.0:50051", grpc::InsecureServerCredentials());
    builder.RegisterService(&service);
    auto server = builder.BuildAndStart();
    server->Wait();
}

10.1 Callback API (现代推荐)

// 服务端 - 异步 callback 风格
class EchoImpl final : public EchoService::CallbackService {
    grpc::ServerUnaryReactor* Echo(
        grpc::CallbackServerContext* ctx,
        const EchoRequest* req,
        EchoResponse* res) override
    {
        res->set_msg(req->msg());
        auto* reactor = ctx->DefaultReactor();
        reactor->Finish(grpc::Status::OK);
        return reactor;
    }
};

10.2 客户端流（client → server 流式发送）

// 客户端
class UploadReactor : public grpc::ClientWriteReactor<Chunk> {
    grpc::ClientContext ctx_;
    UploadResp resp_;
    grpc::Status status_;
    std::queue<Chunk> pending_;
public:
    UploadReactor(Stub* stub) {
        stub->async()->Upload(&ctx_, &resp_, this);
        StartCall();
    }
    void SendChunk(Chunk c) {
        pending_.push(std::move(c));
        if (pending_.size() == 1) StartWrite(&pending_.front());
    }
    void OnWriteDone(bool ok) override {
        pending_.pop();
        if (!pending_.empty()) StartWrite(&pending_.front());
        else StartWritesDone();
    }
    void OnDone(const grpc::Status& s) override { status_ = s; /* notify */ }
};

class EchoImpl : public EchoService {

12.3 QUIC 的关键优势复习

  • 0-RTT 建连恢复（vs TCP+TLS 1-2 RTT）
  • 每流独立可靠，丢包不影响其它流（vs HTTP/2 over TCP 队头阻塞）
  • 连接迁移（IP 变更不断连，移动场景关键）
  • 用户态拥塞控制，可独立部署 BBR / CUBIC

百万连接架构设计

单机百万连接的关键 · 典型架构 · 内存估算 · 实际限制

13.1 单机百万连接的关键

  1. SO_REUSEPORT：多 worker 同端口 listen，内核分流
  2. 每 worker 一 io_context，CPU 亲和性绑定
  3. 每连接最小内存（典型 ~10-30KB / 连接）
  4. 边缘触发 epoll (ET)，减少 syscall
  5. 定时器轮 (timing wheel)，O(1) 添加 / 失效
  6. 用户态协议解析，减少内核态来回

服务网格 (Envoy) 视角

Envoy 架构亮点 · Envoy 的"为什么 C++" · Envoy 与你自家 C++ 服务的关系

Envoy 是 Lyft 出品的现代 L7 代理，C++14 写成，是 Service Mesh 的事实标准数据平面（Istio / Consul / AWS App Mesh 都用 Envoy）。

14.1 Envoy 架构亮点

  • Listener / Filter chain：可插拔过滤器（L3/L4/L7）
  • 每线程独立事件循环（M:N → 1:1 简化）
  • xDS API：动态配置（gRPC streaming），无需重启
  • HTTP/1.1 / 2 / 3 全栈
  • WASM 扩展：用 Wasm 写自家 filter，无需 C++ 编译

14.2 Envoy 的"为什么 C++"

Lyft 工程师公开博文：选 C++ 因为 ① 想要 ms 级 P99；② 内存可控（GC 语言不行）；③ 与 OpenSSL / gRPC / Boost 深度集成。但开发者门槛高，所以提供 WASM filter SDK 让用户用 Rust / Go / TinyGo 写扩展。

14.3 Envoy 与你自家 C++ 服务的关系

  • Envoy 作为 sidecar 反向代理 → 你的 C++ 服务只需关心业务逻辑
  • Envoy 处理 mTLS / 限流 / 熔断 / 重试 / 灰度 → 减少自家代码
  • 但增加一层 latency（~0.5-2 ms / hop）

真实工程案例 5 个

案例 1：行情广播服务（金融） · 案例 2：实时游戏服务器 · 案例 3：微服务后端 · 案例 4：CDN 边缘节点 · 案例 5：物联网网关

15.1 案例 1：行情广播服务（金融）

  • 需求：百万订阅者，5ms 内收到行情更新
  • 设计：Disruptor 内存队列 + Asio multi-context + SO_REUSEPORT + 用户态 SSL
  • 关键：CPU 亲和性 + 大页内存 + busy polling + 无锁队列
  • 结果：P99 推送延迟 2.3ms

15.2 案例 2：实时游戏服务器

  • 需求：10 万并发玩家，30 Hz tick，输入 → 状态 → 广播 ≤ 50ms
  • 设计：QUIC（连接迁移 + 0-RTT）+ msquic + Custom delta-compression
  • 架构：边缘 PoP × N + 后端房间分片
  • 关键：QUIC stream per player，丢包不影响他人

15.3 案例 3：微服务后端

  • 需求：百微服务，gRPC 调用为主，要观测 / 链路追踪
  • 设计：gRPC + Envoy sidecar + OpenTelemetry + Istio
  • 编码：业务用 gRPC callback API + asio-grpc 协程
  • 关键：sidecar 把 mTLS / retry / circuit-breaker 拿出业务代码

15.4 案例 4：CDN 边缘节点

  • 需求：百万 QPS，HTTP/3 优先，全球分布
  • 设计：Cloudflare 风格 — 自家协议栈（quiche on Rust，C++ 应用层）+ XDP/eBPF 包过滤
  • 关键：用户态协议栈消除内核开销，eBPF 处理 DDoS

15.5 案例 5：物联网网关

  • 需求：百万设备 MQTT 长连接，低带宽，断线频繁
  • 设计：MQTT over WebSocket + 自家 broker（基于 Asio）+ 持久化到 Kafka
  • 关键：每连接 strand，会话状态分布式持久化（断线 5min 内复连保留状态）
  • 结果：单机 50 万连接，~10GB RAM
