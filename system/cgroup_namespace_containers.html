01

容器不是虚拟机

没有 Guest OS，只有被"框住"的进程

最大的误解：以为容器是「轻量虚拟机」。虚拟机虚拟的是硬件，每个 VM 跑一整套自己的内核 + OS，靠 Hypervisor 隔离，重但隔离强。容器不虚拟硬件，所有容器共享宿主机的同一个内核，只是被内核的隔离机制「框」起来，让进程以为自己独占整个系统。容器本质就是一个受限的普通进程。

🖥 虚拟机

虚拟硬件 + 独立内核/OS；隔离强、开销大、启动秒级~分钟级。

📦 容器

共享宿主内核；隔离靠 namespace+cgroup；轻量、毫秒启动、密度高。

⚖️ 取舍

容器省资源、快，但共享内核 → 内核漏洞可能逃逸；强隔离场景用轻量 VM（Kata/Firecracker）。

生活类比：独栋别墅 vs 合租公寓

虚拟机像独栋别墅：每家有独立的水电煤（内核/OS），互不影响但盖房成本高。容器像一栋楼里的合租单间：共用整栋楼的水电主管道（宿主内核），但每间房有独立门锁、独立视野（namespace），物业还限制每间的用电额度（cgroup）。合租便宜灵活，但主管道（内核）出问题会影响所有人。

映射：整栋楼水电=宿主内核 · 独立房间视野=namespace · 用电限额=cgroup · 合租便宜=轻量高密度
容器的三大支柱：namespace 管「看得见什么」（隔离视图）、cgroup 管「能用多少」（限制资源）、镜像/OverlayFS 管「根文件系统长什么样」。理解这三点，容器就没有黑魔法了。
02

Namespace：隔离视图

让进程以为自己独占系统

Namespace 把某类全局系统资源包装起来，让不同 namespace 里的进程看到各自独立的实例。容器就是把一个进程放进一组新建的 namespace。

Namespace隔离什么
PID进程号——容器内 PID 1 其实是宿主上的某个大 PID；容器看不到宿主进程
Mount (mnt)挂载点/文件系统视图——容器有自己的根文件系统
Network (net)网卡、IP、路由、端口——容器有独立网络栈
UTS主机名与域名
IPCSystem V IPC、POSIX 消息队列
UserUID/GID 映射——容器内 root 可映射为宿主普通用户（rootless 关键）
Cgroup隐藏宿主 cgroup 层级

相关系统调用

  • clone()：创建新进程时带 CLONE_NEW* 标志，直接放进新 namespace。
  • unshare()：让当前进程脱离并进入新 namespace。
  • setns()：加入一个已存在的 namespace（docker exec 就是把新进程 setns 进目标容器的 namespace）。
  • 亲手体验：unshare --pid --net --mount --fork bash 就能造一个「迷你容器」。
「容器内 PID 1」很特殊：它是 PID namespace 的 init，要负责回收僵尸子进程、转发信号。业务进程直接当 PID 1 常忘了收养孤儿进程 → 用 tini/--init 做轻量 init。这是容器里进程管理的经典坑。
03

Cgroup v2：资源限制与统计

namespace 管隔离，cgroup 管配额

Namespace 让容器「看不见别人」，但不限制它「用多少」——一个容器仍能吃满宿主 CPU/内存。Cgroup（control group）负责限制、统计、隔离一组进程的资源用量。v2 是统一层级（unified hierarchy），已取代 v1。

控制器关键接口作用
cpucpu.max / cpu.weight配额（如每 100ms 最多 50ms）/ 相对权重；超额被 throttle
memorymemory.max / memory.high硬上限（超则 OOM kill）/ 软限（超则加压回收）
ioio.max / io.weight块设备 IOPS/带宽限制
pidspids.max限制进程数，防 fork 炸弹

工程要点

  • CPU throttling：容器设了 cpu.max，突发计算会被限速，表现为「明明 CPU 没满却变慢」——查 cpu.stat 的 throttled_usec。K8s limits 设太紧是常见性能坑。
  • OOM：容器内存超 memory.max → cgroup OOM killer 杀容器内进程（常是 PID 1，容器直接挂）。查 dmesg / memory.events。
  • 感知不到限制：老程序读 /proc/cpuinfo 看到的是宿主全部核数，不知道自己被限到 2 核 → 线程池开太大。需读 cgroup 限额（新版 JVM/Go runtime 已感知）。

内存 cgroup 与 OOM 细节见 NUMA + 高级内存；CPU 权重/配额与调度关系见 进程调度。

04

OverlayFS：镜像分层

为什么 docker pull 能秒级、镜像能共享

容器镜像是只读层的叠加。OverlayFS（联合文件系统）把多个目录「合并」成一个视图：底下是若干只读镜像层（lowerdir），最上面一层可写层（upperdir）。容器看到的是合并后的统一文件系统。

核心机制

  • 分层复用：多个镜像共享相同底层（如都基于 ubuntu:22.04），只存一份 → 省磁盘、拉取快。
  • Copy-on-Write（写时复制）：改只读层的文件时，先复制到可写层再改，不动底层。所以容器改文件不影响镜像、也不影响别的容器。
  • 容器可写层是临时的：容器删除即丢——要持久化必须挂 volume/bind mount。这是「容器无状态」的物理原因。
  • whiteout：删除只读层文件时，在可写层放一个特殊标记「遮住」它，而非真删底层。
生活类比：透明玻璃叠画

镜像层像一叠透明胶片：底下几张是印好的公共图案（只读基础层），最上面一张是空白胶片（可写层）。你要涂改下面的图案时，不能直接画在公共胶片上（别人还要用），而是把那部分描到最上面的空白胶片再改（写时复制）。从正上方看，所有胶片叠加成一张完整的画（合并视图）。抽走你那张空白胶片（删容器），公共胶片完好如初。

映射：公共胶片=只读镜像层 · 空白胶片=可写层 · 描到上层再改=CoW · 抽走上层=删容器
05

容器如何启动（OCI / runc）

从 docker run 到进程诞生

Docker/K8s 只是上层。真正「造容器」的是符合 OCI（Open Container Initiative） 规范的低层运行时，如 runc。启动一个容器大致是这几步：

启动流程

  • ① 准备根文件系统：用 OverlayFS 挂载镜像各层 + 可写层，得到容器 rootfs。
  • ② 创建 namespace：clone() 带各 CLONE_NEW* 标志，进入隔离视图。
  • ③ 设置 cgroup：把新进程加入配好限额的 cgroup。
  • ④ 切换根：pivot_root 把根切到容器 rootfs（比 chroot 更安全）。
  • ⑤ 收敛权限：drop capabilities、应用 seccomp/AppArmor、设 no_new_privs。
  • ⑥ exec 用户进程：execve 运行镜像里的 ENTRYPOINT，容器「活」了。
层次组件职责
编排Kubernetes调度、编排、扩缩容
高层运行时containerd / CRI-O拉镜像、管生命周期、对接 CRI
低层运行时runc（OCI）真正调 namespace/cgroup 造容器进程
隔离沙箱（可选）Kata / gVisor / Firecracker更强隔离（轻量 VM / 用户态内核）
06

容器网络

独立网络栈怎么连出去

每个容器有独立的 network namespace（自己的网卡、IP、路由）。它怎么和宿主/外网通信？

常见模式

  • bridge（默认）：宿主建一个虚拟网桥 docker0；每个容器一对 veth pair（虚拟网线），一头在容器内、一头插网桥；出外网靠 NAT（iptables MASQUERADE）。
  • host：容器直接用宿主网络栈，无隔离、无 NAT 开销，端口直接暴露。
  • none：只有 loopback，完全隔离，自己配。
  • K8s CNI：跨节点 Pod 通信由 CNI 插件（Calico/Cilium/Flannel）实现，Cilium 用 eBPF 做高性能网络与策略。

底层网络编程与 eBPF 见 网络编程 与 eBPF 实战。

07

容器安全

共享内核 = 隔离没有 VM 强

纵深防御

  • Capabilities：把 root 的超级权限拆成细粒度能力，容器默认只保留一小部分，drop 掉危险的（如 CAP_SYS_ADMIN）。别用 --privileged。
  • Seccomp：系统调用白/黑名单，禁掉容器用不到的危险 syscall，缩小攻击面。
  • User namespace（rootless）：容器内 root 映射成宿主普通用户，即使逃逸也不是宿主 root。
  • 只读根 + no_new_privs：根文件系统只读、禁止提权，减少被篡改可能。
  • 强隔离场景：多租户/不可信代码用 gVisor（用户态内核拦截 syscall） 或 Kata/Firecracker（轻量 VM），用一点性能换 VM 级隔离。
记住：容器共享宿主内核，所以「容器逃逸」多是通过内核漏洞或错误配置（特权容器、挂 docker.sock、挂宿主敏感目录）。安全基线：最小权限 + 非 root + seccomp + 不 privileged。
08

checklist

容器排障 / 加固速查

  • 容器「无故变慢」→ 查 cpu.stat 的 throttled（CPU limit 太紧）。
  • 容器突然重启/退出 137 → OOM，查 dmesg / memory.events，调 memory.max 或修内存泄漏。
  • 应用线程池/GC 开太大 → 程序没感知 cgroup 限额，读的是宿主核数；升级 runtime 或显式配置。
  • 僵尸进程堆积 → 业务当 PID 1 没收养孤儿，加 --init/tini。
  • 数据丢失 → 写在容器可写层（CoW，删容器即丢），持久化要挂 volume。
  • 安全 → 非 root 运行、drop 多余 capabilities、加 seccomp、别 --privileged、别挂 docker.sock。
一句话记忆：容器 = namespace（隔离视图）+ cgroup（限制资源）+ OverlayFS（分层根文件系统）包起来的普通进程；共享内核所以轻量但隔离弱。
