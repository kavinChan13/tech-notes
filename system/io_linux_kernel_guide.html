1.1 用户态 vs 内核态

Linux 采用 二环保护模型：用户态（Ring 3）运行应用程序，内核态（Ring 0）运行内核代码。I/O 操作必须陷入内核态执行，因为硬件访问需要特权指令。

streambuf 是 iostream 的心脏——所有字符的实际搬运都在 streambuf 的 get/put 区完成。std::cout << x 本质上是调用 ostream::operator<< → streambuf::sputc → 刷盘时 overflow() 触发 write() 系统调用。

// 自定义 streambuf：重定向 std::cout 到网络 socket
class SocketStreamBuf : public std::streambuf {
    int fd_;
    char buf_[4096];
public:
    SocketStreamBuf(int fd) : fd_(fd) {
        setp(buf_, buf_ + sizeof(buf_));  // 设置 put 区
    }
protected:
    int sync() override {                 // flush
        auto n = pptr() - pbase();
        if (n > 0) ::write(fd_, pbase(), n);
        setp(buf_, buf_ + sizeof(buf_));
        return 0;
    }
    int_type overflow(int_type c) override {
        sync();
        if (c != EOF) *pptr() = c, pbump(1);
        return c;
    }
};

⚠ iostream 的性能陷阱
  • sync_with_stdio(false)：解除 cin/cout 与 stdio 的同步，单线程下性能提升 5-10x
  • cin.tie(nullptr)：解除 cin/cout 绑定，避免每次 cin 前 flush cout
  • endl vs "\n"：endl 触发 flush（系统调用），高频日志场景用 "\n" 更快
  • virtual 函数开销：streambuf 所有方法都是虚函数，热路径可考虑 std::fwrite

3.3 fwrite vs write 性能对比实验

// 写 100 万条日志，每条 ~100 字节
// 方案 A：fwrite (带缓冲)
FILE* f = fopen("log.txt", "w");
for (int i = 0; i < 1'000'000; ++i)
    fprintf(f, "[%d] log line\n", i);
fclose(f);
// 实测: ~80ms (只触发 ~2500 次 write 系统调用)

// 方案 B：write (无缓冲)
int fd = open("log.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
for (int i = 0; i < 1'000'000; ++i) {
    char buf[128];
    int n = snprintf(buf, sizeof(buf), "[%d] log line\n", i);
    write(fd, buf, n);        // 每次都陷入内核！
}
close(fd);
// 实测: ~800ms (100 万次 write 系统调用，慢 10x)

// 方案 C：writev (向量化 I/O)
std::vector<iovec> iovs;
std::vector<std::string> bufs;
for (int i = 0; i < 1'000'000; ++i) {
    bufs.emplace_back(std::format("[{}] log line\n", i));
    iovs.push_back({bufs.back().data(), bufs.back().size()});
}
writev(fd, iovs.data(), iovs.size());  // 1 次调用!
✗ fork 与 stdio 缓冲的陷阱

fork();              // 子进程复制了同一份缓冲区
// 结果：父子进程各打印 "hello"，共 2 次输出！
// 修复：fork 前 fflush(stdout) 或用 write()

▶ 硬链接 vs 软链接

硬链接：多个 dentry 指向同一 inode（i_nlink++），删除文件需 i_nlink=0 才真正释放。不能跨文件系统。

软链接：独立 inode，内容是目标路径的字符串。可跨文件系统，但多一次路径解析。

五种 I/O 模型

5.1 I/O 两阶段模型

任何一次 I/O 操作都包含 两个阶段：

  1. 等待数据准备：内核等待数据到达（从网卡/磁盘）
  2. 数据拷贝：从内核缓冲区拷贝到用户缓冲区

五种 I/O 模型的差异就在于这两个阶段是阻塞还是非阻塞。

▶ 关键区分

同步 vs 异步看的是"阶段 2 数据拷贝"是否由内核自己完成。

阻塞 vs 非阻塞看的是"阶段 1 等数据"调用方是否等待。

所以 I/O 多路复用本质上仍是同步 I/O——只是把"等"的动作从 read() 移到了 select()。

5.3 代码对比

// 1. 阻塞 I/O — 最简单但并发能力最差
char buf[1024];
int n = read(fd, buf, sizeof(buf));  // 无数据则一直阻塞

// 2. 非阻塞 I/O — 忙轮询浪费 CPU
fcntl(fd, F_SETFL, O_NONBLOCK);
while ((n = read(fd, buf, sizeof(buf))) < 0) {
    if (errno != EAGAIN) perror("read"), exit(1);
    // CPU 100% 跑空
}

// 3. I/O 多路复用 (epoll) — C10K 问题的标准答案
int ep = epoll_create1(0);
epoll_event ev{.events = EPOLLIN, .data.fd = fd};
epoll_ctl(ep, EPOLL_CTL_ADD, fd, &ev);
epoll_event events[64];
int nfds = epoll_wait(ep, events, 64, -1);
for (int i = 0; i < nfds; ++i)
    read(events[i].data.fd, buf, sizeof(buf));

// 5. 异步 I/O (io_uring) — 真正的异步
io_uring ring;
io_uring_queue_init(32, &ring, 0);
auto* sqe = io_uring_get_sqe(&ring);
io_uring_prep_read(sqe, fd, buf, sizeof(buf), 0);
io_uring_submit(&ring);
// ... 做其他事 ...
io_uring_cqe* cqe;
io_uring_wait_cqe(&ring, &cqe);  // 数据已到 buf

6.3 LT vs ET (水平触发 vs 边缘触发)

LT水平触发 (Level Trigger)

只要 fd 可读/可写就持续通知。默认模式，编程简单。

  • 读一部分剩余数据仍会通知
  • 兼容阻塞/非阻塞 I/O
  • 类似 select/poll 语义
ET边缘触发 (Edge Trigger)

状态变化时才通知（不可读→可读）。高性能，但要求严格处理。

  • 必须配合非阻塞 fd
  • 必须循环 read 直到 EAGAIN
  • 减少系统调用次数，Nginx/Redis 标配
// ET 模式的正确用法：必须读尽数据
epoll_event ev{.events = EPOLLIN | EPOLLET, .data.fd = fd};
fcntl(fd, F_SETFL, O_NONBLOCK);
epoll_ctl(ep, EPOLL_CTL_ADD, fd, &ev);

while (true) {
    int n = epoll_wait(ep, events, 64, -1);
    for (int i = 0; i < n; ++i) {
        int cfd = events[i].data.fd;
        // ET 模式下必须 while 循环读尽
        while (true) {
            ssize_t r = read(cfd, buf, sizeof(buf));
            if (r > 0) /* 处理 */;
            else if (r == 0) { close(cfd); break; }  // EOF
            else if (errno == EAGAIN) break;   // 读完了
            else { perror("read"); close(cfd); break; }
        }
    }
}
⚠ epoll 惊群问题

多个进程/线程 epoll_wait 同一个 fd（如 Nginx master/worker），连接到来时全部被唤醒，但只有一个能 accept 成功。

解决方案：Linux 4.5+ 的 EPOLLEXCLUSIVE flag；或使用 SO_REUSEPORT（每进程独立 listen fd）；或 Nginx 的 accept mutex。

const char* data = static_cast<const char*>(addr);
for (size_t i = 0; i < st.st_size; ++i)
    process(data[i]);   // 缺页时内核自动加载

munmap(addr, st.st_size);
close(fd);

7.3 mmap 的典型应用

DB数据库

SQLite / LMDB 用 mmap 把整个 B+Tree 映射进内存，依赖内核 LRU 管理缓存

IPC共享内存 IPC

shm_open + mmap 实现进程间零拷贝通信，比管道/socket 快 10x+

ELF可执行文件加载

execve 用 mmap 把 ELF 的 code/data 段按需加载，实现快速启动

⚠ mmap 的陷阱
  • SIGBUS：映射文件后 ftruncate 截断，访问越界区域触发
  • TLB miss：随机访问大文件 mmap 可能比 read 慢（页表项换入换出）
  • 小文件不划算：mmap 有建立 VMA + 页表的固定开销，<4KB 文件用 read 更快
  • MAP_SHARED 同步：必须 msync() 或 munmap() 才保证落盘

9.3 Direct I/O (O_DIRECT)

优优点
  • 绕过 Page Cache，数据直接在用户 buf 和磁盘间 DMA
  • 避免双重缓存（应用自己管理缓存时）
  • 可预测的性能（无页缓存命中率抖动）

10.1 为什么 Linux 长期没有"好用"的异步 I/O

要理解 io_uring，必须先理解它在解决什么问题。异步 I/O 的目标只有一个："提交请求"和"拿到结果"都不让调用线程阻塞。但 Linux 在 io_uring 之前做过两次失败的尝试。

✗POSIX AIO (glibc)

aio_read/aio_write/aio_return 这套接口不是内核实现的——glibc 在用户态偷偷起了一个线程池，把"异步"调用转成阻塞调用放到线程池里跑。

  • 线程池本身也是开销（创建、调度、上下文切换）
  • 规模无法扩展（线程数有限）
  • 行为在不同发行版不一致

所以它在工业界几乎无人使用。

△libaio (io_submit 家族)

内核真正实现的 AIO（io_setup/io_submit/io_getevents），但限制极多：

  • 只对 O_DIRECT 的块设备文件真正异步；对普通缓存 I/O 会退化成同步
  • 不支持网络 I/O
  • 元数据操作（openat/fsync/stat）都会阻塞
  • io_submit 本身可能阻塞（申请内存、获取锁）

MySQL/InnoDB、Ceph 早期版本在用，但写起来繁琐且"不够异步"。

▶ 真正异步的定义

你可能在网上看过这张对比图：同步 vs 异步的差别在于"数据拷贝阶段"是否由调用方完成。对 io_uring 来说更准确的说法是——提交路径无阻塞、完成路径无阻塞、所有 I/O 类型统一。

10.2 io_uring 革命 (Linux 5.1+)

Jens Axboe (Kernel Block 子系统维护者) 设计的新异步 I/O 接口，核心是两个共享内存 Ring Buffer（用户态和内核态共享），提交和完成都不需要系统调用。

10.4 SQE 与 CQE：一次请求的生命周期

理解 io_uring 最关键的两个结构是 SQE (Submission Queue Entry) 和 CQE (Completion Queue Entry)。它们位于一段 mmap 出来的内存（用户态与内核态共享），双方通过 head/tail 指针（Lamport 单生产单消费环形队列）推进，无锁。

// SQE — 用户填写，内核读取 (64 字节，对齐 cache line)
struct io_uring_sqe {
    __u8  opcode;       // READ/WRITE/ACCEPT/SEND/RECV/OPENAT/FSYNC...
    __u8  flags;        // LINK/HARDLINK/ASYNC/DRAIN...
    __u16 ioprio;
    __s32 fd;           // 目标 fd (或 fixed file index)
    __u64 off;          // 文件偏移
    __u64 addr;         // 用户 buffer 地址 (或 iovec 数组)
    __u32 len;          // 长度 (或 iovec 数)
    union { ... } op_flags;  // 按 opcode 解释
    __u64 user_data;    // ★ 回传给 CQE 的 token (常填对象指针)
    ...
};

// CQE — 内核填写，用户读取 (16 字节)
struct io_uring_cqe {
    __u64 user_data;    // 从 SQE 原样搬过来，识别是哪次请求
    __s32 res;          // 结果 (=read 的返回值 / -errno)
    __u32 flags;
};

  • 第 3 步批量提交：一次 syscall 可以塞几十上百个 SQE，摊薄内核切换开销
  • 第 4 步"异步"是真的异步：哪怕目标是普通文件（ext4 读写、fsync、openat、stat），都能异步执行，线程不阻塞
  • 第 7 步无系统调用：CQ 也在共享内存里，用户直接读指针

10.6 提交和收割的主循环模板

// 典型的 io_uring 事件循环 — 一边提交、一边收割 (batched)
void eventLoop(io_uring& ring) {
    while (!stop) {
        // 1) 填 SQE — 尽量一次填多个 (来自应用任务队列)
        while (auto task = taskQueue.try_pop()) {
            auto* sqe = io_uring_get_sqe(&ring);
            if (!sqe) break;                // SQ 满, 下轮再试
            task->prep(sqe);                  // prep_read / prep_write ...
            io_uring_sqe_set_data(sqe, task); // 回传指针当 user_data
        }

        // 2) 一次 syscall 提交 N 个, 并顺便 wait 至少 1 个完成
        io_uring_submit_and_wait(&ring, 1);

        // 3) 批量收割 CQE — 无系统调用, 直接读环形队列
        io_uring_cqe* cqe;
        unsigned head, count = 0;
        io_uring_for_each_cqe(&ring, head, cqe) {
            auto* task = static_cast<Task*>(io_uring_cqe_get_data(cqe));
            task->complete(cqe->res);      // 回调业务层
            ++count;
        }
        io_uring_cq_advance(&ring, count);  // 告诉内核这些 CQE 已消费
    }
}

10.7 进阶特性：让 io_uring 真正起飞的细节

LLinked SQE (操作链)

给 SQE 加 IOSQE_IO_LINK：A 完成后自动触发 B。能把 accept → read → write → close 串成一条链，一次提交全部完成。

FFixed Files / Buffers

MMULTISHOT

一条 IORING_OP_ACCEPT_MULTISHOT / RECV_MULTISHOT 可持续产生 CQE（每来一个新连接/报文就回一个 CQE），连重新提交都省了。Linux 5.19+。

IORING_OP_SEND_ZC：发送时不拷贝用户 buffer，等真正发出 (或 TCP ACK) 再生成完成事件。kTLS / RDMA 类场景的首选。

  • 为什么是两阶段中断（硬 + 软）？硬中断必须短（期间关中断），所以只做"拿队列指针 + 调软中断"；真正重活（协议解析、送到 socket）放软中断。

11.1.2 sk_buff：内核网络数据的"万能载体"

struct sk_buff（简称 skb）贯穿整个协议栈——从网卡驱动一直传到 socket 层都是它。它的巧妙之处在于"头指针可伸缩"：每过一层协议，只需要移动指针暴露出新的 header，不用拷贝数据。

11.4 Reactor / Proactor 模式

RReactor (同步非阻塞)

"I/O 就绪通知"。事件分发器（epoll）通知"可读/可写"，业务逻辑自己调用 read/write。

代表：Nginx, Redis, Netty, muduo, Asio (默认)

PProactor (真异步)

"I/O 完成通知"。用户提交请求，内核完成后回调。无需自己 read/write。

代表：Windows IOCP, io_uring, Boost.Asio (Windows)

// 单线程 Reactor (参考 muduo 简化)
class EventLoop {
    int epfd_;
    std::unordered_map<int, std::function<void()>> readCbs_;
public:
    EventLoop() : epfd_(epoll_create1(EPOLL_CLOEXEC)) {}

    void addRead(int fd, std::function<void()> cb) {
        epoll_event ev{.events = EPOLLIN | EPOLLET, .data.fd = fd};
        epoll_ctl(epfd_, EPOLL_CTL_ADD, fd, &ev);
        readCbs_[fd] = std::move(cb);
    }

    void loop() {
        epoll_event events[1024];
        while (true) {
            int n = epoll_wait(epfd_, events, 1024, -1);
            for (int i = 0; i < n; ++i)
                if (events[i].events & EPOLLIN)
                    readCbs_[events[i].data.fd]();
        }
    }
};

11.6 Reactor 的一次事件循环到底在做什么

很多人面试只能说出"epoll_wait 等事件、分发回调"，但真到写的时候要处理的细节远不止于此。一个可用于生产的 Reactor 主循环大致如下：

void EventLoop::loop() {
    while (!quit_) {
        // 1) 计算本次 epoll_wait 的超时 — 取"下一个到期定时器"的时间
        int timeout_ms = timerQueue_.nextTimeoutMs();

        // 2) 等 I/O 事件
        int n = epoll_wait(epfd_, events_, MAX_EVENTS, timeout_ms);

        // 3) 分发 I/O 事件 (按优先级: 读 > 写 > 错误)
        for (int i = 0; i < n; ++i) {
            auto* ch = static_cast<Channel*>(events_[i].data.ptr);
            ch->handleEvent(events_[i].events);
            // handleEvent 内部 ET 模式需要 while 读到 EAGAIN
        }

        // 4) 处理到期的定时器 (超时/心跳/连接回收)
        timerQueue_.handleExpired();

        // 5) 跨线程任务队列 — 别的线程通过 eventfd 唤醒本 loop
        doPendingFunctors();

        // 6) 迭代结束时的维护 (例如 flush 日志, 关闭待析构连接)
    }
}
▶ eventfd 的用法

工作线程想让 EventLoop 做一件事（比如写数据到 socket），它不能直接碰别人的 epoll——会有竞态。标准做法：创建一个 eventfd 注册到 loop 的 epoll 中，跨线程 write(eventfd, &1, 8) 就能唤醒对方，唤醒后回调里执行任务队列。这是 muduo、libevent、Node.js 都在用的套路。

11.7 C10K → C10M：瓶颈到底在哪？

C10K（1 万并发）在 2000 年代靠 epoll + 非阻塞就能解决；C10M（1 千万并发）则需要跨越多个数量级的瓶颈。

!协议栈本身是瓶颈

每包多层锁、多次分配 skb、每连接维护大量状态 (struct sock ~2KB)。1000 万连接光 sock 结构就是 20 GB。

对策：用户态网络栈 DPDK / VPP / Seastar，或 AF_XDP（内核外旁路）。

!上下文切换 & 锁竞争

一个 epoll fd 被多线程/多核争抢会热点化。

对策：SO_REUSEPORT 每核一个 listen socket；每核一个 EventLoop（share-nothing）；中断亲和与应用亲和对齐。

!内存与缓存

L3 miss 就是 100 ns，百万 QPS 下非常贵。

对策：per-CPU 数据结构、伪共享（false sharing）避开、大页 / 静态分配连接池。

!系统调用开销

epoll_wait + read + write 三连，几十万 QPS 下 syscall 本身就占 20%+ CPU。

对策：io_uring (批量提交 + SQPOLL)；零拷贝发送 (sendfile / MSG_ZEROCOPY)。

12.2 bio 与 request：块层的两个核心对象

文件系统把"写 4 KB 到文件的 offset X"翻译成"写磁盘的第 Y 个扇区"，封装成一个 bio (Block I/O)。bio 进入块层后可能合并 / 拆分成一个或多个 request，排进硬件队列，最终由驱动发给设备。

struct bio {
    struct bio*         bi_next;       // 可链成 bio chain
    struct block_device* bi_bdev;
    unsigned int        bi_opf;        // 操作 + flags (REQ_OP_READ / WRITE / FLUSH / DISCARD)
    struct bvec_iter    bi_iter;       // 当前位置: sector / size / idx
    struct bio_vec*     bi_io_vec;     // ★ 一组 <page, offset, len>: 支持散列的数据页
    bio_end_io_t*       bi_end_io;     // I/O 完成回调
    void*               bi_private;
    ...
};

// request: 一个或多个可合并 bio 的集合, 代表对设备的一次物理 I/O
struct request {
    struct list_head    queuelist;     // 挂在 hctx 上
    struct bio*          bio;           // 首 bio
    struct bio*          biotail;
    unsigned int        __data_len;    // 合并后的总长度
    ...
};
  • 为什么要分 bio 和 request：bio 是"应用视角的 I/O 单元"（一次 write 可能产生一个），request 是"硬件视角的 I/O 单元"（相邻扇区的多次写可合并成一次硬件命令）。
  • 散列列表 (bio_vec)：让一次 I/O 可以跨多个不连续的物理页，正好对应 SG-DMA 的能力——大文件写入不用先拷贝到连续缓冲区。

12.3 plug / unplug：I/O 的"蓄水池"

如果每个 bio 来了就立即下发，无法合并，硬件队列会被大量小 I/O 淹没。块层引入 plugging：

sequenceDiagram participant FS as 文件系统 participant P as 当前进程的 plug 链表 participant S as I/O 调度器 participant HW as 硬件队列 FS->>P: blk_start_plug() — 开启"插头" loop 连续多次 submit_bio FS->>P: bio 暂存到 plug 链表 Note over P: 合并相邻扇区的 bio end FS->>P: blk_finish_plug() — "拔插头" P->>S: 排序 + 调度决策 S->>HW: 下发 request HW-->>FS: I/O 完成回调
  • 合并 (merge)：相邻扇区的读/写合成一个大 request（back merge / front merge）
  • 蓄水时间：plug 是 per-task 的，正常在 unplug 时泄洪；如果阻塞或切换进程，会被主动 unplug 防止延迟
  • 应用影响：顺序大块写通常自动被合并；随机小写则合并率低，这也是"顺序 I/O 比随机 I/O 快"的内核侧原因之一

12.4 I/O 调度器

调度器策略最佳场景
none (noop)FIFO，无重排NVMe SSD（设备自己会重排）
mq-deadline读写队列 + 截止时间SATA SSD / 混合负载
kyber基于延迟目标的令牌桶低延迟要求
bfq公平队列 + 权重桌面交互，多进程争用

每个调度器的设计取舍可以这样记：

Nnone (noop)

只做合并，不做排序。现代 NVMe 设备内部已有自己的并行队列和重排逻辑，软件再做一遍调度反而拖慢延迟。

用它：NVMe、高性能虚拟化块设备

Dmq-deadline

两个维度的队列：按扇区排序（吞吐）+ 按到达时间排序（防止饿死）。读请求有更严格的 deadline（默认 500 ms），因为读通常是同步阻塞调用。

用它：SATA SSD、读延迟敏感的数据库混合负载

Kkyber

基于"延迟目标"的令牌桶：读/写各自设定目标延迟（如读 2 ms、写 10 ms），超标时限制 in-flight 数量。很像"自适应限流"。

用它：追求稳定尾延迟（p99）的场景

Bbfq (Budget Fair Queueing)

按进程/cgroup 分配带宽配额，"谁吵闹谁让路"。代价是 CPU 开销最高（排序队列复杂）。

用它：桌面、多租户容器、需要 I/O QoS 隔离的场景

12.6 IRQ 亲和、NUMA 与 I/O 热路径优化

百万 IOPS 场景下，一个"跨 CPU/跨 NUMA"动作就可能吃掉几微秒延迟。高性能 I/O 的铁律是：提交、完成、业务处理在同一个 CPU，或至少同一个 NUMA 节点。

1中断亲和 (IRQ Affinity)

把 NVMe 的每个 MSI-X 中断绑到特定 CPU：/proc/irq/N/smp_affinity_list。一般发行版会启动 irqbalance，但在极限场景要关掉它手动绑定。

2应用亲和 (CPU Pinning)

taskset -c 0-7 ./app 或 pthread_setaffinity_np。配合 SO_REUSEPORT，每核一个 worker，worker 只处理本核 hctx 上的 I/O 完成。

3NUMA 对齐

numactl -N 0 -m 0 ./app 让进程的 CPU 和内存都在 Node 0；同时检查 NVMe 挂在哪个 PCIe root（lspci -v 看 NUMA node）——跨 node 读一个 4 KB 数据可能多花 100 ns。

4轮询替代中断

IORING_SETUP_IOPOLL 或设置 /sys/block/nvme0n1/queue/io_poll=1。省掉中断到来的唤醒开销，换取 CPU 空转。适合 QD 饱和的场景（ScyllaDB、SPDK）。

  5. blktrace + btt：分别统计 Q→G、G→I、I→D、D→C 各阶段耗时，定位是调度慢还是硬件慢

14.1 基础题

用户态和内核态有什么区别？为什么需要这种隔离？
CPU 的特权等级（Ring 0/3）实现硬件级隔离。内核态可以执行特权指令（访问硬件、修改页表），用户态被限制。好处：(1) 保护内核不被应用破坏；(2) 进程间隔离；(3) 统一的系统资源管理。代价：每次系统调用需要切换上下文（~100ns）。
文件描述符是什么？它在内核中对应什么数据结构？
fd 是进程级小整数，是三级表的索引：进程 FD Table（per-process）→ Open File Table（system-wide，保存 offset/flags）→ inode（文件元数据）。fork 复制 FD Table 但共享 Open File Entry，所以父子进程读写会互相影响 offset，这是 shell pipe 的基础。
stdio 的 fwrite 和系统调用 write 有什么区别？
fwrite 带用户态缓冲（默认 4KB），多次 fwrite 可能只触发一次 write 系统调用；write 直接陷入内核。fwrite 适合高频小 I/O（摊薄系统调用开销），write 适合大块 I/O 或需要精确控制写入时机的场景（如需要原子性保证）。另外，fork 前必须 fflush，否则子进程会复制缓冲区导致重复输出。

14.2 进阶题

select / poll / epoll 的区别？为什么 epoll 更快？
三点关键差异：(1) 数据结构：select 用位图（fd 上限 1024），poll 用数组，epoll 用红黑树+就绪链表。(2) 拷贝：select/poll 每次调用都要把全部 fd 集合从用户态拷贝到内核态；epoll 用 mmap 共享内存，注册一次即可。(3) 复杂度：select/poll 内核需要 O(n) 遍历所有 fd 检查就绪；epoll 通过回调机制（数据到达时内核把 fd 放入就绪链表），epoll_wait 直接拿就绪 fd，O(1)。所以 epoll 适合 fd 多但活跃度低的场景（典型 C10K）。
epoll 的 LT 和 ET 模式有什么区别？各自适用什么场景？
LT (水平触发)：只要缓冲区还有数据就持续通知，编程简单，可以不一次读完。ET (边缘触发)：只在状态变化时通知一次，必须配合非阻塞 fd 并循环读取直到 EAGAIN，否则丢失事件。ET 减少了 epoll_wait 唤醒次数，吞吐更高（Nginx/Redis 默认 ET），但代码更复杂。实际工程上 LT 足够大多数场景使用。
什么是零拷贝？sendfile 和 splice 的区别？
零拷贝指减少或消除 CPU 参与的数据拷贝。传统 read+send 路径有 4 次拷贝（磁盘→内核→用户→socket→网卡）和 4 次上下文切换。sendfile 直接从文件 fd 发送到 socket fd，数据不经过用户态（2 次 DMA + 1 次 CPU 拷贝描述符）。支持 SG-DMA 的网卡可做到 0 CPU 拷贝。splice 要求一端必须是 pipe，通过管道内核 buffer 中转（0 CPU 拷贝），更灵活（任意 fd 类型）。Kafka/Nginx 用 sendfile 发静态内容，代理类应用用 splice 转发流量。
mmap 和 read 读文件哪个更快？
看场景：mmap 快：大文件顺序/随机访问，多进程共享（共享 Page Cache 且零拷贝），需要内存语义（比如 DB 的 B+Tree 遍历）。read 快：小文件（mmap 建立 VMA 开销固定）、一次性读取、流式处理（read 可以直接写入应用 buffer 无需页表查找）。另外 mmap 可能引发 SIGBUS（文件被截断）、TLB miss、不可控的预读等问题，所以不是银弹。
write() 返回后数据在哪里？如何保证数据落盘？
write 返回只保证数据进入 Page Cache，标记为 dirty 页，断电会丢失。要保证落盘必须调用 fsync(fd)（数据 + 元数据）或 fdatasync(fd)（只同步数据，如果大小没变则省一次元数据写入，更快）。极致性能场景可以 Group Commit：攒一批 write 再一次 fsync。数据库（MySQL redo log、PostgreSQL WAL）都是这样设计。如果用 O_SYNC/O_DSYNC，每次 write 都相当于 write + fsync/fdatasync。

14.3 深度题

io_uring 相比 epoll 有什么优势？为什么说它是"真正的"异步 I/O？
epoll 是 I/O 就绪通知（Reactor），业务仍要自己 read/write，这两个操作本身可能阻塞（文件 I/O 尤其）。io_uring 是 I/O 完成通知（Proactor），用户通过共享内存 ring 提交请求，内核完成后写 CQE。优势：(1) 真正异步文件 I/O（epoll 对普通文件没用）；(2) 批量提交摊薄 syscall；(3) SQPOLL 模式完全无系统调用；(4) 统一接口（read/write/accept/fsync/openat 都能异步）；(5) 操作链（依赖自动化）。测试显示 100 万 IOPS 场景 io_uring 可比 libaio 快 3-5 倍。
如何实现一个高性能的日志库？
关键设计点：(1) 异步化：前台线程写无锁环形队列/MPSC 队列，后台线程刷盘。(2) 格式化前移：用 C++20 fmtlib / std::format 避免 iostream 开销，日志点只传参数不做字符串拼接。(3) 批量刷盘：累积 N 条或 T ms 才触发一次 write + fdatasync（Group Commit）。(4) 双缓冲：前台写 buffer A，后台 flush buffer B，切换无锁。(5) 零拷贝：用 writev 批量刷；或用 mmap 映射日志文件。(6) 绕过 iostream：直接 write fd。参考：spdlog (async mode)、NanoLog、Boost.Log。
TCP 连接 accept 后，如何设计线程模型处理请求？
四种主流模型：(1) 单 Reactor 单线程：Redis 风格，适合 CPU 轻业务。(2) 单 Reactor 多线程：主线程 accept/dispatch，worker 线程池处理业务。(3) 主从 Reactor：Netty/muduo 风格，主 Reactor 只 accept，sub Reactor 池处理读写，业务可再分 worker。(4) 多 Reactor 进程：Nginx 风格，用 SO_REUSEPORT 每进程独立 listen，内核负载均衡。选择取决于：业务是否 CPU 密集；是否需要跨连接状态共享；锁竞争容忍度。一般推荐主从 Reactor + 无锁队列。
一次磁盘写入，从 write() 到数据真正落盘经过哪些层？
完整路径：用户 write() → glibc（可能有 stdio 缓冲）→ 系统调用陷入内核 → VFS vfs_write → 文件系统（ext4）ext4_file_write_iter → Page Cache（写入页并标记 dirty，此时 write 返回）→ 后台 kworker/flush-bdflush 触发 → bio 层（构造 bio 对象）→ blk-mq（多队列块层）→ I/O 调度器（mq-deadline 等）→ 设备驱动（NVMe）→ DMA 到物理介质。fsync 会强制触发后面所有步骤完成。整条链路任何一层都可能成为瓶颈，需要配合 blktrace/bpftrace 定位。

14.4 场景设计题

设计一个能处理 100 万并发长连接的服务器，怎么做？
架构：SO_REUSEPORT 多进程 + 每进程 epoll(ET) + 线程池。内存：每连接内存预算 <4KB（100 万连接 = 4GB），考虑连接池复用、避免每连接独立 buffer。文件描述符：ulimit -n 1048576，fs.file-max 调整。内核参数：net.core.somaxconn（backlog）、net.ipv4.tcp_max_syn_backlog、net.ipv4.tcp_tw_reuse。CPU 亲和：worker 绑定 CPU，中断亲和分散到不同核。协议层：TCP_NODELAY + 自定义协议头携带长度避免 read 多次。监控：连接数、每连接 RTT、epoll_wait 次数、系统调用延迟分布。
