1. 引言 — ABI 是什么，为什么大型 C++ 项目绕不开

1.1 ABI vs API

项API (Application Programming Interface)ABI (Application Binary Interface)
层次源码层二进制层
稳定意味着客户代码可重新编译用新版本客户二进制可不重新编译直接用新 .so
涉及函数签名、类定义、宏name mangling、vtable layout、struct padding、calling convention、异常 unwinding、RTTI 表示
谁制定库作者编译器 + OS + 平台（Itanium ABI / MSVC ABI）

1.2 ABI 兼容性的两个方向

  • 源码兼容 (source-compatible)：旧代码能编译过新版本 → 大部分库都做得到
  • 二进制兼容 (ABI-compatible)：旧二进制能链接新 .so 不崩 → 极难，C++ 历史上 libstdc++ / Qt / Boost 各自经历过痛苦切换

1.3 为什么 ABI 对 Tech Lead 是关键

  • 大型公司发版节奏：核心 .so（基础库）一旦升级，依赖它的几百个二进制如果都要重编 → 发版周期从 1 天变成 1 周
  • 线上事故来源：90% 的 "诡异随机崩 / std::string 长度变怪 / dynamic_cast 失败" 都是 ABI 不一致
  • 第三方库选型：选 Boost / Folly / Abseil 时必须想清楚 ABI 策略
  • 跨平台 / 跨语言：C++ → Python (pybind11) / Rust (cxx-rs) / Java (JNI) 全是 ABI 边界
本指南目标：① 解释 Itanium C++ ABI 的关键字段；② 列出大型项目里"会撞 ABI"的所有典型场景；③ 给出每种场景的工程对策（PIMPL / inline namespace / version script / C ABI 包装）。

2. Itanium C++ ABI 详解 ⭐

GCC / Clang / 大多数 Unix 编译器遵循 Itanium C++ ABI。名字来自 1990s 末为 Itanium 设计、后来成了事实标准。它规定了：

2.1 涵盖的 6 大领域

领域规定
Name Mangling函数 / 变量 / 模板特化的"mangled name"格式
vtable 布局虚函数表的内存格式、offset_to_top、RTTI 指针位置
RTTI 表示type_info 类层次、type 比较语义
对象布局多重继承的 subobject 顺序、虚继承的 vbase pointer
异常 unwindinglanding pad、personality function、cleanup 处理
类型构造 / 销毁顺序构造函数 thunk、虚析构语义

2.2 Itanium ABI 与 MSVC ABI 的差异

项Itanium (GCC/Clang on Linux)MSVC ABI (Windows)
Name mangling_ZN... 格式?... 格式
vtable layout2 个负偏移 slot（offset_to_top + type_info）+ 函数指针数组每个 vtable 只含函数指针；RTTI 存别处
this pointer adjustmentvtable 内 thunk调用方调整 (Adjustor thunk 在虚函数表头)
异常实现SJLJ 或 DWARF tablesSEH（结构化异常处理）
typeid 跨 .so 比较比较mangled name 字符串（默认 weak 链接合并）比较type_info 指针（需要 RTTI 在共享内存）

2.3 ABI 版本号

libstdc++ / Clang 都用"_GLIBCXX_USE_CXX11_ABI" 和 ABI version 字段表征兼容性。GCC 默认从 5.0 起切到"新 ABI"（dual ABI 起步）。看你的二进制：

# 看 GCC ABI version
g++ -dumpfullversion -dumpversion

# 看一个 .o / .so 用什么 ABI 编的
readelf -p .GCC.command.line my.o

# 看 libstdc++ 的 dual ABI 选择
nm libfoo.so | grep std::string | head
Itanium ABI 文档不长（~150 页），但极致严格。任何"编译器扩展"或"新 C++ 标准特性"都要先在 ABI 委员会通过才能正式落地，否则就是"unstable"。这就是为什么 C++20 modules / coroutines 早期编译器有但 ABI 不稳。

3. Name Mangling — 函数名怎么变成符号

3.1 为什么需要 mangling

C++ 支持函数重载、命名空间、模板，但链接器只懂"符号 = 字符串"。所以编译器把所有这些信息编码进唯一的 mangled name。

3.2 Itanium mangling 范例

C++ 声明mangled name (Itanium)
void foo()_Z3foov
void foo(int)_Z3fooi
void foo(int, double)_Z3fooid
namespace ns { void bar(); }_ZN2ns3barEv
class A { void m(int); }_ZN1A1mEi
template<> void f<int>()_Z1fIiEvv

3.3 反 mangling 工具

# c++filt — 单个符号
$ c++filt _ZN2ns3barEv
ns::bar()

# 整个文件 / 流
$ nm libfoo.so | c++filt | head
0000000000001234 T MyClass::doWork(int)
0000000000001280 T std::vector<int, std::allocator<int> >::push_back(int const&)

# LLVM 替代
$ llvm-cxxfilt _ZN2ns3barEv

3.4 mangling 的"稳定"承诺

Itanium ABI 承诺给定 C++ 声明的 mangled name 永不变化（除非声明本身改）。这是 C++ ABI 兼容的基石—— 只要你的 .h 文件不动，老 .o 永远能链新 .so。

反例：① 改函数参数类型 → mangled 变 → 老二进制 undefined reference；② 加默认参数（在声明里）→ 通常不变；③ noexcept 自 C++17 起属于函数类型，函数指针 mangled 中包含 noexcept → 老二进制对不上新版本（C++17 之前 noexcept 不参与）。

3.5 extern "C" 抑制 mangling

extern "C" void my_api(int);          // 符号名直接是 "my_api"
extern "C" {
    void f1();                       // 都不 mangle
    void f2(int);
}

// 限制：
// 1. 不能用 C++ 特性 (重载、引用、默认参数、namespace、模板)
// 2. 函数体内仍是 C++，可用 C++ 语言特性
// 3. 类成员函数不能 extern "C"
设计原则：库公开头文件如果想跨编译器 / 跨语言（被 C / Python / Rust 调用），必须用 C ABI（extern "C" + POD struct）。C++ 类型只在内部使用。

4. vtable 与 RTTI 布局

4.1 vtable 在内存里的精确格式（单继承）

// 给定:
struct B { virtual void f(); virtual void g(); virtual ~B(); };

// Itanium ABI vtable 布局:
// 地址 ↓                内容
// vtable - 16:  offset_to_top (= 0 for B itself)
// vtable - 8:   type_info* for B
// vtable + 0:   &B::f             ★ B 对象的 vptr 指向这里
// vtable + 8:   &B::g
// vtable + 16:  &B::~B()  (complete destructor)
// vtable + 24:  &B::~B()  (deleting destructor)

关键点：vptr 指向 vtable + 0（即第一个虚函数），不是 vtable 起始。typeid(obj) 实现为 *(type_info**)(vptr - 8)，dynamic_cast 实现为 __dynamic_cast(obj, offset_to_top, src_type_info, dst_type_info)。

4.2 多重继承的 vtable

struct A { virtual void f(); int a; };
struct B { virtual void g(); int b; };
struct C : A, B { void f() override; void g() override; };

// C 对象布局:
//   [A subobj: vptr_A, a]   ← C* 的首地址
//   [B subobj: vptr_B, b]   ← 偏移 16

// C 的 vtable 实际有"两段":
//   primary vtable (A 接口):
//     offset_to_top = 0
//     type_info* = &typeinfo_for_C
//     [+0]: &C::f
//   secondary vtable (B 接口):
//     offset_to_top = -16    ← 关键! 给"调整 this"用
//     type_info* = &typeinfo_for_C
//     [+0]: &C::g_thunk      ← 不是直接 C::g, 是 thunk

// thunk 内部:
//   sub rdi, 16     ; 调整 this: B subobj → C 起始
//   jmp C::g        ; 跳真正实现

4.3 ABI 影响：在派生类加成员变量不算 ABI 破坏

派生类对象布局是"基类 subobject + 派生类自家数据"。加自家数据放尾部 → 不影响基类的 layout → 老调用者通过基类指针访问仍 OK。

反例：在基类加成员变量 → 派生类整体布局变 → 老二进制的 sizeof / offsetof 错位 → 崩。这就是 ABI 稳定的库（如 Qt）从不给已发布的基类加数据成员，只用 PIMPL。

4.4 加虚函数也是 ABI 破坏

  • 加虚函数 → vtable 多一项 → vtable 大小变 → 所有派生类的 vtable 也要重生成 → 老二进制访问 vtable 索引错位 → 崩
  • "末尾"加虚函数也不行（虽然看似 layout 不变），编译器对 vtable 索引是编译期固化的

4.5 跨 .so 的 typeid 问题

// libA.so 和 libB.so 各自有 typeid(MyClass)
// 两个 type_info 是同一个吗？

// 答: 视实现而定。Itanium ABI 规定 type_info 用
 mangled name 字符串比较, 而且 typeinfo 符号是 weak
// 链接器合并 → 两 so 看到同一个 type_info → typeid 相等 ✓

// 例外: -fvisibility=hidden 后 typeinfo 也 hidden

修复：__attribute__((visibility("default"))) 显式导出 typeinfo，或使用 -fvisibility-ms-compat（Clang）让 typeinfo 默认可见。

5. 异常 ABI — Stack Unwinding

5.1 两种主流实现

实现原理开销典型
SJLJ (setjmp/longjmp)每次进 try 调 setjmp 注册 landing pad正常路径有额外指令嵌入式、旧编译器
DWARF tables (zero-cost)编译器生成调用栈布局表，正常路径无任何代码；抛异常时查表正常 0；异常路径 ms 级GCC/Clang on Linux 默认

5.2 DWARF 异常 ABI 的关键字段

  • .eh_frame 段：每个函数的栈展开信息（用 DWARF CFI 编码）
  • .gcc_except_table：landing pad 表（哪些指令范围在 try 块内、对应的 catch handler）
  • personality function：__gxx_personality_v0，决定具体怎么处理异常 type 匹配
  • type_info 必须在 unwinder 看得到的地方 — 这是异常跨 .so 必须共享 type_info 的原因

5.3 跨 .so 抛异常的两大要求

要求 1：所有相关 .so 必须动态链接同一个 libstdc++ —— 否则 personality function 不一致，unwinder 崩。
要求 2：异常类型的 type_info 必须共享（可见性 default）—— 否则 catch 匹配失败，异常传到 main 后变 std::terminate。

5.4 常见反模式：把 libstdc++ 静态链接

// ❌ 不要这么干 — 每个 .so 自带 libstdc++ 副本
g++ -static-libstdc++ -shared myplugin.cpp -o myplugin.so

// 后果:
//   - myplugin.so 抛的 std::exception 在 main 里 catch 不到
//   - 各 so 之间共享 std::cout 失败
//   - 内存浪费 (每 so 重复几 MB libstdc++)

5.5 noexcept 与 ABI

C++17 起 noexcept 成为函数类型的一部分。重要影响：

// 这两个不是同一类型 (C++17+):
void (*p1)();              // noexcept(false)
void (*p2)() noexcept;     // noexcept(true)

// 函数指针类型的 mangling 也不一样
// 老二进制把 noexcept(false) 指针传给期望 noexcept(true) 的新 API → ABI 不兼容

6. libstdc++ Dual ABI ⭐

6.1 GCC 5 的"切换"

2015 年 GCC 5 把 std::string 从 COW (Copy-on-Write) 改成 SSO (Small String Optimization)。原因：C++11 要求 std::string::data() 返回 const 指针，COW 实现违反新标准。

但不能简单替换—— 大量发布的二进制（Ubuntu / RHEL / 自家产品）都依赖旧 std::string。怎么办？

6.2 Dual ABI 的天才设计：inline namespace

// libstdc++ 的实际做法 (简化)
namespace std {
#if _GLIBCXX_USE_CXX11_ABI == 1
    inline namespace __cxx11 {                // 新 ABI
        class string { /* SSO impl */ };
        class list   { /* 含 size 字段 */ };
    }
#else
    // 旧 ABI (C++03 兼容)
    class string { /* COW impl */ };
    class list   { /* 不含 size */ };
#endif
}

// 用户代码:
std::string s;       // 实际是 std::__cxx11::string OR std::string
//   取决于 -D_GLIBCXX_USE_CXX11_ABI=0 或 1

// Mangled name 也不同 (因为命名空间嵌套不同):
// 新: _ZNSt7__cxx1112basic_stringIcSt11char_traitsIcESaIcEEC1Ev
// 旧: _ZNSsC1Ev
结果：同一个 libstdc++.so.6 里同时存在两套 std::string 实现，老二进制找到老符号，新二进制找到新符号。新旧 ABI 完全隔离，不会"看似匹配实际崩"。

6.3 如何检查 / 选择

# 查 GCC 默认 ABI 是否新
g++ -E -dM -x c++ - < /dev/null | grep _GLIBCXX_USE_CXX11_ABI
// 输出: #define _GLIBCXX_USE_CXX11_ABI 1   (GCC 5+ 默认新)

# 强制用旧 ABI (兼容老二进制)
g++ -D_GLIBCXX_USE_CXX11_ABI=0 ...

# 看符号属于哪一套
nm myapp | c++filt | grep "string" | head
# 含 __cxx11 = 新；不含 = 旧

6.4 Dual ABI 的隐藏陷阱

  • 跨 .so 传递 std::string：双方必须用同一个 ABI 编译 — 否则崩
  • 新代码链旧库：第三方 .so 是旧 ABI 编的，自家代码默认新 ABI → 调用 std::string 接口报"undefined symbol" 或链接乱
  • 大型企业内部：通常强制锁死一个 ABI（要么全旧 要么全新），不混用

6.5 其他 stdlib 的 ABI 切换

事件影响
GCC 5: std::string COW → SSO需要 Dual ABI
GCC 5: std::list 加 size 字段同上
libc++ 2024: std::pair ABI v2_LIBCPP_ABI_VERSION=2 开关
MSVC 在每个 Major version 之间 ABI 不兼容必须重新编译
各家 stdlib 的 hash table 实现（unordered_map）布局不兼容跨 stdlib 不能传

7. 跨 .so 传递 STL 类型的陷阱

7.1 风险矩阵

跨 .so 传递安全原因
int / POD struct (C ABI 兼容)✅C ABI 完全稳定
std::string by value⚠️两边必须同 libstdc++ + 同 dual ABI 设置
std::vector<T> by ref/ptr⚠️同上 + T 的 ABI 也要一致
std::unique_ptr<T>⚠️跨 stdlib 几乎一定崩 (deleter 类型)
std::shared_ptr<T>⚠️控制块布局必须一致
抛异常跨 so⚠️type_info 必须 visibility=default
虚函数跨 so 调用⚠️基类 layout 必须 ABI 稳定
模板实例化结果跨 so⚠️编译器一致 + 同样实例化参数

7.2 经典 case：std::string ABI mismatch

// libA.so (用 -D_GLIBCXX_USE_CXX11_ABI=0 编)
extern "C++" std::string get_name();

// libB.so (默认新 ABI 编)
std::string s = get_name();         // ★ ABI mismatch → 链接错或运行时崩
//                          实际调到旧 ABI 的函数, 但本地按新 ABI 期待返回

// 错误信息:
// undefined reference to `get_name[abi:cxx11]()'
// 或运行时段错误 (布局错位)

7.3 解决方案：C ABI 层 wrapper

// 公共头 mylib.h (C-only, 任何 stdlib 都能用)
#ifdef __cplusplus
extern "C" {
#endif
    const char* mylib_get_name();             // 返回 C 字符串
    void        mylib_free_string(const char*); // 配套 free
#ifdef __cplusplus
}
#endif

// 内部 mylib.cpp - 仍可用任意 C++ 类型
extern "C" const char* mylib_get_name() {
    std::string s = compute();      // 内部用 std::string
    return strdup(s.c_str());          // 出 ABI 边界用 C 字符串
}
extern "C" void mylib_free_string(const char* p) { free(const_cast<char*>(p)); }
C ABI wrapper 是企业级 C++ 库的"ABI 防火墙"。所有跨 .so/.dll 暴露的接口都用 C ABI，C++ 类型只在自己 .so 内部用。换 stdlib / 换编译器 / 升级 GCC 都不影响。

7.4 用 vendoring stdlib 解决

Google / Meta 内部的解决方案：每个二进制都 vendoring 自己的 libstdc++ —— 单一 monorepo + 统一 toolchain，所有代码用同一份 stdlib 编译。静态链 libstdc++ + visibility=hidden，杜绝跨 .so 的 STL 传递问题。代价是二进制大 + 不能用系统 libstdc++ 的安全更新。

8. inline namespace —— ABI 版本演化的标准武器

8.1 基本概念回顾

详细机制见 对象模型 §14.9。本节聚焦 库设计场景。

8.2 给自家库版本化导出

// myapi.h v1 — 首发
namespace myapi {
    inline namespace v1 {
        class Connection {
            int fd_;
        public:
            void connect(const char*);
            void close();
        };
    }
}

// 用户写 myapi::Connection — 实际是 myapi::v1::Connection
// Mangled name 含 v1 → 与未来的 v2 不冲突

8.3 引入 v2（不兼容升级）

// myapi.h v2 — 加成员 + 改实现
namespace myapi {
    namespace v1 {                             // 老版本保留，去掉 inline
        class Connection { /* 原样 */ };
    }
    inline namespace v2 {                       // 新默认
        class Connection {
            int fd_;
            std::string peer_;                  // 新加成员
            int timeout_;                       // 新加成员
        public:
            void connect(const char*);
            void close();
            void set_timeout(int ms);          // 新 API
        };
    }
}

// 老用户:
myapi::v1::Connection c;     // ✅ 老代码还能用，老 .so 也兼容

// 新用户:
myapi::Connection c;         // ✅ 自动是 v2，享新功能
c.set_timeout(5000);

8.4 标准库自己的用法

标准库特性用法
std::__cxx11libstdc++ dual ABI 切换
std::literals::string_literals嵌套 inline namespace 让 using namespace std::literals; 三层粒度可选
std::ranges不少实现用 inline namespace 保护内部细节

8.5 inline namespace 的最佳实践

  • 从第一版开始就用 inline namespace v1 { ... } —— 即使你现在没打算升级
  • 升级时不要删 v1，只把 inline 移到 v2
  • 同一 namespace 不要嵌套太多层 inline（debug 困难）
  • 禁止对已发布的 inline namespace 内部类做 ABI 修改 —— 加新成员/虚函数

9. PIMPL 与 ABI 稳定性

9.1 PIMPL 的 ABI 收益

// ❌ 直接暴露成员 — ABI 不稳
class Widget {
    int a_;
    std::string b_;          // 加 / 改 / 删任一字段都改 sizeof → ABI 破坏
public:
    void do_work();
};

// ✅ PIMPL — 用户只看到 1 个指针
// widget.h
class Widget {
    struct Impl;
    std::unique_ptr<Impl> p_;     // sizeof(Widget) = 1 指针，永不变
public:
    Widget();
    ~Widget();
    void do_work();
};

// widget.cpp — Impl 完整定义在这里
struct Widget::Impl {
    int a;
    std::string b;
    // ★ 这里任意改不影响公开 ABI！
};
PIMPL 是 Qt / ICU / WebKit 等大型库保持几十年 ABI 兼容的核心武器。Qt 的 QObject / QWidget 等所有公共类都是 PIMPL，所以 Qt 5.x 内部能反复重构 implementation 不影响用户。

9.2 PIMPL 的成本

项成本
一次堆分配（创建 Impl）~30 ns
每次访问多一次指针解引用~1 cache line miss 风险
不能 inline 公开成员函数（实现在 .cpp）失去内联机会
headers 中无法 = default 析构（Impl 未定义）必须在 .cpp 里写 Widget::~Widget() = default;，详见智能指针 §3.7

9.3 PIMPL 的最佳实践

// 1. Move-only PIMPL (现代写法)
class Widget {
    struct Impl;
    std::unique_ptr<Impl> p_;
public:
    Widget();
    ~Widget();
    Widget(Widget&&) noexcept;
    Widget& operator=(Widget&&) noexcept;
    Widget(const Widget&) = delete;
};

// 2. Fast PIMPL (用 aligned_storage 避免堆分配)
class Widget {
    alignas(8) char storage_[128];     // 在头里"假装"分配足够空间
    struct Impl;
    Impl* impl() { return reinterpret_cast<Impl*>(storage_); }
public:
    Widget();
    ~Widget();
    // 公开头需要知道 Impl 的 size + alignment, 但不要完整定义
};
// 收益: 零堆分配; 风险: 改 Impl 大小破坏 ABI (要预留 buffer)

9.4 何时不该用 PIMPL

  • 性能极敏感的小对象（每次访问都热路径）
  • 需要 inline 公开成员函数（PIMPL 强迫所有公开函数在 .cpp 实现）
  • 没有跨 .so 边界 / 不需要 ABI 稳定的 internal 模块
  • Template-heavy 类（PIMPL 主要救非模板类）

10.1 visibility 复习

详细规则见 编译指南 · L2。本节聚焦"怎么给 ABI 稳定的库设计 visibility"。

10.2 默认 hidden + 显式导出 API

# CMake 启用
set_target_properties(mylib PROPERTIES
    CXX_VISIBILITY_PRESET hidden
    VISIBILITY_INLINES_HIDDEN ON)

10.3 Version Script —— 更精细的导出 + 版本化

// mylib.ver
LIBMYLIB_1.0 {
    global:
        mylib_init;
        mylib_call;
        // 类的所有成员
        _ZN10ConnectionC1Ev;
        _ZN10Connection7connectEPKc;
        // 或用通配
        _ZN10Connection*;
        // vtable + typeinfo
        _ZTV10Connection;
        _ZTI10Connection;
    local:
        *;                // 其他全部隐藏
};

LIBMYLIB_2.0 {
    global:
        mylib_call_v2;    // 新加的 API 单独一组
    local:
        *;
} LIBMYLIB_1.0;          // 依赖 1.0

# 链接时
g++ -shared -Wl,--version-script=mylib.ver -o libmylib.so.2 *.o

用户链接时拿到的符号自动绑 version 标记，nm -D libmylib.so.2 | grep mylib_call 显示 mylib_call@@LIBMYLIB_1.0。新 API 加在新 version 节，老 API 永远在老 version 节 → 老二进制完全兼容新 .so。这就是 glibc 几十年 ABI 稳定的奥秘。

10.4 versioned symbol 实战：替换实现但保留旧符号

// mylib.cpp
extern "C" {
    int mylib_call_impl_v1(int x) { return x + 1; }    // 老实现
    int mylib_call_impl_v2(int x) { return x * 2; }    // 新实现
}

// 用 .symver 把两个实现绑到同名不同 version
__asm__(".symver mylib_call_impl_v1,mylib_call@LIBMYLIB_1.0");
__asm__(".symver mylib_call_impl_v2,mylib_call@@LIBMYLIB_2.0");   // 默认版本

// 老二进制链 mylib_call@LIBMYLIB_1.0 → 跑老实现
// 新编译的二进制链 mylib_call@@LIBMYLIB_2.0 → 跑新实现

11. soname / dlopen 进阶

11.1 soname 是什么

libfoo.so      → 编译时 symlink (开发用)
libfoo.so.2    → soname (运行时找)
libfoo.so.2.5.1 → 实际文件 (真实文件)

# 看 soname
$ readelf -d libfoo.so.2.5.1 | grep SONAME
0x000000000000000e (SONAME)  Library soname: [libfoo.so.2]

# 看可执行文件依赖什么 soname
$ readelf -d myapp | grep NEEDED
0x0000000000000001 (NEEDED)  Shared library: [libfoo.so.2]

规则：

  • major version 增（2 → 3）= ABI 不兼容 → soname 变
  • minor / patch version 增 = ABI 兼容 → soname 不变，老二进制继续工作
  • 多版本可并存系统中（libfoo.so.2 + libfoo.so.3），各自服务对应程序

11.2 dlopen / dlsym 与 ABI

#include <dlfcn.h>

void* h = dlopen("./plugin.so", RTLD_LAZY);
if (!h) {
    fprintf(stderr, "%s\n", dlerror());
    return -1;
}

// 取符号 — 必须 extern "C" 或知道 mangled name
using CreatorFn = MyPlugin* (*)();
CreatorFn create = reinterpret_cast<CreatorFn>(dlsym(h, "create_plugin"));

if (!create) {
    fprintf(stderr, "missing symbol: %s\n", dlerror());
}

auto* p = create();
p->use();
dlclose(h);

// 插件标准做法 (plugin.cpp):
extern "C" MyPlugin* create_plugin() {
    return new MyPluginImpl();        // new 在 plugin.so 内, delete 也必须在
}
extern "C" void destroy_plugin(MyPlugin* p) {
    delete p;
}
插件 ABI 陷阱：① 主程序和插件必须用同样的 stdlib + ABI；② 跨 dlopen 边界的对象 new 必须配套 delete 在同一边（不同 stdlib 的 new/delete 实现不一）；③ 抛跨 dlopen 边界的异常需 stdlib 共享。

11.3 RTLD_GLOBAL 与 typeinfo 合并

// 默认 RTLD_LOCAL — 插件符号不暴露给后续加载者
dlopen("a.so", RTLD_LAZY);                  // 隔离

// RTLD_GLOBAL — 让 a.so 的符号可见，b.so 也能用
dlopen("a.so", RTLD_LAZY | RTLD_GLOBAL);    // 共享

// 关键: typeinfo 跨 .so 比较的"同一性"
// 要求所有相关 so 都 RTLD_GLOBAL 加载,
// 否则各 so 看到的 typeinfo 不同 → dynamic_cast 跨 so 失败

12.2 MSVC 跨 DLL 导出类

// public header
#ifdef MYLIB_EXPORTS
  #define MYLIB_API __declspec(dllexport)
#else
  #define MYLIB_API __declspec(dllimport)
#endif

class MYLIB_API Widget {
    // 整个类导出 - 含成员函数、vtable
    int a_;
public:
    Widget();
    virtual ~Widget();
};

12.3 "C4251" 警告 — 跨 DLL 传 STL 的官方告警

// MSVC 的经典警告
class MYLIB_API Widget {
    std::vector<int> data_;          // warning C4251: 'std::vector' needs to have
                                       // dll-interface to be used by clients
};

原因：客户 DLL 必须用同 MSVC 版本 + 同 CRT 才能正确处理 std::vector。MSVC 警告你"这是潜在的 ABI 陷阱"。修复同 Linux：① PIMPL；② C ABI；③ 强制锁 toolchain。

12.4 COM 的"老派 ABI 稳定"思路

Windows COM 用纯虚接口 + 引用计数做 ABI：

// COM 接口长这样 - 永远是 vtable-based 纯虚
struct IUnknown {
    virtual HRESULT QueryInterface(REFIID, void**) = 0;
    virtual ULONG   AddRef() = 0;
    virtual ULONG   Release() = 0;
};

这种"接口 = 纯虚类 + 引用计数 + GUID"是 90 年代的方案，但跨编译器跨语言稳定性极强。WinRT / UWP 也基于此。

13. 真实生产事故 5 例

13.1 案例 1：GCC 升级 → std::string ABI 不一致

背景：A 团队的 .so 用 GCC 4.8（旧 ABI），B 团队的应用用 GCC 7（新 ABI）。
症状：链接报 undefined reference to 'A::get_name[abi:cxx11]()'。
原因：B 端按 std::__cxx11::string 找符号，A 端只导出 std::string（旧）。
修复：A 团队用 -D_GLIBCXX_USE_CXX11_ABI=1 重编 .so，或对外暴露 C ABI wrapper。

13.2 案例 2：插件 dlopen 后 dynamic_cast 失败

背景：主程序定义 class Plugin，插件继承它实现具体类。
症状：dynamic_cast<MyPlugin*>(p) 在主程序中总是返回 nullptr，但插件内部用没问题。
原因：主程序和插件各自有 typeinfo for Plugin，因为 -fvisibility=hidden 没显式导出 typeinfo。两个 type_info 地址不同 → dynamic_cast 比较失败。
修复：把 Plugin 类用 __attribute__((visibility("default"))) 标记。

13.3 案例 3：跨 .so 抛异常被 terminate

背景：库 .so 静态链了 libstdc++（-static-libstdc++）。
症状：库内抛 std::runtime_error，主程序 catch (std::exception&) 没生效，整个进程 abort。
原因：库自带一份 libstdc++，std::exception 的 typeinfo 与主程序的不同 → unwinder 找不到匹配 catch。
修复：移除 -static-libstdc++，所有 .so 共享系统 libstdc++。

13.4 案例 4：升级第三方库后 sizeof(MyObj) 变了，老二进制访问越界

背景：第三方库 libfoo.so 给 class Foo 加了一个成员变量，但 soname 没升级。
症状：自家程序不重新编译就用新 .so → Foo 内存布局错乱 → 偶发段错误。
原因：库作者违反 ABI 稳定承诺，加成员没 bump soname major version。
修复：① 短期回退到老 .so；② 长期：自家代码用 PIMPL / C ABI 包装第三方库。

13.5 案例 5：Debug + Release 混链崩

背景：Windows 上 Debug 版主程序 + Release 版的第三方 DLL。
症状：传 std::string 给 DLL 导致随机崩。
原因：MSVC Debug CRT 的 std::string 比 Release 多了"checked iterator"等字段 → sizeof / 布局完全不同。
修复：Debug 主程序只能链 Debug 第三方 DLL，反之亦然。或者 DLL 用 C ABI。

14. ABI 安全设计准则

14.1 库设计的"5 条军规"

  1. 公开头只暴露 C ABI（POD struct + extern "C" 函数）。C++ 类型只在自己 .so 内部用
  2. 必须暴露 C++ 类时用 PIMPL：公开头只放指针 + 几个稳定 inline 接口，所有真实成员藏在 impl
  3. 从第一版就用 inline namespace v1 包装 —— 即使现在不打算升级
  4. visibility=hidden 默认 + 显式 API_EXPORT 公共符号；用 version script 做版本化
  5. 所有交付物锁住 toolchain：编译器版本、libstdc++ 版本、dual ABI 设置统一

14.2 ABI 修改的"红绿表"

变更ABI 影响
加新函数（非虚）🟢 安全
加 inline 函数🟢 安全
给函数加默认参数🟢 安全（声明 mangling 不变）
改函数实现（保签名）🟢 安全
给类加 static 成员函数🟢 安全
给类加成员变量🔴 ABI 破坏
加虚函数 / 改虚函数顺序🔴 ABI 破坏
改基类🔴 ABI 破坏
修改 inline 函数的实现🟡 风险（老二进制有内联拷贝）
把成员函数改成 noexcept (C++17+)🟡 风险（mangling 变化）
把非虚改成虚🔴 ABI 破坏
用 PIMPL 包装的 Impl 内任意改🟢 安全 ★

14.3 abidiff —— 自动检查 ABI 变化

# libabigail 套件 - 比较两个 .so 的 ABI 差异
abidiff libmylib.so.1.0 libmylib.so.2.0
# 输出 ABI 不兼容的变化（加/删/改函数、改 struct 布局等）

# CI 集成: 每次发版前对比 baseline
abidw --out-file v1.abi libmylib.so.v1
abidw --out-file v2.abi libmylib.so.v2
abidiff v1.abi v2.abi

大型项目（systemd / libabigail 自身）每个 CI run 都跑 abidiff，防止意外的 ABI 破坏。
