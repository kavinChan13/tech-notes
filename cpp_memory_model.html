四、C++11 内存模型 (核心：多线程与原子操作)

C++11 正式定义了多线程内存模型，解决了此前"未定义行为泛滥"的问题。核心概念如下：

4.1 为什么需要内存模型？

每个 CPU 核心有自己的缓存和 Store Buffer，写入不会立即对其他核心可见。C++ 内存模型通过 std::atomic 和内存序来控制这种可见性。

五、Happens-Before 关系

Happens-Before 是 C++ 内存模型中最核心的概念——如果操作 A happens-before 操作 B，则 A 的效果保证对 B 可见。

建立 happens-before 的方式：

  • 同一线程中的程序序：前面的语句 happens-before 后面的语句
  • release → acquire 配对：release 写 happens-before 对应的 acquire 读
  • mutex lock/unlock：unlock happens-before 下一次 lock
  • thread create/join：创建线程 happens-before 新线程首语句；线程末尾 happens-before join 返回

六、六种内存序 (Memory Ordering)

C++11 提供了六种 std::memory_order，用于精确控制原子操作的可见性和排序约束：

七、实战代码示例

Relaxed: 简单计数器

// relaxed 只保证原子性，不保证排序——适合不依赖顺序的场景
std::atomic<int> counter{0};

void increment() {
    for (int i = 0; i < 1000; ++i) {
        counter.fetch_add(1, std::memory_order_relaxed);
    }
}

int main() {
    std::thread t1(increment), t2(increment);
    t1.join(); t2.join();
    // counter 最终一定是 2000 —— 原子性保证了这一点
    std::cout << counter.load(std::memory_order_relaxed);  // 2000
}
relaxed 是开销最小的内存序。当你只需要原子性（不被撕裂），不需要和其他变量同步时使用。

Acquire-Release: 生产者-消费者

std::atomic<bool> ready{false};
int data = 0;

void producer() {
    data = 42;                                    // ← 普通写
    ready.store(true, std::memory_order_release);  // ← release：刷出之前所有写
}

void consumer() {
    while (!ready.load(std::memory_order_acquire))  // ← acquire：同步之前所有写
        ;  // 自旋等待
    assert(data == 42);  // ✅ 保证成功！release 保证 data=42 对 acquire 可见
}
release 把"屏障之前的所有写"打包发布；acquire 把这些写全部接收。这是最常用的同步模式。

Sequential Consistency: Dekker 算法

std::atomic<bool> x{false}, y{false};
int r1 = 0, r2 = 0;

void thread_a() {
    x.store(true);   // 默认 seq_cst
    r1 = y.load();    // 默认 seq_cst
}

void thread_b() {
    y.store(true);   // 默认 seq_cst
    r2 = x.load();    // 默认 seq_cst
}

// seq_cst 保证: r1 == 0 && r2 == 0 不可能同时发生！
// 如果用 relaxed 或 acquire/release，这种情况是可能发生的。
seq_cst 建立了一个全局总序——所有线程对所有 seq_cst 操作看到完全相同的顺序。开销最大，但最安全。

Spinlock (自旋锁)

class SpinLock {
    std::atomic_flag flag = ATOMIC_FLAG_INIT;
public:
    void lock() {
        while (flag.test_and_set(std::memory_order_acquire)) {
            // 自旋——acquire 确保临界区操作不会被重排到 lock 之前
        }
    }
    void unlock() {
        flag.clear(std::memory_order_release);
        // release 确保临界区操作不会被重排到 unlock 之后
    }
};
test_and_set 是一个原子 RMW (Read-Modify-Write) 操作。acquire + release 配对保证临界区的操作不会"泄漏"出锁的范围。

双重检查锁定 (DCLP)

std::atomic<Singleton*> instance{nullptr};
std::mutex mtx;

Singleton* getInstance() {
    Singleton* p = instance.load(std::memory_order_acquire);  // 第一次检查
    if (!p) {
        std::lock_guard<std::mutex> lock(mtx);
        p = instance.load(std::memory_order_relaxed);          // 第二次检查
        if (!p) {
            p = new Singleton();
            instance.store(p, std::memory_order_release);      // 发布
        }
    }
    return p;
}

// 更简洁的 C++11 方式（Meyers' Singleton）：
Singleton& getInstance() {
    static Singleton instance;  // C++11 保证线程安全的局部静态初始化
    return instance;
}
C++11 的 static 局部变量初始化是线程安全的，大多数情况下优先使用 Meyers' Singleton。

八、数据竞争与未定义行为

C++ 标准的定义：当两个线程同时访问同一内存位置，且至少一个是写操作，且没有通过原子操作或同步原语（mutex等）建立 happens-before 关系时，就是数据竞争（data race），属于未定义行为（UB）。

十二、常见误区

误区真相
volatile 可以替代 atomic volatile 只防止编译器优化（如不省略读写），但不保证原子性和跨线程可见性。Java 的 volatile 和 C++ 的不同！
int 读写天然是原子的 即使在 x86 上对齐的 int 读写不会被撕裂，但编译器可能重排指令。标准说：不用 atomic/mutex 就是 UB。
加了 mutex 就不需要 atomic 如果所有对变量的访问都在 mutex 保护下，确实不需要 atomic。但如果有无锁的"快速路径"就需要。
relaxed 没用 计数器 (shared_ptr 引用计数递增)、统计等只需要原子性不需要同步的场景，relaxed 最高效。
seq_cst 太慢，不要用 在 x86 上，acquire/release 的读写已经免费了，seq_cst 只在 store 上多一条 MFENCE。先保证正确，再优化。

13.1 Cache 一致性 — MESI 协议

多核 CPU 每个核都有自己的 L1/L2，Cache line（通常 64B）通过 MESI 协议保持一致：

状态含义写操作代价
Modified本核独占、已脏（与内存不一致）直接写
Exclusive本核独占、干净直接写（转 M）
Shared多核共享、干净发 Invalidate，等所有副本失效
Invalid无效先发 Read-for-Ownership

推论：写共享变量代价 ≫ 写独占变量。这是假共享（false sharing）性能塌陷的根源——两个毫不相关的变量被放到同一 cache line，相互拖死。

// 典型假共享
struct Counters {
    std::atomic<long> a;   // 线程 A 疯狂写
    std::atomic<long> b;   // 线程 B 疯狂写 — 同一 cache line ！
};

// 修复：按 cache line 对齐
struct alignas(64) Counters {
    alignas(64) std::atomic<long> a;
    alignas(64) std::atomic<long> b;
};
// C++17:
constexpr std::size_t CL = std::hardware_destructive_interference_size; // 常为 64

对应关系：atomic_thread_fence(memory_order_seq_cst) 在 x86 下生成 mfence；memory_order_acquire/release 在 x86 下可能只是编译器屏障。

// 独立 fence，很少用；绝大多数场景用原子操作自带的 order 参数
std::atomic<int> x{0}, y{0};

// Thread 1
x.store(1, std::memory_order_relaxed);
std::atomic_thread_fence(std::memory_order_release);   // 独立 release
y.store(1, std::memory_order_relaxed);

// Thread 2
if (y.load(std::memory_order_relaxed) == 1) {
    std::atomic_thread_fence(std::memory_order_acquire); // 独立 acquire
    assert(x.load(std::memory_order_relaxed) == 1);   // 成立
}

13.4 DCL 双检锁单例 — 经典案例

C++11 之前 DCL 是著名的反面教材（因为没有内存模型保证），C++11 之后有两种正确写法：

// 方式 1: Meyers' Singleton (C++11 最佳，语言保证线程安全)
Singleton& instance() {
    static Singleton s;     // C++11 保证：静态局部变量初始化线程安全
    return s;
}

// 方式 2: DCL with atomic (跨平台)
static std::atomic<Singleton*> inst{nullptr};
static std::mutex m;

Singleton* get() {
    Singleton* p = inst.load(std::memory_order_acquire);   // ① 快路径
    if (!p) {
        std::lock_guard lk(m);
        p = inst.load(std::memory_order_relaxed);
        if (!p) {
            p = new Singleton();                        // ② 构造
            inst.store(p, std::memory_order_release);      // ③ 发布
        }
    }
    return p;
}

为什么需要 acquire/release：避免其他线程看到 inst != nullptr 但对象还没构造完的"半成品"。release 保证 ② 在 ③ 之前可见，acquire 保证 ① 读到指针后再读对象内容不会提前。

13.6 thread_local 与 TLS 实现

thread_local 变量每个线程一份独立副本，底层通过 TLS (Thread Local Storage) 实现：

  • Linux：%fs 段寄存器指向当前线程 TCB，thread_local int x 访问展开为 mov %fs:offset, %eax
  • Windows：%gs 段寄存器指向 TEB
  • 动态初始化：首次访问时才构造，可能有开销（__tls_init 守护）
  • dlopen 加载的 so 里的 thread_local：GCC 有历史 bug（initial-exec vs global-dynamic TLS 模型），老代码慎用
// 典型用途：无锁日志 / per-thread 缓冲
thread_local char logBuf[4096];
thread_local std::mt19937 rng{std::random_device{}()};  // 每线程独立 RNG

13.8 高频面试 Q&A

Q1: volatile 能用来做多线程同步吗？

❌ 不能。volatile 只阻止编译器优化（缓存到寄存器、合并读写），不提供原子性、也不产生内存屏障。在 ARM 等弱序架构上必然出错。Java 的 volatile 才等同于 C++ 的 atomic。
唯一正确用途：MMIO 内存映射寄存器、信号处理（sig_atomic_t）、setjmp/longjmp 保护的变量。

Q2: happens-before、synchronizes-with、sequenced-before 的关系？

  • sequenced-before：单线程内，分号前后的程序顺序
  • synchronizes-with：一个 release 的 store 与读到该值的 acquire load 之间
  • happens-before：上述二者的传递闭包——是内存可见性的"官方"定义

若 A happens-before B，则 A 的所有副作用在 B 中可见。seq_cst 额外提供全局单一顺序。

Q3: memory_order_consume 为什么不推荐？

consume 试图只保证数据依赖的顺序（比 acquire 更弱、更快，适合 RCU 指针发布）。但：

  • 编译器难以精确追踪数据依赖，几乎所有实现都把 consume 提升为 acquire
  • C++17 起标准建议"暂不使用"，C++26 可能重新设计
  • 想用的话直接写 acquire，清晰且性能差别微小

Q4: 为什么读取未加锁的 non-atomic int 是未定义行为，即使只读？

C++ 标准明确：非同步地并发访问同一非原子对象，且至少一方是写，就是 data race = UB。UB 意味着编译器可以假设不会发生，于是可能：

  • 把读的值缓存到寄存器永不刷新（无限循环）
  • 假设读到的值永远在某个范围内，做激进优化
  • 读撕裂（64 位值在 32 位系统上拆成两个 32 位读）

即使你"肉眼"看代码没问题，编译器 + CPU 联合作用下就可能崩。

Q5: fetch_add(1, relaxed) 安全吗？

安全——如果你只需要原子性而不需要跨线程可见性同步。典型例子：shared_ptr 的引用计数 ++ 可以 relaxed（因为拿到 shared_ptr 本身就已同步过），-- 必须 acq_rel（析构前需要看到所有之前线程的写）。

Q6: 为什么 mutex 内部不需要用户写内存屏障？

std::mutex::lock() 内部隐含 acquire 屏障，unlock() 隐含 release。临界区内的读写不能重排出去。所以只要"读写都在同一 mutex 保护下"，就不用关心内存序。

Q7: std::hardware_destructive_interference_size 和 std::hardware_constructive_interference_size 是什么？

  • destructive：避免假共享的最小对齐（通常 64 或 128B）——独立变量要分开
  • constructive：为了共享 cache line 的最大尺寸——相关数据放一起（如 key+value）

在 Intel 新一代（如 Sapphire Rapids）由于邻近 prefetcher 存在，destructive 值可能是 128B（两条 line）。

Q8: 生产环境如何定位内存序 Bug？

  • ThreadSanitizer (TSan)：-fsanitize=thread，检测 data race（最有效）
  • Relacy Race Detector：枚举所有可能的内存序交织，验证无锁算法
  • CppMem / Herd7：形式化工具，按 C++ 标准公理模拟所有合法执行
  • 压测 + 多平台：x86 上跑 OK 的代码必须在 ARM（如 graviton、M1）上压测
  • ThinkSanity：Meta 开源的内存模型 fuzzer

十四、无锁编程 (Lock-Free Programming) ⭐

无锁编程是内存模型最深的应用领域。核心工具只有 CAS (compare_exchange) + 内存序，但组合出的陷阱极多。以下从原语 → 数据结构 → 回收 → 调优层层展开。

14.1 三级进度保证

工程选择：绝大多数场景 mutex 够用；热路径用 lock-free；实时系统（音频、内核、HFT）追求 wait-free。过早追求 wait-free 通常得不偿失——单次操作更慢，总吞吐反而降低。

14.2 CAS — 无锁编程的"汇编指令"

// compare_exchange 的完整语义
bool compare_exchange_strong(T& expected, T desired,
                              memory_order success,
                              memory_order failure);

// 伪代码等价:
if (this->value == expected) {
    this->value = desired;     // 原子完成
    return true;               // 用 success order 发布
} else {
    expected = this->value;    // 回读当前值
    return false;              // 用 failure order
}
变体差异用法
compare_exchange_weak允许伪失败（spurious fail）循环内首选 — ARM/POWER 上更快（LL/SC 无 retry）
compare_exchange_strong不会伪失败单次 check，不在循环里
// 惯用法：CAS 循环
T expected = atomic_var.load(std::memory_order_relaxed);
T desired;
do {
    desired = compute_new(expected);                     // 基于当前值计算
} while (!atomic_var.compare_exchange_weak(
            expected, desired,
            std::memory_order_release,                   // 成功时发布
            std::memory_order_relaxed));                 // 失败只需重读

关键技巧：CAS 失败时，expected 自动被更新为最新值 —— 不要重新 load。

14.3 CAS vs FAA — 选对原语，性能差 10 倍

简单计数器这种"盲加"场景用 fetch_add (FAA)，而不是 CAS 循环：

方案竞争下的行为吞吐
CAS 循环 while(!cas(x, x+1))N 个线程竞争 → 平均 O(N) 重试随线程数负增长
FAA fetch_add(1)硬件保证一次成功线性扩展（直到总线瓶颈）
// ❌ 反面教材
long expected = counter.load();
while (!counter.compare_exchange_weak(expected, expected + 1)) {}

// ✅ 一行搞定
counter.fetch_add(1, std::memory_order_relaxed);

什么时候只能用 CAS：新值依赖旧值的复杂计算（如链表头替换、位图复杂标志组合）。简单增减、位或/位与都应优先用 FAA / fetch_or / fetch_and。

14.4 无锁栈 — Treiber Stack

template<class T>
class LockFreeStack {
    struct Node { T value; Node* next; };
    std::atomic<Node*> head_{nullptr};

public:
    void push(T v) {
        Node* n = new Node{std::move(v), nullptr};
        n->next = head_.load(std::memory_order_relaxed);
        while (!head_.compare_exchange_weak(
                    n->next, n,
                    std::memory_order_release,      // 发布节点
                    std::memory_order_relaxed)) {}
    }

    bool pop(T& out) {
        Node* old = head_.load(std::memory_order_acquire);
        while (old &&
               !head_.compare_exchange_weak(
                    old, old->next,
                    std::memory_order_acquire,
                    std::memory_order_acquire)) {}
        if (!old) return false;
        out = std::move(old->value);
        delete old;    // ⚠️ ABA / use-after-free 炸弹！
        return true;
    }
};
陷阱：old->next 的读取发生在 CAS 之前——如果另一个线程已经 pop 并 delete 了 old，这次 CAS 就是读释放内存。必须配合下面的节点回收方案。

14.6 无锁队列 — MPMC 的 Michael-Scott & Vyukov

Michael-Scott Queue (基于 CAS 的链表)

// 经典 MPMC 无锁队列（1996），std::queue 的并发版雏形
struct Node { T value; std::atomic<Node*> next; };
std::atomic<Node*> head_, tail_;

void enqueue(T v) {
    Node* n = new Node{std::move(v), nullptr};
    for (;;) {
        Node* t = tail_.load(std::memory_order_acquire);
        Node* nxt = t->next.load(std::memory_order_acquire);
        if (t != tail_.load()) continue;         // 重试
        if (nxt) {
            // tail 落后，帮忙推进
            tail_.compare_exchange_weak(t, nxt);
            continue;
        }
        if (t->next.compare_exchange_weak(nxt, n)) {
            tail_.compare_exchange_weak(t, n);    // 完成推进
            return;
        }
    }
}
// 细节：head/tail 应各自在独立 cache line（假共享）
Michael-Scott 是教科书级算法，但实际性能常被下面的数组版超越（链表节点频繁 new/delete + cache miss）。

Vyukov Bounded MPMC Queue (无锁环形队列)

struct Slot { std::atomic<size_t> seq; T value; };
Slot  buf_[CAP];
std::atomic<size_t> enq_pos_{0}, deq_pos_{0};

bool try_enqueue(T v) {
    size_t pos = enq_pos_.load(std::memory_order_relaxed);
    for (;;) {
        Slot& s = buf_[pos & (CAP-1)];
        size_t seq = s.seq.load(std::memory_order_acquire);
        intptr_t diff = (intptr_t)seq - (intptr_t)pos;
        if (diff == 0) {                         // 槽位空闲
            if (enq_pos_.compare_exchange_weak(pos, pos+1,
                    std::memory_order_relaxed)) {
                s.value = std::move(v);
                s.seq.store(pos + 1, std::memory_order_release);
                return true;
            }
        } else if (diff < 0) return false;      // 队列满
        else pos = enq_pos_.load(std::memory_order_relaxed);
    }
}
// 零动态分配、cache 友好，moodycamel::ConcurrentQueue 的核心思路

14.7 SPSC 队列 — 最快的无锁数据结构

单生产者单消费者 (SPSC) 场景下，完全不需要 CAS，只用两个 atomic 指针即可：

template<class T, size_t N>
class SPSCQueue {
    alignas(64) std::atomic<size_t> head_{0};  // 消费者写
    alignas(64) std::atomic<size_t> tail_{0};  // 生产者写 (避免假共享)
    T buf_[N];

public:
    bool try_push(T v) {                   // 只有 producer 调
        size_t t = tail_.load(std::memory_order_relaxed);
        size_t nxt = (t + 1) % N;
        if (nxt == head_.load(std::memory_order_acquire)) return false;
        buf_[t] = std::move(v);
        tail_.store(nxt, std::memory_order_release);
        return true;
    }
    bool try_pop(T& out) {                  // 只有 consumer 调
        size_t h = head_.load(std::memory_order_relaxed);
        if (h == tail_.load(std::memory_order_acquire)) return false;
        out = std::move(buf_[h]);
        head_.store((h + 1) % N, std::memory_order_release);
        return true;
    }
};
// 典型场景: audio callback ← engine, logger 消费者 ← 多 producer (需改造成 MPSC)

性能：SPSC 单次 enqueue/dequeue ≈ 10-20 ns，比 mutex 队列快 10-100 倍。DPDK、LMAX Disruptor、boost::lockfree::spsc_queue 都基于此。

14.8 RCU — 读多写少的终极方案

Read-Copy-Update：读者零同步开销，写者拷贝新版本、替换指针、延迟释放旧版。

// 简化的 RCU 思路 (伪代码 — 生产环境用 liburcu / folly::rcu)
std::atomic<Config*> g_cfg;

// 读者: 纯 atomic load，零锁开销
void reader() {
    rcu_read_lock();
    Config* c = g_cfg.load(std::memory_order_acquire);
    use(c->value);
    rcu_read_unlock();
}

// 写者: 替换指针后"grace period"等所有旧读者退出
void update(int v) {
    Config* newc = new Config{v};
    Config* old  = g_cfg.exchange(newc, std::memory_order_acq_rel);
    synchronize_rcu();    // 阻塞：等所有读者退出当前临界区
    delete old;           // 此时保证无人再用旧对象
}
场景RCU 是否合适
配置热更新（读 99%、写 1%）✅ 完美匹配
路由表 / ARP 缓存 / Linux 内核 dcache✅ Linux 内核重度使用
写密集的场景❌ synchronize 开销拖累写
严格顺序一致性的场景❌ 读者可能看到旧版本

C++20 替代：std::atomic<std::shared_ptr<T>>。实现简单，但每次 load 都有原子 refcount 开销，性能不如真 RCU。

14.11 工程决策树

14.12 面试高频题

Q1: mutex 是无锁的吗？spinlock 是无锁的吗？

都不是。lock-free 的定义是"任一线程在有限步内能让系统整体前进"——而持锁线程挂起后，其他线程无法前进。
但 spinlock 比 mutex 轻量（不涉及内核切换），在极短临界区 + 多核时比 mutex 快。

Q2: 为什么 CAS 循环要用 weak 不用 strong？

ARM/POWER 用 LL/SC 实现原子操作，"LL 到 SC 之间被中断"就 SC 失败——这就是所谓伪失败（spurious fail）。compare_exchange_weak 允许伪失败，硬件不需要内部 retry；strong 需要库/硬件保证无伪失败，在 LL/SC 架构上就是内部循环。既然外层已经有循环，weak 更高效。

Q3: 如何安全地在无锁栈里 delete 节点？

必须解决 use-after-free：其他线程可能还持有指向该节点的指针（CAS 正在进行）。方案：

  1. Hazard Pointer：每线程声明当前访问的指针，free 前扫描
  2. RCU / Epoch：延迟到所有线程退出读临界区
  3. Tagged Pointer：不能防 use-after-free，只防 ABA——仍需上面任一机制回收
  4. 不 delete：用对象池/arena，整体释放

Q4: std::shared_ptr 的引用计数真的线程安全吗？

✅ 控制块的计数是 atomic 的，++/-- 安全。
❌ shared_ptr 对象本身的 =/reset() 不安全——多线程同写同一 shared_ptr 变量是 data race。
若需要对 shared_ptr 变量的并发读写，C++20 提供 std::atomic<std::shared_ptr<T>>。

Q5: fetch_add 在 x86 和 ARM 上如何实现？

  • x86：lock xadd 单条指令，带 MFENCE 语义
  • ARMv8.1-A LSE：ldadd 单条指令（large system extension）
  • ARMv8 基础：ldaxr + stlxr 循环（LL/SC），relaxed 用 ldxr + stxr

推论：ARM 老平台上 fetch_add 本质是 CAS 循环，并不比自己写 CAS 快；新平台(苹果 M1/M2/M3、AWS Graviton 3+) 已有 LSE，性能接近 x86。

Q6: 为什么 LMAX Disruptor 号称比无锁队列更快？

  • 环形 buffer，预分配——无 malloc/free
  • sequence barrier 协议，读写各自 cache line 独立（零假共享）
  • batch 处理：消费者一次性处理多个 slot，分摊 cache miss
  • 对 JVM 还有对象复用避免 GC 压力（C++ 版本收益略小）

本质是机械同情 (Mechanical Sympathy)：理解 CPU 硬件特性并顺从之。

Q7: 如何证明一个无锁算法正确？

  1. 人工推理：对所有可能的操作交织，画出 happens-before 图
  2. Linearizability 证明：每个操作能找到一个"瞬时点"，等价于串行执行
  3. Relacy / CDSChecker：自动枚举所有合法内存序执行
  4. TSan + 多核多平台压测：发现 data race / ABA
  5. Formal proof：TLA+、Coq——仅关键算法值得投入

实践建议：优先用成熟库（folly、concurrentqueue、liburcu），自己实现的无锁代码几乎肯定有 bug。

Q8: 什么时候一定不要用无锁？

  • 临界区超过几百条指令（长临界区下 mutex 更优，reduce retry 消耗）
  • 团队能力不足、缺少压测/验证基础设施
  • 只跑在单核 / GIL 型语言环境
  • 性能瓶颈经 profile 不在同步（提前优化是万恶之源）
