01BRD / MRD / PRD · 大公司 vs 小公司用法

很多团队三类文档混在一起写，结果"高管看不懂、工程不能落"。先弄清三者关系。

维度BRDMRDPRD
层次战略层战术层执行层
问题"为什么做？回报多少？""市场和用户长什么样？""具体做什么？怎么做？"
主要读者CEO / CFO / VPProduct Director / MarketingEngineering / Design / QA
核心要素市场规模 + ROI + 战略契合Persona + 用户旅程 + 竞品 + JTBD需求列表 + 流程 + AC + 边界
篇幅1-3 页5-15 页10-50 页
大公司必有，立项前必过多数有，独立成文必有
小公司常合并到 PRD §1常合并到 PRD §2必有，写得稍重

实战策略

① 大公司 / 重项目：BRD → 立项 → MRD → PRD，三层文档分开、各 own 各的；
② 小公司 / 普通迭代：1 份 PRD 全包，§1 商业 + §2 市场用户 + §3-§5 产品细节；
③ 0→1 项目：BRD 极重要（融资 / 立项），MRD 必须，PRD 反而可以轻；
④ 已有产品迭代：BRD 极简（1-pager），MRD 可省略，重 PRD。

02不同读者关注什么 · PRD 也要"分层服务"

PRD 长一坨，不同人看不同部分。每段开头要让目标读者一眼能找到自己关心的。

读者会跳着看的部分关心问题
VP / 老板§1 背景 + §5 度量"为什么做？指标涨多少？"
设计§2 用户场景 + §4 流程"用户旅程怎样？信息架构？"
工程§3 功能 + §4 流程 + 边界 / 异常"具体逻辑？接口？异常处理？"
QA§3 功能 + AC + 边界"测试用例怎么写？通过标准是什么？"
运营 / 客服§5 度量 + 上线计划 + FAQ"上线节奏？培训物料？应对话术？"
法务 / 风控合规 / 敏感操作 / 数据流"用户数据怎么处理？合规吗？"

03PRD 5 段式 · §1 背景与目标

这是 PRD 最重要的一段 —— 读者 30 秒内能不能"为什么做这个"对齐。

§1 必备字段

1.1 项目名 + 版本号 + 作者 + 上次更新
1.2 一句话价值（Elevator Pitch）：<30 字
1.3 背景（2-3 段）：
    - 用户痛点（含 JTBD Statement）
    - 业务现状 / 数据现状
    - 为什么是现在做（Why now）
1.4 目标（具体 + 可量化）：
    - 业务目标：北极星指标 / OKR 影响
    - 用户目标：用户体验 / 痛点改善
    - 非目标（什么不做，等同重要）
1.5 成功标准（关联 §5 度量）：
    - 上线 N 个月后，X 指标提升 Y%

好的 §1 范例

1.2 一句话价值：让 30+ 万企业 IM 用户在跨部门讨论时，3 秒分清"必须看"和"可以稍后看"的消息。

1.3 背景：
- 现有"消息洪流"导致 65% 用户每天漏看关键消息（Q1 调研 N=240）
- 公司 IM 周活 80 万但 D7 留存 35%，远低于行业 50%（数据来源：DA 仪表盘）
- 接入 LLM 摘要技术成本本季度降到 1/3（Why now）

1.4 目标：
- 业务目标：D7 留存 35% → 45%（带动北极星 NSM +15%）
- 用户目标：关键消息漏看率从 65% → ≤30%
- 非目标：本期不做语音消息摘要

1.5 成功标准：上线 60 天后，D7 留存 ≥ 45%、用户主动反馈"漏看"的客诉减少 50%

糟糕的 §1 范例

1.2 一句话价值：做一个智能 IM 提升用户体验。

1.3 背景：
- 用户反馈消息太多
- 团队希望提升活跃度
- 老板要求做这个项目

1.4 目标：
- 提升用户体验
- 增加活跃度

1.5 成功标准：上线后效果好

04§2 用户与场景

§2 必备字段

2.1 目标用户（Persona）：
    - 主要 Persona（必须 1 个明确）
    - 次要 Persona（可选 0-2 个）
    - 链接到 Persona 文档

2.2 用户场景 / JTBD：
    - JTBD Statement：When ... I want to ... so I can ...
    - 至少 3 个具体使用场景
    - 不在场景内的不在本期考虑

2.3 用户旅程（关键 1-2 段）：
    - 当前 As-is 流程
    - 改造后 To-be 流程
    - 新增 / 修改 / 删除的触点

2.4 用户量级 / 影响范围：
    - 当前用户基数
    - 这个改动覆盖到多少 % 用户
    - 各端覆盖（iOS / Android / Web）

这一段最容易少：用户量级

很多 PRD 写完功能就忘了说"这个改动影响到多少人"。一定要写：DAU 多少、活跃率多少、几个端、灰度比例。
工程团队最常问的就是这个，决定他们的扩容 / 性能预算。

05§3 范围与功能

§3 必备字段

3.1 范围声明：
    - In Scope：本期会做的功能列表
    - Out of Scope：本期明确不做的（v2 再说）
    - Future considerations：未来可能做但本期不承诺

3.2 功能模块（按业务逻辑分组）：
    - 模块 A
        · 功能 A1（必做）：xxx
        · 功能 A2（可选）：xxx
    - 模块 B
        · 功能 B1（必做）...

3.3 优先级（每个功能必标）：
    - P0 必须有（Must）
    - P1 应该有（Should）
    - P2 可以有（Could）
    - P3 不需要（Won't）—— MoSCoW

3.4 边界与异常：
    - 输入边界（最大长度 / 范围 / 字符限制）
    - 异常情况（网络断开 / 数据为空 / 权限不足）
    - 兼容性（旧版本如何）

3.5 依赖：
    - 上游依赖（其他系统 / API / 数据）
    - 下游影响（哪些系统会被影响）
    - 第三方依赖（外部 SDK / 服务）

反例：把"功能"当"需求"

新手 PM 经常写 "做一个 X 按钮"，但没说"用户在什么场景下点这个按钮 / 点了之后期望什么"。这是实现说明，不是需求说明。

心法：每个功能必须能写出 "User Story + AC" 两段，写不出就退回去想 JTBD。

06§4 体验与流程

§4 必备字段

4.1 信息架构 / 页面结构图（IA）
4.2 关键流程图（Flow Chart）：
    - 主流程（Happy Path）
    - 异常分支
    - 用户决策点
4.3 状态机（如果有复杂状态）：
    - 状态 A → 状态 B 的触发条件
    - 状态切换后的副作用
4.4 文案 + 微交互：
    - 关键文案（按钮 / 提示 / 错误信息）
    - 加载状态 / 空状态 / 错误状态
    - 动效 / 反馈
4.5 视觉规范引用：
    - 链接到 Figma / 设计规范
    - 不在 PRD 里画 mockup（让设计 own）

PRD 和 Figma 怎么分工？

① PRD 写"行为逻辑"：什么情况下显示什么 / 跳到哪。
② Figma 画"视觉表现"：尺寸 / 颜色 / 间距 / 字体。
③ PRD 中只放截图占位 + 链接到 Figma 帧，不要把视觉细节写进 PRD（视觉变了 PRD 必失同步）。

07§5 度量与上线

§5 必备字段

5.1 度量指标：
    - 主指标（关联 §1.4 业务目标）
    - 次指标 2-3 个
    - 护栏指标 3-5 个（性能 / 留存 / Crash / 客诉）

5.2 埋点设计：
    - 事件清单（事件名 / 触发时机 / 属性）
    - 数据流（埋点 → 数据仓库 → 报表）
    - 数据 owner（谁负责拉取 / 监控）

5.3 A/B 实验设计（如适用）：
    - 假设
    - 流量分配 + 样本量
    - 跑期 + 决策 timeline

5.4 上线计划：
    - Internal dogfooding（X 周）
    - 灰度 1% → 10% → 50% → 100%
    - 各阶段观察的指标 + 回滚条件
    - 全量后 30 / 60 / 90 天 review

5.5 风险与应对：
    - Top 3 风险（按 P × I 排序）
    - 每个风险的 mitigation
    - 回滚预案

5.6 培训 / 通知：
    - 客服 FAQ + 话术
    - 营销 PR
    - 内部团队培训

08User Story · INVEST 6 原则

标准模板

作为 [角色 / Persona]，
我想 [功能 / 行为]，
以便 [价值 / 期望结果]。

As a [role],
I want to [action],
so that [benefit].

8.1 INVEST 6 个特征

字母含义验证
Independent独立能不能不依赖其他 story 单独发布？
Negotiable可协商是讨论的开始，不是合同
Valuable有价值能给用户/业务带来价值（不是工程内部任务）
Estimable可估算工程能给出工时估算
Small足够小1 个 sprint 内能做完（理想 ≤ 3 天）
Testable可测试QA 能写出明确的通过标准（即 AC）

09验收标准 · Given-When-Then

User Story 单看是模糊的，必须配 Acceptance Criteria (AC)。最经典格式：BDD 的 Given-When-Then。

例：购物车结算页运费可视化

User Story:
作为已加购的购物车用户，
我想在结算前清楚看到运费，
以便不被支付时的"突发费用"吓到。

Acceptance Criteria:

Scenario 1: 选择默认地址，金额 ≥ 99
  Given 用户购物车有 1 件商品
  And   用户已选默认收货地址
  And   订单金额 ≥ 99 元
  When  用户进入结算页
  Then  运费显示为"0.00 元（满 99 包邮）"
  And   运费图标显示绿色 ✓

Scenario 2: 选择默认地址，金额 < 99
  Given 用户购物车有 1 件商品
  And   订单金额 < 99 元
  When  用户进入结算页
  Then  运费显示为具体金额 + "再凑 X 元包邮" 的引导

Scenario 3: 未选地址
  Given 用户首次结算
  When  用户进入结算页
  Then  运费显示"待选择地址"
  And   不显示具体金额，只显示"地址确定后计算"

好 AC 的 3 个特征

① 每个 Scenario 只测 1 个变量（金额 / 地址 / 优惠券，不要混）；
② Given 写状态，When 写动作，Then 写期望结果，三段清晰不混；
③ 边界 + 异常都覆盖（成功路径 / 失败路径 / 极端值）。

10PRD 评审清单 · 12 项必查

在拉评审会议之前，自己先过一遍这 12 项。过不了就别开会，浪费团队时间。

结构 / 完整性（4 项）

  • ☐ 一句话价值能在 30 秒说清
  • ☐ 背景 / 目标 / 非目标 三件齐
  • ☐ 功能列表全部带优先级（MoSCoW 标注）
  • ☐ 度量指标 + 埋点 已落到 §5

用户 / 价值（3 项）

  • ☐ 目标 Persona 明确（不是"所有用户"）
  • ☐ JTBD Statement 已写
  • ☐ 与北极星指标 / OKR 的关联清晰

可执行性（3 项）

  • ☐ 每个功能配 AC（Given-When-Then）
  • ☐ 边界 / 异常 / 兼容性 已覆盖
  • ☐ 上下游依赖列清

风险 / 上线（2 项）

  • ☐ Top 3 风险 + Mitigation 已列
  • ☐ 上线灰度策略 + 回滚预案

116 类典型反例 · 自查别犯

① 把"老板的话"当需求

"老板说要做一个 X" 直接进 PRD，没有用户证据 / JTBD / 数据。
上线后自然没人用。

心法：所有需求必须有"用户层证据"，老板的话只是触发，不是依据。

② 没非目标

只写"要做什么"，没写"明确不做什么"。结果工程会多想很多，QA 会多测很多。

心法：§1.4 必有"非目标"清单，越具体越好。

③ 优先级全是 P0

"都很重要"等于"都不重要"。开发资源不够时全部砍是同等悲剧。

心法：MoSCoW 强制：P0 ≤ 60%，P1 ≈ 30%，P2 ≈ 10%。

④ 没度量指标

上线后无法判断成功。3 个月后老板问"上次那个功能效果怎样"，没数据可答。

心法：没指标的功能不上线，上线没埋点的功能等于没上线。

⑤ 流程图省略异常分支

只画 happy path，"用户取消 / 网络超时 / 支付失败"等异常没设计。
工程根据"猜"实现，bug 一堆。

心法：每个流程节点都问 "如果失败会怎样" 直到答完所有分支。

⑥ 把视觉细节写进 PRD

"按钮颜色 #4f8cff，圆角 4px"，结果设计稿改了之后 PRD 永远过期。

心法：视觉走 Figma，PRD 只写行为逻辑 + 链接 Figma 帧。

126 道高频面试题（PRD / 文档）

Q1讲一份你写过的 PRD，最难的部分是什么？
高分（STAR）

"S/T：一份跨 4 个团队的复杂 PRD（搜索 + 推荐 + 商详）。
A：① §1 一句话价值反复改了 5 遍才让所有人 30 秒 align；② 跨团队依赖图画了 3 版才把"谁做谁的接口"理清；③ 异常分支补全花了 2 天和 QA 一起出。
R：评审一次过，工程零问题进开发；上线后指标达 80% 目标。
反思：最难的不是写功能，是跨团队对齐"为什么 + 怎么算成功"。"

Q2PRD 你是写得详细还是简略？
高分

"分层来：
① 核心场景 + 异常 / 边界 → 必须详细（AC 完整、流程闭环）；
② 视觉 → 简略（指向 Figma）；
③ 显而易见的常识 → 不写（节省评审时间）。
原则：让工程不需要再问 5 个问题就能开工。"

Q3BRD / MRD / PRD 的关系？
高分

"3 层文档对应 3 个问题：BRD 答'为什么投资'（高管），MRD 答'市场用户长什么样'（产品总监 / 市场），PRD 答'怎么做'（工程 / 设计 / QA）。
大公司必须分开写；小公司常合并到 PRD §1-§2 中。
原则：读者不同 → 详略不同，不能把工程关心的细节写进 BRD 里给老板看。"

Q4PRD 写完，怎么开评审会？
高分

"4 步：
① 会前：12 项清单自查 + 异步发邮件让大家先读；
② 开场 5 分钟：BLUF 讲一句话价值 + 度量目标；
③ 讨论 30-40 分钟：先收异议，再讨论解决，不在会议里改文档；
④ 会后：24h 内出修订版 + Decision Log。"

Q5怎么判断一个 User Story 写得好不好？
高分

"INVEST 6 字诀：Independent / Negotiable / Valuable / Estimable / Small / Testable。
具体验证：
① 工程能不能给工时（可估算）；
② QA 能不能写出 AC（可测试）；
③ 用户拿到这个 story 能不能感受到价值（有价值）；
④ 1-3 天能做完（足够小）。"

Q6需求变更怎么办？
高分

"3 步：
① 记录：写到 Change Log，含影响（进度 / 范围 / 风险）；
② 评估：和 EM / Tech Lead 共同评估代价，给老板 3 个选项（接 / 不接 / 部分）；
③ 决策：让 Sponsor / RACI 上的 A 拍板，写入 Decision Log。
不要口头答应。所有变更走 CR，避免范围蔓延。"
