01是什么 / 为什么

每位直接下属一份 持续累积 的文档（最新议题写最上面）。它是你和他/她之间的 "安全沟通通道" — 团队大部分问题（不满 / 想离职 / 能力短板）不会在公开场合暴露，1on1 是唯一窗口。

核心原则：共享文档，双方都能写、都能看；唯一例外是文档底部的 Private Notes 段，只有 manager 看。

1on1 不是"工作分配会"。员工说 70%，你说 30%。

02字段说明

字段为什么存在 / 怎么用
Cadence / 频次新人和重要岗位每周；稳定老员工双周；规模大可月度（但不能取消）
Their Topics下属先写，不让你变"工作分配会"。70% 时间应聊他/她的议题
My Topics你的议题（反馈、期望、信息同步），30% 时间
Discussion notes会议中实时写，不靠记忆
Action Items每次会后双方各 0-2 个 action，下周开会先 review
Career / Growth每月至少 1 次专门聊职业发展，避免被战术议题挤掉
Feedback Loop你 → 他 + 他 → 你 双向反馈，留痕
Private Notes只有你看的私人观察 — 准备 Performance Review 时无价
Insight 新人入职第 1 周开始建文档；如果你 3 个月才想起建，已经丢失了关键的"初印象 + 早期信号"。

📝填好示例

下面是一份真实场景的填写示例（电信 BU、5G RU 团队、Senior Engineer 候选人）。注意关键技巧：议题下属先写、Action 必有 owner + due、Private Notes 记录离职信号。

1on1 · Alice Wang ↔ Kachan

Cadence: 每周 · 周二 14:00-14:45（芬兰时间）
Started: 2026-01-12 · Last updated: 2026-05-19

2026-05-19

Their topics

  • 上周 5G RU patch CR-7821 我做了 owner，想聊聊 review 被打回 3 次的事
  • 想申请参加 6 月的内部 architect track 培训，时间和 sprint planning 冲突
  • 我感觉 Bob 给我的 review feedback 偏 nitpicky，想一起聊聊怎么处理

My topics

  • 你最近在 design review 上的表达明显进步了，想给你一个具体的正向反馈
  • Q3 想让你 owner 一个跨 BU 的故障排查 feature，先听听你的想法
  • （顺便同步）HR 下周会发布新的 promotion 政策，你符合 senior engineer 评估窗口

Discussion notes

CR-7821 被打回 3 次的根因：Alice 习惯一次提交大 PR（约 800 行），超出 reviewer 的认知带宽。约定下次拆成 3 个小 PR。Architect track 和 sprint planning 的冲突有解（让 Bob 替她主持当周 planning）。她对跨 BU feature 有兴趣，但担心 stakeholder management，约定我先帮她 intro 关键 PM。

Action items

#OwnerActionDueStatus
1Alice下次 CR 单个 PR 控制在 ≤300 行，最多拆 3 个 PR下次 CRopen
2manager把 Alice 引荐给 BU-X 的 PM Maria本周open
3manager帮 Alice 申请 architect track 名额，并调整 sprint planning 安排2026-05-26open
4Alice先和 Bob 单独 1on1 聊一次 review 风格，不行再升级到我2026-05-30open

Feedback exchanged

  • （manager → Alice）："上周 design review 你主动 push back PM 砍 scope 的提议，态度专业、有理有据，是 senior 水准的表现。"
  • （Alice → manager）："希望我们 1on1 不要每次都拖到最后 5 分钟才聊 career，那时基本是草草收尾。"

Long-term tracking

Career goals (review monthly)

  • 12-month: 晋升 Senior Engineer (P6)，能独立 owner 跨 BU 的 feature
  • 6-month: 至少 owner 2 个中等规模 feature，主讲 1 次跨 BU 的 design review
  • Currently working on: 系统设计的纵深 / 跨团队沟通能力

Strengths I've observed

  • 技术深度扎实，特别是 RU radio control 模块的细节理解
  • 写作清晰，design doc 同行评价高
  • 主动承担周末 on-call 事件，态度可靠

Growth areas / IDP

  • PR 体积控制（reviewer 认知带宽）— 已经在 action #1 跟进
  • 跨 BU 沟通经验少 — 通过 Q3 跨 BU feature 项目历练
  • 公开演讲偏弱 — 报名 architect track 训练补齐

Private notes (manager-only)

  • 2026-04-20: 她无意中提到"如果再半年没看到晋升进展，会重新考虑职业方向"，需要关注 retention 风险
  • 2026-05-10: Bob 私下吐槽她"沟通有点强势"，需要观察是文化差异还是真问题
注意 Private Notes 是"只有 manager 看"。如果你的 1on1 是共享 Google Doc / Notion，应该把 Private Notes 放在另一个权限文档里，避免不小心暴露。

⚖️DO / DON'T

DO

  • 让下属先写议题，他没写就直接问"今天想聊什么？"
  • 每次结束必出 action + owner + due，下次开头 review 旧 action
  • 每月至少 1 次专门聊 career，否则被战术议题淹没
  • 困难时段（项目延期 / 绩效差）反而 不能取消，那时最需要
  • 双向反馈，主动问 "我哪里可以做得更好？"
  • 共享文档（你 + 他都能编辑），增加 ownership

DON'T

  • 临时取消 — 哪怕你只 show up 10 分钟也比 cancel 好
  • 变成"项目汇报会" — 那应该在站会上完成
  • 把负面反馈攒到绩效面谈才说 — 当场给
  • 把表扬留在 1on1 — 公开场合给
  • 每次都新建文档 — 失去长期 context，1 年后回看 = 0
  • 笔记里写评价 / 标签（"不靠谱"），写具体行为 + 影响
