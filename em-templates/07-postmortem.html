01是什么 / 为什么

Google SRE 的复盘哲学："假设所有相关人员都做了在当时所知信息下的最佳决策"。

复盘的目的是找 "系统让人做错的机制"，而不是找一个倒霉蛋背锅。

有责怪 = 下次出事没人敢说真话 = 风险升级。
3 个必须从模板剔除的措辞
  • ❌ "XXX 应该 / 没有 / 失败了" → 改成 "流程允许 / 系统未阻止"
  • ❌ "如果 XXX 当时小心一点" → 改成 "工具 / checklist 应该把它变成不可能犯的错"
  • ❌ "人为失误" → 永远改成 "系统缺陷"，再追问 "是什么让人犯错"

02字段说明

段说明
Incident summary1 句话：何时 / 影响什么 / 持续多久
Impact客户 / 业务 / 收入 / SLA 量化
Timeline分钟级时间线，标 Detection / Mitigation / Resolution 三个 milestone
Root cause (5 Whys)连问 5 个为什么挖到机制层 — 不是"工程师没注意"
Contributing factors不是单一根因，是多个 contributing factor叠加
What went well不要只看错误 — detection 快不快 / response 顺不顺也要复盘
What went wrong对事不对人
Action items必须有 Owner + Due + Priority — 区分 4 类：Prevention / Detection / Response / Process
Lessons learned未来防范的"机制层"教训

📝填好示例

Postmortem: Carrier-X RU 管理面失联 47 分钟 · 2026-05-15

Incident ID: INC-2026-031 · Severity: P1 · Status: Final
Author: Kachan · Reviewers: Bob, Carol, Mikko · Date: 2026-05-20

1. Summary

2026-05-15 10:23-11:10（共 47 分钟），在一次例行配置推送后，Carrier-X 商用 5G RU 共 12 台同时丢失了 NETCONF 管理会话。数据面正常工作（运营商业务流量没受影响），但管理面无法操作，直到通过本地 console 重建会话才恢复。

2. Impact

  • Customer impact: 影响 Carrier-X 芬兰区 3 个基站的共 12 台 RU
  • Service impact: 管理面瘫痪；数据面不受影响
  • Duration: 9 分钟（从故障到响应） · 总计 47 分钟
  • Business impact: 没有违反 SLA（SLA 只覆盖数据面）；客户已登记 escalation
  • Detected by: Carrier-X NOC 在 10:32 通过紧急电话报告（我们自己的告警漏判）

3. Timeline (UTC+3)

TimeEvent
10:18工程师 A 通过 NETCONF candidate datastore 推送了一次例行 YANG 配置更新
10:2312 台 RU 同时停止发送 KEEPALIVE
10:24内部 "session loss" 告警触发（默认配置为 P3 —— 第一个问题）
10:32Carrier-X NOC 通过紧急电话联系我们
10:35工程师 A 和 on-call EM 加入事故桥
10:47初步假设：candidate-commit 路径上的 YANG 校验出问题
10:58Mitigation：在 1 台机器上通过本地 console 回滚 YANG，确认能恢复
11:05Mitigation 推广到全部 12 台 RU
11:10所有 RU 恢复可管理状态
11:30解除事故状态，关闭事故频道

4. Root cause (5 Whys)

  1. 为什么 12 台 RU 丢了 NETCONF 会话？→ 一次 YANG 模型更新在 commit 时触发了 session handler 的服务端异常。
  2. 为什么 YANG 更新会触发异常？→ 6.0-beta 新增的一个 leaf 与默认值处理逻辑冲突，抛出 NullPointerException。
  3. 为什么 NPE 会传播到 session handler？→ 异常在 commit 代码里被捕获，但在 pre-commit 的 candidate 校验里没有被捕获；commit 代码又把异常重新抛出。
  4. 为什么只有一条路径捕获了异常？→ candidate 校验和 commit 这两条代码路径上，防御性编程模式不一致。
  5. 为什么这个不一致没被发现？→ 代码层面没有任何检查能确保校验与 commit 两条路径的异常处理对称；我们的 CI 也没有测试 "candidate 校验通过但 commit 失败" 这个场景。

5. Contributing factors

  • "超过 5 台机器 session 丢失" 这个告警被错误配置成 P3（应该是 P1）
  • Carrier-X NOC 比我们自己的告警升级流程快了 9 分钟联系到我们
  • 这次 YANG 改动所在的分支测试覆盖率很低（3 个单元测试，没有 e2e）

6. What went well

  • 一旦加入事故桥，12 分钟内定位到根因
  • Mitigation（通过本地 console 回滚）在 12 台机器上执行得非常顺畅
  • Carrier-X NOC 和我们 NOC 的协作很顺（平时关系打底有用）
  • 数据面完全没受影响 — 设计上的物理隔离起作用了

7. What went wrong

  • 告警等级配错成 P3 让事故被忽略了 9 分钟
  • 我们是从客户那里知道事故的，不是从自己的监控
  • 这个 YANG 改动通过了 pre-commit 检查，本不该通过
  • "RU 集群管理面失联" 这种场景没有任何 runbook

8. Action items

3统一 validation / commit 两条路径上的异常处理模式PreventionBob2026-06-12P0In progress
4编写 runbook "RU 集群管理面失联"ResponseCarol2026-05-29P1Done

6YANG 改动 checklist 必须包含 "validation/commit 对称性" 这一项ProcessBob + Maria2026-05-29P2Open

9. Lessons learned

  • 客户比我们自己的监控更快发现故障 — 这是最大的教训。需要一个不依赖告警等级的 "事故经过时长" 兜底机制。
  • "看起来像在处理错误" 的代码，在不同代码路径上的实际处理可能完全不一致。需要靠 Linter / CI 来强制对称性。
  • 数据面和管理面的物理隔离在这次事故里完全起作用 — 这是 6.0 架构 review 时要继续保留的设计原则。

10. Open questions

  • NETCONF 服务端是否应该有 circuit breaker，在连续 N 个 session 断开后自动 kill 掉错误的变更？（由 Architect 在 2026-06-15 前评估）

⚖️DO / DON'T

DO

  • 对事不对人 — 永远用"流程/工具/系统"，不用"某某没做"
  • 必有 Action Items + Owner + Due + Priority，下次 retro 先 review
  • 区分 Prevention / Detection / Response / Process 4 类 action
  • 包含 "What went well" — 鼓励团队复制好做法
  • Public 给全公司 — 失败的传播比成功更值钱
  • 5 Why 挖到机制层，不要停在"工程师没注意"

DON'T

  • "张三推了错配置导致..." — 致命 blameful 措辞
  • "以后大家小心点" — 0 行 Action Item，废纸
  • 把 P0 复盘藏起来不让其他团队看 — 失去最大学习价值
  • 用 "人为失误" 当根因 — 它永远是表象，不是根因
  • Action items 写 "continue monitoring" — 0 动作
  • 复盘开完不跟进 action — 下次同样事故 100% 重演
