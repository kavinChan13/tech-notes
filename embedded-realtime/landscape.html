① 实时系统全景

"实时" 不是"快"，是"响应时延有可预期上界"。一个 1 GHz 通用 Linux 可能比 100 MHz RTOS 平均更快，但P99.99 / 最坏情况延迟完全不一样。

3 个等级

硬实时 Hard 错过 deadline = 系统失效 / 物理危害。
例：飞控、ABS 刹车控制、医疗注射泵。
典型延迟 deadline：µs 级，jitter 必须 < deadline。
准实时 Firm 错过 deadline = 该任务无效，但系统继续。
例：视频帧抖动、音频包延迟。
典型延迟：ms 级。
软实时 Soft 错过 deadline = 体验下降，可接受。
例：UI 渲染、游戏帧率、网络服务 P99。
典型延迟：10-100 ms。

Linux vs RTOS 何时用

通用 Linux 普通服务器 / 桌面。普通调度，worst-case 100+ ms，软实时勉强能用，硬实时不能。
PREEMPT_RT Linux + RT patch。kernel 全可抢占，worst-case ~ 100 µs，准实时 / 部分硬实时。
RTOS FreeRTOS / Zephyr / VxWorks 等。极小内核，worst-case µs 级，硬实时首选。
Bare metal 无 OS 直接跑。最确定性 + 最难开发。MCU 简单产品（开关 / 传感器）。
选型口诀：
  • µs 级硬实时 → RTOS / bare metal
  • ms 级准实时 + 需要 Linux 生态（网络栈、文件系统、SDK）→ PREEMPT_RT
  • 10-100 ms 软实时 + 高吞吐 → 通用 Linux + 应用层 tune（cgroup / nice / SCHED_FIFO）
  • 混合需求（infotainment + safety）→ Hypervisor + 多 OS（如 QNX + Linux）

② PREEMPT_RT —— Linux 准实时化

Linux 早期就 ~30% 设计不可抢占（spin lock、IRQ handler）→ 普通 Linux worst-case 几百 ms。PREEMPT_RT 是一组 kernel patch（Ingo Molnar / Thomas Gleixner 主导），把不可抢占的部分都改成可抢占，达到 worst-case ~ 100 µs。

核心改动

  • Spin lock → Mutex：原本 spin（busy-wait 不可抢占）改成 sleep-able mutex（可让出 CPU 给更高优先级）
  • IRQ handler → kernel thread：硬件中断顶半部超短，主体逻辑在可抢占的线程里跑
  • Priority Inheritance：低优先级线程持锁阻塞高优先级 → 临时提升低优先级线程的优先级，避免 priority inversion
  • High-Resolution Timer：从 jiffies (1ms) 改成 hrtimer (ns 级)

2026 现状

  • PREEMPT_RT 已合入 mainline Linux（Linux 6.12 LTS, 2024-11）—— 不再是 out-of-tree patch
  • 开启方法：kernel config 选 CONFIG_PREEMPT_RT=y
  • 典型 worst-case latency：50-200 µs（依赖硬件 + 配置）

选型决策树

  1. 需要功能安全认证（ISO 26262 ASIL-D / DO-178C）？→ VxWorks / QNX / µC/OS-III safety certified
  2. 开源 + 大社区 + 通用 MCU？→ FreeRTOS（最稳）/ Zephyr（新）
  3. POSIX 兼容性重要（移植 Linux 代码）？→ NuttX / Zephyr
  4. 极小 footprint（< 8 KB）？→ FreeRTOS / RT-Thread Nano / ThreadX
  5. 汽车 IVI？→ QNX 几乎垄断
  6. 中国生态 / 中文文档？→ RT-Thread

④ AUTOSAR CP / AP

AUTOSAR（AUTomotive Open System ARchitecture）是欧洲车企联盟主导的标准，定义了车端 ECU 软件分层架构。两个分支：

Classic Platform (CP)Adaptive Platform (AP)
定位传统 ECU（动力 / 底盘 / 车身）高性能 ECU（ADAS / 域控制器 / Telematics）
OSOSEK / AUTOSAR OS（静态调度）POSIX-based（典型 Linux / QNX）
语言C（少数 C++）C++14+（强 OO + 动态）
调度静态优先级 + 时间触发动态调度 + 多线程
通信CAN / LIN / FlexRay / SOME/IPSOME/IP / DDS / Ethernet
更新flash 全量 / FOTAOTA（app-level 增量）
典型 ECU气囊 / ABS / 雨刮自动驾驶域 / 智能座舱域
资源< 1 MB RAM / 几 MHz> 100 MB RAM / GHz CPU

AUTOSAR AP 关键概念

  • SOA（Service-Oriented Architecture）—— 不再是静态 ECU 间信号，改为service discovery + RPC
  • SOME/IP（Scalable service-Oriented MiddlewarE over IP）—— 主流车端 RPC 协议，跑在以太网上
  • ARA（AUTOSAR Runtime for Adaptive Applications）—— 一组 C++ API（com / log / per / time / ucm 等）
  • Adaptive Application—— 一个可独立部署 / 升级的可执行（类似 microservice）
  • Persistency—— 标准化的持久化 API（替代直接文件 IO）

2026 现状

  • AUTOSAR CP 仍是主流 ECU 标准（80%+ 量产 ECU）
  • AP 在新一代 EE 架构（中央计算 + zonal）下加速普及
  • 开源实现：Apollo Cyber / EB tresos / Vector MICROSAR / 国产开源协会项目

⑤ 车端 C++ guidelines

车端 C++ 不允许"随便写"—— 多个 guideline 强制限制 C++ 子集。

Guideline来源覆盖标准主要限制
MISRA C++ 2008MISRA UKC++03228 条规则；禁动态内存 / 异常 / template 多种
MISRA C++ 2023MISRA UKC++17更新版，允许 modern 子集；100+ 规则
AUTOSAR C++14AUTOSARC++14340 条规则；很多 align MISRA + 加 OO 规则
HIC++Programming ResearchC++17~ 150 条；商业 lint 工具配套
C++ Core GuidelinesISO 委员会 / BjarneC++ 现代~ 700+ 条；推荐性，非强制

典型车端 C++ 限制

  • 禁动态内存—— 运行期 new / delete / malloc 不允许（unpredictable latency + fragmentation）。改用 memory pool / arena 预分配
  • 禁异常—— exception unwind 不可预期 + 部分 toolchain 不全支持。改用 error code / std::expected
  • 禁 RTTI / dynamic_cast—— 运行期类型识别 overhead + 不确定行为
  • 限制 template depth—— 深递归 / 复杂元编程被 cap
  • 禁裸 raw pointer—— 必须用 smart ptr / span / 引用
  • 禁多重继承—— diamond 问题 + vbptr 开销
  • 所有循环必须 bounded—— 编译期可知上界，否则 WCET 分析失败
  • 禁递归—— 栈深度不可预期
  • 禁 goto—— 跳转破坏静态分析

实战 toolchain

  • Lint 工具：cpplint / clang-tidy / commercial: PC-lint Plus / Coverity / Polyspace / LDRA / Helix QAC
  • Coding standard 强制：CI 集成 lint，违规阻止 merge
  • WCET (Worst Case Execution Time) 分析：商业工具如 aiT / RapiTime
  • Static analysis：Coverity / Klocwork / 商业可证明合规

⑥ 功能安全

车端 / 医疗 / 航空都要做功能安全认证。核心是 "通过过程证明软件不会导致危害"。

标准领域等级最严级别
IEC 61508通用工业 / 基础SIL 1-4SIL 4（最严）
ISO 26262汽车（IEC 61508 衍生）ASIL A-DASIL D（最严，如制动）
DO-178C航空（民用机载）DAL A-EDAL A（飞控）
IEC 62304医疗设备软件Class A/B/CClass C（致命危害）
EN 50128铁路SIL 0-4SIL 4
ISO 21434汽车 cybersecurity——

ASIL 分级核心维度（ISO 26262）

  • S（Severity）—— 危害严重度 S0-S3
  • E（Exposure）—— 暴露频率 E0-E4
  • C（Controllability）—— 司机可控性 C0-C3
  • 组合得 ASIL A / B / C / D，QM = 无安全相关

认证过程典型成本

  • V 模型开发：每个 requirement 对应 design + impl + test + traceability
  • 文档密集：每行代码可能对应 5-10 倍文档
  • 独立审核：第三方（TÜV / SGS）审核
  • 典型 ASIL-D 项目成本：开发 + 认证 ~ 2-5 person-year per 1000 行核心安全代码
真实经验：电信领域虽不直接 ASIL，但电信设备 NEBS / 行业 audit 类似——文档 + 可追溯 + 第三方测试是真正占时间的部分，coding 只 20%。

⑦ 车端 / 实时实战

Telemetry（车端数据采集）

车端采集 sensor / CAN / 域控数据，压缩 + 上传云。低延时 + 不可丢。

// 典型车端 telemetry 流水线
sensor / CAN bus
    ↓ (硬实时, RTOS / PREEMPT_RT)
DataCollector (ring buffer, lock-free)
    ↓ (准实时, ms 级)
LocalCache (memory-mapped file, NOR flash)
    ↓ (异步, 后台线程)
Compressor (zstd / lz4)
    ↓
Uploader (HTTP/2 / MQTT / DDS)
    ↓
Cloud

关键设计：

  • Lock-free SPSC ring buffer（见 cpp #91）—— 上下游解耦
  • 预分配 + monotonic_buffer（见 cpp #108/#110）—— 0 alloc 热路径
  • 多级缓冲 + 断网续传 —— 上传失败本地累积 + 重连后批量
  • 低优先级线程做压缩 / 加密 —— 不影响采集 hot path

SOME/IP 实战

主流车端 SOA 协议。基本概念：

  • Service：一组 method / event / field（类似 RPC service + Pub/Sub）
  • Service Discovery：multicast 发布服务，client 订阅
  • Serialization：二进制格式（类似 protobuf 但更紧凑 + 字段对齐 hardware-friendly）
  • 开源实现：vsomeip（GENIVI / COVESA）

DDS（Data Distribution Service）

  • 面向数据中心 / 高可靠 pub-sub 系统（机器人 / 自动驾驶用）
  • ROS 2 底层就是 DDS
  • 开源：FastDDS / CycloneDDS / OpenDDS
  • vs SOME/IP：DDS 更通用 + 更复杂；SOME/IP 更轻量 + 车端标准

⑧ 嵌入式调试

硬件接口

  • JTAG—— 老牌标准，5+ pin，几乎所有 SoC 支持
  • SWD（Serial Wire Debug）—— ARM 简化版 JTAG，2 pin，主流 Cortex-M 用
  • SWO/ITM—— ARM 单线 trace 输出，比 UART 快
  • ETM / PTM—— 全 trace 引擎，可记录每条指令执行

调试器 + 工具

  • Debug probe：J-Link（SEGGER 商业，强大）/ ST-Link / DAPLink（开源 CMSIS）/ Black Magic Probe（开源）
  • OpenOCD—— 开源 debugger backend，pair with GDB
  • SEGGER J-Trace / Lauterbach TRACE32—— 商业 trace 工具，能 reconstruct 完整执行
  • RTOS-aware debug—— GDB / OpenOCD 知道 FreeRTOS / Zephyr 等的内部数据结构，能列 task / 切换 stack

典型 GDB 远程调试

# 在 host 上跑 OpenOCD（连接 J-Link / ST-Link）
openocd -f interface/stlink.cfg -f target/stm32f4x.cfg

# 在另一个 terminal 跑 gdb
arm-none-eabi-gdb firmware.elf
(gdb) target remote :3333
(gdb) load
(gdb) monitor reset halt
(gdb) break main
(gdb) continue

非侵入式 trace

  • printf trace—— 简单但侵入式（impacts timing）
  • SEGGER RTT—— 非阻塞 ring buffer trace，开销 ~ 100 cycles / call
  • ITM / SWO trace—— 单 pin 输出到调试器，硬件支持时几乎 0 cost
  • Kernel ftrace（Linux）—— 内核级 trace，函数粒度

实战 tuning checklist

  1. 开 -O2 / -Os（embedded 经常 size 重要）
  2. 关闭 RTTI / exception（-fno-rtti -fno-exceptions，binary size -10-20%）
  3. 关键代码用 __attribute__((hot)) 让编译器放到一起 (cache locality)
  4. 启用 LTO（-flto），跨 TU 优化 + 减小 binary
  5. 用 memory pool / arena 替代 malloc
  6. 性能敏感 struct 用 alignas(64) 防 false sharing（多核 SoC）
  7. DMA 大批量 / 双 buffer 切换
  8. 用 __builtin_expect 给热分支提示
  9. 关键变量在 tightly coupled memory（如 STM32 CCM RAM）
  10. const data 放 ROM / flash，节省 RAM
