02

CAN / CAN FD

核心特征 · CAN Frame 结构 · CSMA/CR 仲裁 · CAN FD（2012 发布） · …

CAN（Controller Area Network）—— 1986 Bosch 发明，至今仍是车内最广泛协议。每辆量产车至少有 1 个 CAN bus，typically 5-15 个。

核心特征

  • 差分信号（CAN_H / CAN_L 2 线 + GND）—— 高抗噪
  • Bit rate: Classical CAN 1 Mbps; CAN FD 5+ Mbps (data phase)
  • Multi-master broadcast—— 任意 node 可发，所有 node 收
  • ID-based priority—— ID 小者优先（仲裁机制）
  • CRC + ACK—— 硬件层错误检测
  • Bus length × bit rate trade-off—— 1 Mbps 时最长 40m

CAN Frame 结构

┌────┬────────┬────┬─────┬──────────┬──────────┬───┬──────┬─────┐
│ SOF│  ID    │RTR │ DLC │ Data 0-8 │ CRC 15b  │ACK│ EOF  │ IFS │
│ 1b │ 11/29b │ 1b │ 4b  │  bytes   │          │1b │ 7b   │ 3b  │
└────┴────────┴────┴─────┴──────────┴──────────┴───┴──────┴─────┘
  │
  └─ Start of Frame
  • Standard CAN: 11-bit ID, 0-8 byte data
  • Extended CAN: 29-bit ID
  • CAN FD: 64-byte data + 17-bit CRC

CSMA/CR 仲裁

多 node 同时发时，ID 小者赢（dominant 0 比 recessive 1 强）。Loser 自动退让 + retry。

结果：

  • 低 ID = 高优先级—— 安全相关 frame 用 ID 0x000-0x0FF
  • 没有 collision 损失—— bandwidth 利用率高
  • 但高优先级 frame 可饿死低优先级

CAN FD（2012 发布）

特性Classical CANCAN FD
Max bit rate1 Mbps5-8 Mbps (data phase)
Data length0-8 byte0-64 byte
CRC15-bit17/21-bit
向后兼容—需要 CAN FD-capable 控制器

2024-2026 新车型主流 CAN FD，逐步替代 Classical CAN。

Higher-Layer Protocols

CAN 本身只到 Layer 2。应用层有多种：

  • CANopen—— 工业自动化 + 部分车端
  • J1939—— 重型卡车 / 公交
  • UDS over CAN（ISO 14229）—— 诊断
  • OEM proprietary—— 量产车多用这个，每家定义自己的 message catalog

实战 example: Linux SocketCAN

// Linux 提供 socket-based CAN API
#include <sys/socket.h>
#include <linux/can.h>
#include <linux/can/raw.h>

int s = socket(PF_CAN, SOCK_RAW, CAN_RAW);
ifreq ifr = { .ifr_name = "can0" };
ioctl(s, SIOCGIFINDEX, &ifr);
sockaddr_can addr = { .can_family = AF_CAN, .can_ifindex = ifr.ifr_ifindex };
bind(s, (sockaddr*)&addr, sizeof(addr));

// Send
can_frame frame = { .can_id = 0x123, .can_dlc = 8 };
memcpy(frame.data, payload, 8);
write(s, &frame, sizeof(frame));

// Receive
read(s, &frame, sizeof(frame));

LIN

Low-cost 子总线

LIN（Local Interconnect Network）—— 1999 出，用于低成本 + 低带宽场景（如车门 / 座椅 / 雨刮等 body 部件）。

特征

  • 单线（vs CAN 2 线 + GND）—— cost 低 50%
  • Master-slave—— 1 master + ≤ 16 slave
  • Bit rate: 1-20 kbps
  • 调度: master 按预定 schedule 询问 slave
  • 典型 use: 车门把手 / 后视镜 / 天窗 motor / 雨量传感器

跟 CAN 关系

不取代 CAN —— LIN 是 CAN 的 sub-bus。Gateway ECU 用 CAN 跟其他 ECU 通信，再用 LIN 跟 slave 部件通信。

FlexRay

跟 CAN 关键区别 · Time-Triggered Architecture · 2026 现状

FlexRay—— 2005 出，针对x-by-wire（线控转向 / 线控刹车）等极致安全场景。

跟 CAN 关键区别

维度CAN FDFlexRay
Bit rate5 Mbps10 Mbps × 2 channel = 20 Mbps
触发方式Event-triggered (任意时发)Time-triggered (按 schedule)
Determinism仲裁导致 latency 抖动每 frame 时间确定
容错无冗余双 channel 冗余
成本低高（vs CAN 2-3 倍）
典型应用动力 / 普通 ADASx-by-wire / 安全气囊 main bus

Time-Triggered Architecture

FlexRay 把时间分成 communication cycle（typical 5 ms），每 cycle 进一步分：

  • Static segment—— 预分配 time slot，每 ECU 必有自己的 slot（强 determinism）
  • Dynamic segment—— 类似 CAN 的 event-triggered（灵活但 less deterministic）
  • Symbol window—— 用于 wakeup / time sync
  • Network Idle Time (NIT)—— 同步时间

2026 现状

FlexRay 没有大规模普及。主要原因：

  • 成本高 (vs CAN FD)
  • CAN FD + TSN Ethernet 满足大多数 use case
  • 新车型 BMW / Audi 部分仍用，大众 / Mercedes 部分场景
  • 未来逐渐被 Automotive Ethernet + TSN 替代

Automotive Ethernet

跟 office Ethernet 区别 · 速率世代 · 2026 应用场景 · 实战配置 (Linux)

Automotive Ethernet（也叫 100/1000Base-T1）是车内网络的未来。基于 IEEE 802.3 标准但物理层不同。

跟 office Ethernet 区别

维度Office EthernetAutomotive Ethernet
物理介质4 pair UTP / fiberSingle twisted pair（1 pair）
ConnectorRJ-45专门 MATEnet / H-MTD（小 + 抗振动）
最大距离100m+15m (汽车内足够)
Speed10G+100M / 1000M / 2.5G / 5G / 10G
双工Full duplexFull duplex (over 1 pair using echo cancellation)
EMC普通极严格（汽车 -40 to +125°C）

速率世代

标准速率普及年份
IEEE 802.3bw (100Base-T1)100 Mbps2015+
IEEE 802.3bp (1000Base-T1)1 Gbps2018+
IEEE 802.3ch (NGAUTO 2.5/5/10G)2.5/5/10 Gbps2022-2025
IEEE 802.3cy (25G)25 Gbps2025-2027 (规划)

2026 应用场景

  • ADAS / camera 数据 backbone—— 1G/10G 在 ECU 之间
  • Sensor → 域控制器—— camera / lidar / radar 数据高速传到中央计算
  • OTA / 诊断—— 替代 CAN-based OBD-II 的高速接口
  • IVI / Telematics—— 5G modem 跟主机连接
  • 跨域 backbone—— 各域控制器互联

实战配置 (Linux)

# 100Base-T1 NIC 在 Linux 看起来跟普通以太网类似
ip link show eth0
ip addr add 192.168.1.10/24 dev eth0
ethtool eth0           # 看 link speed / duplex

# 但需要 PHY 驱动支持 BroadR-Reach / 100BASE-T1
# 主流 PHY chip: NXP / TI / Microchip / Marvell

TSN

Time-Sensitive Networking

普通 Ethernet 是 best-effort（无 latency 保证），不适合 safety-critical。TSN 是 IEEE 在 Ethernet 上加的real-time + deterministic 扩展。

核心标准

标准功能
802.1ASTime synchronization (gPTP)—— ns 级时钟同步
802.1QbvTime-Aware Shaper—— 预分配 time slot 给高优先级 traffic
802.1Qbu / 802.3brFrame Preemption—— 高优先 frame 可打断低优先
802.1QciPer-Stream Filtering and Policing
802.1CBFrame Replication and Elimination (FRER)—— 冗余路径
802.1QccStream Reservation + 配置 protocol

TSN 解决什么

  • Latency 上界保证—— 关键 control message 在 X µs 内到达
  • Jitter 极小—— 周期数据时间确定
  • 冗余—— 关键 stream 走两条路径同时发，receiver 选第一个到的
  • 跟普通 traffic 共存—— 同一物理网，不同流量 isolation

实战配置

# Linux TSN 配置示例 (用 tc qdisc)
# 创建 TAPRIO (Time-Aware Priority) qdisc
tc qdisc add dev eth0 parent root handle 100 taprio \
    num_tc 4 \
    map 0 1 2 3 \
    queues 1@0 1@1 1@2 1@3 \
    base-time 1000000000 \
    sched-entry S 0x01 500000 \   # 优先级 0: 500 µs
    sched-entry S 0x02 500000 \   # 优先级 1: 500 µs
    sched-entry S 0x0C 500000     # 优先级 2-3: 500 µs

需要 NIC 硬件支持 TSN（Intel i225 / i226 / NXP LS1028A 等）。

SOME-IP / DDS

SOME-IP（已在 AUTOSAR AP 深入 详述） · DDS (Data Distribution Service) · DDS QoS policies · ROS 2 + DDS

底层 Ethernet 之上需要 SOA (Service-Oriented Architecture) 应用层协议。车端两大主流：

SOME-IP（已在 AUTOSAR AP 深入 详述）

  • AUTOSAR AP 主流，Tier 1 / OEM 主推
  • 固定 16-byte header + 紧凑序列化
  • Service Discovery via multicast UDP 30490
  • 开源 implementation: vsomeip (COVESA)

DDS (Data Distribution Service)

OMG (Object Management Group) 标准，更通用，不限车端。

维度SOME-IPDDS
来源AUTOSAR (车端专门)OMG (通用)
编程模型RPC + Pub/SubPure Pub/Sub (数据中心 model)
QoS有限极丰富（30+ QoS policies）
DiscoverySOME-IP-SD multicastDDS Discovery (RTPS)
开销极轻量较重
典型应用车端 control loop机器人 (ROS 2) / 自动驾驶
开源vsomeipFastDDS / CycloneDDS / OpenDDS

DDS QoS policies

DDS 强大的 QoS 让 publisher / subscriber 协商行为：

  • RELIABILITY: RELIABLE vs BEST_EFFORT
  • DURABILITY: VOLATILE / TRANSIENT / PERSISTENT（新订阅者能否收到旧数据）
  • HISTORY: KEEP_LAST / KEEP_ALL
  • DEADLINE: 数据更新最大间隔
  • LIFESPAN: 数据有效期
  • OWNERSHIP: EXCLUSIVE / SHARED
  • LATENCY_BUDGET: 允许 latency 上界
  • PARTITION: 逻辑分组

ROS 2 + DDS

ROS 2 选 DDS 作为底层 middleware。Auto driving 项目（Autoware / Apollo）大量用：

// ROS 2 Publisher (类似 C++ API)
auto node = rclcpp::Node::make_shared("sensor_publisher");
auto pub = node->create_publisher<sensor_msgs::msg::PointCloud2>(
    "/lidar/points",
    rclcpp::QoS(10).reliable().keep_last(10));

// Subscriber 端
auto sub = node->create_subscription<sensor_msgs::msg::PointCloud2>(
    "/lidar/points",
    rclcpp::QoS(10),
    [](const sensor_msgs::msg::PointCloud2::SharedPtr msg) {
      process(msg);
    });

诊断协议

UDS / OBD-II / DoIP

UDS (ISO 14229)

Unified Diagnostic Services—— 车端诊断的事实标准。所有 ECU 必须支持 UDS。

UDS 关键 service

SIDService用途
0x10DiagnosticSessionControl切换诊断 session (default / extended / programming)
0x11ECUReset重启 ECU
0x14ClearDiagnosticInformation清 DTC (Diagnostic Trouble Code)
0x19ReadDTCInformation读 DTC
0x22ReadDataByIdentifier读 ECU 数据（VIN / serial / config）
0x2EWriteDataByIdentifier写 ECU 数据
0x27SecurityAccessseed-key challenge 解锁
0x31RoutineControl调试 routine（self-test / calibration）
0x34/0x36/0x37RequestDownload/TransferData/RequestTransferExitflash update

OBD-II

  • 法规强制—— 美国 1996+ / 欧洲 EOBD 2001+ 所有车必有 OBD-II 接口
  • 主要用于排放相关诊断
  • UDS 的子集 + 标准化 PID
  • 16-pin 接口在驾驶座下方

DoIP (Diagnostic over IP, ISO 13400)

UDS 跑在 Ethernet 上，主流 OTA / dealership tool 用。

  • TCP 13400 (vehicle ↔ tester)
  • UDP 13400 (announcement / discovery)
  • 替代慢的 K-Line / CAN 诊断
  • 支持多 ECU 并发刷写—— flash 整车从几小时降到几分钟

典型 dev workflow

  1. OEM / Tier 1 提供 DBC 文件（CAN message catalog）
  2. 用 cantools 解析 DBC，生成 C / C++ struct
  3. 用 SocketCAN 在 Linux dev box 模拟 CAN bus
  4. candump / candump -L log traffic 到文件
  5. canplayer replay 文件 → 测试 ECU 行为
  6. Wireshark 解析 + 可视化
  7. CANoe（在 OEM）跑 HIL test + 完整 simulation

6 现状 + 中央计算演化

EE 架构演化 · 主机厂状态 · 未来 5-10 年

EE 架构演化

世代架构网络
Gen 1 (分布式)~100 个独立 ECU多 CAN bus + LIN sub-bus + 个别 FlexRay
Gen 2 (域集中)5-8 个域控制器 (powertrain / chassis / body / infotainment / ADAS)CAN FD 域内 + Ethernet 跨域 backbone
Gen 3 (中央计算 + Zonal)1-2 个中央计算 + 4-6 个 zonal ECU1G/10G Automotive Ethernet + TSN backbone + CAN FD / LIN 末端
Gen 4 (软件定义)1 个超级中央计算 + zonal proxy10G+ Ethernet + 完全 SOA + V2X

主机厂状态

  • Tesla—— 已 Gen 3，自研 ETH backbone
  • 新势力 (蔚来 / 小鹏 / 理想)—— Gen 3 主流
  • BMW / Mercedes 新平台—— Gen 3 (2024+)
  • 传统 OEM (Toyota / GM)—— Gen 2 → Gen 3 过渡中 (2025-2027)

未来 5-10 年

  • CAN / LIN 仍存在—— 末端 sensor / actuator 经济性最佳
  • FlexRay 逐渐淘汰—— 被 TSN Ethernet 替代
  • Automotive Ethernet 主流—— 域间 / 跨域
  • 10G+ in 量产—— 中央计算 ↔ sensor 数据回传
  • V2X (C-V2X)—— 车 ↔ 车 / 车 ↔ infra / 车 ↔ pedestrian
  • 5G/6G modem on-board—— OTA / cloud + edge collaboration
🎯 关键 take-aways
  • 车端有多种网络协议—— CAN / LIN / FlexRay / Ethernet 各 fit 不同子系统
  • CAN / CAN FD 仍是车内 most widely used；CSMA/CR 仲裁 + ID 决定优先级
  • LIN 是 CAN 的低成本 sub-bus，用于 body 电器
  • FlexRay time-triggered + 双 channel 冗余，但被 Ethernet+TSN 替代
  • Automotive Ethernet（100/1000Base-T1 / 10G）是车内网络的未来
  • TSN（gPTP / TAPRIO / FRER）让 Ethernet 也有 real-time + deterministic
  • SOME-IP (AUTOSAR AP) vs DDS (机器人 / 自动驾驶) — 不同 SOA 应用层选择
  • UDS / OBD-II / DoIP 是诊断协议三件套
  • 开发工具: Vector CANoe (商业金标准) + SocketCAN / can-utils (开源)
  • EE 架构演化: 分布式 → 域集中 → 中央计算 + Zonal — 2025+ 主流 Gen 3
