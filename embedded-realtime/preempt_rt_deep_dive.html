1. 为什么 mainline Linux 不能硬实时

普通 Linux 内核默认 preempt 在多处被 disable—— 高优先级任务必须等这些 section 完成才能跑。这个 "等待时间的上界" 是worst-case latency，普通 Linux 可达 100+ ms。

3 个 unpreemptible 段

  1. Spin locks—— 持锁期间禁 preempt（避免死锁）。Critical section 长度 = preempt 延迟下界。
  2. IRQ handlers—— 中断处理 top-half 不可被打断。设备驱动 IRQ 可能跑几 ms（如复杂网卡）。
  3. Softirq / tasklet—— 中断 bottom-half，跑在当前 CPU 的 softirq daemon，可累积大量 work。

典型 worst-case latency 来源

来源典型 worst-caseRT 解决方案
Spin lock 持有µs - ms转 sleeping spinlock
IRQ handlerµs - ms转 IRQ thread
Softirq10 - 100 ms转 ksoftirqd thread + prioritize
Memory allocationµs - 数 ms预 lock + mlockall
Page fault (swap)10 ms - 秒mlockall, no swap
CPU frequency 变µs关 cpu freq scaling
C-state transitionµs - 10 ms禁深度 cstate
System Management Interrupt (SMI)µs - msBIOS 关 / 隔离
Timer tickµsnohz_full 隔离

普通 Linux 跑 cyclictest 1 小时典型分布：

0-10 µs ████████████████████████████████ 88%
10-100 µs ████████ 10%
0.1-1 ms ██ 1.5%
1-10 ms ▌ 0.4%
10-100 ms ▎ 0.1%
> 100 ms ▏ 0.001%（rare 但存在）

软实时 (10-100 ms) 偶尔 OK；µs 级硬实时 永远不达标。

2. RT patch 的 5 大核心改动

PREEMPT_RT patch（Ingo Molnar / Thomas Gleixner 主导，2004 起，2024 mainlined in 6.12）做了 5 件事：

2.1 Sleeping Spinlocks

原本 spin_lock_t 是 busy-wait + 禁 preempt。RT 把大多数 spin lock 转 rt_mutex—— 持锁 thread 可被 preempt，等锁 thread 进 sleep wait queue（不 busy-wait）。

// Mainline kernel: spin lock
spin_lock(&lock);       // disable preempt
work();                // 无法被高优先级抢占
spin_unlock(&lock);     // enable preempt

// RT kernel: 相同 API, 不同实现
spin_lock(&lock);       // rt_mutex_lock; sleeping, preemptible
work();                // 可被高优先级 thread 抢占
spin_unlock(&lock);     // rt_mutex_unlock

API 兼容（driver code 不用改），但语义是 sleeping mutex。

例外：少数真 spin lock 保留（raw_spinlock_t），用在 scheduler / IRQ 进入等无 sleep 的 critical path。Driver 不应该用 raw_spinlock_t。

2.2 Threaded Interrupt Handlers

RT 把大多数 IRQ handler 改为kernel thread。

阶段普通 LinuxRT
Top-halfIRQ context, 不可抢占极短 wakeup + return（µs 级）
Bottom-halfSoftirq, 不可抢占kernel thread, 可调度 / 可抢占

结果：硬件中断 latency 跟调度延迟解耦。可以给关键 IRQ thread 设 SCHED_FIFO 高优先级。

2.3 Priority Inheritance

经典问题 priority inversion：低优先级 thread A 持锁 → 高优先级 thread B 等锁 → 中优先级 thread C 抢 CPU → A 不跑 → B 永远等。著名事故：Mars Pathfinder 1997 系统重启。

RT 的 rt_mutex 实现 Priority Inheritance Protocol (PIP)：

  1. B 等 A 持的锁 → A 的 effective priority 临时提升到 B 的优先级
  2. A 完成 critical section 释放锁 → A 优先级降回
  3. B 拿到锁继续

这样 C（中优先级）不能抢 A → B 等待有上界。

2.4 High-Resolution Timers

普通 Linux 默认 timer 精度 jiffies（1-10 ms）。RT 用 hrtimer（ns 级精度），调度 / sleep / timeout 都精确。

// 应用层用法 (Linux nanosleep)
struct timespec ts = { .tv_sec = 0, .tv_nsec = 50000 };  // 50 µs
clock_nanosleep(CLOCK_MONOTONIC, 0, &ts, NULL);
// 普通 kernel: 醒来时间 = 50µs 到 10ms
// RT kernel: 醒来时间 = 50 µs ± 5 µs

2.5 RCU 改造

RCU (Read-Copy-Update) 是 Linux 高性能并发原语，但 grace period 处理在 softirq → 长 grace period 影响 RT。

RT 加 rcu_nocbs 把 RCU callback 从 softirq 移到 dedicated kernel thread → 可调度优先级，不阻 RT thread。

4. 系统级 + 应用级调优

RT kernel 只是基础。要达到 µs 级 worst-case latency，还需系统配置 + 应用配合。

4.1 内核启动参数（GRUB）

# /etc/default/grub: GRUB_CMDLINE_LINUX

isolcpus=2-7              # 隔离 6 个核，不参与普通调度
nohz_full=2-7             # 隔离的核停 timer tick (无 1ms 唤醒)
rcu_nocbs=2-7             # RCU callback 不在这些核
irqaffinity=0,1           # IRQ 处理在 0/1 核, 不打扰 RT 核
threadirqs                # 所有 IRQ 走 thread (RT 默认这样)

# CPU 频率 / cstate 锁定
processor.max_cstate=1    # 禁深度 cstate, 减唤醒延迟
intel_idle.max_cstate=0
intel_pstate=disable      # 关 P-state，固定频率

# 关 hyper-threading（共享 cache 引 jitter）
# BIOS 关 / 或:
isolcpus=2-7 nosmt

# 关 watchdog (周期 NMI 检查)
nosoftlockup
nmi_watchdog=0

# 关 transparent hugepage (page fault 大)
transparent_hugepage=never

# 极端: spin 不 enter idle
idle=poll                 # 警告: 占核 100% CPU

4.2 运行时 sysctl

# /etc/sysctl.d/99-rt.conf

# Scheduler: 允许 SCHED_FIFO 用满 CPU (不被 throttle)
kernel.sched_rt_runtime_us = -1     # default 950000 (95%)

# Disable swap (page fault disastrous)
vm.swappiness = 0

# Tune memory dirty thresholds
vm.dirty_ratio = 5
vm.dirty_background_ratio = 2

4.3 IRQ Affinity 手动绑定

# 把网卡 IRQ 绑到核 0/1, 不打扰 RT 核 2-7
$ cat /proc/interrupts | grep eth0
 60:  ...  ...  PCI-MSI  eth0-rx-0
 61:  ...  ...  PCI-MSI  eth0-rx-1
 ...

$ echo 1 > /proc/irq/60/smp_affinity   # bitmask 0x01 = core 0
$ echo 2 > /proc/irq/61/smp_affinity   # bitmask 0x02 = core 1

# 或用脚本批量:
$ for irq in $(grep eth0 /proc/interrupts | awk '{print $1}' | tr -d ':'); do
    echo 3 > /proc/irq/$irq/smp_affinity   # cores 0+1
  done

4.4 应用层准备（C++ 程序）

// 启动时一次性 setup
#include <sys/mman.h>
#include <sched.h>
#include <pthread.h>

void setup_rt() {
  // 1. 锁定所有 memory，禁 page fault
  if (mlockall(MCL_CURRENT | MCL_FUTURE) != 0) {
    perror("mlockall");
  }

  // 2. 预 alloc 大 stack (避免栈扩展时 page fault)
  const size_t STACK_SIZE = 8 * 1024 * 1024;
  char buf[STACK_SIZE];
  memset(buf, 0, STACK_SIZE);  // 强制 page commit

  // 3. 设 SCHED_FIFO 高优先级
  struct sched_param sp;
  sp.sched_priority = 80;
  if (sched_setscheduler(0, SCHED_FIFO, &sp) != 0) {
    perror("sched_setscheduler");
  }

  // 4. 绑 CPU
  cpu_set_t mask;
  CPU_ZERO(&mask);
  CPU_SET(2, &mask);  // 跑 core 2
  pthread_setaffinity_np(pthread_self(), sizeof(mask), &mask);
}

// hot loop (永远 sleeping/spinning here)
void rt_loop() {
  struct timespec next;
  clock_gettime(CLOCK_MONOTONIC, &next);
  while (running) {
    do_work();
    next.tv_nsec += 1000000;   // 1 ms cycle
    if (next.tv_nsec >= 1000000000) {
      next.tv_nsec -= 1000000000;
      next.tv_sec += 1;
    }
    clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME, &next, NULL);
  }
}

5.2 rtla（Real-Time Linux Analysis）

Linux 6.x 引入的高级 RT 分析工具：

# Hwlat: 检测硬件级 latency (BIOS / SMI / NMI / firmware)
sudo rtla hwlat top

# Timerlat: timer wakeup latency
sudo rtla timerlat top -P f:80 -c 2

# Osnoise: OS noise (跟踪谁打断 RT thread)
sudo rtla osnoise top -c 2

5.3 ftrace（定位 spike 来源）

cyclictest 看到 max latency 异常时，用 ftrace 找谁造成的。

# 1. 启用 wakeup_rt tracer
echo wakeup_rt > /sys/kernel/debug/tracing/current_tracer
echo 0 > /sys/kernel/debug/tracing/tracing_max_latency

# 2. 跑 workload
sudo cyclictest -i 1000 -p 80 -t 1 -m -D 60

# 3. 看 max latency trace
cat /sys/kernel/debug/tracing/trace
# 输出会显示 latency-causing event:
# 例: kworker/2:0 woke up cyclictest after 78 µs delay

6. 10 个调优陷阱

#陷阱症状修复
1没 mlockall偶发 ms 级 spike（page fault）mlockall(MCL_CURRENT | MCL_FUTURE)
2没禁 cstate10-100 µs spike on idle wakeupkernel arg processor.max_cstate=1
3开 hyper-threadingcache 共享 jitter 5-20 µsBIOS 关 SMT 或 nosmt 启动
4SMI 没隔离µs - ms 级不可预测 spikeBIOS 关或 isolate core，rtla hwlat 验证
5用 std::cout / printfIO 阻塞 ms 级预 format + 异步 log (spdlog async / quill)
6共享 mutex 跟低优先级Priority inversion / unbounded wait用 PI mutex (默认 RT 的 rt_mutex)
7没绑 IRQIRQ 抢 RT 核irqaffinity + /proc/irq 绑核
8RT thread CPU 100% (poll loop)thermal / power 触发降频用 clock_nanosleep / 监控温度
9cgroup CPU throttleRT thread 被 cgroup limitRT cgroup quota 设无限
10NUMA cross-socket access+ 50-100 ns memory latencynumactl --membind + cpu pinning 同节点
SMI 是最难 debug 的。System Management Interrupt 是 BIOS 在用户不可见的高优先级 mode 跑代码（电源 / 温度 / vendor backdoor 管理）。cyclictest 看到 100+ µs spike 但 ftrace 找不到 = SMI。

对策：① BIOS 设置关闭 SMI（risky，部分平台 SMI 是必要）；② 隔离 RT core 让 SMI 不 affect（部分 CPU 支持）；③ 选 server-grade 主板（如 Supermicro / Intel reference），不要 desktop 主板。

7. PREEMPT_RT vs Xenomai vs RTOS

方案架构Worst-case latencyAPI适合
普通 LinuxSingle kernel, preempt limited10-100 msPOSIX软实时 / 通用
PREEMPT_RTSingle kernel, fully preemptible50-200 µsPOSIX准实时 / 部分硬实时
Xenomai (Cobalt)Dual kernel: Linux + RT co-kernel10-50 µsXenomai-specific or POSIX skin硬实时 + Linux 用户态
RTOS (FreeRTOS / VxWorks)单一 RT kernelµs 级Vendor-specific硬实时 + 资源受限
Bare metal无 OSns 级裸 C/C++极致硬实时 + MCU

PREEMPT_RT 选择 fit

  • ✅ 需要 Linux 生态（网络 / 文件系统 / 容器 / SDK）
  • ✅ Worst-case 100-200 µs 接受
  • ✅ 跟现有 Linux app 共用 codebase
  • ❌ 需 sub-10 µs worst-case → Xenomai / RTOS
  • ❌ MCU / 资源受限 → RTOS / bare metal

8. 真实生产案例 · Telemetry 硬延时

platform agent 部分客户用我们处理车端 telemetry—— ECU 采集 sensor / CAN 数据 + 实时上传 OEM cloud。每条数据带 timestamp，硬性要求 ≤ 1 ms 端到端 latency。

初始状态

  • 普通 Ubuntu 20.04 + standard kernel
  • cyclictest 1h: Min 3 / Avg 8 / Max 12000 µs (12 ms worst-case)
  • P99.99 latency 约 8 ms—— hard miss

调优过程

Phase 1: RT kernel + 基础 tune

  • 切 PREEMPT_RT kernel
  • kernel args: isolcpus=2-3 + nohz_full=2-3 + processor.max_cstate=1
  • cyclictest: Max 250 µs (50× improvement)
  • App P99.99 ~ 200 µs—— meeting 1 ms goal but no margin

Phase 2: app 改造

  • mlockall + 预 alloc
  • SCHED_FIFO priority 80
  • CPU 绑核 + NUMA local
  • 异步日志（quill）替代 fprintf
  • cyclictest: Max 120 µs
  • App P99.99 ~ 80 µs—— comfortable margin

Phase 3: BIOS + 硬件

  • 关 SMI (BIOS settings)
  • 关 hyper-threading
  • 固定 CPU frequency 3.0 GHz (intel_pstate=disable + cpufreq performance)
  • 用 huge page (2MB) 减 TLB pressure
  • cyclictest: Max 65 µs
  • App P99.99 ~ 45 µs (180× improvement vs baseline)

Phase 4: IRQ 优化

  • 网卡 IRQ 绑核 0/1 + RPS / RFS 优化
  • 关闭不必要 driver（USB / audio / WiFi）
  • cyclictest: Max 48 µs（差不多到 RT 极限）
  • App P99.99 ~ 35 µs

最终结果

阶段cyclictest Max (1h)App P99.99
Baseline (普通 Linux)12 ms8 ms
+ PREEMPT_RT250 µs (-48×)200 µs (-40×)
+ app tune120 µs80 µs
+ BIOS / HW65 µs45 µs
+ IRQ48 µs35 µs (-228×)

1 ms hard latency requirement 有~ 28× 安全 margin。客户满意 + 通过 OEM 验证。

Lesson learned

  • RT kernel 是必要但不充分——还要 app / BIOS / HW 全栈调
  • SMI 是最隐藏的 latency source——server-grade 主板 + BIOS 关掉
  • app 改造 占总收益约 30%（mlockall / SCHED_FIFO 等）
  • cyclictest 跑 24h+ 比 1h 更能 catch rare spike
  • 持续监控——production deploy 时 metrics 包含 latency histogram，发现退化立刻报警

9. 2026 现状

Mainline 化

PREEMPT_RT 是 20 年的 out-of-tree patch。2024 年 Linux 6.12 LTS 终于把核心 RT 改动 merge 进 mainline。这是 Linux 史上最长 patch series（数十次 review iteration）。

意义：

  • 不再需要单独 patch / 维护 kernel fork
  • 所有 Linux 发行版默认 可启用 RT（CONFIG_PREEMPT_RT=y）
  • 第三方 kernel module / driver 跟 RT 兼容性问题大幅减少
  • Long-term: RT 跟 mainline 同步演进

2026 数据

领域采用
工业自动化~ 60% 新项目用 PREEMPT_RT（vs RTOS 20% / Xenomai 5% / 普通 Linux 15%）
车端（ADAS / cockpit）~ 40% （QNX 30% / AUTOSAR 25% / 其他 5%）
机器人 (ROS 2)主流，ROS 2 推荐 RT kernel
音频 / 实时媒体~ 30%（专业音频 setup 仍 Pro Tools 等专门 RTOS）
金融 HFT多用普通 kernel + 用户态优化（DPDK / kernel bypass），RT kernel 偶用

未来 1-2 年

  • Kernel 6.x 持续 refine RT—— 更好 RCU、io_uring + RT 集成、KASAN-HW 兼容 RT
  • Container（cgroup v3 / k8s）+ RT 更顺畅—— 现在仍需手动 tune
  • ARM64 RT 改进——Apple Silicon / 新 ARM server 平台
  • RT + io_uring 组合让用户态 high-throughput + low-latency 更可行
