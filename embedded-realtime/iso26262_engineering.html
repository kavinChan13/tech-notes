01

为什么车企必须 ISO 26262

背景：从 IEC 61508 而来 · 10 大 Part 概览 · 不合规的代价

ISO 26262（"Road vehicles — Functional safety"）2011 首版，2018 二版。2026 强制 大多数主流市场（欧盟 / 北美 / 中国 / 日本）的新车型。

背景：从 IEC 61508 而来

  • IEC 61508（1998）—— 通用工业功能安全标准
  • 汽车需要更具体的 process + 标准 → ISO 26262 是 IEC 61508 在汽车的 specialization
  • 类似 specialization：医疗 IEC 62304 / 航空 DO-178C / 铁路 EN 50128

10 大 Part 概览

Part主题
Part 1词汇
Part 2功能安全管理（流程 / 文化 / Person Roles）
Part 3概念阶段（Hazard analysis + Functional Safety Concept）
Part 4系统级开发
Part 5硬件级开发
Part 6软件级开发（C++ 工程师最关心）
Part 7生产 + 运营
Part 8支持过程（工具 / 配置 / 变更）
Part 9ASIL + Safety-Oriented Analysis
Part 10指引（无规范要求）
Part 11（2018+）半导体应用指引
Part 12（2018+）摩托车 adaptations

不合规的代价

  • 不能上 OEM 供应商列表（Tier 1 / Tier 2 等）
  • 事故后 product liability 风险（民事 + 刑事）
  • EU、中国 type approval 不通过 → 不能销售
  • 保险 / 公关 / 召回成本

所以车端 C++ 项目无 ISO 26262 = 无市场。

V-Model 全生命周期

概念 → 设计 → 验证

ISO 26262 用 V-Model：左边 develop, 右边 verify。每个开发 artifact 都有 corresponding verification artifact。

开发（Specification） ↔↔↔ 验证（Verification） [3-7] Item Definition / Hazard Analysis [4-9] System Validation ↓ ↑ [3-8] Functional Safety Concept [4-8] System Integration & Test ↓ ↑ [4-6] Technical Safety Concept [4-7] HW-SW Integration & Test ↓ ↑ System Architecture ←→ HW & SW V-models 平行 ↓ ↑ [5] Hardware [6] Software ↓ ↓ ↑ HW design Software Architecture ↓ ↓ HW Detailed Design SW Detailed Design ↓ ↓ Implementation Implementation (silicon / PCB) (C / C++ code) ↓ Unit Test → Component Test → SW Integration

关键 work products（部分）

阶段Work Product
概念Item Definition / Hazard Analysis & Risk Assessment (HARA) / Functional Safety Concept (FSC)
系统Technical Safety Concept (TSC) / Safety Requirements / System Architecture

Safety Case

最终交付的是 Safety Case—— 一份 hundreds of pages 文档，论证 "我们做了什么 + 为什么足够 safe"。包含：

  • 所有 hazard 已识别 + assessed
  • 对应 ASIL 级别 + 推导依据
  • Safety mechanism 设计 + 实现
  • 验证证据（test report / coverage / review）
  • 残余风险论证（acceptably low）
  • 第三方独立审核（Assessor）报告
04

ASIL Decomposition

高级 ASIL 拆成 redundant 低级

实际工程中ASIL D 极难达到（特别软件层）。ISO 26262 允许 ASIL Decomposition—— 把高 ASIL 拆成多个独立通道，每个低 ASIL，redundant 实现。

合法 decomposition 示例

原 ASIL拆成条件
DD(C) + D(A) 或 C(C) + C(A)两 channel 独立
DB(D) + B(D)2 个 ASIL-B 完全 redundant
CB(C) + A(C) 或 C(B) + QM(B)类似
BA(B) + A(B) 或 B(A) + QM(A)类似

D(C) + D(A) 的含义：channel 1 实现到 ASIL D 要求（标 D(C) 表示这是 decomposition from C），channel 2 也实现到 ASIL D（A 衍生）。两 channel 独立运行 + 对比 → 整体满足 ASIL D。

"独立" 的严格要求

  • 不能共享 CPU core / 内存 region / power supply
  • 不同 supplier / 不同 toolchain 更佳
  • Common-cause failure 分析必做（CCF / DFA）
  • 独立性必须由 assessor 验证

典型应用：制动 ASIL D 系统

  • 主 channel：标 D(B) ASIL B 控制器（速度 + 力度）
  • 副 channel：标 D(B) 独立 ASIL B 监督器（合理性检查）
  • 两 channel 在不同 MCU + 不同 PCB partition
  • 不一致 → 进 safe state（释放 brake / 报警）

软件层活动

关键软件活动 · Method Requirement Tables · 典型 SW Architectural Constraints

C++ 工程师最关心的是 ISO 26262 Part 6。它规定每个 ASIL 级别软件活动的method requirement table。

关键软件活动

阶段活动
需求SW Safety Requirements traceability
架构SW Architecture design + Architectural patterns
详细设计Detailed design + Unit specification
实现Source code（按 coding guideline）
验证Unit test / Integration test / Code review
分析Static analysis / Control flow / Data flow

Method Requirement Tables

Part 6 表 6.x 列出每方法在每 ASIL 的推荐度：

  • ++（强烈推荐 / highly recommended）
  • +（推荐 / recommended）
  • 0（无强制 / no recommendation）

例：架构设计 patterns（Table 6.3）

方法ASIL AASIL BASIL CASIL D
Hierarchical structure++++++
Restricted size of components+++++
Strong typing+++++++
Defensive implementation0+++++
Well-trusted design principles++++

Recommended (+) 一般要 satisfy；highly recommended (++) 必须 satisfy 或者 documented rationale 为啥不。

典型 SW Architectural Constraints

  • 禁用 dynamic memory allocation（runtime new / malloc）
  • 禁用 exception（unwind 不可预测）
  • 禁用 RTTI / dynamic_cast（运行期 overhead + 不确定）
  • 禁用 unbounded loop（WCET 分析必须 bounded）
  • 禁用 recursion（栈深不可预期）
  • 限制 template depth + complexity
  • 禁用 multiple inheritance（diamond 风险）
  • 函数小 + 单一职责（推荐 ≤ 30 行 / cyclomatic complexity ≤ 10）

这就是为什么 AUTOSAR C++14 / MISRA C++ 这么严—— 满足 ISO 26262 Table 6.3 / 6.7 等表的具体落地规则。

规则等级

等级含义
Mandatory必须遵守，不允许 deviation
Required必须遵守，可以 documented deviation
Advisory建议遵守，违反需文档说明

典型 MISRA C++ 2023 规则

  • Rule 8.0.1（required）—— union 不允许（type aliasing UB）
  • Rule 8.2.1（required）—— virtual 函数应用 override 或 final
  • Rule 18.1.1（required）—— 指针不能算术 (除 array 索引)
  • Rule 8.10.1（required）—— enum class 优先
  • Rule 6.0.1（required）—— static_cast 不能跨指针类型
  • Rule 16.0.2（required）—— 不用 macro (除 conditional compile)
  • Rule 21.6.1（required）—— 不用 dynamic memory (malloc / new)

Deviation 流程

违反 required rule 时必须走正式 deviation：

  1. 记录 deviation 理由
  2. 分析风险 + 替代验证手段
  3. 项目 safety manager review + sign-off
  4. 第三方 assessor 接受
  5. 所有 deviation 列在 safety case
实际数字: 典型 ASIL-D 项目（~500 万行 C++）有 2000-5000 个 deviation。每个 deviation 流程 typical 1-3 人天 + 多人 review。

工具链认证

Tool Confidence Level (TCL)

"开发用的工具能否信任 it 不引入 bug"？ISO 26262-8 Clause 11 定义 Tool Confidence Level (TCL)。

TCL 推导

因素TI (Tool Impact)TD (Tool Detection)
TI1工具不影响 safety—
TI2工具影响 safety—
TD1—工具 bug 高概率被 detect
TD2—中等概率
TD3—低概率
TD1TD2TD3
TI1TCL1TCL1TCL1
TI2TCL1TCL2TCL3
  • TCL1—— 无 qualification 要求
  • TCL2—— Qualification needed（typically 选有"qualified" vendor 提供的 cert）
  • TCL3—— 严格 qualification + 完整 vendor-provided artifact

典型工具 TCL

工具典型 TCL说明
GCC / ClangTCL3编译器 bug 可能引入；qualified version 贵
商业 qualified compiler (Greenhills MULTI / IAR)已 qualified$$$ 但省 effort
Coverity / PolyspaceTCL3vendor 提供 qualification kit
Git / Make / CMakeTCL1不直接影响 safety output
Code review toolTCL1—
Unit test framework (GoogleTest)TCL1-TCL2测试代码本身被 review
Hardware-in-loop (HIL) toolTCL3测试 setup 错可能漏 bug

开源 GCC / LLVM 怎么用

  • 商业 vendor 提供 qualified GCC fork（如 ACE Embedded TM-GCC, Greenhills, IAR）
  • 开源 LLVM/Clang 部分 vendor (Sonatype / Open Source) 在做 qualification
  • 2026 大多项目 + qualified vendor toolchain；开源 GCC 罕见用在 ASIL-D
  • 例外：很多公司用 GCC + diverse compilation（同 code 用 2 个 compiler 编 → 对比 output → 减少 single tool bug 风险）

测试 + 验证

Coverage / Fault Injection / 形式化验证

Coverage 要求

覆盖率ASIL AASIL BASIL CASIL D
Statement coverage++++++++
Branch coverage+++++++
MC/DC (Modified Condition/Decision)0++++

MC/DC 是航空 DO-178C 借鉴—— 每个 condition 必须能 independently 影响 decision 结果。极严格 + 测试用例多。

测试 levels

层测什么
Unit test单个 function / module
Component test多 module 集成
SW integration testSW component 间
HW-SW integration testSW + target HW（通常 HIL）
System validation整车 / SIL / 实车测试

Fault Injection

故意注入 fault 验证 safety mechanism 真起作用：

  • HW fault: bit flip / stuck-at / open-line（HW fault simulator）
  • SW fault: timing violation / NaN / overflow（test harness 注入）
  • Communication fault: 丢包 / 乱序 / 损坏 message
  • OS fault: scheduling / interrupt / memory corruption

形式化验证（formal verification）

ASIL D 推荐 (++)。工具：

  • Polyspace（MathWorks）—— abstract interpretation
  • Astrée（AbsInt）—— precise over-approximation
  • SPARK / Frama-C—— Ada / C 的 deductive verification
  • CBMC（开源）—— bounded model checking

主要捕：

  • Integer overflow / div-by-zero / null deref
  • Array index out of bound
  • Race condition / deadlock
  • Uninitialized memory
  • Stack overflow

真实成本 + 项目管理

成本分布（典型 ASIL-D 项目，500 万行 C++） · 典型 cost · 项目管理 challenges · PM tips

ISO 26262 项目认证成本占 40-60%，比纯 coding 大。EM / PM 必懂这层 economics。

成本分布（典型 ASIL-D 项目，500 万行 C++）

活动% effort
Requirements + Architecture design~20%
Coding (含 deviation handling)~25%
Unit test + Integration test~25%
Static analysis + formal verification~5%
Documentation (Safety case + traceability)~15%
3rd party Assessor 审核~5%
项目管理 + 配置管理~5%

典型 cost

项目规模ASIL典型 cost (USD)
小型（10 万行）B$500k - $1M
中型（50 万行）C$3M - $8M
大型（500 万行）D$20M - $50M

项目管理 challenges

  • 每行代码对应 5-10 行文档—— 不只 code review，是 requirement traceability + design rationale + test report
  • 独立审核—— TÜV / SGS / DEKRA / Exida 等 assessor 全程参与，按小时收费
  • Tool chain freezing—— 项目周期内不能升级 compiler / OS（每次升级要重新 qualify）
  • Change management 严格—— 每个 commit 要 link 到 requirement / safety analysis
  • 团队 ramp-up 慢—— 新人 understand process 需 3-6 月

PM tips

  • 项目 estimate 必须包含 process / documentation effort（不是纯 coding effort × 2）
  • 预留 10% contingency for assessor finding
  • 早期 engage assessor（不是 ship 前）—— "shift safety left"
  • Tool qualification 是 sunk cost—— 复用 across project
  • 建 reusable "ASIL-D library" 资产长期摊销

SOTIF + Cybersecurity 配套

ISO 21448 SOTIF · ISO/SAE 21434 Cybersecurity · 实施现状

ISO 26262 只管"已知的 unsafe 行为"（如 brake failure）。但 ADAS / 自动驾驶有"系统按 spec 工作但仍 unsafe" 的情况——这是 SOTIF 和 Cybersecurity 的范围。

ISO 21448 SOTIF

Safety Of The Intended Functionality—— 即便系统正常运作但环境 / 用法导致不安全。例：

  • ADAS 在罕见天气 (雪盲) 误识别
  • 感知错把广告牌当行人
  • 用户 over-trust 自动驾驶

SOTIF 跟 ISO 26262 互补 — 26262 管 HW/SW fault，SOTIF 管 spec 不充分。

ISO/SAE 21434 Cybersecurity

2024 强制（UNECE WP.29 R155 法规）。覆盖：

  • 整车 cybersecurity 管理（CSMS）
  • 类似 ISO 26262 的 V-Model，但管 security
  • Threat Analysis + Risk Assessment (TARA)
  • Security 监控 + incident response（车上线后）
  • 跟 26262 互动：security 失效可能导致 safety 失效

2026 实施现状

  • 26262 + SOTIF + 21434 三标合一是新一代 ADAS / 自动驾驶项目必须
  • cost 累加 → 总成本 +30-50% vs 单 26262
  • 大 OEM 都有"Functional Safety + Cybersecurity 联合团队"
  • Tier 1（Bosch / Continental / Aptiv 等）提供整套合规方案
🎯 关键 take-aways
  • ISO 26262 是汽车 functional safety 强制标准，主流市场不合规 = 没市场
  • ASIL A/B/C/D 由 S × E × C 矩阵推导；典型 brake/EPS = ASIL D
  • V-Model 全生命周期：每个 development artifact 对应 verification artifact，最终 Safety Case
  • ASIL Decomposition 让高 ASIL 拆成 redundant 低 ASIL channel —— 工程化必备
  • 软件层 (Part 6) 强制：禁 dynamic mem / 异常 / RTTI / unbounded loop / recursion / multiple inheritance
  • Coding Guideline（MISRA / AUTOSAR C++14）落地软件约束；deviation 流程严格
  • Tool Qualification（TCL）—— compiler / static analyzer 等 ASIL D 需 qualified 版本
  • 测试 + 验证：MC/DC coverage + fault injection + formal verification
  • 认证成本 40-60%—— 一行 code 对应 5-10 行文档
  • 26262 + SOTIF (21448) + Cybersecurity (21434) 三标合一是 2026 自动驾驶项目必备
