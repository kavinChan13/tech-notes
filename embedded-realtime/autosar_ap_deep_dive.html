1. 为什么是 AUTOSAR AP · CP 的限制 + 新一代 EE 架构

AUTOSAR Classic Platform (CP) 已经 20 年，是 80%+ 量产 ECU 的标准。但 2020s 起车端 architecture 正在根本变革：

新一代 EE 架构 (Electrical/Electronic)

世代架构典型
1.0 (分布式)~100 个独立 ECU，每个一个功能2000s 量产车
2.0 (域集中)域控制器（动力 / 底盘 / 智驾 / 座舱）+ Gateway2015-2020s
3.0 (中央计算 + 区域)1-2 个中央计算 + 4-6 个 zonal ECUTesla / 小鹏 / 蔚来 2022+；2024-2030 主流
4.0 (车云一体)边缘计算 + 云端 AI / OTA / fleet learning未来 5-10 年

CP 的限制

  • 静态 binding—— ECU 间通信 compile-time 定死，不能动态发现 / 添加 service
  • 有限计算 / 内存—— typically < 1 MB RAM / 几 MHz CPU；没法跑 AI / 视觉 / V2X
  • 没 POSIX—— OSEK/AUTOSAR OS 静态调度，不支持 multi-threading / 文件系统 / TCP/IP stack
  • C 为主—— C++ 用得少（嵌入式 OEM 不放心 OO + dynamic）
  • FOTA 困难—— Flash 全量升级 ≈ 30 min，OTA 不友好

AP 的目标

  • POSIX-based（典型 Linux / QNX）
  • C++14+（强 OO + 动态）
  • 动态 binding (SOA)—— Service Discovery 让 service 可动态发现
  • OTA-ready—— Application-level 增量更新 (UCM)
  • 满足 ASIL-D—— 跟 CP 同 functional safety 标准
  • 跑在高性能 ECU（> 100 MB RAM / GHz CPU），如 NVIDIA Drive / Qualcomm Snapdragon Ride / TI TDA4

3. ARA::Com 深入 · Service / Event / Method / Field

ara::com 是 AP 最重要的 FC—— 应用间通信。基于 SOA（Service-Oriented Architecture）。

4 大概念

概念对应用法
Service类似 protobuf service定义 interface（events + methods + fields）
EventPub/SubServer 推数据给 subscriber（如 sensor reading）
MethodRPCClient 调 server 函数 + 等返回
Field带 getter/setter + notify 的 property状态共享（如 vehicle speed）

Service Interface 定义（arxml）

用 arxml 工具（如 Vector DaVinci / EB tresos）定义 service interface，自动生成 C++ binding：

// 生成的 C++ binding (简化)
namespace services::vehicle {

struct SpeedData {
  double speed_kmh;
  Timestamp timestamp;
};

// Server side
class VehicleService_Skeleton {
public:
  void OfferService();

  // Event
  ara::com::Event<SpeedData> speed_event_;
  void SendSpeed(SpeedData data) { speed_event_.Send(data); }

  // Method (virtual, override by app)
  virtual ara::core::Future<double> GetMaxSpeed() = 0;

  // Field
  ara::com::Field<bool> cruise_active_;
};

// Client side (proxy)
class VehicleService_Proxy {
public:
  static ara::com::FindServiceHandle StartFindService(
    auto callback);  // 异步发现 service instance

  // Subscribe to event
  void SubscribeSpeedEvent(auto handler);

  // Call method
  ara::core::Future<double> GetMaxSpeed();

  // Read field
  ara::com::FieldSubscription<bool> cruise_active_;
};

}  // namespace

Service Discovery

Client 不预先知道 server IP / port，运行时通过 SD (Service Discovery) 找：

1. Server: OfferService(VehicleService, instance_id=1)
   ↓
   (SD multicast 到 224.224.224.245:30490)

2. Client: StartFindService(VehicleService, [instance_id])
   ↓
   (SD 收到 server 的 offer)

3. Client gets ServiceHandle: 知道 server endpoint (IP, port)

4. Client 调 method / subscribe event 直接走 SOME-IP 通信

Service 上线 / 下线时 SD 自动通知 client → 动态拓扑。

Async 编程模型

所有 ARA::Com 操作异步，返回 ara::core::Future<T>：

auto proxy = VehicleService_Proxy(handle);

// Method call (异步)
auto future = proxy.GetMaxSpeed();
future.then([](auto result) {
  if (result.HasValue()) {
    std::cout << "Max speed: " << result.Value() << "\n";
  } else {
    std::cout << "Error: " << result.Error() << "\n";
  }
});

// Event subscription
proxy.speed_event_.SetReceiveHandler([](SpeedData data) {
  std::cout << "Speed: " << data.speed_kmh << "\n";
});
proxy.speed_event_.Subscribe(/* max_samples= */ 10);

4. SOME-IP 协议 · Service Discovery + 序列化 + Header 详解

ARA::Com 底层通信走 SOME-IP（Scalable service-Oriented MiddlewarE over IP）—— 主流车端 RPC + Pub/Sub 协议。

位置

Application (ARA::Com API)
       ↓
SOME-IP message      (协议层, 16-byte header + payload)
       ↓
UDP / TCP            (Transport)
       ↓
Ethernet             (车内 100Base-T1 / 1000Base-T1)

SOME-IP Header

字段大小含义
Service ID2 byte唯一标识 service 类型
Method/Event ID2 byteservice 内 method/event 编号（高位区分 method/event）
Length4 byte整 message 长度（自此字段后）
Client ID2 byte调用方 client 标识
Session ID2 byte请求 / 响应配对
Protocol Version1 byteSOME-IP version (1)
Interface Version1 byteService interface 版本
Message Type1 byteRequest / Response / Notification / Error
Return Code1 byteOK / NotOK / ServiceNotAvail / ...
Payloadvariable序列化后的 method args / event data

固定 16 byte header + 紧凑 payload —— 比 HTTP/gRPC header 小很多，适合车内带宽受限。

Service Discovery（SOME-IP-SD）

SD 协议 sits on top of SOME-IP，跑在 multicast UDP port 30490：

Message用途
OfferServiceServer 周期性广播"我有此 service"
StopOfferServiceServer 下线通知
FindServiceClient 广播"谁有此 service"
SubscribeEventgroupClient 订阅 event 组
SubscribeEventgroupAck/NackServer 响应订阅

序列化

SOME-IP 用专门序列化格式（不是 protobuf / JSON）：

  • 固定 endian（network order = big-endian）
  • 所有字段紧凑排列，对齐到字节
  • 字符串 length-prefixed（UTF-8）
  • 动态长度 array 带 length
  • 支持 TLV（Tag-Length-Value）用于 backward-compat

对比 protobuf：

  • SOME-IP 序列化简单、固定 schema、紧凑 → 适合 deterministic 车端
  • protobuf 灵活、自描述、跨语言生态强 → 适合云端

开源实现：vsomeip

vsomeip（GENIVI / COVESA 项目，C++）是 SOME-IP 开源实现，被 AP 和非 AP（甚至非汽车）项目广泛用。

// vsomeip 简化 example
#include <vsomeip/vsomeip.hpp>

int main() {
  auto app = vsomeip::runtime::get()->create_application();
  app->init();

  // Offer service
  app->offer_service(0x1234, 0x0001);  // service_id=0x1234, instance_id=0x0001

  // Register message handler
  app->register_message_handler(0x1234, 0x0001, 0x8005,
    [](const std::shared_ptr<vsomeip::message> &msg) {
      // handle incoming method call
      auto response = vsomeip::runtime::get()->create_response(msg);
      app->send(response);
    });

  app->start();  // blocks
}

5. Persistency / Crypto / Identity Access Management

ara::per (Persistency)

标准化的持久化 API—— 替代直接 file IO：

  • Key-Value storage—— 简单类型读写
  • File storage—— 受管理的文件
  • Atomicity guarantees—— 写操作原子（事务）或 fail
  • Power-loss safe—— 断电中途保证数据完整性
  • Encryption-aware—— 跟 ara::crypto 集成
auto kvs = ara::per::OpenKeyValueStorage("vehicle_settings");
kvs.SetKey<double>("max_speed_limit", 130.0);
kvs.GetKey<double>("max_speed_limit").then(...);
kvs.SyncToStorage();  // commit

ara::crypto

密码学 API：对称 / 非对称加密 / 签名 / 验证 / KDF / 安全 random：

  • 跟 HSM (Hardware Security Module) / TPM 集成
  • Key 不出 HSM —— 加密 / 签名都在 HSM 内做
  • 满足 ASIL + cybersecurity (ISO 21434) 要求

ara::iam (Identity and Access Management)

谁能调谁的 service：

  • Application-level identity—— 每个 AA 有 cryptographic identity
  • Access control—— Service / method 级权限
  • Audit log—— 谁在何时调了什么

6. Execution Management + State Machine

ara::exec 管理 AA 生命周期：

State意思
kStartingAA 在初始化
kRunning正常运行
kTerminating正在 shutdown

Function Group + Mode

多个 AA 协调进入不同系统状态：

  • Function Group "Driving" / "Charging" / "Diagnostic" / "Update"
  • Function Group Mode—— 当前组的状态
  • 切换 mode → exec 管理 AA 启停 / 状态迁移

例：车从 "Driving" 切到 "Charging" mode，自动停掉 ADAS 相关 AA，启动充电相关 AA。

7. UCM · 车端 OTA 升级

ara::ucm (Update and Configuration Management) 是 AP 跟 CP 最大差别之一—— application-level OTA。

更新流程

  1. Package 下载—— 从 backend / FOTA server 下增量包
  2. 验证—— 签名 / hash / version check
  3. Pre-update—— 通知 AA 进入 update mode
  4. Install—— 替换 binary / config（多 partition / A-B / inactive partition）
  5. Verify—— 新版本 boot + 自检
  6. Activate / Rollback—— 切到新版本 OR 失败回滚

关键特性

  • A/B partition—— 主流（其他用 Inactive partition）
  • Delta update—— 节省带宽（diff 算法 if pre-known)
  • Rollback—— 新版本失败自动回老版本
  • Driving safety—— 车在动时 update 限制 / 推迟
  • Cybersecurity—— 全链签名 / 验签

  3. 实现 AA business logic

  5. 生成 binary + arxml config
  6. flash 到 ECU

  • build 慢—— 跨多 AA + arxml 代码生成，full build 可能数小时
  • 调试难—— 多 process + IPC + service discovery，bug 难 reproduce
  • 测试 infra—— Hardware-in-the-loop (HIL) setup 复杂
  • 规范变—— AUTOSAR 标准 yearly release，工具 / 代码持续 catch up

9. Functional Safety + Cybersecurity

ISO 26262 + AP

AP 设计支持 ASIL-A 到 D：

  • ASIL decomposition—— 高级 ASIL 拆成多个低级独立 channel
  • Safety mechanism—— Watchdog / health monitor / supervised execution
  • State manager—— Safe state on failure
  • Memory protection—— Process isolation via POSIX MMU

ISO 21434 (Cybersecurity)

AP 必须满足 2024 起强制的 ISO 21434：

  • Secure boot—— 信任链从 HW root
  • Code signing—— 所有 AA + update 包签名
  • Encrypted comm—— 关键 link 加密 (跨车 + 跨域)
  • Intrusion detection—— 异常检测 + 报警
  • Secure update—— 签名验证 + delta 完整性
实施挑战: ASIL-D + ISO 21434 双重合规让 AP 项目认证成本占 40-60%。一个中型 AP 项目（~50 万行 C++）认证 cost typical $5-10M USD。

10. 2026 现状 + 主机厂采用

采用 status

OEM / 区域AP 采用
BMW / Mercedes / VW已量产；新平台全 AP 中央计算 ECU
Bosch / Continental Tier1AP-based 控制器主流；提供给多 OEM
中国新势力（蔚来 / 小鹏 / 理想）部分用 AP，部分自研类 AP；倾向更灵活 frame
特斯拉不用 AUTOSAR—— 全自研 Linux + 自家 framework
日本 OEM (Toyota / Honda)跟随 BMW / Mercedes 节奏，2024-2026 加速
美国传统 OEM (Ford / GM)逐步采纳；Stellantis "STLA Brain" 平台 AP-based

vs 自研 Linux 框架

Tesla / 小鹏 / 部分中国 OEM 选自研 Linux + 自定义 framework，而不用 AP，原因：

  • AP 标准演进慢，跟不上 product roadmap
  • arxml / 商业工具不灵活 + 贵
  • 更想用云端工具链（k8s / gRPC / 现代 dev infra）
  • 软件团队来自 IT / 互联网 → 更熟 Linux + gRPC stack

结果：AP 不是唯一答案。2026 趋势是AP 跟自研 Linux + 自家 framework 并存。

未来 2-3 年

  • AP 23-11 / 24-11 标准持续演进（每年 spring / fall 两版）
  • Open-source AP 实现逐渐 mature（COVESA 等）
  • AI / ML 集成—— ara::ai (草案) 标准化车端 AI inference
  • Cloud 集成—— ara::v2x 跟 5G / V2X 标准化
