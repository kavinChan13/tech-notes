0.1 AI PM 跟传统 PM 的本质区别

维度传统 PMAI PM
需求确定性"做完 X feature → 用户看到 Y""做完 X feature → 用户可能看到 Y (取决于模型质量)"
验收标准functional 测试 pass必须有 eval set + metric 阈值，否则没法验收
风险维度bug / 性能 / 兼容+ 幻觉 / PII 泄漏 / 越狱 / 监管
Unit Economicsmarginal cost ~$0每 query 都 $X · 必须算 gross margin
失败模式明显（崩溃 / 报错）silent (回答慢慢变差 · 没人立刻发现)
跟工程协作跟 SDE Tech Lead 主多了 ML Eng / 数据 / Eval / Safety 4 类对接人

0.2 读完本文你会有的 5 个 deliverable 模板

  1. Week 1 信息收集清单（5 类必收信息 + 5 个必开 1:1）
  2. Month 1 产品决策矩阵（5 项决策 + decision framework）
  3. Quarter 1 必交付：AI PRD v1 + Eval Plan + Cost Model
  4. 5 份危机应对剧本（用户怒删 / vendor 涨价 / 老板压力 等）
  5. AI PM 季度自评清单
AI PM 跟传统 PM 最大不同：必须接受"AI 不可预测性"作为常态。传统 PM 控制 features，AI PM 控制 metrics + guardrails + fallback。第一周如果还在"我让工程师做个完美 AI"的心态，3 个月后必定翻车。
W1
Week 1 · 接手第一周
不下产品决策 · 不开 spec · 只收信息 + 建立信任

1.1 · 5 个必开的 1:1（按优先级）

① 上一任 PM / 你的老板（90 min）Day 1

  • 问：产品的"核心一句话"是什么 / 上 Q 没完成什么 / 哪 3 个用户痛点最重要 / 哪些 stakeholders 必须搞定
  • 关键：拿到老板的明确"成功标准"（north-star metric + 90 day goal）· 避免 3 个月后才发现期望错位
  • 不问：上一任 PM 个人评价 / 团队成员私下八卦（早问显得不专业）

② AI Tech Lead / EM（120 min · 必须 face-to-face 或视频）Day 1-2

  • 问：当前 AI 系统的 3 个最大技术债 / 你最担心的 1 件事 / 跟 PM 协作期望什么（频率 / 颗粒度 / 决策权）/ 你之前最讨厌的 PM 行为是什么
  • 关键：建立"我懂你工作的难"的共识 — AI 团队最讨厌"PM 拍脑袋承诺时间表"。第一周说 3 次 "我不懂 X，请你教我" 比说 3 次 "我们应该 Y" 赢更多信任
  • 送：明确表达 PM 角色 — "我不替你决定技术选型 · 我帮你 ① 跟业务谈优先级 ② unblock 跨团队依赖 ③ 把你的工作 sell 给老板"

③ 数据 / Eval 团队（60 min）Day 3

  • 问：当前有 eval set 吗 / 多大 / 多久更新 / 谁标注 / online metric 跟 offline 一致吗
  • 红灯：如果回答 "没有 eval set" 或 "半年没更新" → 这是 Month 1 必修 P0 决策（见 §2.4）

④ 客户支持 / 用户研究（60 min）Day 3-4

  • 问：用户反馈 top-3 槽点 / 你看到的 AI feature 表现怎么样 / 用户最容易卡在哪个地方 / 有没有用户用了离开
  • 关键：客户支持是 PM 最便宜的真实信号源 · 一线接触用户每天 ~50 次 · 1 次 1:1 比 5 份 dashboard 信息密度高

⑤ 法务 / Safety / Compliance（45 min · 别忽略）Day 4-5

  • 问：产品是否触发监管 (国内生成式 AI 备案 / EU AI Act / GDPR) / 上 Q 有 AI safety incident 吗 / 你看到的最大风险
  • 关键：很多 AI PM 翻车 = 没早建立跟法务的关系 · 出事时才找他们 = 慢 + 关系紧张。第一周就建立 monthly sync

1.2 · 5 类必收的信息（Week 1 末交出 1 页 doc）

信息类具体收什么用来干嘛
产品现状用户量 (DAU/MAU) · retention · 主要使用场景 / funnel判断"哪 funnel 步骤 AI 最有价值"
AI 性能当前 metric (accuracy / CSAT / acceptance rate) · 趋势 · 跟竞品差距判断"先优化质量 vs 扩 feature"
Cost / Unit Economics每月 AI cost · cost / revenue 比 · per-user cost判断是否要 cost optimization 排进 Q1
用户反馈top 槽点 · 高频投诉 · NPS / CSAT · churn 原因判断"用户最痛的 1 件事"
风险 / 合规safety incidents · 监管事项 · vendor 依赖识别 hidden time bomb

1.3 · 识别 3 个最大坑（Week 1 末口袋里要有）

每个 AI 产品都有 3 个"不爆就忍着 · 爆了就是 Sev-1"的坑。Week 1 必须识别 — 不一定 Week 1 解决，但记在 issue list 进入 Month 1 决策。常见 3 类：

🕳 Eval 黑洞

"团队从不跑 eval" 或 "eval set 半年没更新"。下次 model 升级 = 必爆（参考 AT02 事故 #1）。

🕳 Cost 失控潜势

"GPU/API 账单每月超预算 30% + 没人 own"。哪天 CFO 一发邮件就是危机。

🕳 Safety / 合规盲区

"PII 流到 vendor LLM 没 DPA" / "没做 prompt injection 测试"。出 PR 事件 = 公司级危机。

Week 1 通关
  • 5 个 1:1 全部完成（写有结构的笔记，存到自己的 PM dashboard）
  • 5 类信息表填齐（"未知"也要标记 "未知 + 谁能告诉我"）
  • 3 个最大坑识别（不一定解决，记在 issue list）
  • 第一周 memo（500 字）：给团队发 "我在 Week 1 听到的 / 还没听到的 / Month 1 优先级 draft"，建立信任
  • 跟老板第一次 1:1：报告 3 个坑 + Month 1 优先级 draft + 资源 ask
M1
Month 1 · 5 项关键决策
开始下产品决策 · 验证团队信任 · 锁定 Q1 优先级

2.1 · 决策 1：产品定位（Day 8-15）

问 3 个问题

  • ① "我们的 AI 真正解决的痛是什么" — 1 句话 · 不能含 "AI" 这个词（否则你在卖技术不是产品）
  • ② "用户即使 AI 答错也愿意继续用 = ?" — 找出 fallback acceptable 的场景（否则 0 容忍 = 必须 ≥ 99% 准确率 = 难做）
  • ③ "用 AI vs 用规则 / 人工 · 用户感知差什么" — 如果说不出 user-perceivable 差别 = 该 feature 可能不需要 AI

输出：1 页"产品定位 v1" doc（产品价值 / 关键场景 / fallback 模式）· 跟老板 + AI EM align

2.2 · 决策 2：项目优先级（Day 10-20）

3 类项目排清：

项目类排序原则
① 保维持类（现有 AI feature 维护 + 持续 eval）不能停 · 占 30-40% capacity
② 下季度承诺类（已答应业务的 feature）按业务价值排 · 砍 1-2 个 lower-impact 的（让 backlog "看起来满"的副作用是团队失焦）
③ 技术债 / Eval / Safety 投入至少留 20% capacity · 这是 AI PM 不同于传统 PM 的关键

2.3 · 决策 3：Eval 体系评估（Day 12-22）

3 问问出真相（跟 §1.1.③ 调查的延续）：

  • "上周如果改了 prompt，团队怎么知道效果好坏？" 答不上 → eval 体系不完整
  • "如果换 base model（GPT-4o → Claude），决策怎么做？" 答 "感觉" → 没有 model selection eval
  • "online metric 跟 offline eval 一致吗？" 答 "不知道" → 没有 alignment check

3 个问题任 1 个不及格 → Month 1 末必须启动 eval 体系重建。用 AT05 Eval Plan 模板 直接套 · 一个月内出 v1。

2.4 · 决策 4：Cost / Unit Economics 摸底（Day 15-25）

问 finance + AI EM 拉数据，建第一份 cost model（用 AT06 Cost Model 模板）：

必算的 3 个数意义
per-query cost判断 cost 是否随用户量线性飙升
per-user / month cost跟 ARPU 比 · 算 gross margin
cost / revenue 比SaaS 红线一般 < 25% · > 40% 必修

如果 cost / revenue 比 > 30%，立刻进 Q1 必修。典型 3 个 quick win（按 ROI 排）：

  • ① Prompt 优化：砍冗余 system prompt + few-shot → 典型 -20-40% input tokens
  • ② Model routing：简单 query → mini · 复杂才上 frontier → 典型 -40-60% cost
  • ③ Prefix caching + output 限长 → 典型 -10-30%

2.5 · 决策 5：安全 / 合规审查（Day 20-30）

5 条红线检查（任 1 个未通过 → 立刻 escalate）：

  1. PII 流转：用户数据是否合规进入第三方 LLM / training set · DPA 签了吗 · zero-retention 开了吗
  2. 幻觉 disclaimer：高 stakes 场景（医疗 / 法律 / 金融）有 disclaimer 吗
  3. Prompt injection 防护：用户输入有 sanitization 吗 · 有 red team 测试过吗
  4. Output safety：有 content filter / moderation 吗
  5. 监管备案：国内生成式 AI 备案 · EU AI Act risk class 评估
Month 1 通关
  • 5 项决策都有明确结论（哪怕是"保持现状 + 30 天后重审"）
  • 1 张 Month 2-3 优先级矩阵（business / tech / safety 三轴）
  • 1 份 cost quick win 清单（至少 2 个能立刻开干）
  • 1 次产品 all-hands：分享 Month 1 观察 + Q1 方向 · 收集团队反馈
  • 跟老板第二次正式 review：报告 Month 1 + 提前打 Q1 期望
Q1
Quarter 1 · 3 份必交付
把 Month 1 决策落成产物 · 季度末向上汇报有据可依

3.1 · 必交付 1：AI PRD v1（含 Risk / Eval / Cost 4 段）

用 AT01 AI PRD 模板 直接套。跟传统 PRD 关键不同的 4 段：

§5 · Eval plan

用什么 metric / eval set / 验收阈值 / 谁跑 / 多久跑。无 eval = 无验收。

§6 · Failure mode

AI 5 个典型失败场景 + UX 兜底（fallback / 转人工 / disclaimer 等）。

§7 · Cost model

per-request cost · 月度 forecast · cost / revenue ratio · 是否 sustain。

§8 · Risk & Safety

5 条红线（PII / 幻觉 / injection / IP / 监管）+ 法务签字。

仪式：PRD 必须有 Abandon Criteria（什么数据下我们决定砍掉这个 feature，不是改进）— 给团队心理预期，不让"沉没成本"绑架决策。

3.2 · 必交付 2：Eval Plan v1（offline + shadow + canary 3 阶段）

用 AT05 Eval Plan 模板。重点交付：

  • Eval set：200-500 case · cover 常见 (60%) + 边界 (20%) + 长尾 (15%) + 已知 fail (5%)
  • 4 阶段 pipeline：offline (CI) → shadow (pre-launch) → canary (5% → 20% → 100%) → production monitoring
  • Eval set owner：明确谁每月更新 · 不要"团队共同维护"（= 没人维护）

3.3 · 必交付 3：Cost Model + Q2 预算（CFO friendly）

用 AT06 Cost Model 模板。3 张表必交：

表内容给谁看
① 当前 cost 拆解API / GPU / 第三方 / 人力CFO + AI EM
② per-feature unit economics每 feature gross margin · 谁 cost 高谁低CFO + 老板
③ Q2 预算 ask基于业务增长 + cost optimization 的 net forecastCFO（审批）
CFO 最喜欢的 AI PM 是 "能用财务语言谈 AI" 的。如果能拿出 unit economics 表 + sensitivity（DAU × 2 / × 5 / × 10 时 cost）· 预算谈判成功率 +50%。不要等 CFO 来问，主动准备。
Quarter 1 通关
  • AI PRD v1 (4 段 AI 专属) 已发布给 stakeholder review
  • Eval Plan v1 + 第一次跑通 (offline 阶段) + 至少 1 个 model 改动用 eval 验证
  • Cost Model + Q2 预算已跟老板 + CFO 对齐
  • 产品 retro：Q1 哪些 deliverable 达成 / 哪些没达成 / 为什么
  • 个人 90 天复盘 doc（你做对什么 / 错过什么 / Q2 调整）
04

5 个危机场景 · 应对剧本

遇到时不要从零思考 · 直接套这 5 个剧本

AI PM 这个岗位的"危机种类"比传统 PM 多 — 不只是 bug / 性能，还有 model regression / 用户怒删 / vendor 涨价 / 老板焦虑 / 法务紧急。下面 5 个场景按出现频率排，Q1 内大概率会遇到至少 2 个。

场景 1 · 用户在社交媒体晒"AI 给我的离谱答案"
触发
某用户在 X / 知乎晒"我们 AI 给的离谱答案" → 转发 1000+ → PR 团队 / CEO 找你
第 1 小时
3 件事：① 截图存证 ② 团队复现 case（重要 — 区分是 bug 还是 user 截图 PS）③ 给 PR/法务一句话："正在复现 + 4 小时内给响应方案"。不要立刻道歉（也别狡辩 / 解释 "AI 局限"）
第 4 小时
出 1-page incident summary：root cause 假设 / 影响范围（多少用户可能遇到）/ 短期 mitigation（如加 disclaimer）/ 长期 fix。把响应控制权抓在手里 — 否则 PR / Legal 会按"安全"思路写，结果是 "永远不再发生" 这种不可信承诺
第 24 小时
RCA (用 AT02 Postmortem) + 跟 PR/法务 align 对外回应口径。承认 "AI 会出错 + 我们如何降低 + 用户可以 X 反馈"，不要承诺"永远不发生"。主动联系晒图用户私信道歉 + 邀其加入 beta tester — 多数情况能化敌为友
教训
长期：建 in-product feedback button（接 user feedback loop）+ 季度 red team 测试。这就是为什么 Month 1 决策 5 要审 safety
场景 2 · Vendor 涨价 30% / 突然 deprecate 某 model
触发
OpenAI / Anthropic / 国内 vendor 邮件通知 30 天后涨价 30% · 或 deprecate 你正在用的 model 版本
第 1 天
不要 panic。先算实际影响：① 我们 cost 涨多少（用 cost model 表算）② Q2 / Q3 财务影响多少 ③ unit economics 是否还成立
第 1 周
出 3 选 1 plan：A) 吸收 cost（短期 OK · 长期不行）B) 转嫁用户（涨产品价 · 风险大）C) 切替代 model（DeepSeek-V3 / Claude / Gemini）+ A/B test quality。选 C 配合 D 自部署评估 是中长期最稳
第 1 月
启动 model routing 改造（如果还没做）+ 建立"vendor health score"（每季度评估 vendor 稳定性 / 价格 / 合规 · 提前布局 plan B）
教训
永远不要 vendor lock-in。第一周就要确认 model 是 pin 版本（避免 silent 升级）· 架构层让 model 可切换（用 LiteLLM / 自研抽象层）· 跟 AT02 事故 #1 关系直接
场景 3 · 老板转发 "X 公司发布 Sora 同款" 要求我们也做
触发
老板看到竞品 demo / 文章后，每周给你转发 "我们怎么不做这个 / 为什么落后" · 持续 1+ 月
第 1 次
不要直接说"不做"。先问 3 个问题：① "你看重的是技术 demo 还是商业价值？" ② "我们的优先级排序是什么 vs 这个？" ③ "如果做，要砍现有哪个项目？" — 让老板自己排序，他会发现 trade-off
连续 2-3 次
建立"AI Watch List"机制：每周你主动给老板发 1-page "本周值得关注的 AI 进展 + 我们的评估"。抢主动权，从被动变主动 — 老板会少问"我们怎么不做 X"，多问"X 跟我们战略怎么配"
长期
每季度跟老板 align "我们的 AI 战略 + 不做什么"（明确 anti-goals）。老板的 AI 焦虑 = AI PM 必须管理的事 — 这是岗位隐性任务
场景 4 · 团队对 AI feature 失去信心（"这做不出来了"）
触发
某 feature 试了 4 周 · eval 提不上去 · 团队某成员 1:1 时说 "这 feature 可能本质上做不了"
第 1 周
不要立刻 push 团队"继续试"或"换思路"。先承认现实：① 跟 AI EM 拉 ground truth (eval 数据 / 失败 case 分析) ② 区分是 "方法没找到" 还是 "问题本质难"（前者改方向 · 后者砍 feature）
第 2 周
如果是后者：主动跟老板汇报 abandon 决定，引用 PRD 里的 abandon criteria。这是 PM 最难也最有 leverage 的决定 — sunk cost 时砍掉 = 后面 capacity 可以做对的事
第 1 月
做团队 retro：复盘 "为什么我们以为做得了 → 实际做不了"。不追责 · 找出 PRD 阶段判断哪里错（用户研究不够 / 技术调研不够 / abandon criteria 没认真定）
教训
AI 项目 abandon 不是 PM 失败 · 是 PM 成熟。但必须有结构化 retro，避免下次同样错
场景 5 · 法务紧急介入 (监管 audit / GDPR 投诉 / 国内备案问题)
触发
监管来 audit / 用户投诉 GDPR / 国内备案被要求修改 · 法务 cc 老板 cc 你
第 1 小时
不要先解释技术细节。先 ack 法务: "我 own 这事 · 24 hr 内给响应 plan"。然后拉 AI EM + Security 紧急 sync
第 1 天
出 1-page response：① 监管要求是什么 ② 我们现状是什么 ③ gap 是什么 ④ 30 天补救 plan ⑤ 是否需要 hot-fix（如关闭某 feature）
第 1 周
如果要 hot-fix，PM 必须决定 哪些 feature 临时关闭。跟客户支持 + Marketing 同步对外口径（"为符合监管要求，X feature 暂时不可用"）
教训
长期：建立季度合规 review（不要等 audit 才做）+ 跟法务 monthly sync · 把合规 cost 算进 cost model（不要把法务时间当 free）
05

AI PM 的 6 个职业资本积累方向

3-5 年看 · 让你跟传统 PM 拉开差距

同样做 PM 3 年，AI PM 比传统 PM 有 6 个独特的"职业资本"机会。Q1 之后就要有意识投入。

① Eval 体系设计能力

每家 AI 公司都缺懂 eval 的 PM。你在团队建过的 eval pipeline + metric framework 是高度可移植的能力 · 下一份 offer 直接 sell。

② AI Unit Economics 直觉

"能用财务语言谈 AI" 的 PM 会被 promote 到 Director 路径。每月主动给 finance 发 cost dashboard 是关键习惯。

③ Model Risk 管理

2026+ AI 监管收紧（中美欧）。能 own Model Risk Register + 跟法务 / 合规对话的 PM 是大公司 strategic hire。

④ User Research × AI

"用户对 AI 的真实使用模式" 跟 demo / 调研都不同（用户会用奇怪的 prompt / 在意想不到的地方失望）。能做 AI-specific user research 是稀缺技能。

⑤ AI Safety 产品思维

"safety 不是 nice-to-have · 是产品 differentiator"。能把 safety 设计成竞争优势的 PM（vs 把 safety 当成"工程师的事"）会被高度认可。

⑥ Public visibility (writing / talk)

AI 圈热度高，你管的 AI 产品 + 你写的 PM 文章 / talk 更容易被看到。坚持每季度 1 篇深度文章 / 1 次 talk · 2 年后是有名字的 AI PM。

⚠️ 反模式：只做 "AI 翻译官"

  • 很多 AI PM 沦为 "传话筒"（业务 ↔ ML 工程师）· 没自己的 judgment
  • 3 年后没独特技术或战略价值 · 下一份 offer 难谈
  • 必须主动积累上面 6 个之一 · 否则跟传统 PM 没差异
AI PM 跟传统 PM 的根本不同：传统 PM 资本是"用户洞察 + 跨部门协作"，AI PM 多了一层 "AI-specific judgment"（eval / cost / risk / safety / model selection）。有意识积累这层，未来 10 年红利。

8 个 FAQ

第一次当 AI PM 最常见的疑虑
Q1 · 我没 ML 背景，能管 AI 产品吗？
能 — 但 Day 1-30 必须补 4 类概念：① LLM / RAG / Embedding 基本概念 ② Eval / Hallucination / Guardrails ③ AI cost 数量级（per query 多少钱）④ 主要 model 对比（GPT / Claude / 国产 价格 + 能力）。
PM 不需要会推 backprop · 不需要会 fine-tune。PM 的价值是"决策框架 + 用户洞察 + 跨部门沟通"。
推荐前 2 周读：X06 Ch1-3 + X08 Ch1-2 + X12 Ch1-2 + X14 §2.6 AI PM 画像。
Q2 · 团队 ML Eng 跟我说话满口 jargon · 我跟不上怎么办？
不要装懂。说 "我没接触过 X · 你能用 1 句话解释吗" — ML Eng 比你想的更欢迎这种态度（他们烦的是装懂的 PM，不是诚实学习的 PM）。
自己建个 "AI jargon 字典"（10-30 个词）· 每周新词加 1-2 个。1-2 个月后你大部分对话能跟。
重要的不是听懂每个词，是听懂"这件事影响产品 metric 多少"。
Q3 · 团队总说 "再给 3 个月就能做出来"，但永远做不出来怎么办？
AI 项目本质高 uncertainty，"3 个月" 经常是工程师乐观估计。3 招破：
① 要 milestone 拆解：不接受"3 个月" 整体承诺，要"Week 4 / Week 8 / Week 12 各看到什么 metric"
② 要 success criteria：提前定 "什么数算成 / 算败" — 写在 PRD abandon criteria 段
③ 设 kill switch：约定"如果 Week 8 没达 X，重新评估方向"
PM 的工作是 "管理不确定性"，不是 "消除不确定性"。
Q4 · "这 feature 该用 AI 还是用规则" 我怎么判断？
3 个 reality check：
① "规则能解决 80% 吗"？能 → 先做规则版本上线（更快 + 0 cost）· 留 20% 给后续 AI
② "用户能容忍 AI 偶尔错吗"？不能（如金融转账）→ 规则 + 人工兜底，AI 仅辅助
③ "cost / revenue 能 sustain 吗"？算 unit economics · > 30% → 重审是否值得用 AI
反模式："老板要 AI · 那就上 AI"。3 个月后才发现用规则更便宜更准 = 资源浪费。
Q5 · 怎么向老板说 "这 feature 该砍掉"？
数据 + 结构化：
① 用 PRD 里写的 abandon criteria（"我们 Week 8 定的 metric 阈值是 80% accuracy · 当前 67% 已经 4 周无进展"）
② 给 alternative use（"砍掉后释放的 capacity 可以做 X · 预期 ROI 是 Y"）
③ 承认决策风险（"我可能错 · 但等下去的 cost 是 Z"）
老板讨厌的不是砍 feature · 是没想清楚 / 没数据就砍。抛 abandon criteria 比抛"感觉不行"赢 10x trust。
Q6 · 跟 AI EM / Tech Lead 协作分工怎么定？
清晰边界（参考 M07 AI EM Playbook）：
① PM own 什么：what (做什么 feature) · why (用户/业务价值) · when (季度优先级) · success metric · abandon criteria
② EM / Tech Lead own 什么：how (technical 方案) · who (team 人员) · 工程质量 / eval 体系 / cost 优化 method
③ 共同 own：cost target · safety / risk · timeline · hiring 需求
红线：PM 不替 EM 做技术选型（model / framework / infra）· EM 不替 PM 排 feature 优先级。互相尊重对方专业领域。
Q7 · 我管的 AI 产品 cost 月度超预算，CFO 找我 · 我怎么应对？
3 步（跟 §4 场景 2 同框架）：
① 第 1 天：先拉数据归因（"X% 来自业务增长 + Y% 来自 cost-per-request 涨"）· 不要立刻回邮件解释
② 第 2 天：回 CFO 3 段邮件：数据归因 / 短期 mitigation (具体 2-3 个) / 长期 plan (model routing / 量化 / 自部署 timeline)
③ 第 1 周：启动 2-3 个 quick win + 给 CFO 1 张 Q2 cost trajectory 图
长期：每月主动给 finance 发 1-page cost dashboard + forecast。主动报告时你是 owner，被动回应时你是 victim。
Q8 · 我 Q1 干得好，Q2 该怎么 scale up？
Q2 从 "做事" 转 "建系统"。3 个 scale-up 优先级：
① 把 Q1 一次性决策变成机制：eval review / cost review / model selection 进周/月会议程
② 培养 1 个能 backup 你的 APM（不是接班 · 是关键决策能 2 个人扛）
③ 开始投资跨产品影响力：其它 AI 产品的 PRD / Eval / Cost 你能不能提供 template · 推动公司级 AI PM standard
这 3 个做好，你从"feature PM" 升级为"AI 产品系列 owner" → Director path 打开。
最后忠告：AI PM 是 2024-2030 最容易"做出 outsized impact"的 PM 岗位。原因：行业新 + 没有 best practice + 老板都在摸索。你的 Q1 决策可能就成了公司的 AI product 标准。认真做 · 回报 5x。
