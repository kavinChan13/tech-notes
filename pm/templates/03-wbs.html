01是什么 / 为什么

WBS（Work Breakdown Structure）是项目唯一的范围基线。Sprint Backlog 替代不了它，因为 backlog 看不到层级、依赖、范围闭合。

WBS 没拆好，所有计划都是沙上建塔：CPM 关键路径算不准、Cost Baseline 估错、RACI 漏角色、风险登记漏盲区。

WBS 的"100% 原则"：父节点 100% = 所有子节点之和。不能漏，也不能加。

02字段说明

字段为什么存在 / 怎么用
WBS 编号1.0 / 1.1 / 1.1.1 — 树状层级编号，每级 ≤ 9 个子节点
名称（动词 + 名词）"开发用户系统" 而不是 "用户系统" — 说明可交付动作
WBS Dictionary每个工作包的"说明书"：可交付物 / 验收标准 / 假设 / 排除
负责人 (Owner)每个工作包唯一 owner，不能多人
工时估算用 PERT 三点估算 (O+4M+P)/6，不要单点
前置依赖哪些工作包必须完成才能开始本包（CPM 输入）
验收标准 DoD什么算"完成"？必须可验证
状态Not Started / In Progress / Done / Blocked
Insight WBS 第一层应当是"主要可交付物"（用户系统 / 商品系统 / 基础设施），而不是"项目阶段"（需求 / 设计 / 开发 / 测试）。后者是 V 模型不是 WBS — 范围不闭合 + 看不到产品维度。

03WBS 验收 5 问

  1. 父节点工作 = 子节点工作总和？（100% 原则）
  2. 每个叶子是否能在 8-80 工时（1 天到 2 周）完成？
  3. 每个叶子是否有唯一负责人？
  4. 是否有"项目管理 / 风险 / 沟通"这一支？容易漏。
  5. 是否包含"集成测试 / 上线 / 培训 / 文档"这些非开发工作？

⚖️DO & DON'T

✅ DO

  • 按"主要可交付物"拆第一层
  • 叶子节点 8-80 工时（1 天到 2 周）
  • 每个叶子唯一 owner
  • 必有 "项目管理 / 上线 / 培训 / 文档" 这一支
  • 用 PERT 三点估算，不要单点
  • WBS 进基线 = 进版本管理（Confluence / Git）

❌ DON'T

  • 第一层不要按"需求/设计/开发/测试"（V 模型不是 WBS）
  • 叶子不要写"前端开发" — 太大，无法估算
  • 不要直接抄 Sprint Backlog —— 失去层级
  • 不要 1 个叶子 2 个 owner —— 必失败
  • 不要拆超过 7 层 —— 过度细化
