PRD · 智能消息分层 · v1.0

Author: Kachan · Date: 2026-04-30 · Status: Approved

§1 · 背景与目标

1.1 一句话价值：让 30+ 万企业 IM 用户在跨部门讨论时，3 秒分清"必须看"和"可以稍后看"。

1.2 背景：

  • JTBD J-007：消息分层（12 访谈合成）
  • 现有"消息洪流"导致 65% 用户漏看关键消息
  • Carol Persona P-002：每天浪费 35 分钟手动扫消息
  • Why now：① 公司战略 OKR 提到"提升内部效率"；② AI 摘要技术成本 1/3

1.3 目标：

  • 业务：D7 留存 35% → 45%（NSM +15%）
  • 用户：关键消息漏看率 65% → ≤ 30%
  • 非目标：① 不做语音消息摘要（v2）；② 不替换现有 IM（仅增强层）；③ 不做 AI 自动回复

1.4 成功标准：上线 60 天后 D7 留存 ≥ 45% + 客诉"漏看"减少 50%

§3 · 范围与功能（节选 1 个核心功能）

A1 · 智能消息分层（基于 @ + 发送人 + 相关度） — P0

User Story：

作为跨部门 PM (Carol)，我想在打开 IM 时立刻看到"按重要度分层"的消息，以便我能在 30 秒内决定先处理哪 5 条。

Acceptance Criteria：

Scenario 1: @ 提及 + 老板发送
  Given 用户 Carol 打开 IM
  And   有一条消息: 1 小时前老板 @ 了她
  When  Carol 进入消息列表
  Then  这条消息排在 "必看" 区域顶部
  And   显示橙色 ⚠ 角标
  And   悬浮提示 "@ + 直属上级"

Scenario 2: 普通群消息无相关性
  Given 用户在 50 人群
  And   消息无 @ 任何人，发送人非 Carol 直系
  When  Carol 进入消息列表
  Then  这条消息排在 "稍后看" 区域
  And   不显示提示

Scenario 3: 用户手动改变排序权重
  Given Carol 把"老板"权重设为 5（最高）
  When  老板发任意消息
  Then  即使无 @ 也排进 "必看"

边界 / 异常：

  • 无网络：分层缓存最近 24h 数据；无新消息时显示"离线模式"
  • 权限：本人无权限的群消息不参与分层
  • 性能：分层算法在 100ms 内返回，否则降级到传统时间排序

§5 · 度量与上线（节选）

5.1 指标：

  • 主：D7 留存（关联 §1.3 业务目标）
  • 次：每日"必看"区点击率 / 分层准确率（用户调研）/ 平均消息处理时长
  • 护栏：消息延迟 P99 / Crash / 客诉（"分层错"）/ D90 留存

5.4 上线计划：

  • W0-W2 内部 dogfooding（PM 团队 50 人）
  • W3 灰度 1%（约 3000 用户）
  • W5 灰度 10%（A/B vs 老版本，关联 Template 12）
  • W7 灰度 50%
  • W9 全量 + 长期 Holdout 1%
  • 回滚条件：① P99 延迟 +10% 持续 1h；② 客诉率 > 0.1%；③ Crash > 0.05%
  • 30 / 60 / 90 天 Post-Launch Review
Insight 填好示例只展示了 PRD 的"骨架 3 段"。完整 PRD 一般 20-40 页，但每段保留同样的结构。维护成本来自结构化，不是字数。

⚖️DO & DON'T

✅ DO

  • 每段开头让目标读者一眼找到
  • 非目标必填（防范围蔓延的种子）
  • 每个功能配 AC（Given-When-Then）
  • 视觉走 Figma，PRD 只写行为逻辑
  • 头部引用 JTBD ID + Persona ID
  • 每个 PRD 必有 §5 度量（不是上线时再补）

❌ DON'T

  • 不要把视觉细节写进 PRD（"按钮 #4f8cff"）
  • 不要优先级全是 P0
  • 不要省略异常分支 / 边界
  • 不要把"老板的话"当需求（无用户证据）
  • 不要 PRD 写完不开评审会就进开发
