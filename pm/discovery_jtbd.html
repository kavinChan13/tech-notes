00总览：从"需求"到"机会"到"方案"

EM 时代你接的是需求（已经定义好的"做这个功能"）。PM 时代你接的是机会（用户的痛点和场景），需要自己生成"做什么 / 怎么做"。Discovery 是从"机会"到"方案"的闭环。

1️⃣ 发现机会

  • 用户研究（访谈 / 观察 / 数据）
  • JTBD：用户雇用你的产品做什么"job"
  • Opportunity Solution Tree 把机会按权重排序

2️⃣ 验证机会

  • 问题验证（用户真的有这个痛吗？）
  • 规模验证（有多少人有？愿付多少钱？）
  • 竞品验证（已有解决方案差在哪？）

3️⃣ 设计方案

  • 多个 Solution 候选，pick 1-2 个做原型
  • 原型测试（usability / 可接受度）
  • 进入 Definition 阶段写 PRD

01双钻模型 · Discover / Define / Develop / Deliver

英国 Design Council 提出，4 个阶段交替"发散（钻石变宽）"和"收敛（钻石变尖）"，是产品发现的总框架。

   Discover        Define        Develop       Deliver
   ◇━━━━━━━━━━━╳━━━━━━━━━━━◇━━━━━━━━━━━╳━━━━━━━━━━━◇
   发散            收敛            发散            收敛
   问题空间 ──────────────  解决方案空间

① Discover（发散，问题空间）

"用户在干什么？痛点是什么？" 阶段任务：

  • 用户访谈、观察、数据分析
  • 桌面调研、竞品分析
  • JTBD 提取
  • 不要急着想"我们做什么"

② Define（收敛，问题空间）

"真正要解决的核心问题是什么？" 阶段任务：

  • 合并相似洞察
  • 建 Opportunity Solution Tree
  • 写 Problem Statement（一句话）
  • 问题陈述要明确：用户、场景、痛点、影响

③ Develop（发散，方案空间）

"能怎么解决？" 阶段任务：

  • Brainstorm 多个 Solution
  • 每个 Solution 做低保真原型
  • Sketch / Wireframe / Concept Test
  • 不要纠结"哪个最好"，先量出多种可能

④ Deliver（收敛，方案空间）

"选一个方案做出来。" 阶段任务：

  • 原型测试 → 选优（usability / 可接受度）
  • 写 PRD / 进入工程交付
  • 定义成功指标 + A/B 实验设计
  • 上线后做 Post-launch Review

新手最常见的"跳钻"失误

EM/工程师转 PM，最常见的失败模式：跳过 Discover/Define，直接进 Develop。一上来就想"我做个 X 功能"，结果功能上线没人用。

心法：每个新项目至少花 1-2 周做 Discover + Define。这两个阶段省下，后面 8 周开发就是浪费。

02JTBD · Jobs To Be Done

由 Clayton Christensen（《创新者的窘境》作者）推广。核心：用户不"购买产品"，他们"雇用产品来完成某个 job"。把视角从"功能"转到"待完成的工作"，是产品思维的根本转变。

2.1 JTBD 三层结构

Functional 功能性

用户要完成的具体任务。例：从家到公司。

Emotional 情感性

用户在过程中希望感到什么。例：通勤路上想感到放松、不焦虑。

Social 社会性

用户希望被别人怎么看。例：开特斯拉而不是 Honda 是因为想被看作"科技中产"。

2.2 JTBD Statement 模板

标准模板

当 [情境]，
我想 [动机 / 行动]，
以便 [期望结果]。

When ___ ,
I want to ___ ,
so I can ___ .

例 1：通勤

当我早上要从家到公司的 30 分钟里，
我想让大脑不被打扰、获得一些有用信息，
以便到公司时已经准备好开始工作。

可能的"产品候选"：播客 / 听书 / 冥想 App / 看新闻 / 听音乐

例 2：信用卡

当我做一笔 ¥1000 的网购时，
我想让自己有"先享后付"的安全感，
以便不必一次性掏空账户余额。

可能的"产品候选"：信用卡分期 / 花呗 / 京东白条 / 银行延期还款

JTBD 比 Persona / 用户故事强在哪？

① JTBD 不绑定"是谁"，只看"为什么"。同一个 job 不同人都需要（30 岁白领和 50 岁工人都要"通勤路上充电"）。
② JTBD 让你看到真正的竞争对手。奶昔的对手不是别的奶昔，是"早餐通勤的香蕉"（见下例）。
③ JTBD 比"用户痛点"更准，因为它锚定的是"目标 / 期望结果"，而不是"症状"。

03奶昔案例 · JTBD 经典必读

Clayton Christensen 的奶昔故事，是 JTBD 最经典的入门。

背景

美国某连锁快餐想提升奶昔销量。传统做法：调研用户喜欢什么口味、什么粘稠度。结果：调研做了几年，销量没动。

JTBD 团队的发现

实地观察发现：

  • 40% 奶昔卖给"通勤司机"，时间是早上 6-9 点
  • 这些人单独购买，立刻开车走
  • 他们不喝完，会拿到公司继续喝

用 JTBD 重写

当我早上 1 小时无聊的开车通勤时，
我想要一种"能单手吃 / 持续 1 小时不腻 / 不脏手 / 让我感到充实" 的食物，
以便不至于空手开过这段无聊路程。

奶昔的真正"竞品"：香蕉、Bagel、咖啡、甜甜圈

结论

奶昔不是输给"别的奶昔"，而是输给"香蕉、咖啡"等"通勤伴侣"。
改进方向：① 奶昔做得更稠（吸更久不腻）；② 加入"小颗粒"让人感到"在吃东西不是在喝"；③ 早晨快速通道、自助机；④ 移动支付。
结果：早上时段奶昔销量上涨 7×。

从奶昔故事学到的 3 个 PdM 心法

① 用户问"喜欢什么口味"是错问题，应该问"在什么场景下你怎么喝它"。
② 真正竞品来自"同 job 不同 solution"，不要只看行业内对手。
③ 把功能优化绑定到 JTBD，而不是绑定到"用户调研最高分项"。

04Opportunity Solution Tree · 把机会按结构化方式排序

Teresa Torres 在《Continuous Discovery Habits》提出。把"结果 → 机会 → 方案 → 实验"四层串成一棵树，让产品决策可追溯。

                        🌟 Outcome（业务结果）
                        提升结算转化率 12% → 15%
                                  │
            ┌─────────────────────┼─────────────────────┐
            │                     │                     │
       💡 Opportunity 1     💡 Opportunity 2     💡 Opportunity 3
       价格疑虑               信任担忧               时间紧迫
            │                     │                     │
       ┌────┼────┐         ┌────┼────┐         ┌────┼────┐
       │    │    │         │    │    │         │    │    │
   🛠 Sol  Sol  Sol     🛠 Sol  Sol  Sol     🛠 Sol  Sol  Sol
   运费   凑单  价保    评价   品牌  权威    一键  地址  快递
   可视   提示  承诺    前置   背书  认证    支付  记忆  时效
       │
       │
   🧪 Experiment（A/B 测试）
   假设：价格疑虑 → 运费可视化 → 转化 +1pp
   样本量：16,896/组 · 跑期：4 周

OST 4 层结构

  • Outcome 结果：1 个，对齐到 OKR / 北极星指标
  • Opportunity 机会：3-7 个，从用户研究中得出的"用户的痛 / 需求"
  • Solution 方案：每个机会下 2-5 个候选方案
  • Experiment 实验：每个方案配 A/B 测试或 Concept Test

OST 的 3 个原则

  • Outcome 优先：所有 Opportunity 都要回到同一 outcome
  • 横向多样性：每个 Opportunity 下有多个 Solution，避免一条路黑到底
  • 持续修剪：实验完一个分支，把无效的剪掉，留下能打的

05Mom Test · 用户访谈三铁律

Rob Fitzpatrick 在《The Mom Test》提出："如果连你妈都会给你善意但虚假的肯定，你就需要这套规则"。访谈最大的敌人不是用户撒谎，是你的问题让用户没法不撒谎。

① 谈生活，不谈你的想法

错："你觉得我们做一个 X 应用怎么样？"
→ 用户出于礼貌会说"听起来不错"。
对："你最近一次遇到 [那个场景] 是什么时候？"
→ 谈具体经历，没法演戏。

② 问具体的过去，不问未来

错："如果我们做了 X，你会用吗？"
→ "未来意愿"几乎都是高估的。
对："过去 1 个月你是怎么解决这个问题的？花了多少钱 / 时间？"
→ 行为 > 意愿。

③ 少说话，多倾听

访谈黄金比例：用户讲 80%，你讲 20%。
错：边问边介绍自己产品。
对：每个回答后追问"为什么？" / "然后呢？" / "举个例子？"

5.1 用户访谈脚本（30 分钟标准版）

0-3 min · 暖场
  - 自我介绍（轻；不暴露产品）
  - 解释录音 / 隐私
  - "今天主要想了解你们的工作 / 生活，不是测试任何产品"

3-10 min · 背景画像
  - 你日常做什么？
  - 一周里 [相关场景] 发生几次？
  - 你的角色是什么？

10-22 min · 行为追问（核心）
  - "上一次发生 [场景] 是什么时候？带我过一遍当时怎么做的"
  - 在哪一步最痛？
  - 为什么不解决？
  - 现在怎么 work-around？花了多少钱 / 时间？
  - 5 Whys：为什么？为什么？为什么？……

22-27 min · 验证（可选）
  - 如果展示一个简单 mockup："这个看起来能解决你说的痛吗？"
  - 不要解释功能，让对方解读

27-30 min · 闭环
  - 还有什么我应该问但没问的？
  - 你认识 2-3 个有同样问题的人吗？能介绍我聊吗？
  - 如果 6 周后我们做了一个版本，你愿意试用吗？

5.2 5 Whys 与行为追问

例：用户说"我希望搜索更智能"

PM："智能"是什么意思？能不能讲一个具体例子？
用户：上次我搜"红裙子"，结果出来一堆白色的。

PM：你当时为什么搜红裙子？
用户：朋友婚礼。

PM：为什么搜不到让你恼火？
用户：我得一个个翻图，浪费 10 分钟。

PM：现在你怎么解决？
用户：去某宝改成搜"红色 连衣裙 婚礼"。

PM：那你为什么不直接去某宝？
用户：我们公司给的预算只在你这买，但搜得太烦了。

→ 真正洞察：不是"搜索智能"，是"约束条件下的体验差到要切站"。
→ 解法不一定是 NLP，可能是"颜色 + 场合"快捷筛选。

5 Whys 的常见误用

不要问 5 次"为什么"就停。要的是逐层追到具体行为和成本。问"为什么"得到的回答很多是"我习惯" / "我觉得"，要再问"上一次是什么时候 / 怎么做的"才能拿到证据。

心法：每个抽象回答（"我觉得 / 我习惯 / 我希望"）都用具体行为追问替代。

06Persona · 用户画像模板

Persona 是把多次访谈洞察"合成"到一个虚构但典型的用户身上，便于团队共识。Persona 不是市场调研报告，是产品决策工具。

6.1 Persona 标准模板（一页纸）

王

王小敏 · 32 岁 · 互联网产品经理

"我希望一天 8 小时不被无意义会议占满"

背景 Demographics
  • 北京 / 已婚 / 1 个 3 岁孩子
  • 互联网中型公司 (200 人)，5 年经验
  • 带 1 个产品方向，10 个工程师
目标 Goals
  • 提升团队产品决策质量
  • 减少无效会议时间，留出战略思考时间
  • 3 年内升 P7
痛点 Pain Points
  • 每天 6 个会，没时间深度思考
  • 跨部门同步信息成本高
  • 项目状态散落在 Jira / 邮件 / IM
JTBD
  • "当我每天上班时，我想快速看到所有项目状态，以便决定今天最该解决什么。"
动机 Motivations
  • 专业认可 / 晋升 / 成就感 ★★★★★
  • 工作生活平衡 ★★★★
  • 金钱回报 ★★★
目前 Workaround
  • 每天早上自己看 Jira + 整理 Notion
  • 建了一个 Excel 跟踪关键风险
  • 痛苦但不愿换工具（团队成本高）
典型一日
  • 9:00 看 Jira / 邮件 / IM 30 分钟
  • 9:30-12:00 4 个站会 + 评审
  • 14:00-18:00 跨部门 + 决策
  • 20:00 下班路上想"今天解决了什么"

好 Persona 的特征

  • 有具体名字 / 头像 / 一句标语
  • 有 JTBD 而不只是兴趣
  • 有动机优先级（不只列动机）
  • 有"目前 workaround"（最有用！）
  • 更新版本号 / 上次访谈日期

糟糕 Persona 的特征

  • 只有人口学数据（年龄 / 收入 / 性别）
  • 没有动机和 workaround
  • 是"理想用户"，不是"真实用户"
  • 团队从来不引用
  • 2 年没更新

07用户旅程图 · User Journey Map

把用户在某个 JTBD 上的完整体验流程展开成一张图，标出每一步的行为 / 情绪 / 痛点 / 机会点。

想法 "这个能解决我的问题吗？" "靠不靠谱？" "值不值这个价？" "为什么不能 X？" "我会再用吗？"
情绪 😐 好奇 🤔 犹豫 😰 焦虑 😊 满意 / 😡 失望 😄 / 😐 / 😡

旅程图 vs 漏斗

漏斗：客观行为转化数（已发生的事），缺少情绪和痛点。
旅程图：用户主观体验全景，包含情绪、动机、痛点。
最佳实践：先用漏斗找出"哪一步流失最多"，再用旅程图深入"为什么流失"。

088 道高频面试题（Discovery / 用户研究）

Q1你怎么判断一个需求是真的还是假的？
高分

"3 步：① 把'用户说的'翻成 JTBD（场景 + 动机 + 期望结果）；② 看现在 workaround 是什么 + 花了多少钱/时间（最强假设证据）；③ 量化规模（多少人有 + 痛多深）。
Mom Test 三规则：谈生活不谈想法、问具体过去不问未来、少说话多听。
关键：意愿 ≠ 行为。'你会用吗' 几乎都是高估。"

Q2JTBD 和 User Story 的区别？
高分

"User Story 模板：作为 [角色]，我想 [功能]，以便 [价值]。偏'功能粒度'。
JTBD 模板：当 [场景]，我想 [动机]，以便 [结果]。偏'目的粒度'，不绑定具体角色。
JTBD 让你看到真正的竞品：奶昔的对手不是别的奶昔，是'通勤路上的香蕉'。
实战：用 JTBD 找机会，用 User Story 写实施。"

Q3讲一次你做用户研究的经历
高分（STAR）

"S/T：之前一个企业 IM 项目，老板觉得"加视频会议"，我怀疑用户真要的不是这个。
A：约了 12 个目标用户访谈（5 角色），用 Mom Test 规则，重点问"上一次跨部门沟通是怎么做的"。
R：发现真正痛点是"消息太多无法区分优先级"而不是"没有视频会议"。改方向做"消息分层 + AI 摘要"，留存提升 18%。
反思：如果直接听老板的，会做出可有可无功能。Discovery 是 PM 的护身符。"

Q4用户访谈最少访多少人？
高分

"经验法则：5 个起步 / 12-15 个收敛。Nielsen 研究：5 个用户能找到 80% 的可用性问题。
但要点不是数量是多样性：覆盖不同 persona、角色、阶段。
Continuous Discovery（Teresa Torres）建议：每周访谈 2-3 个，常态化，不是项目启动一次就停。"

Q5双钻模型每个阶段的关键产出？
高分

"① Discover：用户访谈纪要、JTBD 列表、行为数据观察；
② Define：Problem Statement、Opportunity Solution Tree、北极星指标；
③ Develop：3-5 个 Solution 候选、低保真原型、Concept Test 结论；
④ Deliver：PRD、A/B 实验设计、Launch Plan、Post-launch Review。"

Q6怎么避免"PM 自嗨需求"？
高分

"4 道防线：
① JTBD 原文：每个需求必须能写出对应的 JTBD Statement（不能写就是没真正理解用户）；
② workaround 证据：用户现在怎么解？花了多少？没付出代价的痛是假痛；
③ Concept Test：原型给 5 个用户看，0 兴趣的不做；
④ Pre-mortem：6 个月后这个需求失败，最可能的原因是什么？"

Q7Persona 怎么用？很多人觉得是"装饰品"
高分

"Persona 不是装饰品的话，要满足 3 条：
① 有 JTBD 和动机，不只人口学；
② 有 workaround（这是最有用的部分）；
③ 团队真的引用：每个 PRD 开头要标'目标 Persona = X'。
团队每季度对着 Persona 做一次"她现在还会这么想吗"的更新，避免石化。"

Q8Discovery 阶段需要 PM 自己动手做用户研究还是请 UX Researcher？
高分

"PM 必须自己做至少一部分，不能完全外包。
① 一手数据是 PM 的产品直觉来源；
② 关键访谈听感受 / 表情 / 沉默，文字纪要传不出来；
③ UXR 帮你做严谨研究和合成，但不能替你产生洞察。
最佳：PM 每月至少做 4 个用户访谈，UXR 做大样本调研 + 合成。"

095 句箴言收口

  1. 用户不"购买产品"，他们雇用产品完成 job。
  2. 意愿 ≠ 行为，问"上次怎么做的"比问"未来会不会用"准 10 倍。
  3. 真正的竞品来自"同 job 不同 solution"，不要只看行业内对手。
  4. 跳过 Discover 直接做开发的 8 周，等于浪费 8 周。
  5. Persona 没有 workaround 部分，就是装饰品。
