WBS · 工作分解结构

WBS 的两大铁律 · 📐 100% 原则 · 🪜 分到工作包（Work Package） · 一个真实例子：电商 App MVP（按可交付物拆 vs 按阶段拆）

WBS（Work Breakdown Structure）是项目唯一的范围基线。Scrum Story 不能替代它，因为 Story 看不到层级、依赖、范围闭合。WBS 没做好，后面所有计划都是沙上建塔。

1.1 WBS 的两大铁律

📐 100% 原则

父节点的 100% 工作 = 所有子节点工作之和。不能漏，也不能加。如果父节点叫"前后端开发"，子节点只列了"登录 / 列表 / 详情"，缺了"个人中心"，整个 WBS 就破了。

检查方法：自下而上回卷，看是不是 100%；多了一项无父归属的活动也违反。

🪜 分到工作包（Work Package）

叶子节点必须分到能独立估算工时、独立分配 owner、独立验收的"工作包"。一般每个工作包 8-80 工时（1 天到 2 周）。

分到 5 ± 2 层为典型。再细就过度，建议落到子任务表（任务清单）里管，不要污染 WBS。

1.2 一个真实例子：电商 App MVP（按可交付物拆 vs 按阶段拆）

选 A：按可交付物拆（推荐）

1.0 电商 App MVP（项目） 1.1 用户系统 1.1.1 注册登录 1.1.2 找回密码 1.1.3 个人中心 1.2 商品系统 1.2.1 商品列表 1.2.2 商品详情 1.2.3 搜索 / 筛选 1.3 交易系统 1.3.1 购物车 1.3.2 下单 / 支付 1.3.3 订单管理 1.4 基础设施 1.4.1 CI/CD 1.4.2 监控告警 1.4.3 日志聚合 1.5 项目管理 1.5.1 立项与计划 1.5.2 风险与沟通 1.5.3 验收与上线

为什么推荐"按可交付物"：每个工作包都有清晰可验收的交付物，跟用户故事/PRD 易对应；后期变更也只动相关分支。

常见 WBS 反例

  • WBS 第一层就是"需求 / 设计 / 开发 / 测试 / 上线" —— 这是 V 模型，不是 WBS。范围不闭合，且把"项目阶段"当成了"产品分解"。
  • 叶子节点写"前端开发"—— 太大，无法估算、无法分配。
  • WBS 直接抄 Sprint Backlog —— 失去了层级，也无法识别关键路径。

验收 WBS 的 5 个问题

  • 父节点工作 = 子节点工作总和？（100% 原则）
  • 每个叶子是否能在 8-80 工时里完成？
  • 每个叶子是否有唯一负责人？
  • 是否有"项目管理 / 风险 / 沟通"这一支？容易漏。
  • 是否包含"集成测试 / 上线 / 培训 / 文档"这些非开发工作？

估算 · PERT 三点法

例题 · 别名 / 变体

单点估算（"3 天"）几乎一定是错的。PERT 三点估算把"乐观 O / 最可能 M / 悲观 P"三个值算出加权均值，并能给出 ±σ 的不确定性区间。

PERT 期望值（Beta 分布近似）

tE = (O + 4M + P) / 6
σ = (P − O) / 6

O (Optimistic)：一切顺利时的耗时；M (Most Likely)：最可能的耗时；P (Pessimistic)：踩坑后的耗时。

直觉：M 占 4/6 权重 ≈ 67%。P-O 越大说明你越不确定。σ 是 1 个标准差，意味着实际耗时落在 [tE-σ, tE+σ] 的概率约 68%。

2.1 例题

题面

"商品搜索"工作包估算 3 个工程师给的：O=4 天，M=7 天，P=18 天。tE 和 σ 分别是多少？

解

tE = (4 + 4×7 + 18) / 6 = (4 + 28 + 18) / 6 = 50 / 6 ≈ 8.33 天
σ  = (18 − 4) / 6 = 14 / 6 ≈ 2.33 天
即：68% 的概率实际耗时 ∈ [6.0, 10.7] 天；
    95% 的概率实际耗时 ∈ [tE−2σ, tE+2σ] = [3.7, 13.0] 天。

从单点估算 7 天 → PERT 期望 8.33 天，多出来的 1.33 天就是"风险溢价"。

2.2 别名 / 变体

方法公式使用场景
PERT (Beta)(O + 4M + P) / 6有历史数据 / 一定经验，最常用
Triangular(O + M + P) / 3无历史数据、对 Beta 不确定时
Wide-band Delphi多专家匿名估算 + 收敛不确定性极高、有专家分歧时
Planning PokerStory Point 序列估Scrum 内部估算，不替代 PERT，是另一个粒度

CPM · 关键路径法

关键概念速记 · 完整例题（你能手算就过关了） · 压缩工期的两个手段（必考） · 赶工 Crashing · 快速跟进 Fast-tracking

关键路径（Critical Path）= 项目网络图中最长的那条路径，决定了项目最早完工日。关键路径上任何一个活动延迟 1 天，整个项目延迟 1 天。

3.1 关键概念速记

缩写全称含义
ESEarliest Start最早开始时间，由前置活动 EF 决定
EFEarliest FinishEF = ES + Duration
LSLatest Start不影响项目完工的最晚开始时间
LFLatest FinishLF = LS + Duration
Float (Slack)总浮动时间LF − EF（或 LS − ES）。Float = 0 → 关键活动
FFFree Float不影响后续活动 ES 的浮动时间

3.2 完整例题（你能手算就过关了）

题面：6 个活动构成的项目

活动前置工期(天)
A 立项—3
B 设计A5
C 后端B10
D 前端B8
E 集成测试C, D4
F 上线E2

① 前向计算（求 ES / EF）

A: ES=0,  EF=0+3=3
B: ES=3,  EF=3+5=8
C: ES=8,  EF=8+10=18
D: ES=8,  EF=8+8=16
E: ES=max(EF_C, EF_D)=18, EF=18+4=22
F: ES=22, EF=22+2=24

项目最早完工：第 24 天。

② 反向计算（求 LF / LS）

F: LF=24, LS=24-2=22
E: LF=22, LS=22-4=18
C: LF=18, LS=18-10=8
D: LF=18, LS=18-8=10  ← 注意 D 的 LS 是 10
B: LF=min(LS_C, LS_D)=8, LS=8-5=3
A: LF=3,  LS=3-3=0

③ 求 Float

A: 0  | B: 0  | C: 0  | D: LS-ES=10-8=2 | E: 0  | F: 0
关键路径 = A → B → C → E → F (Float=0)
非关键 = D（有 2 天 Float，可以晚开始 2 天而不影响项目）

项目经理立刻得到 3 个判断：① 不要把任何资源调走 C；② D 有 2 天浮动，资源可以挪给 C 救急；③ 关键路径上有 5 个活动都"零容错"，是高风险区。

3.3 压缩工期的两个手段（必考）

赶工 Crashing

对关键路径活动追加资源，缩短工期。代价：通常成本上升非线性（加班/外包/外购）。

挑选规则：优先 crash 每天压缩成本最低 的关键路径活动。一直 crash 到工期满足或成本不再划算。

快速跟进 Fast-tracking

把原本串行的活动改并行。代价：返工和质量风险上升。

典型用法：把"详细设计 → 编码"改为"详细设计完 60% 就开始编码"。

陷阱：Near-Critical Path 才是真凶

很多 PM 只盯着关键路径，忽略了 Float 只有 1-2 天的"近关键路径"。一旦它们因为风险延误，会变成新的关键路径，让项目延期。

心法：监控 Float ≤ 总工期 5% 的所有路径，不只盯 Float=0 的那一条。

Gantt 与里程碑 · 让计划"可被汇报"

① 只放工作包级 · ② 必须标里程碑 · ③ 双线显示

Gantt 不是用来给团队看 daily 的（Kanban 才是），它是给干系人和高管看的。3 条原则保证 Gantt 不会变垃圾。

① 只放工作包级

Gantt 横轴时间长，纵轴不要展开到任务级，太密无人看。展开到工作包（WBS 叶子）即可。

② 必须标里程碑

关键评审点（设计评审、UAT、上线）用菱形 ◆ 标记。无里程碑的 Gantt = 没有节奏感的演奏会。

③ 双线显示

同一活动画两条：上：基线（计划）、下：实际。一眼能看到偏差，胜过任何文字汇报。

Gantt + 关键路径标注（建议样式）

活动    | W1  W2  W3  W4  W5  W6  W7  W8
─────────────────────────────────────────────
A 立项  | ████
B 设计  |     ████ ████
C 后端  |               ████ ████ ████ ████ ████  ★关键路径

EVM · 三个基础值 PV / EV / AC

三个基础值（必背） · PV · Planned Value · EV · Earned Value · AC · Actual Cost · 一个例子建立直觉

挣值管理（Earned Value Management）是工程界 50 年验证有效的进度+成本一体化度量法。核心思想：用"挣到的价值"对比"计划要挣的"和"实际花的"，从而同时知道进度和成本健康度。

5.1 三个基础值（必背）

PV · Planned Value

计划价值：截至检查点，按计划应该完成的工作量（以预算计）。又叫 BCWS（Budgeted Cost of Work Scheduled）。

简记：你计划在这个时间点该做完多少钱的活。

EV · Earned Value

挣值：截至检查点，实际完成的工作量按预算计算的价值。又叫 BCWP（Budgeted Cost of Work Performed）。

简记：你实际做完了多少钱的活（按计划价计）。

AC · Actual Cost

实际成本：截至检查点，已完成工作的实际花费。又叫 ACWP（Actual Cost of Work Performed）。

简记：你实际花了多少钱。

5.2 一个例子建立直觉

题面

项目预算 BAC = 100 万，计划工期 10 个月。第 5 个月末检查：

  • 按计划应完成 50% → PV = 50 万
  • 实际完成 40% → EV = 40 万
  • 实际花了 60 万 → AC = 60 万

三句话解读

  1. "该挣 50 万，只挣到 40 万" → 进度落后 10 万 (SV = EV − PV = -10)
  2. "挣 40 万却花了 60 万" → 成本超支 20 万 (CV = EV − AC = -20)
  3. 结论：项目既慢又贵（双红），需要立刻干预。

EVM 最大的反直觉

很多新手把"AC 比 PV 高"当作"超支"，这是错的。AC vs PV 没意义，必须以 EV 为锚点：

• 进度评估：EV vs PV （我做完的 vs 该做完的）
• 成本评估：EV vs AC （我做完的价值 vs 实际花的钱）

06

EVM · 偏差与指数 SV / CV / SPI / CPI

4 象限速判（背下来） · 可视化：S 曲线（PV / EV / AC 同图） · 典型项目 S 曲线（PV / EV / AC）

把 PV/EV/AC 三个值组合，得到 4 个最常用的健康度指标。面试和实战 90% 的题都靠这 4 个。

进度偏差 SV (Schedule Variance)

SV = EV − PV

SV > 0：进度提前；SV = 0：按计划；SV < 0：滞后。

单位是钱，但解读为进度。例：SV = -10 万 ≈ 滞后约 10/(BAC/工期) = 10/(100/10) = 1 个月。

成本偏差 CV (Cost Variance)

CV = EV − AC

CV > 0：成本节约；CV = 0：按预算；CV < 0：超支。

直接的金额偏差。例：CV = -20 万 → 已超支 20 万。

进度绩效指数 SPI

SPI = EV / PV

SPI > 1：超前；SPI = 1：按计划；SPI < 1：落后。

例：SPI = 0.8 → 每计划工作 1 元，只做出 0.8 元的活，进度低 20%。

成本绩效指数 CPI

CPI = EV / AC

CPI > 1：成本好；CPI = 1：刚刚好；CPI < 1：超支。

例：CPI = 0.67 → 每花 1 元只换回 0.67 元的活，钱效率打了 67 折。

6.1 4 象限速判（背下来）

CPI ≥ 1（成本好）CPI < 1（超支）
SPI ≥ 1（不滞后）绿区：又快又省。继续保持。黄区：进度好但花得多。可能赶工赶过头，看看能不能减资源。
SPI < 1（滞后）黄区：钱省但慢。资源不足，需要追加投入。红区：又慢又贵。立刻拉警报、找根因、考虑变更基线。

6.2 可视化：S 曲线（PV / EV / AC 同图）

典型项目 S 曲线（PV / EV / AC）

EV 落后 PV 即进度滞后；AC 高于 EV 即成本超支。三线撕开就是项目失控的信号。

EVM · 预测 EAC / ETC / VAC

综合例题（必做） · 前面的 SPI/CPI 是"现在状态"，EAC/ETC/VAC 是"未来预测"。当老板问"还能按时按预算完成吗"，你需要这一组。

前面的 SPI/CPI 是"现在状态"，EAC/ETC/VAC 是"未来预测"。当老板问"还能按时按预算完成吗"，你需要这一组。

EAC · Estimate at Completion（完工估算）

4 种公式，对应 4 种假设：

EAC₁ = AC + (BAC − EV) ← 假设剩余工作按原预算执行
EAC₂ = BAC / CPI ← 假设剩余工作仍按当前 CPI（最常用）
EAC₃ = AC + (BAC − EV)/CPI ← 同 EAC₂ 的变体写法
EAC₄ = AC + (BAC − EV)/(CPI×SPI) ← 假设剩余 CPI 和 SPI 都不变

新手默认用 EAC₂。EAC₄ 用于已经红区、需要最悲观估算的项目。

ETC · Estimate to Complete（完工尚需估算）

ETC = EAC − AC

"接下来还要花多少钱才能完工"。用于预算请款。

VAC · Variance at Completion（完工偏差）

VAC = BAC − EAC

"按现在节奏，最终预算偏差"。VAC < 0 即超支。

TCPI · To-Complete Performance Index（完工尚需绩效指数）

TCPI = (BAC − EV) / (BAC − AC) ← 想守住原预算
TCPI = (BAC − EV) / (EAC − AC) ← 想守住新 EAC

TCPI > 1.05：未来需要比当前更高效率才能完工，大概率守不住。报警阈值。

7.1 综合例题（必做）

题面（沿用前面 100 万项目）

BAC = 100，PV = 50，EV = 40，AC = 60。计算 SV、CV、SPI、CPI、EAC（4 公式）、ETC、VAC、TCPI（守原预算）。

解

SV   = 40 − 50 = -10      → 滞后 10
CV   = 40 − 60 = -20      → 超支 20
SPI  = 40 / 50 = 0.80     → 进度低 20%
CPI  = 40 / 60 = 0.67     → 钱效率 67%

EAC₁ = 60 + (100−40) = 120     → 假设后续按原预算（最乐观）
EAC₂ = 100 / 0.67    = 149.3   → 假设 CPI 不变（典型）
EAC₃ = 60 + (100−40)/0.67 = 149.6  ≈ EAC₂ 的小数差异
EAC₄ = 60 + (100−40)/(0.67×0.80) = 171.9 ← 最悲观

ETC  = EAC₂ − AC = 149.3 − 60 = 89.3  → 还要花 89.3 万
VAC  = 100 − 149.3 = -49.3            → 大概率超支 49.3 万
TCPI(守原 BAC) = (100−40)/(100−60) = 60/40 = 1.50 → 报警，几乎守不住

结论：SPI/CPI 双低 + TCPI=1.5，立刻给老板发预警，启动变更控制。
建议方案：① 追加预算到 EAC₂ 的 150 万；或 ② 削减 25% 范围 + 资源平衡。

资源平衡 vs 资源平滑 vs 关键链 CCM

Resource Leveling 资源平衡 · Resource Smoothing 资源平滑 · Critical Chain CCM · CCM 的 3 个 buffer 速记

"WBS 也对、关键路径也算了，但资源对不上" 是真实项目里第一大坑。这一节搞清楚 3 个易混概念。

Resource Leveling 资源平衡

当资源被"过度分配"（同一时间需求 > 供给），允许调整 deadline 来对齐资源。

典型：核心架构师每周只能给 20h，但你 plan 了 40h，要么招人，要么延后部分活动。

副作用：项目工期可能延长。

Resource Smoothing 资源平滑

当资源紧张但不允许动 deadline，在非关键路径活动的 Float 内挪动，让资源曲线平滑。

副作用：可能消耗光 Float，让一些活动变成"近关键路径"。

Critical Chain CCM

Goldratt 提出。把"个人安全余量"统统拿出来，在关键链尾部加 Project Buffer，在馈入路径加 Feeding Buffer。

核心理念：人天然把 buffer 藏在估算里，不如集中管理 → 项目 buffer 更小、更可靠。

9.1 CCM 的 3 个 buffer 速记

Buffer 类型位置大小（经验）作用
Project Buffer (PB)关键链末端，紧贴交付日期前关键链工期 × 50%吸收关键链上活动的延误
Feeding Buffer (FB)非关键链汇入关键链处馈入链工期 × 50%避免非关键链延误"传染"到关键链
Resource Buffer (RB)关键资源到来前提前 1-3 天关键资源（顶级专家）准时到位

什么时候用 CCM 而不是 CPM？

CCM 适合：① 资源约束强（专家稀缺）；② 不确定性高；③ 团队习惯把估算"灌水"。
不适合：合同对范围/时点有刚性约束、需要详细 milestone 给客户的项目（CCM 中间没有传统里程碑）。

风险登记 + 5 类应对策略 + 变更控制

风险登记册（Risk Register）字段 · 5 类应对策略（消极风险） · ① 规避 Avoid · ② 转移 Transfer · ③ 减轻 Mitigate · …

EVM 是"现在已经发生的事"，风险管理是"还没发生但可能发生的事"。两者一起才是完整的项目度量体系。

10.2 5 类应对策略（消极风险）

① 规避 Avoid

改变计划，让风险消失。例：取消有合规争议的功能。

② 转移 Transfer

把风险转给第三方承担。例：买保险、外包给专业公司。

③ 减轻 Mitigate

降低概率或影响。例：增加测试覆盖、设双供应商。最常用

④ 接受 Accept

主动接受。积极接受= 设储备金；消极接受= 出事再说。

⑤ 上报 Escalate

风险超出本项目权限，升级给项目集 / 高管处理。例：法律风险、组织级架构问题。

10.3 变更控制 CR / CCB

CR (Change Request) 表必备字段

  • 变更 ID + 提出人 + 日期
  • 变更内容简述（1 句话）
  • 影响分析：进度（±天） / 成本（±钱） / 范围 / 风险
  • 建议方案 + 不批准的后果
  • CCB 决议：批准 / 拒绝 / 推迟 / 部分批准
  • 批准后的新基线版本号

CCB (Change Control Board) 节奏建议

① 小项目：每周 30 分钟，PM + Sponsor + 关键 stakeholder；
② 大项目：双周 1 小时 + 紧急流程；
③ 紧急 CR > 24h 决策不能拖到下周例会，要走"async approval"。

Q1解释一下 EVM 的 3 个基础值，越简单越好

Q2项目过半 SPI=0.7、CPI=0.8，怎么向 VP 汇报？
高分（BLUF + 数据 + 行动）

"项目目前红区：进度低 30%、成本超 25%。EAC₂ 预测 比原 BAC 多 25%。我建议两条路二选一：① 追加预算 X 万守原范围；② 削减 ~20% 范围守预算。本周五前需要决策，否则关键路径会进一步恶化。"

红旗

"嗯……项目有点慢，我们尽力赶……"（没数据、没方案、没决策点，VP 会失去信心）

Q3给一个 6 个月项目，你会怎么做计划？
高分

"5 步走：
① WBS 拆到 5 层、每个工作包 8-80 工时；
② 工作包用 PERT 三点估算，给出期望和不确定区间；
③ 画 CPM 网络图，找出关键路径和近关键路径；

红旗

"开个站会用 Jira 拉 backlog，做 sprint planning……"（这是 Scrum master 答案，不是 PgM 答案；6 个月项目不能只靠 sprint backlog）

Q4关键路径和敏捷有冲突吗？
高分

"不冲突，是不同粒度。Sprint 内用 Kanban / Scrum 管 daily。Release 级用 CPM 找关键路径，识别跨 sprint 依赖。混合（Hybrid）模式在大项目里是主流。"

Q5资源不够，你怎么决策？
高分（决策树）

"先看 deadline 是否硬：
① 硬 deadline → Resource Smoothing（在非关键路径 Float 内调度）；不够再 Crashing（追加资源）。
② 软 deadline → Resource Leveling（动 deadline，不动资源）。
③ 两边都吃紧 → 启动 Change Request，把范围 / 时间 / 成本三角变两条。"

Q6风险评分 P × I = 16，你下一步做什么？
高分

"P×I=16 已属高风险（≥12 报警）。下一步：
① 给风险定 owner；
② 选 5 类策略中之一（最常用 Mitigate）；
③ 写出 contingency plan + 触发器；
④ 估算储备金（contingency reserve）；
⑤ 周度 review，状态进 RAID Log。"

Q7客户突然要加 20% 范围，怎么办？
高分

"启动 CR：
① 评估对进度（关键路径 +X 天）、成本（+Y 万）、风险（关键资源已超载）的影响；
② 提供 3 个方案：A 全接（必延期 + 加预算）/ B 部分接 / C 拒绝并解释；
③ CCB 决议 → 更新基线 → 通知所有 stakeholder。"
关键：不能口头答应，所有变更走 CR 流程，避免范围蔓延。

Q8你认为 PgM 最大的能力是什么？
高分（有底气的回答）

"3 件事：
① 提前 4 周看到坏消息（结构化风险扫描 + EVM 走势）；
② 把模糊变具体（任何承诺要落到 WBS / RACI / Date）；
③ 跨部门讲共同语言（用 PV/EV/AC 对工程，用 NPV/Payback 对财务，用 Roadmap 对产品）。"

红旗

"沟通能力" / "抗压能力"（太空，没有可观察行为，没有专业辨识度）
