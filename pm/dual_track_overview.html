一句话辨析 · PgM / PdM / EM / TPM

很多人把"PM"当一个岗位讲，其实在英文圈是 4 个完全不同的角色

很多人把"PM"当一个岗位讲，其实在英文圈是 4 个完全不同的角色。先搞清楚你想去哪边，否则学错了重点。

角色 一句话定义 最核心问题 衡量指标 主要文档 常见出身
PgMProject / Program Manager 把"已经决定要做的事"按时按质按预算交付 "我们能不能按 deadline 交？阻塞在哪？" OTD、CPI/SPI、风险关闭率、范围偏差 Project Charter、WBS、Schedule、RAID Log、QBR PMO、QA、TPM、Scrum Master
PdMProduct Manager 决定"做什么 / 为什么做 / 给谁做"，对产品成败负责 "用户真的有这个痛点吗？我们做了之后指标会涨吗？" 北极星指标、留存、转化、GMV、NPS、PMF PRD、MRD、用户研究 Report、Roadmap、Launch Plan UX、运营、咨询、技术背景转产品
EMEngineering Manager 对团队和系统的健康负责，长期工程效能 "我的团队能不能持续高质量交付？人和系统都在长大吗？" Eng Velocity、Quality、Retention、System Health 1on1、Career Plan、Tech Strategy、Postmortem Senior Engineer、Tech Lead
TPMTechnical Program Manager 跨多团队的技术大项目协调与交付，PgM 的技术加强版 "这个跨 5 个团队的项目，技术依赖和风险都摸清了吗？" 跨团队 OTD、依赖闭环率、Tech Risk Burndown RFC、Tech Plan、Cross-team Roadmap、Dependency Map Tech Lead、Senior Engineer、PgM 进阶

三个真实问句，立刻分辨

  • "用户为什么要用我们而不是竞品？" → 这是 PdM 的题
  • "按现在的节奏 2 月底能不能交？哪些任务在关键路径上？" → 这是 PgM 的题
  • "团队 P3-P5 各 1 人，半年内有人能升 P6 吗？" → 这是 EM 的题

新手最常见的角色错位

  • 把 PdM 当"需求收集员" → 实际上 PdM 要主动决定不做什么
  • 把 PgM 当"催进度的" → 实际上 PgM 要提前 4 周看到风险
  • 把 EM 当"技术更资深的 PgM" → EM 还要管人和系统，时间维度长得多

知识体系树 · PgM × PdM 全景

🔵 Project Manager 知识树 · 🌸 Product Manager 知识树 · 🟣 共通能力（PgM/PdM/EM 都要）

把 PgM 和 PdM 的核心知识各自树状展开，再标出三者交集（共通能力）。每个叶子节点都对应仓库里的一份配套深度文档。

🔵 Project Manager 知识树

A. 计划 / Planning（命脉）

  • 项目章程 Project Charter — 立项必备
  • WBS 工作分解结构 — 100% 原则、5±2 层
  • CPM 关键路径 — 串联最长链，决定 deadline
  • PERT 三点估算 — (O+4M+P)/6
  • Gantt / Milestone 时间线可视化
  • 资源平衡 / 关键链 CCM — Buffer 管理

B. 执行 / Tracking（铁规）

  • EVM 挣值管理 — PV/EV/AC、SPI/CPI、EAC/ETC 弱项①
  • S 曲线 — 进度健康度的最直观图
  • Burndown / Burnup — Sprint 与 Release 视角
  • Change Request 变更控制

C. 风险 / 质量 / 采购

  • RAID Log（Risk / Assumption / Issue / Decision）
  • 风险矩阵 — 概率 × 影响、5 类应对策略
  • QC 七工具 + 六西格玛 DMAIC（必备入门）
  • 合同类型 FP / CR / T&M、外包风险

D. 干系人 / 沟通

  • 权力-利益方格、Salience 三圈模型
  • RACI / DACI / RAPID 责任矩阵 弱项③
  • Status Report、Steering Committee 节奏

E. 方法论体系

  • PMBOK 7 版（5 大过程组 × 10 大知识领域）
  • PRINCE2（重治理、英式国企/欧洲常见）
  • Agile / Scrum / Kanban / SAFe（你已熟）
  • Hybrid / Disciplined Agile

🌸 Product Manager 知识树

A. 发现 / Discovery（最关键）

  • JTBD Jobs To Be Done — Clayton Christensen
  • 双钻模型 Discover-Define / Develop-Deliver
  • Opportunity Solution Tree — Teresa Torres
  • 用户访谈 5 Whys、Mom Test 三规则
  • Persona + 用户旅程图

B. 定义 / Definition

  • PRD / MRD / BRD 三层文档
  • 用户故事 User Story + INVEST
  • 验收标准 Acceptance Criteria（Given-When-Then）
  • 需求优先级：RICE / Kano / MoSCoW / Value-Effort

C. 度量 / Metrics（命脉）

  • 北极星指标 NSM
  • AARRR / RARRA 海盗指标
  • HEART 框架（Google）
  • 留存曲线 / Cohort 分析 弱项②
  • A/B 测试：假设、样本量、显著性、Guardrail 弱项②
  • SQL 入门（PM 视角，能拉数即可）

D. 商业 / Business

  • 商业模式画布 BMC + 精益画布 Lean Canvas
  • 定位 Positioning（4P / STP）
  • 定价 Pricing — Cost+ / Value-based / 阶梯
  • GTM Go-To-Market 策略
  • SaaS：MRR / ARR / Churn / LTV / CAC

E. 战略 / Lifecycle

  • PMF Product-Market Fit（Sean Ellis 测试）
  • Crossing the Chasm（早期市场→主流市场）
  • 产品生命周期 PLC（导入/成长/成熟/衰退）
  • Roadmap 三种风格（Now-Next-Later / Theme / Outcome）

🟣 共通能力（PgM/PdM/EM 都要）

沟通 / 影响力

  • Pyramid / BLUF / SCQA 三件套
  • Influence without Authority（Cohen-Bradford 6 类货币）弱项③
  • Harvard 谈判 7 要素 + BATNA / ZOPA
  • Bad-news 30min Rule

数据 / 分析

  • SQL 基础（SELECT / JOIN / GROUP / 窗口函数）
  • Excel / Sheets：pivot / vlookup / what-if
  • Tableau / PowerBI / Looker 任选其一
  • 统计基础：均值/中位数、p 值、置信区间

战略 / 商业

  • OKR vs KPI（不同场景下用法）
  • Good Strategy / Bad Strategy（Rumelt）
  • SWOT / PEST / Porter 五力
  • 三张报表的看法（懂术语即可）

① 项目经理（PgM）· 学什么、产什么

必备技能清单（按"够用→精通"分级） · 必产出物清单（PgM 工具箱） · 📄 立项 / 启动 · 📊 计划 / 基线 · ⚠️ 执行 / 监控 · …

② 产品经理（PdM）· 学什么、产什么

必备技能清单（按"够用→精通"分级） · 必产出物清单（PdM 工具箱） · 🔍 Discovery / 发现 · 📝 Definition / 定义 · 📈 Metrics / 度量 · …

PdM 的本质是 "在不确定中决定做什么 / 不做什么"。最大的认知颠覆：需求不是收来的，是发现和验证出来的。

A/B 测试会读 A/B 实验报告设计实验 + 显著性 + Guardrail实验平台治理 + 多实验冲突

战略 / Roadmap会写 Now-Next-LaterOutcome-based Roadmap + OKR3 年产品愿景 + 跨年下注

面向高管能写 1-pager strategy memo能 BLUF / SCQA 汇报能在董事会替产品线辩护

④ 三大弱项攻坚 · 计划/EVM · 数据/AB · 干系人

这是你主动选出来的三大短板

这是你主动选出来的三大短板。这一节给每一个短板一份"为什么、学什么、怎么验证学会了"的攻坚计划，详细深度内容会在配套独立文档里展开。

🟢 弱项 ① 项目计划 / EVM / 关键路径 / 资源平衡 Planning & Earned Value Management

为什么这是大坑：EM 用 Scrum 太久，会本能地反感"长周期计划"，但跨团队 / 跨季度的项目离不开 EVM。一旦上 50 人 / 半年以上的项目，没有 EVM 就是裸奔。

必学 4 件事

  • 计划阶段：WBS（5 层、100% 原则） + CPM（关键路径，最长链）+ PERT（三点估算 (O+4M+P)/6）
  • 挣值管理 EVM 7 个公式：PV/EV/AC、SV=EV-PV、CV=EV-AC、SPI=EV/PV、CPI=EV/AC、EAC=BAC/CPI、ETC=EAC-AC
  • 资源平衡：资源直方图、leveling vs smoothing、关键链法 CCM 的项目缓冲（Project Buffer）/ 馈入缓冲（Feeding Buffer）
  • 变更控制：CR 表 + CCB 评审 + 基线再发布

学会了的标志（自检）

  • 给一个 6 个月项目，你能画出关键路径并指出 3 个最危险的"近关键活动"
  • 项目过半时，老板问"还能按时完成吗？"，你能用 SPI/CPI + EAC 给出有数据支撑的回答
  • 资源冲突时，你能区分"应该平衡（不动 deadline）"还是"应该 Smoothing（动 task float）"

配套文档：pm_evm_planning_guide.html

🟢 弱项 ② 数据分析 / A/B 测试 / SQL / 指标体系 Data Literacy & Experimentation

为什么这是大坑：技术背景的 EM 看技术指标（QPS、P99）很顺手，但产品指标（漏斗、留存、Cohort）和实验思维（假设→样本量→显著性）需要单独训练，否则被产品/数据团队牵着走。

必学 5 件事

  • 指标体系：北极星指标 NSM、AARRR / RARRA、HEART、Guardrail 指标的设计
  • 分析手法：漏斗分析、留存曲线、Cohort、辛普森悖论、归因模型（最后/首次/线性/U 型）
  • SQL 8 招：SELECT / JOIN / GROUP BY / HAVING / WINDOW / CTE / CASE / 时间函数
  • A/B 设计：H0/H1 假设、样本量计算、α 错误（误判率）/β 错误、MDE、Holdout
  • 实验陷阱：Peeking、新奇效应、SRM、多重比较、长期 vs 短期

学会了的标志（自检）

  • 给你一个数据库 schema，你能在 30 分钟内写出 D1/D7/D30 留存查询
  • 你能解释为什么"统计显著 ≠ 业务有意义"
  • 你能识别 A/B 报告里的方法错误（例如样本量不足却宣称显著、SRM 异常）

配套文档：pm_data_ab_testing.html

🟢 弱项 ③ 干系人 / 跨部门影响力 / 谈判 Stakeholder Mgmt & Influence

为什么这是大坑：EM 在团队内部"职权指挥"很顺，但 PgM/PdM 工作 70% 是 "无团队带项目"—— 没人直接汇报给你，你要靠影响力推动。这套技能对很多技术背景的 EM 是真正的"换脑回路"。

必学 4 件事

  • 干系人地图：权力-利益方格、Salience 三圈（合法性/紧迫性/权力）
  • 责任矩阵：RACI（最经典） / DACI（决策密集场景） / RAPID（咨询型）
  • 影响力：Cohen-Bradford "Influence Without Authority" 6 类货币（任务、地位、关系、个人、灵感、组织）
  • 谈判：Harvard 7 要素（Interests/Options/Legitimacy/Alternatives/Communication/Relationship/Commitment） + BATNA / ZOPA / Anchor

学会了的标志（自检）

  • 遇到一个新项目，你能在 30 分钟内画出干系人地图并指出 Top 3 影响关键人
  • 下属抱怨"X 部门不配合"时，你能引导他用 Cohen-Bradford 6 类货币重新设计交换
  • 谈判前，你能写出自己的 BATNA、对方的 BATNA、ZOPA 以及第一锚点

配套文档：pm_stakeholder_influence.html

⑤ 12 周学习路线 · 每周做什么

3 个月每周 ≈ 5~7 小时投入（大约每天 1 小时 + 周末 2 小时）

3 个月每周 ≈ 5~7 小时投入（大约每天 1 小时 + 周末 2 小时）。蓝色 = PgM 主题，粉色 = PdM 主题，绿色 = 弱项攻坚，紫色 = 共通基础。

Week 1
基础读 双线总览（本文）+ 自评能力雷达 + 写一份《我的 PM 转型 Why & 目标 1-pager》。建立笔记仓库（Notion / Obsidian / 你的 tech-notes）。
Week 2
PgMPMBOK 全景：5 大过程组 × 10 大知识领域。读 PMBOK 7 版 + 看《PMBOK Guide 7th》前 3 章。产出：一张 PMBOK 心智图。
Week 3
弱项①WBS + CPM + PERT。挑你过去做过的一个项目，反向画出 WBS 与关键路径，对照 schedule。读 pm_evm_planning_guide.html §1-2。
Week 4
弱项①EVM 7 公式。每天做 2 道 EVM 计算题（书上 / 网上的 PMP 模拟题），用 Excel 把 PV/EV/AC + SPI/CPI 的走势图自己画出来一次。
Week 5
PdMJTBD + Discovery。读《Continuous Discovery Habits》(Teresa Torres)。约 5 个真实用户做访谈（哪怕同事），跑一遍 Mom Test 三规则。
Week 6
PdMPRD 写作。把 W5 用户访谈洞察转成 1 个 Opportunity → 3 个 Solution → 选 1 个写完整 PRD（含 User Story / AC / Edge case）。
Week 7
弱项②指标体系 + SQL。学完 NSM / AARRR / HEART。SQL 刷完 LeetCode 数据库 50 题（前 30 题为主）。读 pm_data_ab_testing.html §1-3。
Week 8
弱项②A/B 测试设计。读 Ronny Kohavi《Trustworthy Online Controlled Experiments》前 5 章。给你的 W6 PRD 设计完整 A/B 实验方案（含样本量计算）。
Week 9
弱项③干系人 + RACI。挑当前一个真实跨部门项目，画干系人地图 + RACI/DACI 矩阵。读 pm_stakeholder_influence.html §1-2。
Week 10
弱项③影响力 + 谈判。读《Influence Without Authority》(Cohen-Bradford)。写 1 篇 1on1 笔记：用 6 类货币重设计一次跨部门交换。看 Harvard PON 谈判教程视频（任意 5 集）。
Week 11
PdM商业 + GTM。画一份 Lean Canvas + Pricing 策略 + Launch Plan。给你 W6 的 PRD 配齐"商业故事"。
Week 12
收口用 pm_interview_cards.html 做模拟面试自测。把过去 11 周所有产出整理成一份 PM 作品集（GitHub Pages / Notion 都行）。重新打雷达图，对比 W1 自评。

路线设计原则（为什么这样安排）

① 先 PgM 再 PdM：PgM 体系性强、考点清晰，给你"快速建立信心"的早期赢面。
② 三大弱项穿插，而不是堆在最后：每补完一个弱项，立刻把它用在你的"项目 / PRD"练习上，否则学了忘。
③ 第 12 周做作品集：找工作 / 转岗时这是你最强的简历筹码，比证书强 10 倍。

⑧ 误区与心法 · EM 转 PM 的 7 个坑

下面 7 条是 EM/Tech Lead 横向打开 PM 能力时最容易翻车的认知陷阱，每条配一句"反向心法"

下面 7 条是 EM/Tech Lead 横向打开 PM 能力时最容易翻车的认知陷阱，每条配一句"反向心法"。

坑 ① 把 PdM 当"需求收集员"

错误以为 PdM 的工作是"把业务方的需求整理给研发"。实际上：

心法：PdM 的核心是决定不做什么，是 trade-off 的最终拍板者。

坑 ② 把 PgM 当"催进度的"

错误以为 PgM 就是开站会、催 owner、画 Gantt。实际上：

心法：PgM 在项目启动当周就要看到 deadline 风险，提前 4 周纠偏，不是事后催。

坑 ③ 用"技术深度"压过"用户语言"

跨部门会议里满嘴 architecture / latency / refactor，用户研究 / 商业语言一上来就发懵。

心法：你过去用技术语言赢，未来用用户语言赢。强迫自己每周写 1 个用户故事。

坑 ④ 用"职权"代替"影响力"

EM 习惯了 "你直属我，我说怎么做"。PgM/PdM 多数项目是无团队的。

心法：把 Cohen-Bradford 的 6 类货币当工具箱，每次推动前问"我能给对方什么"。

坑 ⑤ 学 PMP 当尚方宝剑

以为考完 PMP 就能当项目经理 —— 证只是入场券，做不出项目它就是张纸。

心法：3 个真实交付的项目 > 1 张证书。证可以晚点考甚至不考，作品集要立刻开始攒。

坑 ⑥ 学 Sprint Plan 替代长周期计划

EM/Scrum 出身的人本能地把所有项目都压到 2 周 sprint，跨季度项目就崩了。

心法：Agile 不取消计划，只是把"高度细的计划"推到最后责任时刻。粗粒度的里程碑、关键路径、EVM 还是要做。

坑 ⑦ 数据"看仪表盘 = 数据驱动"

很多 EM 看仪表盘看了 5 年，从来没自己写过一句 SQL，也没设计过 1 个 A/B 实验。

心法：能写 SQL 把数从仓库捞出来，是 PM 的肌肉记忆。每周强迫自己写 3 条复杂 SQL，坚持 10 周。

给自己的 5 句箴言

  1. 先决定不做什么，再决定做什么 —— 优先级是 PM 最重要的产出物，不是写更多 PRD。
  2. 提前 4 周看到坏消息，比当周完美交付更值钱 —— 老板要的是没意外，不是没坏消息。
  3. 没有指标的 PRD 不要写 —— 衡量不到的产出，等于没产出。
  4. 影响力不是说服，是给对方一个赢的理由 —— Cohen-Bradford 6 类货币是工具，不是话术。
  5. 3 个项目 > 3 张证书 —— 边学边做，作品集就是你最好的简历。
