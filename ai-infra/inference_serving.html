01

为什么需要专用推理引擎 · vLLM 比裸跑快 24x

从"能跑"到"能用"中间的 5 个工程鸿沟
🎬生活类比 · 家庭厨房 vs 餐厅厨房
家里炒菜，你一次只能服务一桌，洗菜切菜炒菜全靠一个人，慢悠悠 30 分钟出 1 道菜。

但餐厅厨房不一样 — 同时服务 50 桌。它有 ① 流水线（切菜区、炒菜区、出菜区并行）② 共享原料（很多菜共用一锅高汤）③ 预先备菜（高峰前 prefill）④ 上菜节奏统筹（不让某桌空等）。结果：单菜成本下降 90%，吞吐量提升 30 倍。

vLLM 就是 LLM 的"餐厅厨房"。你用 huggingface transformers 直接跑 model.generate() 就是"家庭厨房" — 能跑但效率极低。vLLM 把"家庭厨房"升级成"餐厅厨房"，throughput 提升 14-24x。
餐厅厨房 → 推理引擎  ·  并行炒菜 → batching  ·  共享高汤 → prefix caching  ·  预先备菜 → prefill 阶段  ·  上菜节奏 → scheduler

vLLM 相对 naive 提速 24x，不是因为算法变了（attention 还是 attention），而是因为工程优化：① 内存不再浪费（PagedAttention）② GPU 不再空闲（Continuous Batching）③ 重复计算被复用（Prefix Caching）。这就是 AI Infra 工程师的价值 — 不写新算法，但能让算法跑出 10x+ 性能。

1.2 朴素推理的"5 个工程鸿沟"

❌ 鸿沟 1 · 内存碎片

每个请求预分配 max_seq_len 大小的 KV-Cache 空间。但 80% 请求只用 30%，剩下 70% 空着也没法给别人 — 显存利用率只有 30-40%。

❌ 鸿沟 2 · 慢的拖累快的

Static batching 把 N 个请求绑一起处理。1 个请求要生成 1000 token，其它 N-1 个只要 10 token — 后者必须空等前者，GPU 大量空闲。

❌ 鸿沟 3 · 重复计算 prompt

很多请求有相同前缀（system prompt / few-shot 示例）。Naive 每次都重新算这段 prefix 的 attention — 纯浪费。

❌ 鸿沟 4 · Prefill 阻塞 Decode

来一个新请求要 prefill（大矩阵乘，几百 ms），这期间其它请求的 decode 全停 — 已有用户体验飞快变慢。

❌ 鸿沟 5 · Decode 浪费算力

Decode 阶段每步只算 1 个 token，GPU 算力大量闲置。能不能"批量生成多个 token"？能（Speculative Decoding）。

vLLM (2023) 一开始主要解决 鸿沟 1 (内存碎片) + 鸿沟 2 (慢的拖累快的)，分别对应 PagedAttention + Continuous Batching。这就是它一战成名的核心创新。后续 vLLM / SGLang / TRT-LLM 又陆续解决了鸿沟 3 / 4 / 5。
02

朴素推理为什么慢 · 拆解到根因

理解瓶颈才能理解 vLLM 的设计选择

2.1 LLM 推理的两阶段 · 一图回顾

回顾 X06 Ch3 的核心结论：LLM 推理分两阶段，瓶颈完全不同。

🔥 Prefill 阶段

  • 处理整个 prompt（n 个 token 一起）
  • n×n 大矩阵乘 — compute-bound
  • GPU 算力打满 ~85%
  • 耗时：~ms 级（n=2k 时 ~200ms）
  • FLOPs ≈ 2 × n × P

🐌 Decode 阶段

  • 每步生成 1 个 token
  • 要读整个 KV-Cache（GB 级）— memory-bound
  • GPU 算力利用率 ~15-25%（带宽瓶颈）
  • 耗时：每 token ~30-50 ms
  • arithmetic intensity 极低

2.2 时间都花在哪了 · perf 拆解

一个典型对话请求（prompt 500 token + 输出 200 token）在 Llama-7B / A100 上的耗时拆解：

阶段占比瓶颈优化方向
Prefill (500 token 一次)~15%computeFlashAttention / FP8
Decode (200 步 × 每步)~80%memory bandwidthGQA / KV 量化 / Spec Decoding
Tokenization / detokenization~2%CPU异步 / 缓存
Sampling (softmax + top-k/p)~3%computefused kernel
Decode 阶段占 80% 时间，这是为什么所有推理引擎优化都主攻 decode。Prefill 只是"热身"，Decode 才是"持久战"。

2.3 KV-Cache 显存的双重压力

KV-Cache（参考 X06 §3.2）一边是 容量（装不下 = 不能服务更多用户），一边是 带宽（读不完 = 每个 token 都慢）。

📏 容量压力

Llama-70B GQA / 32k 上下文 / 单 sequence ≈ 10 GB。A100 80GB 装下 7B 模型（14GB）后只剩 66GB ÷ 10GB = 最多 6 个并发用户。

⚡ 带宽压力

每步 decode 要读完整 KV-Cache。HBM3 带宽 3 TB/s，读 10 GB ≈ 3.3 ms — 这就是单 token latency 下限。增加用户 = 总带宽需求 × N。

最坑的事实：静态预分配 KV-Cache 会浪费 60-70% 显存。请求 A 设了 max_len=2048 实际只用了 200 token — 剩下 1848 个 slot 空着但别的请求不能用。这就是 PagedAttention 要解决的"鸿沟 1"。
03

PagedAttention · 像 OS 那样管理 KV-Cache

vLLM 的核心创新 · 把内存利用率从 30% 提到 96%
🎬生活类比 · 公寓 vs 房间分时租
想象你管理一栋公寓楼，有 100 间房（=100 个 KV-Cache slot）。

朴素方案（连续分配）：来一个长租客（=请求 A）说"我可能住 1 年"，你给他预留连续 10 间房。但他实际只住了 3 间，剩 7 间空着也不能给别人（因为承诺给他了）。一年下来 70% 的房间是空的。

分页方案：来 1 间分 1 间。租客需要扩展时再给他下一间 — 不要求连续。每间房记一个"住户花名册"（block table）。结果：100 间房真的住 96 间，利用率 96%。

这就是 PagedAttention — 借鉴 OS 虚拟内存的分页（paging）思想，把 KV-Cache 从"连续分配"改成"按需分页"。
房间 → KV block (典型 16 token 大)  ·  住户花名册 → block table  ·  公寓楼 → GPU 显存

3.2 借鉴 OS 虚拟内存 · 物理页 vs 逻辑页

PagedAttention 借鉴的就是 OS 虚拟内存的核心思想：逻辑地址连续，但物理地址可以分散。

OS 虚拟内存（对应）

  • 物理页：RAM 上 4KB 大小的页
  • 虚拟页：进程看到的连续地址
  • 页表：虚拟页 → 物理页 映射
  • Page Fault：访问未映射页，OS 分配

PagedAttention（类比）

  • 物理 KV block：显存上 16 token 的连续 KV 槽
  • 逻辑 sequence：请求看到的连续 token
  • Block Table：逻辑 → 物理 block 映射
  • Allocate on demand：需要新 token 时分配新 block

3.3 Block Table · 实现核心

每个 sequence 都有一个 block table，记录它的逻辑 block 0、1、2... 对应哪些物理 block。

# 假设 block_size = 16 (每 block 装 16 token 的 KV)
# Sequence A 现在有 50 token, 需要 ceil(50/16) = 4 block

block_table_A = [
    42,   # 逻辑 block 0 (token 0-15) → 物理 block #42
    17,   # 逻辑 block 1 (token 16-31) → 物理 block #17
    8,    # 逻辑 block 2 (token 32-47) → 物理 block #8
    31,   # 逻辑 block 3 (token 48-49, 半满) → 物理 block #31
]

# 计算 attention 时, kernel 通过 block_table 找到所有物理位置
def paged_attention(query, block_table, kv_cache, seq_len):
    for i in range(len(block_table)):
        phys_block_id = block_table[i]
        kv_block = kv_cache[phys_block_id]   # 物理位置取出 KV
        # 在 query 跟这个 block 上做 attention
        ...

关键设计决策

📐 Block size = 16

不能太小（block table 元数据太多）也不能太大（碎片回来了）。vLLM 默认 16，是实验最优。

🔀 物理 block 池统一管理

所有 sequence 共享一个 free block pool。请求来了从 pool 拿，结束了还回去。跟 OS 物理页管理一模一样。

📦 自定义 CUDA kernel

普通 attention kernel 假设 KV 是连续内存。vLLM 写了专用 PagedAttention kernel，按 block_table 跳着读 KV，是项目核心 C++/CUDA 代码。

3.4 Prefix Caching · 物理 block 共享

有了 block table，多个请求可以共享同一个物理 block。这就是 prefix caching 的基础。

🎬生活类比 · 多个学生共用同一份讲义
想象 100 个学生都在写 ChatGPT 的代码作业，每个学生的 prompt 前 200 字都是相同的 system prompt（"你是助手 / 请按代码块返回..."）。

朴素：每个学生都自己抄一份这 200 字 → 100 份重复内容占内存。
Prefix Caching：把这 200 字的 KV 算一次存起来，所有学生的 block table 前几项都指向同一个物理 block。100 个并发用户的 prefix 只算 1 次！

vLLM Automatic Prefix Caching（2024 加入）就是这么做的。在 chatbot / RAG 等 prompt 高度重复的场景，throughput 再 +30-50%。
面试高频追问："如果两个请求 prompt 前缀相同但后面不同，PagedAttention 怎么处理？" 答：① 检测前缀 hash 相同 ② 它们的 block table 前面几项指向同一物理 block（引用计数 +1）③ 第一个开始"分叉"的位置，给新请求分配新 block，old block 引用计数 -1 但还活着。
04

Continuous Batching · 流水线作业

解决"慢的拖累快的"问题 · throughput 再翻倍
🎬生活类比 · 顺丰快递站的流水线
想象 1 个快递站每天要分拣 1000 个包裹。

Static Batching（旧方案）：把包裹凑成 10 个一批，必须等这 10 个全部分拣完，才能进下一批。如果某个包裹特别复杂（地址难找），其它 9 个都得等。

Continuous Batching（vLLM 方案）：分拣台是流水线，分拣完的包裹立刻离开，空出的位置马上塞新的包裹进来。任何时刻总有 10 个包裹在分拣，但成员一直在变。没有"快的等慢的"，效率翻倍。
快递站 → GPU  ·  分拣台 → batch slot  ·  包裹 → request  ·  复杂包裹 → 长输出请求

4.2 调度器（Scheduler）核心算法

Continuous Batching 的核心是每个 decode step都重新决定"哪些请求进 batch"。算法大概是这样：

def schedule_step(running_seqs, waiting_seqs, free_blocks):
    # Step 1: 检查 running 里有谁完成了 (输出 EOS 或达到 max_len)
    for seq in running_seqs:
        if seq.finished:
            release_blocks(seq.block_table)   # 释放它的 KV block
            running_seqs.remove(seq)

    # Step 2: 检查 running 里谁需要新 block (token 数 % 16 == 0)
    for seq in running_seqs:
        if seq.needs_new_block():
            if free_blocks.empty:
                # 显存满了, 把最低优先级的 seq swap 到 CPU memory
                evict_to_cpu(lowest_priority(running_seqs))
            allocate_block(seq)

    # Step 3: 从 waiting 队列拉新请求进 batch (满足资源时)
    while free_blocks.count() >= waiting_seqs[0].prefill_blocks_needed:
        new_seq = waiting_seqs.pop(0)
        prefill(new_seq)                       # prefill 它的 prompt
        running_seqs.append(new_seq)

    # Step 4: 对所有 running seq 跑一步 decode
    decode_one_step(running_seqs)
关键点 1：scheduler 每个 step 都跑一次（每 ~30ms），不是固定 batch。关键点 2：完成的请求立刻空出 slot 让新请求进来。关键点 3：显存不够时swap 到 CPU（保住已开始的请求，不会丢）。

4.3 Continuous Batching 的边界

✅ 适合

  • 在线 chat / chatbot（请求长度参差不齐）
  • 大量并发用户（> 10）
  • 用户对 throughput 敏感（成本/月）

⚠️ 不太适合

  • 单用户大文档（batch=1 优势消失）
  • 实时高优先级（仍可能被调度延迟）
  • 极端延迟敏感（< 100ms）

Prefill-Decode 分离 · 2 个厨房比 1 个好

SGLang / TensorRT-LLM 2024 新趋势 · 解决 "prefill 阻塞 decode" 问题
🎬生活类比 · 餐厅"备菜区"和"出菜区"分离
餐厅高峰时段，新桌点完单 → 需要洗菜切菜（=prefill），耗时 ~5 分钟。在这 5 分钟里，主厨没法分心去给已上桌的"加水加饭"（=decode）。

聪明餐厅的做法：开两个厨房。一个专门"备菜"（prefill 厨房），一个专门"出菜+加水"（decode 厨房）。两边互不影响，新桌不影响老桌的体验。

Prefill-Decode 分离 = 这个思想直接搬到 GPU：用一组 GPU 专门做 prefill，另一组 GPU 专门做 decode。KV-Cache 通过高速网络传输。

5.1 为什么需要分离

回想 ch2 的结论：Prefill 是 compute-bound（吃算力），Decode 是 memory-bound（吃带宽）。它们对硬件的需求完全不同。

阶段瓶颈理想硬件
Prefill算力（FLOPS）计算密度高的 GPU（H100, B200）/ TPU
Decode显存带宽大显存 + 高带宽 GPU（A100 80GB 也够）

朴素方案是同一组 GPU 干两件事，造成：

  • Prefill 时算力打满，但带宽闲置
  • Decode 时带宽打满，但算力闲置 80%
  • 更糟：prefill 阶段的大矩阵乘独占 GPU，已有 decode 全部停顿（TTFT 飞涨）

5.2 分离架构

典型架构（SGLang / TRT-LLM / vLLM Disaggregated）

用户请求
   ↓
[Load Balancer]
   ↓
[Prefill Worker Pool]    ← H100 × N 专门做 prefill
   ↓ KV-Cache via RDMA / NVLink
[Decode Worker Pool]     ← A100 × M 专门做 decode
   ↓
[Streaming Response]
   ↓
用户
  • KV-Cache 跨节点传输：Prefill 完成后，把 KV-Cache 通过 RDMA / NVLink 传到 Decode worker。代价 ~50ms（视模型大小）
  • 资源解耦：可以独立扩容（高峰期开多 Prefill worker，平峰期回收）
  • SLO 隔离：新请求的 prefill 不影响老请求的 decode latency
2024-2026 趋势：所有头部推理引擎都在做 prefill-decode 分离。代表项目：vLLM Disaggregated / SGLang / DistServe (paper) / TensorRT-LLM disagg。面试可以提到这是"2024 最重要的推理优化方向之一"。
06

Speculative Decoding · 用小模型猜大模型

把 decode 从"一次 1 token"变成"一次 K token"
🎬生活类比 · 实习生先写草稿，主管批量校对
公司里有个明星顾问（=大模型，70B），每咨询一个问题要 30 分钟，但答案权威。每次客户只能问 1 个问题。

聪明做法：雇个实习生（=小模型，1B），他每问回答只要 1 分钟，但答案常出错。实习生先批量回答 5 个问题，明星顾问一口气审核 5 个答案（也是 30 分钟，但同时审 5 个）。如果实习生答对了 4 个，客户净赚 4 倍效率。

这就是 Speculative Decoding：用小模型生成 K 个候选 token，大模型并行验证（用 K+1 次 forward 验证 K 个 token），验证通过的直接用。

6.1 算法核心

def speculative_decode(big_model, small_model, prefix):
    # Step 1: 小模型快速生成 K 个候选 token
    candidates = []
    cur = prefix
    for i in range(K):
        next_tok = small_model.forward(cur).argmax()
        candidates.append(next_tok)
        cur = cur + [next_tok]

    # Step 2: 大模型一次性并行验证这 K 个候选
    # (输入: prefix + K 个候选, 输出: K+1 个 logits)
    big_logits = big_model.forward(prefix + candidates)

    # Step 3: 从左到右接受 / 拒绝
    accepted = []
    for i in range(K):
        big_top = big_logits[i].argmax()
        if big_top == candidates[i]:
            accepted.append(candidates[i])
        else:
            # 第 i 个不匹配, 用大模型的输出, 后面全丢
            accepted.append(big_top)
            break

    return accepted
关键：大模型一次 forward 同时计算 K+1 个位置的 logits（attention 是天然并行的，多算 K 个 token 几乎免费）。所以 1 次大模型 forward = 最多接受 K 个 token，等价于 K 倍加速（如果接受率高）。

6.2 接受率决定加速比

场景典型接受率实际加速
简单文本生成（小模型质量近大模型）70-80%2-3x
代码生成（结构化）60-75%2-2.5x
复杂推理（reasoning model）40-50%1.5-1.8x
多语言切换30-40%1.2-1.5x

6.3 演进 · 各种 Speculative 变体

🎯 Vanilla Spec Decoding

一个独立的小模型猜。简单但小模型要训练。

🦞 Medusa (2024)

在大模型最后一层加几个 "medusa head"，多头并行预测下 K 个 token，不需要独立小模型。

🦅 EAGLE / Lookahead

用大模型自己的早期 layer 当 "draft model"。更高接受率（80%+），2024 SOTA。

vLLM / SGLang / TRT-LLM 都已支持 Spec Decoding。面试常问"它跟普通 batching 哪个更好" — 答：互补，低并发用 Spec Decoding（降单 token latency），高并发用 Continuous Batching（提 throughput）。

维度 vLLM SGLang TensorRT-LLM

License Apache 2.0 Apache 2.0 商业 + 部分开源

选型决策

👶 初学 / POC

vLLM。Apache 开源、模型支持最广、API 跟 OpenAI 兼容（前端代码改 base_url 就能切）。

🎯 Agent / 多轮

SGLang。RadixAttention 对 Agent 场景（大量共享 prefix）有 2-3x 优势。

🏭 大规模生产

TRT-LLM。如果你有 H100 集群 + 愿意花精力做模型编译，能榨干硬件最后 10-20% 性能。

SGLang vs vLLM 内部差异：SGLang 用 RadixAttention（基数树）做 prefix 共享，比 vLLM Automatic Prefix Caching（hash 表 + reference count）更通用 — 不仅 prefix 共享，任意 sub-tree 都能共享。这就是为什么 SGLang 在 Agent 场景（多轮对话有大量重复 sub-prompt）显著胜出。

vLLM 源码导览 · 3 个核心文件

想深入面试就读这 3 个 · 别盲读整个 repo

vLLM repo 有几百个文件，盲读会迷路。下面是 3 个"读完能撑过 Senior 面试"的核心文件，按优先级：

vllm/core/scheduler.py

① 调度器 · Continuous Batching 的灵魂

它做什么：每个推理 step 决定哪些 sequence 进 batch、哪些被 evict、哪些 prefill / decode 优先。

关键函数：Scheduler._schedule_running()、_schedule_prefills()、_schedule_default()。读这 3 个函数（共 ~300 行）就理解了核心调度逻辑。

理解 "preemption（抢占）"机制 — 显存满时怎么决定 evict 哪个 sequence。面试经常问。
vllm/worker/model_runner.py

② Model Runner · 真正跑 forward 的地方

它做什么：拿 scheduler 给的 batched sequences，准备 input tensor、调 model 的 forward、采样。是 Python 跟 CUDA kernel 的桥梁。

关键函数：execute_model()、_prepare_model_input()。注意它怎么把不同长度的 sequence packed 进同一个 batch（这是 Continuous Batching 的实现细节）。

理解 "model_input.input_tokens" 跟 "model_input.input_positions" 的分布 — 这是 packed batch 的核心结构。
csrc/attention/paged_attention_v2.cu

③ PagedAttention CUDA Kernel · 性能命门

它做什么：根据 block_table 跳着读 KV 做 attention。vLLM 性能的根基。

读法建议：先读注释和函数签名理解参数，然后看 paged_attention_v2_kernel 主循环。不需要每行都懂，理解 "为什么用 shared memory" + "block_table 怎么用" 就行。

这是 C++ 老兵转 AI Infra 的最大杠杆 — 你已有的 CUDA / 并行思维直接复用。在简历写"读过 vLLM PagedAttention kernel"是强信号。

9.1 阅读顺序建议

  1. Day 1：跑通 ch8 实战，知道 vLLM 怎么用
  2. Day 2：读 scheduler.py，画一遍调度流程图
  3. Day 3：读 model_runner.py，跟踪一个 request 走完全程
  4. Day 4-5：读 paged_attention_v2.cu，理解 CUDA kernel
  5. Day 6-7：选一个 "good first issue" 提 PR

生产环境优化清单 · 10 个实战 tips

线上跑过才知道的坑
#优化预期收益难度
1开 prefix caching (--enable-prefix-caching)+30-50% throughput（chat 场景）⭐ 低
2调 max-num-seqs（典型 64-256）+20-50% throughput⭐ 低
3启用 FP8 / AWQ 量化+50-100% throughput，质量略降⭐⭐ 中
4启用 Speculative Decoding+1.5-3x latency 降低⭐⭐ 中
5Tensor Parallel 切分（70B 用 2-4 卡）显存够 + 提升 throughput⭐⭐ 中
6调高 gpu-memory-utilization 到 0.92-0.95+10-20% throughput⭐ 低
7Prefill-Decode 分离（多机部署）P95 latency -40%（隔离影响）⭐⭐⭐ 高
8CPU offload + swap（显存不够时）支持更多并发，单 token latency +10-20%⭐⭐ 中
9Multi-LoRA serving（共享 base，每个用户独立 LoRA）1 套 GPU 服务 N 个微调模型⭐⭐⭐ 高
10从 vLLM 切到 SGLang（Agent 场景）+50-200% throughput（高 prefix 共享）⭐⭐ 中
避免的 5 个常见坑：
  1. 把 gpu-memory-utilization 设到 0.99 — 偶发 OOM 杀死整个服务
  2. 开 prefix caching 但 prompt 多变 — 没有共享，反而增加内存碎片
  3. Spec Decoding draft 模型选太大 — 验证比直接 decode 还慢
  4. Tensor Parallel 切到不该切的小模型 — 通信开销大于切分收益
  5. 不监控 GPU 利用率 — 不知道瓶颈在哪就盲调参

15 道 AI Infra 高频面试题

折叠卡片 · 先想再展开 · 招聘官真在考
基础Q1. vLLM 为什么比 huggingface naive 推理快 10x+？
核心
3 个工程优化：① PagedAttention 把显存利用率从 30% 提到 96% → 能装更多并发 ② Continuous Batching 让快请求不等慢请求 → GPU 利用率 25% → 80% ③ 自定义 CUDA kernel 提升单步性能。算法没变，只是工程优化。
基础Q2. PagedAttention 跟 OS 虚拟内存的相似性？
对应
① OS 虚拟页 ↔ 逻辑 KV block / OS 物理页 ↔ 物理 KV block / OS 页表 ↔ block table ② 都解决"地址连续但物理分散"问题 ③ 都用"按需分配 + 共享物理块"提升利用率。
差异
OS 页 4KB / vLLM block 16 token (KB 级)；OS 由 MMU 硬件加速 / vLLM 由专用 CUDA kernel 实现。
进阶Q3. Continuous Batching 跟 Static Batching 区别？什么时候用 Static？
区别
Static 把 N 个请求绑一起进 batch，必须等最慢的；Continuous 每个 step 都重新调度，完成的立刻空出 slot。
什么时候 Static
① 离线 batch 推理（请求长度差不多）② 极端 throughput 优先（GPU 利用率最大化）③ 简单实现。在线 chat 永远用 Continuous。
进阶Q4. 解释 prefix caching 工作原理 + 什么场景效果好？
原理
同一个 prefix 的 KV 算一次存物理 block，多个请求的 block_table 前几项指向同一物理 block（引用计数）。哪个请求开始"分叉"就分配新 block。
最适合场景
① Chat (system prompt 重复) ② Few-shot prompting (示例重复) ③ Agent multi-turn (历史对话重复) ④ RAG (retrieved chunk 重复)。SGLang RadixAttention 对此优化最深。
高频Q5. 为什么 LLM 推理 decode 阶段是 memory-bound？怎么优化？
为什么
每步 decode 只算 1 个 token (FLOPs ≈ 2P)，但要读整个 KV-Cache (GB 级)。arithmetic intensity = FLOPs / Bytes = 极低 → memory-bound。
优化方向
① 减小 KV-Cache：GQA / MLA / KV 量化 ② 提高并发：Continuous Batching 摊薄 KV 读取 ③ 减少 step 数：Speculative Decoding ④ Hide latency：Prefill-Decode 分离
高频Q6. Speculative Decoding 怎么工作？什么场景效果差？
原理
小模型并行生成 K 个候选 token，大模型一次 forward 同时验证 K+1 个位置 (attention 天然并行)。接受率高的话 → 1 次 forward 出 K 个 token。
差场景
① 推理任务 (reasoning model) — 小模型猜不对 ② 多语言切换 — 小模型语言能力差 ③ 高熵任务 (创意写作) — 没有"正确答案"。接受率 < 50% 时 spec decoding 反而变慢。
进阶Q7. Tensor Parallel vs Pipeline Parallel 在推理时的区别？
TP
把 weight 矩阵切到多卡，每层都需要 AllReduce 通信。适合单层大、卡间高速互联 (NVLink)。
PP
把不同 layer 放不同卡，通信只在 layer 边界。适合卡间慢互联 (PCIe)，但 latency 会增加 (流水线 bubble)。
推理实战
单机多卡几乎都用 TP（NVLink 够快）；跨机才用 PP。70B 模型典型 TP=2 or 4。
基础Q8. KV-Cache 量化 INT4 / FP8 的 trade-off？
收益
KV-Cache 显存 ÷ 4 (INT4) 或 ÷ 2 (FP8) → 能装更多并发 / 更长上下文 / 更大 batch → throughput +50-100%。
代价
精度损失：INT4 KV 一般 -1-2% 准确率，FP8 几乎无损。越长上下文 INT4 累积误差越大。
进阶Q9. vLLM 的 preemption 机制是什么？怎么决定 evict 谁？
触发
显存满，新 token 无法分配新 block 时触发。
策略
① Recompute: 丢掉某个 sequence 的所有 KV-Cache（释放最多 block），稍后从头 prefill（适合 prompt 短的）② Swap: 把 KV-Cache 拷到 CPU memory，需要时拷回来（适合 prompt 长的）。vLLM 默认 Recompute。
选谁
按到达时间 FIFO（最晚到的最先被牺牲），或按用户优先级。
高频Q10. vLLM vs SGLang 在 Agent 场景为什么 SGLang 胜出？
SGLang 优势
RadixAttention 用基数树（radix tree）做 prefix 共享 — 不只是前缀共享，任意 sub-tree 都能共享。Agent multi-turn 对话有大量重复 sub-prompt（system + 历史 + 工具描述），radix tree 命中率 70%+。
vLLM 局限
vLLM Automatic Prefix Caching 用 hash 表，只看完整 prefix 命中。Agent 场景 hash 命中率低很多。
进阶Q11. Prefill-Decode 分离架构有什么 trade-off？
收益
① 资源解耦（独立扩容）② SLO 隔离（新请求不影响老请求 latency）③ 硬件匹配（H100 做 prefill / A100 做 decode）④ 整体吞吐 +30-50%。
代价
① KV-Cache 跨节点传输延迟（RDMA 也要 30-50ms）② 部署复杂度上升（需要 LB + 路由）③ 故障域扩大 ④ 适合大规模，小规模反而 overhead 大。
基础Q12. 给定一个 7B 模型 / A100 80GB / 4k 上下文，估算最大并发？
计算
① 模型权重 FP16 = 14 GB ② Activation buffer ≈ 2 GB ③ 剩余显存 = 80 - 14 - 2 = 64 GB ④ KV-Cache per seq (Llama-7B GQA H=8): 2 × 4k × 32 × 8 × 128 × 2 = 0.5 GB ⑤ 最大并发 ≈ 64 / 0.5 = 128 个 sequence。
加分
提到 PagedAttention 实际能装更多（减少碎片），FP8 量化 KV 能翻倍到 256。
进阶Q13. Multi-LoRA serving 是什么？怎么实现？
需求
1 个 base model 给 100 个用户服务，每个用户有自己的 LoRA 微调权重（小，~50 MB）。不能为每个用户单独起一个推理服务（GPU 太贵）。
实现
① base 模型加载一次到 GPU ② 所有 LoRA weights 全在 CPU memory ③ 请求来了根据 user_id 把对应 LoRA swap 到 GPU 临时合并 ④ 算完后释放。vLLM / SGLang 都已支持。
高频Q14. 如果生产 vLLM serving 突然 latency 飙升怎么排查？
排查步骤
① 看 metrics：vLLM 暴露 prometheus metrics（pending requests / running batch size / GPU util / cache hit rate）
② 检查 batch size：突然变小 → 请求太长 / KV-Cache 满了 / preemption 频繁
③ 检查 prefill 占比：突然来了一波长 prompt → prefill 阻塞 decode
④ 检查 GPU util：低 → memory-bound 但 cache 大；高 → compute-bound（应该是 prefill 风暴）
⑤ 看上游：是不是请求模式变了（prompt 突然变长）
实战Q15. 你怎么把一个朴素 model.generate() 服务升级到 vLLM？migration 步骤？
步骤
① 评估：测当前 throughput 和 latency 做 baseline
② 本地 POC：起 vLLM 本地服务，跑 ch8 benchmark，确认性能
③ 客户端改造：vLLM 提供 OpenAI 兼容 API，客户端只改 base_url，代码几乎不动
④ 灰度：5% 流量切到 vLLM，监控 quality / latency / cost
⑤ 调参：根据线上 prompt 分布调 max-num-seqs / prefix-caching / 量化
⑥ 全量：100% 切流，回收旧服务 GPU
回滚预案
留下原服务备份；vLLM API 兼容 OpenAI，回滚只需要切 base_url。Quality 监控异常立即回滚。
