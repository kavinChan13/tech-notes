为什么单卡训不动大模型 · 显存账本

🎬生活类比 · 一个人装不下整个图书馆
训练 70B 模型，不是 1 张 H100 80GB 装不下"模型"（140 GB 装不下），而是"训练状态"远比模型本身大。

一个 70B 模型完整训练需要：
① 模型 weight（FP16）：140 GB
② 梯度（同 weight）：140 GB
③ Optimizer state（Adam: m + v，FP32）：560 GB（4x weight）
④ Activation（中间激活值）：100-500 GB（取决于 batch size 和 seq len）

合计 1-1.5 TB，至少要 15-20 张 H100。这就是为什么训 LLM 必须分布式。

1.1 显存账本（70B 模型示例）

组件大小占模型 X 倍
Weight (FP16)140 GB1x
Gradient (FP16)140 GB1x
Adam state m (FP32)280 GB2x
Adam state v (FP32)280 GB2x
Activation (随 batch / seq)100-500 GB1-4x
合计~1-1.4 TB7-10x weight
显存才是瓶颈，不是算力。即使你愿意训得久，单卡装不下就是装不下。这也是为什么大公司囤 GPU 显存大小（H100 80GB / B200 192GB）远比算力关键。
02

3 维并行 · DP / TP / PP 全景

3 种切法 · 解决 3 种不同瓶颈
🎬生活类比 · 工厂流水线的 3 种切法
一家工厂要生产 1000 个 iPhone，怎么用 4 个车间分？

Data Parallel (DP) = 4 个车间各做 250 部。每个车间设备齐全（=完整模型副本），分一份订单（=数据切片）。最简单。

Tensor Parallel (TP) = 4 个车间分工做"组装一部"。车间 1 装屏幕，车间 2 装芯片，车间 3 装电池，车间 4 装外壳。所有车间一起完成 1 部 iPhone。

Pipeline Parallel (PP) = 4 个车间是流水线。车间 1 出半成品 → 车间 2 加工 → 车间 3 → 车间 4 出成品。每个车间只做一道工序。

千卡训练 = 3 种同时用（3D Parallel）。比如 1024 卡：DP=8 × TP=8 × PP=16 = 1024。

2.1 3 维并行对比表

维度切什么通信量适合典型规模
DP（Data Parallel）不同 GPU 跑不同数据中（梯度 AllReduce）模型能装单卡，加速 throughput2-512 卡
TP（Tensor Parallel）同一层的矩阵切到多卡高（每层 forward/backward 都通信）单层都装不下时2-8 卡（NVLink 内）
PP（Pipeline Parallel）不同层放不同卡低（只在层边界通信）跨机训练 / 极大模型2-64 卡
+ EP (Expert Parallel)MoE expert 切到不同卡高（每 token 路由）MoE 模型必需取决 expert 数
组合规则：TP 内（NVLink），PP 跨机（PCIe/IB），DP 外层。例：32 卡训练，2 节点 × 16 卡，TP=8（节点内 NVLink）/ PP=2（跨节点）/ DP=2（数据切片）。

3.1 DDP 工作流

  1. 初始化：每张 GPU 加载相同模型 weight
  2. Forward：每张 GPU 跑自己的 1/N 数据，得到 loss
  3. Backward：每张 GPU 算自己数据的梯度
  4. AllReduce 同步：所有 GPU 的梯度求平均（通过 NCCL）
  5. Optimizer step：每张 GPU 用同样的平均梯度更新（结果一致）
# PyTorch DDP 代码
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

dist.init_process_group(backend="nccl")        # 初始化 NCCL
torch.cuda.set_device(local_rank)

model = MyModel().cuda(local_rank)
model = DDP(model, device_ids=[local_rank])

# 后续训练跟单卡一样，DDP 自动处理梯度同步
for batch in loader:
    loss = model(batch).backward()       # backward 自动触发 AllReduce
    optimizer.step()

3.2 DDP 的局限

DDP 的核心假设：每张卡能装下完整模型 + 训练状态。对 1B-7B 模型有效，70B 立即失效（每张卡需要 ~1 TB，不可能）。70B+ 模型必须用 FSDP / ZeRO。
04

ZeRO / FSDP · 显存优化王者

让 70B 模型也能"数据并行"
🎬生活类比 · N 个工人共享一套工具箱
DDP 的浪费：4 个工人每人都背着完整一套工具箱（120 公斤），其实同时用的工具只有 5%。

ZeRO 的思路：4 个工人每人只背 1/4 工具。需要时临时从其它工人那里借（gather），用完还回去。每人负重从 120 → 30 公斤，但每次干活前要"借一下"（通信开销）。

这就是 ZeRO（DeepSpeed）/ FSDP（PyTorch 原生）的核心。

4.1 ZeRO 3 阶段

ZeRO-0 (DDP)
什么都不分。每张 GPU 装完整 weight + grad + optimizer。
最简单，但显存占用 = 完整 7-10x model size。
无节省
ZeRO-1
Optimizer state 分到 N 卡。其它仍复制。
Adam optimizer state 占大头（4x model size），分掉后省一大半。
显存 ÷ 4
ZeRO-2
Optimizer state + Gradient 都分。
Forward 后梯度立刻 Reduce-Scatter（部分聚合后分散到各卡）。
显存 ÷ 8
ZeRO-3 (FSDP)
weight + grad + opt-state 全部分。
每张卡只存 1/N 的 weight，需要时 AllGather 临时取完整 weight 算 forward。
显存 ÷ N

4.3 FSDP · PyTorch 原生

FSDP 等于 PyTorch 原生的 ZeRO-3。2023+ 推荐用 FSDP 而不是 DeepSpeed ZeRO，因为：

  • 更好集成 PyTorch eager / compile
  • 更灵活的 wrapping 策略
  • 2024+ FSDP2 增加 per-parameter sharding
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP

model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,   # ZeRO-3 风格
    auto_wrap_policy=transformer_wrap_policy,         # 按 transformer block wrap
    mixed_precision=MixedPrecision(param_dtype=torch.bfloat16),
    cpu_offload=CPUOffload(offload_params=False),   # 显存不够可 offload 到 CPU
)
选型决策：
  • 模型 < 7B / 单卡装得下 → DDP（最快）
  • 模型 7B-70B / 单卡装不下 weight + grad → FSDP / ZeRO-3
  • 模型 100B+ / 单层都装不下 → FSDP + TP 组合
  • 模型 671B (DeepSeek-V3) → 3D Parallel + EP（见 ch8）
05

Tensor Parallel · 切矩阵

单层都装不下时
🎬生活类比 · 4 个人一起搬一个大桌子
一个超大矩阵乘 (10240 × 10240 → 10240 × 10240) 单卡显存装不下中间结果。
TP 思路：把这个矩阵按列切成 4 块（每块 10240 × 2560），4 张卡每张算一块的 GEMM，最后 concat 结果。

5.1 TP 在 Transformer 里怎么切

Megatron-LM 的经典做法（2020 论文）：

  • FFN：第一层 W1 按列切 (column-parallel) / 第二层 W2 按行切 (row-parallel)，每层只需 1 次 AllReduce
  • Attention：把多个 head 切到不同卡 (head-parallel)，每层 1 次 AllReduce
  • Embedding：vocab 维度切
TP 的成本：每层 forward 和 backward 都需要 AllReduce。必须用高速互联（NVLink，不能 PCIe）。所以 TP 一般 ≤ 8（节点内 NVLink），跨节点用 PP。

5.2 TP 限制

TP 通信极频繁（每层 2 次 AllReduce），跨节点 TP 会变成通信瓶颈，速度比 PP 慢。所以 70B 模型典型用：TP=8（节点内 NVLink） + PP=2（节点间） + DP/FSDP 外层。
06

Pipeline Parallel · 切层

跨节点训练的关键
🎬生活类比 · 装配线 + 流水节奏
80 层 Transformer 切 4 段：卡 0 跑 layer 1-20，卡 1 跑 21-40，卡 2 跑 41-60，卡 3 跑 61-80。

问题：朴素流水线下，同一时间只有 1 张卡在工作（其它在等）。GPU 利用率 25%。

解决：把一个 batch 切成 N 个 micro-batch，像流水线一样让 4 张卡同时干不同 micro-batch。GPU 利用率涨到 80%+。

6.1 经典调度策略

算法GPU 利用率显存
Naive PP~25%低
GPipe（micro-batch）~70%高（buffer 所有 activation）
1F1B (PipeDream)~85%中（边 forward 边 backward）
Interleaved 1F1B (Megatron)~90%中
2026 主流：Megatron-LM 的 Interleaved 1F1B。面试提到 "1F1B 调度"加分。

6.2 Pipeline Bubble · 不可避免的 idle 时间

无论怎么调度，流水线开始和结束阶段必有 idle（warm-up 和 cool-down）。Bubble ratio ≈ (PP - 1) / (PP - 1 + num_microbatch)。micro-batch 越多 bubble 越小。

07

Expert Parallel (MoE) · 把专家切到不同卡

MoE 模型有几十-几百个 expert，所有 expert 都要装显存但每 token 只用 k 个。EP 把不同 expert 放不同卡。

🎬生活类比 · 部门按专业分布在不同楼层
公司有 256 个专家，每个问题只问 8 个。不可能 1 个办公室装 256 人。
EP 解：256 个专家分到 32 层楼，每层 8 个。问题来时判断该问哪 8 个，路由到对应楼层。

代价：每层 forward 都要做 All-to-All 通信（把 token 发到对应 expert 所在卡），通信量极大。

7.1 MoE 训练的 4 个挑战

  1. All-to-All 通信瓶颈：每层 2 次（dispatch + combine），跨节点慢
  2. Load imbalance：router 总选热门 expert → 部分卡忙死部分闲死
  3. Activation 显存：dispatch 后 activation 重排，临时 buffer 大
  4. Dead expert：训练后期某些 expert 几乎不被选 → 浪费参数

3D Parallel · 万卡训练真相

DP × TP × PP 三维组合

典型 1024 卡训练 = DP=8 × TP=8 × PP=16

每个绿色块代表一个 (TP × PP) 模型并行单元（8 × 16 = 128 卡），
8 个这样的单元组成 DP（每个单元处理 1/8 数据，梯度同步）

规模典型配置说明

8.2 通信带宽要求

🔵 节点内（NVLink）

~900 GB/s（H100 NVLink 4）。TP / FSDP shard 用这个。

🟣 节点间（InfiniBand）

~50-400 GB/s（IB HDR/NDR）。PP / DP / EP 用这个。

🟠 跨机房

~10-100 Gb/s。训练几乎不可用，只能离线 sync。

NCCL · 集体通信库

分布式训练的"网线"

NCCL (NVIDIA Collective Communications Library) 是分布式训练所有通信的底层。面试常问 NCCL 的原语。

9.1 5 个核心原语

原语做什么用在哪
AllReduce所有卡的 tensor 求和后广播给所有卡DP 梯度同步
ReduceScatter求和但分散到各卡（每卡得 1/N）ZeRO-2 梯度分发
AllGather所有卡的 shard 聚合到完整 tensor，所有卡都拿到FSDP forward 取 weight
All-to-All所有卡互相交换数据MoE expert dispatch
Broadcast1 张卡的 tensor 广播给所有卡初始化时同步 weight

9.2 Ring AllReduce · 经典算法

1
N 张卡组成 ring，每张持有 N/N 的数据
2
Reduce-Scatter 阶段：N-1 步，每步每张卡传 1/N 数据给下家，并和收到的累加
3
AllGather 阶段：N-1 步，每步广播自己的 partial sum
→
总通信量 = 2 × (N-1)/N × T ≈ 2T（T = 数据大小），不随 N 增长，是带宽最优算法

5 个真实千卡训练事故

已脱敏 · 学他人血泪

事故 1 · NCCL hang 24 小时

千卡训练第 2 天 NCCL AllReduce 卡了 24 小时无响应，整集群 idle。

根因：某 GPU 慢慢 ECC 错误，让所有其它 GPU 等它。NCCL 默认无超时。修复：设 NCCL_TIMEOUT 30s + healthcheck 监控 ECC + auto failover。

事故 2 · Loss spike 后训练崩

训到 50k step loss 突然飙升到 NaN，无法恢复。

根因：BF16 训练中某 batch 数据极端 → activation overflow → NaN 传播。修复：梯度 clipping + 数据预过滤 + checkpoint 回滚到 stable。

事故 3 · MoE expert load imbalance

MoE 训练后期发现某些 expert 几乎不被选（活跃度 1%），其它 expert 过载。

根因：router 偏好稳定后强者更强。修复：加大 load balancing loss + auxiliary loss。

事故 4 · Pipeline bubble 大让 GPU 利用率只有 40%

PP=16，但训练 GPU util 只有 40%。期望 80%+。

根因：micro-batch 数量太少（=PP），bubble ratio 50%。修复：micro-batch 数 ≥ 4 × PP，调到 64。util 涨到 85%。

事故 5 · Activation checkpointing 没开训到 OOM

70B 模型 8 卡 FSDP，跑几百 step 后 OOM。

根因：activation 显存没考虑。修复：开 activation checkpointing（forward 时不存中间 activation，backward 时重算），显存 -50% 代价 +30% 计算。

12 道分布式训练高频面试题

基础Q1. 训 70B 模型需要的显存？为什么是 weight 的 7-10x？
账本
weight (1x) + grad (1x) + Adam state (4x，FP32 m+v) + activation (1-4x) = 7-10x weight。70B FP16 = 1-1.4 TB，至少 15-20 张 H100。
基础Q2. DP / TP / PP 区别？什么时候用哪个？
DP
不同 GPU 跑不同数据，每张完整模型。模型装下单卡才用。
TP
同一层矩阵切到多卡。通信极频繁，需要 NVLink，限节点内。
PP
不同层放不同卡。通信少（只在边界），适合跨节点。但有 bubble。
高频Q3. ZeRO 3 个 stage 区别？
3 阶段
① ZeRO-1：optimizer state 分 ② ZeRO-2：optimizer + gradient 分 ③ ZeRO-3 (FSDP)：weight + grad + opt 全分
权衡
越高 stage 显存越省，但通信开销越大（ZeRO-3 每层 forward 都要 AllGather weight）。
进阶Q4. FSDP vs DeepSpeed ZeRO-3 区别？
两者等价
FSDP 是 PyTorch 原生 ZeRO-3 实现。功能一致。
选型
2024+ 推荐 FSDP（PyTorch 团队主推，更好集成 torch.compile / mixed precision）。DeepSpeed 仍在 Microsoft / 历史代码库中常见。
进阶Q5. Tensor Parallel 在 Transformer 里怎么切？
Megatron-LM 经典做法
① FFN 第一层 W1 按列切 + 第二层 W2 按行切，每层 1 次 AllReduce ② Attention 按 head 切（head-parallel）③ Embedding 按 vocab 切
进阶Q6. Pipeline Parallel 的 bubble 是什么？怎么减少？
是什么
PP 流水线启动和收尾时部分卡 idle 的现象。bubble ratio ≈ (PP-1) / (PP-1 + num_microbatch)。
减少
① 增加 micro-batch 数（典型 ≥ 4 × PP） ② 用 1F1B 或 Interleaved 1F1B 调度（边 forward 边 backward） ③ 缩小 PP 维度
基础Q7. NCCL 5 个核心原语？
5 个
AllReduce（DP 梯度）/ ReduceScatter（ZeRO-2）/ AllGather（FSDP forward）/ All-to-All（MoE）/ Broadcast（init）
高频Q8. 为什么 Ring AllReduce 比 naive 好？
Naive
所有卡 → master 汇总 → master 广播。master 是瓶颈，O(N×T) 通信。
Ring
组 ring，每张卡都参与传输。通信量恒定 2T 不随 N 增长，带宽最优。
进阶Q9. MoE EP 训练的 4 个挑战？
挑战
① All-to-All 通信瓶颈（每层 2 次） ② Load imbalance（router 偏好少数 expert） ③ Activation 重排显存 ④ Dead expert
进阶Q10. Activation Checkpointing 是什么？什么时候用？
是什么
Forward 时不存中间 activation，backward 时重新算一遍。显存 -50% 代价 +30% 计算。
什么时候用
显存不够 train 大模型 / 大 batch / 长 seq。大模型训练几乎必开。
高频Q11. 千卡训练 NCCL hang 怎么排查？
5 步
① 看 NCCL log（设 NCCL_DEBUG=INFO 重新跑）
② 找慢卡：dcgmi / nvidia-smi 看哪张卡 util 异常
③ 检查 ECC 错误：硬件慢慢退化常见
④ 网络：ibstatus / ib_send_bw 测 IB 健康
⑤ 设 NCCL_TIMEOUT（默认无穷大）避免下次再 hang
实战Q12. 给一个 70B 模型 + 64 张 H100，配置训练方案？
方案
① 主结构：FSDP（ZeRO-3）+ TP=4
② 布局：8 节点 × 8 卡，每节点 TP=4 × 2，跨节点 FSDP shard
③ 显存：weight 17.5GB + grad 17.5GB + opt 70GB 分到 64 卡 ≈ 1.6GB/卡（绰绰有余）
④ 必开：Activation Checkpointing / BF16 / FlashAttention / Gradient Clipping
