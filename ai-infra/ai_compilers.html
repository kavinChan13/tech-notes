01

为什么需要 AI 编译器

🎬生活类比 · 写代码 vs 优化后的产品代码
你写 PyTorch 代码：y = torch.relu(x @ W + b)
直接跑 = 3 个独立 GPU kernel：matmul → add → relu。每步都要把中间结果写到 HBM 再读回来。

AI 编译器：把这 3 个 op 融合成 1 个 kernel（fused_linear_relu），中间结果不进 HBM，性能 +30-50%。

AI 编译器 = "把易写的高层代码"自动转成"高性能的低层 kernel"。

1.1 优化空间

  • Kernel Fusion：多个 op 合成 1 个（减 HBM 读写）
  • Memory layout 优化：避免冗余 transpose / reshape
  • Constant folding：编译期算出常量
  • Dead code elimination：删没用的 op
  • Auto tiling / vectorization：自动生成 GPU kernel
  • Operator scheduling：调度 op 执行顺序减依赖

🎮 交互演示 · Kernel Fusion 减 HBM 搬运

计算 y = relu(x @ W + b) · 切换 Un-fused ↔ Fused 看 HBM 读写次数 + 中间张量是否实例化的差异。

Kernel 调用次数3
HBM 读 + 写 (MB)10 MB
Launch overhead15 µs
总 latency (估算)28 µs
点上方按钮切换 fusion 模式

1.2 跟传统 C++ 编译器对比

维度C++ Compiler (LLVM)AI Compiler

02

torch.compile · 默认胜出（2024+）

PyTorch 2.0 引入。2025 已成为 AI 编译器主流。两大引擎：

2.1 Dynamo · 抓取计算图

Python 字节码层面 hook，动态把 Python 代码转成 FX graph。优势：支持 dynamic shape + Python control flow（传统 trace 类做不到）。

2.2 Inductor · 生成代码

把 FX graph 编译成 Triton kernel（GPU）或 C++ kernel（CPU）。用 Triton 自动生成 kernel 是 Inductor 的杀手锏 — 跟手写 CUDA 性能接近。

2.3 简单使用

import torch
model = MyModel()
model = torch.compile(model)         # 一行加速

# 第一次跑会编译（慢），之后 cache 跑
output = model(input)

典型加速 1.3-2x（推理），训练 1.5-3x。

03

XLA · Google 体系

XLA (Accelerated Linear Algebra) 是 Google 的 AI 编译器。JAX / TPU 全靠 XLA。TensorFlow 也用。

  • JIT 模式：第一次跑编译，之后 cache
  • AOT 模式：提前编译保存（部署用）
  • 多 backend：CPU / GPU / TPU 都支持
  • HLO IR：XLA 自己的 IR，类似 MLIR linalg dialect
XLA 在 TPU 上跑 LLM 训练效率极高（Google Gemini 训练用），但在 GPU 上 PyTorch + torch.compile 已经胜出。2026 多数公司选 PyTorch ecosystem。
04

TVM · 跨硬件后端

TVM 来自 UW + 陈天奇。设计哲学：1 个 IR + N 个 backend（NVIDIA GPU / AMD GPU / ARM CPU / RISC-V / 各种 NPU / FPGA）。

4.1 适合场景

  • 边缘 / 嵌入式：模型部署到 ARM CPU / 各种小型 NPU
  • 多硬件支持：一份模型部署到 10 种硬件
  • 极致优化：AutoTVM / Ansor 自动 tuning kernel

4.2 局限

开发速度比 torch.compile 慢，NVIDIA GPU 上不如 cuBLAS + cuDNN 手调。所以现在 TVM 主战场是非 NVIDIA 硬件 + 边缘。

05

Triton lang · 写 GPU Kernel 的新方式

OpenAI 主推。Python DSL，编译时自动生成 PTX。是 torch.compile Inductor 的 backend。

5.1 跟 CUDA 对比

维度CUDA C++Triton
语言C++Python DSL
抽象层级底层（thread 级）中层（block 级，自动 tile）
典型 kernel 代码量500-2000 行50-200 行
性能100%（baseline）80-95%
调试难度高中
2026 主流NVIDIA 自己（cuBLAS）PyTorch / vLLM / 学术界

5.2 简单例子

import triton
import triton.language as tl

@triton.jit
def add_kernel(a_ptr, b_ptr, c_ptr, N, BLOCK_SIZE: tl.constexpr):
    pid = tl.program_id(0)
    offsets = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE)
    mask = offsets < N
    a = tl.load(a_ptr + offsets, mask=mask)
    b = tl.load(b_ptr + offsets, mask=mask)
    tl.store(c_ptr + offsets, a + b, mask=mask)

同样的 kernel 用 CUDA 写要 100+ 行 + 处理 grid/block/thread。Triton 30 行搞定。

06

MLIR · 编译器的"基础设施"

MLIR (Multi-Level IR) 来自 LLVM 团队（Chris Lattner 主导）。不是一个编译器，是"做编译器的工具"。

6.1 核心思想

传统编译器：1 个 IR（如 LLVM IR）。MLIR：多层 IR + 可定义 dialect。每个层级表达不同抽象。

例子：TensorFlow → tf dialect → linalg dialect → affine dialect → LLVM IR → GPU。每一层都能做适合该层的优化。

6.2 主流采用

  • TensorFlow / TFLite：底层用 MLIR
  • IREE（Google）：MLIR-based ML runtime
  • Triton：MLIR-based
  • XLA：正逐步迁移到 MLIR
  • Mojo（Modular）：MLIR 主导，专为 AI 设计的新语言
MLIR 是AI 编译器领域的"标准基础设施"。2026 任何新 AI 硬件 / 新编译器都基于 MLIR。如果你做编译器，MLIR 是必修。

C++ 编译器人的友好入口

已经懂 LLVM / 写过编译器的 C++ 工程师 → AI 编译器是最直接的 cross-over：

7.1 复用率高

  • LLVM IR 知识 → MLIR 几乎平移
  • SSA / dataflow analysis → AI graph 优化一样适用
  • register allocation / scheduling → kernel fusion 思想类似
  • backend code gen → GPU kernel 生成

7.2 推荐学习路径

  1. Week 1：读 torch.compile blog + 跑 simple example 体验 graph capture
  2. Week 2：学 Triton（OpenAI 官方 tutorial），写 vector add + GEMM
  3. Week 3：读 MLIR tutorial（toy language compiler）
  4. Week 4：贡献：给 torch.inductor 或 IREE 提一个小 PR

7.3 招聘 hot spots

08

10 面试题

Q1为什么需要 AI 编译器？跟传统 C++ 编译器区别？
AI 编译器把"易写的高层代码（PyTorch）"转成"高性能 GPU kernel"。
vs C++ 编译器：① 输入是计算图 vs AST ② 优化粒度 op 级 vs 指令级 ③ 目标多 backend（GPU/TPU/NPU）
Q2Kernel Fusion 为什么重要？
把 3 个 op (matmul → add → relu) 合成 1 个 kernel，中间结果不进 HBM，减少 HBM↔SRAM 数据搬运。LLM 推理这种 memory-bound 场景提升 30-50%。
Q3torch.compile 跟 jit.trace / jit.script 区别？
jit.trace 只 trace 一次执行路径，不能处理 control flow。
jit.script 解析 Python 子集，限制大。
torch.compile (Dynamo) 在字节码层 hook，支持 dynamic shape + Python control flow，可用性大幅提升。
Q4Triton 跟 CUDA 怎么选？
CUDA：极致性能（cuBLAS / FlashAttention） + 复杂。
Triton：性能 80-95% + 简单 10x。
2026：NVIDIA 自己 + cuBLAS/cuDNN 仍用 CUDA；PyTorch / vLLM / 学术界主要用 Triton。
Q5XLA 跟 torch.compile 怎么选？
XLA：JAX / TPU 必用 / TensorFlow / Google ecosystem。
torch.compile：PyTorch 必用，GPU 主流。
2026 PyTorch 生态胜出，多数公司选 torch.compile。
Q6MLIR 跟 LLVM 区别？为什么需要 MLIR？
LLVM：1 个 IR（低层）。
MLIR：多层 IR + 可定义 dialect。每层表达不同抽象（tensor → linalg → affine → llvm → gpu）。
需要 MLIR：AI 计算图有多个 abstraction level（数学 op → 低层指令），单层 IR 表达不了。
Q7TVM 适合什么场景？为什么 NVIDIA GPU 上不如 PyTorch？
适合：非 NVIDIA 硬件（ARM / 各种 NPU / FPGA）+ 边缘部署 + 跨硬件场景。
NVIDIA GPU 上不如：因为 NVIDIA 自家 cuBLAS / cuDNN 手调极致，TVM AutoTVM 难赢。
Q8AOT vs JIT 编译怎么选？
JIT：第一次跑慢（编译），之后 cache。开发 + 服务端推理主流。
AOT：提前编译保存 binary。边缘部署 + 启动敏感场景（如自动驾驶）。
Q9给一个 PyTorch 模型加速，你会先做什么？
5 步：① profile（torch.profiler 找瓶颈 op）② torch.compile（1 行加速 1.5-3x）③ 量化（INT8/FP8 ÷ 2 显存 × 2 速度）④ FlashAttention（如果有 attention）⑤ 自定义 kernel（最后兜底，写 Triton）
Q10C++ 编译器人转 AI 编译器，最大杠杆点？
复用率高：LLVM/MLIR IR / SSA / dataflow analysis 几乎平移。学习路径：torch.compile blog → Triton tutorial → MLIR toy compiler → 给 IREE/Inductor 提 PR。
招聘 hot spots：NVIDIA / OpenAI Triton / Modular Mojo / 国内华为 MindSpore / 智驾 NPU 团队。
