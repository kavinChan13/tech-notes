读前地图 · 量化是 AI Infra 的杠杆点

1-2 周工作量，砍掉 50-75% 推理 cost

量化 = AI Infra 工程师 ROI 最高的优化方向。1-2 周工作量可以把推理 cost 砍 50-75%。本文 50 分钟把量化讲透。

01

为什么需要量化 · 用同样硬件跑更大模型

🎬生活类比 · 把高清照片压成 JPEG
一张高清照片 50 MB（=FP16 模型）。每个像素用 16 位浮点存。
压成 JPEG 后变 5 MB（=INT4 量化），每个像素只用 4 位。看起来基本一样（人眼分辨不出），文件小 10 倍。

模型量化一模一样：把 FP16 权重（16 位）压成 INT4（4 位），模型大小砍 4 倍，质量损失 1-3%（基本感觉不到）。但能装更大模型 + 推理更快 + 显存够装更多并发。
JPEG 压缩 → 量化  ·  像素 → weight / activation  ·  视觉无损 → PPL / accuracy 几乎不变

1.1 量化的 3 大收益

💾 显存 ÷ 2-4x

Llama-70B FP16 = 140 GB（要 2 张 H100）。INT4 = 35 GB（1 张 H100 装下还有空 KV-Cache）。

⚡ 推理速度 ×2-3

Decode 是 memory-bound（见 X06 §3）。weight 小 4x → 每步要读的字节少 4x → 速度涨 2-3x（未必满 4x 因为还有其它开销）。

💰 Cost ÷ 2-5x

同样 throughput 用少一半 GPU。月成本直接腰斩。这是为什么 vLLM 默认要支持量化。

量化 ≠ 蒸馏。蒸馏是训一个新小模型模仿大模型（参数少、能力降）。量化是同一个模型，只是参数用更少 bit 存（参数数量不变，精度降）。两者经常组合用。

2.2 浮点 vs 整数 · 数值分布差异

浮点数（FP）

数值非均匀分布 — 0 附近密，远处稀。能表示极小到极大范围。
适合：activation 和 weight 都有 outlier 的场景

整数（INT）

数值均匀分布。只有 -128 到 127（INT8）。需要 scale factor 把 weight 映射到这个范围。
适合：weight 分布相对均匀的情况

2.3 INT 量化的核心公式

# 假设 weight W (FP16) 范围 [-2.5, 1.8]
# 量化到 INT8 [-127, 127]:

scale = (W.max() - W.min()) / 255     # 一个量化区间多大
zero_point = -W.min() / scale            # 0 对应哪个 int 值

W_int8 = round(W / scale + zero_point)
       = clip(W_int8, -128, 127)

# 反量化（推理时用）:
W_dequant = (W_int8 - zero_point) * scale
关键 trick：scale 不是全模型一个。Per-channel quantization（每行/列一个 scale）比 per-tensor 精度高很多。2026 SOTA 用 group-wise（每 128 个元素一个 scale），平衡精度和存储开销。

03

PTQ vs QAT · 两条量化路径

🎬生活类比 · 减肥的两种方法
PTQ（Post-Training Quantization）= 已经训好的胖子直接抽脂。快，几小时搞定，但有效果损失。
QAT（Quantization-Aware Training）= 训练过程中就模拟量化，让模型学会"在低精度下也表现好"。慢，要训几天，但效果好。

2024+ LLM 主流用 PTQ，因为：① LLM 训练贵到 QAT 不划算 ② 现代 PTQ 算法（GPTQ / AWQ）已经很好，损失 < 2%
维度PTQQAT
需要训练✅ 不需要❌ 需要
耗时几小时几天-几周
需要数据~128 calibration 样本完整训练数据
精度损失1-3%< 1%
LLM 实战主流稀少

本文剩下章节都讲 PTQ（LLM 量化的主流）。

04

INT4 量化 · GPTQ / AWQ / SmoothQuant 对比

3 种主流 PTQ 算法

4.1 GPTQ · 第一代 INT4 王者（2023）

🎬生活类比 · 一层一层量化 + 误差校正
GPTQ 思路：逐层量化，每层量化时用前面层量化的误差校正。就像盖房子，每砌一层都考虑前面层的偏差来调整。

数学上：用 Hessian 矩阵的 inverse 来决定哪些 weight 量化先后，最小化整层 quantization error。

4.2 AWQ · 关键 weight 保留（2023.6）

🎬生活类比 · VIP 客户高规格服务
AWQ 发现：不是所有 weight 同样重要。1% 的"关键 weight"（对 activation 影响最大）保留 FP16 不量化，其它 99% 量化到 INT4。

AWQ 的关键 insight：activation magnitude 大的对应 channel 的 weight 最重要，量化时优先保护。

4.3 SmoothQuant · activation 量化的钥匙（2023.4）

GPTQ / AWQ 只量化 weight。SmoothQuant 量化 weight + activation（W8A8）。难点：activation 有 outlier，直接量化误差大。

典型 activation 分布（X 轴 = channel index, Y 轴 = magnitude）：

红色 = outlier channel（远高于其它），直接量化会损失大量信息

SmoothQuant 的解：把 outlier 从 activation"挪"到 weight。activation × scale → weight × (1/scale)，数学等价但 activation 分布变平滑，weight 仍能量化。

4.4 3 方法对比

GPTQ
2022.10 · Frantar et al.
量化目标：仅 weight
精度：W4A16（W = INT4, A = FP16）
速度提升：~2-3x
精度损失：1-2%
实现：AutoGPTQ
AWQ
2023.6 · MIT/Tsinghua
量化目标：仅 weight + 1% 保留
精度：W4A16
速度提升：~2-3x
精度损失：0.5-1.5% (优于 GPTQ)
实现：AutoAWQ / vLLM 内置
SmoothQuant
2022.11 · MIT
量化目标：weight + activation
精度：W8A8（INT8 全量化）
速度提升：~1.5-2x
精度损失：< 0.5%
用途：需要 activation 量化的场景
SpQR (2023.6)
+ 2024 GGUF / EXL2
极致压缩：3-bit 甚至 2-bit
精度：保留少量 high-precision outlier
损失：3-bit ~1%，2-bit 5%+
用途：本地 / 端侧 LLM
选型决策：默认 AWQ（精度最好的 INT4），需要 INT8 W8A8 用 SmoothQuant，极致压缩本地用 GGUF（2-3 bit）。vLLM / SGLang / TRT-LLM 三家都默认支持 GPTQ + AWQ。
05

FP8 · H100 时代的训练 + 推理标配

为什么 H100 量化主流转 FP8 而不是 INT8
🎬生活类比 · 8 位浮点 vs 8 位整数的"动态范围"
INT8 像固定刻度的尺：从 -128 到 127 均匀分布。测细节准，但测超大数会爆。
FP8 像对数刻度的尺：能测从 0.0001 到 10000 的超大范围，但同一区间内精度低。

深度学习里 activation 经常有extreme outlier（attention 输出有时是几千），INT8 装不下要 clip 损失大。FP8 能装下，所以 H100 + B200 主推 FP8。

5.1 FP8 vs INT8 数据特性

维度INT8FP8 (E4M3)
范围[-128, 127]±448
动态范围~2.5 个数量级~16 个数量级
0 附近精度±0.5 (固定)±0.001 (高)
大数精度±0.5 (固定)±16 (低)
硬件支持所有 GPUH100, B200, MI300

5.2 FP8 训练的革命

DeepSeek-V3 是首个公开做 FP8 mixed-precision 训练的大规模模型（671B）：

  • 训练成本省 50%（FP8 算力是 FP16 的 2x）
  • 少量 sensitive 算子（如 attention softmax）仍用 BF16
  • 需要细粒度 scale + outlier handling 才能不崩
2026 趋势：大公司训练 + 推理都转 FP8。Llama 3.3 / Qwen2.5 / DeepSeek-V3 / Gemini 都已支持。面试提到"FP8 训练 + 推理一体化"加分。
06

KV-Cache 量化 · Decode 性能的杀手锏

Decode 阶段瓶颈是读 KV-Cache 的带宽（见 X06 §3）。KV-Cache 量化直接降"要读的字节"，提速比 weight 量化更明显。

6.1 KV-Cache 量化的收益

FP16 KV-Cachebaseline
100% / 1x
FP8 KV几乎无损
50% 显存 / 1.5-2x 速度
INT4 KV长上下文略损 1-2%
25% 显存 / 2-3x 速度

6.2 INT4 KV-Cache 的代价

✅ 收益

  • KV-Cache 显存 / 4 → 装 4x 并发或 4x 上下文
  • Decode 带宽消耗 / 4 → 速度 2-3x
  • vLLM / SGLang / TRT-LLM 都已支持

⚠ 代价

  • 短上下文（< 4k）精度几乎无损
  • 长上下文（> 32k）累积误差可见，精度 -1-2%
  • 不同 layer 敏感度不同（前几层比后几层敏感）

Speculative Decoding · 用小模型加速

跟量化互补的另一个加速维度

详见 Y02 Ch6。简言之：小模型预测 K 个 token + 大模型一次性验证，单步等效产出 K 个 token。

跟量化的关系

维度量化Spec Decoding
优化目标每步算/读字节少少跑几步
典型加速2-3x1.5-3x
组合效果互补可叠加，组合 4-9x
2026 主流 production 推理 stack：vLLM + AWQ INT4 weight + INT8 KV-Cache + Speculative Decoding。综合 throughput 比 naive FP16 高 8-15x。
08

实战 · 30 分钟量化 Qwen-7B 到 INT4

8.1 用 AutoAWQ

# 安装
pip install autoawq

# 量化脚本
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_path = "Qwen/Qwen2.5-7B-Instruct"
quant_path = "./qwen2.5-7b-awq"

# 加载
model = AutoAWQForCausalLM.from_pretrained(model_path)
tokenizer = AutoTokenizer.from_pretrained(model_path)

# 量化（用 128 条 calibration 样本）
model.quantize(
    tokenizer,
    quant_config={
        "w_bit": 4,           # 4-bit weight
        "q_group_size": 128,  # 每 128 个元素一个 scale
        "zero_point": True,
        "version": "GEMM",    # GEMM 模式 (vs GEMV)
    }
)

# 保存
model.save_quantized(quant_path)
tokenizer.save_pretrained(quant_path)
# 量化完成，模型大小从 14 GB 降到 4.5 GB

8.2 用 vLLM serving 量化后模型

# vLLM 自动识别 AWQ 格式
vllm serve ./qwen2.5-7b-awq \
    --quantization awq \
    --max-model-len 8192 \
    --gpu-memory-utilization 0.9

# 跑 benchmark 对比
# FP16: ~120 tokens/s
# AWQ INT4: ~280 tokens/s   (2.3x 提速)

8.3 验证精度

# 用 lm-eval-harness 跑 MMLU / GSM8K / 你的业务 eval set
pip install lm-eval

lm_eval --model vllm \
    --model_args pretrained=./qwen2.5-7b-awq,quantization=awq \
    --tasks mmlu,gsm8k \
    --batch_size 16

# 对比 FP16 baseline 看精度损失（典型 -1-2%）
09

5 个量化常见坑

坑 1 · INT4 量化后 reasoning 任务掉得多

7B 模型 INT4 后 GSM8K 数学题准确率从 65% 掉到 42%。普通对话感觉没变。

根因：reasoning 是 long-tail capability，对精度敏感。修复：reasoning 场景用 INT8 或 FP8 而不是 INT4。

坑 2 · Calibration 数据选错

用通用文本做 calibration，量化后在公司专业领域准确率暴跌。

根因：calibration 决定 scale 怎么选。修复：用业务实际数据做 calibration（128 条够），跟生产分布一致。

坑 3 · 长上下文 INT4 KV-Cache 累积误差

短对话好好的，长文档总结质量暴跌。

根因：每步 decode 误差累积，长上下文放大。修复：长上下文场景用 FP8 KV-Cache 而非 INT4，或前几层保留 FP16。

坑 4 · 量化模型反而比 FP16 慢

某团队量化后 latency 反而涨了。

根因：① 推理引擎没启用专门 INT4 kernel（fallback 到通用 kernel） ② batch size 太小（量化收益要靠 batching 摊薄）。修复：确认 vLLM/SGLang 用 marlin / wint4 等专用 kernel。

坑 5 · 量化后微调（QAT 替代）失败

先 SFT 再 AWQ 量化效果不错；想反过来"量化后微调"，效果崩。

根因：INT4 weight 直接 SFT 数值不稳。修复：用 QLoRA — 量化 base + 训练 FP16 LoRA adapter。

12 道量化高频面试题

基础Q1. 量化的 3 个收益？
3 收益
① 显存 ÷ 2-4x（同硬件装更大模型）② 推理速度 ×2-3（decode 是 memory-bound，weight 小 → 读快）③ Cost ÷ 2-5x
基础Q2. PTQ 和 QAT 区别？LLM 主流哪个？
PTQ
训练后量化，不需要训，几小时搞定，精度损失 1-3%。LLM 主流。
QAT
训练时模拟量化，需要训练几天-几周，精度损失 < 1%。LLM 太贵不划算，主要 CV 用。
进阶Q3. GPTQ / AWQ / SmoothQuant 区别？
GPTQ
逐层量化 + Hessian 误差校正。仅 weight INT4。
AWQ
识别 1% "关键 channel"（activation magnitude 大对应的）保留 FP16，其它 INT4。精度优于 GPTQ。
SmoothQuant
同时量化 weight + activation (W8A8)。通过"挪 outlier 从 activation 到 weight"解决 activation 量化难题。
高频Q4. 为什么 H100 推 FP8 而不是 INT8？
动态范围
INT8 范围 [-128,127]，FP8 范围 ±448。LLM activation 经常有 extreme outlier（attention 输出可能几千），INT8 装不下 clip 损失大，FP8 能装下。
硬件
H100 + B200 内置 FP8 Tensor Core，算力是 FP16 的 2x。是 2026 训练 + 推理主流。
进阶Q5. 为什么 KV-Cache 量化对 decode 加速比 weight 量化更明显？
原因
Decode 每步要读整个 KV-Cache（GB 级），weight 是固定的（cache 在 SM 内）。KV-Cache 量化直接减少"每步要读的字节"，瓶颈解除得更彻底。
数据
典型 70B 模型 32k 上下文：weight FP16 = 140GB，KV-Cache FP16 = 84GB。KV 量化收益是 weight 量化的 60% 但更便宜实现。
高频Q6. INT4 量化后什么场景质量掉得最多？
长尾
① Reasoning（数学 / 代码）→ INT4 掉 5-20% ② 长上下文（> 32k）→ 累积误差 ③ 稀少语言（小语种 / 方言）④ edge case
主流场景
普通对话 / 短问答 / 摘要 → 几乎无损（< 1%）
基础Q7. Per-tensor / Per-channel / Group-wise 量化区别？
Per-tensor
整个 tensor 共享 1 个 scale。最简单，精度最差。
Per-channel
每个 channel 一个 scale。精度好很多，存储略增。
Group-wise
每 128 个元素一个 scale。2026 SOTA，精度跟 per-channel 接近但 GEMM 更高效。AWQ / GPTQ 默认。
进阶Q8. SmoothQuant 怎么解决 activation outlier？
核心 trick
把 outlier 从 activation 数学等价地"挪"到 weight。activation × scale_factor → weight × (1/scale_factor)，最终结果一样，但 activation 分布变平滑可量化。
为什么 work
weight 本身是离线静态的，可以容忍更大动态范围；activation 是动态的，需要小动态范围才能 INT8。
进阶Q9. Calibration 数据怎么选？多少条够？
选什么
必须跟生产数据 distribution 一致。用通用 web 文本 calibration 然后跑专业领域 → 大幅掉点。
多少条
典型 128-512 条。再多边际收益小。质量比数量重要。
高频Q10. QLoRA 是什么？怎么 work？
是什么
"量化 base 模型 (INT4) + 训练 FP16 LoRA adapter"。让 70B 模型在 1 张 24GB 消费级 GPU 上能 fine-tune。
关键 trick
① NF4 (NormalFloat 4-bit) 替代 INT4 — 更适合 weight 分布 ② Double Quantization — 量化 scale 自己再量化 ③ Paged Optimizer — 用 GPU-CPU 切换避免 OOM
进阶Q11. Activation 量化为什么比 weight 量化难？
原因
① Activation 是动态的（每 forward 不同），难 calibration ② 有 extreme outlier ③ Outlier channel 不固定（每层不同）
解
SmoothQuant（挪 outlier）、FP8（足够动态范围）、混合精度（outlier 层保留 FP16）
实战Q12. 我有 Llama-70B 想部署到 1 张 H100 80GB，怎么做？
配方
① AWQ INT4 量化 weight：70B × 2 字节 (FP16) = 140GB → AWQ INT4 = 35GB ✅
② FP8 KV-Cache：32k 上下文 KV ≈ 21GB（GQA + FP8）
③ 剩余：80 - 35 - 21 - 5 (activation) = 19GB，能跑 ~3-5 并发 32k 用户
④ 提速：再加 Speculative Decoding（用 Llama-1B draft） + Continuous Batching
⑤ 预期：throughput ~600 tokens/s，P95 latency ~2s，cost vs 用 2 张卡 -50%
加分
提到 reasoning 任务可能要降级到 FP8 weight 而不是 INT4。
