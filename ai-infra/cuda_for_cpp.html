为什么是 GPU · 8 头大象 vs 1 个赛车手

CPU 和 GPU 不是"快慢"关系，是"擅长不同任务"
🎬生活类比 · 赛车手 vs 大象群搬砖
CPU 是赛车手（少而强）：1 个赛车手开 F1，单圈极快（高时钟 / 复杂分支预测 / 大 cache / 乱序执行）。但你让他搬 1000 块砖，他一次只能搬 1 块。

GPU 是大象群（多而简单）：1000 头大象，每头都很笨（简单流水线 / 弱分支预测 / 小 cache），但同时搬 1000 块砖瞬间完工。

关键不是"GPU 比 CPU 快"，而是 GPU 在"同样操作重复几千次"这类任务上效率秒杀 CPU。矩阵乘、卷积、attention 都是这种任务 — 所以深度学习选 GPU。
赛车手 → CPU core (强但少)  ·  大象群 → GPU SP (弱但多)  ·  搬砖 → 数据并行任务  ·  F1 复杂动作 → 复杂分支 / 串行算法

1.1 CPU vs GPU 硬件对比

维度CPU (Intel Xeon)GPU (H100)差异原因
核心数~64~16,896 (SP)GPU 砍掉乱序/分支预测，腾出晶体管做更多核
时钟频率~3 GHz~1.5 GHzGPU 牺牲单核性能换并行密度
L1 Cache / core~32 KB~128 KB (shared)GPU 由程序员显式管理，不是硬件透明缓存
主内存带宽~100 GB/s~3 TB/s (HBM3)GPU 用 HBM 堆叠超高带宽内存
峰值 FLOPS (FP16)~10 TFLOPS~1000 TFLOPSGPU 大量 ALU + Tensor Core
FP16 / 算力比1x~100x这就是为什么 LLM 训练只能用 GPU
GPU = "牺牲单核 + 牺牲缓存自动管理 + 牺牲分支预测 → 换 100x 并行算力 + 30x 内存带宽"。所有 CUDA 编程的痛苦都源于：程序员要做 CPU 上硬件帮你做的事（手动管理 cache、避免分支、对齐访问）。

GPU 硬件模型 · SM / SP / Warp 三层结构

类比 CPU 多核 → GPU 多 SM；CPU 多线程 → GPU 多 Warp

2.1 SM / SP / Warp · 工厂的车间 / 工人 / 班组

🎬生活类比 · GPU 是一座工厂
一座工厂 = 1 个 GPU。
工厂分 132 个车间（=SM, Streaming Multiprocessor）。
每个车间里有 128 个工人（=SP, Streaming Processor）。
工人不是单独干活，而是 32 个一组（=Warp）同步执行同一个动作。

关键洞察：CPU 的"线程"是独立的（各干各的）。
GPU 的"线程"是32 个一组绑定行动的 — 这就是 SIMT (Single Instruction Multiple Threads)。
多核 CPU：N 个独立 core，每个 core 有自己的指令流。线程之间用锁 / atomic 同步。 
for i in 0..N:
    spawn_thread(work, i);
// 4 个 core 跑 4 个线程
GPU：N 个 SM，每个 SM 同时执行多个 warp，每个 warp 32 线程同步执行。 
__global__ void kernel(...){
    int tid = threadIdx.x;
    work(tid);  // 32 线程同时干同一行
}
kernel<<<128,256>>>(...);

2.2 SIMT 执行模型 · 32 个线程绑在一起

🎬生活类比 · 32 人合唱团
想象 32 人合唱团（=1 个 warp），必须同步唱同一句歌词。指挥说"开始唱第 5 段"，所有 32 人同时执行同一条指令，但唱自己负责的那一行声部（=不同数据）。

这就是 SIMT（Single Instruction, Multiple Threads）— 1 条指令 × 32 个数据并行执行。跟 CPU SIMD (SSE/AVX) 思想完全一样，只是规模更大。

2.3 Warp Divergence · 32 人有人不唱了

🎬生活类比 · 合唱团里有人走神
指挥说"高音部唱 A，低音部唱 B"。结果：高音部唱时，低音部要等；低音部唱时，高音部要等。本来一首歌 3 分钟唱完，因为分成两段，变成 6 分钟。

这就是 Warp Divergence。32 个线程遇到 if-else 分支，部分走 if 部分走 else，硬件必须串行执行两个分支 — 同一个 warp 的吞吐量直接腰斩。

⚠️ Warp Divergence：分支导致 2 个 cycle 才完成（理想 1 个），throughput 减半

写 CUDA 的核心原则：避免 warp 内 divergent branch。如果非要分支，让同一个 warp 的 32 个线程走相同分支（例如按 threadIdx.x / 32 分配工作，而不是 threadIdx.x % 2）。

GPU 内存层次 · 5 级 cache 你必须显式管理

CPU 的 L1/L2/L3 全自动，GPU 的 shared memory 要你自己搬
🎬生活类比 · 5 级仓库系统
你的工厂有 5 级仓库，越近的越快但越小：
① 手边的零件盒（=Register）：超快但只能放几样东西
② 桌上的小工具箱（=Shared Memory）：很快但只够当前班组用
③ 车间储物柜（=L1 Cache）：硬件自动管理
④ 仓库（=L2 Cache）：所有车间共享
⑤ 另一栋楼的仓库（=HBM 全局显存）：慢但容量大

跟 CPU 最大的差别：CPU 的 L1/L2/L3 全部硬件自动管理（cache 透明）；GPU 的 Shared Memory 要程序员显式 load/store。这是手动优化最大的杠杆点。

3.1 5 级内存层次 · 带宽和容量

Register
~256 KB/SM
~5 TB/s · 1 cycle · per thread 私有
Shared
~228 KB/SM (H100)
~19 TB/s · ~30 cycle · per block 共享
L1
~128 KB/SM
~19 TB/s · ~30 cycle · 硬件管理
L2
~60 MB
~5 TB/s · ~200 cycle · 所有 SM 共享
HBM
80 GB (H100)
~3 TB/s · ~400 cycle · 全局显存
Register 比 HBM 快 ~400 倍（cycle latency），但容量只有 1/300000。这就是为什么所有 GPU 优化的核心都是 "把数据从 HBM 搬到 Shared / Register 后多次复用"。FlashAttention 就是这个思想的典型代表。

3.2 Coalesced Memory Access · 32 人一起去仓库取货

🎬生活类比 · 32 人取连续编号 vs 分散编号
32 个工人去仓库取零件。
合并访问（=coalesced）：32 人取编号 0-31 的零件（连续）→ 仓库管理员一趟搬完（1 次内存事务）。
分散访问：32 人取编号 0/100/200/.../3100（分散）→ 仓库管理员要跑 32 趟（32 次内存事务）。

性能差 32 倍。这就是为什么 CUDA 强调 "同一个 warp 的 32 线程要访问连续内存"。 
// ❌ 不 coalesced：32 个线程访问跨度 = 32 * sizeof(float) = 128 字节，要 32 次事务
int tid = threadIdx.x;
float x = data[tid * 32];     // thread 0 读 data[0], thread 1 读 data[32]...

// ✅ coalesced：32 个线程访问连续地址，1 次事务搞定
float x = data[tid];           // thread 0 读 data[0], thread 1 读 data[1]...

3.3 Bank Conflict · Shared Memory 的陷阱

C++ false sharing：2 个线程写同一个 cache line 不同变量 → cache line ping-pong，性能腰斩。
CUDA bank conflict：Shared memory 分 32 bank，同一个 warp 的多个线程访问同一 bank → 串行。
解法：让 32 个 thread 访问 32 个不同 bank（padding 数组宽度）。

类比公司组织架构：员工 / 部门 / 公司
🎬生活类比 · CUDA 程序 = 一家公司
Thread（员工）：最小工作单元，干 1 个任务

你启动 kernel 时指定：开几个部门（block 数）+ 每部门几个员工（thread 数） = 总员工数。

4.1 Kernel 启动语法

// CUDA 启动 kernel 的特殊语法 <<<grid, block>>>

    data[tid] = data[tid] * 2;
}

变量含义

blockIdx.x/y/z当前 block 在 grid 内的 ID
blockDim.x/y/z每个 block 有多少 thread
gridDim.x/y/z整个 grid 有多少 block

Tensor Core · 矩阵专用硬件

为什么 H100 FLOPS 比理论值还高 10x
🎬生活类比 · 通用工具 vs 专用机器
通用车床（=普通 SP）可以做任何零件，但做一个圆形零件要 1 小时。
专用圆形冲压机（=Tensor Core）一次性冲一片圆形，5 秒。但只能做圆形。

Tensor Core 就是 GPU 内"专门做矩阵乘的硬件" — 1 cycle 完成 4×4 矩阵 × 4×4 矩阵（普通 SP 要 ~64 cycle）。所有现代 LLM 训练 / 推理都依赖 Tensor Core。

5.1 Tensor Core 演进

架构GPU支持精度峰值 FLOPS (FP16)
Volta (2017)V100FP16~125 TFLOPS
Ampere (2020)A100FP16/BF16/TF32/INT8~312 TFLOPS
Hopper (2022)H100+ FP8~989 TFLOPS
Blackwell (2024)B200+ FP4~2250 TFLOPS
每代 GPU 主升级都是 Tensor Core。Hopper 加 FP8 让 LLM 训练 throughput 翻倍，Blackwell 加 FP4 让推理 throughput 再翻倍。"算力"的实际意义就是 "Tensor Core 算力"。

5.2 怎么用 Tensor Core

  1. 不要自己写 — 直接用 cuBLAS / cuDNN / cutlass 这些库
  2. 对齐尺寸 — 矩阵维度对齐 8 / 16 / 32 才能跑 Tensor Core
  3. 用对精度 — FP16 / BF16 / FP8，不能是 FP32
  4. 检查命中 — Nsight Compute 看 "Tensor Core utilization" metric

写第一个 Kernel · GEMM 从朴素到 cuBLAS 90%

3 段代码看懂 "为什么自己写永远不如 cuBLAS"

6.1 Vector Add · 入门必写

// host (CPU 侧) 代码
int N = 1 << 20;       // 1M 元素
float *h_a, *h_b, *h_c;
cudaMallocManaged(&h_a, N * sizeof(float));
cudaMallocManaged(&h_b, N * sizeof(float));
cudaMallocManaged(&h_c, N * sizeof(float));

// 启动 kernel
int threads = 256;
int blocks = (N + threads - 1) / threads;   // 向上取整
vector_add<<<blocks, threads>>>(h_a, h_b, h_c, N);
cudaDeviceSynchronize();             // 等待 GPU 完成

// device (GPU 侧) kernel
__global__ void vector_add(float* a, float* b, float* c, int N){
    int tid = blockIdx.x * blockDim.x + threadIdx.x;
    if (tid < N) {
        c[tid] = a[tid] + b[tid];
    }
}

6.2 Naive GEMM · 朴素矩阵乘

矩阵乘 C = A × B（A 是 M×K, B 是 K×N, C 是 M×N）。每个输出元素 C[i][j] = Σ A[i][k] × B[k][j]。

// 每个 thread 算 C 的 1 个元素
__global__ void gemm_naive(float* A, float* B, float* C,
                            int M, int N, int K){
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;

    if (row < M && col < N) {
        float sum = 0.0f;
        for (int k = 0; k < K; ++k) {
            sum += A[row * K + k] * B[k * N + col];     // ❌ 每次都从 HBM 读
        }
        C[row * N + col] = sum;
    }
}

// 启动
dim3 threads(16, 16);                // 16x16 = 256 thread / block
dim3 blocks((N+15)/16, (M+15)/16);
gemm_naive<<<blocks, threads>>>(A, B, C, M, N, K);
朴素 GEMM 的问题：每个 thread 算 C 的 1 个元素需要读 K 个 A 值 + K 个 B 值。不同 thread 重复从 HBM 读相同 A/B 元素。性能 ≈ cuBLAS 的 5-10%。

6.3 Tiled GEMM · 用 Shared Memory 复用

关键 insight：把 A、B 的小块（tile）先搬到 Shared Memory，让一个 block 内所有 thread 反复用。每个 HBM 字节被复用 ~TILE_SIZE 次。

#define TILE 16

__global__ void gemm_tiled(float* A, float* B, float* C,
                            int M, int N, int K){
    // ⭐ 关键：分配 Shared Memory（每个 block 一份）
    __shared__ float A_tile[TILE][TILE];
    __shared__ float B_tile[TILE][TILE];

    int row = blockIdx.y * TILE + threadIdx.y;
    int col = blockIdx.x * TILE + threadIdx.x;
    float sum = 0.0f;

    // 分 K/TILE 个 tile 处理
    for (int t = 0; t < (K + TILE - 1) / TILE; ++t) {
        // 每个 thread 协作搬运 1 个元素到 shared memory
        A_tile[threadIdx.y][threadIdx.x] = A[row * K + t * TILE + threadIdx.x];
        B_tile[threadIdx.y][threadIdx.x] = B[(t * TILE + threadIdx.y) * N + col];

        __syncthreads();   // 等所有 thread 搬完

        // 在 shared memory 上做 TILE 次乘加（这步飞快）
        for (int k = 0; k < TILE; ++k) {
            sum += A_tile[threadIdx.y][k] * B_tile[k][threadIdx.x];
        }
        __syncthreads();   // 等所有 thread 算完，才能进下一 tile
    }
    C[row * N + col] = sum;
}

6.4 性能对比

Naive GEMM直接 HBM 读写
~7% of cuBLAS
Tiled GEMM+ Shared Memory
~35% of cuBLAS
+ Register Tiling每 thread 算 4×4
~60% of cuBLAS
+ Tensor Core (WMMA)手写 + tensor core
~85% of cuBLAS
cuBLAS / cutlassNVIDIA 工程师手调
100% baseline
结论：自己手写 GEMM 能跑到 cuBLAS 的 60-85% 已经很厉害。真实工程不要自己写 GEMM，直接用 cuBLAS。但理解优化过程是面试硬通货 — 招聘官会问 "你怎么把 naive GEMM 优化到 60%"。

Roofline 性能模型 · 你卡算力还是卡带宽

优化前必须先识别瓶颈
🎬生活类比 · 餐厅的天花板
餐厅每小时上多少菜，取决于两个上限的较小者：
① 厨师做菜速度（=GPU FLOPS）
② 服务员上菜速度（=HBM 带宽）

如果你的菜做一道要 30 分钟但上菜 10 秒（做菜密集），瓶颈在厨房；如果做一道 1 分钟但上菜要 5 分钟（搬运密集），瓶颈在服务员。

Roofline 就是把这个 trade-off 画成一条折线图。任何 kernel 的性能上限 = min(算力上限, 带宽 × arithmetic intensity)。

7.1 Arithmetic Intensity · 关键指标

定义：AI = FLOPs / Bytes，即每读 1 字节做多少 FLOPs。

kernel典型 AI瓶颈优化方向
Vector Add~0.08带宽没什么好优化的，纯搬数据
GEMM (大矩阵)~50-500算力Tensor Core / 量化
Attention (FlashAttn)~10-30算力 + 带宽tiling 提 AI
LayerNorm~1带宽fused kernel 减读
LLM Decode (attention)~1-3带宽KV-Cache 量化 / GQA / MLA

Triton lang · 新一代选择

OpenAI 主推。用 Python 写 kernel，编译时自动生成 PTX。比 CUDA 简单 10x，性能接近手写 CUDA。

# Triton 写 vector add（比 CUDA 简单）
import triton
import triton.language as tl

@triton.jit
def vec_add(a_ptr, b_ptr, c_ptr, N, BLOCK_SIZE: tl.constexpr):
    pid = tl.program_id(0)
    offsets = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE)
    mask = offsets < N
    a = tl.load(a_ptr + offsets, mask=mask)
    b = tl.load(b_ptr + offsets, mask=mask)
    tl.store(c_ptr + offsets, a + b, mask=mask)

# 启动
N = 1 << 20
vec_add[(triton.cdiv(N, 256),)](a, b, c, N, BLOCK_SIZE=256)
2024-2026 趋势：大量 GPU kernel 从 CUDA 迁移到 Triton（包括 PyTorch 内部）。面试时提到 "Triton 是 CUDA 的高效替代" 加分。但 NVIDIA 自己的 cuBLAS / TensorRT 仍然是手写 CUDA（追求极致）。

12 道 CUDA 高频面试题

折叠卡片 · 先想再展开
基础Q1. CPU 和 GPU 的核心区别是什么？为什么 LLM 用 GPU？
核心
CPU 少而强（~64 core, 复杂分支预测/大 cache），GPU 多而简单（~17k SP, SIMT 执行）。LLM 训练/推理大量同质操作（矩阵乘 / attention），GPU 在这类任务上算力 ~100x、带宽 ~30x。
基础Q2. SIMT 和 SIMD 区别？
SIMD
CPU 的 SSE/AVX，1 条指令操作 1 个 vector（4-16 个数据）。寄存器 wide，编程粒度小。
SIMT
GPU 的 warp，1 条指令 32 个 thread 同时执行（每个 thread 算自己的数据）。编程模型更"线程感" — 你写 thread 代码，硬件自动 32 个并行。
进阶Q3. Warp Divergence 是什么？怎么避免？
是什么
同一 warp 内 32 thread 走不同分支（if-else），硬件必须串行执行两个分支，throughput 减半。
避免
① 按 warp 边界分配工作（threadIdx.x / 32 决定分支，而不是 threadIdx.x % 2）② 用预计算 mask 而不是条件赋值 ③ 把分支移到 host 决定（不同 kernel）
进阶Q4. Coalesced Memory Access 是什么？为什么重要？
是什么
同一 warp 32 thread 访问连续内存（如 thread i 访问 data[i]），硬件合并成 1 次 HBM 事务。反之分散访问要 32 次事务，性能差 32x。
进阶Q5. Shared Memory 跟 L1 Cache 区别？为什么要手动管理 Shared Memory？
硬件
同一块物理硬件（H100 上 228KB），可以划分给 L1 自动管理或 Shared 显式管理。
手动管理的价值
程序员知道数据复用模式（如 GEMM 的 tile），自己搬到 Shared 复用 N 次，比 L1 自动缓存（基于访问历史）更高效。FlashAttention 就是这个思想极致。
高频Q6. 怎么把 naive GEMM 优化到 cuBLAS 60%？步骤？
5 步
① Tiling：把 A/B 小块搬到 Shared Memory，让一个 block 反复用
② Register Tiling：每 thread 算 C 的 4×4 而不是 1×1，复用 register
③ Double Buffering：预取下一 tile，hide HBM 延迟
④ 避免 bank conflict：padding shared 数组宽度
⑤ 用 Tensor Core (WMMA)：再 +25% 性能
基础Q7. Tensor Core 是什么？怎么用？
是什么
GPU 内专门做"矩阵 × 矩阵"的硬件单元。1 cycle 完成 4×4 × 4×4（普通 SP 要 ~64 cycle）。Volta 引入，2026 已演化到 Blackwell FP4。
怎么用
① 用 cuBLAS / cuDNN / cutlass / FlashAttention 自动用 ② 矩阵尺寸对齐 8/16/32 ③ 用 FP16/BF16/FP8 而不是 FP32 ④ Nsight 看 utilization 验证
进阶Q8. Roofline 模型怎么用？arithmetic intensity 是什么？
AI
arithmetic intensity = FLOPs / Bytes，即每读 1 字节做多少 FLOPs。决定 kernel 是 compute-bound 还是 memory-bound。
Roofline
画 X=AI, Y=性能 的折线图。低 AI 区域受带宽限制（斜线），高 AI 区域受算力限制（平线）。kernel 实测性能落在折线下方说明有优化空间。
基础Q9. CUDA Stream 是什么？为什么需要？
是什么
Stream 是 GPU 上的"执行队列"。默认 stream 是串行的；多个 stream 可以并行执行。
用途
① 异步执行（kernel 跟 host 异步）② 并发多个 kernel（不同 stream）③ Hide memcpy 延迟（compute 跟 memcpy 用不同 stream）
高频Q10. 为什么 LLM decode 是 memory-bound？怎么优化？
为什么
Decode 每步只算 1 token (FLOPs ≈ 2P)，但要读整个 KV-Cache（GB 级）。arithmetic intensity = FLOPs / Bytes 极低 → 带宽瓶颈。
优化
① 减小要读的字节：GQA / MLA / KV 量化 ② 提高并发：Continuous Batching 摊薄 KV 读取（详见 Y02）③ 减少 step 数：Speculative Decoding ④ 跨节点：Prefill-Decode 分离
进阶Q11. Triton lang 跟 CUDA 区别？什么时候用？
区别
Triton 是 Python DSL，编译时自动 tiling / 自动 vectorize。代码量是 CUDA 的 1/5，性能 80-95%。
什么时候用
① 写中等复杂度 kernel（attention、norm、激活）② 快速原型 ③ 不追求 cuBLAS 那种极致优化。2024+ PyTorch 内部大量算子已迁移到 Triton。
实战Q12. 我有一个 PyTorch 模型跑得慢，怎么定位 GPU 性能瓶颈？
5 步
① torch.profiler：看哪个算子占比最高
② Nsight Systems：看 timeline，找 idle gap (CPU bottleneck) vs busy GPU (compute/memory bound)
③ Nsight Compute：选最热的 kernel 看 occupancy / cache hit / tensor core util / memory throughput
④ 识别瓶颈类型：算力低 → 用 Tensor Core / 量化 / fuse；带宽高 → 减读写（fuse / tiling）
⑤ 对比理论上限：实测 / Roofline 上限 = optimization headroom
