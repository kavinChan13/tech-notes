3. future / promise / async / packaged_task — 异步四件套

3.1 各自定位

组件角色典型用法
std::future<T>异步结果的消费端get() 取结果 (阻塞或重抛异常)
std::promise<T>异步结果的生产端set_value() / set_exception()

3.2 关系图

3.3 promise + future 经典模式

// 工作线程把结果通过 promise 传给主线程
std::promise<int> p;
std::future<int> f = p.get_future();

std::thread t([p = std::move(p)]() mutable {
    try {
        int result = heavy_work();
        p.set_value(result);
    } catch (...) {
        p.set_exception(std::current_exception());        // 异常也传过去
    }
});

try {
    int r = f.get();        // 阻塞等待 — 异常会被重抛
    use(r);
} catch (const std::exception& e) {
    LOG("failed: {}", e.what());
}
t.join();

3.4 std::async — 一行启动异步

// 启动策略 - 重要细节
auto f1 = std::async(std::launch::async,    work);   // 立刻新线程
auto f2 = std::async(std::launch::deferred, work);   // 同步执行 (在 get/wait 时)
auto f3 = std::async(work);                            // ⚠️ 由实现选择 — 不可控

// 默认策略不可控 → 推荐显式 std::launch::async
std::async 的两个大坑：
  1. 不指定策略时，"async | deferred" 由实现决定 → 可能根本没新线程，不并发
  2. std::async 返回的 future 析构时会等待任务完成(隐式 join 风格) → 看似并行，实际逐个完成。
    std::async(work1); std::async(work2); — 两个 future 都是临时对象 → 立即析构 → 串行! 
// ❌ 可能串行执行 (future 临时对象立即析构等待任务完成)
std::async(std::launch::async, work1);
std::async(std::launch::async, work2);

// ✅ 并行
auto f1 = std::async(std::launch::async, work1);
auto f2 = std::async(std::launch::async, work2);
f1.get(); f2.get();

3.5 packaged_task — 让任意函数变成 future-able 的任务

std::packaged_task<int(int)> task([](int x){ return x * x; });
std::future<int> f = task.get_future();

// 投递到线程池或裸线程
std::thread{std::move(task), 5}.detach();

int r = f.get();      // 25
// packaged_task 是线程池 submit 的核心组件

3.6 shared_future — 多个等待者

std::promise<Config> p;
std::shared_future<Config> sf = p.get_future().share();

// 多个 worker 都等同一个配置加载完成
for (int i = 0; i < 10; ++i) {
    std::thread{[sf]{ auto cfg = sf.get(); use(cfg); }}.detach();
}

// 主线程加载好后通知所有 worker
p.set_value(load_config());
"future / promise 怎么实现?" — 答：内部有共享状态（堆分配，含 mutex + cv + value/exception 槽 + ready 标志）。promise.set_value() 加锁、写值、置 ready、notify_all；future.get() 加锁、cv.wait(ready)、返回值或抛异常。引用计数管理共享状态生命周期。

4. std::shared_mutex 读写锁 (C++17)

4.1 用法

class SessionTable {
    std::shared_mutex mu_;
    std::unordered_map<SessionId, std::shared_ptr<Session>> m_;
public:
    std::shared_ptr<Session> find(SessionId id) const {
        std::shared_lock lk{mu_};               // 读锁 — 多个线程可同时持有
        auto it = m_.find(id);
        return it == m_.end() ? nullptr : it->second;
    }
    void insert(SessionId id, std::shared_ptr<Session> s) {
        std::unique_lock lk{mu_};               // 写锁 — 独占
        m_.emplace(id, std::move(s));
    }
    void erase(SessionId id) {
        std::unique_lock lk{mu_};
        m_.erase(id);
    }
};

4.2 性能 — 反直觉

读写锁不一定比 mutex 快！

场景shared_mutexmutex结论
读多写少 (90% 读)快慢shared_mutex 胜
读写各半≈≈差不多
读多但临界区极短慢快mutex 胜（shared_mutex 自身开销大）
大量并发读快慢shared_mutex 大胜

原因：shared_mutex 实现复杂，本身的 lock/unlock 开销 ~5× 普通 mutex。临界区极短时，"开销 > 节省的等待时间"。

4.3 锁升级 / 降级 — 标准不支持

// ❌ C++ 标准不支持锁升级 (read → write 升级)
std::shared_lock lk{mu_};
if (need_modify()) {
    // 必须先 unlock 再 unique_lock — 中间窗口可能其他线程改动数据
    lk.unlock();
    std::unique_lock wl{mu_};
    // 重新检查条件 (类似 DCLP)
    if (still_need_modify()) do_modify();
}
升级陷阱：两个线程同时持读锁 + 都想升级 → 死锁。所以 C++ 标准故意不提供升级接口。如果业务需要先读再写，一开始就拿写锁。

4.4 何时不要用 shared_mutex

  • 读临界区极短 (一个变量读取) → 直接 atomic
  • 读写比例接近 1:1 → 用普通 mutex
  • 对延迟敏感 → 用 RCU (Read-Copy-Update) 或 hazard pointer
  • 需要锁升级 → 重新设计 (单写者模式 / 单 writer queue)

5. call_once · latch · barrier · semaphore

5.1 std::call_once + once_flag

class DbConnection {
    static std::once_flag init_flag_;
    static std::unique_ptr<DbConnection> instance_;
public:
    static DbConnection& get() {
        std::call_once(init_flag_, []{
            instance_ = std::make_unique<DbConnection>();
        });
        return *instance_;
    }
};

但现代 C++11+ 推荐用局部 static（更简单且自动线程安全）：

static DbConnection& get() {
    static DbConnection inst;        // Magic Static — 标准保证线程安全初始化
    return inst;
}

5.2 std::latch (C++20) — 一次性倒数计数器

std::latch ready{N};                    // 等 N 次 count_down

// 每个 worker 准备好后
for (int i = 0; i < N; ++i) {
    std::thread{[&]{
        prepare();
        ready.count_down();
    }}.detach();
}

ready.wait();                          // 主线程等所有 worker 准备好
start_phase_2();

5.3 std::barrier (C++20) — 可重用的同步点

std::barrier sync{N, []{
    LOG("all threads at sync point");     // 完成回调 (可选)
}};

for (int phase = 0; phase < 10; ++phase) {
    std::thread{[&]{
        do_phase(phase);
        sync.arrive_and_wait();              // 等所有线程都到这一阶段
    }};
}

5.4 std::counting_semaphore / binary_semaphore (C++20)

// 限制并发数 — 类似令牌桶
std::counting_semaphore<100> sem{10};        // 最多 10 个并发, 上限 100

void handle_request() {
    sem.acquire();                            // 拿不到就阻塞
    ScopeExit _{[&]{ sem.release(); }};
    do_work();
}

// 信号量替代 condition_variable 的简单使用
std::binary_semaphore data_ready{0};         // 初值 0 (空)

// producer
prepare_data();
data_ready.release();

// consumer
data_ready.acquire();
use_data();
C++20 把 latch / barrier / semaphore 加进标准 → 之前必须手写或用 boost 的同步原语，现在开箱即用。

6. 线程池 — 完整实现演化 ⭐⭐⭐

线程池是面试必考 + 工程必有的组件。从最简到生产级。

6.1 V1 — 最简线程池（不能返结果）

class SimplePool {
    std::vector<std::thread> workers_;
    std::queue<std::function<void()>> tasks_;
    std::mutex mu_;
    std::condition_variable cv_;
    bool stop_ = false;
public:
    explicit SimplePool(size_t n) {
        for (size_t i = 0; i < n; ++i)
            workers_.emplace_back([this]{ loop(); });
    }
    ~SimplePool() {
        { std::lock_guard lk{mu_}; stop_ = true; }
        cv_.notify_all();
        for (auto& t : workers_) t.join();
    }
    void submit(std::function<void()> f) {
        { std::lock_guard lk{mu_}; tasks_.emplace(std::move(f)); }
        cv_.notify_one();
    }
private:
    void loop() {
        while (true) {
            std::function<void()> task;
            {
                std::unique_lock lk{mu_};
                cv_.wait(lk, [this]{ return stop_ || !tasks_.empty(); });
                if (stop_ && tasks_.empty()) return;
                task = std::move(tasks_.front());
                tasks_.pop();
            }
            task();
        }
    }
};

6.2 V2 — 支持返回值 + 异常透传 (生产级)

class ThreadPool {
    std::vector<std::jthread> workers_;
    std::queue<std::move_only_function<void()>> tasks_;        // C++23 (或 std::function)
    std::mutex mu_;
    std::condition_variable_any cv_;

public:
    explicit ThreadPool(size_t n = std::thread::hardware_concurrency()) {
        workers_.reserve(n);
        for (size_t i = 0; i < n; ++i)
            workers_.emplace_back([this](std::stop_token st){ loop(st); });
    }

    template<class F, class... Args>
    auto submit(F&& f, Args&&... args)
        -> std::future<std::invoke_result_t<F, Args...>>
    {
        using R = std::invoke_result_t<F, Args...>;

        // packaged_task 把返回值/异常都包进 future
        auto task = std::make_shared<std::packaged_task<R()>>(
            [f = std::forward<F>(f),
             args = std::make_tuple(std::forward<Args>(args)...)]() mutable {
                return std::apply(std::move(f), std::move(args));
            });

        std::future<R> fut = task->get_future();
        {
            std::lock_guard lk{mu_};
            tasks_.emplace([task]{ (*task)(); });
        }
        cv_.notify_one();
        return fut;
    }

    void stop() {
        for (auto& w : workers_) w.request_stop();
        cv_.notify_all();
    }

private:
    void loop(std::stop_token st) {
        std::stop_callback cb(st, [this]{ cv_.notify_all(); });
        while (!st.stop_requested()) {
            std::move_only_function<void()> task;
            {
                std::unique_lock lk{mu_};
                cv_.wait(lk, st, [this]{ return !tasks_.empty(); });
                if (st.stop_requested() && tasks_.empty()) return;
                task = std::move(tasks_.front());
                tasks_.pop();
            }
            try { task(); }
            catch (...) { /* packaged_task 内部已吞 — 异常存到 future */ }
        }
    }
};

// 使用 — 任意 callable + 任意参数, 自动获得 future
ThreadPool pool;
auto f1 = pool.submit([](int a, int b){ return a + b; }, 1, 2);
auto f2 = pool.submit([]{ throw std::runtime_error{"oops"}; });

int r = f1.get();             // = 3
try { f2.get(); }
catch (const std::exception& e) { LOG("task failed: {}", e.what()); }

6.4 线程池大小选择公式

经典 (Brian Goetz)：

N_threads = N_cpu * U_cpu * (1 + W/C)
其中:
  N_cpu = CPU 核数
  U_cpu = 目标 CPU 利用率 (0~1)
  W/C   = 等待时间 / 计算时间 比值
  • CPU 密集（W/C ≈ 0）→ 线程数 ≈ N_cpu
  • IO 密集（W/C 高，比如 10）→ 线程数 ≈ N_cpu × 10+
  • 实际：用监控数据调优 (任务延迟 / 队列长度 / CPU 使用率)

7. Producer-Consumer 模式

用有界队列解耦生产者 / 消费者速率。这是所有事件驱动系统的基础。

7.1 BoundedQueue（生产级实现）

template<class T>
class BoundedQueue {
    std::queue<T> q_;
    std::mutex mu_;
    std::condition_variable cv_not_full_, cv_not_empty_;
    size_t cap_;
    bool closed_ = false;
public:
    explicit BoundedQueue(size_t cap) : cap_(cap) {}

    // 阻塞 push — 满了等
    bool push(T v) {
        std::unique_lock lk{mu_};
        cv_not_full_.wait(lk, [this]{ return closed_ || q_.size() < cap_; });
        if (closed_) return false;
        q_.push(std::move(v));
        cv_not_empty_.notify_one();
        return true;
    }

    // 带超时的 push
    bool push_for(T v, std::chrono::milliseconds tm) {
        std::unique_lock lk{mu_};
        if (!cv_not_full_.wait_for(lk, tm, [this]{
            return closed_ || q_.size() < cap_;
        })) return false;
        if (closed_) return false;
        q_.push(std::move(v));
        cv_not_empty_.notify_one();
        return true;
    }

    // 阻塞 pop — 关闭且空时返回 nullopt
    std::optional<T> pop() {
        std::unique_lock lk{mu_};
        cv_not_empty_.wait(lk, [this]{ return closed_ || !q_.empty(); });
        if (q_.empty()) return std::nullopt;       // closed + drained
        T v = std::move(q_.front()); q_.pop();
        cv_not_full_.notify_one();
        return v;
    }

    void close() {
        std::lock_guard lk{mu_};
        closed_ = true;
        cv_not_full_.notify_all();
        cv_not_empty_.notify_all();
    }

    size_t size() const { std::lock_guard lk{mu_}; return q_.size(); }
};

7.2 使用模式

BoundedQueue<Job> q{1024};

// Producer
std::jthread producer{[&]{
    while (running) {
        auto job = make_job();
        if (!q.push(std::move(job))) break;     // closed → 退出
    }
    q.close();      // 通知 consumer 完工
}};

// Consumer
std::jthread consumer{[&]{
    while (auto job = q.pop()) {
        process(*job);
    }
    // pop 返回 nullopt → 队列已空且关闭 → 干净退出
}};

7.3 多 Producer / 多 Consumer

  • BoundedQueue 内部 mutex + cv 已支持任意数量 P/C
  • 性能瓶颈：单一锁。替代方案：
    • SPSC 单生产单消费 → 无锁 ring buffer (~10× 快)
    • MPMC → moodycamel::ConcurrentQueue 或 folly::MPMCQueue
    • 多分片队列 + 工作窃取（见 §11）

7.4 Backpressure (背压) 设计

// 几种 push 满时的策略
enum class FullPolicy {
    Block,            // 阻塞等待 (默认)
    Drop,             // 丢弃新任务
    DropOldest,       // 丢弃最老任务
    Reject,           // 立刻返回失败 (调用方决定重试)
    Slowdown          // 延迟 push (sleep)
};
背压是分布式系统稳定性的关键 — 如果生产端速率长期高于消费端，必须有反馈机制，否则队列无限增长 → OOM。

8. Reader-Writer 模式 — 多种实现

8.1 经典 shared_mutex

class RouteTable {
    std::shared_mutex mu_;
    std::unordered_map<std::string, Handler> routes_;
public:
    Handler find(const std::string& path) const {
        std::shared_lock lk{mu_};
        auto it = routes_.find(path);
        return it == routes_.end() ? nullptr : it->second;
    }
    void add(const std::string& path, Handler h) {
        std::unique_lock lk{mu_};
        routes_[path] = std::move(h);
    }
};

8.2 Copy-on-Write — 极致读性能

"读永不阻塞，写换整个数据结构"。读路径只是原子加载一个 shared_ptr → 几 ns。

class RouteTableCOW {
    using Map = std::unordered_map<std::string, Handler>;
    std::atomic<std::shared_ptr<const Map>> current_;
    std::mutex write_mu_;        // 仅写者间互斥
public:
    RouteTableCOW() : current_(std::make_shared<Map>()) {}

    Handler find(const std::string& path) const {
        auto snapshot = current_.load();         // 原子加载, 几 ns
        auto it = snapshot->find(path);
        return it == snapshot->end() ? nullptr : it->second;
    }

    void add(const std::string& path, Handler h) {
        std::lock_guard lk{write_mu_};
        auto snap = current_.load();
        auto next = std::make_shared<Map>(*snap);   // 复制!
        (*next)[path] = std::move(h);
        current_.store(next);                       // 原子替换
    }
};

适用场景：读 ≫ 写（路由表、配置、metadata）。不适用大数据集（每次写都全量复制）。

8.3 RCU (Read-Copy-Update) — 极端读优化

Linux 内核大量使用。读端零开销（不加锁），写端复制 + 延迟回收。C++ 标准库无原生支持，但 folly / userspace-rcu 提供。

8.4 选型

场景方案
读写比例 90:10，临界区 > 100nsshared_mutex
读 ≫ 写，数据集小COW + atomic shared_ptr
读 ≫ 写，数据集大分片 + per-shard shared_mutex
极端读延迟敏感 (< 50ns)RCU
读临界区极短 (单变量)atomic / atomic_ref

9. Pipeline 流水线模式

把任务分阶段，每阶段一个/多个线程，阶段间用队列连接 → 高吞吐 + 资源隔离。

9.1 实现骨架

template<class In, class Out>
class Stage {
    BoundedQueue<In>& in_;
    BoundedQueue<Out>& out_;
    std::function<Out(In)> transform_;
    std::vector<std::jthread> workers_;
public:
    Stage(BoundedQueue<In>& in, BoundedQueue<Out>& out,
          std::function<Out(In)> t, size_t n_workers)
        : in_(in), out_(out), transform_(std::move(t))
    {
        for (size_t i = 0; i < n_workers; ++i)
            workers_.emplace_back([this](std::stop_token st){ loop(st); });
    }
private:
    void loop(std::stop_token st) {
        while (!st.stop_requested()) {
            auto item = in_.pop();
            if (!item) {                            // 上游关闭
                out_.close();
                return;
            }
            try {
                out_.push(transform_(std::move(*item)));
            } catch (const std::exception& e) {
                LOG_ERROR("stage error: {}", e.what());
            }
        }
    }
};

// 组装 4 阶段流水线
BoundedQueue<RawBuf>  q1{1024};
BoundedQueue<Parsed>   q2{1024};
BoundedQueue<Valid>    q3{1024};
BoundedQueue<Output>   q4{1024};

Stage<RawBuf, Parsed> s1{q1, q2, parse,     2};
Stage<Parsed, Valid>  s2{q2, q3, validate,  4};   // 瓶颈阶段多 worker
Stage<Valid, Output>  s3{q3, q4, transform, 2};
Stage<Output, void>   s4{q4, ..., write,    1};

9.2 性能调优要点

  • 找瓶颈阶段：监控每阶段队列长度 — 长的就是瓶颈
  • 瓶颈阶段加 worker（如果可并行）
  • 批处理：每次 pop N 个一起处理（减少 cv 唤醒次数）
  • 零拷贝：用 shared_ptr 在阶段间传递大对象
  • 避免假共享：不同阶段的队列对象间隔 cache line

10. Active Object / Actor — 异步消息驱动对象

每个对象一个 mailbox + 一个专属线程 → 内部状态无锁访问 (only one thread touches state)。

10.1 Active Object 实现

class Counter {
    std::queue<std::function<void()>> mailbox_;
    std::mutex mu_;
    std::condition_variable cv_;
    std::jthread worker_;
    int count_ = 0;       // 只在 worker 线程内修改

public:
    Counter() : worker_([this](std::stop_token st){ loop(st); }) {}

    // 投递消息 → 立即返回 future
    std::future<int> increment() {
        return post([this]{ return ++count_; });
    }
    std::future<int> get() {
        return post([this]{ return count_; });
    }

private:
    template<class F>
    auto post(F f) -> std::future<std::invoke_result_t<F>> {
        using R = std::invoke_result_t<F>;
        auto task = std::make_shared<std::packaged_task<R()>>(std::move(f));
        auto fut = task->get_future();
        { std::lock_guard lk{mu_}; mailbox_.emplace([task]{ (*task)(); }); }
        cv_.notify_one();
        return fut;
    }

    void loop(std::stop_token st) {
        std::stop_callback cb(st, [this]{ cv_.notify_all(); });
        while (!st.stop_requested()) {
            std::function<void()> msg;
            {
                std::unique_lock lk{mu_};
                cv_.wait(lk, st, [this]{ return !mailbox_.empty(); });
                if (mailbox_.empty()) return;
                msg = std::move(mailbox_.front()); mailbox_.pop();
            }
            msg();
        }
    }
};

// 使用 — 完全无锁的多线程访问
Counter c;
std::vector<std::future<int>> results;
for (int i = 0; i < 1000; ++i)
    results.emplace_back(c.increment());
for (auto& f : results) f.wait();

11. Work-Stealing 工作窃取

11.1 动机

普通线程池一个全局任务队列 → 高并发下队列锁是瓶颈。

Work-Stealing：每个 worker 自己的队列 (deque)，自己 push/pop 自己的尾部；空了去偷别人队列的头。

11.2 关键设计

  • 每个 worker 持有 双端队列
  • 本 worker 从 deque 尾 push/pop（栈式，缓存友好）
  • 空闲 worker 从其他人 deque 头 偷（FIFO 减少冲突）
  • 双端队列内部用原子操作实现 lock-free（Chase-Lev deque）

11.3 工程框架

  • Intel TBB：成熟工业级
  • folly::CPUThreadPoolExecutor：Facebook 实现
  • std::execution::par 后端：标准并行算法
  • 自实现：从零写 Chase-Lev deque 难度极大，建议用现成库
// 概念示意 — 实际生产用 TBB / folly
class WorkStealingPool {
    std::vector<WorkerQueue> queues_;        // 每 worker 一个 lock-free deque
    std::vector<std::jthread> workers_;
    std::atomic<size_t> next_submit_{0};
public:
    void submit(std::function<void()> f) {
        auto idx = next_submit_.fetch_add(1) % queues_.size();
        queues_[idx].push_back(std::move(f));    // round-robin 投递
    }
private:
    void worker_loop(size_t id, std::stop_token st) {
        auto& my_q = queues_[id];
        while (!st.stop_requested()) {
            auto task = my_q.pop_back();          // 自己的尾, lock-free
            if (task) { (*task)(); continue; }

            // 偷别人
            for (size_t i = 0; i < queues_.size(); ++i) {
                if (i == id) continue;
                auto stolen = queues_[i].pop_front();   // 别人的头
                if (stolen) { (*stolen)(); break; }
            }
            // 都没活 — sleep / spin / wait
        }
    }
};
核心收益：① 大多数 push/pop 是 lock-free；② 缓存友好（栈式访问 + 偷别人是少数事件）；③ 自动负载均衡。

12.1 与线程池/future 的关系

线程池 + future 处理"任务 → 结果"，但组合多个异步操作仍是噩梦（嵌套 .then().then().then()）。协程让异步代码看起来像同步。

12.3 简化 Task 实现骨架

template<class T>
class Task {
public:
    struct promise_type {
        std::optional<T> value;
        std::exception_ptr eptr;

        Task get_return_object() {
            return Task{std::coroutine_handle<promise_type>::from_promise(*this)};
        }
        std::suspend_never initial_suspend() noexcept { return {}; }
        std::suspend_always final_suspend() noexcept { return {}; }
        void return_value(T v) { value = std::move(v); }
        void unhandled_exception() { eptr = std::current_exception(); }
    };

    std::coroutine_handle<promise_type> h_;

    // awaiter — 让 co_await Task 工作
    bool await_ready() const { return h_.done(); }
    void await_suspend(std::coroutine_handle<> awaiting) { /* 调度 */ }
    T await_resume() {
        auto& p = h_.promise();
        if (p.eptr) std::rethrow_exception(p.eptr);
        return std::move(*p.value);
    }
};

12.5 应用场景

适合

  • 大量并发 IO（万级以上连接）
  • 异步链式操作（微服务调用）
  • 流式数据处理（generator）
  • 状态机逻辑表达

  • CPU 密集计算 — 还是要线程池

class AggregateError : public std::runtime_error {
    std::vector<std::exception_ptr> errors_;
public:
    AggregateError(std::vector<std::exception_ptr> e)
        : std::runtime_error{"aggregate"}, errors_(std::move(e)) {}
    const auto& errors() const { return errors_; }
};

void wait_all_or_throw(std::vector<std::future<void>>& fs) {
    std::vector<std::exception_ptr> errs;
    for (auto& f : fs) {
        try { f.get(); }
        catch (...) { errs.push_back(std::current_exception()); }
    }
    if (errs.size() == 1) std::rethrow_exception(errs[0]);
    if (errs.size() > 1)  throw AggregateError{std::move(errs)};
}

14. 死锁的预防与检测

(详细诊断流程见 cpp_deadlock_debugging_guide.html — 本节聚焦设计层面的预防。)

14.1 6 条防御规则

  1. 锁的全局顺序：所有线程按同一顺序获取锁 (例如按地址排序)
  2. 同时获取多锁用 std::lock / std::scoped_lock (C++17)
  3. 锁的粒度尽量小：只锁数据，不锁 IO/计算
  4. RAII 锁守卫：lock_guard / unique_lock，永远不裸 lock/unlock
  5. 不在持锁时调用未知代码（用户回调、虚函数）
  6. 不持锁时阻塞等待（不在持锁时 cv.wait 之外的等待）

14.2 同时锁多个 mutex 的正确写法

// ❌ 错误 - 顺序不一致 → 死锁
std::mutex a, b;
void thread1() { std::lock_guard la{a}; std::lock_guard lb{b}; }
void thread2() { std::lock_guard lb{b}; std::lock_guard la{a}; }    // 反向

// ✅ C++17 scoped_lock - 内部用 deadlock-avoiding 算法
void thread1() { std::scoped_lock l{a, b}; }
void thread2() { std::scoped_lock l{b, a}; }    // 同样 OK

// 旧 C++11 写法
std::lock(a, b);
std::lock_guard la{a, std::adopt_lock};
std::lock_guard lb{b, std::adopt_lock};

14.3 锁层次 (Lock Hierarchy)

class HierarchicalMutex {
    std::mutex internal_;
    const uint64_t level_;
    static thread_local uint64_t current_level_;
    uint64_t prev_level_;
public:
    explicit HierarchicalMutex(uint64_t lvl) : level_(lvl) {}

    void lock() {
        if (current_level_ <= level_)
            throw std::logic_error{"violates hierarchy"};
        internal_.lock();
        prev_level_ = current_level_;
        current_level_ = level_;
    }
    void unlock() {
        current_level_ = prev_level_;
        internal_.unlock();
    }
};
thread_local uint64_t HierarchicalMutex::current_level_ = UINT64_MAX;

// 设定层次：高层锁先持有, 低层锁后持有
HierarchicalMutex route_mu{1000};        // 高层
HierarchicalMutex stats_mu{100};         // 低层
// 强制 route_mu 先于 stats_mu — 反序时编译/运行报错

14.4 死锁检测工具

  • ThreadSanitizer (TSan) — gcc/clang -fsanitize=thread，运行时检测潜在数据竞争 + 部分死锁
  • Helgrind / DRD — Valgrind 工具集
  • gdb + thread apply all bt — 进程已死锁时定位
  • perf trace + lockstat — 长时间锁等待

15.2 异步日志 — Producer-Consumer + batch flush

class AsyncLogger {
    BoundedQueue<LogEntry> q_{16384};
    std::jthread flusher_;
public:
    AsyncLogger(const std::string& path) {
        flusher_ = std::jthread([this, path](std::stop_token st){
            std::ofstream f{path};
            std::vector<LogEntry> batch;
            batch.reserve(256);
            while (!st.stop_requested()) {
                // 批量取 — 减少 cv 唤醒次数
                if (auto e = q_.pop()) {
                    batch.push_back(std::move(*e));
                    while (batch.size() < 256) {
                        auto e2 = q_.try_pop();   // 非阻塞
                        if (!e2) break;
                        batch.push_back(std::move(*e2));
                    }
                    for (const auto& e : batch) f << e.format() << '\n';
                    f.flush();
                    batch.clear();
                }
            }
        });
    }
    void log(LogEntry e) { q_.push(std::move(e)); }
};

15.4 限流器 — 令牌桶 (atomic)

class TokenBucket {
    const double rate_;          // tokens / sec
    const double capacity_;
    std::atomic<double> tokens_;
    std::atomic<std::chrono::steady_clock::time_point> last_;
public:
    TokenBucket(double r, double c)
        : rate_(r), capacity_(c), tokens_(c), last_(std::chrono::steady_clock::now()) {}

    bool try_acquire(double n = 1) {
        auto now = std::chrono::steady_clock::now();
        auto prev = last_.exchange(now);
        double dt = std::chrono::duration<double>(now - prev).count();
        double avail = std::min(capacity_, tokens_.load() + dt * rate_);
        if (avail < n) {
            tokens_.store(avail);
            return false;
        }
        tokens_.store(avail - n);
        return true;
    }
};
// 注: 这是简化版 - 严格的并发安全令牌桶需要 CAS 循环

15.5 优雅停服

int main() {
    Server server;
    server.start();

    // 信号 → atomic flag → 主循环检测
    static std::atomic<bool> stopping{false};
    signal(SIGTERM, [](int){ stopping = true; });

    // 优雅停服 4 步

}

16. 易错点速查（10 个高频陷阱）

  1. std::async 临时 future 串行 
    std::async(work1); std::async(work2);     // ⚠️ 串行!
auto f1 = std::async(...); auto f2 = std::async(...);    // ✅ 并行
  2. cv.wait 不带谓词 → 虚假唤醒 
    cv.wait(lk);                   // ⚠️ 虚假唤醒不退
cv.wait(lk, [&]{ return ready; });    // ✅
  3. 持锁调用回调 → 死锁 
    std::lock_guard lk{mu_};
user_callback();          // ⚠️ 用户回调可能拿其他锁 → 死锁
  4. thread 析构没 join → terminate 
    { std::thread t{work}; }    // ⚠️ 析构 abort()
{ std::jthread t{work}; }   // ✅ C++20 自动 join
  5. 引用捕获跨线程 → 悬空 
    {
    int x = 42;
    pool.submit([&]{ use(x); });    // ⚠️ x 可能已销毁
}
  6. future.get() 多次 → terminate
    future 只能 get 一次。多线程共享要用 shared_future。
  7. 持锁时间过长
    锁内做 IO / sleep / 大计算 → 其他线程长时间等待。
  8. shared_mutex 临界区太短
    不如普通 mutex 快 (shared_mutex 自身开销大)。
  9. 线程数过多
    上下文切换开销 ≫ 计算收益。线程数 ≈ N_cpu × (1 + W/C)。
  10. atomic 误用 memory_order_relaxed
    不能保证 happens-before，导致同步失效。详见 cpp_memory_model.html。

关键点：

  1. worker 池 + 任务队列 + mutex + cv (基础)
  2. submit 是变参模板：template<class F, class... Args>
  3. 用 packaged_task<R()> 把 callable + 参数 wrap，自动捕获异常到 future
  4. 用 shared_ptr<packaged_task> 让 lambda 能拷贝（packaged_task 不可拷贝）
  5. std::invoke_result_t 推导返回类型
  6. std::apply + tuple 实现变参完美转发
  7. jthread + stop_token 实现 RAII 停止
  8. 使用方：auto fut = pool.submit(func, args...); fut.get(); 自动重抛

看 §6.2 完整代码。
