1. 引言 — 为什么 C++ 安全是特殊话题

1.1 内存不安全语言的代价

过去 20 年的所有大型 CVE 数据库统计（Microsoft / Google / Chromium 内部报告）一致显示：70% 的高危漏洞源于"内存不安全"—— buffer overflow / use-after-free / double-free / 整数溢出。这些几乎全在 C/C++ 代码里。

  • Microsoft（2019）：MSRC 数据 ~70% 的 CVE 是 memory safety bug
  • Chromium（2020）：70% high+ severity bug 是 memory safety
  • Android（2022）：迁移到 Rust 后新代码 memory safety bug 几乎归零
  • 美国白宫 ONCD（2024）：建议政府软件优先用 memory-safe 语言

1.2 但你不能马上重写 C++ 代码

大型企业有几百万到几千万行 C++ 代码：浏览器内核、数据库引擎、操作系统、游戏引擎、HPC、嵌入式。必须想办法让现有 C++ 代码安全—— 这就是"硬化"（hardening）的意义。

1.3 安全硬化的"洋葱模型"

① 写不出漏洞

现代 C++ 实践、CppCoreGuidelines、std 安全 API、智能指针、static_assert

② 编译期/链接期阻止

-D_FORTIFY_SOURCE / -fstack-protector / RELRO / NOW / CFI / static analysis

③ 运行时缓解

ASLR / PIE / Stack canary / HWASan / GWP-ASan / W^X / seccomp

1.4 本指南的覆盖范围

  • 5 大类 C++ 安全漏洞 + 真实 CVE 案例
  • 编译器 / 链接器 / 系统级所有能上的硬化开关

类别典型 CVE 比例触发 / 利用
① 缓冲区溢出~30%越界写 → 覆盖返回地址 / 函数指针 → RCE

3. 整数溢出与有符号陷阱

3.1 有符号溢出 = UB

int x = INT_MAX;
int y = x + 1;          // ❌ UB! C++ 标准说"有符号整型溢出是未定义行为"
// 编译器可能:
//   - x86 上"wrap-around"成 INT_MIN (硬件行为)
//   - 优化器假设"x > 0 永远成立"→ 删除安全检查 → 漏洞

// 经典：编译器优化掉看似多余的检查
if (x + 1 < x) ...;       // ❌ 编译器: "x+1 不会比 x 小" → 整个 if 被删

3.2 无符号溢出 = 良定义的 wrap-around

unsigned u = 0;
unsigned v = u - 1;     // = 4294967295 (UINT_MAX), 良定义

// 但仍是安全 bug 源:
void copy(char* dst, const char* src, size_t n) {
    if (n - 1 < MAX_SIZE) {          // ❌ n=0 时 n-1 = SIZE_MAX, 永远 ≥ MAX_SIZE
        memcpy(dst, src, n);
    }
}

3.3 真实 CVE：整数溢出导致堆溢出

// 简化的真实 bug 模式 (CVE-2002-0639, CVE-2018-6789 ...)
void process(size_t n, char* user_data) {
    char* buf = (char*)malloc(n * sizeof(MyHeader));  // ❌ n * sizeof 可能溢出
    // 若 n = SIZE_MAX/2 + 1, n*16 = 0 (溢出归零) → malloc(0)
    // 后续 memcpy(buf, src, n*16) 写了一大堆 → 堆溢出
}

// ✅ 修复：用安全乘法或 C++20 std::numeric_limits 检查
if (n > SIZE_MAX / sizeof(MyHeader)) throw std::overflow_error{""};
char* buf = (char*)malloc(n * sizeof(MyHeader));

3.4 整数推广陷阱

signed char sc = -1;
unsigned int ui = sc;       // ★ ui = 0xFFFFFFFF, 不是 -1
// 因为: sc 先 sign-extend 到 signed int (= -1, 0xFFFFFFFF as bits)
// 再转为 unsigned → 保持 bits → 0xFFFFFFFF

if (idx < vec.size()) ...;    // idx 是 int，size 是 size_t (unsigned)
//                                  idx < 0 时被推广为大正数 → 永远 false → 越界访问

3.5 现代 C++ 工具帮忙

// C++20 std::cmp_xxx — 安全跨符号比较
if (std::cmp_less(idx, vec.size())) ...;       // 任意符号都正确

// C++23 std::saturate_cast — 饱和转换
int v = std::saturate_cast<int>(big_uint);    // 超 INT_MAX 钳到 INT_MAX

// __builtin_add_overflow / __builtin_mul_overflow (GCC/Clang)
size_t total;
if (__builtin_mul_overflow(n, sizeof(T), &total)) throw std::overflow_error{};
auto* buf = malloc(total);

// C++26 contracts (P2900)
int add(int a, int b) pre(a <= INT_MAX - b);

3.6 UBSan 检测整数溢出

g++ -fsanitize=signed-integer-overflow,unsigned-integer-overflow ...
// 运行时溢出立即报告 + 调用栈

// 极致硬化：把溢出变成 abort 而不是 wrap-around
clang++ -fsanitize=integer -fno-sanitize-recover=integer ...
所有解析用户输入的 size / length的代码都必须先验证 + 用安全乘法。这是 25 年来 C/C++ 漏洞排行榜前三。

4. 缓冲区溢出 — 栈 / 堆 / 全局

4.1 三种缓冲区溢出

类型位置典型利用
栈溢出函数栈帧的局部数组覆盖返回地址 → RCE
堆溢出malloc/new 的内存覆盖相邻 chunk 元数据 → glibc 利用
BSS/全局溢出全局变量覆盖函数指针 / vtable 指针

4.2 经典栈溢出 (1996 Morris Worm 已用)

// ❌ 经典反例
void vulnerable(const char* user_input) {
    char buf[64];
    strcpy(buf, user_input);     // ★ 无边界 → 长输入溢出 → 覆盖返回地址
}

// 攻击者输入: AAAA...AAAA[shellcode 地址] → RCE

4.3 危险 C 函数 → 安全替代

不安全替代
strcpy(dst, src)strncpy_s / std::string
strcat(dst, src)strncat_s
sprintf(buf, fmt, ...)snprintf / std::format_to_n
gets(buf)禁用（C11 已删）→ fgets / std::getline
scanf("%s", buf)%.NS with width / cin >>
strlen(s) on untrustedstrnlen 限制最大长度
memcpy(d, s, n)验证 n 后用，或 std::copy_n

4.4 现代 C++ 写法

// ✅ 用 std::array / std::span 永远带 size 信息
void safe(std::span<const char> input) {
    std::array<char, 64> buf{};
    auto n = std::min(input.size(), buf.size() - 1);
    std::copy_n(input.begin(), n, buf.begin());
    buf[n] = 0;
}

// ✅ std::string 全自动
void safe2(const std::string& input) { ... }

4.5 堆溢出与 chunk metadata

// glibc heap chunk 头部:
// [prev_size | size_and_flags] [user data]

char* a = (char*)malloc(64);
char* b = (char*)malloc(64);
strcpy(a, very_long_input);      // ❌ 越界 → 覆盖 b 的 chunk header
free(b);                          // → glibc 解 header → 触发 fastbin / tcache 利用

典型利用技术：tcache poisoning、unsorted bin attack、House of Force 等（pwning 入门必学）。

4.6 防御

  • 编译期：-D_FORTIFY_SOURCE=2/3（见 §8）
  • 运行期：ASLR + Stack canary（见 §10）
  • 测试期：ASan（彻底检测越界）
  • fuzz：libFuzzer / AFL++ 找隐藏溢出

5. UAF / Double-Free / Type Confusion

5.1 UAF (Use After Free)

int* p = new int(42);
delete p;
*p = 100;                    // ❌ UAF — 访问已释放内存
// 攻击者: 触发 delete 后立即分配同大小对象 → p 指向新对象 → 控制内容

5.2 经典 UAF 模式：异步回调持有"裸 this"

class Session {
    asio::socket s;
public:
    void read() {
        s.async_read(buf, [this](Error e) {        // ❌ this 可能已死
            process(...);
        });
    }
};

// ✅ 用 enable_shared_from_this（智能指针指南 §8）
class Session : public std::enable_shared_from_this<Session> {
    void read() {
        s.async_read(buf, [self = shared_from_this()](Error e) {
            self->process(...);                           // ✅ self 保活
        });
    }
};

5.3 Type Confusion (C++ 多态独有)

class Base { virtual void f() {} };
class DerivedA : public Base { int a; };
class DerivedB : public Base { int b[100]; };

Base* p = new DerivedA{};
DerivedB* q = (DerivedB*)p;     // ❌ static_cast 错（DerivedA 不是 DerivedB）
q->b[99] = 0;                  // 越界写 + 类型混淆 → 利用

// ✅ 用 dynamic_cast（虽然慢但安全）
if (auto* q = dynamic_cast<DerivedB*>(p)) { ... }

5.4 Chrome 的统计

Chrome 公开数据：高危漏洞约 50% UAF。为此 Google 投入巨资搞 PartitionAlloc + MiraclePtr（智能引用，UAF 自动转 noop）+ MTE（ARM 内存标签）+ 迁移部分到 Rust。详见 Chrome Memory Safety 系列博文。

5.5 防御

  • 智能指针铁律：原始 new/delete 一律禁止（CppCoreGuidelines R.1）
  • ASan / HWASan / MTE：能秒发现 UAF
  • GWP-ASan：生产采样，找出来自真实流量的 UAF（详见 Sanitizers §14b）
  • 架构层：用 Rust / Swift 写新代码（Memory Safe by design）

6. 格式化字符串漏洞

// ❌ 经典反例（CVE 还在持续出）
char buf[128];
printf(user_input);                    // 把用户输入当作格式串
// 攻击者输入 "%x %x %x %x" → 泄漏栈内容
// 输入 "%n" → 任意地址写

// ✅ 正确
printf("%s", user_input);            // 永远把格式串写死

6.1 现代 C++ 永久解决方案

// C++20 std::format — 编译期格式串检查
auto s = std::format("id={} name={}", id, name);   // ✅ 类型安全 + 编译期检查格式

// 用户字符串只作为参数，不能作为格式串
std::format(user_input, ...);    // ❌ 编译错（不接 runtime 格式串）
std::format(std::runtime_format(user_input), ...);  // C++26 显式 opt-in

// C++23 std::println — 一行搞定
std::println("id={} name={}", id, name);

6.2 编译器警告

g++ -Wformat -Wformat-security -Werror ...    # printf(user_input) 直接报错
新代码全用 std::format / std::print，格式串漏洞从源头消除。

7. TOCTOU 与命令注入

7.1 TOCTOU (Time-Of-Check vs Time-Of-Use)

// ❌ 经典 race condition
if (access("/tmp/file", R_OK) == 0) {     // check: 我有读权限
    // ★ 攻击者在这个窗口期把 /tmp/file 改成 /etc/shadow 的 symlink
    int fd = open("/tmp/file", O_RDONLY);   // use: 实际打开的是 /etc/shadow
}

// ✅ 用文件描述符 + faccessat 避开 race
int fd = open("/tmp/file", O_RDONLY | O_NOFOLLOW);
if (fd < 0) { ... }
// 现在所有操作都基于 fd，与路径解耦，无 race

7.2 命令注入

// ❌
std::string cmd = "convert " + user_file + " out.jpg";
system(cmd.c_str());            // user_file = "; rm -rf /" → 灾难

// ✅ 用 fork + exec 配合 argv 数组（无 shell 解释）
pid_t pid = fork();
if (pid == 0) {
    const char* argv[] = {"convert", user_file.c_str(), "out.jpg", nullptr};
    execvp("convert", (char**)argv);
    _exit(127);
}
// 参数作为独立元素传，不被 shell 解释

7.3 其他注入

  • SQL 注入：永远用 prepared statement / parameterized query
  • LDAP / XML 注入：白名单字符 + 转义
  • 路径遍历：检查 .. + canonical path + sandbox 文件系统访问

8. 编译期硬化开关 ⭐

这是最高 ROI的安全提升 —— 改一行编译选项，整个二进制变安全得多。

8.1 完整的"硬化" CXXFLAGS（生产推荐）

# 生产服务通用硬化选项（GCC/Clang）
CXXFLAGS="-O2 -g -Wall -Wextra -Wpedantic \
  -Wformat -Wformat-security \
  -Wconversion -Wsign-conversion \
  -Wnull-dereference \
  -D_GLIBCXX_ASSERTIONS \
  -D_FORTIFY_SOURCE=3 \
  -fstack-protector-strong \
  -fstack-clash-protection \
  -fcf-protection=full \
  -fPIE -pie \
  -fno-delete-null-pointer-checks \
  -fno-strict-overflow \
  -fhardened"             // GCC 14+ 集成开关

8.2 关键开关详解

开关作用开销
-D_FORTIFY_SOURCE=2/3在编译期+运行期检查 memcpy/strcpy/sprintf 等 size 是否安全；3 比 2 更激进~1-3%
-D_GLIBCXX_ASSERTIONSlibstdc++ 的 vector::operator[] 等加边界检查~5-10% (debug-like)
-fstack-protector-strong栈金丝雀 (canary)，函数返回前检查；strong 比 normal 覆盖更多函数~1-3%
-fstack-clash-protection防止 stack clash 攻击（大栈分配跳过 guard page）极小
-fcf-protection=fullIntel CET：indirect branch tracking + shadow stack~2%
-fPIE -piePosition-Independent Executable，启用 ASLR~1%
-fno-delete-null-pointer-checks禁止编译器删除"看似多余"的 NULL 检查（kernel 已被坑过）—
-ftrivial-auto-var-init=zero自动初始化所有 uninitialized 变量为 0~1-3%
-fsanitize=undefinedUBSan（Debug / 早期 CI 用，生产可只开少数子集）10-30%
-fhardened (GCC 14+)一键开启上面大部分（一行替代一坨）—

8.3 _FORTIFY_SOURCE 实例

char buf[10];
strcpy(buf, very_long_string);   // 编译时 GCC 看到溢出 → 编译警告
                                  // 运行时若 size 未知, glibc 替换为 __strcpy_chk → 检测后 abort

# 编译输出:
# warning: '__builtin_strcpy' writing 31 bytes into a region of size 10 [-Wstringop-overflow=]

8.4 -ftrivial-auto-var-init=zero —— 自动初始化

void f() {
    int x;
    return x;        // 未初始化, 但 -ftrivial-auto-var-init=zero 后自动 = 0
}
// 收益: 消除"未初始化变量泄漏栈数据"类漏洞 (Chrome / Linux kernel 都已启用)

8.5 警告开关（一定要开 -Werror）

-Wall -Wextra -Wpedantic
-Wformat -Wformat-security -Wformat-overflow=2
-Wstringop-overflow=4 -Wstringop-truncation
-Wnull-dereference -Wundef -Wshadow
-Wconversion -Wsign-conversion
-Wold-style-cast -Wcast-align
-Wuseless-cast
-Werror              # 警告一律当错误，强制修复

8.6 Clang 专属：-fsanitize=*

# Clang CFI — 控制流完整性
clang++ -fsanitize=cfi -flto -fvisibility=hidden ...
# 检查间接调用 / 虚函数调用的目标是合法函数

9. 链接期保护 (RELRO / NOW / 等)

9.1 RELRO + BIND_NOW

# 完整 link flags
LDFLAGS="-Wl,-z,relro -Wl,-z,now \
  -Wl,-z,noexecstack \
  -Wl,-z,separate-code \
  -Wl,--as-needed"
选项作用
-Wl,-z,relro启动时把 GOT / 全局变量段标为只读，阻止 ROP / GOT overwrite
-Wl,-z,now+ 启动时立即解析所有动态符号，使 GOT 完全只读（"Full RELRO"）
-Wl,-z,noexecstack栈段标为不可执行（NX bit）
-Wl,-z,separate-code分离 .text 和数据段，提高 ASLR 熵
-Wl,--as-needed只链接实际用到的 .so，减少攻击面

9.2 检查二进制是否启用了硬化

# checksec - 一键看所有缓解措施
$ checksec --file=myapp
RELRO           STACK CANARY      NX            PIE             RPATH      RUNPATH
Full RELRO      Canary found      NX enabled    PIE enabled     No RPATH   No RUNPATH

Full RELRO + Canary + NX + PIE 是生产二进制的"四件套"基线。

9.3 -static-pie — 静态链接 + ASLR

g++ -static-pie -fPIE ...    # 静态二进制也能 ASLR (Linux 4.0+)

典型用：容器化 + 强 sandbox 环境，不依赖系统 libc。

10. 运行时缓解 — ASLR / PIE / CFI / Stack Canary / W^X

10.1 ASLR (Address Space Layout Randomization)

每次进程启动，把 stack / heap / mmap 区域 / 可执行映像地址随机化。攻击者无法预测目标地址 → 标准 ROP 利用失败。

# Linux 全局 ASLR 强度
cat /proc/sys/kernel/randomize_va_space
# 0: 关; 1: stack+mmap; 2: 全部 (默认 + 推荐)

# 进程必须是 PIE 才能享受可执行映像 ASLR
g++ -fPIE -pie myapp.cpp -o myapp
file myapp        # "ELF 64-bit LSB pie executable"

10.2 Stack Canary（栈金丝雀）

-fstack-protector 让编译器在每个函数栈帧末尾放一个随机值（canary），函数返回前对比 —— 不一致就是栈溢出，立即 abort。

// 编译器生成的伪代码:
void f(const char* s) {
    char buf[64];
    long canary = __stack_chk_guard;   // 进入时记录
    strcpy(buf, s);
    if (canary != __stack_chk_guard) {
        __stack_chk_fail();             // 不匹配 → 立即终止
    }
}
选项覆盖
-fstack-protector仅保护含字符数组 ≥ 8 的函数
-fstack-protector-strong+ 含数组 / 取局部变量地址的函数（推荐）
-fstack-protector-all所有函数（开销大）

10.3 NX / W^X

"Write XOR Execute" —— 任何内存页要么可写要么可执行，不能同时。栈/堆默认 NX，防止"注入 shellcode 后跳过去执行"。链接时加 -Wl,-z,noexecstack。

10.4 CFI (Control Flow Integrity)

验证每个间接调用 / 虚函数调用的目标是合法函数。攻击者即便控制了函数指针，也只能跳到"合法目标集合"内的函数 → 利用难度暴涨。

# Clang CFI
clang++ -fsanitize=cfi -fvisibility=hidden -flto myapp.cpp -o myapp

# Intel CET (硬件 CFI)
g++ -fcf-protection=full ...
# IBT (Indirect Branch Tracking) + Shadow Stack (返回地址保护)

10.5 SafeStack / Shadow Stack

把"控制数据"（返回地址、saved frame pointer）放到独立的、隔离的栈，"数据数据"（数组、struct）留在普通栈。即便普通栈溢出也覆盖不到返回地址。

clang++ -fsanitize=safe-stack ...           # Clang SafeStack
g++ -mshstk -fcf-protection=return ...      # Intel CET Shadow Stack

10.6 W^X 的扩展：seccomp / pledge

更彻底的"运行时缓解"是系统调用过滤（seccomp-bpf）—— 即便被 RCE，攻击者也调不出 execve / ptrace。详见 操作系统基础 §8。

11.4 CodeQL —— 用查询语言找漏洞

// CodeQL 查询语言示例：找 "strcpy + 用户控制的源" 模式
import cpp

from FunctionCall fc, Expr src
where
  fc.getTarget().getName() = "strcpy" and
  src = fc.getArgument(1) and
  src.isUserControlled()
select fc, "strcpy with user-controlled source — buffer overflow risk"

GitHub 全平台 PR 默认跑 CodeQL，发现 CVE 直接拦住。

12.2 Fuzzing —— 主动找漏洞

// 解析器 / 反序列化器 / 协议处理 必须 fuzz
extern "C" int LLVMFuzzerTestOneInput(const uint8_t* data, size_t size) {
    parse_my_protocol(data, size);
    return 0;
}

# 同时开 ASan + UBSan
clang++ -O1 -g -fsanitize=fuzzer,address,undefined fuzz_target.cpp
./fuzz_target corpus/ -max_total_time=86400      # 跑 24 小时

所有接受外部输入的代码都该有 fuzz target。OSS-Fuzz 给开源项目免费持续 fuzz。

12.3 Continuous Fuzzing 流程

  1. 每个 PR 跑 10 分钟基础 fuzz（CI 必过）
  2. nightly 跑 8 小时深度 fuzz
  3. OSS-Fuzz / 自家 fuzz farm 24/7
  4. 命中 crash → 自动归类 → 创建 issue + 触发 trace + 提示 owner
  5. 90 天宽限期后自动公开（CVE 流程）

13. CppCoreGuidelines —— Bjarne / Herb Sutter 出品

由 C++ 之父 Bjarne Stroustrup 和 ISOCPP 主席 Herb Sutter 主导的"现代 C++ 圣经"。500+ 条规则，按字母分组，所有大厂的 C++ 编码规范几乎都基于它。

13.1 安全相关的最关键 12 条

Rule说明
R.1 Manage resources by RAII禁止裸 new/delete（用智能指针）
R.3 Raw pointer is non-owning所有权用 unique_ptr / shared_ptr 表达
R.11 Avoid calling new/delete用 make_unique / make_shared
ES.20 Always initialize objects消除未初始化变量
ES.46 Avoid narrowing conversions用 narrow_cast / static_cast 显式
ES.50 Don't cast away constconst correctness
I.5 State preconditions用 expects / contracts (C++26)
I.27 Use PIMPL for stable ABI详见 ABI 指南
C.31 Resources acquired in ctor must be released in dtorRAII 完整性
C.83 noexcept swap异常安全基础
CP.20 Use RAII for locks不要手动 lock/unlock
SF.6 Use using namespace directive within scope only避免全局污染

13.2 GSL (Guidelines Support Library)

CppCoreGuidelines 推荐的辅助库 Microsoft GSL：

#include <gsl/gsl>

void f(gsl::not_null<int*> p);              // 编译期 + 运行期保证非 null

gsl::span<int> s{arr, n};                  // 带 size 的视图（C++20 起 std::span 标准化）

auto v = gsl::narrow<int>(big_uint);    // 检查损失的 narrow_cast

gsl::owner<int*> p = new int;             // 标注"我拥有这块内存", 配合 cppcoreguidelines check

13.3 clang-tidy 自动检查

clang-tidy -checks="-*,cppcoreguidelines-*" myfile.cpp
# 输出: 哪条 R.1 / R.11 / ES.20 ... 违反 + 行号

14. MISRA C++ / AUTOSAR C++14 / 23 —— 高安全行业规范

14.1 各规范定位

规范面向状态
MISRA C++ 2008汽车 / 嵌入式 (C++03)遗留
MISRA C++ 2023汽车 / 嵌入式 (C++17)最新
AUTOSAR C++14汽车 (C++14)主流，工业实际广泛使用
AUTOSAR C++23 (in progress)汽车 (C++23)开发中
JSF AV C++航空（联合战斗机 F-35 原始版）军事 / 航空
CERT C++SEI / Carnegie Mellon金融 / 政府

14.2 典型严苛规则

AUTOSAR / MISRA 的规则比 CppCoreGuidelines更严格，禁掉了大量"风险特性"。例如：

  • 禁用动态内存（new / delete）—— 启动时全预分配
  • 禁用异常（噪声大，难追溯路径）
  • 禁用 RTTI / dynamic_cast
  • 禁用多重继承（除接口纯虚类）
  • 禁用宏（除 #include guard）
  • 禁用 goto
  • 禁用 reinterpret_cast 和 union
  • 所有函数 ≤ 50 行（部分规范）
  • 所有 switch 必须有 default
  • 所有 if/else 必须用 {} 大括号
权衡：合规严苛规则会让代码看起来"老派"，禁用许多现代 C++ 优雅特性。但对安全关键软件（汽车 ECU、飞控、医疗设备）是必须的。商业静态分析工具（Coverity / Klocwork / Polyspace）有预置规则集对应这些规范，能自动检查。

14.3 AUTOSAR C++14 实例

// Rule A0-1-3: Every function must be used
static void unused_helper();   // ❌ 违反: 未使用

// Rule A5-1-1: Literal values shall not be used as in-out arguments
f(42);                       // ❌ 字面量传引用

// Rule A12-8-1: Move/copy constructors shall use move/copy assignment
// → 强制 copy-and-swap 风格

// Rule A18-9-3: The std::move shall not be used on objects declared const
const Foo f = ...;
std::move(f);                  // ❌ const 上 move 没意义（实际是 copy）

15. 供应链安全 — 第三方依赖 / CVE / SBOM

15.1 大型 C++ 项目的依赖现状

典型现代 C++ 项目可能依赖：Boost / Abseil / Folly / fmt / spdlog / Protobuf / gRPC / OpenSSL / RocksDB / nghttp2 ... 一个二进制可能间接依赖 50+ 第三方库。任何一个出 CVE 都可能影响你。

15.2 SBOM (Software Bill of Materials)

"软件物料清单" —— 列出你的二进制包含的所有第三方组件 + 版本 + license。美国白宫 EO 14028 / 欧盟 CRA 已要求关键软件强制提供 SBOM。

SBOM 格式

  • SPDX（Linux Foundation）—— 最广泛
  • CycloneDX（OWASP）—— 安全侧友好
  • SWID Tags (ISO)

SBOM 工具

# Syft - 容器/源码扫描生成 SBOM
syft mybinary -o spdx-json > sbom.json

# 用 grype 跟 NVD CVE 数据库对比
grype sbom:sbom.json

# 或 Trivy（最常用）
trivy sbom sbom.json

15.3 vcpkg / conan 自带漏洞数据库

# vcpkg 看依赖的 CVE
vcpkg list --x-json | jq '.packages[] | "\(.package_name)@\(.version)"'
# 配合 osv-scanner 查 OSV.dev

osv-scanner --lockfile=vcpkg.json

15.4 第三方代码引入流程（企业级）

  1. 评估：license 合规、活跃度、维护者信誉、CVE 历史
  2. vendor：fork 进自家 monorepo（保证可控）
  3. CI 扫描：每次 build 跑 trivy / grype，新 CVE 立即告警
  4. 升级策略：安全补丁 24h 内合入，feature 升级按月评估
  5. 降级路径：保留切换到自研 / 替代库的能力

15.5 dependency confusion / typosquatting

近年新攻击：在公开仓库（vcpkg / conan center）发布同名恶意包骗你的 CI 拉取。防御：

  • 锁版本 + 锁哈希（lockfile）
  • 用私有仓库镜像所有第三方依赖
  • 启用包签名验证（cosign）

16. 生产硬化 Checklist（可印出贴墙）

16.1 编码层

  • ✅ 智能指针，禁裸 new/delete
  • ✅ std::string / span / array，禁 C-style 字符串/数组
  • ✅ std::format / println，禁 printf 系列（除性能关键 + 格式串硬编码）
  • ✅ 所有 size / length 用户输入必须先验证（含整数溢出检查）
  • ✅ const correctness 全开
  • ✅ 异步回调用 shared_from_this / weak_ptr
  • ✅ 跨线程数据访问要么 mutex 要么 atomic

16.2 编译选项

  • ✅ -O2 -g -Wall -Wextra -Werror
  • ✅ -D_FORTIFY_SOURCE=3 -D_GLIBCXX_ASSERTIONS
  • ✅ -fstack-protector-strong -fstack-clash-protection
  • ✅ -fcf-protection=full（x86_64）
  • ✅ -fPIE -pie
  • ✅ -ftrivial-auto-var-init=zero
  • ✅ -Wformat-security -Wnull-dereference
  • ✅ GCC 14+ 用 -fhardened 一键开

16.3 链接选项

  • ✅ -Wl,-z,relro -Wl,-z,now
  • ✅ -Wl,-z,noexecstack
  • ✅ -Wl,--as-needed
  • ✅ -fvisibility=hidden + 显式 API_EXPORT

16.4 CI 流水线

  • ✅ 每 PR 跑 clang-tidy（cppcoreguidelines / bugprone 集合）
  • ✅ 每 PR 跑 ASan + UBSan 单元测试
  • ✅ 每 PR 跑 10 分钟 fuzz
  • ✅ Nightly 跑 TSan + MSan + 8 小时深度 fuzz
  • ✅ CodeQL / SonarQube / Coverity 全量扫
  • ✅ 依赖扫描（trivy / osv-scanner）
  • ✅ SBOM 生成 + 归档
  • ✅ 二进制 checksec 验证（RELRO/Canary/NX/PIE 全 ✓）

16.5 运行时

  • ✅ ASLR 全开（kernel.randomize_va_space=2）
  • ✅ 容器 + seccomp 限制 syscall
  • ✅ 容器 + 最小 capability
  • ✅ 非 root 运行
  • ✅ AppArmor / SELinux 策略
  • ✅ 生产采样 GWP-ASan / HWASan（条件允许）
  • ✅ 监控：crash 自动归类 + 警报

16.6 流程

  • ✅ 安全审计：每年第三方渗透测试
  • ✅ 漏洞响应：CVE 24h SLA
  • ✅ 培训：开发者每年 1 次安全编码 + OWASP Top 10 培训
  • ✅ Postmortem：所有安全事件 blameless review
Tech Lead 实务：上面 6 类清单不需要一天全做到。分 4 阶段滚动落地：① 第 1 月：编码 + 编译选项；② 第 2 月：CI 加 ASan/UBSan/clang-tidy；③ 第 3 月：fuzz + 依赖扫描；④ 第 4-6 月：CFI + 生产 sanitizer 采样 + 安全文化。每阶段做完测一波收益再推下一步，避免一刀切。
