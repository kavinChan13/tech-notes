01系统背景与业务目标

本 Capstone 以车端分布式数据采集与遥测系统为案例。这个系统同时涵盖：嵌入式 + Linux SoC + 分布式 + 实时性 + 安全关键——是练习前 5 篇方法论最好的综合载体。

1.1 背景

某汽车 OEM 正在开发下一代 ADAS（高级驾驶辅助系统）平台。需要一套车端数据采集与遥测系统，满足：

  • 从 CAN / CAN-FD / Automotive Ethernet 采集传感器信号（摄像头、雷达、激光雷达）
  • 按配置规则触发式记录数据（如：TTC < 2s 时记录前后 30s 数据）
  • 本地 Flash 缓冲，联网时选择性上传到云端
  • 云端平台负责数据分析 + 模型训练 + OTA 规则下发
  • 系统必须满足ISO 26262 ASIL-B（不直接控制车辆，但与 ADAS 共存）

1.3 核心约束

硬约束（必须满足）

  • 目标 SoC：NVIDIA Orin（ARM Cortex-A78 + 16GB RAM）
  • OS：QNX Hypervisor + Linux Guest（ADAS）+ QNX Guest（Safety）
  • 存储：1TB NVMe（边缘本地），每天最多写 50GB
  • C++ 版本：C++17（C++20 等工具链稳定后迁移）
  • ASIL-B 合规：必须通过 ISO 26262 Part 6 audit
  • OTA 可更新：规则配置 + 软件本身均支持 OTA

软约束（需要权衡）

  • 团队规模：6 人 C++ + 2 人 DevOps + 1 人安全
  • 交付节奏：季度里程碑，1.5 年上线
  • 云端带宽：4G/LTE，平均 10 Mbps，费用敏感
  • Flash 寿命：50GB/day × 5 年 = 91.25 TB 写入（NVMe TBW 需匹配）

2.1 效用树（Utility Tree）

质量属性场景度量优先级难度

Safety 遥测系统异常不干扰 ADAS 计算路径 FFI 验证通过 HH
ASIL-B 合规 audit 通过 ISO 26262 Part 6 HH
Modifiability 新增采集协议（如新型传感器） ≤ 5 人天，不影响其他模块 HM
修改采集规则不需要重新发布整个系统 OTA 规则更新 ≤ 5 分钟 HM
Security 本地录制数据加密，防止物理提取 AES-256 加密，密钥在 HSM HM
Testability 在 HIL 环境可完整模拟信号源 可注入模拟信号替换真实硬件 MM

2.2 架构关键点识别

标记为 [H, H] 的场景是架构关键场景，每一个都需要 ADR：

🔴 Risk（风险点）

  • 信号采集 P99 ≤ 10ms：同步 IPC 架构无法满足 → 需要无锁共享内存
  • RPO = 0：进程重启会丢数据 → 需要 WAL / 原子持久化
  • FFI：遥测进程可能污染 ADAS 内存 → 需要进程隔离 + QNX 内存保护

🟠 Tradeoff（权衡点）

  • 低延迟 IPC vs 进程隔离：共享内存延迟低但隔离弱；消息队列隔离强但延迟高 → ADR-001
  • 数据加密 vs 写入速度：AES-256 加密有 CPU 开销 → 选 NVMe 加密（硬件加速）
  • 规则热更新 vs 系统确定性：动态加载规则引入不确定性 → ADR-006

05ADR-001：进程间通信选型

# ADR-001: 选择无锁共享内存环形缓冲区作为信号流 IPC ## Status Accepted · 2025-09-15 · 平台架构师 + 嵌入式负责人 ## Context Signal Adapter → Rule Engine → Recorder 之间的信号流需要 IPC 方案。 核心需求：P99 延迟 ≤ 10ms（信号采集到规则评估），进程隔离（一个挂不影响另一个）， 系统资源：4 个 CPU 核，8GB 内存可用于遥测系统。 ## Decision Drivers - P99 ≤ 10ms（[H,H] 质量属性场景） - 进程崩溃隔离 - < 5MB IPC 内存占用 - 零第三方 broker 进程 ## Considered Options 1. POSIX 共享内存 + 无锁环形缓冲（自研） 2. SOME/IP ara::com（AUTOSAR Adaptive） 3. ZeroMQ inproc/ipc 4. NATS local broker ## Decision 选择方案 1（POSIX 共享内存 + 无锁环形缓冲）。 ## Justification | 方案 | P99 延迟 | 隔离 | 资源 | 成熟度 | |---|---|---|---|---| | 1. 共享内存 | ~5µs ✅ | 进程隔离 ✅ | <1MB ✅ | 自研 | | 2. ara::com | ~100µs | 完整 ✅ | 10MB+ | 重 ❌ | | 3. ZeroMQ | ~50µs | 进程隔离 | 5MB | OK | | 4. NATS | ~1ms | broker 独立 | 15MB ❌ | OK | SOME/IP/ara::com 资源占用过大；NATS 增加 broker 进程，违反资源约束； ZeroMQ 延迟可接受但比方案 1 高 10x；方案 1 虽然自研，但团队已有 lock-free 经验，代码量< 500 行。 ## Positive Consequences - 信号延迟 P99 ~5µs，远低于 10ms 要求 - 零额外进程，资源占用最小 - 自研代码，对自己有完整控制权 ## Negative Consequences - 自研：需要严格的并发测试（TSan + sanitizers） - 进程崩溃可能导致缓冲区状态不一致 → 启动时做 sanity check，重置环形缓冲 - 无内建背压 → 写满时覆盖旧数据（对实时系统可接受） ## Follow-ups - 信号总线压力 > 80% 触发自动告警 - 下一版本（v2）评估是否切换到 POSIX MQ（支持优先级）

06ADR-002：数据存储选型

# ADR-002: 使用 RocksDB 作为本地时序数据存储 ## Status Accepted · 2025-09-20 ## Context 本地存储需要：写密集（触发时最高 500MB/s），按时间范围读取，支持断电安全，NVMe TBW 寿命约束。 ## Considered Options 1. RocksDB（LSM-Tree，Facebook） 2. SQLite WAL 模式 3. InfluxDB 本地部署 4. 自研时序文件（二进制追加） ## Decision 选择方案 1（RocksDB）+ FIFO Compaction + Snappy 压缩。 ## Justification | 方案 | 写性能 | 空间效率 | 成熟度 | 复杂度 | |---|---|---|---|---| | RocksDB | 优（LSM） | 高（Snappy） | 成熟 ✅ | 中 | | SQLite | 中（B-Tree） | 中 | 成熟 ✅ | 低 | | InfluxDB | 优（TSM） | 高 | 重 ❌ | 高 | | 自研文件 | 优 | 自定义 | — | 高（维护） | RocksDB LSM-Tree 对写密集场景性能最佳； FIFO Compaction 模式天然适合时序数据（按时间删旧数据，无写放大）； Snappy 压缩率约 4:1，显著减少 NVMe 写入量（延长寿命）。 SQLite 在写密集场景性能不足；InfluxDB 资源占用过大。 ## RocksDB 关键配置 compaction_style = kFIFO max_total_wal_size = 1GB compression = kSnappyCompression write_buffer_size = 64MB level0_file_num_compaction_trigger = 4 ## Positive Consequences - 写入性能满足触发时峰值需求 - FIFO 模式自动按时间清理，实现 7 天数据保留 - WAL 保证断电安全（RPO = 0） ## Negative Consequences - RocksDB Compaction 在后台持续 IO，需要限制 compaction 带宽防止影响前台写 - 需要额外配置 HW 加密（RocksDB 本身不加密，由 NVMe 加密层处理） - 跨平台移植需要验证（确认目标 SoC 工具链支持 RocksDB）

07ADR-003：ABI 边界设计

# ADR-003: 插件接口使用 C ABI + Opaque Handle 模式 ## Status Accepted · 2025-10-01 ## Context 信号适配器需要支持多种车辆协议（CAN-FD、SOME-IP、ROS2 Topic）， 这些适配器来自不同供应商，用不同编译器和版本编译。 核心系统需要在不重启的情况下加载新适配器。 ## Decision 适配器插件接口使用 C ABI（extern "C" + 纯 C 函数表）， 内部实现用 C++ + Pimpl。 ## 接口定义 ```c // signal_adapter_api.h — 这是唯一的 ABI 合约 typedef struct SignalAdapterDescriptor { uint32_t api_version; // Must == SIGNAL_ADAPTER_API_VERSION const char* name; // e.g., "can-fd-adapter" const char* version; // semver void (*destroy)(struct SignalAdapterDescriptor*); int (*init)(struct SignalAdapterDescriptor*, const char* config_json); int (*subscribe)(struct SignalAdapterDescriptor*, const char** signal_ids, size_t count, SignalCallback callback, void* userdata); void (*unsubscribe)(struct SignalAdapterDescriptor*, const char** ids, size_t count); } SignalAdapterDescriptor; typedef SignalAdapterDescriptor* (*AdapterCreateFn)(void); ``` ## Positive Consequences - 跨编译器（GCC / Clang / vendor），无 ABI break 风险 - 供应商只需实现 C ABI，内部可用任意语言（C / C++ / Rust） - 适配器可以独立 OTA 更新，无需升级整个遥测系统 ## Negative Consequences - C ABI 无 RAII；每个适配器需要显式 init/destroy → 封装 RAII wrapper - 错误处理只能用返回码（无异常）→ 用 error code + error message out-param - 调试时多一层间接（符号名都是 C 的）

08ADR-004：可观测性方案

# ADR-004: 可观测性方案 — DLT（车内）+ OTel（云端）双轨 ## Status Accepted · 2025-10-10 ## Decision 车内可观测性：DLT（Diagnostic Log and Trace，AUTOSAR 标准） 云端可观测性：OpenTelemetry（OTLP → Grafana Stack） 两者通过 Health Monitor 桥接。 ## 分层设计 Layer 1 (in-vehicle): - DLT for logs（二进制，ECU ID + Context ID 分类） - Atomic counters for metrics（无 lock 开销） - W3C Trace Context 在 DLT message 头传递（简化版 span） Layer 2 (cloud gateway): - Health Monitor 定期把聚合 Metric 转换成 OTel OTLP 发送给云端 Collector - DLT → JSON → Loki（只上报关键事件，不全量） ## 为什么不在车内直接用 OTel - OTel C++ SDK 动态分配，不符合 ASIL-B 约束 - OTLP gRPC 在网络波动时有不确定延迟 - 资源：OTel SDK + Collector 需要额外 ~50MB RAM ## Positive Consequences - 车内轻量（DLT 标准，工具生态完整） - 云端统一 Grafana 可视化（所有服务用同一平台） - 实现渐进：先 DLT，稳定后再考虑 OTel 迁移 ## Negative Consequences - 双轨维护成本 - DLT 和 OTel 的 Trace 模型不完全等价，跨域追踪有信息损失

09ADR-005：C++ 子集与工具链

# ADR-005: 采用 AUTOSAR C++14 子集 + C++17 非违规扩展 ## Status Accepted · 2025-10-15 ## Decision - 基准：AUTOSAR C++14 Guidelines（基于 MISRA C++） - 扩展：C++17 的 std::optional / std::variant / if constexpr（经过审查，无违规） - 禁止：动态分配（runtime new/delete）/ 异常 / RTTI / 递归（除静态 constexpr） - 工具链：GCC 12.3（Linaro cross）+ Clang 16（代码分析） - 静态分析：Coverity（合规检查）+ clang-tidy（代码质量） ## 内存管理策略 - 启动阶段：允许动态分配（初始化资源池、规则缓存） - 运行阶段：零动态分配（所有结构从预分配池获取） - 自定义分配器：etl::pool（Embedded Template Library） ## Positive Consequences - 满足 ASIL-B Clause 8（Software Unit Implementation） - 零动态分配 → WCET 可计算 - Coverity + AUTOSAR checker 覆盖 95% 的合规规则 ## Negative Consequences - std::vector / std::map 无法使用（改用 etl::vector / etl::map） - 异常禁用 → 所有 API 返回 etl::expected<T,E> - 学习成本：团队需要熟悉 ETL 库

10ADR-006：插件化与规则热更新

# ADR-006: 规则配置 OTA 更新采用原子替换 + 双缓冲策略 ## Status Accepted · 2025-11-01 ## Context OTA 更新采集规则时，系统必须继续工作（不允许服务中断）。 规则文件是 Protobuf 序列化，版本化。 ## Decision 使用原子文件替换 + 双缓冲 Rule Cache： 1. OTA 下载新规则到 `/data/rules/rules_new.pb` 2. 校验签名 + 语法检查（沙盒解析） 3. 原子 rename：`rules_new.pb` → `rules.pb`（POSIX rename 是原子操作） 4. Rule Engine 监听文件变化（inotify），触发异步重新加载 5. 双缓冲：加载新规则到 Buffer B；加载完成后原子替换 Buffer A（rule pointer swap） 6. 旧 Buffer A 延迟释放（等待所有正在执行的 match 完成） ## 为什么不用 dlopen 动态加载 规则是配置数据（Protobuf），不是代码 → 不需要 dlopen。 dlopen 引入代码执行不确定性，ASIL-B 下难以验证安全性。 ## Positive Consequences - OTA 期间系统不中断（零停机规则更新） - 原子替换：断电安全（rename 是原子的） - 新规则加载失败 → 保持旧规则不变（graceful degradation） ## Negative Consequences - 双缓冲内存开销（两份规则同时在内存） - inotify 有延迟（约 100-500ms），规则更新有短暂延迟 - 需要验证"指针原子交换"在目标 CPU 上的内存序（用 std::atomic<RuleSet*>）

11.1 Signal Adapter 接口（C++ 内部）

// 信号采集上下文的 Port 接口（依赖倒置）
// domain/ports/i_signal_source.hpp
class ISignalSource {
public:
    virtual ~ISignalSource() = default;
    virtual etl::expected<void, SignalError>
        subscribe(const etl::string_view& signal_id,
                  std::function<void(const SignalSample&)>) = 0;
    virtual etl::expected<void, SignalError>
        unsubscribe(const etl::string_view& signal_id) = 0;
    virtual etl::optional<SignalSample>
        get_latest(const etl::string_view& signal_id) const noexcept = 0;
};

// 核心领域对象（值对象）
struct SignalSample {
    SignalId  id;           // strong typedef
    Timestamp timestamp;   // 高精度，纳秒级
    SignalValue value;     // std::variant<bool, int32_t, float, double, etl::string<64>>
    Quality quality;       // GOOD / DEGRADED / BAD (故障诊断用)
};

// 适配器包装（ACL：防腐层）
class CanFdSignalSource : public ISignalSource {
    const SignalAdapterDescriptor* adapter_;  // C ABI 插件
    DbcDecoder dbc_decoder_;                 // DBC 文件解码器（ACL）
public:
    etl::expected<void, SignalError>
        subscribe(const etl::string_view& signal_id, ...) override {
        // 把我们的 domain signal_id 翻译成 CAN message ID + bit offset
        auto can_sig = dbc_decoder_.resolve(signal_id);
        if (!can_sig) return etl::unexpected{SignalError::NOT_FOUND};
        // 调 C ABI 适配器
        auto rc = adapter_->subscribe(adapter_, can_sig->ids(), ...);
        return rc == 0 ? etl::expected<void, SignalError>{}
                       : etl::unexpected{SignalError::ADAPTER_ERROR};
    }
};

11.2 Rule Engine 接口

// 规则引擎上下文的核心 Aggregate
class TriggerRule {  // Aggregate Root
    RuleId   id_;
    RuleVersion version_;
    ConditionExpr condition_;  // AST，预编译（不是每次重新解析）
    TriggerAction action_;     // 触发后做什么（录制 N 秒）
    RuleStatus status_ = RuleStatus::Active;

public:
    // 判断是否触发（核心业务方法，纯逻辑，零 IO）
    etl::optional<TriggerDecision> evaluate(const MatchContext& ctx) const noexcept;

    // 不变量：Active 状态的规则必须有有效的 ConditionExpr
    RuleId id() const noexcept { return id_; }
    bool is_active() const noexcept { return status_ == RuleStatus::Active; }
};

// Repository 接口（Domain 层定义，Infrastructure 层实现）
class IRuleRepository {
public:
    virtual ~IRuleRepository() = default;
    virtual etl::expected<etl::vector<TriggerRule, MAX_RULES>, LoadError>
        load_all() noexcept = 0;
    virtual RuleVersion current_version() const noexcept = 0;
};

12.1 信号采集上下文的核心 Aggregate

// RecordingSession：核心聚合（触发一次录制的生命周期）
class RecordingSession {  // Aggregate Root
    SessionId        id_;
    TriggerEventId   trigger_;      // 触发这次录制的事件 ID（跨 Aggregate 只存 ID）
    Timestamp        start_;        // 录制开始（pre-buffer 时间）
    Timestamp        end_;          // 录制结束（触发后 N 秒）
    RecordingStatus  status_ = RecordingStatus::Recording;
    etl::vector<SignalId, 256> channels_;  // 录制的信号通道
    StorageLocation  location_;     // RocksDB 里的 key prefix

    // 不变量：Recording 状态下 end_ > start_
    // 不变量：channels_ 不能为空

public:
    // 工厂方法（用 TriggerRule 决定参数）
    static etl::expected<RecordingSession, DomainError>
        create(TriggerEventId trigger_id, const TriggerDecision& decision,
               SessionId id = SessionId::generate());

    // 业务方法
    etl::expected<void, DomainError> finalize(Timestamp now) noexcept {
        if (status_ != RecordingStatus::Recording)
            return etl::unexpected{DomainError::INVALID_TRANSITION};
        end_ = now;
        status_ = RecordingStatus::Completed;
        return {};
    }
    bool is_active() const noexcept { return status_ == RecordingStatus::Recording; }
};

当前 MVP 的 ring buffer 是相对简单的实现（无优先级、无 QoS）。如果 Phase 2 需要更复杂的功能，将采用 Strangler Fig 渐进替换：

  1. 在现有 ISignalBus 接口后面，新增 NewSignalBus 实现
  2. 通过 Feature Flag 将 10% 流量路由到新实现
  3. 比对两种实现的数据完整性和延迟
  4. 稳定后切 100%，删除旧实现
  5. 新实现的 decommission deadline：Phase 2 结束

14架构 Fitness Function

把架构原则写成自动化检查，每次 CI/CD 都运行。这是确保架构不被无声侵蚀的唯一可靠方式。

14.1 依赖方向检查

#!/bin/bash
# scripts/check_arch_deps.sh
# Fitness Function #1: Domain 层不能依赖任何非标准库

# 检查 domain 目录下的 .hpp 不包含 infrastructure / RocksDB / spdlog 等
forbidden=(rocksdb spdlog boost mosquitto DLT)

for lib in "${forbidden[@]}"; do
    if grep -r "#include.*${lib}" src/domain/; then
        echo "❌ FAIL: domain layer includes ${lib}"
        exit 1
    fi
done

echo "✅ PASS: Domain dependency rules"
# Fitness Function #2: ABI 兼容性检查（每次改 signal_adapter_api.h）
# .github/workflows/abi-check.yml
- name: Check Signal Adapter ABI
  run: |
    abi-dumper build/libtelemetry.so -output current.dump
    abi-compliance-checker -lib signal-adapter \
        -old baseline.dump -new current.dump
    # 失败 = ABI break = CI 不通过
# Fitness Function #3: 性能基线（P99 ≤ 10ms）
# tests/perf/signal_flow_bench.cpp
# 每次 PR 运行，延迟超过阈值 fail
BENCHMARK(SignalFlowE2E) {
    // 注入 10000 个 SignalSample，测量到 Rule Engine 处理完的延迟
    auto t0 = now();
    inject_signal(sample);
    auto t1 = rule_engine_.wait_for_match(sample.id);
    auto latency = t1 - t0;

    ASSERT_LT(latency.p99_ns(), 10 * 1000 * 1000);  // 10ms in ns
}
# Fitness Function #4: 禁止 dynamic allocation 在 run phase
# 用 custom allocator 替换 new，运行时检测
namespace {
  std::atomic<bool> run_phase{false};
}

void* operator new(std::size_t n) {
    if (run_phase.load()) {
        spdlog::critical("FATAL: dynamic allocation in run phase! size={}", n);
        std::abort();   // 测试模式下 abort，生产模式下 DLT 告警
    }
    return std::malloc(n);
}
