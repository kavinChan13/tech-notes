怎么答好后端系统设计

招聘官评分的 5 个维度

系统设计面试考的不是「你知道多少组件」，而是在约束下做取舍的结构化思维。同一道题在 QPS 100 和 QPS 100 万下答案完全不同。评分关键：先澄清再动手、会算容量、能画出清晰数据流、每个选型都说出代价、主动谈瓶颈与演进。切忌一上来就堆 Kafka + Redis + K8s。

① 澄清需求（必做）

先问再画：读写比、QPS / DAU、数据规模、延迟 SLA、一致性要求（强一致 vs 最终一致）、成本与现有栈。不澄清直接降档。

② 容量估算

QPS（读/写/峰值）、存储总量（含副本 + 保留期）、带宽、单机能扛多少 → 需要几台。会算账决定后面所有选型。

③ 高层架构 + 数据流

画 Client → LB → 服务 → 缓存 / DB → 异步链路。先给一版能跑的，再逐步加高可用与扩展。

④ 关键取舍

SQL vs NoSQL、同步 vs 异步、强一致 vs 最终一致、推 vs 拉、缓存策略、分库分表。每个选择说代价。

⑤ 瓶颈 / 容错 / 演进

单点、热点、缓存三问题、幂等、限流降级、故障域隔离；Day1 → Year1 怎么扩。

通用答题框架（6 段）

每题都按这个骨架走，不会漏
① 澄清需求 — 功能 / 读写比 / QPS·DAU / 延迟 SLA / 一致性 / 成本 / 现有栈
② 容量估算 — 读写 QPS / 峰值 / 存储总量 / 带宽 / 单机上限 → 机器数
③ 高层架构 — Client → LB → 服务 → 缓存 → DB → 异步（MQ）
④ 数据模型 + 关键 trade-off — 表结构 / 分片键；SQL/NoSQL、推/拉、同步/异步，每个说代价
⑤ 瓶颈 + 容错 — 热点、单点、缓存三问题、幂等、限流降级、故障隔离
⑥ 监控 + 演进 — SLA/指标、扩容路线、Day1 → Year1
先问清「读写比 + 规模」——它几乎决定架构：读多写少 → 加缓存 + 读副本；写多 → 分片 + 异步削峰；强一致 → 单主 / 分布式事务；能容忍最终一致 → 大胆用异步与缓存。这是后端系统设计的第一分叉。

A · 核心 4 题（后端必过）

01
设计短链系统（TinyURL / 短网址）20 min
CORE

澄清需求

  • 写 QPS（生成）vs 读 QPS（跳转）？典型读:写 ≈ 100:1，重读。
  • 短码长度 / 字符集？是否自定义别名？是否需要过期、统计点击？
  • 301 永久跳转（可缓存、省流量）还是 302 临时（可统计）？

容量估算

  • 1 亿新链/月 ≈ 40 写/s，读 4000/s，峰值 ×10。5 年 ≈ 60 亿条。
  • 短码：Base62（a-zA-Z0-9），7 位 ≈ 62⁷ ≈ 3.5 万亿，足够。

架构与数据流

  1. 生成：发号器（Snowflake/号段）产出全局唯一 ID → Base62 编码为短码，写 KV（short → long）。
  2. 跳转：读服务查缓存（Redis）命中直接 302/301；未命中回源 DB 再回填缓存。
  3. 存储：KV 型（短码为主键），读多用只读副本 + 分片。

关键 trade-off

  • 发号 vs 哈希：哈希（MD5 截断）会冲突需处理；发号器无冲突、可控长度，工程更稳。
  • 301 vs 302：301 浏览器缓存省后端压力但无法统计；要点击分析选 302。
  • 预生成 vs 实时：可预生成短码池发放，降低写路径延迟。

边界 + 容错

  • 热点短码：本地缓存 + Redis，防单 key 击穿。
  • 恶意生成：限流 + 风控；短码指向恶意站点需黑名单。
  • 缓存穿透：不存在的短码用空值缓存 / 布隆过滤器。

监控 + 演进

  • 监控：跳转 P99、缓存命中率、发号器可用性、404 比例。
  • 演进：多机房就近解析、点击分析走异步埋点入数仓、自定义域名。
02
设计分布式限流器（Rate Limiter）15 min
CORE

澄清需求

  • 按什么维度限（用户 / IP / API / 租户）？限流值？超限行为（拒绝 429 / 排队 / 降级）？
  • 单机还是全局？允许突发吗？精度要求（严格 vs 近似）？

算法选型

  • 固定窗口：简单但临界点会双倍突发。
  • 滑动窗口日志/计数：精确，内存/计算稍高。
  • 令牌桶：允许突发（桶容量），最常用；漏桶：恒定速率、平滑但不允许突发。

架构与数据流

  1. 集中式：Redis 存计数/令牌，用 Lua 脚本保证「读-判断-写」原子性，避免并发超发。
  2. 本地 + 全局两级：网关本地令牌桶挡大流量，全局配额定期从中心同步，降 Redis 压力。
  3. 接入点：API 网关 / 服务入口中间件。

关键 trade-off

  • 集中精确 vs 本地近似：集中式一致但有网络开销与单点风险；本地快但全局不精确。高 QPS 常用本地为主 + 中心兜底。
  • 拒绝 vs 排队：拒绝保护后端；排队提升体验但可能雪崩。

边界 + 演进

  • Redis 挂了要 fail-open 还是 fail-close？按业务定。演进：分层限流（全局/租户/接口）、自适应限流（根据 RT/负载动态调）、配合熔断降级。参见 限流熔断降级。
03
设计新闻 Feed / 时间线（Twitter Timeline）20 min
CORE

澄清需求

  • DAU？平均关注数？是否有大 V（百万粉）？Feed 排序（时间序 vs 算法排序）？读:写比？

架构与数据流（核心是推 vs 拉）

  1. 写扩散（推 / Fan-out on write）：发帖时写入所有粉丝的收件箱（Redis List）。读极快，适合粉丝少的普通用户。
  2. 读扩散（拉 / Fan-out on read）：读时实时聚合关注对象的帖子。写便宜，适合大 V（避免百万次写扩散）。
  3. 混合：普通用户走推，大 V 走拉，读时合并——业界主流方案。

关键 trade-off

  • 推 vs 拉：推读快写贵、存储放大（每条帖复制 N 份）；拉写便宜读慢。大 V 是压垮纯推模式的元凶。
  • 时间序 vs 排序：时间序简单可缓存；算法排序需特征 + 模型（呼应 推荐特征链路）。

边界 + 容错

  • 热点大 V 发帖：异步扩散 + 限速，避免瞬时写风暴。
  • 收件箱只存 ID + 分页游标，正文另取；控制收件箱长度（只留最近 N 条）。

监控 + 演进

  • 监控 Feed 拉取 P99、扩散延迟、缓存命中。演进：加入排序层、去重、已读状态、多级缓存。
04
设计即时通讯 IM（微信 / WhatsApp）20 min
CORE

澄清需求

  • 单聊 + 群聊？在线状态？消息是否需持久 + 多端同步 + 已读回执？规模（在线连接数）？

架构与数据流

  1. 长连接层：客户端与接入网关建 WebSocket/TCP 长连；网关维护 连接↔用户 路由表（存 Redis）。
  2. 发消息：A → 接入网关 → 逻辑服务（存消息、生成 seq）→ 查 B 在哪个网关 → 推送；B 离线则存离线消息。
  3. 存储：消息按会话分片存（写多，选可水平扩的存储如 HBase/Cassandra）；用递增 seq保证会话内有序 + 拉取补偿。
  4. 多端同步：以服务端 seq 为准，各端按 seq 拉取缺失消息。

关键 trade-off

  • 写扩散 vs 读扩散（群聊）：群小写扩散（每人一份收件箱）；超大群读扩散（共享消息表 + 各自游标），否则一条群消息放大成万次写。
  • 可靠投递：ACK + 重传 + 去重（消息 ID 幂等），保证不丢不重。
  • 有序性：全局有序代价高；只需会话内有序（单会话单分片 + seq）。

边界 + 演进

  • 海量长连接：C10K/C10M 用 epoll + 多路复用（参见 高并发 TCP 服务器）。演进：端到端加密、消息漫游、状态服务拆分、就近接入。

05
设计秒杀 / 高并发抢购18 min
VERTICAL

澄清需求

  • 库存量？瞬时峰值 QPS？是否允许超卖（通常绝不）？下单后支付时限？防刷要求？

架构与数据流（核心：层层削峰 + 库存不超卖）

  1. 前端：按钮置灰 / 答题 / 随机延迟，削掉无效请求；静态资源上 CDN。
  2. 网关：限流（令牌桶）+ 黑名单，只放少量请求进后端。
  3. 库存扣减：Redis 预扣库存（Lua 原子 decr，扣完即拒），挡住绝大多数请求。
  4. 下单：预扣成功者进 MQ 异步下单，DB 最终以乐观锁 / WHERE stock>0 落库，削峰填谷。

关键 trade-off

  • Redis 扣减 vs DB 扣减：DB 行锁扛不住瞬时高并发；Redis 预扣把压力挡在内存层，DB 只处理少量真实订单。
  • 同步 vs 异步下单：异步（MQ）削峰、体验是「排队中」；同步简单但易打垮 DB。
  • 不超卖：Redis 原子扣 + DB 兜底约束，双保险；未支付超时释放库存回补。

边界 + 演进

  • 防超卖、防刷（一人一单幂等）、Redis 与 DB 最终一致对账。演进：库存分桶（拆热点 key）、独立秒杀集群隔离故障域。
06
设计分布式唯一 ID 生成器15 min
VERTICAL

澄清需求

  • 要求：全局唯一、趋势递增（利于 DB 索引）？吞吐？是否要求单调？能否暴露业务量（安全）？

方案对比

  • UUID：本地生成无依赖，但 128 位、无序（拖垮 B+Tree 写）、不可读。
  • DB 自增 / 号段模式：DB 批量发号段（如一次取 1000），本地消费，减少 DB 压力，趋势递增。
  • Snowflake：64 位 = 时间戳 + 机器 ID + 序列号，趋势递增、高吞吐、去中心；依赖时钟。
  • Redis INCR：简单，但引入外部依赖与单点。

关键 trade-off

  • Snowflake 时钟回拨：面试必考——回拨会重复。处理：检测到回拨时等待 / 拒绝 / 借用备用位。
  • 递增 vs 无序：递增利于索引与分页，但会暴露业务量；可加随机位混淆。
  • 号段 vs Snowflake：号段依赖 DB 但绝对递增；Snowflake 无中心但需管理 workerId。

演进

  • 双 buffer 号段（提前预取下一段消除毛刺）；workerId 用 ZK/etcd 自动分配。
07
设计订单支付系统（幂等 + 一致性）18 min
VERTICAL

澄清需求

  • 下单→支付→履约链路？对一致性要求（钱绝不能错）？第三方支付回调？超时未付如何处理？

核心问题与方案

  1. 幂等：客户端带唯一 幂等键（token / 订单号），服务端「唯一约束 + 状态机」保证重复请求只生效一次（防重复扣款）。
  2. 跨服务一致性：扣库存 / 扣款 / 建订单跨库——用 Saga / 本地消息表 / TCC 保证最终一致，避免分布式 2PC 的可用性代价。
  3. 状态机：订单状态（待支付→已支付→已发货→完成/关闭）单向流转，非法转移拒绝。
  4. 回调：第三方支付异步回调 + 主动查询对账兜底（回调可能丢）。

关键 trade-off

  • Saga vs TCC vs 2PC：2PC 强一致但阻塞、可用性差；TCC 侵入大但准；Saga 最终一致、补偿复杂度可控——互联网首选。
  • 本地消息表 vs 事务消息：都保证「本地事务 + 发消息」原子，前者靠 DB 表 + 轮询，后者靠 MQ（RocketMQ 事务消息）。

边界 + 演进

  • 超时未付：延时队列关单 + 释放库存。对账：与第三方 T+1 对账修差。演进：资金流水双记账、风控接入。
08
设计通知 / 推送系统15 min
VERTICAL

澄清需求

  • 渠道（App Push / 短信 / 邮件 / 站内信）？量级（亿级群发）？实时性？去重与频控？

架构与数据流

  1. 接入：业务方调统一通知 API → 写 MQ 削峰。
  2. 编排：模板渲染 + 用户偏好 + 频控 / 免打扰 过滤 → 按渠道路由。
  3. 下发：各渠道 worker 调第三方（APNs/FCM/短信网关），失败重试 + 降级备用通道。
  4. 大群发：分批 + 限速，避免打爆第三方与自身。

关键 trade-off

  • 实时 vs 批量：营销类可批量削峰；验证码类需实时高优先级——用优先级队列分离。
  • 至少一次 vs 精确一次：推送通常至少一次 + 客户端去重（消息 ID）。

边界 + 演进

  • 第三方限额与故障：多通道容灾 + 熔断。演进：触达效果统计、智能发送时间、A/B 文案。

设计分布式缓存（Redis Cluster 类）18 min
PLATFORM

核心设计点

  1. 分片：一致性哈希 / 哈希槽（Redis 16384 slot），节点增删只迁移少量数据。
  2. 高可用：主从复制 + 哨兵 / 自动故障转移；副本提供读扩展。
  3. 淘汰：LRU / LFU / TTL，内存满按策略逐出（参见 LRU 缓存实现）。
  4. 访问模式：Cache-Aside（最常用）/ Read-Through / Write-Through / Write-Behind。

缓存三大问题（必考）

  • 穿透：查不存在的 key 每次打 DB → 布隆过滤器 + 空值缓存。
  • 击穿：热点 key 过期瞬间大量请求打 DB → 互斥锁重建 / 逻辑过期。
  • 雪崩：大量 key 同时过期 → 随机 TTL + 多级缓存 + 限流。

关键 trade-off

  • 一致性：缓存与 DB 双写不一致——「先更 DB 再删缓存」（Cache-Aside）+ 延迟双删 / binlog 订阅同步。强一致极难，通常接受短暂不一致。
  • 热点 key：单 key 打爆单分片 → 本地缓存 + key 拆分 / 复制。

演进

  • 本地缓存（Caffeine）+ 分布式缓存 + DB 的多级缓存体系；热点自动探测与本地化。
10
设计对象存储 / 网盘（分块 + 秒传）15 min
PLATFORM

澄清需求

  • 文件大小分布？总量？上传/下载 QPS？要秒传、断点续传、分享吗？

架构与数据流

  1. 元数据 vs 数据分离：元数据（文件名/大小/块列表/权限）存 DB；文件内容存对象存储（S3/自建）。
  2. 分块上传：大文件切块并行上传 + 断点续传（记录已传块）；小文件合并存储。
  3. 秒传（去重）：按内容哈希（SHA-256）判重，已存在则只加引用，省存储省带宽。
  4. 下载：CDN 加速 + 预签名 URL 直传，绕开应用服务器。

关键 trade-off

  • 内容寻址去重 vs 简单：哈希去重省成本，但需处理哈希计算开销与引用计数删除。
  • 强一致 vs 最终一致：元数据强一致，副本复制最终一致。

演进

  • 纠删码替代多副本降存储成本、冷热分层、跨区域容灾。
11
设计搜索自动补全（Typeahead）15 min
PLATFORM

澄清需求

  • 补全依据（热度 / 个性化）？前缀匹配延迟（< 100ms）？候选量级？更新频率？

架构与数据流

  1. 数据结构：Trie（前缀树） 存前缀，每个节点缓存 Top-K 候选，查询即走前缀返回预计算结果。
  2. 离线构建：从搜索日志离线统计词频 → 定期构建 Trie / 前缀→TopK 映射 → 加载到内存服务。
  3. 在线：内存服务前缀查 Top-K，Redis/本地缓存热点前缀；结果 CDN/边缘缓存。

关键 trade-off

  • 预计算 Top-K vs 实时排序：预计算查询快但更新慢；实时准但贵。补全对新鲜度不敏感 → 预计算为主。
  • Trie 内存 vs 分片：全量 Trie 内存大 → 按前缀分片到多机。

演进

  • 加入个性化（用户历史）、拼写纠错（编辑距离）、多语言分词。

设计延时队列 / 分布式任务调度15 min
PLATFORM

澄清需求

  • 场景（订单超时关单 / 定时任务 / 重试）？延时精度？任务量级？要求不丢不重吗？

方案对比

  • Redis ZSet：score = 到期时间戳，轮询取到期任务；简单高效，最常用。
  • 时间轮（Timing Wheel）：O(1) 增删，适合海量定时器（参见 定时器系统）。
  • MQ 延时消息：RabbitMQ TTL+死信 / RocketMQ 延时级别，交给中间件。
  • DB 轮询：简单但不扛量，仅小规模。

关键 trade-off

  • 精度 vs 成本：高精度需高频轮询；秒级足够则低频省资源。
  • 不丢不重：任务持久化 + 消费幂等 + ACK 后删除；调度器高可用（多实例抢锁，避免重复执行）。

边界 + 演进

  • 调度器单点：分片 + 选主（ZK/etcd）。演进：可视化任务编排（DAG 依赖）、失败重试与告警、任务分片并行。
