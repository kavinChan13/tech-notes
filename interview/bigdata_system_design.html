怎么答好大数据系统设计

招聘官评分的 5 个维度

大数据系统设计和通用系统设计 80% 相通（容量估算 / 数据流 / 高并发 / 容错 / 监控），20% 是数据特殊性：批 vs 流、延迟 vs 吞吐、一致性语义（at-least/exactly-once）、数据规模/倾斜、成本。评分关键不是「答对」，而是展现结构化的取舍思维。

① 澄清需求（必做）

先问再画：数据规模(GB/TB/PB)、QPS/写入速率、延迟要求(T+1 还是秒级)、数据源类型、一致性要求、成本约束、现有技术栈。不澄清直接降档。

② 容量估算

日数据量、峰值写入 TPS、存储总量(含副本/保留期)、查询 QPS、需要多少算力。会算账是数据架构师的基本功（呼应 D01 单机天花板）。

③ 分层与数据流

画出采集→存储→计算→服务的完整链路 + 数仓分层(ODS→ADS, D02)。讲清每段用什么、为什么。

④ 关键取舍

Lambda vs Kappa、批 vs 流、行 vs 列、COW vs MOR、自建 vs 托管、一致性 vs 可用性。每个选择都说出代价。

⑤ 容错/质量/演进

数据丢失/重复怎么办、数据倾斜、迟到数据、数据质量校验、回填、Day1→Year1 演进路径。

FW

通用答题框架（6 段）

每题都按这个骨架走，不会漏
① 澄清需求 — 数据规模 / 延迟(T+1?秒级?) / QPS / 一致性 / 成本 / 现有栈
② 容量估算 — 日增量 / 峰值写入 / 存储总量 / 查询 QPS / 算力
③ 架构与数据流 — 采集 → 存储 → 计算 → 服务，配数仓分层
④ 关键 trade-off — 批/流、Lambda/Kappa、存储格式、引擎选型，每个说代价
⑤ 边界 + 容错 — 不丢不重、倾斜、迟到、回填、降级
⑥ 监控 + 演进 — 数据质量监控、SLA、成本、Day1→Year1
先问清「延迟要求」——它几乎决定整个架构：T+1 离线 → 纯批(Spark+数仓)；秒级 → 引入流(Flink+Kafka+OLAP)；两者都要 → Lambda/Kappa。这是大数据系统设计的第一分叉。

01
设计实时数仓 / 实时 GMV 大屏20 min
CORE

澄清需求

  • 延迟要求？（秒级实时 vs 分钟级）大屏刷新频率？
  • 数据源？（订单库 CDC + 埋点流）日订单量级？峰值 TPS（大促）？
  • 要看哪些指标？（GMV、订单数、分城市/品类）要不要去重 UV？
  • 历史回看 + 数据修正要求？

容量估算

  • 日订单 1000 万，峰值（大促）QPS 可达平日 10–50 倍 → 按 5–10 万 TPS 设计 Kafka/Flink。
  • 状态：按 user 去重需维护亿级 key 状态 → Flink RocksDB 状态后端。

架构与数据流（Kappa 实时链路）

  1. 采集：订单库经 CDC(Debezium/Flink CDC) 入 Kafka；埋点经 SDK→Kafka（D06）。
  2. 计算：Flink 消费，按事件时间开窗聚合 GMV，Watermark 容忍迟到，状态去重 UV（D04）。
  3. 存储/服务：聚合结果写 OLAP（StarRocks/Doris，D07）或直接写 Redis（D12）供大屏点查。
  4. 大屏：前端轮询/WebSocket 读服务层，秒级刷新。

关键 trade-off

  • Lambda vs Kappa：Kappa（纯流）简单但历史修正难；Lambda（批兜底+流）准但维护两套。大促保险起见常用 Lambda（D08）。
  • 去重精确 vs 近似：精确去重状态大、贵；UV 可用 HyperLogLog 近似省 90% 内存（D12）。
  • 写 OLAP vs 写 KV：要多维下钻 → OLAP；只看固定几个数 → Redis 更快。

边界 + 容错

  • 迟到数据：Watermark + allowed lateness；超期数据进侧输出补算。
  • Exactly-once：Flink Checkpoint + Kafka 事务 + 幂等 sink，防大促重复计数（D04/D06）。
  • 大促洪峰：Kafka 削峰 + Flink 反压 + 预扩容；数据修正用批链路 T+1 覆盖。

监控 + 演进

  • Day1：单 Flink 作业 + Redis 大屏。Month3：接 OLAP 支持多维下钻 + 批流双校验。Year1：流批一体(Paimon)、统一指标口径、自动数据质量门。
  • 监控：Kafka lag、Flink checkpoint 时长/失败、端到端延迟、指标波动告警。
02
设计 CDC 入湖管道（MySQL → 湖仓）15 min
CORE

澄清需求

  • 同步多少张表？要不要 schema 变更自动跟随（DDL）？
  • 延迟要求（分钟 vs 秒）？要不要保留删除？要历史版本吗？
  • 下游怎么用（分析查询 / 训练数据 / 实时）？

架构与数据流

  1. 捕获：Debezium/Flink CDC 读 MySQL binlog → Kafka（每表一 topic，含 op 类型）（D06）。
  2. 入湖：Flink 消费 → MERGE INTO 写 Iceberg/Hudi 表（Upsert，删除随 op=DELETE 落地）（D11）。
  3. 布局：MOR 表应对高频更新 + 后台 compaction 维持读性能。
  4. 消费：Trino/StarRocks 直查湖仓表；或继续加工成 DWD/DWS（D02）。

关键 trade-off

  • COW vs MOR：高频小批更新选 MOR（写快）+ compaction；读多写少选 COW（D11）。
  • Hudi vs Iceberg：极高频 Upsert/主键索引 → Hudi；引擎中立/隐藏分区 → Iceberg。
  • 全量+增量：先一次性快照全量，再切增量 binlog（Flink CDC 支持无锁全增量一体）。

边界 + 容错

  • DDL 变更：表格式 schema 演进（field-id）安全加列；不兼容变更需人工介入（D11）。
  • 乱序/重复：按主键 + 版本号（binlog 位点/ts）做幂等 Upsert，保证最终一致。
  • binlog 过期：监控同步延迟，避免 binlog 被清导致断点丢失。

监控 + 演进

  • 监控同步延迟、binlog 位点、Upsert 失败、小文件数。演进：从单表到整库同步 + 自动建表 + 数据质量校验。
03
设计指标平台 / 指标中台15 min
CORE

问题背景

痛点：同一个「GMV」各部门口径不一、到处重复开发、数字对不上。指标平台要做到「一处定义、全局一致、可复用」（呼应 D02 一致性指标 / dbt）。

核心组件

  1. 指标定义层：原子指标(下单金额) + 维度 + 修饰词 → 派生指标(近7天华东GMV)，统一元数据管理。
  2. 语义层：把指标定义翻译成 SQL（dbt/自研），保证口径唯一。
  3. 计算层：离线(Spark 跑 DWS 汇总) + 实时(Flink)，物化常用指标。
  4. 服务层：指标查询 API（OLAP/Redis 后端），供 BI、大屏、报表统一调用。
  5. 治理：指标血缘、负责人、变更审批（D09）。

关键 trade-off

  • 预计算 vs 现算：高频固定指标预聚合(DWS/物化视图)；灵活探查现算。空间换时间。
  • 统一语义层 vs 各自 SQL：语义层前期投入大，但根治口径不一致——这是平台价值核心。

监控 + 演进

  • 监控指标产出延迟、口径变更影响（血缘）、调用量。演进：从报表指标 → 全公司指标资产 + 自助分析。
04
设计海量日志分析平台15 min
CORE

澄清需求

  • 日志量级（TB/天）？用途：检索排障 vs 指标聚合 vs 两者？保留期？延迟要求？

容量估算

  • 10 万 QPS 日志写入、5 TB/天、保留 30 天 → 150 TB（含压缩/副本评估）。冷热分层降本。

架构与数据流

  1. 采集：Filebeat/Vector/Fluent Bit 采服务器日志 → Kafka 削峰（D06 采集端）。
  2. 分流：检索类 → Elasticsearch（倒排，D12）；指标聚合类 → ClickHouse（列存扫描快，D07）。
  3. 可视化：Kibana（检索）+ Grafana（指标大盘）。

关键 trade-off

  • ES vs ClickHouse：全文检索/看上下文 → ES；大规模数值聚合 → ClickHouse 更快更省（D12）。很多公司两者并存。
  • 全量存 vs 采样/降级：成本敏感时对低价值日志采样、冷数据降级到对象存储。

边界 + 演进

  • 洪峰用 Kafka 缓冲 + 背压；ES 写满降级到只入 Kafka 不入 ES。演进：冷热分层、日志结构化、异常自动检测告警。

05
设计用户行为分析（埋点）平台15 min
VERTICAL

澄清需求

  • 埋点量级（亿级事件/天）？要实时漏斗/留存还是 T+1？埋点规范怎么管？

架构与数据流

  1. 埋点采集：客户端/服务端 SDK 按统一事件 schema 上报 → 上报服务 → Kafka（D06）。
  2. 埋点治理：埋点元数据管理 + 上报时 schema 校验（数据质量左移，D06/D09）。
  3. 入湖建模：原始事件入湖(ODS) → 清洗/会话切分(DWD) → 漏斗/留存指标(DWS)（D02）。
  4. 分析：OLAP（ClickHouse）支持自助多维分析；实时漏斗用 Flink。

关键 trade-off

  • 客户端 vs 服务端埋点：客户端拿 UI 行为但易丢/作弊；服务端可靠但拿不到纯前端交互。常混用。
  • 埋点规范严 vs 松：严则前期累、数据干净；松则上线快、后期治理痛。建议规范左移。

边界 + 演进

  • 防丢：SDK 本地缓冲 + 重试；防重：事件去重 ID。演进：实时 + 离线统一口径、AB 实验打通、隐私合规（PII 脱敏，D09）。
06
设计推荐系统的特征数据链路20 min
VERTICAL

澄清需求

  • 特征规模？在线推理 QPS/延迟（毫秒级）？实时特征还是离线为主？训练/服务一致性要求？

架构与数据流

  1. 离线特征：Spark 跑历史画像/统计特征写离线存储（湖仓，含 point-in-time 防穿越，D13）。
  2. 实时特征：Flink 消费行为流算实时特征（近 N 次点击）写在线存储（Redis/HBase，D12）。
  3. 特征平台：一份特征定义同时物化离线(训练)+在线(推理)，消除训练-服务偏差（D13）。
  4. 在线推理：召回→粗排→精排，毫秒级从在线特征存储取特征喂模型。

关键 trade-off

  • 训练-服务一致性：必须同源同口径，否则线上崩——这是面试核心考点（D13）。
  • 实时特征新鲜度 vs 成本：越实时越贵；按特征重要性分级，热特征实时、冷特征 T+1。
  • 在线存储选型：低延迟点查 → Redis；海量画像持久化 → HBase。

边界 + 演进

  • 防数据穿越（point-in-time join）；特征缺失给默认值。演进：特征平台化、特征复用、特征监控（漂移检测）。
07
设计电商订单全链路分析（离线+实时双链路）15 min
VERTICAL

架构与数据流（Lambda）

  1. 实时链路：订单 CDC → Kafka → Flink 实时聚合 → OLAP/Redis（秒级大屏、实时监控）。
  2. 离线链路：订单库 T+1 同步入湖 → Spark 跑 ODS→DWD→DWS→ADS 分层（D02）→ 数仓报表。
  3. 对账：离线结果作为「真相」校正/覆盖实时结果（实时图快、离线保准）。

关键 trade-off

  • Lambda vs Kappa：Lambda 双链路准但维护两套代码、口径易漂移；Kappa 一套流但历史重算难。看团队成熟度（D08）。
  • 退款/取消（撤回）：实时链路要处理撤回流/状态更新，GMV 要能减回去——实时数仓的硬骨头（D08）。

边界 + 演进

  • 数据修正：离线 T+1 覆盖；维度变化用 SCD（D02）。演进：流批一体收敛口径，减少双链路维护成本。
08
设计实时风控 / 反作弊数据链路15 min
VERTICAL

澄清需求

  • 延迟（毫秒级拦截 vs 准实时分析）？规则引擎还是模型？误杀容忍度？

架构与数据流

  1. 实时特征：行为流 → Flink 算滑动窗口特征（近 1 分钟下单次数、IP 聚集度）写在线存储。
  2. 实时决策：规则引擎 + 模型，毫秒级从在线特征取数判定，命中则拦截。
  3. 离线挖掘：湖仓存全量行为，离线挖掘新作弊模式 → 沉淀新规则/特征/样本。
  4. 名单服务：黑/白名单存 Redis/HBase 供毫秒点查（D12）。

关键 trade-off

  • CEP 复杂事件：用 Flink CEP 识别行为序列模式（D04）。
  • 实时拦截 vs 事后追溯：强实时拦截贵且可能误杀；分级——高危实时拦、中危准实时复核。

边界 + 演进

  • 对抗性：作弊模式会变，需离线持续挖掘 + 规则快速热更新。演进：规则→模型→图算法识别团伙。

09
设计湖仓数据平台（存算分离、多引擎）15 min
PLATFORM

分层架构

  1. 存储底座：对象存储(S3/OSS) + 表格式(Iceberg) + REST Catalog（D01/D11）。
  2. 计算层：Spark/Flink（ETL）+ Trino/StarRocks（查询），按需弹性、存算分离。
  3. 编排：Airflow/DolphinScheduler 调度任务依赖与回填（D06）。
  4. 治理：元数据/血缘、数据质量、权限、成本（D09）。
  5. 服务：BI、指标平台、服务层（D12）、特征/AI（D13）。

关键 trade-off

  • 开放表格式 + REST Catalog：一份数据多引擎共享、不锁厂商（D11/D09）。
  • 自建 vs 托管：规模/合规 → 自建；快速起步/小团队 → 托管(EMR/Databricks/MaxCompute)（D09）。
  • 存算分离的代价：跨网读延迟，用本地 SSD 缓存热数据缓解。

演进

  • Day1 单引擎 + HMS；Year1 多引擎共享 Iceberg + 统一 catalog + 自动化治理 + FinOps 成本优化。
10
设计数据质量与治理平台12 min
PLATFORM

核心能力（呼应 D09）

  1. 元数据 + 血缘：自动采集表/字段血缘，支持影响分析（改一张表波及谁）。
  2. 数据质量：质量六维（完整/准确/一致/及时/唯一/有效）+ 质量规则 + 质量门（不达标阻断下游）。
  3. 数据契约：上下游约定 schema/SLA，破坏即告警（呼应 D06 Schema Registry）。
  4. 安全：分级分类、RBAC、PII 脱敏、审计。

关键 trade-off

  • 阻断 vs 告警：质量门太严会卡正常发布，太松形同虚设；按数据重要性分级。
  • 采集血缘方式：解析 SQL（准但难覆盖全引擎）vs 运行时埋点（全但侵入）。

演进

  • 从「事后救火」到「事前契约 + 质量左移」，再到数据网格(domain 自治) + DataOps。
11
设计千万 QPS 用户画像服务层15 min
PLATFORM

澄清需求

  • QPS（千万级）？延迟（P99 < 10ms）？画像字段数/大小？更新频率（T+1 + 实时）？

架构与数据流

  1. 离线构建：Spark 跑画像标签写入服务存储（HBase 存全量画像，D12）。
  2. 实时更新：Flink 把实时标签写入（覆盖/合并）。
  3. 热点加速：Redis 缓存高频访问画像（多级缓存，cache-aside，D12）。
  4. RowKey 设计：散列前缀防热点（D12 演示）。

关键 trade-off

  • Redis vs HBase：极热小画像走 Redis(内存,亚毫秒)；全量大画像走 HBase(持久,可扩)；组合用。
  • 缓存三问题：穿透(Bloom)、击穿(互斥)、雪崩(随机TTL)——千万 QPS 下必须处理（D12）。
  • 一致性：画像可重建，接受最终一致；坏了从离线重灌。

边界 + 演进

  • 大促扩容 + 限流降级（画像缺失返回默认）。演进：特征平台化，画像即在线特征（D13）。
12
设计 A/B 实验数据平台12 min
PLATFORM

核心组件（呼应 pm/data_ab_testing）

  1. 分流：稳定哈希分桶（同一用户始终同组），互斥/正交实验层管理。
  2. 埋点采集：实验曝光 + 转化事件入数据栈（D06）。
  3. 指标计算：离线(Spark)算各组指标 + 显著性检验（p 值/置信区间）。
  4. 看板 + 护栏：实时看核心指标 + 护栏指标（如崩溃率涨自动停实验）。

关键 trade-off

  • 实时 vs 离线指标：实时图早期看趋势，离线保统计严谨（防早停偏差）。
  • 分桶一致性：哈希要稳定且均匀，避免分桶不均导致结论失真（呼应 D12 热点/散列）。

演进

  • 从单指标到多指标 + 分人群下钻 + 序贯检验 + 自动决策。
