怎么答好 C++/Linux 底层设计

评分的 5 个维度

C++/系统岗的设计题偏「白板实现一个组件 / 设计一个高性能服务」，考察点从分布式转向内存布局、并发正确性、I/O 模型、延迟与 cache 友好。评分关键：先定接口与所有权（谁分配谁释放），说清并发同步手段与内存序，能定量分析延迟/吞吐瓶颈，并讲出取舍代价。

① 明确需求 + SLA

吞吐 / 延迟（P99）/ 并发度 / 单线程还是多线程 / 内存预算。定量目标决定数据结构与同步策略。

② 数据结构 + 内存

内存布局（连续 vs 指针）、所有权与生命周期（RAII / 智能指针）、cache line 对齐、避免伪共享。

③ 并发模型

锁 vs 无锁、临界区范围、内存序（acquire/release）、避免死锁与 ABA。先正确再优化。

④ I/O 模型

阻塞 / 非阻塞 / 多路复用（epoll）/ 异步（io_uring）；Reactor vs Proactor；零拷贝。

⑤ 取舍 + 瓶颈

延迟 vs 吞吐、内存 vs 速度、复杂度 vs 收益；说清竞争点、系统调用开销、可观测手段。

通用答题框架（6 段）

白板组件题也按这个骨架走
① 需求 + SLA — 吞吐 / 延迟 P99 / 并发度 / 内存预算 / 单核还是多核
② 接口设计 — 对外 API、所有权约定（谁分配谁释放）、错误处理
③ 数据结构 + 内存布局 — 连续/指针、对齐、cache 友好、生命周期（RAII）
④ 并发 / I/O 模型 + 取舍 — 锁/无锁、内存序、epoll/io_uring，每个说代价
⑤ 边界 + 正确性 — 死锁 / ABA / 伪共享 / 惊群 / 内存泄漏 / 溢出
⑥ 性能验证 + 演进 — benchmark、perf 火焰图、可观测、如何进一步优化
底层设计的第一分叉是「单线程还是多线程 + 锁还是无锁」。能用单线程（如 Redis）就别引锁；必须并发时，先用锁把正确性做对，profile 证明锁是瓶颈后再上无锁——无锁难写、易错、可读性差，是最后手段不是炫技。

01
设计高性能线程池18 min
CORE

需求 + 接口

  • 线程数策略？任务队列是否有界？是否要返回值（std::future）？优雅关闭？
  • 接口：submit(F&&, Args...) -> future<R>、shutdown()。

核心结构

  1. 固定 worker 线程 + 任务队列（queue<function<void()>>）+ mutex + condition_variable。
  2. worker 循环：wait 直到有任务 / 收到停止；取任务、解锁、执行。
  3. submit 用 packaged_task 包装拿 future，push 队列后 notify。

关键 trade-off

  • 单队列 vs 每线程队列 + work-stealing：单队列简单但锁竞争大；work-stealing（如 Tokio/TBB）扩展性好但复杂。
  • 有界 vs 无界队列：无界可能 OOM；有界满时需拒绝策略（阻塞 / 丢弃 / 调用者执行）。
  • 线程数：CPU 密集 ≈ 核数；I/O 密集可更多。避免过多导致上下文切换。

边界 + 正确性

  • 优雅关闭：置 stop 标志 + notify_all + join，处理完在途任务。
  • 异常：worker 内 try-catch，防一个任务抛异常拖垮线程。
  • 惊群：notify_one 而非 notify_all（单任务只唤醒一个）。

性能 + 演进

  • 减少锁持有时间（锁外执行任务）；演进：work-stealing、优先级队列、动态扩缩容。参见 并发模式。
02
设计无锁队列（SPSC / MPMC）18 min
CORE

需求

  • 生产者/消费者数量（单/多）？有界还是无界？延迟极致（低延迟场景）？

SPSC 环形缓冲（最简最快）

  1. 固定大小数组 + head（消费者写）+ tail（生产者写）两个原子索引。
  2. 生产者：tail.load(relaxed) 判满 → 写数据 → tail.store(release)。
  3. 消费者：head.load(relaxed) 判空 → 读数据 → head.store(release)。
  4. release/acquire 保证「数据写入」先于「索引发布」可见。

关键 trade-off / 陷阱

  • 伪共享：head 和 tail 放不同 cache line（alignas(64)），否则两核互相 invalidate，性能暴跌——面试核心考点。
  • MPMC 的 ABA：多生产多消费用 CAS 时，指针被复用导致 ABA → 用带版本号的 tagged pointer / hazard pointer。
  • 内存序：能用 relaxed 就别用 seq_cst；但正确性优先，讲清每个 load/store 为何用该序。
  • 无锁 vs 加锁：无锁减少阻塞但不减少竞争；低竞争下加锁可能更快更简单。

正确性 + 演进

  • ThreadSanitizer 验证无 data race。演进：MPMC 用 Michael-Scott 队列或 Vyukov MPMC bounded queue；回收内存用 hazard pointer / RCU。参见 内存模型。
03
设计内存池 / 对象池分配器15 min
CORE

需求

  • 为什么不用 malloc？高频小对象分配下 malloc 有锁竞争、元数据开销、碎片。目标：快、低碎片、可预测。

核心结构

  1. 固定大小对象池：预分配大块（arena），切成等大 slot，空闲 slot 串成 free list（侵入式，复用对象内存存指针）。
  2. 分配 = free list 头出队（O(1)）；释放 = 头入队（O(1)）。
  3. 多级（类 tcmalloc/slab）：按 size class 分桶；每线程本地缓存（thread-local）避免锁。

关键 trade-off

  • 固定大小 vs 通用：固定池极快无碎片但只服务一种大小；通用分配器灵活但慢。
  • 线程本地 vs 全局：thread-local 免锁但内存不共享（可能浪费）；全局省内存但需锁。
  • 内存归还 OS：池化保留内存降低分配延迟，代价是常驻内存高。

边界 + 演进

  • 对齐（alignas）、越界检测（debug 加哨兵）、防止 double free。演进：STL 自定义 allocator 接入、PMR（std::pmr）monotonic/pool resource。
04
设计线程安全 LRU 缓存15 min
CORE

需求 + 接口

  • get(k) / put(k,v) 都要 O(1)；容量上限；并发访问。

核心结构

  1. 哈希表 + 双向链表：hash map 存 key→链表节点指针（O(1) 定位）；双向链表按访问顺序排（头最新，尾最旧）。
  2. get：查表 → 命中则把节点移到头 → 返回。
  3. put：存在则更新 + 移头；不存在则插头，超容量删尾节点 + 从表移除。

并发 trade-off

  • 全局锁 vs 分段锁：一把大锁简单但热点争用严重；分片（sharding）按 key 哈希分多个桶各自加锁，大幅降竞争——高并发首选。
  • 读写锁：get 也要移动节点（改链表）→ 其实是写操作，读写锁收益有限；用近似 LRU（如 CLOCK / TinyLFU）可让读免改结构。

边界 + 演进

  • 过期 TTL、命中率统计。演进：LFU / TinyLFU（Caffeine）提升命中率；对应分布式版见 分布式缓存。

设计高并发 TCP 服务器（C10K/C10M）20 min
VERTICAL

需求

  • 并发连接数（万级/百万级）？长连接还是短连接？请求特征（CPU 密集/IO 密集）？延迟要求？

I/O 模型演进

  1. 为什么不能一连接一线程：万级连接 = 万级线程，上下文切换 + 栈内存爆炸。
  2. epoll（多路复用）：一个线程管数千 fd，事件驱动；ET vs LT：ET 高效但必须循环读到 EAGAIN。
  3. Reactor 模式：主 Reactor accept，多个 sub-Reactor（每个绑一线程 + 一 epoll）处理读写 → 主从 Reactor 多线程（Netty/muduo 模型）。
  4. io_uring：真正异步、减少系统调用，新一代高性能方案。

关键 trade-off

  • Reactor vs Proactor：Reactor（同步非阻塞，就绪通知）Linux 主流；Proactor（异步完成通知）需 io_uring/IOCP。
  • 线程模型：one loop per thread（每线程独立 epoll）避免锁；连接分配注意负载均衡。
  • 惊群：多线程 accept 同一 fd → 用 SO_REUSEPORT 或 EPOLLEXCLUSIVE 解决。

边界 + 演进

  • 零拷贝（sendfile/splice）、缓冲区管理、优雅关闭、背压。演进：用户态协议栈（DPDK）做极致低延迟。参见 异步网络。
06
设计定时器系统（时间轮）15 min
VERTICAL

需求

  • 海量定时任务（连接超时、重试）？精度？add/cancel 频繁？O(1) 还是 O(log n)？

方案对比

  • 最小堆：add / pop O(log n)，取最近到期 O(1)；实现简单，适合任务量中等。
  • 红黑树（如 nginx）：有序，O(log n)。
  • 时间轮：环形数组 + 指针 tick 前进，add/cancel O(1)；适合海量短时定时器。
  • 分层时间轮：秒/分/时多级轮（类似时钟），支持长跨度定时（Kafka/Netty 用）。

关键 trade-off

  • 堆 vs 时间轮：堆精确、实现简单但 O(log n)；时间轮 O(1) 但有精度（tick 粒度）与空槽开销。
  • 驱动方式：与 epoll 集成用 timerfd / epoll_wait 的 timeout 驱动 tick。

边界 + 演进

  • 任务取消（时间轮用 slot 内标记删除）、到期任务在 I/O 线程执行还是投线程池。分布式延时队列见 延时队列。
07
设计异步低延迟日志库15 min
VERTICAL

需求

  • 吞吐（百万行/s）？不阻塞业务线程？崩溃时不丢日志？多线程写？

核心结构

  1. 前端无锁 / 双缓冲：业务线程把日志写入 thread-local 缓冲或无锁队列，不做 I/O（关键：I/O 移出热路径）。
  2. 后端后台线程：定时/满时把缓冲批量 flush 到磁盘（顺序写、大块写）。
  3. 双缓冲（muduo 模型）：前端写 buffer A，后端交换后落 buffer B，减少锁持有到「交换指针」瞬间。

关键 trade-off

  • 异步 vs 同步：异步低延迟但崩溃可能丢内存中未落盘日志；关键审计日志需同步 / fsync。
  • 格式化时机：在业务线程格式化（占用业务 CPU）vs 后端格式化（传参复杂）；低延迟场景倾向后端格式化甚至二进制日志。
  • 丢弃 vs 阻塞：队列满时丢日志保业务，还是阻塞保完整——按日志级别分级。

边界 + 演进

  • 日志滚动（按大小/时间）、崩溃时 flush（signal handler 谨慎）。演进：结构化日志、采样、接入可观测体系。
08
设计轻量 RPC 框架18 min
VERTICAL

需求

  • 同步/异步调用？序列化格式？服务发现？超时/重试？连接复用？

核心分层

  1. 协议层：自定义帧（magic + 长度 + 序列号 + payload），解决 TCP 粘包/半包（长度前缀法）。
  2. 序列化层：Protobuf（跨语言、紧凑）/ FlatBuffers（零拷贝）/ JSON（可读）。
  3. 传输层：epoll + 连接池复用长连；请求用序列号匹配响应（异步多路复用）。
  4. 服务治理：注册中心（etcd/ZK）服务发现 + 负载均衡 + 超时熔断（参见 限流熔断）。

关键 trade-off

  • 同步 vs 异步：同步简单（一请求占一线程）；异步（future/callback）高吞吐但代码复杂。
  • 序列化选型：Protobuf 通用；追求极致延迟用 FlatBuffers/自定义避免解码拷贝。
  • 连接复用：多路复用（一连接跑多请求）省连接数，需序列号解复用。

边界 + 演进

  • 超时管理、背压、优雅上下线。演进：流式 RPC、拦截器链、多语言 stub 生成（对标 gRPC）。参见 API 设计。

09
设计低延迟撮合引擎（订单簿）18 min
PLATFORM

需求

  • 延迟（微秒级）？吞吐（百万单/s）？确定性（可回放）？公平性（价格-时间优先）？

核心结构

  1. 订单簿：买/卖两侧按价格排序（map<price, queue> 或数组价格档），同价按时间 FIFO。
  2. 撮合：新单与对手方最优价撮合，成交则减量、生成成交回报。
  3. 单线程 + 事件回放：撮合核心单线程（避免锁、保证确定性），输入定序后串行处理，可回放复现。

关键 trade-off（低延迟工程）

  • 单线程 vs 多线程：撮合用单线程保确定性与免锁——LMAX Disruptor 思路；并行只在前置校验/风控。
  • 数据结构：价格档用数组（cache 友好、O(1)）优于 map（指针跳跃、cache miss）。
  • 延迟优化：预分配对象池（零 GC/零 malloc 热路径）、避免异常、绑核（CPU affinity）、内核旁路（kernel bypass）。

边界 + 演进

  • 崩溃恢复用事件日志回放；风控前置。演进：Disruptor 环形队列做输入总线、NUMA 感知。参见 低延迟指南。

设计嵌入式 KV 存储引擎18 min
PLATFORM

需求

  • 读多还是写多？数据量（内存放得下吗）？持久化 + 崩溃恢复？范围查询？

存储模型选型

  • B+Tree（读优化）：原地更新、读快、范围查询好；写放大较高。适合读多写少（对标 InnoDB）。
  • LSM-Tree（写优化）：内存 MemTable + WAL，满则 flush 成有序 SSTable，后台 compaction；写快（顺序写）、读需查多层 + 布隆过滤器。适合写多（对标 RocksDB）。

关键 trade-off

  • 读/写/空间放大三角：LSM 写放大低但读/空间放大高；B+Tree 反之。面试核心。
  • WAL + MemTable：先写 WAL 保证崩溃不丢，再写内存；恢复时重放 WAL。
  • compaction 策略：Leveled（读优、写放大高）vs Tiered（写优、空间放大高）。
  • mmap vs 直接 I/O：mmap 简单借用 page cache 但缺乏控制、缺页不可控；直接 I/O 可控但要自管缓存。

边界 + 演进

  • 并发用 MVCC / 快照隔离；布隆过滤器降无效读。演进：分层存储、压缩、二级索引。参见 存储架构。
11
设计连接池（DB / RPC）12 min
PLATFORM

需求

  • 为什么要池化？建连（TCP + TLS + 认证）昂贵，复用降延迟。目标：控制并发连接数、复用、健康检查。

核心设计

  1. 空闲队列 + 活跃计数：借出（无空闲则新建至 max，达上限则等待/超时）、归还（放回空闲或关闭）。
  2. 参数：min/max 连接、空闲超时、获取超时、最大生命周期。
  3. 健康检查：借出前/定期 ping，剔除失效连接（防拿到已断连接）。

关键 trade-off

  • 池大小：太小排队、太大压垮 DB（连接数 × 后端 = 资源）。经验：与 DB 核数相关，非越大越好。
  • 等待 vs 快速失败：满时阻塞等待（可能雪崩）还是立即失败（保护后端）。

边界 + 演进

  • 连接泄漏检测（借出未归还告警）、TCP keepalive、连接预热。演进：动态调整池大小、多数据源路由。
12
设计 C++ 协程调度器15 min
PLATFORM

需求

  • 为什么协程？以同步写法写异步 I/O，避免回调地狱，比线程轻量（切换无系统调用）。

核心结构（C++20 coroutine）

  1. 协程帧：编译器把 co_await/co_yield 的函数转成状态机，挂起点保存局部状态到堆上的帧。
  2. 调度器：就绪队列 + 事件循环；协程 co_await I/O 时挂起，注册到 epoll，I/O 就绪后 resume。
  3. M:N 模型：M 个协程跑在 N 个线程上（work-stealing 调度），如 folly/coro、cppcoro。

关键 trade-off

  • 有栈 vs 无栈协程：C++20 无栈协程（省内存、切换快，但不能在普通函数中间挂起）；有栈（如 boost.fiber，灵活但每协程一栈占内存）。
  • 协程 vs 线程：协程切换在用户态（纳秒级、无内核参与），适合海量 I/O 并发；但一个协程阻塞会卡住整个线程。
  • 调度公平性：协作式调度需协程主动让出，长任务需手动 yield 防饿死。

边界 + 演进

  • 协程生命周期与悬垂引用（帧在堆上，注意 co_await 的临时对象）。演进：结构化并发、取消传播、绑定 io_uring。参见 协程深入。
