AI Tech Lead / EM 面试指南

5 章 · 20 道高频题 · 技术 + 商业 + 团队 三维度。对位读者：5+ 年工程经验想转 AI 团队 lead / EM。

00

读前定位 · 这本"招聘官最爱筛 AI TL 的话题"

市场上 80% 的 "AI EM" 其实是 "管 AI 子方向的传统 EM"，真正端到端做过 AI 团队 + 产品的 EM < 1%。本文帮你抢这 1%。

AI TL/EM 面试评分 5 维度：① 技术深度（能跟工程师对话）② 商业理解（懂 ROI / Cost）③ 团队建设（招聘 + onboarding + 留人）④ 跨部门（PM / 法务 / SRE）⑤ Roadmap（短中长期规划）。5 维都要过，不是单项很强就行。
01

AI TL/EM 跟传统 TL 的 5 大区别

① GPU 预算管理

传统 EM 主要管人力 cost。AI EM 还要管 GPU cost（典型 1 张 H100/月 ~$3-5k）。要懂 utilization / cost-per-token / per-feature ROI。

② Eval 体系

传统软件成功 = bug-free。AI 没有 bug-free，只有 metric trade-off。EM 要懂 eval pipeline 设计 + multi-objective 权衡。

③ 模型选型

传统 EM 选 DB / framework。AI EM 还要选 base model（GPT-4 vs Claude vs Qwen vs DeepSeek vs 自训）→ 影响 5-10x cost + quality。

④ Safety + 合规

传统 EM 关注 security。AI EM 多一层 safety：幻觉 / 越狱 / PII / 监管（EU AI Act / 国内生成式 AI 备案）。

⑤ 团队人才结构

传统团队全是 SDE。AI 团队混合：ML Eng + Infra Eng + LLM App Eng + 数据 + PM + 标注。沟通成本高。

共同 · uncertainty 管理

AI 项目本质高 uncertainty。EM 要让 leadership 接受"今天不能保证 3 月效果到 90%"，但能讲清楚怎么 manage 不确定性。

02

技术维度 · 8 道高频题

Q1你怎么决定用 OpenAI API 还是自部署开源模型？
框架
4 维度：① cost（高 QPS 自部署省钱）② quality（GPT-4 强但开源差距缩小）③ 合规（国内必须自部署）④ 团队能力（自部署需要 Infra 团队）
典型决策
POC 用 OpenAI 快 → 跑到产品市场契合 → 自部署降 cost → 必要时 fine-tune。不要一开始就自部署（验证之前过度工程）。
Q2团队想做 fine-tune 你怎么评估值不值？
先问
"prompt 工程能不能解决"。多数情况能。fine-tune 是最后手段。
值得的场景
① 风格 / 格式定制（base 学不到）② 私有 domain knowledge ③ 一致性要求高 ④ 推理 cost 占大头（小模型 fine-tune 替代大模型）
不值
① 知识更新快（不如 RAG）② 数据量小（< 1k 样本 fine-tune 容易过拟合）③ 团队没数据工程能力
Q3你怎么 review 团队的 RAG 系统设计？
5 个 must-have 问
① eval set 多少题？② 是 hybrid search 吗？③ 有 cross-encoder rerank 吗？④ Cost / query 多少？⑤ 失败 case 多少 % 是 retrieval 错 vs LLM 错？
答不上的设计是不完整的。
Q4团队 AI 项目准确率 80% 想上线，你 approve 吗？
看场景
① 客服 RAG 80% + 人工兜底 → ✅ ② 医疗诊断 80% → ❌（需要 > 95%） ③ 代码补全 80% → ✅（用户能修） ④ 退款审批 80% → ❌（钱的事）
关键
"错的代价" 决定 threshold。EM 要跟 PM/法务一起定每个场景的"可接受准确率"。
Q5团队报 GPU 不够要新买 100 张 H100，你怎么验证？
5 问
① 当前 utilization 多少？（< 50% → 不批，先优化）② 推理还是训练？训练算 chinchilla budget。③ 量化做了吗？④ 用 vLLM/SGLang 了吗？⑤ Prefix Caching / Spec Decoding 开了吗？
多数 "GPU 不够" 其实是工程效率问题。
Q6怎么评估团队的 AI 工程能力？
5 个 marker
① 有 eval set + CI 拦截退化 ② 至少 1 个量化 metric 改善（如 accuracy +X%）③ 有失败 case 库 + postmortem ④ 有 cost dashboard ⑤ 至少 1 个工程师能讲清 vLLM 内部
Q7团队 AI 服务 P95 latency 8s 流失用户，你怎么排查 + 决策？
技术路径
profile latency 分布 → 找瓶颈 → 应对：① LLM generation 慢 → 换小模型 + Spec Decoding ② network → CDN ③ retrieval → cache + 精排
决策路径
跟 PM 谈 trade-off：accuracy -3% latency -75% 哪个赢？通常 latency 赢（用户不耐烦），但要 A/B test 验证。
Q8团队提出用 Multi-Agent 你怎么判断？
默认 skeptical
Multi-agent 增加 cost + latency + 复杂度。能 single-agent 不要 multi-agent。
值得的场景
① 任务真的需要分工（如 coder + reviewer + tester） ② 单 agent 上下文塞不下 ③ 需要 specialized 角色
03

商业维度 · 6 题

Q9给 CEO 报告 AI 项目 ROI 怎么算？
结构
① 投入：人力月 × N + GPU $X + API $Y
② 产出：DAU + / 转化率 + / 客服成本 - / 客单价 +
③ ROI = (产出 - 投入) / 投入
④ 风险：accuracy 退化 / cost 上涨 / 合规
⑤ 下季 ask：要多少钱 / 解决什么
不讲技术细节，讲钱。
Q10AI 产品 cost 砍 50% 怎么做？
5 招
① Smart Routing：简单问题用小模型（cost -5x）② 量化 INT4（cost -3x）③ Cache 高频 query（命中 30% → cost -30%）④ Continuous Batching + 高 max_num_seqs（utilization +50%）⑤ Multi-LoRA（1 base 服 N 用户）
Q11怎么跟 CFO 谈 GPU 预算？
CFO 关心
不是 GPU 数，是 cost-per-customer / cost-per-feature / payback period。
框架
"现在 1 张 H100 服务 100 DAU。Plan A 加 10 张 H100 = $30k/月，能多服 1000 DAU = 增 $X 收入，payback 3 月。Plan B 不加，用户增长封顶 limit revenue。"
Q12买 vs 自建 vs SaaS 怎么选？
SaaS
(OpenAI API) — POC 快、cost 高、合规风险。用于 0-1 验证。
自建
(vLLM 自部署) — POC 慢、cost 低、可控。规模化后必走。
买
(Cohere / Glean enterprise) — 中庸方案，适合中型企业。
Q13怎么定 AI 产品定价策略？
3 模式
① 按 token / call：透明，但用户难预算 ② seat-based：好预算，但重度用户占便宜 ③ 混合：fixed + overage（主流）
关键
必须搞清楚 cost-per-call 才能定价。很多 AI 公司亏钱因为定价时没算清 cost（Devin $500/月被吐槽就是没算清）。
Q14团队提议自训 base 模型，你 approve 吗？
默认 NO
训 7B 模型至少 $100k + 1 月 + 顶级 ML 团队。除非有强差异化数据 + 长期战略 + 充足资源，否则 fine-tune 现有模型 ROI 高 10-100x。
值得的少数情况
① 公司战略要 own model（如 DeepSeek） ② 行业 specific (法律 / 医学) 而开源没好选项 ③ data moat 强
04

团队维度 · 6 题

Q15怎么 onboarding 新 AI 工程师？
30/60/90
D1-30 跑 1 个 RAG demo + 学 eval 体系 + 读核心文档
D30-60 独立做 1 个 small feature + 我每周 review
D60-90 主导小项目 + 自己跟 PM/数据沟通
关键
让他做有产出的事，每周 1on1 反馈不憋，给犯错空间。
Q16怎么招 senior AI 工程师？面试看什么？
必考
① "你怎么知道你的 AI 系统变好了"（eval 思维）② "讲一次失败 case"（诚实 + ownership）③ 系统设计 1 题（架构思维）④ "你怎么 own 一个 AI 项目从 0 到 1"（项目管理）
红旗
① 只讲技术不讲数字 ② 只讲框架不讲业务 ③ 简历堆 25 个技能 ④ 项目里全是 "参与 / 协助"
Q17AI 团队结构（人才配比）怎么搭？
典型 10 人 LLM App 团队
1 EM + 1 TL + 3-4 LLM App Eng + 1-2 ML Eng (fine-tune) + 1 Infra Eng + 1 Data Eng + 1 PM。不需要每个都 Senior，1-2 个 senior 撑架构其他可以 mid。
Q18团队成员被竞争对手挖怎么办？
短期
① 不忽视 counter-offer 决策（但要量化代价：替换 cost + 信息泄露）② 1on1 真诚理解 why（钱？成长？技术？关系？）③ 必要时给 retention bonus + 调级
长期
不可能留所有人。重点：① 招的时候选 culture fit ② 持续 1on1 知道 morale ③ 关键岗位有 backup ④ 知识沉淀（人走 knowledge 不走）
Q19团队跟 PM / 法务 / SRE 冲突怎么处理？
跟 PM
用 eval 数据说话（"accuracy +5% 但 latency +2x，怎么 trade"）
跟法务
提前 1 月 review LLM 输出（PII / 版权 / 监管）+ 准备 audit log
跟 SRE
定 SLO 三角（quality + latency + cost）+ runbook + on-call
关键
把 AI 不确定性翻译成对方语言（PM 看 metric / 法务看 risk / SRE 看 SLA）
Q20AI 团队怎么沉淀经验？
5 个机制
① ADR（每个技术决策写）② Postmortem（每个 incident 写）③ Skill 库（团队 .cursor/skills/）④ 月度 brown bag（分享）⑤ 公开 eval set（所有人贡献）
05

面试官真在看什么

✅ 加分

  • 能讲 production 数字（DAU / 准确率 / cost / NPS）
  • 能讲失败 case + 学到什么
  • multi-objective 思考（不只 accuracy）
  • 承认 uncertainty + 提 mitigation
  • 讲 "我们" 比讲 "我" 多（团队思维）但不能全 "我们"

❌ 减分

  • 只讲技术不讲业务
  • '感觉变好了' 没数据
  • 项目里没具体决策的 trade-off
  • 从不承认错误
  • 讲战略 vague 没落地路径
  • 过度迷信新技术（什么火都要用）
最重要：AI 招聘官在找的是"能把不确定性变成确定性"的 leader。AI 项目本质高 uncertainty。如果你能在简历 + STAR + 系统设计里反复展现"我面对不确定性怎么做决策、试错、收敛"，你就赢了一半。
