怎么答好 AI 系统设计

招聘官评分的 5 个维度

AI 系统设计跟传统系统设计 80% 一样（容量估算 / 数据流 / 高并发 / 监控），20% 是 AI 特殊性（cost / latency / quality / safety / eval）。评分关键不是"答对"，是"展现你的思维方式"。

5 个评分维度

① 澄清需求（5 min）

不要听完就开干。主动问：QPS / DAU / 用户群体 / 输入特性 / SLO（latency / accuracy / cost）/ 现有 stack / 时间预算。不澄清的候选人直接 -1 档。

② 容量估算（5 min）

数字背诵：A100 80GB / H100 80GB / FP16 = 2 bytes / Llama-7B = 14GB / Qwen-72B = 144GB / 1 token ≈ 3-4 字符 / 1 H100 推理 ≈ 1000 tokens/s。

③ 架构分层（15 min）

从用户请求 → API → 业务逻辑 → AI 服务 → 数据。每层标：技术选型 + 理由 + 替代方案。

④ AI 特殊性（10 min）

必谈：Eval 体系 / Cost 治理 / Safety / Fallback 兜底 / 监控告警 / 灰度发布。这是 senior 跟 junior 区别。

⑤ Trade-off + Future Work（5 min）

主动指出：① 当前方案的 3 个 trade-off ② 1-3 个 future improvement（scale 上去后怎么演化）。展示你"已经做过这事"的真实感。

01
设计 ChatGPT 类对话系统25 min
CORE

澄清需求

  • QPS / DAU / 同时在线用户？
  • 是否要免费 + 付费 tier？
  • 多轮对话保留多久（小时 / 永久）？
  • 支持哪些语言 / 模型？自研还是接 API？

容量估算（假设 1M DAU / 平均 10 轮对话 / 每轮 500 token）

  • 每天 token 量：1M × 10 × 500 = 5B tokens/day
  • QPS：1M × 10 / 86400 ≈ 120 QPS（高峰 ~500 QPS）
  • GPU 需求：每张 H100 跑 7B 模型 ~1000 tokens/s → 需要 ~50 张 H100 应对高峰

架构（7 层）

  1. 前端：Next.js + WebSocket（streaming token）
  2. API Gateway：鉴权 + rate limit + routing
  3. 对话状态服务：Redis 缓存 session + Postgres 持久化（多轮历史）
  4. Prompt 编排：组装 system prompt + 历史 + 当前 query
  5. LLM 推理：vLLM cluster（Qwen-72B / Llama-3-70B 自部署）
  6. Safety Guardrail：输入 prompt injection 检测 + 输出 PII / 暴力检测（用 Llama Guard）
  7. 日志 + 反馈：每次对话存 log + 👍/👎 按钮 → eval pipeline

关键 trade-off

  • 模型选型：闭源 API（GPT-4）质量高但贵 → 开源自部署（Qwen-72B）省 5-10x cost
  • 多轮历史长度：长 = 体验好但 cost 涨 → 典型 20 轮 + 自动 summarization 老对话
  • Streaming vs 完整返回：Streaming TTFT 体验好 → 复杂度高

边界 + Fallback

  • 模型 timeout → 自动 retry 1 次，再失败返回"网络繁忙请稍后"
  • Safety check 拒答 → 返回标准化拒答模板
  • 用户超 rate limit → 提示升级付费
  • 新用户 cold start → 预设示例 prompt 引导

监控 + 演化

  • Day 1：单 region + 单 model + basic dashboard
  • Month 3：多 region + 多模型 routing（简单题用小模型）+ 自动 eval pipeline
  • Year 1：自研 fine-tuned 模型 + Multi-LoRA serving + advanced safety + reasoning model 路由（难题切 o1/R1）
02
设计企业知识库 RAG 系统25 min
CORE

澄清需求

  • 知识库规模？（10万文档 vs 1000万）
  • 文档类型？（PDF / Word / Confluence / 代码）
  • 更新频率？（实时 / 每天 / 每周）
  • 用户数 / QPS？
  • 准确率要求？（80% 还是 95%）

架构（5 层 + 离线 pipeline）

离线 pipeline（每天跑）：

  1. 文档采集（API / 爬虫 / SDK）
  2. 解析 + chunking（按段落 + overlap）
  3. Embedding（BGE-Large-zh / Qwen3-Embedding）
  4. 写入向量库（Qdrant / Milvus）+ BM25 倒排（OpenSearch）+ metadata（version / source）

在线 pipeline（每次查询）：

  1. Query 改写 + 解析（提版本号 / 关键实体）
  2. Hybrid Search（BM25 + Dense + RRF 融合，召回 100）
  3. Cross-Encoder Rerank（精排到 top-5）
  4. Prompt 组装 + LLM 生成（带 citation）
  5. Faithfulness check（基于 context 验证答案）

关键 trade-off

  • 实时 vs 批量更新：实时（如订单查询）需要 stream 索引；批量（如 SOP 文档）每天跑 batch
  • 纯 dense vs hybrid：hybrid +15% recall 但需多部署 BM25 → 默认 hybrid
  • LLM 大小：72B 准确率 +5% 但 cost 10x → 7B + rerank 通常足够

AI 特殊性

  • Eval：建 200 题 eval set + RAGAS 4 metric + CI 拦截退化
  • 失败 case：手动 review 100 个准确率低的 query → 归类（chunk 错 / embedding 错 / LLM 错）→ 针对性修
  • Cost：embedding API 调用聚合（batch 100 一次）

演化

  • Day 1：固定 chunking + 纯 dense + GPT-4 + 100 题 eval
  • Month 3：智能 chunking + hybrid + rerank + 切自部署 Qwen
  • Year 1：GraphRAG（实体关系）+ Self-RAG（自检）+ multi-language + 自动收集失败 case 持续优化
面试加分：能给出"我做的 RAG 上线 6 个月，准确率从 47% → 89%，具体怎么调的"的真实数字 + 失败 case。展示你不是看 LangChain demo 答的。
03
设计百万 QPS LLM 推理服务30 min
CORE

澄清

  • 请求平均/P95 token 长度？
  • SLO：P95 latency 多少？accuracy/quality 要求？
  • 模型大小固定还是可选？
  • 预算（GPU 数 / $ / 月）？
  • region 数？

容量估算（1M QPS / 平均 500 token / Qwen-72B）

  • 每秒 token：1M × 500 = 500M tokens/s
  • 单 H100 跑 72B FP8 量化 ≈ 300 tokens/s（高 batch）→ 需要 1.6M 张 H100 ?!
  • 不对！ 1M QPS 不现实（甚至 ChatGPT 高峰只有几十 K QPS）→ 真实场景需要重新跟面试官确认 QPS
  • 假设修正为 100k QPS → 50M tokens/s → 160k H100 → 仍极大 → 用 7B + INT4 + 大 batch → 单卡 5000 tokens/s → 10k H100

架构（多 region 多层）

  1. Global LB：DNS GeoIP 路由到最近 region
  2. Regional LB：根据 QPS / 健康度 / model availability 分流
  3. Smart Router：简单 query 路由小模型（7B），复杂 query 路由大模型（72B）→ cost ÷ 5
  4. vLLM Cluster：每 node = 1 H100 + vLLM + Continuous Batching + Prefix Caching
  5. Prefill-Decode 分离：H100 prefill cluster + A100 decode cluster
  6. KV-Cache Pool：多级（GPU HBM → CPU RAM → 远端 Redis）
  7. Cache 层：identical query 24h cache + 半相似 cache（embedding 距离）

关键优化（每个都标 throughput 提升）

优化throughput 提升代价
vLLM Continuous Batching+3-4x无（默认）
FP8 量化+1.5-2x精度 -1%
INT4 KV-Cache+1.5-2x长上下文 -1-2%
Smart Routing（小模型分流 60%）+3x需要分类器
Cache 层（命中率 30%）+1.4x缓存 staleness
Prefill-Decode 分离+1.3-1.5x架构复杂
Speculative Decoding+1.5-2x实现复杂

叠加：理论可达 50x naive，实际 10-20x 已经很好。

Fallback + 容灾

  • 单 GPU 故障 → vLLM 自动迁移请求到健康卡
  • 整 region 挂 → 流量切其它 region（接受 +50ms latency）
  • 大模型 down → 降级到小模型 + 标记"基础版"
  • 极端高峰 → priority queue + 低优先级用户排队
04
设计 Code Copilot（Cursor / Copilot 类）25 min
CORE

2 大场景

  • Tab 补全：实时（latency < 300ms） + 高频（每分钟几十次）→ 用小模型 + 高 cache 命中
  • Chat / Agent：复杂任务 → 用大模型 + tools + RAG codebase

关键架构

  1. Codebase Indexing：用户授权后 embedding 整 repo（ast-grep 分块 + bge-code）+ 增量更新
  2. Context Builder：当前文件 + 当前光标位置 + 相关函数（vector search） + LSP 信息
  3. Model Router：Tab 补全用 1B 小模型 / Chat 用 GPT-4 等大模型
  4. Cache 层：Tab 补全识别"小改动可复用 cache"
  5. Agent Loop（Chat 模式）：plan → action → observation → iterate
  6. Tools：read/write file / run terminal / git diff / web search

关键 trade-off

  • Codebase 上传到云 vs 本地处理：合规风险 → 提供 enterprise on-prem 版
  • Tab 补全准确率 vs 体验：50% 准确率烦人，需要 > 80% 才不影响心流
  • Agent 自主度：太多 autonomy 用户失控 → 大改动必须用户 review

Eval

  • Tab 补全 metric：接受率（用户接受补全 vs 拒绝）+ 撤销率（接受后又撤销 = 反向信号）
  • Chat metric：任务完成率 + 用户改写率 + thumbs up/down
  • Benchmark：SWE-bench 跑真实 GitHub issues

设计电商客服 AI Agent25 min
CORE

需求 + 容量

  • 100k 商品 + 500k 用户咨询/天 + 准确率 > 85% + P95 latency < 3s
  • 问题类型：物流（30%）/ 商品（25%）/ 售后（20%）/ 退款（15%）/ 其它（10%）
  • 不能答错的题：法律 / 价格 / 退款政策 → 必须基于 SOP

架构（多 agent 协调）

  1. 意图分类：先小模型分类（物流/商品/售后/退款），路由到对应专家 agent
  2. 专家 Agent：每类 1 个 agent，自带专属 RAG + tools
    • 物流 agent：tools = 查物流 API
    • 商品 agent：RAG = 商品库
    • 退款 agent：RAG = 政策文档 + tools = 提交退款 API
  3. Safety Gate：所有输出过滤 PII / 不当言论
  4. Fallback 转人工：confidence < 0.7 自动转

关键 trade-off

  • Single agent vs Multi-agent：multi-agent 准确率 +10% 但 latency +50% → 选 multi-agent，加缓存抵消 latency
  • RAG vs Fine-tune：政策 RAG（更新快）+ 风格 fine-tune（让 agent 更礼貌）
  • autonomy 程度：纯 AI 风险大 → 80% AI + 20% 转人工，逐步提 AI 占比

上线节奏

  • Week 1-2：5% 灰度（仅简单问题）+ 24h fallback 转人工
  • Week 3-4：收 100 failure case + 优化 + 扩到 20%
  • Month 2-3：扩到 80%（持续质量监控）
  • 关键：从来不 100%（保留 20% 人工 + 边界 case 兜底）

设计 LLM-Enhanced 推荐系统20 min
VERTICAL

核心想法

传统推荐 = 协同过滤 + DNN。LLM 加什么：① 长尾用户冷启动（用 LLM 从用户少量行为生成兴趣向量）② 解释性（"为什么推这个"）③ 复杂 query 理解（"想买适合露营的便宜帐篷"）。

架构

  1. 召回层：传统 collaborative filtering（千万级 item 召回 1000）
  2. LLM 兴趣理解：用户历史 + few-shot → LLM 输出兴趣 embedding（每天 batch 更新）
  3. 精排：传统 DNN（cross-feature + DLRM）+ LLM 兴趣 embedding 作为额外特征
  4. 重排：LLM 看候选 top-50 + 用户画像 → 选 top-5 并生成"推荐理由"
  5. Cache：相同用户 1h 内 LLM 重排 cache

关键 trade-off

  • LLM 重排 cost 10-100x 传统 DNN → 只对 top 用户 / 大订单用 LLM
  • LLM 解释可能"hallucinate" → 强约束基于商品真实属性
07
设计 LLM-Enhanced 搜索（Perplexity / Bing Chat 类）20 min
VERTICAL

跟传统搜索区别

传统：返回 link list。LLM 搜索：返回完整答案 + citation。

架构

  1. Query 理解：LLM 改写 query（解决模糊 / 分解多 sub-query）
  2. Web 检索：调 search engine API（Bing / Google）拿 top-10 链接
  3. 页面抓取 + 提取：fetch HTML → 提取正文（trafilatura / Readability.js）
  4. Rerank：用 Cross-Encoder 选 top-3 最相关页面
  5. LLM 生成答案：基于 top-3 内容生成答案 + 严格 citation
  6. Citation 验证：自动检查答案每个 claim 都能在 source 找到

关键挑战

  • 实时性：page fetch 是 latency 大头 → 并行 + cache 热门 page
  • Hallucination：必须强 citation + faithfulness check + 拒答"找不到证据"
  • Cost：每次搜索 ~10 LLM 调用 → 高频 query 必须 cache

设计多模态搜索（电商图搜 / 视频搜索）20 min
VERTICAL

核心组件

  1. Embedding model：CLIP / SigLIP / Qwen-VL（图文统一 embedding 空间）
  2. Multi-vector index：每张图存图像 embedding + 文字描述 embedding + metadata
  3. Query 路径：① 用户传图 → image embedding ② 用户输文 → text embedding ③ 混合 query → 两者加权
  4. Re-rank：用 VLM 对 top-50 候选打分（"这张图跟 query 有多匹配"）

挑战

  • 视频：分帧 + 关键帧 embedding 减存储
  • 规模：1 亿+ 图片 embedding，需 IVF / HNSW 高效索引
  • 更新：新品上架立刻可搜 → 增量索引

设计智驾感知系统的端到端 VLM 推理25 min
VERTICAL

硬约束（跟通用 AI 完全不同）

  • latency < 100ms (P99)：车要立刻反应
  • 车规级算力：Orin/Thor（200-1000 TOPS，比 H100 弱）
  • 功耗 < 100W：车的电池有限
  • 可解释：失败必须能 debug（不能黑盒 fail）
  • ISO 26262 ASIL-D：功能安全认证

关键设计

  1. 量化激进：INT8 主流 / INT4 实验（边缘设备没 FP16 算力）
  2. 模型小：3B-8B 主流（大模型功耗扛不住）
  3. 视觉 encoder + LLM decoder 分离：encoder（ResNet/ViT）共享给多任务
  4. 多任务 head：检测+分类+规划共享 backbone（减重）
  5. rule-based fallback：AI 失败时退化到传统规控（保安全）
  6. 云端大模型协助：紧急场景上传到云端用大模型分析（5G 网络）— 实验阶段

训练 vs 推理

  • 训练：在云端用万张 H100 train 大 VLM
  • 蒸馏：用大 VLM 蒸馏到 1-3B 小模型
  • 量化：小模型 INT8/INT4
  • 部署：car-side OTA 更新
智驾 AI Infra 跟数据中心 AI Infra 完全不同：追求 latency 上界 + 可靠性 + 安全合规 而非 throughput。这是 C++ 老兵转智驾 AI 比转数据中心 AI 更直接 的原因（已有低延时 + 实时性 + 安全合规经验）。
10
设计具身机器人决策系统（VLA 模型）25 min
VERTICAL

VLA = Vision-Language-Action

从 Vision（摄像头）+ Language（人类指令）→ 输出 Action（机器人关节角度 / 速度）。是 2025+ 具身智能的核心范式。

典型架构

  1. 感知层：多摄像头 + 力学传感器 → 多模态 embedding
  2. VLM 理解层：理解场景 + 用户指令 "把杯子放桌上"
  3. Action Decoder：transformer decoder 输出 action token（30-50Hz）
  4. 低层控制：传统控制器执行 action（PID / MPC）
  5. Safety Layer：碰撞检测 / 力学限制 / Emergency Stop

训练 pipeline

  • 遥操作数据：人远程操作机器人，收集 action 数据（最贵）
  • 仿真：Isaac Sim / MuJoCo 生成大量数据（cheap）
  • Sim2Real：domain randomization 让模型在真实环境鲁棒
  • RL fine-tune：真机器人上做少量 RL 微调

挑战

  • 实时性：30-50Hz 推理（latency < 30ms）
  • 数据稀缺：真机器人数据极贵，大量靠仿真
  • 泛化：训过的场景 OK，没见过的场景失败率高
  • 安全：物理世界出错代价大（撞坏物品 / 伤人）

设计 AI 反作弊系统（检测 LLM 生成内容）20 min
PLATFORM

检测信号

  1. 统计特征：burstiness / perplexity / token distribution 异常（GPTZero 思路）
  2. 水印检测：如果模型加了 logit watermark（OpenAI 探索）
  3. 风格指纹：用户历史 vs 当前样本风格对比
  4. 多模型 ensemble：用多个 detector 投票
  5. 主动诱导：让用户在 sandbox 重写一段，看跟原文一致性

残酷的真相

人类辅助 LLM 修改后基本不可检测。能识别 70% 已经很好，近 100% 准确不现实。
系统设计要假设"假阳性"和"假阴性"都常见 → 永远不要单独决策（reject 一篇 essay），必须有人工审核兜底。

12
设计 AI 产品 A/B 测试平台20 min
PLATFORM

AI A/B 跟传统区别

  • 样本需求大（accuracy diff 小 → 长时间）
  • multi-arm bandit 更优（持续学习用户偏好）
  • quality / latency / cost 三角，难单维评估
  • long-tail effect（10% 长尾用户影响很大）
  • 反馈延迟（用户当下不投诉 ≠ 满意）

设计要点

  1. 实验注册：YAML 定义 variant + 流量分配 + metric
  2. 分流：consistent hash（同用户始终同 variant）
  3. 多维度 metric：quality + cost + P95 latency + retention
  4. 自动 statistical significance：p-value / Bayesian / Sequential test

设计大规模分布式训练平台（千卡级）25 min
PLATFORM

关键组件

  1. 资源调度：K8s + Volcano（gang scheduling，确保 N 卡同时分配）
  2. 训练框架：Megatron-LM / FSDP（封装 3D parallel）
  3. 数据 pipeline：高吞吐数据读取（每秒 GB 级）+ 分布式 dataloader
  4. Checkpoint 系统：异步存 + 多副本（任何机器挂能恢复）
  5. Monitoring：dcgmi（GPU 健康）+ NCCL stats + loss curve
  6. Healthcheck + Failover：检测 ECC / 通信 hang 自动重启
  7. Networking：RDMA / IB 高速互联（节点间 200-400 Gb/s）

关键挑战（99% 时间花在 debug）

  • 硬件故障：1000 卡每天 1-2 张挂正常，必须自动 failover
  • NCCL hang：默认无 timeout，必须设
  • 慢卡：1 张 ECC 错误慢慢腐蚀，让所有其它卡等
  • 数据 stale：训了几天发现数据集有问题，损失百万美元
  • Loss NaN：BF16 数值溢出，必须 gradient clipping

成熟训练平台必备

  • 实验追踪：wandb-like dashboard
  • Hyper-param sweep：grid / Bayesian
  • Cost 报表：按 team / project / experiment 拆分
  • 共享 codebase：所有 team 用同一套训练 framework
