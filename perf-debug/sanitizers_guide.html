1. 引言 — Sanitizer 是 C++ 工程师必备

C++ 三大类 bug：

  • 内存错误：越界 / use-after-free / double-free / 泄漏 → ASan/LSan
  • 未定义行为：有符号溢出 / 空指针解引用 / 类型穿越 → UBSan
  • 并发错误：data race / lock order / 内存序错误 → TSan

这些 bug 在测试期常常不复现（"测试通过 = 没 bug" 是错觉），上线后随机崩溃。Sanitizer 是把它们暴露在测试期的标准工具。

2. Sanitizer 全景与选择

2.1 五大 Sanitizer 速查

名称检测目标开销 (CPU/内存)支持
AddressSanitizer (ASan) 越界 / UAF / double-free / heap/stack/global overflow ~ 2× CPU, 3-4× 内存 gcc 4.8+ / clang 3.1+ / MSVC 16.9+
UndefinedBehaviorSanitizer (UBSan) 有符号溢出 / 空指针 / 越界 / 错误 cast / 严格别名 / 未对齐 / VPTR 错误 ~ 1.2× CPU gcc 4.9+ / clang 3.3+
ThreadSanitizer (TSan) 数据竞争 / 死锁 (部分) / 错误的 mutex 用法 ~ 5-15× CPU, 5-10× 内存 gcc 4.8+ / clang 3.2+
MemorySanitizer (MSan) 未初始化内存读取 ~ 3× CPU, 2-3× 内存 仅 clang 3.3+ (gcc 不支持)
LeakSanitizer (LSan) 内存泄漏 (退出时报告) ~ 0 (内置在 ASan) gcc 4.9+ / clang 3.4+

2.2 何时用哪个

开发期

  • 本地跑测试: ASan + UBSan (一起开)
  • 多线程模块: TSan 单独跑
  • 新项目敏感场景: MSan

CI

  • 每次 PR: ASan + UBSan 必跑
  • 每次 PR: TSan 跑并发测试
  • nightly: 完整测试集 + 各 Sanitizer 单独
  • 发版: LSan 检查无新泄漏

Fuzz Testing

  • libFuzzer + ASan + UBSan + LSan
  • OSS-Fuzz 模式

不能用 Sanitizer 的场景

  • 嵌入式/资源极少
  • Production (开销太大)
  • 确定性要求强的环境 (Sanitizer 改变内存布局)

2.3 通用启用方式

# 编译 + 链接都需要 -fsanitize=...
g++ -O1 -g -fno-omit-frame-pointer -fsanitize=address,undefined \
    myapp.cpp -o myapp_asan

# 直接运行 — 错误时输出报告
./myapp_asan

关键编译选项：

  • -fsanitize=... — 启用 (可逗号分隔多个)
  • -O1 — 不开 O0 (太慢) 也不开 O3 (优化掩盖错误)
  • -g — 含调试符号让报告显示行号
  • -fno-omit-frame-pointer — 准确栈回溯
  • -fno-optimize-sibling-calls — 防 tail-call 让栈不准

3. ASan · AddressSanitizer 实战

3.1 能检测什么

错误类型例子
heap-buffer-overflow越界访问 new 的内存
stack-buffer-overflow越界访问栈上数组
global-buffer-overflow越界访问全局数组
heap-use-after-free访问已释放内存
stack-use-after-return函数返回后访问栈对象
stack-use-after-scope作用域结束后访问局部对象
double-free重复 delete
invalid-freedelete 非 new 的指针
memory-leak(LSan 集成) 退出时未释放

3.2 完整示例 — 一个有 bug 的程序

// bug.cpp
#include <cstdio>
int main() {
    int* arr = new int[10];
    arr[10] = 42;       // ❌ heap-buffer-overflow (off-by-one)
    delete[] arr;
    printf("%d\n", arr[0]);    // ❌ heap-use-after-free
    return 0;
}
$ g++ -O1 -g -fsanitize=address bug.cpp -o bug
$ ./bug

==12345==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x602000000038
WRITE of size 4 at 0x602000000038 thread T0
    #0 0x... in main bug.cpp:5
    #1 0x... in __libc_start_main

0x602000000038 is located 0 bytes to the right of 40-byte region [0x602000000010,0x602000000038)
allocated by thread T0 here:
    #0 0x... in operator new[]
    #1 0x... in main bug.cpp:4

SUMMARY: AddressSanitizer: heap-buffer-overflow bug.cpp:5 in main

报告告诉你：

  • 错误类型 (heap-buffer-overflow)
  • 访问地址 + 越界方向 (右越界 0 字节)
  • 访问栈 (代码 + 行号)
  • 原始内存分配处 (line 4 的 new)

3.3 常用错误的 ASan 报告样例

// use-after-free
int* p = new int(42);
delete p;
*p = 1;
// → AddressSanitizer: heap-use-after-free

// stack-use-after-return
int* bad() {
    int local = 42;
    return &local;       // 返回栈地址
}
*bad() = 1;            // UAR
// 注: 默认 ASan 不抓 UAR (开销大), 需 ASAN_OPTIONS=detect_stack_use_after_return=1

// double-free
int* p = new int(1);
delete p;
delete p;
// → AddressSanitizer: attempting double-free

// global overflow
int arr[10];
int main() { arr[10] = 1; }
// → AddressSanitizer: global-buffer-overflow

3.4 ASan 运行时配置

# 通过环境变量配置
ASAN_OPTIONS="detect_leaks=1:abort_on_error=1:halt_on_error=1:print_stats=1:check_initialization_order=1" ./prog

常用选项：

  • detect_leaks=1 — 启用 LSan (默认 Linux 开)
  • halt_on_error=1 — 第一个错误就退出 (默认)
  • abort_on_error=1 — 错误时 abort() 出 core (调试用)
  • detect_stack_use_after_return=1 — 打开 UAR (慢)
  • strict_string_checks=1 — 严格 string 函数检查
  • verbosity=1 — 详细日志
  • log_path=/tmp/asan — 日志写文件
  • suppressions=asan.supp — 抑制规则文件
实践：开发期默认所有选项. CI 用最严格 (halt_on_error=1, detect_leaks=1)。

4. ASan 原理速览 ⭐

本节是面试速答级原理摘要。完整的 shadow encoding 表、fake stack、quarantine 算法、10 条插桩汇编、详细局限根因表见 sanitizer_internals.html §2 ASan 实现深入。

4.1 三件套

  1. 编译期插桩：编译器在每条 load/store 前插 ~7-10 条 shadow 检查指令
  2. Shadow memory：1 字节 shadow 编码 8 字节真实内存的可访问性（shadow = (real >> 3) + offset）
  3. Redzone + Quarantine：每次 alloc 前后填 128B 红区 shadow poison；free 不立即归还，放进 256MB FIFO 隔离队列 → UAF 在 quarantine 期能 catch

4.2 性能与基本局限（实战视角）

  • CPU ~2×  |  内存 ~3-4×  |  不上 production（用 HWASan / GWP-ASan 替代，见 §14b）
  • 跨红区距离的越界（如 arr[i+1000]）漏检 —— 落入合法 alloc 的 shadow
  • 同一 struct 内字段间越界（intra-object overflow）漏检 —— redzone 只在 alloc 边界
  • UAF 超过 quarantine 容量后漏检
  • 不查未初始化读（→ MSan）/ 不查 race（→ TSan）
面试速答："ASan 怎么发现 UAF?" — ① free 不立即释放、放进 quarantine FIFO；② 整块 shadow 标 0xFD（freed）；③ 后续访问触发 shadow check 立即报告 + 给出 alloc/free/use 三处栈。

完整局限根因表（10 条带原因 + 缓解方法）见 internals §2 ASan 的局限性与根因。

5. UBSan · UndefinedBehaviorSanitizer

5.1 能检测什么

C++ 标准把"未定义行为" 列了 200+ 条。UBSan 能检测其中常见的 ~30 种：

检查项检测的 UB
signed-integer-overflow有符号整数溢出
shift移位超过位宽 / 移位负数
integer-divide-by-zero整数除 0
float-divide-by-zero浮点除 0 (默认关闭)
null空指针解引用
nonnull-attribute违反 __attribute__((nonnull))
vptr错误的多态对象 cast
bounds数组越界 (固定大小)
alignment未对齐访问
object-size访问超出对象大小
return非 void 函数无 return
unreachable到达 unreachable 路径
vla-boundVLA 大小为 0 或负
function函数指针类型错误
enum无效的 enum 值
bool无效的 bool 值 (非 0/1)

5.2 用法

# 全部检查
g++ -O1 -g -fsanitize=undefined myapp.cpp

# 只开几项
g++ -O1 -g -fsanitize=signed-integer-overflow,null myapp.cpp

# 排除某项
g++ -O1 -g -fsanitize=undefined -fno-sanitize=alignment myapp.cpp

5.3 经典报告

// signed overflow
int x = INT_MAX;
int y = x + 1;     // UB!

// 报告:
runtime error: signed integer overflow:
    2147483647 + 1 cannot be represented in type 'int'
    SUMMARY: UndefinedBehaviorSanitizer: signed-integer-overflow at line N
// null deref
int* p = nullptr;
*p;
// runtime error: load of null pointer of type 'int'

// shift
int x = 1 << 32;
// runtime error: shift exponent 32 is too large for 32-bit type 'int'

// vptr (错误的 dynamic_cast 风险)
struct Base { virtual ~Base(){}; };
struct Derived : Base {};
Base* b = new Base;
Derived* d = static_cast<Derived*>(b);
d->virtual_method();
// runtime error: member call on address ... which does not point to an object of type 'Derived'

5.4 UBSan 默认是"trap"

默认行为：触发 UB 时打印消息后继续执行。可改为：

# 触发即 abort (调试用)
g++ -fsanitize=undefined -fno-sanitize-recover=all myapp.cpp

# 或环境变量
UBSAN_OPTIONS="halt_on_error=1:abort_on_error=1" ./prog

5.5 性能与用法

  • 开销 ~ 1.2× (轻量)
  • 建议默认开 (开发 + CI)
  • 常与 ASan 一起用：-fsanitize=address,undefined
  • 有的项目甚至在 Production 开关键检查 (fsanitize=signed-integer-overflow + trap-on-error)
实战经验：UBSan 是性价比最高的 sanitizer — 开销小、能发现的 bug 严重 (UB 是最难调试的 bug 类型)。默认开。

6. TSan · ThreadSanitizer 数据竞争检测

6.1 能检测什么

  • Data race：两个线程同时访问同一内存，至少一个写，且无 happens-before 关系
  • Lock 顺序异常：可能死锁的 lock 顺序
  • Mutex 误用：unlocked unlock / 在持锁线程之外 unlock
  • 已销毁的同步原语使用

6.2 经典 data race 例子

int g = 0;

void thread1() { g = 1; }       // 写
void thread2() { if (g) {} }   // 读

int main() {
    std::thread t1{thread1}, t2{thread2};
    t1.join(); t2.join();
}
$ g++ -O1 -g -fsanitize=thread race.cpp -o race
$ ./race

WARNING: ThreadSanitizer: data race (pid=12345)
  Write of size 4 at 0x... by thread T1:
    #0 thread1() race.cpp:3
  Previous read of size 4 at 0x... by thread T2:
    #0 thread2() race.cpp:4

  Location is global 'g' of size 4 at 0x...

  Thread T1 (tid=12346) created by main thread at:
    #0 std::thread::thread<...> thread:N

SUMMARY: ThreadSanitizer: data race race.cpp:3 in thread1()

6.3 TSan 抓得到的几类问题

// ① 朴素 race
int g = 0;
void a() { g++; }
void b() { g--; }   // race!

// ② 漏锁
std::mutex mu;
int g;
void a() { std::lock_guard lk{mu}; g = 1; }
void b() { g = 2; }   // race! (没加锁)

// ③ memory order 太弱
std::atomic<int> flag{0};
int data = 0;
void producer() { data = 42; flag.store(1, std::memory_order_relaxed); }
void consumer() { while (!flag.load(std::memory_order_relaxed)); use(data); }
// race on data — relaxed 不建立 happens-before

6.4 TSan 能识别的同步原语

  • std::mutex / shared_mutex / recursive_mutex / atomic
  • std::condition_variable
  • std::thread::join / detach
  • future::get / promise::set_value
  • pthread_mutex_t / pthread_rwlock_t / pthread_cond_t
  • POSIX semaphore

6.5 TSan 不能识别的

  • 自定义 spinlock (用普通 atomic 实现) — 需要 __tsan_acquire / __tsan_release 注解
  • 跨进程同步 (共享内存 + futex)
  • 用户态自旋等待 atomic
  • fence 间接建立的 happens-before (relaxed atomic)

6.6 用法

# 编译
g++ -O1 -g -fsanitize=thread myapp.cpp -o myapp_tsan

# 运行时配置
TSAN_OPTIONS="halt_on_error=1:second_deadlock_stack=1" ./myapp_tsan

常用 TSAN_OPTIONS：

  • halt_on_error=1 — 第一个 race 就退出
  • second_deadlock_stack=1 — 死锁报告显示两条调用栈
  • history_size=7 — 内存访问历史长度 (1-7)
  • flush_memory_ms=N — 周期性清理内存

6.7 性能与限制

  • 开销 ~ 5-15× CPU, 5-10× 内存 (很贵)
  • 不能与 ASan 同时用 (二选一)
  • TSan + 长时间测试可能 OOM (内存不断增长)
  • 误报罕见，但"漏报"常见 — race 不一定每次跑都触发
TSan 不保证能发现所有 race — 它只能报告本次执行实际发生的 race。要更彻底，需要多次运行 + 不同负载 / 调度顺序。

7. TSan 原理速览 ⭐

本节是面试速答级摘要。FastTrack 算法（read epoch / write VC 的优化）、shadow word 编码、为何 5-10×、10 条局限根因 见 sanitizer_internals.html §3 TSan FastTrack 算法。

7.1 核心：vector clock + happens-before

每个线程维护一份 vector clock（N 元 timestamp 数组）。同步操作（mutex unlock/lock、atomic release/acquire、cv notify/wait、thread join）合并双方 VC，建立 happens-before 边。

每个内存位置的 shadow 记录"上次访问者 + 当时的 VC"。新访问到来时：

  1. 上次访问者的 VC ≤ 当前 VC → 有 hb，安全
  2. 否则 → race，报警 + dump 两侧栈

7.2 FastTrack 的关键优化

朴素实现每次访问存/比完整 N 元 VC → 100× slowdown。FastTrack 的洞察：多读不构成 race，read 只需 1 个 epoch（tid + clock），write 才存完整 VC → 降到 5-10×。

7.3 性能开销来源（实战视角）

  • 每次内存访问都要更新 + 比对 shadow → 5-15× CPU
  • VC 随线程数线性增长 → 内存 5-10×
  • 访问历史窗口默认 4，可调（长 → 漏报少但内存爆）
面试速答："TSan 怎么知道两个访问之间有没有 happens-before?" — 每线程一份 vector clock；同步原语合并 VC；访问内存时把 shadow 中上次访问的 VC 跟当前比，不可比较或更小则有 hb，更大则是 race。

8. MSan · MemorySanitizer 未初始化内存读

8.1 检测什么

读取未初始化的内存。这是 C++ 隐藏 bug 的大类，比越界更阴险（很多时候不崩，但产生错误结果）。

struct S { int x, y; };

void f() {
    S s;
    s.x = 1;            // y 未初始化
    if (s.y > 0) {}     // ❌ 读未初始化值 → MSan 报警
}

8.2 用法

# 仅 clang 支持
clang++ -O1 -g -fsanitize=memory -fno-omit-frame-pointer myapp.cpp -o myapp_msan
./myapp_msan

8.3 报告

WARNING: MemorySanitizer: use-of-uninitialized-value
    #0 0x... in f() bug.cpp:6
    #1 0x... in main bug.cpp:N

  Uninitialized value was created by an allocation of 'a' in the stack frame
    #0 0x... in main bug.cpp:N

SUMMARY: MemorySanitizer: use-of-uninitialized-value bug.cpp:6 in f()

8.4 原理速览

Bit-level shadow（每 bit 1 shadow bit）+ 延迟报告：读未初始化值不立刻报，而是污染派生值，直到流入分支 / syscall / 输出才报警 → 能定位到 alloc 源头。Origin tracking 进一步保留传播链。

编码原理、延迟报告 trade-off、10 条局限根因（整链路 build / inline asm / JIT / padding 假阳…）见 internals §4 MSan 实现深入。

8.5 MSan 难用的原因（实战视角）

  1. 只支持 clang（gcc 不支持）
  2. 整个程序所有依赖都要用 MSan 重编，包括 libc / libstdc++ / 第三方库 — 否则误报漫天
  3. 实践上：要用 -stdlib=libc++ -L/path/to/msan/libs 链接 MSan-instrumented 标准库
  4. 大型项目集成成本高 —— Google monorepo 能做，普通团队代价高

替代方案：用 Valgrind Memcheck（开销大但不需要重编依赖）+ 编译警告 -Wuninitialized -Wmaybe-uninitialized（编译期能抓部分）。

8.6 何时用 MSan

  • 新项目 / 全 clang / 控制全部依赖 — 强烈推荐
  • 关键数据处理代码 (会传递错误值) — 单独构建 MSan 版本
  • Fuzz testing 配合 — OSS-Fuzz 提供 MSan 标准库
  • 常规项目：用编译警告 + Valgrind 替代

9. LSan · LeakSanitizer 内存泄漏

9.1 工作方式

LSan 集成在 ASan 中（默认开），也可独立使用：

# 集成在 ASan (推荐)
g++ -O1 -g -fsanitize=address myapp.cpp -o myapp
./myapp     # 退出时自动检查泄漏

# 仅 LSan (独立, 开销最小)
g++ -O1 -g -fsanitize=leak myapp.cpp -o myapp
./myapp

9.2 报告

$ ./bug
==12345==ERROR: LeakSanitizer: detected memory leaks

Direct leak of 40 byte(s) in 1 object(s) allocated from:
    #0 0x... in operator new[]
    #1 0x... in main bug.cpp:5

Indirect leak of 80 byte(s) in 2 object(s) allocated from:
    #0 0x... in operator new
    #1 0x... in some_helper bug.cpp:8

SUMMARY: AddressSanitizer: 120 byte(s) leaked in 3 allocation(s).

9.3 检测原理

LSan 在程序退出时执行类似 GC 的可达性分析：

  1. STW (停所有线程)
  2. 从根 (栈 + 全局变量 + 寄存器 + thread_local) 出发
  3. 遍历堆，标记所有可达对象
  4. 未被标记的 = 泄漏

分两类：

  • Direct leak：根本无人指向（典型泄漏）
  • Indirect leak：被 direct leak 的对象指向（连锁泄漏）

9.4 抑制误报

有些"泄漏" 是预期的（程序退出时不释放是性能优化）。用 suppression 文件：

# lsan.supp
leak:third_party_cache_init
leak:libfoo.so
LSAN_OPTIONS="suppressions=$PWD/lsan.supp:print_suppressions=1" ./prog

13. Fuzz Testing + Sanitizer = 黄金组合

13.1 Fuzz Testing 是什么

用大量随机/变异数据喂给程序 → 找 crash / 异常。
配 Sanitizer → 不仅找崩溃，还找潜在问题 (越界 / UAF / UB 等)。

13.2 libFuzzer 入门

// fuzz_target.cpp
#include <cstdint>
#include <cstddef>

extern "C" int LLVMFuzzerTestOneInput(const uint8_t* data, size_t size) {
    // 你的 parser/解码器
    parse_protocol_message(data, size);
    return 0;
}

// 编译
clang++ -O1 -g -fsanitize=fuzzer,address,undefined fuzz_target.cpp -o fuzz_target
./fuzz_target corpus/      # corpus 目录存种子样本

libFuzzer 自动：

  • 变异输入 (bit-flip / byte-shuffle / dict / etc.)
  • 记录覆盖率反馈 (insn coverage / edge coverage)
  • 保留增加覆盖率的样本
  • 遇到 crash → 保存触发输入 + 报告

13.3 OSS-Fuzz

Google 提供的免费持续 fuzz 平台。已知用户：Linux kernel / OpenSSL / SQLite / curl / FFmpeg / V8 / ...

它们用 OSS-Fuzz 持续找 bug，每年发现成千上万的 CVE。

13.4 实战：协议解析器 fuzz

// 给 RPC XML 解析器写个 fuzz target
extern "C" int LLVMFuzzerTestOneInput(const uint8_t* data, size_t size) {
    try {
        std::string_view xml{reinterpret_cast<const char*>(data), size};
        NetconfMessage msg = parse_xml(xml);
        // 触发更深路径
        if (msg.has_filter()) apply_filter(msg);
    } catch (const std::exception&) {
        // 异常是预期的, 不算 fuzz 失败
    }
    return 0;
}
# 编译 + 跑 1 小时
clang++ -O1 -g -fsanitize=fuzzer,address,undefined parse_fuzz.cpp -o parse_fuzz
./parse_fuzz corpus/ -max_total_time=3600

# 找到 crash → 保存到 crash-* 文件
./parse_fuzz crash-abc...        # 重现
任何接受外部输入的解析器 / 反序列化器 / 协议处理都应该写 fuzz target. 24h 跑 fuzz 找出的 bug 数量, 远超人工测试.

13b. Fuzz 进阶 — AFL++ / Honggfuzz / OSS-Fuzz / 结构化 fuzz ⭐

13b.1 主流 fuzzer 横向对比

13b.2 AFL++ 用法（替代/补充 libFuzzer）

# 1. 用 AFL++ 编译器插桩
CC=afl-clang-fast CXX=afl-clang-fast++ make

# 2. 准备 corpus 目录（种子样本，最好 > 100 个）
mkdir corpus
cp testdata/*.xml corpus/

# 3. 启动 fuzz（默认单进程）
afl-fuzz -i corpus -o findings -- ./myapp @@   # @@ 替换为输入文件路径

# 4. 并行 fuzz — 主 + 多 secondary
afl-fuzz -M main -i corpus -o findings -- ./myapp @@ &
afl-fuzz -S worker1 -i corpus -o findings -- ./myapp @@ &
afl-fuzz -S worker2 -i corpus -o findings -- ./myapp @@ &

# 5. 用 power schedule (AFL++ 特性)
afl-fuzz -p explore -i corpus -o findings -- ./myapp @@

# 6. 看 crash
ls findings/main/crashes/
AFL++ vs libFuzzer 选型：
  • 有源码 + 想集成到 CI / OSS-Fuzz → libFuzzer
  • 纯黑盒二进制 / 闭源 → AFL++ + QEMU mode（动态插桩）
  • 需要多种变异策略 / 大型并行 fleet → AFL++
  • 想跨 GitHub Action 持续跑公益开源 → OSS-Fuzz

13b.3 Structure-Aware Fuzzing — 让 fuzz 跨过 magic header / checksum

普通 fuzz 对随机字节输入有效，但对有严格结构的格式（JPEG / protobuf / SQL）大部分变异立刻被拒绝，浪费时间。Structure-aware fuzz 让变异基于"格式 grammar"做。

// libFuzzer + protobuf-mutator —— 直接 fuzz protobuf 消息
#include <src/libfuzzer/libfuzzer_macro.h>
#include "my_proto.pb.h"

DEFINE_PROTO_FUZZER(const MyMessage& msg) {
    // 输入直接是已经合法的 protobuf — 不浪费 cycles 在 magic header
    handle_message(msg);
}

// 编译
clang++ -fsanitize=fuzzer,address my_fuzz.cc my_proto.pb.cc \
   -lprotobuf-mutator-libfuzzer -lprotobuf-mutator -lprotobuf

常见 grammar fuzzer

  • libprotobuf-mutator — 任意 protobuf 类型
  • FuzzBench domains — JSON / SQL / XML / JS
  • Domato — 浏览器 DOM API fuzz
  • Peach / boofuzz — 网络协议 fuzz

13b.4 OSS-Fuzz 接入（让 Google 免费帮你跑）

OSS-Fuzz 给开源项目提供全天候多机 fuzzing。已为 700+ 开源项目找了 1 万+ bug（curl / OpenSSL / SQLite / Envoy / Boost / ICU / FFmpeg ...）

接入步骤（10 行 yaml）

13b.5 Differential Fuzzing — 找正确性 bug（不只是 crash）

普通 fuzz 只发现 crash。Differential fuzz 同时跑两个不同实现同样输入，比较输出 — 找语义差异。

extern "C" int LLVMFuzzerTestOneInput(const uint8_t* data, size_t size) {
    Result a = my_parser::parse(data, size);
    Result b = ref_parser::parse(data, size);
    // 同样输入两个实现应得同样结果
    if (a != b) __builtin_trap();    // 触发 ASan 报告
    return 0;
}

典型用：自家 SIMD 实现 vs 朴素 fallback、新算法 vs 老算法、跨平台浮点。

13b.6 Fuzz 落地经验

  • 种子 corpus 是关键：直接从真实流量 / 单元测试样本起步，比从空开始快 100×
  • 用 -only_ascii=1 / -len_control 等约束参数缩小搜索空间
  • 用 dictionary（-dict=keywords.txt）让 fuzz 知道关键字（HTTP 方法、SQL 关键字、JSON 标记）
  • 同时开多 Sanitizer（ASan + UBSan 默认双开；MSan 独立跑）
  • 覆盖率回归：跑完用 -merge=1 把 corpus 去重，纳入版本控制
  • fuzz CI：每 PR 跑 10 分钟基础 fuzz；nightly 跑 8h 深度 fuzz

14. REAL5 个真实工程案例

14.1 案例 1: ASan 找出 yang 解析器越界

症状：QA 报告偶发 crash，gdb 看 core 是 invalid memory，但 backtrace 不准确。

排查：开 ASan 重跑，第一秒就报 heap-buffer-overflow，定位到 parse_xpath 的 off-by-one。

修复：边界检查改 i <= len → i < len。

教训：早开 ASan 节省 weeks。

14.2 案例 2: TSan 抓出 logger 数据竞争

症状：日志偶尔有错乱（一行被另一行截断）。

排查：TSan 报告 logger 内部 buffer 在两个线程间被并发写入。

修复：把 logger 改成线程安全（per-thread buffer + 异步 flush）。

14.3 案例 3: UBSan 找出整型溢出

症状：高负载下偶尔统计数据为负。

排查：UBSan 报告 signed-integer-overflow，定位到 int total = a * b，a/b 都是 30k 级别。

修复：用 int64_t。

14.4 案例 4: LSan 找到 third-party SDK 泄漏

症状：服务跑 1 周内存涨 100MB。

排查：LSan 报告每次连接后第三方 SDK 的 SDKContext 泄漏。

定位：SDK 文档说要 sdk_destroy_context()。我们的代码没调。

修复：用 RAII 包装 SDK 上下文。

14.5 案例 5: Fuzz + ASan 找出 0day

背景：在 RPC XML 解析器上跑 fuzz + ASan，24 小时。

结果：找到 3 个 crash：
① 嵌套 100+ 层标签触发栈溢出；
② 特定 UTF-8 字节序列导致 use-after-free；
③ 超长 attribute 触发 buffer overflow。

修复：① 限制递归深度；② 修正 string_view 生命周期；③ 加输入长度上限。

价值：3 个潜在 CVE，修复成本远小于线上事故。

所有案例的共同模式：测试期发现 = 几小时；线上事故发现 = 几天 + 巨大代价。Sanitizer 的投入产出比极高。

14b. HWASan + GWP-ASan — 让 Sanitizer 真的能上生产 ⭐

普通 ASan 因为2-3× 内存 + 2× CPU 开销无法上生产。Google/Android/Chrome 团队设计了两个"开销可承受"的变种，让 sanitizer 终于能在 prod 抓内存 bug。

14b.1 全家桶位置定位

Sanitizer开销覆盖率架构定位

HWASan~10% CPU, ~15% mem~95%ARMv8.5+ (Top-Byte Ignore)Android / 嵌入式 prod 候选

14b.2 HWASan — 硬件辅助的轻量 ASan

HWASan（Hardware-assisted AddressSanitizer）利用 ARM 的 Top-Byte Ignore (TBI) ——64 位指针高 8 位被 CPU 忽略可放 tag；或 ARMv8.5+ 的 MTE（硬件比对 tag）。每次 alloc 给指针 + 该 16B 内存区域打同样 4-bit tag，访问时比对失败即报错。

原理对比（数据以 internals §6 为准）

Sanitizer每次访问做什么shadow / tag 比例误报率
ASan查 shadow（每 8 字节 → 1 字节）1/8 (~12.5%)≈ 0（红区精确）
HWASan (TBI/软件)软件比对 ptr tag 与 shadow tag（每 16 字节 → 1 字节 4-bit tag）1/16 (~6%)~6.25%（1/16 tag 撞）
HWASan (MTE/硬件)CPU 自动比对 tag，零软件开销0（硬件存储）~6.25%
# 启用 HWASan（需 ARMv8.5+ 硬件）
clang++ -fsanitize=hwaddress -fno-omit-frame-pointer -O1 -g myapp.cpp -o myapp.hw
./myapp.hw
# 输出形如:
# ==12345==ERROR: HWAddressSanitizer: tag-mismatch on address 0x...
#   READ of size 4 at 0x...

# Android 全平台 HWASan build (Pixel 设备)
m TARGET_PRODUCT=aosp_arm64_hwasan
Android 早已默认开 HWASan 跑 system services 抓内存 bug。Pixel 5+ 设备上的 HWASan-enabled build 是开发期标准镜像。Mali / Adreno GPU 驱动用 HWASan 找到大量历史 bug。

14b.3 GWP-ASan — 采样型生产 ASan

GWP（"Guard With Probability"）-ASan 的设计哲学：不可能全开，但可以采样。每 N 次（典型 N = 1000-10000）malloc 中挑一次走"特殊路径"：

  • 把这次分配单独放到带 guard page 的页面
  • UAF / 越界访问立刻触发 SEGV，dump 完整堆栈 + 分配栈 + 释放栈
  • 其余 99.9% 走普通 allocator，性能影响接近零

使用方式（Chrome / Android 生产实现）

// Chrome 早把 GWP-ASan 集成进 PartitionAlloc：
// 用户态启用
if (base::FeatureList::IsEnabled(features::kPartitionAllocGwpAsanMalloc)) {
    EnableGwpAsanSampling(/*sample_rate=*/ 5000);
}

// 命中 GWP-ASan 触发的 UAF 在 crash report 里大概长这样:
// GWP-ASan: Use After Free
//   Allocated at: malloc -> Vector::reserve -> ...
//   Freed     at: free   -> Vector::~Vector -> ...
//   Accessed  at: read   -> Worker::run     -> ...

独立 GWP-ASan 库

# Scudo allocator (LLVM, Android 默认) 内置 GWP-ASan
clang++ -fsanitize=scudo -fsanitize-recover=scudo myapp.cpp -o myapp
SCUDO_OPTIONS="GWP_ASAN_Enabled=true:GWP_ASAN_SampleRate=5000" ./myapp
Tech Lead 价值：传统的"线上 UAF 调查"是瞎子摸象——core dump 看到的是错误访问点，分配/释放路径完全丢失。GWP-ASan 一旦命中，三个完整堆栈（alloc / free / use）都在 crash report 里，10 分钟定位代替 10 小时大海捞针。

14b.5 真实部署案例

  • Chrome：GWP-ASan + PartitionAlloc + MTE（ARM 内存标签）— 浏览器每月还是抓出 Sandbox 内 UAF
  • Android：System Services 默认 HWASan build；用户态 Scudo + GWP-ASan
  • Google Cloud / Search：内部所有 C++ 服务默认 GWP-ASan，crash 自动 dedup + 上报 + 追溯到具体 commit
  • Linux Kernel：syzkaller + kASan/kHWASan 持续 fuzz，自 2018 起找到 5000+ 内核 bug

HWASan / GWP-ASan 各自局限（1/16 漏检概率、x86 无 TBI、PAC 抢位、采样率 vs catch time 等）见 internals §6 HWASan 局限、§7 GWP-ASan 局限。

结论：Sanitizer 不止是"开发工具"。HWASan + GWP-ASan 改变了"线上内存 bug 不可见"的现实，让 C++ 的"内存安全劣势"在工程实践中收敛到接近 Rust 的可调试水平。但前提是你的团队真的把它部署上线——绝大多数公司只用 ASan 在 CI，是巨大浪费。

15. 局限与陷阱（工程视角）

本节聚焦工程实战中容易踩的坑。每个 Sanitizer 的逐条根因表（约 50 条限制 + 根本原因 + 缓解）见 internals §2-§7 各自局限性与根因，跨家族通用结构性限制见 internals §9。

15.1 一句话局限速记

  • ASan：红区/quarantine 有容量，远距离越界 & 老 UAF 漏
  • TSan：只能发现本次执行真发生的 race —— 多次跑 + 不同负载
  • MSan：要求所有依赖都 instrumented，否则误报
  • UBSan：只检测约 30 种 UB（C++ 标准列了 200+ 种）
  • LSan：进程退出时检测，常驻服务要主动 __lsan_do_recoverable_leak_check()
  • HWASan：1/16 概率漏（4-bit tag），x86 无 TBI 几乎不可用
  • GWP-ASan：单实例几乎不 catch，依赖 fleet + 后台聚合

15.2 行为差异

Sanitizer 改变内存布局 → 程序行为可能不同：

  • 地址变化 → ASLR / 缓存行为变
  • 分配开销大 → 时序敏感的 race 可能不复现
  • 内存增长 → OOM 概率增大
  • 开 ASan 不复现的 bug，可能是 timing / 内存布局相关 (race 例子)

15.3 误报来源

  • 低层 hacks：自实现 allocator / 共享内存 / mmap 上的对象
  • 第三方库：未用 sanitizer 重编
  • JIT / 动态生成代码：CPU 直接执行，绕过 instrumentation
  • signal handler：不是所有 sanitizer 函数 async-signal-safe

15.4 Production 上的 Sanitizer?

大多数公司不在 Production 用 ASan/TSan (开销太大)。例外：

  • Chromium：金丝雀版本开 ASan
  • Android：HWASan (硬件辅助 ASan，开销 2× 内存而非 4×)
  • SafeStack：production 友好，仅 1.1× 开销
  • fsanitize=undefined,trap：触发 UB 直接 trap，零开销，可上 production

15.5 大型项目的实践

  1. 所有依赖统一构建 (vcpkg/conan + sanitizer 选项)
  2. CI 矩阵全跑 (6 个组合至少)
  3. 抑制规则版本化
  4. 关键模块单独 fuzz target + 24/7 持续 fuzz
  5. 线上保留 LSan 的低开销检查

16.4 速答示范（第 4 题）

问：TSan 工作原理是什么? 怎么知道两次访问之间没有 happens-before?

答：

  1. 每个内存位置有 shadow memory，记录"上次访问者 + 时刻 (vector clock)"
  2. 每个线程维护自己的 vector clock，记录"本线程当前时刻 + 已知其他线程的时刻"
  3. 同步操作 (mutex unlock/lock, atomic release/acquire) → 合并双方 vector clock
  4. 每次内存访问时：检查 shadow 里"上次访问"的 vector clock 是否 ≤ 当前线程 vector clock
  5. 如果 ≤ → 上次访问 happens-before 本次 → 安全
  6. 否则 → race

限制：① 只能发现实际发生的 race，不发生的看不到；② 自实现 spinlock 等不被 TSan 识别 → 漏报；③ 性能开销 5-15×。

17.1 Sanitizer 一表速查

Sanitizer检测开销编译选项

17.2 推荐编译模板

# 通用
g++ -O1 -g -fno-omit-frame-pointer -fsanitize=address,undefined ...

# 并发测试
g++ -O1 -g -fno-omit-frame-pointer -fsanitize=thread,undefined ...

# Fuzz
clang++ -O1 -g -fsanitize=fuzzer,address,undefined ...

17.3 常用环境变量

ASAN_OPTIONS="halt_on_error=1:abort_on_error=1:detect_leaks=1:print_stats=1"
UBSAN_OPTIONS="halt_on_error=1:abort_on_error=1:print_stacktrace=1"
TSAN_OPTIONS="halt_on_error=1:abort_on_error=1:second_deadlock_stack=1"
LSAN_OPTIONS="suppressions=$PWD/lsan.supp:print_suppressions=1"

17.4 工程化 Checklist

  • ☐ CMake 选项 ENABLE_ASAN / ENABLE_TSAN
  • ☐ CI 矩阵: debug / release / asan / tsan
  • ☐ 抑制规则文件 (lsan.supp / tsan.supp 等)
  • ☐ 测试覆盖率 ≥ 70% (sanitizer 才有用武之地)
  • ☐ 关键模块写 fuzz target
  • ☐ Production 用 -fsanitize=undefined -ftrap-on-error (低开销)
  • ☐ 团队规范: 所有抑制规则配 issue + 截止日期

17.5 Quick Decision Tree

遇到崩溃 / 内存问题:           → ASan + LSan
遇到偶发并发问题:              → TSan
遇到数值/逻辑诡异:              → UBSan
新项目 / 全 clang:              → 加 MSan
要测试外部输入鲁棒性:           → libFuzzer + ASan
不能改源码 / 不能重编:          → Valgrind
