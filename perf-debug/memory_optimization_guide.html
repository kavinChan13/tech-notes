01

内存层级与延迟

所有内存优化的第一性原理：越近越快越小越贵

理解延迟数量级差，才知道优化往哪使劲

1.1 金字塔与延迟数字

无论 C++ 还是 GPU，内存都是一座金字塔：顶层快、小、贵（寄存器 / SRAM），底层慢、大、便宜（DRAM / HBM / 磁盘）。优化的本质就是 —— 让热数据尽量待在金字塔上层。

图 1.1 · 内存金字塔。C++ 的优化目标是把热数据留在 Cache；GPU 的 FlashAttention 同理 —— 把数据从 HBM 搬到 SRAM 后反复复用。

下面这张动画条形图把延迟数量级"画"出来 —— 注意条长是对数感知的近似，重点是体会 malloc、page fault 这些"抖动源"和 cache 命中之间的鸿沟（向下滚动触发动画）：

L1 cache 命中
~1 ns
L3 cache 命中
~10 ns
DRAM 访问
~100 ns
malloc 小块
50–200 ns 抖
page fault (minor)
~5 µs 抖
page fault (major)
~ms 极抖
优化排序原则：先消除"抖动大"的操作（malloc / page fault / 锁竞争），再优化"稳定但慢"的（cache miss）。均值低不代表 P99 低。

1.2 优化的两条主线

主线 A · 减少分配 / 搬运

  • 对象池 / arena / 预分配 → 免运行时 malloc
  • 移动语义 / SSO → 免拷贝
  • 零拷贝 / mmap → 免内核态搬运
  • KV-Cache / FlashAttention → 免重复计算/搬运

主线 B · 提升局部性 / 命中率

  • cache 友好布局（SoA / 对齐 / 紧凑）
  • 避免 false sharing
  • 大页减 TLB miss · NUMA 本地访问
  • 量化 / 紧凑数据结构 → 装得下、读得快

① C++ LAYER 进程内存优化

对象布局 · false sharing · 分配器 · 碎片

从单个对象的字节，到整个堆的形态

2.1 对象布局与对齐

编译器为满足对齐要求会在成员之间插入 padding（空洞）。同样的字段，声明顺序不同，对象大小能差 30%+。把大字段排前面、小字段聚一起，就能挤掉空洞。

图 2.1 · 字段重排消除 padding：同样 4 个字段，从 24B 降到 16B。用 pahole 可直接看出空洞位置。

见 cpp/object_model.html（布局/EBO/pahole）· cpp/memory_model.html

2.2 False Sharing（伪共享）

两个线程改的是不同变量，但它们恰好落在同一条 64B cache line 上 —— 每次写都让对方的缓存行失效，在两个核之间来回弹（ping-pong），性能暴跌。下图红色高亮就是被反复争抢的那条 cache line：

图 2.2 · False sharing 动画：两核反复让对方缓存行失效（红块闪烁 = 失效）。perf c2c 可定位。

见 cpp/memory_model.html §13 · performance_guide.html · low_latency_guide.html

2.3 分配器与内存池

默认 std::allocator（底层 malloc）有全局锁、延迟不确定、会触发 brk/mmap 突变。热路径上应换成专用分配策略：

策略分配开销释放适用
std::allocator (malloc)50–200 ns + 抖同左通用
jemalloc / tcmalloc / mimalloc更稳更稳多线程服务
monotonic / arena几 ns（指针推进）0（整体释放）请求级短生命周期
对象池 / slab~10 ns（free list）~10 ns固定大小对象
栈 / SBO inline00小对象
arena→一个请求开始时分配一大块，期间所有小对象在块内"指针推进"，请求结束整块一次性丢弃 —— 零碎释放、零碎片。

见 low_latency_guide.html §3–8（pmr / arena / 对象池 / 零分配）· 协程帧分配器

2.4 内存碎片

反复 malloc/free 不同大小后，空闲内存被切成许多无法利用的小洞：明明总空闲够，却分不出一块连续的大内存。这就是为什么 free 之后 RSS 不降 —— 碎片把堆"锁住"了。

图 2.4 · 外部碎片：空闲总量足够但不连续。这是"RSS 不下降"最常见的非泄漏成因。

见 memory_leak_guide.html（假泄漏诊断矩阵 / glibc malloc 内部）

2.5 SSO / SBO 小对象优化

std::string、std::function 这些类型，如果数据足够小，会直接内嵌进对象自己的栈空间（SSO / SBO），完全不碰堆 —— 短字符串场景下省掉一次 malloc + 一次 free + 提升 cache 局部性。只有超过阈值才退化为堆分配。

图 2.5 · SSO/SBO：小对象内嵌、零堆分配；超阈值才落堆。

见 object_model.html（SBO/launder）· stl_containers · modern_libs_guide.html

2.6 智能指针控制块

shared_ptr 的成本藏在控制块里。make_shared 把控制块和对象合并成一次分配（更快、cache 更友好）；但代价是只要还有 weak_ptr 存活，整块内存（含已析构对象的空间）都不归还。

图 2.6 · 控制块布局与 make_shared 权衡。

见 smart_pointers_deep_dive.html（控制块/aliasing/intrusive_ptr）

② SYSTEM LAYER 系统级内存优化

虚拟内存 · NUMA · 大页 · cgroup / OOM

从进程的"假"地址，到物理 DRAM 的真实代价

3.1 虚拟内存与页表

每个进程看到的都是连续的虚拟地址，但物理内存是零散的页。MMU 通过多级页表把虚拟页翻译成物理页，TLB 缓存这个翻译。第一次访问某页才真正分配物理页（延迟分配 / 缺页）。

图 3.1 · 虚拟→物理翻译。优化点：大页减少页表层级和 TLB miss；mlock 锁页防换出；预触（pre-touch）避免运行时缺页尖刺。

见 system/os_fundamentals.html · system/numa_memory_advanced.html

3.2 NUMA

多路 CPU 服务器上，内存被分给不同 socket（NUMA node）。CPU 访问本地内存快，访问另一个 node 的远程内存要走 QPI/UPI，慢 30–50%。无意识跨 node 访问是隐形性能杀手。

图 3.2 · NUMA：让线程和它的数据待在同一 node。手段：first-touch 分配、numactl --membind、CPU pinning、IRQ 亲和。

3.3 大页 / TLB

4KB 普通页

  • 1GB 数据 = 262144 个页表项
  • TLB 装不下 → 频繁 miss → 走页表
  • 随机访问大数据集时延迟抖

2MB / 1GB 大页

  • 同样 1GB 只需 512 / 1 个项
  • TLB 命中率飙升 → 减少翻译开销
  • 数据库 / vLLM / 低延迟服务常开
THP（透明大页）有 page collapse 尖刺，DB/Redis 常建议设 madvise 或 never，改用显式 HugePages。

3.4 cgroup / OOM

容器里最反直觉的坑：page cache 也算进 memory.current，于是"看起来没满却被 OOM kill"。理解 cgroup v2 的内存计数是排障关键。

OOM→查 dmesg 看 oom_score → 区分 anon（真用量）vs cached（可回收）→ 调 memory.high 限流而非直接 kill

见 numa_memory_advanced.html（NUMA/大页/cgroup 最全）· incident_playbook.html（OOM 排障剧本）

3.5 零拷贝 / mmap

传统"读文件再发网络"要在内核缓冲区 ↔ 用户缓冲区之间来回拷贝 4 次、切换上下文多次。零拷贝（sendfile / splice / mmap）让数据不经过用户态，直接在内核里从 Page Cache 走到网卡 —— 大文件传输 / 高吞吐服务的关键。

图 3.5 · 零拷贝。其他手段：mmap 把文件映射进地址空间免 read 拷贝；io_uring 注册 buffer + SEND_ZC；MSG_ZEROCOPY。

见 io_linux_kernel_guide.html（零拷贝/mmap/io_uring 最全）· networking_guide.html

4.1 训练显存账本

4.2 KV-Cache

推理时为避免重复计算，每生成一个 token 都把它的 Key/Value 缓存下来。问题是：KV-Cache 随序列长度线性增长，长上下文 + 高并发下它才是显存第一杀手（而非权重）。

图 4.2 · KV-Cache 随序列增长（动画）。压缩手段：MHA→GQA/MQA→MLA 缩减 KV 头数、KV 量化 INT8/INT4。

4.3 PagedAttention（vLLM）

传统推理给每个请求静态预留最大长度的 KV 空间 → 60–70% 浪费。PagedAttention 借鉴 OS 虚拟内存：把 KV 切成固定大小的 block，按需分配、用 block table 索引 —— 显存利用率从 ~30% 提到 ~96%。

图 4.3 · PagedAttention = KV-Cache 的"虚拟内存"。配合 Continuous Batching 让吞吐倍增。

见 ai-infra/inference_serving.html · transformer_llm_architecture.html

4.5 FlashAttention 分块

标准 attention 要在 HBM（大而慢）里实例化一个 n×n 的注意力矩阵 —— 序列一长就爆显存、且反复读写 HBM。FlashAttention 把 Q/K/V 切成小块（tiling）搬进 SRAM（小而快），用 online softmax 边算边累积，从不完整存下 n×n 矩阵 —— 显存从 O(n²) 降到 O(n)，还更快。

图 4.5 · FlashAttention 分块 + online softmax：不实例化 n×n，显存 O(n²)→O(n)。

见 transformer_llm_architecture.html · 李宏毅 · FlashAttention

4.6 ZeRO 三阶段分片

数据并行（DDP）每张卡都存一份完整的权重+梯度+优化器状态 → 极度浪费。ZeRO 的核心是：这些状态本来就重复，把它们切片分到各卡，用时再 AllGather 临时聚合。逐阶段切得更狠，单卡显存逐级下降。

图 4.6 · ZeRO 三阶段：逐步把优化器状态→梯度→权重切片分到各卡。ZeRO-3 ≈ PyTorch FSDP。

见 distributed_training.html（ZeRO/FSDP/TP/PP 全景）

面试题集（分层）

三层主线的高频题 · 标准答案 + 加分点 + 陷阱

点开每题查看要点，覆盖 C++ / 系统 / AI

基础

基础栈分配和堆分配有什么区别？为什么栈快？
栈
移动栈指针即完成分配/释放，生命周期随作用域，几乎零开销，且天然 cache 友好；缺点是容量小（典型 1–8MB）、大小需编译期可知。
堆
走 malloc，需维护空闲链表、可能加锁、可能触发 brk/mmap，延迟不确定且会碎片化。
延伸
能用栈/SBO 就别用堆；热路径用对象池/arena 把堆分配"摊销"掉。
基础什么是内存对齐？为什么调整字段顺序能省内存？
对齐
每种类型要求其地址是自身大小的整数倍，编译器为此插入 padding（空洞）。
省内存
把大字段排前、小字段聚一起可减少空洞；同样字段不同顺序大小能差 30%+。用 pahole 看空洞。
基础程序 RSS 一直涨，就一定是内存泄漏吗？
不一定
常见非泄漏成因：内存碎片、容器 capacity 只增不减、allocator/thread arena 缓存、page cache 增长、预热填充。
区分
先 ASan/LSan 排真泄漏；再用 malloc_trim / 切 allocator / 静止观察 判断是否碎片或缓存。看 USS 比看 RSS 更准。
基础虚拟内存和物理内存的关系？Page Fault 是什么？
关系
进程看到连续虚拟地址，MMU 经页表翻译到零散物理页，TLB 缓存翻译。malloc 通常只分配虚拟地址，首次写才真正给物理页（延迟分配）。
Page Fault
minor：首次访问已映射页（~µs）；major：需从磁盘/swap 换入（~ms，极抖）。低延迟系统用 mlock/预触消除。

中级

中级什么是 false sharing？如何定位和解决？
现象
多个线程改各自的变量，但它们落在同一 64B cache line 上，写操作互相让对方缓存行失效，在核间 ping-pong，多核反而不扩展。
定位
perf c2c 看 cache line 争抢热点。
解决
alignas(64) 或 std::hardware_destructive_interference_size 把热变量隔到独立 cache line；或 per-thread 局部聚合。
中级monotonic/arena、对象池、std::pmr 分别适合什么场景？
arena
一次请求内大量短命小对象：指针推进分配、请求结束整块释放，零碎片零零碎释放。
对象池/slab
固定大小对象反复创建销毁：free list 复用，~10ns。
pmr
C++17 多态分配器，让同一容器类型在运行时切换内存来源（如挂 monotonic_buffer_resource 用栈 buffer 做零堆分配）。
中级NUMA 是什么？没意识到 NUMA 会有什么后果？怎么优化？
NUMA
多 socket 机器上内存分属不同 node，本地访问快，跨 node 远程访问慢 30–50%。
后果
线程在 node0、数据在 node1（如内存由其他线程 first-touch），所有访问走 UPI，吞吐/延迟劣化且抖动。
优化
first-touch 让用数据的线程分配它；numactl --membind/--cpunodebind、CPU pinning、IRQ 亲和、NVMe 与 node 对齐。
中级容器明明没满，为什么进程被 OOM kill 了？
关键
cgroup v2 的 memory.current 把 page cache 也算进去；应用 anon 内存没满，但 anon + cached 触达 memory.max 就会 OOM。
排查
dmesg 看 oom_score；区分 anon（真用量）/ cached（可回收）；用 memory.high 限流回收而非硬 kill。
中级大页（HugePage）为什么能提速？THP 又为什么常被建议关掉？
提速
2MB/1GB 页让同样数据需要的页表项大减 → TLB 命中率高 → 翻译开销小，随机访问大数据集尤其受益。
THP 坑
透明大页的 page collapse 会带来不可预测的延迟尖刺，DB/Redis 常设 madvise 或 never，改用显式 HugePages。

高级（体现深度）

高级make_shared 一定比 shared_ptr(new T) 好吗？什么时候不要用？
好处
控制块+对象单次分配，少一次 malloc、cache 更友好，还异常安全。
陷阱
对象与控制块同块内存；strong 归零对象析构后，只要还有 weak_ptr，整块（含大对象空间）都不释放。
结论
大对象 + 长生命周期 weak_ptr 观察者场景，改回 shared_ptr(new T) 分离分配。
高级LLM 推理时显存被什么占满？为什么 PagedAttention 能提升利用率？
占用
长上下文 + 高并发下，KV-Cache（随序列线性增长）才是第一杀手，常超过权重本身。
老做法
给每个请求按最大长度静态预留 KV → 60–70% 浪费。
PagedAttention
借鉴 OS 虚拟内存：KV 切成固定 block、按需分配、block table 索引，还能 prefix 共享 → 利用率 ~30% 提到 ~96%，配合 continuous batching 吞吐倍增。
高级训练 70B 模型单卡放不下，显存都花在哪？有哪些手段省？
账本
权重 + 梯度 + 优化器状态（Adam 的 m/v，FP32）+ 前向激活值 ≈ 7–10× 权重，激活值常占大头。
手段
ZeRO-1/2/3 或 FSDP 分片；Activation Checkpointing（省激活 ~50%、算力 +30%）；BF16 混合精度；TP/PP/EP 并行；CPU/NVMe offload。
权衡
分片和重算都是"用通信/算力换显存"，要结合带宽和 batch 调。
高级FlashAttention 为什么既省显存又更快？它的核心 trick 是什么？
省显存
不在 HBM 实例化整个 n×n 注意力矩阵，显存 O(n²)→O(n)。
更快
attention 是 memory-bound，瓶颈在 HBM 读写。FlashAttention 把 Q/K/V 分块（tiling）搬进 SRAM、用 online softmax 边算边累积，大幅减少 HBM 往返 —— 本质和 C++ 的"提升 cache 局部性"同源。
高级量化能省显存，代价是什么？哪些场景要谨慎？
收益
FP16→INT8/INT4，显存 ÷2~÷4，且 decode 是 memory-bound 故还提速。
代价
可能掉点，reasoning 任务、长上下文 KV 累积误差尤其敏感；calibration 域偏移、kernel 未真正启用都是坑。
方法
权重：GPTQ/AWQ/SmoothQuant；KV：INT8/INT4 量化；训练微调：QLoRA（INT4 base + FP16 LoRA + Paged Optimizer）。

速答示范（false sharing）

问：两个原子计数器在多线程下越加越慢，为什么？怎么改？
答：大概率是 false sharing —— 两个计数器虽是不同变量，但相邻、落在同一条 64B cache line 上。每个核写自己的计数器都会让另一个核缓存的整条 line 失效，触发 MESI 一致性流量，在核间反复 ping-pong。用 perf c2c 能看到对应 cache line 的 HITM 热点。修复：给每个计数器 alignas(64)（或用 hardware_destructive_interference_size）让它们独占 cache line；更好的是 per-thread 局部累加、最后再汇总，连原子操作都省了。
