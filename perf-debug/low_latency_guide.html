低延时场景对 C++ 的特殊要求

"普通 C++ 工程"和"低延时 C++"是两套思维。普通工程关心 p99 / QPS，低延时关心 worst case latency 和确定性。

普通工程

  • p99 < 100 ms 可接受
  • 偶尔 GC / 长尾不致命
  • 资源弹性扩展
  • 动态内存随便用
  • 单元测试覆盖足矣

低延时 / 后端 / HFT

  • p99.99 < 100 µs / p99 < 50 µs
  • 无 long tail (任何抖动都是 bug)
  • 资源静态预分配
  • 避免一切动态分配
  • 需要 worst-case 测试 + 压力延迟测量

1.1 你的领域 — RPC / RAN

RPC 网络管理协议本身延迟要求一般（毫秒级），但有实时分支：

  • RU (无线单元) 控制平面：管理消息可以慢，但不能阻塞
  • 定时器精度：心跳 / keepalive 不能漂移
  • 故障 (alarm) 上报：µs ~ ms 级别
  • config 下发：批量但要可控
  • 性能管理 (PM) 数据采集：高吞吐 + 实时

即便不是 HFT，"零分配 + 缓存友好 + 避免锁竞争" 这些原则对你都受益。

2.1 为什么要做"预算"

低延时系统的开发第一步是把 SLA 拆成预算：每个组件分配多少 µs。然后每个组件在预算内做事。

SLA: end-to-end p99.99 < 100 µs

  网络收 RX:           10 µs
  Reactor 分发:         5 µs
  解析 (parse):        15 µs
  业务逻辑:            30 µs
  序列化:              15 µs
  网络发 TX:           10 µs
  内核栈/调度抖动:    15 µs
  ━━━━━━━━━━━━━━━━━━━━━━━
  总:                 100 µs ✓

2.2 关键数字 (低延时口径)

在 cpp_performance_guide.html 的 Jeff Dean 表基础上，低延时还要熟记这些：

低延时编程的核心是：消除"抖动大"的操作，即便它们的均值很低。

2.3 worst-case vs average

误区："平均延迟 5 µs 已经很好"。
现实：p99.99 才是用户看到的。如果 99.99% 是 5 µs，但 0.01% 是 5 ms — 在 1M QPS 下每秒 100 次抖动, 用户感知是偶发卡顿。

低延时设计要把"抖动源"逐个消除：

  • 动态内存分配 → 用 pool / arena
  • 锁竞争 → 用 lock-free / per-thread
  • 系统调用 → 批处理 / io_uring / DPDK kernel-bypass
  • page fault → mlock 锁内存
  • 线程调度 → CPU pinning + RT 优先级
  • CPU 频率波动 → governor=performance + 关 turbo
  • CPU 中断 → IRQ affinity / isolcpus
  • NUMA 跨节点 → 数据/线程绑同节点

3.1 默认 std::allocator 的问题

  • 底层用 ::operator new → glibc malloc / jemalloc / tcmalloc
  • 有全局锁 (or per-arena 锁) → 多线程争抢
  • 分配延迟不确定 (取决于堆状态)
  • 触发 brk/mmap 时延迟突变
  • 典型耗时 50-200 ns，但 worst case 可达 µs

3.2 分配器分类

类型分配复杂度释放复杂度适用
std::allocator (malloc)~ 50-200 ns + 抖~ 50-200 ns + 抖通用
jemalloc / tcmalloc同上但更稳同上多线程服务器
monotonic_buffer_resource~ 几 ns (指针推进)0 (整体释放)请求处理 / 短生命周期

stack / inline 分配00SBO

3.3 分配器选型决策

问: 这个对象的生命周期?
├─ 与请求处理同步 → arena / monotonic
├─ 短期 + 大量同尺寸 → pool / slab
├─ 长期 + 不可预测 → tcmalloc / jemalloc
└─ 超短期 + 小 → SBO / 栈分配

问: 频率?
├─ 极高频 (每 RTT/每 µs) → 必须自定义 (零分配)
├─ 中频 (每秒千次) → tcmalloc 足够
└─ 低频 → std::allocator 都行

问: 多线程?
├─ 多线程访问同一池 → per-thread arena 或 lock-free
├─ 单线程 → 简单 monotonic 即可
└─ 跨线程传递 → 注意所有权

std::pmr

核心组件 · 基础用法 · 请求处理模式 · 默认 resource 切换 · std::pmr 的限制

多态 allocator (C++17)

std::pmr 让同一个容器类型能用不同分配器，运行时切换 — 这是 C++17 自定义 allocator 的标准方式。

4.1 核心组件

类用途
memory_resource抽象基类: allocate / deallocate / is_equal
polymorphic_allocator<T>包装 memory_resource* 的 allocator
monotonic_buffer_resource顺序推进, 整体释放
unsynchronized_pool_resource对象池, 单线程
synchronized_pool_resource对象池, 多线程
new_delete_resource()默认 new/delete 后端
null_memory_resource()分配总失败 (调试)

4.2 基础用法

#include <memory_resource>

// 用栈上 buffer 作为分配池
char buf[8192];
std::pmr::monotonic_buffer_resource pool{buf, sizeof(buf)};

std::pmr::vector<int> v{&pool};
for (int i = 0; i < 100; ++i) v.push_back(i);
// vector 在 buf 上扩展, 0 次 malloc

std::pmr::string s{"hello world", &pool};
std::pmr::map<int, std::pmr::string> m{&pool};
m.emplace(1, std::pmr::string{"a", &pool});

// 池满时 fallback 到 default (new_delete_resource)

4.3 请求处理模式

// 每个请求一个 arena, 处理完整体释放 — 极快 + 无碎片
void handle_request(Request req) {
    char buf[16 * 1024];
    std::pmr::monotonic_buffer_resource arena{buf, sizeof(buf)};
    std::pmr::polymorphic_allocator<> alloc{&arena};

    std::pmr::vector<Field> fields{alloc};
    std::pmr::map<std::pmr::string, std::pmr::string> index{alloc};

    parse(req, fields, index);
    Response resp = build_response(fields, index);
    send(resp);
    // arena 析构时 — 0 次 free 调用, 整体释放
}
实测：典型 RPC 处理：默认 allocator ~ 5 µs (含 malloc 抖动)；arena allocator ~ 1.5 µs (稳)。延迟3× 提升 + 无抖动。

4.4 默认 resource 切换

// 让所有 std::pmr::xxx 默认用我的 resource
std::pmr::set_default_resource(&my_pool);

std::pmr::vector<int> v;       // 自动用 my_pool

4.5 std::pmr 的限制

  • 每个 pmr 容器要存一个 memory_resource* → sizeof 多 8 字节
  • 分配/释放走虚函数 → 1-2 ns 间接开销
  • 不能跨 resource 移动 (除非重新分配)
  • std::pmr::xxx 与 std::xxx 是不同类型，无法互转
规则：① 请求级别用 monotonic_buffer_resource；② 进程级别长寿对象用 synchronized_pool_resource；③ 不需要切换时退回 std::allocator。

5.1 简化实现

class ArenaAllocator {
    char* current_;
    char* end_;
    char* base_;
public:
    ArenaAllocator(char* buf, size_t n) : current_(buf), end_(buf + n), base_(buf) {}

    void* allocate(size_t n, size_t align = alignof(std::max_align_t)) {
        // 对齐 current_ 到 align 边界
        uintptr_t addr = reinterpret_cast<uintptr_t>(current_);
        uintptr_t aligned = (addr + align - 1) & ~(align - 1);
        char* result = reinterpret_cast<char*>(aligned);
        if (result + n > end_) throw std::bad_alloc{};
        current_ = result + n;
        return result;
    }

    void deallocate(void*, size_t) {}    // no-op

    void reset() { current_ = base_; }       // 整体释放
};

5.2 多块 chunk 扩展

固定大小 buf 容易爆。生产用多块 chunk：

class ChunkedArena {
    std::vector<std::unique_ptr<char[]>> chunks_;
    char* current_;
    char* end_;
    size_t chunk_size_ = 64 * 1024;

    void grow() {
        auto chunk = std::make_unique<char[]>(chunk_size_);
        current_ = chunk.get();
        end_ = current_ + chunk_size_;
        chunks_.push_back(std::move(chunk));
        chunk_size_ *= 2;            // 几何增长 — vector 风格
    }
public:
    void* allocate(size_t n, size_t align) {
        // ... 对齐逻辑同上 ...
        if (result + n > end_) {
            if (n > chunk_size_) chunk_size_ = n * 2;
            grow();
            // 重新计算
        }
        // ...
    }

    void reset() {
        chunks_.clear();
        current_ = end_ = nullptr;
    }
};

5.3 集成到 std::pmr

class MyArenaResource : public std::pmr::memory_resource {
    ChunkedArena arena_;
protected:
    void* do_allocate(size_t bytes, size_t align) override {
        return arena_.allocate(bytes, align);

    void do_deallocate(void*, size_t, size_t) override {}
    bool do_is_equal(const std::pmr::memory_resource& o) const noexcept override {
        return this == &o;

固定大小对象池 · RAII handle 包装 · slab allocator

6.1 固定大小对象池

同尺寸对象的高频分配/释放 — pool 是最优解。

template<class T>
class FixedPool {
    union Slot {
        T value;
        Slot* next;
    };
    Slot* free_list_ = nullptr;
    std::vector<std::unique_ptr<Slot[]>> chunks_;
    size_t chunk_size_ = 256;

    void grow() {
        auto chunk = std::make_unique<Slot[]>(chunk_size_);
        for (size_t i = 0; i < chunk_size_ - 1; ++i)
            chunk[i].next = &chunk[i + 1];
        chunk[chunk_size_ - 1].next = free_list_;
        free_list_ = chunk.get();
        chunks_.push_back(std::move(chunk));
        chunk_size_ *= 2;
    }

public:
    template<class... Args>
    T* create(Args&&... args) {
        if (!free_list_) grow();
        Slot* s = free_list_;
        free_list_ = s->next;
        return new (&s->value) T(std::forward<Args>(args)...);     // placement new
    }

    void destroy(T* obj) {
        obj->~T();
        Slot* s = reinterpret_cast<Slot*>(obj);
        s->next = free_list_;
        free_list_ = s;
    }
};

// 使用
FixedPool<Frame> frame_pool;
Frame* f = frame_pool.create(seq, payload);
// ...
frame_pool.destroy(f);
// 分配/释放都 ~ 10 ns

6.2 RAII handle 包装

template<class T>
class PoolHandle {
    FixedPool<T>* pool_;
    T* obj_;
public:
    PoolHandle(FixedPool<T>* p, T* o) : pool_(p), obj_(o) {}
    ~PoolHandle() { if (obj_) pool_->destroy(obj_); }
    T* operator->() { return obj_; }
    // 禁拷贝, 允许 move
};

6.3 slab allocator (Linux 内核风格)

slab = 多个 size class 的 pool 集合。常见做法：2^n 桶（8/16/32/64/128/256/...）。

class SlabAllocator {
    static constexpr size_t kSizeClasses[] = {8, 16, 32, 64, 128, 256, 512, 1024, 2048, 4096};
    std::array<FixedPool<char[]>, std::size(kSizeClasses)> pools_;

    size_t size_class_for(size_t n) {
        for (size_t i = 0; i < std::size(kSizeClasses); ++i)
            if (n <= kSizeClasses[i]) return i;
        return SIZE_MAX;       // 大块走 mmap/malloc
    }
public:
    void* allocate(size_t n) {
        size_t klass = size_class_for(n);
        if (klass == SIZE_MAX) return std::malloc(n);
        return pools_[klass].acquire_raw();
    }
};

// jemalloc / tcmalloc / mimalloc 内部就是更复杂的 slab 实现

6.4 与 §9 (cpp_concurrency_patterns_guide) 对象池区别

那里的 ObjectPool 主要为对象生命周期管理 (RAII 自动归还)；这里的 FixedPool/SlabAllocator 主要为分配性能。两者可结合：FixedPool 提供分配，handle 提供 RAII。

7.2 实现 small_vector

template<class T, size_t N>
class SmallVector {
    union {
        alignas(T) char sbo_[N * sizeof(T)];   // SBO 缓冲
        T* heap_;                                  // 大于 N 时堆指针
    };
    size_t size_ = 0;
    size_t cap_ = N;

    bool on_heap() const { return cap_ > N; }
    T* data() {
        return on_heap() ? heap_
                          : reinterpret_cast<T*>(sbo_);
    }

public:
    void push_back(const T& v) {
        if (size_ == cap_) grow();
        new (data() + size_++) T(v);
    }
    // ... 析构 / 移动 / 拷贝 ...
};

// 用法
SmallVector<Field, 8> fields;     // 多数情况 ≤ 8 字段, 0 次 malloc
fields.push_back(...);

7.3 SBO 收益

RPC 解析典型字段数 4-8。用 SmallVector<Field, 8> 替代 std::vector：
每次 RPC 节省 1-2 次 malloc ≈ 100-400 ns。1M QPS 下节省 100-400 ms 累计 CPU。

7.4 SBO 的代价

  • 对象 sizeof 变大（含 SBO buffer + 状态位）
  • 移动构造可能不能简单 swap 指针（要拷贝 SBO 内容）
  • 容器扩容到 heap 后 SBO 部分浪费
  • 对齐复杂（union 需要正确处理 trivial vs non-trivial）
选择：① 标准库 (string / function / any) 自带 SBO，受益自动；② 自定义容器优先用 boost::small_vector / folly::small_vector，不要从零写；③ 阈值 N 选项要基于实测数据分布 (绝大多数 case ≤ N)。

零分配编程 (Zero-allocation)

HFT / 实时系统的圣杯：业务热路径上一次 malloc 都没有 → 延迟稳定 + 无 GC 抖动。

8.1 怎么做

  1. 所有 buffer 启动时预分配
  2. 用 arena / pool 替代 new
  3. STL 容器要么 reserve 要么 pmr
  4. 禁用 std::string 中的扩展 — 用 string_view
  5. 避免 std::function (有时会堆分配) — 用模板 / 函数指针
  6. 异常处理: throw 会分配 → 关键路径用错误码 / std::expected
  7. 避免 std::shared_ptr构造 (control block 堆分配) — 用 unique_ptr 或 intrusive_ptr
  8. 日志:避免同步 fmt::format（会 string 拼接）— 用结构化日志 + 异步

8.4 经典模式

// 启动时静态预分配 — 跑起来后所有结构都已存在
class MessageRouter {
    static constexpr size_t kMaxRoutes = 4096;
    std::array<Route, kMaxRoutes> routes_;
    std::atomic<size_t> count_{0};
public:
    // 注册阶段 (启动期)
    bool register_route(Route r) {
        size_t i = count_.fetch_add(1);
        if (i >= kMaxRoutes) return false;
        routes_[i] = std::move(r);
        return true;
    }
    // 热路径 (无分配)
    Handler* find(MessageType t) const {
        for (size_t i = 0; i < count_.load(); ++i)
            if (routes_[i].type == t) return routes_[i].handler;
        return nullptr;
    }
};
"怎么验证你的代码真的零分配?" — 答：① LD_PRELOAD 一个 hook malloc 的 .so, 关键路径触发 abort；② strace -e brk,mmap 看系统调用；③ 用 jemalloc 的 stats 看 alloc 计数；④ valgrind --tool=massif 看 heap 增长。

Lock-free 数据结构

SPSC ring buffer (单生产单消费) · MPMC 复杂度 · lock-free 的代价

SPSC ring buffer · MPMC 复杂度 · lock-free 的代价

9.1 SPSC ring buffer (单生产单消费)

最简单 + 最快的 lock-free 队列。

template<class T, size_t N>
class SPSCQueue {
    static_assert((N & (N-1)) == 0, "N must be power of 2");
    alignas(64) std::atomic<size_t> head_{0};      // 写者
    alignas(64) std::atomic<size_t> tail_{0};      // 读者
    // ↑ cache line 对齐, 防假共享
    std::array<T, N> buf_;

public:
    bool push(T v) {                    // 仅 producer 调用
        size_t h = head_.load(std::memory_order_relaxed);
        size_t next = (h + 1) & (N - 1);
        if (next == tail_.load(std::memory_order_acquire))
            return false;       // 满
        buf_[h] = std::move(v);
        head_.store(next, std::memory_order_release);
        return true;
    }

    bool pop(T& out) {                 // 仅 consumer 调用
        size_t t = tail_.load(std::memory_order_relaxed);
        if (t == head_.load(std::memory_order_acquire))
            return false;       // 空
        out = std::move(buf_[t]);
        tail_.store((t + 1) & (N - 1), std::memory_order_release);
        return true;
    }
};

关键点：

  • head/tail 各自 cache line 对齐 — 避免假共享
  • N 必须是 2 的幂 — 用 & (N-1) 取模
  • release-acquire pair 保证 happens-before
  • 无锁 — push/pop 都是 ~ 10 ns
SPSC ring buffer push/pop ~ 10 ns vs 加锁队列 ~ 50-100 ns。

9.2 MPMC 复杂度

多生产多消费的 lock-free 队列难写，需要：

  • CAS 循环
  • 解决 ABA 问题 (双 word CAS / hazard pointer / epoch-based reclamation)
  • 正确的 memory_order
  • 验证: TLA+ / 形式化证明

现实：不要自己写。用现成库：

  • moodycamel::ConcurrentQueue — header-only, 高性能
  • folly::MPMCQueue — Facebook
  • boost::lockfree::queue
  • concurrentqueue::BlockingConcurrentQueue

9.3 lock-free 的代价

  • 代码极难调试
  • 性能不一定比锁好 (高竞争时 CAS 重试)
  • memory_order 错误难发现 (TSan 不一定抓得到)
  • 可读性差
"lock-free 比 lock 一定快" 是误区。中低竞争下，std::mutex 加无竞争路径几乎免费 (~ 25 ns)。仅当锁真成为瓶颈（perf 显示锁等待 > 30%）才考虑 lock-free。

(基础见 cpp_performance_guide.html § 5-7。这里讲低延时更激进的优化。)

10.1 极致 cache 优化清单

  • 所有热点结构 cache line 对齐 (alignas(64))
  • 多线程 atomic / 计数器: 每个 cache line 一个
  • 热数据放结构体前部 (访问频率排序)
  • 冷数据 (调试 / metadata) 单独结构 / 单独 alloc
  • 避免分支 (cmov 替代 if) — 减少 ICache miss
  • 函数 inline (减少 ICache miss)
  • 小函数避免被优化器拒绝 inline (用 always_inline)

10.2 prefetch 主动预取

// 提前告诉 CPU 下一次要读的地址 → 预热 cache
for (size_t i = 0; i < v.size(); ++i) {
    __builtin_prefetch(&v[i + 8], 0, 1);    // 提前 8 步
    process(v[i]);
}
// gcc/clang: __builtin_prefetch(addr, rw=0/1, locality=0-3)
// MSVC: _mm_prefetch

用法：

  • 顺序遍历的 cache miss 不是问题（hardware prefetcher 自动处理）
  • 跳跃式访问（hash map / linked list）prefetch 收益大
  • 过度 prefetch → 反向污染 cache

10.3 hot/cold 分离

// ❌ 冷热混在一起 — 加载 hot 字段时也加载冷数据 (浪费 cache)
struct Connection {
    int fd;                        // hot
    std::atomic<int64_t> bytes_in;  // hot
    std::atomic<int64_t> bytes_out; // hot
    std::string peer_address;     // cold (仅诊断)
    std::map<std::string, std::string> metadata;  // cold
    char debug_log[4096];          // cold
};

// ✅ 拆分
struct ConnectionHot {
    int fd;
    std::atomic<int64_t> bytes_in, bytes_out;
    ConnectionCold* cold;             // 指向冷数据
};
struct ConnectionCold {
    std::string peer;
    std::map<std::string, std::string> metadata;
    char debug_log[4096];
};

普通 page 4 KB；huge page 2 MB / 1 GB。优势：

  • TLB miss 大幅减少 (覆盖范围 ×500)
  • page table walk 开销减少
  • 典型收益 5-15% (内存密集场景)

CPU 亲和性 (pinning) 与 NUMA

11.1 为什么 pinning

  • 线程在 CPU 间迁移 → cache miss + TLB miss + 跨 NUMA
  • 调度器即便不迁移，也会让线程"插队等待" → 抖动
  • pin 到固定核 → 缓存稳定 + 可预测延迟

11.2 用 pthread_setaffinity_np

cpu_set_t cpuset;
CPU_ZERO(&cpuset);
CPU_SET(2, &cpuset);     // 绑到 CPU 2
pthread_setaffinity_np(pthread_self(), sizeof(cpuset), &cpuset);

// 用户态包装
void pin_to_cpu(int cpu) {
    cpu_set_t mask;
    CPU_ZERO(&mask);
    CPU_SET(cpu, &mask);
    pthread_setaffinity_np(pthread_self(), sizeof(mask), &mask);
}

// jthread 启动时
std::jthread t([]{
    pin_to_cpu(2);
    // ... worker loop ...
});

// 命令行 (运行时)
taskset -c 2 ./myapp
numactl --cpunodebind=0 --membind=0 ./myapp

11.3 isolcpus + nohz_full

更激进：在内核启动参数把某些核从普通调度器隔离，仅手动绑核的进程才能跑：

# /etc/default/grub
GRUB_CMDLINE_LINUX_DEFAULT="isolcpus=2,3 nohz_full=2,3 rcu_nocbs=2,3"

# isolcpus: 把 CPU 2,3 从调度器移除
# nohz_full: 关闭这些核的 timer tick (减少中断)
# rcu_nocbs: 把 RCU 回调从这些核移走

效果：CPU 2/3 几乎只跑你 pin 上去的线程，jitter 极小。

11.4 NUMA

多 socket 服务器：每个 CPU socket 有自己的本地内存 (NUMA node)。跨 node 访问内存延迟2-3× 本地。

# 看 NUMA 拓扑
numactl --hardware
lscpu

# 绑定线程 + 内存到同一 NUMA 节点
numactl --cpunodebind=0 --membind=0 ./myapp

# C++ 中显式分配
#include <numa.h>
void* p = numa_alloc_onnode(size, node);

# 检查当前线程的 NUMA
int node = numa_node_of_cpu(sched_getcpu());
规则：① 高频线程 pin 到固定核；② 数据分配在同一 NUMA 节点；③ 关键应用配 isolcpus + nohz_full；④ 不同 NUMA 节点的服务考虑独立部署。

11.5 中断绑核 (IRQ affinity)

网卡中断会随机绑到核 → 干扰你的业务。把中断推到管理核：

# 把网卡中断绑到 CPU 0,1 (留 2,3 给业务)

系统调用最小化

syscall 的代价 · 减少 syscall 的方法 · io_uring

12.1 syscall 的代价

syscall典型耗时
getpid (vDSO 优化)~ 50 ns
clock_gettime (vDSO)~ 20 ns
read/write 1 byte~ 500-1000 ns
epoll_wait~ 1 µs
futex (慢路径)~ 1-10 µs
fork~ ms 级

每次 syscall 都有 user/kernel 切换 → cache pollute + register save。

12.2 减少 syscall 的方法

  • 批处理：writev / readv / sendmmsg / recvmmsg 一次多个
  • io_uring：异步批量 IO，最大化 syscall 摊销
  • SO_BUSY_POLL：避免 epoll 切上下文，busy poll 网卡 ring
  • kernel-bypass：DPDK / XDP / SPDK 完全绕开内核
  • vDSO：clock_gettime / getpid / gettimeofday 已自动用
  • 用户态 timer：自己维护时间，少调 clock_gettime

12.3 io_uring (Linux 5.1+)

详见 CPP_IO_Linux_Kernel_Guide.html。要点：

  • 用共享内存环提交/完成 IO，不需要每次 syscall
  • 批量 SQE / CQE，单次 syscall 可处理几十个 IO
  • 轮询模式可以完全无 syscall处理 IO
  • RPC 高 QPS 场景理论最优

12.4 user-space timer

// 业务热路径不调 clock_gettime, 用 RDTSC 或一个全局 atomic time
// 一个专门线程每 1ms 更新这个时间
std::atomic<int64_t> g_now_us{0};

void time_keeper() {
    while (running) {
        g_now_us.store(clock_now_us(), std::memory_order_relaxed);
        std::this_thread::sleep_for(1ms);
    }
}

// 业务侧
int64_t now = g_now_us.load(std::memory_order_relaxed);     // ~ 1 ns

代价: 时间精度降到 ms 级，但热路径节省 syscall。

批处理 (batching)

网络 IO 批处理 · 队列批量 pop · 日志批量 flush

批处理是低延时 + 高吞吐共存的关键。每个操作的固定开销 (syscall / cache miss / 锁) 摊到 N 个事件上。

13.1 网络 IO 批处理

// 一次发多包
struct mmsghdr msgs[32];
// 准备 32 个消息...
sendmmsg(fd, msgs, 32, 0);

// 一次收多包
recvmmsg(fd, msgs, 32, 0, nullptr);

// 一次写多 buffer
struct iovec iov[8];
writev(fd, iov, 8);

13.2 队列批量 pop

template<class T>
size_t BoundedQueue<T>::pop_batch(T* out, size_t max) {
    std::unique_lock lk{mu_};
    cv_not_empty_.wait(lk, [this]{ return closed_ || !q_.empty(); });
    size_t n = std::min(max, q_.size());
    for (size_t i = 0; i < n; ++i) {
        out[i] = std::move(q_.front()); q_.pop();
    }
    cv_not_full_.notify_all();
    return n;
}

// 调用方
T batch[32];
size_t n = q.pop_batch(batch, 32);
for (size_t i = 0; i < n; ++i) process(batch[i]);
// 一次锁获取处理 32 个事件

13.3 日志批量 flush

class BatchedLogger {
    std::vector<LogEntry> pending_;       // 批缓冲
    std::ofstream f_;
public:
    void log(LogEntry e) {
        pending_.push_back(std::move(e));
        if (pending_.size() >= 256) flush();
    }
    void flush() {
        for (const auto& e : pending_) f_ << e.format() << '\n';
        f_.flush();          // 一次 syscall, 不是每条一次
        pending_.clear();
    }
};

13.4 trade-off

批处理增加吞吐但可能增加单事件延迟（要等凑够批）。设计上：

  • min-batch + max-wait：要么凑够 N 个，要么等 t µs
  • 低负载时小 batch (低延迟)
  • 高负载时大 batch (高吞吐)
  • 自适应调整

14.3 RT 调度优先级

struct sched_param p;
p.sched_priority = 50;       // 1-99
sched_setscheduler(0, SCHED_FIFO, &p);
// SCHED_FIFO: 实时, 不被普通进程抢占
// SCHED_RR: 实时 round-robin

# 命令行

RT 调度的危险：① 优先级太高的死循环会冻结整机；② RT 进程与普通进程抢 CPU 时绝对优先；③ 必须配 cpu_quota 限制以防失控；④ 测试环境要慎用。

14.4 PREEMPT_RT 内核

主线 Linux 不是硬实时。如果延迟要求 < 100 µs (无任何抖动)，需要 PREEMPT_RT 补丁内核：

  • 所有内核数据结构可被抢占
  • spinlock 转 mutex
  • 中断也可被抢占
  • 典型抖动从 ms 降到 µs
  • 代价: 平均吞吐略降 5-10%

电信场景 (RAN 实时部分) 常用 PREEMPT_RT。

17.1 案例 1: platform agent 的延迟优化

背景：RU 规模上万，每个 RU 每分钟下发配置 + 上报告警。Agent 总 QPS 设计 100k+, 每个 RPC 响应 p99 < 5 ms。

关键优化：

  1. 每个 RPC 处理用 monotonic_buffer_resource arena → 0 次 malloc
  2. 解析器全用 string_view 引用 socket buffer
  3. 路由表用 FlatMap (vector + sort) 替代 unordered_map (条目少 cache 友好)
  4. session 表分片 16 → 减少锁竞争
  5. Reactor 单线程 + worker 线程池
  6. 异步日志 + 批量 flush

效果：p99 从 50 ms 降到 3 ms (16×)；内存占用减半 (无碎片)。

17.2 案例 2: 告警 (Alarm) 上报路径

要求：告警生成到 NMS 收到 < 100 µs (硬实时分支).

设计：

  • 告警预分配池 (启动期 64K 个 AlarmEntry, 0 次 malloc)
  • SPSC 队列把告警从硬件中断线程 → 推送线程
  • 推送线程 pin 到 isolcpus 隔离核
  • 用 PREEMPT_RT 内核 + SCHED_FIFO 优先级
  • 网络发送用 io_uring 批处理
  • 用 fmt::format_to_n 序列化到栈 buffer (零 string 分配)

效果：max latency < 50 µs 99.999% 时间。

17.3 案例 3: PM 数据采集

背景：每秒数百万条 KPI 采样上报。需要高吞吐 + 不影响主业务延迟。

设计：

  • 每个采样源 thread_local 缓冲 (无锁)
  • 每秒批量提交到 MPMC 队列 (moodycamel)
  • 专门的 aggregator 线程 pull 消费 + 聚合
  • Aggregator 用 SoA 布局 + SIMD 化 reduce
  • 输出用 protobuf + zstd 压缩
  • 网络发送 io_uring 批处理

17.4 经验总结

RPC / RAN 场景实际不要求 HFT 级别 (ns/百 ns)。但稳定性要求很高 — 不能因为某次 GC / page fault 让告警延迟突跳到 500ms。
核心模式：① 启动期预分配；② 热路径零 malloc；③ 关键线程 pin 核；④ 异步化所有 IO；⑤ 用直方图监控延迟分布而不是平均值。

内核旁路 (Kernel Bypass)

17b.1 四大主流方案 · 17b.2 DPDK — 用户态网络全家桶 · 17b.3 RDMA — 远程内存直接读写 · 17b.4 XDP / eBPF — 在内核里拦截，保留协议栈 · 17b.5 选型决策

DPDK / RDMA / XDP / OnLoad ⭐

普通 socket 一次 send/recv 经历 syscall → 内核协议栈 → 网卡驱动，总延迟约 10-30 μs。HFT、5G UPF、CDN 边缘等场景要求 < 5 μs。解决方案：让用户态直接和网卡 DMA 通信，绕过内核协议栈。

技术定位典型延迟使用门槛

17b.2 DPDK — 用户态网络全家桶

核心机制：

  • UIO / VFIO 驱动把网卡从内核接管到用户态
  • 大页内存（2 MB / 1 GB）做 DMA buffer，避免 TLB miss
  • Poll Mode Driver (PMD)：用户态轮询网卡描述符，不用中断
  • 线程独占 CPU（busy polling），换吞吐与延迟
  • 用户态实现 TCP/UDP/ARP/ICMP 等（mTCP / F-Stack / Seastar）
// 最小 DPDK rx loop（概念示意）
#include <rte_eal.h>
#include <rte_ethdev.h>
#include <rte_mbuf.h>

int main(int argc, char** argv) {
    rte_eal_init(argc, argv);                        // 大页/亲和性/PMD 初始化
    struct rte_mempool* pool = rte_pktmbuf_pool_create(
        "mbuf", 8192, 256, 0, RTE_MBUF_DEFAULT_BUF_SIZE, rte_socket_id());

    rte_eth_dev_configure(0, 1, 1, &port_conf);
    rte_eth_rx_queue_setup(0, 0, 1024, 0, &rx_conf, pool);
    rte_eth_tx_queue_setup(0, 0, 1024, 0, &tx_conf);
    rte_eth_dev_start(0);

    // 热路径：100% CPU 轮询，无中断、无 syscall
    struct rte_mbuf* bufs[32];
    for (;;) {
        uint16_t nb = rte_eth_rx_burst(0, 0, bufs, 32);
        for (uint16_t i = 0; i < nb; i++) {
            // 直接拿到 packet 内存 ptr，自己解析 ethernet/ip/udp
            process_packet(bufs[i]);
            rte_pktmbuf_free(bufs[i]);
        }
    }
}
DPDK 应用必须整核独占。一个网卡 → 多 RX queue → 多 lcore（每核一个 worker），用 RSS 哈希分流，无锁。配合 Seastar / F-Stack 这种用户态网络栈，C++ 服务能跑到 100 Mpps。

17b.3 RDMA — 远程内存直接读写

InfiniBand / RoCE / iWARP 三类硬件。客户端可以直接读/写服务端内存，CPU 不参与。延迟 ~1 μs（同机房），吞吐 200+ Gbps。

三种通信原语

原语语义典型用
SEND/RECV两端配合，类似 socket控制平面、握手
RDMA WRITE主动方写到远端内存，远端 CPU 完全不感知分布式 KV、共享日志
RDMA READ主动方从远端内存读分布式缓存、跨主机锁
Atomic FA/CAS跨主机原子操作分布式计数器、锁
// RDMA 用 ibverbs API（高层有 librdmacm / UCX / OFI）
// 1. 创建 Protection Domain、注册内存 (memory region)
ibv_pd* pd  = ibv_alloc_pd(ctx);
ibv_mr* mr  = ibv_reg_mr(pd, buf, len, IBV_ACCESS_REMOTE_WRITE | IBV_ACCESS_LOCAL_WRITE);

// 2. 建 Queue Pair (RC mode 类似 TCP)
ibv_qp* qp  = ibv_create_qp(pd, &qp_init);

// 3. 把本地 (buf, len, rkey) 通过其他通道（TCP）告知对端
// 4. 对端发起 RDMA WRITE：直接写入我的 buf！
//    我的 CPU 全程未感知，可被 poll completion queue 知道完成
ibv_poll_cq(cq, 10, wc);
典型 C++ 框架：UCX（NVIDIA/RPI 主导，含 MPI 后端）、libfabric / OFI。典型用户：HPC 集群通信、分布式存储（Ceph / DAOS / WekaIO）、分布式数据库（TiDB Async commit）、AI 训练（NCCL）。

17b.4 XDP / eBPF — 在内核里拦截，保留协议栈

XDP（eXpress Data Path）让 eBPF 程序在驱动收到包后立即运行，可决定：

  • XDP_DROP — 丢包（DDoS 防护，10x 快于 iptables）
  • XDP_PASS — 交给内核协议栈
  • XDP_TX — 立即回弹（L4 负载均衡）
  • XDP_REDIRECT — 转发到另一网卡或 AF_XDP 用户态 socket

典型用户：Cloudflare（10M pps DDoS 过滤）、Facebook Katran（L4 LB）、Cilium（K8s 网络）。

硬件时间戳 · PTP / PHC

17c.1 三档时间精度 · 17c.2 PTP 工作原理（极简） · 17c.3 C++ 代码读硬件时戳

当 μs 级时间精度成为刚需 ⭐

金融做市要求"事件时间戳精度优于 1 μs"（用于审计、回放、判别因果）。NTP 同步精度只有毫秒级，软件 gettimeofday 误差受 NTP / VDSO / CPU 频率变化影响。解决方案：PTP（Precision Time Protocol, IEEE 1588）+ 网卡硬件时钟（PHC, PTP Hardware Clock）。

17c.1 三档时间精度

方案精度成本
NTP~ms (千 μs)免费，普通网卡
软件 PTP（ptp4l）~10-100 μs免费
硬件 PTP（网卡 PHC）~50-500 ns支持 PHC 的网卡（Mellanox / Solarflare / Intel I210+）
White Rabbit / GPS~1 ns金融交易所、CERN

17c.2 PTP 工作原理（极简）

主时钟周期性发 SYNC + FOLLOW_UP，从时钟测出"网络单程延迟"和"主从时钟差"，校准本地 PHC。关键：网卡硬件给收发包打时戳（< 100 ns 抖动），不依赖操作系统调度。

# 一台机器作主，多台机器作从（典型机房部署）
# 主
sudo ptp4l -i eth0 -m -H -s    # -H = hardware ts

# 从（同子网）
sudo ptp4l -i eth0 -m -H       # 自动发现主

# 把 PHC 反向同步到系统时钟（让 gettimeofday 也准）
sudo phc2sys -s eth0 -O 0 -m

# 看实时偏差
pmc -u -b 0 'GET CURRENT_DATA_SET'
# offsetFromMaster: 28 ns   ← 已经收敛到 28 ns

// 直接读网卡 PHC（精度 ~ns）
#include <linux/ptp_clock.h>
int phc_fd = open("/dev/ptp0", O_RDONLY);
clockid_t clk = ((~(clockid_t)phc_fd) << 3) | 3;

clock_gettime(clk, &ts);

isolcpus / nohz_full / cpuset / IRQ affinity ⭐

低延时编程的"最后一公里"：把整个 CPU 核从 Linux 调度器和定时中断里拔出来，专属给业务线程独占。这样可以消灭：内核线程抢占、ticks 中断 (~250-1000 Hz)、IRQ 中断、负载均衡迁移。

17d.1 内核启动参数三件套

参数作用例
isolcpus=N-M调度器不在这些核上放普通任务（你必须显式 pin）isolcpus=4-15
nohz_full=N-M这些核单进程运行时关闭 1kHz tick（消除最重要的抖动源）nohz_full=4-15
rcu_nocbs=N-MRCU 回调不在这些核上跑（把 RCU offload 到别处）rcu_nocbs=4-15
irqaffinity=0-3所有 IRQ 默认只送给 0-3 核—
intel_pstate=disable禁用动态频率，避免降频—
processor.max_cstate=1禁深度 C-state，避免唤醒延迟—

/etc/default/grub 示例（4 个 hot core）

GRUB_CMDLINE_LINUX="isolcpus=4-7 nohz_full=4-7 rcu_nocbs=4-7 \
  irqaffinity=0-3 intel_pstate=disable processor.max_cstate=1 \
  intel_idle.max_cstate=0 mce=ignore_ce idle=poll"

17d.2 业务程序如何使用这些隔离核

// ① 设置 CPU affinity，把业务线程钉到 isolated 核
cpu_set_t set;
CPU_ZERO(&set);
CPU_SET(4, &set);
pthread_setaffinity_np(thread, sizeof(set), &set);

// ② 锁内存，禁止换出（防 swap / 缺页）
mlockall(MCL_CURRENT | MCL_FUTURE);

// ③ 提升优先级（可选，配合 SCHED_FIFO/RR）
sched_param sp{.sched_priority = 80};
sched_setscheduler(0, SCHED_FIFO, &sp);   // 需 CAP_SYS_NICE

// ④ 预触摸所有内存避免后续缺页
for (char* p = buf; p < buf + size; p += 4096) *p = 0;

17d.3 IRQ 亲和性 —— 别让网卡中断打扰

# 找网卡 IRQ 号
grep eth0 /proc/interrupts | awk -F':' '{print $1}'

# 把它们绑到 housekeeping 核（cpu0-3），不打扰 4-7
for irq in 56 57 58 59; do
  echo 0-3 > /proc/irq/$irq/smp_affinity_list
done

# 设置 RPS / RFS，把 packet 处理也限定在 housekeeping 核
echo f > /sys/class/net/eth0/queues/rx-0/rps_cpus

17d.5 PREEMPT_RT 内核 —— 终极硬实时

如果还不够，上 PREEMPT_RT 实时内核补丁（Linux 6.12 起部分合入主线）：让几乎所有内核临界区可被抢占，硬实时延迟可保证在 < 100 μs（甚至 < 10 μs）。代价：吞吐率下降 5-15%。

17d.6 完整低延时部署 checklist

  1. BIOS：关闭 HT、关闭 turbo / SpeedStep、关闭 C-states、内存设最高频
  2. 内核 cmdline：isolcpus + nohz_full + rcu_nocbs + intel_pstate=disable + idle=poll
  3. 把所有 IRQ / kworker 赶到 housekeeping 核
  4. 业务进程：mlockall + pin 核 + SCHED_FIFO + 预触内存
  5. 网卡：DPDK 或 SO_BUSY_POLL + ethtool 调 rx/tx ring
  6. 定时器：用 timerfd + epoll，不要 setitimer/SIGALRM
  7. 禁用 transparent huge pages（除非业务受益）：echo never > /sys/kernel/mm/transparent_hugepage/enabled
  8. 用 cyclictest / hwlatdetect 验证抖动收敛
  9. P99/P999/P9999 用 HDR Histogram 实时监控
价值排序：隔核 (大头) > mlockall > nohz_full > IRQ 亲和 > SCHED_FIFO > PREEMPT_RT。HFT、5G UPF、PTP grandmaster 这种场景每一条都要上；普通低延时服务一般做前三条已经够。
