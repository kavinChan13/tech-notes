1. 引言 — 为什么 CMake 仍是 C++ 工程首选

C++ 构建系统过去三十年百花齐放（Make / Autotools / SCons / Bazel / Meson / xmake / CMake / build2 / ninja...）。CMake 胜出的原因：

  • 跨平台：Linux / macOS / Windows + 多种编译器（gcc/clang/MSVC/Intel/Apple）
  • IDE 集成：CLion / VS / VSCode / Qt Creator 一等公民
  • 生态：90% 主流 C++ 库都有 FindXXX.cmake 或 export config
  • 包管理器适配：vcpkg / conan 都基于 CMake
  • 持续演进：3.x 起的 "Modern CMake" 让代码大幅简化

2. 现代 CMake 哲学 vs 老式

2.1 老式 CMake (CMake 2.x 时代)

# 老式 - directory-based, 全局变量满天飞
include_directories(/usr/include/foo)            # 全局污染所有 target!
add_definitions(-DFOO=1)                         # 全局
set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -O2")  # 全局, 拼字符串
link_directories(/usr/lib/foo)                   # 全局

add_executable(myapp main.cpp)
target_link_libraries(myapp foo)                 # 链接 foo (假设全局可见)

问题：

  • 全局污染 — 一个 target 的设置影响所有其他 target
  • 变量字符串拼接 — 难维护、难调试
  • 子项目相互干扰 — 加一个 add_subdirectory 改变整个构建
  • 不可重用

2.2 现代 CMake (3.5+ 起，3.15+ 推荐)

# 现代 - target-based, 每个 target 自带属性
add_executable(myapp main.cpp)

target_include_directories(myapp PRIVATE /usr/include/foo)
target_compile_definitions(myapp PRIVATE FOO=1)
target_compile_options(myapp PRIVATE -O2)
target_link_libraries(myapp PRIVATE foo)         # foo 是 imported target

关键差别：

  • 所有设置挂在 target 上而不是全局
  • 用 PUBLIC / PRIVATE / INTERFACE 控制传递性
  • 变量基本不用，全靠 target 属性
  • 可重用、可组合、可隔离

2.3 现代 CMake 五大原则

原则含义
1. Treat CMake as code用代码风格写 CMake，不是配置文件
2. Targets are everything一切以 target 为中心，避免全局变量
3. Use PUBLIC/PRIVATE/INTERFACE明确传递性，避免泄漏
4. Modern features over legacy用 target_xxx 不用 add_xxx
5. CMake 3.15+ baseline低于这版本的"现代"很少 ROI

2.4 推荐版本要求

cmake_minimum_required(VERSION 3.20)        # 推荐. 含 CMakePresets, fetchContent 完善
project(myapp VERSION 1.0 LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)               # 禁用 GNU 扩展, 用纯标准
团队规范：cmake_minimum_required(VERSION 3.20) 起。CMake 2.x 已经无人用，CMake 3.5-3.14 是过渡期，3.15+ 是现代 CMake 真正开始。

add_library(mylib STATIC

    VERSION 1.0.0
    DESCRIPTION "My awesome project"
    LANGUAGES CXX
)

    enable_testing()

endif()

if(BUILD_BENCHMARKS)
    add_subdirectory(benchmarks)
endif()

4. Targets — 一切以 target 为中心

4.1 三种 target 类型

类型命令用途
可执行文件add_executable(name srcs...)程序入口
静态库add_library(name STATIC srcs...)编译时链接
动态库add_library(name SHARED srcs...)运行时加载
对象库add_library(name OBJECT srcs...)不打包，仅 .o 集合（共享中间产物）
接口库add_library(name INTERFACE)纯头文件库 / 仅传递设置
导入库add_library(name STATIC IMPORTED)引用预编译的库
别名add_library(ns::name ALIAS name)命名空间风格

4.2 INTERFACE 库 — 纯头文件库

# 头文件库 — 没有 .cpp, 只传递设置
add_library(myheader_only INTERFACE)
target_include_directories(myheader_only INTERFACE include/)
target_compile_features(myheader_only INTERFACE cxx_std_20)
target_compile_definitions(myheader_only INTERFACE MY_HEADER_ONLY=1)

# 别人这样用
add_executable(app main.cpp)
target_link_libraries(app PRIVATE myheader_only)
# app 自动获得: include 路径, C++20 标准, MY_HEADER_ONLY 宏

4.3 别名 target — 命名空间风格

add_library(myproject_core STATIC ...)
add_library(myproject::core ALIAS myproject_core)

# 别人用
target_link_libraries(app PRIVATE myproject::core)
# 一致的命名 - 与 find_package 找到的 target 名一致
所有对外暴露的 target 都该提供 namespace::name 别名。这样：① 写法统一；② 链接错误更清晰（"找不到 myproject::core" 一眼明了）；③ 可被 find_package / FetchContent 替代时不需要改用户代码。

4.4 OBJECT 库 — 共享中间产物

# 一份代码同时给静态库和动态库用 — 避免重复编译
add_library(mycore OBJECT
    src/foo.cpp src/bar.cpp
)
target_include_directories(mycore PUBLIC include/)

add_library(mylib_static STATIC $<TARGET_OBJECTS:mycore>)
add_library(mylib_shared SHARED $<TARGET_OBJECTS:mycore>)

5. PUBLIC / PRIVATE / INTERFACE 传递性 ⭐⭐⭐

这是现代 CMake 的核心概念。理解了这三个关键字，CMake 就掌握了 80%。

5.1 三个关键字的含义

关键字本 target 用下游 target 用
PRIVATE✅ 用❌ 不传递
INTERFACE❌ 不用✅ 传递
PUBLIC✅ 用✅ 传递

"下游" = 用 target_link_libraries(B PRIVATE/PUBLIC A) 链接 A 的 B。

5.2 决策法则（Daniel Pfeifer 的经典口诀）

对于 target A 的某个属性 X，问自己 2 个问题：

  1. A 自己的 .cpp 需要 X 吗?
  2. A 的头文件暴露了 X 吗? (链接 A 的人能看到 X)
A 自用?头文件暴露?用
是否PRIVATE
否是INTERFACE
是是PUBLIC

5.3 实例分析

# mylib 的实现里用了 fmt 但不在 mylib 的头文件里出现
add_library(mylib src/foo.cpp)
target_link_libraries(mylib PRIVATE fmt::fmt)
# ↑ mylib.cpp 用 fmt → 自用. 但 mylib.h 里不出现 fmt → 不传递. PRIVATE.

# mylib 的头文件里 #include <fmt/format.h> (类型/宏暴露)
add_library(mylib src/foo.cpp)
target_link_libraries(mylib PUBLIC fmt::fmt)
# ↑ 用户的代码也得能 include fmt → 传递. PUBLIC.

# 头文件库 (interface lib) 完全不编译, 只传递
add_library(json_helper INTERFACE)
target_link_libraries(json_helper INTERFACE nlohmann_json::nlohmann_json)
# ↑ 自身没 .cpp, 不需要 fmt. 但用户能 include 我的 .hpp → 必须传 nlohmann_json. INTERFACE.

5.4 头文件路径同样规则

add_library(mylib src/foo.cpp)

target_include_directories(mylib
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>     # build 时
        $<INSTALL_INTERFACE:include>                                # install 后
    PRIVATE
        ${CMAKE_CURRENT_SOURCE_DIR}/src       # 仅本库自己 #include
)

含义：

  • include/ — 是 mylib 的公共 API 路径，下游也得能 #include → PUBLIC
  • src/ — 是 mylib 的内部头，只本库需要 → PRIVATE
  • BUILD_INTERFACE / INSTALL_INTERFACE 生成器表达式分别处理"本地构建" 和 "安装后" 两种路径

5.5 编译选项 / 定义同样规则

target_compile_definitions(mylib
    PUBLIC  MYLIB_PUBLIC_MACRO=1     # 暴露在 mylib.h, 用户也得有
    PRIVATE MYLIB_INTERNAL_MACRO=1   # 仅 mylib.cpp 用
)
target_compile_options(mylib
    PUBLIC  -fvisibility=hidden       # 用户也要 (符号可见性一致)
    PRIVATE -Wall -Wextra             # 仅本库构建时启用
)
"解释一下 PUBLIC / PRIVATE / INTERFACE 区别。"
答：PRIVATE = 自用不传，INTERFACE = 不用只传，PUBLIC = 自用 + 传。
判断：① 我的 .cpp 用吗？② 我的头文件暴露吗？两个 yes/no 决定。

6. 头文件 / 编译选项 / 链接

6.1 头文件路径

    PRIVATE src/internal/

# 单个特性
target_compile_features(mylib PUBLIC
    cxx_constexpr
    cxx_lambda_init_captures

6.3 编译选项

target_compile_options(mylib PRIVATE
    -Wall
    -Wextra
    -Wpedantic
    -Wshadow
    -Wnon-virtual-dtor

6.4 编译器警告 module 化

# cmake/CompilerWarnings.cmake
function(set_project_warnings target_name)
    set(MSVC_WARNINGS
        /W4 /permissive- /w14242 /w14254 /w14263 /w14265 /w14287
        /w14289 /w14296 /w14311 /w14545 /w14546 /w14547 /w14549
        /w14555 /w14619 /w14640 /w14826 /w14905 /w14906 /w14928
    )
    set(CLANG_WARNINGS
        -Wall -Wextra -Wpedantic
        -Wshadow -Wnon-virtual-dtor
        -Wold-style-cast -Wcast-align -Wunused
        -Woverloaded-virtual -Wconversion -Wsign-conversion
        -Wdouble-promotion -Wformat=2
        -Wimplicit-fallthrough
    )
    set(GCC_WARNINGS
        ${CLANG_WARNINGS}
        -Wmisleading-indentation -Wduplicated-cond
        -Wduplicated-branches -Wlogical-op -Wuseless-cast
    )

    if(MSVC)
        set(WARNINGS ${MSVC_WARNINGS})
    elseif(CMAKE_CXX_COMPILER_ID MATCHES "Clang")
        set(WARNINGS ${CLANG_WARNINGS})
    elseif(CMAKE_CXX_COMPILER_ID STREQUAL "GNU")
        set(WARNINGS ${GCC_WARNINGS})
    endif()

    target_compile_options(${target_name} PRIVATE ${WARNINGS})
endfunction()

# 调用
add_executable(myapp main.cpp)
set_project_warnings(myapp)

6.5 链接库

target_link_libraries(myapp
    PRIVATE
        mylib                    # 同项目 target
        fmt::fmt                  # find_package / FetchContent 找到的
        Threads::Threads          # 标准 pthread (跨平台)
        ${CMAKE_DL_LIBS}          # dl 库 (动态加载)
)

6.6 链接选项 (linker flags)

target_link_options(myapp PRIVATE
    -Wl,--as-needed              # 不链接未使用的库
    -Wl,--gc-sections             # 移除未使用的 section
)

7. 子目录 · FetchContent · ExternalProject

7.1 add_subdirectory — 同仓库子模块

# 顶层 CMakeLists
add_subdirectory(src)
add_subdirectory(tests)

# src/CMakeLists.txt
add_library(mylib STATIC foo.cpp bar.cpp)
target_include_directories(mylib PUBLIC ${CMAKE_CURRENT_SOURCE_DIR}/include)

7.2 FetchContent — 在线拉取并构建

CMake 3.11+ 推荐方式。直接从 git/url 拉取依赖，作为子目录构建。

include(FetchContent)

FetchContent_Declare(
    fmt
    GIT_REPOSITORY https://github.com/fmtlib/fmt.git
    GIT_TAG        10.2.1
    GIT_SHALLOW    TRUE
)

FetchContent_Declare(
    googletest
    GIT_REPOSITORY https://github.com/google/googletest.git
    GIT_TAG        v1.14.0
)

# 一次拉取多个 (3.14+)
FetchContent_MakeAvailable(fmt googletest)

# 现在可直接用
target_link_libraries(myapp PRIVATE fmt::fmt)

优势：

  • 无需安装到系统
  • 跨平台一致
  • 版本锁定在代码里 (可重现构建)
  • CI 友好

劣势：

  • 每次干净构建都要拉源码
  • 编译时间长 (要编依赖)
  • 依赖代码进入你的构建系统 (可能版本冲突)

7.3 ExternalProject — 高级版本控制

更老的 API，下载 + 单独编译 (不集成到本工程的 build graph)。已被 FetchContent 大体替代，仅在需要独立 build pipeline 时用。

7.4 选型决策

需求推荐

跨平台 + 版本锁定FetchContent
团队统一管理vcpkg / conan
大型公司 monorepovendoring (复制源码) + add_subdirectory

9. install / export — 让别人通过 find_package 用你的库

如果你写库给别人用，要让他们能 find_package(mylib)。这需要正确 install + export。

9.1 完整流程

cmake_minimum_required(VERSION 3.20)
project(mylib VERSION 1.0 LANGUAGES CXX)

include(GNUInstallDirs)
include(CMakePackageConfigHelpers)

# --- 库本身 ---
add_library(mylib STATIC src/foo.cpp src/bar.cpp)
add_library(mylib::mylib ALIAS mylib)

target_compile_features(mylib PUBLIC cxx_std_20)
target_include_directories(mylib
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:${CMAKE_INSTALL_INCLUDEDIR}>
)

# --- 安装规则 ---
install(TARGETS mylib
    EXPORT mylibTargets
    LIBRARY DESTINATION ${CMAKE_INSTALL_LIBDIR}
    ARCHIVE DESTINATION ${CMAKE_INSTALL_LIBDIR}
    INCLUDES DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}
)
install(DIRECTORY include/ DESTINATION ${CMAKE_INSTALL_INCLUDEDIR})

# --- 导出 target ---
install(EXPORT mylibTargets
    FILE mylibTargets.cmake
    NAMESPACE mylib::
    DESTINATION ${CMAKE_INSTALL_LIBDIR}/cmake/mylib
)

# --- 生成 mylibConfig.cmake ---
configure_package_config_file(
    ${CMAKE_CURRENT_SOURCE_DIR}/cmake/mylibConfig.cmake.in
    ${CMAKE_CURRENT_BINARY_DIR}/mylibConfig.cmake
    INSTALL_DESTINATION ${CMAKE_INSTALL_LIBDIR}/cmake/mylib
)
write_basic_package_version_file(
    ${CMAKE_CURRENT_BINARY_DIR}/mylibConfigVersion.cmake
    VERSION ${PROJECT_VERSION}
    COMPATIBILITY SameMajorVersion
)
install(FILES
    ${CMAKE_CURRENT_BINARY_DIR}/mylibConfig.cmake
    ${CMAKE_CURRENT_BINARY_DIR}/mylibConfigVersion.cmake
    DESTINATION ${CMAKE_INSTALL_LIBDIR}/cmake/mylib
)

9.2 mylibConfig.cmake.in 模板

# cmake/mylibConfig.cmake.in
@PACKAGE_INIT@

include(CMakeFindDependencyMacro)
find_dependency(fmt 10.0)            # 如果 mylib 依赖 fmt 且 PUBLIC 链接

include("${CMAKE_CURRENT_LIST_DIR}/mylibTargets.cmake")
check_required_components(mylib)

9.3 用户使用

# 安装 mylib
cmake -S . -B build -DCMAKE_INSTALL_PREFIX=/usr/local
cmake --install build

# 用户的 CMakeLists
find_package(mylib 1.0 REQUIRED)
target_link_libraries(myapp PRIVATE mylib::mylib)
规则：库的 PUBLIC 依赖必须在 Config.cmake 里 find_dependency()，否则用户 find_package 你的库时找不到传递依赖。

15. Sanitizer + Coverage 在 CMake 中的集成

15.1 Sanitizers.cmake module

# cmake/Sanitizers.cmake
function(enable_sanitizers target)
    if(NOT (CMAKE_CXX_COMPILER_ID STREQUAL "GNU" OR
            CMAKE_CXX_COMPILER_ID MATCHES ".*Clang"))
        return()
    endif()

    set(SANITIZERS "")
    if(ENABLE_ASAN)
        list(APPEND SANITIZERS "address")
    endif()
    if(ENABLE_UBSAN)
        list(APPEND SANITIZERS "undefined")
    endif()
    if(ENABLE_TSAN)
        if(ENABLE_ASAN OR ENABLE_MSAN)
            message(FATAL_ERROR "TSan cannot combine with ASan/MSan")
        endif()
        list(APPEND SANITIZERS "thread")
    endif()
    if(ENABLE_MSAN)
        list(APPEND SANITIZERS "memory")
    endif()

    list(JOIN SANITIZERS "," SAN_LIST)
    if(SAN_LIST)
        target_compile_options(${target} PRIVATE
            -fsanitize=${SAN_LIST}
            -fno-omit-frame-pointer
            -fno-optimize-sibling-calls
        )
        target_link_options(${target} PRIVATE -fsanitize=${SAN_LIST})
    endif()
endfunction()

# 调用
add_executable(myapp main.cpp)
enable_sanitizers(myapp)

15.2 Coverage.cmake module

# cmake/Coverage.cmake
function(enable_coverage target)
    if(NOT ENABLE_COVERAGE)
        return()
    endif()
    target_compile_options(${target} PRIVATE
        -O0 -g --coverage
    )
    target_link_options(${target} PRIVATE --coverage)
endfunction()

15.3 在 CI 中使用

18b.2 Ninja —— CMake 的最佳后端

Ninja（Evan Martin，Google 2010）是极简快速的构建执行器。CMake 生成 ninja 文件，ninja 解析速度远快于 Make。

# 让 CMake 用 ninja 后端
cmake -G Ninja -B build
ninja -C build           # 并行编译，默认 = cpu 核数
ninja -C build -j 32 -l 28   # 32 并发，load avg 28 时不再加

# 看 ninja 实际跑了多久（CRITICAL PATH 分析）
ninja -C build -t graph > build.dot
ninja -C build -t deps   # 看每文件依赖
ninja -C build -t recompact   # 压缩 .ninja_deps

# CMakePresets.json 一行设置默认 generator
"generator": "Ninja"
单看构建图启动时间，ninja 比 Make 快 5-50×。所有 CI 都该默认 Ninja。

18b.3 Unity / Jumbo Build —— 把多个 .cpp 合一

把 10 个 .cpp 用 #include 合并成一个 TU 编译，大幅减少模板实例化、头文件解析重复。Chromium 早期用 "jumbo build" 把构建时间砍半。

# CMake 3.16+ 一行启用 Unity Build
set_target_properties(mylib PROPERTIES
    UNITY_BUILD ON
    UNITY_BUILD_BATCH_SIZE 8)        # 每 8 个 .cpp 合一

# 或全局开
set(CMAKE_UNITY_BUILD ON)
副作用：① 匿名命名空间符号冲突（两个 .cpp 都有 static void foo()）；② 同名宏 / using 互相污染；③ 增量编译变差（改 1 个 .cpp 重编一批）。一般在 CI 全量 build 开 Unity，本地开发关。

18b.4 Precompiled Headers (PCH)

# CMake 3.16+ 一行
target_precompile_headers(mylib PRIVATE
    <string> <vector> <memory> <unordered_map>
    "common/internal.h")

典型项目大模板（<regex>、<chrono>、Boost）编译 PCH 后增量加速 20-50%。注意：PCH 跨编译器选项失效（debug/release 各一份）。

sccache --show-stats

18b.6 distcc / icecream — 分布式编译

把 .cpp 编译任务分发到一组机器跑，本地只做预处理 + 链接。10 台 CI 机就是 10 倍编译能力。

# distcc（最老）
export CC="distcc gcc"
export DISTCC_HOSTS="localhost build1 build2 build3"
make -j 60

# icecream（KDE 出品，更智能，自动节点发现）
sudo apt install icecc
export PATH=/usr/lib/icecc/bin:$PATH
make -j 100        # 节点自动加入 farm

# 与 ccache 组合 — 先看 cache，没命中才分发
CMAKE_CXX_COMPILER_LAUNCHER="ccache;distcc"

18b.7 Bazel —— Google 风格的密封 + 远程缓存 + 远程执行

Bazel 三个杀手锏：

  1. 密封性 (Hermetic)：每条编译命令的输入由 BUILD 文件完全声明，无隐式依赖 → 内容哈希做 cache key 100% 准
  2. Remote Cache：把构建产物存远程（gRPC + CAS），全公司共享。新 dev 拉代码后第一次 build 几乎全 cache hit
  3. Remote Build Execution (RBE)：编译命令本身也分发到远程集群跑，本地几乎只做 IO
# 最小 BUILD.bazel
cc_library(
    name = "foo",
    srcs = ["foo.cc"],
    hdrs = ["foo.h"],
    deps = ["//bar", "@absl//absl/strings"],
    visibility = ["//visibility:public"],
)

cc_binary(
    name = "myapp",
    srcs = ["main.cc"],
    deps = [":foo"],
)

cc_test(
    name = "foo_test",
    srcs = ["foo_test.cc"],
    deps = [":foo", "@googletest//:gtest_main"],
)

Bazel 优势

  • 万亿行级 monorepo的事实标准（Google / Uber / Twitter / Lyft / Pinterest）
  • 跨语言：C++ + Java + Go + Python + Rust 一套构建
  • Remote Cache 让团队级构建从 30 分钟降到 30 秒（cache hit）
  • 支持分级构建：只 build/test 受改动影响的目标

Bazel 劣势

  • 启动成本高：BUILD 文件 + WORKSPACE 学习曲线陡
  • 整合 GCC/Clang 工具链需写 toolchain rules（CROSSTOOL）
  • 与 CMake 生态（vcpkg / conan / find_package）互通成本极高
  • 本地小项目 overhead 大于收益
选型经验：项目 < 50 万行 C++、外部依赖多 → CMake + Ninja + ccache；> 100 万行 + monorepo + 多语言 → 考虑 Bazel / Buck2；中间状态可看 Meson。

18b.8 C++20 Modules —— 构建系统的下一站

Modules 解决 C++ 编译模型的根本问题：头文件被重复解析。一个 #include <vector> 在每个 TU 都重新展开 ~25 万行预处理代码。Modules 改成"编译一次，二进制 BMI（Binary Module Interface）导入"。

// math.cppm — module interface unit
export module math;

export int square(int x) { return x * x; }
int internal_helper() { ... }   // 不 export = 模块私有

// main.cpp — consumer
import math;
import std;            // C++23: 整个标准库一行导入

int main() {
    std::cout << square(5) << "\n";
}

编译器与构建系统支持现状（2026）

工具状态
Clang 17+✅ 完整 modules + std module
GCC 14+✅ 大部分 features
MSVC 19.36+✅ 主导实现
CMake 3.28+✅ FILE_SET CXX_MODULES（实验稳定）
Ninja 1.11+✅ dyndep 支持模块依赖发现
Bazel⚠️ 实验性，2025 起改进

CMake 启用 modules 示例

cmake_minimum_required(VERSION 3.28)
project(modules_demo CXX)
set(CMAKE_CXX_STANDARD 23)
set(CMAKE_CXX_MODULE_STD ON)

add_library(math_mod)
target_sources(math_mod PUBLIC
    FILE_SET CXX_MODULES FILES math.cppm
)

add_executable(app main.cpp)
target_link_libraries(app PRIVATE math_mod)

Modules 实测收益

  • 大量使用 <regex> / <chrono> / Boost 的 TU，编译加速 20-50%
  • 纯小文件 / 已用 PCH 的项目，收益 5-15%
  • 但 BMI 不像 .o 那样可缓存（依赖编译器版本/选项），分布式编译复杂度上升
现实建议（2026）：modules 已可以在新项目里试用，但大型既有项目"全量迁移"成本巨大（要重写所有公共头）。渐进策略：① 新增叶子库用 modules；② 公共依赖暂保留传统 header；③ 等 5 年看 Bazel/CMake 大型生态成熟再考虑全面切换。

18b.10 真实项目编译加速 checklist

  1. Ninja 替换 Make（5% 起步）
  2. ccache + 在 CI 上配 S3 远程后端（命中率 80%+ 时大幅加速）
  3. PCH 关键头（<string>/<vector>/项目核心头）
  4. Unity Build（CI 全量 OK，本地按需）
  5. IWYU 清理多余 include
  6. 大型类型用 PIMPL / extern template
  7. distcc / icecc / sccache 分布式编译（成本 vs 收益评估）
  8. Modules：先在新增叶子库试水
  9. 对超大型 monorepo，评估迁移 Bazel/Buck2
典型成绩：50 万行 C++ 服务，全量编译时间 40 分钟 → 6 分钟（Ninja + ccache 远程 + PCH + IWYU + Unity）。性价比最高的组合是"Ninja + ccache S3 + PCH"，零代码侵入。

19. 易错点速查（10 个高频陷阱）

  1. 用 PUBLIC 时不传递依赖 
    target_link_libraries(mylib PUBLIC fmt::fmt)
# 别忘 mylibConfig.cmake 里加 find_dependency(fmt)
  2. include_directories 全局污染
    用 target_include_directories 替代。
  3. set CMAKE_CXX_FLAGS 拼字符串
    用 target_compile_options 替代。
  4. ALIAS target 名缺命名空间 
    add_library(mylib::mylib ALIAS mylib)   # 必须有命名空间
  5. install 时忘了 EXPORT
    install(TARGETS … EXPORT mylibTargets) 才能用 find_package。
  6. FetchContent 没用 GIT_SHALLOW
    首次拉取超慢。加上后只拉单 commit。
  7. CMakePresets 缺少 schema 字段
    "version": 6 不可少。
  8. 没 set CMAKE_EXPORT_COMPILE_COMMANDS
    clang-tidy / VSCode 的 IntelliSense 都要它。
  9. cmake_minimum_required 版本太低
    < 3.15 没法享受现代 CMake 大量改进。
  10. add_executable / add_library 先于 find_package
    可以但混乱。约定俗成：先 project, 再 find_package, 再 add_*。

20.3 速答示范（第 2 题）

问：PUBLIC / PRIVATE / INTERFACE 三个关键字含义?

答：

  • PRIVATE = 自己用，不传给下游
  • INTERFACE = 自己不用，只传给下游
  • PUBLIC = 自己用 + 传给下游 (PRIVATE + INTERFACE)

判断方法：

  1. 本 target 的 .cpp 用吗?
  2. 本 target 的头文件暴露吗? (用户能不能看到这个属性)

都用 → PUBLIC；只 1 用 → PRIVATE；只 2 用 → INTERFACE。

例子：

target_link_libraries(mylib
    PRIVATE  spdlog::spdlog    # mylib.cpp 用 spdlog, 但头文件不暴露
    PUBLIC   fmt::fmt          # mylib.h 里用了 fmt 类型 → 用户也得有
)
