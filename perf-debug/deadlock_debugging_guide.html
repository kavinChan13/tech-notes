死锁的四个必要条件 (Coffman Conditions, 1971)

死锁 (Deadlock) 是指两个或多个线程互相持有对方所需的资源，且都不释放，从而进入永久阻塞状态

死锁 (Deadlock) 是指两个或多个线程互相持有对方所需的资源，且都不释放，从而进入永久阻塞状态。 1971 年 E.G. Coffman 等人证明：死锁发生 当且仅当 同时满足以下四个条件。打破任意一条即可破除死锁。

条件含义常见破除策略
1. 互斥 (Mutual Exclusion) 资源同一时刻只能被一个线程独占 读写锁、无锁结构、不可变对象、RCU
2. 持有并等待 (Hold and Wait) 线程持有资源 A 的同时申请资源 B 一次性申请所有资源 (std::lock)、等不到就先释放已持有的
3. 不可抢占 (No Preemption) 资源不能被强制夺走，必须由持有者主动释放 try_lock、timed_mutex 超时退避
4. 循环等待 (Circular Wait) 形成 T1→T2→...→Tn→T1 的资源等待环 统一全局锁序 (Lock Hierarchy)

2.1 Lock Ordering Violation (AB-BA 锁序反转) 最常见

两个线程以相反顺序获取同一对锁，是生产环境最常见的死锁形态。

✗ 死锁写法

std::mutex mA, mB;

void thread1() {
    std::lock_guard g1(mA);  // 1. 拿 A
    std::this_thread::sleep_for(10ms);
    std::lock_guard g2(mB);  // 2. 等 B  ⏳
}

void thread2() {
    std::lock_guard g1(mB);  // 1. 拿 B
    std::this_thread::sleep_for(10ms);
    std::lock_guard g2(mA);  // 2. 等 A  💀 死锁
}

✓ 正确写法 — std::scoped_lock (C++17)

std::mutex mA, mB;

void safe_op() {
    // scoped_lock 内部用 deadlock-avoidance 算法
    // 同时持有所有锁，顺序无关
    std::scoped_lock lk(mA, mB);
    // ... 临界区 ...
}

// C++14 等价写法
std::unique_lock l1(mA, std::defer_lock);
std::unique_lock l2(mB, std::defer_lock);
std::lock(l1, l2);              // 原子获取，避免 AB-BA

2.2 自死锁 (Self-Deadlock) — 同一线程递归加锁非递归 mutex 易忽略

✗ 死锁写法

class Counter {
    std::mutex mu_;
    int n_ = 0;
public:
    void inc() {
        std::lock_guard lk(mu_);
        n_++;
    }
    void incIfPositive() {
        std::lock_guard lk(mu_);
        if (n_ > 0) inc();   // 💀 同线程再次锁 mu_
    }
};

✓ 正确写法 — 抽出私有无锁版本

class Counter {
    std::mutex mu_;
    int n_ = 0;

    void incLocked() { n_++; }     // 不加锁版
public:
    void inc() {
        std::lock_guard lk(mu_);
        incLocked();
    }
    void incIfPositive() {
        std::lock_guard lk(mu_);
        if (n_ > 0) incLocked();
    }
};
std::recursive_mutex 也能解决，但通常是设计问题的信号——它会让“到底持有几次锁”变得不可读。优先重构。

2.3 条件变量误用：Spurious Wakeup & Lost Signal 常见

✗ Lost Signal — 永久阻塞

std::mutex mu;
std::condition_variable cv;
bool ready = false;

// Producer
producer():
    ready = true;     // ❌ 没加锁修改
    cv.notify_one();      // ❌ 可能在 consumer 进入 wait 前发出

// Consumer
consumer():
    std::unique_lock lk(mu);
    cv.wait(lk);          // 💀 信号可能永远丢失，且无谓词检查

✓ 正确写法 — 谓词 + 锁内修改

// Producer
producer():
    {
        std::lock_guard lk(mu);
        ready = true;     // 锁内修改状态
    }
    cv.notify_one();

// Consumer
consumer():
    std::unique_lock lk(mu);
    cv.wait(lk, []{ return ready; });
    // ↑ 带谓词版本自动处理 spurious wakeup +
    //   防止 lost signal (notify 在 wait 前到也 OK)
三大铁律：① 修改条件变量关联的状态必须持锁；② wait() 必须传谓词；③ notify 可在锁外（性能更好）但状态更改必须在锁内。

2.4 Reader-Writer 锁升级死锁 隐蔽

当多个线程持有共享读锁，同时尝试升级为独占写锁时，所有线程互相等待对方释放读锁——典型读写锁升级死锁。

✗ 升级死锁

std::shared_mutex m;
int data = 0;

void readerThenWrite() {
    std::shared_lock rd(m);    // 多线程同时进来
    if (data == 0) {
        rd.unlock();
        std::unique_lock wr(m); // 💀 互相等待
        if (data == 0) data = 42;
    }
}

✓ 直接独占 + Double-checked

void readerThenWrite() {
    {
        std::shared_lock rd(m);
        if (data != 0) return;     // fast path
    }
    std::unique_lock wr(m);            // 升级前完全释放
    if (data == 0) data = 42;          // 持锁后再校验
}
C++ 标准 shared_mutex 没有原子升级 API。Boost 提供 upgrade_lock，但需所有线程都用相同语义，否则仍可能死锁。

2.5 lock_guard 在异常路径下其实是安全的——但 raw lock/unlock 不是 陷阱

✗ 锁泄漏 (raw lock/unlock)

std::mutex m;

void badProcess() {
    m.lock();                    // 手动加锁
    if (cond) throw std::runtime_error("bad");
    // ↑ 异常抛出后……
    process();
    m.unlock();                  // 💀 永远不会执行 → 锁泄漏
}                                //   后续所有等 m 的线程死锁

✓ RAII 自动释放

void goodProcess() {
    std::lock_guard lk(m);     // 析构时自动 unlock
    if (cond) throw std::runtime_error("bad");
    // 栈展开 → ~lock_guard() → 锁释放
    process();
}
原则：永远不要直接调用 mutex::lock()/unlock()，永远使用 lock_guard / unique_lock / scoped_lock 这些 RAII 包装。

2.6 跨组件回调中的隐式锁依赖 最难定位

在大型代码库（如本工作区的 PlatformAgent）中，Controller / Facade / Observer / Dispatcher 等组件互相调用回调时， 最容易在持锁状态下回调进对方的另一个加锁路径，形成隐式 AB-BA。

真实案例 1：FailedConfigurationsFacade

来源：src/Controller/FailedConfigurationsFacade.cpp

// 简化后的真实代码片段
class FailedConfigurationsFacade {
    std::mutex mutex_;
    boost::shared_ptr<Mgmt::MoFailed_configurations> mo_;

    void add(const std::string& rulDn) {
        std::lock_guard<std::mutex> lk(mutex_);
        if (mo_) {
            mo_->updateSomething(rulDn);   // 🚨 若该回调反向调用 facade 其他成员函数
        }                                  //    → 自死锁；若它再去抢另一个全局锁 → AB-BA
    }
    void remove(const std::string& rulDn) {
        std::lock_guard<std::mutex> lk(mutex_);
        // ...
    }
};

真实案例 2：Dispatcher 队列

来源：src/PlatformAgent/Common/Dispatcher.hpp

class Dispatcher {
    std::mutex mutex_;
    std::condition_variable cv_;
    std::vector<std::shared_ptr<IScenario>> scenarios_;
    std::thread thread_;

    void run() {
        while (running_) {
            std::shared_ptr<IScenario> sc;
            {
                std::unique_lock lk(mutex_);
                cv_.wait(lk, [&]{ return !scenarios_.empty() || !running_; });
                sc = scenarios_.back();
                scenarios_.pop_back();
            }                          // ✓ 锁在这里释放
            sc->execute();             // 🚨 千万别在持 mutex_ 时执行回调！
        }                              //    若 execute() 内部又调度回 Dispatcher → 死锁
    }
};

真实案例 3：ThreadSafeList 的 forEach

来源：src/PlatformAgent/Common/ThreadSafeList.hpp — 这是工作区里一个正确的范例。

template <typename F>
void forEach(F functor) {
    decltype(data_) temp;
    {
        std::lock_guard lk{mutex_};
        temp.swap(data_);              // ① 持锁仅做指针交换
    }
    for (auto const& item : temp) {
        functor(item);                 // ② 不持锁回调用户 functor
    }                                  //    → 用户 functor 可放心抢任何锁
    {
        std::lock_guard lk{mutex_};
        data_.splice(data_.end(), temp);
    }
}
设计模式：Copy-out / Iterate-without-lock——把数据拷出来再迭代，是消除回调死锁最简单可靠的手段。

pthread_mutex_t 内部结构 (glibc >= 2.30)

struct __pthread_mutex_s {
    int  __lock;        // 0=空闲, 1=锁定, 2+=有等待者 (futex)
    unsigned __count;   // 递归 mutex 的重入计数
    int  __owner;       // 持有线程的 LWP/TID  ← 死锁分析关键
    unsigned __nusers;  // 等待 + 持有的线程总数
    int  __kind;        // 0=NORMAL, 1=RECURSIVE, 2=ERRORCHECK
    ...
};
死锁判据：
① 多个线程都阻塞在 __lll_lock_wait / pthread_cond_wait；
② 没有任何线程在 runnable 状态（CPU 占用 0%）；
③ __owner 字段构成环。

3.3 ThreadSanitizer (TSan) — 主动发现潜在死锁

编译加 -fsanitize=thread -g -O1，TSan 会在运行时记录每个线程的锁获取顺序， 一旦发现两个锁的获取顺序在不同线程中不一致就立即告警——即使本次运行没真死锁。

$ g++ -fsanitize=thread -g -O1 deadlock.cpp -o demo -lpthread
$ ./demo
==================

  Mutex M2 acquired here while holding mutex M1 in thread T1:
    #0 pthread_mutex_lock
    #1 thread1() deadlock.cpp:14:5

4.1 症状识别 (Symptom Triage)

症状含义
进程卡死、命令行无响应主线程也阻塞在 mutex（典型）
top 显示 CPU 0%、内存稳定所有线程都在 sleep 状态 → 强信号
日志输出停止，但未崩溃区别于 crash（segfault 会立即退出）
压力测试偶发不稳定，重启即恢复典型死锁（non-deterministic）
少数线程 100% CPU 而其他卡住更可能是 livelock / 自旋锁，非典型死锁

4.5 生产环境 — Watchdog 自动取证

等死锁发生再人工抢救已经晚了。生产代码必须主动检测：

class Watchdog {
    std::atomic<int64_t> lastTick_{0};
    std::thread monitor_;
public:
    void tick() {                         // 工作线程定期调用
        lastTick_.store(now_ms(), std::memory_order_relaxed);
    }
    void start(int timeoutMs) {
        monitor_ = std::thread([=]{
            while (running_) {
                std::this_thread::sleep_for(1s);
                auto elapsed = now_ms() - lastTick_.load();
                if (elapsed > timeoutMs) {
                    LOG_FATAL("watchdog: hang for " << elapsed << "ms");
                    // 关键：触发 SIGABRT → core dump → 上传到 crash 服务
                    std::abort();
                }
            }
        });
    }
};

5.1 统一锁序 (Lock Hierarchy / Level-based Ordering)

给每个锁分配一个数值“层级”，约定只能从低层级向高层级申请，运行时违反就报错。

class HierarchicalMutex {
    std::mutex internal_;
    const unsigned long hierarchyValue_;
    unsigned long previousHierarchyValue_;
    static thread_local unsigned long thisThreadHierarchyValue_;

    void checkForHierarchyViolation() {
        if (thisThreadHierarchyValue_ <= hierarchyValue_)
            throw std::logic_error("mutex hierarchy violated");
    }
    void updateHierarchyValue() {
        previousHierarchyValue_ = thisThreadHierarchyValue_;
        thisThreadHierarchyValue_ = hierarchyValue_;
    }
public:
    explicit HierarchicalMutex(unsigned long v)
        : hierarchyValue_(v), previousHierarchyValue_(0) {}

    void lock() {
        checkForHierarchyViolation();
        internal_.lock();
        updateHierarchyValue();
    }
    void unlock() {
        thisThreadHierarchyValue_ = previousHierarchyValue_;
        internal_.unlock();
    }
    bool try_lock();
};

// 使用：层级越大越“高”，只允许从高往低锁
HierarchicalMutex high_(10000);
HierarchicalMutex mid_ (6000);
HierarchicalMutex low_ (3000);

// ✓ 正确：10000 → 6000 → 3000
{ std::lock_guard a(high_); std::lock_guard b(mid_); ... }

// ✗ 运行时立即抛异常：3000 之后再要 10000
{ std::lock_guard a(low_);  std::lock_guard b(high_); ... }

5.2 std::lock / std::scoped_lock — 一次性原子获取多锁

// C++11/14: std::lock + adopt
void swap(X& a, X& b) {
    if (&a == &b) return;
    std::lock(a.m_, b.m_);                      // 死锁规避算法
    std::lock_guard<std::mutex> la(a.m_, std::adopt_lock);
    std::lock_guard<std::mutex> lb(b.m_, std::adopt_lock);
    std::swap(a.value_, b.value_);
}

// C++17: scoped_lock 一行搞定
void swap(X& a, X& b) {
    std::scoped_lock lk(a.m_, b.m_);          // ✓ 推荐
    std::swap(a.value_, b.value_);
}
std::lock 工作原理：以“先 lock 一个、其余 try_lock”的方式尝试，任何一个失败就把已持有的全部释放、换另一个起点重试，直到全部成功。

5.3 try_lock + Backoff — 打破“不可抢占”

while (true) {
    std::unique_lock l1(mA, std::defer_lock);
    std::unique_lock l2(mB, std::defer_lock);
    if (l1.try_lock() && l2.try_lock()) {
        // ... 临界区 ...
        break;
    }
    // 任意一把没拿到 → unique_lock 析构会自动释放已拿的
    std::this_thread::sleep_for(std::chrono::microseconds(rand() % 100));
    // ↑ 随机退避，避免活锁 (livelock)
}
注意：朴素 try_lock 循环会引发活锁——两线程同时让步、同时重试。
必须加随机退避 (exponential / jittered backoff)，类似 TCP CSMA-CA。

5.4 缩小临界区 / 拷出后处理

// ✗ 死锁高危：在持锁期间执行回调 / I/O
void notifyAll_BAD() {
    std::lock_guard lk(m_);
    for (auto& obs : observers_)
        obs->onUpdate();           // 💀 obs 内部可能反过来 lock 我们
}

// ✓ 拷出 + 释锁后回调
void notifyAll_GOOD() {
    decltype(observers_) snapshot;
    {
        std::lock_guard lk(m_);
        snapshot = observers_;     // 持锁仅做容器拷贝
    }
    for (auto& obs : snapshot)
        obs->onUpdate();           // 不持锁回调，安全
}

5.5 RAII / 异常安全

  • 所有 mutex 必须用 lock_guard / unique_lock / scoped_lock 包装。
  • 禁止 m.lock(); ... m.unlock(); 这种手写代码——编译期 lint 即可拦截。
  • unique_lock 比 lock_guard 灵活（可 unlock / 移动），但仅在需要时使用，否则用更轻量的 lock_guard。
  • 条件变量必须搭配 unique_lock（cv 需要释锁等待）。

Q1. 死锁的四个必要条件是什么？打破任意一个的具体做法？
  • 互斥：用无锁数据结构（lock-free queue）、不可变对象 (immutable)、RCU；读多写少改用 shared_mutex。
  • 持有并等待：std::lock / std::scoped_lock 一次性原子获取所有锁。
  • 不可抢占：try_lock + 随机退避；timed_mutex::try_lock_for 超时返回。
  • 循环等待：建立全局锁层级 (Lock Hierarchy)，规定从低到高单调申请。
Q2. std::lock 是怎么避免死锁的？

它实现的是“try-and-back-off”算法：先 lock() 第一个 mutex，然后对其余的 try_lock()。 若任何一个 try 失败，就把已经持有的全部 unlock()，等一个非常短的随机时间后重新选起点。 直到一次能把所有 mutex 都获取成功才返回。这相当于“原子获取多锁”的语义，无论多个线程以什么顺序调用 std::lock(mA, mB) 都不会死锁。

Q3. lock_guard 和 unique_lock 有什么区别？什么时候用哪个？
lock_guardunique_lock

默认用 lock_guard；需要手控解锁、配合 cv、或与 std::lock+defer_lock 配合时改用 unique_lock；需要锁多个 mutex 用 scoped_lock。

Q4. std::recursive_mutex 解决什么问题？为什么不推荐？

解决“同一线程在持锁期间需要再次进入加锁路径”的问题（典型如递归算法、回调）。 不推荐的原因：
① 性能比普通 mutex 慢 ~1.5× ~ 2×（要维护 owner 和 count）；
② 掩盖设计问题——通常说明加锁粒度过大、或者公私接口没有分离；
③ 让“到底持锁多少次”变得不可读，不利于审查；
④ 与 condition_variable 配合复杂（必须用 condition_variable_any）。
更好做法：把临界区抽成 fooLocked() 私有版本（无锁），公开 foo() 加锁后调用 fooLocked()。

Q5. 死锁和活锁 (Livelock) / 饥饿 (Starvation) 的区别？
  • 死锁：线程都不动了，CPU 0%，互相等待，永久阻塞。
  • 活锁：线程都在动，CPU 100%，但不断互相让步、永远完不成工作。典型：朴素 try_lock 循环、TCP 电梯效应。解：随机退避。
  • 饥饿：某些线程永远轮不上锁，但其他线程在正常工作。典型：写者在大量读者中拿不到 shared_mutex 的写锁。解：公平队列锁、写者优先策略。
Q6. 在生产环境怎么定位一个线上死锁？
  1. 监控报警：心跳超时 / RPS 跌零；top -H 看线程是否全部 D/S 状态。
  2. 取现场：sudo gcore -o /tmp/core $PID，不杀进程。如果服务无法继续，kill -SIGABRT $PID 触发 systemd-coredump。
  3. 下载符号：把 .so 调试符号、core、二进制本身一起 sync 到分析机。
  4. GDB 分析：thread apply all bt full，p mutex.__data.__owner 画 wait-for 图。
  5. 修复：调整锁序、改 scoped_lock、缩小临界区、或拆模块用消息队列。
  6. 预防：CI 跑 TSan；引入 lock hierarchy；上线前压测。
Q7. 条件变量为什么必须配合谓词使用？什么是虚假唤醒 (Spurious Wakeup)？

虚假唤醒：POSIX 标准规定 pthread_cond_wait 可能在没有任何 signal/broadcast 的情况下返回——这是底层为了实现简单（futex 可能因信号、超时、被错误唤醒）。
因此必须用 while 循环检查条件，而不是 if：

// ✗ 错
if (!ready) cv.wait(lk);

// ✓ 对（手写循环）
while (!ready) cv.wait(lk);

// ✓ 推荐（lambda 谓词版本，标准库内部就是 while）
cv.wait(lk, []{ return ready; });

另外，谓词还能防止 lost signal：如果 producer 在 consumer 进入 wait() 前就 notify_one()，没谓词就会丢失信号；带谓词时 consumer 进 wait 前先检查谓词，已满足就直接跳过等待。

Q8. 怎么在编码阶段就预防死锁？
  • 规范：约定全局锁序文档，锁名带层级前缀（如 l1_config_mu_）。
  • 禁手写：禁用裸 lock()/unlock()，code review 自动拒绝。
  • scoped_lock 优先：但凡 ≥ 2 把锁，强制 std::scoped_lock。
  • 临界区原则：① 不在锁内做 I/O；② 不在锁内回调用户函数；③ 不在锁内发送消息或申请其他锁。
  • 不变量审查：每个共享数据结构标注 “guarded by mutex_xxx”（GCC __attribute__((guarded_by(mu))) 或 Clang Thread Safety Analysis）。
  • 测试：CI 跑 TSan，发版前压测 24h+。
Q9. 什么是 Lock-free / Wait-free / Obstruction-free？
  • Obstruction-free：任意线程在没有竞争时能在有限步内完成。最弱保证，不能保证整体进度。
  • Lock-free：系统整体能在有限步内取得进度，但单个线程可能永远饥饿。例如 CAS 循环算法。
  • Wait-free：每个线程都能在有限步内完成。最强保证，工程实现极难。

从 Lock-free 到死锁免疫并不等同——lock-free 不会死锁，但可能 livelock（无限重试）。

Q10. shared_mutex 为什么容易死锁？

三种典型陷阱：

  1. 读升写死锁：N 个线程持读锁后同时申请写锁——所有人都等其他人释放读锁。
  2. 写者饥饿：读者源源不断进入，写者永远拿不到写锁。需要公平队列锁实现 (gcc 11+ 已部分修复)。
  3. 嵌套读读死锁：某些 shared_mutex 实现不允许同线程递归 lock_shared()，会死锁；规范未严格定义。

建议：除非读写比 > 10:1 且临界区耗时 > 1µs，否则 std::mutex 已足够；性能更易预测。

Q11. Linux 下 mutex 是怎么实现的？为什么死锁时 CPU 占用是 0？

glibc 的 pthread_mutex 基于 futex (fast userspace mutex)：

  • 无竞争时，lock() 仅是一次 cmpxchg（原子指令），完全在用户态完成，开销 ~10ns。
  • 有竞争时，调用 futex(FUTEX_WAIT) 系统调用让线程进入内核睡眠队列。
  • 持有者 unlock() 时调 futex(FUTEX_WAKE) 唤醒一个等待者。

因此死锁时所有线程都阻塞在内核 futex_wait_queue_me，CPU 调度器给它们 0 时间片——表现为 CPU 0%、进程仍存活。这也是为什么 strace -e futex 是诊断利器。

Q12. 用 C++ 实现一个简单的 DeadlockDetector 思路？

核心：维护一张全局 有向图「线程 → 等待的锁 → 锁的持有者 → ...」，每次 lock() 加边、检查是否成环。

class DeadlockDetectingMutex {
    std::mutex internal_;
    std::atomic<std::thread::id> owner_{};
public:
    void lock() {
        auto me = std::this_thread::get_id();
        Graph::registerWait(me, this);    // 加边
        if (Graph::hasCycle())               // DFS / Tarjan
            throw std::system_error(...);
        internal_.lock();
        owner_.store(me);
        Graph::registerHold(me, this);    // 改边
    }
    void unlock() {
        Graph::unregister(std::this_thread::get_id(), this);
        owner_.store({});
        internal_.unlock();
    }
};

真实库：Boost.Thread 的实验性 strict_lock、Abseil 的 Mutex 自带死锁检测、Folly 的 folly::SharedMutex。
