01

Valgrind 是什么 · 一图看懂六工具家族

DBT 框架 + 6 个独立工具 · Memcheck 只是其中一个

很多 C++ 工程师把 "Valgrind" 等同于 "Memcheck"，但 Valgrind 实际是一个 动态二进制插桩 (DBI / DBT) 框架，自带 6 个官方工具。Memcheck 只是其中名气最大的那个。理解这个分层关系，是看懂后面所有原理章节的前提。

1.1 项目历史 · 为什么诞生

2002
Julian Seward 起步
最初叫 Valgrind（北欧神话中通往瓦尔哈拉的大门），目标只有一个：在不重编不修改源码的前提下查 C/C++ 程序的内存错误。这是当时主流（Purify 等）做不到的——它们都需要二进制重写或链接器介入。
2004 · v3.0
VEX IR 引入
把"指令翻译"从 Memcheck 中抽出来，独立成 VEX IR（中间表示）。从此 Valgrind 变成一个框架：核心 (Coregrind) 负责翻译/调度，工具 (tools) 只插桩业务逻辑。这才有了后续的 Helgrind / Massif / Callgrind。
2007 · v3.3
DRD / Massif 加入
DRD（Bart Van Assche）作为 Helgrind 的"科学派"对照实现；Massif 作为堆使用画像工具进入主线。
2012 · 关键拐点
Sanitizer 横空出世
Google 发布 ASan / TSan / MSan —— 编译期插桩，比 Valgrind 快 10×。Valgrind 在"日常 CI"场景被快速边缘化。但Valgrind 没消失，因为它有一个杀手级特性：不需要重编。
至今 · v3.x 维护期
仍在更新 · 由 Mark Wielaard (Red Hat) 等维护
2024 Q4 仍有 v3.24 发布（aarch64 / RISC-V 改进、新内核 syscall 支持）。在第三方闭源 .so / 嵌入式 / 老编译器场景仍是主力。

2.2 VEX IR · 让"加新工具"成本极低的中间表示

VEX 是 Valgrind 自己定义的体系结构无关中间表示。你可以理解成"Valgrind 的 LLVM IR"，但它的设计目标完全不同：不是为了优化，是为了方便插桩。

核心思想：把 x86 的复杂指令拆成一组只读/只写一次的 micro-op，每个 micro-op 都是一个独立 statement。这样 tool 想在"读内存之前插一个 hook"就只要拦截 IR 里的 LoadStmt，不用懂 x86 编码细节。

// x86 原指令（一条复杂指令）
movq  %rax, (%rbx)        ; store rax to memory at rbx

// VEX IR (拆成 3 条 statement)
t1 = GET:I64(rax)         ; 读寄存器
t2 = GET:I64(rbx)         ; 读寄存器
STle(t2) = t1             ; little-endian store

// 插桩后（Memcheck 在 store 前后加 hook）
t1 = GET:I64(rax)
t2 = GET:I64(rbx)
DIRTY helperc_STOREV64(t2, t1)   ; ← Memcheck 插的 hook：标记 t2 处 8 字节为已定义
STle(t2) = t1

VEX IR 的 4 类 Statement

  • IRStmt_Put / IRStmt_Get · 读写寄存器（含 EFLAGS）
  • IRStmt_WrTmp · 给 tmp 赋值（VEX 里寄存器叫 tmp）
  • IRStmt_Store · 写内存（前面例子的 STle）
  • IRStmt_Dirty · 调用 C 写的 helper（这是 tool 主要的插桩点）

体系结构无关的好处

  • 同一份 tool 代码跑在 x86 / ARM / aarch64 / RISC-V / s390x
  • 新加 ISA 只要写"前端译码 (frontend)" + "后端反编译 (backend)"
  • Memcheck / Helgrind 等工具完全不知道底下是什么 CPU
  • 对比 ASan：每加一个新架构都要重写 instrumentation pass
VEX 命名由来：来自 "Verifiable EXecutable"（最早的设计目标含验证），但后续演化成纯翻译框架。Julian Seward 在 2004 论文 "Valgrind: A Framework for Heavyweight Dynamic Binary Instrumentation" 里给出了完整设计动机。

2.3 Coregrind 架构 · 工具是"插件"

Coregrind 是 Valgrind 的核心。它给每个工具暴露一组 VG_(track_*) 回调，工具只要"按需注册感兴趣的事件"。这是整个项目能扩展出 6 个工具的根本原因。

图 2.2 · Coregrind 架构。核心子系统 5 大块 + 插件层。每个 tool 是个独立的 .so，通过 VG_(track_new_mem_heap) / VG_(track_die_mem_heap) / VG_(track_post_mem_write) 等 ~30 个回调与核心通信。
// 一个最小的 tool（Lackey 简化版）骨架：
// 完整代码在 valgrind 源码 lackey/lk_main.c

static void my_post_mem_write(CorePart part, ThreadId tid,
                               Addr a, SizeT len) {
    // 每次内存被写入都会调到这里
    // Memcheck 在这里把 V-bit 标记为"已定义"
    VG_(printf)("write %lu bytes at %p\n", len, (void*)a);
}

static void my_pre_clo_init(void) {
    VG_(track_post_mem_write)(my_post_mem_write);
    // 只关心写内存？只注册这一个回调，剩下全 0 开销。
}
VG_DETERMINE_INTERFACE_VERSION(my_pre_clo_init)
关键设计哲学：tool 写起来"像写普通 C"，看起来在调用回调。但实际上：tool 的代码会被静态链到 Coregrind 二进制（不是动态加载），运行时合成 CPU 上跑客户机代码 + tool callback。这是为什么 Valgrind 不需要 host 系统装任何 tool runtime。

2.4 Shadow memory · A-bit + V-bit 双层影子

Memcheck 能精确报"读未初始化内存"到位级精度，全靠 shadow memory。它的设计是 Valgrind 最经典的算法贡献之一。

图 2.3 · Memcheck 双层影子内存。1 字节用户内存 ↔ 1 个 A-bit + 8 个 V-bits。这是为什么 Memcheck 内存开销 ~ 2-3×（每用户字节多用 9 位）。Shadow 自己在 Coregrind 私有堆里，二级页表式管理。
// 一个经典的"V-bit 位级追踪"例子：
// 读 char 时，如果只赋值了 high nibble，会精确报"低 4 位 undef"
struct Bits { unsigned hi:4; unsigned lo:4; };
Bits b;
b.hi = 3;            // 写：V-bits = 1111_0000  (低 4 位仍 undef)
int x = b.lo;       // 读：低 4 位 undef → Memcheck 报 "use of uninitialized value"
对比 ASan：ASan 的 shadow 是 1 字节 shadow / 8 字节用户内存（粒度=8B），不能查未初始化，只查"是否 addressable + 是否 redzone/quarantine"。这就是为什么"Memcheck 能查 uninit、ASan 不能"——shadow 模型粒度不同。MSan 才补回了 V-bit。

2.5 syscall 拦截 · 客户机如何与内核对话

客户机程序调用 read(fd, buf, n) 时，不能直接执行 syscall —— 因为内核会把数据写入 buf，但 Memcheck 必须知道这件事，才能把那 n 个字节的 V-bit 标为 "defined"。Coregrind 的解决办法：所有 syscall 走它的 wrapper。

// 客户机原代码：
int n = read(fd, buf, 100);

// 翻译后实际跑的伪代码（Coregrind 包装）：
// 1. 进入 wrapper
VG_(client_syscall)(SYS_read, fd, buf, 100);
   // 1a. PRE_HOOK: 通知 Memcheck "buf 即将被写入 100 字节"
   VG_(track_pre_mem_write)(buf, 100);
   // 1b. 真正发 syscall（host 系统的 syscall）
   ssize_t r = real_syscall(SYS_read, fd, buf, 100);
   // 1c. POST_HOOK: 通知 Memcheck "实际写了 r 字节，把 V-bit 标 defined"
   VG_(track_post_mem_write)(buf, r);

每个 syscall 都有 PRE / POST 两个 hook

Coregrind 维护一个 SyscallInfo 表，每个 syscall 有专门的 wrapper。比如：

  • read → POST 后标 buf 为 defined
  • write → PRE 前检查 buf 全部 defined，否则报"writing uninit data"
  • mmap → 把新映射区标 A=1, V=defined
  • munmap → 把对应区域标 A=0
  • brk → 调整 heap 边界

意外好处：Memcheck 知道你在"写未初始化数据到 socket"

这是 Memcheck 比 ASan 更全面的一点：

struct Header h;
h.magic = 0xCAFE;       // 只填 magic
write(sock, &h, sizeof h);
// → Memcheck: "syscall write writing uninit byte at offset 4"
//   信息泄露漏洞预警

2.6 为什么慢 10-30× · 一张性能 breakdown 图

这是面试高频题。答案不是简单"动态翻译"四个字，而是多个开销叠加。

图 2.4 · Memcheck 性能开销分解。结论：3 + 4 项是 dominate 因素（V-bit 检查 + shadow 访问），加起来 ~ 12-15×。这也意味着关掉 origin tracking、减小 --num-callers、用 None tool 来跑都能加速 30%+。
L5 MID "动态翻译"具体翻什么？为什么不像 QEMU 那么慢？
答案
QEMU 是跨架构翻译（x86 → ARM 等），需要语义对齐每条指令；Valgrind 是同架构翻译（x86 → x86），只是中间套一层 IR 让 tool 插桩。所以 QEMU 慢 50-100×，Valgrind 只慢 4-5×（不带 tool）。Memcheck 多出来的减速来自插桩本身。
深挖
Valgrind 与 DynamoRIO / Pin 的对比：DynamoRIO 是 Hewlett-Packard 的产品，在线生成 trace 比 Valgrind 快（~ 2× native + tool overhead）；但 DynamoRIO 没有 IR 抽象，写新 tool 比 Valgrind 难。Valgrind 选了"易扩展"的工程取舍。

3.2 完整命令行 · 黄金参数集

# 标准排查内存错误的"黄金组合"
valgrind \
  --tool=memcheck \                # 默认工具，省略也行
  --leak-check=full \              # 详细列出每个 leak（不写就只给 summary）
  --show-leak-kinds=all \          # 4 种 leak 都报：definitely / indirectly / possibly / reachable
  --track-origins=yes \            # 追溯 uninit value 的来源（多花 50% 时间但值得）
  --num-callers=30 \             # 栈最多 30 帧（默认 12，不够）
  --keep-stacktraces=alloc-and-free \  # UAF 报告同时显示分配栈+释放栈
  --read-var-info=yes \            # 读 DWARF 变量名，报告里显示真实变量名
  --error-exitcode=1 \           # 有错就 exit 1（CI 必备）
  --xml=yes --xml-file=vg.xml \    # 输出 XML 给 CI parser
  ./your-program arg1 arg2

# 调试期可加上：
  --vgdb=yes --vgdb-error=0 \    # 第一次出错就停下来等 GDB（vgdb 联调）
  --suppressions=valgrind.supp \   # 抑制规则文件
  --gen-suppressions=all           # 把每个错误的 suppression 模板打印出来（写规则用）

3.3 跑一个 buggy 程序 · 报告解读全套

下面这个 50 行程序故意塞了 6 类错误。这是教学界常用的 Memcheck 教学例。逐个看 Memcheck 怎么报。

// buggy.c
// 编译: gcc -g -O0 buggy.c -o buggy
// 跑:   valgrind --leak-check=full --track-origins=yes --num-callers=20 ./buggy

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int* leak_func() {
    int* p = malloc(sizeof(int) * 10);    // ① 没 free → leak
    p[0] = 42;
    return p;
}

int main() {
    int x;
    printf("%d\n", x);              // ② uninit read

    int* arr = malloc(40);
    arr[10] = 5;                       // ③ heap OOB write (索引到 #10，但只有 10 个 int)
    int y = arr[11];                  // ④ heap OOB read

    free(arr);
    arr[0] = 7;                        // ⑤ UAF
    free(arr);                          // ⑥ double free

    int* leaked = leak_func();        // 调用泄漏函数（① 计入这里）
    return y + leaked[0];
}

报告字段速查

  • ==PID== · 输出行前缀，方便从 stdout 里 grep 出来
  • at 0x... · 出错指令地址 + 函数名 + 文件:行号
  • by 0x... · 调用栈上一帧（多帧组成栈，从顶到底）
  • Address 0x... is N bytes after/inside a block · 关键诊断信息
  • alloc'd at / free'd at · 谁分的、谁释的（要 --keep-stacktraces）

常见误读

  • "0 bytes after a block" 字面看像没问题，实际意思是"紧跟在合法块的下一字节" → 越界 1 字节
  • "still reachable" 不一定是泄漏 —— 进程退出时 OS 反正会回收。Google abseil 故意这么用（leaky singleton）
  • uninit read 没显示 origin → 说明没开 --track-origins=yes
  • 报"at ???"无符号 → 没编 -g 或 strip 了

图 3.2 · A/V-bit 状态机。这是 Memcheck 的"心智模型"：3 个状态、~ 6 种转换。Coregrind 给每个事件钩了对应回调，Memcheck 在回调里改 shadow。

"未定义传染"机制 · 为什么 --track-origins 这么有用

Memcheck 不仅追踪"该位是否未定义"，还追踪"这个未定义来自哪里"（开 --track-origins=yes 后）。原理：每 4 字节 V-bits 旁边再附一个 origin tag（32-bit），指向"分配栈"。值在赋值/算术运算时跟着流动。

int f() {
    int x;          // 行号 2: x 的 V-bits=undef, origin=alloc-site
    return x + 1;   // 算术：result 仍 undef，origin 跟着传
}
int main() {
    int y = f();    // y = undef，origin 仍指向 f() 行 2
    printf("%d", y);// 报"uninit"，origin 显示行号 2 ✓
}
性能代价：origin tracking 让 Memcheck 慢 ~ 50%（多写 4 字节 / 4 用户字节）。建议只在调试时开，CI 默认不开。报告里没"Uninitialised value was created by..."这行，就是没开。

Helgrind & DRD · 数据竞争与死锁检测

happens-before 关系 · lockset 算法 · vector clock · 与 TSan 的对比

Helgrind 与 DRD 是两个独立的多线程错误检测工具。两者目标重合 80%，但用了不同算法 —— Helgrind 偏 lockset + happens-before 混合，DRD 是纯 vector clock。理解差异之后，就知道"为什么有时一个工具报、另一个不报"。

4.1 Helgrind · lockset + happens-before 混合算法

Helgrind 用两套互补的算法。lockset 主要查"两个线程访问共享变量但没共同的锁"；happens-before 主要补 lockset 漏报（条件变量、原子操作等同步原语建立的"非锁"序）。

图 4.1 · Helgrind 的 lockset 算法。对每个共享地址，记录"访问它时持有的锁集合 (LS)"。两次访问的 LS 交集为空 → 报 race。简单暴力但可能误报（用 atomic / barrier 同步时漏算了 happens-before）。

Helgrind 还会查的另外两类问题

① pthread API 误用

  • unlock 一个没 lock 的 mutex
  • destroy 一个仍被持有的 mutex
  • signal 一个没 wait 的 cond
  • 对 PTHREAD_MUTEX_DEFAULT 做递归 lock

本质：基于 pthread API 状态机做不变式检查

② 锁序反转 (lock order inversion)

  • 线程 A：lock(M1) → lock(M2)
  • 线程 B：lock(M2) → lock(M1)
  • → 即使当前没死锁，Helgrind 也报 "lock order violated"

原理：维护全局 lock graph，检测 cycle

③ data race

  • 就是图 4.1 那种情况
  • 对每个内存位置维护"上次访问的 LS + tid"
  • 新访问 LS 与旧 LS 交集 = ∅ → race

最常用，最贵（每个 load/store 都查）

4.2 DRD · 纯 vector clock 实现

DRD（Bart Van Assche 维护）是 Helgrind 的"科学派"对照实现。它完全用 vector clock建立 happens-before 关系，不用 lockset。优势：condition_variable / barrier / atomic / pthread_join 等同步语义建立的 hb 关系处理得更精准。

// DRD 的核心数据结构（伪代码）：
// 每个线程维护一个 N-元 vector clock
vector_clock { int ts[N]; };  // N = 线程数

// 每个内存位置维护"上次读 / 上次写"的 vector clock
struct MemAccess {
    vector_clock last_read;
    vector_clock last_write;
    int last_writer_tid;
};

// 同步事件 (lock release, cv signal, atomic store-release):
//   把"自己的 vc"传递给 acquirer，acquirer 取每个槽位 max

// 数据访问 (load/store):
//   if (!happens_before(prev_access, my_vc)) → 报 race
bool happens_before(vc a, vc b) {
    for (int i = 0; i < N; i++)
        if (a.ts[i] > b.ts[i]) return false;
    return true;
}
vector clock 的精彩之处：它能识别"任意同步原语建立的序"，不局限于锁。例：用 atomic<int>.store(release) + atomic<int>.load(acquire) 同步两个线程时，DRD 能正确识别为已同步、不报 race；Helgrind 的 lockset 在这里漏报或误报。

Massif · 堆使用画像

不是查泄漏 · 查"什么时候、谁、占了多少堆" · ms_print 输出解读

Massif 容易被误解成"另一个查泄漏的工具"。其实它的目标完全不同：给你一张"程序运行过程中堆使用的时间序列图"。每隔一定间隔采一次堆快照，记录"现在共占多少堆 + top 10 调用栈各占多少"。在 heaptrack 出现之前，它是 Linux 上唯一能做这件事的工具。

5.1 用法 · 一条命令两段输出

# 跑：
valgrind --tool=massif \
         --time-unit=B \              # 时间轴 = bytes-allocated（最直观）
         --detailed-freq=1 \         # 每个采样点都记详细栈
         --max-snapshots=100 \       # 最多 100 个采样点（默认 100，太密 → 调小）
         --threshold=1.0 \           # 调用栈贡献 < 1% 的不展开
         ./your-program

# 输出：massif.out.<PID>

# 文本可视化：
ms_print massif.out.12345

# 图形可视化：
massif-visualizer massif.out.12345     # Qt GUI

关键开关详解

选项作用典型值
--time-unit=时间轴单位B bytes alloc / i instructions / ms 真实时间
--pages-as-heap=yes把 mmap 也算进堆查"jemalloc/tcmalloc 内部用的 mmap"必备
--stacks=yes把栈也算进画像查"递归吃光栈"
--detailed-freq=N每 N 个采样点记一次详细栈1 (全记) ~ 10 (省内存)
--threshold=X调用栈贡献 < X% 不展开1.0 (默认) ~ 0.1 (更细)
--peak-inaccuracy=X峰值精度1.0 (默认)，CI 可设 0.5

选型口诀：① 线下 + 一次性看堆形态 → Massif（5 分钟出图）；② 线下需要 diff → heaptrack；③ 线上 / 长跑 / 大流量 → jeprof（如果用了 jemalloc）。Massif 已经被 heaptrack 在大多数场景替代，但不能装第三方工具的环境（嵌入式、隔离网）Massif 仍然有用——它已经在 Valgrind 里了。

Callgrind & Cachegrind · 调用图与 cache 模拟

指令计数 profiler · L1/LL cache 模拟 · 与 perf/vtune 的本质差异

这两个是 Valgrind 家族里的"性能分析工具"。它们给的不是 wall-clock 时间，而是指令数 + cache miss 数 + 分支错失数。优势：结果完全可重现（不像 perf 受其他进程干扰）。劣势：不算时间（一条 LOAD 在 cache 命中时 1 cycle，miss 时 200 cycles，但 Cachegrind 都算 1 inst）。

6.1 Callgrind · 调用图 + 指令计数

# 跑：
valgrind --tool=callgrind \
         --dump-instr=yes \         # 指令级精度（不只是行级）
         --collect-jumps=yes \      # 收集分支信息
         --cache-sim=yes \          # 同时启用 cache 模拟（融合 Cachegrind）
         --branch-sim=yes \         # 分支预测模拟
         ./prog

# 输出：callgrind.out.<PID>

# 文本可视化：
callgrind_annotate --auto=yes callgrind.out.12345 | less

# 图形可视化（强烈推荐）：
kcachegrind callgrind.out.12345
图 6.1 · Callgrind 调用图。每个节点：函数名 + 累计指令占比 + cache miss 率。从 main 一路展开，找 hot 节点。kcachegrind GUI 还能 zoom in 到指令级，看 inner loop 的 cycle 分布。
关键陷阱：Callgrind 的 "cycle" 不是 wall-clock cycle。它假设每条 inst = 1 cycle（除非开 --cache-sim 才把 miss penalty 算进去）。真要看墙钟时间，用 perf record；Callgrind 的优势是"逻辑层"分析 + 结果可重现。

6.2 Cachegrind · 软件模拟 cache 行为

Cachegrind 在 合成 CPU 上模拟一个 cache 模型：L1i / L1d / LL（last-level，通常对应 L2 或 L3）。每次 load/store 都算"命中还是 miss"，最后报告每行代码的 miss 数。

valgrind --tool=cachegrind \
         --I1=32768,8,64 \         # L1i: 32KB, 8-way, 64B line
         --D1=32768,8,64 \         # L1d: 32KB, 8-way, 64B line
         --LL=8388608,16,64 \      # LL: 8MB, 16-way, 64B line
         ./prog

# 默认：自动检测当前 CPU 的 cache config（cpuid）
# 但你可以传任意 config 来"模拟另一个 CPU"

cg_annotate --auto=yes cachegrind.out.12345 > report.txt
// 报告样例（节选）：
                Ir         I1mr      ILmr           Dr        D1mr       DLmr           Dw        D1mw       DLmw

  10,000,000            0         0    5,000,000           0          0    5,000,000           0          0  // vector::push_back 内联展开
   2,500,000           12         0    2,000,000     312,500     125,000      500,000     125,000     100,000  // hot loop 行 ⚠

// 列含义：
//   Ir   = instructions retired
//   I1mr = L1i miss
//   ILmr = LL miss (instructions)
//   Dr   = data reads
//   D1mr = L1d miss (read)
//   DLmr = LL miss (read)
//   Dw   = data writes
//   D1mw = L1d miss (write)
//   DLmw = LL miss (write)

Cachegrind 的局限

  • 只有 3 级 cache，不模 TLB
  • 不模 prefetcher（现代 CPU 有 4 个 prefetch unit）
  • 不模 store buffer / write combining
  • 分支预测模型简化（仅 2-bit saturating counter）
  • 不算 SMT / hyperthreading 干扰

但它仍然有用

  • 对比两个版本的相对 miss 数（重构验证）
  • 结果稳定可重现（适合 CI 性能 baseline）
  • 不需要 PMU 权限（Docker / Cloud VM 友好）
  • 支持模拟"另一个 CPU 的 cache"做选型预估

6.3 vs perf / vtune · 决策表

场景推荐工具理由

常被忽略的用法：很多 C++ 库的 micro-benchmark 报告会用 Cachegrind 给出"L1d miss 减少 N%"的结论。这是比 perf 更可信的指标，因为 perf 在 noisy 机器上误差 ±10%，Cachegrind 是 0 误差（同一份代码同一份数据 miss 数永远一样）。

"会跑命令"只是入门。把 Valgrind 用到工程级别，必须掌握 3 个高级技能：① 用 suppression 文件把已知误报压下去（否则 CI 会被淹）；② 用 Client Requests 宏让被测代码主动告诉 Valgrind "这里别管"或"这里检查一下"；③ 用 vgdb 在 Valgrind 内挂 GDB 联调（出错点的现场检查）。

7.1 Suppression · 误报治理工程化

为什么需要 suppression

在真实项目里，跑 Memcheck 经常会看到几十上百条来自系统库 / 第三方 lib / 编译器优化的"看似 bug 但其实是误报"的报告。比如：

  • glibc 的 __GI___libc_init_first 在内部用了 unitialized memory 但行为安全
  • libstdc++ 的 std::deque 内部 union 触发 "possibly lost"
  • 动态加载器 ld.so 在第一次解析 PLT 时的"reachable" 块
  • C++ 静态初始化（ICU、abseil、protobuf）的 leaky singleton

不压下去 → CI 报错率 100% → 团队学会忽略报告 → 真 bug 也被忽略。这是 suppression 治理的真实痛点。

suppression 文件格式

// project.supp
{
   deque-internal-union            // suppression 名（自起，便于追溯）
   Memcheck:Cond                    // 错误类型（必须）
   // 调用栈匹配规则（从顶到底）：
   fun:_M_initialize_map               // fun: 函数名
   fun:std::deque*::*deque*            // 通配符 *
   // ...
   obj:/usr/lib/x86_64-linux-gnu/libstdc++.so.6  // obj: 共享库路径
   ...
}

{
   leaky-abseil-singleton
   Memcheck:Leak
   match-leak-kinds: reachable        // 仅匹配 still reachable 类
   fun:malloc
   fun:absl::base_internal::*
   ...
}

工程化治理建议

✓ 推荐做法

  • 项目根放 valgrind.supp（业务自己的）+ third-party.supp（库的）
  • 每条 suppression 都注释来源（哪个版本的库、为什么是误报）
  • 定期 review（季度）：库升级后旧 suppression 可能失效
  • CI 用 --suppressions=project.supp --error-exitcode=1
  • 同时保留 --gen-suppressions=all 输出，PR diff 时检查是否新增

✗ 反模式

  • 用大通配符 ... 配整段栈（容易把真 bug 也吃掉）
  • 所有 suppression 名字都叫 fix-later（永远 fix 不了）
  • 把项目自己代码的 leak 也加 suppression（应该修 bug）
  • 不写注释（半年后没人记得为什么压下去）
  • 把 suppression 写在源码里 而不是单独文件（不能版本管理）

7.2 Client Requests · 让被测程序"主动通信"

Client Requests 是一组宏（在 <valgrind/valgrind.h> / <valgrind/memcheck.h> / <valgrind/helgrind.h>），允许被测程序主动告诉 Valgrind "这块内存的状态"。原理：宏展开成一段特殊的 x86 指令序列（"魔法指令"），native CPU 执行就是 NOP，但在 Valgrind 翻译时被识别为客户机请求。

#include <valgrind/memcheck.h>

// === 自定义 allocator 的 5 个核心宏 ===

// 告诉 Memcheck "这是个新分配的块"
VALGRIND_MALLOCLIKE_BLOCK(addr, size, redzone_size, is_zeroed);

// 告诉 Memcheck "这块被释放了"
VALGRIND_FREELIKE_BLOCK(addr, redzone_size);

// 强制把范围标 NoAccess（自定义 allocator 的空闲块）
VALGRIND_MAKE_MEM_NOACCESS(addr, size);

// 强制标 Defined（人为消除"误报"）
VALGRIND_MAKE_MEM_DEFINED(addr, size);

// 检查范围内全部已定义，否则报错
VALGRIND_CHECK_MEM_IS_DEFINED(addr, size);

// 检查 Valgrind 是否在跑（用于条件编译热路径）
int in_vg = RUNNING_ON_VALGRIND;

典型用例 1 · 自定义 pool allocator

// 自己写的 slab allocator：从一个大 mmap 块里切小块
class Pool {
    void* arena;          // mmap 一大块
    size_t size;
public:
    Pool(size_t n) {
        arena = mmap(...);
        // 默认 mmap 后是 Defined。我们想让 Memcheck 把整块标 NoAccess，
        // 这样未通过 alloc() 拿到的访问会被 Memcheck 抓到。
        VALGRIND_MAKE_MEM_NOACCESS(arena, n);
    }

    void* alloc(size_t sz) {
        void* p = pick_free_slot(sz);
        VALGRIND_MEMPOOL_ALLOC(arena, p, sz);  // 标这块 Undefined（已分配但未写）
        return p;
    }

    void dealloc(void* p) {
        VALGRIND_MEMPOOL_FREE(arena, p);        // 标这块 NoAccess
        return_to_freelist(p);
    }
};

// 这样 Memcheck 就能查出"用户在 free 后还在访问 pool 里的块"。
// 没有这些宏的话，Memcheck 看到整块 mmap 都是 Defined，啥都查不到。

典型用例 2 · 让 Helgrind 知道"自己实现的同步原语"

#include <valgrind/helgrind.h>

// 自己写的 spinlock，Helgrind 不认识它，会误报 race。
void my_lock(my_lock_t* l) {
    while (atomic_test_and_set(&l->flag));
    ANNOTATE_RWLOCK_ACQUIRED(l, 1);   // 告诉 Helgrind "这是 acquire"
}
void my_unlock(my_lock_t* l) {
    ANNOTATE_RWLOCK_RELEASED(l, 1);   // 告诉 Helgrind "这是 release"
    atomic_clear(&l->flag);
}

// 现在 Helgrind 能正确建立 happens-before 关系，不再误报。
性能成本：在 native 跑时，所有 CLIENT REQUEST 宏都是几条 NOP-like 指令，< 1ns 开销。在 Valgrind 跑时才被识别。所以可以放心地留在生产代码里，不需要 #ifdef。

7.3 vgdb · 在 Valgrind 中用 GDB

跑 Memcheck 时遇到错误，往往想"停下来检查现场变量"。vgdb 让这件事变成可能：Valgrind 内置 gdbserver，让外部 GDB 连进去。

# Terminal 1: 跑被测程序，让它在第一个错误时停下
valgrind --vgdb=yes --vgdb-error=0 ./prog
# 输出："TO CONTROL THIS PROCESS USING vgdb (which you probably
#       don't want to do, unless you know exactly what you're doing,
#       or are doing some strange experiment):"
#       "  /usr/lib/.../bin/vgdb --pid=12345 ..."

# Terminal 2: 启 GDB，连过去
gdb ./prog
(gdb) target remote | vgdb --pid=12345
(gdb) backtrace          # 看现场调用栈
(gdb) print *some_var     # 看变量值
(gdb) monitor leak_check full   # 触发 Memcheck 的 leak check（不退出程序）
(gdb) monitor get_vbits 0xADDR  # 看某地址的 V-bits
(gdb) continue           # 继续跑到下一个错误

vgdb 的 monitor 命令（Memcheck 专属）

命令作用
monitor leak_check full立即跑一次 leak check，不结束程序
monitor block_list列出当前所有未释放的 heap 块
monitor who_points_at 0xADDR找谁指向给定地址（查谁泄漏）
monitor get_vbits 0xADDR len看某段内存的 V-bits
monitor make_memory defined 0xADDR len动态把某段标 Defined
monitor xtmemory dump.kcg导出当前的"全部分配栈"给 kcachegrind
"长跑泄漏 attach 排查"实战：跑了几小时怀疑泄漏 → 不想重启 → 在 GDB 里 monitor leak_check full 立即看 → monitor block_list 看哪些没释放 → 选可疑块 monitor who_points_at 找引用源。这个工作流是 Valgrind 在生产排查里少有的几个还能用的场景。

性能优化与生产权衡

让它跑快一点 · 何时绝对不能用 · 替代方案矩阵

在第 2 章我们看了"为什么慢"，这一章看"能不能让它没那么慢"。结论：通过参数调优可以让 Memcheck 减速从 25× 降到 ~ 8×（约 3× 提速）。但即便这样，也不可能在生产环境跑。所以最重要的还是知道什么时候不该用 Valgrind。

8.1 加速技巧 · 把 25× 降到 8×

开关作用提速代价
--track-origins=no关掉 uninit value 来源追溯~ 1.5×报告里没"created by"行
--num-callers=8栈帧数从 30 降到 8~ 1.2×调用栈短，定位难
--read-var-info=no关掉 DWARF 变量名解析~ 1.1×报告显示寄存器名而非变量名
--keep-stacktraces=alloc不存 free 栈（只 alloc）~ 1.15×UAF 报告没释放栈
--free-fill=no不填充 free 后的内存~ 1.1×UAF 报告 value 不易识别
--vex-iropt-level=0关掉 IR 优化变慢不要用
--smc-check=stack只检测栈上自修改代码~ 1.05×JIT 代码可能漏检
--errors-for-leak-kinds=definite只 fail definitely lost—仅 CI 决策用，不影响速度
 
# 极速模式（损失精度换速度）：
valgrind --tool=memcheck \
         --track-origins=no \
         --num-callers=8 \
         --read-var-info=no \
         --keep-stacktraces=alloc \
         --free-fill=no \
         ./prog
# 减速 ~ 8× （vs 全开 ~ 25×）

# 完整精度模式（CI 默认）：
valgrind --leak-check=full \
         --show-leak-kinds=all \
         --track-origins=yes \
         --num-callers=30 \
         --keep-stacktraces=alloc-and-free \
         ./prog
# 减速 ~ 25-30×
实战平衡点：CI 跑全套用完整精度（值得多花时间）；本地 reproduce 排查用"中间档"——开 origins 但 num-callers=15。极速模式只在"测试本身就跑很久 + bug 已经定位到一段代码"时用。

8.2 为什么不能用 Valgrind 跑生产 · 6 个理由

① 减速 10-30×

100 QPS 服务变 4 QPS。SLA 必崩，超时全部命中，调用方反复重试压死服务。

② 内存涨 2-3×

shadow memory 自己要内存。原本 4GB 服务变 12GB，cgroup 直接 OOM-kill。

③ 改时序行为

原本会触发的 race condition 在 Valgrind 下时序变了不复现。反过来 Valgrind 内部串行化又可能制造原本不存在的 bug。

④ JIT / self-modifying code

JVM / V8 / .NET 等 JIT 引擎反复改自己生成的代码。Valgrind 要不停清 translation cache，性能再差 3-5×，且容易出错。

⑤ syscall 兼容性

新 syscall 进内核 (io_uring / cachestat / pidfd_send_signal) 后 Valgrind 要等几个版本才支持。生产用了新 API 直接报"Unhandled syscall"罢工。

⑥ 不能 attach

必须从一开始就在 Valgrind 下启动。已经在跑的进程不能 attach。生产环境很少能容忍重启。

9.2 真实事故 5 例 · Valgrind 是唯一答案的故事

Case 1 · 第三方闭源 SDK 导致段错误

L7 · 4G/5G 设备 · 2022
现象
商用基站设备集成某 vendor 的 RAN 协议栈 SDK（动态库形式，无源码）。客户现场偶发 SIGSEGV，平均一周一次，core dump 显示 PC 在 libran_pdcp.so 内部，地址 0xFFFFFFFFDEADBEEF（典型 UAF 标志）。
尝试
① ASan 无效（不能重编 vendor 的库，混编后 missed shadow 报满屏 false positive 无法用）；② core dump 看不出来源；③ 静态分析工具看不到 .so 内部。
突破
复现环境跑 valgrind --leak-check=full --keep-stacktraces=alloc-and-free ./bs_emu。Memcheck 在 SIGSEGV 之前就报"Invalid read of size 8 at libran_pdcp.so + 0x1234, allocated by 0x... freed by 0x..."。带着这份报告去找 vendor，对方一周后承认是 SDK 在某种重配场景下没正确处理 PDU 引用计数，发了 patch。
教训
对第三方闭源库，Valgrind 是唯一能给出可读报告的工具——它不需要 vendor 重编。这种事故里 Valgrind 不是"调试工具"，是"和 vendor 撕扯的证据"。

Case 2 · 嵌入式 ARMv7 上 ASan 不可用

L6 · 车载娱乐系统 · 2021
现象
车机 HMI 在长跑（48 小时回归测试）后偶发 freeze。日志显示 dbus 进程消失，怀疑 OOM。RAM 总共 1GB，dbus RSS 上涨到 200MB 不正常。
尝试
① ASan 在该 GCC 4.7 + 定制 musl 上构建失败（缺少 ASan runtime）；② 升级编译器风险太大（已 SIL2 认证）；③ heaptrack 不在 yocto BSP 里，拉进来需要 vendor 审批两个月。
突破
Valgrind 是 yocto 里默认就有的（valgrind recipe）。Cross-compile 后 push 到目标板，valgrind --tool=memcheck --leak-check=full --soname-synonyms=somalloc=*musl* ./dbus-daemon。20 分钟跑出报告：dbus 内部一个用户回调注册时分配了 256B，没在 unregister 路径释放。
教训
嵌入式上 "工具能不能用"比"工具好不好"更重要。Valgrind 在 yocto/buildroot 里覆盖率最高（甚至 ARMv5 都有），ASan 在很多嵌入式 toolchain 还跑不起来。

Case 3 · MSan 报不出来的 syscall 信息泄漏

L7 · 安全审计 · 2023
现象
安全审计工具抓到某 RPC 服务的 binary 协议在某些请求下回包泄露 stack 内容（包尾随机 8 字节看起来像 RIP / RBP 寄存器值）。
尝试
① MSan 跑了一遍 → 没报（因为 MSan 不监控 socket write 时的 buffer 是否全 defined）；② code review 没看出来（结构体 padding 看起来合理）。
突破
Valgrind Memcheck 跑同一份代码 → 立即报"Syscall param write(buf) points to uninitialised byte(s) at write(...) → encode_response(...) → handle_request(...)"。原因：协议结构体填字段时漏填了一个 4-byte padding，整个结构体直接 write 到 socket → 4 字节 stack 数据外泄。
教训
Memcheck 的 "syscall 参数检查" 是独有能力。MSan 监控的是用户代码 read uninit；Memcheck 还会监控 syscall 把 uninit data 写出去——后者是信息泄漏漏洞的常见根因。

Case 4 · cache 友好重构需要"可重现"的指标

L6 · 高频交易 · 2024
现象
order book 数据结构从 std::map 改成 cache-friendly 的 SoA 布局，PR 评审需要客观证据"L1d miss 真的少了"。
尝试
① perf stat 在共享 CI 机器上反复跑，每次结果差 ±15%（有其他作业共用机器）；② 隔离机器排队 2 周；③ 用同事桌面机做 baseline → 不可比。
突破
valgrind --tool=cachegrind --cache-sim=yes 跑两个版本。结果显示：旧版 D1 miss = 12.3M，新版 = 4.1M（-66.7%）。同一机器跑 5 次每次结果一模一样。直接放进 PR description，秒批。
教训
"可重现的相对指标"在工程评审里比"绝对的真实数字"更值钱。Cachegrind 的纯软件模拟正好满足这个需求。

Case 5 · ASan 漏报、Helgrind 抓到的 lock-free bug

L7 · 分布式存储 · 2023
现象
自研 lock-free queue 在压测下偶发崩溃，core dump 显示 free list 链表破坏。怀疑 ABA 问题但 reproduce 困难。
尝试
① TSan 跑 → 0 race 报告（TSan 信任程序员标注的 atomic）；② code review 反复看没问题；③ stress test 一周不复现。
突破
valgrind --tool=helgrind --history-level=full ./stress。Helgrind 比 TSan 更"偏执"，对 lock-free 算法默认假设没同步。报了一段：read of head 在 push() 中和 write of head 在 pop() 中竞争。仔细看代码：作者用的 memory_order_relaxed 在某分支应该是 acquire。改了之后 stress 跑 24 小时无问题。
教训
TSan 信任程序员的 atomic memory order；Helgrind 不信任。对 lock-free / 自定义同步原语，Helgrind 的"偏执"反而是优点——它会逼你证明每一处同步都对。
5 例的共同特征：① 都是主线工具失败后才用 Valgrind；② Valgrind 提供的不是"快速"，是"独有能力"——闭源、嵌入式、syscall 检查、可重现 cache、偏执 race 检查。理解这一点，就理解了 Valgrind 在 2026 年仍然不可替代的真正原因。

大厂 C++ / 系统 / SRE 面试里 Valgrind 是高频考点（尤其问到"为什么慢"的细节）。这一章把题分成 4 个层级，每题给"答案 + 深挖"。如果 80% 能答出来，Tech Lead 面试的"调试与工具链"部分基本稳了。

10.1 L4 · 初级（4 题）"会用、知道是什么"

L4 INIT Q1. 一句话讲 Valgrind 是什么、最常用是哪个工具？
答案
Valgrind 是动态二进制翻译框架（DBT），自带 6 个工具，不需要重编。最常用的是 Memcheck（默认工具），查内存错误和泄漏。其他 5 个：Helgrind/DRD（race）、Massif（heap profile）、Callgrind/Cachegrind（perf 分析）。
深挖
"DBT 框架"具体什么意思？→ 把客户机指令翻译成 VEX IR，让 tool 插桩，再降回 host 指令在合成 CPU 上跑。这层抽象让"加新工具"成本极低（写组 callback 就行）。
L4 INIT Q2. 怎么用 Memcheck 查泄漏？典型命令是？
答案
valgrind --leak-check=full --show-leak-kinds=all --track-origins=yes --num-callers=30 ./prog。结果关注 "definitely lost"（必修）+ "indirectly lost"（连带）；"possibly lost" 通常误报；"still reachable" 通常忽略。
深挖
4 类 leak 怎么区分？→ definitely=丢指针没 free；indirectly=父亲丢了连带儿子；possibly=指针偏到块中间（多继承 vtable）；reachable=还指着但程序退出没释放（leaky singleton）。
L4 INIT Q3. --track-origins=yes 干什么用、为什么不默认开？
答案
追溯"未初始化值"来源。报告里会多一行 "Uninitialised value was created by a stack/heap allocation at..."，告诉你这个 uninit 数据是从哪行代码冒出来的，不只告诉你"哪里读到了"。
深挖
不默认开是因为性能开销 ~ 50%。Memcheck 内部要给每 4 字节再附一个 32-bit origin tag（指向分配栈），赋值/算术时跟着流动。
L4 INIT Q4. ASan / TSan 出来之后，Valgrind 还有用吗？
答案
有。ASan 杀掉了 Valgrind 在 CI / 日常开发的位置（快 10×），但"不能重编"的场景仍然只能 Valgrind：① 第三方闭源 .so；② 嵌入式 / 老 toolchain；③ 想跑可重现的 cache 模拟。
深挖
2026 年 Valgrind 的"独占主场"6 类：闭源 .so / 老 toolchain / MSan 不可用 / Cachegrind 可重现指标 / production-like 数据 sanity check / ASan 漏抓的双重保险。

10.2 L5 · 中级（4 题）"懂原理"

L5 MID Q5. 为什么 Memcheck 慢 10-30×？
答案
开销叠乘：① 合成 CPU 解释 ~ 4×；② V-bit 检查（每个 load/store 查 shadow） ~ 7×；③ shadow memory 自己访存挤掉 cache ~ 2×；④ stacktrace 抓取 ~ 1.2×；⑤ IR 翻译（cache 命中后接近 0）。乘起来理论 100×，实测 10-30× 是因为 I/O / sleep 等被减速少。
深挖
对比 ASan：ASan 编译期插桩，每次访存只多 1-2 条 host 指令检查 shadow（cache friendly），所以 2-3×。Valgrind 慢主要不是因为"翻译"，而是因为"插桩太多 + shadow 抖 cache"。
L5 MID Q6. Memcheck 的 shadow memory 怎么工作？为什么能查 uninit？
答案
对每一字节用户内存，shadow 维护：1 个 A-bit（是否可寻址）+ 8 个 V-bits（每位是否已定义）。malloc → A=1, V=undef；写入 → V=defined；free → A=0；读 → 检查 A=1 ∧ V=defined。位级精度。
深挖
① 内存开销：每用户字节多 9 位 → 总 ~ 2-3× 内存。
② 对比 ASan：ASan shadow 是 1B/8B 粒度，不能查 uninit（只查 addressable + redzone）。MSan 才补回了 V-bit 模型。
③ 为什么 V-bits 是 1 字节用 8 位？因为 C/C++ 允许位操作（如 bitfield、union），需要位级追踪。
L5 MID Q7. Helgrind 跟 TSan 的区别？什么时候用 Helgrind 而不是 TSan？
答案
① Helgrind 是 DBT，不需要重编；TSan 编译期插桩，必须重编。② Helgrind 慢 20-100×；TSan 5-15×。③ Helgrind 用 lockset + happens-before 混合算法，能独家查锁序反转；TSan 用 vector clock (FastTrack) 只查 race。④ TSan 信任程序员的 atomic mem-order，Helgrind 不信任，对 lock-free 更"偏执"。
深挖
什么时候选 Helgrind：① 不能重编（第三方 .so）；② 想抓 lock-free 算法的 mem-order bug（Helgrind 偏执反成优点）；③ 想查锁序反转（TSan 不查）。
什么时候选 DRD：cv / atomic 同步多 → DRD 误报少。
L5 MID Q8. Memcheck 能查栈越界吗？为什么？
答案
不能（绝大多数情况）。栈是 OS 一次性分给进程的合法内存，Memcheck 只能从 A-bit 看到"整页 mapped"。栈帧之间没有"redzone"，越界 1 个 int 写到了相邻局部变量上 → A=1, V=defined → Memcheck 看不出问题。
深挖
ASan 能查栈越界，因为它在编译期给每个栈对象前后插入 redzone（典型 32B），用 shadow 标 poison。这是"DBT 抓不到、需要编译期信息"的典型场景。
Memcheck 能查的栈相关错误：use-after-return（栈帧析构后某些场景能抓）、栈整体越界（栈溢出 SIGSEGV）。

10.3 L6 · 高级（3 题）"懂设计 + 工程化"

L6 SR Q9. VEX IR 是什么？为什么 Valgrind 要专门做一个 IR？
答案
VEX 是 Valgrind 自己定义的体系结构无关中间表示。每条 host 指令翻成多条 VEX IR statement（Get/Put/WrTmp/Store/Dirty 等），每个 statement 只读/写一次。这层抽象让 tool 写在 IR 上，不用懂 x86/ARM/RISC-V 细节，加新 ISA 只要写前后端。
深挖
VEX vs LLVM IR：① VEX 设计目标是"方便插桩"，不是"优化"；② VEX 比 LLVM IR 更接近物理指令（保留 EFLAGS 等），翻译保真；③ VEX 更小，能在线翻译快速生成。
VEX vs Pin / DynamoRIO：Pin 直接在 host 指令上加 hook（无 IR），写 tool 难；Valgrind 选了"易扩展"取舍。
L6 SR Q10. 怎么把 Valgrind 集成到 CI 里？suppression 怎么治理？
答案
① CI 命令：valgrind --error-exitcode=1 --suppressions=project.supp --xml=yes ...；② 把 suppression 拆成业务自己的 + 第三方库的两份文件，每条加注释（来源 / 为什么是误报）；③ 季度 review，因为库升级会失效；④ 同时跑 --gen-suppressions=all，PR diff 检查是否新增可疑 suppression。
深挖
反模式 5 条：① 用 ... 通配符吃掉一大段栈（容易误吃真 bug）；② 名字都叫 fix-later；③ 把项目自己代码的 leak 也压（应该修）；④ 不写注释；⑤ 把 suppression 写在源码 #pragma 里（不能版本管理）。
L6 SR Q11. 自己写了个 pool allocator，Memcheck 看不到 UAF。怎么办？
答案
用 Client Requests 宏。在 alloc 路径调 VALGRIND_MEMPOOL_ALLOC(pool, p, size)，dealloc 路径调 VALGRIND_MEMPOOL_FREE(pool, p)。这样 Memcheck 把 pool 的整块底层 mmap 看成 NoAccess，只在程序"实际拿到"slot 时才标 Defined → 用户在 dealloc 后还访问就能被抓到。
深挖
① 性能成本：Client Request 宏在 native 跑就是几条 NOP-like 指令，< 1ns 开销，可以放生产代码不需要 #ifdef。
② 同类宏 helgrind 也有：ANNOTATE_RWLOCK_ACQUIRED/RELEASED 让 Helgrind 认识自己实现的同步原语。
③ 实现原理：宏展开成"魔法 x86 序列"（一组特殊的 ROL 指令组合），native CPU 执行就是 NOP，但 Valgrind 翻译时识别为 client request 转走。

10.4 L7 · 专家（2 题）"系统设计 + 边界条件"

L7 EXPERT Q12. 在嵌入式 ARMv7 + 1GB RAM 上跑 Valgrind，会遇到什么问题？怎么解决？
答案（开放设计题）
4 个问题 + 解法：
① Memory 2-3×：原本 200MB 服务变 600MB → 接近 1GB 限制，OOM。解：缩小 testcase；或者跑 None tool（只测核心 DBT 开销）；或者只对部分进程开。
② 性能 10-30×：原本 10s 测试变 5min。解：缩小输入；或并行多个小 testcase。
③ ARMv7 syscall 支持：定制内核新 syscall Valgrind 可能不识别。解：升级到 valgrind 3.21+；或者 fallback 到不用该 syscall 的代码路径测。
④ 编译 cross-toolchain：要 yocto / buildroot recipe，或自己编。解：用 yocto 自带 recipe，或预编好的 bin。
考察
是否真的在嵌入式上跑过，是否懂多约束工程权衡。
额外加分：知道 valgrind 在 ARMv5 / MIPS / PowerPC 这类老架构上仍然是少数能用的工具，是 vendor 集成测试的"最后一道防线"。
L7 EXPERT Q13. 设计：怎么给团队建一套"渐进式" Valgrind 误报治理流程？让团队从不愿用到日常用？
答案（开放设计题）
4 阶段渐进：
① Phase 1 · Baseline · 跑一次完整 valgrind，把所有当前的报告全部转成 suppression（Big Bang）。CI 从 0 开始，新增任何错误才 fail。
② Phase 2 · Differential · 每月 review 一次 suppression list，挑 top 10 重复出现的真 bug 修掉，suppression 数量净减少。
③ Phase 3 · Per-team · 拆 suppression 文件按子模块归属，谁的代码谁负责。
④ Phase 4 · Mature · 把 Valgrind 替换成 ASan + LSan 跑 CI（快 10×），保留 Valgrind 作为夜间深度扫描 + 第三方 .so 专项检查。

关键：不要一上来就要求 0 suppression，那必死；先稳住边界，再慢慢往里收。
考察
① 工程领导力 —— 怎么推动一个"高摩擦工具"在团队里落地。
② 是否懂 suppression 的双面性：既是治理利器，也是"债务沉淀池"。
③ 是否能把 Valgrind 放在大工具栈里看 —— 它不是终态，是过渡阶段，最终被 ASan + eBPF 替代到边缘场景。
