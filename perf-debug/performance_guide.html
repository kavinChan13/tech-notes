测量先行

三大法则 · 标准优化流程 · 微基准 — Google Benchmark · 容易被忽略的几个测量原则

性能优化的三大法则

2.1 三大法则

法则 1: 测量, 不要猜

"过早优化是万恶之源" — Knuth。
没有 profiling 数据 → 不要优化任何东西。
"你认为是热点的代码" 与 "实际热点" 重叠率 < 30%。

法则 2: 优化前定基线

每次优化前 → 跑一次 benchmark 记录基线。
每次优化后 → 对比。看不到 ≥ 5% 改进就回退（噪音范围）。

法则 3: 优化用户感知的指标

p99 延迟 / 吞吐量 / 资源使用，而不是单次 nanosecond。
用户 30 ms 等待感知不到差异，但 100 ms → 1s 是巨大体验差。

法则 4 (额外): 20/80 法则

20% 的代码占 80% CPU 时间。优化这 20%。其它代码即便慢 10× 也无所谓。

2.2 标准优化流程

  1. 定义目标：p99 < 50 ms / QPS > 10000 / 内存 < 1 GB
  2. 建立基线：当前指标值 + 测试条件
  3. profiling：perf record → 火焰图，找 top 3 热点
  4. 假设 + 实验：针对热点提改进方案
  5. 实施：尽量隔离一个变量改
  6. 对比验证：再 benchmark，对比基线
  7. 提交或回退：明显改进保留，否则回退
  8. 监控：上线后持续观察，防止回退

2.3 微基准 — Google Benchmark

#include <benchmark/benchmark.h>

static void BM_VectorPushBack(benchmark::State& st) {
    for (auto _ : st) {
        std::vector<int> v;
        for (int i = 0; i < st.range(0); ++i)
            v.push_back(i);
        benchmark::DoNotOptimize(v);   // 防编译器优化掉
    }
}
BENCHMARK(BM_VectorPushBack)->Range(1<<10, 1<<20);

BENCHMARK_MAIN();
// 编译运行
$ g++ -std=c++20 -O3 bench.cpp -lbenchmark -lpthread -o bench
$ ./bench

BM_VectorPushBack/4096    21456 ns
BM_VectorPushBack/16384   89234 ns

Google Benchmark 自动处理：① 找统计稳定的运行次数；② 防止编译器优化 dead code；③ 报告含均值/标准差。

2.4 容易被忽略的几个测量原则

  • warm-up：第一次运行有缓存冷启动，要 burn-in 几次
  • 固定 CPU 频率：禁用 turbo boost / cpufreq governor 设 performance
  • 关闭 ASLR + 隔离 CPU：减少噪音
  • 测多次取分布：单次运行 ±20% 噪音正常
  • 小样本不可信：跑 100 万次以上才稳定
  • 注意编译器把代码优化掉：用 DoNotOptimize / ClobberMemory

CPU 微架构基础（你必须知道的）

现代 CPU 的几个关键事实 · 性能的三大杀手 · IPC (Instructions Per Cycle)

流水线 · cache miss · branch miss · IPC

4.1 现代 CPU 的几个关键事实

  • 流水线 (Pipeline)：每个指令分 ~14-20 个阶段（Intel Skylake 14，Apple M1 ~16）
  • 超标量 (Superscalar)：每周期可发射 4-6 条指令并行执行
  • 乱序执行 (Out-of-Order)：CPU 自动重排指令以填满流水线
  • 分支预测 (Branch Prediction)：预测 if/while 分支，错了就 flush 流水线 (~10-20 周期)
  • 预取 (Prefetcher)：硬件自动预测下次内存访问，提前加载到 cache
  • SIMD：单指令多数据 — SSE/AVX (x86), NEON (ARM)

4.2 性能的三大杀手

杀手开销 (周期)典型起因
Cache miss (L3 → DRAM)~ 200-300随机内存访问 / 大数据集
分支预测错误~ 10-20不可预测的 if / 数据相关分支
指令依赖链 (data dependency)~ 阻塞乱序执行循环里的串行更新

结论：现代 CPU 的速度不取决于指令条数，而取决于内存访问模式 + 分支可预测性。

4.3 IPC (Instructions Per Cycle)

评估代码"是否充分利用 CPU" 的关键指标：

  • 理想 IPC = 4 (Intel) / 6 (Apple M1)
  • 实测 IPC < 1 → CPU 大量等待（cache miss / 分支错 / 依赖链）
  • 用 perf stat 测：perf stat -e instructions,cycles ./prog
$ perf stat -e instructions,cycles,cache-references,cache-misses,branches,branch-misses ./bench

  3,541,233,442  instructions
  1,234,567,890  cycles
                   #    2.87  IPC                   ← 很好
     12,345,678  cache-misses              #    9.2% of all cache refs
        345,678  branch-misses             #    0.5% of all branches
"怎么判断一个程序的 CPU 使用是否健康?" — 答：① IPC ≥ 1.5（越高越好）；② cache-miss 率 < 5%；③ branch-miss 率 < 1%。perf stat 一把出完。

5.1 缓存层次（典型 Intel Skylake）

层级大小延迟共享
L1 (data)32 KB / core~ 4 cycles (1 ns)每核独立
L1 (instr)32 KB / core~ 4 cycles每核独立

5.2 Cache Line

缓存的基本传输单位 — 典型 64 字节。

  • 读 1 字节也会加载整个 64 字节
  • 设计数据结构时按 cache line 对齐 / 紧凑布局 → 局部性提升
  • 查看：cat /sys/devices/system/cpu/cpu0/cache/index0/coherency_line_size
  • C++17: std::hardware_destructive_interference_size

5.3 缓存友好的两个原则

① 时间局部性

"刚访问的数据再次被访问" — 编译器/CPU 自动利用。
你能做的：处理完一块再切换到下一块，不要东一榔头西一棒。

② 空间局部性

"相邻地址被一起访问" — 顺序遍历数组就是典型。
你能做的：用连续容器（vector），避免链表，访问模式顺序化。

5.4 经典例子：行优先 vs 列优先

constexpr int N = 4096;
int a[N][N];

// ✅ 行优先 — cache 友好 (顺序访问)
for (int i = 0; i < N; ++i)
    for (int j = 0; j < N; ++j)
        a[i][j] = i + j;            // 步长 = sizeof(int) = 4 字节

// ❌ 列优先 — cache miss 风暴
for (int j = 0; j < N; ++j)
    for (int i = 0; i < N; ++i)
        a[i][j] = i + j;            // 步长 = N*sizeof(int) = 16 KB → 每次都 cache miss
实测 4096×4096 int 矩阵：行优先 ~30 ms vs 列优先 ~200 ms。差 7 倍，仅仅是访问顺序不同！

5.5 Cache 友好的容器

容器cache 友好度原因
std::vector★★★★★单块连续内存，最优
std::array★★★★★同上，且栈分配
std::deque★★★分段连续，跨段时 cache miss
std::list★每节点独立分配，每步 cache miss
std::map★★红黑树，节点分散
std::unordered_map★★bucket 数组 + 链表节点
absl::flat_hash_map★★★★open addressing，节点连续
规则：默认用 vector。即便算法复杂度更高，缓存友好性常常胜出。N < 1000 时几乎所有场景 vector 优于 list / set。

False Sharing 假共享 ★ (并发性能杀手)

问题 · 重现 · 解决：cache line 对齐 · 易触发场景 · 检测工具

Cache line ping-pong · alignas(64) · padding

6.1 问题

两个变量被不同线程读写，但恰好在同一 cache line → CPU 缓存一致性协议（MESI）会让 cache line 在核间反复传输 → 性能暴跌。

6.2 重现

// ❌ 假共享 — 4 个 counter 在同一 cache line
struct Counters { std::atomic<long> c[4]; };
Counters g;

void work(int tid) {
    for (int i = 0; i < 100'000'000; ++i)
        g.c[tid].fetch_add(1, std::memory_order_relaxed);
}
// 4 线程每个写自己的 counter — 应该完全并行
// 实测: 比单线程慢 5-10× (假共享)

6.3 解决：cache line 对齐

struct PaddedCounter {
    alignas(std::hardware_destructive_interference_size)   // 64 字节对齐
    std::atomic<long> v;
};
PaddedCounter counters[4];     // 每个独占一个 cache line

// 或手动 padding
struct PaddedCounterManual {
    std::atomic<long> v;
    char pad[64 - sizeof(std::atomic<long>)];
};
实测：4 线程 100M 自增 — 假共享版 ~3 s，padding 版 ~0.4 s。8× 加速，仅仅靠对齐！

6.4 易触发场景

  • 线程池 worker 的per-worker stats 紧挨着
  • std::vector<atomic<T>> 中的元素
  • per-thread 计数器数组
  • spinlock 数组
  • RingBuffer 的 head/tail 指针没对齐

6.5 检测工具

  • perf c2c record/report — Cache to Cache，专测假共享
  • perf stat -e cache-misses — 高 miss 率是信号
  • Intel VTune — 直观显示 false sharing 热点
"怎么判断你的代码有 false sharing?" — 答：① perf c2c report 显示 hot cache line + multi-core 写；② 多线程加速比 ≪ 线程数；③ atomic 操作明显比单线程慢。修复：cache line padding。

数据布局优化

SoA vs AoS · 内存对齐与 padding

SoA vs AoS · 对齐 · padding

7.1 SoA vs AoS

风格布局遍历某字段时典型场景
AoS (Array of Structs)vector<Particle> 每个 particle 是一组字段cache miss 多 (跨字段)OOP 风格
SoA (Struct of Arrays)struct Particles { vector<float> x, y, z; }顺序访问, cache 友好, SIMD 友好游戏 / 数据处理
// AoS — OOP 风格
struct Particle {
    float x, y, z;
    float vx, vy, vz;
    int id;
    char name[32];
};
std::vector<Particle> particles(N);

// 只想更新 x 坐标
for (auto& p : particles) p.x += dt * p.vx;
// 每次跳 sizeof(Particle) = 56 字节 → 加载整个对象

// SoA — 数据并行风格
struct Particles {
    std::vector<float> x, y, z;
    std::vector<float> vx, vy, vz;
    std::vector<int> id;
};

// 同样更新 x
for (size_t i = 0; i < ps.x.size(); ++i)
    ps.x[i] += dt * ps.vx[i];
// 顺序访问 → cache 满载 + 编译器自动 SIMD 化
数据并行场景下 SoA 通常 2-5× 快于 AoS。但代码可读性差一些。

7.2 内存对齐与 padding

详细原理见 cpp_object_model.html §2 内存对齐与 Padding。这里只讲性能层面的优化。

① 字段重排减少 padding

// ❌ 24 字节 (含 padding)
struct Bad {
    char  a;        // 1 + 7 padding
    double b;      // 8
    char  c;        // 1 + 7 padding
};

// ✅ 16 字节
struct Good {
    double b;      // 8
    char  a, c;     // 2 + 6 padding
};
// 大小从 24 → 16, vector 内存少 33%

② SIMD 友好对齐

// AVX 要求 32 字节对齐, 否则用慢的 unaligned load
alignas(32) float data[1024];

③ Cache line 对齐 (并发场景)

struct alignas(64) PerWorkerStats {
    std::atomic<int64_t> processed;
    std::atomic<int64_t> errors;
    // ... padding 自动到 64 字节
};
规则：① 大 struct → 字段按大小降序排（大字段前），减少 padding；② 数据并行 → 考虑 SoA；③ 多线程 atomic 数组 → cache line 对齐。

8.1 分支预测器原理

现代 CPU 在条件分支跳转之前就预测会走哪条路径，提前执行后续指令。预测错 → 流水线 flush → ~10-20 周期惩罚。

预测器精度：

  • 规律分支（while 循环、固定模式）→ 99%+ 准确
  • 随机分支（数据相关）→ 50% 准确（近似随机）
  • 偏置分支（99% 走一边）→ 99% 准确

8.2 经典实验：sort 数据后变快

std::vector<int> v(32768);
// 填充 0~256 随机值
std::generate(v.begin(), v.end(), []{ return rand() % 256; });

// 不排序
long sum = 0;
for (int x : v) if (x > 128) sum += x;       // 6 ms

// 排序后
std::sort(v.begin(), v.end());
for (int x : v) if (x > 128) sum += x;       // 1.5 ms — 4× 快!
原理：排序后的数据让 if 分支变成"前半段都 false 后半段都 true" → 分支预测 ~99% 准确。乱序数据预测 50% → 大量 flush。

8.3 [[likely]] / [[unlikely]] (C++20)

if (rc > 0) [[likely]] {                  // 99% 走这里 → 编译器把它放热路径
    handle_data(rc);
} else [[unlikely]] {
    handle_error(rc);
}

// gcc/clang 老写法
if (__builtin_expect(rc > 0, 1)) { /* hot */ }

编译器据此重排 basic block：把 likely 路径放在 fallthrough，unlikely 放跳转目标 → 指令缓存友好 + 分支预测器从一开始就预测对。

8.4 Branchless (无分支) 编程

// ❌ 有分支
int max = (a > b) ? a : b;       // CPU 视情况而定

// ✅ 无分支 — 编译器通常自动转 cmov 指令
int max = std::max(a, b);

// ❌ 数据相关分支
for (int x : v) if (x > 0) sum += x;

// ✅ branchless — 用乘法替代
for (int x : v) sum += x * (x > 0);

// 数据排序去除分支
int abs_value = (x ^ (x >> 31)) - (x >> 31);   // branchless abs
误区：不是所有分支都该消除。
① 高度可预测分支几乎零成本 — 不必费力变 branchless；
② branchless 代码可能让两个路径都执行 → 总计算量更大；
③ 编译器很聪明，std::max/std::min/?: 通常自动 cmov。
手动 branchless 仅在测出分支预测错率高 + branchless 真有提升时使用。

11.2 inline 关键字 ≠ 强制 inline

inline 关键字主要用途：

  • 原始：建议编译器 inline (现在编译器自己决定)
  • 实际：避免链接时 ODR violation
  • 真正强制：__attribute__((always_inline)) (gcc/clang) / __forceinline (MSVC)
inline int add(int a, int b) { return a + b; }    // 编译器自己决定

__attribute__((always_inline))                          // gcc/clang 强制
inline int hot_fn() { /* ... */ }

__attribute__((noinline))                                // 强制不 inline
int cold_fn() { /* ... */ }

11.3 LTO 后链接器也能 inline

没 LTO：跨翻译单元函数无法 inline (编译器看不到对方代码)。
开 LTO：整个程序视角，跨 .o 也能 inline。

11.4 虚函数与 devirtualization

class Codec {
public:
    virtual void encode(...) = 0;
};

class JsonCodec final : public Codec {       // ⭐ final
    void encode(...) override;
};

// final 让编译器在已知具体类型时 devirtualize → inline
JsonCodec c;
c.encode(...);   // 编译器知道是 JsonCodec final → 直接调用 + 可能 inline
规则：① 让编译器决定 inline，不要瞎标 always_inline；② 派生类标 final 帮助 devirtualize；③ 模板代码天然可 inline。

12.3 PGO (Profile-Guided Optimization)

12.4 自动向量化 (auto-vectorization)

O3 / O2 + -mavx2 让编译器自动用 SIMD 指令。

// 编译器可向量化的循环
void add(float* a, float* b, float* c, size_t n) {
    for (size_t i = 0; i < n; ++i)
        c[i] = a[i] + b[i];
    // → 编译为 vaddps 指令, 一次处理 8 个 float (AVX 256)
}

让编译器愿意向量化的条件：

  • 循环次数编译期可分析
  • 访问模式简单 (顺序 + stride 1)
  • 无数据依赖 (可并行)
  • 类型简单 (POD / float / int)
  • 用 __restrict__ 告诉编译器指针不重叠
  • 开 -O3 + -march=native

13.2 锁优化的几个方向

13.3 分片示例

// ❌ 一把全局锁 — 高并发瓶颈
class SessionTable {
    std::mutex mu_;
    std::unordered_map<SessionId, Session> m_;
};

// ✅ 分片 — 16 把锁, 不同 session 无锁竞争
class ShardedSessionTable {
    static constexpr size_t N = 16;
    struct Shard {
        std::mutex mu;
        std::unordered_map<SessionId, Session> m;
    };
    std::array<Shard, N> shards_;

    Shard& shard_for(SessionId id) {
        return shards_[std::hash<SessionId>{}(id) % N];
    }
public:
    void insert(SessionId id, Session s) {
        auto& sh = shard_for(id);
        std::lock_guard lk{sh.mu};
        sh.m[id] = std::move(s);
    }
};

13.4 Lock-Free 计数

// 全局热点计数 — 多线程竞争 atomic
std::atomic<long> g_count{0};
void hit() { g_count.fetch_add(1, std::memory_order_relaxed); }

// ✅ thread_local 累加 + 周期性刷新到全局 — 完全无竞争
thread_local long tl_count = 0;
std::atomic<long> g_count{0};

void hit() { ++tl_count; }
void flush() { g_count += tl_count; tl_count = 0; }    // 周期性调用
实测：4 线程 100M 计数 — atomic 版 ~1 s（cache line 争抢），thread_local 版 ~50 ms。20× 加速。

微基准 (microbenchmark) 的陷阱

正确的微基准设置 · 微基准的"测出"结论常常误导。十大坑：

微基准的"测出"结论常常误导。十大坑：

  1. 编译器把 dead code 优化掉 
    for (auto _ : st) {
    int x = heavy();
    // x 没用到 → 整个 heavy() 被优化掉
}
// 修复:

    int x = heavy();
    benchmark::DoNotOptimize(x);
}
  2. 常量折叠：add(2, 3) 编译期算成 5
  3. 缓存命中过高：循环数据集 < L1，不代表实际性能
  4. 分支预测过准：固定模式让 predictor 100% 准
  5. turbo boost：单次基准时 CPU 升频，多线程时降
  6. CPU 频率波动：governor / 温度
  7. 测量小到测不准：纳秒级测量需要大循环
  8. 内存对齐：栈位置随机 / ASLR 影响
  9. 预热不足：第一次跑 cache 冷
  10. 用户态/内核态切换：syscall 测量含上下文切换

效果：CPU 降到 30%，QPS 翻倍。

症状：8 线程加 1 单线程的 8 倍，但实测只快 2 倍。

原因：每线程 stats 在同一 cache line。

修复：alignas(64) struct Stats {...};

效果：8 线程吞吐提升到 6.5 倍（接近理论值）。

16.3 案例 3: vector::push_back 大 buffer 拷贝

修复：异步日志（producer-consumer + 后台 flusher）。

效果：p99 从 200ms 降到 5ms。

所有案例的共同模式：测量 → 定位 → 简单改动 → 大幅收益。没有一个用了汇编 / SIMD / lock-free 这种 "高级"技术。常见性能问题都是基本功。

从自动向量化到 intrinsics ⭐

SIMD（Single Instruction Multiple Data）是 CPU 一条指令同时处理多个数据。x86-64 上 SSE / AVX2（256 bit, 8×float）/ AVX-512（512 bit, 16×float），ARMv8 NEON / SVE。用对了能让数值循环提速 4-16×。

16b.2 让自动向量化生效 —— 6 个前提条件

  1. 循环边界编译期可知或运行期常量（vectorizer 要预测迭代数）
  2. 无指针别名—— 用 __restrict__ 告诉编译器"两个指针不重叠"
  3. 循环体无函数调用（除非内联）
  4. 数据对齐—— alignas(64) + std::assume_aligned (C++20)
  5. 无数据依赖—— 迭代 i+1 不依赖 i 的结果
  6. 无早退分支—— break / continue / return 会阻止向量化（部分能转 masked op）

16b.4 三个能让 GCC/Clang 立刻向量化的提示

// ① __restrict__ —— 标记指针无别名
void axpy(float* __restrict__ y, const float* __restrict__ x,
          float a, int n) {
    for (int i = 0; i < n; ++i) y[i] += a * x[i];
}
// 不加 restrict：编译器假设 y 与 x 可能重叠 → 不能并行 → 失败

// ② alignas + assume_aligned (C++20)
alignas(64) float buf[1024];
float* p = std::assume_aligned<64>(buf);
for (int i = 0; i < 1024; ++i) p[i] *= 2.0f;
// 对齐后能用更宽的 SIMD（aligned load 比 unaligned 快）

// ③ #pragma GCC ivdep / #pragma omp simd
#pragma GCC ivdep                                       // 告诉 GCC "我担保没依赖"
for (int i = 0; i < n; ++i) ...;

#pragma omp simd reduction(+:sum) aligned(p:64)        // OpenMP 直接强制 SIMD
for (int i = 0; i < n; ++i) sum += p[i] * p[i];

16b.5 实例：dot product 三种写法对比

// 方案 1：朴素 C++（依赖自动向量化）
float dot_naive(const float* a, const float* b, int n) {
    float s = 0;
    for (int i = 0; i < n; ++i) s += a[i] * b[i];
    return s;
}
// -O3 后 GCC 13+ 会自动向量化到 AVX2，~1 个 cycle / 4 float

// 方案 2：std::experimental::simd / C++26 std::simd
#include <experimental/simd>
namespace stdx = std::experimental;
float dot_stdsimd(const float* a, const float* b, int n) {
    using V = stdx::native_simd<float>;
    V acc{0};
    int i = 0;
    for (; i + V::size() <= n; i += V::size()) {
        V va{&a[i], stdx::vector_aligned};
        V vb{&b[i], stdx::vector_aligned};
        acc += va * vb;
    }
    float s = stdx::reduce(acc);                    // 跨 lane 求和
    for (; i < n; ++i) s += a[i] * b[i];        // 尾巴
    return s;
}

// 方案 3：原生 AVX2 intrinsics（最快但只 x86）
#include <immintrin.h>
float dot_avx2(const float* a, const float* b, int n) {
    __m256 acc = _mm256_setzero_ps();
    int i = 0;
    for (; i + 8 <= n; i += 8) {
        __m256 va = _mm256_loadu_ps(&a[i]);
        __m256 vb = _mm256_loadu_ps(&b[i]);
        acc = _mm256_fmadd_ps(va, vb, acc);          // fused multiply-add
    }
    // 水平归约 8 个 float
    alignas(32) float tmp[8];
    _mm256_storeu_ps(tmp, acc);
    float s = tmp[0]+tmp[1]+tmp[2]+tmp[3]+tmp[4]+tmp[5]+tmp[6]+tmp[7];
    for (; i < n; ++i) s += a[i] * b[i];
    return s;
}

16b.7 SIMD 反模式

  • 盲目 intrinsics：忽略编译器自动向量化能力，写一堆 _mm256_* 反而更慢（cache 用量、寄存器分配）
  • 横向操作过多：跨 lane 的 hadd / permute 极慢（10+ cycles），尽量保持纵向运算
  • 分支严重：if 多的循环 → masked op 也救不了，先用 partition / sort 让数据"同质"
  • 未对齐 + AVX-512：AVX-512 对未对齐特别敏感，可能比 AVX2 还慢
  • 误用 AVX-512：Intel 部分 CPU 上 AVX-512 触发频率降级 → 整个 core 变慢
建议流程：① 写朴素清晰 C++；② -O3 -fopt-info-vec 检查；③ 没向量化 → 用 __restrict__ / alignas / #pragma omp simd；④ 仍不行 → Highway / std::simd；⑤ 最后才考虑 intrinsics。绝大多数情况 ②-③ 已经够。

用 profile 优化二进制 ⭐

编译器优化是"静态启发式"，不知道你的程序到底走哪条热路径。PGO（Profile-Guided Optimization）让你跑一次真实负载，把 profile 喂回编译器或 binary 工具，做更准确的 inline / 函数排序 / 分支预测。生产级 C++ 服务普遍能拿到 5-15% 端到端加速。

16c.1 PGO 经典三段流水线

# GCC PGO
# 阶段 1: 插桩编译（生成可写 profile 的二进制）
g++ -O2 -fprofile-generate=./pgo-data myapp.cpp -o myapp.inst
# 阶段 2: 跑真实/代表性负载，落 profile 文件
./myapp.inst < representative_input.bin
#   → ./pgo-data/*.gcda
# 阶段 3: 用 profile 重新优化编译
g++ -O2 -fprofile-use=./pgo-data myapp.cpp -o myapp.opt

# Clang PGO
clang++ -O2 -fprofile-instr-generate=p.profraw myapp.cpp -o myapp.inst
./myapp.inst ; llvm-profdata merge -o p.profdata p.profraw
clang++ -O2 -fprofile-instr-use=p.profdata myapp.cpp -o myapp.opt

16c.3 PGO 的痛 → AutoFDO 救场

插桩 PGO 三大问题：① 插桩二进制慢（不能直接上线）；② 需"代表性负载"，难找；③ 工作流复杂，工程师抗拒。

AutoFDO（Google 2010s 开源）：直接用 perf record 在生产环境采集 sampling profile（无插桩，开销 < 1%），转成 LLVM/GCC 能吃的 profile 格式。

# 生产机上：开 LBR 采样跑 30 秒
sudo perf record -e BR_INST_RETIRED.NEAR_TAKEN:uppp -b -- ./myapp prod.bin
sudo perf script -F +brstack > perf.script

# 转 LLVM profile
create_llvm_prof --binary=./myapp --profile=perf.data --out=myapp.afdo

# 用它重新编译
clang++ -O2 -fprofile-sample-use=myapp.afdo myapp.cpp -o myapp.opt
AutoFDO 已经成为 Google / Meta / Microsoft 生产 C++ 服务的标配编译流程。CI 每周拉一次最近 24h 的 prod sampling，作为下次发版的 PGO 输入，形成"生产数据 → 编译优化"闭环。

16c.4 BOLT — 二进制级 post-link 优化（Meta，2019）

BOLT（Binary Optimization and Layout Tool）是 Meta 开源的后链接优化工具：编译器已经做完了，BOLT 拿成品 ELF + perf profile，重新排函数和基本块。本质是"用 profile 重做 layout"，对大型 binary 收益惊人。

三步用法

# 0. 编译时加 emit-relocs（保留 BOLT 需要的重定位信息）
clang++ -O2 -Wl,--emit-relocs myapp.cpp -o myapp

# 1. 在生产或代表性环境采样
sudo perf record -e cycles:u -j any,u -- ./myapp prod.bin
perf2bolt -p perf.data -o myapp.fdata myapp

# 2. BOLT 优化
llvm-bolt myapp -o myapp.bolt -data=myapp.fdata -reorder-blocks=ext-tsp \
  -reorder-functions=hfsort+ -split-functions -split-all-cold \
  -split-eh -dyno-stats

# 3. 部署 myapp.bolt

16c.7 PGO/BOLT 常见坑

  • 调试体验下降：函数顺序乱了，gdb backtrace 不连续 → 保留 -g + 单独的 debug 包
  • profile 老化：业务流量模式变化后旧 profile 失效，反而降速 → 定期重采集
  • cold 路径变慢：BOLT 把冷代码放到 binary 尾部，第一次走 cold path 时 icache miss 暴增 → 业务平均 OK，p999 可能恶化
  • build 时间增加：PGO 三段式至少 build 2 次；BOLT 增加 5-30 分钟
  • perf 权限：生产采集需要 perf_event_paranoid 调低 + CAP_PERFMON（5.8+）

17.4 速答示范（第 4 题）

问：什么是 cache line? 典型大小?

答：

  • Cache line = 缓存的基本传输单位，CPU 在 cache 与 DRAM 之间一次只搬这么多字节
  • 典型 64 字节（x86/ARM 主流）；某些 ARM 是 128 字节
  • 读 1 字节也会加载整个 64 字节 → 顺序访问 8 个相邻 int 几乎免费
  • 查看：cat /sys/devices/system/cpu/cpu0/cache/index0/coherency_line_size
  • C++17：std::hardware_destructive_interference_size (避免假共享) / hardware_constructive_interference_size (主动共享)

面试官可能追问："cache line 概念怎么影响数据结构设计?" — 答：① 数据布局紧凑（同时访问的字段放一起）；② 多线程 atomic 数组要对齐到 cache line（防假共享）；③ 优先用连续容器（vector / array）。
