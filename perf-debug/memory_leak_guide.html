内存泄漏的本质再定义

三视角分歧 — VSZ / RSS / PSS / USS · 三视角下的"泄漏"定义 · 六种 RSS/USS 增长形态

为什么"泄漏"这个词在工程语境下是多义的？三视角分歧带来 80% 误诊

「程序内存一直涨」≠「内存泄漏」。这是工程实践与教科书的最大鸿沟。教科书定义是不可达的已分配内存，但生产现场遇到的 RSS 增长有 6 种本质截然不同的成因。本章重新定义"泄漏"，让你避免 80% 的误诊。

1.1 三视角分歧 — VSZ / RSS / PSS / USS

已有文档 cpp_os_fundamentals.html 提到 "RSS"，但从未拆解它和 VSZ/PSS/USS 的差异。在做内存泄漏判断时，看错指标 = 误诊。

图 1.1 · 四种内存指标的层次关系。判泄漏 首选 USS，看容器账单用 PSS，单进程粗判用 RSS，VSZ 几乎从不用来判泄漏。

⚠ 经典误诊场景

某服务在 K8s 上 RSS 显示 2GB，每个 pod 共 4 个进程同 binary。运维以为泄漏到 8GB 极限，重启 pod。

真相：libc/libstdc++/libssl 总共 1.5GB 被四个进程共享，真实 USS = 500MB × 4 = 2GB，根本没泄漏。

教训：容器内必看 PSS。

✓ 推荐采集命令

# 单进程：USS 才是泄漏判定指标
cat /proc/$PID/smaps_rollup
# 关注 Private_Clean + Private_Dirty

# 多进程容器：PSS 才是账单
cat /proc/$PID/smaps | grep "Pss:" | awk '{s+=$2}END{print s}'

# 时间序列（每秒采一次）
while :; do awk '/Rss/{r+=$2}/Pss/{p+=$2}END{printf"%d %d %d\n",systime(),r,p}' /proc/$PID/smaps; sleep 1; done

1.2 三视角下的"泄漏"定义

同一段代码，三个视角看到的"泄漏"完全不同：

视角「泄漏」定义看到现象对应工具容易误判
编译器/教科书 分配后没有 free/delete Valgrind 报 "definitely lost" Valgrind / ASan-LSan 容器、cache、单例不算（实际不算泄漏）
C++ 运行时 对象不可达（unreachable） 无任何指针指向它 LSan / 引用图分析 shared_ptr 循环引用：可达但永远不释
操作系统/运维 RSS/USS 单调上涨且不回落 top 看监控曲线斜率为正 smaps / 监控大盘 allocator cache 增长被误判为泄漏
实战上，OS 视角的"泄漏" 才是真正会触发告警和 OOM 的"泄漏"。教科书视角的泄漏如果体积很小、且不增长（比如初始化时少 free 一次），线上从不会成为问题。本文从此章起，"泄漏" 默认指 OS 视角。

1.3 六种 RSS/USS 增长形态

把"内存持续上涨"展开，有 6 种本质截然不同的形态。识别形态是排查第一步。

A 线性持续增长

RSS 与 time 成线性关系，且从不回落。每个请求/事件残留固定字节。

→ 典型真泄漏（容器/链表/observer）
→ 处理：heap profile 取 diff

B 阶梯式增长

每次某动作发生时 RSS 跳一个台阶。常见：连接建立、新订阅、cache miss。

→ 单次事件未清理 + 长生命周期容器
→ 处理：事件追踪 + 关联

C 锯齿但中线上移

有 GC/清理动作让 RSS 回落，但每轮峰值比上一轮高。

→ 内存碎片 或 cache 缓慢失控
→ 处理：jemalloc 切换 / 调 trim

D S 曲线 (Logistic)

起步快，最后平稳收敛。典型 warm-up + cache 填充 模式。

→ 不是泄漏，是 cache 预热
→ 处理：确认稳态值是否符合预期

E 周期性脉冲

每隔 N 分钟突涨突降。典型 定时任务/批处理。

→ 通常不是泄漏，是峰值
→ 处理：评估峰值合理性

F 突然跳高

某一时刻 RSS 直接 +1G/+10G。

→ 大对象分配 / 异常输入 / DoS
→ 处理：当时的请求日志

本文之后所有"排查方法"都建立在先判定形态之上。形态 D/E 几乎不需要排查（不是泄漏）；形态 A/B/F 才是真正需要工具上场的场景。

2.1 Arena — 多线程下的并发分配单元

图 2.1 · ptmalloc2 的 arena 模型 —— 这是"为什么多线程程序 RSS 看起来虚高"的根因。
关联已有知识 · cpp_os_fundamentals.html 提到 "tcmalloc 用 thread-local 缓存降低锁竞争"。但它没说：从 glibc 2.26 起，ptmalloc2 自己也有 tcache（per-thread cache），不需要切换 tcmalloc 也能享受类似优化。

这也是为什么很多老博客说"换 tcmalloc 性能爆炸"，但你今天换上反而没什么效果 —— 因为 glibc 已经偷师了。

2.2 Bins 与 tcache — free 后内存去哪了？

当你 free(p)，内存并不会立刻还给内核。它根据 size 落入不同的 bin，等待下一次 malloc 复用。这是"RSS 不下降"的第一根因。

Bin 类型数量size 范围结构归还内核?用途
tcache64 单 size≤ 0x408 (≈1KB)单链 LIFO否（per-thread 缓存） 无锁极速分配
fastbin10 单 size16~80B单链 LIFO否（不合并） 小对象快通道
unsorted bin1 个任意双向链否（缓冲池） 刚释放的"还没归类"
smallbin62 个< 512 (32位)/1024 (64位)双向链 FIFO否 小对象按 size 分桶
largebin63 个≥ smallbin 上限双向链 + 跨度索引否（除非 top chunk 触发 trim） 大对象按 size 区间分桶
top chunk1 个arena 堆顶剩余—可能（malloc_trim 时） 堆扩张/收缩的接口
mmap 大块—≥ 128KB独立 vma是（立刻 munmap） 大对象旁路

关键 1 — tcache 的"无锁陷阱"

tcache 是每线程独立的，free 进 tcache 的内存其他线程拿不到。在线程多、对象小的服务里，这能让 RSS 看起来虚高 30%~50%。

缓解：GLIBC_TUNABLES=glibc.malloc.tcache_count=0 关闭 tcache（性能下降）

关键 2 — 大于 128KB 才走 mmap

如果你频繁 new char[100KB]，全部进 largebin，永远不还内核。把分配尺寸调到 130KB 以上（或调小 MMAP_THRESHOLD），RSS 会立即跟随业务下降。

// 运行时动态调整
mallopt(M_MMAP_THRESHOLD, 65536);  // 64KB 起就走 mmap

2.3 chunk 布局 — 堆里每一个对象长什么样？

malloc 返回的指针不是真正的起点，前面藏着 16 字节 metadata（chunk header）。这部分知识在面试中常以"off-by-one 改 prev_size 攻击"出现。

图 2.2 · ptmalloc chunk 二态布局。注意 free chunk 的 fd/bk 指针复用了用户数据区，所以 free 后改写指针会导致下一次 malloc 返回畸形地址（这是堆溢出攻击的基础）。

2.4 M_MMAP_THRESHOLD / malloc_trim / 为什么 RSS 不下降

这是面试和线上排查最高频问题。完整答案需要把上面所有概念串起来：

核心问答：为什么 free 了全部对象 RSS 还是不下降？

四个原因，按优先级排：

  1. tcache/fastbin/unsorted/small/largebin 全部保留在用户态 —— 只是数据结构上是"free"，物理页一字未还。
  2. top chunk 不释放 —— 即使 top 很大，只要它位于堆顶，glibc 不会 brk(-size) 收缩堆（除非 top 超过 M_TRIM_THRESHOLD，默认 128KB，且它前面所有 chunk 必须全部 free）。
  3. 主 arena 堆是连续 brk —— 中间有一个 in-use chunk，就把后面全部空闲也锁住了，因为 brk 只能从堆顶动。
  4. mmap 大块例外 —— ≥ 128KB 的分配走 mmap，free 时立刻 munmap，这种情况 RSS 立刻下降。
图 2.3 · 主 arena 堆的"碎片锁堆"。只要中间存在一个长寿对象，后面所有 free 的物理页都还不回去。这是 ptmalloc2 的根本设计权衡。
leaky_service.cpp// 直觉：以为 free 了就好
void handleRequest() {
  auto* big = new char[64 * 1024];  // 64KB < 128KB → 走 brk arena
  ... process(big) ...;
  delete[] big;
}
// 长稳跑一周：RSS 8GB → 32GB（fastbin 碎片化），
// 但 valgrind/asan 都说"没泄漏"
service_with_trim.cpp// 方案 1：让中等对象走 mmap
mallopt(M_MMAP_THRESHOLD, 32 * 1024);  // 32KB 起走 mmap

// 方案 2：每个请求结束后强制还内存（高延迟，慎用）
void handleRequest() {
  ...
  if (request_count++ % 1000 == 0) {
    malloc_trim(0);  // 扫描 + 归还（<1ms 通常）
  }
}

// 方案 3：切换 jemalloc（推荐，零代码改动）
// LD_PRELOAD=/usr/lib/libjemalloc.so ./service
env-tuning.sh# glibc tunables（运行时调参）
export GLIBC_TUNABLES="\
  glibc.malloc.arena_max=2:\
  glibc.malloc.tcache_count=0:\
  glibc.malloc.mmap_threshold=65536:\
  glibc.malloc.trim_threshold=131072"

# 含义：
# arena_max=2     ：限制 arena 数量，减少 thread arena 缓存
# tcache_count=0  ：关闭 tcache（每个线程不再 cache 小块）
# mmap_threshold  ：64K 以上的分配走 mmap（立刻归还）
# trim_threshold  ：top chunk > 128K 时自动 trim
关联 · 如果你想理解 mmap 为什么能立刻归还，需要复习 cpp_os_fundamentals.html §4 mmap。本质上是 mmap 区域是独立的 vma，munmap 直接从页表删除映射、释放物理页。

3.1 三个决定性区分实验

面对 RSS 上涨，第一时间不要打开 Valgrind。先做这三个实验：

实验 A · malloc_trim 测试

在进程内通过 kill -SIGUSR1 触发 malloc_trim(0)，看 RSS 变化。

  • 下降 > 30% → allocator cache
  • 下降 < 5% → 真泄漏或长寿对象多

实验 B · 切 allocator 实验

LD_PRELOAD=libjemalloc.so 重启服务。

  • 峰值下降 > 40% → ptmalloc 碎片
  • 下降 < 10% → 不是 allocator 问题

实验 C · 静止观察

让服务不接任何流量 1 小时，看 RSS 是否还在涨。

  • 仍涨 → 后台定时任务/心跳泄漏
  • 不涨 → 与业务请求强相关

3.2 容器 capacity 陷阱（高频面试）

cap_pitfall.cppclass SessionMgr {
  std::vector<Session> sessions_;     // 峰值 100 万会话
  std::unordered_map<int, Conn> conns_;
public:
  void cleanup() {
    sessions_.clear();   // 仍占用 100 万个对象的内存！
    conns_.clear();      // bucket 数组依然在
  }
};
// 现象：业务侧"清理"了，RSS 不变。
// 误诊：怀疑泄漏 → 浪费一周
cap_correct.cppclass SessionMgr {
  std::vector<Session> sessions_;
  std::unordered_map<int, Conn> conns_;
public:
  void cleanup() {
    std::vector<Session>{}.swap(sessions_);  // 真清空 capacity
    std::unordered_map<int, Conn>{}.swap(conns_);
    // 或者 C++11 的 shrink_to_fit (不保证)
  }
};
vector::shrink_to_fit() 是 non-binding 请求，标准允许实现不做任何事。要真正归还内存，用 swap with temp 才是 100% 可靠的。

单点泄漏（漏一个 new）不可怕，结构性泄漏才是。本章讲泄漏如何"沿着对象引用图传播"，让一个看似无关的小 bug 把整个子系统的内存全锁住。

4.1 三层泄漏分类

业务设计错 • 日志缓冲打开了 unbounded • 队列消费速率 < 生产 • retry/超时积压 • 异步 task 越启越多 • coroutine 没 co_await • 滑动窗口未清 → 任何工具都报不出 → 需要业务指标对齐 → 最难定位

4.2 shared_ptr 循环引用 — 图解传播

这是 L2 泄漏的"教科书案例"，但真正的工程现场比教科书的二节点循环复杂得多。常常是 4~7 个节点构成的长环：

图 4.2 · 多节点循环的"断点选择"。关键设计原则："后向引用 / 通知方"应当用 weak_ptr。callback 引用 session 是天然的"后向"，最适合断开。
shared_ptr 控制块内部布局（深度问答）

一个 shared_ptr<T> 在堆上有两块内存：

shared_ptr<T>:                  ┌────────────┐
   ├─ T* ptr ────────────────►│ T 对象       │  (用 make_shared 时合并到同块)
   └─ ControlBlock* cb ──────►│ T 对象       │
                                  ├────────────┤
                              ControlBlock:
                                  ├─ atomic<long> strong_count
                                  ├─ atomic<long> weak_count  // = weak_ptr 数量 + (strong > 0 ? 1 : 0)
                                  ├─ deleter
                                  └─ allocator
  • strong → 0 时：调用 ~T()，但控制块不释放（因为可能还有 weak_ptr 要查 expired）
  • weak → 0 时：释放控制块
  • 所以 weak_ptr 自己也占内存，泄漏 weak 也会泄漏控制块（约 24~40 字节）
  • make_shared 把 T 和控制块合并 → 节约 1 次分配 + 局部性好
    但有副作用：weak_ptr 还在时整个 T 的内存都不能释放（因为合并了），反而比 shared_ptr(new T) 多占内存

4.3 lambda capture 与 callback 的三大死亡陷阱

陷阱 1 · 异步 lambda 持 this

class Service {
  vector<char> buf_;  // 大对象
  void tick() {
    timer_.async_wait(
      [this](...){ use(buf_); });
  }
};
// Service 释放 → lambda 在 worker
// 触发 use-after-free / 反过来要
// shared_from_this 保活 → 泄漏

陷阱 2 · shared_from_this 死循环

class Conn : enable_shared_from_this {
  function<void()> cb_;
  void init() {
    cb_ = [self=shared_from_this()]{
      self->do_xxx();  // 自己持自己
    };
  }
};
// cb_ → self → cb_ → ...
// 永不析构

陷阱 3 · observer 未注销

EventBus::register(
   [obj=shared_ptr{x}]{
     obj->on_event();
   });
// EventBus 是单例 →
// shared_ptr 永远持有 obj
// obj 永生 → 业务积累
// 一个个全死不掉
设计规则：异步回调里出现的 lambda 默认用 weak_ptr 捕获，在 lambda 内 lock() 提升。这种模式被 Asio/folly/seastar 广泛使用： 
timer_.async_wait([weak = weak_from_this()](...){
  if (auto self = weak.lock()) { self->do_work(); }
});

4.4 协程泄漏 — C++20 coroutine 的新陷阱

C++20 协程是 新的泄漏富矿。std::coroutine_handle 类似于裸指针：

coroutine_leak.cpptask<void> processOne() {
  co_await network_op();
  co_return;
}

void handler() {
  processOne();   // ⚠ 返回值丢弃 → coroutine_handle 泄漏
                  //    协程帧 (frame) 残留在堆上
}

// 正确：
void handler() {
  auto t = processOne();
  co_await std::move(t);  // 闭合 await
}

协程帧泄漏的特点：

  • 每次泄漏几百字节 (frame ≈ 200-500B)
  • ASan/LSan 完全报不出来 —— 因为 frame 通常用 operator new，promise 内部还引用着自己
  • 表现是 RSS 缓慢线性涨
诊断协程泄漏，常用手段是统计未完成 task 的数量，或在 promise 内嵌一个全局原子计数器 +/-。

现代 C++ 工程防御体系

Ownership 的四种基本姿态 · 容易被低估的 6 条防御规约 · 智能指针误区清单 · 单例 / 全局对象的"善终"

为什么用了智能指针和 RAII，"现代 C++ 服务仍然大量泄漏"？

RAII 不是银弹。智能指针只是工具，不是设计。本章给出一套ownership-first 的设计规约，结合 6 条工程铁律，让你的代码从源头难以泄漏。

5.1 Ownership 的四种基本姿态

姿态表达语义典型用法
独占 unique_ptr<T>, T&& 只能有一个所有者，move 转移 工厂返回、参数 sink、PIMPL
共享 shared_ptr<T> 多个所有者，最后一个释放 共享缓存、生命周期不确定的资源
观察 weak_ptr<T>, T*, T&, span<T> 不参与生命周期管理，仅借用 参数传递、回调、observer
值 T 拷贝/移动语义，自动析构 小对象、容器元素
铁律 1：函数参数默认用 观察姿态（const T&, span, string_view），只在需要转移所有权时才接 unique_ptr/shared_ptr。
铁律 2：返回值用独占（unique_ptr 或值），不要返回 raw new。
铁律 3：shared_ptr 只用于真正需要共享生命周期的少数场景；80% 的代码应该是 unique_ptr。

5.2 容易被低估的 6 条防御规约

① = delete 不安全的构造

class Owner {
  Resource* r_;
public:
  Owner(const Owner&) = delete;
  Owner& operator=(const Owner&) = delete;
  ~Owner() { delete r_; }
};
// 强制 caller 思考所有权移交

② Pinning · 避免 dangling capture

class Service : public enable_shared_from_this<Service> {
  void tick() {
    timer_.async_wait(
      [weak=weak_from_this()](...) {
        if (auto s = weak.lock())
          s->tick();
      });
  }
};

③ Sink-style 接口

// 不要这样（容易漏 release）：
void add(Conn* c);   // 谁 own?

// 这样：
void add(unique_ptr<Conn> c);
// caller 必须 move，所有权显式

④ 对生命周期写出 contract

// 用 [[gsl::lifetimebound]] 或注释
string_view getName() const
    [[gsl::lifetimebound]] {
  return name_;
}
// caller 知道："这个 view 跟着 this 活"

⑤ 限制 shared_ptr 的"传染"

// 不要让 shared_ptr 渗透到 API 边界
// 内部用 shared，对外暴露 view
class Cache {
  unordered_map<Key, shared_ptr<Val>> m_;
public:
  const Val& get(const Key&);  // 不暴露 shared
};

⑥ 周期性 audit 全局容器

// 任何全局容器都必须有：
//   ① 最大容量上限
//   ② 元素 TTL 或淘汰策略
//   ③ 清理路径（信号触发或定时器）
class Registry {
  LRU<Key, Val, 10000> m_;  // 强制 cap
};

误区错误代码正确做法
用 shared_ptr 当默认 shared_ptr<T> p = make_shared<T>(); 然后只在一处使用 改 unique_ptr。少 1 次原子操作 + 不会引入循环

5.4 单例 / 全局对象的"善终"

很多服务的"启动时分配但不 free"被 ASan 视为泄漏（"still reachable"），即使逻辑上没问题。但问题在于：如果单例持有了大量子对象（如连接池、cache），它们的析构顺序不可控，可能在静态析构期间访问已经销毁的依赖，导致 crash 而非泄漏。

unsafe_singleton.cppLogger& logger() {
  static Logger g;   // 析构顺序未定义
  return g;
}
// 进程退出时：
//   1) g 析构
//   2) 但其它单例的析构里还在 LOG()
//   3) 触发 use-after-destruction
leaky_singleton.cppLogger& logger() {
  static Logger* g = new Logger();
  return *g;   // 故意不 delete
}
// ASan 视为 "still reachable"，可忽略
// 进程退出由 OS 直接回收页
// 完全避免析构顺序问题

// 配合 LSan 抑制：
// __lsan_default_suppressions() { return "leak:Logger\n"; }
Google 的 abseil/protobuf 内部大量采用 "leaky singleton" 模式 —— 故意不析构。前提：① 内存量小且不增长；② 抑制 LSan 报告；③ 进程退出时 OS 兜底回收。

7.1 pmap / smaps / smaps_rollup — 零成本快查

已有文档 cpp_crash_debugging_guide.html 没专门讲。这是第一线最常用的工具，因为它不需要任何插桩、零损耗、瞬间出结果。

pmap-cookbook.sh# 1. 总览（最常用）
pmap -x $PID | tail -1
#                   ↑ 总 RSS / 总 mapped

# 2. 看 heap 区有多大（连续 anon 大块通常是 ptmalloc arena）
pmap -x $PID | grep '\[heap\]'

# 3. anon 段（每个 thread arena 一段，jemalloc 也用 anon）
pmap -x $PID | awk '$NF=="" {print}'

# 4. 看每个段的 dirty page（真正占物理内存的）
cat /proc/$PID/smaps | awk '
  /^[0-9a-f]+/ {addr=$0}
  /Rss/        {rss=$2}
  /Private_Dirty/ {pd=$2; print pd, rss, addr}
' | sort -rn | head
如何识别 thread arena 段?

thread arena 是通过 mmap(64MB, PROT_NONE) 预留地址空间，然后逐段 mprotect 为 RW。所以你会在 maps 里看到很多 64MB 倍数 的 anon 区。

cat /proc/$PID/maps | awk '$NF=="" {
  split($1, a, "-")
  sz = (strtonum("0x"a[2]) - strtonum("0x"a[1])) / 1024 / 1024
  if (sz > 30 && sz < 70) print sz" MB  "$1
}'

每多 1 个这样的段，说明多一个 thread arena。线程池过大 → arena 数量爆 → RSS 虚高。

7.2 heaptrack — 强力 diff 工具

heaptrack 是 KDE 出品，把 malloc/free 全程记录到磁盘，事后用 heaptrack_print / GUI 分析。最强能力是 "diff 两个时间点"。

heaptrack-workflow.sh# 启动时挂钩（也可 attach 到已有进程）
heaptrack ./my_service

# 或 attach（不重启服务）
heaptrack --pid $PID

# 输出文件：/tmp/heaptrack.my_service.PID.gz

# 文本分析
heaptrack_print /tmp/heaptrack.*.gz | less

# GUI（最好用）
heaptrack_gui /tmp/heaptrack.*.gz

# 关键视图：
#  · Bottom-Up: 哪个 malloc 调用栈分配最多
#  · Top-Down: 沿主流程看分配
#  · Flame Graph: 直观看占用
#  · Memory Leaks: "什么没 free"（按调用栈聚合）

✓ heaptrack 的杀手锏：diff

  • 在 t1（基线）抓一次：heaptrack_print -d t1.gz > base.txt
  • 在 t2（30 分钟后）再抓：heaptrack_print -d t2.gz > now.txt
  • 对比："new 出来但 t2 时仍未 free 的对象"按调用栈聚合
  • 排在最前的几个，几乎一定是泄漏源

⚠ heaptrack 的两个坑

  • 磁盘消耗大：分配热点高的服务 1 小时产生 5~20GB 日志
  • 慢路径会显著拉长：每个 malloc 多 1 次 unwind backtrace
  • 线程多时启停成本大（要 hook 所有 thread）
  • 关于 采样：可以设置 HEAPTRACK_TRACE_INTERVAL=N 抽样降低开销
同类工具：Bytehound —— 同样是 LD_PRELOAD 型分配追踪，但自带 Web UI、可导出 heaptrack/JSON/火焰图、支持把采样流式传到另一台机器，部分工业界团队用它替代/补充 heaptrack。定位与命令示意见 valgrind_guide §5.4（命令以 上游 README 为准）。注意它和 heaptrack 一样只做堆画像，不替代 ASan/Memcheck 查非法访问。

7.3 jemalloc profile — 生产首选

jemalloc 的 heap profile 是大厂线上首选。它采样记录分配调用栈（不记录每一次），损耗低于 2%，可以一直开着。

jemalloc-prof-setup.sh# 1. 启用 prof（必须在启动时设置或编译时打开）
export MALLOC_CONF="prof:true,prof_active:true,lg_prof_sample:19,prof_prefix:/tmp/jeprof.out"
#                    采样 1/2^19 ≈ 1/500K 次分配

LD_PRELOAD=/usr/lib/libjemalloc.so.2 ./my_service

# 2. 运行期 dump（kill -USR1 让 je 写一份 profile）
#    可选：mallctl("prof.dump", ...) 在代码里触发
kill -USR1 $PID  # 写出 /tmp/jeprof.out.PID.*.heap

# 3. 分析（jeprof 是 gperftools 自带 pprof 的兄弟工具）
jeprof --show_bytes --inuse_space \
   ./my_service \
   /tmp/jeprof.out.PID.0.heap

# 4. 输出 svg flamegraph
jeprof --svg ./my_service /tmp/jeprof.out.PID.0.heap > flame.svg

# 5. 取差值（最关键）
jeprof --base=/tmp/jeprof.out.PID.0.heap \
        --svg ./my_service \
        /tmp/jeprof.out.PID.5.heap > diff.svg
解读思路：在 SVG 火焰图中，横向宽度 = inuse_space（仍未 free 的字节）。如果某个调用栈在 diff 图里特别宽，意味着这条路径分配后没释放。用 diff 模式排除"长期持有"的合理对象（它们在 base 和现在都存在）。

7.4 ASan / LSan — CI 必备

已有 cpp_crash_debugging_guide.html §5.1 介绍过基础。这里补充泄漏专属用法：

asan-leak-only.sh# 编译
g++ -O1 -g -fsanitize=address -fno-omit-frame-pointer src.cpp

# 仅检查泄漏（不检查越界）
export ASAN_OPTIONS="detect_leaks=1:print_stats=1:atexit=1"

# 自定义抑制（忽略已知 leaky singleton）
export LSAN_OPTIONS="suppressions=lsan.supp:print_suppressions=1"

# lsan.supp 内容：
# leak:Logger::instance
# leak:libfoo.so

# Stand-alone LSan（不要 ASan 的开销）
g++ -fsanitize=leak src.cpp   # 损耗 < 5%

# 周期 dump（无须退出进程）：
# __lsan_do_recoverable_leak_check();
ASan 为什么快？(高频面试)

这是面试高频题。ASan ≈ 2-3× 损耗，Valgrind 是 10-30×。差异源于：

  • ASan：编译时插桩 —— 每条访存指令前插入"查 shadow byte"。本地访问，缓存友好，只多一两条指令。
  • Valgrind：运行时翻译（IR + JIT） —— 每条原始指令翻成多条 IR 跑在合成 CPU 上。CPU 缓存抖动严重。
  • ASan 用地址空间标红区（每 8 字节用 1 字节 shadow）发现越界。Valgrind 用纯软件 V-bit + A-bit 完整跟踪每字节。
  • 所以：ASan 适合 CI 默认开；Valgrind 适合线下深度排查（二进制 + 无源码 + 不可重编）。

7.5 eBPF / uprobe / bpftrace — 不重启抓调用栈

这是最近 3 年线上排查的"新一代神兵"。优势：不需要重启、不需要 LD_PRELOAD、不需要重新编译。挂上去、采几分钟、卸下。

bpftrace-malloc-stack.bt// 采集所有 malloc 调用栈，按调用栈聚合 size
uprobe:/usr/lib/libc.so.6:malloc {
  @bytes[ustack(perf), comm] = sum(arg0);
}

interval:s:60 {
  print(@bytes);
  clear(@bytes);
}

// 运行：
// bpftrace malloc-stack.bt -p $PID
// 1 分钟后输出：每条调用栈累计分配了多少字节

典型 bpftrace 配方

  • 大对象追踪：uprobe:libc:malloc / arg0 > 1048576 / { printf("%s %d\n", comm, arg0); ustack(); }
  • 未匹配 malloc/free：用 @alloc[arg0_ret] = ustack() 进 malloc，uretprobe 上 free 时减去
  • brk/mmap 系统调用追踪：tracepoint:syscalls:sys_enter_brk 配合 ustack

# 每 30s 采一次，连采 5 次

7.6 自研 malloc hook（终极后门）

所有商用工具都不合用时，自己写 hook。所有大厂都有自研版：

my_malloc_hook.cpp// 三种 hook 方式：

// 方案 A：__libc_malloc + LD_PRELOAD（最简单）
extern "C" void* malloc(size_t n) {
  static auto real = (void*(*)(size_t)) dlsym(RTLD_NEXT, "malloc");
  void* p = real(n);
  record_alloc(p, n, capture_stack());
  return p;
}

// 方案 B：glibc 提供的 __malloc_hook（已废弃但仍可用）

// 方案 C：jemalloc 的 hooks API（最优雅）
extent_hooks_t hooks = {...};
mallctl("arena.0.extent_hooks", nullptr, nullptr, &hooks, sizeof(hooks));
自研 hook 的三个坑：① 不要在 hook 里调用 malloc → 死循环；② 多线程下 backtrace 不安全（gdb's libunwind 在 dlopen 时可能死锁），用 libunwind 加 cache；③ 子进程 fork() 后 hook 状态可能错乱（CoW + lock 状态）。

RSS 上涨 → Page Cache 受压 → reclaim → OOM 的完整链路 · cgroup v2 memory 子系统五个关键文件 · Transparent Huge Page (THP) 与"内存膨胀…

"泄漏不一定 OOM · OOM 不一定泄漏" —— 完整理解二者的因果链

为什么"泄漏不一定 OOM"？答：swap、cgroup soft limit、overcommit。
为什么"OOM 不一定泄漏"？答：单次突发大对象、内存膨胀失控、其他进程偷内存。
本章从内核角度把这条链补全。

8.1 RSS 上涨 → Page Cache 受压 → reclaim → OOM 的完整链路

图 8.1 · 从 malloc 到 SIGKILL 的完整链路。容器内 OOM 由 cgroup 触发（系统 free 可能仍很多），裸机 OOM 由全局 reclaim 失败触发。

8.2 cgroup v2 memory 子系统五个关键文件

文件含义用途
memory.current当前内存使用容器实时账单
memory.max硬上限超过即触发 cgroup OOM
memory.high软上限超过后进程被 throttle（不立刻杀）
memory.low保护下限系统回收时优先不动这里的页
memory.events事件计数high/max 触发了几次（监控指标）
memory.stat详细分项anon/file/slab/sock 各自占多少
看 K8s 容器是否"被自家 cgroup OOM"：进 pod 看 cat /sys/fs/cgroup/memory.events。oom_kill > 0 表示本 cgroup 有过 OOM 事件。配合 kubectl describe pod 看 lastState reason=OOMKilled 确认。

8.3 Transparent Huge Page (THP) 与"内存膨胀"

THP 默认开启的系统上，内核会把连续的 2MB 物理页合并成"巨页"。这有放大 RSS 的副作用：

  • 你 malloc 了 1KB，内核为了 THP 给了你一个 2MB 巨页
  • RSS 立刻 +2MB
  • 典型表现：服务启动后 RSS 比静态计算的"对象总和"大 30%~100%
  • 缓解：echo madvise > /sys/kernel/mm/transparent_hugepage/enabled （只对 madvise 区域用 THP）
Redis/MongoDB 等内存敏感服务的官方文档都推荐关闭 THP。原因不只是膨胀，还有 fork 时 CoW 的"copy-on-fault 巨页拆分"延迟尖刺。

8.4 NUMA 与"看起来不平衡的 RSS"

NUMA 系统上，每个 socket 有自己的内存。kernel 默认 first-touch policy：哪个 CPU 先写，页就分配在那个 socket 的内存上。

  • 线程绑到 node 0，但被调度到 node 1 → 跨 socket 访问 → 慢 + 远端 RSS 上涨
  • 用 numastat -p $PID 看每个 node 的页分布
  • 大对象用 mbind 或 numa_alloc_* 显式指定 node
  • 泄漏定位时：如果某 node 的 RSS 远高于其它 node，可能是"在该 node 上跑的线程"在泄漏

案例 1 · IM 网关 RSS 每天涨 1G · arena 碎片化 假泄漏
现象
线上 IM 网关服务，处理 200 万长连接。重启后 RSS 8GB，每天上涨 1G，一周后 OOM。Valgrind 跑不动（线下重放性能不够），ASan 报告 0 leak。
初判
RSS 形态：稳定线性（形态 A）。但是 LSan/Valgrind 都不报泄漏。怀疑：① 真泄漏 L2（可达但永生）② allocator 碎片。
关键操作
① 触发 malloc_trim(0) → RSS 下降 1.5GB（说明大量空闲未还）
② 看 /proc/PID/maps → 数到 32 个 thread arena（线程池 32 个）
③ MALLOC_ARENA_MAX=2 重启 → RSS 稳定在 4GB，每天涨 200MB（仍涨但慢）
④ 切 jemalloc → RSS 稳定 3.5GB，不再增长
真相
不是泄漏，是 glibc ptmalloc2 在多线程 + 频繁小对象分配 + 跨线程 free 的模式下产生大量碎片。每个 arena 独立、tcache per-thread 让对象"分散到各处"，碎片永久无法回收。
修复
① 切 jemalloc（最低风险，LD_PRELOAD）② 业务上把热点对象池化（slab）。
教训
RSS 涨 ≠ 泄漏。先做 malloc_trim 实验，3 秒判断是不是 allocator 问题。
案例 2 · 长稳测试夜间增长 · shared_ptr 循环 真泄漏 L2
现象
72 小时长稳测试，前 48 小时正常，48 小时后 RSS 突然加速增长。LSan 报告 0 个 leak。
初判
形态：突变后线性（形态 A 变种）。可达但永生 → 怀疑 shared_ptr 循环。
关键操作
① jemalloc profile diff 出"OrderBook 对象"和"SubscriberList"是 top 增长
② gdb attach + info shared_ptr (pretty printer) 看 use_count
③ 发现一个 shared_ptr<Subscriber> 持有 shared_ptr<OrderBook>，OrderBook 又持有该 Subscriber → 循环
④ 触发条件：业务在 48h 后某条件成立才加入这个订阅关系
修复
Subscriber 持 OrderBook 改为 weak_ptr。CI 加入静态分析 + 单元测试主动构造循环场景。
教训
循环引用是 L2 泄漏典型，LSan 报不出。稳定性测试必须 > 业务最长周期（48h 才暴露）。jemalloc profile diff 是看 L2 的金手段。
案例 3 · thread cache 误诊 · 切 tcmalloc "更糟" 误诊
现象
团队怀疑 ptmalloc 慢，切 tcmalloc，RSS 涨幅增大了。
初判
tcmalloc 的 thread cache 默认大于 glibc tcache，cache 池更大 = RSS 更大。
关键操作
① MallocExtension::GetStats 输出 → 看到 thread cache 占 800MB
② 调 TCMALLOC_MAX_TOTAL_THREAD_CACHE_BYTES=64MB
③ 再次部署，RSS 回到正常区间
真相
不是 allocator 问题，是 默认配置不合适。tcmalloc 默认 thread cache 64MB × N 线程 = 几 GB。在 RSS 受限容器内必须调小。
教训
切 allocator 不是无脑动作，必须调参。jemalloc 也类似（dirty_decay_ms）。
案例 4 · async lambda 持 this · 连接泄漏 真泄漏 L1+L2
现象
WebSocket 服务，客户端断开后服务端 RSS 不下降。同时 /proc/PID/fd 数量持续上涨。
关键操作
① fd 上涨 → 复习 FileDescriptor_DeepDive.html §11.2 fd 泄漏检测
② 抓 core dump → gdb 看 info threads → 大量 worker 在 epoll_wait

教训
fd 泄漏 + 内存泄漏经常同源。资源都挂在同一个对象图上。
案例 5 · coroutine task 未 await · 帧泄漏 L1（工具盲区）
现象
C++20 协程改造后的 RPC 服务，RSS 缓慢线性增长。ASan/LSan/heaptrack 都报 0 leak。

案例 6 · observer 没注销 · 单例事件总线 L2 业务

10.1 初级（5 题）— P5/L4 校招新人门槛

L4 INIT Q1. new 之后忘了 delete 是不是泄漏？
答案
是。但只是 L1（definitely lost）。如果你 new 之后保留了指针在某个全局，那是 L2（still reachable）。要区分。
考察
能否区分 "无人持有" 和 "有人持有但永不释放"
追问
如果是单例 leaky 设计呢？→ 答："故意不 delete 由 OS 回收，这不算 bug"
L4 INIT Q2. unique_ptr 和 shared_ptr 区别？
答案
① unique_ptr 独占所有权，只能 move 转移；shared_ptr 引用计数，可共享。② unique_ptr 零开销（等价于裸指针 + RAII）；shared_ptr 有控制块 + 原子计数。③ 默认 80% 应该用 unique_ptr。
考察
是否知道性能差异（不只是语义差异）
常见错误
说 "都行随便用一个" → 大扣分
L4 INIT Q3. 什么是 RAII？
答案
Resource Acquisition Is Initialization。在对象构造时获取资源，析构时释放。利用栈展开保证异常安全下也能释放。
深挖
① 为什么 C 语言做不到？→ 没有析构函数
② Java 的 try-with-resources 是 RAII 吗？→ 不完全是（无栈对象，无确定性时机）
③ Go 的 defer 呢？→ 函数级，不是对象级
L4 INIT Q4. 怎么用 Valgrind 查泄漏？
答案
valgrind --tool=memcheck --leak-check=full --show-leak-kinds=all ./prog
关注 "definitely lost"，"still reachable" 通常忽略。
深挖
Valgrind 太慢怎么办？→ ASan/LSan。原理差异 = 编译插桩 vs 运行时翻译。性能 2-3× vs 10-30×
L4 INIT Q5. 栈对象需要 delete 吗？
答案
不需要。栈对象自动析构。delete 栈对象会 crash（错误的 free 给了 allocator 一个非堆地址）。
常见错误
说 "delete 栈对象不会有问题，反正会被忽略" → 完全错

10.2 中级（6 题）— P6/L5 一线工程师

L5 MID Q6. shared_ptr 为什么会循环引用？怎么解决？
答案
A 持 shared<B>，B 持 shared<A>，引用计数永远 ≥ 1。解决：把其中一边改 weak_ptr。规则：后向引用 / 通知方 用 weak。
深挖
① weak_ptr 内部怎么实现？→ 控制块里的 weak_count，可独立于 strong_count。
② weak_ptr.lock() 是无锁的吗？→ 是 CAS 循环（lock-free）。
③ make_shared 和 shared_ptr(new) 在内存布局上有什么区别？→ 前者合并 T 和控制块；后者两次分配。前者更省、cache 友好，但 weak 还在时 T 也不能释放。
常见错误
只说"改 weak_ptr"不解释哪一边改 → 中等扣分
L5 MID Q7. malloc 为什么不立即归还内存给内核？
答案
① 性能：归还后下次 malloc 又要 syscall。② brk 只能从堆顶动，中间空闲还不了。③ free 后进入 bins（fastbin/smallbin/largebin/unsorted/tcache）等待复用。④ 仅 ≥128KB 的 mmap 块 free 时立刻 munmap。
深挖
① M_MMAP_THRESHOLD 是什么？默认值？怎么调？→ 128KB，mallopt 或 GLIBC_TUNABLES
② malloc_trim 做什么？→ 扫描堆顶，把 top chunk 中可还的部分通过 brk(-N) 还给内核
③ tcache 为什么独立 per-thread？→ 无锁分配，配合 fastbin 形成两级 cache。glibc 2.26 引入。
L5 MID Q8. RSS 不下降怎么办？
答案
四步排查：① malloc_trim(0) 试试 ② 切 jemalloc 试试 ③ jemalloc/tcmalloc heap profile diff ④ 排查是否容器 capacity 没收缩（vector::clear 不缩容）。
深挖
有没有可能 free 了 RSS 仍上升？→ 有。free 进 fastbin，但下一次 malloc 走了不同 size，反而又申请新页。这是典型碎片场景。
L5 MID Q9. ASan 为什么比 Valgrind 快这么多？
答案
① ASan 编译期插桩，每条访存前查 shadow byte。② Valgrind 是整程序动态翻译，每条原始 inst 翻成多条 IR 在合成 CPU 跑。③ ASan 损耗 2-3×，Valgrind 10-30×。
深挖
① ASan 的 shadow memory 怎么映射？→ 每 8 字节用户内存 → 1 字节 shadow，固定偏移（shadow = (mem >> 3) + 0x7FFF_8000）
② ASan 能查未初始化内存吗？→ 不能，那是 MSan
③ ASan 能查越界但 Valgrind 漏报吗？→ 能。ASan 的 redzone 设计；Valgrind 看不到栈/全局变量越界（只看 heap）
L5 MID Q10. allocator 如何减少锁竞争？
答案
三级 cache：① per-thread tcache（无锁）② per-arena fastbin（同 arena 内一把锁）③ 多 arena（不同线程分散到不同 arena）。tcmalloc/jemalloc 类似但更激进。
深挖
① jemalloc 的 size class 设计？→ 几何级数，约 200 个 class
② tcmalloc 的 page heap 锁颗粒？→ 每个 size class 一把
③ 为什么不能完全无锁？→ 跨线程 free 必然要锁（dest arena）
L5 MID Q11. vector::clear 之后内存会还吗？
答案
不会。capacity 不变。clear 只析构对象，buffer 还在。要真释放：vector<T>{}.swap(v) 或 v.shrink_to_fit()（后者标准上不强制）。
同理 unordered_map::clear 不缩 bucket 数组。

10.3 高级（6 题）— P7/L6 资深 / Staff

L6 SR Q12. 线上不能重启，怎么定位泄漏？
答案
三个手段（按优先级）：① 如果用了 jemalloc/tcmalloc，它们的 profile 模式开了就在跑，直接 SIGUSR dump → jeprof diff。② attach heaptrack --pid（10-20% 损耗，10 分钟够定位）。③ bpftrace / eBPF memleak（采样最低损耗）。不能用 Valgrind/ASan（需重启 + 高损耗）。
深挖
① 没开 prof 怎么办？→ 切流量到一个 canary 实例，重启它开 prof
② 容器内连 bpftrace 都装不了？→ ssh 到宿主跑 sidecar，用 cgroup 路径过滤
L6 SR Q13. 为什么"泄漏不一定 OOM，OOM 不一定泄漏"？
答案
泄漏 ↛ OOM：① 速度极慢 / 服务进程会 cycle ② swap 兜底 ③ cgroup 没设 hard limit ④ overcommit + 写得稀疏，未必触发 page fault。
OOM ↛ 泄漏：① 一次性 alloc 大对象（输入恶意）② 业务突发峰值 ③ 同宿主其它进程偷内存 ④ 内存膨胀 (THP/碎片) ⑤ allocator cache 失控
L6 SR Q14. delete this 为什么危险？
答案
① this 之后任何成员访问 = UAF ② 编译器可能在调用 delete 后仍访问 vptr（虚函数解析） ③ 你必须保证 this 是 new 出来的（栈对象 / 数组 / 静态对象都 crash）④ 异常安全断裂。
正确做法：不要 delete this。用 shared_from_this 或外部生命周期管理。
深挖
什么场景必须 delete this？→ 极少。比如 COM 风格的 AddRef/Release 模式，Release==0 时 delete this。但现代 C++ 用 intrusive_ptr 替代。
L6 SR Q15. weak_ptr 的内部实现？
答案
weak_ptr 持有 同一个控制块 的指针，不增加 strong_count，但增加 weak_count。
strong→0 时调 ~T()，但控制块不释放（因为 weak_ptr 还可能查询）。
weak→0 时才释放控制块。
lock() 是 CAS 循环："读 strong_count，若 >0 则 +1，否则失败"。
深挖
① make_shared 时控制块和 T 合并，那 strong=0、weak>0 时，T 已析构但内存不能释放，相当于浪费？→ 是的。这是 make_shared 的缺点。
② 为什么 use_count() 不线程安全？→ 读了立刻就过期，应该用 expired() 替代逻辑判断
③ enable_shared_from_this 怎么工作？→ 内部有个 weak_ptr<T> _Wptr，shared_from_this() 就是 _Wptr.lock()
L6 SR Q16. 你怎么设计一个长稳服务的内存治理体系？
答案（架构题）
四道防线：
① 静态防御：clang-tidy + cppcoreguidelines + 自研 lint（强制 weak 捕获 / 限制 shared_ptr 边界）
② CI 防御：每个 PR 跑 ASan/LSan + 长稳模拟 30 分钟
③ 运行时观测：jemalloc profile 常开 + USS 监控 + slope alarm（24h 斜率 > 阈值告警）
④ 应急工具链：jeprof / heaptrack / bpftrace 在标准基础镜像里，SOP 文档化
考察
是否能从单点工具上升到"体系"的视角
L6 SR Q17. 怎么实现一个 malloc hook？两个坑？
答案
① LD_PRELOAD + dlsym(RTLD_NEXT, "malloc") 接管。② 或 glibc 的 __malloc_hook（已废弃但能用）。③ 或 jemalloc 的 extent_hooks（最优雅）。
坑 1 · 递归
hook 内不能调 printf/malloc → 死循环。用 syscall 直写 fd。
坑 2 · backtrace 不安全
多线程下 unwind 可能死锁（信号 + dl_iterate_phdr 持锁）。用 frame pointer 自己走或 libunwind cache。

L7+ EXPERT Q18. ptmalloc 的 chunk header 布局？P 位用来做什么？
答案
每个 chunk 头 16 字节（64 位）：prev_size | size+flags。
flag 低 3 位：P (PREV_INUSE)、M (IS_MMAPPED)、A (NON_MAIN_ARENA)。
P 位：标记前一个 chunk 是否在使用。free 时检查下一块的 P 位决定是否合并。如果前一块也 free，本块 prev_size 字段才有效（指向前块起始）。
深挖
off-by-one 改 P 位为什么导致漏洞？→ 攻击者改 size 字段的 P 位，glibc 误判可合并，把用户数据当 free chunk 链表处理，链表操作 unlink 改写指定地址 → write-what-where。这是 heap fengshui 攻击的基础。
L7+ EXPERT Q19. 在 NUMA 系统上发现某 socket RSS 远高于其它 socket，怎么排查？
答案
① numastat -p $PID 看分布
② 检查线程亲和性 (taskset/numactl)：是不是有线程绑到该 socket 但跨 socket alloc？
③ first-touch policy：哪个 CPU 写，页落在哪 node。负载分配不均可能导致"hot socket"
④ 检查 NUMA-aware allocator 配置（jemalloc 有 percpu arenas）
⑤ 如果是泄漏，泄漏的线程亲和性会反映在某 node 上
L7+ EXPERT Q20. C++20 协程的 frame 在哪？怎么排查协程泄漏？
答案
frame 由 operator new 在堆上分配（编译器可优化成栈，需 HALO 启用）。包含：所有局部变量、promise、suspended state。
排查：① 在 promise 的 ctor/dtor 嵌全局原子计数 ② handle 在 final_suspend 后是否被 destroy
③ jeprof / heaptrack 看 operator new 的 size 分布，frame 通常是个小固定值
深挖
HALO 是什么？→ Heap Allocation eLision Optimization，编译器在 inline 后能证明 frame 不逃逸时把它栈分配。需要 -O2+ 且 await 表达式都内联。
L7+ EXPERT Q21. 容器里的 OOMKilled 但宿主机 free 内存很多，可能原因？
答案
cgroup memory.max 触发了。看 memory.events.oom_kill。原因：
① cgroup limit 设得太低
② 单进程实际 USS 已达 limit（不是宿主总量）
③ Java/Go 等运行时未感知 cgroup（C++ 通常无此问题）
④ kernel memcg "reparenting" 后 LRU 评估异常（罕见）
⑤ 进程 oom_score_adj 太高被优先选中
L7+ EXPERT Q22. 设计：如何在不重编不重启的前提下，给一个跑了 3 天的服务挂上"分配热点追踪"？
答案（开放设计题）
