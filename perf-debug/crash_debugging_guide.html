Crash 信号分类

信号产生的内核流程示意 · 哪些信号会产生 core dump?

Linux 下进程异常终止的根因，几乎都来自内核或运行时通过 signal 机制把异常事件投递给了进程。理解信号是 crash 排查的起点。

动手前先做关键字预判，避免追错方向（电信现场尤其重要）：
  • Abort signal received — 进程是被外部 kill 的，不是它自己挂的。往上几行查 crash framework / 系统日志，多半是内存/消息队列耗尽。
  • Shutdown in progress — 关机过程中崩溃，这是结果不是原因，往上找真正的关机触发。
  • 多域二进制崩溃（如 myapp_exe、mymodule_main）只需 c++filt 解符号即可分诊到对应域，不必下 NotStripped 包。详见第九章。
信号编号典型场景内核如何产生
SIGSEGV 11 空指针/野指针解引用、写只读段、访问未映射地址、栈溢出 CPU 触发 page fault → do_page_fault 检查 VMA 不存在或权限错误 → force_sig_info(SIGSEGV)
SIGABRT 6 std::terminate()、assert()、abort()、glibc 检测到堆破坏 (double free / corruption) 用户态 abort() → raise(SIGABRT) → 内核投递；不可被忽略
SIGBUS 7 未对齐访问 (ARM)、mmap 文件被截断后访问、写满磁盘的 mmap page fault 但物理上不能满足（比如对应文件已经被 truncate）→ SIGBUS
SIGFPE 8 整数除 0、INT_MIN / -1、浮点异常 (启用 feenableexcept 后) CPU #DE/#XF 异常 → do_trap → force_sig_fault(SIGFPE, ...)
SIGILL 4 函数指针损坏、跳转到非代码段、CPU 指令集不支持 (老 CPU 跑 AVX 程序)、栈被破坏后 ret 到非法地址 CPU #UD → do_invalid_op → SIGILL
SIGPIPE 13 对端关闭后还在 write，常见于长连接 / socket 通信 TCP 收到 RST 后 write 返回 EPIPE 并发 SIGPIPE，未处理则进程退出
SIGKILL 9 OOM Killer、kill -9、cgroup OOM、systemd 强制结束 不能被捕获、不会产生 core dump（除非 systemd-coredump 配合）
SIGTERM 15 正常终止请求；如未处理也会让进程退出（通常不算 crash） 用户态 / systemd / kill 发送

信号产生的内核流程示意

// 简化版：x86-64 缺页异常 → SIGSEGV 的链路
asm_exc_page_fault          // IDT 入口
  → handle_page_fault
  → do_user_addr_fault
  → find_vma()                // 未找到 VMA
  → bad_area_nosemaphore()
  → force_sig_info(SIGSEGV, &si, current);
  // 进程从内核态返回前，arch_do_signal 投递信号
  // 默认动作：core dump + 进程退出
所有同步信号 (SIGSEGV/SIGBUS/SIGFPE/SIGILL/SIGABRT) 都是线程局部的：哪个线程触发，信号就投递给哪个线程。这就是为什么多线程 core 一定要看 thread apply all bt。

哪些信号会产生 core dump?

  • 会：SIGSEGV, SIGABRT, SIGBUS, SIGFPE, SIGILL, SIGQUIT, SIGSYS, SIGTRAP, SIGXCPU, SIGXFSZ
  • 不会：SIGTERM, SIGINT, SIGHUP, SIGPIPE, SIGKILL（除非配合 systemd-coredump）

ulimit 与 RLIMIT… · /proc/sys/kerne… · systemd-coredump

2.1 ulimit 与 RLIMIT_CORE

每个进程都有 RLIMIT_CORE，决定 core 文件最大字节数；为 0 时完全不写 core。

# 查看当前 shell 配置
ulimit -c          # 0 表示禁用

# 临时打开（仅当前 shell）
ulimit -c unlimited

# 永久打开（推荐写入 /etc/security/limits.conf）
*  soft  core  unlimited
*  hard  core  unlimited

# systemd 服务则在 unit 文件中
[Service]
LimitCORE=infinity

2.2 /proc/sys/kernel/core_pattern

这个 sysctl 控制 core 文件名/路径，支持管道（首字符 | 表示交给某进程处理）。

# 落地到固定目录（生产推荐：唯一文件名 + 不覆盖）
echo '/var/cores/core.%e.%p.%t' > /proc/sys/kernel/core_pattern

# 占位符
%e  可执行文件名 (basename)
%p  PID
%t  时间戳 (epoch)
%h  hostname
%s  导致 core 的信号编号
%c  RLIMIT_CORE 上限
%E  完整可执行路径 (用 ! 代替 /)

# 通过管道发给 systemd-coredump（现代发行版默认）
|/usr/lib/systemd/systemd-coredump %P %u %g %s %t %c %h
写入到 NFS、容器只读层、磁盘满的目录都会导致 core 写不出。生产环境请把目录单独挂载，并提前把 /proc/sys/fs/suid_dumpable 设为 2，否则 setuid 程序不写 core。

2.3 systemd-coredump

# 列出最近所有 core
coredumpctl list

# 查看某个 core 的元数据 + journal
coredumpctl info 12345          # PID
coredumpctl info /usr/bin/myapp # 二进制名

# 直接进入 gdb 调试
coredumpctl gdb 12345

# 把 core 文件抽取出来（默认压缩在 /var/lib/systemd/coredump/）
coredumpctl dump 12345 -o ./core.bin

2.4 Core 文件格式 (ELF NOTE)

core 本质是一个 ELF Core 文件 (e_type = ET_CORE)，由若干 PT_LOAD 段和一个关键的 PT_NOTE 段构成：

# /proc/<pid>/coredump_filter 位掩码
0x01  匿名私有
0x02  匿名共享
0x04  文件映射私有
0x08  文件映射共享
0x10  ELF header
0x20  巨页私有
0x40  巨页共享

# 查看
cat /proc/self/coredump_filter   # 默认 0x33 (= 51)

# 验证一个 core 文件
file core.12345
readelf -n core.12345 | less     # 看 NOTE
readelf -l core.12345 | head     # 看 segments

2.5 生产环境 core dump 收集策略

  • 独立分区：/var/cores/，避免写满 root；按周期 logrotate 压缩
  • 大小裁剪：把 coredump_filter 设为 0x33（默认）即可；千万别设 0xFF（连共享文件映射都写，可能 GB 级）
  • 命名唯一：core.%e.%p.%t.%h 防止同名覆盖
  • 外送服务：core_pattern 设为 |/opt/agent/uploadcore，将 core + binary + libs 一起打包送中央存储
  • Build-ID 索引：保存所有发布版本的 .debug，用 build-id 反查（见 GDB 章节）
  • 容器：核心要点 — core_pattern 在宿主机命名空间生效；容器内 ulimit 必须打开

3.4 调试 -O2 优化代码的技巧

优化后会出现：变量被<optimized out>、栈帧合并 (inline / tail-call)、行号跳跃。
  • 保留调试信息：-O2 -g（甚至 -Og）；务必不要 strip，或保留 .debug 副本
  • 用 info frame 看 prologue 是否被消除（rbp 不一定有）
  • 关键函数加 __attribute__((noinline)) 让栈可读
  • 编译加 -fno-omit-frame-pointer，性能损失 1-3%，回报巨大
  • 使用 bt -frame-arguments all、set print frame-arguments all
  • 反汇编 + 寄存器 ↔ 源代码：disas /m 同时显示源码与汇编
  • 变量 optimized out 时：用 x/g $rdi / $rsi 直接看 ABI 寄存器（System V x86-64：前 6 个整型/指针参数依次是 rdi rsi rdx rcx r8 r9）

GDB 自带 libstdc++ pretty printer，让 STL 容器显示成人类可读形式。配置：

4.2 Use-After-Free / Double Free SIGSEGV SIGABRT

❌ BAD
// UAF — 异步回调引用了局部对象
void startAsyncTask() {
    Worker w;
    threadPool_.post([&w] {
        w.run();           // w 已销毁
    });
}

// Double free — 同一裸指针被两个智能指针接管
Foo* p = new Foo();
std::shared_ptr<Foo> a(p);
std::shared_ptr<Foo> b(p);    // 两个独立计数!
// a 析构 → delete p
// b 析构 → delete p again → SIGABRT
✅ GOOD
void startAsyncTask() {
    auto w = std::make_shared<Worker>();
    threadPool_.post([w] {                  // 共享所有权
        w->run();
    });
}

// 永远只对一份 raw ptr 调一次 make_shared
auto a = std::make_shared<Foo>();
auto b = a;                              // 引用计数 = 2

// 自管对象需要 enable_shared_from_this
class Worker : public std::enable_shared_from_this<Worker> {
    void postSelf() {
        pool_.post([self = shared_from_this()]{
            self->run();
        });
    }
};
glibc 的 double-free 检测原理

glibc malloc (ptmalloc) 会在 free() 时检查：

  1. chunk 的 prev_inuse / size 字段是否合理
  2. 对 fastbin 范围的 chunk 检查 fastbin->fd != newchunk （防止刚 free 的 chunk 又 free）
  3. tcache 检查 chunk->key == tcache_key

触发后输出 double free or corruption (fasttop)、malloc(): invalid size 等信息后调用 abort() → SIGABRT。在 GDB 里：bt 会看到 __libc_message → abort → raise → __pthread_kill 链路。

❌ BAD — 栈溢出
void parseMsg(const char* in) {

// 堆溢出 — off-by-one 写

✅ GOOD

    std::array<char, 128> buf{};
    auto n = std::min(in.size(), buf.size() - 1);
    std::memcpy(buf.data(), in.data(), n);
    buf[n] = '\0';
}

std::vector<char> v(len);   // 自动管理边界
std::fill(v.begin(), v.end(), 0);

4.4 Data Race (并发竞争) SIGSEGV SIGABRT SIGBUS

❌ BAD
class Counter {
    int n_ = 0;
public:
    void inc() { ++n_; }      // 竞争！
    int get() { return n_; }
};

// 一锁两用，保护范围错位
void ReplyService::stop() {
    running_ = false;          // 没锁
    cv_.notify_all();
}
void ReplyService::wait() {
    std::unique_lock lk(mutex_);
    cv_.wait(lk, [&]{ return !running_; });  // 可能错过通知
}
✅ GOOD
class Counter {
    std::atomic<int> n_{0};
public:
    void inc() { n_.fetch_add(1, std::memory_order_relaxed); }
    int get() { return n_.load(); }
};

// condition_variable 三件套
void stop() {
    {
        std::lock_guard lk(mutex_);
        running_ = false;
    }
    cv_.notify_all();
}
void wait() {
    std::unique_lock lk(mutex_);
    cv_.wait(lk, [&]{ return !running_; });
}

6.4 Defensive nullptr Check 的边界

"到处判 null"会把代码变得肮脏且掩盖 bug。原则：

  • API 边界：外部传入的指针必须判（如本工作区 ReplyService(...session) 构造）
  • 内部约定：能用引用就用引用，能用 std::optional/std::variant 就别用裸指针
  • 智能指针：仍然要判，因为 shared_ptr<T> 也可能存空
  • 不变量：通过类型表达，例如 NotNull<T*> wrapper（见 GSL）

6.5 线程安全设计模式

  • Thread-confinement：对象只在一个线程访问，根本不需要锁
  • Immutable：构造后只读；读端无锁，写端用 copy-on-write 或换指针
  • Producer-Consumer + Queue：用 lock-free 队列，避免多锁交叉
  • 分层锁（如 ReplyService）：上层锁保证语义，下层锁保护字段
  • std::shared_mutex：读多写少；写时记得用 unique_lock
  • std::scoped_lock：一次锁多个 mutex，避免死锁（C++17）
  • 原子标志 + condition_variable：通知 + 唤醒检查；务必在持锁时修改标志，否则会丢失通知
// 一次锁两个，自动避免死锁（内部用 std::lock 算法）
void transfer(Account& a, Account& b, int v) {
    std::scoped_lock lk(a.m, b.m);
    a.bal -= v; b.bal += v;
}

SIGSEGV：访问的虚拟地址没有合法的 VMA 映射，或访问权限不符（如写只读段）

Q1：SIGSEGV 和 SIGBUS 有什么区别？

SIGSEGV：访问的虚拟地址没有合法的 VMA 映射，或访问权限不符（如写只读段）。最常见就是空指针/野指针解引用。

SIGBUS：虚拟地址有合法 VMA，但物理上无法满足。典型场景：
  1) mmap 文件后文件被 truncate，再访问超出文件长度的页面；
  2) 在 ARM/SPARC 等严格对齐架构上未对齐访问 (load 一个未 4 字节对齐的 int)；
  3) 写入磁盘已满的 mmap。

Q2：Core dump 文件里有哪些关键信息？为什么有时调试看到 ?? ??

Core 文件是 ELF Core 类型，包含 PT_NOTE（NT_PRSTATUS 寄存器、NT_PRPSINFO 进程信息、NT_SIGINFO 出错地址、NT_FILE mmap 表、NT_AUXV 辅助向量、NT_FPREGSET 浮点寄存器）和 PT_LOAD（按 coredump_filter 选择的内存段：栈、堆、数据、可选的代码段）。

看到 ?? ?? 通常因为：
  1) 二进制 / .so 不带调试符号（被 strip）；
  2) build-id 不匹配；
  3) 栈被破坏，回溯失败（bt 找不到 frame）；
  4) JIT / 动态生成代码（如 V8、JVM）。

Q3：怎么区分 "double free" 和 "use-after-free"？

都触发 SIGABRT，但报错文案不同：

  • double free：glibc 报 double free or corruption，在 free() 检查 fastbin/tcache 链表时发现刚刚释放过的 chunk。
  • use-after-free：glibc 不一定报错；表现为 SIGSEGV（地址被 unmap）、随机崩溃、数据错乱。要靠 ASan / Valgrind 才能精确定位。

定位首选 ASan：-fsanitize=address，会同时给出"分配栈 / 释放栈 / 访问栈"三条调用栈。

Q4：在多线程程序里 GDB 看到 main 线程在 abort，怎么找真正的肇事线程？

步骤：

  1. info threads 列所有线程
  2. thread apply all bt 拿到所有线程栈快照
  3. 在所有栈里搜：__cxa_throw、std::terminate、__assert_fail、__stack_chk_fail、raise、__libc_message、应用层 panic 函数
  4. 找到栈顶位于这些"陷阱函数"的线程，那就是肇事者
  5. 切过去 thread N，沿 frame 向上看业务代码

注意：很多场景肇事线程触发同步信号后异步通知了主线程 / signal handler 线程，主线程上看到的 abort 不是真正的源头。

Q5：std::shared_ptr 是线程安全的吗？

分两层：

  • 控制块（引用计数）：线程安全。多个线程对不同 shared_ptr 实例但指向同一对象的拷贝/赋值/析构是安全的。
  • 同一个 shared_ptr 实例：不是线程安全。两个线程同时对同一个对象 reset/赋值，需要外部同步，或者使用 std::atomic<std::shared_ptr<T>> (C++20) / std::atomic_load/store (旧)。
  • 指向的对象：和普通对象一样，需要业务自己保证线程安全。
Q6：noexcept 的真正含义是什么？写错有什么后果？

noexcept 是承诺该函数不会抛异常。如果违反：

  • 编译器可生成更优代码（不需要异常清理路径，move 操作可被容器优先选择）
  • 真抛了：直接调用 std::terminate → SIGABRT，不进行栈展开，析构函数也不跑

典型规则：

  • 析构函数默认 noexcept(true)，绝不要在析构里抛
  • 移动构造 / 移动赋值标 noexcept（否则 vector 扩容时会退化为 copy）
  • swap / 简单 getter 标 noexcept
Q7：Stack Overflow 在 Linux 上是怎么被检测到的？

Linux 在每个线程栈的最低端放一个或多个 guard page（PROT_NONE）。当栈增长触到这页：

  1. CPU 触发 page fault
  2. 内核 do_page_fault 检查 VMA 是 PROT_NONE
  3. 对栈段还会判断 expand_stack 是否能扩展；若已到上限则发 SIGSEGV

编译器加 -fstack-protector-strong 后，函数 prologue 在栈上放 canary，epilogue 检查；被破坏则调用 __stack_chk_fail → SIGABRT。

线程栈大小通过 pthread_attr_setstacksize 或 ulimit -s 调整。

Q8：什么时候应该用 std::atomic 而不是 mutex？
  • 单一标志位 / 计数器：用 atomic（如 running_、refcount）
  • 读多写少 / 单生产单消费：atomic 配合正确的 memory_order，无锁队列
  • 多字段需要保持一致：必须用 mutex，原子操作只能是单一变量
  • 需要等待事件：condition_variable + mutex；atomic 配 wait() (C++20) 也可以

误区：以为 atomic 一定比 mutex 快。实际上：高竞争下 mutex 让线程睡眠节省 CPU；atomic 自旋反而浪费。 总线锁 (LOCK 前缀) 也不便宜，约 10-100ns。

Q9：condition_variable 的 spurious wakeup 怎么处理？

POSIX 允许 pthread_cond_wait 在没有 notify 时也唤醒。所以必须在 while 循环中检查谓词：

std::unique_lock lk(m);
cv.wait(lk, [&]{ return ready; });   // 等价 while (!ready) cv.wait(lk);

另一个常见 bug：notify 之前没在持锁状态下修改谓词，导致 notify 在 wait 之前发生（lost wakeup）。正确做法：

{
    std::lock_guard lk(m);
    ready = true;          // 持锁修改
}
cv.notify_one();           // 释放锁后通知（可优化等待者唤醒后立刻拿锁）
Q10：生产环境进程偶发 crash，如何系统化排查？
  1. 取证：保存 core 文件 + 二进制 + 所有 .so + dmesg + journalctl + 业务日志（同一时间窗）
  2. 初判：file core、readelf -n core | grep PRSTATUS 看信号；dmesg 看 segfault 行
  3. 定栈：gdb + thread apply all bt；用 build-id 对齐符号；区分崩溃线程
  4. 分类：根据信号 + 栈顶函数划归 4.x 节中的某一类（空指针 / UAF / 越界 / race / 栈溢 / 异常）
  5. 复现：在测试机用 ASan/TSan 重新编译；输入回放 / fuzz / 压测
  6. 修复：明确根因后改代码；写回归用例
  7. 加固：在该路径加 contract 检查、metric、超时；添加 sanitizer CI；把 core_pattern + ulimit 检查加进部署 healthcheck

8.3 自带轻量 backtrace 的进程内 handler

// 当系统不允许写 core（容器、嵌入式）时，至少把栈写到日志
#include <execinfo.h>
#include <cxxabi.h>

extern "C" void crashHandler(int sig, siginfo_t* info, void* /*ctx*/) {
    void* frames[64];
    int n = backtrace(frames, 64);
    // async-signal-safe: 用 backtrace_symbols_fd 直接 write
    backtrace_symbols_fd(frames, n, STDERR_FILENO);
    dprintf(STDERR_FILENO, "sig=%d addr=%p\n", sig, info->si_addr);
    // 恢复默认动作并 re-raise，以便系统 core dump
    signal(sig, SIG_DFL);
    raise(sig);
}

void installCrashHandler() {
    struct sigaction sa{};
    sa.sa_flags     = SA_SIGINFO | SA_RESTART;
    sa.sa_sigaction = crashHandler;
    sigemptyset(&sa.sa_mask);
    for (int s : {SIGSEGV, SIGABRT, SIGBUS, SIGFPE, SIGILL})
        sigaction(s, &sa, nullptr);
}
handler 内部能调用的函数受限于 async-signal-safe 集合（man 7 signal-safety）：不能 malloc、不能 std::cout、不能持锁。backtrace_symbols 不安全（malloc），用 backtrace_symbols_fd。

while read addr; do
    addr2line -e ./myapp -f -i -C $addr
done < backtrace.txt

# 或用 llvm-symbolizer（更快，支持 split-dwarf）
llvm-symbolizer -obj=./myapp -inlining=true -demangle=true 0x4012ab

9.5 Stage 2：addr2line — 地址翻译到源码行

Stage 1 还原了符号名，但看不到源文件:行号，也找不到内联展开链。要做这一步，必须找到该构建产物对应的 NotStripped 二进制（含 DWARF 调试信息）。

为什么必须用 NotStripped 版本？

发布到现场的 ELF 通常被 strip 砍掉了下面这些节，体积小但没法做行号映射：

ELF 节用途strip 后
.dynsym动态链接所需的导出符号（最少集合）保留
.symtab所有内部/静态符号删除
.debug_infoDWARF：类型、变量、函数 DIE 树删除
.debug_line指令地址 ↔ (file,line,column) 状态机表删除
.debug_str调试用字符串池删除

怎么拿到 NotStripped 包

  • 通用 Linux：发行版的 -dbg / -debuginfo 包，或自家构建产物保留的 .debug 副本
  • build-id 索引：readelf -n binary | grep "Build ID" 拿到 hash，查 /usr/lib/debug/.build-id/<前2位>/<余下>.debug
  • 企业内部构建系统：
    • 从 issue tracker 取 build ID → 内部构建系统 → 找名字含 NotStripped 的 zip → 解压
    • 典型路径：拿应用 branch+revision → 到内部 artifact 服务器找对应 HW 架构 → 下载 myapp_dbg.txz → xz -d < myapp_dbg.txz | tar xvf -

用 addr2line 解码

# 单帧
addr2line -isfCe build/myapp_dbg 0x35158c
#  -e  指定带符号的可执行文件
#  -i  内联展开（关键！否则只看到最外层）
#  -s  只打 basename
#  -f  打印函数名
#  -C  自动 demangle

# 批量解码 myapp_exe 的所有偏移
grep -oE 'myapp_exe\+0x[0-9a-f]+' stack_demangled.txt \
  | sed 's/.*+//' \
  | while read off; do
      echo "== $off =="
      addr2line -isfCe build/myapp_dbg $off
    done

# 更快的替代品（支持 split-dwarf / DWP）
llvm-symbolizer -obj=build/myapp_dbg -inlining=true -demangle=true 0x35158c

读懂 addr2line 的输出

$ addr2line -isfCe build/myapp_dbg 0x35158c
operator()                              ← 内联帧 1：lambda 调用
function_template.hpp:759
??                                      ← 内联帧 2：编译器生成的辅助代码
tools.hpp:156                      ← 真正想要的源码位置

多行输出对应一个 VA 上多层内联展开的调用链，从顶（最内层）到底（最外层）。?? 行通常是匿名 lambda 的 operator() 或编译器合成函数，把它当作"中间桥梁"忽略即可，第一个有具体行号的源文件就是 bug 现场。

9.6 责任组件识别：剥离信号处理框架

调用栈的栈顶若干帧通常是信号捕获/异常抛出框架，它们是"症状"不是"病因"。要找责任方，必须先把这些框架帧剥掉。

常见的"信号陷阱函数"白名单（出现在栈顶就跳过）

  • libc：__libc_message, raise, abort, __pthread_kill, __assert_fail, __stack_chk_fail, signal trampoline
  • libstdc++：__cxa_throw, std::terminate, __verbose_terminate_handler
  • 内部 Crash 框架：error_exception_raise, linux_sig_handler, error_report_raise
  • 自家 crash handler / sentry / breakpad 钩子
责任分诊原则：跳过上述框架帧后，第一个属于应用层（你的可执行文件或业务库）的帧就是 bug 责任方。myapp 例中 #5~#10 全是 DeviceMgr*，所以结论 = "DeviceMgr is most suitable component to check further"。

9.7 端到端实操脚本

#!/usr/bin/env bash
# decode_crash.sh
#   用法: ./decode_crash.sh raw_syslog.txt build/myapp_dbg

set -euo pipefail
RAW=$1                            # 从设备拉来的 syslog 片段
DBG=$2                            # 已解压的 NotStripped 二进制

# 1) 清洗成纯净栈
sed -E 's/.*pid:[0-9]+\) //' "$RAW" > stack.txt

# 2) Stage 1：批量 demangle
c++filt -n < stack.txt > stack_demangled.txt

# 3) Stage 2：解 myapp_exe 内的偏移到 file:line
EXE_NAME=$(basename "$DBG" | sed 's/dbg$//')
echo "=== addr2line for $EXE_NAME ===" > resolved.txt
grep -oE "${EXE_NAME}\+0x[0-9a-f]+" stack_demangled.txt \
  | sort -u \
  | while read tag; do
      OFF=${tag##*+}
      printf '\n--- %s ---\n' "$tag"
      addr2line -isfCe "$DBG" "$OFF"
    done >> resolved.txt

# 4) 打印责任分诊建议
echo
echo "== 应用层第一帧（=责任组件） =="
awk '/^#/ { if ($0 !~ /libcrash|libc\.so|libapp|libcore|__cxa|terminate|abort|raise/) { print; exit } }' stack_demangled.txt

9.8 跨架构 Cross-Debug 配置（MIPS / PowerPC / ARM Core 在 x86 上调）

嵌入式 / 工业现场设备常见 MIPS、PowerPC、ARM 等架构，但开发机几乎都是 x86_64。直接拿本机 gdb 打开非 x86 的 core 会失败——必须用架构匹配的 GDB + 目标 sysroot + 平台库搜索路径。这就是 cross-debug。

9.8.1 凑齐 4 件套

组件作用典型来源
业务二进制 + .debug 符号定位自家代码的函数名 / 行号内部构建系统上对应 build 的 NotStripped 包
平台层 .so（如 libcrash）解析栈中 error_exception_raise 等框架帧SDK 的 C_Platform/crash_fw/lib/<arch>/

架构匹配的 gdb读取目标架构的寄存器/指令SDK 的 dbg-tools/.../bin/<arch>-gdb
同源码二次编译，二进制不一致！
即使源码、编译器、依赖完全相同，两次构建的产物 build-id 也不同（含时间戳、构建者、临时路径等元数据）。debug 符号必须严格来自跑出 core 的那一次构建，否则地址 → 行号映射会错位且不会报错。

金标准：第 N 帧函数的源码里应该能看到对第 N-1 帧函数的调用点。

  • 如果 demangled 出来全是乱码 → 架构端序错了 (用 little-endian gdb 读 big-endian core)

9.9 Crash & Signal Handler 日志字段解读

典型的公司内部 Crash & Signal Handler 框架，在崩溃时会用统一格式打印一段 "exception report" 到 syslog。读懂这些字段能在不解码栈的情况下就完成初步分诊。典型样例：

实战诊断模式

模式 1：description 含 Abort signal received + 栈里有 __libc_malloc / _Znwm

典型链路：operator new → __libc_malloc → glibc 检测到堆元数据破坏 → abort → crash handler 捕获

#5  gsignal              ← libc 信号转发
#6  abort                ← 谁调了 abort？
#10 __libc_malloc        ← ★ malloc 在分配时检测到堆破坏
#11 _Znwm = operator new
#12 std::_Rb_tree::...   ← C++ 容器在分配节点
#15 DBObject::addAttribute
#17 BS::cloneObject(bool)  ← ★ 真正业务入口

模式 2：description 含 Shutdown in progress

这个崩溃是关机过程中的副作用，不是关机的原因。往 syslog 上方搜真正的关机触发：

  • Reset request received
  • Watchdog expired
  • 管理面主动下发 reboot RPC
  • cgroup OOM / kernel panic
模式 3：description 含 please contact DCS/URI/HWR/...

多域二进制（myapp_exe / mymodule_main）的 框架已经做过自动分诊，直接按提示转给对应团队即可。最多用 c++filt 把栈解可读，不必下 NotStripped 包做 addr2line。

9.11 边界情况与陷阱

  • 同名异版 .so：现场 libcore.so.1 与本地下载的 build 不同，偏移会错位。务必按 build-id 严格匹配（readelf -n 或 eu-unstrip）。
  • 同源码二次编译：见 §9.8.1 警告——build-id 必然不同，必须用产生 core 的那次构建产物。
  • arm 架构 thumb 位：ARM 偶函数地址末位是 1（thumb 标志），addr2line 时需 off & ~1，否则查不到。
  • 端序错误：用小端 gdb 读大端 core，demangled 出来全是乱码或栈深度异常——回 §9.8.2 重新选 gdb。
  • 位置无关 vs 静态链接：可执行文件如果是 PIE，输出里的"myapp_exe+0x..." 已经是文件偏移，可直接用；非 PIE 二进制偶尔需要先减去 .text 起始（罕见）。
  • debug 文件分离：现代发行版常用 --only-keep-debug 把符号拆到 xxx.debug 单文件 + 主二进制里塞 .gnu_debuglink。addr2line -e 直接给 .debug 文件即可。
  • GDB 命令顺序错：先 core-file 再设 sysroot 会让一半栈显示 ??——见 §9.8.4 SVG。
  • 栈被破坏导致回溯到一半就乱：表现为后面帧地址明显不在任何 .so 范围内、或者 mangled 名乱码。这种情况只能配合现场 core 用 x/gx $rsp 手动还原，离线日志救不了。
  • JIT / 动态生成代码：地址不属于任何 ELF，addr2line 必然 ??。需要 perfmap 或运行时 unwinder 配合。

10.1 rr — Mozilla 的"录制即调试器"，复现 race 神器

rr（"record and replay"）由 Mozilla 出品，是 Firefox / Linux 内核开发者的最爱。它先录制一次进程的全部执行（含线程调度、信号、syscall 返回），然后允许你反复确定性回放，甚至反向单步（reverse-step / reverse-continue）。

核心使用

# 1. 安装 (Linux 5.x+)，准备内核参数
sudo apt install rr
sudo sysctl -w kernel.perf_event_paranoid=1

# 2. 录制 — 跑一次问题进程（产生 trace 目录）
rr record ./myapp problematic_input.bin
# 命中 crash → 自动保存 trace

# 3. 回放 — 进入 GDB 风格的交互
rr replay
(rr) c           # run 到 crash
(rr) bt          # 看 backtrace
(rr) reverse-step # ★ 反向单步！
(rr) reverse-continue # ★ 反向跑到上一个断点
(rr) watch -l ptr # 反向 watchpoint
(rr) reverse-cont # 自动找到 ptr 最后一次被修改的位置

典型杀手场景

场景没 rr 怎么调有 rr
偶发 race，难复现加日志，等几天，可能再也不出现录一次复现 → 100% 反复回放
UAF：某指针突然指向坏内存printf 把所有写操作追踪一遍watch -l ptr; reverse-continue 直接到上次写点
"为什么这个变量是 0?"逐函数加日志reverse-cont 到最后一次赋值
多线程死锁难以复现录一次，反复研究调度顺序
rr 的开销大约 1.2-2× 慢，且只支持单线程或有限并行（默认序列化所有线程到单核运行）。生产服务量级一般用不上，但开发期复现一个 bug 后就再也别让它跑掉。

rr + Chaos Mode（专门构造 race）

# 强制随机化线程调度，提高 race 出现概率
rr record --chaos ./myapp
# 跑 50 次：
for i in {1..50}; do
  rr record --chaos ./myapp || break      # 命中后保留 trace
done

10.3 GDB Python Pretty-Printer — 把 STL 输出变人话

裸 GDB 打印 std::vector<int> 会显示内部 _M_start / _M_finish / _M_end_of_storage 指针 —— 完全没法看。Pretty-printer 是 GDB Python 扩展，把容器输出成"人可读"形式。

启用标准库 pretty-printer（libstdc++ / libc++ 都自带）

# 编辑 ~/.gdbinit
python
import sys
sys.path.insert(0, '/usr/share/gcc-12/python')
from libstdcxx.v6.printers import register_libstdcxx_printers
register_libstdcxx_printers(None)
end

# 现在 print std::vector<int>{1,2,3} 显示:
#   $1 = std::vector of length 3, capacity 4 = {1, 2, 3}
# 而不是:
#   $1 = {<std::_Vector_base> = {_M_impl = {_M_start = 0x...,
#         _M_finish = 0x..., _M_end_of_storage = 0x...}}}

给自家类型写 pretty-printer

# my_printers.py
class RingBufferPrinter:
    def __init__(self, val):
        self.val = val

    def to_string(self):
        head = int(self.val['head_'])
        tail = int(self.val['tail_'])
        cap  = int(self.val['cap_'])
        size = (tail - head) % cap
        return f'RingBuffer(size={size}, cap={cap}, head={head})'

    def children(self):
        # 让 print 显示元素
        cap = int(self.val['cap_'])
        buf = self.val['buf_']
        for i in range(int(self.val['head_']), int(self.val['tail_'])):
            yield f'[{i}]', buf[i % cap]

def my_lookup(val):
    if str(val.type) == 'RingBuffer<int>':
        return RingBufferPrinter(val)
    return None

gdb.pretty_printers.append(my_lookup)
# .gdbinit 加载
source ~/proj/.gdb/my_printers.py

# 现在
(gdb) p my_buffer
$1 = RingBuffer(size=3, cap=64, head=10) = {[10] = 42, [11] = 99, [12] = 7}
大型 C++ 项目应该把 pretty-printer 当作代码一部分：放 .gdb/ 目录，CI 强制跑，复杂数据结构（自家 HashMap / SkipList / Ringbuffer / Variant）都有 pretty-printer，所有团队成员调试同一种数据看到的输出一致。

10.4 bpftrace / eBPF — 生产环境无侵入观测 ⭐⭐⭐

eBPF 是 Linux 内核里的一个虚拟机，能动态注入小程序到几乎任何内核 / 用户态函数入口/出口。bpftrace 是它的 awk 风格脚本前端。

对 C++ 工程的革命性：你不再需要为了观测某行为而重启 / 加日志 / 改代码。直接在正在跑的生产进程上挂探针，看每一次函数调用、参数、返回值、延迟分布。

核心用法（一行命令解决问题）

# 1. 统计每秒 read() syscall 的次数和延迟分布
sudo bpftrace -e '
  tracepoint:syscalls:sys_enter_read { @start[tid] = nsecs; }
  tracepoint:syscalls:sys_exit_read /@start[tid]/ {
    @lat = hist((nsecs - @start[tid]) / 1000);
    delete(@start[tid]);
  }
  interval:s:1 { print(@lat); clear(@lat); }
'
# 实时输出 us 级延迟直方图，零侵入

# 2. 跟踪某进程内某 C++ 函数（用 uprobe）
sudo bpftrace -e '
  uprobe:/path/to/myapp:_ZN6Server13handleRequestEv {
    @[ustack(5)] = count();   /* 调用栈聚合 */
  }
  interval:s:5 { print(@); clear(@); }
'

# 3. 抓所有打开的文件（不需 strace 那样停整个进程）
sudo bpftrace -e '
  tracepoint:syscalls:sys_enter_openat {
    printf("%-6d %-16s %s\n", pid, comm, str(args->filename));
  }
'

# 4. 谁在执行 send 系统调用 + 字节数
sudo bpftrace -e '
  tracepoint:syscalls:sys_enter_sendto {
    @bytes[comm] = sum(args->len);
  }
'
