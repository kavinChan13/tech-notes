01是什么 / 为什么

AI Feature PRD 是新 AI 能力的立项与验收基线。它比传统 PRD 多承担一件事：把不确定性写清楚 — 不是所有 AI feature 都能做到产品想要的质量，"做不到"也是有效结论。

传统 PRD 假设「定义清晰 → 实现可达成」。AI PRD 必须额外回答：怎么测达标、错了怎么办、花多少钱、有什么合规风险。

AI PRD 的本质：把不确定性透明化。没有 Eval / Failure mode / Cost / Risk 四段，等于没有验收标准。

02字段说明

段 / 字段为什么存在 / 怎么用
1. Problem & User解决谁的什么痛点；没有用户场景 = 技术 demo
2. Goal & Non-goalNon-goal 必写 — AI feature scope 膨胀最快
3. Solution overview一句话方案 + 架构图；RAG / Agent / API 模式要说清
4. UX flow必须设计 AI thinking / failed / empty 三种状态
5. 🤖 AI Eval planAI 专属必填 — 指标 / eval set / 验收阈值 / 谁跑 eval
6. 🤖 Failure modesAI 专属必填 — 至少 5 个失败场景 + UX 兜底
7. 🤖 Cost modelAI 专属必填 — per-request cost + 规模化总账 + cost/收入比
8. 🤖 Risk & SafetyAI 专属必填，法务必看 — PII / 幻觉 / 越狱 / 监管
9. Rollout plan灰度阶段 + Abandon criteria（什么数据下砍掉 feature）
Insight 读者看完 PRD 应知道：这个 feature 90% 概率能做到 80% 质量，10% 概率根本做不到 — 以及做不到时怎么办。

MARKDOWN模板
# [Feature Name] · AI Feature PRD
作者：[Your Name]    审阅：[EM/Tech Lead/Legal]    日期：[YYYY-MM-DD]    状态：Draft / In Review / Approved
## 1. Problem & User
- [必填] 解决谁的什么痛点：[一句话]
- 当前用户怎么 work around：[1-2 句]
- 痛点的 frequency / severity：[数据 or 调研]
## 2. Goal & Non-goal
- Goals (本期): [3 个具体 measurable 目标]
- Non-goals: [明确本期不做的 3 件事，避免 scope creep]
- Success criteria: [business metric + AI metric 两组]
## 3. Solution overview
- 一句话方案：[xxx + LLM + xxx 用什么模式]
- 架构图：[贴图，前端 → API → LLM 调用 → 后处理 → 响应]
- 跟现有系统关系：[1-2 段]
## 4. UX flow
- 主流程：[step 1 → 2 → 3]
- 关键界面 mockup：[贴图]
- Loading / Error / Empty state 设计：[特别要说明 "AI thinking" / "AI failed" / "no result" 三种状态]
## 5. 🤖 AI Eval plan (必填)
- 指标: [choose: accuracy / faithfulness / recall@K / human-rating / 用户满意度 等]
- Eval set: [多少 case / 怎么 curate / 谁标注 / 多久更新]
- 验收阈值: [必须达到 X% 才能上线，低于 Y% 直接 kill]
- 在线监控指标：[user feedback / abandon rate / regen rate]
- 谁负责跑 eval：[人 + 频率]
## 6. 🤖 Failure modes (必填)
- 5 个典型失败场景 + UX 兜底：
  1. [场景，如 "查不到答案"] → [兜底，如 "fallback 人工客服 链接"]
  2. [幻觉] → [加 disclaimer + "举报错误"按钮]
  3. [超时 > 10s] → [显示 progress + 允许取消]
  4. [输出被 safety filter 拦截] → ["内容受限，请换说法"]
  5. [输入越狱/恶意] → [识别 + log + neutral 回复]
## 7. 🤖 Cost model (必填)
- 单次请求 cost：[模型: $X/M tokens × tokens/请求 = $Y/请求]
- 预期 DAU × 平均 N 请求/天 = $Z/天 = $Z×30/月 = $A/月
- Cost / 收入比：[%，必须 < 25% 否则商业不可持续]
- Cost optimization 计划：[Phase 1 prompt 优化 / Phase 2 caching / Phase 3 量化或自部署]
## 8. 🤖 Risk & Safety (必填，法务必看)
- PII 流转: 用户 PII 是否进入第三方 LLM？[Yes/No 必答] → 如 Yes 需要 [DPA / zero-retention / 合规审查]
- 幻觉: 高 stakes 场景吗？[如医疗/法律/金融需要 disclaimer + 人工兜底]
- Prompt injection: 用户输入是否会触发 system prompt 暴露？[input sanitization 方案]
- IP / 版权: 是否会生成被版权保护的内容？[过滤或免责]
- 监管: 国内生成式 AI 备案 / EU AI Act 适用性？[跟法务 align]
## 9. Rollout plan
- Phase 1 (内测)：[%用户 + 持续 X 天 + 看 eval]
- Phase 2 (灰度)：[%用户 + 看在线 metric 是否 align offline eval]
- Phase 3 (全量)：[判断标准]
- Abandon criteria: [什么数据下我们决定砍掉这个 feature，不是改进 — 而是放弃]
- Monitoring dashboard 链接：[URL]

填好示例 · AI 客服 FAQ Bot

企业 SaaS 公司给客户做 AI 客服

# Customer Support AI Bot · AI Feature PRD
作者：J.Lee    审阅：EM, Legal, Head of Support    日期：2026-03-15    状态：In Review

## 1. Problem & User
- 支持团队每天回 ~800 条 ticket，60% 是常见问题（密码 / 计费 / 配置）
- 当前用户平均等 25 min · 一线客服 80% 时间花在重复问题，自己也烦
- 调研：62% 客户表示"如果能即时拿到答案，比慢慢等人工更好"

## 2. Goal & Non-goal
- Goals: ① 50% 常见问题 24/7 即时答 ② 一线客服 capacity 释放 30% ③ CSAT > 4.0/5
- Non-goals: 不做"复杂排障"（如 API 集成问题）/ 不做销售线索引导
- Success: 上线 30 天后 deflection rate > 40% + CSAT not worse than 人工

## 3. Solution overview
- RAG over 1500 篇 Help Center 文章 + 6 个月历史 ticket 资料
- 架构：用户提问 → embed → 向量搜 → top-5 chunks → GPT-4o mini 答 + 引用源
- 接现有 Intercom widget，failure 时一键转人工

## 4. UX flow
- 用户输入 → 显示 "AI 正在查阅 X 篇文档" → 答案 + 源链接 + "有用/没用"按钮
- 没用 → 自动转人工 + 把对话 context 传过去（不让用户重述）
- Mockup: [Figma 链接]

## 5. 🤖 AI Eval plan
- 指标：faithfulness (LLM-Judge) · answer relevance · citation accuracy · 召回率
- Eval set: 200 个真实 ticket（覆盖 8 类问题）· Support 团队标注 "正确答案" · 每月 +20 case
- 验收阈值：faithfulness > 90% + citation accuracy > 95% + 召回 > 75%
- 在线监控：thumbs up/down · 转人工 rate · 平均对话轮数
- 跑 eval：QA 团队每周一次 (~30 min) · 灰度期间每周一次

## 6. 🤖 Failure modes
1. 答不出（无相关文档）→ "我没找到答案，正在转人工" + 转人工
2. 答错（幻觉）→ 引用源永远显示 + "举报答案错误"按钮 + 反馈进 eval set
3. 超时（> 8s）→ Loading 动画 + "查询时间较长，请稍候"
4. Safety 拦截（用户骂人 / 试图越狱）→ 中性回复 + log
5. 用户连问 3 次类似问题（说明 AI 没真懂）→ 自动转人工

## 7. 🤖 Cost model
- GPT-4o mini ($0.15/M input + $0.60/M output) · 平均 5K input + 200 output / query
- per query cost = (5000×0.15 + 200×0.60) / 1M = $0.00087
- 估 DAU 2000 × 平均 3 query = 6000 query/day · = $5.22/天 · = $156/月
- Cost / 收入比：customer 月费 $30 × 60% deflection × 100 customer = $1800 saved support cost / 月 → cost ratio < 10% ✅
- 优化：Phase 2 加 prompt caching (省 30%) · Phase 3 高频问题 cache (省额外 20%)

## 8. 🤖 Risk & Safety
- PII 流转: ticket 中可能有 email/account ID → 我们 redact 后再传 LLM · DPA 已签 (OpenAI Enterprise + zero-retention)
- 幻觉: 中度风险（不是医疗 / 法律），但商业语境会影响信任 → 强制引用源 + 鼓励反馈
- Prompt injection: 用户输入做 input filter (常见 jailbreak pattern) · system prompt 严格分层
- IP / 版权: 答案来源都是自家 Help Center · 不涉及第三方版权
- 监管: 国内备案不适用（B2B 海外）· GDPR 已对齐

## 9. Rollout plan
- Phase 1 (内测，week 1-2): 内部 50 人 · pass eval threshold 才进入 Phase 2
- Phase 2 (灰度，week 3-6): 5% → 20% → 50% customers · 每周对比 deflection / CSAT
- Phase 3 (全量，week 7+): 100% · 持续 monitor
- Abandon criteria: 若灰度 4 周 CSAT < 3.5 或 deflection < 25% → 暂停 + 重新评估
- Dashboard: [Looker 链接]

DO / DON'T

DO

  • 第 5/6/7/8 段（AI 4 段）必填 + 用数字
  • "Failure modes" 段写至少 5 个失败场景 + UX 兜底
  • "Cost model" 段算出 unit economics + cost ratio
  • "Abandon criteria" 必须有 — 给团队心理预期
  • 法务必须 review 第 8 段 (Risk)，签字才上线
  • 每次 PRD review 前先跑一次 eval，用数据说话

DON'T

  • "我们用了 LLM 所以一定 work" — 没有 eval = 没有验收
  • "AI 应该不会出错" — AI 必然出错，没 failure mode 段就上线 = 灾难
  • "Cost 可以以后再算" — 上线后才发现 unit economics 亏钱
  • "Safety 让 SRE 看" — 你 PM 必须 own 这段，不能甩锅
  • 没 "Non-goals" + 没 "Abandon criteria" = scope 必定膨胀 + 砍不掉
  • 用 "GPT-4 应该能搞定" 当方案 — 必须有具体的 eval 数据

Insight AI PRD 的最大价值不是"定义做什么"，是"定义不做什么 + 失败什么样 + 失败怎么办"。这 3 点写好，团队 90% 的争论都没了。
