为什么 unit economics 是 AI 团队生死线

传统 SaaS：cost 跟用户数对数关系（marginal cost 接近 0）。AI SaaS：cost 跟用户数线性关系（每个 query 都要付 GPU 时间）。这是 AI 商业模式的核心挑战 — 不算 unit economics = 不知道哪天会亏死。

0.1 三个 reality check 问题

问你自己（或你的 AI PM / EM）：

  1. "我们每个 query 多少钱？" — 答不上 = 没做 unit economics
  2. "我们每个用户每月 cost / revenue 比例？" — 答 "感觉 OK" = 没真算过
  3. "用户量翻 10x 时 cost 翻多少倍？" — 答 "应该 10 倍吧" = 没考虑 caching / scaling 效应
Insight 最快揭示 AI 团队 cost awareness 水平的问题："我们的 gross margin per query 是多少？" — 大多数 AI 团队答不上。能秒答的 EM 是稀缺品。

0.2 AI Cost 跟传统 SaaS 的 3 个本质区别

维度传统 SaaSAI SaaS
Marginal cost~$0 / user$X / query × N query/user/month
Cost predictability高（固定 infra）低（vendor 涨价 / 模型升级 / context 长度波动）
Unit economics 是否能转正规模化后必然转正不一定 — 需要主动 cost optimization

3 种 cost 结构

① 纯 API (OpenAI / Anthropic / 通义 / Doubao)

谁适合：早期 POC / 小流量 / 不想搞 infra

Cost 结构：
API_token_cost × tokens/req × N_req

  • + 简单 · 起步快 · 0 infra
  • + vendor 帮你 scale
  • - per-token unit cost 最贵
  • - 大流量后 cost runaway
  • - Vendor lock-in / 数据合规

② 自部署 (vLLM / SGLang / TRT-LLM)

谁适合：大流量 / 合规要求 / 团队有 AI Infra 能力

Cost 结构：
GPU_cost/hr × hours × N_replicas / queries_served

  • + per-token cost 砍 50-70%
  • + 数据完全自控
  • + 可深度优化 (量化 / batching)
  • - 需要 AI Infra 团队
  • - 低流量时 utilization 低 = 单位 cost 反而高

③ 混合 (Routing)

谁适合：成熟团队 / 多种 task 类型

Cost 结构：
分流到 API / 自部署 / 小模型，按 task 性质

  • + 最优 cost (按需选 cheap / expensive)
  • + 兜底能力强（vendor 挂可切自部署）
  • - 复杂度高 · 需要 routing logic + eval
  • - 需要分类器决定 query → which model

1.1 3 种结构的"何时切换" 决策框架

核心公式 + 2026 Pricing 参考

2.1 核心公式（必会）

MARKDOWN模板
## API 模式
per_request_cost = (input_tokens × input_price + output_tokens × output_price) / 1M

## 自部署模式
per_request_cost = (gpu_hourly_cost × replica_count) / (qps × 3600)
其中：qps = throughput per replica（受 batch / quantization 影响）

## 月度总 cost
monthly_cost = per_request_cost × DAU × avg_requests_per_user × 30

## Unit Economics
cost_per_user_per_month = per_request_cost × avg_requests_per_user × 30
gross_margin_per_user = revenue_per_user - cost_per_user_per_month
gross_margin_ratio = gross_margin_per_user / revenue_per_user

## ROI of cost optimization
monthly_savings = (old_cost - new_cost) × DAU × avg_req × 30
payback_period = engineering_cost / monthly_savings

8 个 cost 优化手段（按 ROI 排序）

  1. Prompt 优化（最高 ROI） · 砍掉冗余 system prompt + few-shot · 典型省 20-40% input tokens
  2. Prefix caching · 同 system prompt 重复 query · OpenAI / Anthropic 都支持 cache 50% off · 典型省 30-50%
  3. Model routing · 简单 query 走便宜 model (mini / Haiku / Flash) · 复杂走 frontier · 典型省 40-60%
  4. Output 限制 · 设 max_tokens · structured output · 砍 output 长度 · 典型省 10-30%
  5. RAG chunk 优化 · 别一次塞 20 个 chunk · 加 rerank 留 top-3 · 典型省 30-50% input tokens
  6. 切自部署 + 量化 · INT4 + vLLM · 大流量后才划算 · per-req cost 砍 70-90%
  7. Batch API · 非实时任务（数据处理 / eval）用 OpenAI Batch API · 省 50%
  8. Vendor 谈判 · 月 spend > $10k 后 negotiate enterprise pricing + zero retention · 典型省 10-30%
Insight 优化顺序：先做前 4 个（应用层）· 流量起来再考虑切 infra（5-8）。前 4 个 1-2 周可见效，5-8 需要 1-3 月工程投入。

完整示例 · AI 客服 SaaS Cost Model

企业 SaaS 公司 (B2B SaaS, $50/customer/月)

# Customer Support AI SaaS · Cost Model

## Sheet 1: Assumptions
| Variable | Value | Source |
|---|---|---|
| Active customers | 200 (current) → 2000 (12mo target) | Sales pipeline |
| End users / customer | 50 avg | Usage analytics |
| AI queries / end-user / day | 2.5 | Product analytics (last 30 days) |
| Avg input tokens / query | 1800 (含 RAG context) | Sample 200 production reqs |
| Avg output tokens / query | 250 | Sample 200 production reqs |
| Revenue / customer / month | $50 | SaaS pricing |
| Other infra (DB / hosting) | $0.50 / customer / month | Finance |

## Sheet 2: Per-Request Cost (3 scenarios)

### Scenario A · Pure API (gpt-4o-mini)
input_cost  = 1800 × $0.15 / 1M = $0.000270
output_cost = 250  × $0.60 / 1M = $0.000150
per_req_cost = $0.000420

### Scenario B · Self-host Llama-70B INT4 (vLLM)
- 假设 1 个 H100 replica / 月 cost = $2.5/hr × 720 hr = $1800
- 实测 throughput: 200 QPS (INT4 + batched)
- per_req_cost = $1800 / (200 × 3600 × 24 × 30) = $1800 / 518M reqs/month = $0.00000347 ← 极便宜
- BUT: 需要 utilization 高才划算。低流量时 idle GPU 烧钱

### Scenario C · Hybrid (70% mini, 30% Llama via routing)
- 简单 query (70%) → gpt-4o-mini @ $0.000420 each
- 复杂 query (30%) → 自部署 Llama @ $0.00000347 each
- weighted per_req_cost = 0.7 × 0.000420 + 0.3 × 0.00000347 = $0.000295

## Sheet 3: Monthly Forecast

### Current (200 customers)
- queries/day = 200 × 50 × 2.5 = 25,000
- queries/month = 750,000

| | API Only | Self-host | Hybrid |
|---|---|---|---|
| Per-req | $0.000420 | $0.00000347 | $0.000295 |
| Monthly cost | $315 | $1,800* | $221 |
| *Note: Self-host idle - 不划算（流量太低）|

### Target (2000 customers, 12mo)
- queries/day = 2000 × 50 × 2.5 = 250,000
- queries/month = 7.5M

| | API Only | Self-host | Hybrid |
|---|---|---|---|
| Per-req | $0.000420 | $0.00000347 | $0.000295 |
| Monthly cost | $3,150 | $1,800 | $2,212 |
| Annual cost | $37,800 | $21,600 | $26,544 |

→ 10x DAU 后自部署最优（cost 不随用户量线性涨 · 因为 GPU 已经在那）
→ Hybrid 介于中间，运维更简单

## Sheet 4: Unit Economics

### Current (200 customers, hybrid)
- Revenue / customer / month = $50
- AI cost / customer / month = $221 / 200 = $1.11
- Other infra / customer / month = $0.50
- Gross margin = $50 - $1.11 - $0.50 = $48.39 (96.8%) ✅

### Target (2000 customers, hybrid)
- Revenue / customer / month = $50
- AI cost / customer / month = $2,212 / 2000 = $1.11 (same per-customer)
- Other infra / customer / month = $0.50
- Gross margin = 96.8% ✅

### Sensitivity check
- 如果 vendor 涨价 30%：AI cost +$0.33/customer · GM 跌 0.7% → 仍 acceptable
- 如果重度用户 query × 5 (= 12.5 queries/user/day)：cost ×5 = $5.55/customer · GM 87% → 仍 OK 但要 watch
- Break-even (cost = revenue)：query/user/day = ~110 · 极小概率达到

## Conclusion
- 现在 API Only 也 acceptable (cost 微不足道)
- DAU 涨 5x 后切 Hybrid (cost 减少 30% + 风险分散)
- DAU 涨 20x 后纯自部署 (cost 减少 50% + utilization 够)
- 主要 cost risk 不是 LLM cost · 是 input token 膨胀（context 一旦长 30%，cost 涨 30%）

DO / DON'T

DO

  • 每月主动给 CFO / Finance 发 1-page cost dashboard
  • per-feature unit economics 必算（不只 total cost）
  • 建 sensitivity table（DAU × 2 / 5 / 10 时的 cost）
  • 每个新 AI feature 必估 cost (PRD §7 必填)
  • Cost dashboard 加 alert（超出预算 X% 自动 ping EM）
  • Vendor pricing 改动后 24 hr 内更新 model
  • Cost optimization 跟 OKR 挂钩（不要做完没人记）

DON'T

  • 只看 total cost 不算 per-user · 业务一涨就翻车
  • 切自部署因为"便宜" · 没算 idle GPU 烧的钱
  • 盲信 vendor benchmark · 实测 per-req cost 才数
  • cost 跟 quality 完全分开优化 · 必须 trade-off
  • Cost 报告只给 Finance 不给团队 · 团队没 cost awareness
  • 季度才看一次 cost · 月度才能 early warning
  • "自部署一定省钱" · 流量不够时反而更贵
