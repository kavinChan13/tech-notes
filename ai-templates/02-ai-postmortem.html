01是什么 / 为什么

AI Postmortem 是AI 翻车专属 RCA 文档。传统 incident 假设「有明确 bug + 明确修法」；AI 翻车经常是数据漂移、模型升级、prompt 边界 — 代码一行都没改也会挂。

比传统 postmortem 多 Data / Model / Prompt 三维归因。「AI 偶尔会这样」不是 RCA，是放弃。

AI postmortem 必须给出 hypothesis 并设计可验证实验 — 否则 3 个月后再撞同一面墙。

Root Cause · ModelAI 专属 — vendor 升级 / 自部署版本 / 量化变化
Root Cause · PromptAI 专属 — 越狱 / 边界输入 / system prompt 改动

事故 #1 · 模型升级回归

vendor 把 GPT-4o 静默升级，召回率掉了 15 个点

触发

OpenAI 升级 gpt-4o pinned version（没通知）

影响

~40% 用户 query 答案质量降低 · 持续 48 hr

严重度

Sev-2（无 outage 但 CSAT 跌 0.5 分）

Timeline 关键节点
  1. T+0：OpenAI 升级（团队没收到通知）
  2. T+18 hr：CSAT dashboard 显示当日 4.1 → 3.6（异常）
  3. T+24 hr：on-call 调查 — 排除流量异常 / 代码改动 / 用户分布
  4. T+30 hr：跑 regression eval → 发现 faithfulness 从 92% → 76%（明显回归）
  5. T+36 hr：发现 OpenAI status page "we updated gpt-4o-2024-08-06"（隐藏在 changelog）
  6. T+40 hr：rollback 到 gpt-4o-2024-05-13（pin 老版本）
  7. T+48 hr：CSAT 恢复 4.0
5 维归因
  1. Code: ❌ 无改动
  2. System: ❌ 无 outage
  3. Data: ❌ eval set 没变
  4. Model: ✅ 主因 — vendor 静默升级
  5. Prompt: ❌ 无改动
Action Items
  1. P0: 所有 model 调用强制 pin 具体版本号（gpt-4o-2024-05-13） · DRI: AI Infra · 1 周
  2. P0: 建立 auto regression eval（每天跑 100 case · 异常告警） · DRI: ML · 2 周
  3. P1: 订阅 vendor changelog · 评估每次升级 · DRI: PM · 持续
  4. P2: 团队培训 "model versioning policy" · DRI: EM · 1 个月
永远 pin model 版本，永远跑 regression eval。"使用最新 model" 是给业余开发者的，生产系统必须 vendor 升级 = 你主动 opt-in 的决策，不是被动接受。

事故 #2 · RAG 召回掉点

升级 embedding 模型后，召回率从 80% 降到 60%

触发

团队把 embedding 从 text-embedding-ada-002 升到 text-embedding-3-large

影响

RAG 问答正确率 89% → 74% · 持续 5 天才发现

严重度

Sev-2

Timeline 关键节点
  1. Day 0：团队为"性能提升"升级 embedding（看的是 OpenAI 官方 benchmark）
  2. Day 0-3：重 embed 整个 knowledge base · 部署
  3. Day 4：客户支持反馈 "AI 经常说不知道"
  4. Day 5：跑 RAGAS eval 发现 recall@5 从 80% → 60% · 但 OpenAI benchmark 上 3-large 应该更好
  5. Day 6：分析发现 — text-embedding-3-large 维度 3072 vs ada-002 1536，向量空间结构不同，原 query 跟新 chunk embedding "分布漂移" · 同时部分 query 类型反而退步（短 query 性能差）
  6. Day 7：rollback ada-002 · 召回恢复
5 维归因
  1. Code: ❌ 仅 embedding model name 换了
  2. System: ❌
  3. Data: ✅ 主因 — embedding 分布漂移
  4. Model: ✅ 副因 — embedding model 不只是"质量等级"差异，结构也不同
  5. Prompt: ❌
Action Items
  1. P0: 任何 embedding / model 升级 必须先在 staging 跑 RAGAS · DRI: ML · 立刻
  2. P0: 建立"召回率"实时监控 + 阈值告警 · DRI: AI Infra · 1 周
  3. P1: 收集真实 user query 加入 eval set（不只 synthetic） · DRI: ML · 1 个月
vendor benchmark ≠ 你的 use case 表现。Embedding 升级必须用你自己的 eval set 验证，不能信公开 benchmark。"3-large 比 ada 好"是平均，对你的 domain 可能反而差。

事故 #3 · Prompt Injection 攻击

用户在 query 里加 "ignore all previous instructions"，bot 开始泄漏 system prompt

触发

X (Twitter) 上一个安全研究者发了 PoC video

影响

system prompt + 部分 RAG knowledge 被泄漏 · 病毒传播 12 hr 内 50k+ 阅读

严重度

Sev-1（声誉 + 安全双重影响）

Timeline 关键节点
  1. T+0：研究者发推 + video demo
  2. T+2 hr：PR 团队 alert · 团队复现 · 确认确实漏
  3. T+3 hr：紧急上线 input filter（屏蔽 "ignore previous" / "your instructions" 等 pattern）
  4. T+6 hr：发现 attacker 用 unicode 变体绕过 → 加 unicode normalize
  5. T+10 hr：加 output filter（如果输出含 system prompt 特征字符串就过滤）
  6. T+24 hr：PR 团队官方回应 + 致谢研究者 + 邀请其加入 bug bounty
5 维归因
  1. Code: ✅ 副因 — 缺 input/output filter
  2. System: ❌
  3. Data: ❌
  4. Model: ✅ 副因 — LLM 本质对 system prompt 没"硬隔离"，所有 instruction 都在同一个 context
  5. Prompt: ✅ 主因 — 用户输入绕过 system prompt
Action Items
  1. P0: input/output filter 完善 + 持续更新 attack pattern lib · DRI: AI Infra · 立刻
  2. P0: 上 red team（季度对抗测试） · DRI: Safety · 1 个月
  3. P0: bug bounty program 启动 · DRI: PM + Legal · 2 周
  4. P1: system prompt 中减少敏感信息（如内部 URL / 内部产品代号） · DRI: PM · 1 个月
  5. P2: 团队 prompt security 培训 · DRI: EM · 2 个月
LLM 没有真正的 "system prompt isolation"。任何让用户输入直接进 context 的产品都会被 injection。必须假设"system prompt 会被看到"，不放敏感信息 + 加 input/output filter + 持续 red team。

事故 #4 · 量化质量下降

INT4 量化省 75% cost，但 reasoning 任务掉点 8%

触发

团队为省 cost · Llama 70B FP16 量化到 INT4 (AWQ)

影响

整体 metric 看起来 OK · 但 reasoning-heavy 客户投诉准确率明显下降

严重度

Sev-2

Timeline 关键节点
  1. Day 0：team 跑 standard benchmark (MMLU / HellaSwag) → INT4 vs FP16 diff < 1%
  2. Day 0：上线 INT4 · GPU cost 砍 75% ✅
  3. Day 7：高级套餐客户（用复杂 reasoning 任务的）开始投诉
  4. Day 10：分析发现 — INT4 在长 chain-of-thought 任务上 accuracy 81% → 73%（基准没覆盖这类）
  5. Day 12：实施 multi-quant routing — reasoning task 用 FP16，简单 task 用 INT4
  6. Day 14：质量恢复 · cost 仍砍 55%（不是 75% 但 acceptable）
5 维归因
  1. Code: ✅ 副因 — 量化部署
  2. System: ❌
  3. Data: ✅ 副因 — eval set 没覆盖 reasoning 任务
  4. Model: ✅ 主因 — INT4 在 long reasoning 上 outlier 量化误差累积导致 drift
  5. Prompt: ❌
Action Items
  1. P0: eval set 必须分类 (simple QA / reasoning / coding / long-context)，量化前每类都测 · DRI: ML · 立刻
  2. P1: 部署 multi-quant routing pipeline · DRI: AI Infra · 2 周
  3. P2: 评估 FP8 vs INT4 trade-off（FP8 质量损失更小但 cost 节省少） · DRI: AI Infra · 1 个月
量化的精度损失不均匀 — long context / reasoning 任务受影响最大。Standard benchmark (MMLU 等) 不能反映这些。必须按 use case 分类 eval。详见 Y04 §9 量化坑。

事故 #5 · GPU 预算失控

用户量涨 50% + per-request cost 涨 30%，月底超支 95%

触发

季度 finance review 才发现 GPU 账单超预算 95%

影响

无 outage 但 financial impact 高 · CFO 邮件 cc 老板

严重度

Sev-2（financial）

Timeline 关键节点
  1. M1：预算 $50k/月 · 用了 $58k（超 16%，团队没在意）
  2. M2：用了 $72k（超 44%，仍然没人发 alert · cost dashboard 没人看）
  3. M3：用了 $97k（超 94%） · 月底 finance review 拉警报
  4. Day 1 (after detection)：拉数据归因 — ① 用户量涨 50% (业务好事) ② per-request cost 涨 30% (新 feature 用了更复杂 prompt + 没 caching)
  5. Day 3：紧急 quick win — prompt 优化（砍 25% tokens） + 加 prefix cache（额外砍 15%） + model routing（简单 query 用便宜 model 砍 20%）
  6. Week 2：cost 回到 $65k/月（仍超原预算 但符合业务增长）
  7. Week 4：跟 CFO align 新预算 $75k/月 + 月度 cost review 机制
5 维归因
  1. Code: ✅ 副因 — 新 feature 上线没考虑 cost impact
  2. System: ✅ 主因 — 没 cost monitoring + alert
  3. Data: ❌
  4. Model: ❌
  5. Prompt: ✅ 副因 — system prompt 长度膨胀
Action Items
  1. P0: cost dashboard 上线 + 周/月预算 alert · DRI: AI Infra + Finance · 1 周
  2. P0: 每个新 AI feature PRD 必须有 Cost Model 章 (见 AT01 §7) · DRI: PM · 立刻
  3. P1: monthly cost review 进 EM 周会议程 · DRI: EM · 1 个月
  4. P1: 建立 unit economics dashboard（per-feature gross margin） · DRI: Finance + PM · 1 个月
  5. P2: 跟 CFO 建立 quarterly cost review 机制 · DRI: EM · 1 个季度
AI cost 失控 = 监控失控。不是因为优化没做，是因为"没人盯账单"。EM 必须把 cost 看成跟 latency 一样的 SLI — daily dashboard + 月度 review。详见 M06 AI EM Playbook §4 场景 3。

DO / DON'T

DO

  • 用 5 维归因（Code/System/Data/Model/Prompt） · 不要漏 Data/Model/Prompt
  • 每个 Action Item 必须有 DRI + due date + tracking link
  • 区分 short-term mitigation vs long-term fix · 两者都写
  • 写 "What went well" 段 — 团队心理建设
  • Legal review 如果有 PII / 监管影响
  • 放 24-72 hr 完成 — 拖周以上事实就模糊了

DON'T

  • "AI 偶尔会这样" 当 root cause — 这是放弃
  • 追责具体人（postmortem 是 blameless · 制度问题不是人的问题）
  • action items 全 P1 / P2 — 必有 P0 否则等于没修
  • 不区分 vendor 模型 vs 自训模型 — 应对方式完全不同
  • 跳过 timeline — 写流水账反而最有用
  • postmortem 写完就归档 — 必须季度 review action items 进度

7.1 跟其它模板的关系

  • 事故 #1 / #2 / #4 都涉及 eval gap → 写新 AI feature 时按 AT01 AI PRD §5 强制写 Eval Plan
  • 事故 #5 cost 失控 → 写 PRD 时按 AT01 §7 强制写 Cost Model
  • 事故 #3 injection → 写 PRD 时按 AT01 §8 写 Risk + 参考 X05 AI Safety
  • 事故应对剧本 → 参考 M06 AI EM Playbook §4 5 个危机场景
Insight AI postmortem 的最高境界：每个事故都"产生 2 件事" — ① 立刻修当下问题 ② 系统性 prevent 同类问题。只做 ① 不做 ② = 你团队 3 个月再撞同样的墙。
