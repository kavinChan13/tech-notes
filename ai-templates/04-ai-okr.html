为什么 AI 团队 OKR 不一样

传统团队 OKR 主要 3 维：交付 / 质量 / 效率。AI 团队 OKR 至少有 4 维 + 1 个独特 trade-off。

0.1 跟传统 OKR 的 4 个区别

维度传统团队AI 团队
1. 不确定性"做完 X 就 done""做完 X 不代表 metric 涨" — 必须 Key Result 包含 metric 不只 deliverable
2. + Eval 维度无必须有 eval-related KR（如召回率 / faithfulness / CSAT）
3. + Safety 维度偶尔有必须有 safety / 0-incident KR（不能漏）
4. Cost trade-offcost 独立"quality 必须 cost-aware" — 不能为了 quality 翻 5x cost

0.2 季度 OKR vs Sprint 的边界

AI 团队 KR 的 "可达成性" 区别于传统："做完 X 任务" ≠ "达成 X 指标"。OKR 必须看指标，否则团队会做完 5 个 task 但 metric 没动 — OKR 看似完成实则失败。这是 AI 团队 OKR 最大陷阱。
01

4 类 AI 团队 OKR pattern

90% AI 团队的 OKR 都能套这 4 类
🎯 Pattern 1 · Quality 类

"提升 AI 输出质量"

最常见。关键是用 eval-driven metric，不要用"上线 X feature"这种 task-based。

O: 让我们的 [feature] 答得更准更可信 KR1: faithfulness (RAGAS) 85% → 92% (用现有 eval set / 月度跑) KR2: 用户 thumbs-up rate 64% → 75% (in-product feedback) KR3: 客服 escalation rate 18% → 10% (business metric)

陷阱：只用 LLM-Judge 一个指标 — 会被 prompt 调教 game · 必须 2-3 个 cross-validate。

💰 Pattern 2 · Cost 类

"降低 AI 单位经济性 / 提升 ROI"

CFO 最爱看的。不要只看 absolute cost，要看 unit economics（per-request / per-user / per-feature）。

O: 把 [feature] 的 unit economics 转正 KR1: per-request cost $0.012 → $0.006 (-50%) (model routing + caching) KR2: gross margin per query -15% → +30% (收入端不动 cost 端砍) KR3: GPU utilization 35% → 70% (infra 效率)

陷阱：砍 cost 同时 quality 不能掉 — 必须加 "quality 不退步 > X%" 作为 guardrail metric。

🚀 Pattern 3 · Scale 类

"扩大用户量 / 提升 throughput / 降低 latency"

业务增长场景。多关注 P95/P99，不只 average，否则掩盖长尾问题。

O: 让 [feature] 扛得住 10x 流量 KR1: P95 latency 4.2s → 1.5s KR2: 单实例 QPS 50 → 200 (vLLM 优化 + 量化) KR3: 高峰期 error rate 5% → < 0.5%

陷阱：吞吐涨了 latency 也涨 — 必须 2 个 KR 都达标，不能只看 1 个。

🛡 Pattern 4 · Safety / Compliance 类

"风险降至可接受水位 / 合规"

最容易被忽略 - 但出事就是 Sev-1。必须每个季度都至少 1 个 safety KR，不管业务多忙。

O: 建立 [feature] 的 safety + compliance 底线 KR1: prompt injection 红队测试 100 case 通过率 75% → 95% KR2: PII 漏出 incident 0（baseline 上季度 2 个） KR3: 完成国内备案 + EU AI Act 评估 (binary)

陷阱：Safety KR 不容易"量化"但必须量化，否则就变成"做了 audit" 等 task-based — 没意义。

4 类不是互斥，典型团队季度 OKR 是 "2 quality + 1 cost + 1 safety" 或 "1 quality + 1 scale + 1 cost + 1 safety"。每 Q 4-5 个 Objective，每个 3-4 个 KR，超过 = 散。

# [Team Name] Q[N] [Year] OKR Owner: [EM] Reviewers: [VP/Head] Date: [YYYY-MM-DD] Status: Draft / Aligned / In Progress / Closed ## Context (1 段) - 团队定位 + 本 Q 重点 - 上 Q 结果回顾（哪些 OKR 达成 / 没达成 / 为什么） ## Objective 1: [Quality 类 - 必有] [一句话目标 - 用户视角 / 业务视角，不用技术 jargon] KR1.1: [Eval metric 从 X% → Y%] (DRI: [name]) KR1.2: [User feedback metric 从 X → Y] KR1.3: [Business metric 从 X → Y] ## Objective 2: [Cost 类 - 推荐有] [一句话目标] KR2.1: [per-unit cost 从 $X → $Y] KR2.2: [unit economics 从 X% → Y%] KR2.3: [Guardrail metric: quality 不退 > X%] (必须有 guardrail！) ## Objective 3: [Scale 类 - 业务高速增长时必有] KR3.1: [P95 latency 从 Xs → Ys] KR3.2: [QPS / throughput 从 X → Y] KR3.3: [Error rate 从 X% → Y%] ## Objective 4: [Safety / Compliance 类 - 必有] (每 Q 都要) KR4.1: [Red team test pass rate 从 X% → Y%] KR4.2: [Incident count 0 / 监管事件 0] KR4.3: [Compliance deliverable: 备案 / 审计 (binary)] ## Stretch (optional) [1 个 stretch Objective - 完成是惊喜，不完成不扣分] ## Anti-goals (本 Q 明确不做的事) - [3-5 条 - 避免 scope creep] ## Resource ask - Headcount: [N people / N% of existing] - GPU: [N H100 / 月] - Dependencies: [Cross-team 依赖] ## Tracking - Weekly: [每周 EM 更新 KR % 在 dashboard] - Mid-Q review: [Week 6 跟老板对齐进度 / 调整] - End-of-Q: [Week 12 close + 写复盘]

示例 1 · AI 客服团队季度 OKR

SaaS 公司客服 AI Bot 团队 Q2 OKR
# Customer Support AI Team · Q2 2026 OKR
Owner: J.Lee (EM)    Reviewers: VP Eng + Head of Support    Status: Aligned

## Context
- 上 Q 完成 AI Bot v1 上线 · deflection rate 38% (target 40%, 略 miss)
- 主要短板：长尾问题答错率高 · cost 占收入 18%（目标 < 10%）
- 本 Q 重点：把 v1 调成"能 sustain 的"产品，preparing scale 到所有客户

## Objective 1: 让 AI 客服 deflect 一半常见问题（quality）
- KR1.1: faithfulness (RAGAS) 85% → 92% (DRI: M.Wang)
- KR1.2: CSAT (in-product 5-star) 3.8 → 4.2 (DRI: PM)
- KR1.3: Deflection rate 38% → 50%
- KR1.4: 长尾问题（< 5 occurrences / month）答对率 45% → 70%

## Objective 2: 单位经济转正（cost）
- KR2.1: per-conversation cost $0.038 → $0.015 (-60%)
  · Mechanism: model routing (简单 query → gpt-4o-mini) + prefix caching
- KR2.2: cost / revenue 18% → < 10%
- KR2.3: Guardrail: KR1.1 (faithfulness) 不退 > 1 个点（92% → > 91%）

## Objective 3: Safety + 合规闭环（safety）
- KR3.1: prompt injection 红队 50 个 attack case，通过率 60% → 95%
- KR3.2: 0 PII 漏出 incident（上 Q 1 个）
- KR3.3: 完成 SOC2 audit + GDPR 评估（binary）

## Stretch
- O4 (stretch): 启动 AI 客服多语言（英 → +日 +韩）
  · KR4.1: 日韩 deflection rate > 30%（不强求达到中文水平）

## Anti-goals
- 本 Q 不做 voice agent（资源不够）
- 不做主动出击 / 推销（产品定位限制）
- 不做 customer-specific fine-tune（unit economics 算不过来）

## Resource ask
- Headcount: 当前 4 人 · 申请 +1 ML eng (扩长尾 handling)
- GPU: $5k/月 (跟 SRE 已锁)
- Dependencies: 数据团队提供 labeled long-tail set (Week 4 due)

## Tracking
- Weekly: M.Wang 更新 dashboard 每周一
- Mid-Q (Week 6): 跟 VP 同步进度
- End-of-Q (Week 12): 复盘 + Q3 plan draft

示例 2 · AI Infra 团队季度 OKR

某 LLM 应用公司的 AI Infra 团队 Q3 OKR
# AI Infra Team · Q3 2026 OKR
Owner: T.Zhang (EM)    Reviewers: CTO + VP AI    Status: Aligned

## Context
- 上 Q 完成 vLLM 自部署 · 替代 60% OpenAI 流量
- 本 Q 重点：① 完成剩余 40% 替换 ② 砍 cost 到下个 milestone ③ 准备 Q4 scale 5x

## Objective 1: 完成 OpenAI 替换 + 性能不退（quality + scale）
- KR1.1: 自部署流量占比 60% → 95%
- KR1.2: P95 latency 1.8s → 1.4s (vs OpenAI baseline 1.2s)
- KR1.3: 用户感知质量（A/B test）不退 > 2%（offline eval + online metric）

## Objective 2: 砍 inference cost 50%（cost）
- KR2.1: per-request cost $0.005 → $0.0025
  · Mechanism: INT4 量化（high-confidence task）+ continuous batching 优化 + FP8 KV-cache
- KR2.2: GPU utilization 45% → 75% (P50 over 24h)
- KR2.3: Guardrail: quality 在 long reasoning task 不退 > 3%（防止量化掉点 — 见 Y04 §9）

## Objective 3: 扛住 Q4 5x 流量（scale）
- KR3.1: 完成多机 tensor parallel 部署 (2 → 8 GPU per replica)
- KR3.2: peak QPS capacity 800 → 4000
- KR3.3: failover testing：单 replica 挂时降级时间 < 30s

## Objective 4: Infra safety + observability（safety）
- KR4.1: 完成 GPU OOM / model crash 的 auto recovery (binary)
- KR4.2: 完成 cost dashboard + 月度异常 alert (binary)
- KR4.3: 0 P0 incident due to infra issue（上 Q 1 个 OOM 导致 30 min outage）

## Stretch
- O5 (stretch): 评估 H100 → B200 升级 ROI
  · KR5.1: 产出 Q4 capacity planning doc（含 H100 vs B200 vs 混合 cost model）

## Anti-goals
- 本 Q 不引入新 inference engine（坚持 vLLM，不上 SGLang / TRT-LLM 评估）
- 不做训练 infra（公司不训自有模型）

## Resource ask
- Headcount: 当前 6 人 · 申请 +2 (1 CUDA / 1 SRE)
- GPU: 当前 80 张 H100 · Q3 末 100 张（业务增长配套）
- Dependencies: ML team 提供 量化后 model checkpoint (Week 3)

## Tracking
- Weekly cost + perf dashboard
- Bi-weekly cross-team sync with App teams (依赖 AI Infra 的下游)
- Mid-Q review with CTO

DO / DON'T

✅ DO

  • 每 Q 必有 quality + safety O · cost / scale 二选一
  • 每个 cost / scale O 配 guardrail KR（防止 trade-off 失控）
  • KR 必须是 metric 不是 task（"上线 X" 不是 KR · "X metric 涨 Y%" 才是）
  • Baseline + Target 都写清楚（"X 从 A → B"）
  • 每个 KR 有 DRI（一个人，不是团队）
  • Anti-goals 必写（防止 scope creep）
  • Mid-Q review 调整 (3 个月不动死板)

❌ DON'T

  • "上线 RAG v2" 当 KR — 是 task，不是 KR · 改成"RAGAS faithfulness X→Y"
  • "提升用户满意度" 当 KR — 太虚 · 必须有具体 metric + 数字
  • O 超过 5 个 / KR 超过 20 个 — 散，团队没 focus
  • Cost O 没 guardrail — 容易"砍 cost 翻 quality 车"
  • Safety O 隔 Q 才做 — 出事就完了
  • 季度初定完锁死 — 必须 mid-Q 调整
  • 所有 KR 都 stretch 70% — 不真实，团队会摆烂

5.1 跟其它模板/文档的关系

  • 季度规划全景 → M07 AI EM Playbook §3 Quarter 1 必交付
  • OKR 写完后 → 跟通用 EM 04 30/60/90 plan 落地
  • 每个 KR 用 metric → 参考 X12 AI 评测体系 怎么定 metric
  • cost KR → 配合 AT06 AI Cost Model
  • eval KR → 配合 AT05 Eval Plan
  • 季度末复盘 → 通用 EM 模板 06 QBR
OKR 的最大价值是 "逼自己想清楚下季度做什么 + 为什么"。写 OKR 痛苦 = 思考不清 — 不是模板不好，是团队还没真正决定方向。OKR 写不出来时不要硬填，回头跟老板 align "我们 Q 内的 1 件最重要的事到底是什么"。
