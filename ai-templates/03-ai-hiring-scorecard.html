01是什么 / 为什么

AI Hiring Scorecard 是每位候选人的结构化评分卡。比通用 EM scorecard 多 Eval Mindset / Cost Awareness / Real-world Limits 三维 — 用来区分 demo 工程师 vs 生产 AI 工程师。

核心原则：任何 AI 维度 < 2 分 → 直接 No Hire，不管其它维度多高。简历漂亮但 1 个月不出活，通常是这 3 维没过。

测试题："你怎么知道你做的 RAG 比之前好？" — 答"感觉"是 Junior，答"eval set X% → Y%"是 Senior。

02字段说明

维度 / 字段为什么存在 / 怎么用（1-5 分）
岗位核心技术 (30%)RAG / Agent / Infra 等核心概念能否白板拆解 + trade-off
实战经验 (25%)生产经验 + 能讲具体数字；只做过 tutorial = Lvl-1
Eval Mindset (15%)AI 专属 — 主动建 eval pipeline，非"用户没投诉"
Cost Awareness (10%)AI 专属 — 能算 unit economics + 知道优化手段
Real-world Limits (10%)AI 专属 — 能讲自己项目的失败 / 限制，非"AI 啥都能干"
沟通 / Culture fit (10%)解释清晰、主动提问、听得懂反馈
Must-ask (×3)每岗位 3 个必问题 + 候选人原话记录
Red flags面试中观察到的警示信号（见 practices 章 8 条通用）
DecisionStrong Hire / Hire / Maybe / No Hire + 加权总分 rationale
Insight 决策门槛：总分 < 3.0 No Hire · 3.0-3.5 Maybe · 3.5-4.0 Hire · > 4.0 Strong Hire。但 AI 三维任一 < 2 则一票否决。

岗位 1 · LLM App Engineer

用 LLM API + RAG / Agent 框架做产品级 AI feature · 是 2026 最大需求量岗位
薪资 (一线 / 5+ yoy): 50-100w Junior - Senior

Must-have skills

  • 熟悉 OpenAI / Anthropic / 国内主流 model API + 至少 1 个 RAG / Agent 框架（LangChain / LlamaIndex / 自研）
  • 能写完整的 production AI feature（不止 demo）
  • Python / TypeScript 至少一个熟练
  • 能设计 + 跑 RAGAS / LLM-Judge 类 eval

Nice-to-have

  • 有 vector DB 选型经验 (pgvector / Pinecone / Weaviate / Qdrant)
  • 有 MCP / function calling 实战
  • 有 agent 多步推理设计经验

LLM App 候选人

  1. "讲一个你做的 RAG 项目，准确率从 X% 提到 Y%，具体改了哪 3 个东西？" 考察生产经验 + chunking/embedding/rerank 哪个有真心得。答不出数字 = 没做过
  2. "你怎么知道你的 prompt 改完比之前好？" 考察 Eval Mindset。答 "试了几个 case" = Junior · 答 "建了 100 case eval set 跑了 LLM-Judge / human rating 对比" = Senior
  3. "100k DAU 系统的 LLM cost 大概多少 / 你怎么砍 50%？" 考察 Cost Awareness。能算账 + 知道 caching/routing/量化 = Senior

LLM App

  • 主动说 "我做的 X feature 失败了，因为 Y" — 会反思
  • 能区分 "demo 准确率" vs "生产准确率" — 知道分布差异
  • 读过 vLLM / sglang 至少 1 个 OSS 项目 — 求知欲
  • 会问 "你们的 eval set 谁维护" — 有 production sense

LLM App

  • "我用 LangChain 搭了 ChatBot" — 没有数字 / 没 eval = demo 工程师
  • 说自己"懂 fine-tune" 但说不清 LoRA / SFT / RLHF 区别
  • 问 "vector DB 选什么" 时只答品牌名 (Pinecone)，不能说选型逻辑
  • 认为 "GPT-4 能搞定一切" — 没有 limit awareness

岗位 2 · AI Infra Engineer ⭐

推理引擎 / 训练框架 / CUDA / 量化 / 分布式 · 2026 senior 缺口最大方向 · C++ 老兵最优杠杆 (见 D1)
薪资 (一线 / 5+ yoy): 70-150w Senior - Staff ⚡ 极度稀缺

Must-have skills

  • C++17/20 熟练 · 多线程 / lock-free / 性能优化经验
  • CUDA 基础 (能写 kernel) 或者强烈愿意 + 能力学
  • 读过 vLLM / SGLang / TRT-LLM 至少 1 个的源码 · 能讲核心 mechanism
  • 懂 Transformer / KV-Cache / attention 基本原理（不需要会推 backprop）

Nice-to-have

  • NCCL / 分布式训练经验
  • 量化经验 (GPTQ/AWQ/FP8)
  • Triton kernel 写过
  • vLLM / SGLang / 同类 OSS contributor

AI Infra 候选人

  1. "PagedAttention 解决什么问题？为什么需要 block_table？" 考察推理引擎核心原理。答不上 = 没真碰过 vLLM。能讲 "解决 KV-Cache 内存碎片 + block_table 是逻辑→物理映射" = OK · 能讲 prefix sharing / preemption / swap-out 等更深 = Strong Hire
  2. "为什么 attention 是 memory-bound，FlashAttention 怎么解决？" 考察 GPU 优化思维 + 是否真懂 roofline model。看 X06 §6 / Y01 §7 的概念
  3. "INT4 量化时哪些任务最容易掉点？怎么验证？" 考察实战经验。答 "reasoning / long-context 任务最容易 · 必须按 use case 分类 eval" = Senior · 笼统答 "做 benchmark" = Junior

AI Infra

  • C++ 老兵 5+ yoy + 主动学 CUDA — 这是最稀缺的 candidate
  • 能讲清自己对 vLLM 某个文件做过深度阅读 / contribution
  • 主动说 "我没碰过 ML modeling 但懂 inference engine internals" — 有自知之明
  • 有 OSS 贡献（哪怕 documentation PR）

AI Infra

  • "我用过 vLLM API" — 用 vs 懂内部完全不同
  • 问 "PagedAttention" 时只能复述 paper 的话 — 没实战
  • 说自己"懂 CUDA" 但白板写不出 vector add kernel
  • 纯 ML 背景但拒绝学 C++ / 系统编程 — AI Infra 必须懂底层

岗位 3 · AI Tech Lead / EM

管 5-15 人 AI 团队 · 兼有技术决策 + 团队 + 跨部门职责 · 跟 IC 路径很不同
薪资 (一线 / 8+ yoy): 80-200w Senior - Director

Must-have skills

  • 5+ 年技术管理经验 · 至少 2 年 AI 团队管理（或强 AI IC 经验）
  • 能跟 ML / Infra / App / PM 多 persona 沟通
  • 懂 eval / cost / safety / hiring 4 个 AI EM 核心议题（参考 M06）
  • 能讲过去 hire 过的 1-2 个成功 case + 失败 case

Nice-to-have

  • 有 cost / unit economics 跟 finance 谈过经验
  • 有 AI safety incident handling 经验
  • 有 cross-functional 项目交付经验 (跟法务 / SRE / data 协作)

AI TL/EM 候选人

  1. "你怎么决定团队的 eval 体系深度？eval 占团队 capacity 多少？" 考察 eval mindset + 资源分配判断。Junior 答 "看情况"。Senior 答 "根据 use case 风险分级，high-stakes 用 RAGAS + human eval，低 stakes 用 LLM-Judge，总占 15-20% capacity"
  2. "你团队 GPU 预算超 30%，CFO 找你怎么办？" 考察 cost ownership + 向上沟通。看 M06 §4 场景 3
  3. "招一个 PhD vs 招一个 5 yoy SWE 转 AI，你会怎么决策？" 考察对 AI 人才结构理解。没有标准答案 — 关键看候选人能不能讲清楚 trade-off (PhD 强在 modeling / SWE 强在工程化)

AI TL/EM

  • 能讲过 hire 后回归看 hiring decision 的 case (反思)
  • 能讲 1 个让团队失望但事后证明正确的决策
  • 主动谈 cost / business metric · 不只技术
  • 承认 "AI EM 比 backend EM 不确定性高" — 心态成熟

AI TL/EM

  • 从未 fire 过人 — 经验不够
  • 跟 IC 讲技术时听不懂 jargon — 容易被忽悠
  • 没有 model risk / safety incident handling 经验
  • 谈 metric 只谈 throughput / latency · 不谈 cost / CSAT / business KPI

其它 3 个岗位 · 简版 scorecard

岗位 4 · ML Engineer (Modeling)

SFT / LoRA / RM / RLHF / DPO · 负责模型本身训练 · 跟 AI Infra 不同（不是 serving）
薪资: 60-130w Senior

ML Engineer 候选人

  1. "SFT vs DPO vs RLHF · 你做过哪个，什么场景选哪个？" 考察 alignment 实战
  2. "你 fine-tune 过 model · 怎么判断"是数据问题"还是"超参数问题"？" 考察 debugging
  3. "DeepSeek R1 的 reasoning 训练方式跟传统 RLHF 区别？" 考察是否跟 SOTA paper

岗位 5 · AI Research Engineer

前沿研究 + paper 输出 · 大模型架构 / 训练范式 · 顶尖公司才招
薪资: 100-300w+ PhD 优先 / 顶会 paper ⚠ 不一定适合大多数公司

Research 候选人

  1. "你的 paper 跟前人 X 比，新意在哪？" 看是否能讲清 novelty (vs 包装话术)
  2. "如果给你 1000 张 H100 1 个月，你会复现什么 paper / 训什么模型？" 看 research taste
  3. "你最近看的 1 篇让你改变想法的 paper 是什么？为什么？" 看持续学习 + intellectual humility

Research

  • paper 多但都是 incremental / 没引用 — paper 数 ≠ impact
  • 能讲 paper 但不能讲工程实现 — 落不了地
  • 对 SOTA model (Llama / DeepSeek / Claude / GPT) 不熟 — 脱节

岗位 6 · AI PM

产品定义 + eval + business · 跟传统 PM 关键差异：必须懂 AI 不确定性
薪资: 40-120w Mid - Senior PM

AI PM 候选人

  1. "AI feature 准确率 80%，PM 你 approve 吗？" 看是否懂 "看场景定阈值"（参考 I08 §2 Q4）
  2. "AI PRD 跟普通 PRD 多哪几段？" 看是否懂 AI 特殊性。期望答 Eval / Failure mode / Cost / Risk 4 段（参考 AT01）
  3. "你怎么写 abandon criteria？" 看是否能给"AI 失败时砍掉的勇气"

AI PM

  • 承诺"AI 会让 X 涨 100%" — 没数据 / 不懂不确定性
  • 不愿意定 eval set / abandon criteria — 怕担责
  • 不懂 LLM cost 数量级 — 写不出可行的 cost model

8 个通用 Red Flag

  1. 说不出"我做的项目失败了什么" — 没经验 / 不反思
  2. "AI 应该 / 会 / 能"开头的答案多（vs "我做过 / 我看过 / 数据是"） — 在猜不是在答
  3. 不能 ballpark estimate AI cost（哪怕 order-of-magnitude） — 没碰过生产
  4. 对当下 SOTA 不熟（不知道 GPT-4 / Claude 3.5 / DeepSeek V3 谁强谁弱）— 脱节 / 不持续学习
  5. "用 GPT-4 就行" 当方案（不考虑 cost / latency / privacy / 失败模式）— 不会做产品决策
  6. 认为 eval 是"测试团队的事" — 没建过 eval pipeline，没真的 own 过质量
  7. 讲技术时不主动谈 trade-off（只说 "这是 best practice"）— Junior level
  8. 对面试反馈防御（"不是我的错")— Culture fit 差，难协作
Insight AI 招聘比传统招聘容错率低 — 1 个 mis-hire 在 AI 团队的影响 = 普通团队的 3-5 倍。原因：AI 团队规模小 + 决策权重大 + 试错周期长。宁缺勿滥比快速 fill 重要。

决策框架 + DO / DON'T

7.1 Debrief 决策框架

每场面试结束后 24 hr 内必须完成 scorecard 填写。Debrief 会议流程：

  1. 第一轮 · 各自 hire/no-hire 表态（不解释为什么）— 看是否一致
  2. 第二轮 · 每人讲 1 个最强证据 + 1 个最担心（基于 scorecard 维度）
  3. 第三轮 · 讨论分歧 — 不是说服，是 surface 不同视角
  4. 第四轮 · Hiring manager 决定（不投票）+ 记 decision log

7.2 入职 12 个月后 review

必须做 — 检验你的 scorecard 准不准：

  • 当时 scorecard 高分维度，是否在工作中确实 strong？
  • 当时 scorecard 低分维度，是否在工作中确实 weak？
  • 有哪些 scorecard 没测到的能力（要加进下一版）？
  • 有哪些 scorecard 测到的能力其实不重要（要砍掉）？

DO

  • 开 role 前先锁 scorecard（不要面试中改）
  • 所有面试官独立填分 (no influence) · debrief 才汇总
  • 每个分数必有证据（候选人说了什么 / 做了什么）
  • 3 个 AI 维度 < 2 → 直接 No Hire 不论其它
  • 入职 12 个月后回 review scorecard 准确度
  • "为什么 No Hire" 也要写清楚 — 避免半年后又面同人

DON'T

  • "感觉对" 当决策依据 — 用 framework 不用 vibes
  • panel 投票决定 — Hiring manager own decision
  • 因为 pipeline 干 = lower bar — 反而招来 mis-hire 拖团队
  • 用 "他们公司很 fancy" 当加分 — pedigree ≠ 能力
  • 把 culture fit 当排除非主流候选人的借口 — 是 culture add 不是 culture fit
  • 面试问 "你的弱点是什么" 这种 BS 问题

  • 通用 EM 流程：EM 模板 08 Hiring Scorecard + 11 Promotion Packet

Insight 最后忠告：scorecard 是工具不是目的。用 scorecard 是为了让招聘决策 可解释 + 可复盘 + 可迭代，不是为了 "客观"（招聘永远不可能 100% 客观）。3 个月用 scorecard hire 5 个人，回头比对 → 你会发现自己的判断在哪些维度准 / 在哪些不准。这才是真正的成长。
