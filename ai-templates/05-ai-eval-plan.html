为什么 eval 是 AI 项目的"地基"

没 eval 的 AI 项目 = 不称体重的减肥。3 个月后团队完全不知道自己变好还是变差。这不是哲学问题，是工程问题。

0.1 没 eval 的 5 个典型症状

症状背后问题
"我改完 prompt 感觉好多了"没数据 = 心理暗示
"换模型试试，行就上"没基准 = 永远在调，永远不知好坏
"用户没投诉就行"用户大部分会沉默离开 · sample bias 严重
"上线后再看 metric"线上回归 = 用户已经受影响 = 太晚
"我们 LLM 应用太复杂 eval 不了"越复杂越要 eval · 是借口
Eval 不是给"完美 AI"建的，是给"不完美 AI 在不完美变化中"建的。vendor model 升级 / prompt 改动 / data 漂移每周都发生，没 eval 等于盲驾。

0.2 Eval 体系的 ROI 倒推

典型团队 cost-benefit：

  • 投入：1-2 ML eng × 4 周建第一版 eval（约 50-80 人/天）
  • 避免的事故：1 次 silent regression 影响 30% 用户 1 周 = ~$50k-200k cost (depends scale)
  • 加速的决策：model selection / prompt 调试 / data 修正 — 时间砍 5x
  • 典型 ROI：3-6 个月回本 · 长期 5-10x ROI

4 阶段 Eval Pipeline

从 offline 到 production · 每个 model / prompt 改动都过这 4 关

① Offline Eval

DEV 阶段 · 每次改动

What：预先 curated test set 跑 + auto metric · 不接真实流量

Tools：RAGAS / LlamaIndex eval / 自建 pytest

Tempo：每次 PR 自动跑（CI）

Pass criteria：核心 metric 不退步 · 改进 metric 涨 > X%

  • 速度快 (~ 1-5 min)
  • 反馈 fast loop
  • 但 ≠ 真实分布

② Shadow Eval

STAGE 阶段 · 上线前

What：用真实流量但不返回给用户 · 比对新旧 model 输出

How：production traffic 复制 1% 到 new model · 留 sample 给 ML team 看

Tempo：上线前 3-7 天

Pass criteria：随机抽 100 case · 人工评分 ≥ 老 model

  • 真实分布
  • 不影响用户
  • 但需要 infra 支持

③ Canary / A-B

部分上线 · 1-2 周

What：5% → 20% → 50% 灰度 · 跑 A-B

Metrics：online metric (CTR / thumbs-up / conversion / CSAT)

Pass criteria：所有 metric 不退 + 至少 1 个改进 statistically significant

Rollback：任何 P0 metric 跌 > 5% · 自动 rollback

  • 真实业务 metric
  • 但需要 traffic 量足够

④ Production Monitoring

长期 · 全量后

What：持续监控 · alert 异常

Metrics：质量 (sampled human-eval) · cost · latency · failure rate · safety incident

Tempo：daily dashboard + weekly review + monthly deep-dive

Alert：核心 metric 跌 > X% / P95 latency > Ys / cost spike > 30%

  • catch silent regression
  • data drift detection
4 阶段不能跳 — 跳 stage 阶段（直接 dev → canary）= 高概率事故。每阶段的 cost 递增（offline 自动跑几乎免费 → production human eval 贵），fail fast 应该在前 2 阶段就拦截。

场景 1 · RAG 客服 Eval Plan

企业 SaaS 公司客服 AI Bot
# Customer Support RAG Bot · Eval Plan

## 1. Scope
- 评估对象：RAG-based FAQ bot (over 1500 Help Center 文章)
- 目标：① 检测 prompt / model / embedding 改动是否回归 ② 持续追踪 production quality

## 2. Metrics
| 维度 | Metric | 怎么算 | P0 阈值 | Target |
|---|---|---|---|---|
| Faithfulness | 答案是否来自检索内容 | RAGAS | > 85% | > 92% |
| Answer Relevance | 答是否答到问 | LLM-Judge (Claude) | > 80% | > 90% |
| Citation Accuracy | 引用源是否正确 | Rule-based (URL match) | > 90% | > 95% |
| Context Recall | 该召回的有没召回 | RAGAS | > 70% | > 85% |
| Latency P95 | 用户感知 | NewRelic | < 4s | < 2s |
| Cost / query | 单查询 | OpenAI billing | < $0.005 | < $0.002 |

## 3. Eval Set Construction
- Size: 200 cases (start) → grow to 500 over 6 months
- Coverage:
  · 常见 8 类问题 (账户 / 计费 / 配置 / API / 安全 / 集成 / 移动端 / 通用)
  · 每类 20 case + 边界 case 40 个
- 来源:
  · 60% 真实 historical tickets (sampled 6 months)
  · 30% Support team 手工 craft (覆盖未来 expected case)
  · 10% LLM 合成 (低质量 prompt / 越狱 / 边界)
- 标注: Support 团队 2 人独立标 "正确答案" · disagreement 走第 3 人
- 更新: 每月 Support team 增加 10-20 真实 ticket + 移除过时 case
- 版本: Git LFS · `eval/support_qa_v2.jsonl`

## 4. 4-Phase Pipeline

### Phase 1 · Offline (CI)
- Tool: RAGAS via pytest fixture
- Trigger: 每次 prompts/*.py 改动 + 每次 retrieval config 改动
- Runtime: ~ 4 min (200 case × 2s each)
- Pass: faithfulness > 90% + recall > 75% + 任 1 metric 比 baseline 涨

### Phase 2 · Shadow (Pre-launch, 5 天)
- Sample: 5% production traffic 复制到 staging
- 人工评估: QA team 抽 100 case · 给 5-star rating
- Pass: 平均 rating 新 ≥ 老

### Phase 3 · Canary (10 天)
- 灰度: 5% (3 天) → 20% (3 天) → 100% (4 天 观察)
- Online metric: thumbs-up rate · 转人工 rate · 对话轮数
- Rollback: thumbs-up rate 跌 > 5% 24h 内 / 转人工 rate 涨 > 10%

### Phase 4 · Production
- Dashboard: Looker · refresh every hour
- Sampled human eval: 每周 50 case · QA team
- Alert: faithfulness daily sample 跌 > 5% / cost spike > 30%
- Weekly: PM + ML 一起 review failure case + 加入 eval set

## 5. Resources
- 1 ML eng × 4 周 build · 之后 ~10% capacity 维护
- Annotation: Support team 0.5 day/week
- Cost: LLM-Judge API ~$200/month

## 6. Risks
- LLM-Judge (Claude) bias: 每月 sample 20 case human cross-check
- Eval set staleness: 月度强制更新机制
- 长尾盲点: 每月跑 1 次完整 traffic 分布分析 · 补 long-tail eval

场景 2 · Agent 多步任务 Eval Plan

企业内部 Agent 自动处理报销 / 假期申请类任务
# Internal Workflow Agent · Eval Plan

## 1. Scope
- 评估对象：multi-step agent (~3-7 tool calls / task)
- 难点：agent 中间步骤错也可能最终结果对 · 必须 step-level + task-level 都评

## 2. Metrics
| 维度 | Metric | 怎么算 |
|---|---|---|
| Task success | 最终结果正确 | rule-based check (e.g. ticket created with right fields) |
| Trajectory faithfulness | 中间步骤合理 | LLM-Judge over reasoning trace |
| Tool call accuracy | 工具调用对 | exact match against expected calls |
| Token efficiency | 用了多少 token 完成 task | log + 分类比 baseline |
| Hallucination rate | 编造 tool / API | regex + tool registry check |
| Safety | 是否执行越权操作 | rule-based + LLM-Judge |

## 3. Eval Set Construction
- Size: 100 tasks (start) → 300
- Coverage:
  · 简单 task (1-2 step): 30%
  · 中等 (3-5 step): 50%
  · 复杂 / 长链 (6+ step): 20%
- 来源: 真实 employee request (匿名化) + handcraft adversarial
- 每个 case 包含: input · expected tool calls · expected final state · acceptable alternatives

## 4. 4-Phase Pipeline
（同 RAG 但 Phase 1 必须比对完整 trajectory，不只 final answer）

### 特殊点：Step-level eval
- 每个 tool call 后 dump state · 跟 expected diff
- 鼓励 "different path 同样到达" → 设置 "acceptable_alternatives" 字段

### Phase 3 Canary
- 灰度更慢：5% (1 week) → 25% (1 week) → 100% (2 week)
- Online metric: task completion rate · 平均 step 数 · human override rate
- Rollback: human override 涨 > 15% → 立刻 rollback

## 5. Special: Replay testing
- 每周从 production 抽 50 失败 task · 在新 model / 新 prompt 上 replay
- 看是否能修复同样问题

## 6. Risks
- Step-level eval cost 高（每 task ~10x simple eval cost）→ 用 cheap LLM-Judge + sample 而非全跑
- "Different path 一样对" 难定义 → 用 outcome-based + manual approval for new path

场景 3 · Coding Assistant Eval Plan

公司内部 AI Coding Assistant（基于 Cursor / Claude）
# Internal AI Coding Assistant · Eval Plan

## 1. Scope
- 评估对象：code completion + chat-edit + autonomous agent 3 mode
- 难点：代码"正确"标准复杂 (能跑 ≠ 正确 / 符合规范)

## 2. Metrics
| 维度 | Metric | 怎么算 |
|---|---|---|
| Code correctness | 生成代码能跑 + 测试过 | 跑 unit test |
| Code quality | 符合代码规范 | linter + LLM-Judge against style guide |
| Suggestion accept rate | 用户接受 % | IDE telemetry |
| Time-to-first-token | 流式响应速度 | latency log |
| Per-suggestion cost | 用了多少 token | API billing |
| Security | 不生成漏洞代码 | Snyk / SonarQube / 自建 rule |

## 3. Eval Set Construction
- Size: 200 case → 500
- Coverage:
  · 语言: Python / TS / Go / Rust / C++ (按公司 stack 比例)
  · 任务类型: completion (40%) / refactor (25%) / new function (20%) / bug fix (15%)
  · 复杂度: 简单 (50%) / 中 (30%) / 复杂 / cross-file (20%)
- 来源: 公司真实 PR (extract before/after diff) + HumanEval-style synthetic + 已知 bug

## 4. 4-Phase Pipeline

### Phase 1 · Offline
- Tool: Custom harness · 跑测试 + linter + security scan
- Runtime: ~ 15 min (代码跑 test 慢)
- Pass: code 能跑 > 85% + security clean

### Phase 2 · Shadow
- Sample: 部分 internal dev opt-in 使用新版本但 telemetry 不影响 reporting
- 跑 1 周

### Phase 3 · Canary
- 灰度: 10% engineer → 50% → 100%
- Online: acceptance rate · undo rate · 错误率
- Rollback: undo rate 涨 > 20%

### Phase 4 · Production
- Weekly: 1-on-1 with 10 sampled users · 收集 qualitative feedback
- Monthly: 跟 git commit data correlation (suggested code 多少进 production)

## 5. 特殊：Security gating
- 任何 suggestion 包含 known vulnerability pattern → block + log
- 季度 red team test (Coding Agent specific): SQL injection prompt / hardcoded secret 等

## 6. Risks
- 用户 acceptance ≠ 代码质量（用户可能 accept 然后回头改）→ 需要 "long-term retention" metric
- Security false positive 会让 dev 烦 → 调阈值 + allowlist 机制

DO / DON'T

✅ DO

  • 4 阶段都建立（即使每阶段简化版） · 不能只做 offline 就上线
  • Eval set 必须有 owner + 月度更新机制
  • 每个 metric 必须有 P0 阈值（ship-block）+ Target
  • Online metric 跟 offline 至少季度做 alignment check
  • Failure case 加入 regression test（"再也不踩同样坑"）
  • Eval cost 是 budget 的一部分 · 不要省
  • 跑 eval 自动化（PR / cron） · 不靠人记得

❌ DON'T

  • 只用 1 个 metric · 容易被 game
  • 只用 LLM-Judge · 偏差严重，要跟 human rating cross-check
  • Eval set 半年不更新 · stale 后没意义
  • Eval 跟 production logic 用同一个 prompt · 完全没 generalization 测试
  • "我们项目小 eval 不需要" · 错 — 小项目也会回归
  • 跳 shadow 阶段直接 canary · 高概率事故
  • 没有 alert + dashboard · 等用户投诉才知道
