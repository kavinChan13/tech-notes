01分布式系统的根本问题：为什么这么难

分布式系统的所有复杂性都根源于两个物理现实：（1）节点会故障；（2）网络会延迟、丢包、分区。一旦承认这两点，大量"在单机上天经地义"的假设就全部崩塌。

1.1 "分布式计算的八大谬误"（Peter Deutsch, Sun Microsystems 1994）

程序员常犯的错误假设

  1. 网络是可靠的
  2. 延迟为零
  3. 带宽是无限的
  4. 网络是安全的
  5. 拓扑结构不会变化
  6. 只有一个管理员
  7. 传输成本为零
  8. 网络是同质化的

正确的工程假设

  • 网络会丢包、延迟、分区
  • 延迟从微秒到秒级都可能
  • 节点会崩溃、重启、变慢
  • 消息可能重复、乱序、丢失
  • 时钟不同步（NTP 只能缓解）
  • 进程可能暂停（GC、swap、容器迁移）

1.2 分布式系统的三大不可能

FLP 不可能定理（1985）

在异步系统中，即使只有一个节点可能故障，也不存在确定性的共识算法。

实践含义：所有实际共识算法（Raft/Paxos）都必须在某些条件下牺牲进度（liveness）或引入随机化。

CAP 定理（2000）

分布式系统在网络分区时只能二选一：一致性（C）或可用性（A）。

实践含义：没有"完美"的分布式系统，只有根据业务需求选择牺牲哪个。

拜占庭容错（Byzantine Fault）

如果节点可以发送错误/恶意消息，需要 3f+1 个节点才能容忍 f 个拜占庭故障。

大多数内部系统不考虑拜占庭故障（假设节点可以崩溃但不会撒谎）。区块链例外。

1.3 分布式系统的时间问题

时间是分布式系统里最危险的假设：

物理时钟（wall clock）不可信

  • NTP 同步误差通常 1-50ms，极端情况更大
  • VM 迁移、休眠、时钟调整都可能导致时钟跳跃
  • 两个节点的 System.currentTimeMillis() 可能相差数百毫秒

永远不要用物理时钟来决定事件的先后顺序。

逻辑时钟（Logical Clock）

  • Lamport 时钟：每次事件 +1，发消息带时间戳；接收时取 max+1。建立了 happens-before 关系，但不能推断并发。
  • 向量时钟（Vector Clock）：每个节点维护一个向量，能准确识别并发与因果关系。
  • 混合逻辑时钟（HLC）：结合物理时钟和逻辑时钟，CockroachDB 使用。

1.4 部分失败（Partial Failure）

单机故障通常是全或无：要么正常，要么死机。分布式系统的故障是部分失败——：

  • 请求发出了，但不知道对方收没收到
  • 对方处理了，但响应丢失了
  • 对方还在处理，只是很慢（慢节点 vs 死节点不可区分）
  • 某台机器的网卡出问题，只接收不发送
架构师的思维转变："在分布式系统里，你永远不知道一个操作是否成功——只能知道它可能成功了"。这就是为什么所有分布式操作都必须设计为幂等的（重试安全），以及为什么需要明确区分"至少一次"、"至多一次"、"恰好一次"语义。

02CAP / PACELC / BASE：正确理解，避免常见误读

2.1 CAP 定理精确含义

Eric Brewer（2000）提出，Gilbert & Lynch（2002）证明。

C — Consistency

这里的一致性是Linearizability（可线性化），不是 ACID 的"一致性"（C）！

含义：每次读都能读到最新的写，或者出错。所有节点像一台机器。

A — Availability

每次请求都会得到非错误响应（不保证是最新数据）。

注意："可用"的严格定义是响应不能是超时或 error——可以返回旧数据。

P — Partition Tolerance

系统在网络分区时继续运行（节点间消息可以任意延迟/丢失）。

对于分布式系统，P 是必须保留的（网络分区是不可避免的物理现实），所以实际只是 C vs A 的选择。

最常见的 CAP 误读

❌ 误读 1："选 AP 就没有一致性"

错。没有网络分区时，AP 系统可以提供强一致性；只有分区发生时，才必须选择：响应旧数据（牺牲 C）还是拒绝请求（牺牲 A）。

❌ 误读 2："关系数据库都是 CA"

不存在"CA 系统"。P 不是一个可以"不选"的属性——分布式系统必须容忍网络分区，否则是单机系统。

❌ 误读 3："CAP 是永恒的三选二"

CAP 只在网络分区发生时才产生冲突。正常情况下 C 和 A 不矛盾。

❌ 误读 4：CAP 的 C = ACID 的 C

完全不同。ACID 的 C 是"事务后数据满足约束"；CAP 的 C 是 Linearizability（操作像原子一样）。

2.2 PACELC：CAP 的改进版

Daniel Abadi (2012) 提出，更全面地描述了分布式系统的权衡：

PACELC：If there is a Partition, choose between Availability and Consistency; Else (no partition), choose between Latency and Consistency.

即使没有网络分区，强一致性（Linearizability）也需要额外的同步开销（更高延迟）。

DynamoDBPAEL可用性优先，最终一致
CassandraPAEL可调一致性（默认 EL）
ZooKeeperPCEC强一致，分区时停服
etcdPCECRaft，强一致
CockroachDBPCEL/EC默认 EC，可配置
MySQL（单主）PCEC强一致，但非分布式

2.3 BASE：AP 系统的设计哲学

BASE 是 Basically Available, Soft State, Eventually Consistent 的缩写，与 ACID 对立。

Basically Available

系统保证可用性，但可能是部分可用（降级）或响应旧数据。

例：Amazon 购物车允许返回旧数据，保证响应。

Soft State

系统状态可能在没有新输入的情况下自发改变（因为后台的复制/同步还在进行）。

Eventually Consistent

如果没有新的更新，最终所有副本会收敛到同一个值。时间界限通常是"几秒到几分钟"。

BASE 不是"放弃一致性"，而是"接受短暂不一致换取高可用"。设计 BASE 系统时，架构师的工作是：
  1. 识别哪些数据是允许最终一致的（用户头像、浏览历史）
  2. 识别哪些数据必须强一致（账户余额、库存扣减）
  3. 为"最终一致"的场景设计冲突解决策略（Last-Write-Wins、CRDT、version vector）

03一致性模型谱系：从最强到最弱

"一致性"这个词在分布式系统里有十几种精确含义。本章从最强到最弱梳理常见一致性模型，给出可测试的定义，以及每种模型在实际系统中的对应。

3.1 一致性谱系（强 → 弱）

Linearizability（可线性化 / 强一致）
操作表现如同在单机上顺序执行；读取总能看到最新值。最强。ZooKeeper, etcd, Spanner
Sequential Consistency（顺序一致）
所有操作按某种全局顺序执行，且每个进程内的顺序保持。比 Linearizability 弱（不要求实时）。
Causal Consistency（因果一致）
有因果关系的操作按因果顺序看到；并发操作顺序可不同。MongoDB（会话内），COPS
Read Your Writes（读己之写）
一个进程总能读到自己最新的写。MySQL 主从的同 session 保证。
Monotonic Read（单调读）
如果读到了某个值，之后的读不会读到更老的值（不会回退）。
Monotonic Write（单调写）
同一进程的写操作按顺序执行（不乱序）。
Eventual Consistency（最终一致）
没有新写入的情况下，最终所有副本收敛。DynamoDB, Cassandra（默认），DNS

3.2 Linearizability 的精确定义

Linearizability = 每个操作表现得好像在整个系统中原子地、瞬间地发生，且在其 invocation 和 completion 之间的某个时间点生效。

// 时序图：检验 Linearizability
// 每个操作有 start 和 end 时间，操作效果必须在 [start, end] 内发生

// A: write(x, 1)   |------|
// B:        read(x)         |------|
// B 读到 1 是 Linearizable（A 在 B 开始之前完成）

// A: write(x, 1)   |-----|
// B: read(x)  |-----------| (与 A 并发)
// B 读到 0 或 1 都合法（并发执行，允许 B 在 A 写之前"生效"）

// 违反 Linearizability：
// A: write(x, 1)  |-----|
// B: read(x)              |---| → 结果=0  (A 已完成，B 读到旧值 = VIOLATION!)

3.3 最终一致性的陷阱

"最终一致性"不是"随便不一致"的借口。最终一致系统必须回答以下问题：
  • 多长时间后才"最终"收敛？（通常需要 SLA，如"99.9% 在 5 秒内"）
  • 冲突如何解决？（Last-Write-Wins? CRDT? 应用层 merge?）
  • 哪些操作是幂等的？（重试时不会双写）
  • 读到旧值时用户体验是什么？（会产生 confusion / 数据错误吗？）

常见冲突解决策略

策略原理适用缺点
Last-Write-Wins (LWW) 时间戳最大的写入胜出 用户配置、Profile 时钟不同步 → 错误覆盖
Version Vector 追踪每个节点的写入版本，检测并发 Riak，Amazon Cart 并发冲突需应用层解决
CRDT（无冲突复制数据类型） 设计成天然可合并的数据结构（计数器、集合、树） 协作编辑、计数、在线状态 只适用于特定数据类型
Operational Transformation (OT) 转换并发操作使其可交换 Google Docs 复杂，难以正确实现

04Raft 共识算法完整讲解

Raft 是 Diego Ongaro（2014 年斯坦福博士论文）设计的共识算法，目标是比 Paxos 更易理解和实现。它是 etcd、CockroachDB、TiKV、InfluxDB 等众多系统的核心。本章完整覆盖 Raft 的三个核心问题：Leader 选举、日志复制、安全性，以及成员变更和日志压缩。

4.1 Raft 基础：三种角色

Leader（领导者）

集群在正常情况下只有一个 Leader。

  • 处理所有客户端请求
  • 把日志条目复制给 Follower
  • 定期发 heartbeat 维持权威

Follower（跟随者）

被动响应，不主动发起请求。

  • 接受并复制 Leader 的日志
  • 响应 Leader 和 Candidate 的请求
  • 超时没收到 heartbeat → 变为 Candidate

Candidate（候选人）

Leader 选举期间的临时状态。

  • 发起 RequestVote 给所有节点
  • 获得多数票 → 成为 Leader
  • 选举失败或超时 → 重新发起

4.2 Term（任期）：Raft 的逻辑时钟

Raft 把时间划分成一系列 term（任期），每个 term 以单调递增的整数标识：

时间轴：
Term 1: [选举]→[Leader运行...]
Term 2: [选举失败（split vote）]
Term 3: [选举]→[Leader运行...]

关键规则：
1. 每个节点有 currentTerm，单调递增
2. 如果收到比自己更高 term 的消息 → 更新自己的 term，退为 Follower
3. 如果收到比自己更低 term 的消息 → 拒绝（stale leader）

4.3 Leader 选举：完整流程

投票规则（关键：保证安全性）

// Follower 投票给 Candidate 的条件（两者都必须满足）：
// 1. candidate.term >= self.currentTerm
// 2. candidate 的日志"至少和自己一样新"（up-to-date）

// 日志"更新"的比较：
bool candidate_log_up_to_date(
    int candidate_last_term, int candidate_last_index,
    int my_last_term,         int my_last_index) {
    // 先比较最后一条日志的 term
    if (candidate_last_term != my_last_term)
        return candidate_last_term > my_last_term;
    // term 相同，再比较日志长度
    return candidate_last_index >= my_last_index;
}

// 每个 follower 在同一 term 内只投一票（先到先得）
// votedFor: 记录本 term 已投给谁

选举超时（Election Timeout）

每个 Follower 有一个随机化的选举超时（通常 150-300ms 之间随机）。收到 Leader heartbeat 就重置。超时未收到 → 变为 Candidate，term+1，发起投票。

随机化超时的目的：避免多个 Follower 同时超时，同时发起选举，导致 split vote（没有人能获得多数票）。随机化使得一个节点通常比其他节点早超时，能在别人反应之前先获得多数票。

4.4 日志复制：AppendEntries RPC

// Leader 的处理流程：
// 1. 客户端发送命令
// 2. Leader 把命令追加到自己的日志（uncommitted）
// 3. 并行发 AppendEntries RPC 给所有 Follower
// 4. 当 多数节点（包括 Leader）写入成功后 → 提交（committed）
// 5. Leader 执行命令，回复客户端
// 6. 后续的 AppendEntries 告知 Follower 提交点，Follower 执行命令

// AppendEntries RPC 参数：
struct AppendEntriesRequest {
    int term;              // Leader 的当前 term
    int leaderId;
    int prevLogIndex;      // 紧接在新条目前的日志条目 index
    int prevLogTerm;       // prevLogIndex 条目的 term
    std::vector<LogEntry> entries;  // 要追加的条目（heartbeat 时为空）
    int leaderCommit;     // Leader 已提交的最高 index
};

日志一致性检验（Log Matching Property）

Follower 收到 AppendEntries 时，会检查 (prevLogIndex, prevLogTerm)：

  • 如果自己在 prevLogIndex 处的 term 与 prevLogTerm 不匹配 → 拒绝
  • Leader 发现 Follower 落后 → 回退 prevLogIndex，重试
  • 通过这个机制，Leader 会找到和 Follower 的最后一个共同点，然后强制 Follower 复制 Leader 之后的日志

4.5 安全性：为什么不会产生脑裂

关键不变量：已提交的日志永远不会丢

这通过两个规则保证：

规则 1：多数派提交

只有多数节点（Quorum）都写入了某条日志后，Leader 才能提交它。

新 Leader 至少会从多数节点中选出，而多数节点中一定至少有一个节点有这条已提交的日志。

规则 2：选举时日志检查

新 Leader 只能从日志最新的节点中产生（投票规则中的 up-to-date 检查）。

确保新 Leader 一定包含了所有已提交的日志。

Leader Only Commits from Current Term（重要微妙点）

// 边角 case：Raft 不允许 Leader 提交"来自旧 term 的日志"
// 即使这条旧日志已经在多数节点上了

// 原因：
// S1 (Leader, term 2) 把 log[2]={term:2, cmd:X} 复制给了 S2
// S1 挂了，S5 (term 3) 成为 Leader
// S5 在 log[2] 写了 {term:3, cmd:Y}（覆盖旧日志）
// S5 挂了，S1 重新当选，看到 log[2]={term:2} 在多数节点上
// 如果 S1 提交 log[2]={term:2}，S5 之后又可能覆盖它 → 已提交日志被覆盖！

// Raft 的解决：Leader 只提交 currentTerm 的日志
// 旧 term 的日志通过"顺带"新 term 的提交来间接提交

4.6 成员变更（Membership Change）

在集群运行期间增减节点是危险操作——如果处理不当可能同时存在两个 Leader（脑裂）。

Raft 使用联合共识（Joint Consensus）或单步变更（Single-Server Change）解决。最常用的是单步变更：

  • 每次只加或减一个节点
  • 新配置通过日志复制到多数节点后生效
  • 在此期间新旧配置的多数派有交集 → 不会出现两个 Leader

05Paxos 与 ZAB：与 Raft 的对比

5.1 Paxos（单 Decree）核心

Leslie Lamport 1989（发表 1998）。Paxos 回答的问题：如何让 N 个可能故障的节点对一个值达成共识？

两阶段协议

// === Phase 1: Prepare ===
// Proposer 发送 Prepare(n) 给 Acceptor，n 是提案编号
// Acceptor 如果 n > maxPromise，回复 Promise(n, acceptedProposal?)
// Acceptor 承诺不再接受 < n 的提案

// === Phase 2: Accept ===
// Proposer 收到多数 Promise，选择值 v：
//   - 如果有任何 Promise 带了 acceptedProposal，取 proposal number 最大的那个值
//   - 否则用自己的值
// Proposer 发送 Accept(n, v)
// Acceptor 接受（除非已经 Promise 了更高的 n）
// Learner 从 Acceptor 学习到最终的值

Paxos vs Raft 的核心差异

维度PaxosRaft
Leader无强 Leader（任何节点都可以提案）强 Leader（所有请求通过 Leader）
日志顺序日志洞（hole）可能存在日志严格顺序
易理解性极难（"世界上最难理解的算法"）设计目标就是易理解

5.2 ZAB（ZooKeeper Atomic Broadcast）

ZAB 是 ZooKeeper 的核心协议，和 Raft 同类（强 Leader）但先于 Raft 出现。

ZAB 与 Raft 的相似

  • 强 Leader（ZooKeeper 里叫 Leader）
  • Leader 发现（Leader Election）
  • 日志复制（Synchronization Phase）
  • 多数派提交

ZAB 与 Raft 的不同

  • ZAB 保证 FIFO 全序广播；Raft 保证日志顺序
  • ZAB 的 epoch 类比 Raft 的 term
  • ZAB 有专门的 Discovery / Synchronization / Broadcast 三阶段
  • ZAB 对日志洞（gap）有不同处理
架构师的选择标准：需要共识服务，首选 etcd（Raft）或 ZooKeeper（ZAB），不要自己实现共识算法——这是分布式系统里最难正确实现的部分，极小的细节错误就会导致数据丢失。

06分布式事务：2PC / 3PC / TCC

分布式事务是"让多个服务的操作要么全部成功要么全部回滚"。没有完美方案——每种方案都在一致性、可用性、复杂度之间做权衡。

6.1 为什么不能直接用数据库事务

单数据库的 ACID 事务非常强大，但在微服务 / 多数据库场景下不可用：

  • 每个服务有自己的数据库（"Database per Service"原则）
  • 跨数据库的事务需要分布式协调
  • 分布式协调的成本非常高（性能下降 10-100x）

6.2 两阶段提交（2PC）

2PC 的问题

问题描述影响
Coordinator 单点 Phase 2 开始后 Coordinator 挂了，Participant 永远不知道该 Commit 还是 Abort 事务悬挂（In-doubt 状态），需人工介入
阻塞性 Participant Prepared 后必须持有锁，等待 Phase 2 决定 高并发下锁竞争严重，吞吐低
同步等待 Coordinator 必须等所有 Participant 响应 最慢的节点决定整个事务延迟
跨网络 2 个 RTT Prepare + Commit 各需要一次网络往返 延迟至少是 2 × RTT

6.3 三阶段提交（3PC）

在 Phase 1 和 Phase 2 之间加了一个 PreCommit 阶段，让 Participant 在 Coordinator 挂后能自主决定，解决了 2PC 的阻塞问题。

但 3PC 引入了新问题：在网络分区时可能导致不一致提交。实际上几乎没有系统使用纯粹的 3PC。

6.4 TCC（Try-Confirm-Cancel）

TCC 是业务层的补偿事务，完全避开数据库的分布式事务：

Try（尝试）

预留资源，不真正执行。

例：库存预留 10 件（不减库存，冻结库存）

Confirm（确认）

真正执行业务逻辑。

例：将冻结的库存真正扣减

Cancel（取消）

释放预留的资源（补偿）。

例：解冻库存，归还 10 件

TCC 的关键要求

  • Try 必须幂等：重试多次结果一致
  • Confirm 必须幂等：重试不会双扣
  • Cancel 必须幂等：重试不会多退
  • 空回滚：Try 超时后收到 Cancel，即使没做 Try 也能安全 Cancel
  • 防悬挂：Cancel 后来迟到的 Try 不能执行

TCC 与 2PC 对比

维度2PCTCC
锁数据库行锁（Prepared 期间持有）业务层资源预留（冻结），不用 DB 锁
实现复杂度低（框架支持）高（每个接口要实现 3 个方法）
侵入性低高（业务逻辑需要改造）
适用同一数据库 / XA跨服务、高并发
性能差（阻塞）好（无 DB 锁）

07Saga / Outbox / Event Sourcing：最终一致性事务

7.1 Saga 模式

Saga：把一个分布式事务拆成一系列本地事务，每个本地事务都有对应的补偿事务（compensating transaction）。如果某一步失败，从后向前执行补偿，回滚到初始状态。

两种 Saga 编排方式

Choreography（编舞）

每个服务监听事件，完成自己的任务后发布新事件。

流程：OrderService → OrderPlaced → InventoryService → InventoryReserved → PaymentService → PaymentCompleted → OrderConfirmed

优点：低耦合，无中心化瓶颈

缺点：难追踪整个事务流，难调试，隐式业务流程

Orchestration（编排）

有一个中心化的 Saga Orchestrator 协调所有步骤。

优点：流程清晰，易追踪，易处理异常

缺点：Orchestrator 是新的耦合点；可能成为瓶颈

Saga 补偿示例

// 订单 Saga：下单 → 扣库存 → 支付
// 每一步有对应的补偿

class CreateOrderSaga {
    enum class Step { CreateOrder, ReserveInventory, ProcessPayment };
    std::vector<Step> completed_steps;

    // 正向步骤
    Result create_order(OrderId id, ...) {
        auto r = order_service_.create(id, ...);
        if (r.ok()) completed_steps.push_back(Step::CreateOrder);
        return r;
    }
    Result reserve_inventory(OrderId id) {
        auto r = inventory_service_.reserve(id);
        if (r.ok()) completed_steps.push_back(Step::ReserveInventory);
        return r;
    }
    Result process_payment(OrderId id, Money amount) {
        auto r = payment_service_.charge(id, amount);
        if (r.ok()) completed_steps.push_back(Step::ProcessPayment);
        return r;
    }

    // 补偿（从后向前，每步都必须幂等）
    void compensate() {
        for (auto it = completed_steps.rbegin(); it != completed_steps.rend(); ++it) {
            switch (*it) {
            case Step::ProcessPayment: payment_service_.refund(saga_id_); break;
            case Step::ReserveInventory: inventory_service_.release(saga_id_); break;
            case Step::CreateOrder: order_service_.cancel(saga_id_); break;
            }
        }
    }

    // 执行 saga
    void execute(OrderId id, ...) {
        if (!create_order(id, ...).ok()) { compensate(); return; }
        if (!reserve_inventory(id).ok()) { compensate(); return; }
        if (!process_payment(id, amount).ok()) { compensate(); return; }
    }
};
Saga 的三个关键陷阱：
  1. 补偿不能保证原子性：补偿执行到一半也可能失败 → 需要补偿本身也是幂等的，并有重试机制
  2. 脏读：其他事务可能在 Saga 完成前读到"中间态"数据 → 需要在应用层处理（语义锁、乐观并发）
  3. 没有隔离性：多个并发 Saga 可能互相干扰 → 需要明确设计并发控制

7.2 Outbox 模式：可靠地发布事件

Outbox 模式：在同一个数据库事务中，同时写入业务数据和"待发布事件"到 outbox 表。后台进程负责把 outbox 事件发布到消息队列。保证"业务操作"和"事件发布"的原子性。

为什么需要 Outbox？

最直觉的做法是"操作成功后发消息"，但这不可靠：

❌ 不可靠的做法

tx.begin();
order_repo.save(order);  // 写 DB
tx.commit();
event_bus.publish(OrderPlaced{...}); // 如果这里崩溃？
// DB 已提交，事件没发 → 数据不一致！

✅ Outbox 模式

tx.begin();
order_repo.save(order);  // 写业务 DB
outbox.insert(OrderPlaced{...});  // 同一事务写 outbox
tx.commit();  // 原子！要么都成功，要么都失败
// 后台 Outbox Poller 读 outbox，发消息，标记已发

Outbox Poller 实现

// outbox 表结构
// CREATE TABLE outbox (
//   id         UUID PRIMARY KEY,
//   topic      TEXT,
//   payload    JSONB,
//   created_at TIMESTAMPTZ,
//   published  BOOLEAN DEFAULT FALSE
// );

class OutboxPoller {
    IOutboxRepository& outbox_;
    IEventBus& bus_;
public:
    void poll() {
        auto events = outbox_.find_unpublished(limit=100);
        for (const auto& e : events) {
            try {
                bus_.publish(e.topic, e.payload);
                outbox_.mark_published(e.id);  // 发成功才标记
            } catch (...) {
                // 失败了，下次重试（至少一次语义）
                log.warn("Failed to publish event {}", e.id);
            }
        }
    }
    // 定时调用 poll()，或用 Change Data Capture (CDC) 监听 DB 变更
};

Change Data Capture（CDC）方式

更高效的 Outbox 实现：不用轮询，而是通过 Debezium 监听 PostgreSQL 的 WAL（Write-Ahead Log），自动把 outbox 表的变更发到 Kafka。

Outbox + CDC 是事件驱动微服务的最可靠发布方案。CDC 工具：Debezium（支持 PostgreSQL/MySQL/MongoDB 等）。不需要改业务代码，只需在 DB 层监听变更。

08分布式 ID 生成

8.1 需求分析

分布式 ID 需要满足：

  • 全局唯一：多节点不冲突
  • 趋势递增（可选）：对数据库 B+ 树插入友好
  • 信息隐藏（可选）：不暴露业务量（防竞争对手计算订单数）
  • 高性能：不成为瓶颈
  • 单调性（可选）：相关事件的 ID 有先后顺序

8.3 Snowflake 深入

// Snowflake 64 位结构：
// 1 bit:  符号位（0，始终为正）
// 41 bit: 时间戳（毫秒，2^41 ≈ 69 年）
// 10 bit: 机器 ID（worker ID，最多 1024 台机器）
// 12 bit: 序列号（同一毫秒内最多 4096 个 ID）
// 总计：最高 1024 × 4096 = 4,194,304 个 ID/ms = 40 亿 ID/s

class Snowflake {
    static constexpr int64_t EPOCH = 1700000000000LL; // 自定义 epoch（ms）
    static constexpr int64_t WORKER_BITS   = 10;
    static constexpr int64_t SEQUENCE_BITS = 12;
    static constexpr int64_t MAX_SEQUENCE  = (1 << SEQUENCE_BITS) - 1;  // 4095

    int64_t worker_id_;
    int64_t sequence_ = 0;
    int64_t last_ts_  = -1;
    std::mutex mu_;

    int64_t current_ms() {
        using namespace std::chrono;
        return duration_cast<milliseconds>(system_clock::now().time_since_epoch()).count();
    }

public:
    explicit Snowflake(int64_t worker_id) : worker_id_(worker_id) {}

    int64_t next_id() {
        std::lock_guard<std::mutex> lock{mu_};
        int64_t now = current_ms() - EPOCH;
        if (now == last_ts_) {
            sequence_ = (sequence_ + 1) & MAX_SEQUENCE;
            if (sequence_ == 0) {
                // 同一毫秒内序列号耗尽，等到下一毫秒
                while (current_ms() - EPOCH == now) {}
                now = current_ms() - EPOCH;
            }
        } else {
            sequence_ = 0;
        }
        last_ts_ = now;
        return (now << (WORKER_BITS + SEQUENCE_BITS))
             | (worker_id_ << SEQUENCE_BITS)
             | sequence_;
    }
};
Snowflake 的时钟回拨问题：如果系统时钟回拨，生成的 ID 可能重复。对策：
  • 检测到时钟回拨时，等待时钟追上（允许小幅回拨 < 10ms）
  • 或者记录上次使用的时间戳，回拨时拒绝生成，触发告警
  • 用 NTP 配置，不允许时钟突然跳变（用 slew 模式而不是 step 模式）

8.4 UUID v7（2023 RFC 9562）

// UUID v7 结构（128 bit）：
// 48 bit: Unix timestamp（毫秒）— 趋势递增
// 4 bit:  version = 7
// 12 bit: rand_a（可用作序列号）
// 2 bit:  variant
// 62 bit: rand_b（随机）

// 优势：
// - 标准 UUID 格式，现有工具都能处理
// - 毫秒级趋势递增，对 DB 索引友好
// - 不需要 worker ID，无中心化
// - 充分的随机性（62 bit）

// 推荐：新系统首选 UUID v7，而不是 v4

09分布式锁：Redis / etcd / ZooKeeper

9.1 分布式锁的需求

什么时候需要分布式锁？当多个进程/节点互斥访问共享资源时，且单机锁不够用：

  • 防止同一个任务被多个实例同时执行（分布式定时任务）
  • 限流（某用户的请求在集群级别串行化）
  • 协调资源访问（共享设备、共享文件）
先问自己：真的需要分布式锁吗？能否用数据库乐观锁（版本号）或幂等性设计来替代？分布式锁是一个强耦合、高延迟、有单点风险的方案，能避则避。

9.2 Redis 分布式锁（SET NX）

// 正确的 Redis 锁实现（SET NX + PX + Lua）

// 加锁（原子性：SET key value NX PX timeout）
SET lock:resource client_uuid NX PX 30000
// NX: 只在 key 不存在时设置
// PX 30000: 30 秒超时（防止锁持有者崩溃导致死锁）
// value = 唯一 ID（确保只有锁的持有者才能释放）

// 释放锁（Lua 脚本保证原子性：检查 + 删除）
// eval "if redis.call('get',KEYS[1])==ARGV[1]
//      then return redis.call('del',KEYS[1])
//      else return 0 end"
// 1 lock:resource client_uuid

// C++ 包装
class RedisDistLock {
    RedisClient& redis_;
    std::string  key_;
    std::string  token_;      // 唯一标识符
    std::chrono::milliseconds ttl_;
public:
    bool try_acquire() {
        token_ = generate_uuid();
        auto result = redis_.set(key_, token_,
            SetOptions{.nx=true, .px=ttl_.count()});
        return result == "OK";
    }
    void release() {
        static const char* lua = R"""(
            if redis.call('get',KEYS[1])==ARGV[1]
            then return redis.call('del',KEYS[1])
            else return 0 end
        )""";
        redis_.eval(lua, {key_}, {token_});
    }
};

Redis 锁的问题

Redis 锁的三个安全问题：
  1. GC 停顿导致锁过期：进程 GC 暂停 → 锁超时被释放 → 另一个进程获锁 → 两个进程都以为自己持有锁。解决：持有锁后必须给操作加版本号（fencing token），数据库拒绝旧版本的写。
  2. Redis 主从延迟：主节点写锁后崩溃，从节点未复制 → 新主节点里没有这个锁 → 另一个进程又获锁。解决：RedLock（多数派写多个独立 Redis）
  3. 网络分区：持锁方与 Redis 分区 → 锁超时 → 另一方获锁 → 双写。
Redis 锁适合"容忍偶发不一致"的场景（防重复、限流）；不适合"绝对互斥"的场景（用 etcd/ZooKeeper）。

9.3 etcd / ZooKeeper 分布式锁（更强保证）

// etcd 锁：基于 Lease + Revision 实现有序等待
// 1. 创建 Lease（TTL 内有效）
// 2. 用 Lease 创建 key（prefix + UUID）
// 3. 查看所有同 prefix 的 key，按 revision 排序
// 4. 如果自己的 revision 最小 → 获锁
// 5. 否则 watch revision 比自己小的那个 key，等待它消失

// 这个机制保证：
// - 公平（先申请先得，FIFO）
// - 持有者崩溃 → Lease 过期 → key 自动删除 → 下一个等待者获锁
// - 基于 Raft，强一致，不存在 Redis 的主从延迟问题
方案一致性延迟可用性适用
Redis 单节点弱低中限流、防重复（容忍偶发失效）
RedLock中（有争议）中中比单节点更强，仍有边角 case
etcd强（Raft）中中关键资源互斥、Leader 选举
ZooKeeper强（ZAB）中中成熟稳定，大规模使用

10.2 ISR（In-Sync Replicas）机制

ISR：与 Leader 足够同步的副本集合。足够同步的定义：在最近 replica.lag.time.max.ms（默认 30s）内发送过 fetch 请求，且没有落后 Leader 超过 replica.lag.max.messages。

HW（High Watermark）：消费者可见的最高偏移量

Leader: [0][1][2][3][4][5]   ← LEO (Log End Offset) = 6
Replica1: [0][1][2][3]       ← LEO = 4
Replica2: [0][1][2][3][4]    ← LEO = 5
ISR = {Leader, Replica1, Replica2}
HW = min(ISR members' LEO) = 4
消费者只能读到 offset < HW 的消息（即 0-3）

ISR 的动态收缩：如果 Replica 落后太多 → 从 ISR 中移除 → HW 可能提升（因为落后的不参与计算）。

10.3 Producer 的 acks 配置

acks 值含义消息丢失风险延迟
acks=0不等确认，fire-and-forget高最低
acks=1Leader 写入即返回（Follower 未必写入）中（Leader 挂则丢）中

10.4 Exactly-Once 语义（EOS）

Kafka 0.11+ 支持 Exactly-Once，通过两个机制：

幂等 Producer

每个 Producer 有 Producer ID + Sequence Number。Broker 拒绝重复的 (ProducerID, PartitionID, SequenceNumber)，保证单分区内恰好一次。

enable.idempotence=true

事务 API

Producer 可以原子性地写多个分区。Consumer 设置 isolation.level=read_committed 只读取已提交的事务消息。

producer.initTransactions();
producer.beginTransaction();
producer.send(record1);
producer.send(record2);
producer.commitTransaction();  // 或 abortTransaction()

10.5 消费者组 Rebalance

当消费者组内有成员加入 / 退出 / 崩溃时，触发 Rebalance——重新分配 Partition 到 Consumer。

Rebalance 的代价：期间所有消费者停止消费（Stop-The-World）→ 造成消费延迟。

减少 Rebalance 的配置：

# Consumer 配置
session.timeout.ms=90000       # 增大心跳超时，避免误判
heartbeat.interval.ms=10000
max.poll.interval.ms=300000    # 增大 poll 间隔，避免处理慢导致踢出

# Kafka 2.4+ 的 Cooperative Rebalance（渐进式，不需要 Stop-The-World）
partition.assignment.strategy=CooperativeStickyAssignor

10.6 Log Compaction（日志压缩）

不同于定时删除（TTL），Log Compaction 保留每个 key 的最新值，删除旧值：

原始日志:
offset 0: user_1 → {name:"Alice"}
offset 1: user_2 → {name:"Bob"}
offset 2: user_1 → {name:"Alice Smith"}  ← 更新
offset 3: user_3 → {name:"Charlie"}
offset 4: user_2 → null   ← 删除（tombstone）

压缩后（只保留每个 key 的最新值）:
offset 2: user_1 → {name:"Alice Smith"}
offset 3: user_3 → {name:"Charlie"}
(user_2 被墓碑删除后也会最终清理)

适用场景：将 Kafka 作为物化视图存储——新消费者可以从头读取，得到当前状态的完整快照。

11服务发现与健康检查

11.1 服务发现的两种模式

Client-Side Discovery

客户端查询注册中心，获取服务实例列表，自己做负载均衡。

例：Netflix Eureka + Ribbon

优点：灵活，客户端可实现复杂策略（版本感知、区域感知）

缺点：客户端和注册中心耦合，多语言实现成本高

Server-Side Discovery

客户端请求发给 Load Balancer / API Gateway，LB 查注册中心转发。

例：Kubernetes Service + kube-proxy / Envoy

优点：客户端无感，与语言无关

缺点：多一跳，LB 是单点（需 HA）

高频面试题 · 分布式系统

所有公司架构师面试的必考点：CAP、Raft、分布式事务、Kafka。
Q1 请准确解释 CAP 定理。它的 C 和 ACID 的 C 一样吗？
P0主题：CAP 基础难度：⭐⭐⭐必问题
核心答案（先纠正常见误读）

CAP 的精确表述：分布式系统在网络分区时，只能在一致性（C）和可用性（A）之间二选一。不是任何时候都二选一——没有分区时 C 和 A 都能拿到。

  • C = Linearizability：每次读都返回最新写入值，或者错误。所有节点表现如同一台机器。
  • A：每次请求都得到非错误响应（可能是旧数据）。
  • P：网络分区时系统继续工作（节点间消息可丢失/延迟）。

CAP 的 C ≠ ACID 的 C：

  • CAP 的 C：操作的"原子可见性"（Linearizability）
  • ACID 的 C：事务后数据满足约束（业务规则不被破坏）
陷阱 1：选 "CA" 系统

"我用关系数据库，它是 CA 系统" — 错。CA 系统不存在，因为分布式系统必须容忍网络分区。"传统单机 RDBMS" 严格说不是分布式系统。

陷阱 2：选 AP 就没有一致性

错。AP 系统在没有分区时可以提供强一致；只是在分区发生时优先返回（可能是旧的）数据。

追问：PACELC 是什么？

If Partition, A or C; Else, Latency or Consistency. — 即使没有分区，强一致性（Linearizability）也需要额外同步开销（更高延迟）。这是 CAP 没说清的一半。DynamoDB 是 PA/EL，etcd 是 PC/EC。

Q2 讲一下 Raft 的 Leader 选举过程。为什么 candidate 必须"日志最新"才能当选？
P0主题：共识算法难度：⭐⭐⭐⭐
选举过程
  1. Follower 选举超时（150-300ms 随机），自增 currentTerm，变 Candidate
  2. 给所有节点发 RequestVote(term, lastLogIndex, lastLogTerm)
  3. 每个 Follower 在同一 term 内只投一票（先到先得），且要求 candidate 日志 ≥ 自己
  4. Candidate 获得多数派（> N/2）选票 → 成为 Leader
  5. 立即发心跳（AppendEntries 空 entries）宣告权威，其他节点退回 Follower
为什么必须"日志最新"才能当选（核心问题）

Raft 的最关键不变量：已经提交的日志永远不能被覆盖/丢失。

已提交 = 多数节点写入。新 Leader 一定也来自多数派，所以多数派的交集里至少有一个节点拥有所有已提交日志。投票规则保证只有"拥有所有已提交日志"的节点才能当选 Leader——这样新 Leader 不会回滚已提交的日志。

日志"更新"的比较规则
  1. 先比 lastLogTerm（最后一条日志的 term），大的更新
  2. term 相同，再比 lastLogIndex（日志长度），长的更新
追问：为什么选举超时要随机化？

避免多个 Follower 同时超时同时发起选举导致 split vote（没人能拿到多数票）。随机化让一个节点通常比其他节点早一点超时，能先获得多数票。

面试加分：Pre-Vote 优化

不稳定网络下，孤立节点会反复自增 term 发起选举，term 涨得很快。当它重新加入集群时会扰乱当前 Leader。Pre-Vote 是先用"试探票"问 majority 自己有没有机会当选，有才正式发起选举。这是面试加分点。

Q3 2PC、Saga、TCC、Outbox 这四个分布式事务方案怎么选？
P0主题：分布式事务难度：⭐⭐⭐⭐
核心答案 · 决策矩阵
方案一致性性能实现复杂度典型适用
2PC强差低同 DB 内（XA），不适合微服务
Saga最终好中跨服务长流程（订单/物流/支付）
TCC最终好高（每接口 3 个方法）高并发资金类（账户冻结）
Outbox最终好低业务写 + 事件发布原子保证
选型关键问题
  • 能接受最终一致吗？不能 → 2PC；能 → Saga/TCC/Outbox
  • 需要事务-事件双写一致？→ Outbox（最常用的"小事务"）
  • 有金融级别的并发要求？→ TCC（资源预留比锁更高效）
  • 长流程跨服务编排？→ Saga（编舞或编排）
陷阱：Saga 没有隔离性

Saga 执行中，其他事务可能读到"中间态"数据。必须在应用层处理：语义锁（状态字段标记"处理中"）、乐观并发（版本号）、补偿事务幂等性。

面试加分：Outbox 实战

"我们用 Debezium 监听 PostgreSQL 的 WAL，自动把 outbox 表的变更发到 Kafka——这样应用层只需把事件写到同一事务的 outbox 表，发布完全异步且零业务代码改动。" 这是 CDC + Outbox 的标准组合。

Q4 Kafka 怎么实现 Exactly-Once Semantics？需要什么配合？
P0主题：消息中间件难度：⭐⭐⭐⭐
核心答案

Kafka EOS 由两个机制组成：

  1. 幂等 Producer（enable.idempotence=true）：每个 Producer 有 PID + Sequence Number，Broker 拒绝重复 (PID, Partition, Seq)。保证单分区内恰好一次。
  2. 事务 API（initTransactions / beginTransaction / commitTransaction）：跨分区原子写。Consumer 设置 isolation.level=read_committed 只读已提交。
EOS 的完整链路
// 典型 read-process-write 模式（Consumer-Producer 一体）
producer.initTransactions();
while (running) {
    ConsumerRecords records = consumer.poll(timeout);
    producer.beginTransaction();
    for (record : records) {
        ProducerRecord output = process(record);
        producer.send(output);
    }
    producer.sendOffsetsToTransaction(offsets, consumerGroupId);
    producer.commitTransaction();
}
陷阱：EOS 不是端到端的

Kafka EOS 只保证从 Producer 到 Broker，再从 Broker 到 Consumer 的处理过程。如果 Consumer 处理后写入下游系统（DB / 另一个 MQ），那段链路Kafka 管不了，需要业务自己保证幂等。

追问：性能代价多大？

开启 EOS 大约 10-20% 吞吐下降。对绝大多数业务可接受。但极致性能场景（如百万 TPS 实时管道）可能用 At-Least-Once + 业务幂等。

Q5 Snowflake 算法的核心是什么？时钟回拨怎么处理？
P1主题：分布式 ID难度：⭐⭐⭐
核心结构（64-bit）
1 bit  : 符号位（0，恒正）
41 bit : 时间戳（毫秒，2^41 ≈ 69 年）
10 bit : 机器 ID（workerId，最多 1024 台）
12 bit : 序列号（同毫秒内最多 4096 个 ID）

吞吐上限 = 1024 × 4096 / ms = 4M ID/ms ≈ 4B ID/s
时钟回拨处理（三种策略）
  1. 小回拨（< 10ms）：等待时钟追上再生成 ID
  2. 大回拨：拒绝生成 ID，抛异常 + 触发告警，等运维介入
  3. 预防性：用 NTP slew 模式（缓慢调整）而不是 step 模式（瞬间跳跃）
面试加分：推荐 UUID v7 替代

"如果是新系统，我会用 UUID v7（2023 RFC 9562 标准）替代 Snowflake——它毫秒时间戳前缀，趋势递增对 B+ Tree 索引友好；标准 UUID 格式（128 bit）所有工具都支持；不需要分配 workerId，无中心化。"

Q6 用 Redis 做分布式锁，可能有什么问题？怎么解决？
P0主题：分布式锁难度：⭐⭐⭐⭐
Redis 锁的三大问题（Martin Kleppmann 的著名分析）
  1. GC/STW 暂停导致锁过期：客户端获锁后 GC 暂停 30 秒，锁过期被另一客户端拿到，两者同时认为自己持锁 → 双写。
  2. 主从延迟丢锁：客户端在主节点写锁后主节点崩溃，从节点未复制成功 → 新主节点不知道这个锁 → 另一客户端再获锁。
  3. 网络分区：持锁方与 Redis 分区，锁超时被释放，另一方获锁，但持锁方继续操作。
解决方案
  • Fencing Token（推荐）：锁返回单调递增 token，每次写资源带 token，资源系统拒绝旧 token。即使两个客户端"以为"自己持锁，资源系统只接受新 token。
  • RedLock：写到 N/2+1 个独立 Redis 实例才算成功。降低主从延迟问题，但有时钟假设争议。
  • 选 etcd/ZK：基于 Raft/ZAB 强一致，不存在主从延迟。
什么时候用什么
  • 容忍偶发失效（限流、防重复点击）→ Redis 单实例足够
  • 绝对互斥（库存扣减、Leader 选举）→ etcd / ZooKeeper
  • 需要 fencing token → 锁服务返回单调递增版本号，下游资源校验

       return redis.call('del',KEYS[1]) else return 0 end"
