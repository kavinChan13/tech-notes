1. 模板 & 泛型编程概览

模板（Template） 是 C++ 实现 泛型编程（Generic Programming） 的核心机制。它让你编写与类型无关的代码，由编译器根据使用时的具体类型自动生成对应版本。

模板的核心价值

特性说明项目中的体现

2. 函数模板 (Function Template)

函数模板允许你编写一个函数"蓝图"，编译器在调用时根据参数自动推导类型并生成具体函数。

编写模板
template<typename T>
→
调用时
serialize(myInt, payload)
→
编译器推导
T = int
→
实例化
生成 serialize<int>

项目实例 1：序列化/反序列化 MessageHelpers.hpp

// src/Core/proxy/MessageHelpers.hpp
template <typename T>
inline void serialize(const T& var, Payload& payload)
{
    payload.insert(payload.end(),
        reinterpret_cast<const char*>(&var),
        reinterpret_cast<const char*>(&var) + sizeof(T));
}

template <typename T>
inline void deserialize(T& var, Reader& reader)
{
    memcpy(&var, &*reader, sizeof(T));
    reader += sizeof(T);
}

📌 解读：serialize<T> 和 deserialize<T> 不关心 T 是 int、uint32_t 还是自定义结构体——只要它是 POD 类型就可以直接 memcpy。编译器会针对每种调用类型生成独立版本。

项目实例 2：线性插值 VoltageScenario

// src/NetconfAgent/Scenario/VoltageManagement/ConfigureInputVoltageThresholdsScenario.hpp
template <typename T>
T linearInterpolate(T inputValue, T inputMin, T inputMax,
                     T outputMin, T outputMax)
{
    if (inputValue <= inputMin) return outputMin;
    if (inputValue >= inputMax) return outputMax;

    return outputMin + (inputValue - inputMin)
           * (outputMax - outputMin) / (inputMax - inputMin);
}

📌 这个函数可以同时处理 double、float、int 等任意数值类型，一份代码覆盖所有场景。

3. 类模板 (Class Template)

类模板允许你参数化整个类的行为和数据类型。STL 中的 std::vector<T>、std::map<K,V> 都是类模板。

项目实例 1：线程安全封装 ReadWriteLockedValue<T>

// src/Core/ReadWriteLockedValue.hpp
template <typename T>
class ReadWriteLockedValue
{
public:
    explicit ReadWriteLockedValue(const T& value = T()) : value_(value) {}

    T get() const {
        std::shared_lock<std::shared_mutex> lock(mutex_);
        return value_;
    }

    void set(T&& value) {
        std::unique_lock<std::shared_mutex> lock(mutex_);
        value_ = std::forward<T>(value);
    }

    // 嵌套的成员函数模板 —— 接受任意可调用对象
    template <typename Func>
    auto read(Func func) const {
        std::shared_lock<std::shared_mutex> lock(mutex_);
        return func(value_);
    }

    template <typename Func>
    auto write(Func func) {
        std::unique_lock<std::shared_mutex> lock(mutex_);
        return func(value_);
    }

private:
    mutable std::shared_mutex mutex_;
    T value_;
};

📌 设计亮点：

  • T 作为类模板参数，封装了任意类型的线程安全读写
  • read(Func) 和 write(Func) 是成员函数模板——类模板中再嵌套函数模板，双重泛型
  • 使用 std::shared_mutex 实现读写分离锁策略
  • 使用 std::forward 实现完美转发

项目实例 2：消息适配器 Adapter<T>

// src/Core/proxy/Adapter.hpp
template <typename TMessageType>
struct Adapter : public TMessageType  // ← 继承自模板参数！
{
    std::string getName() const;
    std::string toString() const;

    void to(Payload& payload) const {
        static_assert(sizeof(TMessageType) == sizeof(Adapter<TMessageType>),
            "Default Adapter assume const size message");
        payload.resize(sizeof(TMessageType));
        memcpy(&payload.front(), this, sizeof(TMessageType));
    }

    void from(const Payload& payload) {
        static_assert(sizeof(TMessageType) == sizeof(Adapter<TMessageType>),
            "Default Adapter assume const size message");
        memcpy(this, &payload.front(), sizeof(TMessageType));
    }
};

📌 CRTP-like 模式：Adapter<T> 直接继承 T——这意味着 Adapter<SApiInetVsftpStopReq> 既是一个 SApiInetVsftpStopReq，又具备序列化/反序列化的能力。默认实现用 memcpy，特化版本可以自定义。

4. 模板特化 (Template Specialization)

模板特化允许你为特定类型提供不同的实现。分为全特化（完全指定类型）和偏特化（部分指定类型）。

4.1 全特化：MessageInfo SApiInetVsftpStop.hpp

// 主模板声明（永远不会直接使用，强制使用特化版本）
template <typename TMsg>
struct MessageInfo;  // 只有声明，没有定义！

// 全特化：为 SApiInetVsftpStopReq 提供完整定义
template <>
struct MessageInfo<::SApiInetVsftpStopReq>
{
    enum { RequestId  = API_INET_VSFTP_STOP_REQ_MSG };
    enum { ResponseId = API_INET_VSFTP_STOP_RESP_MSG };
    using RequestMsg  = ::SApiInetVsftpStopReq;
    using ResponseMsg = ::SApiInetVsftpStopResp;
    enum { DefaultTimeoutDuration = 30 };
    static const TaskId Task = HWAPI_IP_SERVICE;
};

📌 精妙设计：主模板只声明不定义。如果有人对一个没有特化的消息类型使用 MessageInfo，编译器会报错「不完整类型」——这是一个编译期约束，确保每种消息都必须提供配置信息。

4.2 全特化：Adapter 自定义序列化

// 对 VsftpStopReq 全特化 Adapter，自定义序列化逻辑
template <>
struct Adapter<::SApiInetVsftpStopReq> : public ::SApiInetVsftpStopReq
{
    TTransactionID transactionId;

    void to(Payload& payload) const {
        payload.reserve(sizeof(::SApiInetVsftpStopReq));
        serialize(transactionId, payload);  // 只序列化 transactionId
    }

    void from(const Payload& payload) {
        auto reader = payload.begin();
        deserialize(transactionId, reader);
    }
};

📌 默认 Adapter<T> 用 memcpy 整体拷贝；但 VsftpStop 消息的特化版只序列化 transactionId。这就是全特化的威力——同一接口，不同实现。

// src/Core/FmService/Faults/Fault2930Handler.cpp

    {

📌 if constexpr（C++17）在编译期决定走哪个分支——不满足条件的分支代码完全不会被编译。结合 static_assert 可以在编译期捕获不支持的类型。

8. 模板最佳实践

✅ DO

  • 使用 static_assert 提供友好的编译期错误信息（如 MakeMsg 中的 Req/Resp 校验）
  • 使用 if constexpr 替代 SFINAE 实现编译期分支（C++17，如 Fault2930Handler）
  • 使用显式实例化 当类型集合有限且希望隐藏实现时（如 convertValue）
  • 使用类型约束（C++20 Concepts 或 static_assert）限制模板可接受的类型
  • 保持模板代码简短，将复杂逻辑提取为非模板辅助函数

❌ DON'T

  • 不要把所有模板定义放在 .cpp 中却忘了显式实例化 → 链接错误
  • 不要过度使用模板——如果只有一两个类型，直接重载可能更清晰
  • 不要忽略编译器的模板错误信息——从最后一个实例化点开始向上阅读
  • 避免模板嵌套过深导致编译时间爆炸和难以阅读的错误信息

9. 面试深度专题 ⭐ Senior C++

本章节面向高级 C++ 工程师面试，聚焦模板的底层机制、常见陷阱、现代 C++ 演进，以及大厂高频考点。

9.1 SFINAE — 模板元编程的灵魂

SFINAE（Substitution Failure Is Not An Error）：模板参数替换失败时，编译器不会报错，只会把该重载从候选集中默默移除。这是实现"编译期 if"和类型检查的核心。

// 经典 SFINAE：检测类型是否有 serialize() 成员
template <typename T>
auto hasSerialize(int) -> decltype(std::declval<T>().serialize(), std::true_type{});

template <typename T>
auto hasSerialize(...) -> std::false_type;

template <typename T>
constexpr bool has_serialize_v = decltype(hasSerialize<T>(0))::value;

// 使用 std::enable_if 启用/禁用重载
template <typename T, std::enable_if_t<std::is_integral_v<T>, int> = 0>
T safe_add(T a, T b) { return a + b; }     // 仅整型

template <typename T, std::enable_if_t<std::is_floating_point_v<T>, int> = 0>
T safe_add(T a, T b) { return a + b + 1e-9; } // 浮点

9.2 Concepts (C++20) — SFINAE 的现代替代

C++20 引入 Concepts，让模板约束像函数签名一样清晰，错误信息可读性大幅提升。

❌ SFINAE 年代

template <typename T,
  std::enable_if_t<
    std::is_integral_v<T> &&
    !std::is_same_v<T,bool>,
    int> = 0>
T next(T x) { return ++x; }

// 错误信息: 长达几十行
// "no matching function..."

✅ C++20 Concepts

template <typename T>
concept Integer =
    std::integral<T> &&
    !std::same_as<T,bool>;

template <Integer T>
T next(T x) { return ++x; }

// 错误信息：清晰一句话
// "T does not satisfy Integer"

9.3 CRTP — 静态多态模式 零开销抽象

CRTP（Curiously Recurring Template Pattern）：基类模板以派生类自身为模板参数。用静态绑定替代虚函数，既复用代码又避免虚表开销。

template <typename Derived>
class Shape {
public:
    double area() const {
        // 静态下转型 — 编译期解析，无虚表查找
        return static_cast<const Derived*>(this)->areaImpl();
    }
};

class Circle : public Shape<Circle> {
    double r_;
public:
    double areaImpl() const { return 3.14 * r_ * r_; }
};

// 对比: 虚函数每次调用有间接跳转 + 阻碍内联
// CRTP 可完全内联 → 性能与手写无差别

典型应用：std::enable_shared_from_this、Boost.Iterator、Eigen 表达式模板

9.4 Type Erasure — 运行时泛型

类型擦除：在编译期不暴露具体类型，运行期通过"虚接口 + 模板包装"统一处理。标准库典范：std::function、std::any、std::shared_ptr 的 deleter。

// 简化版 std::function 实现
template <typename Sig> class Function;

template <typename R, typename... Args>
class Function<R(Args...)> {
    struct Concept {                              // 内部虚接口
        virtual R call(Args...) = 0;
        virtual ~Concept() = default;
    };
    template <typename F>
    struct Model : Concept {                       // 模板实现
        F f_;
        Model(F f) : f_(std::move(f)) {}
        R call(Args... a) override { return f_(std::forward<Args>(a)...); }
    };
    std::unique_ptr<Concept> impl_;
public:
    template <typename F>
    Function(F f) : impl_(std::make_unique<Model<F>>(std::move(f))) {}
    R operator()(Args... a) { return impl_->call(std::forward<Args>(a)...); }
};

9.5 变长模板 & Fold Expressions (C++17)

// ① 递归展开 (C++11 风格)
void print() {}
template <typename T, typename... Rest>
void print(const T& t, const Rest&... rest) {
    std::cout << t << ' ';
    print(rest...);   // 递归 → 需要终止函数
}

// ② 折叠表达式 (C++17) — 一行搞定
template <typename... Args>
void print(const Args&... args) {
    ((std::cout << args << ' '), ...);   // 逗号折叠
}

// ③ 四种折叠方向
(args + ...)        // 一元右折叠:  a1 + (a2 + (a3 + a4))
(... + args)        // 一元左折叠:  ((a1 + a2) + a3) + a4
(args + ... + 0)    // 二元右折叠:  带初值，支持空包
(0 + ... + args)    // 二元左折叠

// ④ 完美转发多参数
template <typename... Args>
auto makeVector(Args&&... args) {
    std::vector<std::common_type_t<Args...>> v;
    (v.push_back(std::forward<Args>(args)), ...);
    return v;
}

// 效果：编译时间下降 20-50%，目标文件更小

9.7 Two-Phase Name Lookup —— 完整规则

模板的名字查找分两阶段（C++ 标准 [temp.res]），这套规则是跨编译器移植的最大坑之一。MSVC 历史上不严格实施，导致代码在 GCC/Clang 上才暴露问题。

① 第一阶段：模板定义点（非依赖名字）

"非依赖名字"指与模板参数无关、不需要等到实例化才能确定的名字。模板定义时立即查找，遵循普通名字查找规则（不带 ADL）。

② 第二阶段：实例化点（依赖名字）

"依赖名字"指出现 T、T::xxx、f(t) （t 类型依赖 T）等的名字。直到模板被实例化为某个具体类型时才查找，此时会启用 ADL。

// 经典示例 — 易踩坑
void g(int) { std::cout << "int g\n"; }

template <typename T>
void f(T t) {
    g(42);    // 第 1 阶段：找到上面的 g(int)
    g(t);     // 第 2 阶段：依赖 T，到实例化时再查
}

namespace ns {
    struct X {};
    void g(X) { std::cout << "ns::g(X)\n"; }
}

f(ns::X{});  // 输出: "int g" 然后 "ns::g(X)" — 第 2 个走 ADL 找到 ns::g

依赖类型必须用 typename / template 消歧符

template <typename T>
void f() {
    typename T::iterator it;          // ✅ typename 告诉编译器这是类型
    T::iterator it2;                  // ❌ 编译器以为是静态成员（C++20 起部分场景可省略）

    t.template get<int>();         // ✅ .template / ::template / ->template 消歧
                                       // 否则 < 会被当成小于号
}

典型移植 bug：MSVC 通过 GCC 失败

template <typename T>
struct Base { void init() {} };

template <typename T>
struct Derived : Base<T> {
    void run() {
        init();                // ❌ GCC/Clang: 找不到 init
                                //    第 1 阶段查 init 时 Base<T> 是依赖类型，跳过
        this->init();          // ✅ 加 this-> 让 init 变依赖名字
        Base<T>::init();      // ✅ 或显式限定
    }
};
C++20 变化：[temp.res.general]/2 放宽规则——在"明显是类型"的位置（如 using A = T::B;、void f(T::B);、T::B var; 作为函数参数等），typename 可以省略。但模板体内的局部变量声明仍需要。

9.8 ADL（Argument-Dependent Lookup）与 Hidden Friend

ADL（参数依赖查找，又叫 Koenig lookup）：调用未限定函数名 f(x) 时，编译器除了普通查找，还会在 x 的实参类型所属命名空间查找。这是 C++ 实现"自由函数像成员函数一样自然"的关键。

ADL 让你能写 std::cout << x 而不是 std::operator<<(std::cout, x)

namespace ns {
    struct Box { int v; };
    std::ostream& operator<<(std::ostream& os, Box b) {
        return os << "Box{" << b.v << "}";
    }
}

ns::Box b{42};
std::cout << b;     // 找 std::operator<<(ostream&, ns::Box) → 没有
                    // ADL 在 ns 里找 operator<< → 找到

ADL 的 4 个"参数所属命名空间"规则

  1. 参数类型自身的命名空间（含别名）
  2. 类的直接和间接基类的命名空间
  3. 类模板的模板实参所属命名空间（vector<ns::Box> → 也找 ns）
  4. 类的外围类的命名空间

Hidden Friend Idiom —— 现代 C++ 的关键模式

在类内 friend 定义的函数只能通过 ADL 找到，不污染外层命名空间，且编译加速明显（不参与无关重载决议）。

// ❌ 老写法：所有 operator== 都进 ns 的查找池
namespace ns {
    struct Point { int x, y; };
    bool operator==(Point a, Point b) { return a.x==b.x && a.y==b.y; }
}

// ✅ 新写法：hidden friend
namespace ns {
    struct Point {
        int x, y;
        // 只有通过 Point 实参 ADL 才能找到，且无显式命名空间限定
        friend bool operator==(Point a, Point b) noexcept {
            return a.x==b.x && a.y==b.y;
        }
        friend std::ostream& operator<<(std::ostream& os, Point p) {
            return os << p.x << ',' << p.y;
        }
    };
}
三大收益：
  • 编译加速：模板实例化时，编译器只查与实参类型相关的函数，hidden friend 不出现在重载集中（除非 ADL 触发），重载决议复杂度骤降
  • 避免歧义重载：不参与无关命名空间的查找
  • 更好的概念匹配：自定义 operator< + Hidden friend 让类型天然满足 std::totally_ordered

Eric Niebler / Howard Hinnant / Niall Douglas 等核心专家都强烈推荐对所有非成员二元运算符使用 hidden friend。std::format 的特化、operator<=> 的实现几乎清一色用这个模式。

禁用 ADL 的技巧（如有需要）

std::vector<int>{1,2,3};
(::std::move)(x);          // 括号包起来 → 不走 ADL
using std::swap;
swap(a, b);             // 标准 ADL swap 模式 — 优先用户提供的 swap

9.9 Tag Dispatch —— 编译期分派的老牌方案

Tag dispatch：用空标签类型作为参数把"运行时分支"变成"编译期重载选择"。在 SFINAE / Concepts / if constexpr 出现前，这是分派算法实现的主力。

典型示例：根据迭代器类别选最优实现

// 标签类型（标准库的迭代器分类）
struct input_iterator_tag          {};
struct forward_iterator_tag        : input_iterator_tag {};
struct bidirectional_iterator_tag  : forward_iterator_tag {};
struct random_access_iterator_tag  : bidirectional_iterator_tag {};

// 分派实现
template <class Iter>
auto distance_impl(Iter a, Iter b, std::input_iterator_tag) {
    // O(N)：只能 ++ 遍历
    typename std::iterator_traits<Iter>::difference_type n = 0;
    while (a != b) { ++a; ++n; }
    return n;
}
template <class Iter>
auto distance_impl(Iter a, Iter b, std::random_access_iterator_tag) {
    // O(1)：直接相减
    return b - a;
}

// 公共接口
template <class Iter>
auto my_distance(Iter a, Iter b) {
    return distance_impl(a, b,
        typename std::iterator_traits<Iter>::iterator_category{});
}

Tag dispatch vs if constexpr vs Concepts —— 三方对比

项Tag Dispatchif constexpr (C++17)Concepts (C++20)
语法清晰中★★★★★★★★★
错误信息差（重载失败）中★★★★★（一句话）
支持继承层次（兼容性）★★★★★（标签继承）不直接支持concept refinement
能否 ADL 扩展★★★★★（用户可定义新 tag）不能concept 可特化
最低标准C++98C++17C++20
什么时候仍该用 tag dispatch：① 库代码要兼容老编译器；② 需要第三方用户为自家类型定义"该类型属于哪个 tag" → tag 系统是 ABI 稳定的扩展点；③ iterator_traits、allocator_traits 等标准库基础设施就是这种模式。

9.10 Detection Idiom（void_t / is_detected）

Detection idiom：编译期检测"类型 T 是否有成员/操作 X"。比 SFINAE 重载更直接、更可读。

核心机制：std::void_t（C++17）

template <class...>
using void_t = void;          // "把任何类型映射到 void" 的工具

// 检测 T 是否有嵌套 type — primary template (false)
template <class T, class = void>
struct has_type : std::false_type {};

// 偏特化：T::type 有效时匹配
template <class T>
struct has_type<T, std::void_t<typename T::type>> : std::true_type {};

static_assert( has_type<std::vector<int>>::value);  // ✅
static_assert(!has_type<int>::value);                // ✅

通用 is_detected 模板（基于 void_t）

template <class Default, class AlwaysVoid,
          template<class...> class Op, class... Args>
struct detector { using value_t = std::false_type; using type = Default; };

template <class Default,
          template<class...> class Op, class... Args>
struct detector<Default, std::void_t<Op<Args...>>, Op, Args...>
    { using value_t = std::true_type; using type = Op<Args...>; };

template <template<class...> class Op, class... Args>
using is_detected = typename detector<void, void, Op, Args...>::value_t;

// 使用：检测 T 是否能调 .serialize()
template <class T>
using serialize_t = decltype(std::declval<T>().serialize());

template <class T>
constexpr bool has_serialize = is_detected<serialize_t, T>::value;

C++20 之后：Concepts 一句话搞定

template <class T>
concept Serializable = requires(T t) { t.serialize(); };

static_assert(Serializable<MyClass>);
历史地位：detection idiom 是 C++11/14/17 时代实现"类型 X 是否支持 Y"检查的工业标准。WG21 的 LFTSv2 把 std::experimental::is_detected 标准化了一段时间，但 C++20 起 Concepts 取代了大部分场景。读老代码 / 维护 boost 风格库时必备。

9.11 CTAD Deduction Guides —— 类模板参数推导

C++17 引入 类模板参数推导（CTAD），std::pair p{1, "abc"}; 不用再写 std::pair<int, const char*>。背后机制是推导指引 (deduction guide)。

编译器自动推导（隐式 guide）

template <class T>
struct Wrapper {
    T v;
    Wrapper(T x) : v(std::move(x)) {}
};

Wrapper w1{42};              // ✅ T 推导为 int
Wrapper w2{"hello"};         // ✅ T 推导为 const char* (可能不是你想要)

显式 Deduction Guide 接管推导

template <class T>
struct Wrapper { ... };

// 把所有 const char* 推导成 std::string
Wrapper(const char*) -> Wrapper<std::string>;

Wrapper w{"hello"};           // ✅ Wrapper<std::string>

真实库案例 —— std::pair 与 std::array

// 标准库 std::pair 的 deduction guide（简化）
template<class T1, class T2>
pair(T1, T2) -> pair<T1, T2>;

// std::array 的 deduction guide — 通过 fold 推导大小
template<class T, class... U>
array(T, U...) -> array<T, 1 + sizeof...(U)>;

std::array a{1,2,3,4};      // std::array<int, 4>

典型用法 —— 让自家类型支持 CTAD

// 锁守卫 — 让 std::lock_guard{mu} 直接工作
template <class M>
class LockGuard {
    M& mu_;
public:
    explicit LockGuard(M& m) : mu_(m) { mu_.lock(); }
    ~LockGuard() { mu_.unlock(); }
};
LockGuard(std::mutex&) -> LockGuard<std::mutex>;       // 必要时显式

std::mutex m;
LockGuard g{m};                // ✅ 自动推导 LockGuard<std::mutex>

// 异构容器 — 推导出 variant
template <class... Ts>
struct Overloaded : Ts... { using Ts::operator()...; };
template <class... Ts>
Overloaded(Ts...) -> Overloaded<Ts...>;     // ★ 让 visitor 写法成立

std::visit(Overloaded{
    [](int i)    { ... },
    [](std::string s) { ... }
}, var);

陷阱：聚合类型 + CTAD

template <class T>
struct Box { T v; };
Box b1{42};                     // C++17: ❌ 编译错；C++20: ✅ 聚合 CTAD

// C++17 的 fix：
template <class T> Box(T) -> Box<T>;
建议：① 不要过度依赖隐式 guide，遇到字符串/容器歧义就显式写 guide；② 模板类公开接口若想被用户类型自然使用，主动提供 deduction guide；③ C++20 起聚合类型有自动 guide，但显式写更稳。

class vector_base {

9.13 高频面试题 Q&A

Q1: 模板代码为什么必须放在头文件？

编译器在实例化时需要看到模板的完整定义。C++ 编译是逐 TU 独立进行的，链接期不能再生成代码。如果定义放 .cpp 里，其他 TU 看不到就无法实例化 → 链接错误。

例外：使用 explicit instantiation 在 .cpp 内预先实例化所有用到的类型，可以把定义留在 .cpp — 常用于减少编译依赖与编译时间。

Q2: typename 和 class 在模板参数里的区别？

作为模板参数声明时，二者完全等价：template<class T> ≡ template<typename T>。

但在模板体内，必须用 typename 指示"这是一个依赖类型"：

template<typename T>
void f() {
    typename T::iterator it;  // 必须加 typename！
    // 否则编译器以为 T::iterator 是静态成员
}

C++20 起，许多"明显是类型"的场合可省略 typename。

Q3: 两阶段名字查找（Two-Phase Name Lookup）是什么？

① 第一阶段（模板定义时）：查找非依赖名字（与模板参数无关的）。
② 第二阶段（实例化时）：查找依赖名字（涉及模板参数的）——结合 ADL 参数依赖查找。

template <typename T>
void f(T t) {
    g(42);    // 第一阶段：必须此时可见的 g
    g(t);     // 第二阶段：实例化时根据 T 的命名空间再查
}

MSVC 传统上不严格实施两阶段查找（更宽容），导致跨编译器移植时会暴露问题 — 面试常考。

Q4: 为什么函数模板不能偏特化？

语言设计决定：函数偏特化会与函数重载冲突，二者的决议规则不同（偏特化按特殊性，重载按参数匹配）。标准委员会选择让函数重载承担这个角色。

替代方案：

  • 用函数重载（最常见）
  • 用类模板偏特化 + 静态成员函数包装
  • 用 if constexpr (C++17) 在函数内分派
  • 用 Concepts 约束 (C++20)
Q5: 万能引用（Forwarding Reference）与右值引用的区别？ 
void f(T&& x);               // 右值引用：T 是具体类型
template<typename T>
void g(T&& x);               // 万能引用！因为 T 是模板参数

auto&& x = foo();             // 万能引用 (auto&&)

关键是引用折叠规则：T& && → T&，T&& && → T&&。

  • 传左值 → T 被推导为 T& → 参数类型 T&
  • 传右值 → T 被推导为 T → 参数类型 T&&

必须配合 std::forward<T>(x) 才能正确转发。用 std::move 转发万能引用会把左值变右值 — 高危错误！

Q6: std::vector<bool> 为什么被称为"C++ 的污点"？

它是 vector 的全特化，为节省空间用位压缩（1 个 bit 存 1 个 bool）。代价：

  • operator[] 返回的是代理对象 reference，不是真正的 bool&
  • 无法取 bool* 地址：&v[0] 编译错误
  • 不满足 Container 接口概念 → 泛型代码失效

替代：std::deque<bool>、std::vector<char>、std::bitset、std::array<bool, N>。

Q7: if constexpr 和普通 if 的区别？

关键差异：if constexpr 在编译期选择分支，未选中的分支不参与模板实例化。

template<typename T>
auto getValue(T t) {
    if constexpr (std::is_pointer_v<T>) {
        return *t;       // 非指针类型时此分支不编译
    } else {
        return t;
    }
}
// 普通 if 的话，两个分支都要编译 → *int 直接报错

替代了 C++17 前的 tag dispatch 和 SFINAE 方案，代码可读性飞跃。

Q8: 模板编译慢怎么办？（大厂面试重点）
  1. 前置声明 + PImpl — 减少头文件依赖
  2. extern template — 避免重复实例化
  3. 显式实例化 — 把模板定义搬进 .cpp
  4. 减少递归深度 — 用 fold expression、pack indexing 替代递归模板
  5. 预编译头 (PCH) — GCC -include、MSVC /Yc /Yu
  6. Module (C++20) — 取代头文件的未来方案
  7. 分布式编译 — distcc、icecream、sccache
  8. unity build — 合并多个 cpp 成一个编译单元
