1. 模板 & 泛型编程概览

模板（Template） 是 C++ 实现 泛型编程（Generic Programming） 的核心机制。它让你编写与类型无关的代码，由编译器根据使用时的具体类型自动生成对应版本。

模板的核心价值

特性说明项目中的体现

2. 函数模板 (Function Template)

函数模板允许你编写一个函数"蓝图"，编译器在调用时根据参数自动推导类型并生成具体函数。

编写模板
template<typename T>
→
调用时
serialize(myInt, payload)
→
编译器推导
T = int
→
实例化
生成 serialize<int>

项目实例 1：序列化/反序列化 MessageHelpers.hpp

// src/Core/proxy/MessageHelpers.hpp
template <typename T>
inline void serialize(const T& var, Payload& payload)
{
    payload.insert(payload.end(),
        reinterpret_cast<const char*>(&var),
        reinterpret_cast<const char*>(&var) + sizeof(T));
}

template <typename T>
inline void deserialize(T& var, Reader& reader)
{
    memcpy(&var, &*reader, sizeof(T));
    reader += sizeof(T);
}

📌 解读：serialize<T> 和 deserialize<T> 不关心 T 是 int、uint32_t 还是自定义结构体——只要它是 POD 类型就可以直接 memcpy。编译器会针对每种调用类型生成独立版本。

项目实例 2：线性插值 VoltageScenario

// src/NetconfAgent/Scenario/VoltageManagement/ConfigureInputVoltageThresholdsScenario.hpp
template <typename T>
T linearInterpolate(T inputValue, T inputMin, T inputMax,
                     T outputMin, T outputMax)
{
    if (inputValue <= inputMin) return outputMin;
    if (inputValue >= inputMax) return outputMax;

    return outputMin + (inputValue - inputMin)
           * (outputMax - outputMin) / (inputMax - inputMin);
}

📌 这个函数可以同时处理 double、float、int 等任意数值类型，一份代码覆盖所有场景。

3. 类模板 (Class Template)

类模板允许你参数化整个类的行为和数据类型。STL 中的 std::vector<T>、std::map<K,V> 都是类模板。

项目实例 1：线程安全封装 ReadWriteLockedValue<T>

// src/Core/ReadWriteLockedValue.hpp
template <typename T>
class ReadWriteLockedValue
{
public:
    explicit ReadWriteLockedValue(const T& value = T()) : value_(value) {}

    T get() const {
        std::shared_lock<std::shared_mutex> lock(mutex_);
        return value_;
    }

    void set(T&& value) {
        std::unique_lock<std::shared_mutex> lock(mutex_);
        value_ = std::forward<T>(value);
    }

    // 嵌套的成员函数模板 —— 接受任意可调用对象
    template <typename Func>
    auto read(Func func) const {
        std::shared_lock<std::shared_mutex> lock(mutex_);
        return func(value_);
    }

    template <typename Func>
    auto write(Func func) {
        std::unique_lock<std::shared_mutex> lock(mutex_);
        return func(value_);
    }

private:
    mutable std::shared_mutex mutex_;
    T value_;
};

📌 设计亮点：

  • T 作为类模板参数，封装了任意类型的线程安全读写
  • read(Func) 和 write(Func) 是成员函数模板——类模板中再嵌套函数模板，双重泛型
  • 使用 std::shared_mutex 实现读写分离锁策略
  • 使用 std::forward 实现完美转发

项目实例 2：消息适配器 Adapter<T>

// src/Core/proxy/Adapter.hpp
template <typename TMessageType>
struct Adapter : public TMessageType  // ← 继承自模板参数！
{
    std::string getName() const;
    std::string toString() const;

    void to(Payload& payload) const {
        static_assert(sizeof(TMessageType) == sizeof(Adapter<TMessageType>),
            "Default Adapter assume const size message");
        payload.resize(sizeof(TMessageType));
        memcpy(&payload.front(), this, sizeof(TMessageType));
    }

    void from(const Payload& payload) {
        static_assert(sizeof(TMessageType) == sizeof(Adapter<TMessageType>),
            "Default Adapter assume const size message");
        memcpy(this, &payload.front(), sizeof(TMessageType));
    }
};

📌 CRTP-like 模式：Adapter<T> 直接继承 T——这意味着 Adapter<SApiInetVsftpStopReq> 既是一个 SApiInetVsftpStopReq，又具备序列化/反序列化的能力。默认实现用 memcpy，特化版本可以自定义。

4. 模板特化 (Template Specialization)

模板特化允许你为特定类型提供不同的实现。分为全特化（完全指定类型）和偏特化（部分指定类型）。

4.1 全特化：MessageInfo SApiInetVsftpStop.hpp

// 主模板声明（永远不会直接使用，强制使用特化版本）
template <typename TMsg>
struct MessageInfo;  // 只有声明，没有定义！

// 全特化：为 SApiInetVsftpStopReq 提供完整定义
template <>
struct MessageInfo<::SApiInetVsftpStopReq>
{
    enum { RequestId  = API_INET_VSFTP_STOP_REQ_MSG };
    enum { ResponseId = API_INET_VSFTP_STOP_RESP_MSG };
    using RequestMsg  = ::SApiInetVsftpStopReq;
    using ResponseMsg = ::SApiInetVsftpStopResp;
    enum { DefaultTimeoutDuration = 30 };
    static const TaskId Task = HWAPI_IP_SERVICE;
};

📌 精妙设计：主模板只声明不定义。如果有人对一个没有特化的消息类型使用 MessageInfo，编译器会报错「不完整类型」——这是一个编译期约束，确保每种消息都必须提供配置信息。

4.2 全特化：Adapter 自定义序列化

// 对 VsftpStopReq 全特化 Adapter，自定义序列化逻辑
template <>
struct Adapter<::SApiInetVsftpStopReq> : public ::SApiInetVsftpStopReq
{
    TTransactionID transactionId;

    void to(Payload& payload) const {
        payload.reserve(sizeof(::SApiInetVsftpStopReq));
        serialize(transactionId, payload);  // 只序列化 transactionId
    }

    void from(const Payload& payload) {
        auto reader = payload.begin();
        deserialize(transactionId, reader);
    }
};

📌 默认 Adapter<T> 用 memcpy 整体拷贝；但 VsftpStop 消息的特化版只序列化 transactionId。这就是全特化的威力——同一接口，不同实现。

// src/Core/FmService/Faults/Fault2930Handler.cpp

    {
        faultPreprocessor_->preprocessFault<lim::MoCablink_r>(...);

    else

📌 if constexpr（C++17）在编译期决定走哪个分支——不满足条件的分支代码完全不会被编译。结合 static_assert 可以在编译期捕获不支持的类型。

8. 模板最佳实践

✅ DO

  • 使用 static_assert 提供友好的编译期错误信息（如 MakeMsg 中的 Req/Resp 校验）
  • 使用 if constexpr 替代 SFINAE 实现编译期分支（C++17，如 Fault2930Handler）
  • 使用显式实例化 当类型集合有限且希望隐藏实现时（如 convertValue）
  • 使用类型约束（C++20 Concepts 或 static_assert）限制模板可接受的类型
  • 保持模板代码简短，将复杂逻辑提取为非模板辅助函数

❌ DON'T

  • 不要把所有模板定义放在 .cpp 中却忘了显式实例化 → 链接错误
  • 不要过度使用模板——如果只有一两个类型，直接重载可能更清晰
  • 不要忽略编译器的模板错误信息——从最后一个实例化点开始向上阅读
  • 避免模板嵌套过深导致编译时间爆炸和难以阅读的错误信息

9. 面试深度专题 ⭐ Senior C++

本章节面向高级 C++ 工程师面试，聚焦模板的底层机制、常见陷阱、现代 C++ 演进，以及大厂高频考点。

9.1 SFINAE — 模板元编程的灵魂

SFINAE（Substitution Failure Is Not An Error）：模板参数替换失败时，编译器不会报错，只会把该重载从候选集中默默移除。这是实现"编译期 if"和类型检查的核心。

// 经典 SFINAE：检测类型是否有 serialize() 成员
template <typename T>
auto hasSerialize(int) -> decltype(std::declval<T>().serialize(), std::true_type{});

template <typename T>
auto hasSerialize(...) -> std::false_type;

template <typename T>
constexpr bool has_serialize_v = decltype(hasSerialize<T>(0))::value;

// 使用 std::enable_if 启用/禁用重载
template <typename T, std::enable_if_t<std::is_integral_v<T>, int> = 0>
T safe_add(T a, T b) { return a + b; }     // 仅整型

template <typename T, std::enable_if_t<std::is_floating_point_v<T>, int> = 0>
T safe_add(T a, T b) { return a + b + 1e-9; } // 浮点

9.2 Concepts (C++20) — SFINAE 的现代替代

C++20 引入 Concepts，让模板约束像函数签名一样清晰，错误信息可读性大幅提升。

❌ SFINAE 年代

template <typename T,
  std::enable_if_t<
    std::is_integral_v<T> &&
    !std::is_same_v<T,bool>,
    int> = 0>
T next(T x) { return ++x; }

// 错误信息: 长达几十行
// "no matching function..."

✅ C++20 Concepts

template <typename T>
concept Integer =
    std::integral<T> &&
    !std::same_as<T,bool>;

template <Integer T>
T next(T x) { return ++x; }

// 错误信息：清晰一句话
// "T does not satisfy Integer"

9.3 CRTP — 静态多态模式 零开销抽象

CRTP（Curiously Recurring Template Pattern）：基类模板以派生类自身为模板参数。用静态绑定替代虚函数，既复用代码又避免虚表开销。

template <typename Derived>
class Shape {
public:
    double area() const {
        // 静态下转型 — 编译期解析，无虚表查找
        return static_cast<const Derived*>(this)->areaImpl();
    }
};

class Circle : public Shape<Circle> {
    double r_;
public:
    double areaImpl() const { return 3.14 * r_ * r_; }
};

// 对比: 虚函数每次调用有间接跳转 + 阻碍内联
// CRTP 可完全内联 → 性能与手写无差别

典型应用：std::enable_shared_from_this、Boost.Iterator、Eigen 表达式模板

9.4 Type Erasure — 运行时泛型

类型擦除：在编译期不暴露具体类型，运行期通过"虚接口 + 模板包装"统一处理。标准库典范：std::function、std::any、std::shared_ptr 的 deleter。

// 简化版 std::function 实现
template <typename Sig> class Function;

template <typename R, typename... Args>
class Function<R(Args...)> {
    struct Concept {                              // 内部虚接口
        virtual R call(Args...) = 0;
        virtual ~Concept() = default;
    };
    template <typename F>
    struct Model : Concept {                       // 模板实现
        F f_;
        Model(F f) : f_(std::move(f)) {}
        R call(Args... a) override { return f_(std::forward<Args>(a)...); }
    };
    std::unique_ptr<Concept> impl_;
public:
    template <typename F>
    Function(F f) : impl_(std::make_unique<Model<F>>(std::move(f))) {}
    R operator()(Args... a) { return impl_->call(std::forward<Args>(a)...); }
};

9.5 变长模板 & Fold Expressions (C++17)

// ① 递归展开 (C++11 风格)
void print() {}
template <typename T, typename... Rest>
void print(const T& t, const Rest&... rest) {
    std::cout << t << ' ';
    print(rest...);   // 递归 → 需要终止函数
}

// ② 折叠表达式 (C++17) — 一行搞定
template <typename... Args>
void print(const Args&... args) {
    ((std::cout << args << ' '), ...);   // 逗号折叠
}

// ③ 四种折叠方向
(args + ...)        // 一元右折叠:  a1 + (a2 + (a3 + a4))
(... + args)        // 一元左折叠:  ((a1 + a2) + a3) + a4
(args + ... + 0)    // 二元右折叠:  带初值，支持空包
(0 + ... + args)    // 二元左折叠

// ④ 完美转发多参数
template <typename... Args>
auto makeVector(Args&&... args) {
    std::vector<std::common_type_t<Args...>> v;
    (v.push_back(std::forward<Args>(args)), ...);
    return v;
}

9.7 高频面试题 Q&A

Q1: 模板代码为什么必须放在头文件？

编译器在实例化时需要看到模板的完整定义。C++ 编译是逐 TU 独立进行的，链接期不能再生成代码。如果定义放 .cpp 里，其他 TU 看不到就无法实例化 → 链接错误。

例外：使用 explicit instantiation 在 .cpp 内预先实例化所有用到的类型，可以把定义留在 .cpp — 常用于减少编译依赖与编译时间。

Q2: typename 和 class 在模板参数里的区别？

作为模板参数声明时，二者完全等价：template<class T> ≡ template<typename T>。

但在模板体内，必须用 typename 指示"这是一个依赖类型"：

template<typename T>
void f() {
    typename T::iterator it;  // 必须加 typename！
    // 否则编译器以为 T::iterator 是静态成员
}

C++20 起，许多"明显是类型"的场合可省略 typename。

Q3: 两阶段名字查找（Two-Phase Name Lookup）是什么？

① 第一阶段（模板定义时）：查找非依赖名字（与模板参数无关的）。
② 第二阶段（实例化时）：查找依赖名字（涉及模板参数的）——结合 ADL 参数依赖查找。

template <typename T>
void f(T t) {
    g(42);    // 第一阶段：必须此时可见的 g
    g(t);     // 第二阶段：实例化时根据 T 的命名空间再查
}

MSVC 传统上不严格实施两阶段查找（更宽容），导致跨编译器移植时会暴露问题 — 面试常考。

Q4: 为什么函数模板不能偏特化？

语言设计决定：函数偏特化会与函数重载冲突，二者的决议规则不同（偏特化按特殊性，重载按参数匹配）。标准委员会选择让函数重载承担这个角色。

替代方案：

  • 用函数重载（最常见）
  • 用类模板偏特化 + 静态成员函数包装
  • 用 if constexpr (C++17) 在函数内分派
  • 用 Concepts 约束 (C++20)
Q5: 万能引用（Forwarding Reference）与右值引用的区别？ 
void f(T&& x);               // 右值引用：T 是具体类型
template<typename T>
void g(T&& x);               // 万能引用！因为 T 是模板参数

auto&& x = foo();             // 万能引用 (auto&&)

关键是引用折叠规则：T& && → T&，T&& && → T&&。

  • 传左值 → T 被推导为 T& → 参数类型 T&
  • 传右值 → T 被推导为 T → 参数类型 T&&

必须配合 std::forward<T>(x) 才能正确转发。用 std::move 转发万能引用会把左值变右值 — 高危错误！

Q6: std::vector<bool> 为什么被称为"C++ 的污点"？

它是 vector 的全特化，为节省空间用位压缩（1 个 bit 存 1 个 bool）。代价：

  • operator[] 返回的是代理对象 reference，不是真正的 bool&
  • 无法取 bool* 地址：&v[0] 编译错误
  • 不满足 Container 接口概念 → 泛型代码失效

替代：std::deque<bool>、std::vector<char>、std::bitset、std::array<bool, N>。

Q7: if constexpr 和普通 if 的区别？

关键差异：if constexpr 在编译期选择分支，未选中的分支不参与模板实例化。

template<typename T>
auto getValue(T t) {
    if constexpr (std::is_pointer_v<T>) {
        return *t;       // 非指针类型时此分支不编译
    } else {
        return t;
    }
}
// 普通 if 的话，两个分支都要编译 → *int 直接报错

替代了 C++17 前的 tag dispatch 和 SFINAE 方案，代码可读性飞跃。

Q8: 模板编译慢怎么办？（大厂面试重点）
  1. 前置声明 + PImpl — 减少头文件依赖
  2. extern template — 避免重复实例化
  3. 显式实例化 — 把模板定义搬进 .cpp
  4. 减少递归深度 — 用 fold expression、pack indexing 替代递归模板
  5. 预编译头 (PCH) — GCC -include、MSVC /Yc /Yu
  6. Module (C++20) — 取代头文件的未来方案
  7. 分布式编译 — distcc、icecream、sccache
  8. unity build — 合并多个 cpp 成一个编译单元
